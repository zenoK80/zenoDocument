# 🗄️ SQL 기본 명령어와 내장 SQL — 데이터베이스 핵심 개념 완전 정복

데이터베이스를 처음 배우는 분들이라면 "SQL이 뭔지는 알겠는데, 정작 시험 문제나 실무에서 보면 헷갈린다"는 경험을 자주 하게 됩니다. 이 문서에서는 SQL의 기본 명령어부터 내장 SQL, 관계형 연산, DBMS 구성 요소, 파일 시스템, RAID, 데이터베이스 특성까지 — 시험에 자주 등장하는 핵심 개념들을 초보자도 완벽히 이해할 수 있도록 하나씩 풀어드립니다.

---

## 🔑 1. SQL 명령어 분류 — DDL, DML, DCL이란?

### SQL 명령어는 왜 분류할까요?

SQL(Structured Query Language, 구조화 질의 언어)은 데이터베이스를 다루는 언어입니다. 하지만 데이터베이스를 다루는 방식은 크게 세 가지로 나뉩니다. **테이블을 만들고 구조를 정의**하는 일, **데이터를 조회·삽입·수정·삭제**하는 일, 그리고 **권한을 부여하고 제어**하는 일입니다. 이 세 가지 역할을 각각 DDL, DML, DCL로 분류합니다. 마치 식당을 운영할 때 "가게 자체를 짓고 리모델링하는 일(DDL)", "손님에게 음식을 주문받고 서빙하는 일(DML)", "직원 출입 권한을 관리하는 일(DCL)"로 역할을 나누는 것과 비슷합니다.

### DDL (Data Definition Language, 데이터 정의어)

DDL은 데이터베이스의 **구조(스키마)**를 정의하거나 변경하는 명령어입니다. 테이블을 새로 만들고(`CREATE`), 구조를 바꾸고(`ALTER`), 테이블 자체를 삭제(`DROP`)하는 등의 작업이 여기에 해당합니다.

| 명령어 | 설명 | 예시 |
|--------|------|------|
| `CREATE` | 테이블, 뷰, 인덱스 등 새로운 객체를 생성 | `CREATE TABLE 학생(...)` |
| `ALTER` | 기존 객체의 구조를 변경 | `ALTER TABLE 학생 ADD 학년 INT` |
| `DROP` | 기존 객체를 삭제 | `DROP TABLE 학생` |
| `TRUNCATE` | 테이블의 모든 데이터를 삭제(구조는 유지) | `TRUNCATE TABLE 학생` |

:::warning DDL과 DML 혼동 주의
시험에서 자주 나오는 함정 문제입니다. **`SELECT`는 DDL이 아니라 DML**입니다! `SELECT`는 데이터를 **조회**하는 명령어이므로 데이터 조작어(DML)에 해당합니다. DDL은 테이블의 **구조**를 다루는 명령어임을 꼭 기억하세요.
:::

> ✅ **핵심 포인트**: `SELECT`는 DDL이 아닙니다. DDL에는 `CREATE`, `ALTER`, `DROP`이 포함되며, `SELECT`는 DML(데이터 조작어)에 해당합니다.

```sql
-- ❌ 잘못된 이해: SELECT를 DDL로 분류하는 경우
-- DDL: CREATE, ALTER, DROP, SELECT (X)

-- ✅ 올바른 분류
-- DDL (데이터 정의어): CREATE, ALTER, DROP
CREATE TABLE 직원 (
    직원번호 INT PRIMARY KEY,   -- 기본키 설정
    이름 VARCHAR(50),            -- 문자열 타입 컬럼
    부서 VARCHAR(30)             -- 부서 컬럼
);

-- DML (데이터 조작어): SELECT, INSERT, UPDATE, DELETE
SELECT * FROM 직원;             -- 데이터 조회 (DML)
INSERT INTO 직원 VALUES (1, '홍길동', '개발팀');  -- 데이터 삽입 (DML)
```

### DML (Data Manipulation Language, 데이터 조작어)

DML은 실제 데이터를 다루는 명령어입니다. 데이터를 조회하고(`SELECT`), 새 데이터를 삽입하고(`INSERT`), 기존 데이터를 수정하고(`UPDATE`), 데이터를 삭제(`DELETE`)합니다. 마치 엑셀 파일에서 내용을 읽고 쓰고 지우는 행위와 유사합니다.

## 🔍 2. NULL 값 처리 — IS NULL vs = NULL

### NULL이란 무엇인가요?

**NULL**은 "값이 없음"을 의미하는 특수한 상태입니다. 0이나 빈 문자열('')과는 다릅니다. 0은 숫자 0이라는 값이 존재하는 것이고, 빈 문자열은 길이가 0인 문자열이라는 값이 있는 것입니다. NULL은 아예 **값 자체가 존재하지 않는** 상태입니다. 마치 시험지에 답을 쓰지 않은 것과 같습니다. 0점(0)을 받은 것도 아니고, 공백을 쓴 것도 아니라 — 아예 응시하지 않은(NULL) 상태인 것입니다.

### NULL 값을 검색할 때 주의사항

NULL을 비교할 때는 일반적인 등호(`=`)를 사용할 수 없습니다. NULL은 "알 수 없는 값"이기 때문에, NULL = NULL조차도 TRUE가 되지 않습니다. 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용해야 합니다.

```sql
-- ❌ 잘못된 방법: NULL을 = 로 비교 (결과가 항상 빈 결과 반환)
SELECT * FROM myTable WHERE column1 = NULL;
-- NULL은 어떤 값과도 "=" 로 비교할 수 없습니다
-- 심지어 NULL = NULL도 FALSE 입니다

-- ❌ 잘못된 방법: 존재하지 않는 EQUALS 키워드 사용
SELECT * FROM myTable WHERE column1 EQUALS NULL;
-- SQL에는 EQUALS 라는 키워드가 없습니다

-- ❌ 잘못된 방법: NOT NULL을 조건절에 잘못 사용
SELECT * FROM myTable WHERE column1 NOT NULL;
-- 이 문법은 테이블 정의(DDL)에서 제약조건으로 사용하는 것입니다

-- ✅ 올바른 방법: IS NULL 사용
SELECT * FROM myTable WHERE column1 IS NULL;
-- column1의 값이 NULL인 행만 반환합니다

-- ✅ NULL이 아닌 값 찾기
SELECT * FROM myTable WHERE column1 IS NOT NULL;
-- column1에 값이 존재하는 행만 반환합니다
```

:::danger NULL 비교 오류
`WHERE column1 = NULL`은 SQL에서 **절대로 원하는 결과를 반환하지 않습니다**. NULL은 등호(`=`)로 비교할 수 없으며, 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용해야 합니다. 이 실수는 실무에서도 매우 자주 발생하는 버그의 원인입니다.
:::

## 🧹 3. DISTINCT — 중복 제거의 마법

### 중복 제거가 왜 필요한가요?

데이터베이스에서 데이터를 조회하다 보면 같은 값이 여러 행에 반복적으로 나타나는 경우가 많습니다. 예를 들어 수천 명의 직원 데이터에서 "어떤 부서들이 있는지"만 보고 싶을 때, 같은 부서명이 수십~수백 번씩 반복되어 나타나면 확인하기가 매우 불편합니다. 이럴 때 **중복된 행을 하나만 남기고 제거**해주는 것이 `DISTINCT`입니다.

### DISTINCT vs 다른 명령어 비교

시험에서 "중복을 제거하는 SQL 명령어"를 묻는 문제가 자주 출제됩니다. 비슷해 보이는 명령어들과의 차이를 명확히 알아야 합니다.

| 명령어 | 역할 | 중복 제거 여부 |
|--------|------|----------------|
| `DISTINCT` | 결과에서 중복된 행 제거 | ✅ 예 |
| `ALL` | 중복 포함 모든 행 반환 (기본값) | ❌ 아니오 |
| `ORDER BY` | 결과를 특정 컬럼 기준으로 정렬 | ❌ 아니오 |
| `GROUP BY` | 특정 컬럼 기준으로 그룹화(집계 함수와 함께 사용) | 부분적으로 |

```sql
-- 예시 테이블: 직원 테이블
-- | 이름   | 부서   |
-- | 홍길동 | 개발팀 |
-- | 김철수 | 개발팀 |
-- | 이영희 | 마케팅 |
-- | 박민준 | 개발팀 |
-- | 최수진 | 마케팅 |

-- ❌ ALL 사용 시 (기본값) - 중복 포함
SELECT ALL 부서 FROM 직원;
-- 결과: 개발팀, 개발팀, 마케팅, 개발팀, 마케팅 (중복 있음)

-- ✅ DISTINCT 사용 시 - 중복 제거
SELECT DISTINCT 부서 FROM 직원;
-- 결과: 개발팀, 마케팅 (중복 제거됨)

-- ORDER BY는 정렬만 할 뿐 중복을 제거하지 않음
SELECT 부서 FROM 직원 ORDER BY 부서;
-- 결과: 개발팀, 개발팀, 개발팀, 마케팅, 마케팅 (정렬만 됨)

-- GROUP BY는 집계를 위한 그룹화이지 단순 중복 제거가 아님
SELECT 부서, COUNT(*) FROM 직원 GROUP BY 부서;
-- 결과: 개발팀 3명, 마케팅 2명 (집계 목적)
```

:::tip DISTINCT 활용 팁
`DISTINCT`는 `SELECT` 바로 뒤에 한 번만 작성합니다. 여러 컬럼에 `DISTINCT`를 적용하면 **해당 컬럼들의 조합**이 중복되는 행을 제거합니다. 즉, `SELECT DISTINCT 부서, 직급 FROM 직원`은 부서와 직급의 조합이 같은 행들만 제거합니다.
:::

## 🏗️ 4. 관계형 연산 — 조인, 디비전, 합병 가능 연산

### 4-1. 조인(Join) — 두 테이블을 연결하는 방법

#### 조인이 왜 필요한가요?

데이터베이스에서는 데이터를 효율적으로 관리하기 위해 여러 테이블로 나누어 저장합니다. 마치 도서관에서 책 정보는 책 목록에, 대출 정보는 대출 장부에 따로 관리하는 것처럼요. 이때 "누가 어떤 책을 빌렸는지" 알려면 책 목록과 대출 장부를 **연결해서 조회**해야 하는데, 이 연결 작업이 바로 **조인(Join)**입니다.

#### 내추럴 조인(Natural Join) — 중복 속성을 자동으로 제거

**내추럴 조인(Natural Join)**은 두 릴레이션(테이블)에서 **동일한 이름을 가진 속성(컬럼)**을 기준으로 자동으로 조인하고, **중복되는 속성을 하나만 남기고 제거**하는 조인 방법입니다. 일반적인 동등 조인(Equi Join)은 두 테이블의 공통 컬럼이 결과에 두 번 나타나는데, 내추럴 조인은 이 중복을 자동으로 제거해줍니다.

> 🎯 **시험 포인트**: "두 개 이상의 릴레이션에서 동일 조인의 결과 중 중복되는 속성을 하나만 남기고 제거하는 조인"은 **내추럴 조인(Natural Join)**입니다.

```sql
-- 예시: 학생 테이블과 수강 테이블을 조인

-- 학생 테이블
-- | 학번 | 이름   | 학과   |
-- | 101  | 홍길동 | 컴퓨터 |
-- | 102  | 김철수 | 전기   |

-- 수강 테이블
-- | 학번 | 과목명   | 학점 |
-- | 101  | 데이터베이스 | A |
-- | 101  | 알고리즘     | B |

-- ❌ 동등 조인(Equi Join) - 학번 컬럼이 두 번 나타남
SELECT * FROM 학생, 수강 WHERE 학생.학번 = 수강.학번;
-- 결과: | 학생.학번 | 이름 | 학과 | 수강.학번 | 과목명 | 학점 |
--        학번이 두 번 나타남!

-- ✅ 내추럴 조인(Natural Join) - 중복 속성 자동 제거
SELECT * FROM 학생 NATURAL JOIN 수강;
-- 결과: | 학번 | 이름 | 학과 | 과목명 | 학점 |
--        학번이 한 번만 나타남 (중복 제거됨)
```

### 4-2. 디비전(Division, ÷) — 나누기 연산

#### 디비전 연산이란?

디비전(Division) 연산은 관계 대수에서 "S의 모든 조건을 만족하는 R의 튜플"을 구하는 연산입니다. 예를 들어 "모든 과목을 수강한 학생"을 구하는 경우처럼, 어떤 집합의 **모든 원소와 관련된 데이터**를 찾을 때 사용합니다.

**디비전 연산의 핵심 특성**을 정리하면 다음과 같습니다:

- **차수(Degree, 속성의 수)**: 나누어지는 릴레이션 R의 차수가 (m+n)이고, 나누는 릴레이션 S의 차수가 n이면, 결과 릴레이션의 차수는 **m**이 됩니다.
- **공통 속성**: 나누어지는 R은 나누는 S의 **모든 속성(Y)**을 포함해야 합니다.
- **도메인(Domain, 값의 범위)**: 공통 속성은 이름이 같은 것이 아니라, **대응되는 속성끼리 도메인(값의 범위)이 같아야** 합니다.
- **결과의 의미**: 결과 릴레이션의 각 튜플은 S의 모든 튜플과 결합했을 때, 그 결합 결과가 모두 R에 존재해야 합니다.

:::warning 디비전 연산 오해 주의
시험에서 자주 나오는 함정이 있습니다. "공통되는 속성은 이름이 같아야 한다"는 설명은 **틀린 설명**입니다. 디비전 연산에서 중요한 것은 속성의 **이름이 같은 것**이 아니라, **도메인(값의 범위)이 같아야** 한다는 것입니다. 대응되는 속성들은 이름이 달라도 도메인만 같으면 됩니다.
:::

### 4-3. 합병 가능(Union Compatible) 연산

**합병 가능(Union Compatible)**이란 두 릴레이션이 집합 연산(합집합, 교집합, 차집합)을 수행하기 위해 만족해야 하는 조건입니다. 조건은 두 가지입니다:

1. 두 릴레이션의 **속성(컬럼) 수가 같아야** 합니다.
2. 대응되는 속성끼리 **도메인(값의 범위)이 같아야** 합니다.

합집합(`UNION`), 교집합(`INTERSECT`), 차집합(`EXCEPT/MINUS`) 연산은 합병 가능 조건을 만족해야 수행할 수 있습니다. 반면 **카티션 프로덕트(Cartesian Product, 곱집합)**는 합병 가능 조건이 **필요 없습니다**. 카티션 프로덕트는 단순히 두 릴레이션의 모든 행을 조합하는 연산이므로, 두 릴레이션의 속성 수나 도메인이 달라도 수행할 수 있습니다.

> 🎯 **시험 포인트**: 합병 가능 연산에는 합집합, 교집합, 차집합이 포함되며, **카티션 프로덕트는 합병 가능 연산이 아닙니다**.

```sql
-- 합병 가능 연산 예시

-- 테이블 A: 2023년 수강 학생 | 학번 | 이름 |
-- 테이블 B: 2024년 수강 학생 | 학번 | 이름 |
-- 두 테이블의 속성 수(2개)와 도메인이 같으므로 합병 가능

-- ✅ 합집합 (UNION) - 두 테이블의 모든 학생 (중복 제거)
SELECT 학번, 이름 FROM A
UNION
SELECT 학번, 이름 FROM B;

-- ✅ 교집합 (INTERSECT) - 두 해 모두 수강한 학생
SELECT 학번, 이름 FROM A
INTERSECT
SELECT 학번, 이름 FROM B;

-- ✅ 차집합 (EXCEPT) - 2023년에만 수강한 학생
SELECT 학번, 이름 FROM A
EXCEPT
SELECT 학번, 이름 FROM B;

-- 카티션 프로덕트는 합병 가능 조건 불필요
-- 두 테이블의 모든 행을 조합 (속성 수가 달라도 가능)
SELECT * FROM A, B;  -- A의 모든 행 × B의 모든 행
```

## 🔒 5. 무결성 제약조건 — 데이터의 정확성을 지키는 규칙

### 무결성이란 무엇인가요?

**무결성(Integrity)**이란 데이터베이스에 저장된 데이터가 **정확하고 일관성 있게 유지되는 것**을 의미합니다. 마치 회사 장부에서 "존재하지 않는 직원에게 급여를 지급"하거나 "나이가 -5살인 사람"이 등록되는 것을 막는 것처럼, 데이터의 논리적 오류를 방지하는 규칙입니다.

### 주요 무결성 제약조건 비교

| 무결성 종류 | 설명 | 예시 |
|------------|------|------|
| **개체 무결성** | 기본키(Primary Key)는 NULL이거나 중복될 수 없음 | 학번은 반드시 고유해야 함 |
| **참조 무결성** | 외래키(Foreign Key)는 참조하는 테이블의 기본키 값과 일치하거나 NULL이어야 함 | 수강 테이블의 학번은 학생 테이블에 존재해야 함 |
| **도메인 무결성** | 속성 값은 해당 속성의 도메인(정의된 범위) 내의 값이어야 함 | 나이는 0~150 사이의 정수여야 함 |
| **널 무결성** | 특정 속성은 NULL 값을 가질 수 없음 | 이름 컬럼은 반드시 값이 있어야 함 |

### 참조 무결성이 핵심인 이유

**참조 무결성(Referential Integrity)**은 특히 중요합니다. 외래키(Foreign Key, 다른 테이블의 기본키를 참조하는 키)를 변경하려면 참조 대상 테이블의 기본키도 변경되거나, 해당 관계가 유효한 상태를 유지해야 합니다.

문제에서 "릴레이션 R1에 속한 외래키를 변경하려면 참조하고 있는 R2의 기본키도 변경해야 한다"는 설명은 바로 **참조 무결성**을 의미합니다.

> 🎯 **시험 포인트**: 외래키와 기본키의 연관 관계 유지 규칙 = **참조 무결성**

## 🏛️ 6. DBMS 구성 요소 — 데이터베이스를 관리하는 엔진

### DBMS란 무엇인가요?

**DBMS(Database Management System, 데이터베이스 관리 시스템)**는 데이터베이스를 생성하고 관리하는 소프트웨어입니다. MySQL, Oracle, PostgreSQL, MS SQL Server 같은 프로그램들이 DBMS입니다. DBMS는 단순히 데이터를 저장하는 것이 아니라, 다양한 기능을 수행하는 여러 구성 요소로 이루어져 있습니다.

### DBMS의 필수 기능 3가지

DBMS의 필수 기능은 크게 세 가지입니다:

1. **정의 기능(Definition)**: 데이터베이스의 구조(스키마)를 정의합니다. 어떤 테이블에 어떤 컬럼이 있고, 어떤 타입인지를 정의합니다.
2. **조작 기능(Manipulation)**: 데이터를 검색, 삽입, 수정, 삭제하는 기능입니다.
3. **제어 기능(Control)**: 데이터의 정확성과 안전성을 유지합니다. 무결성 검사, 접근 권한 관리, 병행 제어, 회복 작업 등이 포함됩니다.

:::info 사전 기능은 DBMS 필수 기능이 아님
"사전 기능"은 DBMS의 필수 기능으로 보기 어렵습니다. DBMS의 핵심 필수 기능은 **정의, 조작, 제어** 세 가지입니다. 시험에서 "DBMS의 필수 기능이 아닌 것"을 묻는 문제에서 "사전 기능"이 오답으로 자주 등장합니다.
:::

### DBMS 제어 기능이 담당하는 역할

**제어 기능**은 다음과 같은 중요한 작업을 담당합니다:

- **무결성 제약조건 만족 여부 검사**: 입력 데이터가 정의된 규칙을 지키는지 확인
- **사용자 접근 권한 검사**: 어떤 사용자가 어떤 데이터에 접근할 수 있는지 관리
- **트랜잭션(Transaction, 작업 묶음)의 병행 제어(Concurrency Control)**: 여러 사용자가 동시에 데이터를 수정할 때 충돌 방지
- **회복 작업(Recovery)**: 시스템 장애 발생 시 데이터를 정상 상태로 복구

### DML 예비 컴파일러 — 내장 SQL을 추출하는 구성요소

**DML 예비 컴파일러(Precompiler)**는 호스트 프로그래밍 언어(C, Java, COBOL 등)로 작성된 응용 프로그램 속에 **삽입(내장, Embedded)되어 있는 DML 명령문을 추출**하는 DBMS 구성요소입니다. 쉽게 말해, Java 코드 안에 섞여 있는 SQL 문장들을 골라내서 별도로 처리할 수 있도록 분리하는 역할을 합니다.

> 🎯 **시험 포인트**: "호스트 프로그래밍 언어 속에 삽입된 DML 명령문을 추출하는 구성요소" = **DML 예비 컴파일러(Precompiler)**

## 📝 7. 내장 SQL (Embedded SQL) — 프로그램 코드 안의 SQL

### 내장 SQL이란 무엇인가요?

일반적으로 SQL은 데이터베이스 도구(예: MySQL Workbench, SQL*Plus)에서 직접 입력하여 실행합니다. 이를 **대화식 SQL(Interactive SQL)**이라고 합니다. 반면 **내장 SQL(Embedded SQL)**은 C, Java, COBOL 같은 프로그래밍 언어로 작성된 **응용 프로그램 코드 안에 SQL 문장을 직접 삽입**하는 방식입니다. 마치 한국어 문서 안에 영어 문장을 삽입하는 것처럼, 프로그램 코드 안에 SQL 문장이 포함됩니다.

### 내장 SQL의 핵심 특징

내장 SQL을 사용할 때는 반드시 알아야 할 규칙들이 있습니다:

**① EXEC SQL 키워드 사용**

내장 SQL 문장은 일반 대화식 SQL 문장과 구별하기 위해 앞에 반드시 **`EXEC SQL`**을 붙입니다. 컴파일러가 이 키워드를 보고 "아, 여기서부터는 SQL 문장이구나!"라고 인식합니다.

**② 호스트 변수(Host Variable) 사용 가능**

**호스트 변수(Host Variable)**란 내장 SQL에서 사용하는 **주언어(C, Java 등)의 변수**입니다. SQL 문장 안에서 프로그램의 변수 값을 참조하거나, SQL 실행 결과를 프로그램 변수에 저장할 때 사용합니다. 호스트 변수는 SQL 문 안에서 **콜론(`:`)을 앞에 붙여** 일반 SQL 속성(컬럼 이름)과 구별합니다.

**③ 호스트 변수와 데이터베이스 필드는 같은 이름을 가질 수 있음**

호스트 변수와 데이터베이스의 컬럼 이름이 같아도 됩니다. 다만 SQL 문 안에서 콜론(`:`)을 붙인 것이 호스트 변수, 붙이지 않은 것이 데이터베이스 필드로 구별됩니다.

**④ 호스트 변수의 데이터 타입은 대응하는 SQL 데이터 타입과 일치해야 함**

내장 SQL에서 사용하는 호스트 변수의 데이터 타입은 해당 데이터베이스 필드의 SQL 데이터 타입과 **반드시 일치해야** 합니다. 예를 들어 데이터베이스의 컬럼이 `INTEGER` 타입이면, 이에 대응하는 호스트 변수도 정수형 타입이어야 합니다.

:::warning 내장 SQL 오답 주의
시험에서 "내장 SQL에 대한 설명으로 옳지 않은 것"을 묻는 문제가 자주 나옵니다. **"SQL문은 주언어 변수(호스트 변수)의 참조를 포함할 수 없다"는 설명은 완전히 틀린 설명입니다.** 내장 SQL의 핵심 기능 중 하나가 바로 호스트 변수를 통해 프로그램 변수와 SQL을 연결하는 것이기 때문입니다.
:::

```c
/* 내장 SQL 예시 - C 언어에서의 사용 */

/* 호스트 변수 선언 */
EXEC SQL BEGIN DECLARE SECTION;
    int 학번;           /* 호스트 변수: 학번 (정수형) */
    char 이름[50];      /* 호스트 변수: 이름 (문자 배열) */
    char 학과[30];      /* 호스트 변수: 학과 (문자 배열) */
EXEC SQL END DECLARE SECTION;

/* 학번에 값 할당 */
학번 = 101;  /* C 언어에서 변수에 값 저장 */

/* ✅ 올바른 내장 SQL 문: EXEC SQL + 호스트 변수 참조 (콜론 사용) */
EXEC SQL SELECT 이름, 학과        /* SQL 키워드와 컬럼명 */
         INTO :이름, :학과        /* 결과를 호스트 변수에 저장 (콜론 앞에 붙임) */
         FROM 학생               /* 테이블명 */
         WHERE 학번 = :학번;     /* 호스트 변수 참조 (콜론으로 구별) */

/* 위 코드 설명:
   - EXEC SQL: 내장 SQL 시작을 알리는 키워드
   - :이름, :학과: 콜론이 붙은 것은 호스트 변수 (C 변수)
   - 학번 = :학번 에서 왼쪽 학번은 DB 컬럼, :학번은 C 변수
   - INTO 절: SQL 결과를 호스트 변수에 담는 역할
*/
```

## 🌳 8. B-트리 (B-Tree) — 데이터베이스 인덱스의 핵심

### B-트리란 무엇인가요?

**B-트리(Balanced Tree, 균형 트리)**는 데이터베이스에서 **인덱스(색인)**를 구현할 때 가장 많이 사용하는 자료구조입니다. 마치 책의 목차처럼, B-트리를 사용하면 많은 데이터 중에서 원하는 데이터를 빠르게 찾을 수 있습니다. "차수(Order)가 3인 B-트리"는 한 노드(저장 단위)에 최대 2개의 키(값)를 가질 수 있고, 최대 3개의 자식 노드를 가질 수 있는 B-트리입니다.

### 차수 3인 B-트리에 키 삽입 과정

차수가 3인 B-트리에 키 값 25, 40, 120, 10, 55를 순서대로 삽입하는 과정을 살펴보겠습니다.

B-트리의 규칙:
- 차수가 3이므로 한 노드에 최대 **2개의 키** 저장 가능
- 키가 3개가 되면 **분열(Split)** 발생: 가운데 키가 위로 올라가고 양쪽으로 나뉨

**삽입 과정:**

1. **25 삽입**: 루트 노드에 [25] 저장
2. **40 삽입**: 루트 노드에 [25, 40] 저장 (2개라 OK)
3. **120 삽입**: [25, 40, 120] → 3개가 되어 **분열 발생!**
   - 가운데 값 **40**이 루트로 올라감
   - 왼쪽 자식: [25], 오른쪽 자식: [120]
4. **10 삽입**: 40보다 작으므로 왼쪽 자식 [25]에 삽입 → [10, 25]
5. **55 삽입**: 40보다 크므로 오른쪽 자식 [120]에 삽입 → [55, 120]

최종 결과:
- 루트: **[40]**
- 왼쪽 자식: [10, 25]
- 오른쪽 자식: [55, 120]

> 🎯 **시험 포인트**: 차수 3인 B-트리에서 25, 40, 120, 10, 55 삽입 시 루트 값은 **40**입니다.

## 💾 9. 저장 장치 — SSD, RAID, 파일 구조

### 9-1. SSD (Solid State Drive)

**SSD(Solid State Drive, 솔리드 스테이트 드라이브)**는 **반도체를 이용한 정보 저장 장치**입니다. 기계적 움직임이 있는 HDD(Hard Disk Drive, 하드 디스크 드라이브)와 달리 SSD는 전기적으로 데이터를 저장합니다.

SSD의 주요 특징:
- **속도**: HDD에 비해 데이터 읽기/쓰기 속도가 훨씬 빠름
- **기계적 지연 없음**: 회전하는 디스크나 움직이는 헤드가 없으므로 기계적 지연(Mechanical Latency) 없음
- **실패율(Failure Rate) 낮음**: 움직이는 부품이 없으므로 물리적 충격에 강함
- **발열·소음 적음**: 기계적 부품이 없어 열과 소음이 적음
- **소형화·경량화**: 얇고 가볍게 제작 가능

> 🎯 **시험 포인트**: "하드디스크에 비해 속도가 빠르고 기계적 지연·실패율·발열·소음이 적으며 소형화·경량화 가능한 반도체 기반 저장 장치" = **SSD**

### 9-2. 디스크 접근 시간의 구성 요소

디스크에서 데이터를 읽을 때 걸리는 시간은 크게 세 부분으로 나뉩니다:

| 접근 시간 요소 | 설명 | 단축 방법 |
|---------------|------|-----------|
| **탐구 시간 (Seek Time)** | 디스크 헤드를 원하는 트랙(실린더)으로 이동하는 시간 | 같은 실린더/트랙에 데이터 배치 |
| **회전 지연 시간 (Rotational Latency)** | 원하는 섹터가 헤드 아래로 올 때까지 디스크가 회전하는 시간 | 연속된 섹터에 데이터 배치 |
| **데이터 전송 시간 (Transfer Time)** | 실제 데이터를 읽거나 쓰는 시간 | 전송 속도 향상 |

**같은 실린더 또는 같은 트랙에 연속적으로 데이터 블록을 배치**함으로써 단축할 수 있는 것은 **탐구 시간(Seek Time)**입니다. 헤드가 이동하지 않아도 되기 때문에 탐구 시간이 0에 가까워집니다.

### 9-3. RAID (Redundant Array of Independent Disks)

**RAID**는 여러 개의 하드 디스크를 하나처럼 사용하는 기술입니다. 성능 향상이나 데이터 안전성(이중화) 목적으로 사용합니다.

| RAID 레벨 | 특징 | 목적 |
|----------|------|------|
| **RAID 0** | **스트라이핑(Striping)** — 데이터를 여러 디스크에 분산 저장 | 속도 향상 (안전성 없음) |
| **RAID 1** | 미러링(Mirroring) — 동일한 데이터를 2개 디스크에 복사 | 안전성 향상 (속도 무관) |
| **RAID 0+1** | RAID 0과 RAID 1을 조합 | 속도 + 안전성 |
| **RAID 6** | 이중 패리티 — 2개 디스크가 동시에 고장나도 복구 가능 | 고안전성 |

> 🎯 **시험 포인트**: "여러 하드 디스크에 분산 저장하는 스트라이핑(Striping) 방식" = **RAID 0**

### 9-4. 파일 구조 — 급여 관리 시스템에 적합한 구조

파일 구조의 종류와 특징:

| 파일 구조 | 특징 | 적합한 상황 |
|----------|------|-------------|
| **순차 파일** | 레코드를 순서대로 저장·접근 | 전체 데이터 순차 처리 |
| **인덱스 순차 파일** | 순차 접근 + 인덱스를 통한 직접 접근 모두 가능 | 수시 조회/수정 + 전체 순차 출력이 모두 필요한 경우 |
| **직접 파일** | 해시 함수로 레코드 위치를 직접 계산하여 접근 | 빠른 직접 접근만 필요한 경우 |
| **힙 파일** | 삽입 순서대로 저장, 특별한 구조 없음 | 단순 삽입 위주 |

**급여 관리 프로그램 조건 분석:**
- ① 특정 직원의 급여내역을 **수시로 조회** → 직접 접근 필요
- ② 특정 직원의 급여내역을 **수시로 수정** → 직접 접근 필요
- ③ 전 직원의 급여명세서를 **매월 출력** → 순차 접근 필요

이 세 가지 조건을 모두 만족하는 파일 구조는 **인덱스 순차 파일(Index Sequential File)**입니다. 순차 접근과 직접 접근을 모두 지원하기 때문입니다.

### 9-5. 비트맵 인덱스 (Bitmap Index)

**비트맵 인덱스**는 인덱스 값을 **비트(0 또는 1)로 변환하여 저장**하는 인덱스 방식입니다. 특정 컬럼의 값 종류가 매우 적을 때(예: 성별, 지역 코드 등) 효율적입니다.

**비트맵 인덱스의 특징:**
- 테이블이 매우 크고 **낮은 분포도(Cardinality, 고유값의 수가 적음)**를 가질 때 효과적
- 인덱스 값을 비트맵(0, 1)으로 변환하여 저장
- **OLAP(Online Analytical Processing, 분석 처리)** 환경에서 많이 사용
- **삽입이나 갱신이 많은 테이블에는 부적합** — 비트맵을 자주 업데이트해야 하므로 성능 저하

:::warning 비트맵 인덱스 적용 시 주의
"삽입이나 갱신이 많이 발생하는 테이블에 적용한다"는 설명은 **틀린 설명**입니다. 비트맵 인덱스는 데이터 변경이 잦은 OLTP(온라인 트랜잭션 처리) 환경보다는 조회 위주의 OLAP 환경에 적합합니다. 잦은 삽입/갱신은 비트맵 재계산 비용을 유발하여 오히려 성능을 저하시킵니다.
:::

## 🗂️ 10. 논리적 데이터 모델과 데이터베이스 특성

### 10-1. 논리적 데이터 모델의 종류

**논리적 데이터 모델**은 데이터를 어떤 방식으로 표현하고 구조화할지를 정의하는 모델입니다. 주요 종류는 다음과 같습니다:

| 데이터 모델 | 데이터 표현 방식 | 특징 |
|------------|----------------|------|
| **관계형 데이터 모델** | 테이블(릴레이션)의 집합 | 현재 가장 널리 사용, SQL 기반 |
| **네트워크 데이터 모델** | 그래프(오너-멤버 관계) 구조 | 복잡한 관계 표현 가능 |
| **계층적 데이터 모델** | 트리(부모-자식) 계층 구조 | 단순한 계층 관계에 적합 |
| **객체지향 데이터 모델** | 객체와 상속 구조 | 복잡한 데이터 타입 표현 가능 |

:::warning 계층적 vs 네트워크 데이터 모델 혼동 주의
시험에서 자주 나오는 함정입니다. **계층적 데이터 모델**은 트리 구조(부모-자식 관계)로, 한 부모가 여러 자식을 가질 수 있지만 자식은 하나의 부모만 가집니다. **네트워크 데이터 모델**은 그래프 구조로, 더 복잡한 다대다 관계도 표현할 수 있습니다. "오너-멤버 관계의 계층적 그래프 구조"라는 설명은 사실 **네트워크 데이터 모델**의 특징이므로 혼동하지 않도록 주의하세요.
:::

### 10-2. 데이터베이스의 일반적 특성 4가지

데이터베이스는 파일 시스템과 달리 다음과 같은 일반적인 특성을 가집니다:

**① 실시간 접근성(Real-time Accessibility)**
질의(Query)에 대하여 **실시간으로 처리하고 응답**할 수 있어야 합니다. 사용자가 데이터를 요청하면 즉각적으로 결과를 반환합니다.

**② 계속적인 변화(Continuous Evolution)**
삽입(INSERT), 삭제(DELETE), 갱신(UPDATE)을 통해 **항상 최신 상태의 데이터를 유지**합니다. 데이터베이스는 정적인 것이 아니라 끊임없이 변화하는 동적인 존재입니다.

**③ 동시 공용(Concurrent Sharing)**
**여러 사용자가 동시에** 같은 데이터베이스를 사용할 수 있습니다. 여러 직원이 동시에 같은 회사 데이터베이스에 접근할 수 있는 것처럼요.

**④ 내용에 의한 참조(Content Reference)**
데이터를 찾을 때 **데이터의 물리적 위치(주소)가 아니라, 데이터의 값(내용)**을 기준으로 검색합니다. "학번이 101인 학생"이라고 값으로 검색하지, "몇 번째 블록의 몇 번째 레코드"라고 위치로 검색하지 않습니다.

> 🎯 **시험 포인트**: "데이터를 위치나 주소에 의해 참조한다"는 설명은 **데이터베이스의 특성이 아닙니다**. 데이터베이스는 **내용(값)에 의해 참조**합니다. "주소에 의한 참조"는 데이터베이스의 일반적 특성에 해당하지 않습니다.

### 10-3. 데이터베이스 기본 개념 — 개체, 속성, 도메인

데이터베이스 설계의 기본 개념들도 정확히 이해해야 합니다:

| 개념 | 설명 |
|------|------|
| **개체(Entity)** | 현실 세계의 독립적인 객체. 예: 학생, 강좌, 교수 |
| **속성(Attribute)** | 개체의 특성을 나타내는 항목. 예: 학생의 학번, 이름, 학과 |
| **개체 타입(Entity Type)** | 동일한 속성들을 가진 개체들의 **틀(설계도)**. 즉, 개체의 구조를 정의한 것 |
| **개체 어커런스(Entity Occurrence)** | 개체 타입에 속하는 **실제 값(인스턴스)**. 예: (101, 홍길동, 컴퓨터) |
| **도메인(Domain)** | 한 속성이 가질 수 있는 **모든 가능한 값들의 집합** |
| **속성 관계** | 하나의 개체를 기술하는 속성과 속성 사이의 관계 |
| **개체 관계** | 개체 집합과 개체 집합 사이의 관계 |

:::warning 개체 타입 vs 개체 어커런스 구분
시험에서 자주 혼동되는 개념입니다. **개체 타입(Entity Type)**은 "학생"이라는 틀(설계도)이고, **개체 어커런스(Entity Occurrence)**는 "(101, 홍길동, 컴퓨터)"처럼 실제 데이터 값입니다. 문제에서 "개체에 속해 있는 실제 값들로 구성된 개체 어커런스의 집합"을 "개체 타입"이라고 설명하면 **틀린 설명**입니다. 개체 타입은 실제 값들의 집합이 아니라, 개체의 구조를 정의한 **틀**입니다.
:::

## ⚖️ 11. 데이터베이스 시스템 vs 파일 시스템

### 파일 시스템의 문제점과 DBMS의 장점

과거에는 데이터를 파일에 저장하고 프로그램마다 직접 접근하는 **파일 시스템** 방식을 사용했습니다. 하지만 이 방식에는 여러 문제점이 있었고, 이를 해결하기 위해 **DBMS(데이터베이스 관리 시스템)**가 등장했습니다.

| 비교 항목 | 파일 시스템 | DBMS |
|----------|------------|------|
| **데이터 중복** | 프로그램마다 별도 파일로 같은 데이터 중복 저장 | **데이터 중복 최소화** (완전 제거는 아님) |
| **데이터 독립성** | 파일 구조 변경 시 프로그램도 모두 수정 필요 | **데이터 독립성 유지** (구조 변경 시 프로그램 영향 최소화) |
| **동시 접근** | 여러 사용자 동시 접근 시 충돌 위험 | 병행 제어로 동시 접근 지원 |
| **일관성** | 데이터 불일치 발생 가능 | 무결성 제약으로 일관성 유지 |
| **보안** | 파일 수준의 접근 제어 | 세밀한 접근 권한 관리 |

**데이터베이스 시스템의 핵심 특징**:

> ✅ **자료의 중복 배제(Redundancy Elimination) + 자료의 독립성 유지(Data Independence)**

:::warning DBMS가 데이터 중복을 완벽히 제거할 수 없음
"DBMS는 데이터 중복을 완벽하게 제거할 수 있다"는 설명은 **틀린 설명**입니다. DBMS는 데이터 중복을 **최소화(Minimize)**하거나 **배제(Eliminate)**하는 것을 목표로 하지만, 성능이나 설계 이유로 일부 중복이 발생할 수 있습니다. "완벽한 제거"는 현실적으로 불가능합니다.
:::

### 논리적 구조와 물리적 구조

**논리적 구조(Logical Structure)**: 사용자가 생각하는 데이터의 논리적 표현입니다. 테이블, 컬럼, 관계 등 사용자가 보는 관점의 구조입니다.

**물리적 구조(Physical Structure)**: 실제 저장 장치(하드 디스크 등)에 데이터가 저장되는 방식입니다. 물리적 구조에서 취급하는 레코드는 **저장 레코드(Stored Record)**라고 합니다.

:::info 물리적 구조는 논리적 구조에서 직접 취급하지 않음
"물리적 구조는 논리적 구조에서도 취급할 수 있다"는 설명은 **틀린 설명**입니다. 데이터베이스의 3단계 스키마 구조(외부/개념/내부)에 따르면, 물리적 구조(내부 스키마)는 논리적 구조(개념 스키마)와 분리되어 있습니다. 이 분리가 바로 데이터 독립성의 기반입니다.
:::

## ✅ 핵심 정리

이 문서에서 다룬 핵심 개념들을 정리합니다:

- **DDL 명령어**: `CREATE`, `ALTER`, `DROP`이며, `SELECT`는 DDL이 아니라 DML(데이터 조작어)
- **NULL 비교**: `= NULL`은 사용 불가, 반드시 `IS NULL` / `IS NOT NULL` 사용
- **DISTINCT**: `SELECT` 결과에서 중복된 행을 제거하는 명령어 (`ALL`, `ORDER BY`, `GROUP BY`와 구별)
- **내추럴 조인(Natural Join)**: 동일 속성을 기준으로 자동 조인하고 중복 속성을 하나만 남기고 제거
- **디비전(÷) 연산**: R의 차수 (m+n), S의 차수 n → 결과 차수 m; 공통 속성은 이름이 아닌 **도메인**이 같아야 함
- **합병 가능 연산**: 합집합, 교집합, 차집합; **카티션 프로덕트는 합병 가능 연산이 아님**
- **참조 무결성**: 외래키-기본키 간의 참조 관계를 일관성 있게 유지하는 무결성 규칙
- **DBMS 필수 기능**: 정의 기능, 조작 기능, 제어 기능 (사전 기능은 필수 기능 아님)
- **DML 예비 컴파일러**: 호스트 언어 속에 내장된 DML 명령문을 추출하는 DBMS 구성요소
- **내장 SQL**: `EXEC SQL`로 시작, 호스트 변수는 콜론(`:`)으로 표시, 호스트 변수 참조 **가능**
- **B-트리(차수 3, 키 25→40→120→10→55 삽입)**: 루트 값은 **40**
- **SSD**: 반도체 기반, 빠른 속도, 기계적 지연·발열·소음 없음, 소형화 가능
- **RAID 0**: 스트라이핑(Striping) 방식으로 여러 디스크에 분산 저장, 속도 향상 목적
- **탐구 시간(Seek Time)**: 같은 실린더/트랙에 연속 배치하여 단축 가능한 디스크 접근 시간
- **인덱스 순차 파일**: 직접 접근(수시 조회/수정)과 순차 접근(전체 출력) 모두 지원
- **비트맵 인덱스**: OLAP 환경에 적합, 삽입/갱신이 많은 테이블에는 **부적합**
- **데이터베이스 특성**: 실시간 접근, 계속적 변화, 동시 공용, **내용에 의한 참조** (주소에 의한 참조 ❌)
- **DBMS vs 파일 시스템**: 데이터 **중복 배제** + 데이터 **독립성 유지** (완전한 중복 제거는 불가)
- **개체 타입**: 개체의 구조를 정의한 틀(설계도), 실제 값들의 집합이 아님
