# 🗄️ 데이터베이스 기본 개념과 특성 완전 정복

데이터베이스는 현대 소프트웨어 개발의 근간이 되는 기술입니다. 스마트폰 앱, 쇼핑몰 웹사이트, 은행 시스템 등 우리 주변의 거의 모든 디지털 서비스 뒤에는 데이터베이스가 동작하고 있습니다. 이 문서에서는 데이터베이스의 기본 개념부터 SQL 문법, 관계 대수(릴레이션 간의 연산), 파일 구조, DBMS의 구성 요소까지 시험에 자주 나오는 핵심 개념들을 초보자도 이해할 수 있도록 하나씩 풀어서 설명합니다.

:::info 이 문서를 읽기 전에
이 문서는 데이터베이스 관련 시험 문제들을 중심으로, 각 개념의 **원리와 이유**를 함께 설명합니다. 단순 암기보다 개념을 이해하면 응용 문제도 쉽게 풀 수 있습니다.
:::

## 🔗 조인(Join) 연산 — 릴레이션을 연결하는 방법

### 조인이란 무엇인가?

데이터베이스에서는 정보를 여러 개의 테이블(릴레이션)에 나누어 저장합니다. 예를 들어 학생 정보는 `학생` 테이블에, 수강 정보는 `수강` 테이블에 따로 보관하는 방식입니다. 그런데 "홍길동 학생이 어떤 과목을 수강하고 있나?"처럼 두 테이블을 합쳐서 봐야 할 때가 있습니다. 이때 사용하는 연산이 바로 **조인(Join)** 입니다. 마치 두 개의 명단을 공통된 이름을 기준으로 합치는 것과 같습니다.

### 조인의 종류와 자연 조인(Natural Join)

조인에는 여러 종류가 있는데, 그 중에서 시험에 자주 등장하는 개념이 **자연 조인(Natural Join)** 입니다.

| 조인 종류 | 설명 | 중복 속성 처리 |
|---|---|---|
| 동등 조인(Equi Join) | 공통 속성의 값이 같은 튜플을 연결 | 중복 속성이 그대로 남음 |
| **자연 조인(Natural Join)** | 동등 조인 결과에서 중복 속성을 제거 | 중복 속성 하나만 남김 |
| 세타 조인(Theta Join) | 비교 연산자(=, >, < 등)로 조인 | 중복 속성 남음 |

> **자연 조인(Natural Join)** 은 두 개 이상의 릴레이션에서 공통된 속성(컬럼)을 기준으로 연결하되, **중복되는 속성을 하나만 남기고 제거**하는 조인 방법입니다.

예를 들어 `학생(학번, 이름, 학과코드)` 테이블과 `학과(학과코드, 학과명)` 테이블을 동등 조인하면 결과에 `학과코드`가 두 번 나타납니다. 자연 조인은 이 중복된 `학과코드`를 하나만 남겨 깔끔하게 결과를 만들어 줍니다. 마치 두 서류를 스테이플러로 찍을 때 같은 내용의 항목을 하나로 합치는 것과 같습니다.

:::tip 핵심 요약
- **동등 조인**: 공통 속성으로 연결, 중복 컬럼 **남김**
- **자연 조인**: 공통 속성으로 연결, 중복 컬럼 **제거** ← 시험 정답
:::

## 🌳 B-트리(B-Tree) — 데이터를 빠르게 찾는 나무 구조

### B-트리가 왜 필요한가?

데이터를 저장할 때 단순히 순서대로 쌓아 놓으면, 특정 값을 찾을 때 처음부터 하나씩 비교해야 합니다. 데이터가 100만 개라면 최악의 경우 100만 번 비교해야 합니다. 이런 문제를 해결하기 위해 **트리(Tree, 나무) 구조**를 사용합니다. B-트리는 데이터베이스 인덱스(색인, 빠른 검색을 위한 목차)에 가장 많이 사용되는 자료구조입니다.

### B-트리의 구조와 차수(Order)

**차수(Order)** 는 B-트리에서 각 노드(마디)가 가질 수 있는 **최대 자식 노드의 수**를 의미합니다. 차수가 3인 B-트리라면 각 노드는 최대 3개의 자식을 가질 수 있고, 최대 2개의 키 값을 저장할 수 있습니다.

> B-트리의 한 노드에 저장되는 최대 키의 수 = 차수 - 1

B-트리에 키 값을 삽입할 때는 다음 규칙을 따릅니다:
1. 항상 리프 노드(가장 아래 마디)에 삽입합니다.
2. 노드가 꽉 차면 **분할(Split)** 이 일어나고, 중간 값이 부모 노드로 올라갑니다.

### 차수 3인 B-트리에 값 삽입하기

차수가 3인 B-트리에 25, 40, 120, 10, 55 순서로 삽입하는 과정을 단계별로 따라가 보겠습니다.

**1단계: 25 삽입**
```
[25]
```
노드에 25 하나만 저장. 아직 여유 있음.

**2단계: 40 삽입**
```
[25, 40]
```
차수 3이므로 한 노드에 키 2개까지 가능. 정상 삽입.

**3단계: 120 삽입 → 분할 발생!**
차수 3인 B-트리의 노드에는 키가 최대 2개(차수-1)까지 들어갈 수 있습니다. 120을 넣으면 `[25, 40, 120]`이 되어 키가 3개가 됩니다. 이때 **분할(overflow, 넘침)**이 발생합니다.

세 값 중 **중간값인 40**이 위로 올라가 루트가 되고, 나머지는 양쪽 자식이 됩니다.
```
    [40]
   /    \
[25]   [120]
```

**4단계: 10 삽입**
10은 40보다 작으므로 왼쪽 자식인 `[25]`로 이동하여 삽입합니다.
```
     [40]
    /    \
[10,25]  [120]
```

**5단계: 55 삽입**
55는 40보다 크고 120보다 작으므로 오른쪽 자식인 `[120]`으로 이동하여 삽입합니다.
```
      [40]
     /    \
[10,25]  [55,120]
```

따라서 **최종 루트 값은 40**입니다.

:::tip B-트리 삽입 요약
- 차수 3 → 노드당 최대 키 2개
- 넘치면 중간 값이 부모로 올라가며 분할
- 루트가 분할되면 트리의 높이가 1 증가
:::

## 💾 저장 장치 — SSD와 디스크 구조

### SSD란 무엇인가?

전통적인 하드디스크 드라이브(HDD)는 내부에 금속 원판이 빠르게 회전하면서 자기(磁氣)로 데이터를 읽고 씁니다. 이 방식은 기계적인 부품이 움직이기 때문에 소음이 나고, 속도에 한계가 있으며, 충격에 약합니다. 이를 해결한 것이 바로 **SSD(Solid State Drive, 반도체 저장 장치)** 입니다.

> **SSD(Solid State Drive)** 는 반도체(플래시 메모리)를 이용한 정보 저장 장치로, HDD에 비해 속도가 빠르고 기계적 지연이 없으며 발열·소음이 적고 소형화·경량화가 가능합니다.

마치 카세트테이프(HDD)와 USB 메모리(SSD)의 차이를 생각하면 됩니다. 카세트테이프는 원하는 노래를 찾으려면 테이프를 감아야 하지만(기계적 동작), USB는 즉시 접근할 수 있습니다.

| 항목 | HDD | SSD |
|---|---|---|
| 저장 방식 | 자기 원판(기계) | 반도체(플래시) |
| 속도 | 느림 | 빠름 |
| 소음 | 있음 | 없음 |
| 발열 | 많음 | 적음 |
| 충격 저항 | 약함 | 강함 |
| 가격(용량 대비) | 저렴 | 비쌈 |

### 디스크 접근 시간의 구성요소

디스크에서 데이터를 읽을 때 걸리는 시간은 세 가지로 구성됩니다.

- **탐구 시간(Seek Time)**: 읽기/쓰기 헤드가 원하는 트랙(원판의 원형 경로)으로 이동하는 시간
- **회전 지연 시간(Rotational Latency)**: 원판이 회전하여 원하는 섹터(데이터 저장 구역)가 헤드 아래로 올 때까지 기다리는 시간
- **데이터 전송 시간(Transfer Time)**: 실제로 데이터를 읽거나 쓰는 시간

이 중에서 **같은 실린더(여러 원판에서 같은 위치의 트랙 묶음) 또는 같은 트랙에 연속적으로 데이터를 배치**하면 줄일 수 있는 것은 **회전 지연 시간**입니다. 데이터가 같은 트랙에 연속으로 있으면 헤드가 거의 움직이지 않아도 되고, 원판이 한 바퀴 도는 동안 연속으로 읽을 수 있기 때문입니다.

### RAID — 여러 디스크를 묶어 쓰는 기술

**RAID(Redundant Array of Independent Disks, 독립 디스크들의 중복 배열)** 는 여러 개의 하드 디스크를 묶어서 하나처럼 사용하는 기술입니다.

| RAID 레벨 | 방식 | 특징 |
|---|---|---|
| **RAID 0** | 스트라이핑(Striping) | 데이터를 여러 디스크에 분산 저장 → 빠른 속도, 장애 복구 불가 |
| RAID 1 | 미러링(Mirroring) | 동일 데이터를 두 디스크에 복사 → 안전하지만 비용 2배 |
| RAID 0+1 | 스트라이핑 + 미러링 | 속도와 안전성 모두 확보 |
| RAID 6 | 이중 패리티 | 두 개 디스크 동시 장애도 복구 가능 |

> **RAID 0(스트라이핑)** 은 데이터를 여러 개의 하드 디스크에 분산하여 저장함으로써 빠른 입출력이 가능하게 하는 방식입니다. 단, 디스크 하나가 망가지면 전체 데이터를 잃을 수 있습니다.

:::warning RAID 0 주의사항
RAID 0은 속도는 빠르지만 **데이터 보호 기능이 전혀 없습니다**. 디스크 하나라도 고장나면 모든 데이터가 손실됩니다. 중요한 데이터에는 절대 단독으로 사용하지 마세요.
:::

## 🏗️ DBMS의 구성 요소와 기능

### DBMS란 무엇인가?

**DBMS(Database Management System, 데이터베이스 관리 시스템)** 는 데이터베이스를 만들고, 관리하고, 사용할 수 있도록 도와주는 소프트웨어입니다. Oracle, MySQL, PostgreSQL, Microsoft SQL Server 등이 대표적인 DBMS입니다. 마치 도서관에서 책을 분류하고 대출/반납을 관리하는 사서 시스템과 같습니다.

### DBMS의 필수 기능 세 가지

DBMS가 반드시 갖추어야 할 기능은 크게 세 가지입니다.

**① 정의 기능(Definition)**: 데이터베이스의 구조와 형식을 정의합니다. 어떤 테이블을 만들고, 각 컬럼의 데이터 타입을 지정하는 것이 이에 해당합니다. DDL(Data Definition Language, 데이터 정의어)이 이 역할을 담당합니다.

**② 조작 기능(Manipulation)**: 데이터를 삽입, 삭제, 수정, 검색하는 기능입니다. DML(Data Manipulation Language, 데이터 조작어)이 이 역할을 담당합니다.

**③ 제어 기능(Control)**: 데이터의 정확성과 안전성을 유지하는 기능입니다. 무결성 검사, 접근 권한 제어, 트랜잭션 병행 제어, 장애 복구 등이 포함됩니다.

:::note 사전 기능은 DBMS의 필수 기능이 아닙니다
"사전(Dictionary) 기능"은 DBMS의 필수 3대 기능(정의, 조작, 제어)에 포함되지 않습니다. 시험에서 "DBMS의 필수 기능이 아닌 것"을 고르라고 하면 **사전 기능**을 선택하세요.
:::

### DBMS의 구성 요소별 역할

DBMS 내부에는 여러 구성 요소가 있으며, 각각 다른 역할을 합니다.

**데이터 제어 언어 처리기(DCL Processor)** 또는 **데이터베이스 제어기(Database Controller)** 는 다음을 수행합니다:
- **무결성 제약조건(데이터의 정확성 규칙) 만족 여부 검사**
- **사용자 접근 권한 검사** (이 사람이 이 데이터를 볼 권한이 있는지)
- **트랜잭션의 병행 제어** (여러 사용자가 동시에 같은 데이터를 수정할 때 충돌 방지)
- **회복 작업 수행** (장애 발생 시 데이터를 이전 상태로 복원)

**DML 예비 컴파일러(DML Precompiler)** 는 응용 프로그램(자바, C, Python 등으로 만든 프로그램) 안에 삽입된 SQL 명령문(내장 SQL)을 추출하는 역할을 합니다. 마치 일반 문서 속에 숨어있는 수식들을 찾아내는 것과 같습니다.

| DBMS 구성 요소 | 역할 |
|---|---|
| DDL 컴파일러 | DDL 문장을 해석하여 데이터 사전에 저장 |
| **DML 예비 컴파일러** | 응용 프로그램 속 내장 SQL 추출 |
| **데이터베이스 제어기** | 무결성, 접근 권한, 병행 제어, 회복 |
| 질의 처리기 | 사용자의 질의(Query)를 해석하고 실행 |
| 런타임 데이터베이스 처리기 | 실제 데이터 접근 수행 |

## 📝 SQL 핵심 문법 완벽 정리

### SQL의 분류 — DDL, DML, DCL

SQL 명령어는 역할에 따라 크게 세 가지로 분류됩니다.

**DDL(Data Definition Language, 데이터 정의어)**: 테이블 구조를 만들고 수정하는 명령어
- `CREATE`: 새 테이블 생성
- `ALTER`: 테이블 구조 변경
- `DROP`: 테이블 삭제

**DML(Data Manipulation Language, 데이터 조작어)**: 데이터를 다루는 명령어
- `SELECT`: 데이터 조회
- `INSERT`: 데이터 삽입
- `UPDATE`: 데이터 수정
- `DELETE`: 데이터 삭제

**DCL(Data Control Language, 데이터 제어어)**: 권한과 트랜잭션을 관리하는 명령어
- `GRANT`: 권한 부여
- `REVOKE`: 권한 회수
- `COMMIT`: 트랜잭션 확정
- `ROLLBACK`: 트랜잭션 취소

:::danger SELECT는 DDL이 아닙니다
시험에서 "DDL 명령어가 아닌 것"을 고르라고 하면 **SELECT**를 선택하세요. SELECT는 데이터를 조회하는 **DML**에 해당합니다. CREATE, ALTER, DROP은 모두 DDL입니다.
:::

### NULL 값 검사하기

NULL(널 값)은 "값이 없음" 또는 "알 수 없음"을 의미합니다. 이는 0이나 빈 문자열("")과 다릅니다. 마치 설문지에서 답을 적지 않은 칸과 같습니다.

NULL 값을 검사할 때는 일반 비교 연산자(=, !=)를 사용할 수 없습니다. NULL은 어떤 값과도 같지도 다르지도 않기 때문입니다. 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용해야 합니다.

```sql
-- ❌ 잘못된 방법: NULL과의 = 비교는 항상 거짓(FALSE)을 반환합니다
SELECT * FROM myTable WHERE column1 = NULL;

-- ❌ 잘못된 방법: EQUALS는 SQL 표준 문법이 아닙니다
SELECT * FROM myTable WHERE column1 EQUALS NULL;

-- ❌ 잘못된 방법: NOT NULL만 단독으로 사용하는 문법은 없습니다
SELECT * FROM myTable WHERE column1 NOT NULL;

-- ✅ 올바른 방법: IS NULL을 사용하여 NULL 값을 가진 행을 찾습니다
SELECT * FROM myTable WHERE column1 IS NULL;
-- column1의 값이 NULL인 모든 행을 myTable에서 선택합니다
```

### DISTINCT — 중복 제거하기

데이터를 조회할 때 같은 값이 여러 번 나타날 수 있습니다. 예를 들어 학과 목록을 조회할 때 "컴퓨터공학과" 학생이 100명이라면 "컴퓨터공학과"가 100번 반복됩니다. 이때 **DISTINCT**를 사용하면 중복을 제거하고 유일한 값만 출력됩니다.

```sql
-- ❌ DISTINCT 없이 조회: 중복 값이 모두 출력됩니다
SELECT department FROM student;
-- 결과: 컴퓨터공학과, 컴퓨터공학과, 전기공학과, 컴퓨터공학과 ...

-- ✅ DISTINCT 사용: 중복 없이 유일한 값만 출력됩니다
SELECT DISTINCT department FROM student;
-- 결과: 컴퓨터공학과, 전기공학과, 경영학과 ...
-- DISTINCT는 SELECT 바로 다음에 위치해야 합니다
```

다른 선택지와의 비교:

| SQL 키워드 | 역할 |
|---|---|
| ALL | 중복 포함 전체 출력 (기본값, 생략 가능) |
| **DISTINCT** | **중복 제거** ← 정답 |
| ORDER BY | 정렬 (중복 제거 아님) |
| GROUP BY | 그룹화 (중복 제거가 주목적이 아님) |

### 내장 SQL(Embedded SQL)

**내장 SQL(Embedded SQL)** 은 C, Java 등의 프로그래밍 언어로 작성된 프로그램 안에 SQL 문장을 직접 삽입하는 방식입니다. 마치 한국어로 쓴 문서 중간에 영어 문장이 삽입된 것과 같습니다.

내장 SQL의 특징:
- 일반 대화식 SQL 앞에 **`EXEC SQL`** 을 붙여서 내장 SQL임을 표시합니다
- **호스트 변수(host variable, 주언어에서 선언한 변수)** 를 SQL 문 안에서 참조할 수 있습니다 (`:변수명` 형태로 사용)
- 주언어 변수와 데이터베이스 필드(컬럼)는 **같은 이름을 가질 수 있습니다** (콜론으로 구분 가능)
- 호스트 변수의 데이터 타입은 이에 대응하는 데이터베이스 필드의 SQL 데이터 타입과 **일치해야 합니다**

```c
/* ✅ 내장 SQL 예시 (C 언어 + SQL) */
int emp_id = 100;      /* 호스트 변수 선언 */
char emp_name[50];     /* 조회 결과를 받을 호스트 변수 */

/* EXEC SQL을 앞에 붙여 내장 SQL임을 표시 */
EXEC SQL SELECT ename INTO :emp_name  /* :emp_name은 호스트 변수 참조 */
         FROM emp
         WHERE empno = :emp_id;       /* :emp_id는 호스트 변수 참조 */
```

:::danger 절대 틀리면 안 되는 내장 SQL 오답
"SQL문은 주언어 변수(호스트 변수)의 참조를 **포함할 수 없다**"는 설명은 **틀렸습니다**. 내장 SQL은 호스트 변수를 참조할 수 **있습니다**. 이것이 내장 SQL의 핵심 기능입니다. 시험에서 이 문장을 보면 "옳지 않은 것"으로 선택하세요.
:::

## 🧮 관계 대수(Relational Algebra) — 릴레이션 연산의 기초

### 관계 대수란 무엇인가?

**관계 대수(Relational Algebra)** 는 릴레이션(테이블)에 적용할 수 있는 연산들의 집합입니다. 수학에서 숫자에 덧셈, 뺄셈을 적용하듯이, 데이터베이스에서는 테이블에 합집합, 교집합, 조인 등의 연산을 적용합니다. SQL 문장이 내부적으로 어떻게 동작하는지 이해하는 데 도움이 됩니다.

### 합병 가능 조건(Union Compatible)

일부 관계 연산(합집합, 교집합, 차집합)은 두 릴레이션이 **합병 가능(Union Compatible)** 해야 합니다. 합병 가능 조건이란:
1. 두 릴레이션의 **차수(degree, 속성의 수)가 같아야** 합니다.
2. **대응하는 속성의 도메인(값의 범위/타입)이 같아야** 합니다.

마치 두 엑셀 시트를 합칠 때 컬럼 수와 타입이 맞아야 하는 것과 같습니다.

합병 가능한 연산: **합집합(∪), 교집합(∩), 차집합(-)** — 이 세 가지는 합병 가능 조건을 요구합니다.

**카티션 프로덕트(Cartesian Product, 교차 곱)** 는 두 릴레이션의 모든 튜플을 조합하는 연산으로, **합병 가능 조건을 필요로 하지 않습니다**. 그냥 두 테이블의 모든 행을 짝지으면 됩니다.

:::tip 시험 포인트
"합병 가능한 연산이 아닌 것"을 고르라고 하면 **카티션 프로덕트**를 선택하세요.
:::

### 디비전(Division, R ÷ S) 연산

**디비전(Division)** 은 관계 대수 연산 중 가장 이해하기 까다로운 연산입니다. 간단히 말하면 "R에서 S의 모든 조건을 만족하는 것만 골라내는 연산"입니다. 예를 들어 "모든 과목을 수강한 학생을 찾아라"처럼 전체를 만족해야 하는 질의에 사용됩니다.

디비전 R ÷ S의 특성:
- R의 **차수(속성 수)가 (m+n)** 이고, S의 차수가 **n** 이면, 결과 릴레이션의 차수는 **m** 입니다.
- R은 S의 **모든 속성(Y)을 공통으로 포함**합니다.
- 공통 속성은 **도메인(값의 범위)이 같아야** 합니다.

:::danger 디비전의 흔한 오해
"대응되는 속성은 이름이 같아야 한다"는 것은 **틀린 설명**입니다. 디비전에서 공통 속성이 중요한 것은 **이름이 아니라 도메인(값의 타입과 범위)이 같아야 한다**는 것입니다. 속성 이름은 달라도 됩니다. 시험에서 "도메인이 같다는 것이 아니라 이름이 같다는 것"이라는 보기가 나오면 이것이 오답입니다.
:::

## 🔒 무결성 제약조건 — 데이터의 정확성을 지키는 규칙

### 무결성(Integrity)이란?

**무결성(Integrity)** 은 데이터베이스에 저장된 데이터가 항상 정확하고 일관되게 유지되어야 한다는 원칙입니다. 마치 은행 계좌에서 출금 금액이 잔액보다 클 수 없다는 규칙처럼, 데이터에도 지켜야 할 규칙이 있습니다.

### 참조 무결성(Referential Integrity)

**참조 무결성** 은 외래 키(다른 테이블의 기본키를 참조하는 키)와 관련된 규칙입니다.

예를 들어 `주문` 테이블에 `고객번호`라는 외래 키가 있고, 이것이 `고객` 테이블의 기본키를 참조한다고 합시다. 이때:
- `주문` 테이블의 `고객번호`를 변경하려면 `고객` 테이블에 해당 번호가 존재해야 합니다.
- `고객` 테이블에서 특정 고객을 삭제하려면, 그 고객을 참조하는 `주문` 레코드를 먼저 처리해야 합니다.

> **참조 무결성**: 릴레이션 R1의 외래 키를 변경하려면, 이를 참조하고 있는 R2의 기본키도 그에 맞게 처리되어야 한다는 제약 조건입니다.

| 무결성 종류 | 설명 |
|---|---|
| 개체 무결성(Entity Integrity) | 기본 키는 NULL이 될 수 없고, 중복될 수 없음 |
| **참조 무결성(Referential Integrity)** | 외래 키는 참조하는 테이블의 기본키 값이거나 NULL이어야 함 |
| 도메인 무결성(Domain Integrity) | 속성 값은 정의된 도메인 내의 값이어야 함 |
| 널 무결성(Null Integrity) | 특정 속성은 NULL을 허용하지 않음 |

## 📂 파일 구조와 인덱스 — 데이터 저장 방식의 이해

### 파일 구조의 종류

데이터를 파일에 어떻게 저장하느냐에 따라 검색과 수정 속도가 크게 달라집니다. 마치 서류를 서랍에 그냥 쌓아두는 것과, 번호 순서로 정리해서 보관하는 것의 차이입니다.

**① 순차 파일(Sequential File)**: 데이터를 순서대로 쌓아 저장합니다. 전체를 처음부터 끝까지 읽을 때는 빠르지만, 특정 데이터를 찾거나 수정하려면 처음부터 훑어야 해서 느립니다.

**② 인덱스 순차 파일(Indexed Sequential File, ISAM)**: 순차 파일에 인덱스(목차)를 추가한 형태입니다. 특정 데이터를 빠르게 찾을 수 있고, 전체 순차 처리도 가능합니다. 수시로 조회·수정하면서도 전체 출력이 필요한 경우에 적합합니다.

**③ 직접 파일(Direct/Random File)**: 해시 함수(Hash Function, 키를 주소로 변환하는 함수)를 이용해 특정 위치에 직접 접근합니다. 빠른 검색이 가능하지만 순차 처리가 어렵습니다.

**④ 힙 파일(Heap File)**: 데이터를 순서 없이 빈 공간에 쌓아 저장합니다. 삽입은 빠르지만 검색은 느립니다.

### 급여 관리 시스템에 적합한 파일 구조

다음 조건의 급여 관리 시스템을 설계한다면:
1. 특정 직원의 급여 내역을 **수시로 조회**
2. 특정 직원의 급여 내역을 **수시로 수정**
3. 전 직원의 급여 명세서를 **매월 전체 출력**

이 경우 **수시 검색/수정**(조건 1, 2)과 **순차 전체 처리**(조건 3)를 동시에 만족해야 합니다. 이런 요구사항에 가장 적합한 것은 **인덱스 순차 파일**입니다.

> **인덱스 순차 파일**은 인덱스를 통한 빠른 직접 접근(수시 조회·수정)과 순차 처리(전체 출력)를 모두 지원하기 때문에, 이 시나리오에서 가장 적합한 선택입니다.

### 비트맵 인덱스(Bitmap Index)

**비트맵 인덱스** 는 컬럼의 값을 비트(0과 1)로 변환하여 저장하는 인덱스 방식입니다. 예를 들어 "성별" 컬럼에서 '남'=1, '여'=0으로 표시하는 것입니다.

비트맵 인덱스의 특징:
- **테이블이 매우 크고 값의 종류가 적을(낮은 카디널리티/분포도) 때** 효과적입니다.
- 인덱스 값을 비트맵(0, 1)으로 변환하여 저장합니다.
- **OLAP(Online Analytical Processing, 대용량 데이터 분석)** 환경에 많이 사용됩니다.
- **삽입이나 갱신이 많이 발생하는 테이블에는 부적합**합니다. 데이터가 바뀔 때마다 비트맵 전체를 재구성해야 하기 때문입니다.

:::warning 비트맵 인덱스 오답 주의
"삽입이나 갱신이 많이 발생하는 테이블에 적용한다"는 것은 **틀린 설명**입니다. 비트맵 인덱스는 변경이 적고 읽기 위주의 환경(OLAP)에 적합합니다. 변경이 많은 OLTP 환경에는 일반 B-트리 인덱스가 더 적합합니다.
:::

## 🏛️ 데이터베이스의 특성과 DBMS vs 파일 시스템

### 데이터베이스의 4가지 일반적 특성

데이터베이스는 파일 시스템과 구별되는 고유한 특성을 갖고 있습니다.

**① 실시간 접근(Real-time Accessibility)**: 사용자의 질의에 대해 실시간으로 처리하고 응답합니다. 마치 은행 ATM기에서 잔액을 조회하면 즉시 결과가 나오는 것처럼입니다.

**② 계속적인 변화(Continuous Evolution)**: 삽입, 삭제, 갱신을 통해 항상 최신 데이터 상태를 유지합니다. 재고 관리 시스템처럼 데이터가 끊임없이 업데이트됩니다.

**③ 동시 공용(Concurrent Sharing)**: 다수의 사용자가 동시에 같은 데이터베이스를 이용할 수 있습니다. 여러 직원이 동시에 같은 고객 정보를 조회하는 것이 가능합니다.

**④ 내용에 의한 참조(Content Reference)**: 데이터를 찾을 때 **저장 위치나 주소가 아닌 데이터의 값(내용)** 으로 검색합니다. "주소가 서울인 고객을 찾아라"처럼 내용을 기준으로 검색합니다.

:::danger 절대 틀리면 안 되는 핵심 개념
데이터베이스의 특성 중 **"주소에 의한 참조"는 잘못된 설명**입니다. 데이터베이스는 데이터의 위치나 주소로 데이터를 찾는 것이 아니라, **데이터의 내용(값)으로 데이터를 찾습니다**. 이것이 파일 시스템(주소로 접근)과의 큰 차이입니다. 시험에서 "데이터베이스의 특성이 아닌 것"으로 **"주소에 의한 참조"** 를 반드시 고르세요.
:::

### DBMS vs 파일 시스템 비교

DBMS가 파일 시스템에 비해 갖는 가장 핵심적인 특징은 두 가지입니다.

**자료의 중복 배제(Minimum Redundancy)**: 파일 시스템에서는 부서 시스템, 급여 시스템, 인사 시스템이 각각 직원 정보를 별도로 저장하여 같은 데이터가 여러 곳에 중복됩니다. DBMS는 데이터를 중앙에서 통합 관리하여 중복을 최소화합니다.

**자료의 독립성 유지(Data Independence)**: 데이터의 구조(스키마)가 바뀌어도 응용 프로그램을 수정하지 않아도 됩니다. 파일 시스템에서는 파일 구조가 바뀌면 그 파일을 사용하는 모든 프로그램을 수정해야 합니다.

> **핵심**: 데이터베이스 시스템은 파일 시스템에 비해 **자료의 중복을 배제**하고 **자료의 독립성을 유지**합니다.

### DBMS의 장단점

| 항목 | 파일 시스템 | DBMS |
|---|---|---|
| 데이터 중복 | 많음 | 최소화 (완전 제거는 불가능) |
| 데이터 일관성 | 유지하기 어려움 | 유지하기 쉬움 |
| 데이터 독립성 | 낮음 | 높음 |
| 동시 접근 | 어려움 | 지원 |
| 무결성 유지 | 어려움 | 쉬움 |
| 표준화 | 비교적 쉬움 | **복잡한 관계로 표준화 어려움** |
| 초기 비용 | 낮음 | 높음 |

:::warning DBMS의 한계 — 데이터 중복 완전 제거는 불가능
"DBMS는 데이터 중복을 **완벽하게** 제거할 수 있다"는 설명은 **틀렸습니다**. DBMS는 중복을 **최소화**할 수 있지만, 실제로 완전한 제거는 현실적으로 불가능합니다. 성능상의 이유로 의도적으로 중복을 허용하는 경우(반정규화)도 있습니다.
:::

## 📐 논리적 데이터 모델과 데이터베이스 구조

### 논리적 데이터 모델의 종류

**논리적 데이터 모델(Logical Data Model)** 은 데이터를 어떤 구조로 표현할지에 대한 개념적 틀입니다.

**① 관계형 데이터 모델(Relational Data Model)**: 데이터를 **테이블(행과 열로 구성된 표)**의 집합으로 표현합니다. Oracle, MySQL 등 현재 가장 많이 사용되는 모델입니다.

**② 네트워크 데이터 모델(Network Data Model)**: 데이터를 **그래프(노드와 간선으로 구성된 구조)** 로 표현합니다. 부모-자식 관계가 여러 개 허용됩니다.

**③ 계층적 데이터 모델(Hierarchical Data Model)**: 데이터를 **트리 구조**로 표현합니다. 부모-자식 관계가 하나씩만 허용됩니다. 오너-멤버 관계가 아닌 **부모-자식 트리 구조**입니다.

**④ 객체지향 데이터 모델(Object-Oriented Data Model)**: 데이터를 **객체(Object)와 상속(Inheritance) 구조**로 표현합니다.

:::danger 계층적 데이터 모델 오답 주의
"계층적 데이터 모델 — 오너-멤버 관계의 계층적 그래프 구조로 표현"이라는 설명은 **틀렸습니다**. **오너-멤버(Owner-Member) 관계는 네트워크 데이터 모델**의 특징입니다. 계층적 데이터 모델은 트리(Tree) 구조로 표현하며, 부모-자식 관계가 1:N으로 제한됩니다.
:::

### 논리적 구조와 물리적 구조

| 구조 | 설명 | 레코드 유형 |
|---|---|---|
| 논리적 구조 | 사용자 관점에서 본 데이터의 표현 방식 | 논리적 레코드 |
| 물리적 구조 | 저장 장치(디스크) 관점에서 본 데이터의 실제 배치 | 저장 레코드(stored record) |

:::warning 물리적 구조와 논리적 구조의 독립성
"물리적 구조는 논리적 구조에서도 취급할 수 있다"는 설명은 **틀렸습니다**. 데이터베이스의 핵심 목표 중 하나가 바로 **데이터 독립성**으로, 물리적 구조와 논리적 구조는 서로 독립적으로 관리됩니다. 물리적 저장 방식이 바뀌어도 논리적 구조는 영향을 받지 않아야 합니다.
:::

## 🧩 데이터베이스 기본 용어 정리

### 핵심 개념들의 정확한 의미

데이터베이스를 공부할 때 자주 혼동되는 개념들을 명확하게 정리합니다.

**속성(Attribute)**: 테이블의 각 열(컬럼)에 해당하며, 하나의 개체를 설명하는 특성입니다. 예를 들어 "학생" 개체의 "학번", "이름", "나이"가 속성입니다.

**개체(Entity)**: 실세계에서 구별할 수 있는 대상입니다. "학생", "강좌", "교수" 등이 개체입니다.

**개체 타입(Entity Type)**: 동일한 속성들로 구성된 개체들의 집합 유형입니다. 예를 들어 "학생"이라는 개체 타입은 {학번, 이름, 나이} 등의 속성 구조를 가집니다.

**개체 어커런스(Entity Occurrence)**: 개체 타입에 속하는 **실제 개별 데이터(값)**입니다. "홍길동, 20101001, 22세"처럼 실제 데이터 하나하나가 어커런스입니다.

:::danger 개체 타입의 정확한 정의
"개체 타입 — 개체에 속해 있는 실제 값들로 구성된 개체 어커런스의 집합"이라는 설명은 **틀렸습니다**. 개체 어커런스의 집합은 **개체 집합(Entity Set)** 이고, 개체 타입은 개체 어커런스들이 공유하는 **속성의 구조(틀)**를 의미합니다. 실제 값들의 집합은 개체 타입이 아닙니다.
:::

**도메인(Domain)**: 어느 한 속성이 가질 수 있는 **값들의 총칭**(범위)입니다. 예를 들어 "나이" 속성의 도메인은 1~150 사이의 정수이며, "성별" 속성의 도메인은 {'남', '여'}입니다.

**속성 관계(Attribute Relationship)**: 하나의 개체를 기술하는 **속성과 속성 사이의 관계**입니다. 예를 들어 "학생" 개체 내에서 "학번"과 "이름" 사이의 관계입니다.

**개체 관계(Entity Relationship)**: **개체 집합과 개체 집합 사이의 관계**입니다. "학생"과 "강좌" 사이의 "수강" 관계가 이에 해당합니다.

## ✅ 핵심 정리

이 문서에서 학습한 내용을 항목별로 정리합니다.

- **자연 조인(Natural Join)**: 동등 조인 결과에서 중복 속성을 하나만 남기고 제거하는 조인 방법
- **B-트리 차수 3, 키 25-40-120-10-55 삽입 시 루트 값**: 40 (120 삽입 시 분할, 40이 루트로 승격)
- **SSD**: 반도체(플래시 메모리)를 이용한 저장 장치로 HDD보다 빠르고 소음/발열이 적음
- **DBMS 제어기 역할**: 무결성 검사, 접근 권한 검사, 병행 제어, 회복 작업 수행
- **DML 예비 컴파일러**: 응용 프로그램 속에 삽입된 내장 SQL 명령문을 추출하는 역할
- **내장 SQL 오답**: "호스트 변수 참조 불가"는 틀림 — 내장 SQL은 호스트 변수 참조 가능
- **NULL 검색 문법**: `IS NULL` 사용 (= NULL, EQUALS NULL, NOT NULL 단독 사용은 오답)
- **DISTINCT**: SELECT에서 중복 값을 제거하는 SQL 키워드
- **DDL이 아닌 것**: SELECT (SELECT는 DML에 해당)
- **디비전 오답**: "이름이 같아야 한다" — 틀림, 도메인(타입/범위)이 같아야 함
- **합병 불가능 연산**: 카티션 프로덕트 (합집합, 교집합, 차집합은 합병 가능)
- **참조 무결성**: 외래 키가 참조하는 기본키가 변경되면 외래 키도 따라서 처리되어야 함
- **계층적 데이터 모델 오답**: 오너-멤버 관계는 네트워크 모델의 특징 (계층 모델은 트리 구조)
- **인덱스 순차 파일**: 수시 조회/수정 + 전체 순차 처리를 모두 지원
- **비트맵 인덱스 오답**: 삽입/갱신이 많은 곳에 부적합 (OLAP, 읽기 위주 환경에 적합)
- **회전 지연 시간 단축**: 같은 실린더/트랙에 연속으로 데이터 배치하면 단축 가능
- **RAID 0**: 스트라이핑 방식, 여러 디스크에 분산 저장으로 빠른 입출력 (복구 불가)
- **DBMS vs 파일 시스템 핵심**: 자료의 중복 **배제** + 자료의 독립성 **유지**
- **DBMS 필수 기능 아닌 것**: 사전 기능 (정의, 조작, 제어가 3대 필수 기능)
- **데이터베이스 특성이 아닌 것**: 주소에 의한 참조 (데이터베이스는 내용/값으로 참조)
- **개체 타입 오답**: 실제 값들의 집합(어커런스 집합)이 아닌, 속성 구조(틀)를 의미함
- **물리적 구조 오답**: 논리적 구조에서 물리적 구조를 취급할 수 없음 (독립적)
- **DBMS 데이터 중복 제거 오답**: 중복을 완벽하게 제거할 수 있다 — 틀림, 최소화만 가능
- **DBMS 표준화**: 데이터 간 복잡한 관계로 오히려 표준화가 어려울 수 있음 (단점)
