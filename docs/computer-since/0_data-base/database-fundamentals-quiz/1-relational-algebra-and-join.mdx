# 🗄️ 관계 연산자와 조인 — 데이터베이스 기초 완전 정복

데이터베이스를 처음 공부하다 보면 "관계 연산자", "조인", "무결성", "DBMS 구성요소" 같은 단어들이 마치 외계어처럼 느껴질 수 있습니다. 하지만 걱정하지 마세요! 이 문서는 비전공자나 완전 입문자도 이해할 수 있도록, 실생활 비유와 풍부한 예시를 곁들여 처음부터 끝까지 친절하게 설명합니다. 시험 대비용 핵심 개념부터 실무에서 왜 이 개념이 필요한지까지 모두 다루겠습니다. 끝까지 함께 읽어보세요! 🚀

## 🔗 조인(Join) — 테이블을 합치는 방법

### 조인이란 무엇인가?

데이터베이스에서 **조인(Join)**은 두 개 이상의 테이블(릴레이션)을 연결하여 관련 있는 데이터를 함께 보여주는 연산입니다. 마치 두 개의 퍼즐 조각을 맞춰서 하나의 완성된 그림을 만드는 것처럼, 서로 다른 테이블에 흩어진 정보를 하나로 합쳐서 의미 있는 결과를 만들어냅니다.

예를 들어, "직원 테이블"에는 직원 ID와 이름이 있고, "급여 테이블"에는 직원 ID와 급여 금액이 있다고 해봅시다. 이 두 테이블을 직원 ID를 기준으로 합치면 "이름과 급여"를 동시에 볼 수 있습니다. 이것이 바로 조인입니다.

### 조인의 종류와 특징

조인에는 여러 종류가 있는데, 시험에서 자주 등장하는 중요한 것들을 정리해보겠습니다.

| 조인 종류 | 설명 | 특징 |
|---|---|---|
| **동등 조인 (Equi Join)** | 공통 속성의 값이 같은 튜플을 연결 | 결과에 중복 컬럼이 생김 |
| **자연 조인 (Natural Join)** | 동등 조인 후 중복 속성을 하나만 남김 | 중복 컬럼 자동 제거 |
| **세타 조인 (Theta Join)** | =, >, &lt;, ≥, ≤, ≠ 등 다양한 조건으로 연결 | 동등 조인보다 범위가 넓음 |
| **외부 조인 (Outer Join)** | 조건에 맞지 않는 행도 NULL로 포함 | 데이터 누락 방지 |
| **카티션 프로덕트 (Cartesian Product)** | 두 테이블의 모든 행의 조합 | 조건 없이 모든 경우의 수 생성 |

### ✅ 핵심 문제 ① — 자연 조인(Natural Join)

> **문제**: 두 개 이상의 릴레이션으로부터 상호 연관성을 구하기 위한 연산자로서, 동일 조인의 결과 중복되는 속성이 생긴다. 이때 중복되는 속성을 하나만 남기고 제거하는 조인 방법은?

**정답: 자연 조인(Natural Join)**

동등 조인(Equi Join)을 수행하면 두 테이블에서 조인에 사용된 컬럼이 결과에 두 번 나타납니다. 예를 들어, 직원 테이블과 부서 테이블을 `dept_id`로 동등 조인하면 결과 테이블에 `dept_id`가 두 번 등장합니다. **자연 조인은 이런 중복 속성(컬럼)을 자동으로 감지하여 하나만 남기고 나머지를 제거**합니다. 마치 같은 내용이 두 번 적힌 보고서에서 한 쪽을 지우는 것과 같습니다.

```sql
-- ❌ 동등 조인: dept_id가 두 번 등장
SELECT *
FROM 직원 e, 부서 d
WHERE e.dept_id = d.dept_id;
-- 결과: e.dept_id, d.dept_id 둘 다 나옴 → 중복!

-- ✅ 자연 조인: dept_id가 한 번만 등장
SELECT *
FROM 직원
NATURAL JOIN 부서;
-- 결과: dept_id가 한 번만 표시됨 → 깔끔!
```

:::info 자연 조인 vs 동등 조인
- **동등 조인**: 조건 지정, 공통 컬럼이 결과에 두 번 나타남
- **자연 조인**: DBMS가 자동으로 같은 이름의 컬럼을 찾아 조인하고, 중복 컬럼은 하나만 남김
:::

## 🌳 B-트리(B-Tree) — 데이터를 빠르게 찾는 균형 트리

### B-트리란 무엇인가?

**B-트리(Balanced Tree, 균형 트리)**는 데이터베이스 인덱스(색인)에서 가장 많이 사용되는 자료구조입니다. 마치 도서관의 책 목록처럼, 데이터를 특정 규칙에 따라 정리해두어 원하는 값을 빠르게 찾을 수 있게 해줍니다. "차수(Order)"는 하나의 노드(마디)에 들어갈 수 있는 최대 자식 수를 의미합니다.

### ✅ 핵심 문제 ② — B-트리 삽입과 루트 값

> **문제**: 차수가 3인 B-트리에서 키 값 25, 40, 120, 10, 55 순으로 삽입하였을 때 B-트리의 루트 값은?

**정답: 40**

차수가 3인 B-트리는 한 노드에 최대 2개의 키(Key)를 가질 수 있습니다(차수 - 1 = 2). 노드가 꽉 차면 중간 값이 위로 올라가는 **분열(Split)** 이 발생합니다. 단계별로 살펴보겠습니다.

1. **25 삽입** → 루트: [25]
2. **40 삽입** → 루트: [25, 40]
3. **120 삽입** → 노드가 꽉 참! [25, 40, 120] → 중간값 40이 루트로 올라가고 분열
   - 루트: [40], 왼쪽 자식: [25], 오른쪽 자식: [120]
4. **10 삽입** → 40보다 작으므로 왼쪽 자식에 삽입 → 왼쪽 자식: [10, 25]
5. **55 삽입** → 40보다 크므로 오른쪽 자식에 삽입 → 오른쪽 자식: [55, 120]

최종 루트 값은 **40**입니다.

:::tip B-트리 핵심 규칙
- 차수가 n이면 한 노드에 최대 **n-1개**의 키를 저장
- 노드가 꽉 차면 **중간값이 부모 노드로 올라감** (분열)
- 모든 리프 노드는 같은 레벨에 위치 → "균형"을 유지
:::

## 💾 저장 장치 — SSD와 RAID

### ✅ 핵심 문제 ③ — SSD(Solid State Drive)

> **문제**: 하드디스크 드라이브에 비하여 속도가 빠르고 기계적 지연이나 실패율, 발열·소음도 적으며, 소형화·경량화 할 수 있는 장점이 있는 반도체를 이용한 정보 저장 장치는?

**정답: SSD (Solid State Drive, 반도체 저장 장치)**

일반 하드디스크(HDD)는 자석이 달린 원판(플래터)이 빙빙 돌면서 데이터를 읽고 씁니다. 마치 LP 레코드판을 바늘로 긁어 음악을 재생하는 것처럼요. 반면, **SSD는 전기적 신호로 데이터를 저장하는 반도체 칩을 사용**합니다. 돌아가는 부품이 없으니 소음도 없고, 발열도 적고, 충격에도 강하며 속도도 월등히 빠릅니다.

| 비교 항목 | HDD (하드디스크) | SSD (반도체 저장 장치) |
|---|---|---|
| 작동 방식 | 기계적 회전 (플래터) | 반도체 전기 신호 |
| 속도 | 느림 | 매우 빠름 |
| 소음/발열 | 있음 | 거의 없음 |
| 충격 저항 | 약함 | 강함 |
| 가격 | 저렴 | 상대적으로 비쌈 |
| 소형화 | 한계 있음 | 매우 작게 만들 수 있음 |

### ✅ 핵심 문제 ⑰ — RAID 0 (Striping)

> **문제**: 데이터의 빠른 입출력이 가능하도록 여러 개의 하드 디스크에 분산되어 저장하는 Striping 방식의 RAID는 무엇인가?

**정답: RAID 0**

**RAID(Redundant Array of Independent Disks, 독립 디스크 중복 배열)**는 여러 개의 하드디스크를 묶어서 사용하는 기술입니다. **RAID 0**은 데이터를 여러 디스크에 나눠서(Striping) 동시에 읽고 쓰는 방식입니다. 마치 짐을 여러 사람이 나눠 들면 더 빨리 이동할 수 있는 것처럼요. 속도는 매우 빠르지만, 하나의 디스크가 고장나면 모든 데이터를 잃어버리는 단점이 있습니다.

| RAID 종류 | 방식 | 특징 |
|---|---|---|
| **RAID 0** | 스트라이핑(분산 저장) | 속도 최고, 안정성 없음 |
| **RAID 1** | 미러링(복사 저장) | 안정성 높음, 용량 절반 |
| **RAID 0+1** | 스트라이핑 + 미러링 | 속도와 안정성 모두 확보 |
| **RAID 6** | 이중 패리티 | 디스크 2개 동시 고장도 복구 가능 |

### ✅ 핵심 문제 ⑯ — 디스크 접근 시간

> **문제**: 같은 실린더 또는 같은 트랙에 연속적으로 데이터 블록을 배치함으로써 단축할 수 있는 디스크 접근 시간 요소는?

**정답: 회전 지연시간 (Rotational Latency)**

하드디스크가 데이터를 읽으려면 세 가지 시간이 필요합니다.

- **탐구 시간(Seek Time)**: 디스크 헤드가 원하는 트랙(원판의 원형 줄)으로 이동하는 시간
- **회전 지연 시간(Rotational Latency)**: 원판이 돌아서 원하는 섹터가 헤드 아래로 올 때까지 기다리는 시간
- **데이터 전송 시간(Transfer Time)**: 실제로 데이터를 읽고 전송하는 시간

같은 실린더나 트랙에 연속으로 데이터를 배치하면, 헤드가 이동하지 않아도 되고 원판이 조금만 돌면 다음 데이터가 바로 나옵니다. 이렇게 하면 **회전 지연 시간**을 크게 줄일 수 있습니다.

## 🏗️ DBMS 구성 요소와 기능

### DBMS란 무엇인가?

**DBMS(Database Management System, 데이터베이스 관리 시스템)**는 데이터베이스를 만들고, 관리하고, 사용할 수 있게 해주는 소프트웨어입니다. 마치 도서관의 사서처럼, 데이터를 체계적으로 정리하고 필요할 때 빠르게 찾아주며, 여러 사람이 동시에 이용할 수 있도록 관리해줍니다. Oracle, MySQL, PostgreSQL, Microsoft SQL Server 등이 대표적인 DBMS입니다.

### ✅ 핵심 문제 ④ — DBMS 제어 구성 요소

> **문제**: 무결성 제약조건 만족여부, 사용자 접근권한 검사, 트랜잭션의 병행제어, 회복작업 수행을 하는 DBMS의 구성 요소는?

**정답: 데이터베이스 제어 (Database Control) 구성 요소 / 런타임 데이터베이스 처리기**

DBMS는 크게 세 가지 핵심 기능을 가집니다.

- **정의 기능 (Definition)**: 데이터의 구조(스키마)를 정의합니다. "이 테이블에는 어떤 컬럼이 있고, 어떤 타입인가?"를 결정합니다.
- **조작 기능 (Manipulation)**: 데이터를 검색, 삽입, 수정, 삭제합니다. SQL의 SELECT, INSERT, UPDATE, DELETE가 여기에 해당합니다.
- **제어 기능 (Control)**: 데이터의 정확성과 안전성을 유지합니다. 무결성 검사, 접근 권한 관리, 트랜잭션(작업 묶음) 제어, 장애 복구 등이 포함됩니다.

**무결성 제약조건, 접근권한 검사, 병행 제어, 회복 작업**은 모두 "제어" 기능에 해당합니다.

:::warning 시험 포인트
DBMS의 필수 기능은 **정의, 조작, 제어** 세 가지입니다. "사전 기능"은 필수 기능이 아닙니다! (문제 ⑲ 참고)
:::

### ✅ 핵심 문제 ⑤ — DML 예비 컴파일러

> **문제**: 호스트 프로그래밍 언어로 작성된 응용 프로그램 속에 삽입되어 있는 DML 명령문들을 추출하는 DBMS 구성요소는?

**정답: DML 예비 컴파일러 (DML Precompiler)**

프로그래머가 C, Java 같은 프로그래밍 언어 코드 안에 SQL 문장을 직접 집어넣는 것을 **내장 SQL(Embedded SQL)**이라고 합니다. 이렇게 섞여 있는 코드에서 SQL 부분만 쏙 빼내는 역할을 하는 것이 **DML 예비 컴파일러**입니다. 마치 도시락 통에서 반찬과 밥을 분리하는 것처럼, 프로그램 코드에서 DML(데이터 조작 언어) 부분만 추출하여 별도로 처리합니다.

### ✅ 핵심 문제 ⑲ — DBMS의 필수 기능

> **문제**: DBMS의 필수 기능으로 보기 어려운 것은? (정의 기능 / 조작 기능 / 제어 기능 / 사전 기능)

**정답: 사전 기능**

앞서 설명했듯이 DBMS의 필수 기능은 **정의, 조작, 제어** 세 가지입니다. "사전 기능"이라는 것은 DBMS의 표준 필수 기능 분류에 포함되지 않습니다. 데이터 딕셔너리(Data Dictionary, 데이터 사전)는 DBMS의 부가적인 구성요소이지 필수 기능으로 분류되지 않습니다.

## 📝 SQL 기본 문법 완전 정복

### SQL이란 무엇인가?

**SQL(Structured Query Language, 구조화 질의 언어)**은 데이터베이스를 다루기 위한 표준 언어입니다. 마치 데이터베이스와 대화하는 언어라고 생각하면 됩니다. "이 테이블에서 나이가 30살 이상인 사람들을 보여줘!"라고 말하는 것을 SQL 문장으로 표현하는 것이죠.

### ✅ 핵심 문제 ⑦ — NULL 값 검사

> **문제**: column1의 값이 널 값(NULL Value)인 경우를 찾기 위한 올바른 SQL 문장은?

**정답: `SELECT * FROM myTable WHERE column1 IS NULL;`**

**NULL**은 "아무 값도 없음"을 나타내는 특수한 상태입니다. NULL은 0이나 빈 문자열("")과도 다릅니다. 0은 숫자 0이고, ""는 빈 문자열이지만, NULL은 "값 자체가 존재하지 않음"을 의미합니다. 이 때문에 NULL을 비교할 때는 일반 등호(=)를 사용할 수 없고, 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용해야 합니다.

```sql
-- ❌ 잘못된 방법: NULL은 = 로 비교 불가
SELECT * FROM myTable WHERE column1 = NULL;
-- 이 쿼리는 항상 결과가 없음! NULL = NULL도 FALSE임

-- ❌ 잘못된 방법: SQL에 EQUALS 키워드 없음
SELECT * FROM myTable WHERE column1 EQUALS NULL;
-- 문법 오류 발생!

-- ❌ 잘못된 방법: 문법이 틀림
SELECT * FROM myTable WHERE column1 NOT NULL;
-- IS NOT NULL 이어야 함

-- ✅ 올바른 방법: IS NULL 사용
SELECT * FROM myTable WHERE column1 IS NULL;
-- column1이 NULL인 모든 행을 반환함
```

:::danger NULL 비교의 함정
SQL에서 `NULL = NULL`의 결과는 **TRUE가 아니라 NULL(알 수 없음)**입니다! 이것이 가장 많은 초보자들이 실수하는 부분입니다. NULL과의 모든 비교 연산은 NULL을 반환하므로, 반드시 `IS NULL` / `IS NOT NULL`을 사용하세요.
:::

### ✅ 핵심 문제 ⑧ — 중복 제거 DISTINCT

> **문제**: 중복을 제거하는 SQL 명령어는? (ALL / ORDER BY / DISTINCT / GROUP BY)

**정답: DISTINCT**

`DISTINCT`는 SELECT 결과에서 중복된 행을 제거하고 유일한 값만 보여줍니다. 마치 출석부에서 같은 이름이 여러 번 적혀 있을 때 한 번만 남기는 것과 같습니다.

```sql
-- 예시 데이터: 주문 테이블에 고객 ID가 중복으로 존재
-- 고객ID: 1, 2, 1, 3, 2, 1

-- ❌ ALL (기본값): 중복 포함해서 모두 출력
SELECT ALL 고객ID FROM 주문;
-- 결과: 1, 2, 1, 3, 2, 1 (중복 있음)

-- ✅ DISTINCT: 중복 제거 후 출력
SELECT DISTINCT 고객ID FROM 주문;
-- 결과: 1, 2, 3 (중복 없음!)

-- 참고: ORDER BY는 정렬, GROUP BY는 그룹화
SELECT DISTINCT 고객ID FROM 주문 ORDER BY 고객ID;
-- 결과: 1, 2, 3 (중복 제거 + 오름차순 정렬)
```

각 옵션의 역할을 비교해보겠습니다.

| SQL 키워드 | 역할 | 중복 제거 여부 |
|---|---|---|
| **ALL** | 모든 결과 출력 (기본값) | ❌ 제거 안 함 |
| **DISTINCT** | 중복 행 제거 | ✅ 제거함 |
| **ORDER BY** | 결과를 정렬 | ❌ (정렬만 함) |
| **GROUP BY** | 같은 값끼리 묶어서 집계 | ❌ (그룹화만 함) |

### ✅ 핵심 문제 ⑨ — DDL vs DML

> **문제**: DDL 명령어로 옳지 않은 것은? (CREATE / ALTER / DROP / SELECT)

**정답: SELECT**

SQL 명령어는 크게 DDL과 DML로 나뉩니다.

- **DDL (Data Definition Language, 데이터 정의 언어)**: 테이블, 인덱스 같은 데이터베이스 구조 자체를 만들고 수정하고 삭제합니다.
  - `CREATE`: 새 테이블/인덱스 등을 생성
  - `ALTER`: 기존 구조를 수정 (컬럼 추가, 타입 변경 등)
  - `DROP`: 테이블/인덱스 등을 삭제
  - `TRUNCATE`: 테이블의 모든 데이터 삭제 (구조는 유지)

- **DML (Data Manipulation Language, 데이터 조작 언어)**: 데이터를 검색하고 변경합니다.
  - `SELECT`: 데이터 조회 ← **DDL이 아닌 DML!**
  - `INSERT`: 데이터 삽입
  - `UPDATE`: 데이터 수정
  - `DELETE`: 데이터 삭제

```sql
-- DDL 예시: 테이블 구조를 만들고 수정하고 삭제
CREATE TABLE 직원 (
    직원ID INT PRIMARY KEY,    -- 기본키 설정
    이름 VARCHAR(50),           -- 문자열 50자
    급여 INT                    -- 정수형 급여
);

ALTER TABLE 직원 ADD 부서 VARCHAR(30); -- 부서 컬럼 추가

DROP TABLE 직원; -- 직원 테이블 완전 삭제

-- DML 예시: 데이터를 다루는 명령어
SELECT * FROM 직원;             -- 데이터 조회 (DDL 아님!)
INSERT INTO 직원 VALUES (1, '김철수', 3000000, '개발팀');
UPDATE 직원 SET 급여 = 3500000 WHERE 직원ID = 1;
DELETE FROM 직원 WHERE 직원ID = 1;
```

:::note DCL도 있어요!
SQL에는 DDL, DML 외에도 **DCL(Data Control Language, 데이터 제어 언어)**이 있습니다. `GRANT`(권한 부여), `REVOKE`(권한 취소) 등이 여기에 해당합니다.
:::

### ✅ 핵심 문제 ⑥ — 내장 SQL (Embedded SQL)

> **문제**: 내장 SQL에 대한 설명으로 옳지 않은 것은?

**정답: "SQL문은 주언어 변수(host variable)의 참조를 포함할 수 없다"는 설명이 옳지 않음**

**내장 SQL(Embedded SQL)**은 C, Java 같은 일반 프로그래밍 언어 코드 안에 SQL 문장을 직접 삽입하는 방식입니다. 올바른 내장 SQL의 특징은 다음과 같습니다.

- ✅ 내장 SQL문 앞에는 `EXEC SQL`을 붙입니다 → **맞는 설명**
- ❌ SQL문은 주언어 변수(호스트 변수)를 참조할 수 없다 → **틀린 설명!** 실제로는 콜론(:)을 앞에 붙여서 호스트 변수를 참조할 수 있습니다.
- ✅ 주언어 변수와 데이터베이스 필드(컬럼)는 같은 이름을 가질 수 있습니다 → **맞는 설명** (콜론으로 구분)
- ✅ 호스트 변수의 데이터 타입은 대응하는 SQL 데이터 타입과 일치해야 합니다 → **맞는 설명**

```c
/* 내장 SQL 예시 (C 언어) */

/* 호스트 변수 선언 */
EXEC SQL BEGIN DECLARE SECTION;
    int 직원ID;           /* 주언어(C) 변수 선언 */
    char 이름[50];
    int 급여;
EXEC SQL END DECLARE SECTION;

직원ID = 1; /* C 코드에서 변수에 값 대입 */

/* SQL 안에서 콜론(:)으로 호스트 변수 참조 가능! */
EXEC SQL SELECT 이름, 급여
         INTO :이름, :급여         /* :변수명으로 호스트 변수 참조 */
         FROM 직원
         WHERE 직원ID = :직원ID;   /* C 변수 값 1을 SQL에서 사용 */
```

## 🔢 관계 대수 — 릴레이션 연산의 핵심

### 관계 대수란?

**관계 대수(Relational Algebra)**는 데이터베이스에서 원하는 데이터를 얻기 위한 수학적인 연산 방법입니다. 마치 수학에서 덧셈, 뺄셈, 곱셈 같은 연산이 있듯이, 릴레이션(테이블)에도 합칠 수 있고, 나눌 수 있고, 빼낼 수 있는 연산이 있습니다. SQL은 이 관계 대수를 기반으로 만들어졌습니다.

### ✅ 핵심 문제 ⑩ — Division(나누기) 연산

> **문제**: Division(R ÷ S)에 대한 설명으로 옳지 않은 것은?

**정답: "공통되는 애트리뷰트별로 도메인이 같다는 것이 아니라 대응되는 애트리뷰트는 모두 이름들이 같다는 것이다"라는 설명이 옳지 않음**

**Division(나누기) 연산**은 "S에 있는 모든 튜플과 관계된 R의 튜플을 찾아라"는 연산입니다. 실생활로 비유하면 "모든 과목을 수강한 학생을 찾아라"와 같습니다.

올바른 Division 특징:
- R의 차수(컬럼 수)가 m+n이고, S의 차수가 n이면, 결과의 차수는 m입니다 → **맞는 설명**
- R은 S의 모든 속성을 공통으로 포함합니다 → **맞는 설명**
- 공통 속성은 **이름이 같아야 하는 것이 아니라 도메인(값의 범위)이 같아야** 합니다 → "이름이 같다"고 한 설명이 **틀림!**
- 결과 릴레이션의 각 튜플은 S의 모든 튜플과 결합할 때 R에 존재합니다 → **맞는 설명**

### ✅ 핵심 문제 11 — 합병 가능 연산자

> **문제**: 합병 가능한 연산자로 옳지 않은 것은? (합집합 / 교집합 / 차집합 / 카티션 프로덕트)

**정답: 카티션 프로덕트 (Cartesian Product)**

**합병 가능(Union Compatible)**이란 두 릴레이션을 집합 연산(합집합, 교집합, 차집합)할 때 만족해야 하는 조건입니다.
- 두 릴레이션의 차수(컬럼 수)가 같아야 함
- 대응되는 속성의 도메인(값의 범위)이 같아야 함

합집합(∪), 교집합(∩), 차집합(-)은 모두 합병 가능 조건을 필요로 합니다. 하지만 **카티션 프로덕트(×)**는 두 릴레이션의 모든 행 조합을 만드는 연산이므로, 합병 가능 조건이 필요하지 않습니다. 크기가 서로 다른 테이블끼리도 카티션 프로덕트가 가능합니다.

## 🔒 무결성과 데이터 모델

### 무결성이란?

**무결성(Integrity)**은 데이터베이스에 저장된 데이터가 정확하고 일관성이 있어야 한다는 규칙입니다. 마치 주민등록부에 실제로 존재하지 않는 사람의 정보가 들어가면 안 되는 것처럼, 데이터베이스에도 논리적으로 올바른 데이터만 저장되도록 강제하는 규칙이 무결성 제약조건입니다.

### ✅ 핵심 문제 12 — 참조 무결성

> **문제**: 릴레이션 R1에 속한 외래키를 변경하려면 이를 참조하고 있는 R2의 기본키도 변경해야 하는 것은?

**정답: 참조 무결성 (Referential Integrity)**

무결성의 종류를 정리해보겠습니다.

| 무결성 종류 | 설명 | 예시 |
|---|---|---|
| **개체 무결성** | 기본키(PK)는 NULL이거나 중복될 수 없음 | 학번이 비어있거나 같은 학번이 두 개 있으면 안 됨 |
| **참조 무결성** | 외래키(FK)는 참조하는 테이블의 기본키 값과 일치해야 함 | 주문 테이블의 고객ID는 고객 테이블에 존재하는 ID여야 함 |
| **도메인 무결성** | 속성값은 정의된 도메인(범위) 안에 있어야 함 | 나이는 0~150 사이여야 함, 성별은 '남'/'여'만 가능 |
| **널 무결성** | 특정 속성은 NULL이 될 수 없음 | 이름은 반드시 값이 있어야 함 |

**참조 무결성**은 외래키(Foreign Key, 다른 테이블의 기본키를 참조하는 키)와 기본키(Primary Key) 사이의 일관성을 유지하는 규칙입니다. 외래키 값은 반드시 참조하는 테이블의 기본키에 존재하는 값이어야 합니다.

### ✅ 핵심 문제 13 — 논리적 데이터 모델

> **문제**: 논리적 데이터 모델에 대한 설명으로 옳지 않은 것은?

**정답: "계층적 데이터 모델 — 오너-멤버 관계의 계층적 그래프 구조"라는 설명이 옳지 않음**

오너-멤버(Owner-Member) 관계는 **네트워크 데이터 모델**의 특징입니다. 올바른 설명은 다음과 같습니다.

| 데이터 모델 | 올바른 설명 |
|---|---|
| **관계형 데이터 모델** | 데이터를 테이블(행과 열)의 집합으로 표현. 가장 많이 사용됨 |
| **네트워크 데이터 모델** | 오너-멤버 관계로 데이터를 그래프(망) 구조로 표현 |
| **계층적 데이터 모델** | 부모-자식 관계의 트리(나무) 구조로 표현. 한 부모에 여러 자식 가능, 반대는 불가 |
| **객체지향 데이터 모델** | 객체와 상속 구조로 데이터를 표현 |

:::info 계층 vs 네트워크 모델
- **계층적 모델**: 트리 구조. 한 자식은 오직 하나의 부모만 가질 수 있습니다. (1:N 관계만 가능)
- **네트워크 모델**: 그래프 구조. 오너-멤버 관계로, 한 자식이 여러 부모를 가질 수 있습니다. (M:N 관계 가능)
:::

## 📁 파일 구조와 인덱스

### 다양한 파일 구조

파일 시스템에서 데이터를 어떻게 저장하느냐에 따라 검색 속도와 수정 편의성이 크게 달라집니다.

### ✅ 핵심 문제 14 — 파일 구조 선택

> **문제**: 특정 직원의 급여내역을 수시로 조회/수정하고, 전 직원 급여명세서를 매월 출력하는 경우 가장 적절한 파일 구조는?

**정답: 인덱스 순차 파일 (Indexed Sequential File)**

| 파일 구조 | 특징 | 적합한 경우 |
|---|---|---|
| **순차 파일** | 데이터가 순서대로 저장. 처음부터 끝까지 읽음 | 전체 데이터를 순서대로 처리할 때 |
| **인덱스 순차 파일** | 인덱스(색인)로 빠른 검색 + 순차 처리 모두 가능 | 검색/수정도 많고 전체 출력도 필요할 때 ✅ |
| **직접 파일** | 해시 함수로 주소를 계산해서 직접 접근 | 특정 레코드 검색이 매우 빠름, 순차 처리 어려움 |
| **힙 파일** | 새 데이터를 빈 공간에 아무렇게나 저장 | 데이터 양이 적거나 전체 스캔만 할 때 |

이 문제의 조건은:
1. 특정 직원 **수시 조회** → 빠른 임의 검색 필요
2. 특정 직원 **수시 수정** → 빠른 접근 필요
3. 전 직원 **매월 순차 출력** → 전체 순차 처리 필요

이 세 가지를 모두 만족하는 것은 **인덱스 순차 파일**입니다. 인덱스로 특정 레코드를 빠르게 찾고, 순차적으로 전체를 읽는 것도 가능하기 때문입니다.

### ✅ 핵심 문제 15 — 비트맵 인덱스

> **문제**: 비트맵 인덱스에 대한 설명으로 옳지 않은 것은?

**정답: "삽입이나 갱신이 많이 발생하는 테이블에 적용한다"는 설명이 옳지 않음**

**비트맵 인덱스(Bitmap Index)**는 각 인덱스 값을 0과 1의 비트(bit) 배열로 표현하는 인덱스 방식입니다. 예를 들어, 성별 컬럼에서 '남'은 1, '여'는 0으로 표현하는 식입니다.

올바른 비트맵 인덱스 특징:
- ✅ 테이블이 매우 크고, 컬럼의 분포도(카디널리티)가 낮을 때 적합 (예: 성별, 지역, 상태코드 등 값의 종류가 적을 때)
- ✅ 인덱스 값을 비트맵(0, 1)으로 변환하여 저장
- ❌ 삽입이나 갱신이 많은 테이블에 적합하다 → **틀림!** 비트맵 인덱스는 데이터 변경 시 비트맵 전체를 재구성해야 하므로, **읽기(조회)가 많고 변경이 적은 OLAP 환경**에 적합합니다.
- ✅ OLAP(Online Analytical Processing, 온라인 분석 처리) 환경에 많이 사용

:::warning 비트맵 인덱스 사용 주의
비트맵 인덱스는 **삽입(INSERT), 수정(UPDATE), 삭제(DELETE)가 잦은 OLTP 환경에는 부적합**합니다. 데이터 변경 시마다 비트맵을 재구성해야 해서 오히려 성능이 저하될 수 있습니다.
:::

## 🏛️ 데이터베이스의 특성과 개념

### 데이터베이스 vs 파일 시스템

데이터베이스가 기존 파일 시스템보다 우수한 이유, 그리고 한계는 무엇인지 알아보겠습니다.

### ✅ 핵심 문제 18 — 데이터베이스의 핵심 특징

> **문제**: 데이터베이스 시스템이 파일 시스템에 비교되는 특징으로 옳은 것은?

**정답: 자료의 중복 배제와 자료의 독립성 유지**

파일 시스템의 문제점은 같은 데이터가 여러 파일에 중복으로 저장되어 일관성이 깨지기 쉽다는 것입니다. 예를 들어, 회사에서 직원 정보를 급여 파일, 인사 파일, 복지 파일에 각각 따로 저장하면 한 직원의 주소가 바뀔 때 세 파일 모두 수정해야 합니다. 데이터베이스는 이러한 **데이터 중복을 최소화(배제)**하고, 응용 프로그램이 데이터 저장 방식에 영향을 받지 않도록 **데이터 독립성**을 보장합니다.

### ✅ 핵심 문제 21 — 데이터베이스의 특성

> **문제**: 데이터베이스의 특성으로 옳지 않은 것은?

**정답: "데이터 참조 시 위치나 주소에 의하여 데이터를 찾는다"는 설명이 옳지 않음**

데이터베이스의 올바른 특성:

- ✅ **실시간 접근(Real-time Accessibility)**: 질의(쿼리)에 대해 실시간으로 처리하고 응답
- ✅ **계속적인 변화(Continuous Evolution)**: 삽입, 삭제, 갱신으로 항상 최신 상태 유지
- ✅ **동시 공용(Concurrent Sharing)**: 여러 사용자가 동시에 이용 가능
- ❌ **주소에 의한 참조**: 데이터베이스는 위치나 주소가 아닌 **데이터 값(내용)으로 참조**합니다. → **틀린 설명!**

> 데이터베이스의 네 가지 특성: **실시간 접근, 계속적인 변화, 동시 공용, 내용에 의한 참조** (주소가 아님!)

### ✅ 핵심 문제 25 — 데이터베이스 일반 특성

> **문제**: 데이터베이스의 일반적인 특성에 대한 설명으로 옳지 않은 것은? (실시간 접근 / 계속적인 변화 / 동시공용 / 주소에 의한 참조)

**정답: 주소에 의한 참조**

앞서 설명했듯이, 데이터베이스는 주소가 아닌 **내용(값)**으로 데이터를 참조합니다. 파일 시스템에서는 "3번째 레코드"처럼 위치로 접근할 수 있지만, 데이터베이스에서는 `WHERE 이름 = '김철수'`처럼 **데이터의 내용**으로 검색합니다.

### ✅ 핵심 문제 22 — 논리적/물리적 구조

> **문제**: 데이터베이스의 논리적 구조와 물리적 구조에 대한 설명으로 옳지 않은 것은?

**정답: "물리적 구조는 논리적 구조에서도 취급할 수 있다"는 설명이 옳지 않음**

- **논리적 구조**: 사용자 관점에서 본 데이터의 모습. 테이블, 컬럼 등 개념적인 구조
- **물리적 구조**: 실제 저장 장치에 어떻게 저장되는지. 저장 레코드(Stored Record), 파일 구조 등
- 물리적 구조는 논리적 구조와 **분리**됩니다. 물리적 저장 방식이 바뀌어도 사용자가 보는 논리적 구조는 바뀌지 않아야 합니다(데이터 독립성!). 따라서 논리적 구조에서 물리적 구조를 직접 취급하지 않습니다.

### ✅ 핵심 문제 23, 24 — DBMS의 장단점

> **문제 23**: DBMS를 이용하는 경우의 장점으로 옳지 않은 것은?
> **정답: "데이터 중복을 완벽하게 제거할 수 있다"는 설명이 옳지 않음**

> **문제 24**: DBMS의 장단점으로 옳지 않은 것은?
> **정답: "데이터 간의 복잡한 관계로 인하여 표준화가 어렵다"는 설명이 옳지 않음**

DBMS는 데이터 중복을 **최소화**할 수 있지만, **완전히 제거**하는 것은 불가능합니다. 성능상의 이유나 설계의 한계로 일부 중복이 발생할 수 있습니다. 또한 DBMS는 복잡한 관계를 표준화된 방식(관계형 모델, SQL 표준 등)으로 다루기 때문에 오히려 **표준화가 용이**합니다.

### ✅ 핵심 문제 20 — 데이터베이스 기본 개념

> **문제**: 데이터베이스 기본 개념에 대한 설명으로 옳지 않은 것은?

**정답: "개체 타입 — 개체에 속해 있는 실제 값들로 구성된 개체 어커런스의 집합"이라는 설명이 옳지 않음**

올바른 개념 정리:

| 용어 | 올바른 설명 |
|---|---|
| **속성 관계** | 하나의 개체를 기술하는 속성과 속성 사이의 관계 (예: 직원의 이름과 직원ID의 관계) |
| **개체 관계** | 개체 집합과 개체 집합 사이의 관계 (예: 직원과 부서의 관계) |
| **개체 타입** | 개체가 가지는 속성들의 집합을 정의한 틀/구조 (예: 직원 = {직원ID, 이름, 급여}) |
| **개체 어커런스** | 개체 타입에 속하는 실제 인스턴스(실제 값) |
| **도메인** | 어느 한 속성이 가질 수 있는 모든 가능한 값들의 집합 |

개체 타입은 "실제 값들의 집합"이 아니라 **속성들의 정의(틀)**이고, 실제 값들의 집합은 **개체 어커런스(Entity Occurrence)** 또는 **개체 인스턴스**라고 합니다.

## ⚠️ 주의사항과 시험 대비 핵심 팁

### 자주 혼동하는 개념들

시험에서 자주 틀리는 포인트들을 정리했습니다.

:::warning 혼동 주의 1: 자연 조인 vs 동등 조인
- **동등 조인**: 공통 컬럼이 결과에 **두 번** 나타남
- **자연 조인**: 공통 컬럼이 결과에 **한 번**만 나타남 (중복 제거)
- "중복 속성을 제거"한다는 설명이 나오면 → **자연 조인!**
:::

:::warning 혼동 주의 2: DDL vs DML
- **DDL**: CREATE, ALTER, DROP, TRUNCATE → 구조 변경
- **DML**: SELECT, INSERT, UPDATE, DELETE → 데이터 변경
- SELECT는 DDL이 아닙니다! SELECT는 DML입니다!
:::

:::warning 혼동 주의 3: 계층 모델 vs 네트워크 모델
- **계층 모델**: 트리 구조, 한 자식에 하나의 부모
- **네트워크 모델**: 그래프 구조, **오너-멤버** 관계, 한 자식에 여러 부모 가능
- "오너-멤버 관계"는 계층 모델이 아닌 **네트워크 모델**!
:::

:::danger 절대 혼동 금지: NULL 비교
- `WHERE column1 = NULL` → **항상 틀린 코드!**
- `WHERE column1 IS NULL` → **올바른 코드!**
- NULL은 = 로 비교할 수 없습니다. 항상 `IS NULL` / `IS NOT NULL` 사용!
:::

:::tip 데이터베이스 특성 암기법
데이터베이스의 4가지 특성을 **"실계동내"**로 암기하세요!
- **실**시간 접근
- **계**속적인 변화
- **동**시 공용
- **내**용에 의한 참조 (주소 ❌, 값/내용 ✅)
:::

:::tip DBMS 필수 기능 암기법
**"정조제"** — 정의, 조작, 제어 세 가지가 DBMS의 필수 기능입니다. "사전 기능"은 필수 기능이 아닙니다!
:::

## 📌 핵심 정리

이 문서에서 배운 핵심 개념들을 한눈에 정리합니다.

- **자연 조인(Natural Join)**: 동등 조인 후 중복 속성을 하나만 남기고 제거하는 조인 방법
- **B-트리 분열**: 차수 3인 B-트리에서 25, 40, 120 삽입 시 40이 루트로 올라가며 분열, 최종 루트는 40
- **SSD**: 반도체를 이용한 정보 저장 장치, HDD 대비 빠르고 소음·발열 적음
- **RAID 0**: 스트라이핑(Striping) 방식, 여러 디스크에 분산 저장하여 빠른 입출력 가능
- **회전 지연 시간**: 같은 실린더/트랙에 연속 배치 시 단축 가능한 디스크 접근 시간 요소
- **DBMS 제어 기능**: 무결성 검사, 접근 권한, 트랜잭션 병행 제어, 회복 작업 담당
- **DML 예비 컴파일러**: 응용 프로그램에 삽입된 DML 명령문을 추출하는 DBMS 구성요소
- **내장 SQL**: `EXEC SQL` 접두어 사용, 콜론(:)으로 호스트 변수 참조 가능
- **NULL 비교**: 반드시 `IS NULL` / `IS NOT NULL` 사용, `= NULL`은 절대 사용 금지
- **DISTINCT**: 중복 행 제거 SQL 명령어 (ALL은 중복 포함, ORDER BY는 정렬, GROUP BY는 그룹화)
- **DDL**: CREATE, ALTER, DROP — 구조 정의 언어 / **DML**: SELECT, INSERT, UPDATE, DELETE — 데이터 조작 언어
- **Division 연산**: 공통 속성은 이름이 같아야 하는 것이 아니라 도메인(값 범위)이 같아야 함
- **합병 가능 연산**: 합집합, 교집합, 차집합 → 합병 가능 / 카티션 프로덕트 → 합병 가능 불필요
- **참조 무결성**: 외래키는 참조하는 테이블의 기본키 값과 일치해야 함
- **계층 모델 vs 네트워크 모델**: 오너-멤버 관계는 계층 모델이 아닌 네트워크 모델
- **인덱스 순차 파일**: 수시 조회·수정 + 전체 순차 출력이 모두 필요한 경우 최적
- **비트맵 인덱스**: 변경이 잦은 테이블에는 부적합, OLAP(분석) 환경에 적합
- **데이터베이스 4대 특성**: 실시간 접근, 계속적인 변화, 동시 공용, 내용(값)에 의한 참조
- **DBMS 필수 기능**: 정의, 조작, 제어 — 사전 기능은 필수 기능 아님
- **데이터베이스 vs 파일 시스템**: 데이터 중복 배제 + 데이터 독립성 유지가 데이터베이스의 핵심 특징
- **DBMS 중복 제거**: 완벽하게 제거가 아닌 **최소화**가 올바른 표현
- **개체 타입**: 실제 값들의 집합이 아닌 속성들의 정의(틀), 실제 값의 집합은 개체 어커런스
- **논리적 구조**: 물리적 구조와 독립적으로 유지되어야 하며 서로 직접 취급하지 않음
