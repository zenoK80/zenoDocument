# 🗄️ DBMS 구성 요소와 무결성 제약조건

데이터베이스를 공부하다 보면 "DBMS가 정확히 어떤 일을 하는가?", "무결성이란 무엇인가?", "SQL 명령어는 어떻게 분류되는가?" 같은 질문에 부딪히게 됩니다. 이 문서는 그런 질문들에 명확하게 답하기 위해, 시험에 자주 등장하는 핵심 개념들을 초보자도 이해할 수 있는 언어로 하나씩 풀어나갑니다. 비유와 예시를 아끼지 않고, 각 개념이 왜 필요한지부터 차근차근 설명할 테니 편하게 따라오세요.

## 🏗️ DBMS란 무엇이고, 왜 필요한가?

### 파일 시스템의 한계

컴퓨터가 처음 데이터를 저장하던 방식은 **파일 시스템(File System)** 이었습니다. 마치 서랍장에 서류를 넣어두는 것처럼, 각 프로그램이 자기 서류 파일을 따로따로 만들어 보관했습니다. 급여 프로그램은 급여 파일을, 인사 프로그램은 인사 파일을 따로 만들었죠.

이 방식의 가장 큰 문제는 **데이터 중복(Data Redundancy)** 입니다. 직원 홍길동의 이름, 주소, 전화번호가 급여 파일에도, 인사 파일에도, 복리후생 파일에도 각각 저장됩니다. 홍길동이 이사를 가면 모든 파일을 일일이 찾아 수정해야 하고, 하나라도 빠트리면 데이터가 서로 달라지는 **불일치 문제**가 생깁니다.

또한 파일 시스템에서는 데이터가 특정 프로그램의 구조에 맞게 설계되기 때문에, 다른 프로그램이 같은 데이터를 쓰려면 복잡한 변환 작업이 필요합니다. 이것을 **데이터 종속성(Data Dependency)** 이라고 합니다.

> **DBMS(Database Management System, 데이터베이스 관리 시스템)** 는 이런 문제를 해결하기 위해 등장했습니다. 데이터를 한 곳에 통합해서 관리하고, 중복을 줄이며, 여러 사람이 동시에 안전하게 사용할 수 있는 환경을 제공합니다.

### 데이터베이스 시스템의 핵심 특성

데이터베이스 시스템이 파일 시스템과 구별되는 가장 중요한 두 가지 특성은 다음과 같습니다.

| 특성 | 파일 시스템 | 데이터베이스 시스템 |
|---|---|---|
| 데이터 중복 | 중복 유지 (각 프로그램마다 별도 파일) | **중복 배제** (통합 관리) |
| 데이터 독립성 | 독립성 없음 (프로그램에 종속) | **독립성 유지** (프로그램과 데이터 분리) |
| 동시 접근 | 제한적 | 다수 사용자 동시 접근 가능 |
| 일관성 | 유지 어려움 | 유지 용이 |

**자료의 중복 배제**란 같은 데이터를 여러 곳에 반복 저장하지 않는다는 의미입니다. **자료의 독립성 유지**란 데이터의 구조가 바뀌어도 응용 프로그램을 수정할 필요가 없다는 의미입니다. 마치 도서관 사서가 책을 어느 서가에 옮겨도, 도서관 카탈로그 시스템만 업데이트하면 이용자는 여전히 같은 방법으로 책을 찾을 수 있는 것처럼요.

:::info 데이터베이스의 4대 일반 특성
데이터베이스는 다음 네 가지 특성을 갖습니다:
- **실시간 접근(Real-time Accessibility)**: 질의(Query)에 대해 실시간 처리 및 응답이 가능합니다.
- **계속적인 변화(Continuous Evolution)**: 삽입, 삭제, 갱신을 통해 항상 최신 데이터를 유지합니다.
- **동시 공용(Concurrent Sharing)**: 다수의 사용자가 동시에 같은 데이터를 이용할 수 있습니다.
- **내용에 의한 참조(Content Reference)**: 데이터를 위치나 주소가 아닌 **데이터 값(내용)** 으로 찾습니다. "홍길동을 찾아라"처럼 값으로 검색하는 것이죠.
:::

:::warning 자주 틀리는 함정!
"데이터베이스는 데이터를 위치나 주소로 찾는다"는 설명은 **틀린 설명**입니다. 데이터베이스는 반드시 **데이터 값(내용)에 의해 참조**됩니다. 위치나 주소로 접근하는 것은 낮은 수준의 파일/메모리 접근 방식입니다.
:::

## ⚙️ DBMS의 구성 요소와 필수 기능

### DBMS의 3대 필수 기능

DBMS가 제공해야 하는 필수 기능은 크게 세 가지입니다. 이를 **정의 기능**, **조작 기능**, **제어 기능**이라고 합니다. 여기서 **"사전 기능"은 DBMS의 필수 기능이 아닙니다.**

| 기능 | 설명 | 역할 비유 |
|---|---|---|
| **정의 기능** | 데이터 구조(스키마)를 정의하고 명세 | 건물의 설계도를 그리는 것 |
| **조작 기능** | 데이터 검색, 삽입, 삭제, 수정 | 건물 안에서 물건을 옮기고 배치하는 것 |
| **제어 기능** | 무결성, 보안, 병행제어, 회복 담당 | 건물 경비원과 안전 관리자 역할 |

### 제어 기능을 담당하는 DBMS 구성 요소

**제어(Control)** 기능은 DBMS에서 가장 중요한 역할 중 하나입니다. 제어 기능을 담당하는 구성 요소는 다음 네 가지 작업을 수행합니다.

1. **무결성 제약조건 만족 여부 검사**: 저장되는 데이터가 미리 정해놓은 규칙을 지키는지 확인합니다. 예를 들어, 학생의 학점이 0~4.5 사이여야 한다는 규칙이 있을 때, 5.0이라는 값이 들어오면 거부합니다.

2. **사용자 접근 권한 검사**: 이 데이터에 접근할 자격이 있는 사람인지 확인합니다. 마치 아파트 공동현관 비밀번호처럼, 권한이 있는 사람만 데이터를 보거나 수정할 수 있습니다.

3. **트랜잭션(Transaction, 작업 묶음)의 병행 제어**: 여러 사람이 동시에 같은 데이터를 수정할 때 충돌이 나지 않도록 조율합니다. 마치 은행 창구에서 같은 계좌에 두 사람이 동시에 출금하려 할 때 순서를 정해주는 것과 같습니다.

4. **회복(Recovery) 작업 수행**: 시스템이 갑자기 다운되거나 오류가 발생했을 때 데이터를 정상 상태로 복구합니다. 마치 자동저장 기능처럼 문제가 생겨도 이전 안전한 상태로 돌아갈 수 있습니다.

### DML 예비 컴파일러 (Precompiler)

**호스트 프로그래밍 언어(Host Language)** 란 Java, C, Python처럼 일반적인 프로그래밍 언어를 말합니다. 이런 언어로 작성된 프로그램 안에 SQL 명령문을 직접 삽입해서 쓰는 방식을 **내장 SQL(Embedded SQL)** 이라고 합니다.

문제는 Java 컴파일러가 SQL 문법을 모른다는 것입니다. 그래서 Java 코드와 SQL이 뒤섞인 소스 파일을 본격적으로 컴파일하기 전에, **SQL 명령문만 골라내서 처리해주는 전처리기**가 필요합니다. 이것이 바로 **DML 예비 컴파일러(DML Precompiler)** 입니다.

> 쉽게 말해, 한국어와 영어가 섞인 문서에서 영어 부분만 따로 추출해 번역하는 도구라고 생각하면 됩니다. DML 예비 컴파일러는 소스 코드에서 SQL 명령문을 추출해 데이터베이스가 이해할 수 있는 형태로 변환합니다.

## 📋 내장 SQL(Embedded SQL)과 SQL 기초 문법

### 내장 SQL이란?

**내장 SQL(Embedded SQL)** 은 C, Java 같은 일반 프로그래밍 언어 코드 안에 SQL 문을 직접 삽입하는 방식입니다. 데이터베이스 작업이 필요한 부분에서만 SQL을 쓰고, 나머지 로직은 호스트 언어로 작성합니다.

내장 SQL의 핵심 규칙들을 알아봅시다.

**✅ 올바른 내장 SQL 규칙:**

```c
/* 내장 SQL 예시 (C 언어) */

/* 규칙 1: 내장 SQL문 앞에는 반드시 EXEC SQL을 붙인다 */
EXEC SQL SELECT salary INTO :host_salary FROM emp WHERE emp_id = :host_id;

/* 규칙 2: 호스트 변수(host variable)는 콜론(:)을 앞에 붙여 참조한다 */
/* :host_salary, :host_id 가 바로 호스트 변수(주언어 변수)다 */

/* 규칙 3: 주언어 변수와 DB 필드는 같은 이름을 가질 수 있다 */
/* 예: C변수 salary와 DB컬럼 salary가 같은 이름이어도 된다 */
/* 콜론(:)으로 구분하기 때문에 혼동되지 않는다 */
```

내장 SQL에서 **자주 출제되는 함정**이 있습니다. 바로 "SQL문은 주언어 변수(host variable)의 참조를 포함할 수 없다"는 설명입니다. 이것은 **틀린 설명**입니다. 내장 SQL은 오히려 호스트 변수를 참조하기 위해 만들어진 것이며, 콜론(`:`)을 붙여 호스트 변수를 SQL 안에서 자유롭게 참조할 수 있습니다.

:::warning 내장 SQL의 중요한 제약
내장 SQL에서 호스트 변수의 **데이터 타입**은 해당 데이터베이스 필드의 SQL 데이터 타입과 **반드시 일치**해야 합니다. 예를 들어, DB에서 `salary` 컬럼이 `INTEGER` 타입이면, C 언어에서도 `int` 타입 변수를 사용해야 합니다. 타입이 맞지 않으면 런타임 오류나 데이터 손실이 발생할 수 있습니다.
:::

### NULL 값 처리 — IS NULL vs = NULL

SQL을 처음 배울 때 가장 많이 실수하는 부분 중 하나가 바로 **NULL 값 검색**입니다. NULL은 "값이 없음"을 나타내는 특별한 상태입니다. 일반 숫자나 문자열처럼 `=` 연산자로 비교할 수 없습니다.

왜냐하면 NULL은 "알 수 없는 값(Unknown)"이기 때문입니다. "알 수 없는 값 = 알 수 없는 값"의 결과는 TRUE도 FALSE도 아닌 "알 수 없음(Unknown)"입니다. 그래서 NULL 비교에는 특별한 연산자 `IS NULL`을 사용해야 합니다.

```sql
-- ❌ 잘못된 방법: NULL을 = 으로 비교하면 항상 결과가 없음
SELECT * FROM myTable WHERE column1 = NULL;
-- 이 쿼리는 문법 오류는 아니지만, 항상 빈 결과를 반환한다
-- NULL = NULL은 Unknown이기 때문

-- ❌ 잘못된 방법: EQUALS라는 SQL 연산자는 존재하지 않는다
SELECT * FROM myTable WHERE column1 EQUALS NULL;
-- 문법 오류 발생

-- ❌ 잘못된 방법: NOT NULL은 NULL이 아닌 것을 찾는 구문이다
SELECT * FROM myTable WHERE column1 NOT NULL;
-- NULL인 것을 찾는 것이 아니라 NULL이 아닌 것을 찾게 됨 (게다가 올바른 문법도 아님)

-- ✅ 올바른 방법: IS NULL을 사용해야 한다
SELECT * FROM myTable WHERE column1 IS NULL;
-- column1의 값이 NULL인 모든 행을 반환
```

반대로 NULL이 아닌 값을 찾으려면 `IS NOT NULL`을 사용합니다.

```sql
-- ✅ NULL이 아닌 값 찾기
SELECT * FROM myTable WHERE column1 IS NOT NULL;

-- ✅ NULL을 가진 행 수 세기 (COUNT는 NULL을 제외하므로 주의)
SELECT COUNT(*) FROM myTable;           -- 전체 행 수 (NULL 포함)
SELECT COUNT(column1) FROM myTable;    -- column1이 NULL이 아닌 행 수만 세어짐
```

### DISTINCT — 중복 제거 명령어

데이터베이스에는 중복된 값이 많을 수 있습니다. 예를 들어, 직원 테이블에서 부서 목록을 조회하면 같은 부서 이름이 여러 번 나올 수 있습니다. 이때 **DISTINCT** 키워드를 사용하면 중복을 제거하고 유일한 값만 가져올 수 있습니다.

```sql
-- 직원 테이블에서 부서 목록 조회 (중복 있음)
SELECT department FROM employees;
-- 결과: 개발팀, 개발팀, 영업팀, 개발팀, 인사팀, 영업팀 ...

-- ✅ DISTINCT로 중복 제거
SELECT DISTINCT department FROM employees;
-- 결과: 개발팀, 영업팀, 인사팀 (각 부서가 한 번씩만 표시)
```

중복 제거와 관련하여 헷갈리기 쉬운 키워드들을 비교해 봅시다:

| 키워드 | 역할 | 예시 |
|---|---|---|
| **DISTINCT** | 중복 행 제거 | `SELECT DISTINCT dept FROM emp` |
| **ALL** | 중복 포함 전체 출력 (기본값) | `SELECT ALL dept FROM emp` |
| **ORDER BY** | 결과를 특정 컬럼 기준으로 정렬 | `ORDER BY salary DESC` |
| **GROUP BY** | 특정 컬럼 기준으로 그룹화 | `GROUP BY dept` |

### DDL vs DML — SQL 명령어 분류

SQL 명령어는 역할에 따라 크게 **DDL**, **DML**, **DCL**로 나뉩니다.

**DDL(Data Definition Language, 데이터 정의어)** 는 데이터베이스 구조(테이블, 인덱스 등)를 **만들고, 수정하고, 삭제**하는 명령어입니다. 마치 건물의 구조를 설계하고 변경하는 것과 같습니다.

- `CREATE`: 테이블이나 인덱스를 새로 만듭니다.
- `ALTER`: 기존 테이블 구조를 변경합니다.
- `DROP`: 테이블이나 인덱스를 완전히 삭제합니다.

**`SELECT`는 DDL 명령어가 아닙니다.** SELECT는 데이터를 조회하는 **DML(Data Manipulation Language, 데이터 조작어)** 에 속합니다.

```sql
-- ✅ DDL 명령어 예시

-- CREATE: 학생 테이블 생성
CREATE TABLE students (
    student_id INT PRIMARY KEY,   -- 기본키, 정수형
    name VARCHAR(50) NOT NULL,    -- 이름, 최대 50자, NULL 불가
    grade FLOAT                   -- 학점, 소수점 포함
);

-- ALTER: 테이블에 컬럼 추가
ALTER TABLE students ADD COLUMN email VARCHAR(100);

-- DROP: 테이블 완전 삭제 (복구 불가!)
DROP TABLE students;

-- ❌ DDL이 아닌 것: SELECT는 DML이다
-- SELECT * FROM students; -- 이것은 DDL이 아님
```

```sql
-- ✅ DML 명령어 예시

-- SELECT: 데이터 조회 (DML)
SELECT * FROM students WHERE grade >= 3.0;

-- INSERT: 데이터 삽입 (DML)
INSERT INTO students VALUES (1001, '홍길동', 4.2);

-- UPDATE: 데이터 수정 (DML)
UPDATE students SET grade = 4.5 WHERE student_id = 1001;

-- DELETE: 데이터 삭제 (DML)
DELETE FROM students WHERE student_id = 1001;
```

:::tip DDL vs DML 쉽게 기억하는 법
- **DDL**: 테이블 **구조** 를 다룬다 → CREATE, ALTER, DROP (건물 설계/개조/철거)
- **DML**: 테이블 안의 **데이터** 를 다룬다 → SELECT, INSERT, UPDATE, DELETE (건물 안에서 가구 배치/추가/이동/제거)
:::

## 🔗 무결성 제약조건 (Integrity Constraints)

### 무결성이란 무엇인가?

**무결성(Integrity)** 이란 데이터베이스에 저장된 데이터가 항상 **정확하고 일관되며 유효한 상태**를 유지하는 것을 말합니다. "결함이 없다(無缺)"는 한자 뜻 그대로, 데이터에 오류나 불일치가 없는 상태입니다.

왜 무결성이 필요할까요? 병원 데이터베이스에서 환자의 혈액형이 'Z형'으로 저장되거나, 주문 테이블에 존재하지 않는 고객 번호가 입력된다면 어떻게 될까요? 의료 사고나 업무 오류로 이어질 수 있습니다. 무결성 제약조건은 이런 잘못된 데이터가 처음부터 저장되지 못하도록 막는 **데이터 품질의 수문장** 역할을 합니다.

### 참조 무결성 (Referential Integrity)

**참조 무결성**은 관계형 데이터베이스에서 테이블 간의 관계를 일관성 있게 유지하는 규칙입니다. **외래키(Foreign Key)** 를 통해 두 테이블이 연결될 때 적용됩니다.

외래키란 무엇일까요? 마치 학생증처럼 생각해 보세요. 도서관 대출 기록에는 '학번'이 적혀 있습니다. 이 학번은 학생 테이블의 기본키(Primary Key)를 참조합니다. 도서관 대출 기록 테이블 입장에서 이 학번이 바로 **외래키(Foreign Key)** 입니다.

**참조 무결성의 핵심 규칙**: 외래키 값은 반드시 참조하는 테이블의 기본키 값으로 존재하거나, NULL이어야 합니다.

문제에서 나온 내용을 좀 더 정확히 설명하겠습니다. "릴레이션 R1에 속한 애트리뷰트의 조합인 외래키를 변경하려면, 이를 참조하고 있는 R2의 릴레이션의 기본키도 변경해야 한다"는 설명은 **참조 무결성**과 관련된 내용입니다.

```sql
-- 참조 무결성 예시

-- 부서 테이블 (참조되는 쪽 - 부모 테이블)
CREATE TABLE departments (
    dept_id INT PRIMARY KEY,      -- 기본키
    dept_name VARCHAR(50)
);

-- 직원 테이블 (참조하는 쪽 - 자식 테이블)
CREATE TABLE employees (
    emp_id INT PRIMARY KEY,
    emp_name VARCHAR(50),
    dept_id INT,
    -- dept_id는 departments 테이블의 dept_id를 참조하는 외래키
    FOREIGN KEY (dept_id) REFERENCES departments(dept_id)
);

-- ❌ 참조 무결성 위반: 존재하지 않는 부서(dept_id=999)에 직원 삽입
INSERT INTO employees VALUES (1, '홍길동', 999);
-- 오류 발생! departments 테이블에 dept_id=999가 없음

-- ✅ 올바른 삽입: 존재하는 부서에만 직원 할당 가능
INSERT INTO departments VALUES (10, '개발팀');
INSERT INTO employees VALUES (1, '홍길동', 10);  -- 정상 처리
```

### 개체 무결성 (Entity Integrity)

**개체 무결성**은 모든 테이블의 **기본키(Primary Key)** 는 반드시 유일한 값을 가져야 하며, **NULL이 될 수 없다**는 규칙입니다. 마치 주민등록번호처럼, 모든 사람은 유일한 번호를 가져야 하고 번호가 없는(NULL) 사람은 있을 수 없습니다.

### 도메인 무결성 (Domain Integrity)

**도메인 무결성**은 각 속성(컬럼)의 값이 **미리 정의된 도메인(값의 범위)** 에 속해야 한다는 규칙입니다. 예를 들어, 성별 컬럼은 '남' 또는 '여'만 허용, 나이 컬럼은 0~150 사이의 정수만 허용하는 식입니다.

| 무결성 종류 | 적용 대상 | 규칙 |
|---|---|---|
| **개체 무결성** | 기본키 | NULL 불가, 유일해야 함 |
| **참조 무결성** | 외래키 | 참조 대상 기본키에 반드시 존재해야 함 |
| **도메인 무결성** | 각 속성 값 | 미리 정의된 도메인 범위 내 값만 허용 |
| **널 무결성** | 특정 속성 | NULL을 허용하지 않도록 지정 |

## 📊 관계 대수(Relational Algebra) 핵심 연산

### 조인(Join) — 테이블 합치기

**조인(Join)** 은 두 개 이상의 테이블을 연관시켜 하나의 결과를 만드는 연산입니다. 마치 퍼즐 조각을 맞추는 것처럼, 공통된 컬럼을 기준으로 두 테이블을 합칩니다.

조인의 종류 중 **동등 조인(Equi Join)** 은 두 테이블에서 같은 값을 가진 행을 연결합니다. 그런데 동등 조인을 하면 조인 기준이 된 컬럼이 결과에 두 번 나타납니다. 예를 들어, 직원 테이블과 부서 테이블을 `dept_id`로 조인하면 결과에 `dept_id` 컬럼이 두 개 생깁니다.

이때 **중복되는 속성(컬럼) 하나를 제거**해서 깔끔하게 만든 것이 **자연 조인(Natural Join)** 입니다.

> **자연 조인(Natural Join)**: 동등 조인의 결과에서 중복되는 속성을 하나만 남기고 제거하는 조인 방법. 두 테이블에서 이름이 같은 컬럼을 자동으로 조인 조건으로 사용하고, 그 컬럼은 결과에 한 번만 나타납니다.

### 집합 연산 — 합병 가능한 연산자

관계 대수에서 집합 연산(합집합, 교집합, 차집합)을 수행하려면 두 릴레이션(테이블)이 **합병 가능(Union-Compatible)** 해야 합니다. 합병 가능하다는 것은 두 테이블의 **차수(Degree, 컬럼 수)가 같고**, **대응하는 속성의 도메인(데이터 타입과 의미)이 같아야** 한다는 조건입니다.

합집합(`∪`), 교집합(`∩`), 차집합(`-`)은 모두 합병 가능 조건이 필요합니다.

그러나 **카티션 프로덕트(Cartesian Product, 곱집합)** 는 두 테이블의 모든 행을 서로 조합하는 연산이기 때문에, 합병 가능 조건이 **필요하지 않습니다**. 두 테이블의 구조가 달라도 카티션 프로덕트는 수행할 수 있습니다.

| 연산 | 합병 가능 조건 필요? |
|---|---|
| 합집합 (UNION) | ✅ 필요 |
| 교집합 (INTERSECT) | ✅ 필요 |
| 차집합 (EXCEPT) | ✅ 필요 |
| **카티션 프로덕트** | ❌ 불필요 |

### 디비전(Division) 연산

**디비전(Division, R ÷ S)** 는 가장 이해하기 어려운 관계 대수 연산 중 하나입니다. "S에 있는 모든 것과 관련된 R의 튜플을 찾아라"는 의미를 가집니다.

마치 "모든 필수과목을 수강한 학생을 찾아라"처럼, S의 조건을 모두 만족하는 R의 행을 추출합니다.

디비전의 핵심 규칙:
- 나누어지는 릴레이션 R의 차수가 (m+n)이고, 나누는 릴레이션 S의 차수가 n이면, 결과 릴레이션의 차수는 **m**이 됩니다.
- R은 S의 모든 속성(Y)을 포함합니다.
- 중요한 조건: **공통 속성은 이름이 같은 것이 아니라, 도메인(값의 범위/타입)이 같아야 합니다.** "이름이 같아야 한다"는 설명은 **틀린 설명**입니다.

:::warning 디비전 연산 주의사항
"대응되는 애트리뷰트는 모두 이름들이 같다"는 설명은 **옳지 않습니다**. 디비전 연산에서 공통 속성의 조건은 이름의 동일함이 아니라 **도메인(값의 범위)이 같아야** 한다는 것입니다. 실제로 이름이 달라도 도메인이 같으면 연산이 가능합니다.
:::

## 💾 저장 구조와 인덱스 — 데이터를 빠르게 찾는 방법

### 파일 구조의 종류

데이터를 저장하는 파일 구조는 어떻게 사용하느냐에 따라 가장 적합한 형태가 다릅니다. 마치 물건을 보관하는 방식이 자주 꺼내보느냐, 순서대로 처리하느냐에 따라 달라지는 것처럼요.

**시나리오**: 급여 관리 프로그램에서 ①특정 직원 급여를 수시로 조회, ②수시로 수정, ③전 직원 급여명세서를 매월 출력해야 한다면 어떤 파일 구조가 적합할까요?

| 파일 구조 | 특징 | 적합한 상황 |
|---|---|---|
| **순차 파일** | 데이터를 순서대로 저장, 전체 스캔에 적합 | 전체 처리만 할 때 |
| **인덱스 순차 파일** | 순차 접근 + 인덱스를 통한 직접 접근 모두 가능 | 수시 조회/수정 + 전체 출력 **모두** 필요할 때 |
| **직접 파일** | 해시 함수로 직접 접근, 랜덤 접근 최적 | 특정 레코드만 빠르게 찾을 때 |
| **힙 파일** | 입력 순서대로 저장, 단순 | 정렬이나 검색 필요 없을 때 |

위 시나리오에서는 수시 조회/수정(랜덤 접근)과 전체 출력(순차 접근) **모두** 필요합니다. 따라서 두 가지를 모두 지원하는 **인덱스 순차 파일(Indexed Sequential File)** 이 가장 적합합니다.

### 디스크 접근 시간 구성 요소

하드 디스크에서 데이터를 읽는 시간은 세 가지로 나뉩니다. 마치 도서관에서 책을 찾는 과정처럼 생각해 보세요.

1. **탐구 시간(Seek Time)**: 읽기/쓰기 헤드가 해당 **트랙(원형 경로)** 으로 이동하는 시간입니다. 마치 도서관에서 책이 있는 서가 앞까지 걸어가는 시간과 같습니다.

2. **회전 지연 시간(Rotational Latency)**: 헤드가 트랙에 도달한 후, 원하는 데이터가 헤드 아래로 올 때까지 디스크가 회전하기를 기다리는 시간입니다. 마치 회전하는 선반에서 원하는 물건이 내 앞에 올 때까지 기다리는 것과 같습니다.

3. **데이터 전송 시간(Data Transfer Time)**: 실제로 데이터를 읽어 메모리로 전송하는 시간입니다.

**같은 실린더 또는 같은 트랙에 연속적으로 데이터 블록을 배치하면 단축할 수 있는 것은 바로 탐구 시간(Seek Time)입니다.** 헤드를 덜 움직여도 되기 때문입니다.

:::info 실린더(Cylinder)란?
여러 장의 디스크 플래터에서 같은 위치(반지름)에 있는 트랙들의 집합을 **실린더**라고 합니다. 실린더 단위로 데이터를 배치하면 헤드를 거의 움직이지 않고도 많은 데이터를 읽을 수 있어 탐구 시간을 크게 줄일 수 있습니다.
:::

### SSD — 반도체 저장 장치

**SSD(Solid State Drive)** 는 하드 디스크 드라이브(HDD)와 달리 **반도체(메모리 칩)** 를 이용해 데이터를 저장하는 장치입니다. 기계적으로 움직이는 부품이 없기 때문에 다음과 같은 장점이 있습니다:

- ⚡ **속도가 빠름**: HDD보다 데이터 접근 속도가 훨씬 빠릅니다.
- 🔕 **기계적 지연이나 소음이 없음**: 회전하는 플래터와 움직이는 헤드가 없으므로 소음과 진동이 없습니다.
- 🌡️ **발열이 적음**: 기계 마찰이 없어 열이 덜 납니다.
- 💧 **실패율(고장률)이 낮음**: 충격에 강합니다.
- 📱 **소형화·경량화 가능**: 노트북, 스마트폰 등에 적합합니다.

### RAID — 여러 디스크를 하나처럼

**RAID(Redundant Array of Independent Disks, 독립 디스크 배열)** 는 여러 개의 하드 디스크를 묶어 하나의 저장 장치처럼 사용하는 기술입니다.

**RAID 0**은 데이터를 여러 디스크에 **분산 저장(Striping)** 하는 방식입니다. 마치 짐을 여러 사람이 나눠 드는 것처럼, 데이터를 여러 디스크에 나눠 저장하고 동시에 읽고 씁니다. 덕분에 **입출력 속도가 매우 빠릅니다**. 그러나 디스크 하나가 고장나면 전체 데이터를 잃는 위험이 있습니다(복구 불가).

| RAID 레벨 | 방식 | 특징 |
|---|---|---|
| **RAID 0** | Striping (분산) | 속도 최고, 안전성 없음 |
| **RAID 1** | Mirroring (복사) | 데이터 이중화, 안전성 높음 |
| **RAID 0+1** | Striping + Mirroring | 속도와 안전성 모두 확보 |
| **RAID 6** | 이중 패리티 | 두 개의 디스크 동시 고장에도 복구 가능 |

### B-트리와 비트맵 인덱스

**B-트리(B-Tree)** 는 데이터베이스에서 가장 널리 사용되는 인덱스 자료구조입니다. 마치 책의 목차처럼, 원하는 데이터를 빠르게 찾아갈 수 있도록 키 값들을 정렬된 트리 구조로 유지합니다.

**차수가 3인 B-트리**에서는 각 노드에 최대 2개의 키와 3개의 자식 포인터를 가질 수 있습니다. 키 값 25, 40, 120, 10, 55 순으로 삽입할 때의 과정을 살펴봅시다.

1. 25 삽입: 루트 노드에 [25]
2. 40 삽입: 루트 노드에 [25, 40]
3. 120 삽입: 노드가 가득 차면 분할(Split) 발생. [25, 40, 120] → 중간 값 **40**이 루트로 올라가고, [25]와 [120]이 자식이 됨
4. 10 삽입: 25보다 작으므로 왼쪽 자식 [25]에 삽입 → [10, 25]
5. 55 삽입: 40~120 사이이므로 오른쪽 자식 [120]에 삽입 → [55, 120]

최종 루트 값은 **40**입니다.

**비트맵 인덱스(Bitmap Index)** 는 각 고유 값에 대해 비트열(0과 1)로 인덱스를 구성하는 방식입니다.

:::danger 비트맵 인덱스 오해 주의
"삽입이나 갱신이 많이 발생하는 테이블에 적용한다"는 설명은 **완전히 잘못된 설명**입니다. 비트맵 인덱스는 갱신 시 비트맵 전체를 재구성해야 하므로 **삽입/갱신이 많은 OLTP 환경에는 매우 부적합합니다**. 비트맵 인덱스는 **낮은 분포도(카디널리티)** 를 가진 컬럼(예: 성별, 지역 코드 등)이 있는 매우 큰 테이블에서, **읽기 위주의 OLAP 환경**에서 사용하는 것이 적합합니다.
:::

## 🗂️ 논리적 데이터 모델의 종류

### 데이터 모델이란?

**데이터 모델(Data Model)** 은 데이터의 구조와 관계를 표현하는 방식입니다. 마치 지도가 현실 세계를 2D로 표현하는 것처럼, 데이터 모델은 현실의 정보를 컴퓨터가 이해할 수 있는 구조로 나타냅니다.

**논리적 데이터 모델**의 종류를 정확히 알아야 합니다.

| 데이터 모델 | 구조 표현 방식 | 특징 |
|---|---|---|
| **관계형 데이터 모델** | 테이블(릴레이션)의 집합 | 현재 가장 널리 사용, SQL로 조작 |
| **네트워크 데이터 모델** | 그래프(Graph) 구조 | 다대다(N:M) 관계 직접 표현 가능 |
| **계층적 데이터 모델** | 트리(Tree) 구조 | 부모-자식 관계, 일대다(1:N)만 표현 가능 |
| **객체지향 데이터 모델** | 객체(Object)와 상속(Inheritance) 구조 | 복잡한 데이터 타입과 행동 포함 |

여기서 중요한 함정이 있습니다. **계층적 데이터 모델**은 "오너-멤버 관계"가 아닙니다. **오너-멤버(Owner-Member) 관계**는 **네트워크 데이터 모델**의 특징입니다. 계층적 데이터 모델은 **부모-자식(Parent-Child)** 관계의 **트리 구조**로 데이터를 표현합니다.

:::warning 자주 혼동되는 개념
- **계층적 데이터 모델**: 트리(Tree) 구조, **부모-자식** 관계
- **네트워크 데이터 모델**: 그래프(Graph) 구조, **오너-멤버** 관계

"계층적 데이터 모델이 오너-멤버 관계의 계층적 그래프 구조로 표현한다"는 설명은 **틀린 설명**입니다. 오너-멤버 관계는 네트워크 모델의 특징입니다.
:::

## 📚 데이터베이스 기본 개념 용어 정리

### 핵심 용어들

데이터베이스에서 자주 등장하는 기본 용어들을 정확히 이해하고 있어야 합니다.

**도메인(Domain)**: 관계 데이터 모델에서 어느 한 속성이 가질 수 있는 **값들의 집합(총칭)** 입니다. 예를 들어, '성별' 속성의 도메인은 {'남', '여'}, '학점' 속성의 도메인은 0.0에서 4.5 사이의 실수 집합입니다.

**개체 타입(Entity Type)**: 공통된 특성을 가진 개체들의 **집합적 정의**입니다. 예를 들어, '학생'이라는 개체 타입은 학번, 이름, 학년 등의 속성을 가집니다. **개체 어커런스(Entity Occurrence)** 는 개체 타입에 속하는 실제 개별 개체(인스턴스)입니다.

:::warning 자주 틀리는 개체 타입 정의
"개체 타입: 개체에 속해 있는 실제 값들로 구성된 개체 어커런스의 집합"이라는 설명은 **틀린 설명**입니다. 이것은 개체 타입이 아니라 **개체 집합(Entity Set)** 또는 **개체 어커런스** 에 대한 설명에 가깝습니다. 개체 타입은 어커런스들이 공유하는 **구조(속성 목록)의 정의**입니다. 실제 값들의 집합은 개체 집합(인스턴스들의 모음)입니다.
:::

**속성 관계(Attribute Relationship)**: 하나의 개체를 기술하는 속성들 사이의 관계입니다. 예를 들어, 학생 개체에서 '학번'과 '이름' 사이의 관계.

**개체 관계(Entity Relationship)**: 개체 집합과 개체 집합 사이의 관계입니다. 예를 들어, '학생' 집합과 '강의' 집합 사이의 '수강' 관계.

### 논리적 구조 vs 물리적 구조

**논리적 구조(Logical Structure)** 는 사용자 관점에서 데이터가 어떻게 보이는지, 즉 **사용자가 생각하는 데이터의 논리적 표현**입니다. 테이블, 뷰, 인덱스 같은 개념들이 여기에 해당합니다.

**물리적 구조(Physical Structure)** 는 데이터가 실제 저장 장치(디스크)에 어떻게 저장되는지, 즉 **저장 장치 입장에서 본 데이터의 구조**입니다. 물리적 구조에서 다루는 레코드는 **저장 레코드(Stored Record)** 라고 합니다.

중요한 것은 **물리적 구조는 논리적 구조에서 취급할 수 없습니다.** 사용자는 논리적 구조만 보고 데이터를 다루며, 실제 저장 방식(물리적 구조)은 DBMS가 알아서 처리합니다. 이것이 바로 **데이터 독립성**의 핵심입니다.

### DBMS 장단점 — 자주 틀리는 함정

DBMS를 사용하면 데이터 중복을 최소화할 수 있지만, **완벽하게 제거**하기는 어렵습니다. 성능이나 특정 업무 요구 때문에 의도적으로 일부 데이터를 중복 저장하는 경우도 있습니다.

:::warning DBMS 장점 함정 문제
"DBMS를 사용하면 데이터 중복을 **완벽하게** 제거할 수 있다"는 설명은 **틀린 설명**입니다. DBMS는 중복을 **최소화**하는 것이지, 완전히 없애는 것은 아닙니다. 실제로 성능 최적화나 분산 처리를 위해 의도적인 중복(반정규화)이 사용되기도 합니다.

또한, "데이터 간의 복잡한 관계로 인하여 표준화가 어렵다"는 것은 DBMS의 **단점**이지 장점이 아닙니다. DBMS가 장단점을 함께 가진다는 점을 꼭 기억하세요.
:::

## ✅ 주의사항 및 시험 대비 핵심 팁

### 자주 출제되는 함정 총정리

시험에서 자주 나오는 헷갈리는 개념들을 한 번에 정리합니다.

:::tip 헷갈리는 개념 구분법

**SQL 문법 관련:**
- NULL 비교: `= NULL` ❌ → `IS NULL` ✅
- DDL vs DML: SELECT는 DDL이 아니라 **DML**
- 중복 제거: `ALL` ❌ → `DISTINCT` ✅

**DBMS 구성 요소:**
- 무결성 검사 + 접근권한 + 병행제어 + 회복 → **제어 기능 담당 구성요소**
- 내장 SQL에서 DML 추출 → **DML 예비 컴파일러(Precompiler)**

**무결성 제약조건:**
- 기본키 → **개체 무결성** (NULL 불가, 유일해야 함)
- 외래키 → **참조 무결성** (참조 대상 기본키에 존재해야 함)
- 속성 값 범위 → **도메인 무결성**

**데이터 모델:**
- 트리 구조, 부모-자식 → **계층적 모델**
- 그래프 구조, 오너-멤버 → **네트워크 모델** ← 자주 혼동!

**RAID:**
- Striping(분산, 속도 우선) → **RAID 0**
- Mirroring(복사, 안전 우선) → **RAID 1**

**비트맵 인덱스:**
- 삽입/갱신 많은 테이블에 부적합 ← 반드시 기억!
- OLAP 환경, 낮은 분포도 컬럼에 적합

**합병 가능한 연산:**
- 합집합, 교집합, 차집합 → 합병 가능 조건 필요
- **카티션 프로덕트** → 합병 가능 조건 **불필요**

**데이터베이스 특성:**
- 데이터 참조 방식: 위치/주소 ❌ → **내용(값)에 의한 참조** ✅
:::

## 📝 핵심 정리

이 문서에서 배운 핵심 개념들을 한 눈에 정리합니다.

- **DBMS는 데이터 중복 배제와 데이터 독립성 유지**를 파일 시스템과 구별되는 핵심 특성으로 제공한다.
- **DBMS 3대 필수 기능**은 정의 기능, 조작 기능, 제어 기능이며, "사전 기능"은 필수 기능이 아니다.
- **제어 기능**: 무결성 검사, 접근 권한 검사, 트랜잭션 병행 제어, 회복 작업을 담당한다.
- **DML 예비 컴파일러**: 호스트 언어 소스 코드에서 내장 SQL 명령문을 추출하는 구성 요소다.
- **내장 SQL**: `EXEC SQL`을 앞에 붙이고, 호스트 변수는 콜론(`:`)으로 참조하며, 호스트 변수 타입은 DB 필드 타입과 일치해야 한다. SQL 내에서 호스트 변수 참조는 **가능**하다.
- **NULL 비교**: 반드시 `IS NULL`을 사용해야 하며, `= NULL`은 항상 빈 결과를 반환한다.
- **DISTINCT**: SQL에서 중복을 제거하는 키워드다.
- **SELECT는 DDL이 아닌 DML**이다. DDL은 CREATE, ALTER, DROP이다.
- **참조 무결성**: 외래키는 참조하는 테이블의 기본키에 반드시 존재해야 하는 규칙이다.
- **개체 무결성**: 기본키는 NULL이 될 수 없고 유일해야 한다.
- **자연 조인(Natural Join)**: 동등 조인에서 중복 속성 하나를 제거한 조인 방식이다.
- **카티션 프로덕트**는 합병 가능 조건이 필요 없는 유일한 집합 연산이다.
- **디비전 연산**: 공통 속성 조건은 "이름이 같아야 함"이 아니라 "도메인이 같아야 함"이다.
- **인덱스 순차 파일**: 수시 조회/수정(랜덤 접근)과 전체 출력(순차 접근) 모두 필요할 때 적합하다.
- **탐구 시간(Seek Time)**: 같은 실린더/트랙에 데이터를 연속 배치하면 단축할 수 있다.
- **SSD**: 반도체를 이용한 저장 장치로 속도 빠름, 소음/발열 적음, 소형화 가능하다.
- **RAID 0**: Striping 방식으로 여러 디스크에 분산 저장하여 입출력 속도를 높인다.
- **비트맵 인덱스**: OLAP 환경, 낮은 분포도 컬럼에 적합하며 삽입/갱신이 많은 환경에는 **부적합**하다.
- **계층적 데이터 모델**: 트리(부모-자식) 구조이며, 오너-멤버 관계는 **네트워크 모델**의 특징이다.
- **데이터베이스 참조**: 위치/주소가 아닌 **데이터 값(내용)** 으로 참조한다.
- **DBMS의 데이터 중복**: 완벽 제거가 아닌 **최소화**가 정확한 표현이다.
- **물리적 구조**: 저장 레코드를 다루며, 논리적 구조에서는 직접 취급하지 않는다.
