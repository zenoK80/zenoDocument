# 💾 디스크 저장 구조, 인덱스 및 RAID — 데이터베이스 저장의 모든 것

데이터베이스를 공부하다 보면 "데이터가 도대체 어디에, 어떻게 저장되는 걸까?"라는 질문이 자연스럽게 떠오릅니다. 화면에 보이는 표(테이블)는 논리적인 표현일 뿐이고, 실제 데이터는 하드디스크나 SSD 같은 물리적인 저장 장치에 특정한 구조로 기록됩니다. 이 문서에서는 **디스크 저장 구조**, **인덱스(색인)**, **RAID(디스크 묶음 기술)** 를 중심으로, 데이터베이스와 파일 시스템의 차이, B-트리 인덱스, 비트맵 인덱스, 관계 대수 연산자, SQL 핵심 명령어, DBMS의 구성 요소까지 초보자도 이해할 수 있도록 아주 자세하게 설명합니다.

---

## 🗂️ 1. 데이터베이스 vs 파일 시스템 — 왜 DBMS가 필요한가?

### 파일 시스템의 한계

컴퓨터가 처음 등장했을 때, 데이터는 단순한 **파일(file)** 형태로 저장되었습니다. 예를 들어 급여 관리 프로그램, 인사 관리 프로그램, 회계 프로그램이 각각 따로 파일을 만들어서 직원 이름, 부서, 급여 정보를 저장했습니다. 이렇게 되면 같은 직원 정보가 세 군데 파일에 중복으로 저장되고, 한 곳에서 수정하면 다른 곳은 여전히 옛날 정보를 갖고 있어 **데이터 불일치(inconsistency)** 가 발생합니다. 마치 집 안 여러 군데에 똑같은 메모지를 붙여뒀다가 한 장만 고치고 나머지는 잊어버리는 상황과 같습니다.

### DBMS의 등장

**DBMS(Database Management System, 데이터베이스 관리 시스템)** 는 이런 문제를 해결하기 위해 등장했습니다. DBMS는 데이터를 중앙에서 통합 관리하고, 여러 사용자가 동시에 접근할 수 있도록 지원합니다.

> **DBMS의 핵심 특징: "자료의 중복 배제와 자료의 독립성 유지"**
> 파일 시스템은 데이터가 중복되고 프로그램에 종속되지만, DBMS는 중복을 최소화하고 데이터와 프로그램을 독립적으로 유지합니다.

### 데이터베이스의 4가지 일반적 특성

데이터베이스는 다음 네 가지 특성을 가집니다.

| 특성 | 설명 |
|------|------|
| **실시간 접근(Real-time Accessibility)** | 사용자의 질의(질문)에 즉시 응답할 수 있어야 합니다 |
| **계속적인 변화(Continuous Evolution)** | 삽입, 삭제, 갱신을 통해 항상 최신 데이터를 유지합니다 |
| **동시 공용(Concurrent Sharing)** | 여러 사용자가 동시에 같은 데이터를 이용할 수 있습니다 |
| **내용에 의한 참조(Content Reference)** | 데이터의 위치나 주소가 아닌, **데이터 값(내용)** 으로 데이터를 찾습니다 |

:::warning 자주 틀리는 포인트
데이터베이스는 **"주소에 의한 참조"가 아니라 "내용(값)에 의한 참조"** 를 합니다.  
파일 시스템은 특정 주소나 위치로 데이터를 찾지만, 데이터베이스는 `WHERE 이름 = '홍길동'`처럼 **값으로** 데이터를 검색합니다. 시험에서 "주소에 의하여 데이터를 찾는다"는 선택지가 나오면 틀린 설명입니다.
:::

### DBMS의 장단점 비교

DBMS가 파일 시스템보다 항상 좋은 것은 아닙니다. 장단점을 함께 알아야 합니다.

| 구분 | DBMS의 장점 | DBMS의 한계 |
|------|------------|------------|
| 중복 | 데이터 중복 **최소화** (완벽 제거는 불가) | 설계가 복잡해질 수 있음 |
| 일관성 | 데이터 일관성 유지 용이 | 초기 구축 비용이 높음 |
| 독립성 | 응용 프로그램과 데이터 독립 | 전문 관리자(DBA) 필요 |
| 동시성 | 여러 사용자 동시 접근 지원 | 복잡한 관계로 표준화 어려울 수 있음 |

:::danger 절대 착각하지 마세요
**"DBMS는 데이터 중복을 완벽하게 제거한다"** — 이것은 **틀린** 설명입니다!  
DBMS는 중복을 **최소화(minimize)** 할 뿐, 완벽한 제거는 현실적으로 불가능합니다. 외래키 참조나 정규화 과정에서 일부 중복이 남을 수 있습니다.
:::

## 🏗️ 2. DBMS의 구성 요소와 내장 SQL

### DBMS의 필수 기능

DBMS는 크게 세 가지 **필수 기능**을 갖습니다.

- **정의 기능(Definition)**: 데이터의 구조(스키마)를 정의합니다. DDL(Data Definition Language)을 통해 테이블을 만들고 수정하고 삭제합니다.
- **조작 기능(Manipulation)**: 데이터를 검색, 삽입, 수정, 삭제합니다. DML(Data Manipulation Language)이 여기에 해당합니다.
- **제어 기능(Control)**: 데이터의 정확성과 안전성을 유지합니다. 무결성 검사, 보안, 병행 제어, 회복 작업이 포함됩니다.

:::info DBMS의 필수 기능은 3가지
정의(Definition), 조작(Manipulation), 제어(Control) — 이 세 가지가 DBMS의 **필수 기능**입니다.  
"사전 기능(Dictionary Function)"은 DBMS의 구성 요소 중 하나일 수 있지만, **필수 기능**으로 분류되지 않습니다.
:::

### DBMS 주요 구성 요소

DBMS 내부에는 다양한 역할을 하는 구성 요소들이 있습니다.

**① 데이터 제어 처리기 (Data Control Processor)**

마치 도서관 사서처럼, 누가 어떤 데이터에 접근할 수 있는지 감독합니다. 구체적으로는 다음 역할을 수행합니다.
- **무결성 제약조건(Integrity Constraint)** 만족 여부 검사: 데이터가 올바른 형식을 지키는지 확인합니다
- **사용자 접근 권한(Access Control)** 검사: 허가받지 않은 사용자가 데이터에 접근하지 못하도록 막습니다
- **트랜잭션(Transaction, 작업 묶음)의 병행 제어**: 여러 사용자가 동시에 같은 데이터를 수정할 때 충돌이 생기지 않도록 조율합니다
- **회복(Recovery) 작업 수행**: 시스템 오류가 생겼을 때 데이터를 이전 상태로 복구합니다

**② DML 프리컴파일러 (DML Precompiler)**

응용 프로그램(C언어, Java 등으로 작성된 프로그램) 안에 숨어 있는 SQL 명령문을 찾아서 추출하는 역할을 합니다. 마치 긴 편지 안에서 특정 키워드가 포함된 문장만 골라내는 것과 같습니다. 이렇게 추출된 SQL은 별도로 처리됩니다.

### 내장 SQL (Embedded SQL)

**내장 SQL(Embedded SQL)** 은 C, Java, COBOL 같은 **호스트 프로그래밍 언어(주언어)** 로 작성된 프로그램 안에 SQL 문장을 직접 삽입해서 사용하는 방식입니다. 마치 한국어 문장 안에 영어 단어를 섞어 쓰는 것처럼, 일반 프로그램 코드 안에 데이터베이스 명령문을 끼워 넣습니다.

내장 SQL의 주요 특징을 알아봅시다.

```c
/* 내장 SQL 예시 - C 언어 안에 SQL 삽입 */

/* ✅ 올바른 내장 SQL 사용 예시 */
EXEC SQL                          /* 내장 SQL 시작을 알리는 키워드 */
    SELECT salary                 /* 급여 컬럼 조회 */
    INTO :host_salary             /* 결과를 호스트 변수에 저장 (콜론으로 구분) */
    FROM employee                 /* employee 테이블에서 */
    WHERE emp_id = :host_emp_id;  /* 호스트 변수 값으로 조건 지정 */
```

내장 SQL에 대해 반드시 알아야 할 규칙들이 있습니다.

**규칙 1**: 내장 SQL문 앞에는 반드시 **`EXEC SQL`** 을 붙입니다. 이것이 "지금부터는 SQL 문장이에요"라고 컴파일러에게 알려주는 신호입니다.

**규칙 2**: SQL문은 **호스트 변수(host variable, 주언어 변수)를 참조할 수 있습니다.** 이때 호스트 변수 앞에는 콜론(`:`)을 붙여서 SQL 키워드와 구분합니다. 예를 들어 `:host_salary`처럼 씁니다.

**규칙 3**: 주언어 변수와 데이터베이스 필드(컬럼)는 **같은 이름을 가질 수 있습니다.** 같은 이름이더라도 콜론(`:`)이 있으면 호스트 변수, 없으면 데이터베이스 필드로 구분합니다.

**규칙 4**: 호스트 변수의 데이터 타입은 대응하는 데이터베이스 필드의 SQL 데이터 타입과 **일치해야 합니다.** 예를 들어 데이터베이스 컬럼이 `INTEGER`면 호스트 변수도 정수형이어야 합니다.

:::warning 내장 SQL 오답 포인트
**"SQL문은 주언어 변수(host variable)의 참조를 포함할 수 없다"** — 이것은 **틀린** 설명입니다!  
내장 SQL은 오히려 주언어 변수를 **적극적으로 참조**하기 위해 사용합니다. 프로그램의 변수 값을 SQL 조건절에 넣을 수 있기 때문에 내장 SQL이 유용한 것입니다.
:::

## 🔍 3. SQL 핵심 명령어 완벽 정리

### DDL vs DML 분류

SQL 명령어는 크게 **DDL(Data Definition Language, 데이터 정의어)** 와 **DML(Data Manipulation Language, 데이터 조작어)** 로 나뉩니다.

| 구분 | 명령어 | 역할 |
|------|--------|------|
| **DDL** | `CREATE` | 테이블, 인덱스, 뷰 등의 객체를 **생성** |
| **DDL** | `ALTER` | 기존 객체의 구조를 **변경** |
| **DDL** | `DROP` | 객체를 **삭제** |
| **DML** | `SELECT` | 데이터를 **조회** |
| **DML** | `INSERT` | 데이터를 **삽입** |
| **DML** | `UPDATE` | 데이터를 **수정** |
| **DML** | `DELETE` | 데이터를 **삭제** |

:::tip DDL vs DML 쉽게 기억하는 법
- **DDL**: "구조(틀)를 다루는 명령어" — 테이블이라는 그릇을 만들고, 모양을 바꾸고, 없앱니다
- **DML**: "내용(데이터)을 다루는 명령어" — 그릇 안의 내용물을 넣고, 꺼내고, 바꾸고, 버립니다

`SELECT`는 데이터를 **조회**하는 명령어이므로 **DML**에 속합니다. DDL이 아닙니다!
:::

### NULL 값 처리

NULL은 "값이 없음"을 나타내는 특별한 상태입니다. 0이나 빈 문자열('')과는 다릅니다. 마치 "모름" 또는 "해당 없음"을 표시하는 것과 같습니다.

NULL 값을 검색할 때는 반드시 **`IS NULL`** 또는 **`IS NOT NULL`** 을 사용해야 합니다.

```sql
-- ❌ 잘못된 방법 - 이 방식은 절대 작동하지 않습니다
SELECT * FROM myTable WHERE column1 = NULL;
-- NULL은 어떤 값과도 '='로 비교할 수 없습니다
-- NULL = NULL 조차도 FALSE 입니다

-- ❌ 잘못된 방법 - EQUALS라는 SQL 키워드는 존재하지 않습니다
SELECT * FROM myTable WHERE column1 EQUALS NULL;

-- ❌ 잘못된 방법 - NOT NULL은 테이블 정의 시 제약조건입니다
SELECT * FROM myTable WHERE column1 NOT NULL;

-- ✅ 올바른 방법 - IS NULL 사용
SELECT * FROM myTable WHERE column1 IS NULL;
-- column1 컬럼의 값이 NULL인 모든 행을 가져옵니다

-- ✅ NULL이 아닌 값 찾기
SELECT * FROM myTable WHERE column1 IS NOT NULL;
-- column1 컬럼에 값이 있는(NULL이 아닌) 모든 행을 가져옵니다
```

> **NULL은 "값이 없음"이기 때문에 어떤 값과도 같다/다르다 비교가 불가능합니다.** 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용하세요.

### DISTINCT, ORDER BY, GROUP BY 차이

```sql
-- 🔹 DISTINCT: 중복된 값을 제거하고 유일한 값만 반환
SELECT DISTINCT department     -- 부서 목록에서 중복 제거
FROM employee;
-- 예: 영업부, 영업부, 개발부, 개발부 → 영업부, 개발부

-- 🔹 ALL: 중복 포함 모든 값 반환 (기본값)
SELECT ALL department          -- 중복 포함 모든 부서명 반환
FROM employee;

-- 🔹 ORDER BY: 결과를 특정 컬럼 기준으로 정렬
SELECT name, salary
FROM employee
ORDER BY salary DESC;          -- 급여 기준 내림차순 정렬

-- 🔹 GROUP BY: 특정 컬럼 기준으로 행을 묶어서 집계
SELECT department, AVG(salary) -- 부서별 평균 급여 계산
FROM employee
GROUP BY department;           -- 부서별로 그룹 묶기
```

각 명령어의 역할을 정리하면 다음과 같습니다.

| 명령어 | 역할 | 비유 |
|--------|------|------|
| `DISTINCT` | **중복 제거** | 같은 카드 여러 장 중 한 장만 남기기 |
| `ALL` | 중복 포함 전체 반환 | 모든 카드 다 보여주기 |
| `ORDER BY` | **정렬** | 카드를 숫자 순서대로 나열하기 |
| `GROUP BY` | **그룹별 집계** | 카드를 무늬(하트, 스페이드 등)별로 묶기 |

## 🌳 4. B-트리 인덱스와 파일 구조

### 인덱스(Index)란 무엇인가?

**인덱스(Index, 색인)** 는 책 뒷면의 찾아보기 페이지와 같습니다. 책에서 "트랜잭션"이라는 단어가 나오는 페이지를 찾을 때, 처음부터 끝까지 다 읽는 것보다 찾아보기에서 바로 해당 페이지를 찾는 것이 훨씬 빠릅니다. 데이터베이스에서 인덱스도 마찬가지로, 특정 데이터를 빠르게 찾을 수 있도록 도와주는 자료구조입니다.

### B-트리 (Balanced Tree)

**B-트리(B-Tree, Balanced Tree)** 는 데이터베이스 인덱스에서 가장 많이 사용되는 자료구조입니다. "균형 잡힌 트리"라는 의미로, 모든 단말 노드(잎사귀)가 같은 깊이에 있어 어떤 데이터를 찾아도 비슷한 시간이 걸립니다.

**차수(Order)가 3인 B-트리** 는 하나의 노드에 최대 2개의 키 값을 가질 수 있고, 최대 3개의 자식 노드를 가질 수 있습니다. (차수 = 최대 자식 노드 수)

차수 3인 B-트리에서 키 값 **25, 40, 120, 10, 55** 를 순서대로 삽입하는 과정을 살펴봅시다.

```
[단계 1] 25 삽입
  루트: [25]

[단계 2] 40 삽입
  루트: [25, 40]

[단계 3] 120 삽입
  → 노드에 키가 3개(25, 40, 120)가 되면 오버플로우!
  → 중간 값(40)을 부모로 올리고 분리(split) 발생
  
  루트: [40]
  왼쪽 자식: [25]   오른쪽 자식: [120]

[단계 4] 10 삽입
  → 10 < 40이므로 왼쪽 자식([25])으로 이동
  왼쪽 자식: [10, 25]

[단계 5] 55 삽입
  → 40 < 55 < 120이므로 오른쪽 자식([120])으로 이동
  → 오른쪽 자식: [55, 120]

최종 트리:
        [40]          ← 루트 = 40
       /    \
   [10,25]  [55,120]
```

따라서 **루트 값은 40** 입니다.

:::tip B-트리 삽입 핵심 원리
1. 값을 삽입할 적절한 위치를 찾습니다
2. 노드에 값을 넣었을 때 **최대 허용 키 수를 초과**하면 분리(split)가 발생합니다
3. 분리 시 **중간 값이 부모 노드로 올라갑니다**
4. 이 과정이 루트까지 올라갈 수 있습니다
:::

### 비트맵 인덱스 (Bitmap Index)

**비트맵 인덱스(Bitmap Index)** 는 컬럼의 각 값에 대해 0과 1로 이루어진 비트맵을 만들어 인덱스를 구성하는 방법입니다. 예를 들어 "성별" 컬럼처럼 값의 종류가 적은(남/여) 경우에 효과적입니다.

```
예시: 성별 컬럼의 비트맵 인덱스

행번호:  1  2  3  4  5  6
성별:    남 여 남 남 여 남

남성 비트맵: 1  0  1  1  0  1  ← 남성인 행에 1 표시
여성 비트맵: 0  1  0  0  1  0  ← 여성인 행에 1 표시
```

비트맵 인덱스의 특징을 정확히 알아야 합니다.

| 특징 | 설명 |
|------|------|
| **낮은 분포도(Low Cardinality)** 컬럼에 적합 | 값의 종류가 적을 때 효율적 (예: 성별, 등급, 상태코드) |
| **큰 테이블**에 적합 | 테이블 크기가 클수록 효과적 |
| **OLAP(온라인 분석 처리)** 환경에 적합 | 읽기 중심의 분석 쿼리에 강함 |
| **삽입/갱신이 많은 테이블에 부적합** | 값이 바뀔 때마다 비트맵 전체를 재구성해야 하므로 성능 저하 |

:::warning 비트맵 인덱스 오답 포인트
**"삽입이나 갱신이 많이 발생하는 테이블에 적용한다"** — 이것은 **틀린** 설명입니다!  
비트맵 인덱스는 데이터 변경이 잦으면 성능이 크게 떨어집니다. 주로 데이터 변경이 적고 읽기 위주인 **데이터 웨어하우스(DW)** 나 **OLAP** 환경에서 사용합니다.
:::

### 파일 구조 종류

데이터를 디스크에 저장하는 방식(파일 구조)은 여러 종류가 있으며, 사용 목적에 따라 적합한 구조가 다릅니다.

| 파일 구조 | 특징 | 적합한 용도 |
|-----------|------|------------|
| **순차 파일(Sequential File)** | 데이터를 순서대로 저장, 순서대로만 읽을 수 있음 | 전체 데이터를 순서대로 처리할 때 (예: 월급명세서 일괄 출력) |
| **인덱스 순차 파일(ISAM)** | 순차 접근 + 인덱스를 통한 직접 접근 모두 가능 | **특정 레코드 검색/수정**과 **전체 순차 처리**를 **모두** 해야 할 때 |
| **직접 파일(Direct File)** | 해시 함수로 주소 계산, 특정 레코드 직접 접근 | 특정 레코드를 매우 빠르게 검색/수정할 때 |
| **힙 파일(Heap File)** | 아무 순서 없이 저장, 전체 탐색 필요 | 단순 삽입, 정렬 불필요 시 |

**실전 문제 풀이**: 급여 관리 프로그램에서 아래 조건을 모두 만족해야 한다면?
- ① 특정 직원의 급여내역을 **수시로 조회**
- ② 특정 직원의 급여내역을 **수시로 수정**
- ③ 전 직원의 급여명세서를 **매월 전체 출력**

조건 ①②는 특정 레코드에 빠른 직접 접근이 필요하고, 조건 ③은 전체를 순서대로 처리해야 합니다. 직접 접근과 순차 접근을 **모두** 지원하는 **인덱스 순차 파일(ISAM)** 이 가장 적합합니다.

## 💿 5. 디스크 접근 시간과 RAID

### 디스크 접근 시간의 구성 요소

하드디스크 드라이브(HDD)에서 데이터를 읽으려면 세 단계의 시간이 필요합니다. 마치 LP 레코드판에서 원하는 노래를 찾는 것처럼 생각하면 됩니다.

```
디스크 전체 접근 시간 = 탐구 시간 + 회전 지연 시간 + 데이터 전송 시간
```

**① 탐구 시간 (Seek Time, 헤드 이동 시간)**

디스크의 읽기/쓰기 헤드를 원하는 **트랙(동심원)** 이 있는 **실린더(cylinder)** 위치로 이동하는 데 걸리는 시간입니다. 마치 LP판 위의 바늘을 원하는 트랙 위로 이동시키는 것과 같습니다.

> **같은 실린더 또는 같은 트랙에 연속적으로 데이터 블록을 배치**하면 헤드를 거의 이동시키지 않아도 되므로 **탐구 시간(Seek Time)** 을 크게 줄일 수 있습니다.

**② 회전 지연 시간 (Rotational Latency)**

헤드가 올바른 트랙 위에 위치했더라도, 디스크가 회전하면서 원하는 섹터(데이터 위치)가 헤드 아래로 올 때까지 기다리는 시간입니다. 원반이 돌아오기를 기다리는 것입니다.

**③ 데이터 전송 시간 (Data Transfer Time)**

실제로 데이터를 읽거나 쓰는 데 걸리는 시간입니다. 헤드 아래로 지나가는 데이터가 메모리로 전송되는 시간입니다.

:::info 핵심 정리
"같은 실린더/트랙에 연속 배치로 단축되는 것은?" → **탐구 시간(Seek Time)**입니다.  
헤드를 이동할 필요가 없거나 최소화되기 때문입니다.
:::

### SSD vs HDD

**SSD(Solid State Drive, 반도체 저장 장치)** 는 기존 하드디스크(HDD)와 달리 기계적인 움직임 없이 반도체 메모리(플래시 메모리)에 데이터를 저장합니다.

| 비교 항목 | HDD | SSD |
|-----------|-----|-----|
| **저장 매체** | 자기 디스크(회전) | 반도체(플래시 메모리) |
| **속도** | 느림 | **빠름** |
| **기계적 지연** | 있음 (탐구, 회전) | **없음** |
| **실패율** | 상대적으로 높음 | **낮음** |
| **발열/소음** | 있음 | **적음** |
| **크기/무게** | 큼/무거움 | **소형/경량화 가능** |

SSD의 특징을 한마디로 정리하면: "속도가 빠르고, 기계적 지연·실패율·발열·소음이 적으며, 소형화·경량화가 가능한 **반도체를 이용한 정보 저장 장치**"입니다.

### RAID (Redundant Array of Independent Disks)

**RAID(레이드)** 는 여러 개의 하드디스크를 하나처럼 묶어서 사용하는 기술입니다. 마치 여러 명의 택배 직원이 동시에 각각 다른 주소로 배달하면 전체 배달 속도가 빨라지는 것처럼, 여러 디스크가 동시에 데이터를 처리하면 속도가 향상됩니다.

RAID의 주요 레벨을 알아봅시다.

| RAID 레벨 | 방식 | 특징 |
|-----------|------|------|
| **RAID 0** | **스트라이핑(Striping)** | 여러 디스크에 데이터를 분산 저장 → 속도 최고, 하지만 하나라도 고장 나면 전체 데이터 손실 |
| **RAID 1** | **미러링(Mirroring)** | 동일한 데이터를 두 디스크에 복사 → 안전성 최고, 하지만 용량 절반만 사용 |
| **RAID 0+1** | 스트라이핑 + 미러링 결합 | 속도와 안전성을 모두 추구 |
| **RAID 6** | 이중 패리티(Parity) | 2개의 디스크가 동시에 고장 나도 복구 가능 |

**RAID 0 — 스트라이핑(Striping) 방식 상세 설명**

```
데이터: A B C D E F G H

디스크1: A  C  E  G   ← 짝수번째 데이터 조각
디스크2: B  D  F  H   ← 홀수번째 데이터 조각

→ 두 디스크가 동시에 읽고 쓰므로 속도가 2배 향상!
→ 하지만 디스크1이 고장나면 A, C, E, G를 잃어버림 → 전체 데이터 복구 불가
```

> **RAID 0의 핵심**: 데이터를 여러 디스크에 **분산 저장(Striping)** 하여 **빠른 입출력**을 달성하지만, **데이터 보호(fault tolerance)** 기능은 없습니다.

:::danger RAID 레벨 혼동 주의
- **RAID 0** → 오직 **속도(Striping)** 만 추구, 안전성 없음
- **RAID 1** → 오직 **안전성(Mirroring)** 만 추구, 속도/용량 희생
- "빠른 입출력을 위해 여러 디스크에 분산 저장하는 Striping 방식"은 반드시 **RAID 0** 입니다!
:::

## 🔗 6. 관계 대수 연산자와 릴레이션

### 조인(Join) 연산

**조인(Join)** 은 두 개 이상의 테이블(릴레이션)에서 관련 있는 데이터를 연결하여 하나의 결과로 만드는 연산입니다. 마치 두 개의 레고 블록을 조합해서 새로운 모양을 만드는 것과 같습니다.

조인의 종류 중 **자연 조인(Natural Join)** 은 두 릴레이션에서 **같은 이름의 속성(컬럼)을 기준으로 자동으로 조인**하고, 그 결과에서 **중복되는 속성을 하나만 남기고 제거**합니다.

반면 **동등 조인(Equi Join)** 은 같은 값을 가진 속성으로 조인하지만 중복 속성이 결과에 그대로 남습니다. 자연 조인은 동등 조인의 결과에서 중복 컬럼을 제거한 것이라고 볼 수 있습니다.

```
예시: 학생 테이블과 수강 테이블 조인

학생 테이블:
학번  이름
001   홍길동
002   이순신

수강 테이블:
학번  과목명
001   데이터베이스
001   알고리즘
002   데이터베이스

동등 조인 결과 (학번 중복 포함):
학생.학번  이름    수강.학번  과목명
001       홍길동  001       데이터베이스
001       홍길동  001       알고리즘
002       이순신  002       데이터베이스

자연 조인 결과 (중복 학번 제거):
학번  이름    과목명
001  홍길동  데이터베이스
001  홍길동  알고리즘
002  이순신  데이터베이스
```

### 합병 가능(Union Compatible) 연산자

두 릴레이션 간의 **합집합(∪), 교집합(∩), 차집합(-)** 연산을 수행하려면 두 릴레이션이 **합병 가능(Union Compatible)** 조건을 만족해야 합니다.

**합병 가능 조건**:
1. 두 릴레이션의 **속성(컬럼) 수(차수)가 같아야** 합니다
2. 대응하는 속성의 **도메인(값의 범위/타입)이 같아야** 합니다

**카티션 프로덕트(Cartesian Product, 교차 곱)** 는 두 릴레이션의 모든 가능한 조합을 만드는 연산입니다. 이 연산은 합병 가능 조건을 필요로 하지 않습니다. 두 릴레이션의 차수가 달라도 됩니다.

:::warning 합병 가능 연산자 오답 포인트
합병 가능한 연산자는 **합집합, 교집합, 차집합** 세 가지입니다.  
**카티션 프로덕트는 합병 가능 조건이 필요 없습니다!** 따라서 "합병 가능한 연산자"를 묻는 문제에서 카티션 프로덕트가 등장하면 그것이 **옳지 않은 것(오답)**입니다.
:::

### 디비전(Division, ÷) 연산

**디비전(Division, R ÷ S)** 은 관계 대수에서 가장 이해하기 어려운 연산 중 하나입니다. "R에서 S의 모든 튜플과 매칭되는 R의 튜플만 결과로 반환"하는 연산입니다.

디비전의 규칙을 정확히 이해합시다.

- **R의 차수**: (m+n), **S의 차수**: n → **결과의 차수**: m
- R은 S의 모든 속성(Y)을 **공통으로 포함**하고 있습니다
- 공통 속성에서 **도메인(값의 범위)이 같아야** 합니다 — **이름이 같아야 한다는 것이 아닙니다**

:::danger 디비전 연산 오답 포인트
**"대응되는 애트리뷰트는 모두 이름들이 같다는 것이다"** — 이것은 **틀린** 설명입니다!  
디비전에서 중요한 것은 대응되는 속성의 **도메인(값의 범위/타입)이 같아야** 한다는 것입니다. 이름이 같아야 하는 것이 아닙니다.
:::

### 참조 무결성 (Referential Integrity)

**참조 무결성(Referential Integrity)** 은 외래키(Foreign Key)와 기본키(Primary Key) 사이의 관계를 보호하는 제약조건입니다.

쉽게 설명하면: 릴레이션 R1의 외래키가 릴레이션 R2의 기본키를 참조할 때, R2에 존재하지 않는 기본키 값을 R1의 외래키가 가리킬 수 없다는 규칙입니다. 마치 존재하지 않는 주소로 택배를 보낼 수 없는 것과 같습니다.

무결성의 종류를 구분해서 알아야 합니다.

| 무결성 종류 | 설명 | 예시 |
|------------|------|------|
| **개체 무결성(Entity Integrity)** | 기본키는 NULL이나 중복 값을 가질 수 없음 | 학번은 반드시 있어야 하고 중복 불가 |
| **참조 무결성(Referential Integrity)** | 외래키는 참조하는 테이블의 기본키에 존재하는 값이어야 함 | 수강 테이블의 학번은 학생 테이블에 있는 학번이어야 함 |
| **도메인 무결성(Domain Integrity)** | 속성 값은 정의된 도메인 범위 안의 값이어야 함 | 나이는 0 이상 150 이하여야 함 |

## 🧩 7. 논리적 데이터 모델과 데이터베이스 구조

### 논리적 데이터 모델의 종류

**논리적 데이터 모델(Logical Data Model)** 은 데이터를 어떤 방식으로 표현하고 구조화할지에 대한 추상적인 설계 방법입니다. 실제 물리적 저장 방식과는 별개로, "데이터가 어떻게 보이는가"를 정의합니다.

**① 관계형 데이터 모델(Relational Data Model)**

데이터를 **테이블(표)의 집합**으로 표현합니다. 우리가 일반적으로 알고 있는 행(Row)과 열(Column)로 이루어진 스프레드시트 형태입니다. SQL을 사용하며, 현재 가장 널리 사용되는 모델입니다.

**② 네트워크 데이터 모델(Network Data Model)**

데이터를 **그래프(Graph) 구조**로 표현합니다. 레코드(노드)들이 여러 방향으로 연결될 수 있어 복잡한 관계를 표현할 수 있습니다. 하나의 자식이 여러 부모를 가질 수 있습니다.

**③ 계층적 데이터 모델(Hierarchical Data Model)**

데이터를 **트리(Tree) 구조**로 표현합니다. 부모-자식 관계(오너-멤버 관계)로 연결되며, 하나의 부모가 여러 자식을 가질 수 있지만 자식은 반드시 하나의 부모만 가집니다.

:::warning 계층적 vs 네트워크 데이터 모델 혼동 주의
**계층적 데이터 모델**: 트리 구조, **오너-멤버** 관계, 자식은 **하나의** 부모만 가짐  
**네트워크 데이터 모델**: 그래프 구조, 자식이 **여러 부모**를 가질 수 있음  

시험에서 "계층적 데이터 모델 — 오너-멤버 관계의 계층적 그래프 구조"라고 나오면, 엄밀히는 계층적 모델이 **트리 구조**인데 "계층적 그래프"라고 표현하므로 맥락을 잘 파악해야 합니다.
:::

**④ 객체지향 데이터 모델(Object-Oriented Data Model)**

데이터를 **객체(Object)** 와 **상속(Inheritance)** 구조로 표현합니다. 프로그래밍의 객체지향 개념을 데이터 모델에 적용한 것입니다.

### 데이터베이스의 논리적 구조 vs 물리적 구조

| 구분 | 논리적 구조 | 물리적 구조 |
|------|------------|------------|
| **관점** | 사용자가 생각하는 데이터의 논리적 표현 | 저장 장치 입장에서 본 데이터 구조 |
| **레코드 유형** | 논리적 레코드 | **저장 레코드(Stored Record)** |
| **다루는 방식** | 사용자와 응용 프로그램이 다룸 | DBMS와 운영체제가 다룸 |

:::danger 물리적 구조에 대한 오해
**"물리적 구조는 논리적 구조에서도 취급할 수 있다"** — 이것은 **틀린** 설명입니다!  
논리적 구조와 물리적 구조는 서로 **독립적(독립성)**입니다. 물리적 구조가 변경되어도 논리적 구조에 영향을 주지 않는 것이 **데이터 독립성**의 핵심입니다. 논리적 구조에서 물리적 구조의 세부사항을 취급하지 않습니다.
:::

## ⚠️ 8. 주의사항 및 시험 대비 핵심 팁

### 자주 혼동하는 개념 모음

**① 자연 조인 vs 동등 조인**

- **동등 조인**: 같은 값으로 조인, **중복 속성 남음**
- **자연 조인**: 동등 조인 결과에서 **중복 속성 제거**

**② DISTINCT vs GROUP BY**

- `DISTINCT`: 단순히 **중복 행 제거** 후 반환
- `GROUP BY`: 그룹별로 묶어서 **집계 함수(SUM, AVG, COUNT 등)** 와 함께 사용

```sql
-- DISTINCT: 중복 부서명 제거
SELECT DISTINCT department FROM employee;

-- GROUP BY: 부서별 직원 수 집계 (집계 함수와 함께 사용)
SELECT department, COUNT(*) as 직원수
FROM employee
GROUP BY department;
```

**③ NULL 처리의 3가지 원칙**

```sql
-- ✅ NULL 비교는 항상 IS NULL / IS NOT NULL 사용
WHERE column1 IS NULL        -- NULL 값 찾기
WHERE column1 IS NOT NULL    -- NULL이 아닌 값 찾기

-- ❌ 이런 방식은 절대 사용 금지
WHERE column1 = NULL         -- 항상 FALSE 반환
WHERE column1 != NULL        -- 항상 FALSE 반환
WHERE column1 <> NULL        -- 항상 FALSE 반환
```

**④ DBMS 3대 필수 기능**

```
정의(Definition) → CREATE, ALTER, DROP (DDL)
조작(Manipulation) → SELECT, INSERT, UPDATE, DELETE (DML)
제어(Control) → 무결성, 보안, 병행제어, 회복
```

**⑤ 데이터베이스 4대 특성 (마지막은 항상 "내용에 의한 참조")**

- 실시간 접근 ✅
- 계속적인 변화 ✅
- 동시 공용 ✅
- **내용에 의한 참조** ✅ (주소에 의한 참조 ❌)

### 개체 타입 vs 개체 인스턴스

**개체 타입(Entity Type)** 은 개체의 **구조(틀)**를 정의하는 것입니다. 예를 들어 "직원"이라는 개체 타입은 학번, 이름, 부서 같은 속성들의 집합으로 정의됩니다.

반면 **개체 어커런스(Entity Occurrence, 개체 인스턴스)** 는 개체 타입의 **실제 값**입니다. "홍길동, 영업부, 010-1234-5678"처럼 실제로 존재하는 한 직원의 데이터가 하나의 어커런스입니다.

:::warning 개체 타입에 대한 오답 포인트
**"개체 타입 — 개체에 속해 있는 실제 값들로 구성된 개체 어커런스의 집합"** — 이것은 **틀린** 설명입니다!  
개체 타입은 실제 값이 아니라 **속성들의 집합(구조/틀)**입니다. 실제 값들의 집합은 **개체 어커런스(개체 인스턴스)** 또는 **개체 집합(Entity Set)**이라고 합니다.
:::

## 📝 핵심 정리

이 문서에서 학습한 핵심 개념들을 정리합니다.

- **DBMS 특징**: 자료의 중복 **배제**와 자료의 **독립성 유지**가 핵심 (중복 유지 ❌, 독립성 배제 ❌)
- **데이터베이스 참조**: 주소(위치)가 아닌 **내용(값)**에 의한 참조
- **DBMS 필수 기능 3가지**: 정의 기능, 조작 기능, 제어 기능 (사전 기능은 필수 기능 아님)
- **데이터 제어 처리기**: 무결성 검사, 접근권한 검사, 트랜잭션 병행제어, 회복 작업 담당
- **DML 프리컴파일러**: 응용 프로그램 속 SQL 명령문을 **추출**하는 구성요소
- **내장 SQL**: `EXEC SQL` 앞에 붙임, 호스트 변수 앞에 `:` 사용, 호스트 변수 **참조 가능**
- **NULL 검색**: 반드시 `IS NULL` 사용, `= NULL`은 작동하지 않음
- **DISTINCT**: 중복 제거 SQL 명령어 (ALL은 중복 포함)
- **SELECT**: DDL이 아닌 **DML** 명령어
- **자연 조인**: 동등 조인 결과에서 **중복 속성 제거**, 이것이 핵심 특징
- **합병 가능 연산자**: 합집합, 교집합, 차집합 (카티션 프로덕트는 합병 가능 조건 불필요)
- **디비전 연산**: 도메인이 같아야 하는 것이지, 이름이 같아야 하는 것이 아님
- **참조 무결성**: 외래키 변경 시 참조하는 기본키도 함께 관리
- **인덱스 순차 파일(ISAM)**: 직접 접근(검색/수정)과 순차 접근(전체 출력) 모두 지원
- **비트맵 인덱스**: 낮은 분포도 컬럼, 큰 테이블, OLAP에 적합 — 삽입/갱신 많은 환경에는 **부적합**
- **B-트리 삽입**: 오버플로우 시 중간 값이 부모로 올라가며 노드 분리 발생
- **탐구 시간(Seek Time)**: 같은 실린더/트랙에 연속 배치 시 단축 가능
- **SSD**: 반도체 저장 장치, HDD 대비 속도 빠름, 기계적 지연·실패율·발열·소음 적음
- **RAID 0**: 스트라이핑(Striping) 방식, 여러 디스크 분산 저장으로 빠른 입출력
- **개체 타입**: 속성들의 구조/틀 (실제 값의 집합이 아님)
- **물리적 구조**: 저장 장치 관점의 구조, 레코드는 **저장 레코드(Stored Record)**
- **DBMS 중복 제거**: 완벽한 제거가 아닌 **최소화**
