# 🗄️ 논리적 데이터 모델과 파일 구조

데이터베이스를 처음 공부하면 "릴레이션", "조인", "인덱스", "RAID" 같은 단어들이 한꺼번에 쏟아져서 당황스러울 수 있습니다. 이 문서에서는 시험에 자주 등장하는 핵심 개념들을 **초보자도 이해할 수 있도록** 하나씩 차근차근 풀어드립니다. 실생활 비유와 코드 예시, 정오 판별 훈련까지 함께 준비했으니 끝까지 읽어보세요! 😊

## 1️⃣ 관계형 연산자 — 조인(Join)과 디비전(Division)

### 🔗 조인(Join)이란 무엇인가?

**조인**은 두 개 이상의 테이블(릴레이션)을 서로 연결해서 하나의 결과로 합치는 연산입니다. 마치 학교 명단표(학생 테이블)와 성적표(성적 테이블)를 학생 번호로 연결해서 "이름과 성적이 함께 보이는 표"를 만드는 것과 같습니다.

조인의 종류 중 **동일 조인(Equi Join)**은 두 테이블에서 같은 값을 가진 열(컬럼)을 기준으로 연결합니다. 그런데 여기에 문제가 생깁니다. 두 테이블에 모두 `학생번호`라는 열이 있으면, 결과 테이블에도 `학생번호` 열이 **두 개** 중복되어 나타납니다. 이처럼 **중복된 속성(컬럼) 하나를 제거**하고 깔끔하게 정리한 조인 방식을 **자연 조인(Natural Join)** 이라고 합니다.

> 💡 **핵심 정리**: 동일 조인(Equi Join)에서 중복되는 속성을 하나만 남기고 제거한 조인 = **자연 조인(Natural Join)**

```sql
-- ✅ 자연 조인 예시: 중복 속성 자동 제거
SELECT *
FROM 학생 NATURAL JOIN 성적;
-- 두 테이블 모두에 있는 '학번' 컬럼이 하나만 표시됨

-- ❌ 동일 조인(Equi Join) 예시: 중복 속성 남음
SELECT *
FROM 학생, 성적
WHERE 학생.학번 = 성적.학번;
-- '학번' 컬럼이 두 번 등장하여 중복 발생
```

:::info 자연 조인 vs 동일 조인
| 구분 | 동일 조인 (Equi Join) | 자연 조인 (Natural Join) |
|---|---|---|
| 조인 조건 | `=` 연산자 사용 | 이름이 같은 컬럼 자동 연결 |
| 중복 속성 | 남아있음 | 자동으로 하나 제거 |
| 결과 컬럼 수 | 두 테이블 합산 | 중복 제거 후 합산 |
:::

### ➗ 디비전(Division) 연산이란?

**디비전(Division, R ÷ S)** 은 두 테이블 사이에서 "S에 있는 **모든** 항목에 대응하는 R의 튜플(행)"을 찾아내는 연산입니다. 예를 들어 "모든 과목을 수강한 학생"을 찾는 것처럼, S의 모든 조건을 만족하는 R의 부분을 결과로 돌려줍니다.

디비전에 관해 자주 출제되는 **틀린 보기**가 있습니다:

> ❌ "대응되는 애트리뷰트는 모두 **이름들이 같다**는 것이다."

이것은 틀린 설명입니다. 디비전 연산에서 공통 애트리뷰트(속성)는 **이름이 같은 것이 아니라 도메인(값의 범위)이 같아야** 합니다. 즉, 이름이 달라도 같은 종류의 값을 담는 속성이라면 대응될 수 있습니다.

:::warning 디비전 연산 주의사항
디비전 결과 릴레이션의 **차수(컬럼 수)**는 `m`입니다.
- R의 차수: `m + n`
- S의 차수: `n`
- 결과의 차수: `m`

마치 분수 나눗셈처럼 분모에 해당하는 S의 속성 `n`개를 빼고 나머지 `m`개만 결과로 나옵니다.
:::

### 🔀 합병 가능한 연산자

**합병 가능(Union-Compatible)** 조건은 두 릴레이션의 **차수(컬럼 수)가 같고, 대응되는 도메인이 같아야** 한다는 것입니다.

합집합(∪), 교집합(∩), 차집합(−)은 모두 합병 가능 조건을 요구합니다.

**카티션 프로덕트(Cartesian Product, ×)** 는 두 테이블의 모든 행을 서로 곱하는 연산인데, 이 연산은 **합병 가능 조건이 필요하지 않습니다.** 두 테이블의 컬럼 수나 도메인이 달라도 쌍(Pair)을 만들 수 있기 때문입니다.

| 연산 | 합병 가능 조건 필요? |
|---|---|
| 합집합 (∪) | ✅ 필요 |
| 교집합 (∩) | ✅ 필요 |
| 차집합 (−) | ✅ 필요 |
| 카티션 프로덕트 (×) | ❌ 불필요 |

## 2️⃣ B-트리(B-Tree) 구조와 삽입 과정 🌳

### B-트리란 무엇인가?

**B-트리(Balanced Tree)**는 데이터베이스와 파일 시스템에서 대용량 데이터를 **균형 있게** 저장하고 빠르게 검색하기 위해 사용하는 트리 자료구조입니다. 일반 이진 트리(한 노드에 두 개의 자식)와 달리, B-트리는 **하나의 노드에 여러 개의 키(Key)와 자식 포인터**를 가질 수 있습니다.

마치 도서관 서가를 생각해보세요. 일반 이진 트리는 책을 두 그룹씩만 나눠서 찾아가는 방식이고, B-트리는 서가를 여러 구역으로 나눠서 한 번에 더 많은 후보를 좁혀가는 방식입니다. 덕분에 훨씬 적은 단계로 원하는 데이터에 도달할 수 있습니다.

### 차수(Order)와 키 삽입 규칙

**차수(Order) 3**인 B-트리는 한 노드에 **최대 2개의 키**와 **최대 3개의 자식**을 가질 수 있습니다. 키가 2개를 초과하면 **분할(Split)**이 일어납니다.

:::info B-트리 차수 규칙
- 차수가 `t`인 B-트리에서 한 노드가 가질 수 있는 **최대 키 수** = `t - 1`
- 차수 3 → 최대 키 수 = **2개**
- 키가 3개가 되면 **중간값을 부모로 올리고 분할**합니다
:::

### 키 삽입 시뮬레이션: 25, 40, 120, 10, 55

차수가 3인 B-트리에 25, 40, 120, 10, 55 순서로 삽입하는 과정을 단계별로 따라가봅시다.

**Step 1: 25 삽입**
```
[25]
```
노드 하나에 키 하나. 아무 문제 없습니다.

**Step 2: 40 삽입**
```
[25, 40]
```
키가 2개. 아직 최대(2개) 이하이므로 분할 없음.

**Step 3: 120 삽입**
```
[25, 40, 120]  ← 키가 3개! 분할 발생
```
중간값은 **40**입니다. 40을 부모로 올리고 25와 120을 자식으로 분리합니다:
```
      [40]
     /    \
  [25]   [120]
```

**Step 4: 10 삽입**
10은 40보다 작으므로 왼쪽 자식 [25]로 내려갑니다:
```
      [40]
     /    \
[10, 25]  [120]
```

**Step 5: 55 삽입**
55는 40보다 크므로 오른쪽 자식 [120]으로 내려갑니다:
```
      [40]
     /    \
[10, 25]  [55, 120]
```

최종 B-트리의 **루트 값은 40** 입니다.

> 🎯 **정답**: 차수 3인 B-트리에 25, 40, 120, 10, 55를 순서대로 삽입했을 때 루트 값은 **40**

## 3️⃣ 저장 장치: SSD, RAID, 디스크 접근 시간 💾

### SSD(Solid State Drive)란?

**SSD(Solid State Drive, 반도체 저장 장치)**는 기존 하드디스크(HDD)와 달리 **회전하는 원판(자기 디스크) 없이 반도체 메모리 칩**을 이용해 데이터를 저장하는 장치입니다.

HDD가 마치 LP 레코드판에 바늘을 올려서 음악을 재생하는 방식이라면, SSD는 USB 메모리처럼 **전기 신호만으로** 데이터를 읽고 씁니다. 그 덕분에 다음과 같은 장점이 있습니다:

- ⚡ **속도가 빠르다**: 기계적 움직임이 없어서 읽기/쓰기 속도가 월등히 빠릅니다
- 🔇 **소음이 없다**: 회전하는 부품이 없어 조용합니다
- 🌡️ **발열이 적다**: 기계 마찰이 없으므로 열 발생이 적습니다
- 💪 **내구성이 높다**: 물리적 충격에 강합니다
- 📦 **소형화·경량화 가능**: 반도체 칩 기반이라 얇고 가볍게 만들 수 있습니다

:::tip SSD vs HDD 비교
| 특성 | HDD | SSD |
|---|---|---|
| 저장 방식 | 자기 디스크 회전 | 반도체 메모리 |
| 속도 | 느림 | 빠름 |
| 소음 | 있음 | 없음 |
| 발열 | 많음 | 적음 |
| 기계적 고장 | 있음 | 없음 |
| 소형화 | 어려움 | 가능 |
:::

### 디스크 접근 시간의 구성 요소

하드디스크에서 데이터를 읽으려면 세 가지 시간이 필요합니다:

1. **탐구 시간(Seek Time)**: 읽기 헤드(바늘)를 원하는 **트랙(원판의 동심원 중 하나)** 위로 이동시키는 시간
2. **회전 지연 시간(Rotational Latency)**: 원하는 **섹터(데이터가 있는 위치)**가 헤드 아래로 올 때까지 기다리는 시간
3. **데이터 전송 시간(Transfer Time)**: 실제로 데이터를 읽거나 쓰는 시간

이 중에서 **같은 실린더(cylinder, 여러 원판의 같은 위치 트랙의 집합) 또는 같은 트랙에 연속적으로 데이터 블록을 배치하면 단축할 수 있는 것**은 바로 **탐구 시간(Seek Time)**입니다.

헤드를 이미 원하는 트랙 위치에 갖다 놓은 상태라면 헤드를 다시 움직일 필요가 없기 때문입니다. 마치 책갈피를 꽂아두면 매번 처음부터 페이지를 넘길 필요가 없는 것처럼요.

### RAID — 여러 디스크를 하나처럼 🗂️

**RAID(Redundant Array of Independent Disks, 독립 디스크 배열)**는 여러 개의 하드디스크를 묶어서 **하나의 고성능·고신뢰성 저장 장치**처럼 사용하는 기술입니다.

RAID의 주요 레벨을 비교해봅시다:

| RAID 레벨 | 방식 | 특징 |
|---|---|---|
| **RAID 0** | 스트라이핑(Striping) | 데이터를 여러 디스크에 분산 저장 → 속도 최대화, 복구 불가 |
| **RAID 1** | 미러링(Mirroring) | 데이터를 두 디스크에 동일하게 복사 → 신뢰성 최대화 |
| **RAID 0+1** | 스트라이핑 + 미러링 | 속도와 신뢰성 모두 확보 |
| **RAID 6** | 이중 패리티(Parity) | 두 개의 디스크가 동시에 고장나도 복구 가능 |

> 🎯 **스트라이핑(Striping) 방식** — 데이터를 여러 디스크에 분산 저장하여 **입출력 속도를 극대화**하는 방식은 **RAID 0**입니다.

RAID 0은 마치 여러 명의 직원이 각각 다른 서랍을 담당해서 동시에 파일을 꺼내오는 것과 같습니다. 한 명이 모든 서랍을 혼자 뒤지는 것보다 훨씬 빠르겠죠!

:::danger RAID 0의 치명적 단점
RAID 0은 속도는 빠르지만 **데이터 복구 기능이 없습니다.** 디스크 중 하나라도 고장나면 **모든 데이터를 잃습니다.** 중요한 데이터를 저장할 때는 절대 RAID 0만 사용하지 마세요.
:::

## 4️⃣ DBMS의 구성 요소와 SQL 핵심 문법 🛠️

### DBMS의 주요 구성 요소

**DBMS(Database Management System, 데이터베이스 관리 시스템)**는 여러 구성 요소로 이루어져 있습니다. 시험에서는 특히 두 가지 구성 요소가 자주 출제됩니다.

#### ① 데이터 제어 담당 구성 요소

다음 기능들을 수행하는 DBMS 구성 요소는 **데이터 제어어(DCL) 처리 모듈** 또는 **런타임 데이터베이스 처리기**입니다:

- **무결성 제약조건 만족 여부 검사**: 데이터가 올바른 형식을 갖췄는지 확인 (예: 나이에 음수 입력 방지)
- **사용자 접근 권한 검사**: 이 사람이 이 데이터를 읽거나 수정할 자격이 있는지 확인
- **트랜잭션(작업 묶음)의 병행 제어**: 여러 사람이 동시에 같은 데이터를 수정할 때 충돌 방지
- **회복 작업 수행**: 시스템 오류 발생 시 데이터를 이전 상태로 되돌리는 작업

#### ② DML 명령문 추출 구성 요소

호스트 프로그래밍 언어(C, Java 등)로 작성된 응용 프로그램 속에 섞여 있는 SQL 명령문(DML)을 뽑아내는 DBMS 구성 요소를 **DML 예비 컴파일러(DML Precompiler)**라고 합니다.

예를 들어 C 프로그램 중간에 `EXEC SQL SELECT ...` 같은 SQL 문장이 들어있을 때, 이것을 일반 C 코드와 분리해서 처리하는 역할을 합니다.

### 내장 SQL(Embedded SQL)의 이해

**내장 SQL(Embedded SQL)**은 C, Java, COBOL 같은 일반 프로그래밍 언어 코드 안에 **SQL 명령문을 직접 삽입**하는 방식입니다.

내장 SQL의 규칙:
- SQL 문장 앞에 **`EXEC SQL`** 키워드를 반드시 붙여야 합니다
- **호스트 변수(host variable)**: 일반 프로그래밍 언어에서 선언한 변수를 SQL 문장 안에서 참조할 수 있습니다
- 호스트 변수와 데이터베이스 필드는 **같은 이름을 가질 수 있습니다** (충돌 방지를 위해 `:변수명` 형식으로 구분)
- 호스트 변수의 데이터 타입은 대응하는 데이터베이스 필드의 SQL 데이터 타입과 **일치해야 합니다**

:::warning 내장 SQL 틀린 설명 식별하기
다음 보기 중 **틀린 것**은?

> "SQL문은 주언어 변수(host variable)의 참조를 포함할 수 없다."

→ 이것이 **틀린 설명**입니다! 내장 SQL은 호스트 변수를 **참조할 수 있습니다.** 오히려 호스트 변수 참조야말로 내장 SQL의 핵심 기능 중 하나입니다.
:::

```c
// ✅ 올바른 내장 SQL 예시 (C 언어)
int emp_id = 1001;           // 호스트 변수 선언
char emp_name[50];           // 결과를 받을 호스트 변수

// EXEC SQL로 내장 SQL 시작, :변수명으로 호스트 변수 참조
EXEC SQL SELECT 이름 INTO :emp_name
         FROM 직원
         WHERE 사원번호 = :emp_id;

// emp_name에 조회 결과가 담겨있음
printf("직원 이름: %s\n", emp_name);
```

### NULL 값 처리와 DISTINCT

#### NULL 검사 방법

**NULL**은 "값이 없음"을 나타내는 특수한 상태입니다. NULL은 0도 아니고 빈 문자열도 아닙니다. 그냥 "모름" 또는 "해당 없음"입니다.

NULL을 검사할 때는 **반드시 `IS NULL` 또는 `IS NOT NULL`** 을 사용해야 합니다.

```sql
-- ✅ 올바른 NULL 검사
SELECT * FROM myTable WHERE column1 IS NULL;

-- ❌ 잘못된 NULL 검사 (항상 결과 없음)
SELECT * FROM myTable WHERE column1 = NULL;
-- NULL은 어떤 값과도 '=' 비교가 되지 않음. NULL = NULL 도 FALSE!

-- ❌ 존재하지 않는 문법
SELECT * FROM myTable WHERE column1 EQUALS NULL;

-- ❌ 문법 오류 (NOT NULL은 WHERE절에서 이렇게 쓰지 않음)
SELECT * FROM myTable WHERE column1 NOT NULL;
```

:::danger NULL 비교의 함정
`NULL = NULL`의 결과는 `TRUE`가 아닌 `UNKNOWN`입니다! NULL은 "알 수 없는 값"이기 때문에 두 개의 "알 수 없는 값"이 같은지도 알 수 없습니다. 반드시 `IS NULL`을 사용하세요.
:::

#### DISTINCT로 중복 제거

SQL에서 **중복된 행을 제거**하려면 `DISTINCT` 키워드를 사용합니다.

```sql
-- ✅ DISTINCT: 중복 행 제거
SELECT DISTINCT 부서명 FROM 직원;
-- 같은 부서명이 여러 번 나오더라도 한 번씩만 출력

-- ALL: 중복 포함 (기본값, 생략 가능)
SELECT ALL 부서명 FROM 직원;
-- 같은 부서명이 여러 번 출력됨

-- ORDER BY: 정렬 (중복 제거 아님)
SELECT 이름 FROM 직원 ORDER BY 이름;

-- GROUP BY: 그룹화 (집계 함수와 함께 사용)
SELECT 부서명, COUNT(*) FROM 직원 GROUP BY 부서명;
```

### DDL vs DML vs DCL 명령어 분류

**DDL(Data Definition Language, 데이터 정의어)**은 테이블 구조 자체를 만들고 수정하고 삭제하는 명령어입니다.

| 분류 | 명령어 | 설명 |
|---|---|---|
| DDL | `CREATE` | 테이블, 인덱스 등 **생성** |
| DDL | `ALTER` | 테이블 구조 **수정** |
| DDL | `DROP` | 테이블, 인덱스 등 **삭제** |
| DML | `SELECT` | 데이터 **조회** |
| DML | `INSERT` | 데이터 **삽입** |
| DML | `UPDATE` | 데이터 **수정** |
| DML | `DELETE` | 데이터 **삭제** |
| DCL | `GRANT` | 권한 **부여** |
| DCL | `REVOKE` | 권한 **회수** |

> 🎯 **`SELECT`는 DDL이 아닌 DML**입니다! 테이블 구조를 바꾸는 것이 아니라 데이터를 읽어오는 명령이기 때문입니다.

## 5️⃣ 무결성 제약조건과 논리적 데이터 모델 🔒

### 무결성(Integrity) 제약조건의 종류

**무결성(Integrity)**이란 데이터베이스에 저장된 데이터가 **정확하고 일관되게 유지되도록** 보장하는 규칙입니다. 마치 은행 계좌에서 잔액이 음수가 되어선 안 된다는 규칙처럼, 데이터의 신뢰성을 지키는 울타리입니다.

#### 참조 무결성(Referential Integrity)

문제에서 묘사한 상황을 보겠습니다:

> "릴레이션 R1에 속한 애트리뷰트의 조합인 **외래키**를 변경하려면 이를 참조하고 있는 R2의 릴레이션의 **기본키**도 변경해야 한다."

이것은 **참조 무결성(Referential Integrity)**의 개념입니다. 외래키(Foreign Key, 다른 테이블의 기본키를 참조하는 키)는 참조 대상인 기본키가 존재할 때만 유효합니다. 만약 참조 대상 기본키가 바뀌거나 삭제되면 외래키도 함께 관리해야 합니다.

비유하자면, 학생 테이블에 "소속 학과 코드"가 외래키로 있고, 학과 테이블의 학과 코드가 기본키일 때 — 학과 코드를 바꾸면 학생 테이블의 소속 학과 코드도 함께 바꿔야 한다는 규칙입니다.

| 무결성 종류 | 설명 | 예시 |
|---|---|---|
| **개체 무결성** | 기본키는 NULL이 될 수 없고 중복될 수 없음 | 학번은 반드시 있어야 하고 유일해야 함 |
| **참조 무결성** | 외래키는 참조하는 기본키가 반드시 존재해야 함 | 없는 학과 코드를 소속 학과로 지정 불가 |
| **도메인 무결성** | 속성 값은 정해진 도메인(범위) 내여야 함 | 나이 컬럼에 음수나 문자 불가 |
| **널 무결성** | 특정 속성에 NULL이 허용되지 않아야 함 | 이름 컬럼은 반드시 값이 있어야 함 |

### 논리적 데이터 모델의 종류

**논리적 데이터 모델(Logical Data Model)**은 데이터를 사람이 이해하기 쉬운 구조로 표현하는 방법입니다. 물리적으로 어떻게 저장되는지는 고려하지 않고, **어떤 구조로 데이터를 표현할 것인가**에 집중합니다.

주요 논리적 데이터 모델 네 가지를 살펴봅시다:

#### 1. 관계형 데이터 모델(Relational Data Model)
데이터베이스를 **테이블(표)의 집합**으로 표현합니다. 현재 가장 널리 사용되는 모델입니다. MySQL, Oracle, PostgreSQL 등이 이 모델을 사용합니다.

#### 2. 네트워크 데이터 모델(Network Data Model)
데이터베이스를 **그래프(Graph) 구조**로 표현합니다. 노드(개체)와 엣지(관계)로 이루어져 있으며, 한 자식이 여러 부모를 가질 수 있습니다.

#### 3. 계층적 데이터 모델(Hierarchical Data Model)
데이터베이스를 **트리(Tree) 구조**로 표현합니다. 부모-자식 관계(오너-멤버 관계)로 이루어진 계층적 구조입니다. 한 자식은 오직 하나의 부모만 가질 수 있습니다.

:::warning 계층적 데이터 모델 설명 주의
"계층적 데이터 모델 — 데이터베이스를 오너–멤버 관계의 **계층적 그래프** 구조로 표현한다"는 설명은 **틀린 설명**입니다!

계층적 데이터 모델은 **그래프가 아닌 트리(Tree) 구조**입니다. 오너-멤버 관계로 표현하는 것은 맞지만, 이는 "그래프"가 아니라 "트리" 구조입니다. **그래프 구조**는 네트워크 데이터 모델의 설명입니다.
:::

#### 4. 객체지향 데이터 모델(Object-Oriented Data Model)
데이터베이스를 **객체(Object)와 상속(Inheritance) 구조**로 표현합니다. 프로그래밍의 객체지향 개념을 데이터베이스에 적용한 모델입니다.

## 6️⃣ 파일 구조와 인덱스 📂

### 파일 구조의 종류와 특징

파일 구조는 데이터를 디스크에 **어떤 방식으로 저장하고 접근**할지를 결정하는 방법입니다. 마치 도서관에서 책을 번호순으로 꽂을지, 장르별로 꽂을지, 아니면 그냥 랜덤으로 꽂을지를 정하는 것과 같습니다.

| 파일 구조 | 특징 | 장점 | 단점 |
|---|---|---|---|
| **순차 파일** | 데이터를 순서대로 저장 | 순서대로 읽기(배치 처리)에 최적 | 특정 레코드 검색·수정 느림 |
| **인덱스 순차 파일** | 순차 저장 + 인덱스 테이블 | 순차 접근 + 임의 접근 모두 가능 | 인덱스 공간 추가 필요 |
| **직접 파일** | 해시 함수로 직접 주소 계산 | 임의 접근 매우 빠름 | 순차 접근 어려움 |
| **힙 파일** | 빈 공간에 순서 없이 삽입 | 삽입 빠름 | 검색 매우 느림 |

### 급여관리 프로그램에 적합한 파일 구조는?

문제의 조건을 분석해봅시다:

1. ✅ **특정 직원의 급여내역을 수시로 조회** → **임의 접근(Random Access)** 이 필요합니다
2. ✅ **특정 직원의 급여내역을 수시로 수정** → **임의 접근(Random Access)** 이 필요합니다
3. ✅ **전 직원의 급여명세서를 매월 출력** → **순차 접근(Sequential Access)** 이 필요합니다

세 가지 조건을 모두 만족하려면 **임의 접근과 순차 접근을 모두 지원**하는 파일 구조가 필요합니다.

> 🎯 **정답: 인덱스 순차 파일(ISAM, Indexed Sequential Access Method)**

인덱스 순차 파일은 인덱스를 이용해 특정 레코드를 빠르게 찾을 수 있고(임의 접근), 동시에 전체 레코드를 순서대로 읽는 것도 가능합니다(순차 접근). 급여관리처럼 두 가지 접근 방식이 모두 필요한 업무에 딱 맞는 구조입니다.

### 비트맵 인덱스(Bitmap Index)

**비트맵 인덱스**는 각 컬럼의 값에 대해 **0과 1의 비트맵(Bitmap)**으로 인덱스를 저장하는 방식입니다. 예를 들어 성별(남/여) 같은 컬럼에서 "여자"에 해당하는 행은 1, 나머지는 0으로 표시합니다.

비트맵 인덱스가 적합한 환경:
- 테이블이 **매우 크고**, 컬럼의 **분포도(카디널리티)가 낮을 때** (값의 종류가 적을 때, 예: 성별, 지역 코드)
- **읽기 위주의 OLAP(Online Analytical Processing, 분석 처리) 환경**

:::warning 비트맵 인덱스 틀린 설명 식별하기
> "삽입이나 갱신이 많이 발생하는 테이블에 적용한다."

이것은 **틀린 설명**입니다! 비트맵 인덱스는 데이터 변경(삽입, 수정, 삭제)이 발생할 때마다 비트맵 전체를 갱신해야 하므로 **오버헤드(부담)가 매우 큽니다.** 따라서 **변경이 적고 조회가 많은 OLAP 환경**에서 사용하는 것이 적합합니다.
:::

## 7️⃣ 데이터베이스 시스템의 특성과 개념 📚

### 데이터베이스 vs 파일 시스템

**파일 시스템(File System)**은 각 응용 프로그램이 **자기만의 파일**을 따로 관리합니다. 인사팀 프로그램은 인사 파일을, 급여팀 프로그램은 급여 파일을 각자 보관합니다. 이러면 같은 직원 정보가 여러 파일에 중복 저장되어 관리가 어려워집니다.

반면 **데이터베이스 시스템(DBMS)**은:

- ✅ **자료의 중복을 최소화(배제)**: 같은 데이터를 한 곳에만 저장
- ✅ **데이터 독립성 유지**: 저장 방식이 바뀌어도 응용 프로그램은 수정할 필요 없음

> 🎯 **데이터베이스 시스템의 특징**: **"자료의 중복 배제 + 자료의 독립성 유지"**

:::note 데이터 독립성(Data Independence)이란?
데이터가 저장되는 방식(물리적 구조)을 바꾸더라도 응용 프로그램(논리적 구조)은 그대로 작동하는 성질입니다. 마치 창고 내부 진열 방식을 바꿔도 고객이 주문하는 방식은 변하지 않는 것과 같습니다.
:::

### 데이터베이스의 4대 특성

데이터베이스는 다음 네 가지 일반적 특성을 가집니다:

1. **실시간 접근(Real-time Accessibility)**: 질의(Query, 데이터 요청)에 대해 실시간으로 처리하고 응답합니다
2. **계속적인 변화(Continuous Evolution)**: 삽입, 삭제, 수정을 통해 항상 최신 상태를 유지합니다
3. **동시 공용(Concurrent Sharing)**: 여러 사용자가 동시에 같은 데이터베이스를 사용할 수 있습니다
4. **내용에 의한 참조(Content Reference)**: 데이터의 **위치나 주소가 아닌, 데이터 값(내용)**으로 데이터를 참조합니다

:::warning 데이터베이스 특성 틀린 설명 찾기
다음 중 **틀린 것**은?

> "데이터 참조 시 데이터 값에 의해서는 참조될 수 없으므로 위치나 주소에 의하여 데이터를 찾는다."

이것이 **틀린 설명**입니다! 데이터베이스는 정반대로, **위치나 주소가 아닌 데이터의 내용(값)**으로 참조합니다. `WHERE 이름 = '홍길동'`처럼 값으로 검색하는 것이 데이터베이스의 특징입니다.
:::

### DBMS의 필수 3대 기능

**DBMS**가 반드시 제공해야 하는 기능은 세 가지입니다:

| 기능 | 설명 | 해당 언어 |
|---|---|---|
| **정의 기능(Definition)** | 데이터 구조(스키마)를 정의하는 기능 | DDL |
| **조작 기능(Manipulation)** | 데이터를 검색·삽입·수정·삭제하는 기능 | DML |
| **제어 기능(Control)** | 무결성, 보안, 동시성 제어, 회복 기능 | DCL |

> 🎯 **"사전 기능"은 DBMS의 필수 기능이 아닙니다.** 정의, 조작, 제어 — 이 세 가지가 DBMS의 3대 필수 기능입니다.

### 데이터베이스의 논리적 구조 vs 물리적 구조

| 구분 | 설명 | 예시 |
|---|---|---|
| **논리적 구조** | 사용자 입장에서 본 데이터의 표현 방식 | 테이블, 행, 열 |
| **물리적 구조** | 저장 장치 입장에서 본 실제 저장 방식 | 블록, 섹터, 저장 레코드 |

물리적 구조에서 다루는 레코드는 **저장 레코드(Stored Record)**라고 합니다.

:::warning 논리적 구조 vs 물리적 구조 혼동 주의
> "물리적 구조는 논리적 구조에서도 취급할 수 있다."

이것은 **틀린 설명**입니다! 논리적 구조와 물리적 구조는 **서로 독립적**으로 분리되어 있습니다. 논리적 구조에서는 물리적 저장 방식을 직접 다루지 않습니다. 이것이 바로 **데이터 독립성**의 핵심입니다.
:::

### 데이터베이스 기본 개념 용어 정리

| 용어 | 설명 |
|---|---|
| **속성(Attribute)** | 개체(Entity)를 설명하는 하나의 특성. 예: 학생의 이름, 나이 |
| **개체 타입(Entity Type)** | 같은 속성을 가진 개체들의 **틀(형식)** |
| **개체 어커런스(Entity Occurrence)** | 개체 타입에 속하는 **실제 값** (인스턴스) |
| **도메인(Domain)** | 한 속성이 가질 수 있는 **값들의 전체 범위** |
| **속성 관계** | 하나의 개체를 기술하는 속성들 사이의 관계 |
| **개체 관계** | 개체 집합과 개체 집합 사이의 관계 |

:::warning 개체 타입의 올바른 정의
> "개체 타입: 개체에 속해 있는 실제 값들로 구성된 개체 어커런스의 집합"

이것은 **틀린 설명**입니다! **개체 타입**은 실제 값들의 집합이 아니라 **개체의 구조(형식)를 정의한 틀**입니다. 실제 값들로 구성된 것은 **개체 어커런스(Entity Occurrence)** 또는 **개체 집합(Entity Set)**입니다.

비유: 개체 타입은 "붕어빵 틀"이고, 개체 어커런스는 "실제로 만들어진 붕어빵" 하나하나입니다.
:::

### DBMS 사용의 장단점

파일 시스템 대비 DBMS를 사용할 때의 장단점을 정확히 알아야 합니다:

**✅ 장점**:
- 여러 사용자가 동시에 같은 데이터를 사용할 수 있습니다
- 데이터의 일관성(Consistency)을 유지할 수 있습니다
- 응용 프로그램 개발 및 유지보수가 용이합니다
- 데이터 중복을 최소화(줄일 수)할 수 있습니다

:::warning DBMS 장점에 대한 흔한 오해
> "데이터 중복을 완벽하게 제거할 수 있다."

이것은 **틀린 설명**입니다! DBMS는 데이터 중복을 **최소화(minimize)**할 수는 있지만, **완전히 제거**하는 것은 불가능합니다. 성능 향상이나 시스템 설계상의 이유로 일부 의도적 중복이 존재할 수 있습니다.
:::

**❌ 단점**:
- 초기 구축 비용이 높습니다
- 데이터 간의 복잡한 관계로 인해 **표준화가 어렵습니다**

:::note DBMS 단점 — 표준화
"데이터 간의 복잡한 관계로 인하여 표준화가 어렵다"는 것은 DBMS의 **단점**입니다. 이것을 장점으로 오해하지 않도록 주의하세요.
:::

## 📝 핵심 정리

이 문서에서 배운 핵심 개념들을 정리합니다:

- **자연 조인(Natural Join)**: 동일 조인의 중복 속성을 하나만 남기고 제거한 조인 방법
- **디비전 연산 주의점**: 대응 애트리뷰트는 이름이 같은 것이 아니라 **도메인이 같아야** 함
- **합병 가능 연산자**: 합집합, 교집합, 차집합은 합병 조건 필요; **카티션 프로덕트는 불필요**
- **B-트리 차수 3 삽입(25,40,120,10,55)**: 최종 루트 값은 **40**
- **SSD**: 반도체를 이용한 저장 장치로 속도 빠름, 소음·발열·기계적 고장 없음
- **디스크 탐구 시간**: 같은 실린더/트랙에 데이터 배치로 단축 가능한 접근 시간
- **RAID 0**: 스트라이핑 방식으로 속도 극대화; 복구 기능 없음
- **DML 예비 컴파일러**: 응용 프로그램 속 DML 명령문을 추출하는 DBMS 구성 요소
- **내장 SQL**: 호스트 변수 참조 **가능**, `EXEC SQL` 접두어 필수
- **NULL 검사**: 반드시 `IS NULL` 사용; `= NULL` 사용 불가
- **DISTINCT**: SQL에서 중복 행을 제거하는 키워드
- **SELECT**: DDL이 아닌 **DML** 명령어
- **참조 무결성**: 외래키는 참조하는 기본키가 반드시 존재해야 함
- **계층적 데이터 모델**: 트리 구조(그래프 아님); 오너-멤버 관계
- **인덱스 순차 파일**: 임의 접근 + 순차 접근 모두 필요할 때 적합
- **비트맵 인덱스**: 삽입/갱신이 많은 테이블에는 **부적합**, OLAP 환경에 적합
- **데이터베이스의 내용에 의한 참조**: 위치/주소가 아닌 **데이터 값**으로 참조
- **DBMS 3대 기능**: 정의·조작·제어 기능; **사전 기능은 필수 기능이 아님**
- **물리적 구조**: 논리적 구조에서 취급 **불가** (독립성 원칙)
- **DBMS 중복 제거**: 완전 제거가 아닌 **최소화**
- **데이터베이스 특성**: 실시간 접근, 계속적 변화, 동시 공용, **내용에 의한 참조**
