# 🔑 슈퍼키와 후보키 구분 및 예시

데이터베이스를 처음 공부하면 "키(Key)"라는 단어가 너무 많이 나와서 헷갈리기 쉽습니다. 슈퍼키, 후보키, 기본키, 대체키, 외래키… 이름만 들어도 머릿속이 복잡해지죠. 하지만 걱정하지 마세요. 이 문서에서는 각 키의 개념을 **실생활 비유와 구체적인 예시**를 통해 단계적으로 설명합니다. 특히 슈퍼키와 후보키를 구분하는 핵심 개념인 **유일성**과 **최소성(비분해성)**을 완벽히 이해할 수 있도록 아주 자세하게 풀어드리겠습니다.

---

## 📌 키(Key)란 무엇인가? — 개념 소개

### 키가 필요한 이유

데이터베이스의 테이블(릴레이션)에는 수많은 행(튜플, row)이 저장됩니다. 예를 들어 학생 테이블에 수백 명의 학생 정보가 들어 있다고 생각해 보세요. 이때 **특정 학생 한 명을 정확하게 찾아내려면** 어떻게 해야 할까요?

이름으로 찾으면 "김민준"이 여러 명일 수 있고, 주소로 찾으면 같은 동네에 사는 사람이 여럿일 수 있습니다. 그래서 데이터베이스에서는 **각 행(튜플)을 유일하게 식별할 수 있는 특별한 속성(컬럼)**이 필요한데, 이것을 바로 **키(Key)** 라고 합니다.

마치 우리가 주민등록번호로 전국 어디서든 특정 사람 한 명을 정확하게 찾아낼 수 있는 것처럼, 데이터베이스의 키는 테이블 안에서 특정 행 하나를 정확히 집어낼 수 있는 식별자 역할을 합니다.

### 키의 종류 한눈에 보기

키는 범위에 따라 계층 구조를 이루고 있습니다. 가장 넓은 개념부터 좁은 개념 순서로 정리하면 다음과 같습니다.

| 키 종류 | 조건 | 설명 |
|---|---|---|
| **슈퍼키 (Super Key)** | 유일성만 만족 | 튜플을 유일하게 식별할 수 있는 모든 속성(집합) |
| **후보키 (Candidate Key)** | 유일성 + 최소성 | 슈퍼키 중에서 최소성까지 만족하는 것 |
| **기본키 (Primary Key)** | 후보키 중 하나 선택 | 설계자가 후보키 중에서 딱 하나만 지정 |
| **대체키 (Alternate Key)** | 후보키 중 기본키 제외 | 후보키 중 기본키로 선택되지 않은 나머지 |
| **외래키 (Foreign Key)** | 다른 테이블의 기본키 참조 | 테이블 간의 관계를 맺어주는 키 |

> **핵심 포인트:** 슈퍼키가 가장 큰 범위이고, 그 안에 후보키가 있으며, 후보키 안에 기본키와 대체키가 들어있습니다. 즉, **후보키는 슈퍼키의 부분집합**이고, **기본키와 대체키는 후보키의 부분집합**입니다.

## 🧩 유일성과 최소성 — 슈퍼키와 후보키를 나누는 핵심 기준

슈퍼키와 후보키를 구분하려면 반드시 **유일성(Uniqueness)**과 **최소성(Minimality, 비분해성)** 이라는 두 가지 개념을 먼저 이해해야 합니다. 이 두 개념이 이 단원에서 가장 중요하며, 처음에는 헷갈릴 수 있으니 아주 천천히 짚어봅니다.

### 유일성(Uniqueness)이란?

**유일성**이란, 릴레이션(테이블) 안에 있는 **모든 튜플(행)에서 해당 키 값이 모두 다른 것**을 의미합니다. 즉, 같은 값이 두 번 이상 등장하지 않아야 한다는 뜻입니다.

예를 들어, 반 친구들의 학번이 모두 다르다면 "학번은 유일성을 만족한다"고 말할 수 있습니다. 반면, 이름은 "김민준"이 두 명 있을 수 있으므로 유일성을 만족하지 못합니다.

> **유일성 = "이 속성의 값으로 각 행(튜플)을 구별할 수 있다"**

### 최소성(비분해성, Minimality)이란?

**최소성**은 조금 더 까다로운 개념입니다. 다른 말로 **비분해성(Irreducibility)** 이라고도 하는데, 뜻은 이렇습니다.

> **키를 구성하는 속성들 중에서 어떤 속성 하나라도 제거하면, 유일성이 깨지는 성질**

즉, 키가 여러 속성의 조합으로 이루어졌을 때, **어느 하나라도 빼면 더 이상 유일하게 식별할 수 없게 된다**면 그 키는 최소성을 만족하는 것입니다. 반대로, 속성 하나를 빼도 여전히 유일하게 식별할 수 있다면 **그 키는 불필요한 속성을 가지고 있는 것**이고, 최소성을 만족하지 못합니다.

**실생활 비유로 이해하기:**
마치 자물쇠를 여는 열쇠 조합을 생각해 보세요. 세 자리 비밀번호 "1-2-3"이 있을 때, "1-2-3" 전부가 있어야만 자물쇠가 열린다면, 세 숫자 모두 반드시 필요한 것이고 이것이 최소성을 만족합니다. 하지만 "1-2"만으로도 자물쇠가 열린다면, "3"은 굳이 필요 없는 것이고 이는 최소성을 만족하지 못합니다.

### 두 개념의 비교 정리

| 구분 | 유일성 | 최소성(비분해성) |
|---|---|---|
| **의미** | 모든 튜플에서 키 값이 서로 다름 | 키 구성 속성 중 하나라도 빼면 유일성이 깨짐 |
| **슈퍼키** | ✅ 반드시 만족 | ❌ 만족하지 않아도 됨 |
| **후보키** | ✅ 반드시 만족 | ✅ 반드시 만족 |

:::info 핵심 요약
- **슈퍼키** = 유일성만 만족하면 OK
- **후보키** = 유일성 **AND** 최소성(비분해성) 모두 만족해야 OK
- 모든 후보키는 슈퍼키이지만, 모든 슈퍼키가 후보키인 것은 아닙니다.
:::

## 🔬 슈퍼키와 후보키 판별 — 핵심 원리와 단계별 예시

이제 실제 데이터를 가지고 슈퍼키와 후보키를 직접 판별하는 방법을 연습해 봅시다. 두 가지 예제를 통해 완벽히 이해해 보겠습니다.

### 예제 1: 복합키가 후보키가 되는 경우

아래와 같은 릴레이션 R이 있다고 가정합니다.

| A | B |
|---|---|
| A1 | B1 |
| A1 | B2 |
| A2 | B1 |

이 릴레이션에서 어떤 속성(조합)이 슈퍼키이고 후보키인지 판별해 봅시다.

**① 속성 A만 키가 될 수 있을까?**

A 컬럼을 보면 A1, A1, A2로 **A1이 두 번 중복**됩니다. 유일성을 만족하지 못하므로, A는 키가 될 수 없습니다.

**② 속성 B만 키가 될 수 있을까?**

B 컬럼을 보면 B1, B2, B1으로 **B1이 두 번 중복**됩니다. 마찬가지로 유일성을 만족하지 못하므로, B도 키가 될 수 없습니다.

**③ 복합 속성 {A, B}는 키가 될 수 있을까?**

A와 B를 합쳐서 보면 (A1,B1), (A1,B2), (A2,B1)으로 **세 개 모두 다른 값**을 가집니다. 유일성을 만족하므로, {A, B}는 키가 될 수 있습니다!

**④ {A, B}는 슈퍼키인가?**

유일성을 만족하므로 → ✅ **슈퍼키입니다.**

**⑤ {A, B}는 후보키인가? (최소성 검사)**

이제 최소성을 검사합니다. {A, B}에서 속성을 하나씩 제거해 봅니다.

- **A를 제거하면** → B만 남습니다. B는 B1, B2, B1으로 중복이 있으므로 유일성이 깨집니다. ❌
- **B를 제거하면** → A만 남습니다. A는 A1, A1, A2로 중복이 있으므로 유일성이 깨집니다. ❌

두 속성 모두, 하나를 제거하면 유일성이 깨지므로 **{A, B}는 최소성(비분해성)을 만족**합니다.

따라서 {A, B}는 슈퍼키이면서 동시에 **후보키**입니다. ✅

```sql
-- 예제 1 릴레이션 R 데이터
-- A  | B
-- A1 | B1  → (A1, B1) : 유일
-- A1 | B2  → (A1, B2) : 유일
-- A2 | B1  → (A2, B1) : 유일
-- A 단독: A1 중복 → 키 불가
-- B 단독: B1 중복 → 키 불가
-- {A,B} 복합: 모두 다름 → 슈퍼키 O, 후보키 O (최소성 만족)
```

### 예제 2: 복합키가 슈퍼키는 되지만 후보키는 되지 못하는 경우

이번에는 다른 릴레이션 R을 살펴봅니다.

| A | B |
|---|---|
| A1 | B1 |
| A2 | B2 |
| A1 | B3 |

**① 속성 A만 키가 될 수 있을까?**

A 컬럼: A1, A2, A1 → **A1이 중복**됩니다. 유일성 불만족, 키 불가.

**② 속성 B만 키가 될 수 있을까?**

B 컬럼: B1, B2, B3 → **세 값이 모두 다릅니다.** 유일성 만족! ✅ B는 키가 될 수 있습니다.

**③ 복합 속성 {A, B}는 키가 될 수 있을까?**

(A1,B1), (A2,B2), (A1,B3) → 모두 다른 값, 유일성 만족. ✅ {A,B}도 키가 될 수 있습니다.

**④ {A, B}는 슈퍼키인가?**

유일성을 만족하므로 → ✅ **슈퍼키입니다.**

**⑤ {A, B}는 후보키인가? (최소성 검사)**

{A, B}에서 속성을 하나씩 제거해 봅니다.

- **A를 제거하면** → B만 남습니다. B는 B1, B2, B3으로 모두 다릅니다. **B만으로도 키가 됩니다!** ✅

A를 제거했더니 B만으로도 유일하게 식별할 수 있습니다. 이 말은 **{A, B}에서 A는 사실 없어도 된다**는 뜻입니다. 즉, **최소성(비분해성)을 만족하지 못합니다.** ❌

따라서 {A, B}는 슈퍼키는 되지만 **후보키는 될 수 없습니다.** 왜냐하면 B 하나만으로도 유일하게 식별 가능하기 때문입니다.

:::warning 최소성 판별 시 주의사항
최소성 검사는 "**속성 하나를 제거했을 때 유일성이 무조건 깨지는가?**"를 확인하는 것입니다.
만약 하나를 제거해도 여전히 유일하게 식별 가능하다면, 그 복합키는 **불필요한 속성을 포함하고 있는 것**이므로 후보키가 될 수 없습니다.
:::

```sql
-- 예제 2 릴레이션 R 데이터
-- A  | B
-- A1 | B1
-- A2 | B2
-- A1 | B3

-- B 단독: B1, B2, B3 → 모두 다름 → 슈퍼키 O, 후보키 O (단일 속성이므로 최소성 자동 만족)
-- {A,B} 복합: 모두 다름 → 슈퍼키 O
-- {A,B}에서 A 제거 → B만으로도 유일 → 최소성 불만족 → 후보키 X
```

### 예제 3: 실제 학생 테이블로 슈퍼키와 후보키 찾기

이제 실제 데이터에 가까운 예시로 연습해 봅시다. 아래와 같은 학생 테이블이 있습니다.

| 학번 | 이름 | 주소 | 주민번호 | 전공 |
|---|---|---|---|---|
| 2024001 | 김민준 | 서울 강남구 | 040101-3XXXXXX | 컴퓨터공학 |
| 2024002 | 이서연 | 부산 해운대구 | 050202-2XXXXXX | 전기공학 |
| 2024003 | 박도윤 | 서울 마포구 | 030303-1XXXXXX | 컴퓨터공학 |

각 속성의 특성을 먼저 파악합니다.

- **학번**: 모든 학생이 서로 다른 고유 번호를 가집니다 → 유일성 만족 ✅
- **이름**: "김민준"처럼 동명이인이 존재할 수 있습니다 → 유일성 불만족 ❌
- **주소**: 같은 집에 사는 가족이 같은 학교에 다닐 수 있습니다 → 유일성 불만족 ❌
- **주민번호**: 대한민국 모든 국민이 서로 다른 주민번호를 가집니다 → 유일성 만족 ✅
- **전공**: 같은 전공을 가진 학생이 여러 명입니다 → 유일성 불만족 ❌

**단일 속성 키 후보 분석:**

- `{학번}` → 유일성 O, 더 이상 분해 불가 → **슈퍼키 ✅, 후보키 ✅**
- `{이름}` → 유일성 X → 키 불가 ❌
- `{주소}` → 유일성 X → 키 불가 ❌
- `{주민번호}` → 유일성 O, 더 이상 분해 불가 → **슈퍼키 ✅, 후보키 ✅**
- `{전공}` → 유일성 X → 키 불가 ❌

**복합 속성 키 후보 분석:**

`{학번, 이름}` → 학번이 유일하니 이를 포함한 조합도 유일합니다 → 슈퍼키 ✅. 하지만 학번을 제거해도 이름만으로는 키가 안 되고, **이름을 제거하면 학번만으로 키가 됩니다**. 즉, 이름이 없어도 식별 가능 → 최소성 불만족 → **후보키 ❌**

`{주민번호, 전공}` → 주민번호가 유일하니 이를 포함한 조합도 유일합니다 → 슈퍼키 ✅. 하지만 전공을 제거해도 주민번호만으로 키가 됩니다 → 최소성 불만족 → **후보키 ❌**

`{학번, 주민번호}` → 둘 다 유일하니 조합도 유일합니다 → 슈퍼키 ✅. 학번을 제거해도 주민번호만으로 키가 됩니다. 주민번호를 제거해도 학번만으로 키가 됩니다 → 어느 쪽을 빼도 남은 것만으로 키 가능 → 최소성 불만족 → **후보키 ❌**

**결론:**

이 학생 테이블에서 **후보키는 `{학번}`과 `{주민번호}` 두 가지**입니다. 나머지 학번이나 주민번호를 포함한 모든 복합 조합들은 슈퍼키는 되지만 후보키는 되지 못합니다.

> **기본키는 후보키 중에서 설계자가 딱 하나만 선택하는 것입니다.** 이 경우 `{학번}`을 기본키로 선택한다면, `{주민번호}`는 **대체키(Alternate Key)** 가 됩니다.

:::tip 슈퍼키 판별 꿀팁
슈퍼키를 찾을 때는 모든 속성 조합(부분집합)을 만들어보고, 각 조합이 유일성을 만족하는지 확인합니다. 속성이 n개라면 만들 수 있는 조합의 수는 `2ⁿ - 1`개(공집합 제외)입니다. 예를 들어 속성이 5개라면 `2⁵ - 1 = 31`가지 조합을 검토해야 합니다.
:::

## 🗂️ 기본키, 대체키, 외래키 — 실무에서 사용하는 키들

### 기본키(Primary Key)

**기본키**는 후보키 중에서 **데이터베이스 설계자가 단 하나만 선택한 키**입니다. 마치 여러 명의 후보 중에서 딱 한 명을 대표로 뽑는 것과 같습니다.

기본키는 아래의 두 가지 조건을 반드시 지켜야 합니다. 이를 **개체 무결성(Entity Integrity)** 이라고 합니다.

1. **기본키는 절대 NULL(빈 값) 값을 가질 수 없습니다.** 기본키가 없으면 그 행을 식별할 방법이 없기 때문입니다.
2. **기본키는 유일한 값만 가져야 합니다.** 중복된 기본키 값은 허용되지 않습니다.

```sql
-- 학생 테이블 생성 시 기본키 지정 예시
CREATE TABLE Student (
    학번 VARCHAR(10) NOT NULL,   -- NULL 불허 (개체 무결성)
    이름 VARCHAR(20),
    주민번호 CHAR(14) UNIQUE,    -- UNIQUE = 대체키로 지정
    전공 VARCHAR(30),
    PRIMARY KEY (학번)           -- 기본키 지정
);
```

### 대체키(Alternate Key)

**대체키**는 후보키 중에서 기본키로 선택되지 못한 나머지 후보키들입니다. 예를 들어 후보키가 `{학번}`과 `{주민번호}` 두 개라면, `{학번}`을 기본키로 지정했을 때 `{주민번호}`가 대체키가 됩니다.

대체키와 기본키의 중요한 차이점 하나가 있습니다.

| 구분 | 기본키(Primary Key) | 대체키(Alternate Key) |
|---|---|---|
| NULL 허용 | ❌ 절대 불허 | ✅ NULL 허용 가능 |
| 유일성 | ✅ 반드시 유일 | ✅ 반드시 유일 |
| 개수 | 테이블당 1개만 가능 | 여러 개 가능 |
| SQL 키워드 | `PRIMARY KEY` | `UNIQUE` |

**대체키는 NULL 값을 가질 수 있습니다.** 예를 들어 외국 유학생의 경우 한국 주민번호가 없을 수 있으므로, 주민번호 컬럼에 NULL이 들어갈 수 있습니다.

:::note 대체키 SQL 표현
SQL에서 대체키는 `UNIQUE` 제약 조건으로 표현합니다. `UNIQUE`는 중복 값을 허용하지 않지만, NULL은 허용합니다(NULL은 "값 없음"으로 간주하여 중복으로 처리하지 않습니다).
:::

### 외래키(Foreign Key)

**외래키**는 다른 테이블의 기본키를 참조하기 위해 가져온 키입니다. 마치 "이 학생의 담당 교수가 누구인지"를 연결하기 위해, 교수 테이블의 교수번호를 학생 테이블에 가져다 쓰는 것과 같습니다.

외래키의 핵심 규칙은 다음과 같습니다.

1. **외래키의 값은 반드시 참조하는 테이블의 기본키 값 중 하나여야 합니다.** 존재하지 않는 기본키 값을 외래키로 가질 수 없습니다.
2. **외래키는 NULL 값을 가질 수 있습니다.** 아직 어떤 값을 참조할지 결정하지 못했을 때 NULL을 사용합니다.
3. **외래키의 데이터 타입(도메인)은 참조하는 기본키와 동일해야 합니다.**

```sql
-- 교수 테이블: 교수번호가 기본키
CREATE TABLE Professor (
    교수번호 INT PRIMARY KEY,    -- 기본키
    교수명 VARCHAR(20),
    학과번호 INT,               -- 학과 테이블의 기본키를 참조하는 외래키
    FOREIGN KEY (학과번호) REFERENCES Department(학과번호)
    ON DELETE CASCADE           -- 참조된 기본키 삭제 시 함께 삭제
    ON UPDATE CASCADE           -- 참조된 기본키 수정 시 함께 수정
);

-- 학과 테이블: 학과번호가 기본키
CREATE TABLE Department (
    학과번호 INT PRIMARY KEY,   -- 기본키
    학과명 VARCHAR(30),
    학과장_교수번호 INT,         -- 교수 테이블의 기본키를 참조하는 외래키
    FOREIGN KEY (학과장_교수번호) REFERENCES Professor(교수번호)
);
```

**외래키 관계 이해하기:**

- R1 테이블이 R2 테이블의 기본키를 외래키로 가질 때
- R1은 **참조하는 릴레이션(Referencing Relation)** 이라고 합니다
- R2는 **참조되는 릴레이션(Referenced Relation)** 이라고 합니다

:::info 자기 참조(Self-Referencing)
외래키는 반드시 다른 테이블을 참조해야 하는 것이 아닙니다. **같은 테이블 내에서 자신의 기본키를 참조하는 것도 가능**합니다. 이를 **자기 참조(Self-Referencing)** 라고 합니다.

예를 들어, 직원 테이블에서 "상사 직원번호" 컬럼이 같은 테이블의 "직원번호" 기본키를 참조하는 경우가 이에 해당합니다. 직원들 사이의 상하 관계를 하나의 테이블로 표현할 때 유용합니다.
:::

## ⚠️ 참조 무결성과 외래키 옵션 — 주의사항 및 팁

### 참조 무결성(Referential Integrity)이란?

**참조 무결성**은 외래키가 참조하는 기본키의 값이 항상 유효해야 한다는 제약 조건입니다. 쉽게 말해, **"없는 교수번호를 지도교수로 가진 학생은 존재할 수 없다"** 는 원칙입니다.

이 무결성을 유지하기 위해, 참조되는 테이블(R2)의 기본키 값이 **변경되거나 삭제될 때** 참조하는 테이블(R1)의 외래키를 어떻게 처리할지 정해야 합니다. 이것이 바로 외래키의 **옵션(Option)** 입니다.

### 외래키의 4가지 처리 옵션

참조되는 기본키가 **삭제(DELETE)** 또는 **수정(UPDATE)** 될 때, 외래키를 어떻게 처리할지 4가지 옵션이 있습니다.

| 옵션 | 설명 | 비유 |
|---|---|---|
| **CASCADE** | 기본키와 함께 외래키도 같이 삭제/수정 | 부모가 이사 가면 자식도 함께 이사 |
| **SET NULL (Nullified)** | 기본키가 삭제/수정되면 외래키를 NULL로 변경 | 담당자가 퇴사하면 담당자란을 공백으로 |
| **RESTRICT (No Action)** | 외래키에서 참조 중이면 기본키 삭제/수정 거부 | 대출 중인 책은 반납 전까지 폐기 불가 |
| **SET DEFAULT** | 기본키가 삭제/수정되면 외래키를 사전에 정해놓은 기본값으로 변경 | 담당자 퇴사 시 자동으로 "미배정" 처리 |

```sql
-- CASCADE 예시: 학과가 폐과되면 해당 학과 교수 정보도 함께 삭제
CREATE TABLE Professor (
    교수번호 INT PRIMARY KEY,
    교수명 VARCHAR(20),
    학과번호 INT,
    FOREIGN KEY (학과번호) REFERENCES Department(학과번호)
    ON DELETE CASCADE    -- 학과 삭제 시 교수도 함께 삭제
    ON UPDATE CASCADE    -- 학과번호 변경 시 교수의 학과번호도 함께 변경
);

-- SET NULL 예시: 학과가 폐과되면 교수의 학과번호를 NULL로 처리
CREATE TABLE Professor (
    교수번호 INT PRIMARY KEY,
    교수명 VARCHAR(20),
    학과번호 INT,        -- NULL 허용 필요
    FOREIGN KEY (학과번호) REFERENCES Department(학과번호)
    ON DELETE SET NULL   -- 학과 삭제 시 학과번호를 NULL로 변경
);

-- RESTRICT 예시: 교수가 속한 학과는 삭제 불가
CREATE TABLE Professor (
    교수번호 INT PRIMARY KEY,
    교수명 VARCHAR(20),
    학과번호 INT,
    FOREIGN KEY (학과번호) REFERENCES Department(학과번호)
    ON DELETE RESTRICT   -- 참조 중인 학과는 삭제 거부
);
```

:::danger 외래키 없이 데이터 관리하면 생기는 문제
외래키 제약 조건 없이 데이터를 관리하면 **존재하지 않는 교수번호를 참조하는 학생 레코드**가 생길 수 있습니다. 이런 상황을 "고아 레코드(Orphan Record)"라고 하며, 데이터의 신뢰성이 완전히 무너집니다. 반드시 외래키 제약 조건과 적절한 옵션을 설정하세요.
:::

### 무결성의 종류 총정리

데이터베이스에서 보장해야 할 무결성은 크게 네 가지입니다.

**① 개체 무결성 (Entity Integrity)**

기본키는 NULL 값을 가질 수 없고, 중복될 수 없다는 제약입니다. `PRIMARY KEY` 제약으로 자동 적용됩니다.

**② 참조 무결성 (Referential Integrity)**

외래키는 참조하는 테이블의 기본키에 존재하는 값이거나 NULL이어야 한다는 제약입니다. `FOREIGN KEY` 제약으로 적용됩니다.

**③ 도메인 무결성 (Domain Integrity)**

각 속성의 값은 해당 속성이 가질 수 있는 값의 범위(도메인) 안에 있어야 한다는 제약입니다. 예를 들어 점수 속성에 0~100 범위를 벗어난 120이 입력되거나, 날짜 속성에 2024-02-30 같은 존재하지 않는 날짜가 입력되는 것을 막습니다. 또한 "근무상태"가 1(정상), 2(휴직), 3(퇴직)으로 정의되어 있다면 4를 입력하면 오류가 발생합니다.

```sql
-- 도메인 무결성 예시: CHECK 제약으로 점수 범위 제한
CREATE TABLE Grade (
    학번 VARCHAR(10),
    과목코드 VARCHAR(10),
    점수 INT CHECK (점수 >= 0 AND 점수 <= 100),  -- 0~100만 허용
    PRIMARY KEY (학번, 과목코드)
);

-- 도메인 무결성 예시: NOT NULL로 필수 입력 강제
CREATE TABLE Student (
    학번 VARCHAR(10) PRIMARY KEY,
    이름 VARCHAR(20) NOT NULL,   -- 이름은 반드시 입력해야 함
    전공 VARCHAR(30)             -- 전공은 입력 안 해도 됨 (NULL 허용)
);
```

**④ 의미 무결성 (Semantic Integrity)**

데이터베이스에 저장된 데이터가 **업무 규칙이나 의미적 제약**에 부합해야 한다는 제약입니다. 예를 들어 쇼핑몰에서 "배송 수량은 주문 수량보다 클 수 없다"거나, "올해 연봉이 작년 연봉보다 낮아질 수 없다"는 비즈니스 규칙이 이에 해당합니다. 이러한 제약은 `CHECK` 제약이나 트리거(Trigger)로 구현합니다.

```sql
-- 의미 무결성 예시: 배송 수량은 주문 수량보다 클 수 없음
CREATE TABLE Order_Detail (
    주문번호 INT,
    상품코드 VARCHAR(20),
    주문수량 INT NOT NULL,
    배송수량 INT DEFAULT 0,
    -- 배송수량이 주문수량을 초과하면 입력 거부
    CHECK (배송수량 <= 주문수량),
    PRIMARY KEY (주문번호, 상품코드)
);
```

:::tip 무결성은 시스템이 자동으로 지켜줍니다
무결성 제약 조건을 한 번 설정해 놓으면, 사용자가 매번 직접 확인하지 않아도 됩니다. 데이터베이스 시스템이 **INSERT, UPDATE, DELETE 등의 연산이 실행될 때마다 자동으로 제약 조건을 검사**하고, 조건을 위반하면 해당 연산을 거부합니다. 마치 경비원이 24시간 자동으로 출입을 통제하는 것과 같습니다.
:::

## 🧠 잘못된 예시 vs 올바른 예시 — 실수 방지 가이드

### 키 판별 시 흔히 하는 실수

:::warning 주의: 현재 데이터만 보고 유일성을 판단하지 마세요!

❌ **잘못된 판단:** 현재 테이블에 이름이 모두 다르니까 이름이 기본키가 될 수 있다.

✅ **올바른 판단:** 이름은 앞으로 동명이인이 입력될 가능성이 있으므로 유일성을 보장할 수 없고, 기본키가 될 수 없다.

키를 선정할 때는 **현재 데이터뿐 아니라 미래에 추가될 수 있는 모든 가능한 데이터를 고려**해야 합니다.
:::

**❌ 잘못된 키 설계 예시:**

```sql
-- 잘못된 예: 이름을 기본키로 사용
CREATE TABLE Student (
    이름 VARCHAR(20) PRIMARY KEY,  -- ❌ 동명이인 문제 발생 가능
    전공 VARCHAR(30),
    학년 INT
);
-- 나중에 "김민준"이 두 명 입력되면 오류 발생!
```

**✅ 올바른 키 설계 예시:**

```sql
-- 올바른 예: 학번을 기본키로 사용, 이름은 일반 속성으로
CREATE TABLE Student (
    학번 VARCHAR(10) PRIMARY KEY,  -- ✅ 항상 유일함이 보장되는 학번을 기본키로
    이름 VARCHAR(20) NOT NULL,     -- 이름은 일반 속성 (NULL만 불허)
    전공 VARCHAR(30),
    학년 INT CHECK (학년 >= 1 AND 학년 <= 4)  -- 도메인 무결성
);
```

### 외래키 설정 시 흔히 하는 실수

**❌ 잘못된 외래키 예시:**

```sql
-- 잘못된 예: 외래키 제약 없이 참조 컬럼만 추가
CREATE TABLE Student (
    학번 VARCHAR(10) PRIMARY KEY,
    이름 VARCHAR(20),
    교수번호 INT   -- ❌ 외래키 제약 없음 → 존재하지 않는 교수번호 입력 가능!
);
```

**✅ 올바른 외래키 예시:**

```sql
-- 올바른 예: 외래키 제약 조건 명시
CREATE TABLE Student (
    학번 VARCHAR(10) PRIMARY KEY,
    이름 VARCHAR(20) NOT NULL,
    교수번호 INT,    -- NULL 허용 (아직 지도교수 미배정 가능)
    -- ✅ 외래키 제약 명시: Professor 테이블의 교수번호만 참조 가능
    FOREIGN KEY (교수번호) REFERENCES Professor(교수번호)
    ON DELETE SET NULL   -- 교수 삭제 시 학생의 교수번호는 NULL로
    ON UPDATE CASCADE    -- 교수번호 변경 시 학생 테이블도 함께 변경
);
```

## 📊 핵심 정리

이 문서에서 배운 핵심 개념을 정리합니다.

- **키(Key)**: 릴레이션에서 특정 튜플(행)을 유일하게 식별할 수 있는 속성 또는 속성들의 집합
- **유일성**: 모든 튜플에서 키 값이 서로 다른 성질 → 슈퍼키와 후보키 모두 필수 조건
- **최소성(비분해성)**: 키 구성 속성 중 어느 하나라도 제거하면 유일성이 깨지는 성질 → 후보키만의 추가 조건
- **슈퍼키**: 유일성만 만족하는 속성(집합) → 후보키를 포함하는 더 넓은 개념
- **후보키**: 유일성 + 최소성(비분해성)을 모두 만족하는 속성(집합)
- **기본키**: 후보키 중 설계자가 딱 하나만 선택한 키 → NULL 불허, 중복 불허 (개체 무결성)
- **대체키**: 후보키 중 기본키로 선택되지 않은 나머지 → NULL 허용 가능
- **외래키**: 다른 테이블의 기본키를 참조하는 속성 → NULL 허용, 참조 무결성 준수 필요
- **자기 참조(Self-Referencing)**: 외래키가 같은 테이블의 기본키를 참조하는 것도 가능
- **참조 무결성 옵션**: CASCADE(연쇄 처리), SET NULL(NULL로 변경), RESTRICT(거부), SET DEFAULT(기본값으로 변경)
- **개체 무결성**: 기본키는 NULL이 될 수 없고 유일해야 함
- **도메인 무결성**: 속성값은 정해진 도메인 범위 안에 있어야 함
- **참조 무결성**: 외래키는 참조 대상 기본키에 존재하는 값이거나 NULL이어야 함
- **의미 무결성**: 데이터는 업무적 의미와 규칙에 부합해야 함
- **무결성 제약은 자동 관리**: 설정 후 데이터베이스 시스템이 자동으로 검사하고 위반 시 거부
