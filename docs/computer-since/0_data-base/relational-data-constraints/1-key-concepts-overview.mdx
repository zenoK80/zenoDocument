# 🔑 키의 개념과 종류 개요 — 관계 데이터의 제약을 이해하는 첫걸음

데이터베이스를 배우다 보면 "기본키", "외래키"라는 단어를 자주 듣게 됩니다. 그런데 막상 "슈퍼키가 뭐예요?", "후보키와 기본키는 어떻게 달라요?"라는 질문을 받으면 선뜻 대답하기 어렵죠. 이 문서에서는 데이터베이스의 핵심 개념인 **키(Key)** 의 종류와 그 관계를 처음 배우는 분도 완전히 이해할 수 있도록 풍부한 예시와 함께 아주 꼼꼼하게 설명합니다.

---

## 📌 관계 데이터의 제약이란 무엇인가?

### 왜 "제약"이 필요한가?

데이터베이스는 단순히 데이터를 저장하는 창고가 아닙니다. **정확하고 일관된 데이터**를 유지해야 하는 시스템입니다. 예를 들어 학생 정보를 저장하는 시스템에서 같은 학번을 가진 학생이 두 명 등록되거나, 존재하지 않는 교수를 지도교수로 지정하는 일이 생긴다면 어떻게 될까요? 시스템 전체의 신뢰성이 무너지게 됩니다.

이런 문제를 막기 위해 데이터베이스는 **제약 조건(Constraint)** 을 둡니다. 마치 은행 ATM기가 잔액보다 많은 금액은 출금하지 못하도록 막는 것처럼, 데이터베이스도 규칙에 맞지 않는 데이터는 아예 저장 자체를 거부합니다.

### 제약 조건의 가장 기본 — 키(Key)

관계 데이터의 제약 중 가장 기초가 되는 것이 바로 **키에 대한 제약**입니다. ER 다이어그램을 릴레이션(테이블)으로 변환할 때 우리는 기본키와 외래키를 사용해 테이블 간의 관계를 맺어줍니다.

- **1:N 관계**에서는 1쪽의 기본키를 N쪽의 외래키로 가져와서 관계를 표현합니다.
- **M:N 관계**에서는 양쪽 테이블의 기본키를 모두 가져와 별도의 관계 테이블을 만들고, 복합키(두 개 이상의 속성 조합)를 기본키로 사용합니다.

이처럼 키는 단순한 식별자가 아니라, **테이블 간의 관계를 정의하고 데이터의 무결성(데이터가 정확하고 올바른 상태)을 지켜주는 핵심 장치**입니다.

> 💡 **무결성(Integrity)** 이란 데이터가 오류 없이 정확하고 일관된 상태를 유지하는 성질을 말합니다. 데이터베이스에서는 이 무결성을 지키는 것이 가장 중요한 목표 중 하나입니다.

## 🗝️ 키의 종류와 계층 구조

### 키의 전체 그림 먼저 보기

키를 처음 배울 때 가장 헷갈리는 부분은 종류가 여러 개라는 점입니다. 먼저 전체 구조를 한눈에 파악해 보겠습니다.

| 키 종류 | 유일성 | 최소성(비분해성) | 설명 |
|---|---|---|---|
| **슈퍼키 (Super Key)** | ✅ 만족 | ❌ 불필요 | 유일성만 만족하면 됨 |
| **후보키 (Candidate Key)** | ✅ 만족 | ✅ 만족 | 유일성 + 최소성 모두 만족 |
| **기본키 (Primary Key)** | ✅ 만족 | ✅ 만족 | 후보키 중 설계자가 선택한 단 하나의 키 |
| **대체키 (Alternate Key)** | ✅ 만족 | ✅ 만족 | 후보키 중 기본키로 선택되지 않은 나머지 키 |
| **외래키 (Foreign Key)** | 불필요 | 불필요 | 다른 테이블의 기본키를 참조하는 속성 |

이 계층 구조를 그림으로 표현하면 다음과 같습니다.

```
슈퍼키 (가장 넓은 범위)
  └─ 후보키 (슈퍼키의 부분집합)
        ├─ 기본키 (후보키 중 딱 하나)
        └─ 대체키 (기본키가 아닌 나머지 후보키)
```

:::info 핵심 포인트
슈퍼키가 후보키보다 **더 큰 범위**를 갖습니다. 즉, 모든 후보키는 슈퍼키이지만, 모든 슈퍼키가 후보키인 것은 아닙니다. 후보키는 슈퍼키의 **부분집합**입니다.
:::

## 🔍 유일성과 최소성(비분해성) — 키를 구분하는 두 가지 기준

### 두 개념이 왜 중요한가?

슈퍼키와 후보키를 구분하는 핵심 기준은 딱 두 가지입니다: **유일성**과 **최소성(비분해성)**. 이 두 개념을 정확히 이해해야 나머지 키의 개념이 자연스럽게 따라옵니다.

### 유일성(Uniqueness)이란?

**유일성**이란 릴레이션(테이블)에 있는 모든 튜플(행)에 대해 키 값이 모두 다른 것을 의미합니다. 마치 주민등록번호처럼, 모든 사람이 서로 다른 번호를 가지고 있어서 그 번호만으로 특정 한 사람을 정확히 찾아낼 수 있는 것과 같습니다.

> 릴레이션에 있는 **모든 튜플의 키 값이 서로 다르면** 유일성을 만족한다고 말합니다.

예를 들어, 학생 테이블에서 "이름" 속성은 유일성을 만족할까요? 같은 이름을 가진 학생이 둘 이상 있을 수 있기 때문에 유일성을 만족하지 못합니다. 반면 "학번"은 모든 학생이 서로 다른 번호를 가지므로 유일성을 만족합니다.

### 최소성(Minimality) / 비분해성(Irreducibility)이란?

**최소성** 또는 **비분해성**은 조금 더 이해가 필요한 개념입니다. 이름이 두 개인데, 같은 의미입니다.

> 키를 구성하는 속성(애트리뷰트) 집합에서 **어떤 속성을 하나라도 제거하면 유일성이 깨지는** 성질을 최소성(비분해성)이라고 합니다.

쉽게 말하면, "이 키가 유일하게 역할을 하려면 지금 이 속성들이 **최소한** 모두 필요하다"는 의미입니다. 하나라도 빼면 더 이상 유일한 식별이 불가능해지는 것이죠.

**비분해성**이라는 이름도 같은 의미입니다 — "분해하면 키가 될 수 없다", 즉 더 이상 쪼갤 수 없는 최소 단위로 구성된 키라는 뜻입니다.

## 🧪 예시로 완벽하게 이해하는 슈퍼키 vs 후보키

### 예시 1: 두 속성 모두 중복이 있는 경우 (최소성 만족)

아래와 같은 릴레이션 R이 있다고 가정합니다.

| A | B |
|---|---|
| A1 | B1 |
| A1 | B2 |
| A2 | B1 |

**① A만 키가 될 수 있나요?**

A 열을 보면 `A1, A1, A2`로 A1이 중복됩니다. 유일성을 만족하지 못하므로 A만으로는 **키가 될 수 없습니다**.

**② B만 키가 될 수 있나요?**

B 열을 보면 `B1, B2, B1`으로 B1이 중복됩니다. 유일성을 만족하지 못하므로 B만으로도 **키가 될 수 없습니다**.

**③ A, B 복합키는 키가 될 수 있나요?**

A와 B를 합친 값을 보면 `(A1,B1), (A1,B2), (A2,B1)`으로 모두 다릅니다. 유일성을 만족하므로 **키가 될 수 있습니다**.

**④ A,B 복합키는 슈퍼키인가요? 후보키인가요?**

- 슈퍼키 여부: 유일성을 만족하므로 ✅ **슈퍼키 조건 만족**
- 후보키 여부(최소성 검사):
  - A를 제거하면 → B만 남는데, B는 유일하지 않음 → 키가 안 됨 ✅
  - B를 제거하면 → A만 남는데, A는 유일하지 않음 → 키가 안 됨 ✅
  - **어떤 속성을 하나라도 빼면 키가 안 되므로, 최소성(비분해성) 만족**

따라서 A,B 복합키는 **슈퍼키이면서 동시에 후보키**입니다.

:::tip 핵심 판단 방법
최소성(비분해성) 검사는 이렇게 하면 됩니다:
1. 키를 구성하는 각 속성을 하나씩 제거해 봅니다.
2. 제거했을 때 **남은 속성만으로 키가 될 수 없으면** → 최소성 만족 ✅
3. 제거했을 때 **남은 속성만으로도 키가 될 수 있으면** → 최소성 불만족 ❌
:::

### 예시 2: 한 속성만 유일한 경우 (최소성 불만족)

이번엔 아래와 같은 릴레이션 R이 있습니다.

| A | B |
|---|---|
| A1 | B1 |
| A2 | B2 |
| A1 | B3 |

**① B만 키가 될 수 있나요?**

B 열을 보면 `B1, B2, B3`로 모두 다릅니다. 유일성을 만족하므로 **B 혼자서 키가 될 수 있습니다**.

**② A,B 복합키는 슈퍼키인가요? 후보키인가요?**

A,B를 합친 값 `(A1,B1), (A2,B2), (A1,B3)`은 모두 다르므로 유일성을 만족합니다. 슈퍼키 조건은 만족합니다.

그러나 최소성을 검사해 봅니다:
- A를 제거하면 → B만 남는데, **B 혼자서 키가 될 수 있음** ❌

A를 제거해도 B만으로 키가 되므로, A,B 복합키는 **최소성(비분해성)을 만족하지 못합니다**.

따라서 A,B 복합키는 **슈퍼키는 되지만, 후보키는 되지 않습니다**.

:::warning 자주 하는 실수
"복합키(두 속성의 조합)이면 무조건 후보키다"라고 생각하기 쉽습니다. 하지만 복합키라도 **구성 요소 중 하나만으로 키가 될 수 있다면 최소성을 만족하지 못해 후보키가 아닌 슈퍼키에 머물게 됩니다.** 반드시 최소성 검사를 해야 합니다.
:::

## 🎓 실제 학생 데이터로 배우는 슈퍼키와 후보키

### 실제 테이블 예시

아래와 같은 학생 테이블이 있다고 가정합니다.

| 학번 | 이름 | 주소 | 주민번호 | 전공 |
|---|---|---|---|---|
| 2024001 | 김철수 | 서울 강남구 | 000101-1234567 | 컴퓨터공학 |
| 2024002 | 이영희 | 부산 해운대구 | 991215-2345678 | 경영학 |
| 2024003 | 박민준 | 서울 강남구 | 010303-1456789 | 컴퓨터공학 |

### 어떤 속성이 유일성을 갖는가?

현실 세계의 지식을 적용해 봅니다. 이 테이블에 데이터가 많아진다면:

- **이름**: 동명이인(같은 이름을 가진 사람)이 존재할 수 있음 → 유일성 ❌
- **주소**: 같은 주소에 여러 사람이 살 수 있음 (위 예시에서도 서울 강남구가 중복) → 유일성 ❌
- **전공**: 같은 전공 학생이 여럿 존재 (위 예시에서도 컴퓨터공학 중복) → 유일성 ❌
- **학번**: 모든 학생에게 유일하게 부여됨 → 유일성 ✅
- **주민번호**: 모든 사람에게 유일하게 부여됨 → 유일성 ✅

### 슈퍼키 찾기

유일성만 만족하면 슈퍼키입니다. 학번이 유일하므로, 학번을 포함하는 **모든 속성 조합**은 유일성을 유지합니다.

- `{학번}` → 유일 → 슈퍼키 ✅
- `{주민번호}` → 유일 → 슈퍼키 ✅
- `{학번, 이름}` → 학번이 유일하니 조합도 유일 → 슈퍼키 ✅
- `{학번, 주소}` → 슈퍼키 ✅
- `{학번, 주민번호}` → 슈퍼키 ✅
- `{학번, 전공}` → 슈퍼키 ✅
- `{학번, 이름, 주소}` → 슈퍼키 ✅
- `{주민번호, 이름}` → 주민번호가 유일하니 조합도 유일 → 슈퍼키 ✅
- ... (학번이나 주민번호를 포함하는 모든 조합)

이처럼 슈퍼키는 매우 많습니다.

### 후보키 찾기 — 최소성 검사

이제 슈퍼키들 중에서 최소성을 만족하는 것만 후보키가 됩니다.

**`{학번}` 단독의 경우:**
- 학번 하나만으로 유일성 만족 ✅
- 더 이상 분해할 속성이 없음 (혼자이므로) → 최소성 만족 ✅
- **→ 후보키 ✅**

**`{주민번호}` 단독의 경우:**
- 주민번호 하나만으로 유일성 만족 ✅
- 더 이상 분해할 속성이 없음 → 최소성 만족 ✅
- **→ 후보키 ✅**

**`{학번, 이름}` 복합의 경우:**
- 유일성 만족 ✅ (슈퍼키는 됨)
- 최소성 검사: 이름을 제거 → `{학번}`만으로 키가 됨 → 최소성 불만족 ❌
- **→ 슈퍼키만 되고, 후보키는 안 됨 ❌**

**`{주민번호, 전공}` 복합의 경우:**
- 유일성 만족 ✅ (슈퍼키는 됨)
- 최소성 검사: 전공을 제거 → `{주민번호}`만으로 키가 됨 → 최소성 불만족 ❌
- **→ 슈퍼키만 되고, 후보키는 안 됨 ❌**

:::note 정리
이 학생 테이블에서 후보키는 `{학번}`과 `{주민번호}` 단 두 개입니다. 학번이나 주민번호를 포함하는 나머지 모든 조합들은 슈퍼키이긴 하지만 최소성을 만족하지 못하므로 후보키가 될 수 없습니다.
:::

## 👑 기본키, 대체키 — 후보키에서 역할 나누기

### 기본키(Primary Key)란?

여러 후보키 중에서 **데이터베이스 설계자가 대표로 선택한 단 하나의 키**를 기본키라고 합니다. 마치 여러 신분증(주민등록증, 여권, 운전면허증) 중에서 가장 공식적으로 쓰이는 주민등록증처럼, 기본키는 해당 테이블에서 각 행(튜플)을 공식적으로 식별하는 대표 키입니다.

**기본키의 핵심 특성:**

1. **유일성**: 테이블 내 모든 행에서 값이 달라야 합니다.
2. **NOT NULL(널 불가)**: 기본키는 절대로 null(비어있는 값) 값을 가질 수 없습니다. 이것이 **개체 무결성(Entity Integrity)** 입니다.

> 기본키가 null이라는 것은 "어떤 행인지 알 수 없다"는 뜻입니다. 신원이 불명확한 데이터는 신뢰할 수 없으므로, 기본키는 반드시 값이 있어야 합니다.

### 대체키(Alternate Key)란?

후보키가 여러 개일 때, 기본키로 선택되지 않은 나머지 후보키들을 **대체키**라고 합니다. "기본키를 대체할 수 있는 키"라는 의미입니다.

예를 들어, 학생 테이블에서 후보키가 `{학번}`과 `{주민번호}` 두 개라면:
- 설계자가 `{학번}`을 기본키로 선택 → **기본키: 학번**
- 나머지 `{주민번호}` → **대체키: 주민번호**

**기본키와 대체키의 차이:**

| 구분 | 기본키 | 대체키 |
|---|---|---|
| 개수 | 테이블당 반드시 단 1개 | 0개 이상 여러 개 가능 |
| NULL 허용 여부 | **절대 불가** (NOT NULL) | **허용 가능** |
| 유일성 | 만족 | 만족 |
| 최소성 | 만족 | 만족 |
| SQL 키워드 | `PRIMARY KEY` | `UNIQUE` |

:::warning 중요 차이
기본키는 절대 NULL 값을 가질 수 없지만, **대체키는 NULL 값을 가질 수 있습니다**. 이것이 기본키와 대체키의 가장 중요한 실질적 차이입니다. 헷갈리지 않도록 주의하세요!
:::

### 후보키가 3개라면?

만약 어떤 테이블에서 후보키가 `{학번}`, `{주민번호}`, `{이메일}` 세 개라면:
- 기본키: 셋 중 하나 선택 (예: `{학번}`) → **1개**
- 대체키: 나머지 둘 → `{주민번호}`, `{이메일}` → **2개**

후보키 수에서 1을 빼면 대체키의 수가 됩니다.

## 🔗 외래키(Foreign Key) — 테이블 간의 다리

### 외래키란 무엇인가?

**외래키**는 다른 테이블의 기본키 값을 가져와서 두 테이블 사이의 관계를 맺어주는 속성입니다. 마치 도서관 책 대출 기록에 "대출한 사람의 회원번호"가 적혀 있는 것처럼, 한 테이블이 다른 테이블의 특정 행을 "참조"하기 위해 사용합니다.

```
교수 테이블 (R1)          학과 테이블 (R2)
┌─────────┬──────┬──────┐  ┌──────┬──────┐
│교수번호  │교수명│학과번호│  │학과번호│학과명│
│(기본키) │      │(외래키)│  │(기본키)│      │
├─────────┼──────┼──────┤  ├──────┼──────┤
│P001     │김교수│D001  │  │D001  │컴공  │
│P002     │이교수│D002  │  │D002  │경영  │
│P003     │박교수│NULL  │  │D003  │수학  │
└─────────┴──────┴──────┘  └──────┴──────┘
         R1의 학과번호(외래키)는
         R2의 학과번호(기본키)를 참조
```

- **참조하는 릴레이션(Referencing Relation)**: 외래키를 가지고 있는 테이블 (위에서 R1, 교수 테이블)
- **참조되는 릴레이션(Referenced Relation)**: 외래키가 참조하는 기본키를 가진 테이블 (위에서 R2, 학과 테이블)

### 외래키의 핵심 규칙

**규칙 1: 외래키 값은 참조되는 기본키 값에 반드시 존재해야 합니다.**

학과 테이블에 없는 학과번호 `D999`를 교수 테이블의 학과번호에 넣을 수 없습니다. 존재하지 않는 학과에 교수를 배정하는 것은 현실적으로도 말이 안 되죠. 이것이 **참조 무결성(Referential Integrity)** 입니다.

**규칙 2: 외래키는 NULL 값을 가질 수 있습니다.**

위 표에서 박교수의 학과번호가 NULL인 것을 볼 수 있습니다. 이는 "아직 어느 학과로 배정될지 결정되지 않았음"을 의미합니다. 기본키와 달리 외래키는 NULL을 허용합니다.

**규칙 3: 외래키와 참조되는 기본키는 도메인(데이터 타입)이 같아야 합니다.**

학과번호가 참조되는 쪽에서는 숫자 타입인데, 외래키 쪽에서는 문자열 타입으로 저장하면 안 됩니다. 컴퓨터는 값을 이진수로 변환해서 비교하기 때문에, 같은 `10`이라도 숫자 10과 문자 "10"은 서로 다른 의미를 가집니다.

**규칙 4: 외래키는 반드시 다른 테이블의 기본키일 필요는 없습니다 — 같은 테이블도 가능합니다.**

:::note 셀프 레퍼런싱(Self-Referencing)
같은 테이블 내에서 자기 자신의 기본키를 참조하는 외래키를 만들 수 있습니다. 이를 **자기 참조(Self-Referencing)** 라고 합니다.

예를 들어, 사원 테이블에서 "상사 사원번호" 열이 같은 테이블의 "사원번호" 기본키를 참조하면, 누가 누구의 상사인지를 한 테이블에서 표현할 수 있습니다.
:::

## ⚙️ 참조 무결성을 지키는 4가지 옵션

### 왜 옵션이 필요한가?

외래키가 참조하는 기본키 값이 수정되거나 삭제될 때, 외래키를 가진 테이블은 어떻게 처리해야 할까요? 예를 들어, 학과 테이블에서 `D001` 학과가 삭제된다면, 교수 테이블에서 `D001`을 참조하던 행들은 어떻게 되어야 할까요?

이런 상황을 처리하기 위해 외래키를 설정할 때 **4가지 옵션** 중 하나를 선택할 수 있습니다.

### 옵션 1: CASCADE (연쇄 처리)

기본키가 수정되거나 삭제되면, 그것을 참조하는 외래키 값도 **함께 수정되거나 삭제**됩니다. 마치 도미노처럼 연쇄적으로 처리됩니다.

```
학과 테이블에서 D001 삭제
         ↓ CASCADE
교수 테이블에서 학과번호가 D001인 모든 행도 함께 삭제
```

### 옵션 2: SET NULL (NULL로 설정, Nullify)

기본키가 수정되거나 삭제되면, 그것을 참조하던 외래키 값을 **NULL로 변경**합니다. "참조 대상이 없어졌으니 아직 미정"이라는 의미입니다.

```
학과 테이블에서 D001 삭제
         ↓ SET NULL
교수 테이블에서 학과번호가 D001이었던 행의 학과번호를 NULL로 변경
```

### 옵션 3: RESTRICT / NO ACTION (삭제 거부)

기본키 값을 참조하는 외래키가 존재하는 한, **기본키의 수정이나 삭제를 아예 거부**합니다. 에러 메시지를 띄우고 작업을 취소시킵니다.

```
학과 테이블에서 D001을 삭제하려 함
         ↓ RESTRICT
오류 발생! "D001을 참조하는 외래키가 존재하므로 삭제할 수 없습니다."
작업 취소.
```

### 옵션 4: SET DEFAULT (기본값으로 설정)

기본키가 수정되거나 삭제되면, 외래키 값을 **미리 약속된 기본값으로 변경**합니다.

```
학과 테이블에서 D001 삭제
         ↓ SET DEFAULT
교수 테이블에서 학과번호가 D001이었던 행의 학과번호를 기본값(예: D000)으로 변경
```

| 옵션 | SQL 키워드 | 동작 방식 | 사용 시기 |
|---|---|---|---|
| CASCADE | `ON DELETE CASCADE` | 참조하는 행도 함께 삭제 | 종속 관계가 강할 때 |
| SET NULL | `ON DELETE SET NULL` | 외래키를 NULL로 변경 | 선택적 관계일 때 |
| RESTRICT | `ON DELETE RESTRICT` | 삭제 자체를 거부 | 데이터 보호가 중요할 때 |
| SET DEFAULT | `ON DELETE SET DEFAULT` | 기본값으로 변경 | 기본 분류가 있을 때 |

## 🛡️ 무결성의 4가지 종류

### 개체 무결성(Entity Integrity)

**기본키는 NULL 값을 가질 수 없고, 유일해야 한다**는 규칙입니다. 기본키가 NULL이면 어떤 행인지 식별할 수 없기 때문입니다. SQL에서 `PRIMARY KEY`를 지정하면 자동으로 `NOT NULL`과 `UNIQUE` 제약이 함께 적용됩니다.

:::danger 절대 하면 안 되는 것
기본키 열에 NULL 값이나 중복 값을 입력하면 안 됩니다. 이는 개체 무결성 위반으로, 데이터베이스가 자동으로 입력을 거부합니다.
:::

### 참조 무결성(Referential Integrity)

**외래키 값은 참조되는 테이블의 기본키에 존재하는 값이거나 NULL이어야 한다**는 규칙입니다. 존재하지 않는 기본키 값을 외래키로 참조할 수 없습니다.

예를 들어, 학과 테이블에 없는 학과번호를 교수의 소속 학과번호로 입력할 수 없습니다. 실제로 존재하지 않는 학과에 교수를 배정할 수는 없으니까요.

참조 무결성은 사용자가 매번 확인하는 것이 아니라, **한 번 설정해 놓으면 시스템이 자동으로 검사하고 규칙 위반 시 거부**합니다. 일종의 자동 감시 장치라고 볼 수 있습니다.

### 도메인 무결성(Domain Integrity)

**각 속성의 값은 정의된 도메인(값의 범위와 타입) 안에 속해야 한다**는 규칙입니다. "도메인"이란 해당 속성이 가질 수 있는 값의 범위를 의미합니다.

실생활 예시를 들면:

- **시험 점수**가 0~100점 범위인데, **120점**이 입력되면 → 거부 ❌
- **주문 날짜**에 **2020년 2월 30일**이 입력되면 (2월은 최대 29일) → 거부 ❌
- **근무 상태**가 1(정상), 2(휴직), 3(퇴직) 세 가지인데, **4**가 입력되면 → 거부 ❌
- **상품명**에 `NOT NULL` 제약이 있는데, 빈 값으로 입력하면 → 거부 ❌

도메인 무결성은 데이터 타입, 길이, 허용 값 범위, NOT NULL 등의 제약 조건으로 구현됩니다.

### 의미 무결성(Semantic Integrity)

**데이터가 현실 세계의 의미에 맞는 조건을 만족해야 한다**는 규칙입니다. 앞의 세 가지 무결성이 시스템이 자동으로 처리할 수 있는 형식적 규칙이라면, 의미 무결성은 비즈니스 규칙(Business Rule)에 해당하는 의미적 제약입니다.

예시:
- 쇼핑몰에서 **배송 수량은 주문 수량보다 클 수 없다** → 주문 10개인데 배송 12개는 말이 안 됨
- 직원의 **올해 연봉은 작년 연봉보다 적을 수 없다** → 연봉이 삭감되는 것을 방지
- **예약 퇴실일은 예약 입실일보다 이후여야 한다** → 퇴실이 입실보다 먼저일 수는 없음

이런 의미적 제약을 데이터베이스 수준에서 설정해 놓으면, 잘못된 데이터가 입력되는 것을 자동으로 방지할 수 있습니다.

## 💻 SQL로 키 정의하기

### 기본키, 대체키, 외래키 생성 문법

아래는 SQL에서 실제로 키를 정의하는 방법입니다. 자세한 SQL 문법은 이후 SQL 명령어 학습 시 다루지만, 여기서는 키 관련 핵심 키워드를 먼저 익혀둡니다.

```sql
-- 학생 테이블 생성 예시
CREATE TABLE Student (
    student_id    VARCHAR(10)   NOT NULL,         -- 학번 (NULL 불가)
    name          VARCHAR(50)   NOT NULL,         -- 이름 (NULL 불가)
    address       VARCHAR(100),                   -- 주소 (NULL 허용)
    ssn           CHAR(14)      UNIQUE,           -- 주민번호: 대체키 (UNIQUE 키워드 사용)
    major         VARCHAR(50),                   -- 전공
    dept_id       VARCHAR(10),                   -- 학과번호: 외래키

    -- 기본키 지정
    PRIMARY KEY (student_id),

    -- 외래키 지정 (학과 테이블의 dept_id를 참조)
    FOREIGN KEY (dept_id) REFERENCES Department(dept_id)
        ON DELETE SET NULL       -- 학과 삭제 시 학과번호를 NULL로 변경
        ON UPDATE CASCADE        -- 학과번호 변경 시 함께 변경
);
```

**코드 한 줄씩 설명:**

- `student_id VARCHAR(10) NOT NULL` : 학번은 최대 10자리 문자열, 반드시 값이 있어야 함
- `ssn CHAR(14) UNIQUE` : 주민번호는 14자리 고정 문자열, 중복 불가 → **대체키** 역할
- `PRIMARY KEY (student_id)` : 학번을 **기본키**로 지정. NOT NULL + UNIQUE가 자동 적용됨
- `FOREIGN KEY (dept_id) REFERENCES Department(dept_id)` : 학과번호를 **외래키**로 지정하고 Department 테이블의 dept_id(기본키)를 참조함
- `ON DELETE SET NULL` : 참조된 학과가 삭제되면 이 학과번호를 NULL로 변경
- `ON UPDATE CASCADE` : 참조된 학과번호가 변경되면 이 값도 함께 변경

```sql
-- ❌ 잘못된 예시: 기본키에 NULL 삽입 시도
INSERT INTO Student (student_id, name, major)
VALUES (NULL, '김철수', '컴퓨터공학');
-- 오류 발생: student_id는 PRIMARY KEY이므로 NULL 불가

-- ✅ 올바른 예시: 기본키에 유일한 값 삽입
INSERT INTO Student (student_id, name, major, dept_id)
VALUES ('2024001', '김철수', '컴퓨터공학', 'D001');
-- 정상 처리: 학과번호 D001이 Department 테이블에 존재할 때

-- ❌ 잘못된 예시: 존재하지 않는 학과번호로 외래키 위반
INSERT INTO Student (student_id, name, major, dept_id)
VALUES ('2024002', '이영희', '경영학', 'D999');
-- 오류 발생: D999는 Department 테이블에 존재하지 않음 (참조 무결성 위반)

-- ✅ 올바른 예시: 외래키에 NULL 삽입 (아직 학과 미정)
INSERT INTO Student (student_id, name, major, dept_id)
VALUES ('2024003', '박민준', '수학', NULL);
-- 정상 처리: 외래키는 NULL을 허용함
```

:::tip 키워드 정리
- **기본키** → SQL에서 `PRIMARY KEY`
- **대체키** → SQL에서 `UNIQUE`
- **외래키** → SQL에서 `FOREIGN KEY ... REFERENCES ...`
:::

## ⚠️ 주의사항과 실수하기 쉬운 포인트

### 주의사항 1: 슈퍼키와 후보키를 혼동하지 마세요

슈퍼키와 후보키는 다릅니다. 유일성만 만족하면 슈퍼키, 유일성 **+** 최소성을 모두 만족해야 후보키입니다. 슈퍼키가 더 넓은 개념이고, 후보키는 슈퍼키의 부분집합입니다.

### 주의사항 2: 복합키가 항상 후보키인 것은 아닙니다

두 속성을 합쳤을 때 유일성이 생겨도, 그 중 하나만으로도 유일성이 보장된다면 최소성 위반으로 후보키가 아닌 슈퍼키에만 해당됩니다.

### 주의사항 3: 기본키와 대체키의 NULL 허용 여부를 헷갈리지 마세요

- **기본키**: NULL **절대 불가**
- **대체키**: NULL **허용 가능**
- **외래키**: NULL **허용 가능** (참조할 대상이 아직 미정인 경우)

### 주의사항 4: 외래키는 다른 테이블의 기본키 값만 참조해야 합니다

외래키에 입력되는 값은 반드시 참조되는 테이블의 기본키에 실제로 존재하는 값이거나 NULL이어야 합니다. 존재하지 않는 값을 넣으면 참조 무결성 위반으로 오류가 발생합니다.

:::danger 절대 하면 안 되는 것
참조 무결성 설정 없이 외래키 관계를 맺는 것은 매우 위험합니다. 설정을 제대로 하지 않으면 삭제된 데이터를 참조하는 "고아 레코드(Orphan Record)"가 생겨 데이터 정합성이 깨질 수 있습니다.
:::

### 주의사항 5: 도메인은 현실 세계의 규칙을 반영해야 합니다

데이터베이스를 설계할 때 각 속성의 도메인을 현실에 맞게 설정하는 것이 중요합니다. "2월 30일", "점수 120점" 같은 현실에서 불가능한 값이 저장되지 않도록 CHECK 제약 조건을 활용하세요.

## 📋 핵심 정리

- **슈퍼키**: 유일성만 만족하면 되는 키. 범위가 가장 넓다.
- **후보키**: 유일성 + 최소성(비분해성)을 모두 만족하는 키. 슈퍼키의 부분집합.
- **유일성**: 릴레이션의 모든 튜플에서 키 값이 서로 다른 성질.
- **최소성(비분해성)**: 키의 구성 속성 중 하나라도 제거하면 유일성이 깨지는 성질.
- **기본키**: 후보키 중 설계자가 선택한 단 하나의 대표 키. NULL 절대 불가.
- **대체키**: 후보키 중 기본키로 선택되지 않은 나머지. NULL 허용 가능.
- **외래키**: 다른 테이블의 기본키를 참조하는 속성. NULL 허용. 참조 무결성을 지킴.
- **개체 무결성**: 기본키는 NULL이 될 수 없고 유일해야 한다는 규칙.
- **참조 무결성**: 외래키 값은 참조되는 기본키에 존재하거나 NULL이어야 한다는 규칙.
- **도메인 무결성**: 속성 값은 정의된 도메인(타입, 범위) 안에 속해야 한다는 규칙.
- **의미 무결성**: 비즈니스 규칙에 따른 의미적 제약 조건을 만족해야 한다는 규칙.
- **CASCADE**: 기본키 변경/삭제 시 외래키도 함께 변경/삭제.
- **SET NULL**: 기본키 변경/삭제 시 외래키를 NULL로 변경.
- **RESTRICT**: 외래키가 참조 중이면 기본키 변경/삭제를 거부.
- **SET DEFAULT**: 기본키 변경/삭제 시 외래키를 기본값으로 변경.
