# 🔑 유일성과 최소성(비분해성) — 키의 두 가지 조건 완전 정복

관계형 데이터베이스를 공부하다 보면 **"키(Key)"** 라는 개념이 자주 등장합니다. 그런데 키에는 슈퍼키, 후보키, 기본키, 대체키, 외래키처럼 여러 종류가 있어서 처음 접하는 분들은 혼란스러울 수 있습니다. 이 문서에서는 키를 제대로 이해하기 위한 가장 핵심적인 두 가지 개념, 바로 **유일성(Uniqueness)** 과 **최소성(Minimality, 비분해성)** 을 중심으로, 슈퍼키와 후보키의 차이, 기본키와 대체키, 그리고 외래키와 무결성 제약까지 단계적으로 설명합니다.

> 🎯 이 문서를 끝까지 읽으면, "왜 기본키가 중요한가?", "슈퍼키와 후보키는 어떻게 다른가?", "유일성과 최소성이 무엇인가?"를 명확하게 설명할 수 있게 됩니다.

## 📌 관계 데이터의 제약이란 무엇인가?

### 왜 데이터에 제약이 필요한가?

데이터베이스는 단순히 데이터를 저장하는 창고가 아닙니다. **"올바른 데이터"** 만 저장되도록 보호하는 역할도 합니다. 예를 들어, 학생 테이블에 같은 학번을 가진 학생이 두 명 존재한다면 어떨까요? 어떤 학생의 성적을 조회해야 할지 알 수 없게 됩니다. 이런 상황을 막기 위해 데이터베이스에는 **제약 조건(Constraint)** 이라는 규칙을 설정합니다.

관계 데이터의 제약 중 가장 기본이 되는 것은 바로 **키(Key)에 대한 제약** 입니다. 키는 테이블에서 각 행(튜플)을 유일하게 식별하는 역할을 합니다. 마치 주민등록번호가 대한민국에서 각 사람을 유일하게 구분하는 것처럼, 키는 데이터베이스 테이블에서 각 레코드를 유일하게 구분하는 식별자 역할을 합니다.

### 키 제약이 없으면 생기는 문제

키 제약이 없다면 다음과 같은 심각한 문제가 발생할 수 있습니다.

- **중복 데이터**: 동일한 학번을 가진 학생이 두 명 이상 등록될 수 있습니다.
- **데이터 불일치**: 어떤 데이터가 진짜인지 알 수 없어 신뢰성이 무너집니다.
- **참조 오류**: A 테이블이 B 테이블의 없는 값을 참조하면 시스템이 혼란에 빠집니다.

이런 문제들을 막기 위해 데이터베이스는 **무결성(Integrity)** 이라는 개념을 도입했고, 그 무결성을 지키는 핵심 도구가 바로 **키** 입니다.

:::info 무결성(Integrity)이란?
무결성이란 데이터가 "오염되지 않고, 정확하며, 일관성 있게 유지되는 성질"을 말합니다. 마치 음식이 상하지 않도록 냉장고에 보관하는 것처럼, 데이터베이스는 무결성 제약을 통해 데이터가 잘못되지 않도록 보호합니다.
:::

## 🗝️ 키(Key)의 종류와 계층 구조

### 키의 종류 한눈에 보기

키를 처음 공부할 때 가장 헷갈리는 부분은 종류가 너무 많다는 것입니다. 전체 구조를 먼저 머릿속에 그려 놓으면 이해가 훨씬 쉬워집니다. 아래 표를 먼저 살펴보고 각각을 하나씩 이해해 봅시다.

| 키 종류 | 유일성 | 최소성(비분해성) | NULL 허용 | 개수 제한 |
|---|---|---|---|---|
| **슈퍼키(Super Key)** | ✅ 만족 | ❌ 불필요 | - | 제한 없음 |
| **후보키(Candidate Key)** | ✅ 만족 | ✅ 만족 | - | 제한 없음 |
| **기본키(Primary Key)** | ✅ 만족 | ✅ 만족 | ❌ 불가 | 테이블당 1개 |
| **대체키(Alternate Key)** | ✅ 만족 | ✅ 만족 | ✅ 허용 | 제한 없음 |
| **외래키(Foreign Key)** | - | - | ✅ 허용 | 제한 없음 |

### 키 계층 구조 이해하기

키의 관계를 집합으로 표현하면 다음과 같습니다.

```
슈퍼키 (가장 큰 범위)
  └── 후보키 (슈퍼키의 부분집합)
        ├── 기본키 (후보키 중 딱 1개 선택)
        └── 대체키 (기본키로 선택되지 않은 나머지 후보키들)
```

> 💡 **핵심 포인트**: 모든 후보키는 슈퍼키이지만, 모든 슈퍼키가 후보키인 것은 아닙니다. 후보키가 되려면 슈퍼키 조건에 **최소성(비분해성)** 이라는 추가 조건을 만족해야 합니다.

이 계층 구조를 이해하는 핵심 열쇠가 바로 오늘 학습할 **유일성** 과 **최소성(비분해성)** 입니다.

## 🔍 유일성(Uniqueness)과 최소성(Minimality) 핵심 개념

### 유일성(Uniqueness)이란?

유일성은 말 그대로 **"모든 튜플(행)에서 키 값이 서로 달라야 한다"** 는 조건입니다. 마치 각 사람의 주민등록번호가 절대 겹치지 않는 것처럼, 키로 사용되는 속성(열)의 값은 테이블의 모든 행에서 서로 달라야 합니다.

예를 들어, 아래와 같은 학생 테이블이 있다고 가정해 봅시다.

| 학번 | 이름 | 전공 |
|---|---|---|
| 2024001 | 김철수 | 컴퓨터공학 |
| 2024002 | 이영희 | 경영학 |
| 2024003 | 박민준 | 컴퓨터공학 |

이 경우, **학번** 은 2024001, 2024002, 2024003으로 모두 다르므로 **유일성을 만족** 합니다. 반면 **전공** 은 "컴퓨터공학"이 두 번 등장하므로 **유일성을 만족하지 않습니다.**

> 📌 **정의**: 릴레이션(테이블)에 있는 모든 튜플(행)에 대해 키 값이 모두 다르면, 해당 속성(또는 속성 집합)은 **유일성을 만족** 한다고 합니다.

### 최소성(Minimality) — 비분해성이라고도 합니다

최소성은 유일성보다 조금 더 이해가 필요한 개념입니다. 최소성은 다른 말로 **비분해성(Irreducibility)** 이라고도 부릅니다.

**최소성의 정의**: 키를 구성하는 속성들의 집합에서 **어떤 속성 하나라도 제거하면 유일성이 깨지는 성질** 을 말합니다.

쉽게 말하면, **"키를 이루는 모든 속성이 꼭 필요하다"** 는 것입니다. 하나라도 빼면 더 이상 키 역할을 못 한다면 → 최소성 만족! 하나를 빼도 여전히 키 역할을 한다면 → 최소성 불만족!

마치 자물쇠를 열기 위한 비밀번호가 "1234"인데, "123"만으로도 문이 열린다면 "4"는 불필요한 숫자입니다. 반면 "1234" 중 어느 하나를 빼도 문이 열리지 않는다면, 이 네 자리 모두가 꼭 필요한 것이죠. 이것이 최소성입니다.

:::tip 비분해성(非分解性)이란?
"분해(分解)"는 나누는 것을 의미합니다. "비분해성"은 "나눌 수 없는 성질"이라는 뜻입니다. 즉, 키를 구성하는 속성들을 분해(하나씩 제거)했을 때, 키의 기능(유일성)이 사라진다면 — 이 키는 비분해성, 즉 최소성을 가진다고 합니다.
:::

## 🧪 슈퍼키 vs 후보키 — 예제로 완전히 이해하기

### 예제 1: AB 복합키가 후보키가 되는 경우

다음과 같은 릴레이션 R이 있다고 가정합니다.

| A | B |
|---|---|
| A1 | B1 |
| A1 | B2 |
| A2 | B1 |

**Step 1: A 단독으로 키가 될 수 있는가?**

A 열을 보면 A1, A1, A2 — A1이 두 번 등장합니다. 중복이 있으므로 유일성을 만족하지 않습니다. ❌ A는 키가 될 수 없습니다.

**Step 2: B 단독으로 키가 될 수 있는가?**

B 열을 보면 B1, B2, B1 — B1이 두 번 등장합니다. 중복이 있으므로 유일성을 만족하지 않습니다. ❌ B는 키가 될 수 없습니다.

**Step 3: AB 복합키로 키가 될 수 있는가?**

(A1, B1), (A1, B2), (A2, B1) — 세 가지 조합이 모두 다릅니다. ✅ AB는 유일성을 만족합니다!

**Step 4: AB는 슈퍼키인가?**

유일성을 만족하므로 ✅ 슈퍼키 조건 만족!

**Step 5: AB는 후보키인가? (최소성 검사)**

후보키가 되려면 최소성도 만족해야 합니다. AB에서 하나씩 빼봅니다.

- A를 제거 → B만 남음 → B는 키가 될 수 없음 (B1 중복) ❌
- B를 제거 → A만 남음 → A는 키가 될 수 없음 (A1 중복) ❌

A를 빼도 안 되고, B를 빼도 안 됩니다. **A와 B 모두 꼭 필요합니다!** 이것이 최소성(비분해성) 만족입니다. ✅

**결론: AB는 슈퍼키이면서 동시에 후보키입니다.**

```
[정리]
- AB → 유일성 ✅, 최소성 ✅ → 슈퍼키 ✅, 후보키 ✅
- A → 유일성 ❌ → 키 불가
- B → 유일성 ❌ → 키 불가
```

### 예제 2: AB 복합키가 슈퍼키만 되는 경우

이번엔 다른 릴레이션 R을 봅니다.

| A | B |
|---|---|
| A1 | B1 |
| A2 | B2 |
| A1 | B3 |

**Step 1: A 단독 키 가능 여부**

A1, A2, A1 → A1이 중복. ❌ 키 불가.

**Step 2: B 단독 키 가능 여부**

B1, B2, B3 → 모두 다릅니다! ✅ B는 유일성 만족. B는 키가 될 수 있습니다.

**Step 3: AB 복합키 가능 여부**

(A1, B1), (A2, B2), (A1, B3) → 모두 다릅니다. ✅ AB도 유일성 만족.

**Step 4: AB는 슈퍼키인가?**

유일성 만족 → ✅ 슈퍼키 조건 만족!

**Step 5: AB는 후보키인가? (최소성 검사)**

- A를 제거 → B만 남음 → B1, B2, B3 모두 다름 → **B 혼자서도 키가 됩니다!** ✅

B 혼자서 키가 될 수 있다는 것은, AB에서 A가 불필요하다는 의미입니다. 즉, A는 있어도 되고 없어도 된다는 뜻이죠. 이 경우 **최소성(비분해성)을 만족하지 않습니다.** ❌

**결론: AB는 슈퍼키는 되지만 후보키는 될 수 없습니다.**

:::warning 최소성 판단 주의사항
최소성 검사 시, 집합을 구성하는 속성 중 **단 하나라도** 제거했을 때 유일성이 유지된다면 → 최소성 불만족입니다. 즉, "모든 속성이 다 빠져도 키가 된다"가 아니라, "**어느 하나라도** 빼면 키가 된다"면 최소성을 만족하지 못하는 것입니다.
:::

### 두 예제 비교 정리

| 구분 | 예제 1 (A1,B1 / A1,B2 / A2,B1) | 예제 2 (A1,B1 / A2,B2 / A1,B3) |
|---|---|---|
| A 단독 키 가능? | ❌ (A1 중복) | ❌ (A1 중복) |
| B 단독 키 가능? | ❌ (B1 중복) | ✅ (B1,B2,B3 유일) |
| AB 유일성? | ✅ | ✅ |
| AB 최소성? | ✅ (A빼도, B빼도 키 불가) | ❌ (A 빼면 B가 키 됨) |
| AB → 슈퍼키? | ✅ | ✅ |
| AB → 후보키? | ✅ | ❌ |

## 🎓 실제 데이터로 배우는 키 찾기

### 학생 테이블 예시

실제 학생 테이블을 가지고 키를 찾아보겠습니다.

| 학번 | 이름 | 주소 | 주민번호 | 전공 |
|---|---|---|---|---|
| 2024001 | 김철수 | 서울 강남구 | 040101-1234567 | 컴퓨터공학 |
| 2024002 | 이영희 | 부산 해운대구 | 980303-2345678 | 경영학 |
| 2024003 | 박민준 | 서울 강남구 | 010505-1357924 | 컴퓨터공학 |

이 테이블에서 어떤 속성(또는 속성 조합)이 키가 될 수 있는지 분석해 봅시다.

**각 속성의 유일성 분석:**

- **학번**: 2024001, 2024002, 2024003 → 모두 다름 ✅ 유일성 만족
- **이름**: 현재 데이터는 다르지만, 동명이인이 존재할 수 있음 ❌ 유일성 미보장
- **주소**: 서울 강남구가 두 번 등장 ❌ 유일성 불만족
- **주민번호**: 각자 다른 고유값 ✅ 유일성 만족
- **전공**: 컴퓨터공학이 두 번 등장 ❌ 유일성 불만족

:::note 이름은 왜 키가 될 수 없나요?
현재 데이터에서는 이름이 모두 다르게 보이지만, 데이터베이스는 **현재 데이터만** 이 아니라 **앞으로 입력될 모든 데이터** 를 고려합니다. 현실 세계에서 같은 이름을 가진 사람이 얼마든지 존재할 수 있으므로, 이름은 유일성을 보장할 수 없어 키로 사용할 수 없습니다.
:::

### 슈퍼키 찾기

유일성만 만족하면 슈퍼키가 됩니다. 학번이 유일하므로, 학번을 포함하는 모든 조합은 슈퍼키입니다. 주민번호도 유일하므로, 주민번호를 포함하는 모든 조합도 슈퍼키입니다.

슈퍼키 목록 (일부):
- 학번
- 주민번호
- 학번 + 이름
- 학번 + 주소
- 학번 + 전공
- 학번 + 주민번호
- 주민번호 + 이름
- 주민번호 + 전공
- 학번 + 이름 + 주소
- 학번 + 이름 + 주소 + 주민번호 + 전공 (모든 속성 포함)
- ... (수십 가지 조합)

### 후보키 찾기 — 최소성 검사

슈퍼키 중에서 최소성(비분해성)까지 만족하는 것이 후보키입니다.

**학번 단독:**
- 더 이상 제거할 속성이 없습니다 (단일 속성). 최소성 ✅
- → **후보키 ✅**

**주민번호 단독:**
- 더 이상 제거할 속성이 없습니다 (단일 속성). 최소성 ✅
- → **후보키 ✅**

**학번 + 이름:**
- 학번을 제거하면 이름만 남는데, 이름은 유일성을 보장할 수 없어 키가 안 됩니다.
- 이름을 제거하면 학번만 남는데, **학번 혼자서 키가 됩니다!**
- → 이름이 불필요하므로 최소성 ❌ → **슈퍼키는 되지만 후보키는 안 됨**

**학번 + 주민번호:**
- 학번을 제거하면 주민번호만 남는데, 주민번호 혼자 키가 됩니다!
- → 최소성 ❌ → **슈퍼키는 되지만 후보키는 안 됨**

**주민번호 + 전공:**
- 전공을 제거하면 주민번호만 남는데, 주민번호 혼자 키가 됩니다!
- → 최소성 ❌ → **슈퍼키는 되지만 후보키는 안 됨**

> 🏆 **결론**: 이 학생 테이블에서 후보키는 **학번** 과 **주민번호** 두 가지입니다. 나머지 복합 조합들은 모두 슈퍼키는 되지만 최소성을 만족하지 못해 후보키가 되지 못합니다.

### 기본키와 대체키 결정

후보키가 2개(학번, 주민번호)이므로, 데이터베이스 설계자가 이 중 하나를 **기본키(Primary Key)** 로 선택합니다. 보통 학생 테이블에서는 **학번** 을 기본키로 선택합니다.

- **기본키**: 학번 (설계자가 선택한 1개)
- **대체키(Alternate Key)**: 주민번호 (기본키로 선택되지 않은 나머지 후보키)

:::info 기본키 선택 기준
기본키는 후보키 중에서 설계자가 선택하는 것이므로 어떤 것을 고르든 이론적으로는 문제없습니다. 다만 실무에서는 보통 변경 가능성이 낮고, 의미가 명확하고, 짧고 단순한 것을 기본키로 선택합니다. 주민번호보다 학번이 더 짧고, 개인정보 보호 측면에서도 유리합니다.
:::

## 🔗 외래키(Foreign Key)와 참조 무결성

### 외래키란 무엇인가?

외래키(Foreign Key)는 다른 테이블의 기본키를 참조하는 속성(열)입니다. 마치 도서관에서 책을 대출할 때 "회원번호"를 적는 것처럼, 외래키는 다른 테이블과의 관계를 연결하는 다리 역할을 합니다.

예를 들어, 교수 테이블과 학과 테이블이 있다고 합시다.

**교수 테이블 (R1)**

| 교수번호 | 교수명 | 학과번호 |
|---|---|---|
| P001 | 홍길동 | D001 |
| P002 | 이순신 | D002 |
| P003 | 강감찬 | NULL |

**학과 테이블 (R2)**

| 학과번호 | 학과명 | 학과장교수번호 |
|---|---|---|
| D001 | 컴퓨터공학과 | P001 |
| D002 | 경영학과 | P002 |

여기서 교수 테이블의 **학과번호** 는 학과 테이블의 기본키(학과번호)를 참조합니다. 이때 교수 테이블의 학과번호가 **외래키** 가 됩니다.

- **R1(교수 테이블)**: 참조하는 릴레이션 (Referencing Relation)
- **R2(학과 테이블)**: 참조되는 릴레이션 (Referenced Relation)

### 외래키의 중요한 규칙

**규칙 1: 외래키 값은 참조되는 테이블의 기본키 값이어야 합니다.**

교수 테이블의 학과번호에 D003을 입력하려고 했는데, 학과 테이블에 D003이 존재하지 않는다면 입력이 거부됩니다. 이것이 **참조 무결성(Referential Integrity)** 입니다.

**규칙 2: 외래키는 NULL 값을 가질 수 있습니다.**

P003(강감찬 교수)의 학과번호가 NULL인 것을 보면, 아직 어느 학과에 배정할지 결정이 안 된 상태입니다. 이렇게 **"아직 참조할 값을 결정하지 못했을 때"** 외래키에 NULL을 허용합니다. 이 점이 기본키(NULL 절대 불가)와 다른 중요한 차이점입니다.

**규칙 3: 외래키와 참조되는 기본키는 같은 도메인이어야 합니다.**

학과번호가 문자형(VARCHAR)이라면 외래키인 학과번호도 문자형이어야 합니다. 숫자 10과 문자 "10"은 컴퓨터 내부에서 다른 값으로 처리되기 때문입니다.

```sql
-- ✅ 올바른 외래키 설정 예시
CREATE TABLE 교수 (
    교수번호 VARCHAR(10) PRIMARY KEY,  -- 기본키 설정
    교수명 VARCHAR(50) NOT NULL,
    학과번호 VARCHAR(10),              -- 외래키가 될 컬럼
    FOREIGN KEY (학과번호) REFERENCES 학과(학과번호)  -- 외래키 선언
        ON DELETE SET NULL             -- 학과가 삭제되면 NULL로 변경
        ON UPDATE CASCADE              -- 학과번호가 변경되면 같이 변경
);

-- ❌ 잘못된 예시: 도메인이 다른 외래키
CREATE TABLE 교수 (
    교수번호 VARCHAR(10) PRIMARY KEY,
    학과번호 INT,  -- 학과 테이블의 학과번호가 VARCHAR인데 INT로 선언 → 오류!
    FOREIGN KEY (학과번호) REFERENCES 학과(학과번호)
);
```

### 자기 참조(Self-Referencing)

외래키는 반드시 다른 테이블을 참조할 필요는 없습니다. 같은 테이블 내에서 자기 자신을 참조하는 것도 가능하며, 이를 **자기 참조(Self-Referencing)** 라고 합니다.

예를 들어, 직원 테이블에서 상사-부하 관계를 표현할 때 유용합니다.

```sql
-- ✅ 자기 참조(Self-Referencing) 예시
CREATE TABLE 직원 (
    직원번호 VARCHAR(10) PRIMARY KEY,  -- 직원의 고유 번호 (기본키)
    직원명 VARCHAR(50) NOT NULL,       -- 직원 이름
    상사번호 VARCHAR(10),              -- 자기 테이블의 직원번호를 참조 (외래키)
    FOREIGN KEY (상사번호) REFERENCES 직원(직원번호)  -- 자기 참조!
);

-- 데이터 예시
-- P001 홍길동 (상사번호 NULL: 최고 직책, 상사 없음)
-- P002 이영희 (상사번호 P001: 홍길동이 이영희의 상사)
-- P003 박민준 (상사번호 P001: 홍길동이 박민준의 상사)
```

## ⚙️ 무결성 제약의 종류와 참조 무결성 옵션

### 무결성의 네 가지 종류

데이터베이스에서 지켜야 하는 무결성은 크게 네 가지로 나뉩니다.

**1. 개체 무결성(Entity Integrity)**

기본키에 관한 제약입니다. 기본키는 절대 NULL 값을 가질 수 없고, 반드시 유일한 값을 가져야 합니다. 사람으로 치면 주민등록번호가 반드시 있어야 하고, 두 사람이 같은 번호를 가질 수 없는 것과 같습니다.

| 속성 | 기본키 | 대체키 |
|---|---|---|
| 유일성 | ✅ 필수 | ✅ 필수 |
| NULL 허용 | ❌ 절대 불가 | ✅ 허용 |

**2. 참조 무결성(Referential Integrity)**

외래키에 관한 제약입니다. 외래키 값은 반드시 참조되는 테이블의 기본키 값과 일치하거나 NULL이어야 합니다. 마치 도서관 대출 기록에 존재하지 않는 회원번호가 들어갈 수 없는 것과 같습니다.

**3. 도메인 무결성(Domain Integrity)**

각 속성이 가질 수 있는 값의 범위(도메인)에 관한 제약입니다. 예를 들어, 점수 속성은 0~100 사이의 값만 허용하고 120이 입력되면 자동으로 거부됩니다. 날짜 속성에 "2024-02-30"처럼 존재하지 않는 날짜가 입력되면 거부됩니다. 근무 상태가 1(정상), 2(휴직), 3(퇴직) 중 하나여야 한다면 4가 입력될 수 없습니다.

```sql
-- ✅ 도메인 무결성 예시
CREATE TABLE 성적 (
    학번 VARCHAR(10) NOT NULL,                    -- NULL 불가 제약
    과목명 VARCHAR(50) NOT NULL,
    점수 INT CHECK (점수 >= 0 AND 점수 <= 100),   -- 0~100 범위 제약
    근무상태 INT CHECK (근무상태 IN (1, 2, 3)),   -- 1,2,3만 허용
    주문일 DATE NOT NULL                          -- 날짜 형식 제약
);

-- ❌ 잘못된 데이터 입력 예시 (아래는 모두 에러 발생)
-- INSERT INTO 성적 VALUES ('2024001', '데이터베이스', 120, 1, '2024-02-30');
-- 점수 120은 범위 초과, 2024-02-30은 존재하지 않는 날짜
```

**4. 의미 무결성(Semantic Integrity)**

데이터들 사이의 논리적 관계에 관한 제약입니다. 예를 들어, 쇼핑몰에서 배송 수량이 주문 수량을 초과할 수 없다는 규칙, 또는 직원의 올해 연봉이 작년 연봉보다 낮을 수 없다는 규칙처럼, 업무 규칙(Business Rule)을 데이터베이스 수준에서 강제합니다.

```sql
-- ✅ 의미 무결성 예시 (CHECK 제약으로 업무 규칙 강제)
CREATE TABLE 주문 (
    주문번호 INT PRIMARY KEY,
    주문수량 INT NOT NULL,
    배송수량 INT DEFAULT 0,
    -- 배송 수량은 주문 수량을 초과할 수 없음
    CHECK (배송수량 <= 주문수량)
);
```

### 참조 무결성 위반 시 처리 옵션 4가지

참조되는 테이블(R2)의 기본키 값이 변경되거나 삭제될 때, 이를 참조하던 외래키(R1)를 어떻게 처리할지 결정하는 옵션이 4가지 있습니다.

아래 상황을 가정하겠습니다.

- R1(교수 테이블): 학과번호 D001을 외래키로 가지고 있음
- R2(학과 테이블): 학과번호 D001이 기본키

이때 R2에서 D001을 삭제하려 하면 어떻게 될까요?

| 옵션 | SQL 키워드 | 동작 설명 |
|---|---|---|
| **연쇄 삭제** | `CASCADE` | R2의 D001이 삭제되면 R1에서 D001을 참조하는 행도 함께 삭제 |
| **NULL 설정** | `SET NULL` (Nullified) | R2의 D001이 삭제되면 R1의 학과번호를 NULL로 변경 |
| **삭제 거부** | `RESTRICT` / `NO ACTION` | R1이 D001을 참조하고 있으므로 R2의 삭제 자체를 거부 |
| **기본값 설정** | `SET DEFAULT` | R2의 D001이 삭제되면 R1의 학과번호를 미리 약속된 기본값으로 변경 |

```sql
-- ✅ CASCADE 옵션 예시
-- 학과가 삭제되면 그 학과의 교수 데이터도 함께 삭제됨
CREATE TABLE 교수 (
    교수번호 VARCHAR(10) PRIMARY KEY,
    교수명 VARCHAR(50) NOT NULL,
    학과번호 VARCHAR(10),
    FOREIGN KEY (학과번호) REFERENCES 학과(학과번호)
        ON DELETE CASCADE    -- 학과 삭제 시 교수도 같이 삭제
        ON UPDATE CASCADE    -- 학과번호 변경 시 교수의 학과번호도 같이 변경
);

-- ✅ SET NULL 옵션 예시
-- 학과가 삭제되면 교수의 학과번호를 NULL로 변경 (아직 미정 상태)
CREATE TABLE 교수 (
    교수번호 VARCHAR(10) PRIMARY KEY,
    교수명 VARCHAR(50) NOT NULL,
    학과번호 VARCHAR(10),
    FOREIGN KEY (학과번호) REFERENCES 학과(학과번호)
        ON DELETE SET NULL   -- 학과 삭제 시 교수의 학과번호를 NULL로
);

-- ✅ RESTRICT 옵션 예시
-- 교수가 소속된 학과는 삭제할 수 없음 (안전 최우선)
CREATE TABLE 교수 (
    교수번호 VARCHAR(10) PRIMARY KEY,
    교수명 VARCHAR(50) NOT NULL,
    학과번호 VARCHAR(10),
    FOREIGN KEY (학과번호) REFERENCES 학과(학과번호)
        ON DELETE RESTRICT   -- 참조하는 데이터가 있으면 삭제 거부
);
```

:::danger CASCADE 옵션 사용 시 주의
`CASCADE` 옵션은 편리하지만 위험할 수 있습니다. 상위 테이블에서 하나의 행을 삭제했을 때, 이를 참조하는 하위 테이블의 수많은 행이 연쇄적으로 삭제될 수 있습니다. 실수로 중요한 데이터가 대규모로 삭제되는 사고가 발생할 수 있으므로, 업무 요구사항을 충분히 검토한 후 신중하게 사용해야 합니다.
:::

### SQL에서 기본키, 대체키, 외래키 생성하기

```sql
-- ✅ 학생 테이블 생성: 기본키, 대체키, 외래키 모두 포함
CREATE TABLE 학생 (
    학번 VARCHAR(10) NOT NULL,          -- 학번: 기본키 컬럼 (NULL 불가)
    이름 VARCHAR(50) NOT NULL,           -- 이름: 반드시 입력해야 함
    주소 VARCHAR(100),                   -- 주소: NULL 허용
    주민번호 CHAR(14) UNIQUE,           -- 주민번호: 대체키 (UNIQUE = 유일값, NULL 허용)
    전공코드 VARCHAR(10),               -- 전공코드: 외래키가 될 컬럼
    PRIMARY KEY (학번),                  -- 기본키 선언
    FOREIGN KEY (전공코드) REFERENCES 전공(전공코드)  -- 외래키 선언
        ON DELETE SET NULL               -- 전공 삭제 시 NULL로 변경
        ON UPDATE CASCADE                -- 전공코드 변경 시 함께 변경
);

-- 코드 설명:
-- PRIMARY KEY (학번): 학번을 기본키로 지정. NULL 불가, 중복 불가.
-- UNIQUE (주민번호): 주민번호를 대체키로 지정. 중복 불가, NULL은 허용.
-- FOREIGN KEY: 외래키 선언. REFERENCES는 참조할 테이블과 컬럼을 지정.
-- ON DELETE SET NULL: 참조되는 행이 삭제되면 외래키 값을 NULL로.
-- ON UPDATE CASCADE: 참조되는 키 값이 변경되면 외래키 값도 같이 변경.
```

## ⚠️ 주의사항과 실수하기 쉬운 포인트

### 주의사항 1: 슈퍼키와 후보키 혼동 금지

:::warning 슈퍼키 ≠ 후보키
가장 많이 하는 실수는 "유일하면 후보키"라고 생각하는 것입니다. 유일성만 만족하면 슈퍼키이고, 유일성 + 최소성을 모두 만족해야 후보키입니다. "학번 + 이름"은 유일하므로 슈퍼키이지만, 학번만으로도 유일하므로 이름이 불필요하여 최소성을 만족하지 못해 후보키가 아닙니다.
:::

### 주의사항 2: 기본키와 대체키의 NULL 처리 차이

기본키는 **절대 NULL 불가** 이지만, 대체키(후보키 중 기본키로 선택되지 않은 것)는 **NULL을 허용** 합니다. 이는 시험에서도 자주 출제되는 중요한 차이점입니다.

```sql
-- ✅ 기본키: NULL 절대 불가
학번 VARCHAR(10) NOT NULL PRIMARY KEY,  -- NOT NULL이 기본으로 적용됨

-- ✅ 대체키: NULL 허용 가능
주민번호 CHAR(14) UNIQUE,  -- UNIQUE만 지정, NULL 허용 (아직 모르는 경우)
```

### 주의사항 3: 외래키 NULL의 올바른 이해

외래키에 NULL이 있다는 것은 "잘못된 데이터"가 아닙니다. 이것은 "**아직 참조할 값이 결정되지 않은 상태**"를 의미합니다. 예를 들어 새로 채용된 교수가 어느 학과에 배정될지 아직 결정되지 않았을 때, 학과번호 외래키에 NULL을 넣는 것은 정상적인 상황입니다.

:::tip 최소성 검사 방법 — 간단한 요령
복합키의 최소성을 검사할 때는 다음 순서를 따르세요.
1. 복합키에서 속성을 하나씩 제거해 봅니다.
2. 제거한 후에도 나머지 속성만으로 유일성이 유지된다면 → 최소성 불만족 (그 속성은 불필요)
3. 어떤 속성을 제거해도 유일성이 깨진다면 → 최소성 만족 (모든 속성이 꼭 필요)
:::

### 주의사항 4: 무결성은 시스템이 자동으로 관리

참조 무결성, 도메인 무결성 등의 제약 조건은 **설정만 해두면 시스템이 자동으로 검사하고 관리** 합니다. 개발자나 사용자가 매번 직접 확인할 필요가 없습니다. 어떤 연산(입력, 수정, 삭제)이 발생하면 DBMS(데이터베이스 관리 시스템)가 자동으로 제약 조건을 검사하여, 만족하면 처리하고 만족하지 않으면 거부합니다. 마치 은행 ATM이 잔액 부족 시 출금을 자동으로 거부하는 것과 같습니다.

## 📚 핵심 정리

이 문서에서 학습한 핵심 개념들을 정리합니다.

- **유일성(Uniqueness)**: 릴레이션의 모든 튜플에서 키 값이 서로 달라야 하는 성질
- **최소성(Minimality) = 비분해성**: 키를 구성하는 속성 중 하나라도 제거하면 유일성이 깨지는 성질 (모든 속성이 꼭 필요)
- **슈퍼키(Super Key)**: 유일성만 만족하면 되는 키 (최소성 불필요)
- **후보키(Candidate Key)**: 유일성 + 최소성 모두 만족하는 키
- **기본키(Primary Key)**: 후보키 중 설계자가 선택한 단 하나의 키. NULL 절대 불가
- **대체키(Alternate Key)**: 후보키 중 기본키로 선택되지 않은 나머지 키. NULL 허용
- **외래키(Foreign Key)**: 다른 테이블의 기본키를 참조하는 속성. NULL 허용
- **계층 관계**: 슈퍼키 ⊃ 후보키 ⊃ 기본키(or 대체키)
- **개체 무결성**: 기본키는 NULL 불가, 유일성 필수
- **참조 무결성**: 외래키 값은 참조되는 기본키 값과 일치하거나 NULL이어야 함
- **도메인 무결성**: 속성 값은 정해진 도메인(범위, 타입, 형식) 안에 있어야 함
- **의미 무결성**: 업무 규칙에 따른 데이터 간 논리적 일관성 유지
- **참조 무결성 옵션**: CASCADE(연쇄), SET NULL(NULL 변경), RESTRICT(거부), SET DEFAULT(기본값)
- **자기 참조(Self-Referencing)**: 같은 테이블 내에서 기본키를 외래키로 참조하는 것도 가능
- **무결성 자동 관리**: 제약 조건 설정 후 DBMS가 자동으로 검사 및 처리
