# 🔗 조인 연산자의 종류 완전 정복: 세타 조인, 동일 조인, 자연 조인

데이터베이스를 배우다 보면 반드시 마주치는 개념이 있습니다. 바로 **조인(Join)** 입니다. 조인은 단순히 이론적인 개념에 머무르지 않고, 실무에서 가장 자주 사용하는 핵심 연산입니다. 이번 문서에서는 관계대수(Relational Algebra)의 확장 개념을 바탕으로, 세타 조인 → 동일 조인 → 자연 조인으로 이어지는 조인 연산자의 흐름을 처음부터 차근차근 이해해 봅니다. 나아가 세미 조인과 외부 조인까지 실제 데이터 예시와 함께 살펴보겠습니다.

:::info 이 문서를 읽기 전에 알아두면 좋은 것
이 문서는 관계대수의 기본 개념(셀렉션, 프로젝션, 합집합, 교집합, 차집합, 카티션 프로덕트 등)을 이미 배운 분들을 대상으로 합니다. 조인 연산자는 **순수관계 연산자** 중 하나이며, 실무에서 가장 널리 쓰이는 연산입니다.
:::

## 📚 관계대수와 조인 연산자: 기본 개념 소개

### 관계대수란 무엇인가요?

**관계대수(Relational Algebra)** 란 데이터베이스에서 원하는 데이터를 꺼내오기 위해 사용하는 수학적인 연산 체계입니다. 마치 요리사가 레시피에 따라 순서대로 재료를 손질하고 조리하듯이, 관계대수는 **어떤 순서로 어떤 연산을 수행할 것인지**를 명확히 정의합니다. 이처럼 연산의 순서가 중요하다는 점에서 관계대수는 **절차적(Procedural)** 언어라고 부릅니다.

관계대수의 연산자는 크게 두 가지로 나뉩니다.

| 분류 | 연산자 종류 |
|---|---|
| **일반 집합 연산자** | 합집합(∪), 교집합(∩), 차집합(−), 카티션 프로덕트(×) |
| **순수관계 연산자** | 셀렉션(σ), 프로젝션(π), 조인(⋈), 디비전(÷) |

이 중에서 오늘 우리가 집중적으로 살펴볼 것은 **순수관계 연산자 중 조인(Join)** 입니다.

### 조인이란 무엇인가요?

**조인(Join)** 은 두 개 이상의 테이블(릴레이션)을 특정 조건에 따라 연결하여 하나의 결과로 만들어내는 연산입니다. 마치 두 개의 명단을 공통된 기준(예: 학번, 사번)으로 합쳐서 더 풍부한 정보를 만들어내는 것과 같습니다.

예를 들어, **학생 테이블**에는 학번, 이름, 학과가 있고, **성적 테이블**에는 학번, 과목명, 점수가 있다고 가정해 봅시다. 이 두 테이블을 학번을 기준으로 연결하면 "어떤 학생이 어떤 과목에서 몇 점을 받았는지"를 한눈에 볼 수 있는 결과를 만들 수 있습니다. 이것이 바로 조인의 핵심 역할입니다.

> 💡 조인은 실무 데이터베이스에서 가장 빈번하게 사용되는 연산입니다. 조인을 이해하지 못하면 실제 SQL 쿼리를 작성하기 매우 어렵습니다.

## 🤔 왜 조인의 종류가 여러 개인가요? — 배경과 필요성

### 단순한 조인의 한계

처음에는 "그냥 두 테이블을 연결하면 되는 거 아닌가요?"라고 생각할 수 있습니다. 하지만 현실에서는 다양한 상황이 존재합니다.

- 두 테이블을 **정확히 같은 값**으로 연결하고 싶은 경우
- 한 테이블의 값이 다른 테이블의 값보다 **크거나 작은** 조건으로 연결하고 싶은 경우
- 연결 후 **중복 속성을 제거**하고 싶은 경우
- 조인에 **참여하지 못한 행(row)도 결과에 포함**시키고 싶은 경우

이처럼 각기 다른 요구사항을 충족시키기 위해 여러 종류의 조인이 발전했습니다. 조인의 종류를 이해하면, 상황에 맞는 적절한 연산을 선택할 수 있게 됩니다.

### 조인 종류의 전체 구조

이 문서에서 다루는 조인의 종류를 먼저 큰 그림으로 살펴봅시다.

```
조인(Join)
├── 세타 조인 (Theta Join): 비교 연산자(>, <, >=, <=, =, ≠) 모두 사용 가능
│   └── 동일 조인 (Equi Join): 세타 조인 중 '=' 조건만 사용, 속성 중복 발생
│       └── 자연 조인 (Natural Join): 동일 조인에서 중복 속성 제거
├── 세미 조인 (Semi Join): 자연 조인 후 한쪽 릴레이션의 속성만 출력
└── 외부 조인 (Outer Join): 조인에 참여 못한 튜플도 NULL로 포함
    ├── 좌측 외부 조인 (Left Outer Join)
    ├── 우측 외부 조인 (Right Outer Join)
    └── 완전 외부 조인 (Full Outer Join)
```

## 🔍 핵심 원리: 세타 조인, 동일 조인, 자연 조인

### 세타 조인 (Theta Join) — 가장 넓은 개념의 조인

**세타 조인(θ-Join)** 은 조인의 가장 일반적인 형태입니다. 두 릴레이션을 연결할 때 사용하는 비교 조건(θ, 세타)으로 다음과 같은 연산자를 **모두** 사용할 수 있습니다.

| 비교 연산자 | 의미 |
|---|---|
| `=` | 같다 |
| `≠` | 같지 않다 |
| `>` | 크다 |
| `<` | 작다 |
| `>=` | 크거나 같다 |
| `<=` | 작거나 같다 |

마치 수학 시간에 배운 비교 연산자 전부를 조인 조건으로 사용할 수 있는 것이 세타 조인입니다. 예를 들어 "A 테이블의 나이 값이 B 테이블의 나이 값보다 큰 경우만 연결해라"라는 조건도 세타 조인으로 표현할 수 있습니다.

:::note 세타(θ)란?
세타(θ)는 그리스 문자로, 수학에서 **임의의 비교 연산자**를 통칭할 때 사용합니다. 즉, 세타 조인은 "어떤 비교 조건이든 사용할 수 있는 조인"이라는 의미입니다.
:::

### 동일 조인 (Equi Join) — 세타 조인의 특별한 경우

**동일 조인(Equi Join)** 은 세타 조인 중에서 **조인 조건이 반드시 등호(`=`)인 경우**를 말합니다. 두 테이블에서 같은 값을 가진 행끼리만 연결합니다. 마치 두 반 학생 명단에서 이름이 같은 학생끼리만 짝을 지어주는 것과 같습니다.

동일 조인의 핵심적인 특징이자 문제점이 하나 있습니다. 바로 **조인에 사용된 속성(컬럼)이 결과에 중복으로 나타난다**는 점입니다.

예를 들어, R 테이블과 S 테이블을 모두 `C` 속성의 값이 같은 조건으로 조인하면, 결과 테이블에 R의 `C` 열과 S의 `C` 열이 **두 번** 나타납니다. 두 열은 똑같은 값을 가지고 있으므로 하나는 불필요한 중복입니다.

```
동일 조인 결과 예시:
A  | R.C | S.C | D      ← R.C와 S.C 값이 항상 동일 → 중복!
a1 | c1  | c1  | d1
a2 | c2  | c2  | d2
```

### 자연 조인 (Natural Join) — 중복을 제거한 깔끔한 조인

**자연 조인(Natural Join)** 은 동일 조인에서 **중복되는 속성(컬럼) 하나를 제거**한 형태입니다. 두 테이블에서 공통으로 존재하는 속성의 이름이 같고 값도 같은 행끼리 연결하되, 중복된 속성은 한 번만 출력합니다.

> 💡 자연 조인 = 동일 조인 + 중복 속성 제거

자연 조인은 실무에서 매우 자주 사용되며, SQL의 `NATURAL JOIN` 키워드나 `INNER JOIN ... ON` 구문의 기반이 되는 개념입니다.

```
자연 조인 결과 예시:
A  | C  | D      ← 중복된 C 열이 하나로 합쳐짐
a1 | c1 | d1
a2 | c2 | d2
```

### 세 가지 조인의 포함 관계 정리

세 조인의 관계를 포함 관계로 정리하면 다음과 같습니다.

```
세타 조인 (가장 넓음)
    └── 동일 조인 (= 조건만 사용)
            └── 자연 조인 (중복 속성 제거)
```

:::tip 기억하는 방법
- **세타 조인**: 조건이 뭐든 다 됨 (>, &lt;, =, ≠ ...)
- **동일 조인**: 조건이 반드시 "같다(=)"인 세타 조인, 단 속성 중복 있음
- **자연 조인**: 동일 조인인데 중복 속성을 깔끔하게 제거한 것
:::

## 🧪 실제 예시로 배우는 자연 조인 실행 과정

### 예제 테이블 설정

실제 데이터를 가지고 자연 조인이 어떻게 동작하는지 단계별로 살펴보겠습니다.

**릴레이션 R** (속성: A, B, C)

| A  | B  | C  |
|----|----|-----|
| a1 | b1 | c1  |
| a2 | b1 | c1  |
| a3 | b2 | c2  |
| a4 | b2 | c3  |

**릴레이션 S** (속성: B, C, D)

| B  | C  | D  |
|----|----|----|
| b1 | c1 | d1 |
| b1 | c1 | d2 |
| b2 | c3 | d3 |

### 1단계: 공통 속성 찾기

자연 조인을 수행하려면 먼저 **두 릴레이션이 공통으로 가지고 있는 속성**을 찾아야 합니다.

- R의 속성: A, B, C
- S의 속성: B, C, D
- **공통 속성: B, C**

따라서 자연 조인의 조건은 **"R의 B값과 S의 B값이 같고, R의 C값과 S의 C값도 같은 행끼리 연결"** 이 됩니다.

### 2단계: 조건에 맞는 행 찾기

이제 R의 각 행과 S의 각 행을 비교하여 B와 C 값이 동시에 같은 쌍을 찾습니다.

- R의 `(a1, b1, c1)` → S에서 `(b1, c1, ?)` 찾기 → `(b1, c1, d1)`, `(b1, c1, d2)` 두 개 매칭 ✅
- R의 `(a2, b1, c1)` → S에서 `(b1, c1, ?)` 찾기 → `(b1, c1, d1)`, `(b1, c1, d2)` 두 개 매칭 ✅
- R의 `(a3, b2, c2)` → S에서 `(b2, c2, ?)` 찾기 → 없음 ❌ (조인 참여 불가)
- R의 `(a4, b2, c3)` → S에서 `(b2, c3, ?)` 찾기 → `(b2, c3, d3)` 매칭 ✅

### 3단계: 결과 생성 (중복 속성 제거)

동일 조인이라면 B와 C가 두 번씩 나타나겠지만, **자연 조인은 중복 속성을 제거**하여 B와 C를 한 번만 출력합니다.

**자연 조인 결과: R ⋈ S**

| A  | B  | C  | D  |
|----|----|----|-----|
| a1 | b1 | c1 | d1  |
| a1 | b1 | c1 | d2  |
| a2 | b1 | c1 | d1  |
| a2 | b1 | c1 | d2  |
| a4 | b2 | c3 | d3  |

총 5개의 튜플(행)이 결과로 나옵니다. `a3`이 포함된 행은 S에 매칭되는 값이 없어서 결과에 포함되지 않습니다.

```sql
-- 자연 조인을 SQL로 표현하면
-- (개념 이해용, 실제 NATURAL JOIN 키워드와 동일한 의미)
SELECT A, R.B, R.C, D  -- 중복 속성 B, C는 한 번만 출력
FROM R
INNER JOIN S ON R.B = S.B AND R.C = S.C;  -- 공통 속성의 값이 같은 조건
```

:::warning 자연 조인에서 주의할 점
자연 조인은 **이름이 같은 속성을 자동으로 조인 조건으로 사용**합니다. 따라서 두 테이블에 이름이 같지만 의미가 다른 속성이 있다면 의도치 않은 결과가 나올 수 있습니다. 실무에서는 이 점을 항상 주의해야 합니다.
:::

## 🌓 세미 조인 (Semi Join): 한쪽 결과만 필요할 때

### 세미 조인이란?

**세미 조인(Semi Join)** 은 자연 조인을 수행한 후, **한쪽 릴레이션의 속성만 결과로 출력**하는 연산입니다. 마치 두 명단을 비교해서 "공통으로 조건을 만족하는 사람이 누구인지"를 왼쪽 명단 기준으로만 알고 싶을 때 사용하는 것과 같습니다.

세미 조인 기호는 자연 조인 기호에서 **한쪽이 열려 있는 형태**로 표시합니다.

- 자연 조인: `R ⋈ S` (양쪽 닫힌 나비넥타이 모양)
- 세미 조인: `R ⋉ S` (왼쪽만 닫힌 모양 → R의 결과만 출력)
- 세미 조인: `R ⋊ S` (오른쪽만 닫힌 모양 → S의 결과만 출력)

### 세미 조인의 공식 정의

세미 조인의 정의를 수식으로 표현하면 다음과 같습니다.

> **R ⋉ S = πₓ(R ⋈ S)**
>
> 즉, R과 S를 자연 조인한 결과에서 **R의 속성(X)만 프로젝션(추출)**한 것

또는 이렇게도 표현할 수 있습니다.

> **R ⋉ S = R ⋈ πₓ∩ᵧ(S)**
>
> 즉, S 릴레이션에서 R과 S의 **공통 속성만 먼저 프로젝션**한 뒤, 그 결과를 R과 자연 조인

두 표현은 동일한 결과를 만들어냅니다.

### 세미 조인 실행 예시 (단계별)

앞서 사용한 R과 S 테이블로 `R ⋉ S`를 수행해 봅시다.

**1단계: S에서 공통 속성(B, C)만 프로젝션**

S의 속성 중 R과 공통인 B, C만 추출합니다.

| B  | C  |
|----|----|
| b1 | c1 |
| b1 | c1 |  ← 중복!
| b2 | c3 |

릴레이션에서는 **동일한 튜플은 중복을 제거**해야 한다는 규칙이 있습니다. 따라서 중복된 `(b1, c1)`은 하나만 남깁니다.

**프로젝션 결과 (중복 제거 후):**

| B  | C  |
|----|----|
| b1 | c1 |
| b2 | c3 |

**2단계: 프로젝션 결과를 R과 자연 조인**

이제 위 결과와 R을 자연 조인합니다. 공통 속성은 여전히 B, C입니다.

- R의 `(a1, b1, c1)` → `(b1, c1)` 매칭 ✅
- R의 `(a2, b1, c1)` → `(b1, c1)` 매칭 ✅
- R의 `(a3, b2, c2)` → `(b2, c2)` 없음 ❌
- R의 `(a4, b2, c3)` → `(b2, c3)` 매칭 ✅

**3단계: R의 속성만 출력**

세미 조인은 최종 결과로 **R의 속성(A, B, C)만** 출력합니다. D는 포함되지 않습니다.

**세미 조인 결과: R ⋉ S**

| A  | B  | C  |
|----|----|----|
| a1 | b1 | c1 |
| a2 | b1 | c1 |
| a4 | b2 | c3 |

자연 조인 결과(5개 행)와 비교했을 때, 세미 조인은 3개 행만 나오고 D 열이 없습니다.

```sql
-- 세미 조인을 SQL로 표현하면
SELECT DISTINCT R.A, R.B, R.C  -- R의 속성만 출력
FROM R
WHERE EXISTS (                  -- S에 매칭되는 행이 존재하는지 확인
    SELECT 1
    FROM S
    WHERE R.B = S.B AND R.C = S.C  -- 공통 속성 조건
);
```

### 세미 조인이 필요한 이유: 분산 데이터베이스

"그냥 자연 조인하면 되지 왜 굳이 세미 조인을 쓰나요?"라는 의문이 생길 수 있습니다. 세미 조인은 특히 **분산 데이터베이스(Distributed Database)** 환경에서 그 진가를 발휘합니다.

**예시 상황:**
- 서울 서버: 학생(Student) 데이터 보유
- 부산 서버: 교수(Professor) 데이터 보유
- 목표: 학생 정보만 화면에 출력 (교수 정보 불필요)

**자연 조인을 사용하는 경우:**
부산의 교수 데이터 **전체**를 서울로 전송하거나, 서울의 학생 데이터 **전체**를 부산으로 전송해야 합니다. 데이터 양이 많으면 네트워크 전송 비용이 엄청납니다.

**세미 조인을 사용하는 경우:**
부산 서버에서 **공통 속성(조인 키)만** 추출하여 서울로 전송합니다. 서울은 그 작은 데이터를 받아서 로컬에서 자연 조인을 수행하고 학생 정보만 출력합니다. **전송 데이터 양을 최소화**할 수 있습니다.

> 💡 세미 조인은 분산 데이터베이스에서 **네트워크 전송 비용을 줄이기 위한 최적화 기법**으로 널리 사용됩니다.

### 세미 조인의 교환 법칙: 성립하지 않는다!

관계대수에서 **교환 법칙(Commutative Law)** 이 성립하지 않는 연산이 세 가지 있습니다.

| 연산 | 교환 법칙 성립 여부 | 이유 |
|---|---|---|
| 차집합 (R − S) | ❌ 성립 안 함 | R−S ≠ S−R (결과가 다름) |
| 디비전 (R ÷ S) | ❌ 성립 안 함 | R÷S ≠ S÷R (결과가 다름) |
| **세미 조인 (R ⋉ S)** | ❌ 성립 안 함 | 출력되는 속성이 달라짐 |

세미 조인에서 교환 법칙이 성립하지 않는 이유는 명확합니다.

- `R ⋉ S` → 결과로 **R의 속성**만 출력
- `S ⋉ R` → 결과로 **S의 속성**만 출력

두 결과는 완전히 다른 데이터를 담고 있으므로, 절대 교환할 수 없습니다.

:::danger 세미 조인 교환 법칙 절대 주의
`R ⋉ S`와 `S ⋉ R`은 **완전히 다른 결과**를 냅니다. 세미 조인을 사용할 때는 반드시 어느 쪽의 데이터를 출력할 것인지를 먼저 결정하고, 그에 맞게 피연산자의 순서를 정해야 합니다.
:::

## 🌐 외부 조인 (Outer Join): 조인에 참여 못한 데이터도 보고 싶을 때

### 왜 외부 조인이 필요한가?

자연 조인은 **조인 조건을 만족하는 행끼리만** 결과를 만들어냅니다. 따라서 조인 조건에 맞는 상대방이 없는 행은 결과에서 아예 사라집니다. 이것은 때로는 문제가 됩니다.

예를 들어, 학생 테이블과 수강 테이블을 조인할 때, **아직 수업을 신청하지 않은 학생**은 조인 결과에서 사라집니다. 하지만 "수업을 신청하지 않은 학생이 누구인지"를 알고 싶다면 어떻게 해야 할까요? 이럴 때 사용하는 것이 바로 **외부 조인(Outer Join)** 입니다.

> 외부 조인은 **조인에 참여하지 못한 행도 NULL 값으로 채워서 결과에 포함**시켜 주는 조인입니다.

### 외부 조인의 세 가지 종류

외부 조인은 어느 쪽의 데이터를 모두 보존하느냐에 따라 세 가지로 나뉩니다.

| 종류 | 기호 | 설명 |
|---|---|---|
| **좌측 외부 조인** | ⟕ (L+) | 왼쪽 릴레이션의 모든 행 보존, 오른쪽 미참여 행은 NULL |
| **우측 외부 조인** | ⟖ (R+) | 오른쪽 릴레이션의 모든 행 보존, 왼쪽 미참여 행은 NULL |
| **완전 외부 조인** | ⟗ (F+) | 양쪽 릴레이션의 모든 행 보존, 미참여 행은 모두 NULL |

### 외부 조인 예제

다음 두 테이블로 외부 조인의 세 가지 형태를 모두 살펴보겠습니다.

**릴레이션 R** (속성: A, B, C)

| A  | B  | C  |
|----|----|-----|
| a1 | b1 | c1  |
| a2 | b2 | c2  |

**릴레이션 S** (속성: C, D)

| C  | D  |
|----|----|
| c1 | d1 |
| c3 | d2 |

공통 속성은 `C`입니다. 자연 조인 결과:
- R의 `c1` ↔ S의 `c1` → 매칭 ✅
- R의 `c2` → S에 `c2` 없음 ❌
- S의 `c3` → R에 `c3` 없음 ❌

**자연 조인 결과:**

| A  | B  | C  | D  |
|----|----|----|----|
| a1 | b1 | c1 | d1 |

보다시피 `a2`와 `c3` 관련 행은 결과에서 사라졌습니다.

#### 좌측 외부 조인 (Left Outer Join)

왼쪽 R의 **모든 행을 보존**합니다. R에서 조인에 참여하지 못한 `(a2, b2, c2)`도 결과에 포함되며, 오른쪽 S의 속성 D는 **NULL**로 채워집니다.

**좌측 외부 조인 결과: R ⟕ S**

| A  | B  | C  | D    |
|----|----|----|------|
| a1 | b1 | c1 | d1   |
| a2 | b2 | c2 | NULL |  ← S에 c2가 없으므로 D는 NULL

```sql
-- 좌측 외부 조인 SQL
SELECT R.A, R.B, R.C, S.D
FROM R
LEFT OUTER JOIN S ON R.C = S.C;
-- R의 모든 행이 출력되고, 매칭되지 않는 S.D는 NULL
```

#### 우측 외부 조인 (Right Outer Join)

오른쪽 S의 **모든 행을 보존**합니다. S에서 조인에 참여하지 못한 `(c3, d2)`도 결과에 포함되며, 왼쪽 R의 속성 A, B는 **NULL**로 채워집니다.

**우측 외부 조인 결과: R ⟖ S**

| A    | B    | C  | D  |
|------|------|----|-----|
| a1   | b1   | c1 | d1  |
| NULL | NULL | c3 | d2  |  ← R에 c3가 없으므로 A, B는 NULL

```sql
-- 우측 외부 조인 SQL
SELECT R.A, R.B, R.C, S.D
FROM R
RIGHT OUTER JOIN S ON R.C = S.C;
-- S의 모든 행이 출력되고, 매칭되지 않는 R.A, R.B는 NULL
```

#### 완전 외부 조인 (Full Outer Join)

좌측 외부 조인과 우측 외부 조인을 **합쳐놓은 형태**입니다. 양쪽 릴레이션의 모든 행이 결과에 포함됩니다.

**완전 외부 조인 결과: R ⟗ S**

| A    | B    | C  | D    |
|------|------|----|------|
| a1   | b1   | c1 | d1   |
| a2   | b2   | c2 | NULL |  ← 좌측 외부 조인 결과
| NULL | NULL | c3 | d2   |  ← 우측 외부 조인 결과

```sql
-- 완전 외부 조인 SQL
SELECT R.A, R.B, R.C, S.D
FROM R
FULL OUTER JOIN S ON R.C = S.C;
-- 양쪽 모든 행이 출력되고, 미참여 속성은 NULL
```

:::tip NULL 값이 보이면 외부 조인을 사용한 것!
조인 결과에 **NULL 값이 포함되어 있다면**, 그것은 외부 조인을 수행한 결과입니다. NULL은 "해당 조건에 맞는 상대방 데이터가 없다"는 의미입니다.
:::

### 잘못된 사용 vs 올바른 사용

❌ **잘못된 예시: 내부 조인(자연 조인)만 사용하여 전체 현황 파악 시도**

```sql
-- ❌ 잘못된 예: 수강 신청을 하지 않은 학생이 결과에서 사라짐
SELECT 학생.이름, 수강.과목명
FROM 학생
INNER JOIN 수강 ON 학생.학번 = 수강.학번;
-- 수강 신청을 안 한 학생은 결과에 포함되지 않음 → 전체 학생 현황 파악 불가
```

✅ **올바른 예시: 좌측 외부 조인으로 모든 학생 포함**

```sql
-- ✅ 올바른 예: 수강 신청 여부에 관계없이 모든 학생 출력
SELECT 학생.이름, 수강.과목명  -- 출력할 속성 지정
FROM 학생
LEFT OUTER JOIN 수강 ON 학생.학번 = 수강.학번;
-- 수강 신청을 안 한 학생은 과목명이 NULL로 표시됨
-- 예: '김철수' | NULL  → 수강 신청 안 함을 알 수 있음
```

## 🔗 외부 합집합 (Outer Union): 구조가 다른 테이블 합치기

### 외부 합집합이란?

일반적인 합집합(Union)은 **합병 가능(Union-compatible)** 한, 즉 속성의 수와 도메인(자료형)이 같은 두 릴레이션만 합칠 수 있습니다. 하지만 **외부 합집합(Outer Union)** 은 속성이 서로 다른 두 릴레이션도 합칠 수 있습니다.

**예시:**

| 성명 | 과목 | 성적 |
|------|------|------|
| 홍길동 | 수학 | A |

| 성명 | 과목 | 학과코드 |
|------|------|--------|
| 이순신 | 영어 | CS01 |

위 두 테이블은 속성 구조가 달라 일반 합집합을 사용할 수 없습니다. 외부 합집합을 적용하면 **서로 없는 속성은 NULL로 채워서** 합칩니다.

**외부 합집합 결과:**

| 성명 | 과목 | 성적 | 학과코드 |
|------|------|------|--------|
| 홍길동 | 수학 | A | NULL |
| 이순신 | 영어 | NULL | CS01 |

:::info 핵심 요약
외부 합집합은 **합병 불가능한 두 릴레이션을 강제로 합치되, 서로 없는 속성 값은 NULL로 채워주는 연산**입니다.
:::

## ⚙️ 관계 해석 (Relational Calculus): 가볍게 이해하기

### 관계 해석이란?

**관계 해석(Relational Calculus)** 은 관계대수와 달리 **"어떻게 가져올 것인지(How)"가 아니라 "무엇을 가져올 것인지(What)"**를 선언적으로 기술하는 방식입니다. 이를 **비절차적(Non-procedural)** 언어라고 합니다.

| 구분 | 관계대수 | 관계 해석 |
|---|---|---|
| 방식 | 절차적 (어떻게 가져올까?) | 비절차적 (무엇을 가져올까?) |
| 연산 순서 | 중요 | 크게 중요하지 않음 |
| 표현 단위 | 릴레이션 전체 | 튜플 또는 도메인 |
| SQL과의 관계 | SQL의 이론적 기반 | SQL의 이론적 기반 |

관계대수로 표현한 것은 관계 해석으로도 동일하게 표현할 수 있으며, 반대도 성립합니다. 즉, **두 체계는 서로 대응 관계**에 있습니다.

:::note 관계 해석의 실용적 중요성
관계 해석은 이론적으로는 중요하지만, 실무와 시험에서의 비중은 관계대수보다 낮습니다. 기본 개념과 구조를 가볍게 이해하는 수준으로 학습하면 충분합니다.
:::

### 튜플 관계 해석 (Tuple Relational Calculus)

**튜플 관계 해석** 은 릴레이션에서 원하는 데이터를 **튜플(행) 단위**로 표현하는 방식입니다. 연산 단위가 튜플이며, **QBE(Query-By-Example)** 언어가 대표적인 튜플 관계 해석 기반 언어입니다.

튜플 관계 해석에서 사용하는 주요 논리 연산자:

| 기호 | 의미 | 설명 |
|---|---|---|
| ∧ | AND (그리고) | 두 조건 모두 만족 |
| ∨ | OR (또는) | 두 조건 중 하나 이상 만족 |
| ¬ | NOT (부정) | 조건을 부정 |
| ∀ | 전체 정량자 (For All) | 모든 경우에 대해 만족 |
| ∃ | 존재 정량자 (Exists) | 적어도 하나 존재하면 만족 |

**표현 예시: 학생 릴레이션에서 이름 출력**

```
{ s.name | Student(s) }
```

- `s`: student 테이블을 나타내는 튜플 변수
- `s.name`: s 튜플에서 name 속성을 출력
- `Student(s)`: s가 Student 테이블에 속한다는 조건

**표현 예시: 학과가 '컴퓨터'인 학생의 이름 출력**

```
{ s.name | Student(s) ∧ s.department = '컴퓨터' }
```

- `Student(s)`: s는 Student 테이블의 튜플
- `s.department = '컴퓨터'`: 학과가 컴퓨터인 조건
- `s.name`: 조건을 만족하는 학생의 이름 출력

### 도메인 관계 해석 (Domain Relational Calculus)

**도메인 관계 해석** 은 튜플 변수 대신 **도메인 변수(속성 값의 범위를 나타내는 변수)** 를 사용하는 방식입니다. 표현 방법이 조금 다를 뿐 개념은 튜플 관계 해석과 동일합니다.

**표현 예시: 학생 릴레이션에서 이름 출력**

```
{ <ds_name> | Student(ds_name, ...) }
```

- `ds_name`: Student 테이블의 name 속성에 대한 도메인 변수
- 화면에 출력할 속성을 꺾쇠(`< >`) 안에 명시

**표현 예시: 학과가 '컴퓨터'인 학생의 이름 출력**

```
{ <ds_name> | Student(ds_name, d_dept) ∧ d_dept = '컴퓨터' }
```

- `d_dept`: department 속성에 대한 도메인 변수
- 튜플 관계 해석과 동일한 결과를 다른 방식으로 표현

## ⚠️ 주의사항 및 핵심 팁

### 조인 종류 선택 가이드

실무나 시험에서 어떤 조인을 선택해야 할지 헷갈릴 때 다음 기준을 참고하세요.

```
결과에 NULL이 포함되어야 하는가?
├── YES → 외부 조인 사용
│   ├── 왼쪽 데이터 모두 보존 → LEFT OUTER JOIN
│   ├── 오른쪽 데이터 모두 보존 → RIGHT OUTER JOIN
│   └── 양쪽 데이터 모두 보존 → FULL OUTER JOIN
└── NO → 내부 조인 사용
    ├── 비교 조건이 '='인가?
    │   ├── YES + 중복 제거 원함 → 자연 조인 (NATURAL JOIN)
    │   └── YES + 중복 그대로 → 동일 조인 (EQUI JOIN)
    └── NO (>, <, ≠ 등) → 세타 조인 (THETA JOIN)

한쪽 결과만 필요 + 분산 DB 최적화?
└── 세미 조인 (SEMI JOIN)
```

### 자주 하는 실수들

:::warning 실수 1: 자연 조인과 동일 조인 혼동
자연 조인과 동일 조인은 조인 조건은 같지만 결과가 다릅니다. **동일 조인은 중복 속성이 있고**, **자연 조인은 중복 속성이 제거됩니다.** 이 차이를 반드시 기억하세요.
:::

:::warning 실수 2: 세미 조인의 교환 법칙 착각
`R ⋉ S`와 `S ⋉ R`은 **완전히 다른 결과**입니다. 세미 조인은 교환 법칙이 성립하지 않으므로, 어느 쪽의 데이터를 원하는지 명확히 결정한 후 피연산자 순서를 정해야 합니다.
:::

:::warning 실수 3: 프로젝션 후 중복 제거 누락
세미 조인 수행 시 S에서 공통 속성을 프로젝션하면 **중복 튜플이 발생할 수 있습니다.** 릴레이션은 동일한 튜플을 중복으로 가질 수 없으므로, **반드시 중복을 제거해야 합니다.**
:::

:::danger 절대 혼동하지 말 것: 자연 조인과 카티션 프로덕트
카티션 프로덕트(Cartesian Product)는 두 테이블의 **모든 행을 조건 없이 모두 조합**합니다. 자연 조인은 공통 속성의 값이 같은 행끼리만 연결합니다. 이 둘을 혼동하면 전혀 다른 결과가 나옵니다.
:::

## 📋 핵심 정리

이 문서에서 배운 핵심 개념들을 정리합니다.

- **관계대수**는 절차적 언어이며, 일반 집합 연산자와 순수관계 연산자로 구분된다.
- **세타 조인(θ-Join)** 은 모든 비교 연산자(>, &lt;, =, ≠ 등)를 조인 조건으로 사용할 수 있는 가장 일반적인 조인이다.
- **동일 조인(Equi Join)** 은 세타 조인 중 조인 조건이 등호(`=`)인 경우이며, 결과에 중복 속성이 발생한다.
- **자연 조인(Natural Join)** 은 동일 조인에서 중복 속성을 제거한 형태로, 가장 널리 사용되는 조인이다.
- **세미 조인(Semi Join)** 은 자연 조인 후 한쪽 릴레이션의 속성만 결과로 출력하며, 분산 데이터베이스에서 전송 비용 절감에 효과적이다.
- **세미 조인은 교환 법칙과 결합 법칙이 성립하지 않는다.** `R ⋉ S ≠ S ⋉ R`
- **외부 조인(Outer Join)** 은 조인에 참여하지 못한 행도 NULL 값으로 채워 결과에 포함시키는 조인이다.
- **좌측 외부 조인**은 왼쪽 릴레이션의 모든 행을 보존하고, 오른쪽 미참여 속성을 NULL로 채운다.
- **우측 외부 조인**은 오른쪽 릴레이션의 모든 행을 보존하고, 왼쪽 미참여 속성을 NULL로 채운다.
- **완전 외부 조인**은 좌측 + 우측 외부 조인을 합친 것으로, 양쪽 모든 행을 보존한다.
- **외부 합집합**은 구조가 다른(합병 불가능한) 두 릴레이션을 합칠 때 사용하며, 없는 속성은 NULL로 채운다.
- **관계 해석**은 비절차적 언어이며, 튜플 관계 해석과 도메인 관계 해석으로 나뉜다.
- 관계대수와 관계 해석은 **서로 대응 관계**에 있으며, 한 방식으로 표현한 것을 다른 방식으로도 표현할 수 있다.
- 이러한 수학적 이론들이 실제 **SQL 명령어의 이론적 토대**가 된다.
