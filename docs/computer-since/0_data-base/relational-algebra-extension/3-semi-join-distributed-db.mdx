# 🔗 세미 조인의 활용: 분산 데이터베이스에서의 데이터 전송 최적화

데이터베이스를 공부하다 보면 다양한 종류의 **조인(Join)** 연산을 만나게 됩니다. 지난 시간에 우리는 세타 조인, 동일 조인, 자연 조인까지 학습했습니다. 이번 시간에는 그 연장선에서 **세미 조인(Semi Join)** 이라는 특별한 조인 연산을 깊이 있게 살펴봅니다. 세미 조인은 특히 **분산 데이터베이스(Distributed Database, 여러 곳에 나뉘어 저장된 데이터베이스)** 환경에서 네트워크로 전송해야 하는 데이터의 양을 획기적으로 줄여주는 강력한 도구입니다.

이 문서에서는 세미 조인이 무엇인지, 왜 필요한지, 어떻게 동작하는지를 실제 테이블 예시와 함께 아주 자세히 설명합니다. 처음 접하는 분이라도 걱정하지 마세요. 하나하나 차근차근 따라오시면 됩니다. 🙌

## 📚 조인 연산 복습: 세미 조인을 배우기 전에 알아야 할 것들

세미 조인을 이해하려면 먼저 **자연 조인(Natural Join)** 을 확실히 이해해야 합니다. 세미 조인은 자연 조인을 기반으로 정의되기 때문입니다. 잠깐 이전에 배운 내용을 복습해 보겠습니다.

### 관계대수 연산자의 종류

관계대수(Relational Algebra)에는 크게 두 가지 계열의 연산자가 있습니다.

| 분류 | 연산자 종류 |
|---|---|
| **일반 집합 연산자** | 합집합(∪), 교집합(∩), 차집합(−), 카티션 프로덕트(×) |
| **순수 관계 연산자** | 셀렉션(σ), 프로젝션(π), 조인(⋈), 디비전(÷) |

이 중에서 실무에서 가장 많이 사용되는 연산자는 단연 **조인(Join)** 입니다. 조인에도 종류가 있는데, 아래와 같이 정리할 수 있습니다.

| 조인 종류 | 설명 |
|---|---|
| **세타 조인(θ-Join)** | `>`, `<`, `≥`, `≤`, `=`, `≠` 등 모든 비교 조건을 사용한 조인 |
| **동일 조인(Equi Join)** | 조인 조건이 `=`(같음)인 조인. 단, 결과에 공통 속성이 중복됨 |
| **자연 조인(Natural Join)** | 동일 조인에서 중복되는 공통 속성을 하나만 남기고 제거한 조인 |

:::info 자연 조인 핵심 요약
자연 조인(Natural Join)이란 두 릴레이션(테이블)에서 **공통으로 존재하는 속성(컬럼)의 값이 같은 튜플(행)끼리 결합**하고, 그 공통 속성은 한 번만 출력하는 조인입니다. 결과에는 두 릴레이션의 모든 속성이 포함됩니다.
:::

### 자연 조인 예시로 감 잡기

자연 조인이 어떻게 동작하는지 구체적인 예시를 통해 살펴봅시다. 아래 두 릴레이션 R과 S가 있다고 가정합니다.

**릴레이션 R (속성: A, B, C)**

| A | B | C |
|---|---|---|
| a1 | b1 | c1 |
| a2 | b1 | c1 |
| a3 | b1 | c2 |
| a4 | b2 | c3 |

**릴레이션 S (속성: B, C, D)**

| B | C | D |
|---|---|---|
| b1 | c1 | d1 |
| b1 | c1 | d2 |
| b2 | c3 | d3 |

R에는 속성 A, B, C가 있고, S에는 속성 B, C, D가 있습니다. **양쪽에 공통으로 존재하는 속성은 B와 C**입니다. 자연 조인은 이 B와 C의 값이 같은 튜플끼리 결합합니다.

- R의 (b1, c1) ↔ S의 (b1, c1, d1) → 결합 가능 ✅
- R의 (b1, c1) ↔ S의 (b1, c1, d2) → 결합 가능 ✅
- R의 (b1, c2) ↔ S에 (b1, c2)가 없음 → 결합 불가 ❌
- R의 (b2, c3) ↔ S의 (b2, c3, d3) → 결합 가능 ✅

**자연 조인 결과 (속성: A, B, C, D)**

| A | B | C | D |
|---|---|---|---|
| a1 | b1 | c1 | d1 |
| a1 | b1 | c1 | d2 |
| a2 | b1 | c1 | d1 |
| a2 | b1 | c1 | d2 |
| a4 | b2 | c3 | d3 |

총 5개의 튜플이 결과로 나옵니다. 자연 조인은 결과에 **R의 속성과 S의 속성이 모두 포함**됩니다. 공통 속성인 B, C는 중복 제거 후 한 번만 출력됩니다. a3의 (b1, c2)는 S에 대응하는 값이 없으므로 결과에 포함되지 않았습니다.

## 🤔 세미 조인이란 무엇인가? — 개념 소개

### 세미 조인의 정의

이제 본론인 **세미 조인(Semi Join)** 을 살펴봅시다. 세미 조인의 정의를 한 문장으로 표현하면 다음과 같습니다.

> **세미 조인(Semi Join)**이란, 두 릴레이션 R과 S를 자연 조인했을 때 **한쪽 릴레이션(R)의 튜플만을 결과로 출력**하는 조인 연산입니다.

자연 조인은 결과에 R과 S의 속성이 **모두** 나옵니다. 하지만 세미 조인은 결과에 **닫혀 있는 쪽(기준이 되는 쪽)의 속성만** 나옵니다. 마치 조인에 참여한 영수증에서 내가 필요한 정보만 오려내는 것과 같습니다.

### 세미 조인의 기호 표현

자연 조인은 `⋈` 기호로 표현하는데, 세미 조인은 이 기호의 **한쪽이 열린 형태**로 표현합니다.

- 자연 조인: `R ⋈ S` → R과 S 모두의 속성이 결과에 포함
- R에 대한 세미 조인: `R ⋉ S` → **R의 속성만** 결과에 포함
- S에 대한 세미 조인: `R ⋊ S` → **S의 속성만** 결과에 포함

:::tip 기호 기억법 💡
세미 조인 기호(`⋉` 또는 `⋊`)에서 **닫혀 있는(뾰족한) 쪽의 릴레이션** 속성이 결과로 나온다고 기억하세요. `R ⋉ S`에서 왼쪽(R 쪽)이 닫혀 있으므로 R의 속성만 나오고, `R ⋊ S`에서 오른쪽(S 쪽)이 닫혀 있으므로 S의 속성만 나옵니다.
:::

### 세미 조인의 수식 정의

세미 조인을 수식으로 엄밀하게 정의하면 두 가지 방법으로 표현할 수 있습니다.

**방법 1: 프로젝션을 이용한 정의**

```
R ⋉ S = R ⋈ π(X∩Y)(S)
```

이것을 말로 풀어쓰면:
1. S 릴레이션에 대해 **R과 S의 공통 속성(X∩Y)만 프로젝션(추출)** 합니다.
2. 그 추출된 결과를 R과 **자연 조인**합니다.
3. 최종 결과는 **R의 속성만** 포함됩니다.

**방법 2: 자연 조인 후 프로젝션**

```
R ⋉ S = π_X(R ⋈ S)
```

이것을 말로 풀어쓰면:
1. R과 S를 먼저 **자연 조인**합니다.
2. 그 결과에서 **R의 속성(X)만 프로젝션(추출)** 합니다.

두 방법 모두 동일한 결과를 만들어냅니다. 접근 방식만 다를 뿐입니다.

## 🌐 왜 세미 조인이 필요한가? — 분산 데이터베이스에서의 필요성

### 중앙 집중식 vs. 분산 데이터베이스

세미 조인이 **왜 필요한지** 이해하려면, 먼저 **분산 데이터베이스(Distributed Database)** 가 무엇인지 알아야 합니다.

**중앙 집중식 데이터베이스**는 마치 모든 책이 한 도서관에 모여 있는 것과 같습니다. 필요한 책을 찾을 때 한 곳만 가면 됩니다. 데이터가 한 서버에 모두 있으므로 조인 연산을 할 때 네트워크를 통한 데이터 이동이 필요 없습니다.

반면, **분산 데이터베이스**는 책이 여러 도서관에 나뉘어 보관된 것과 같습니다. 예를 들어, 소설책은 서울 도서관에, 과학책은 부산 도서관에 있는 상황입니다. 두 종류의 책 정보를 함께 조회하려면 어느 한쪽의 책 목록을 다른 도서관으로 보내거나, 한쪽 도서관이 다른 도서관으로 이동해야 합니다.

### 분산 환경에서의 조인 문제

현실적인 예시로 살펴봅시다. **서울 서버에는 학생 테이블**이 있고, **부산 서버에는 교수 테이블**이 있다고 가정합니다.

```
[서울 서버]                    [부산 서버]
┌─────────────────────┐       ┌─────────────────────┐
│   학생 테이블 (100만 행) │  ←→  │   교수 테이블 (1만 행)  │
└─────────────────────┘  네트  └─────────────────────┘
                         워크
```

이 두 테이블을 조인하려면 어느 한쪽의 데이터를 네트워크를 통해 다른 쪽으로 전송해야 합니다. 만약 최종 결과로 **학생 정보만 필요한 상황**이라면 어떨까요?

- **자연 조인 방식**: 부산의 교수 테이블(1만 행, 모든 컬럼)을 서울로 전송 → 서울에서 조인 후 학생 속성만 추출
- **세미 조인 방식**: 부산의 교수 테이블에서 **공통 속성(예: 학과 코드)만** 추출해서 서울로 전송 → 서울에서 자연 조인

세미 조인 방식에서는 교수 테이블 전체가 아니라 **공통 속성 컬럼 하나만** 전송합니다. 교수 테이블에 이름, 나이, 연봉, 연구실, 연락처 등 수십 개의 컬럼이 있더라도 **조인에 필요한 컬럼 하나만** 보내면 되므로, 전송 데이터 양이 획기적으로 줄어드는 것입니다!

> **분산 데이터베이스에서 세미 조인을 사용하는 핵심 이유는 네트워크를 통해 전송해야 하는 데이터의 양을 최소화하기 위함입니다.** 전송 데이터가 줄어들면 네트워크 비용이 감소하고, 전체 쿼리 응답 속도가 빨라집니다.

:::info 분산 DB와 세미 조인
분산 데이터베이스 시스템에서 세미 조인은 가장 널리 활용되는 조인 최적화 기법 중 하나입니다. 특히 대용량 데이터를 다루는 환경에서 네트워크 전송 비용은 전체 쿼리 성능에 결정적인 영향을 미치기 때문에, 세미 조인을 통한 데이터 전송 최소화는 매우 중요한 전략입니다.
:::

## ⚙️ 세미 조인의 핵심 원리 — 단계별 동작 과정

### 세미 조인이 동작하는 3단계

세미 조인은 내부적으로 다음 세 단계를 거쳐 동작합니다.

**1단계: 공통 속성 파악**

R과 S에서 공통으로 존재하는 속성이 무엇인지 파악합니다. 이것이 조인의 비교 기준이 됩니다.

**2단계: S에서 공통 속성만 프로젝션(추출)**

S 릴레이션 전체가 아니라, **공통 속성 컬럼만** 뽑아냅니다. 이때 중복된 튜플은 제거합니다(릴레이션의 정의상 중복 튜플은 존재할 수 없기 때문입니다).

**3단계: R과 프로젝션 결과를 자연 조인**

R 릴레이션과 2단계에서 만든 결과를 자연 조인합니다. 결과에는 **R의 속성만** 포함됩니다.

### 실제 데이터로 단계별 계산해보기

앞서 사용한 R과 S 릴레이션으로 `R ⋉ S`(R에 대한 세미 조인)를 단계별로 계산해 봅시다.

**📌 입력 릴레이션**

**릴레이션 R (속성: A, B, C)**

| A | B | C |
|---|---|---|
| a1 | b1 | c1 |
| a2 | b1 | c1 |
| a3 | b1 | c2 |
| a4 | b2 | c3 |

**릴레이션 S (속성: B, C, D)**

| B | C | D |
|---|---|---|
| b1 | c1 | d1 |
| b1 | c1 | d2 |
| b2 | c3 | d3 |

---

**📌 1단계: 공통 속성 파악**

R의 속성: {A, B, C}
S의 속성: {B, C, D}
공통 속성(교집합): **{B, C}**

---

**📌 2단계: S에서 공통 속성(B, C)만 프로젝션**

S에서 B와 C 컬럼만 추출합니다.

S를 B, C로 프로젝션하면:

| B | C |
|---|---|
| b1 | c1 |
| b1 | c1 | ← 중복!
| b2 | c3 |

릴레이션에서 중복 튜플은 반드시 제거해야 합니다. 따라서 최종 프로젝션 결과는:

| B | C |
|---|---|
| b1 | c1 |
| b2 | c3 |

---

**📌 3단계: R과 프로젝션 결과를 자연 조인**

R과 위에서 만든 프로젝션 결과를 자연 조인합니다. 공통 속성은 여전히 B, C입니다.

- R의 (a1, **b1, c1**) ↔ 프로젝션 결과의 (**b1, c1**) → 조인 성공 ✅
- R의 (a2, **b1, c1**) ↔ 프로젝션 결과의 (**b1, c1**) → 조인 성공 ✅
- R의 (a3, **b1, c2**) ↔ 프로젝션 결과에 (b1, c2) 없음 → 조인 실패 ❌
- R의 (a4, **b2, c3**) ↔ 프로젝션 결과의 (**b2, c3**) → 조인 성공 ✅

**세미 조인 `R ⋉ S` 최종 결과 (속성: A, B, C — R의 속성만 포함)**

| A | B | C |
|---|---|---|
| a1 | b1 | c1 |
| a2 | b1 | c1 |
| a4 | b2 | c3 |

**3개의 튜플**이 결과로 나옵니다. a3의 (b1, c2)는 S에 대응하는 값이 없으므로 제외됩니다.

### 자연 조인 결과와 세미 조인 결과 비교

같은 R과 S에 대해 자연 조인과 세미 조인의 결과를 비교해 보겠습니다.

| 구분 | 자연 조인 (R ⋈ S) | 세미 조인 (R ⋉ S) |
|---|---|---|
| **결과 속성** | A, B, C, D (R과 S 모두) | A, B, C (R만) |
| **결과 튜플 수** | 5개 | 3개 |
| **포함 여부** | a1×d1, a1×d2, a2×d1, a2×d2, a4×d3 | a1, a2, a4 |
| **용도** | 양쪽 데이터가 모두 필요할 때 | 한쪽 데이터만 필요할 때 |

> 세미 조인은 "S와 조인했을 때 매칭되는 R의 튜플이 무엇인지"만 알아내는 연산입니다. S의 속성 값은 최종 결과에 포함되지 않습니다.

## ↔️ 세미 조인의 교환 법칙과 결합 법칙 — 주의해야 할 특성

### 교환 법칙이 성립하지 않는다 🚫

세미 조인에서 가장 중요한 주의사항은 **교환 법칙(Commutative Law)이 성립하지 않는다**는 점입니다.

우리가 일반 덧셈에서는 `3 + 5 = 5 + 3`이 성립합니다. 이것이 교환 법칙입니다. 자연 조인도 `R ⋈ S = S ⋈ R`처럼 교환 법칙이 성립합니다. 결과에 R과 S의 속성이 모두 포함되기 때문에 순서를 바꿔도 동일한 결과가 나옵니다.

하지만 세미 조인은 다릅니다.

```
R ⋉ S ≠ S ⋉ R
```

- `R ⋉ S`: 결과에 **R의 속성**만 포함 → A, B, C
- `S ⋉ R`: 결과에 **S의 속성**만 포함 → B, C, D

결과로 나오는 속성 자체가 달라지므로 교환 법칙이 절대 성립하지 않습니다.

:::warning 교환 법칙 불성립 주의
세미 조인을 사용할 때는 반드시 **어느 쪽 릴레이션의 결과를 원하는지** 명확히 결정한 후, 해당 릴레이션을 **닫힌 쪽(왼쪽 또는 오른쪽)에 배치**해야 합니다. 순서를 잘못 쓰면 완전히 다른 결과가 나옵니다.
:::

### 결합 법칙도 성립하지 않는다 🚫

결합 법칙(Associative Law)도 성립하지 않습니다. 덧셈에서는 `(1+2)+3 = 1+(2+3)`이 성립하지만, 세미 조인에서는 순서를 바꾸면 결과 속성 자체가 달라질 수 있습니다.

관계대수에서 **교환 법칙과 결합 법칙이 성립하지 않는 연산자**는 다음 세 가지입니다:

| 연산자 | 이유 |
|---|---|
| **차집합(−)** | `R − S ≠ S − R` |
| **디비전(÷)** | `R ÷ S ≠ S ÷ R` |
| **세미 조인(⋉)** | `R ⋉ S ≠ S ⋉ R` |

### 자연 조인과의 변환 관계

세미 조인은 자연 조인을 이용해 다음과 같이 표현할 수도 있습니다.

```
R ⋉ S = π_X(R ⋈ S)
```

여기서 주의할 점이 있습니다. 자연 조인 `R ⋈ S`는 교환 법칙이 성립하므로 `R ⋈ S = S ⋈ R`입니다. 그런데 세미 조인으로 변환하면 이야기가 달라집니다.

- `R ⋉ S = π_X(R ⋈ S)` → R의 속성(X)으로 프로젝션
- `S ⋉ R = π_Y(S ⋈ R)` → S의 속성(Y)으로 프로젝션

`R ⋈ S`와 `S ⋈ R`의 결과 집합은 같더라도, 마지막에 **어느 속성으로 프로젝션하느냐**에 따라 결과가 달라집니다. 이것이 바로 세미 조인에서 교환 법칙이 성립하지 않는 근본적인 이유입니다.

## 💻 SQL로 표현하는 세미 조인 — 실제 코드 예시

### 기본 SQL 예시

실제 데이터베이스 시스템에서 세미 조인의 개념을 SQL로 표현하는 방법을 살펴보겠습니다. 아래 예시는 앞서 다룬 R과 S 테이블을 기반으로 합니다.

:::note SQL과 세미 조인
표준 SQL에는 `SEMI JOIN`이라는 키워드가 따로 없습니다. 대신 `EXISTS`, `IN`, 또는 서브쿼리(subquery, 쿼리 안에 쿼리)를 사용하여 세미 조인의 동작을 구현합니다.
:::

**테이블 생성 및 데이터 입력**

```sql
-- R 테이블 생성
CREATE TABLE R (
    A VARCHAR(10),  -- A 속성
    B VARCHAR(10),  -- B 속성 (S와 공통)
    C VARCHAR(10)   -- C 속성 (S와 공통)
);

-- S 테이블 생성
CREATE TABLE S (
    B VARCHAR(10),  -- B 속성 (R과 공통)
    C VARCHAR(10),  -- C 속성 (R과 공통)
    D VARCHAR(10)   -- D 속성
);

-- R 테이블에 데이터 입력
INSERT INTO R VALUES ('a1', 'b1', 'c1');
INSERT INTO R VALUES ('a2', 'b1', 'c1');
INSERT INTO R VALUES ('a3', 'b1', 'c2');  -- 이 튜플은 세미 조인에서 제외됨
INSERT INTO R VALUES ('a4', 'b2', 'c3');

-- S 테이블에 데이터 입력
INSERT INTO S VALUES ('b1', 'c1', 'd1');
INSERT INTO S VALUES ('b1', 'c1', 'd2');
INSERT INTO S VALUES ('b2', 'c3', 'd3');
```

**세미 조인 구현 방법 1: EXISTS 사용 (권장)**

```sql
-- ✅ 올바른 세미 조인 구현 (EXISTS 방식)
-- "S에 매칭되는 튜플이 존재하는 R의 모든 속성을 출력"
SELECT R.A, R.B, R.C        -- R의 속성만 선택 (세미 조인의 핵심)
FROM R                       -- 기준 테이블은 R
WHERE EXISTS (               -- EXISTS: S에 조건을 만족하는 행이 하나라도 있으면 참
    SELECT 1                 -- 실제 값은 필요 없고 존재 여부만 확인
    FROM S                   -- S 테이블에서
    WHERE R.B = S.B          -- R과 S의 B 값이 같고
      AND R.C = S.C          -- R과 S의 C 값도 같은 것이 존재하면
);
-- 결과: (a1, b1, c1), (a2, b1, c1), (a4, b2, c3) — 3개 튜플
```

**세미 조인 구현 방법 2: IN 사용**

```sql
-- ✅ 올바른 세미 조인 구현 (IN + 서브쿼리 방식)
-- "S에 존재하는 (B, C) 조합을 가진 R의 튜플만 출력"
SELECT R.A, R.B, R.C                      -- R의 속성만 선택
FROM R                                     -- 기준 테이블 R
WHERE (R.B, R.C) IN (                      -- (B, C) 조합이
    SELECT DISTINCT S.B, S.C               -- S에서 추출한 공통 속성(B, C) 목록에 있으면
    FROM S                                 -- S 테이블에서
);
-- 결과: 동일하게 (a1, b1, c1), (a2, b1, c1), (a4, b2, c3)
```

**잘못된 방법 — 자연 조인과 혼동하지 마세요**

```sql
-- ❌ 잘못된 방법: 이것은 자연 조인이지 세미 조인이 아님
-- R과 S의 속성이 모두 출력되므로 세미 조인과 다름
SELECT R.A, R.B, R.C, S.D   -- S.D까지 포함 → 세미 조인이 아님!
FROM R
JOIN S ON R.B = S.B AND R.C = S.C;
-- 결과: 5개 튜플 출력 (자연 조인 결과, a1-d1, a1-d2, a2-d1, a2-d2, a4-d3)
```

위의 잘못된 예시를 보면 S.D 컬럼이 포함되어 결과가 5개 튜플로 늘어납니다. 세미 조인은 R의 속성만 출력해야 하며, 결과 튜플도 R 기준으로 3개여야 합니다.

### 분산 환경을 시뮬레이션하는 세미 조인 예시

분산 데이터베이스 시나리오를 SQL로 표현해 보겠습니다. 서울 서버에 학생 테이블이 있고, 부산 서버에 수강신청 테이블이 있다고 가정합니다.

```sql
-- [서울 서버] 학생 테이블
CREATE TABLE 학생 (
    학번 VARCHAR(10),   -- 학생 고유 번호 (수강신청 테이블과 공통)
    이름 VARCHAR(20),   -- 학생 이름
    학과 VARCHAR(20),   -- 소속 학과
    학년 INT            -- 재학 학년
);

-- [부산 서버] 수강신청 테이블
CREATE TABLE 수강신청 (
    학번   VARCHAR(10),  -- 학생 고유 번호 (학생 테이블과 공통)
    과목번호 VARCHAR(10), -- 수강 과목 번호
    성적   CHAR(2)       -- 취득 성적
);
```

```sql
-- ✅ 세미 조인 활용: "C413 과목을 수강한 학생의 정보(학생 테이블 속성만)를 조회"
-- 분산 환경에서는 부산 서버로부터 '학번' 컬럼만 전송받아 서울에서 조인
SELECT 학생.학번, 학생.이름, 학생.학과, 학생.학년  -- 학생 테이블 속성만 출력
FROM 학생                                          -- 기준: 서울 서버의 학생 테이블
WHERE EXISTS (                                    -- 수강신청 테이블에 해당하는 행이 있으면
    SELECT 1
    FROM 수강신청                                  -- 부산 서버의 수강신청 테이블
    WHERE 학생.학번 = 수강신청.학번                -- 학번이 일치하고
      AND 수강신청.과목번호 = 'C413'               -- 과목번호가 C413인 경우
);
-- 결과: 성적, 과목번호 같은 수강신청 테이블 정보 없이 학생 정보만 출력
-- 분산 환경에서는 부산→서울 전송 데이터: 학번 컬럼만! (과목번호, 성적 컬럼 전송 불필요)
```

## 🔄 외부 조인(Outer Join) — 세미 조인과 함께 알아두어야 할 확장 조인

세미 조인과 함께 관계대수의 확장으로 다루어지는 중요한 개념이 바로 **외부 조인(Outer Join)** 입니다. 이 두 개념은 함께 이해하면 조인 연산 전체를 더욱 명확하게 파악할 수 있습니다.

### 외부 조인이 왜 필요한가?

자연 조인의 한계를 먼저 생각해봅시다. 자연 조인은 **두 릴레이션에서 공통 속성의 값이 일치하는 튜플만** 결과에 포함합니다. 즉, **조인에 참여하지 못한 튜플은 결과에서 사라집니다.**

예를 들어, 학생 테이블과 수강신청 테이블을 자연 조인하면 **한 번도 수강신청을 하지 않은 학생은 결과에 나타나지 않습니다.** 하지만 "수강신청 여부와 관계없이 모든 학생을 조회하되, 수강신청한 학생은 수강 정보도 함께 보고 싶다"는 요구사항이 있다면 어떻게 할까요? 이럴 때 사용하는 것이 바로 **외부 조인(Outer Join)** 입니다.

> **외부 조인이란, 조인에 참여하지 못한 튜플도 결과에 포함시키되, 해당하는 상대방 속성 값을 `NULL`로 채워서 출력하는 조인입니다.**

조인 결과에 `NULL` 값이 보인다면, 그것은 외부 조인이 수행된 증거입니다.

### 외부 조인의 세 가지 종류

외부 조인은 어느 쪽 릴레이션의 튜플을 모두 보존하느냐에 따라 세 가지로 나뉩니다.

| 종류 | 기호 | 설명 |
|---|---|---|
| **좌측 외부 조인 (Left Outer Join)** | `R ⟕ S` 또는 `R LEFT+ S` | **왼쪽(R)의 모든 튜플** 보존. S에 매칭 없는 R 튜플의 S 속성은 NULL |
| **우측 외부 조인 (Right Outer Join)** | `R ⟖ S` 또는 `R RIGHT+ S` | **오른쪽(S)의 모든 튜플** 보존. R에 매칭 없는 S 튜플의 R 속성은 NULL |
| **완전 외부 조인 (Full Outer Join)** | `R ⟗ S` 또는 `R FULL+ S` | **양쪽 모든 튜플** 보존. 매칭 없는 쪽 속성은 모두 NULL |

### 외부 조인 데이터 예시

다음 두 릴레이션으로 세 가지 외부 조인을 모두 계산해 봅시다.

**릴레이션 R (속성: A, B, C)**

| A | B | C |
|---|---|---|
| a1 | b1 | c1 |
| a2 | b2 | c2 |

**릴레이션 S (속성: C, D)**

| C | D |
|---|---|
| c1 | d1 |
| c3 | d2 |

공통 속성은 **C**입니다. R에서 c1은 S의 c1과 매칭되지만, R의 c2는 S에 없습니다. S의 c3는 R에 없습니다.

**① 자연 조인 결과 (R ⋈ S)**

| A | B | C | D |
|---|---|---|---|
| a1 | b1 | c1 | d1 |

c2와 c3는 매칭이 없어 결과에서 제외됩니다. 단 1개의 튜플만 남습니다.

**② 좌측 외부 조인 결과 (R ⟕ S)**

| A | B | C | D |
|---|---|---|---|
| a1 | b1 | c1 | d1 |
| a2 | b2 | c2 | NULL |

R의 모든 튜플이 보존됩니다. a2의 c2는 S에 매칭이 없으므로 D 값이 **NULL**로 채워집니다.

**③ 우측 외부 조인 결과 (R ⟖ S)**

| A | B | C | D |
|---|---|---|---|
| a1 | b1 | c1 | d1 |
| NULL | NULL | c3 | d2 |

S의 모든 튜플이 보존됩니다. S의 c3는 R에 매칭이 없으므로 A, B 값이 **NULL**로 채워집니다.

**④ 완전 외부 조인 결과 (R ⟗ S)**

| A | B | C | D |
|---|---|---|---|
| a1 | b1 | c1 | d1 |
| a2 | b2 | c2 | NULL |
| NULL | NULL | c3 | d2 |

좌측 외부 조인과 우측 외부 조인의 결과를 합쳐놓은 것과 같습니다. R과 S의 모든 튜플이 결과에 포함됩니다.

### SQL로 구현하는 외부 조인

```sql
-- ✅ 좌측 외부 조인 (Left Outer Join)
-- "R의 모든 행을 보여주고, S에 매칭이 없으면 NULL로 채움"
SELECT R.A, R.B, R.C, S.D   -- 모든 속성 선택
FROM R                        -- 왼쪽 기준 테이블
LEFT OUTER JOIN S             -- 좌측 외부 조인
ON R.C = S.C;                 -- C 값이 같은 것끼리 조인

-- ✅ 우측 외부 조인 (Right Outer Join)
-- "S의 모든 행을 보여주고, R에 매칭이 없으면 NULL로 채움"
SELECT R.A, R.B, R.C, S.D
FROM R
RIGHT OUTER JOIN S            -- 우측 외부 조인
ON R.C = S.C;

-- ✅ 완전 외부 조인 (Full Outer Join)
-- "R과 S의 모든 행을 보여주고, 매칭이 없는 쪽은 NULL로 채움"
SELECT R.A, R.B, R.C, S.D
FROM R
FULL OUTER JOIN S             -- 완전 외부 조인
ON R.C = S.C;
```

:::warning NULL 값 처리 주의
외부 조인 결과에는 NULL 값이 포함될 수 있습니다. 이후 해당 결과를 필터링하거나 집계할 때 NULL 처리를 반드시 고려해야 합니다. 예를 들어, `WHERE D = 'd1'`로 필터링하면 NULL인 행은 자동으로 제외됩니다.
:::

## ⚠️ 주의사항 및 실용적인 팁

### 세미 조인 사용 시 주의사항

:::danger 교환 순서 혼동 금지
`R ⋉ S`와 `S ⋉ R`은 전혀 다른 결과를 냅니다. 세미 조인을 쓸 때는 **"나는 어느 릴레이션의 결과가 필요한가?"** 를 먼저 결정한 후, 해당 릴레이션을 **닫힌 쪽(왼쪽)에 배치**해야 합니다. 이 순서를 잘못 쓰면 완전히 엉뚱한 결과가 나옵니다.
:::

:::tip 세미 조인 활용 체크리스트
세미 조인을 사용하기 전 다음을 확인하세요:
- ✅ 최종 결과로 **한쪽 릴레이션의 속성만** 필요한가?
- ✅ 두 릴레이션 사이에 **공통 속성(조인 키)** 이 존재하는가?
- ✅ 분산 환경에서 **네트워크 전송 비용을 줄여야** 하는 상황인가?
- ✅ SQL로 구현할 때 `EXISTS` 또는 `IN` 서브쿼리를 사용했는가?
:::

### 자연 조인 vs. 세미 조인 선택 기준

| 상황 | 추천 조인 |
|---|---|
| 양쪽 테이블의 속성이 모두 필요한 경우 | **자연 조인** |
| 한쪽 테이블의 속성만 필요하고, 상대 테이블의 매칭 여부만 확인할 경우 | **세미 조인** |
| 분산 환경에서 네트워크 전송량을 줄여야 할 경우 | **세미 조인** |
| 조인에 참여하지 못한 튜플도 결과에 포함해야 할 경우 | **외부 조인** |

### 프로젝션 후 중복 제거를 잊지 마세요

세미 조인의 2단계(공통 속성 프로젝션)에서 **중복 튜플 제거**를 반드시 수행해야 합니다. 릴레이션의 정의상 동일한 튜플은 하나만 존재해야 하기 때문입니다. SQL에서는 `DISTINCT` 키워드로 이를 보장합니다.

```sql
-- ✅ DISTINCT로 중복 제거 (올바른 방법)
SELECT DISTINCT S.B, S.C   -- 공통 속성만 추출하고 중복 제거
FROM S;

-- ❌ DISTINCT 없이 프로젝션 (중복 발생 가능)
SELECT S.B, S.C            -- 중복된 (b1, c1)이 두 번 나올 수 있음
FROM S;
```

### 성능 관점에서의 세미 조인

분산 데이터베이스 환경에서 세미 조인의 성능 이점을 정리합니다.

```
[자연 조인 방식의 데이터 전송]
부산 서버 → 서울 서버:
- 컬럼 수: 전체 컬럼 (예: 학번, 과목번호, 성적, 교수명, 강의실, ... N개)
- 행 수: 전체 행 (예: 100만 행)
- 전송 데이터: N컬럼 × 100만행 = 매우 큼

[세미 조인 방식의 데이터 전송]
부산 서버 → 서울 서버:
- 컬럼 수: 공통 속성만 (예: 학번 1개)
- 행 수: 중복 제거 후 (예: 5만 행)
- 전송 데이터: 1컬럼 × 5만행 = 매우 작음 ✅
```

이처럼 세미 조인은 분산 환경에서 네트워크 I/O를 극적으로 줄일 수 있는 강력한 최적화 기법입니다.

## 📖 관계 해석 간략 소개 — 관계대수와의 대응 관계

이 문서의 원본 강의에서는 관계 해석(Relational Calculus)도 함께 다루었으므로, 간략히 소개합니다.

### 관계 해석이란?

**관계 해석(Relational Calculus)** 은 관계대수와 함께 관계형 데이터베이스의 이론적 기반을 이루는 개념입니다. 두 개념의 핵심 차이는 다음과 같습니다.

| 구분 | 관계대수 | 관계 해석 |
|---|---|---|
| **처리 방식** | 절차적 (어떻게 가져올지 명시) | 비절차적 (무엇을 원하는지만 명시) |
| **연산 순서** | 중요함 | 크게 중요하지 않음 |
| **표현 방식** | 연산자 기호 사용 | 논리식(WFF, Well-Formed Formula) 사용 |
| **분류** | 일반 집합 연산자 / 순수 관계 연산자 | 튜플 관계 해석 / 도메인 관계 해석 |

:::note 관계대수와 관계 해석의 동등성
관계대수로 표현한 것은 관계 해석으로도 표현할 수 있고, 그 반대도 마찬가지입니다. 두 체계는 서로 다른 방식이지만 표현 능력이 동등합니다. 이것은 에드거 F. 코드(Edgar F. Codd)가 제안하여 증명한 중요한 이론입니다.
:::

### 튜플 관계 해석 표현식 예시

```
-- 학생 릴레이션에서 학과가 '컴퓨터'인 학생의 이름을 출력
{ s.이름 | STUDENT(s) ∧ s.학과 = '컴퓨터' }

-- 해석:
-- s는 STUDENT 릴레이션의 튜플 변수
-- STUDENT(s): s가 학생 릴레이션에 속한다
-- s.학과 = '컴퓨터': 학과 속성값이 '컴퓨터'인 것
-- s.이름: 그 조건을 만족하는 튜플의 이름 속성 출력
```

관계 해석은 이론적 배경 이해에 중요하지만, 실무에서는 SQL이라는 실제 언어로 구현되어 사용되므로 개념적 이해 수준으로 파악하면 충분합니다. SQL의 `SELECT-FROM-WHERE` 구조는 관계 해석의 비절차적 특성을 그대로 계승한 것입니다.

## 📝 핵심 정리

이 문서에서 배운 핵심 개념을 정리합니다.

- **세미 조인(Semi Join)** 은 두 릴레이션을 자연 조인한 결과에서 **한쪽 릴레이션의 속성만 출력**하는 조인 연산이다.
- 세미 조인의 정의: `R ⋉ S = R ⋈ π(X∩Y)(S)` — S에서 공통 속성만 프로젝션한 결과를 R과 자연 조인.
- 세미 조인은 **분산 데이터베이스 환경에서 네트워크 전송 데이터량을 최소화**하기 위해 주로 사용된다.
- `R ⋉ S`는 R의 속성만, `S ⋉ R`은 S의 속성만 결과로 나온다 — **닫힌 쪽 릴레이션의 속성이 결과에 포함**된다.
- 세미 조인은 **교환 법칙과 결합 법칙이 성립하지 않는다** — 차집합, 디비전과 함께 비교환적 연산자다.
- 프로젝션 수행 시 **중복 튜플은 반드시 제거**해야 한다 — 릴레이션의 정의에 따라 중복 불가.
- **외부 조인(Outer Join)** 은 조인에 참여하지 못한 튜플도 NULL 값을 채워 결과에 포함시키는 조인이다.
- 외부 조인의 세 종류: **좌측(Left), 우측(Right), 완전(Full) 외부 조인** — 어느 쪽을 보존하느냐로 구분.
- SQL에서 세미 조인은 **`EXISTS` 또는 `IN` 서브쿼리**로 구현한다.
- **관계 해석(Relational Calculus)** 은 관계대수와 표현 능력이 동등하며, 비절차적 방식으로 데이터를 기술한다.
- 관계 해석은 **튜플 관계 해석**과 **도메인 관계 해석**으로 나뉘며, SQL의 이론적 기반이 된다.
