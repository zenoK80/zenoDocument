# 🗄️ 이상현상과 정규화의 개념 — 1NF부터 5NF까지 완전 정복

데이터베이스를 공부하다 보면 "정규화(Normalization)"라는 단어를 반드시 만나게 됩니다. 처음 접하는 분들은 "정규화? 뭔가 복잡한 수학 용어 같은데..."라고 느낄 수 있지만, 사실 정규화는 아주 단순한 목표를 가지고 있습니다. 바로 **데이터를 깔끔하고 일관성 있게 유지하기 위한 과정**입니다.

이 문서에서는 왜 정규화가 필요한지, 어떤 단계가 있는지, 그리고 각 단계에서 어떤 문제를 해결하는지를 실제 예시와 함께 아주 자세하게 살펴봅니다. 초보자도 충분히 이해할 수 있도록 비유와 예시를 최대한 풍부하게 활용할 것이니 천천히 따라오세요.

---

## 🚨 이상현상(Anomaly)이란 무엇인가?

### 이상현상의 정의

**이상현상(Anomaly)** 이란, 데이터베이스 테이블이 잘못 설계되었을 때 데이터를 삽입, 삭제, 수정하는 과정에서 발생하는 **비정상적인 부작용**을 말합니다. 마치 서랍 하나에 양말, 셔츠, 서류를 전부 뒤섞어 넣어두면 양말 하나를 꺼내려다 서류가 같이 딸려 나오는 것처럼, 테이블이 잘못 설계되면 의도하지 않은 결과가 연쇄적으로 발생합니다.

이상현상은 크게 세 가지로 나눕니다.

| 이상현상 종류 | 설명 | 비유 |
|---|---|---|
| **삽입 이상(Insertion Anomaly)** | 새로운 데이터를 넣으려면 불필요한 다른 데이터도 함께 넣어야 하는 문제 | 새 교수를 등록하려면 반드시 학생이 있어야 하는 상황 |
| **삭제 이상(Deletion Anomaly)** | 특정 데이터를 지우면 의도치 않게 다른 중요한 데이터까지 사라지는 문제 | 수강 취소를 했더니 담당 교수 정보까지 지워지는 상황 |
| **갱신 이상(Update Anomaly)** | 데이터를 수정할 때 여러 군데를 동시에 바꿔야 하고, 일부를 빠뜨리면 데이터가 불일치하는 문제 | 교수의 소속 학과를 바꿨는데 한 곳만 바뀌고 나머지는 그대로인 상황 |

### 왜 이상현상이 발생하는가?

이상현상은 본질적으로 **하나의 테이블 안에 서로 다른 주제의 정보가 뒤섞여 있을 때** 발생합니다. 예를 들어 "학생의 수강 정보"와 "교수의 담당 과목 정보"를 하나의 테이블에 모두 집어넣으면, 학생 정보를 건드릴 때 교수 정보가 영향을 받고, 교수 정보를 건드릴 때 학생 정보가 영향을 받는 식으로 서로 얽혀버립니다.

> 💡 **핵심 요약**: 이상현상은 테이블이 잘못 설계되었을 때 발생하며, 그 원인은 속성들 사이에 불필요한 **종속 관계(Dependency)** 가 복잡하게 얽혀 있기 때문입니다. 이를 해결하는 것이 바로 **정규화(Normalization)** 입니다.

## 🏗️ 정규화(Normalization)란 무엇인가?

### 정규화의 정의와 목적

**정규화(Normalization)** 란, 이상현상을 제거하기 위해 하나의 테이블을 **논리적으로 올바른 여러 개의 테이블로 분해하는 과정**입니다. 여기서 중요한 포인트가 있습니다. 분해를 한다고 해서 속성들 사이의 **종속 관계(속성과 속성 사이의 결정 관계)** 를 없애버리는 것이 아닙니다. 종속 관계는 **보존하면서** 테이블을 적절히 나누는 것이 정규화의 핵심입니다.

마치 복잡하게 얽혀있는 전선들을 "서로 연결은 유지하되, 용도별로 깔끔하게 정리"하는 것에 비유할 수 있습니다. 연결 자체는 끊지 않고, 어지럽게 뒤섞인 것을 체계적으로 분류하는 작업이죠.

### 정규화의 단계

정규화는 단계별로 진행되며, 각 단계를 **정규형(Normal Form, NF)** 이라고 부릅니다. 단계가 높아질수록 더욱 엄격한 조건을 만족해야 합니다.

| 정규형 | 해결하는 문제 | 조건 |
|---|---|---|
| **제1정규형 (1NF)** | 비정규 릴레이션 → 원자값 | 모든 속성이 원자값(하나의 값)을 가져야 함 |
| **제2정규형 (2NF)** | 부분 함수 종속 제거 | 1NF + 완전 함수 종속 |
| **제3정규형 (3NF)** | 이행적 함수 종속 제거 | 2NF + 이행 종속 없음 |
| **BCNF** | 결정자가 키가 아닌 경우 해결 | 3NF + 모든 결정자가 후보키 |
| **제4정규형 (4NF)** | 다치 종속 제거 | BCNF + 다치 종속 없음 |
| **제5정규형 (5NF)** | 조인 종속 제거 | 4NF + 조인 종속 없음 |

:::info 정규화와 실무
실무에서는 일반적으로 **제3정규형(3NF)** 수준까지 정규화를 진행하면 대부분의 이상현상이 사라집니다. 4NF와 5NF는 이론적으로 중요하지만, 실제 프로젝트에서 마주치는 경우는 상대적으로 드뭅니다. 개념 정도만 이해하고 넘어가도 충분합니다.
:::

## 🔢 제1정규형 (1NF): 원자값의 세계

### 제1정규형의 정의

**제1정규형(1NF, First Normal Form)** 은 정규화의 가장 기본 단계입니다. 정의는 간단합니다.

> **어떤 릴레이션에 속한 모든 속성의 값이 전부 원자값(Atomic Value)을 가지면 제1정규형이다.**

**원자값(Atomic Value)** 이란 더 이상 나눌 수 없는 하나의 단일 값을 의미합니다. 쉽게 말해, 하나의 칸(셀)에 오직 하나의 값만 들어있어야 한다는 뜻입니다.

### 원자값이 아닌 경우 — 비정규형

아래처럼 하나의 속성에 여러 값이 들어있으면 **비정규형(Non-Normal Form)** 입니다.

```
❌ 비정규형 예시

| 학번 | 수강과목              |
|------|----------------------|
| A1   | 수학, 영어, 과학     |
| A2   | 국어, 체육           |
```

위 테이블에서 `수강과목` 열에는 여러 과목이 한꺼번에 들어있습니다. 이것은 원자값이 아닙니다.

```
✅ 제1정규형 — 원자값으로 변환

| 학번 | 수강과목 |
|------|---------|
| A1   | 수학    |
| A1   | 영어    |
| A1   | 과학    |
| A2   | 국어    |
| A2   | 체육    |
```

이제 각 칸에는 딱 하나의 값만 들어있습니다. 이것이 원자값이고, 이 상태가 제1정규형입니다.

:::note 관계형 데이터베이스의 기본 전제
관계형 데이터베이스(MySQL, Oracle, PostgreSQL 등)의 모든 테이블은 기본적으로 **제1정규형을 만족해야** 합니다. 하나의 셀에 여러 값이 들어가는 구조 자체가 관계형 DB의 설계 원칙에 어긋납니다.
:::

### 부분 함수 종속 — 1NF의 문제점

제1정규형을 만족하더라도 여전히 이상현상이 발생할 수 있습니다. 그 원인은 바로 **부분 함수 종속(Partial Functional Dependency)** 입니다.

**함수 종속(Functional Dependency)** 이란, A가 B를 결정하는 관계입니다. 예를 들어, 학번이 같으면 학생 이름이 반드시 같다면, "학번 → 이름"이라는 함수 종속이 성립합니다. 마치 주민등록번호가 같으면 반드시 같은 사람이듯, 결정자(왼쪽)가 같으면 종속자(오른쪽)도 반드시 같습니다.

**완전 함수 종속(Full Functional Dependency)** 은 기본키 전체가 어떤 속성을 결정할 때를 말합니다. 예를 들어, 기본키가 (학번, 과목번호)의 복합키일 때, (학번, 과목번호) 둘 다 있어야 "성적"이 결정된다면 이것이 완전 함수 종속입니다.

**부분 함수 종속(Partial Functional Dependency)** 은 기본키의 **일부분만**으로 어떤 속성이 결정될 때를 말합니다. 예를 들어, (학번, 과목번호)가 기본키인데, 학번만으로 지도교수와 학과가 결정된다면, 이것이 부분 함수 종속입니다.

```
[1NF의 부분 함수 종속 다이어그램]

기본키: (학번, 과목번호)

(학번, 과목번호) ──→ 성적   ← 완전 함수 종속 ✅
    학번         ──→ 지도교수 ← 부분 함수 종속 ⚠️
    학번         ──→ 학과    ← 부분 함수 종속 ⚠️
    지도교수      ──→ 학과    ← 이행 종속 (나중에 다룸)
```

### 1NF 실제 예제 — 수강지도 테이블

아래 `수강지도` 테이블을 예시로 봅시다.

```sql
-- 수강지도 테이블 (1NF 상태)
-- 기본키: (학번, 과목번호) 복합키

| 학번 | 지도교수 | 학과     | 과목번호 | 성적 |
|------|---------|---------|---------|------|
| 100  | P1      | 컴퓨터   | C413    | A    |
| 100  | P1      | 컴퓨터   | E412    | A    |
| 200  | P2      | 전기     | C123    | B    |
| 300  | P3      | 컴퓨터   | C312    | A    |
| 300  | P3      | 컴퓨터   | C324    | C    |
| 300  | P3      | 컴퓨터   | C413    | A    |
| 400  | P1      | 컴퓨터   | C312    | A    |
| 400  | P1      | 컴퓨터   | C324    | A    |
| 400  | P1      | 컴퓨터   | C413    | B    |
| 400  | P1      | 컴퓨터   | E412    | A    |
```

이 테이블에서 **함수 종속 관계**를 정리하면 다음과 같습니다.

- `(학번, 과목번호)` → `성적` (**완전 함수 종속**: 두 개 모두 있어야 성적이 결정됨)
- `학번` → `지도교수` (**부분 함수 종속**: 학번만으로 지도교수가 결정됨)
- `학번` → `학과` (**부분 함수 종속**: 학번만으로 학과가 결정됨)
- `지도교수` → `학과` (이행 종속: 지도교수로 학과가 결정됨)

이 테이블은 모든 값이 원자값이므로 **제1정규형**을 만족합니다. 하지만 부분 함수 종속이 존재하므로 이상현상이 발생합니다.

#### 😱 삽입 이상 (Insertion Anomaly)

```sql
-- ❌ 문제 상황: 500번 학생의 지도교수가 P4라는 정보를 삽입하고 싶다
-- 기본키는 (학번, 과목번호)이므로 과목번호가 NULL이면 삽입 불가!

INSERT INTO 수강지도 (학번, 지도교수, 학과, 과목번호, 성적)
VALUES (500, 'P4', '경영', NULL, NULL);
-- → 오류 발생! 기본키(과목번호)에 NULL 값은 개체 무결성 위반
```

500번 학생의 지도교수 정보를 등록하고 싶은데, 과목을 아직 수강하지 않았다면 과목번호가 없습니다. 그런데 과목번호는 기본키의 일부이므로 NULL이 될 수 없습니다. 결국 지도교수 정보를 삽입하려면 **없는 과목번호를 임의로 채워 넣는 수밖에 없는** 상황이 생깁니다. 이것이 **삽입 이상**입니다.

#### 😱 삭제 이상 (Deletion Anomaly)

```sql
-- ❌ 문제 상황: 200번 학생이 C123 과목을 수강 취소했다
-- 200번 학생이 듣는 과목이 C123 하나뿐이라면?

DELETE FROM 수강지도
WHERE 학번 = 200 AND 과목번호 = 'C123';

-- → 이 행을 지우면 P2 교수가 전기과 소속이라는 정보도 함께 사라짐!
-- 나는 수강 취소만 하고 싶었는데, 교수 정보까지 날아가 버린 것
```

200번 학생이 C123 과목 하나만 수강 중이었다면, 수강 취소 시 그 행 전체가 삭제됩니다. 그 결과 "P2 교수는 전기과 소속"이라는 정보도 함께 사라집니다. 이것이 **삭제 이상**입니다.

#### 😱 갱신 이상 (Update Anomaly)

```sql
-- ❌ 문제 상황: 400번 학생의 지도교수가 P1에서 P3로 변경되었다
-- 400번 학생이 4개 과목을 수강 중이므로, 4개 행을 모두 수정해야 함

UPDATE 수강지도 SET 지도교수 = 'P3' WHERE 학번 = 400;
-- 만약 4개 중 3개만 변경되고 1개를 빠뜨리면?
-- → 같은 학생인데 지도교수가 P3인 행과 P1인 행이 공존하는 데이터 불일치 발생
```

400번 학생이 4개 과목을 수강 중이어서 지도교수 정보가 4번 중복 저장되어 있습니다. 지도교수가 바뀌면 4곳을 모두 수정해야 하는데, 하나라도 빠뜨리면 **같은 학생의 지도교수가 두 가지로 나타나는 모순**이 생깁니다. 이것이 **갱신 이상**입니다.

:::warning 이상현상의 근본 원인
이 모든 이상현상의 원인은 하나입니다. **부분 함수 종속**, 즉 기본키의 일부(학번)만으로 다른 속성(지도교수, 학과)이 결정되는데, 그것을 같은 테이블 안에 섞어 놓았기 때문입니다.
:::

## 🔧 제2정규형 (2NF): 부분 함수 종속 제거

### 제2정규형의 정의

**제2정규형(2NF, Second Normal Form)** 은 다음 두 조건을 모두 만족해야 합니다.

1. **제1정규형(1NF)을 만족한다** — 모든 속성이 원자값
2. **완전 함수 종속이다** — 부분 함수 종속이 존재하지 않는다

쉽게 말해, **기본키의 일부만으로 결정되는 속성은 없어야 한다**는 뜻입니다. 기본키 전체가 있어야만 다른 속성들이 결정되는 상태를 만드는 것입니다.

### 1NF → 2NF 변환 방법

부분 함수 종속을 제거하려면 테이블을 분해해야 합니다. 분해 원칙은 간단합니다.

- **완전 함수 종속 관계**는 하나의 테이블로 유지
- **부분 함수 종속 관계**는 별도 테이블로 분리

```
[1NF 상태 — 하나의 테이블]
(학번, 과목번호) → 성적      ← 완전 함수 종속
학번            → 지도교수  ← 부분 함수 종속 (문제!)
학번            → 학과      ← 부분 함수 종속 (문제!)

         ↓ 분해 ↓

[2NF — 두 개의 테이블로 분리]

테이블 1: 수강(학번, 과목번호, 성적)
  (학번, 과목번호) → 성적  ← 완전 함수 종속 ✅

테이블 2: 지도(학번, 지도교수, 학과)
  학번 → 지도교수  ← 완전 함수 종속 ✅
  학번 → 학과      ← 완전 함수 종속 ✅
  지도교수 → 학과  ← (이행 종속 — 아직 남아있음 ⚠️)
```

```sql
-- ✅ 2NF로 분해된 두 테이블

-- 테이블 1: 수강 릴레이션
-- 기본키: (학번, 과목번호)
| 학번 | 과목번호 | 성적 |
|------|---------|------|
| 100  | C413    | A    |
| 100  | E412    | A    |
| 200  | C123    | B    |
| 300  | C312    | A    |
| 400  | C312    | A    |
-- 이 테이블은 이상현상 없음 ✅

-- 테이블 2: 지도 릴레이션
-- 기본키: 학번
| 학번 | 지도교수 | 학과   |
|------|---------|-------|
| 100  | P1      | 컴퓨터 |
| 200  | P2      | 전기   |
| 300  | P3      | 컴퓨터 |
| 400  | P1      | 컴퓨터 |
-- 이 테이블에는 아직 이행 종속이 존재 ⚠️
-- 학번 → 지도교수 → 학과 (이행 종속)
```

### 2NF에도 여전히 남아있는 문제 — 이행 종속

2NF로 분해하면 부분 함수 종속은 사라집니다. 하지만 `지도` 테이블을 보면 새로운 문제가 보입니다.

- `학번` → `지도교수` (학번이 지도교수를 결정)
- `지도교수` → `학과` (지도교수가 학과를 결정)
- 따라서 `학번` → `학과` (학번이 학과를 결정)

이처럼 A가 B를 결정하고, B가 C를 결정하면, A가 C를 결정하는 관계를 **이행적 함수 종속(Transitive Functional Dependency)** 이라고 합니다. 마치 A → B → C 처럼 거쳐가는 종속 관계입니다.

이 이행 종속 때문에 2NF 상태의 `지도` 테이블에서도 이상현상이 발생합니다.

#### 2NF의 이상현상 예시

**삽입 이상**: 새로운 교수 P4가 경영학과에 부임했습니다. 그런데 아직 담당 학생이 없습니다.

```sql
-- ❌ 삽입 불가: 기본키(학번)가 없으면 삽입이 안 됨
INSERT INTO 지도 (학번, 지도교수, 학과)
VALUES (NULL, 'P4', '경영');
-- → 기본키에 NULL 불가! 삽입 이상 발생
```

**삭제 이상**: 200번 학생이 자퇴했습니다.

```sql
-- ❌ 200번 학생 삭제 시, P2 교수가 전기과라는 정보도 함께 삭제됨
DELETE FROM 지도 WHERE 학번 = 200;
-- → 나는 학생 정보만 지우고 싶었는데, 교수 소속 정보도 날아감!
```

**갱신 이상**: P1 교수의 소속 학과가 컴퓨터에서 전기과로 바뀌었습니다.

```sql
-- ❌ P1 교수를 담당 학생 수만큼 모두 변경해야 함
-- 100번, 400번 학생의 지도교수가 P1이므로 2번 수정 필요
UPDATE 지도 SET 학과 = '전기' WHERE 지도교수 = 'P1';
-- 만약 한 행만 수정되면? → 같은 P1 교수인데 학과가 다르게 나타남!
```

> 이행 종속이 존재하는 것이 제2정규형의 특징입니다. 이 이행 종속을 제거해야 제3정규형으로 올라갈 수 있습니다.

## 🧹 제3정규형 (3NF)과 BCNF: 이행 종속과 결정자 문제 해결

### 제3정규형의 정의

**제3정규형(3NF, Third Normal Form)** 은 다음 조건을 모두 만족해야 합니다.

1. **제2정규형(2NF)을 만족한다** — 원자값, 완전 함수 종속
2. **이행적 함수 종속이 없다** — A → B → C 형태의 종속 관계가 없어야 함

즉, 제3정규형은 **원자값 + 부분 함수 종속 없음 + 이행 종속 없음**의 세 조건을 모두 만족하는 상태입니다.

### 2NF → 3NF 변환 방법

이행 종속을 제거하려면 이행 종속을 이루는 관계를 별도 테이블로 분리합니다.

```
[2NF 상태 — 지도 테이블]
학번 → 지도교수  (X → Y)
지도교수 → 학과  (Y → Z)
따라서 학번 → 학과 이행 종속 존재 ⚠️

         ↓ 분해 ↓

[3NF — 두 개의 테이블로 분리]

테이블 1: 학생지도(학번, 지도교수)
  학번 → 지도교수  ✅

테이블 2: 교수학과(지도교수, 학과)
  지도교수 → 학과  ✅
```

```sql
-- ✅ 3NF로 분해된 테이블들

-- 테이블 1: 학생지도 릴레이션
-- 기본키: 학번
| 학번 | 지도교수 |
|------|---------|
| 100  | P1      |
| 200  | P2      |
| 300  | P3      |
| 400  | P1      |

-- 테이블 2: 교수학과 릴레이션
-- 기본키: 지도교수
| 지도교수 | 학과   |
|---------|-------|
| P1      | 컴퓨터 |
| P2      | 전기   |
| P3      | 컴퓨터 |
```

이제 새 교수 P4(전기과)가 부임하면, 학생 정보 없이도 `교수학과` 테이블에 바로 삽입할 수 있습니다.

```sql
-- ✅ 삽입 이상 해결: 학생 없이도 교수 정보 등록 가능
INSERT INTO 교수학과 (지도교수, 학과) VALUES ('P4', '전기');
-- → 아무 문제 없음! ✅

-- ✅ 갱신 이상 해결: P1 교수 학과 변경 시 한 곳만 바꾸면 됨
UPDATE 교수학과 SET 학과 = '전기' WHERE 지도교수 = 'P1';
-- → 딱 한 번만 수정하면 됨! ✅
```

:::tip 3NF까지 정규화하면 충분한가?
일반적인 실무 환경에서는 **제3정규형까지 정규화하면 대부분의 이상현상을 제거할 수 있습니다**. 하지만 특수한 상황(후보키가 여러 개이고 중첩되는 경우)에서는 3NF를 만족하면서도 이상현상이 발생할 수 있습니다. 이때 필요한 것이 BCNF입니다.
:::

### BCNF (Boyce-Codd Normal Form): 강한 제3정규형

**BCNF(보이스-코드 정규형)** 는 제3정규형보다 더 엄격한 조건을 요구합니다. 3NF가 후보키가 하나이고 단순한 구조를 가정하는 데 반해, BCNF는 다음과 같은 복잡한 경우를 해결합니다.

- 후보키(테이블을 유일하게 식별할 수 있는 속성의 최소 집합)가 **여러 개**인 경우
- 후보키들이 **2개 이상의 속성**으로 구성된 경우
- 후보키들이 **서로 중첩(겹치는 속성 포함)** 되는 경우

BCNF의 핵심 규칙은 딱 하나입니다.

> **💡 모든 결정자(X → Y에서 X)는 반드시 후보키(Candidate Key)여야 한다.**

이행 종속이니 부분 함수 종속이니 복잡하게 따지는 것이 아니라, **"결정자인데 키가 아니면 그 결정자를 키로 만들어서 분해하면 된다"** 는 간단한 원칙입니다.

### BCNF 실제 예제

아래 `수강교수` 테이블을 봅시다.

```sql
-- 수강교수 테이블
-- 기본키: (학번, 과목)
-- 각 교수는 오직 한 과목만 담당 (교수 → 과목)
-- 한 학생이 같은 과목을 여러 교수에게 들을 수 없음 (학번, 과목 → 교수)

| 학번 | 과목       | 교수 |
|------|----------|------|
| 100  | 프로그래밍 | P1   |
| 100  | 자료구조   | P2   |
| 200  | 프로그래밍 | P1   |
| 200  | 자료구조   | P3   |
| 300  | 프로그래밍 | P1   |
| 300  | 자료구조   | P2   |
```

함수 종속 관계를 분석하면 다음과 같습니다.

```
(학번, 과목) → 교수   ← 완전 함수 종속 (기본키)
교수        → 과목   ← 결정자! 그런데 교수는 기본키가 아님 ⚠️
```

- 부분 함수 종속? 없습니다. (기본키 전체가 교수를 결정)
- 이행 종속? 없습니다. (A → B → C 구조가 아님)
- **그러면 3NF를 만족합니다.**

그런데도 이상현상이 발생합니다!

**삽입 이상**: 새 교수 P5가 자료구조를 담당하게 되었습니다.

```sql
-- ❌ P5가 자료구조 담당이라는 정보를 삽입하려면 학번이 필요함
INSERT INTO 수강교수 (학번, 과목, 교수) VALUES (NULL, '자료구조', 'P5');
-- → 기본키(학번)에 NULL 불가! 삽입 이상 발생
```

**삭제 이상**: 100번 학생이 프로그래밍 과목을 취소했습니다.

```sql
-- ❌ 100번 학생의 프로그래밍 행을 지우면
-- P1 교수가 프로그래밍을 담당한다는 정보가 남은 행들에 있지만
-- 만약 100번이 유일한 학생이었다면 P1의 담당과목 정보도 사라짐
DELETE FROM 수강교수 WHERE 학번 = 100 AND 과목 = '프로그래밍';
```

**갱신 이상**: P1 교수의 담당 과목이 변경됩니다.

```sql
-- ❌ P1 교수가 여러 행에 등장하므로, 모든 행을 수정해야 함
-- 일부만 수정되면 같은 P1 교수의 담당 과목이 다르게 표시됨
UPDATE 수강교수 SET 과목 = '데이터베이스' WHERE 교수 = 'P1';
```

이 이상현상의 원인은 **"교수"가 "과목"의 결정자인데, 교수가 기본키(후보키)가 아니기 때문**입니다. BCNF 원칙에 따라 모든 결정자는 후보키여야 하므로, "교수"를 후보키로 만들어 분해합니다.

```
[BCNF 분해]

현재: (학번, 과목, 교수)
교수 → 과목 (결정자인 교수가 키가 아님 → 분해!)

         ↓ 분해 ↓

테이블 1: 수강(학번, 교수)
  기본키: (학번, 교수)

테이블 2: 교수과목(교수, 과목)
  기본키: 교수
```

```sql
-- ✅ BCNF로 분해된 테이블

-- 테이블 1: 수강 릴레이션
-- 기본키: (학번, 교수)
| 학번 | 교수 |
|------|------|
| 100  | P1   |
| 100  | P2   |
| 200  | P1   |
| 200  | P3   |
| 300  | P1   |
| 300  | P2   |

-- 테이블 2: 교수과목 릴레이션
-- 기본키: 교수
| 교수 | 과목       |
|------|----------|
| P1   | 프로그래밍 |
| P2   | 자료구조   |
| P3   | 자료구조   |

-- ✅ 이제 P5 교수가 자료구조 담당이라는 정보를 학생 없이도 삽입 가능!
INSERT INTO 교수과목 (교수, 과목) VALUES ('P5', '자료구조');
```

이렇게 BCNF로 분해하면 더 이상 이상현상이 발생하지 않습니다.

:::note BCNF vs 3NF
BCNF는 3NF보다 **더 강한 조건**을 요구하기 때문에 "강한 제3정규형(Strong 3NF)"이라고도 불립니다. 3NF를 만족한다고 해서 반드시 BCNF를 만족하는 것은 아니지만, BCNF를 만족하면 반드시 3NF도 만족합니다.
:::

## 🌐 제4정규형(4NF)과 제5정규형(5NF): 고급 정규화

### 제4정규형 (4NF): 다치 종속 제거

:::info 학습 난이도 안내
4NF와 5NF는 실무에서 자주 마주치는 개념은 아닙니다. 이론적으로 이런 개념이 있다는 것을 이해하는 수준으로 학습하면 충분합니다.
:::

**다치 종속(Multi-Valued Dependency, MVD)** 이란, 하나의 결정자가 여러 개의 값을 가지는 종속자를 결정하는 관계입니다. 함수 종속은 결정자 하나에 종속자 하나(1:1)인데, 다치 종속은 결정자 하나에 종속자 여러 개(1:N)인 경우를 말합니다.

다치 종속이 성립하는 조건은 다음과 같습니다.

- 릴레이션에 **속성이 3개 이상** 있을 때
- `A, B, C` 세 속성이 있을 때, `A`의 값이 같으면 그에 대응하는 `B` 값의 집합이 `C`와 **독립적으로** 결정될 때
- 즉, `A →→ B` (A가 B를 다치 결정) 이고 `A →→ C` (A가 C를 다치 결정)이면서 **B와 C는 서로 무관한** 경우

**실제 예시**: 한 과목을 여러 교수가 가르칠 수 있고, 한 과목에 여러 교재가 사용될 수 있습니다. 그리고 교수와 교재는 서로 독립적입니다.

```sql
-- 다치 종속이 있는 테이블 (비정규형)
-- 기본키: (과목, 교수, 교재) — 세 가지 모두 있어야 유일 식별 가능

| 과목      | 교수 | 교재 |
|---------|------|------|
| 자료구조  | P1   | T1   |
| 자료구조  | P1   | T2   |
| 자료구조  | P2   | T1   |
| 자료구조  | P2   | T2   |
| 데이터베이스 | P3 | T1   |
| 데이터베이스 | P3 | T2   |
| 데이터베이스 | P3 | T3   |

-- 과목 →→ 교수  (과목이 교수를 다치 결정)
-- 과목 →→ 교재  (과목이 교재를 다치 결정)
-- 교수와 교재는 서로 독립 (무관)
```

이 상태에서도 이상현상이 발생합니다.

**삽입 이상**: 데이터베이스 과목에 새 교수 P4가 배정되었습니다. 교재가 3권이므로 3번 삽입해야 합니다.

```sql
-- ❌ P4가 데이터베이스를 가르친다는 정보를 넣으려면 교재 수만큼 삽입해야 함
INSERT INTO 과목교수교재 VALUES ('데이터베이스', 'P4', 'T1');
INSERT INTO 과목교수교재 VALUES ('데이터베이스', 'P4', 'T2');
INSERT INTO 과목교수교재 VALUES ('데이터베이스', 'P4', 'T3');
-- → 교재 수만큼 반복 삽입해야 하는 비효율 발생
```

해결 방법은 다치 종속을 이루는 두 관계를 별도 테이블로 분리하는 것입니다.

```sql
-- ✅ 4NF로 분해

-- 테이블 1: 과목교수 (과목 →→ 교수)
| 과목       | 교수 |
|---------|------|
| 자료구조   | P1   |
| 자료구조   | P2   |
| 데이터베이스 | P3 |

-- 테이블 2: 과목교재 (과목 →→ 교재)
| 과목       | 교재 |
|---------|------|
| 자료구조   | T1   |
| 자료구조   | T2   |
| 데이터베이스 | T1 |
| 데이터베이스 | T2 |
| 데이터베이스 | T3 |
```

이렇게 분리하면 P4 교수를 등록할 때 `과목교수` 테이블에 한 행만 추가하면 됩니다. 교재와는 무관하게 독립적으로 관리할 수 있습니다.

### 제5정규형 (5NF): 조인 종속 제거

**제5정규형(5NF, Fifth Normal Form)** 은 **조인 종속(Join Dependency)** 을 제거하는 단계입니다. 조인 종속이란, 릴레이션을 여러 개로 분해했다가 다시 조인(JOIN)했을 때 원래 릴레이션이 정확히 복원되어야 하는 성질과 관련됩니다.

특히 **후보키가 아닌 속성들로 조인을 수행할 때**, 원래 데이터에 없던 가짜 데이터(가짜 튜플, Spurious Tuple)가 생성될 수 있습니다. 이를 방지하기 위해, **반드시 후보키(기본키 또는 외래키)를 통해서만 조인이 이루어지도록** 해야 한다는 것이 5NF의 핵심입니다.

```
[조인 종속 문제 예시]

원본 릴레이션: (S, P, C)
→ (S, P), (P, C), (C, S)로 3분해
→ 이 세 테이블을 다시 조인하면?

(S, P) ⋈ (P, C) ⋈ (C, S)의 결과에
원래 없던 (S2, P1) 같은 행이 생겨날 수 있음 → 이것이 가짜 튜플(Spurious Tuple)
```

5NF는 이러한 문제를 방지하기 위해, **분해된 릴레이션들이 반드시 후보키를 포함하도록** 설계하고, **후보키를 통해서만 조인에 참여**하도록 만듭니다. 이렇게 하면 조인 결과가 항상 원본 릴레이션과 동일하게 복원됩니다.

:::note 5NF의 실용성
5NF는 이론적으로 완벽한 정규화의 마지막 단계이지만, 실제 데이터베이스 설계에서 5NF까지 적용하는 경우는 매우 드뭅니다. 개념을 이해하고 있다는 것만으로 충분합니다.
:::

## ⚠️ 주의사항과 실전 팁

### 정규화의 장점과 단점

정규화는 만능이 아닙니다. 단계가 높아질수록 테이블이 더 많이 분해됩니다. 이는 다음과 같은 **트레이드오프(Trade-off)** 를 만듭니다.

| 구분 | 정규화 단계 높을수록 | 정규화 단계 낮을수록 |
|---|---|---|
| **데이터 무결성** | ✅ 이상현상 적음, 일관성 높음 | ❌ 이상현상 많음, 불일치 위험 |
| **조인 횟수** | ❌ 테이블 많아서 JOIN 많이 필요 | ✅ 테이블 적어서 JOIN 적음 |
| **성능(쿼리 속도)** | ❌ 많은 JOIN으로 느려질 수 있음 | ✅ 간단한 SELECT로 빠를 수 있음 |
| **저장 공간** | ✅ 중복 데이터 없어 공간 절약 | ❌ 중복 데이터로 공간 낭비 |

:::warning 과도한 정규화 주의
정규화를 지나치게 많이 하면 오히려 성능이 저하될 수 있습니다. 실무에서는 **3NF 또는 BCNF 수준**이 적절한 균형점인 경우가 많습니다. 성능이 중요한 경우에는 의도적으로 정규화를 일부 되돌리는 **역정규화(Denormalization)** 를 적용하기도 합니다.
:::

### 정규화 단계별 핵심 체크리스트

```
✅ 1NF 체크: 모든 속성 값이 원자값(하나의 값)인가?
             → NO라면: 행을 나누어 원자값으로 분리

✅ 2NF 체크: 기본키의 일부로만 결정되는 속성이 없는가?
             → NO라면: 부분 함수 종속 관계를 별도 테이블로 분리

✅ 3NF 체크: A → B → C 형태의 이행 종속이 없는가?
             → NO라면: 이행 종속 관계를 별도 테이블로 분리

✅ BCNF 체크: 모든 결정자가 후보키인가?
              → NO라면: 결정자를 후보키로 만들어 분리

✅ 4NF 체크: 다치 종속이 없는가?
             → NO라면: 다치 종속을 이루는 두 관계를 각각 별도 테이블로 분리

✅ 5NF 체크: 조인 종속이 없는가?
             → NO라면: 후보키를 포함하는 릴레이션으로만 분해
```

:::danger 종속성을 제거하면 안 됩니다
정규화는 종속 관계 자체를 없애는 것이 아닙니다! **종속 관계는 반드시 보존**해야 합니다. 단지 테이블을 분해할 뿐이며, 분해된 테이블들을 JOIN하면 원래 데이터를 항상 복원할 수 있어야 합니다. 이를 **무손실 분해(Lossless Decomposition)** 라고 합니다.
:::

### 정규형 간의 관계 요약

```
비정규형
   ↓ (원자값으로 만들기)
1NF — 원자값 O, 부분 함수 종속 존재
   ↓ (부분 함수 종속 제거)
2NF — 원자값 O, 부분 함수 종속 X, 이행 종속 존재
   ↓ (이행 종속 제거)
3NF — 원자값 O, 부분 함수 종속 X, 이행 종속 X
   ↓ (결정자를 모두 후보키로)
BCNF — 모든 결정자가 후보키
   ↓ (다치 종속 제거)
4NF — 다치 종속 X
   ↓ (조인 종속 제거)
5NF — 조인 종속 X (가장 완전한 형태)
```

## 📝 핵심 정리

- **이상현상(Anomaly)** 은 테이블이 잘못 설계되었을 때 데이터 삽입·삭제·수정 시 발생하는 비정상적인 부작용이다.
- 이상현상의 종류는 **삽입 이상, 삭제 이상, 갱신 이상** 세 가지다.
- **정규화(Normalization)** 는 이상현상을 제거하기 위해 테이블을 논리적으로 분해하는 과정이며, 종속 관계는 보존된다.
- **제1정규형(1NF)**: 모든 속성이 원자값(단일 값)을 가지는 상태이며, 부분 함수 종속이 존재할 수 있다.
- **제2정규형(2NF)**: 1NF를 만족하고 **부분 함수 종속을 제거**한 상태이며, 이행 종속이 존재할 수 있다.
- **제3정규형(3NF)**: 2NF를 만족하고 **이행적 함수 종속을 제거**한 상태로, 실무에서 일반적으로 목표하는 수준이다.
- **BCNF(강한 제3정규형)**: 3NF보다 엄격하며, **모든 결정자를 후보키**로 만들어 이상현상을 제거한다.
- **제4정규형(4NF)**: **다치 종속(1:N 결정 관계)** 을 제거하며, 독립적인 다치 종속 관계를 별도 테이블로 분리한다.
- **제5정규형(5NF)**: **조인 종속을 제거**하며, 후보키를 통해서만 조인이 이루어지도록 설계한다.
- 정규화 단계가 높을수록 **무결성은 향상**되지만, **조인 횟수 증가로 성능이 저하**될 수 있다.
- 실무에서는 성능과 무결성의 균형을 고려해 **3NF 또는 BCNF** 수준의 정규화가 일반적으로 권장된다.
