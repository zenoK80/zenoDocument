# 🗂️ 정규화(Normalization): 1정규형부터 5정규형까지 완벽 이해하기

데이터베이스를 공부하다 보면 "정규화"라는 단어를 자주 마주치게 됩니다. 처음에는 복잡하고 어렵게 느껴지지만, 차근차근 이해하다 보면 "아, 이게 결국 데이터를 깔끔하게 정리하는 방법이구나!"라는 깨달음을 얻게 됩니다. 이 문서에서는 정규화의 개념부터 시작해서 1정규형, 2정규형, 3정규형, BCNF, 4정규형, 5정규형까지 초보자도 이해할 수 있도록 아주 자세히 설명하겠습니다.

---

## 🧹 정규화란 무엇인가? — 이상현상을 해결하는 과정

### 왜 정규화가 필요한가?

정규화를 이해하려면 먼저 **이상현상(Anomaly)** 이 무엇인지 알아야 합니다. 이상현상이란, 데이터베이스 테이블이 잘못 설계되었을 때 데이터를 삽입하거나, 삭제하거나, 수정할 때 생기는 **의도치 않은 문제들**을 말합니다.

마치 물건을 정리하지 않고 한 방에 다 쑤셔 넣은 상태를 생각해 보세요. 양말을 찾으려다 옷이 다 쏟아지거나, 셔츠 하나를 꺼냈더니 다른 물건들이 엉켜서 같이 딸려 나오는 상황과 비슷합니다. 이런 **엉킴 현상**이 데이터베이스에서는 이상현상으로 나타납니다.

이상현상에는 세 가지 종류가 있습니다:

| 이상현상 종류 | 설명 | 실생활 비유 |
|---|---|---|
| **삽입 이상(Insertion Anomaly)** | 꼭 필요하지 않은 데이터까지 함께 입력해야만 새 데이터를 넣을 수 있는 문제 | 회원가입을 하려면 반드시 구매 이력이 있어야 하는 상황 |
| **삭제 이상(Deletion Anomaly)** | 하나의 데이터를 지웠더니 필요한 다른 데이터까지 함께 사라지는 문제 | 영수증을 버렸더니 거래처 전화번호도 같이 사라지는 상황 |
| **갱신 이상(Update Anomaly)** | 데이터를 수정할 때 중복된 곳을 모두 바꾸지 않으면 데이터가 서로 불일치하는 문제 | 주소가 여러 곳에 적혀 있는데 한 곳만 바꿔서 정보가 엇갈리는 상황 |

> **정규화(Normalization)** 란, 이러한 이상현상을 해결하기 위해 하나의 릴레이션(테이블)을 **속성 간의 종속관계(함수 종속성)를 보존하면서** 더 작은 릴레이션들로 분해해 나가는 과정입니다.

:::info 종속성은 사라지지 않는다
정규화를 하면 테이블이 여러 개로 나뉘지만, **속성과 속성 사이의 종속관계(종속성)는 그대로 보존**됩니다. 즉, "학번이 지도교수를 결정한다"는 관계는 분해 이후에도 유지됩니다. 단지 그 관계를 담는 테이블이 별도로 분리될 뿐입니다.
:::

### 정규형의 종류와 흐름

정규화는 단계적으로 진행됩니다. 각 단계를 **정규형(Normal Form)** 이라고 부릅니다.

```
비정규 릴레이션
    ↓ 원자값만 갖도록 변환
  1정규형 (1NF) — 부분함수 종속 존재
    ↓ 부분함수 종속 제거
  2정규형 (2NF) — 이행적 함수 종속 존재
    ↓ 이행적 함수 종속 제거
  3정규형 (3NF) — 결정자가 키가 아닌 경우 존재 가능
    ↓ 결정자를 모두 키로 만들기
  BCNF (보이스-코드 정규형) — 다치 종속 존재 가능
    ↓ 다치 종속 제거
  4정규형 (4NF) — 조인 종속 존재 가능
    ↓ 조인 종속 제거
  5정규형 (5NF) — 완전 정규화
```

이제 각 단계를 하나씩 자세하게 살펴보겠습니다.

## 1️⃣ 1정규형(1NF): 원자값으로 구성된 릴레이션

### 원자값(Atomic Value)이란?

**1정규형(First Normal Form, 1NF)** 의 핵심은 바로 **원자값(Atomic Value)** 입니다. 원자값이란, 하나의 속성(열, 컬럼)이 **오직 하나의 값만**을 가지는 것을 말합니다.

"원자(Atom)"라는 단어는 물질의 가장 작은 단위를 뜻합니다. 즉, 더 이상 쪼갤 수 없는 하나의 값이라는 의미입니다. 마치 계란 한 판에 계란이 30개 들어 있다고 할 때, 한 칸에 계란 하나만 들어 있어야 하는데, 한 칸에 계란 3개를 억지로 우겨 넣는 것은 원자값이 아닌 상태와 같습니다.

:::warning 원자값을 위반하는 예시
아래 테이블을 보세요. `성적` 속성 하나에 여러 개의 값이 들어 있습니다. 이것은 **1정규형을 위반**합니다.

| 학번 | 성적 |
|---|---|
| A1 | 10, 20 |
| A2 | 30, 40 |

한 칸(하나의 속성값)에 여러 값이 들어 있으면 안 됩니다!
:::

올바른 1정규형은 이렇게 되어야 합니다:

| 학번 | 과목 | 성적 |
|---|---|---|
| A1 | 국어 | 10 |
| A1 | 수학 | 20 |
| A2 | 국어 | 30 |
| A2 | 수학 | 40 |

각 셀(칸)에는 **단 하나의 값**만 존재합니다. 이것이 원자값이고, 이 조건을 만족하면 1정규형입니다.

> **1정규형의 정의**: 릴레이션에 속한 모든 속성의 도메인(값의 범위)이 원자값(단일 값)으로만 구성되어 있는 상태

:::note 관계형 데이터베이스의 기본 조건
관계형 데이터베이스의 모든 테이블은 기본적으로 1정규형을 만족해야 합니다. 국어 점수 칸에 "30, 70"처럼 두 개의 값이 동시에 들어가는 경우는 관계형 DB에서 허용되지 않습니다.
:::

### 부분함수 종속(Partial Functional Dependency)이란?

1정규형은 원자값 조건을 만족하지만, **부분함수 종속(Partial Functional Dependency)** 이라는 문제를 가지고 있을 수 있습니다. 부분함수 종속이 바로 이상현상의 주요 원인입니다.

먼저 **완전함수 종속**과 **부분함수 종속**을 비교해 보겠습니다.

| 개념 | 설명 | 예시 |
|---|---|---|
| **완전함수 종속** | 기본키 전체가 있어야만 어떤 속성이 결정되는 관계 | (학번 + 과목번호) → 성적 |
| **부분함수 종속** | 기본키의 일부분만으로도 어떤 속성이 결정되는 관계 | 학번 → 지도교수 (기본키의 일부인 학번만으로 결정) |

마치 금고 문을 열려면 두 개의 열쇠(학번 + 과목번호)가 모두 필요한 경우가 완전함수 종속이고, 열쇠 하나(학번)만으로도 금고 일부가 열려버리는 경우가 부분함수 종속입니다.

```
[완전함수 종속]
(학번, 과목번호) ──────────→ 성적  ✅ 기본키 전체가 필요

[부분함수 종속]
(학번, 과목번호) ─┐
                  └→ (학번만으로) ──→ 지도교수  ⚠️ 기본키 일부만으로 결정
```

### 1정규형 예제: 수강지도 릴레이션

실제 예제를 통해 1정규형을 살펴보겠습니다. 아래는 `수강지도` 릴레이션입니다.

**테이블 구조:**

| 학번 | 지도교수 | 학과 | 과목번호 | 성적 |
|---|---|---|---|---|
| 100 | P1 | 컴퓨터 | C100 | A |
| 100 | P1 | 컴퓨터 | C200 | B |
| 200 | P2 | 전기 | C123 | A |
| 300 | P3 | 컴퓨터 | C100 | C |
| 300 | P3 | 컴퓨터 | C200 | A |
| 300 | P3 | 컴퓨터 | C300 | B |
| 400 | P1 | 컴퓨터 | C100 | A |
| 400 | P1 | 컴퓨터 | C200 | B |
| 400 | P1 | 컴퓨터 | C300 | A |
| 400 | P1 | 컴퓨터 | C400 | B |

- **기본키(Primary Key)**: `학번 + 과목번호` (복합키)
- **함수 종속 관계**:
  - `(학번, 과목번호)` → `성적` (완전함수 종속)
  - `학번` → `지도교수` (부분함수 종속! ⚠️)
  - `학번` → `학과` (부분함수 종속! ⚠️)
  - `지도교수` → `학과`

**함수 종속 다이어그램:**

```
학번 ──────────────→ 지도교수
학번 ──────────────→ 학과
지도교수 ──────────→ 학과
(학번, 과목번호) ──→ 성적
```

이 다이어그램에서 `학번`만으로도 `지도교수`와 `학과`가 결정됩니다. 이것이 **부분함수 종속**입니다. 기본키는 `(학번, 과목번호)`인데, 기본키의 일부인 `학번` 혼자서 다른 속성들을 결정하고 있으니까요.

### 1정규형의 이상현상 분석

이제 이 테이블에서 이상현상이 실제로 발생하는지 확인해 보겠습니다.

#### ❌ 삽입 이상 (Insertion Anomaly)

**상황**: 500번 학생의 지도교수가 P4라는 정보를 입력하고 싶습니다.

```sql
-- 500번 학생의 지도교수 정보를 삽입하려고 시도
INSERT INTO 수강지도 (학번, 지도교수, 학과)
VALUES (500, 'P4', '경영');

-- ❌ 오류 발생!
-- 기본키인 '과목번호'가 NULL이 될 수 없어서 삽입 불가!
-- 개체 무결성(Entity Integrity) 위반
```

학생의 지도교수 정보를 추가하려면 반드시 **과목번호**가 있어야 합니다. 하지만 500번 학생은 아직 수강 신청을 하지 않았습니다. 결국 **존재하지도 않는 과목번호를 억지로 넣어야만** 삽입이 가능합니다. 이것이 삽입 이상입니다.

#### ❌ 삭제 이상 (Deletion Anomaly)

**상황**: 200번 학생이 C123 과목의 수강을 철회했습니다.

| 학번 | 지도교수 | 학과 | 과목번호 | 성적 |
|---|---|---|---|---|
| ~~200~~ | ~~P2~~ | ~~전기~~ | ~~C123~~ | ~~A~~ |

200번 학생은 C123 과목 하나만 수강하고 있었습니다. 이 행(row)을 삭제하면 수강 정보뿐만 아니라 **P2 교수가 전기과 소속이라는 정보도 함께 사라집니다.** 나는 과목 철회만 하려고 했는데, 중요한 교수 정보가 연쇄적으로 삭제되어 버립니다. 이것이 삭제 이상입니다.

#### ❌ 갱신 이상 (Update Anomaly)

**상황**: 400번 학생의 지도교수가 P1에서 P3로 변경되었습니다.

400번 학생은 과목을 4개 수강하고 있으므로, 테이블에 4개의 행이 있습니다. 지도교수를 바꾸려면 **4개의 행을 모두 수정**해야 합니다. 만약 3개만 수정하고 1개를 빠뜨리면:

| 학번 | 지도교수 | 학과 | 과목번호 | 성적 |
|---|---|---|---|---|
| 400 | **P3** | 컴퓨터 | C100 | A |
| 400 | **P3** | 컴퓨터 | C200 | B |
| 400 | **P3** | 컴퓨터 | C300 | A |
| 400 | **P1** | 컴퓨터 | C400 | B |  ← 수정 누락!

같은 400번 학생인데 지도교수가 P3이기도 하고 P1이기도 한 **데이터 불일치(데이터 모순)** 가 발생합니다. 이것이 갱신 이상입니다.

:::danger 이상현상의 근본 원인
이 모든 이상현상의 근본 원인은 **부분함수 종속** 때문입니다. `학번`만으로도 `지도교수`와 `학과`가 결정되는데, `(학번, 과목번호)`를 기본키로 하는 하나의 테이블에 이 모든 정보가 섞여 있기 때문입니다. 해결책은 부분함수 종속을 **제거**하는 것입니다!
:::

### 1정규형 → 2정규형으로 변환하기

부분함수 종속을 제거하려면 **릴레이션을 분해(Decompose)** 해야 합니다.

```
[분해 전 — 1정규형]
수강지도(학번, 지도교수, 학과, 과목번호, 성적)

[분해 후 — 2정규형]
수강(학번, 과목번호, 성적)        ← 완전함수 종속
지도(학번, 지도교수, 학과)        ← 완전함수 종속
```

- **`수강` 릴레이션**: `(학번, 과목번호)` → `성적` — 기본키 전체가 성적을 결정 (완전함수 종속 ✅)
- **`지도` 릴레이션**: `학번` → `지도교수`, `학번` → `학과` — 기본키인 학번이 혼자서 결정 (완전함수 종속 ✅)

이렇게 분해하면 부분함수 종속이 제거되고, **2정규형**이 됩니다.

## 2️⃣ 2정규형(2NF): 부분함수 종속을 제거한 상태

### 2정규형의 정의

> **2정규형(Second Normal Form, 2NF)**: 1정규형을 만족하면서 (원자값으로 구성), 모든 비주요 속성이 기본키에 **완전함수 종속**되어 있는 상태. 즉, 부분함수 종속이 없는 상태.

쉽게 말하면, **"기본키의 일부분만으로는 어떤 속성도 결정할 수 없다"** 는 것입니다.

:::tip 2정규형 체크리스트
- ✅ 원자값으로만 구성되어 있는가?
- ✅ 부분함수 종속이 없는가? (비주요 속성이 기본키 전체에 종속되는가?)
- ⚠️ 이행적 함수 종속은 있을 수 있음
:::

### 이행적 함수 종속(Transitive Functional Dependency)이란?

2정규형이 되면 부분함수 종속은 사라지지만, 새로운 문제인 **이행적 함수 종속(Transitive Functional Dependency)** 이 남아 있을 수 있습니다.

이행적 함수 종속이란, `X → Y`이고 `Y → Z`이면, `X → Z`도 성립하는 관계를 말합니다. 마치 "A가 B의 상사이고, B가 C의 상사이면, A가 C의 상사이기도 하다"는 논리와 같습니다.

앞서 분해한 `지도` 릴레이션을 살펴보겠습니다.

```
지도(학번, 지도교수, 학과)

학번 → 지도교수  (X → Y)
지도교수 → 학과  (Y → Z)
∴ 학번 → 학과   (X → Z, 이행적 함수 종속!)
```

`학번`이 `지도교수`를 결정하고, `지도교수`가 `학과`를 결정하므로, 결국 `학번`이 `학과`를 결정합니다. 이것이 이행적 함수 종속입니다.

### 2정규형의 이상현상 분석

2정규형이 되었지만, 이행적 함수 종속이 있는 `지도` 릴레이션에서는 여전히 이상현상이 발생합니다.

**지도 릴레이션 데이터:**

| 학번 | 지도교수 | 학과 |
|---|---|---|
| 100 | P1 | 컴퓨터 |
| 200 | P2 | 전기 |
| 300 | P3 | 컴퓨터 |
| 400 | P1 | 컴퓨터 |

#### ❌ 삽입 이상

**상황**: 새로 부임한 P4 교수가 경영학과 소속이라는 정보를 입력하고 싶습니다.

```sql
-- P4 교수의 학과 정보를 입력하려고 시도
INSERT INTO 지도 (지도교수, 학과)
VALUES ('P4', '경영');

-- ❌ 오류! 기본키인 '학번'이 NULL이 될 수 없음!
-- 교수 정보를 입력하려면 반드시 학생의 학번이 있어야 함
```

교수 정보를 입력하려면 학생 학번이 있어야만 합니다. 학생이 없는 교수는 정보를 등록할 수 없는 비정상적인 상황이 발생합니다.

#### ❌ 삭제 이상

**상황**: 200번 학생이 자퇴하여 학생 정보를 삭제합니다.

```sql
DELETE FROM 지도 WHERE 학번 = 200;
-- 이 과정에서 P2 교수가 전기과 소속이라는 정보도 함께 삭제됨!
```

200번 학생 정보를 지웠더니, P2 교수가 전기과 교수라는 소중한 정보도 함께 사라집니다.

#### ❌ 갱신 이상

**상황**: P1 교수의 소속 학과가 컴퓨터에서 전기과로 변경되었습니다.

| 학번 | 지도교수 | 학과 |
|---|---|---|
| 100 | P1 | **전기** |  ← 변경 완료
| 400 | P1 | **컴퓨터** |  ← 변경 누락!

P1 교수를 지도교수로 둔 학생이 100번, 400번 두 명이라 두 행을 모두 수정해야 합니다. 하나라도 빠뜨리면 같은 P1 교수인데 소속 학과가 다르게 기록되는 데이터 불일치가 발생합니다.

### 2정규형 → 3정규형으로 변환하기

이행적 함수 종속을 제거하려면 다시 분해합니다.

```
[분해 전 — 2정규형]
지도(학번, 지도교수, 학과)
  학번 → 지도교수 → 학과  (이행 종속)

[분해 후 — 3정규형]
학생지도(학번, 지도교수)   ← X → Y 관계 보존
교수학과(지도교수, 학과)   ← Y → Z 관계 보존
```

`X → Z` 관계(학번 → 학과)는 `X → Y`와 `Y → Z`를 통해 언제든지 유추할 수 있으므로 별도로 저장할 필요가 없습니다.

## 3️⃣ 3정규형(3NF): 이행적 함수 종속까지 제거한 상태

### 3정규형의 정의

> **3정규형(Third Normal Form, 3NF)**: 2정규형을 만족하면서(원자값, 부분함수 종속 없음), 이행적 함수 종속도 없는 상태.

3정규형이 되면 원자값 조건, 부분함수 종속 제거, 이행적 함수 종속 제거가 모두 이루어진 상태입니다.

:::tip 실무에서의 3정규형
**실무에서는 대부분 3정규형 수준까지 정규화를 진행하면 일반적으로 이상현상이 발생하지 않습니다.** 3정규형까지 정규화된 데이터베이스는 충분히 안정적으로 운영될 수 있습니다.
:::

### 3정규형의 결과

앞서의 예제를 3정규형으로 분해하면:

```
수강(학번, 과목번호, 성적)      ← 1정규형 → 2정규형 과정에서 분리
학생지도(학번, 지도교수)        ← 2정규형 → 3정규형 과정에서 분리
교수학과(지도교수, 학과)        ← 2정규형 → 3정규형 과정에서 분리
```

이제 각 테이블을 보면:

- 새 교수가 부임해도 학번 없이 `교수학과` 테이블에 교수 정보를 바로 입력할 수 있습니다. ✅
- P1 교수의 학과를 변경할 때 `교수학과` 테이블에서 딱 한 번만 수정하면 됩니다. ✅
- 학생이 자퇴해도 교수 정보는 `교수학과` 테이블에 그대로 남습니다. ✅

이상현상이 모두 해결되었습니다!

그런데, 1정규형~3정규형은 **기본키(후보키)가 하나이거나, 복합키라도 후보키가 중첩되지 않는 단순한 경우**를 전제로 합니다. 만약 후보키가 여러 개이고, 복합키이며, 서로 중첩되는 복잡한 구조라면 3정규형만으로는 이상현상이 해결되지 않을 수 있습니다. 이때 등장하는 것이 **BCNF**입니다.

---

## 🔐 BCNF(보이스-코드 정규형): 결정자를 모두 키로

### BCNF란 무엇인가?

**BCNF(Boyce-Codd Normal Form, 보이스-코드 정규형)** 는 3정규형을 강화한 형태로, **"강한 제3정규형(Strong 3NF)"** 이라고도 부릅니다. BCNF의 핵심 원칙은 매우 단순합니다.

> **BCNF**: 릴레이션에서 **모든 결정자(Determinant)는 반드시 후보키(Candidate Key)** 여야 한다.

**결정자**란 다른 속성의 값을 결정하는 속성(또는 속성 집합)입니다. `학번 → 지도교수`에서 `학번`이 결정자이고, `지도교수`가 종속자입니다.

BCNF는 "결정자인데 키가 아닌 것"이 있으면 그것을 키로 만들어 주는 것입니다. 부분함수 종속이니, 이행 종속이니 복잡한 개념을 따지는 것이 아니라, **"결정자인데 키가 아니면 키로 만들어라!"** 는 단순하고 강력한 규칙입니다.

### BCNF가 필요한 상황

다음과 같은 상황에서 3정규형만으로는 이상현상이 해결되지 않고 BCNF가 필요합니다:
- 후보키(Candidate Key)가 여러 개인 경우
- 후보키가 2개 이상의 속성으로 구성된 복합키인 경우
- 후보키들이 서로 중첩되는(공유하는 속성이 있는) 경우

### BCNF 예제

아래 `수강교수` 릴레이션을 봅시다.

**테이블 구조:**

| 학번 | 과목 | 교수 |
|---|---|---|
| 100 | 프로그래밍 | P1 |
| 100 | 자료구조 | P2 |
| 200 | 프로그래밍 | P1 |
| 200 | 데이터베이스 | P3 |
| 300 | 프로그래밍 | P1 |
| 300 | 프로그래밍 | P1 |

**함수 종속 관계:**
- `(학번, 과목)` → `교수` : 기본키(복합키)가 교수를 결정
- `교수` → `과목` : **교수가 과목을 결정** (교수 한 명은 한 과목만 담당)

**함수 종속 다이어그램:**

```
(학번, 과목) ──────────→ 교수
     교수    ──────────→ 과목
```

- 부분함수 종속? ❌ 없음
- 이행적 함수 종속? ❌ 없음 (`학번 → 교수 → 과목`이 되려면 `학번 → 교수`가 성립해야 하는데, `학번` 혼자서는 교수를 결정할 수 없음)

**그렇다면 이것은 3정규형입니다.** 그런데도 이상현상이 발생합니다!

#### ❌ 삽입 이상

```sql
-- P5 교수가 자료구조 과목을 담당한다는 정보를 삽입하려고 시도
INSERT INTO 수강교수 (교수, 과목)
VALUES ('P5', '자료구조');

-- ❌ 오류! 기본키인 '학번'이 NULL이 될 수 없음!
-- 교수 정보를 추가하려면 반드시 학번(학생)이 있어야 함
```

#### ❌ 삭제 이상

```sql
-- 100번 학생이 프로그래밍 과목을 철회
DELETE FROM 수강교수 WHERE 학번 = 100 AND 과목 = '프로그래밍';

-- P1 교수가 프로그래밍을 담당한다는 정보도 함께 사라질 수 있음!
```

#### ❌ 갱신 이상

P1 교수의 담당 과목이 프로그래밍에서 데이터베이스로 바뀌면, P1이 등장하는 모든 행을 수정해야 합니다. 일부만 수정하면 데이터 불일치가 발생합니다.

### 이상현상의 원인 분석

문제의 원인은 **`교수`가 `과목`을 결정하는 결정자인데, `교수`는 키(기본키 또는 후보키)가 아니라는 것**입니다.

```
기본키: (학번, 과목)
결정자: 교수  ← 교수가 과목을 결정하는 결정자인데, 후보키로 설정되지 않음!
```

### BCNF로 변환하기

해결책은 간단합니다. **결정자인 `교수`를 키로 만들어 주면** 됩니다!

```
[분해 전 — 3정규형이지만 BCNF 아님]
수강교수(학번, 과목, 교수)

[분해 후 — BCNF]
학생수강(학번, 교수)     ← 기본키: (학번, 교수)
교수과목(교수, 과목)     ← 기본키: 교수 (결정자를 키로!)
```

**분해 후 데이터:**

`학생수강` 릴레이션:

| 학번 | 교수 |
|---|---|
| 100 | P1 |
| 100 | P2 |
| 200 | P1 |
| 200 | P3 |

`교수과목` 릴레이션:

| 교수 | 과목 |
|---|---|
| P1 | 프로그래밍 |
| P2 | 자료구조 |
| P3 | 데이터베이스 |

이제:
- 새 교수 P5가 자료구조를 담당한다면 `교수과목` 테이블에 `(P5, 자료구조)`를 바로 삽입할 수 있습니다. ✅
- P1 교수의 담당 과목이 바뀌면 `교수과목` 테이블에서 한 번만 수정하면 됩니다. ✅
- 학생을 삭제해도 `교수과목` 테이블의 교수 정보는 그대로입니다. ✅

이상현상이 완전히 해결되었습니다!

## 4️⃣ 4정규형(4NF): 다치 종속 제거

### 다치 종속(Multi-valued Dependency)이란?

**4정규형**은 **다치 종속(Multi-valued Dependency)** 을 제거한 상태입니다. 다치 종속은 실무에서 자주 다루지는 않지만, 개념적으로 이해해 두면 좋습니다.

**함수 종속**이 "결정자 하나 → 종속자 하나(또는 정해진 하나의 값)"의 관계라면, **다치 종속**은 "결정자 하나 → 종속자 **여러 개**"의 관계입니다. 화살표로 표현할 때 함수 종속은 `→` (단일 화살표), 다치 종속은 `↠` (이중 화살표)로 표현합니다.

| 종속 종류 | 표현 | 예시 | 의미 |
|---|---|---|---|
| 함수 종속 | `X → Y` | `학번 → 이름` | 학번이 같으면 이름도 하나로 같음 |
| 다치 종속 | `X ↠ Y` | `과목 ↠ 교수` | 하나의 과목에 여러 명의 교수가 대응 |

다치 종속이 존재하는 조건:
- 속성이 3개 이상인 릴레이션에서만 존재합니다.
- `A ↠ B | C` 의 형태로, `A`가 `B`를 결정하는데 그 값이 여러 개이고, `A`가 `C`를 결정하는데 그 값도 여러 개이며, `B`와 `C`는 서로 독립적인 관계입니다.

:::note 함수 종속은 다치 종속의 특수한 경우
함수 종속은 결정되는 값이 오직 하나인 다치 종속의 특수한 경우입니다. 즉, 다치 종속 ⊃ 함수 종속의 관계입니다.
:::

### 다치 종속 예제

다음 `강의정보` 릴레이션을 보겠습니다.

| 과목 | 교수 | 교재 |
|---|---|---|
| 자료구조 | P1 | T1 |
| 자료구조 | P1 | T2 |
| 자료구조 | P2 | T1 |
| 자료구조 | P2 | T2 |
| 데이터베이스 | P3 | T1 |
| 데이터베이스 | P3 | T2 |
| 데이터베이스 | P3 | T3 |

- 자료구조 과목을 가르치는 교수: P1, P2 (여러 명)
- 자료구조 과목의 교재: T1, T2 (여러 권)
- 교수와 교재는 서로 독립적 (어떤 교수가 어떤 교재를 쓰는지는 무관)

이 경우 기본키는 `(과목, 교수, 교재)` 세 가지 전체가 되어야 합니다. 그리고 이상현상이 발생합니다.

#### ❌ 삽입 이상

P4 교수가 데이터베이스 과목을 새로 담당하게 되었습니다. 그러면 데이터베이스 과목의 교재가 3권이므로 **교재 수만큼(3번)** 삽입해야 합니다. 교수 정보 하나를 추가하는 데 불필요한 작업이 생깁니다.

#### ❌ 갱신 이상

자료구조 담당 교수 P1이 T1 교재를 다른 교재로 바꾸려면, P1이 담당하는 모든 행을 수정해야 합니다. 중복 데이터로 인한 갱신 문제입니다.

### 4정규형으로 변환하기

해결책은 교수와 교재를 독립적인 테이블로 분리하는 것입니다.

```
[분해 전]
강의정보(과목, 교수, 교재)

[분해 후 — 4정규형]
과목교수(과목, 교수)   ← 과목 ↠ 교수 관계
과목교재(과목, 교재)   ← 과목 ↠ 교재 관계
```

`과목교수` 릴레이션:

| 과목 | 교수 |
|---|---|
| 자료구조 | P1 |
| 자료구조 | P2 |
| 데이터베이스 | P3 |

`과목교재` 릴레이션:

| 과목 | 교재 |
|---|---|
| 자료구조 | T1 |
| 자료구조 | T2 |
| 데이터베이스 | T1 |
| 데이터베이스 | T2 |
| 데이터베이스 | T3 |

이제 P4 교수를 추가할 때 `과목교수` 테이블에 `(데이터베이스, P4)` 한 행만 추가하면 됩니다. ✅

## 5️⃣ 5정규형(5NF): 조인 종속 제거

### 조인 종속(Join Dependency)이란?

**5정규형(Fifth Normal Form, 5NF)** 은 **조인 종속(Join Dependency)** 을 제거한 상태입니다. 이 개념은 가장 이론적이며 실무에서는 거의 다루지 않습니다. 따라서 개념적으로만 이해하고 넘어가셔도 충분합니다.

조인 종속이란, 릴레이션 R을 여러 개의 부분 릴레이션으로 분해했을 때, 그 분해된 릴레이션들을 다시 조인(합치기)하면 원래의 릴레이션 R로 정확하게 복원되는 특성을 말합니다.

**문제 상황**: 만약 분해 시 **기본키나 후보키를 포함하지 않는 속성**으로 분해하고 다시 조인하면, 원래 없던 **가짜 데이터(Spurious Tuple, 스퓨리어스 튜플)** 가 생길 수 있습니다.

### 5정규형 예제

릴레이션 `SPJ(공급자, 부품, 프로젝트)`를 세 개의 이진 릴레이션으로 분해했다고 가정해 봅시다.

```
SPJ(S, P, J) 분해
→ SP(S, P)
→ PJ(P, J)
→ SJ(S, J)
```

이 세 개를 다시 조인하면:

```sql
SP ⨝ PJ ⨝ SJ
```

이 조인 결과가 원래 SPJ와 동일하면 정보 손실이 없는 것이고, 만약 원래 없던 튜플이 생기면 이상 현상이 발생한 것입니다. 기본키를 포함하지 않는 임의의 속성으로 조인하면 아래처럼 없던 행이 나타날 수 있습니다:

| S | P | J |
|---|---|---|
| S1 | P1 | J1 |
| **S2** | **P1** | **J2** |  ← 원래 없던 가짜 데이터!

### 5정규형의 해결책

5정규형은 **반드시 후보키(기본키 포함)를 통해서만 조인에 참여**하도록 릴레이션을 분해하는 것입니다. 후보키를 기준으로 분해하면 조인 후에도 원래 릴레이션으로 손실 없이 복원됩니다.

```
[5정규형의 원칙]
릴레이션을 분해할 때 반드시 후보키를 포함하는 릴레이션으로만 분해하라.
그리고 그 분해된 릴레이션들을 후보키(기본키, 외래키)를 통해서만 조인하라.
```

:::note 5정규형은 이론적 완결
5정규형은 이론적으로 완벽한 정규화의 끝이지만, 실무에서는 성능 문제 때문에 3정규형 또는 BCNF 수준에서 정규화를 마치는 경우가 대부분입니다.
:::

## ⚠️ 정규화의 주의사항과 실무 팁

### 정규화를 더 할수록 릴레이션 수가 늘어난다

정규화를 진행할수록 테이블이 더 많이 분해됩니다. 테이블이 많아지면 **조인(Join, 여러 테이블을 합치는 연산)** 을 더 많이 해야 합니다. 조인이 많아지면 **데이터베이스 성능이 저하**될 수 있습니다.

| 정규화 수준 | 데이터 무결성 | 성능(조인 횟수) |
|---|---|---|
| 낮음 (비정규) | 낮음 (이상현상 많음) | 높음 (조인 적음) |
| 중간 (3NF) | 높음 (이상현상 거의 없음) | 중간 |
| 높음 (5NF) | 매우 높음 | 낮음 (조인 매우 많음) |

> 정규화를 많이 하면 할수록 **무결성(데이터의 정확성)은 좋아지지만, 성능은 떨어질 수 있습니다.** 실무에서는 이 둘의 균형을 적절히 맞추는 것이 중요합니다.

:::tip 실무에서의 정규화 전략
실무에서는 일반적으로 **3정규형(3NF) 또는 BCNF 수준**까지 정규화를 진행합니다. 그 이상의 정규화(4NF, 5NF)는 데이터 구조가 특별히 복잡한 경우에만 고려합니다. 때로는 성능을 위해 의도적으로 정규화를 낮추는 **반정규화(Denormalization)** 를 적용하기도 합니다.
:::

:::warning 반정규화는 신중하게
반정규화(Denormalization)는 성능 향상을 위해 의도적으로 중복 데이터를 허용하는 것입니다. 하지만 이상현상이 다시 발생할 수 있으므로, 반정규화를 적용할 때는 반드시 이상현상 발생 가능성을 충분히 검토해야 합니다.
:::

### 정규화 단계별 핵심 정리 표

| 정규형 | 조건 | 제거 대상 |
|---|---|---|
| **비정규형** | 원자값 아닌 속성 존재 | — |
| **1정규형(1NF)** | 원자값만 존재 | — (하지만 부분함수 종속 존재 가능) |
| **2정규형(2NF)** | 1NF + 부분함수 종속 없음 | 부분함수 종속 제거 |
| **3정규형(3NF)** | 2NF + 이행적 함수 종속 없음 | 이행적 함수 종속 제거 |
| **BCNF** | 3NF + 모든 결정자가 후보키 | 결정자인데 키가 아닌 것 제거 |
| **4정규형(4NF)** | BCNF + 다치 종속 없음 | 다치 종속 제거 |
| **5정규형(5NF)** | 4NF + 조인 종속 없음 | 조인 종속 제거 |

## 🧠 핵심 정리

- **정규화**는 이상현상(삽입 이상, 삭제 이상, 갱신 이상)을 해결하기 위해 릴레이션을 단계적으로 분해하는 과정이다.
- **1정규형(1NF)**: 모든 속성이 원자값(단일 값)을 가지는 상태이며, 부분함수 종속이 존재할 수 있다.
- **부분함수 종속**: 기본키의 일부만으로도 어떤 속성이 결정되는 관계 (예: 학번만으로 지도교수 결정).
- **2정규형(2NF)**: 1NF를 만족하면서 부분함수 종속을 제거하여 완전함수 종속만 남긴 상태.
- **이행적 함수 종속**: `X → Y`, `Y → Z`이면 `X → Z`가 성립하는 관계.
- **3정규형(3NF)**: 2NF를 만족하면서 이행적 함수 종속을 제거한 상태. 실무에서 일반적으로 이 수준까지 정규화한다.
- **BCNF**: 모든 결정자가 반드시 후보키여야 하는 조건. 3정규형보다 강한 조건이며 "강한 제3정규형"이라고도 불린다.
- **4정규형(4NF)**: BCNF를 만족하면서 다치 종속(하나의 결정자가 여러 값을 결정하는 관계)을 제거한 상태.
- **5정규형(5NF)**: 4NF를 만족하면서 조인 종속(후보키가 아닌 속성으로 조인 시 가짜 데이터 발생)을 제거한 상태.
- **정규화의 트레이드오프**: 정규화 수준이 높을수록 데이터 무결성은 향상되지만, 조인 횟수 증가로 성능이 저하될 수 있다.
- **종속성은 보존**: 정규화로 테이블을 분해해도 속성 간의 종속관계(함수 종속성)는 반드시 보존되어야 한다.
