# 🗂️ 부분함수 종속 예제: 수강지도 릴레이션 분석

데이터베이스를 처음 공부하다 보면 "정규화"라는 단어를 자주 만나게 됩니다. 정규화는 단순히 테이블을 쪼개는 작업이 아니라, **데이터를 건강하게 유지하기 위한 필수적인 설계 과정**입니다. 이번 문서에서는 실제 "수강지도" 릴레이션을 예제로 삼아, 부분함수 종속이 왜 문제가 되는지, 그리고 각 정규형이 어떻게 그 문제를 해결해 나가는지를 단계별로 아주 자세하게 살펴보겠습니다.

처음 보시는 분도 걱정하지 마세요. 마치 **냉장고 속 재료를 체계적으로 정리하듯**, 하나하나 차근차근 설명해 드리겠습니다. 🥦

## 🧩 1. 정규화란 무엇인가? — 개념 소개

### 정규화가 등장한 배경

데이터베이스에서 데이터를 저장하다 보면 예상치 못한 문제들이 생깁니다. 예를 들어 학생 정보를 저장하는 테이블에서, 학생이 수강 과목을 취소했을 때 **지도교수 정보까지 함께 사라지는 상황**이 발생할 수 있습니다. 이런 현상들을 우리는 **이상현상(Anomaly)** 이라고 부릅니다.

이상현상에는 크게 세 가지가 있습니다.

| 이상현상 종류 | 설명 | 실생활 비유 |
|---|---|---|
| **삽입 이상** | 꼭 필요하지 않은 데이터가 있어야만 새 데이터를 넣을 수 있음 | 집 주소를 등록하려는데 반드시 차량 번호도 입력해야 하는 상황 |
| **삭제 이상** | 하나의 데이터를 지우면 의도치 않은 다른 데이터도 함께 사라짐 | 영수증을 버렸더니 거래처 연락처도 같이 사라지는 상황 |
| **갱신 이상** | 하나의 정보를 수정하려면 중복된 여러 곳을 모두 바꿔야 함 | 회사 주소가 바뀌었는데 직원 명부마다 일일이 수정해야 하는 상황 |

> **정규화(Normalization)** 란, 이런 이상현상들을 제거하기 위해 릴레이션(테이블)을 체계적으로 분해하는 과정입니다. 이때 중요한 점은, **속성과 속성 사이의 종속 관계는 그대로 보존**하면서 분해한다는 것입니다.

### 정규형의 단계

정규화는 한 번에 끝나는 것이 아니라, 여러 단계를 거칩니다.

```
비정규형 → 1정규형(1NF) → 2정규형(2NF) → 3정규형(3NF) → BCNF → 4정규형(4NF) → 5정규형(5NF)
```

각 단계로 올라갈수록 데이터의 **무결성(데이터가 정확하고 일관되게 유지되는 성질)** 은 높아지지만, 그만큼 테이블 수가 늘어나고 **조인(여러 테이블을 연결하는 작업)** 도 많아지게 됩니다. 실무에서는 일반적으로 **3정규형 또는 BCNF 수준**으로 정규화를 수행하면 이상현상을 충분히 제거할 수 있습니다.

## 🔍 2. 1정규형과 부분함수 종속 — 왜 문제가 생기는가?

### 1정규형(1NF)이란?

**1정규형(1NF)** 의 정의는 간단합니다. 릴레이션에 속한 모든 속성(컬럼)이 **원자값(Atomic Value)** 만을 가지고 있으면 1정규형입니다.

**원자값**이란, 하나의 속성이 오직 **한 개의 값**만을 가지는 것을 의미합니다.

예를 들어 아래와 같은 상황을 생각해 보세요.

❌ **잘못된 예시 — 원자값 위반**

| 학번 | 점수 |
|---|---|
| A1 | 30, 70 |
| A2 | 40, 90, 100 |

위 테이블에서 "점수" 속성에 여러 개의 값(30, 70)이 들어 있습니다. 이것은 원자값이 아닙니다. 마치 **하나의 서랍에 여러 종류의 물건이 뒤섞인 것**처럼, 나중에 꺼낼 때 어떤 값을 사용해야 할지 알 수 없게 됩니다.

✅ **올바른 예시 — 원자값 유지**

| 학번 | 점수 |
|---|---|
| A1 | 30 |
| A1 | 70 |
| A2 | 40 |
| A2 | 90 |
| A2 | 100 |

이렇게 하나의 속성에 하나의 값만 들어가야 원자값을 가진다고 말할 수 있고, 이 조건을 만족하면 **1정규형**이 됩니다. 관계형 데이터베이스의 모든 테이블은 기본적으로 1정규형을 만족해야 합니다.

### 부분함수 종속이란?

1정규형은 원자값을 만족하지만, 그 안에 **부분함수 종속(Partial Functional Dependency)** 이 존재할 수 있습니다. 이것이 이상현상의 핵심 원인입니다.

부분함수 종속을 이해하려면 먼저 **완전함수 종속**을 알아야 합니다.

- **완전함수 종속**: 기본키 전체가 있어야만 다른 속성의 값이 결정되는 경우
- **부분함수 종속**: 기본키의 일부분만으로도 다른 속성의 값이 결정되는 경우

비유를 들어볼게요. 학교 수업을 생각해 봅시다. "어떤 학생(학번)이 어떤 과목(과목번호)을 들었을 때의 성적"은 반드시 **학번과 과목번호 둘 다** 알아야 결정됩니다. 이것이 완전함수 종속입니다.

하지만 "어떤 학생(학번)의 지도교수"는 **학번 하나만으로** 결정됩니다. 과목번호는 전혀 관계없습니다. 이처럼 기본키의 일부(학번)만으로 다른 속성(지도교수)이 결정될 때, 이를 **부분함수 종속**이라고 합니다.

```
완전함수 종속: (학번, 과목번호) → 성적       ✅ 두 개 모두 필요
부분함수 종속: 학번 → 지도교수              ⚠️ 학번 하나만으로 결정됨
부분함수 종속: 학번 → 학과                 ⚠️ 학번 하나만으로 결정됨
```

> 1정규형은 원자값을 가지고 있지만 **부분함수 종속이 존재**하는 릴레이션입니다. 이 부분함수 종속 때문에 삽입, 삭제, 갱신 이상현상이 발생합니다.

## 📋 3. 수강지도 릴레이션 분석 — 핵심 원리

### 릴레이션 구조 파악

이제 실제 예제인 **수강지도 릴레이션**을 분석해 봅시다.

| 속성명 | 설명 |
|---|---|
| **학번** | 학생을 식별하는 번호 (기본키의 일부) |
| **과목번호** | 수강 과목을 식별하는 번호 (기본키의 일부) |
| **지도교수** | 해당 학생의 지도교수 |
| **학과** | 해당 학생이 속한 학과 |
| **성적** | 해당 과목의 성적 |

**기본키**: (학번, 과목번호) — 두 속성의 복합키(두 개 이상의 속성이 합쳐져 기본키 역할을 하는 것)

### 함수 종속 관계 정리

이 릴레이션에서 함수 종속 관계는 다음과 같습니다.

```
(학번, 과목번호) → 성적    // 복합키 전체가 성적을 결정 (완전함수 종속)
학번 → 지도교수           // 학번만으로 지도교수 결정 (부분함수 종속 ⚠️)
학번 → 학과              // 학번만으로 학과 결정 (부분함수 종속 ⚠️)
지도교수 → 학과           // 지도교수가 학과를 결정
```

이를 함수 종속 다이어그램으로 표현하면 아래와 같습니다.

```
[학번] ──────────────────→ [지도교수] ──→ [학과]
  │                              │
  └──────────────────────────→ [학과]
  │
[과목번호] ─→ (학번+과목번호) ──→ [성적]
```

:::info 함수 종속 다이어그램 보는 법
화살표 왼쪽이 **결정자(Determinant)**, 오른쪽이 **종속자(Dependent)** 입니다. "A → B"는 "A 값이 같으면 B 값도 반드시 같다"는 의미입니다. 마치 주민등록번호가 같으면 이름도 반드시 같은 것처럼요.
:::

### 실제 데이터 예시

아래는 수강지도 릴레이션의 실제 데이터입니다.

| 학번 | 지도교수 | 학과 | 과목번호 | 성적 |
|---|---|---|---|---|
| 100 | P1 | 컴퓨터 | C413 | A |
| 100 | P1 | 컴퓨터 | E412 | A |
| 200 | P2 | 전기 | C123 | B |
| 300 | P3 | 컴퓨터 | C312 | A |
| 300 | P3 | 컴퓨터 | C413 | A |
| 300 | P3 | 컴퓨터 | C324 | B |
| 400 | P1 | 컴퓨터 | C312 | A |
| 400 | P1 | 컴퓨터 | C324 | A |
| 400 | P1 | 컴퓨터 | C413 | B |
| 400 | P1 | 컴퓨터 | E412 | C |

보이시나요? **100번 학생의 지도교수 P1, 학과 컴퓨터 정보가 두 번** 반복되고 있습니다. **400번 학생의 지도교수 P1, 학과 컴퓨터 정보는 무려 네 번**이나 중복됩니다. 이 중복이 바로 이상현상의 씨앗입니다.

## ⚠️ 4. 1정규형의 이상현상 — 실제 예시로 이해하기

### 삽입 이상(Insertion Anomaly)

**상황**: 500번 학생의 지도교수가 P4라는 정보를 저장하고 싶습니다.

```sql
-- ❌ 잘못된 상황: 과목번호 없이 학생 정보 삽입 불가
INSERT INTO 수강지도 (학번, 지도교수, 학과)
VALUES (500, 'P4', '경영');
-- 오류 발생! 기본키인 과목번호가 NULL이 될 수 없음
```

기본키는 **(학번, 과목번호)** 입니다. 기본키의 일부인 과목번호가 `NULL`이 되면 **개체 무결성(Entity Integrity, 기본키는 NULL이 될 수 없다는 규칙)** 을 위반합니다. 따라서 학생 정보를 넣으려면 **억지로 가짜 과목번호를 넣거나, 아예 삽입을 포기**해야 합니다.

:::danger 개체 무결성 위반
기본키(Primary Key)는 절대로 NULL 값을 가질 수 없습니다. 따라서 복합키의 일부가 NULL인 상태로 삽입하려 하면 데이터베이스가 이를 거부합니다. 이것이 삽입 이상의 근본 원인입니다.
:::

마치 **도서관에서 책을 등록하려는데 반드시 대출자 정보가 있어야만 책 정보를 입력할 수 있는 상황**과 같습니다. 대출자가 없는 새 책은 등록조차 못하는 황당한 상황이 되는 것입니다.

### 삭제 이상(Deletion Anomaly)

**상황**: 200번 학생이 C123 과목을 수강 철회했습니다.

| 학번 | 지도교수 | 학과 | 과목번호 | 성적 |
|---|---|---|---|---|
| ~~200~~ | ~~P2~~ | ~~전기~~ | ~~C123~~ | ~~B~~ |

200번 학생은 C123 과목 **한 과목만** 수강하고 있었습니다. 그래서 이 튜플(행)을 삭제하면 수강 정보뿐만 아니라, **P2 교수가 전기과 소속이라는 정보**까지 함께 사라져 버립니다.

```sql
-- ❌ 의도치 않은 연쇄 삭제
DELETE FROM 수강지도
WHERE 학번 = 200 AND 과목번호 = 'C123';
-- 결과: P2 교수가 전기과라는 정보도 같이 삭제됨 😱
```

학생은 단지 수강을 취소했을 뿐인데, 교수 정보까지 날아가 버리는 것입니다. 마치 **퇴사한 직원의 기록을 지웠더니 그 직원이 속했던 부서 정보 자체가 사라지는 것**과 같은 황당한 상황입니다.

### 갱신 이상(Update Anomaly)

**상황**: 400번 학생의 지도교수가 P1에서 P3로 변경되었습니다.

| 학번 | 지도교수 | 학과 | 과목번호 | 성적 |
|---|---|---|---|---|
| 400 | **P1 → P3** | 컴퓨터 | C312 | A |
| 400 | **P1 → P3** | 컴퓨터 | C324 | A |
| 400 | **P1 → P3** | 컴퓨터 | C413 | B |
| 400 | **P1 → P3** | 컴퓨터 | E412 | C |

400번 학생이 수강하는 과목이 4개이므로, **4개 행을 모두 업데이트**해야 합니다. 만약 3개만 수정하고 마지막 1개를 깜빡 잊는다면 어떻게 될까요?

```sql
-- ❌ 일부만 갱신하면 데이터 불일치 발생
UPDATE 수강지도 SET 지도교수 = 'P3' 
WHERE 학번 = 400 AND 과목번호 = 'C312'; -- 1번만 수정

-- 결과: 같은 400번 학생인데 어떤 행은 P3, 어떤 행은 P1으로 불일치 발생
```

같은 학생인데 지도교수가 어떤 과목에서는 P3이고, 다른 과목에서는 P1으로 남아있게 됩니다. 이런 데이터는 **어느 것이 올바른 데이터인지 알 수 없으므로 신뢰할 수 없게** 됩니다.

:::warning 이상현상의 근본 원인
이 세 가지 이상현상이 발생하는 근본 원인은 **부분함수 종속** 때문입니다. 학번만으로도 지도교수와 학과가 결정되는데, 이것이 학번+과목번호를 기본키로 가진 하나의 테이블에 함께 존재하기 때문에 문제가 생기는 것입니다.
:::

## 🔧 5. 정규화 과정 — 단계별 해결책

### 1정규형 → 2정규형: 부분함수 종속 제거

이상현상의 원인인 **부분함수 종속을 제거**하면 2정규형이 됩니다. 방법은 간단합니다. **부분함수 종속이 존재하는 속성들을 별도의 테이블로 분리**하면 됩니다.

**분해 원칙**:
- 기본키 전체에 완전함수 종속인 속성들 → 하나의 테이블
- 기본키의 일부에만 종속인 속성들 → 별도의 테이블로 분리

```
[1정규형] 수강지도(학번, 과목번호, 지도교수, 학과, 성적)
         기본키: (학번, 과목번호)

          ↓ 부분함수 종속 제거 (분해)

[2정규형] 수강(학번, 과목번호, 성적)      ← 완전함수 종속만 남김
          기본키: (학번, 과목번호)

[2정규형] 지도(학번, 지도교수, 학과)      ← 부분함수 종속이었던 것을 분리
          기본키: 학번
```

✅ **2정규형 분해 후 데이터**

**수강 릴레이션**

| 학번 | 과목번호 | 성적 |
|---|---|---|
| 100 | C413 | A |
| 100 | E412 | A |
| 200 | C123 | B |
| 300 | C312 | A |
| 400 | C312 | A |

**지도 릴레이션**

| 학번 | 지도교수 | 학과 |
|---|---|---|
| 100 | P1 | 컴퓨터 |
| 200 | P2 | 전기 |
| 300 | P3 | 컴퓨터 |
| 400 | P1 | 컴퓨터 |

이제 부분함수 종속은 사라졌습니다. 하지만 **지도 릴레이션**을 자세히 보면, 새로운 문제가 숨어 있습니다.

```
학번 → 지도교수    // 학번이 지도교수를 결정
지도교수 → 학과    // 지도교수가 학과를 결정
∴ 학번 → 학과     // 학번이 학과를 결정 (이행적 함수 종속 ⚠️)
```

이것이 바로 **이행적 함수 종속(Transitive Functional Dependency)** 입니다. X → Y이고 Y → Z이면 X → Z가 성립하는 관계입니다. 마치 "서울이 대한민국에 있고, 대한민국이 아시아에 있으면, 서울은 아시아에 있다"는 논리 관계와 같습니다.

:::note 2정규형의 특징
2정규형은 **부분함수 종속은 없지만, 이행적 함수 종속이 존재**하는 상태입니다. 이행적 함수 종속도 이상현상을 일으킵니다.
:::

### 2정규형에서 발생하는 이상현상

**삽입 이상**: 새로운 교수 P4가 경영학과 소속으로 부임했습니다. 그런데 지도 릴레이션에서 **기본키는 학번**입니다. 담당 학생이 없으면 학번이 NULL이 되므로 교수 정보를 삽입할 수 없습니다.

**삭제 이상**: 200번 학생이 자퇴하여 지도 릴레이션에서 해당 튜플을 삭제하면, P2 교수가 전기과 소속이라는 정보까지 함께 사라집니다.

**갱신 이상**: P1 교수의 학과가 컴퓨터에서 전기과로 바뀌면, P1 교수를 지도교수로 가진 모든 행(100번, 400번 학생)을 수정해야 합니다. 일부만 수정하면 불일치가 발생합니다.

### 2정규형 → 3정규형: 이행적 함수 종속 제거

**이행적 함수 종속을 제거**하면 3정규형이 됩니다. 분해 방법은 다음과 같습니다.

```
[2정규형] 지도(학번, 지도교수, 학과)
          학번 → 지도교수 → 학과 (이행적 함수 종속 존재)

          ↓ 이행적 함수 종속 제거 (분해)

[3정규형] 학생지도(학번, 지도교수)    ← 학번 → 지도교수
          기본키: 학번
          
[3정규형] 교수학과(지도교수, 학과)    ← 지도교수 → 학과
          기본키: 지도교수
```

✅ **3정규형 분해 후 데이터**

**학생지도 릴레이션**

| 학번 | 지도교수 |
|---|---|
| 100 | P1 |
| 200 | P2 |
| 300 | P3 |
| 400 | P1 |

**교수학과 릴레이션**

| 지도교수 | 학과 |
|---|---|
| P1 | 컴퓨터 |
| P2 | 전기 |
| P3 | 컴퓨터 |

이제 이상현상이 사라졌습니다!

- **삽입**: 새 교수 P4가 경영학과 소속으로 부임하면 교수학과 릴레이션에 `(P4, 경영)` 한 행만 추가하면 됩니다. 학번이 필요 없습니다. ✅
- **삭제**: 200번 학생이 자퇴해도 교수학과 릴레이션의 `(P2, 전기)` 정보는 그대로 유지됩니다. ✅
- **갱신**: P1 교수의 학과가 바뀌면 교수학과 릴레이션에서 딱 한 행만 수정하면 됩니다. ✅

:::tip 실무에서의 정규화 수준
실무에서는 **3정규형 수준으로 정규화**하면 일반적인 이상현상은 대부분 제거됩니다. 4정규형, 5정규형은 이론적으로는 존재하지만, 실무 적용 빈도는 낮습니다.
:::

### BCNF(Boyce-Codd Normal Form): 강한 제3정규형

그런데 3정규형이어도 이상현상이 생기는 특수한 경우가 있습니다. 바로 **후보키가 여러 개이고, 후보키들이 서로 중첩되는 경우**입니다. 이를 해결하기 위해 **BCNF(보이스-코드 정규형)** 가 등장합니다.

**BCNF가 필요한 상황의 예시:**

| 학번 | 과목 | 교수 |
|---|---|---|
| 100 | 프로그래밍 | P1 |
| 100 | 자료구조 | P2 |
| 200 | 프로그래밍 | P1 |
| 300 | 프로그래밍 | P3 |
| 300 | 자료구조 | P2 |

이 릴레이션의 함수 종속 관계:
```
(학번, 과목) → 교수    // 복합키가 교수를 결정 (기본키)
교수 → 과목           // 교수가 과목을 결정 (⚠️ 결정자이지만 키가 아님)
```

부분함수 종속도 없고, 이행적 함수 종속도 없습니다. 따라서 3정규형입니다. 하지만 이상현상이 발생합니다!

- **삽입 이상**: 새 교수 P5가 자료구조 담당으로 왔지만, 담당 학생(학번)이 없으면 입력 불가
- **삭제 이상**: 100번 학생이 프로그래밍 과목을 철회하면 P1이 프로그래밍 담당이라는 정보 소멸
- **갱신 이상**: P1의 담당 과목이 바뀌면 P1이 포함된 모든 행을 수정해야 함

**이상의 원인**: `교수 → 과목`에서 **교수가 결정자(Determinant)임에도 불구하고 후보키(Candidate Key)가 아니기** 때문입니다.

**BCNF의 규칙**: 모든 결정자는 반드시 후보키여야 한다.

```
[3정규형] 수강지도(학번, 과목, 교수)
          기본키: (학번, 과목)
          교수 → 과목 (결정자이지만 후보키 아님 ⚠️)

          ↓ 결정자를 후보키로 만들기 (분해)

[BCNF] 학생교수(학번, 교수)     ← 기본키: (학번, 교수)
[BCNF] 교수과목(교수, 과목)     ← 기본키: 교수
```

✅ **BCNF 분해 후 데이터**

**학생교수 릴레이션**

| 학번 | 교수 |
|---|---|
| 100 | P1 |
| 100 | P2 |
| 200 | P1 |
| 300 | P3 |
| 300 | P2 |

**교수과목 릴레이션**

| 교수 | 과목 |
|---|---|
| P1 | 프로그래밍 |
| P2 | 자료구조 |
| P3 | 프로그래밍 |

이제 새 교수 P5가 왔을 때 교수과목 릴레이션에 `(P5, 자료구조)`만 추가하면 되고, P1의 담당 과목이 바뀌면 교수과목 릴레이션 한 행만 수정하면 됩니다. 이상현상이 완전히 사라졌습니다! ✅

> **BCNF는 3정규형보다 더 강한 조건**을 요구하기 때문에 "강한 제3정규형(Strong 3NF)"이라고도 불립니다.

### 4정규형(4NF): 다치종속 제거

4정규형과 5정규형은 실무에서 자주 다루지는 않지만, 개념은 알아두면 좋습니다.

**다치종속(Multi-valued Dependency)** 이란, 하나의 결정자가 **여러 개의 값을 결정**하는 관계입니다. 함수종속의 화살표를 `→`로 표시한다면, 다치종속은 `↠` (이중 화살표)로 표시합니다.

예를 들어, 과목 하나에 교수도 여러 명이고, 교재도 여러 권인 경우를 생각해 봅시다.

| 과목 | 교수 | 교재 |
|---|---|---|
| 자료구조 | P1 | T1 |
| 자료구조 | P1 | T2 |
| 자료구조 | P2 | T1 |
| 자료구조 | P2 | T2 |
| 데이터베이스 | P3 | T3 |
| 데이터베이스 | P3 | T4 |
| 데이터베이스 | P3 | T5 |

```
과목 ↠ 교수   // 하나의 과목에 여러 교수 대응
과목 ↠ 교재   // 하나의 과목에 여러 교재 대응
교수와 교재는 서로 독립(무관)
```

이런 경우 새 교수 P4가 데이터베이스를 담당하면, 교재 수(3권)만큼 3개 행을 삽입해야 합니다. 이것이 삽입 이상입니다.

**해결책**: 교수와 교재를 독립된 두 테이블로 분리합니다.

```
[4정규형] 과목교수(과목, 교수)   ← 과목 ↠ 교수
[4정규형] 과목교재(과목, 교재)   ← 과목 ↠ 교재
```

### 5정규형(5NF): 조인종속 제거

**조인종속(Join Dependency)** 은 릴레이션을 여러 개로 분해한 후, 그것들을 다시 **조인(Join)** 했을 때 원래의 릴레이션과 동일하게 복원되는 성질과 관련됩니다.

문제는 기본키를 포함하지 않는 속성들로 조인을 수행하면 **원래 없던 가짜 튜플(Spurious Tuple)** 이 생성될 수 있다는 것입니다.

5정규형은 이를 방지하기 위해 **반드시 후보키(기본키)를 포함한 릴레이션으로만 분해하고, 그것들을 통해서만 조인에 참여**하도록 합니다. 이렇게 하면 원하지 않는 가짜 데이터가 생성되는 것을 막을 수 있습니다.

:::info 4정규형, 5정규형의 실무 적용
4정규형과 5정규형은 이론적으로는 중요하지만, **실무에서는 3정규형 또는 BCNF 수준에서 정규화를 마치는 경우가 대부분**입니다. 따라서 개념적으로 이해하는 수준으로만 학습하셔도 충분합니다.
:::

## 📊 6. 정규형 전체 비교 정리

### 정규형별 특징 한눈에 보기

| 정규형 | 조건 | 제거하는 종속 | 실무 적용 |
|---|---|---|---|
| **1NF** | 원자값으로만 구성 | - | 기본 조건 |
| **2NF** | 1NF + 완전함수 종속 | 부분함수 종속 제거 | 권장 |
| **3NF** | 2NF + 비이행적 함수 종속 | 이행적 함수 종속 제거 | 강력 권장 ✅ |
| **BCNF** | 3NF + 모든 결정자가 후보키 | 비후보키 결정자 제거 | 강력 권장 ✅ |
| **4NF** | BCNF + 비자명 다치종속 없음 | 다치종속 제거 | 선택적 |
| **5NF** | 4NF + 조인종속 없음 | 조인종속 제거 | 이론적 |

### 정규화의 장단점

정규화는 장점만 있는 것이 아닙니다. 양날의 검처럼, 분명한 단점도 존재합니다.

**장점 ✅**
- 이상현상(삽입, 삭제, 갱신) 제거
- 데이터 무결성(정확성, 일관성) 향상
- 데이터 중복 최소화로 저장 공간 절약

**단점 ⚠️**
- 테이블이 많아질수록 **조인 횟수 증가** → 쿼리 성능 저하
- 테이블 구조가 복잡해져 **이해하기 어려워질 수 있음**
- 지나친 정규화는 오히려 **성능 문제**를 야기할 수 있어, 경우에 따라 **역정규화(De-normalization)** 를 고려해야 함

:::tip 역정규화(De-normalization)
성능을 위해 의도적으로 정규화를 되돌리는 것을 역정규화라고 합니다. 예를 들어 조회가 매우 빈번한 테이블에서 조인 횟수를 줄이기 위해 일부 중복을 허용하는 방식입니다. "무조건 정규화"가 정답은 아니며, 상황에 맞는 설계가 중요합니다.
:::

## 🛠️ 7. 코드로 보는 정규화 — SQL 예시

### 1정규형 → 2정규형 분해 SQL

```sql
-- ❌ 1정규형 상태: 부분함수 종속 존재
-- 수강지도 테이블 (정규화 전)
CREATE TABLE 수강지도 (
    학번        INT,
    과목번호    VARCHAR(10),
    지도교수    VARCHAR(20),   -- 학번에만 종속 (부분함수 종속 ⚠️)
    학과        VARCHAR(20),   -- 학번에만 종속 (부분함수 종속 ⚠️)
    성적        CHAR(1),       -- (학번, 과목번호)에 완전함수 종속
    PRIMARY KEY (학번, 과목번호)  -- 복합키
);

-- ✅ 2정규형 상태: 부분함수 종속 제거 후 분해
-- 수강 테이블: (학번, 과목번호) → 성적 (완전함수 종속만 남김)
CREATE TABLE 수강 (
    학번        INT,
    과목번호    VARCHAR(10),
    성적        CHAR(1),           -- 복합키 전체에 종속
    PRIMARY KEY (학번, 과목번호)   -- 복합키
);

-- 지도 테이블: 학번 → 지도교수, 학과 (부분함수 종속을 별도 테이블로 분리)
CREATE TABLE 지도 (
    학번        INT PRIMARY KEY,   -- 단일 기본키
    지도교수    VARCHAR(20),       -- 학번에 완전함수 종속
    학과        VARCHAR(20)        -- 학번에 종속 (단, 이행종속 존재 ⚠️)
);
```

### 2정규형 → 3정규형 분해 SQL

```sql
-- ❌ 2정규형 상태: 이행적 함수 종속 존재
-- 지도 테이블에서: 학번 → 지도교수 → 학과 (이행적 함수 종속 ⚠️)
CREATE TABLE 지도 (
    학번        INT PRIMARY KEY,
    지도교수    VARCHAR(20),
    학과        VARCHAR(20)        -- 이행적 함수 종속: 지도교수를 통해 결정됨
);

-- ✅ 3정규형 상태: 이행적 함수 종속 제거 후 분해
-- 학생지도 테이블: 학번 → 지도교수 (직접 종속만 유지)
CREATE TABLE 학생지도 (
    학번        INT PRIMARY KEY,
    지도교수    VARCHAR(20),       -- 학번에 직접 종속
    FOREIGN KEY (지도교수) REFERENCES 교수학과(지도교수)  -- 외래키로 연결
);

-- 교수학과 테이블: 지도교수 → 학과 (이행 고리를 별도 테이블로 분리)
CREATE TABLE 교수학과 (
    지도교수    VARCHAR(20) PRIMARY KEY,  -- 결정자가 기본키
    학과        VARCHAR(20)              -- 지도교수에 직접 종속
);
```

```sql
-- ✅ 3정규형에서 새 교수 삽입 (삽입 이상 없음!)
-- 새 교수 P4가 경영학과 소속으로 부임
INSERT INTO 교수학과 (지도교수, 학과)
VALUES ('P4', '경영');
-- 학번 없이도 자유롭게 삽입 가능 ✅

-- ✅ 3정규형에서 교수 학과 변경 (갱신 이상 없음!)
-- P1 교수의 학과를 컴퓨터 → 전기로 변경
UPDATE 교수학과
SET 학과 = '전기'
WHERE 지도교수 = 'P1';
-- 딱 한 번의 UPDATE로 모든 관련 정보가 일관되게 변경 ✅
```

## 💡 8. 주의사항과 실전 팁

### 정규화할 때 흔히 하는 실수들

**실수 1: 기본키 파악을 소홀히 하기**

정규화의 시작은 **기본키(Primary Key)를 정확히 파악**하는 것입니다. 기본키가 단일 속성인지 복합 속성인지에 따라 부분함수 종속의 존재 여부가 결정됩니다. 기본키가 단일 속성이라면 부분함수 종속은 원천적으로 존재할 수 없습니다. 부분함수 종속은 **복합키**를 가질 때만 발생합니다.

```
단일 기본키: 학번 → (지도교수, 학과, 성적)
→ 부분함수 종속 불가능 (기본키가 하나뿐이므로)

복합 기본키: (학번, 과목번호) → (지도교수, 학과, 성적)
→ 부분함수 종속 가능! (학번만으로, 또는 과목번호만으로 결정되는 속성이 있을 수 있음)
```

**실수 2: 이행종속 방향을 거꾸로 이해하기**

이행종속은 반드시 **X → Y → Z** 방향으로 흐릅니다. `학번 → 지도교수 → 학과`에서 `지도교수 → 학번`은 성립하지 않습니다. 한 교수가 여러 학생을 지도할 수 있기 때문입니다. 방향을 항상 명확히 확인하세요.

**실수 3: BCNF와 3NF 혼동하기**

3정규형이면 이상현상이 없다고 착각할 수 있습니다. 하지만 **후보키가 여러 개이고 중첩되는 경우**에는 3정규형이어도 이상현상이 발생할 수 있습니다. 이때는 BCNF를 적용해야 합니다.

:::warning BCNF 적용 시 주의
BCNF는 3정규형보다 강하지만, BCNF로 분해하면 일부 함수 종속이 보존되지 않는 경우도 있습니다. 무결성이 높아지는 대신 일부 제약조건을 별도로 관리해야 할 수도 있습니다.
:::

**실수 4: 분해 후 조인 가능성 확인을 빠뜨리기**

정규화로 테이블을 분해할 때는, 나중에 원본 릴레이션을 **무손실 분해(Lossless Decomposition)** 로 복원할 수 있는지 확인해야 합니다. 즉, 분해된 테이블들을 조인했을 때 원래 데이터가 정확히 복원되어야 합니다. 외래키 관계가 올바르게 설정되어 있는지 꼭 확인하세요.

### 정규화 체크리스트

아래 순서대로 확인하면 정규화 수준을 판별할 수 있습니다.

```
1. 모든 속성이 원자값인가?
   ├─ No  → 비정규형 (먼저 1NF로 만들기)
   └─ Yes → 1NF 이상

2. 부분함수 종속이 존재하는가?
   ├─ Yes → 1NF (부분함수 종속 제거 → 2NF)
   └─ No  → 2NF 이상

3. 이행적 함수 종속이 존재하는가?
   ├─ Yes → 2NF (이행종속 제거 → 3NF)
   └─ No  → 3NF 이상

4. 결정자이지만 후보키가 아닌 속성이 있는가?
   ├─ Yes → 3NF이지만 BCNF 아님 (결정자를 후보키로 → BCNF)
   └─ No  → BCNF

5. 다치종속이 존재하는가?
   ├─ Yes → BCNF이지만 4NF 아님 (다치종속 제거 → 4NF)
   └─ No  → 4NF

6. 조인종속이 존재하는가?
   ├─ Yes → 4NF이지만 5NF 아님 (조인종속 제거 → 5NF)
   └─ No  → 5NF
```

## 📌 핵심 정리

이번 문서에서 배운 핵심 개념들을 정리합니다.

- **이상현상**은 삽입 이상, 삭제 이상, 갱신 이상 세 가지로 구분되며, 잘못된 테이블 설계에서 발생한다.
- **정규화**는 이상현상을 제거하기 위해 릴레이션을 분해하는 과정이며, 이때 속성 간의 종속 관계는 보존된다.
- **1정규형(1NF)** 은 모든 속성이 원자값(단일 값)을 가지는 릴레이션이며, 부분함수 종속이 존재할 수 있다.
- **부분함수 종속**은 복합키의 일부만으로 다른 속성이 결정될 때 발생하며, 1NF에서 이상현상의 주원인이다.
- **2정규형(2NF)** 은 부분함수 종속을 제거한 상태로, 모든 비키 속성이 복합키 전체에 완전함수 종속된다.
- **이행적 함수 종속**은 X → Y → Z 관계로, 2NF에서 이상현상의 원인이 된다.
- **3정규형(3NF)** 은 이행적 함수 종속까지 제거한 상태로, 실무에서 가장 많이 목표로 삼는 정규화 수준이다.
- **BCNF**는 모든 결정자가 후보키여야 한다는 조건으로, 3NF보다 강하며 "강한 제3정규형"이라고도 불린다.
- **4정규형(4NF)** 은 다치종속(하나의 결정자가 여러 값을 결정)을 제거한 상태이다.
- **5정규형(5NF)** 은 조인종속을 제거하여 후보키를 통해서만 조인에 참여하도록 한 상태이다.
- **정규화의 단점**은 테이블 수 증가로 인한 조인 횟수 증가와 그에 따른 성능 저하이므로, 필요에 따라 역정규화를 고려할 수 있다.
- **수강지도 릴레이션 분석**을 통해, 복합키 `(학번, 과목번호)` 환경에서 `학번 → 지도교수`, `학번 → 학과`의 부분함수 종속이 세 가지 이상현상 모두를 유발함을 확인할 수 있다.
