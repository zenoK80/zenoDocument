# 🗄️ 개념적·논리적·물리적 설계 단계와 설계 목표

데이터베이스를 만든다는 건 단순히 코드를 작성하는 것이 아닙니다. 마치 아파트를 짓기 전에 설계 도면을 그리듯이, 데이터베이스도 단계적인 **설계 과정**을 거쳐야 합니다. 이번 문서에서는 데이터베이스 설계의 전체 흐름과 핵심 세 단계인 **개념적 설계**, **논리적 설계**, **물리적 설계**에 대해 아주 자세하게 살펴보겠습니다.

---

## 🏗️ 데이터베이스 생명주기(Life Cycle)란?

### 왜 생명주기를 알아야 할까요?

데이터베이스는 단 하루 만에 뚝딱 만들어지는 것이 아닙니다. 집을 짓는 것처럼 **계획 → 설계 → 시공 → 거주 → 유지보수** 라는 흐름이 있습니다. 데이터베이스도 마찬가지로 탄생부터 운영, 그리고 개선에 이르기까지 일정한 순환 과정을 거치는데, 이를 **데이터베이스 생명주기(Database Life Cycle)** 라고 부릅니다.

이 생명주기를 이해하면 "지금 내가 어느 단계에 있는가?", "다음에 무엇을 해야 하는가?"를 명확히 알 수 있습니다. 개발 현장에서 일하게 되면 이 순서를 기반으로 프로젝트가 진행되기 때문에, 반드시 머릿속에 새겨두어야 합니다.

### 생명주기 5단계

데이터베이스 생명주기는 총 **5단계**로 구성됩니다.

| 단계 | 이름 | 핵심 활동 |
|---|---|---|
| 1단계 | 요구 조건 분석 | 사용자 요구사항 수집 및 문서화 |
| 2단계 | 설계 | 개념적 → 논리적 → 물리적 설계 |
| 3단계 | 구현 | DDL 명령어로 실제 DB 생성 |
| 4단계 | 운영 | 실제 사용자에게 서비스 제공 |
| 5단계 | 감시 및 개선 | 성능 모니터링 및 문제 개선 |

> **핵심 포인트:** 생명주기의 각 단계는 일직선이 아닙니다. 운영하다가 문제가 발견되면 설계 단계로 돌아가서 다시 보완할 수 있습니다. 이처럼 **피드백(feedback)** 이 가능한 순환 구조라는 점이 중요합니다.

:::info 생명주기는 순환합니다
설계 단계에서 요구 조건이 부족하면 다시 요구 조건 분석 단계로 돌아갈 수 있고, 운영 단계에서 설계의 문제점이 발견되면 설계 단계로 피드백하여 전체 시스템의 효율성과 성능을 지속적으로 개선해 나갈 수 있습니다.
:::

### 각 단계별 상세 설명

**① 요구 조건 분석**

데이터베이스를 만들기 전 가장 먼저 해야 할 일은 "무엇을 위해, 누가, 어떻게 쓸 것인가?"를 파악하는 것입니다. 이 단계에서는 데이터베이스에 저장할 **범위**를 결정하고, **사용자가 누구인지**를 파악합니다. 특히 중요한 점은 현재 사용자뿐 아니라 **잠재적 사용자(미래에 이 시스템을 쓰게 될 사람들)** 도 반드시 고려해야 한다는 것입니다. 지금 당장은 직원 50명이 쓰는 시스템이더라도, 3년 후에 500명이 쓰게 될 수도 있으니까요.

**② 설계**

요구 조건 분석이 끝나면 본격적인 설계에 들어갑니다. 이 단계가 바로 이 문서의 핵심 주제인 **개념적 설계 → 논리적 설계 → 물리적 설계** 입니다. 아파트의 설계 도면을 그리는 것과 같습니다.

**③ 구현**

설계가 완료되면 `CREATE TABLE`, `CREATE DATABASE` 같은 **DDL(Data Definition Language, 데이터 정의어)** 명령어를 사용하여 실제 데이터베이스를 생성합니다. 빈 껍데기의 데이터베이스 구조가 만들어지는 단계입니다.

**④ 운영**

실제 사용자의 요구에 따라 서비스를 제공하는 단계입니다. 직원들이 실제로 데이터를 넣고, 조회하고, 수정하고, 삭제하는 일상적인 업무가 이루어집니다.

**⑤ 감시 및 개선**

운영 중에 성능이 떨어지거나, 보안 취약점이 발견되거나, 새로운 요구 조건이 생기면 시스템을 모니터링하고 개선하는 단계입니다. IT 컨설턴트들이 현장에서 시스템을 진단하고 보고서를 만드는 활동이 바로 이 단계에 해당합니다.

## 🔍 요구 조건 분석 단계 자세히 보기

### 요구 조건 분석이란 무엇인가요?

요구 조건 분석은 마치 건축사가 집을 짓기 전에 건축주에게 "몇 층으로 지을까요?", "방은 몇 개 필요하세요?", "예산은 얼마나 되시나요?"를 꼼꼼히 물어보는 것과 같습니다. 데이터베이스를 만들기 위해 필요한 정보를 체계적으로 수집하고 정리하는 과정입니다.

이 단계에서 수집하는 정보는 다음과 같습니다.
- **사용자가 누구인지**: 어떤 부서의 누가 이 시스템을 쓸 것인가
- **어떤 업무인지**: 재고 관리, 인사 관리, 학사 관리 등 어떤 업무를 처리할 것인가
- **데이터의 종류**: 어떤 데이터를 저장하고 관리할 것인가
- **데이터의 흐름**: 데이터가 어디서 생겨서 어디로 흘러가는가

### 요구 조건의 세 가지 분류

요구 조건은 크게 세 가지로 나눌 수 있습니다.

**① 정적인 정보 구조에 대한 요구 조건**

이것은 데이터베이스의 **구조(테이블)** 에 관한 요구 조건입니다. 예를 들어 학교 시스템을 만든다면, "학생 테이블", "교수 테이블", "과목 테이블"이 필요하다는 것을 파악합니다. 또한 각 테이블이 어떤 속성(컬럼)을 가지는지, 테이블 간의 관계는 어떤지(일대일인지, 일대다인지, 다대다인지)도 이 단계에서 분석합니다. 집에 비유하면, 방이 몇 개인지, 화장실 위치는 어디인지, 거실 크기는 어떻게 할지를 결정하는 것과 같습니다.

**② 동적인 데이터베이스에 대한 요구 조건**

이것은 데이터베이스에서 **어떤 작업(트랜잭션)** 이 일어나는지에 관한 요구 조건입니다. 트랜잭션(transaction)이란 데이터베이스에서 이루어지는 작업의 단위를 말합니다. 예를 들어 학교 시스템이라면:

- 학생이 수강 신청을 한다 → **삽입(INSERT)**
- 학생이 수강 변경을 한다 → **갱신(UPDATE)**
- 학생이 수강 철회를 한다 → **삭제(DELETE)**
- 학생이 성적을 조회한다 → **검색(SELECT)**

이러한 작업들이 얼마나 자주, 얼마나 많이 발생하는지를 파악해야 최적화된 설계를 할 수 있습니다.

**③ 범 기관적 제약 조건**

현실적인 제약입니다. 사용자 요구를 전부 만족시키려면 개발 비용이 5억 원인데, 우리 회사의 예산이 1억 원뿐이라면? 이런 현실적인 제약(예산, 기간, 인력, 기관의 경영 목표, 정책 등)을 고려해서 요구 조건의 우선순위를 조정하고 타협점을 찾아야 합니다.

### 요구 조건 명세서와 분석 도구들

요구 조건 분석이 끝나면 반드시 **공식적인 요구 조건 명세서**가 만들어져야 합니다. 이것은 결재를 받은 공식 문서로, 이 문서를 기반으로 실제 개발이 진행됩니다. 나중에 개발 중에 "이건 처음에 요청한 내용이 아닌데요?"라는 분쟁을 방지하는 역할도 합니다.

이 명세서를 작성하는 데 사용하는 도구들이 있습니다.

**HIPO 다이어그램(Hierarchy Input Process Output Diagram)**

HIPO 다이어그램은 시스템의 기능을 계층 구조로 표현한 도표입니다. 업무가 어떤 흐름으로 이루어지고, 큰 업무가 어떤 세부 업무들로 나뉘는지를 한눈에 파악할 수 있게 해줍니다. 예를 들어 "재고 관리 시스템"이 최상위에 있고, 그 아래에 "거래 처리", "재고량 업데이트", "보고서 출력"이 있으며, 각각이 다시 더 세부적인 업무로 나뉘는 형태입니다.

**데이터 플로우 다이어그램(Data Flow Diagram, DFD)**

데이터가 어떤 프로세스를 거쳐 흘러가는지를 그림으로 표현합니다. 예를 들어 판매부에서 고객 주문서를 접수하면, 그 데이터가 신용 검토 프로세스를 거쳐 최종 처리되는 흐름을 도식화합니다.

**SADT(Structured Analysis and Design Technique)**

대규모의 복잡한 시스템을 구조적으로 분석할 때 사용하는 방법론입니다. 입력 데이터(Input), 제어 데이터(Control), 처리 메커니즘(Mechanism), 출력 결과(Output)의 관계를 구조적으로 표현합니다.

:::tip 실무에서는?
실제 회사에서 IT 개발 업무를 하게 되면, 무작정 코딩부터 시작하는 것이 아니라 수많은 문서 작업이 먼저 이루어집니다. 그 중 가장 기본이 되는 문서가 바로 **공식 요구 조건 명세서**입니다. 이 문서 없이 개발을 시작하면 나중에 "이건 제가 원한 게 아닌데요"라는 문제가 반드시 발생합니다.
:::

## 🎨 개념적 설계 단계

### 개념적 설계란 무엇인가요?

요구 조건 분석이 끝나면 드디어 설계 단계에 들어갑니다. 설계의 첫 번째 단계인 **개념적 설계(Conceptual Design)** 는 **사용자 관점**에서 데이터베이스의 전체적인 구조를 표현하는 단계입니다.

쉽게 비유하자면, 건축가가 집을 짓기 전에 "이 집은 3층 구조로, 1층에는 거실과 주방, 2층에는 침실 3개, 3층에는 다락방이 있겠다"라는 큰 그림을 그리는 것과 같습니다. 아직 시멘트 어떤 것을 쓸지, 배관은 어떻게 할지는 결정하지 않은 단계입니다.

개념적 설계의 핵심 결과물은 **ER 다이어그램(Entity-Relationship Diagram, 개체-관계 다이어그램)** 입니다.

### ER 다이어그램이란?

ER 다이어그램은 데이터베이스에 존재하는 **개체(Entity, 실체)** 들과 그들 사이의 **관계(Relationship)** 를 그림으로 표현한 것입니다. 예를 들어 학교 시스템이라면 다음과 같이 표현할 수 있습니다.

- **학생** 개체: 학번(기본키), 이름, 학과
- **교수** 개체: 교수번호(기본키), 이름, 소속학과
- **지도 관계**: 교수 한 명은 여러 명의 학생을 지도한다(1:N 관계)

이 ER 다이어그램은 **어떤 DBMS를 사용하든 상관없이 독립적으로 작성**할 수 있다는 것이 큰 특징입니다. Oracle을 쓰든, MySQL을 쓰든, MSSQL을 쓰든, 관계형이든 계층형이든 상관없이 동일한 ER 다이어그램이 나옵니다.

### 개념적 설계에서 사용하는 추상화 개념

개념적 설계를 잘 하려면 **추상화(Abstraction)** 라는 개념이 필요합니다. 추상화란 복잡한 현실을 단순화해서 핵심만 뽑아내는 것입니다. 마치 지도가 실제 지형을 그대로 보여주는 것이 아니라, 필요한 정보만 단순하게 표현하는 것처럼요.

추상화에는 두 가지 방식이 있습니다.

**집단화(Aggregation)**

여러 개의 **속성(attribute)** 들을 묶어서 하나의 **개체(entity)** 로 만드는 것입니다. 예를 들어 학번, 이름, 학과, 나이, 연락처라는 여러 속성들을 하나로 묶어 "학생"이라는 개체를 만드는 것이 집단화입니다. 개별 요소들을 그룹으로 묶어 하나의 더 큰 단위를 만드는 개념입니다.

**일반화(Generalization)**

여러 **개체** 들의 공통적인 특성을 뽑아내어 더 일반적인 개체를 만드는 것입니다. 예를 들어 "대학원생"과 "학부생"이라는 두 개체가 있을 때, 이 둘의 공통점인 "학생"이라는 더 일반적인 개체를 도출하는 것이 일반화입니다.

> **둘 다 추상화입니다.** 집단화는 속성들 → 개체, 일반화는 개체들 → 더 일반적인 개체. 공통점은 "공통된 것을 뽑아내어 단순하게 표현한다"는 것입니다.

### 개념적 설계에서의 트랜잭션 모델링

개념적 설계 단계에서는 **구조(ER 다이어그램)** 만 만드는 것이 아니라, **어떤 작업(트랜잭션)** 이 이루어질지도 고차원적으로 정의합니다. "이 데이터베이스에서는 검색이 주로 이루어질 것이다", "삽입과 갱신 작업이 빈번하게 일어날 것이다" 와 같이 트랜잭션의 종류와 성격을 개략적으로 명세하는 것을 **트랜잭션 모델링**이라고 합니다.

:::note 개념적 설계는 DBMS 독립적!
개념적 설계의 결과물인 ER 다이어그램은 어떤 DBMS(데이터베이스 관리 시스템)를 사용할지와 완전히 독립적입니다. Oracle을 쓸지, MySQL을 쓸지 아직 결정하지 않아도 됩니다. 이 결정은 논리적 설계 단계에서 이루어집니다.
:::

## 🔧 논리적 설계 단계

### 논리적 설계란 무엇인가요?

개념적 설계가 끝나서 ER 다이어그램이 만들어지면, 이제 이것을 실제 데이터베이스에서 사용할 수 있는 형태로 변환해야 합니다. 이 과정을 **논리적 설계(Logical Design)** 라고 합니다.

관계형 데이터베이스(우리가 흔히 아는 표 형태의 DB)를 사용한다면, ER 다이어그램을 **테이블(표)** 형태로 변환합니다. 계층형 데이터베이스라면 트리(tree) 형태로, 네트워크형 데이터베이스라면 그래프(graph) 형태로 변환합니다.

즉, 논리적 설계는 **어떤 DBMS를 사용하느냐에 따라 달라지는 설계**입니다. 이를 "목표 DBMS에 종속적이다" 또는 "목표 DBMS에 의존적이다"라고 표현합니다.

| 데이터베이스 유형 | 변환 결과 형태 |
|---|---|
| 관계형 DB (Oracle, MySQL, MSSQL 등) | 테이블(표) 형태 |
| 계층형 DB | 트리(나무 구조) 형태 |
| 네트워크형 DB | 그래프(그물망 구조) 형태 |

### 논리적 설계에서 하는 핵심 작업들

**① 정규화(Normalization) 수행**

논리적 설계의 핵심 작업 중 하나가 바로 **정규화**입니다. 정규화란 데이터베이스를 최적화하기 위해 테이블 구조를 정리하고 중복을 제거하는 과정입니다. 예를 들어 하나의 테이블에 너무 많은 정보가 들어있어서 데이터가 중복되거나 이상 현상이 발생할 때, 이를 여러 테이블로 분리하여 깔끔하게 만드는 것입니다.

:::tip 정규화는 나중에 자세히!
정규화는 굉장히 중요하고 복잡한 주제입니다. 이 문서에서는 "논리적 설계 단계에서 정규화가 이루어진다"는 것만 기억해두세요. 구체적인 정규화 방법과 개념은 별도의 강의에서 자세하게 다룹니다.
:::

**② 트랜잭션 인터페이스 설계**

개념적 설계에서 트랜잭션을 "모델링"했다면, 논리적 설계에서는 그 트랜잭션들을 실제로 **어떻게 연결하고 인터페이스(interface, 연결 방식)를 만들지** 설계합니다. 각 트랜잭션이 어떤 테이블에 접근하고, 어떤 경로로 데이터를 처리할지의 큰 그림을 그립니다.

**③ 목표 DBMS 선정**

논리적 설계 단계에서는 실제로 어떤 DBMS 제품을 사용할지도 결정합니다. 이때 고려하는 요소들은 다음과 같습니다.

| 고려 요소 | 설명 |
|---|---|
| 기능 | 필요한 기능을 제공하는가 |
| 라이선스 비용 | 구매/사용 비용이 얼마인가 |
| 유지보수 비용 | 버전 업그레이드, 유지보수에 드는 비용 |
| 하드웨어 비용 | 서버 등 인프라 구입 비용 |
| 마이그레이션 비용 | 기존 DB에서 새 DB로 전환하는 비용 |
| 교육 비용 | 직원들이 새 시스템을 배우는 데 드는 비용 |
| 운영 비용 | 실제 운영 중 드는 인건비 및 유지비 |

**④ 성능 평가**

논리적 설계 단계에서는 다음 요소들을 고려하여 설계의 성능을 평가합니다.
- **데이터의 양**: 얼마나 많은 데이터가 저장될 것인가
- **처리 빈도수**: 얼마나 자주 데이터가 조회/변경되는가
- **작업량**: 동시에 얼마나 많은 작업이 이루어지는가
- **레코드 접근량 및 전송량**: 한 번에 얼마나 많은 데이터를 읽고 쓰는가
- **DB의 전체 크기**: 데이터베이스의 총 용량이 얼마나 될 것인가

## ⚙️ 물리적 설계 단계

### 물리적 설계란 무엇인가요?

논리적 설계가 끝나면 마지막 설계 단계인 **물리적 설계(Physical Design)** 로 넘어갑니다. 이 단계는 **저장장치 관점**에서의 설계입니다. 즉, 실제로 데이터가 하드디스크에 어떻게 저장되고, 어떤 방식으로 빠르게 접근할 수 있을지를 결정합니다.

집 짓기에 비유하면, 논리적 설계가 "방의 배치와 구조"를 결정했다면, 물리적 설계는 "어떤 자재를 쓸지, 전기 배선은 어떻게 할지, 단열재는 무엇을 쓸지"를 결정하는 단계입니다. 사용자 눈에는 보이지 않지만, 성능에 직접적으로 영향을 미치는 매우 중요한 단계입니다.

### 물리적 설계에서 하는 핵심 작업들

**① 저장 레코드 양식 결정**

데이터를 실제 저장 장치에 어떤 형태로 저장할지 결정합니다. 이 단계에서 다루는 데이터를 **저장 레코드(stored record)** 라고 부릅니다.

**② 접근 경로(Access Path) 설계**

접근 경로란 데이터를 찾아가는 방법입니다. 마치 도서관에서 책을 찾을 때 번호 순서대로 찾는 것(순차 구조)이 빠른지, 아니면 색인(index)을 활용해 찾는 것이 빠른지를 결정하는 것과 같습니다.

주요 접근 방법으로는 다음과 같은 것들이 있습니다.
- **순차 구조**: 데이터를 처음부터 순서대로 접근
- **인덱스(Index)**: 별도의 색인을 만들어 빠른 검색을 지원
- **해싱(Hashing)**: 특정 계산식을 통해 데이터 위치를 바로 찾아가는 방식

**③ 트랜잭션 상세 설계**

개념적 설계에서 트랜잭션을 **모델링**하고, 논리적 설계에서 트랜잭션 **인터페이스를 설계**했다면, 물리적 설계에서는 트랜잭션의 **상세 설계**를 합니다. 실제로 데이터에 접근하는 세부적인 방법과 순서까지 결정합니다.

### 물리적 설계에서 중요한 성능 지표 3가지

물리적 설계를 잘 하기 위해서는 다음 세 가지 성능 지표를 반드시 고려해야 합니다.

**① 응답 시간(Response Time)**

사용자가 시스템에 어떤 요청을 보냈을 때, 최종 결과를 돌려받을 때까지 걸리는 시간입니다. 예를 들어 네이버에서 "오늘의 날씨"를 검색하고 버튼을 눌렀을 때, 결과가 화면에 나타나기까지 걸리는 시간이 바로 응답 시간입니다. 당연히 이 시간이 짧을수록 좋은 시스템입니다.

**② 저장 공간의 효율화(Storage Efficiency)**

물리적으로 데이터를 저장하는 공간을 얼마나 효율적으로 사용하느냐입니다. 너무 낭비해도 안 되고, 너무 꽉 채워도 문제가 됩니다. **블로킹 팩터(blocking factor)**, 즉 블록 사이즈를 얼마로 설정하느냐에 따라 I/O(입출력) 횟수가 달라지고, 이것이 전체 성능에 큰 영향을 줍니다.

**③ 트랜잭션 처리도(Transaction Throughput)**

**처리도(Throughput)** 란 단위 시간당 처리할 수 있는 트랜잭션의 수를 말합니다. 예를 들어 "Throughput이 1,000이다"라는 말은 1초에 1,000개의 작업을 처리할 수 있다는 뜻입니다. 이 수치가 높을수록 더 많은 사용자를 동시에 처리할 수 있는 고성능 시스템입니다.

```
예시:
- Throughput = 100  → 초당 100건 처리 가능 (소규모 시스템)
- Throughput = 1,000 → 초당 1,000건 처리 가능 (중규모 시스템)
- Throughput = 10,000 → 초당 10,000건 처리 가능 (대규모 시스템)
```

## 🛠️ 구현 단계: 드디어 실제 데이터베이스를 만든다!

### 구현이란 무엇인가요?

설계가 모두 끝나면 드디어 실제 데이터베이스를 만드는 **구현(Implementation)** 단계에 들어갑니다. 설계 도면이 완성됐으니, 이제 실제로 집을 짓는 단계입니다.

구현은 **DDL(Data Definition Language, 데이터 정의어)** 명령어를 사용하여 이루어집니다. DDL의 대표적인 명령어는 다음 세 가지입니다.

| 명령어 | 역할 | 예시 |
|---|---|---|
| `CREATE` | 새로운 테이블/DB 생성 | `CREATE TABLE 학생 (...)` |
| `ALTER` | 기존 구조 변경 | `ALTER TABLE 학생 ADD 이메일 VARCHAR(100)` |
| `DROP` | 테이블/DB 삭제 | `DROP TABLE 학생` |

아래는 실제 테이블을 생성하는 SQL 예시입니다.

```sql
-- 학생 테이블 생성 예시

-- ❌ 잘못된 예시: 기본키(PRIMARY KEY) 설정 없이 테이블 생성
CREATE TABLE 학생 (
    학번 VARCHAR(10),   -- 학번 컬럼 (문자열 10자리)
    이름 VARCHAR(20),   -- 이름 컬럼 (문자열 20자리)
    학과 VARCHAR(30)    -- 학과 컬럼 (문자열 30자리)
    -- 기본키가 없어서 중복 데이터가 들어올 수 있음!
);

-- ✅ 올바른 예시: 기본키를 명확히 지정한 테이블 생성
CREATE TABLE 학생 (
    학번   VARCHAR(10)  NOT NULL,    -- 학번: 반드시 값이 있어야 함 (NULL 불가)
    이름   VARCHAR(20)  NOT NULL,    -- 이름: 반드시 값이 있어야 함
    학과   VARCHAR(30),              -- 학과: 선택 입력 가능
    나이   INT,                      -- 나이: 정수형 숫자
    CONSTRAINT PK_학생               -- 제약 조건 이름 지정
        PRIMARY KEY (학번)           -- 학번을 기본키로 설정 (유일하고 NULL 불가)
);

-- 교수 테이블 생성
CREATE TABLE 교수 (
    교수번호  VARCHAR(10)  NOT NULL,  -- 교수번호: 반드시 값이 있어야 함
    이름      VARCHAR(20)  NOT NULL,  -- 이름: 반드시 값이 있어야 함
    소속학과  VARCHAR(30),            -- 소속 학과
    CONSTRAINT PK_교수
        PRIMARY KEY (교수번호)        -- 교수번호를 기본키로 설정
);

-- 학생 테이블에 지도교수 외래키 추가
ALTER TABLE 학생
    ADD 지도교수번호 VARCHAR(10),     -- 지도교수번호 컬럼 추가
    ADD CONSTRAINT FK_지도교수        -- 외래키 제약 조건 이름
        FOREIGN KEY (지도교수번호)    -- 학생 테이블의 지도교수번호 컬럼이
        REFERENCES 교수(교수번호);    -- 교수 테이블의 교수번호를 참조함
```

**코드 한 줄씩 설명:**

- `CREATE TABLE 학생`: "학생"이라는 이름의 테이블을 새로 만듭니다.
- `학번 VARCHAR(10) NOT NULL`: 학번은 최대 10자리 문자열이며, 반드시 값이 있어야 합니다(NULL 불가).
- `CONSTRAINT PK_학생 PRIMARY KEY (학번)`: 학번을 기본키로 지정합니다. 기본키는 값이 반드시 있어야 하고(NOT NULL), 중복될 수 없습니다(UNIQUE).
- `ALTER TABLE 학생 ADD 지도교수번호`: 이미 만들어진 학생 테이블에 새 컬럼을 추가합니다.
- `FOREIGN KEY (지도교수번호) REFERENCES 교수(교수번호)`: 학생의 지도교수번호는 반드시 교수 테이블에 존재하는 교수번호여야 합니다. 이것이 참조 무결성을 지켜주는 외래키입니다.

### 구현 단계에서 평가하는 것들

구현이 완료되면 다음 항목들을 평가합니다.

- **질의 응답 시간**: 데이터를 검색 요청했을 때 첫 결과가 화면에 나타나기까지 걸리는 시간
- **트랜잭션 갱신 시간**: 데이터를 수정(UPDATE)하는 데 걸리는 시간
- **보고서 작성 시간**: 데이터를 취합하여 보고서를 출력하는 데 걸리는 시간
- **저장 공간 사용량**: 주기억장치(RAM)와 보조기억장치(하드디스크)를 얼마나 사용하는지
- **CPU 및 I/O 성능**: CPU 사용률과 입출력 횟수 및 성능

## 🎯 데이터베이스 설계의 목표

### 왜 설계 목표를 명확히 해야 할까요?

설계를 시작하기 전에 "이 데이터베이스를 만들어서 무엇을 달성할 것인가?"라는 목표를 명확히 해야 합니다. 목표 없이 설계를 하면 방향을 잃게 됩니다. 데이터베이스 설계의 핵심 목표들을 살펴보겠습니다.

### 핵심 설계 목표

**① 사용자 요구 조건 만족**

가장 중요한 목표입니다. 아무리 기술적으로 훌륭한 데이터베이스라도 사용자가 원하는 것을 제공하지 못한다면 아무 의미가 없습니다. 사용자들이 요청한 기능들이 모두 동작해야 하고, 사용자들이 원하는 방식으로 데이터를 검색하고 관리할 수 있어야 합니다.

**② 자연스럽고 이해하기 쉬운 정보 구조 제공**

관계형 데이터베이스의 테이블 형태는 우리가 일상에서 보는 Excel 표와 비슷하기 때문에 직관적으로 이해하기 쉽습니다. 이처럼 사용자가 쉽게 이해하고 활용할 수 있는 구조로 설계되어야 합니다.

**③ 성능 요소 고려**

사용자 요구 사항만 고려하면 안 됩니다. 가성비, 즉 **투입되는 자원 대비 얼마만큼의 성능이 나오는지**도 반드시 고려해야 합니다. 구체적으로는 다음 요소들을 고려합니다.

- **응답 시간**: 얼마나 빠르게 결과를 보여줄 것인가
- **처리 시간**: 작업 하나를 처리하는 데 얼마나 걸리는가
- **저장 공간**: 얼마나 많은 저장 공간이 필요한가
- **동시 접속자 수**: 몇 명이 동시에 사용할 것인가 (1만 명? 5만 명? 30만 명?)

## 🛡️ 데이터베이스 설계 시 고려해야 할 품질 요소

### 무결성 (Integrity) - 데이터의 정확성을 지켜라!

**무결성**이란 한마디로 **데이터의 정확성**입니다. 잘못된 데이터가 들어오지 못하도록 막고, 데이터 간의 관계가 항상 올바르게 유지되도록 하는 것입니다.

무결성에는 세 가지 종류가 있습니다.

**① 개체 무결성(Entity Integrity)**

기본키(Primary Key, 각 행을 유일하게 식별하는 값)에 관한 규칙입니다.
- 기본키는 반드시 값이 있어야 합니다 (NULL 불가)
- 기본키는 중복될 수 없습니다 (UNIQUE)

예를 들어 학번이 기본키라면, 학번이 없는 학생은 존재할 수 없고, 같은 학번을 가진 두 학생이 존재할 수 없습니다. 주민번호가 없는 대한민국 국민이 없는 것과 마찬가지입니다.

```sql
-- ❌ 잘못된 예시: 기본키 없이 데이터 삽입 (중복 허용)
INSERT INTO 학생 VALUES ('홍길동', '컴퓨터공학과');  -- 학번 없음!
INSERT INTO 학생 VALUES ('홍길동', '컴퓨터공학과');  -- 똑같은 데이터 중복 삽입 가능

-- ✅ 올바른 예시: 기본키가 있는 경우
INSERT INTO 학생 (학번, 이름, 학과) VALUES ('2024001', '홍길동', '컴퓨터공학과');
-- 만약 아래를 실행하면 기본키 중복 오류 발생!
-- INSERT INTO 학생 (학번, 이름, 학과) VALUES ('2024001', '김철수', '전자공학과');
-- ERROR: Duplicate entry '2024001' for key 'PRIMARY'
```

**② 참조 무결성(Referential Integrity)**

외래키(Foreign Key, 다른 테이블의 기본키를 참조하는 값)에 관한 규칙입니다. 외래키가 참조하는 값은 반드시 참조 대상 테이블에 실제로 존재해야 합니다.

예를 들어 교수 테이블에 'A1', 'A2', 'A3' 교수만 있는데, 어떤 학생의 지도교수 번호가 'A4'라면 이는 참조 무결성 위반입니다. 존재하지 않는 교수를 참조하고 있기 때문입니다.

```sql
-- 교수 테이블에 있는 데이터
-- 교수번호: A1 (김교수), A2 (이교수), A3 (박교수)

-- ❌ 잘못된 예시: 존재하지 않는 교수번호 참조
INSERT INTO 학생 (학번, 이름, 지도교수번호)
VALUES ('2024001', '홍길동', 'A4');  -- 'A4' 교수는 존재하지 않음!
-- ERROR: Foreign key constraint violation!

-- ✅ 올바른 예시: 실제로 존재하는 교수번호 참조
INSERT INTO 학생 (학번, 이름, 지도교수번호)
VALUES ('2024001', '홍길동', 'A1');  -- 'A1' 교수는 존재함 ✓

-- 외래키는 NULL 값을 가질 수 있음 (아직 지도교수가 배정되지 않은 경우)
INSERT INTO 학생 (학번, 이름, 지도교수번호)
VALUES ('2024002', '김영희', NULL);  -- NULL: 아직 지도교수 미배정
```

:::warning 외래키와 NULL
기본키는 절대로 NULL 값을 가질 수 없지만, **외래키는 NULL 값을 가질 수 있습니다**. 외래키가 NULL이라는 것은 "아직 참조할 대상을 결정하지 못했다"는 의미입니다. 예를 들어 학생이 아직 지도교수가 배정되지 않은 경우 지도교수번호 컬럼이 NULL이 될 수 있습니다.
:::

**③ 도메인 무결성(Domain Integrity)**

각 컬럼(속성)에 입력될 수 있는 값의 범위(도메인)를 지키는 것입니다. 예를 들어 "점수"는 0~100 사이의 값만 입력되어야 하고, "성별"은 '남' 또는 '여'만 입력되어야 합니다.

```sql
-- ✅ 올바른 예시: 도메인 무결성을 위한 제약 조건 추가
CREATE TABLE 성적 (
    학번    VARCHAR(10)  NOT NULL,           -- 학번
    과목코드 VARCHAR(10) NOT NULL,           -- 과목 코드
    점수    INT CHECK (점수 >= 0 AND 점수 <= 100),  -- 점수: 0~100만 허용
    성별    CHAR(1) CHECK (성별 IN ('남', '여')),   -- 성별: '남' 또는 '여'만 허용
    PRIMARY KEY (학번, 과목코드)             -- 복합 기본키
);

-- ❌ 도메인 무결성 위반 예시
INSERT INTO 성적 (학번, 과목코드, 점수)
VALUES ('2024001', 'CS101', 150);  -- 점수가 100을 초과! 오류 발생
-- ERROR: Check constraint violation!
```

### 일관성 (Consistency) - 항상 같은 결과를 보장하라!

**일관성**이란 동일한 조건으로 데이터를 조회했을 때 항상 같은 결과가 나와야 한다는 것입니다. 내가 내 집 주소를 바꾸지 않았는데, 오늘 조회와 내일 조회에서 다른 주소가 나온다면 그 데이터베이스는 일관성을 잃은 것입니다.

### 회복 (Recovery) - 장애 발생 시 원상태로 복구하라!

시스템 장애(정전, 서버 다운, 프로그램 오류 등)가 발생했을 때 데이터베이스를 장애 이전의 정상 상태로 복구할 수 있어야 합니다. 마치 자동차 블랙박스처럼, 데이터베이스도 모든 변경 사항을 로그로 기록해두고 필요시 복구할 수 있어야 합니다.

### 보안 (Security) - 허가된 사람만 접근하게 하라!

데이터베이스를 무단 접근이나 해킹으로부터 보호해야 합니다. 보안 기법으로는 다음과 같은 것들이 있습니다.

- **임의 접근 제어(DAC, Discretionary Access Control)**: 아이디와 비밀번호를 통해 접근을 제어하는 방식. 일반적인 로그인 시스템이 이에 해당합니다.
- **강제 접근 제어(MAC, Mandatory Access Control)**: 문서 자체에 등급을 부여하여 접근을 제한하는 방식. 군사 기밀 문서의 "극비", "대외비", "비밀", "일반" 등급처럼 데이터에 보안 등급을 부여하고 사용자의 보안 등급에 따라 접근을 허용하거나 차단합니다.

### 효율성과 확장성

- **효율성**: 응답 시간은 짧게, 저장 공간은 최적화하여, 생산성은 높게 유지해야 합니다.
- **확장성**: 처음에 사용자 100명으로 시작한 시스템이 10만 명으로 늘어나더라도, 기존 시스템 운영에 큰 영향 없이 확장할 수 있어야 합니다. 확장성을 고려하지 않고 설계하면 나중에 시스템을 아예 새로 만들어야 하는 상황이 생깁니다.

:::danger 확장성을 무시하면 안 됩니다!
처음에 소규모로 시작하더라도 반드시 미래의 확장을 고려하여 설계해야 합니다. 확장성 없이 설계된 시스템은 사용자가 늘어날 때 새로 개발해야 하는데, 이는 엄청난 비용과 시간 낭비를 초래합니다.
:::

## 🗺️ 세 단계 설계의 전체 흐름 한눈에 보기

### 데이터 중심 설계 vs 처리 중심 설계

데이터베이스 설계 작업은 두 가지 관점에서 함께 이루어져야 합니다.

**데이터 중심 설계**: 어떤 데이터를 저장할 것인가? 데이터의 구조가 어떠한가? (집의 방 배치, 구조와 같은 개념)

**처리 중심 설계**: 그 데이터로 어떤 작업을 할 것인가? (집에서 사람이 어떻게 생활하고 이동하는가와 같은 개념)

이 두 가지를 모두 고려해야 합니다. 구조만 좋고 실제 사용하기 불편하거나, 사용하기 편하지만 구조가 엉망이라면 좋은 데이터베이스가 아닙니다.

### 세 단계 설계 비교 총정리

| 구분 | 개념적 설계 | 논리적 설계 | 물리적 설계 |
|---|---|---|---|
| **관점** | 사용자 관점 | 데이터베이스 조직 관점 | 저장장치 관점 |
| **주요 결과물** | ER 다이어그램 | 테이블 스키마 (관계형의 경우) | 인덱스, 저장 구조 |
| **DBMS 종속성** | 독립적 (어떤 DBMS든 무관) | 종속적 (DBMS 유형에 따라 다름) | 종속적 (특정 DBMS에 최적화) |
| **트랜잭션 처리** | 트랜잭션 모델링 | 트랜잭션 인터페이스 설계 | 트랜잭션 상세 설계 |
| **주요 작업** | ER 모델링, 추상화 | 정규화, DBMS 선정 | 인덱스 설계, 접근 경로 결정 |
| **추상화 수준** | 가장 높음 (가장 추상적) | 중간 | 가장 낮음 (가장 구체적) |

### 전체 프로세스 흐름

```
[작은 세계 / 현실]
       ↓
  요구사항 수집
       ↓
[요구 조건 분석]
  (요구조건 명세서 작성)
       ↓
[개념적 설계] ← DBMS 독립적
  (ER 다이어그램 작성)
       ↓
[논리적 설계] ← DBMS 종속적
  (테이블 설계 + 정규화 + DBMS 선정)
       ↓
[물리적 설계] ← DBMS 종속적
  (인덱스 설계 + 저장 구조 결정)
       ↓
[구현]
  (DDL: CREATE, ALTER, DROP)
       ↓
[운영]
  (실제 서비스 제공)
       ↓
[감시 및 개선]
  (성능 모니터링 + 피드백 → 앞 단계로 돌아가 보완)
```

:::tip 핵심 암기 포인트
- **개념적 설계** → ER 다이어그램 작성 → **DBMS 독립적**
- **논리적 설계** → 테이블 변환 + 정규화 → **DBMS 종속적**
- **물리적 설계** → 인덱스 + 저장 구조 → **DBMS 종속적**

이 세 가지 차이점을 확실히 구별할 수 있어야 합니다!
:::

## 📝 핵심 정리

- **데이터베이스 생명주기**는 요구조건 분석 → 설계 → 구현 → 운영 → 감시 및 개선의 5단계로 이루어지며, 각 단계 간 피드백이 가능한 순환 구조입니다.
- **요구 조건 분석**에서는 정적 요구 조건(데이터 구조), 동적 요구 조건(트랜잭션 종류/빈도), 범 기관적 제약 조건(예산, 정책)을 모두 고려하여 공식 요구 조건 명세서를 작성합니다.
- **개념적 설계**는 사용자 관점에서 ER 다이어그램을 작성하는 단계로, 어떤 DBMS를 사용하든 독립적으로 수행할 수 있습니다.
- **논리적 설계**는 ER 다이어그램을 목표 DBMS에 맞는 스키마(테이블, 트리, 그래프 등)로 변환하고, 정규화와 트랜잭션 인터페이스 설계를 수행하는 단계로 DBMS에 종속됩니다.
- **물리적 설계**는 저장장치 관점에서 실제 데이터 저장 방식, 접근 경로(인덱스, 해싱 등), 트랜잭션 상세 설계를 결정하며 응답 시간, 저장 공간 효율화, 처리도(Throughput)를 핵심 지표로 삼습니다.
- **구현**은 `CREATE`, `ALTER`, `DROP` 등 DDL 명령어로 실제 데이터베이스를 생성하고, 성능(응답 시간, 갱신 시간)과 저장 공간을 평가하는 단계입니다.
- **설계의 세 가지 목표**는 사용자 요구 조건 만족, 이해하기 쉬운 정보 구조 제공, 성능(응답 시간·처리 시간·저장 공간·동시 접속자 수) 고려입니다.
- **무결성**에는 개체 무결성(기본키는 유일하고 NULL 불가), 참조 무결성(외래키는 실제 존재하는 값만 참조), 도메인 무결성(속성 값은 정해진 범위 내)의 세 종류가 있습니다.
- **외래키**는 기본키와 달리 NULL 값을 가질 수 있으며, 이는 "아직 참조 대상이 결정되지 않았음"을 의미합니다.
- **데이터 중심 설계**(구조)와 **처리 중심 설계**(트랜잭션)를 모두 고려해야 완성도 높은 데이터베이스를 설계할 수 있습니다.
- **확장성**을 처음부터 고려하지 않으면, 사용자가 늘어날 때 시스템 전체를 새로 개발해야 하는 큰 비용이 발생할 수 있습니다.
