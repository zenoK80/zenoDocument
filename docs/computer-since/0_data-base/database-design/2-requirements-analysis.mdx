# 🏗️ 데이터베이스 설계: 요구조건 분석부터 운영·감시까지 완전 정복

데이터베이스를 만든다는 것은 단순히 SQL 명령어 몇 줄을 입력하는 것이 아닙니다. 마치 아파트를 짓기 전에 설계 도면을 그리고, 입주자의 요구사항을 반영하고, 건축 허가를 받는 것처럼 **체계적인 단계와 절차**가 필요합니다. 이 문서에서는 데이터베이스가 탄생하고 운영되기까지의 전체 흐름, 즉 **데이터베이스 생명주기(Life Cycle)**를 처음부터 끝까지 아주 자세하게 살펴봅니다.

---

## 🌱 데이터베이스 생명주기란 무엇인가?

### 생명주기(Life Cycle)의 개념

**생명주기(Life Cycle)**란 말 그대로 어떤 시스템이 태어나서 죽을 때까지의 전체 과정을 의미합니다. 사람으로 따지면 태어나고, 자라고, 일하고, 나이 들어 은퇴하는 과정과 비슷합니다. 데이터베이스도 마찬가지로 처음 **필요성이 제기되는 순간**부터 시작해서, **설계**, **구현**, **운영**, 그리고 **개선**에 이르는 일련의 단계를 거칩니다.

데이터베이스 생명주기는 크게 다음 5단계로 구성됩니다:

| 단계 | 이름 | 한 줄 설명 |
|---|---|---|
| 1단계 | **요구조건 분석** | 무엇을 만들어야 하는지 파악하는 단계 |
| 2단계 | **설계** (개념적 → 논리적 → 물리적) | 어떻게 만들 것인지 도면을 그리는 단계 |
| 3단계 | **구현** | 실제로 데이터베이스를 생성하는 단계 |
| 4단계 | **운영** | 사용자들이 실제로 사용하는 단계 |
| 5단계 | **감시 및 개선** | 문제를 발견하고 보완하는 단계 |

> 💡 **핵심 포인트:** 이 5단계는 단순히 순서대로 한 번만 진행되는 것이 아닙니다. 어느 단계에서든 문제가 발견되면 **이전 단계로 피드백(되돌아가기)**하여 수정할 수 있습니다. 이것이 생명주기의 핵심입니다.

### 아파트 건축에 빗댄 비유

데이터베이스 설계 과정을 아파트 건축에 비유하면 훨씬 이해하기 쉽습니다.

- **요구조건 분석** = 입주 예정자들이 "방은 3개, 화장실은 2개, 주차장은 넓게 해주세요"라고 요청하는 과정
- **설계** = 건축가가 그 요구를 바탕으로 설계 도면을 그리는 과정
- **구현** = 실제 벽돌을 쌓고 건물을 짓는 과정
- **운영** = 입주자들이 실제로 살기 시작하는 과정
- **감시 및 개선** = 살다 보니 불편한 점이 생겨 리모델링하는 과정

이처럼 데이터베이스도 **충분한 분석과 설계 없이 무턱대고 구현하면** 나중에 엄청난 수정 비용이 발생합니다. 처음부터 제대로 된 절차를 따르는 것이 왜 중요한지 이해가 되셨나요?

---

## 🔍 1단계: 요구조건 분석 — "무엇을 만들어야 하는가?"

### 요구조건 분석이란?

요구조건 분석(Requirements Analysis)은 데이터베이스 프로젝트의 **출발점**입니다. 사용자들이 "이런 시스템이 필요해요", "우리 부서에서 이런 업무 처리를 위해 데이터베이스가 필요해요"라고 하는 **다양한 요구 사항을 수집하고 분석하는 단계**입니다.

이 단계에서 가장 중요한 것은 단순히 **현재 사용자의 요구**만 파악하는 것이 아니라, **미래에 시스템을 사용할 잠재적 사용자의 요구**까지 고려해야 한다는 점입니다. 왜냐하면 현재 사용자만 고려해서 시스템을 만들었다가, 나중에 새로운 사용자나 새로운 업무가 추가되면 기존 시스템과 맞지 않아 처음부터 다시 만들어야 하는 상황이 발생할 수 있기 때문입니다.

:::info 잠재적 사용자를 왜 고려해야 할까요?
예를 들어, 현재 직원이 50명인 회사가 데이터베이스를 만든다고 가정해 봅시다. 만약 2년 후에 직원이 500명으로 늘어날 계획이 있다면, 처음부터 500명을 수용할 수 있는 규모로 설계해야 합니다. 그렇지 않으면 2년 후에 데이터베이스를 완전히 새로 만들어야 하는 상황이 생깁니다.
:::

### 요구조건 분석에서 수집하는 정보

요구조건 분석 단계에서 수집해야 할 정보는 크게 다음과 같습니다:

- **사용자 정보**: 누가 이 데이터베이스를 사용하는가? (현재 사용자 + 잠재적 사용자)
- **업무 정보**: 어떤 업무를 처리하기 위해 데이터베이스가 필요한가?
- **데이터 종류**: 어떤 종류의 데이터를 저장해야 하는가?
- **데이터 흐름**: 데이터가 어떻게 입력되고, 처리되고, 출력되는가?
- **데이터베이스의 범위**: 어디까지 개발할 것인가? (개발 범위 결정)

### 요구조건의 세 가지 유형

요구조건은 크게 **정적인 요구조건**, **동적인 요구조건**, **범 기관적 제약 조건**으로 나눌 수 있습니다.

#### 1) 정적인 정보 구조에 대한 요구조건

**정적인(Static) 요구조건**은 데이터베이스의 **구조**에 관한 것입니다. 마치 집을 지을 때 방이 몇 개인지, 화장실은 어디에 위치하는지를 정하는 것과 같습니다.

예를 들어 학교 데이터베이스를 만든다고 하면:
- 어떤 테이블(표)들이 필요한가? (학생 테이블, 교수 테이블, 과목 테이블 등)
- 각 테이블에는 어떤 속성(열/컬럼)들이 있는가? (학번, 이름, 학과 등)
- 테이블들 사이의 관계는 어떠한가? (학생과 교수의 관계, 학생과 과목의 관계 등)
- 그 관계는 **일대일(1:1)**, **일대다(1:N)**, **다대다(N:M)** 중 어느 것인가?

이처럼 데이터베이스의 **정적인 구조**를 정의하는 것이 정적인 요구조건 분석입니다.

#### 2) 동적인 데이터베이스 처리에 대한 요구조건

**동적인(Dynamic) 요구조건**은 데이터베이스에서 **어떤 작업이 얼마나 자주 일어나는가**에 관한 것입니다. 집으로 비유하면, 사람이 집 안에서 어떻게 이동하고 생활하는지(동선)와 같습니다.

학교 데이터베이스를 예로 들면:
- 학생이 **수강 신청**을 한다 → 데이터 **삽입(INSERT)** 작업
- 학생이 **수강 변경**을 한다 → 데이터 **갱신(UPDATE)** 작업
- 학생이 **수강 철회**를 한다 → 데이터 **삭제(DELETE)** 작업
- 학생이 **성적을 조회**한다 → 데이터 **검색(SELECT)** 작업

이러한 삽입, 삭제, 갱신, 검색 작업들을 우리는 **트랜잭션(Transaction, 작업 묶음)**이라고 부릅니다. 동적인 요구조건 분석에서는 이러한 트랜잭션이 얼마나 자주, 얼마나 많이 발생하는지를 파악합니다. 검색이 주로 많이 일어나는 시스템인지, 아니면 삽입과 삭제가 빈번한 시스템인지에 따라 설계 방향이 달라지기 때문입니다.

#### 3) 범 기관적 제약 조건

**범 기관적(Organizational) 제약 조건**은 조직 내부의 **현실적인 제약**에 관한 것입니다. 아무리 좋은 설계를 해도 예산이 부족하거나 회사 방침에 맞지 않으면 실현하기 어렵습니다.

예를 들어:
- **예산 제약**: 모든 요구사항을 만족하려면 5억 원이 필요한데, 회사 예산은 1억 원밖에 없다면 → 요구사항을 조정해야 합니다
- **경영 목표**: "이 데이터베이스로 고객 만족도를 5% 향상시켜야 한다", "직원 업무 효율성을 10% 올려야 한다"
- **정책 및 규정**: 회사의 보안 정책, 개인정보보호법 등 각종 규정 준수

이러한 현실적인 제약 조건들을 무시하고 설계하면, 나중에 아무리 훌륭한 시스템을 만들어도 실제로 사용할 수 없는 상황이 됩니다.

### 공식적인 요구조건 명세서

요구조건 분석의 최종 산출물은 **공식적인 요구조건 명세서**입니다. 이것은 단순한 메모가 아니라 **결재를 받은 공식 문서**로, 이후 모든 개발의 기준이 됩니다.

:::tip 실무에서의 요구조건 명세서
실제 IT 회사에서 개발 업무를 할 때는 무작정 개발하지 않습니다. 개발을 시작하기 전에 반드시 요구조건 명세서를 포함한 다양한 문서들이 작성되고 승인을 받습니다. 이 문서가 없으면 나중에 "이건 제가 요청한 것과 다른데요?"라는 분쟁이 발생할 수 있습니다. 요구조건 명세서는 개발자와 고객 사이의 계약서와 같은 역할을 합니다.
:::

### 요구조건 분석 도구들

요구조건을 문서화하기 위해 여러 가지 도구들이 사용됩니다.

#### HIPO 다이어그램 (Hierarchy Input Process Output)

**HIPO 다이어그램**은 시스템의 기능들을 **계층 구조(트리 형태)**로 표현한 도표입니다. 마치 회사의 조직도처럼, 큰 업무가 작은 세부 업무들로 나뉘는 구조를 시각적으로 보여줍니다.

이름에서 알 수 있듯이:
- **H (Hierarchy)**: 계층 구조
- **I (Input)**: 입력 데이터
- **P (Process)**: 처리 과정
- **O (Output)**: 출력 결과

예를 들어 재고 관리 시스템이라면:
- 최상위: **재고 관리 시스템**
  - 중간 레벨: **거래 처리** / **재고량 업데이트** / **보고서 출력**
    - 하위 레벨: 각 중간 레벨이 다시 세부 기능으로 분리

이 다이어그램을 보면 전체 시스템의 구조와 업무 흐름을 한눈에 파악할 수 있습니다.

#### 데이터 플로우 다이어그램 (Data Flow Diagram, DFD)

**데이터 플로우 다이어그램(DFD)**은 데이터가 시스템 내에서 **어떻게 흘러가는지**를 그림으로 표현한 것입니다. 판매부, 고객 서비스부 등 각 부서와 그 사이를 오가는 데이터의 흐름을 화살표로 표현합니다. 예를 들어 고객이 주문서를 접수하면 → 신용도 검토 → 재고 확인 → 배송 지시 등의 데이터 흐름을 시각화합니다.

#### SADT (Structured Analysis and Design Technique)

**SADT**는 소프트 테크(SofTech)사에서 개발한 구조적 분석 및 설계 기법입니다. 아주 복잡하고 대규모인 시스템을 분석할 때 사용하며, 입력 데이터, 제어 데이터, 메커니즘, 출력 결과를 구조적인 도형으로 표현합니다.

:::note 참고
HIPO 다이어그램, DFD, SADT는 데이터베이스 과목보다는 **시스템 분석 및 설계**, **소프트웨어 공학** 과목에서 더 자세하게 다룹니다. 데이터베이스에서는 이런 도구들이 요구조건 분석 단계에서 사용된다는 것만 이해하고 넘어가면 됩니다.
:::

## 📐 2단계: 설계 — "어떻게 만들 것인가?"

### 설계의 세 가지 레벨

요구조건 분석이 끝나면 본격적인 **설계 단계**에 들어갑니다. 데이터베이스 설계는 크게 **개념적 설계**, **논리적 설계**, **물리적 설계** 세 가지로 나뉩니다. 이 세 가지는 마치 건축 설계에서 큰 그림을 먼저 그리고, 점점 세부적인 도면을 완성해 나가는 것과 같습니다.

| 설계 단계 | 관점 | DBMS 종속 여부 | 주요 산출물 |
|---|---|---|---|
| **개념적 설계** | 사용자 관점 | DBMS 독립적 | ER 다이어그램 |
| **논리적 설계** | 데이터베이스 조직 관점 | DBMS 종속적 | 테이블(스키마) |
| **물리적 설계** | 저장 장치 관점 | DBMS 종속적 | 인덱스, 저장 구조 |

### 설계의 두 가지 축: 데이터 중심 vs 처리 중심

설계 작업은 두 가지 측면을 동시에 고려해야 합니다.

**데이터 중심 설계**는 "어떤 데이터를 저장할 것인가?"에 초점을 맞춥니다. 어떤 테이블이 필요한지, 각 테이블에 어떤 속성들이 있는지를 정의합니다. 집으로 비유하면 방의 개수, 화장실 위치, 부엌 크기 등 집의 **구조**를 설계하는 것입니다.

**처리 중심 설계**는 "데이터베이스에서 어떤 작업을 할 것인가?"에 초점을 맞춥니다. 검색이 많은지, 삽입이 많은지, 동시에 몇 명의 사용자가 접속하는지를 고려합니다. 집으로 비유하면 사람이 집 안에서 어떻게 이동하고 생활하는지 **동선**을 고려하는 것입니다.

:::warning 두 가지 모두 고려해야 합니다!
데이터 구조만 고려하고 트랜잭션을 무시하거나, 반대로 트랜잭션만 고려하고 데이터 구조를 소홀히 하면 성능이 크게 저하될 수 있습니다. 항상 **두 가지를 균형 있게** 고려해야 합니다.
:::

### 개념적 설계: 큰 그림 그리기

**개념적 설계(Conceptual Design)**는 사용자의 관점에서 데이터베이스의 전체 구조를 그리는 단계입니다. 이 단계에서는 오라클을 쓸지, MySQL을 쓸지, MSSQL을 쓸지 전혀 신경 쓰지 않아도 됩니다. 어떤 DBMS(데이터베이스 관리 시스템)를 사용하더라도 독립적으로 적용할 수 있는 **범용적인 설계**를 하는 단계입니다.

개념적 설계의 핵심 산출물은 **ER 다이어그램(Entity-Relationship Diagram, 개체-관계 다이어그램)**입니다. ER 다이어그램은 데이터베이스에 존재하는 **개체(Entity)**들과 그들 사이의 **관계(Relationship)**를 시각적으로 표현한 그림입니다.

예를 들어 학교 데이터베이스의 ER 다이어그램에는:
- **학생** 개체 (학번, 이름 등의 속성을 가짐)
- **교수** 개체 (교수번호, 이름, 소속 학과 등의 속성을 가짐)
- **학생과 교수 사이의 "지도" 관계** (한 명의 교수는 여러 명의 학생을 지도할 수 있음 → 1:N 관계)

이런 내용들이 도형으로 표현됩니다.

#### 추상화: 복잡한 현실을 단순하게

개념적 설계에서 중요한 개념이 바로 **추상화(Abstraction)**입니다. 추상화란 복잡한 내용을 감추고, 공통의 속성을 모아 하나의 단순한 개념으로 표현하는 것입니다.

예를 들어 실제 학생에 대한 정보는 매우 많습니다:
- 학번, 이름, 연락처, 학과, 나이, 취미, 출신 고등학교, 주소, 혈액형...

이 모든 것을 다 나열하는 대신, 우리는 필요한 것들만 골라서 **"학생"**이라는 하나의 개체(Entity)로 묶어서 표현합니다. 이것이 추상화입니다.

추상화에는 두 가지 방식이 있습니다:

- **집단화(Aggregation)**: 여러 **속성들**을 묶어서 하나의 개체를 만드는 것
  - 예: 학번 + 이름 + 학과 → **학생** 개체
- **일반화(Generalization)**: 여러 **개체들**의 공통점을 묶어서 더 일반적인 개체를 만드는 것
  - 예: 대학원생 + 학부생 → **학생** 개체 (두 개체의 공통점을 추상화)

두 가지 모두 공통적인 것을 뽑아내는 작업이기 때문에, 넓은 의미에서 모두 **추상화**의 개념에 포함됩니다.

#### 트랜잭션 모델링

개념적 설계 단계에서는 ER 다이어그램을 그리는 것뿐만 아니라, **트랜잭션 모델링(Transaction Modeling)**도 함께 수행합니다. 트랜잭션 모델링이란 "이 데이터베이스에서 어떤 작업들이 이루어질 것인가?"를 **고차원적으로 정의**하는 것입니다. 검색 위주인지, 갱신이 잦은지, 삽입과 삭제가 많은지를 파악하여 이후 설계의 방향을 잡는 데 활용됩니다.

### 논리적 설계: 테이블로 변환하기

**논리적 설계(Logical Design)**는 개념적 설계에서 만든 ER 다이어그램을 **실제 데이터베이스 구조(스키마, Schema)**로 변환하는 단계입니다. 관계형 데이터베이스를 사용한다면 ER 다이어그램의 각 개체가 **테이블(Table)**로 변환됩니다.

이 단계부터는 어떤 DBMS를 사용할 것인지에 따라 결과가 달라집니다. 즉, **DBMS에 종속적**입니다.

| DBMS 유형 | 논리적 설계 결과 |
|---|---|
| 관계형 DB (Oracle, MySQL, MSSQL 등) | 테이블(Table) 형태 |
| 계층형 DB | 트리(Tree) 형태 |
| 네트워크형 DB | 그래프(Graph) 형태 |

#### 어떤 DBMS를 선택해야 할까?

논리적 설계 단계에서는 어떤 DBMS를 선택할지도 결정해야 합니다. DBMS 선택 시 고려해야 할 요소들은 다음과 같습니다:

- **기능**: 해당 DBMS가 필요한 기능을 모두 지원하는가?
- **비용**: 라이선스 비용, 하드웨어 구입 비용, 교육비, 운영 비용
- **유지보수**: 버전 업그레이드 비용, 기술 지원 가능 여부
- **마이그레이션 비용**: 기존 데이터베이스에서 새 DBMS로 전환하는 데 드는 비용
- **인건비 및 교육비**: 직원들이 새 DBMS를 배우는 데 드는 시간과 비용

#### 정규화(Normalization): 데이터 최적화

논리적 설계 단계에서 중요한 작업 중 하나가 바로 **정규화(Normalization)**입니다. 정규화란 데이터베이스의 **중복을 제거하고 데이터를 효율적으로 구조화**하는 과정입니다. 논리적 설계 단계에서 이 정규화 작업이 수행됩니다. 정규화의 자세한 내용은 이후 별도 단원에서 다룹니다.

#### 논리적 설계에서의 성능 평가

논리적 설계 단계에서는 다음과 같은 항목들을 평가합니다:
- **데이터의 양**: 전체 데이터가 얼마나 될 것인가?
- **처리 빈도수**: 각 트랜잭션이 얼마나 자주 발생하는가?
- **작업량**: 전체 작업의 규모는 어느 정도인가?
- **레코드 접근량 및 전송량**: 한 번의 작업에서 몇 개의 레코드를 다루는가?
- **DB 크기**: 전체 데이터베이스의 용량은 어느 정도인가?

#### 트랜잭션 인터페이스 설계

개념적 설계에서 트랜잭션을 **모델링**했다면, 논리적 설계에서는 그 트랜잭션들을 **서로 연결하는 인터페이스를 설계**합니다. 즉, 각 트랜잭션이 어떤 테이블에 어떻게 접근하는지를 구체적으로 정의합니다.

### 물리적 설계: 저장 장치 최적화

**물리적 설계(Physical Design)**는 논리적 설계에서 만든 테이블 구조를 실제 **저장 장치(하드디스크 등)**에 어떻게 효율적으로 저장할 것인지를 결정하는 단계입니다. 이 단계에서는 **인덱스(Index, 색인)**를 추가하는 것과 같은 성능 향상 작업이 이루어집니다.

물리적 설계는 데이터베이스 성능에 **아주 큰 영향**을 미칩니다. 아무리 논리적 설계가 잘 되어 있어도, 물리적 설계가 잘못되면 시스템이 느려지거나 저장 공간을 낭비할 수 있습니다.

#### 물리적 설계에서 고려하는 요소들

- **저장 레코드 양식**: 데이터를 어떤 형태로 저장할 것인가?
- **순서**: 데이터를 어떤 순서로 정렬하여 저장할 것인가?
- **접근 경로(Access Path)**: 데이터에 어떻게 접근할 것인가?
  - 순차 구조(Sequential): 처음부터 끝까지 순서대로 찾기
  - 해싱 구조(Hashing): 특정 공식으로 위치를 바로 계산하여 찾기
  - 인덱스 구조(Index): 색인을 만들어 빠르게 찾기
- **저장 공간 할당**: 저장 공간을 얼마나, 어떻게 할당할 것인가?

#### 물리적 설계의 세 가지 핵심 기준

**1. 응답 시간(Response Time)**

응답 시간이란 사용자가 어떤 요청을 보내고 나서 결과를 받을 때까지 걸리는 **총 시간**입니다. 예를 들어 네이버에서 "오늘의 날씨"를 검색하고 결과가 화면에 나타날 때까지 걸리는 시간이 바로 응답 시간입니다. 응답 시간은 **짧을수록 좋습니다.**

**2. 저장 공간의 효율화(Space Efficiency)**

물리적 설계에서는 저장 공간을 얼마나 효율적으로 사용할 것인지도 중요합니다. 너무 많은 공간을 낭비해도 안 되고, 그렇다고 너무 빡빡하게 사용해도 성능에 문제가 생길 수 있습니다. 블록 크기(블로킹 팩터)를 얼마로 설정하느냐에 따라 I/O(입출력) 횟수가 달라지고, 이것이 성능에 큰 영향을 줍니다.

**3. 트랜잭션 처리량(Throughput)**

**처리량(Throughput)**이란 단위 시간당 처리할 수 있는 트랜잭션의 수입니다. 예를 들어 처리량이 1,000이라면, 1초에 1,000개의 트랜잭션을 처리할 수 있다는 의미입니다. 처리량은 **높을수록 좋습니다.** 일반적으로 "초당 몇 개의 트랜잭션을 처리하는가(TPS, Transactions Per Second)"로 시스템 성능을 평가합니다.

물리적 설계에서는 **트랜잭션 상세 설계**도 수행합니다. 개념적 설계에서 트랜잭션을 모델링하고, 논리적 설계에서 트랜잭션 인터페이스를 설계했다면, 물리적 설계에서는 그 트랜잭션이 실제 저장 장치 수준에서 **어떻게 실행될 것인지를 상세하게 설계**합니다.

## 🔨 3단계: 구현 — "실제로 데이터베이스를 만들자!"

### 구현이란?

**구현(Implementation)** 단계는 앞에서 그린 설계 도면을 바탕으로 **실제로 데이터베이스를 생성**하는 단계입니다. 건축으로 비유하면, 설계 도면을 보고 실제로 벽돌을 쌓고 건물을 짓는 과정입니다.

구현 단계에서는 **DDL(Data Definition Language, 데이터 정의어)**이라는 SQL 명령어를 사용합니다.

### DDL 명령어로 데이터베이스 만들기

DDL에는 크게 세 가지 명령어가 있습니다:

| 명령어 | 의미 | 용도 |
|---|---|---|
| `CREATE` | 생성 | 테이블, 뷰, 데이터베이스 등을 새로 만든다 |
| `ALTER` | 변경 | 기존 테이블의 구조를 수정한다 |
| `DROP` | 삭제 | 테이블이나 데이터베이스를 삭제한다 |

```sql
-- ✅ 올바른 예시: 학생 테이블 생성
CREATE TABLE 학생 (
    학번     CHAR(10)    NOT NULL,   -- 학번: 10자리 문자, NULL 불가 (필수값)
    이름     VARCHAR(50) NOT NULL,   -- 이름: 최대 50자 문자, NULL 불가 (필수값)
    학과     VARCHAR(50),            -- 학과: 최대 50자 문자, NULL 허용
    학년     INT,                    -- 학년: 정수형, NULL 허용
    지도교수번호 CHAR(10),           -- 지도교수번호: 10자리 문자 (외래키로 활용 예정)
    PRIMARY KEY (학번)              -- 학번을 기본키로 지정 (유일하고 NULL 불가)
);

-- ✅ 교수 테이블 생성
CREATE TABLE 교수 (
    교수번호  CHAR(10)    NOT NULL,  -- 교수번호: 10자리 문자, NULL 불가 (필수값)
    이름      VARCHAR(50) NOT NULL,  -- 이름: 최대 50자 문자, NULL 불가 (필수값)
    소속학과  VARCHAR(50),           -- 소속학과: 최대 50자 문자, NULL 허용
    PRIMARY KEY (교수번호)          -- 교수번호를 기본키로 지정
);

-- ✅ 외래키 추가: 학생의 지도교수번호가 교수 테이블을 참조하도록 설정
ALTER TABLE 학생
ADD CONSTRAINT fk_지도교수
FOREIGN KEY (지도교수번호) REFERENCES 교수(교수번호);
-- 학생의 지도교수번호는 반드시 교수 테이블에 존재하는 교수번호여야 함 (참조무결성)
```

```sql
-- ❌ 잘못된 예시: 기본키 없이 테이블 생성
CREATE TABLE 학생 (
    이름     VARCHAR(50),   -- 이름만 있고 기본키가 없음
    학과     VARCHAR(50)    -- 동일한 이름을 가진 학생을 구분할 방법이 없음
);
-- 위와 같이 기본키가 없으면 데이터 중복이 발생할 수 있고
-- 특정 학생을 정확히 식별하는 것이 불가능합니다
```

### 구현 단계의 평가 항목

데이터베이스를 생성한 후에는 다음과 같은 항목들을 평가합니다:

- **저장 공간 사용량**: 주기억장치(RAM)와 보조기억장치(하드디스크)를 얼마나 점유하는가?
- **질의 응답 시간**: 검색 쿼리(질의)를 실행하고 첫 번째 결과가 화면에 나타날 때까지 걸리는 시간
- **트랜잭션 갱신 시간**: 데이터를 업데이트하는 데 걸리는 시간 (예: 회원 정보 수정)
- **보고서 작성 시간**: 대량의 데이터를 처리하여 보고서를 생성하는 데 걸리는 시간
- **CPU 및 I/O 현황**: CPU 사용률, 입출력 횟수 등 시스템 자원 사용 현황

이런 평가 결과를 분석하여 시스템이 목표 성능을 달성하는지 확인합니다. 만약 성능이 기대치에 미치지 못하면 이전 설계 단계로 돌아가 수정합니다.

## 📡 4단계: 운영과 5단계: 감시 및 개선

### 운영(Operation)

**운영** 단계는 실제로 사용자들이 데이터베이스를 사용하는 단계입니다. 직원들이 업무에 데이터베이스를 활용하고, 고객들이 서비스를 이용하는 바로 그 시점부터 운영이 시작됩니다. 운영 단계에서는 사용자의 요구에 따라 **지속적으로 서비스를 제공**합니다.

### 감시 및 개선(Monitoring & Improvement)

**감시 및 개선** 단계는 운영 중인 데이터베이스를 **지속적으로 모니터링**하고, 문제가 발견되면 개선하는 단계입니다. 다음과 같은 상황에서 감시 및 개선이 필요합니다:

- **성능 저하**: 응답 시간이 점점 느려지는 경우
- **보안 취약점 발견**: 해킹 시도나 보안 문제가 발생한 경우
- **권한 문제**: 접근 권한 설정에 오류가 생긴 경우
- **새로운 요구조건 등장**: 새로운 업무나 기능이 추가되어야 하는 경우
- **확장 필요**: 사용자 수 증가로 데이터베이스를 확장해야 하는 경우

:::tip IT 컨설턴트의 역할
실제로 IT 컨설턴트들이 하는 주요 업무 중 하나가 바로 이 "감시 및 개선" 작업입니다. 운영 중인 시스템에 방문하여 응답 시간, 처리량, 저장 공간 사용률 등을 분석하고, 개선이 필요한 부분을 진단하여 보고서를 작성합니다. "이 부분을 이렇게 수정하면 성능이 30% 향상됩니다"와 같은 컨설팅 결과를 제시합니다.
:::

### 피드백(Feedback): 생명주기의 핵심

생명주기에서 가장 중요한 개념 중 하나가 바로 **피드백(Feedback)**입니다. 어느 단계에서든 문제가 발견되면 적절한 이전 단계로 돌아가 수정할 수 있습니다.

- 설계 중에 요구조건이 부족하다는 것을 알게 되면 → **요구조건 분석 단계로 피드백**
- 운영 중에 구현 단계에서 오류가 발견되면 → **구현 단계로 피드백**
- 운영 중에 설계 자체에 문제가 있다면 → **설계 단계로 피드백**

이처럼 생명주기는 단순한 일직선이 아니라, 지속적으로 개선하고 발전하는 **순환 구조**를 가지고 있습니다.

## 🛡️ 데이터베이스 설계의 목표와 고려사항

### 설계의 핵심 목표

데이터베이스를 설계할 때 반드시 달성해야 할 목표들이 있습니다.

**첫째, 사용자 요구조건 만족**: 데이터베이스를 사용하는 것은 결국 사람이기 때문에, 사용자들의 요구를 만족시키는 것이 가장 중요한 목표입니다. 아무리 기술적으로 훌륭한 데이터베이스라도 사용자가 원하는 기능을 제공하지 못하면 의미가 없습니다.

**둘째, 자연스럽고 이해하기 쉬운 정보 구조**: 데이터베이스는 사람이 이해하고 사용할 수 있어야 합니다. 관계형 데이터베이스가 테이블 형태로 데이터를 표현하는 것도, 사람들이 표 형태에 익숙하기 때문입니다.

**셋째, 성능과 비용의 균형(가성비)**: 모든 요구사항을 100% 만족시키는 것이 이상적이지만, 현실에서는 예산과 성능 사이의 균형을 찾아야 합니다. 동시 접속자 수, 응답 시간, 처리량 등을 고려하여 적절한 수준의 성능을 달성하는 설계를 해야 합니다.

### 무결성(Integrity): 데이터의 정확성 보장

**무결성(Integrity)**이란 데이터의 **정확성과 일관성**을 유지하는 것입니다. 데이터베이스에는 세 가지 종류의 무결성이 있습니다.

#### 개체 무결성(Entity Integrity)

**기본키(Primary Key)**에 관한 규칙입니다. 기본키는 각 행(레코드)을 고유하게 식별하는 값으로, 다음 두 가지 조건을 반드시 만족해야 합니다:

1. **유일성(Uniqueness)**: 중복된 값을 가질 수 없다
2. **NOT NULL**: NULL 값을 가질 수 없다 (반드시 값이 있어야 함)

```sql
-- ✅ 올바른 예시: 기본키 제약조건이 있어 개체 무결성 보장
CREATE TABLE 학생 (
    학번 CHAR(10) NOT NULL,     -- NOT NULL: 반드시 값이 있어야 함
    이름 VARCHAR(50) NOT NULL,
    PRIMARY KEY (학번)          -- 학번은 중복될 수 없음 (유일성 보장)
);

-- ❌ 잘못된 예시: 같은 학번을 가진 학생이 두 명 삽입되는 경우
INSERT INTO 학생 VALUES ('2024001', '홍길동');
INSERT INTO 학생 VALUES ('2024001', '김철수');  -- 오류! 학번 중복
-- 이처럼 개체 무결성이 없으면 동일한 학번을 가진 학생이 여러 명 생길 수 있습니다
```

학번, 주민등록번호, 직원번호 등이 기본키의 대표적인 예입니다. 같은 학번을 가진 두 학생이 존재할 수 없고, 학번이 없는 학생도 존재할 수 없습니다.

#### 참조 무결성(Referential Integrity)

**외래키(Foreign Key)**에 관한 규칙입니다. 외래키란 다른 테이블의 기본키를 참조하는 속성으로, 참조하는 값이 반드시 참조 대상 테이블에 존재해야 합니다.

예를 들어, 학생 테이블의 "지도교수번호"가 교수 테이블의 "교수번호"를 참조한다면:
- 교수번호가 a1, a2, a3인 교수만 존재하는데
- 학생의 지도교수번호에 a4가 들어있으면 → **참조 무결성 위반!**

```sql
-- ✅ 올바른 예시: 참조 무결성 보장
-- 교수 테이블에 존재하는 교수번호만 학생의 지도교수번호로 사용 가능
INSERT INTO 교수 VALUES ('A001', '김교수', '컴퓨터공학과');  -- 교수 추가
INSERT INTO 학생 VALUES ('2024001', '홍길동', '컴퓨터공학과', 1, 'A001');  -- 정상: A001 교수 존재

-- ❌ 잘못된 예시: 존재하지 않는 교수번호를 참조
INSERT INTO 학생 VALUES ('2024002', '김철수', '컴퓨터공학과', 1, 'Z999');
-- 오류! Z999라는 교수번호를 가진 교수가 존재하지 않음
-- 외래키 제약조건이 없으면 이런 잘못된 데이터가 삽입될 수 있습니다
```

:::note 외래키와 NULL 값
기본키는 절대 NULL 값을 가질 수 없지만, 외래키는 NULL 값을 가질 수 있습니다. 외래키의 NULL 값은 "아직 참조할 대상을 결정하지 못했다"는 의미입니다. 예를 들어 신입생의 경우 아직 지도 교수가 배정되지 않았을 수 있으므로, 지도교수번호(외래키)가 NULL일 수 있습니다.
:::

#### 도메인 무결성(Domain Integrity)

**속성 값의 유효 범위**에 관한 규칙입니다. 각 속성이 가질 수 있는 값의 범위나 형식을 제한합니다.

```sql
-- ✅ 올바른 예시: 도메인 무결성 보장
CREATE TABLE 성적 (
    학번    CHAR(10)    NOT NULL,
    과목코드 CHAR(10)   NOT NULL,
    점수    INT         CHECK (점수 >= 0 AND 점수 <= 100),  -- 점수는 0~100만 허용
    성별    CHAR(2)     CHECK (성별 IN ('남', '여')),       -- 성별은 '남' 또는 '여'만 허용
    PRIMARY KEY (학번, 과목코드)
);

-- ❌ 잘못된 예시: 유효 범위 밖의 값 삽입 시도
INSERT INTO 성적 VALUES ('2024001', 'CS101', 150, '남');  -- 오류! 점수 150은 100 초과
INSERT INTO 성적 VALUES ('2024001', 'CS101', 85, '기타'); -- 오류! '기타'는 허용되지 않는 값
```

### 일관성(Consistency): 모순 없는 데이터

**일관성(Consistency)**이란 같은 조건으로 데이터를 조회했을 때 항상 **같은 결과**가 나와야 한다는 것입니다. 내가 바꾸지 않은 내 집 주소를 아침에 조회하고 저녁에 다시 조회했을 때 다른 주소가 나온다면, 그것은 데이터의 일관성이 깨진 것입니다.

### 기타 중요한 설계 고려사항

| 고려사항 | 설명 |
|---|---|
| **회복(Recovery)** | 장애 발생 시 원래 상태로 복구하는 기능. 하드디스크 고장이나 정전 시에도 데이터를 잃지 않아야 함 |
| **보안(Security)** | 인가받지 않은 사용자의 접근을 차단. 아이디/비밀번호 기반 접근 제어 기법 및 강제 접근 제어(기밀 문서 등급 부여) 등 사용 |
| **효율성(Efficiency)** | 응답 시간은 짧게, 저장 공간은 최적화, 생산성은 높게 |
| **확장성(Scalability)** | 사용자 수나 데이터 양이 늘어나도 시스템 운영에 지장이 없도록 확장 가능한 구조로 설계 |

:::danger 확장성을 무시하면 생기는 문제
처음에 사용자 100명을 기준으로 설계한 데이터베이스가 10만 명의 사용자를 수용하려면 완전히 새로 만들어야 할 수 있습니다. 이는 엄청난 비용과 시간 낭비를 초래합니다. 설계 단계에서 **미래의 확장 가능성**을 반드시 고려해야 합니다.
:::

## 🗺️ 전체 데이터베이스 설계 프로세스 한눈에 보기

지금까지 배운 전체 과정을 정리하면 다음과 같습니다:

```
[현실 세계 (사용자 요구사항)]
         ↓
[1단계] 요구조건 분석
         - 정보 수집 (사용자, 업무, 데이터 종류, 데이터 흐름)
         - 제약 조건 식별
         - 공식적인 요구조건 명세서 작성
         - 도구: HIPO 다이어그램, DFD, SADT
         ↓
[2단계] 설계
    ┌─ 개념적 설계 ─── ER 다이어그램 작성 (DBMS 독립적)
    │                   트랜잭션 모델링
    │                   추상화 (집단화, 일반화)
    │         ↓
    ├─ 논리적 설계 ─── ER 다이어그램 → 테이블 변환 (DBMS 종속적)
    │                   정규화 수행
    │                   트랜잭션 인터페이스 설계
    │                   DBMS 선택 (Oracle, MySQL, MSSQL 등)
    │         ↓
    └─ 물리적 설계 ─── 인덱스, 저장 구조 결정 (DBMS 종속적)
                        접근 경로 선택 (순차/해싱/인덱스)
                        응답 시간, 저장 공간, 처리량 최적화
                        트랜잭션 상세 설계
         ↓
[3단계] 구현
         - DDL (CREATE, ALTER, DROP)로 데이터베이스 생성
         - 저장 공간 평가, 응답 시간 평가
         ↓
[4단계] 운영
         - 실제 사용자에게 서비스 제공
         ↓
[5단계] 감시 및 개선
         - 성능 모니터링
         - 문제 발견 시 이전 단계로 피드백하여 개선
```

> 💡 **중요한 차이점:** 개념적 설계는 어떤 DBMS를 사용하든 **독립적**으로 적용할 수 있습니다. 반면 논리적 설계와 물리적 설계는 어떤 DBMS를 사용하느냐에 따라 결과가 달라지므로 **DBMS에 종속적(의존적)**입니다.

## ⚠️ 주의사항과 초보자를 위한 팁

### 자주 하는 실수들

:::warning 요구조건 분석을 소홀히 하지 마세요
많은 초보 개발자들이 빨리 코딩을 시작하고 싶은 마음에 요구조건 분석을 대충 넘기는 경우가 있습니다. 하지만 건물 설계도 없이 건물을 짓다가 나중에 설계를 바꾸면 이미 지어진 벽을 허물고 다시 쌓아야 하는 것처럼, 데이터베이스도 나중에 구조를 변경하면 기존 데이터를 마이그레이션(이전)하는 작업이 매우 복잡하고 비용이 많이 듭니다. 요구조건 분석에 충분한 시간을 투자하세요.
:::

:::warning 잠재적 사용자를 항상 고려하세요
현재 사용자만을 위한 설계는 단기적으로는 괜찮아 보일 수 있지만, 서비스가 성장하면서 심각한 문제가 발생할 수 있습니다. 처음 설계 시 미래의 확장성을 충분히 고려하여 유연한 구조를 만드는 것이 중요합니다.
:::

:::danger 무결성 제약조건을 빼먹지 마세요
기본키, 외래키, 도메인 제약조건 없이 테이블을 만들면 잘못된 데이터가 데이터베이스에 쌓이게 됩니다. 한 번 잘못된 데이터가 쌓이면 나중에 이를 수정하는 것이 매우 어렵습니다. 처음부터 적절한 무결성 제약조건을 설정하는 것이 훨씬 현명합니다.
:::

### 생산적인 설계를 위한 팁

:::tip 설계 단계별로 문서를 남기세요
각 설계 단계에서 결정한 내용을 문서로 남겨두면, 나중에 문제가 생겼을 때 어떤 단계에서 어떤 결정이 왜 내려졌는지 추적하기 쉽습니다. 팀 프로젝트에서는 특히 모든 팀원이 같은 설계 문서를 공유하고 이해해야 합니다.
:::

:::tip 개념적 설계부터 시작하세요
개념적 설계(ER 다이어그램)를 먼저 충분히 검토하고 팀원들과 합의한 다음에 논리적, 물리적 설계로 넘어가세요. 개념적 설계에서의 실수는 논리적, 물리적 설계에서 모두 영향을 미치기 때문에, 이 단계를 철저히 하는 것이 전체 프로젝트의 성패를 좌우합니다.
:::

:::tip 성능은 처음부터 고려하세요
"일단 작동하게 만들고, 나중에 최적화하자"는 생각은 위험합니다. 물리적 설계 단계에서 인덱스와 저장 구조를 처음부터 올바르게 설계해야 나중에 성능 문제로 전체 구조를 다시 바꾸는 상황을 방지할 수 있습니다.
:::

## 📌 핵심 정리

- **데이터베이스 생명주기**는 요구조건 분석 → 설계 → 구현 → 운영 → 감시 및 개선의 5단계로 구성된다
- **요구조건 분석**은 현재 사용자뿐만 아니라 잠재적 사용자의 요구까지 고려하여 데이터베이스 범위와 사용자를 결정하는 단계다
- **요구조건의 유형**은 정적인 요구조건(구조), 동적인 요구조건(트랜잭션), 범 기관적 제약조건(예산, 정책) 세 가지로 나뉜다
- **공식적인 요구조건 명세서**는 결재를 받은 공식 문서로 이후 모든 개발의 기준이 된다
- **설계 도구**로는 HIPO 다이어그램, 데이터 플로우 다이어그램(DFD), SADT 등이 있다
- **설계**는 개념적 설계, 논리적 설계, 물리적 설계 세 단계로 나뉜다
- **개념적 설계**는 ER 다이어그램을 작성하며, 어떤 DBMS에도 독립적으로 적용 가능하다
- **추상화**는 집단화(속성 → 개체)와 일반화(여러 개체 → 일반 개체)로 나뉘며, 개념적 설계의 핵심 개념이다
- **논리적 설계**는 ER 다이어그램을 DBMS에 맞는 스키마(테이블 등)로 변환하며, 정규화와 트랜잭션 인터페이스 설계를 수행한다
- **물리적 설계**는 저장 구조, 인덱스, 접근 경로를 결정하며 응답 시간, 저장 공간 효율화, 처리량(Throughput)을 최적화한다
- **구현**은 DDL(CREATE, ALTER, DROP) 명령어로 실제 데이터베이스를 생성하는 단계다
- **무결성**은 개체 무결성(기본키: 유일성+NOT NULL), 참조 무결성(외래키: 올바른 참조), 도메인 무결성(값의 유효 범위)으로 나뉜다
- **논리적·물리적 설계**는 사용하는 DBMS에 따라 달라지므로 DBMS에 종속적(의존적)이다
- 생명주기의 어느 단계에서든 문제 발견 시 이전 단계로 **피드백**하여 개선할 수 있다
