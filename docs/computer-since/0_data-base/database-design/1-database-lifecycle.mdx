# 🗄️ 데이터베이스 생명주기(Life Cycle) 개요

데이터베이스는 그냥 어느 날 갑자기 "뚝딱" 만들어지는 것이 아닙니다. 마치 아파트를 짓기 전에 설계 도면을 그리고, 부지를 조사하고, 자재를 선정하는 것처럼, 데이터베이스도 **체계적인 단계**를 거쳐서 탄생합니다. 이 문서에서는 데이터베이스가 어떤 단계를 거쳐 만들어지고, 운영되고, 개선되는지 — 즉, 데이터베이스의 **생명주기(Life Cycle)** 전반을 아주 쉽고 자세하게 살펴봅니다.

---

## 🌱 데이터베이스 생명주기란 무엇인가?

### 생명주기(Life Cycle)의 개념

**생명주기(Life Cycle)** 란 어떤 시스템이나 제품이 처음 기획되어 사용되고, 개선·폐기될 때까지의 전 과정을 말합니다. 사람으로 비유하면 태어나서 → 성장하고 → 일하다가 → 은퇴하는 과정과 비슷합니다. 데이터베이스도 마찬가지로 "탄생 → 설계 → 구현 → 운영 → 개선"이라는 흐름을 가집니다.

데이터베이스 생명주기를 한 문장으로 정리하면 이렇습니다:

> **요구조건 분석 → 설계(개념적·논리적·물리적) → 구현 → 운영 → 감시 및 개선**

이 5단계가 순서대로 진행되기도 하지만, **각 단계에서 문제가 발생하면 이전 단계로 피드백(되돌아가 수정)** 할 수 있습니다. 예를 들어, 구현 단계에서 설계에 오류가 발견되면 다시 설계 단계로 돌아가 수정합니다. 운영 중에 새로운 사용자 요구가 생기면 요구조건 분석 단계부터 다시 시작할 수도 있습니다. 이처럼 생명주기는 단순한 직선이 아니라 **순환하는 구조**입니다.

### 왜 생명주기 개념이 필요한가?

데이터베이스를 만들 때 "일단 만들고 보자"는 식으로 접근하면 어떻게 될까요? 나중에 사용자가 "이런 기능이 필요해요"라고 요구할 때마다 전체를 다시 뜯어고쳐야 하는 상황이 생깁니다. 이는 마치 설계 도면 없이 집을 짓다가 "화장실 위치가 잘못됐네" 하고 벽을 다시 허무는 것과 같습니다. 비용도, 시간도 엄청나게 낭비됩니다.

생명주기 개념을 도입하면 **각 단계에서 해야 할 일이 명확해지고**, 문제가 발생했을 때 어느 단계로 돌아가야 하는지도 분명해집니다. 결과적으로 더 안정적이고 효율적인 데이터베이스를 만들 수 있습니다.

## 📋 1단계: 요구조건 분석

### 요구조건 분석이란?

**요구조건 분석(Requirements Analysis)** 은 데이터베이스를 만들기 전에 "우리가 무엇을 만들어야 하는가?"를 명확히 파악하는 단계입니다. 건물로 비유하면, 건축주에게 "몇 층짜리 건물을 원하시나요? 방은 몇 개 필요하세요? 주차장은요?" 하고 꼼꼼히 물어보는 과정입니다.

이 단계에서는 크게 세 가지를 확인합니다:

1. **데이터베이스의 범위(Scope)**: 어디까지 개발할 것인가? 어떤 데이터를 저장할 것인가?
2. **사용자(User)**: 누가 이 데이터베이스를 사용할 것인가?
3. **제약 조건(Constraints)**: 예산, 기간, 기관 정책 등 지켜야 할 조건은 무엇인가?

특히 **잠재적 사용자(Potential User)** 까지 고려해야 한다는 점이 중요합니다. 지금 당장 사용하는 사람만 생각하면, 미래에 사용자가 늘어나거나 새로운 업무가 추가될 때 기존 시스템이 맞지 않는 문제가 발생합니다. 마치 지금 가족이 4명이라고 해서 방 2개짜리 집을 지었는데, 나중에 가족이 6명이 되면 집이 너무 좁아지는 것과 같습니다.

### 요구조건의 세 가지 종류

요구조건은 성격에 따라 세 가지로 나눌 수 있습니다.

| 종류 | 설명 | 예시 |
|---|---|---|
| **정적인 정보 구조** | 어떤 테이블이 필요하고, 어떤 관계가 있는가 | 학생-교수-과목 테이블, 1:N 관계 |
| **동적인 요구 조건** | 어떤 작업(트랜잭션)이 얼마나 자주 발생하는가 | 수강신청(삽입), 성적조회(검색), 수강변경(갱신) |
| **범 기관적 제약 조건** | 예산, 정책, 경영 목표 등 | 개발 예산 1억 원, 업무 효율 10% 향상 목표 |

**정적인 정보 구조**는 쉽게 말해 "어떤 테이블을 만들 것이고, 그 테이블들이 서로 어떤 관계를 맺는가"를 파악하는 것입니다. 예를 들어 대학 시스템이라면, 학생 테이블·교수 테이블·과목 테이블이 필요하고, 학생과 교수는 "지도" 관계, 학생과 과목은 "수강" 관계가 있다는 것을 파악합니다.

**동적인 요구 조건**은 "이 데이터베이스에서 어떤 작업이 주로 일어나는가"를 파악합니다. 수강신청은 데이터를 **삽입(INSERT)** 하는 것이고, 성적을 조회하는 것은 **검색(SELECT)** 이며, 수강과목을 바꾸는 것은 **갱신(UPDATE)**, 수강을 취소하는 것은 **삭제(DELETE)** 입니다. 이런 작업들을 통틀어 **트랜잭션(Transaction, 작업 묶음)** 이라고 합니다. 삽입이 많은지, 검색이 많은지에 따라 설계 방향이 달라집니다.

**범 기관적 제약 조건**은 아무리 좋은 시스템을 설계해도 예산이 5억인데 회사 예산이 1억밖에 없다면 현실적으로 불가능하다는 뜻입니다. 기관의 경영 목표, 정책, 규정에 맞춰 설계 방향을 조정해야 합니다.

### 요구조건 분석의 결과물: 공식 명세서

요구조건 분석이 완료되면 반드시 **공식적인 요구조건 명세서(Requirements Specification Document)** 가 만들어져야 합니다. 이 문서는 결재(승인)를 받은 공식 문서로, 이것을 기준으로 실제 개발이 진행됩니다.

:::info 요구조건 명세서란?
실무에서 개발자가 "그냥 구두로 들었는데요..."라고 말하면 큰 문제가 생깁니다. 나중에 "그런 말 한 적 없는데요"라는 분쟁이 생기기 때문입니다. 공식적인 요구조건 명세서는 이런 분쟁을 막고, 개발의 방향을 명확히 하는 핵심 문서입니다. 실무에 입사하면 가장 먼저 접하게 되는 문서 중 하나입니다.
:::

요구조건 명세서를 작성하는 데 활용하는 도구들이 있습니다.

**① HIPO 다이어그램 (Hierarchy Input Process Output Diagram)**

HIPO는 **계층(Hierarchy) + 입력(Input) + 처리(Process) + 출력(Output)** 의 약자입니다. 시스템의 기능들을 각각의 독립된 모듈(기능 단위)로 만들고, 그 모듈들 간의 관계를 계층 구조로 표현한 도표입니다.

예를 들어 **재고 관리 시스템**이라면:
- 최상위: 재고 관리 시스템
  - 중간: 거래 처리 / 재고량 업데이트 / 재고 보고서 출력
    - 하위: 거래 처리 → 입고 처리, 출고 처리
    - 하위: 재고량 업데이트 → 실시간 업데이트, 일괄 업데이트
    - 하위: 보고서 출력 → 월간 보고서, 주간 보고서

이렇게 표현하면 시스템 전체 업무 흐름을 한눈에 파악할 수 있습니다.

**② 데이터 흐름도 (DFD, Data Flow Diagram)**

데이터가 어떤 프로세스를 거쳐 어디로 흘러가는지를 그림으로 표현한 도구입니다. 예를 들어 판매부에서 고객 주문서가 들어오면, 고객 서비스 부서에서 신용도 검토를 하고, 주문이 처리되는 흐름을 시각적으로 보여줍니다.

**③ SADT (Structured Analysis and Design Technique)**

SADT는 SofTech사에서 개발한 방법론으로, 대규모의 복잡한 시스템 구조를 계층적으로 분석할 때 사용합니다. 입력 데이터(Input), 제어 데이터(Control), 처리 메커니즘(Mechanism), 출력 결과(Output)를 구조화된 도형으로 표현합니다.

:::note 참고
HIPO, DFD, SADT는 **시스템 분석 및 설계** 또는 **소프트웨어 공학** 과목에서 더 자세히 다룹니다. 데이터베이스 과목에서는 이 도구들을 통해 만들어진 요구조건 명세서를 가지고 **실제 DB를 설계하고 구현하는 것**이 핵심 목표이므로, 여기서는 개념만 이해하면 충분합니다.
:::

## 🎨 2단계: 설계 — 개념적·논리적·물리적 설계

### 설계 단계의 전체 구조

요구조건 분석이 끝나면 이제 본격적으로 **설계** 단계에 들어갑니다. 건축으로 치면 설계 도면을 그리는 단계입니다. 데이터베이스 설계는 크게 세 단계로 나뉩니다:

```
개념적 설계 → 논리적 설계 → 물리적 설계
(사용자 관점)  (DB 조직 관점)  (저장장치 관점)
```

> **개념적 설계**는 "무엇을 저장할 것인가?", **논리적 설계**는 "어떤 구조로 저장할 것인가?", **물리적 설계**는 "실제 하드웨어에 어떻게 저장할 것인가?"를 다룹니다.

설계 작업은 크게 두 가지 관점에서 이루어집니다:

| 관점 | 설명 | 건축 비유 |
|---|---|---|
| **데이터 중심 설계** | 어떤 데이터를 저장할 것인가, 데이터의 구조가 어떠한가 | 방 배치, 층수, 구조 결정 |
| **처리 중심 설계** | 데이터에 어떤 작업을 할 것인가 (검색, 삽입, 삭제 등) | 사람의 동선, 생활 방식 고려 |

이 두 가지를 모두 고려해야 합니다. 구조(데이터 중심)만 고려하고 트랜잭션(처리 중심)을 무시하면, 실제 운영 시 성능 문제가 생깁니다. 반대로 트랜잭션만 생각하고 데이터 구조를 소홀히 해도 마찬가지입니다.

### 개념적 설계 (Conceptual Design)

**개념적 설계**는 **사용자 관점**에서 "어떤 데이터가 필요하고, 데이터들 사이에 어떤 관계가 있는가"를 표현하는 단계입니다. 이 단계의 핵심 결과물은 **ER 다이어그램(Entity-Relationship Diagram, 개체-관계 다이어그램)** 입니다.

:::tip 개념적 설계의 핵심 특징
개념적 설계는 **어떤 DBMS를 사용하든 상관없이 독립적으로 만들 수 있습니다**. Oracle을 쓰든, MySQL을 쓰든, MSSQL을 쓰든, 계층형 DB를 쓰든, 개념적 설계 단계에서 나온 ER 다이어그램은 동일합니다.
:::

개념적 설계에서 가장 중요한 개념은 **추상화(Abstraction)** 입니다. 추상화란 복잡한 현실 세계의 정보를 단순하고 공통적인 개념으로 표현하는 것입니다.

예를 들어, 실제 학생 한 명에게는 학번, 이름, 연락처, 학과, 나이, 취미, 출신 고등학교 등 수많은 정보가 있습니다. 이 모든 것을 다 표현하는 대신, 데이터베이스에 필요한 **공통 속성만 뽑아서 "학생"이라는 하나의 개체(Entity)로 표현**합니다. 이것이 추상화입니다.

추상화에는 두 가지 방식이 있습니다:

**① 집단화 (Aggregation)**
여러 **속성(Attribute, 특성값)** 들을 묶어서 하나의 **개체(Entity)** 를 만드는 것입니다.

예시: `학번`, `이름`, `학과` → 묶어서 **학생** 개체 생성

**② 일반화 (Generalization)**
여러 **개체들**의 공통적인 특성을 모아서 더 일반적인 상위 개체를 만드는 것입니다.

예시: `학부생`과 `대학원생` → 공통점 추출 → **학생** 개체 생성

두 방식 모두 공통적인 것을 뽑아내는 작업이므로, 넓은 의미에서 모두 **추상화**에 포함됩니다.

#### ER 다이어그램 예시

```
[학생]                     [교수]
- 학번 (기본키, PK)  지도   - 교수번호 (기본키, PK)
- 이름              ────   - 이름
- 학과                     - 소속 학과
```

위 그림에서 `학생`과 `교수`는 **개체(Entity)**, `지도`는 **관계(Relationship)**, `학번`·`이름` 등은 **속성(Attribute)** 입니다. 한 명의 교수는 여러 명의 학생을 지도할 수 있으므로 **1:N 관계**가 됩니다.

개념적 설계에서는 또한 **트랜잭션 모델링** 을 수행합니다. 이 데이터베이스에서 어떤 작업(검색·삽입·갱신·삭제)이 이루어질지를 고차원적(추상적)으로 정의합니다. 나중에 논리적·물리적 설계 단계에서 이 정보를 더 구체화합니다.

### 논리적 설계 (Logical Design)

**논리적 설계**는 개념적 설계에서 만든 ER 다이어그램을 **실제 DBMS에서 사용할 수 있는 구조로 변환**하는 단계입니다. 관계형 데이터베이스(RDBMS)를 사용한다면 ER 다이어그램을 **테이블(Table)** 형태로 변환합니다.

:::warning 논리적 설계는 DBMS에 종속됩니다
개념적 설계와 달리, 논리적 설계는 **어떤 DBMS를 사용하느냐에 따라 달라집니다**.
- 관계형 DB (Oracle, MySQL, MSSQL) → **테이블** 형태
- 계층형 DB → **트리(Tree)** 형태
- 네트워크형 DB → **그래프(Graph)** 형태
:::

논리적 설계에서 수행하는 핵심 작업은 두 가지입니다:

**① 정규화 (Normalization)**
데이터베이스의 중복을 제거하고 데이터 구조를 최적화하는 작업입니다. 정규화를 통해 불필요한 데이터 중복, 삽입·삭제·갱신 시 발생하는 오류(이상 현상)를 방지합니다. 정규화에 대한 자세한 내용은 이후 챕터에서 다룹니다.

**② 트랜잭션 인터페이스 설계 (Transaction Interface Design)**
개념적 설계에서 모델링한 트랜잭션들을 실제로 어떻게 연결하고 처리할지 설계합니다.

논리적 설계 단계에서는 다음과 같은 성능 평가 지표도 함께 고려합니다:

| 평가 항목 | 설명 |
|---|---|
| 데이터의 양 | 저장될 데이터가 얼마나 많은가 |
| 처리 빈도수 | 트랜잭션이 얼마나 자주 발생하는가 |
| 작업량 | 한 번에 처리해야 하는 작업의 규모 |
| 레코드 접근 및 전송량 | 데이터를 얼마나 자주 읽고 쓰는가 |
| DB의 크기 | 전체 데이터베이스의 용량 |

또한 이 단계에서 **어떤 DBMS를 선택할 것인가**도 결정합니다. DBMS 선택 기준은 다음과 같습니다:

- **기능**: 원하는 기능을 지원하는가?
- **초기 비용**: 라이선스, 서버 하드웨어 구입 비용
- **마이그레이션(이전) 비용**: 기존 DBMS에서 새 DBMS로 데이터를 옮기는 비용
- **교육비**: 직원들이 새 DBMS를 배우는 데 드는 비용
- **유지보수 및 운영 비용**: 버전 업그레이드, 지속적인 운영 비용

### 물리적 설계 (Physical Design)

**물리적 설계**는 논리적 설계 결과물인 테이블 구조를 **실제 저장 장치(하드디스크 등)에 어떻게 저장할 것인가**를 결정하는 단계입니다. 건축으로 비유하면, 설계 도면이 완성된 후 "이 기둥은 어떤 재료로, 어떻게 배치할 것인가"를 결정하는 것과 같습니다.

물리적 설계에서 다루는 핵심 내용:

- **저장 레코드 양식**: 데이터를 어떤 형태로 저장 장치에 기록할 것인가
- **레코드 순서**: 데이터를 어떤 순서로 정렬해서 저장할 것인가
- **접근 경로(Access Path)**: 데이터에 어떻게 접근할 것인가 (인덱스 사용 여부, 순차 구조/해싱 구조 선택 등)
- **저장 공간 할당**: 데이터베이스에 얼마만큼의 디스크 공간을 배정할 것인가

물리적 설계에서 특히 **인덱스(Index)** 를 추가하는 것이 대표적인 작업입니다. 인덱스는 마치 책의 목차처럼, 데이터를 빠르게 찾을 수 있도록 도와주는 구조입니다.

이 단계에서 **트랜잭션의 상세 설계** 도 완료됩니다. 개념적 설계에서 "모델링"하고, 논리적 설계에서 "인터페이스를 설계"했다면, 물리적 설계에서는 가장 구체적인 수준으로 트랜잭션을 정의합니다.

물리적 설계 시 고려하는 성능 지표:

**① 응답 시간 (Response Time)**
사용자가 요청을 보낸 시점부터 결과를 받을 때까지 걸리는 시간입니다. 예를 들어, 네이버 검색창에 "오늘 날씨"를 입력하고 엔터를 누른 순간부터 결과 화면이 뜰 때까지의 시간입니다. 응답 시간은 **짧을수록 좋습니다**.

**② 저장 공간의 효율화**
저장 공간을 너무 낭비해서도, 너무 빽빽하게 채워서도 안 됩니다. **블로킹 팩터(Blocking Factor)** — 블록 크기를 얼마로 설정하느냐에 따라 I/O(입출력) 횟수가 달라지고, 이것이 성능에 큰 영향을 미칩니다.

**③ 트랜잭션 처리량 (Throughput)**
**Throughput(처리도)** 란 단위 시간당 처리할 수 있는 트랜잭션의 수를 말합니다. 예를 들어 "Throughput = 1,000"이라면 1초에 1,000건의 작업을 처리할 수 있다는 뜻입니다. 이 값이 높을수록 더 많은 사용자를 동시에 처리할 수 있는 고성능 시스템입니다.

## 🔨 3단계: 구현 (Implementation)

### 구현이란?

**구현(Implementation)** 단계는 설계가 완료된 후 실제로 **SQL 명령어를 사용해서 데이터베이스를 생성하는 과정**입니다. 건축으로 비유하면 설계 도면을 보고 실제로 건물을 짓는 단계입니다.

이 단계에서는 주로 **DDL(Data Definition Language, 데이터 정의어)** 명령어를 사용합니다:

```sql
-- ✅ 데이터베이스 생성
CREATE DATABASE university_db;  -- 대학교 데이터베이스 생성

-- ✅ 테이블 생성 예시: 학생 테이블
CREATE TABLE student (
    student_id   VARCHAR(10) PRIMARY KEY,  -- 학번 (기본키, 유일한 식별자)
    name         VARCHAR(50) NOT NULL,      -- 이름 (NULL 허용 안 함)
    department   VARCHAR(50),              -- 학과
    age          INT                        -- 나이
);

-- ✅ 테이블 생성 예시: 교수 테이블
CREATE TABLE professor (
    professor_id VARCHAR(10) PRIMARY KEY,  -- 교수번호 (기본키)
    name         VARCHAR(50) NOT NULL,      -- 교수 이름
    department   VARCHAR(50)               -- 소속 학과
);

-- ✅ 테이블 수정 예시: 학생 테이블에 이메일 컬럼 추가
ALTER TABLE student ADD email VARCHAR(100);

-- ✅ 테이블 삭제 예시 (주의: 데이터도 함께 삭제됨)
DROP TABLE student;
```

:::danger DROP은 신중하게
`DROP TABLE`을 실행하면 테이블과 그 안의 **모든 데이터가 즉시 삭제**됩니다. 실무에서는 반드시 백업 후에 실행하세요. 실수로 삭제한 데이터는 복구가 매우 어렵습니다.
:::

구현이 완료되면 **비어 있는 데이터베이스 구조(스키마, Schema)** 가 만들어집니다. 이 시점에서는 테이블의 틀만 만들어진 것이고, 실제 데이터는 아직 아무것도 없는 상태입니다.

### 구현 단계에서 평가하는 항목

구현이 완료되면 아래 항목들을 실제로 측정하고 평가합니다:

- **저장 공간**: 주기억 장치(RAM)와 보조기억 장치(HDD/SSD)를 얼마나 사용하는가
- **질의 응답 시간**: SQL 질의를 시작해서 결과가 화면에 처음 나타날 때까지의 시간
- **트랜잭션 갱신 시간**: 데이터를 갱신(UPDATE)하는 데 걸리는 시간
- **CPU 및 I/O 사용량**: 처리 과정에서 CPU와 디스크 입출력이 얼마나 발생하는가
- **보고서 작성 시간**: 결과 데이터를 보고서 형태로 출력하는 데 걸리는 시간

:::info IT 컨설턴트의 역할
실제 현업에서 IT 컨설턴트들이 하는 일 중 하나가 바로 이 구현 단계의 평가입니다. 현재 운영 중인 시스템을 방문해서 응답 시간, I/O 횟수, 처리 성능 등을 분석하고, "이 부분을 개선하면 성능이 20% 향상됩니다"와 같은 보고서를 만들어냅니다.
:::

## ⚙️ 4~5단계: 운영과 감시 및 개선

### 4단계: 운영 (Operation)

**운영(Operation)** 단계는 완성된 데이터베이스를 **실제 사용자들이 서비스에 활용하는 단계**입니다. 건물이 완공되어 사람들이 입주해서 생활하는 것과 같습니다. 실제 업무 처리, 데이터 조회, 보고서 생성 등 모든 서비스가 이 단계에서 이루어집니다.

### 5단계: 감시 및 개선 (Monitoring & Refinement)

**감시 및 개선(Monitoring & Refinement)** 단계는 데이터베이스를 운영하면서 발생하는 문제를 **지속적으로 모니터링하고 개선**하는 과정입니다. 어떤 시스템이든 처음 만든 상태 그대로 영원히 완벽할 수는 없습니다. 다음과 같은 상황이 발생하면 개선이 필요합니다:

- **성능 저하**: 사용자가 늘면서 응답 시간이 길어질 때
- **보안 취약점 발견**: 해킹 시도나 데이터 유출 위험이 감지될 때
- **권한 문제**: 잘못된 접근 권한 설정이 발견될 때
- **새로운 요구사항 발생**: 새로운 업무 프로세스나 기능이 필요할 때
- **확장 필요**: 데이터 규모나 사용자 수가 크게 증가할 때

이 단계에서 중요한 것은 **피드백 구조**입니다. 감시 중에 발견된 문제의 성격에 따라 적절한 이전 단계로 돌아갑니다:

```
운영 중 성능 문제 발견 → 물리적 설계 단계로 피드백
운영 중 구조적 문제 발견 → 논리적 설계 단계로 피드백
운영 중 새로운 요구사항 → 요구조건 분석 단계로 피드백
```

이처럼 데이터베이스 생명주기는 **선형(한 방향)이 아닌 순환 구조**를 가집니다.

## 🎯 데이터베이스 설계의 목표

### 왜 설계 목표를 명확히 해야 하는가?

설계를 시작하기 전에 "무엇을 위한 설계인가?"를 명확히 해야 방향을 잃지 않습니다. 데이터베이스 설계의 목표는 다음과 같습니다.

**① 사용자 요구 조건 만족**
가장 중요한 목표입니다. 데이터베이스를 사용하는 것은 결국 사람이기 때문입니다. "사용자가 원하는 것을 정확하게 제공할 수 있는가?"가 설계 성공의 가장 중요한 기준입니다.

**② 자연스럽고 이해하기 쉬운 정보 구조**
관계형 데이터베이스의 테이블 구조처럼, 누가 봐도 직관적으로 이해할 수 있는 구조여야 합니다. 복잡하고 난해한 구조는 유지보수를 어렵게 만들고, 개발자 교체 시 인수인계도 힘들어집니다.

**③ 성능 고려 (가성비)**
무조건 고성능 시스템이 좋은 것이 아닙니다. **투입되는 자원(비용, 시간) 대비 얼마만큼의 성능**이 나오는지를 고려해야 합니다. 동시 접속 사용자가 1만 명인지, 10만 명인지에 따라 응답 시간, 처리 시간, 저장 공간이 모두 달라지므로, 예상 사용 규모에 맞는 적절한 설계를 해야 합니다.

## 🔐 데이터베이스 설계 시 고려해야 할 핵심 속성

### 무결성 (Integrity)

**무결성(Integrity)** 이란 데이터의 **정확성과 일관성을 보장**하는 것입니다. 삽입·삭제·갱신 작업이 일어날 때 항상 제약 조건을 만족하도록 유지되어야 합니다.

무결성은 세 가지로 나뉩니다:

**① 개체 무결성 (Entity Integrity)**
**기본키(Primary Key)** 는 반드시 유일한 값을 가져야 하며, NULL(값 없음)이 될 수 없습니다.

> 예시: 학생의 학번은 절대 중복될 수 없고, 반드시 값이 있어야 합니다. 주민번호 없는 대한민국 국민이 존재할 수 없는 것과 같습니다.

**② 참조 무결성 (Referential Integrity)**
**외래키(Foreign Key)** 가 참조하는 값은 반드시 실제로 존재하는 값이어야 합니다.

> 예시: 학생의 지도교수 번호가 A4인데, 실제 교수 테이블에는 A1, A2, A3만 있다면 잘못된 참조입니다. 존재하지 않는 교수를 참조하게 됩니다.

```sql
-- ❌ 잘못된 예시: 존재하지 않는 교수번호 참조
INSERT INTO student VALUES ('2024001', '김철수', '컴퓨터공학', 20, 'A4');
-- A4 교수는 교수 테이블에 없으므로 참조무결성 위반!

-- ✅ 올바른 예시: 실제 존재하는 교수번호 참조
INSERT INTO student VALUES ('2024001', '김철수', '컴퓨터공학', 20, 'A1');
-- A1 교수는 교수 테이블에 존재하므로 정상 삽입
```

:::note 외래키와 NULL
기본키는 NULL을 가질 수 없지만, 외래키는 NULL을 가질 수 **있습니다**. 외래키가 NULL이라는 것은 "아직 참조할 대상을 결정하지 못했다"는 의미입니다. 예를 들어, 지도교수가 아직 배정되지 않은 신입생의 경우 지도교수 외래키가 NULL일 수 있습니다.
:::

**③ 도메인 무결성 (Domain Integrity)**
각 속성(컬럼)에 입력될 수 있는 값의 범위나 형식을 제한합니다.

> 예시: 시험 점수는 0~100 사이의 값만 입력 가능, 성별은 '남' 또는 '여'만 입력 가능

```sql
-- ✅ 도메인 무결성을 적용한 테이블 생성 예시
CREATE TABLE exam_result (
    student_id VARCHAR(10),           -- 학번
    subject    VARCHAR(50),           -- 과목명
    score      INT CHECK (score >= 0 AND score <= 100),  -- 점수: 0~100만 허용
    gender     CHAR(1) CHECK (gender IN ('M', 'F'))      -- 성별: M 또는 F만 허용
);
```

### 일관성 (Consistency)

**일관성(Consistency)** 이란 동일한 조건으로 질의(쿼리)를 했을 때 **항상 같은 결과**가 나와야 한다는 것입니다.

> 예시: 내 집 주소를 오전에 조회하든 오후에 조회하든 바꾼 적이 없다면 항상 같은 주소가 나와야 합니다. 조회할 때마다 다른 결과가 나온다면 일관성이 깨진 것입니다.

### 회복 (Recovery)

**회복(Recovery)** 이란 시스템 장애(정전, 하드웨어 오류, 소프트웨어 버그 등)가 발생했을 때 **데이터베이스를 장애 이전 상태로 복구**할 수 있어야 한다는 것입니다. 백업과 트랜잭션 로그를 통해 이를 구현합니다.

### 보안 (Security)

**보안(Security)** 이란 권한이 없는 사용자의 접근으로부터 데이터를 안전하게 보호하는 것입니다.

- **접근 제어 기법**: 아이디와 비밀번호로 데이터베이스 접근을 제한합니다.
- **강제 접근 제어**: 군사 기밀이나 국가 기밀처럼 문서에 등급(극비, 대외비, 비밀, 일반)을 부여하여 해당 등급을 가진 사람만 접근할 수 있도록 합니다.

### 효율성 (Efficiency)

응답 시간은 **짧을수록**, 저장 공간은 **최적화할수록**, 생산성은 **높을수록** 좋습니다. 세 가지 모두를 균형 있게 고려해야 합니다.

### 확장성 (Scalability)

**확장성(Scalability)** 이란 기존 시스템 운영에 영향을 주지 않으면서 데이터베이스 규모를 키울 수 있어야 한다는 것입니다.

> 예시: 처음에 사용자가 100명이었는데 서비스가 성공하여 10만 명이 되었다고 가정합니다. 이때 기존 시스템을 완전히 갈아엎지 않고도 서버를 늘리거나 구조를 확장할 수 있어야 합니다. 확장성을 고려하지 않은 설계는 성장의 발목을 잡는 족쇄가 됩니다.

## ⚠️ 주의사항 및 실무 팁

### 설계 단계별 산출물을 명확히 하라

각 설계 단계가 끝날 때마다 **명확한 결과물(산출물)** 을 남겨야 합니다. 이것이 없으면 다음 단계로 넘어갈 수 없고, 문제 발생 시 어느 단계에서 실수했는지 추적하기 어렵습니다.

| 단계 | 산출물 |
|---|---|
| 요구조건 분석 | 공식 요구조건 명세서, HIPO 다이어그램, DFD |
| 개념적 설계 | ER 다이어그램 |
| 논리적 설계 | 정규화된 테이블 구조, DBMS 선정 |
| 물리적 설계 | 인덱스 설계, 저장 구조 명세 |
| 구현 | 실제 생성된 데이터베이스 스키마 |

### 피드백을 두려워하지 마라

많은 초보자들이 "이미 다음 단계로 넘어왔는데 이전 단계로 돌아가는 것은 실패"라고 생각합니다. 하지만 생명주기에서 **피드백은 자연스럽고 당연한 과정**입니다. 빠른 피드백이 오히려 나중에 발생할 더 큰 문제를 예방합니다.

:::tip 피드백은 실패가 아닙니다
생명주기의 각 단계에서 문제를 발견하고 이전 단계로 돌아가 수정하는 것은 시스템의 품질을 높이는 정상적인 과정입니다. 늦게 발견할수록 수정 비용이 기하급수적으로 커지므로, 초기에 발견하고 빠르게 피드백하는 것이 훨씬 경제적입니다.
:::

### 현재 사용자만 생각하지 마라

요구조건 분석 단계에서 **잠재적 사용자**를 반드시 고려해야 합니다. 현재 100명의 사용자만을 위해 설계하면, 나중에 10,000명이 사용할 때 시스템이 다운되거나 재설계가 필요해집니다.

:::warning 잠재적 사용자를 무시한 설계의 위험성
국내 한 스타트업의 사례처럼, 처음에 소규모로 설계된 데이터베이스가 서비스가 갑자기 성장하면서 동시 접속자를 감당하지 못해 서비스가 마비되는 일이 종종 발생합니다. 설계 초기에 확장성을 고려하는 것이 훨씬 효율적입니다.
:::

### 개념적 설계는 DBMS에 독립적임을 기억하라

개념적 설계(ER 다이어그램)는 Oracle을 쓰든 MySQL을 쓰든 동일하게 적용됩니다. 반면 논리적 설계와 물리적 설계는 선택한 DBMS에 종속됩니다. 이 차이를 명확히 이해해야 각 단계에서 올바른 작업을 수행할 수 있습니다.

## 📌 핵심 정리

- **데이터베이스 생명주기**는 요구조건 분석 → 설계 → 구현 → 운영 → 감시 및 개선의 **5단계**로 이루어진다.
- 각 단계는 선형이 아닌 **순환 구조**로, 문제 발생 시 이전 단계로 피드백하여 개선할 수 있다.
- **요구조건 분석**은 DB 범위, 사용자(잠재적 사용자 포함), 제약 조건을 파악하며, 결과물로 **공식 요구조건 명세서**가 나온다.
- 요구조건 명세서 작성에는 **HIPO 다이어그램**, **DFD(데이터 흐름도)**, **SADT** 등의 도구를 활용한다.
- **설계 단계**는 개념적 설계 → 논리적 설계 → 물리적 설계의 세 단계로 세분화된다.
- **개념적 설계**는 ER 다이어그램을 작성하며, 어떤 DBMS를 사용하든 **독립적**으로 만들 수 있다.
- **논리적 설계**는 ER 다이어그램을 DBMS에 맞는 구조(테이블 등)로 변환하며, **정규화**를 수행한다.
- **물리적 설계**는 실제 저장 장치 수준의 설계로, 인덱스 추가, 저장 구조 선택, 응답 시간·Throughput 등을 최적화한다.
- **구현 단계**에서는 `CREATE`, `ALTER`, `DROP` 등의 **DDL 명령어**로 실제 DB를 생성한다.
- 데이터베이스 설계 시 **무결성(개체·참조·도메인), 일관성, 회복, 보안, 효율성, 확장성**을 반드시 고려해야 한다.
- **개체 무결성**: 기본키는 유일하며 NULL이 될 수 없다.
- **참조 무결성**: 외래키는 실제 존재하는 값만 참조해야 하며, NULL은 가능하다.
- **도메인 무결성**: 속성값은 정해진 범위나 형식 내의 값만 입력되어야 한다.
- **Throughput(처리도)** 는 단위 시간당 처리 가능한 트랜잭션 수로, 시스템 성능의 기준 지표이다.
- 논리적·물리적 설계는 선택한 **DBMS에 종속(의존적)** 이다.
