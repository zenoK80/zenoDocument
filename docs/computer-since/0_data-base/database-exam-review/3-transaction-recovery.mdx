# 🗄️ 트랜잭션 회복 기법: 체크포인트, ARIES, 트랜잭션 상태

데이터베이스는 우리 일상 곳곳에 숨어 있습니다. 온라인 쇼핑몰에서 결제할 때, 은행 앱에서 이체할 때, 병원에서 진료 기록을 조회할 때—모두 데이터베이스가 작동하고 있습니다. 그런데 만약 결제 도중 서버가 갑자기 꺼진다면 어떻게 될까요? 돈은 빠져나갔는데 주문은 등록이 안 되는 상황이 생길 수 있습니다. 이런 끔찍한 상황을 막아주는 것이 바로 **트랜잭션 회복 기법**입니다.

이 문서는 데이터베이스 시험 문제를 바탕으로, 버퍼링, 체크포인트, ARIES, 트랜잭션 상태, 로킹, 정규화, 보안까지 데이터베이스의 핵심 개념들을 처음 공부하는 분도 이해할 수 있도록 아주 친절하게 설명합니다. 차근차근 따라오세요! 😊

## 🚀 1. 버퍼링(Buffering): 성능을 높이는 메모리 활용 기법

### 버퍼링이란 무엇인가요?

**버퍼링(Buffering)**이란 데이터를 매번 즉시 저장 장치(하드디스크)에 기록하는 대신, **메인 메모리(RAM)의 임시 공간(버퍼)에 모아두었다가 한꺼번에 저장**하는 기법입니다.

왜 이게 필요할까요? 하드디스크에 데이터를 쓰는 것(I/O 작업)은 메모리에 쓰는 것보다 수백~수천 배 느립니다. 마치 물건을 사러 마트에 갈 때, 필요한 것이 생길 때마다 한 번씩 다녀오는 것과 필요한 것을 목록에 모아두었다가 한 번만 다녀오는 것의 차이와 같습니다. 당연히 한 번에 모아서 다녀오는 것이 훨씬 효율적이죠.

> 💡 **버퍼링 = 택배 여러 건을 모아서 한 번에 배달하는 것과 같은 원리!**
> 개별 배달(매번 I/O)보다 묶음 배달(버퍼링)이 훨씬 효율적입니다.

### 버퍼링의 동작 원리

데이터베이스에서 `write` 연산(쓰기 작업)이 발생할 때마다 저장 장치에 직접 저장하면 **I/O(입출력) 횟수**가 폭발적으로 증가합니다. I/O는 CPU 연산에 비해 극도로 느리기 때문에, 이것이 데이터베이스 성능 저하의 주범이 됩니다.

버퍼링은 이 문제를 다음 순서로 해결합니다:

1. `write` 연산 요청이 들어온다
2. 데이터를 즉시 디스크에 쓰지 않고, **메인 메모리의 버퍼 블록**에 임시 저장한다
3. 버퍼가 가득 차거나 특정 조건(체크포인트 등)이 되면, **버퍼의 내용을 한꺼번에 디스크에 플러시(flush)** 한다

이렇게 하면 디스크 I/O 횟수를 획기적으로 줄일 수 있습니다.

:::info 버퍼링과 회복의 관계
버퍼링 덕분에 성능이 좋아지지만, 한 가지 문제가 생깁니다. 메모리(RAM)는 전원이 꺼지면 내용이 사라집니다! 버퍼에 임시 저장된 데이터가 디스크에 기록되기 전에 시스템이 다운되면 데이터가 손실될 수 있습니다. 이 문제를 해결하기 위해 **회복(Recovery) 기법**이 필요합니다.
:::

## 📊 2. 연관 규칙: 지지도와 신뢰도 계산

### 연관 규칙이란?

**연관 규칙(Association Rule)**은 데이터 마이닝(대량의 데이터에서 유용한 패턴을 찾는 기술)에서 사용하는 개념으로, "A를 구매한 사람은 B도 구매한다"와 같은 패턴을 수치로 표현합니다. 마트의 "우유를 사는 사람은 주스도 함께 산다"는 분석이 대표적인 예입니다.

연관 규칙에서 가장 중요한 두 가지 지표는 **지지도**와 **신뢰도**입니다.

### 지지도(Support)와 신뢰도(Confidence) 계산법

| 지표 | 공식 | 의미 |
|------|------|------|
| **지지도(Support)** | 동시 구매 건수 ÷ 전체 트랜잭션 수 | 두 상품이 함께 구매되는 비율 |
| **신뢰도(Confidence)** | 동시 구매 건수 ÷ A 구매 건수 | A를 산 사람 중 B도 산 비율 |

**예시 문제**: 총 1,000개 트랜잭션, 우유(Milk) 구매 700개, 주스(Juice) 구매 500개, 동시 구매 350개일 때:

```
지지도 = 350 / 1,000 = 0.35 = 35%
신뢰도 = 350 / 700  = 0.50 = 50%
```

- **지지도 35%**: 전체 거래 중 35%에서 우유와 주스가 함께 팔렸다는 뜻
- **신뢰도 50%**: 우유를 산 사람 중 절반이 주스도 함께 샀다는 뜻

:::tip 외우는 팁
- **지지도** = "전체 중 같이 산 비율" → 분모가 **전체 트랜잭션**
- **신뢰도** = "A 산 사람 중 B도 산 비율" → 분모가 **A 구매 건수**
:::

## 🔐 3. 동시성 제어: 로킹(Locking)의 모든 것

### 로킹이 왜 필요한가?

데이터베이스는 여러 사용자가 **동시에** 접근합니다. 예를 들어, 두 사람이 동시에 같은 좌석을 예매하려 한다면 어떻게 될까요? 둘 다 "빈 좌석"이라고 읽은 후 동시에 예약을 완료하면, **한 좌석에 두 명이 예약**되는 최악의 상황이 벌어집니다. 이런 문제를 막기 위해 **로킹(Locking, 잠금)**이 필요합니다.

로킹은 마치 화장실 문에 "사용 중" 표시를 거는 것과 같습니다. 누군가 화장실을 사용하는 동안(잠금) 다른 사람은 기다려야 합니다(대기). 사용이 끝나면(잠금 해제) 다음 사람이 들어갈 수 있습니다.

### 로킹 단위와 트레이드오프

**로킹 단위(Granularity)**란 잠금을 거는 데이터의 크기를 말합니다. 데이터베이스 전체를 잠글 수도 있고, 테이블 하나, 레코드 하나, 심지어 필드 하나만 잠글 수도 있습니다.

| 로킹 단위 | 동시 수행 | 오버헤드 | 충돌 가능성 | 일관성 |
|-----------|-----------|----------|-------------|--------|
| **크면** (예: DB 전체) | ↓ 낮음 | ↓ 낮음 | ↑ 높음 | ↑ 높음 |
| **작으면** (예: 레코드) | ↑ 높음 | ↑ 높음 | ↓ 낮음 | ↓ 낮음 |

- **로킹 단위가 크면**: 한 번에 큰 덩어리를 잠그므로 관리(오버헤드)는 쉽지만, 다른 사용자들이 기다려야 하는 시간이 길어집니다(동시 수행 낮음). 대신 전체 데이터가 일관되게 유지됩니다(일관성 높음).
- **로킹 단위가 작으면**: 세밀하게 잠그므로 여러 사용자가 동시에 다른 부분에 접근할 수 있지만(동시 수행 높음), 잠금을 관리하는 비용이 커집니다(오버헤드 높음).

### 공유 락(S락)과 배타 락(X락)

로킹에는 두 가지 종류가 있습니다:

- **공유 락(Shared Lock, S락)**: 읽기 전용 잠금입니다. 여러 트랜잭션이 동시에 S락을 보유할 수 있습니다. 마치 도서관에서 같은 책을 여러 사람이 동시에 읽는 것과 같습니다.
- **배타 락(Exclusive Lock, X락)**: 읽기/쓰기 잠금입니다. 한 트랜잭션만 보유할 수 있으며, 다른 락과 절대 공존할 수 없습니다. 마치 책에 형광펜으로 밑줄을 그으려면 혼자 봐야 하는 것과 같습니다.

| | **S락 보유 중** | **X락 보유 중** |
|---|---|---|
| **S락 요청** | ✅ 양립 가능 | ❌ 대기 |
| **X락 요청** | ❌ 대기 | ❌ 대기 |

> 🔑 **핵심**: S락끼리는 함께 사용할 수 있지만, X락이 끼어들면 무조건 기다려야 합니다!

### 2단계 로킹 규약(2PL: Two-Phase Locking)

**2단계 로킹 규약(2-Phase Locking Protocol)**은 트랜잭션의 직렬 가능성(Serializability, 동시 실행 결과가 순차 실행과 같음을 보장)을 보장하는 규칙입니다.

2단계 로킹은 이름 그대로 두 단계로 나뉩니다:

1. **확장 단계(Growing Phase)**: lock만 걸 수 있고, unlock은 불가능합니다. 트랜잭션이 필요한 자원을 하나씩 잠그는 단계입니다.
2. **수축 단계(Shrinking Phase)**: unlock만 할 수 있고, 새로운 lock은 불가능합니다. 한번 unlock이 시작되면 절대 다시 lock을 걸 수 없습니다.

```
[확장 단계]          [수축 단계]
lock A  →  lock B  →  unlock A  →  unlock B
         ↑
     이 시점부터 새로운 lock 불가!
```

:::warning 2PL 핵심 규칙
**"unlock을 한 번이라도 했으면, 그 이후로는 절대 다시 lock을 걸 수 없다!"**
이를 위반하면 직렬 가능성이 깨질 수 있습니다.
:::

❌ **잘못된 예시**: unlock 이후 다시 lock을 거는 경우
```
lock A → unlock A → lock B → ...  (2PL 위반!)
```

✅ **올바른 예시**: 모든 lock을 먼저 걸고, 그 다음에 unlock을 시작
```
lock A → lock B → unlock A → unlock B  (2PL 준수)
```

### 교착상태(Deadlock)와 해결 방법

**교착상태(Deadlock)**란 두 트랜잭션이 서로 상대방이 보유한 자원을 기다리며 영원히 멈춰버리는 상태입니다. 마치 두 차가 좁은 골목에서 서로 마주보고 "네가 먼저 빠져라"라고 우기는 상황과 같습니다.

교착상태 해결 방법은 세 가지입니다:

- **예방(Prevention)**: 교착상태가 아예 발생하지 않도록 조건을 없앱니다
- **탐지(Detection)**: 교착상태가 발생했는지 주기적으로 감지합니다
- **회복(Recovery)**: 교착상태가 탐지되면 트랜잭션 중 하나를 취소하여 풀어줍니다

:::note 비선점(Non-preemption)은 독립적인 해결책이 아닙니다
"비선점"은 교착상태 **예방 전략의 하위 개념**으로, 독립적인 해결책이 아닙니다. 시험에서 교착상태 해결책을 고르라고 하면 **예방, 탐지, 회복** 세 가지만 기억하세요!
:::

## ✅ 4. 트랜잭션 회복 기법의 핵심: 체크포인트와 ARIES

### 트랜잭션 상태 이해하기

**트랜잭션(Transaction)**은 데이터베이스의 상태를 변환시키는 하나의 논리적 작업 단위입니다. 예를 들어 "A 계좌에서 B 계좌로 1만원 이체"는 하나의 트랜잭션입니다. 트랜잭션은 다음과 같은 상태를 거칩니다:

```
[활동(Active)]
     ↓
[부분 완료(Partially Committed)]  ←── 마지막 연산 완료, 아직 디스크에 반영 전
     ↓
[완료(Committed)]  ←── 디스크에 영구 반영 완료

또는...

[활동(Active)]
     ↓
[실패(Failed)]  ←── 오류 발생
     ↓
[철회(Aborted)]  ←── 모든 변경 취소, 롤백 완료
```

> 🚨 **중요**: "실패 → 부분 완료"로 가는 경로는 **절대 존재하지 않습니다!** 실패한 트랜잭션은 반드시 철회(Aborted)로 가야 합니다.

트랜잭션 상태 변화의 합법적인 경로:

| 변화 | 가능 여부 | 이유 |
|------|-----------|------|
| 활동 → 부분완료 → 완료 | ✅ 정상 완료 경로 | 모든 작업 성공 |
| 부분완료 → 실패 → 철회 | ✅ 가능 | 마지막 단계에서 문제 발생 |
| 활동 → 실패 → 철회 | ✅ 정상 실패 경로 | 처리 중 오류 발생 |
| 활동 → 실패 → **부분완료** | ❌ **불가** | 실패 후 부분완료로 가는 경로 없음 |

:::danger 절대 틀리면 안 되는 것
"활동 → 실패 → 부분완료" 경로는 **존재하지 않습니다!** 실패한 트랜잭션은 반드시 **철회(Abort)**로 가야 합니다. 이를 헷갈리면 시험에서 반드시 틀립니다.
:::

### 체크포인트(Checkpoint) 기법

**체크포인트(Checkpoint)**란 특정 시점에 "현재까지의 작업은 안전하게 디스크에 저장되었다"고 표시하는 기법입니다. 마치 게임에서 세이브 포인트를 찍는 것과 같습니다. 게임이 강제 종료되더라도 세이브 포인트부터 다시 시작하면 됩니다.

시스템이 장애(Failure)로 다운된 후 회복할 때, **체크포인트를 기준으로** 각 트랜잭션을 어떻게 처리할지 결정합니다:

| 상황 | 처리 방법 | 이유 |
|------|-----------|------|
| 체크포인트 **이전**에 Commit된 트랜잭션 | **아무것도 안 함** | 이미 디스크에 안전하게 저장됨 |
| 체크포인트 **이후**에 Commit된 트랜잭션 | **REDO (재실행)** | Commit했지만 디스크 반영 전에 다운됐을 수 있음 |
| Commit **안 된** 트랜잭션 | **UNDO (취소)** | 완료되지 않은 작업은 되돌려야 함 |

이것을 타임라인으로 표현하면:

```
──────────[체크포인트]──────────[장애 발생]──────
T1: ──Commit──|                               → 아무것도 안 함 ✓
T2:          ──|────────Commit──              → REDO 필요
T3:          ──|─────────────────(미완료)     → UNDO 필요
```

- **T1**: 체크포인트 이전에 이미 커밋 완료 → 디스크에 확실히 저장됨 → **무시**
- **T2**: 체크포인트 이후에 커밋 → 디스크 미반영 가능성 → **REDO**
- **T3**: 커밋 안 됨 → 불완전한 작업 → **UNDO**

:::tip REDO vs UNDO 쉽게 기억하기
- **REDO (다시 실행)**: "했어야 했는데 아직 안 된 것" → Commit은 했지만 디스크 미반영
- **UNDO (되돌리기)**: "하다가 만 것" → Commit 못 하고 장애 발생
:::

### ARIES 회복 기법

**ARIES(Algorithm for Recovery and Isolation Exploiting Semantics)**는 IBM에서 개발한 고급 회복 알고리즘입니다. 현재 대부분의 상용 데이터베이스 시스템에서 사용하는 표준 회복 기법입니다.

ARIES의 가장 중요한 특징은 다음과 같습니다:

1. **회복 수행 후 로그 파일을 생성**하여 회복 시간을 단축합니다
2. **Commit된 트랜잭션은 REDO** (재실행)합니다
3. **Commit 안 된 트랜잭션은 UNDO** (취소)합니다
4. **UNDO 시에도 로그를 남깁니다** → 이것이 핵심!

4번이 특히 중요합니다. ARIES는 UNDO 작업을 수행할 때도 **CLR(Compensation Log Record, 보상 로그 레코드)**를 로그에 기록합니다. 이렇게 하는 이유는, UNDO 도중에 또 다른 장애가 발생했을 때 이미 UNDO한 작업을 **두 번 UNDO하지 않도록** 보호하기 위해서입니다.

:::warning ARIES에서 자주 출제되는 함정
"ARIES는 UNDO 시 로깅을 하지 않는다" → **이것은 틀린 설명입니다!**
ARIES는 UNDO 시에도 반드시 **CLR(보상 로그 레코드)**를 남깁니다.
시험에서 "UNDO 시 로깅X"라고 나오면 무조건 **틀린 설명**입니다!
:::

ARIES의 3단계 회복 과정:

```
1단계: 분석(Analysis)
   - 로그를 처음부터 읽어 어떤 트랜잭션이 REDO/UNDO 필요한지 파악

2단계: REDO
   - 로그를 순서대로 재실행하여 장애 직전 상태로 복원

3단계: UNDO
   - 미완료 트랜잭션들을 역순으로 취소 (CLR 로그 기록)
```

## 🔍 5. 고급 질의 처리와 최적화

### 고급 질의어 처리 4단계

SQL 쿼리가 입력되면 데이터베이스 내부에서 다음 4단계를 거쳐 처리됩니다. 이것은 마치 외국어로 쓰인 레시피를 이해하고 요리하는 과정과 비슷합니다.

**1단계: 구문 분석(Parsing, 파싱)**
SQL 문장을 분석하고 문법적으로 이상이 있는지 검사합니다. 마치 작문한 글에서 맞춤법 검사를 하는 것과 같습니다. `SELEKT` 같은 오타나 잘못된 문법은 이 단계에서 걸러집니다.

**2단계: 내부 표현 변환**
파싱이 끝난 SQL을 데이터베이스 내부에서 처리하기 쉬운 형태(쿼리 트리, Query Tree)로 변환합니다. 이 쿼리 트리는 연산들의 계층 구조를 나타냅니다.

**3단계: 후보 프로시저 선택**
쿼리 트리를 실행하는 여러 가지 방법(후보 프로시저) 중에서 적절한 것을 선택합니다.

**4단계: 경험적 최적화 규칙 적용**
선택된 방법을 더 빠르게 실행할 수 있도록 최적화합니다. 데이터를 최대한 일찍 줄여서 처리량을 최소화합니다.

:::info 처리 순서 암기법
**파(싱) → 내(부변환) → 후(보선택) → 경(험적최적화)**
"파내후경" 으로 외워보세요!
:::

### 경험적 최적화 규칙

최적화 단계에서 적용되는 주요 규칙들입니다:

- ✅ **SELECT 연산은 가능한 한 아래로**: 데이터를 일찍 걸러내어 처리할 양을 줄입니다
- ✅ **가장 제한적인 SELECT는 하위 노드에**: 데이터를 가장 많이 줄이는 조건을 먼저 실행합니다
- ✅ **카티션 프로덕트(Cartesian Product)가 리프 노드에서 일어나지 않도록**: 카티션 프로덕트는 두 테이블의 모든 조합을 만들어 결과가 폭발적으로 커지므로, 가능하면 조인으로 대체합니다
- ✅ **카티션 프로덕트 + SELECT → 조인으로 변환**: 더 효율적인 조인 연산으로 대체합니다

:::warning 자주 출제되는 함정
"가장 제한적인 SELECT는 **상위** 노드로" → **이것은 틀린 설명입니다!**
제한적인 SELECT일수록 **하위(아래) 노드**에 배치해야 합니다. 아래에서 먼저 데이터를 줄여야 위쪽에서 처리할 양이 적어지기 때문입니다.
:::

## 🛡️ 6. 데이터베이스 보안

### 보안의 여러 위협 유형

데이터베이스 보안을 위협하는 다양한 공격 유형을 이해해야 합니다.

### 추론(Inference) 공격

**추론(Inference)**이란 보안 등급이 없는 일반 사용자가 합법적으로 접근 가능한 공개 정보를 조합하여 기밀 정보를 간접적으로 알아내는 것입니다.

직접 기밀 데이터에 접근하지 않았음에도 불구하고, 공개 정보의 조합만으로 비밀을 알아낸다는 점에서 매우 교묘한 공격입니다.

**예시**:
```
공개 정보 1: 개발팀의 평균 월급 = 500만원
공개 정보 2: 개발팀 총 인원 = 5명
공개 정보 3: 팀장을 제외한 4명의 총 월급 = 1,800만원

→ 유추: 팀장 월급 = 5 × 500 - 1,800 = 700만원 (기밀!)
```

이처럼 개별 공개 정보들을 조합해 비밀 정보를 알아내는 것이 추론 공격입니다.

### DDoS 공격

**DDoS(Distributed Denial of Service, 분산 서비스 거부 공격)**는 악성코드에 감염된 수천, 수만 대의 **좀비 PC**를 동원하여 특정 서버에 동시에 대량의 트래픽을 보내, 서버가 정상적인 서비스를 못 하도록 마비시키는 공격입니다.

| 구분 | DoS | DDoS |
|------|-----|------|
| **공격 주체** | 1대의 PC | 수천~수만 대의 좀비 PC |
| **D의 의미** | - | Distributed(분산) |
| **방어 난이도** | 상대적으로 쉬움 | 매우 어려움 |

### SQL 인젝션(SQL Injection)

**SQL 인젝션**은 사용자 입력창에 악의적인 SQL 코드를 삽입하여 데이터베이스를 조작하는 공격입니다. SQL 변형, 코드 투입, 함수 투입을 통해 데이터베이스를 공격합니다.

```sql
-- 정상적인 로그인 쿼리
SELECT * FROM users WHERE id = '입력값' AND pw = '입력값';

-- SQL 인젝션 공격 예시
-- 아이디 입력: admin' OR '1'='1
SELECT * FROM users WHERE id = 'admin' OR '1'='1' AND pw = '';
-- '1'='1'은 항상 참이므로 비밀번호 없이 로그인 가능!
```

:::danger SQL 인젝션 방어법
SQL 인젝션은 **준비된 구문(Prepared Statement)**이나 **파라미터 바인딩**을 사용하면 예방할 수 있습니다. 절대로 사용자 입력을 SQL 문자열에 직접 연결하지 마세요!
:::

### 권한 관리: GRANT와 REVOKE

데이터베이스에서 사용자 권한을 관리하는 두 가지 명령어입니다:

```sql
-- ✅ 권한 부여: GRANT ... TO
GRANT SELECT ON STUDENT TO U1;
-- U1 사용자에게 STUDENT 테이블 조회(SELECT) 권한을 부여

-- ✅ 권한 회수: REVOKE ... FROM
REVOKE SELECT ON STUDENT FROM U1;
-- U1 사용자로부터 STUDENT 테이블 조회 권한을 회수
```

- **GRANT**: 권한을 **줄 때** → `TO` 사용
- **REVOKE**: 권한을 **뺄 때** → `FROM` 사용

:::tip GRANT vs REVOKE 암기법
- **GRANT** → **TO** (선물을 **~에게** 주다)
- **REVOKE** → **FROM** (권한을 **~으로부터** 빼앗다)
:::

## 📐 7. 무결성 규정과 정규화

### 무결성 규정의 구성요소

**무결성(Integrity)**이란 데이터베이스의 데이터가 정확하고 일관성 있는 상태를 유지하는 것을 말합니다. 무결성 규정은 이를 보장하기 위한 제약 조건들의 집합입니다.

무결성 규정은 4가지 구성요소로 이루어집니다:

| 구성요소 | 역할 | 예시 |
|----------|------|------|
| **rule name** | 규정의 이름 | `CONST-YEAR` |
| **trigger condition** | **검사 시기** (언제 검사할 것인가) | `AFTER UPDATE`, `BEFORE INSERT` |
| **constraints** | 만족해야 할 조건 | `CHECK (VALUE IN (1,2,3,4))` |
| **violation action** | 조건 위반 시 처리 | `ROLLBACK`, `ELSE DO` |

**도메인 무결성 규정 예시**:

```sql
-- ✅ 도메인 무결성 규정 예시
CREATE DOMAIN DYEAR SMALLINT
    CONSTRAINT CONST-YEAR      -- rule name: 규정 이름
    CHECK (VALUE IN (1,2,3,4)); -- constraints: 학년은 1,2,3,4 중 하나여야 함

-- 이 규정은 학년(DYEAR) 속성이 반드시 1, 2, 3, 4 중 하나의 값만
-- 가질 수 있도록 제한합니다. 5학년이나 0학년은 입력 불가!
```

이 예시는 **도메인 무결성 규정**으로, 속성값이 특정 범위나 집합 내의 값만 가질 수 있도록 제한합니다.

무결성 규정에서 제약 조건을 기술할 때 사용하는 명령어는 **`CHECK`**입니다.

### 정규화(Normalization)와 이상현상

**이상현상(Anomaly)**은 정규화되지 않은 릴레이션을 조작할 때 발생하는 문제입니다. 이상현상의 **근본적인 원인**은 "여러 가지 사실(종속 관계)을 하나의 릴레이션으로 무리하게 표현"하기 때문입니다.

예를 들어, "학생 정보"와 "수강 정보"와 "교수 정보"를 하나의 테이블에 억지로 넣으면:
- **삽입 이상**: 학생 없이는 과목을 추가할 수 없는 상황
- **삭제 이상**: 특정 학생을 삭제하면 그 학생만 듣던 과목 정보까지 사라지는 상황  
- **갱신 이상**: 교수 이름이 바뀌면 해당 교수의 모든 수강 레코드를 다 바꿔야 하는 상황

이런 이상현상을 해결하는 방법이 **정규화(Normalization)**입니다. 정규화는 릴레이션을 올바른 형태로 분해하여 각각의 종속 관계를 분리합니다.

### 정규형 판별 예시

**문제**: R(A, B, C), 기본키: (A, B), 함수적 종속성: {A,B} → C, C → B

```
단계 1: 2NF 확인
- {A,B} → C: 기본키 전체에서 C가 종속 = 완전 함수 종속 ✅
- 부분 함수 종속(기본키의 일부에 종속)이 없으므로 2NF 만족!

단계 2: 3NF 확인
- C → B: 일반 속성 C가 다른 일반 속성 B를 결정함 = 이행적 종속 ❌
- 결정자 C가 기본키가 아니므로 3NF 위반!

결론: 이 릴레이션은 제2정규형(2NF)
```

### 정규화의 장단점

정규화가 무조건 좋은 것은 아닙니다:

- **장점**: 데이터 중복 감소, 이상현상 제거, 데이터 무결성 향상
- **단점**: 테이블이 여러 개로 분해되어 JOIN이 많아짐 → **응답 속도 저하**

:::warning 정규화와 성능
"정규화된 릴레이션은 빠른 응답을 보인다" → **이것은 틀린 설명입니다!**
정규화하면 오히려 JOIN이 많아져 응답이 느려질 수 있습니다.
빠른 응답이 목적이라면 오히려 **역정규화(De-normalization)**를 고려해야 합니다.
:::

### 스키마 변환의 원리

스키마 변환(정규화 과정에서 릴레이션을 올바른 형태로 변환)의 3가지 원리:

1. **정보 표현의 무손실**: 변환 후에도 원래 정보를 모두 표현할 수 있어야 합니다
2. **최소의 데이터 중복 허용**: 중복을 최소화합니다
3. **분리의 원칙**: 독립적인 종속성은 분리된 릴레이션으로 표현합니다

:::danger 통합의 원칙은 스키마 변환 원리가 아닙니다!
스키마 변환의 원리는 **무손실, 최소 중복, 분리**입니다. "통합의 원칙"은 스키마 변환 원리에 해당하지 않습니다!
:::

## 📊 8. DB 성능 튜닝 기법

### 실체화된 뷰(Materialized View)

**실체화된 뷰(Materialized View)**는 일반 뷰와 달리 쿼리 결과를 **실제로 디스크에 저장**해두는 특수한 뷰입니다. 일반 뷰는 쿼리할 때마다 실시간으로 계산하지만, 실체화된 뷰는 미리 계산된 결과를 저장해두기 때문에 조회 속도가 훨씬 빠릅니다.

마치 자주 쓰는 계산 결과를 계산기로 매번 계산하는 대신 메모장에 적어두고 꺼내보는 것과 같습니다.

> **실체화된 뷰는 성능을 향상시킵니다.** "저하시킨다"는 설명은 틀렸습니다!

### 역정규화(De-normalization)

**역정규화**는 정규화로 분해된 테이블을 의도적으로 다시 합치는 기법입니다. JOIN을 줄여 조회 성능을 높이기 위해 사용합니다. 데이터 중복이 생기지만, 성능이 중요한 경우에는 유용합니다.

### UNION vs UNION ALL

```sql
-- ❌ UNION: 중복을 제거하는 정렬 작업이 추가로 발생 (느림)
SELECT name FROM student_A
UNION
SELECT name FROM student_B;

-- ✅ UNION ALL: 중복 제거 없이 단순 결합 (빠름)
-- 중복이 없다는 것이 확실할 때, 또는 중복이 있어도 괜찮을 때 사용
SELECT name FROM student_A
UNION ALL
SELECT name FROM student_B;
```

`UNION ALL`은 중복 제거 과정이 없으므로 `UNION`보다 빠릅니다. 중복이 없는 데이터나 중복을 허용하는 경우에는 `UNION ALL`을 사용하는 것이 성능에 유리합니다.

## 🎯 핵심 정리

이 문서에서 배운 핵심 개념들을 한 눈에 정리합니다.

- **버퍼링(Buffering)**: 데이터를 메모리 버퍼에 모아 한꺼번에 저장하여 I/O 횟수를 줄이고 성능을 향상시키는 기법
- **지지도**: 전체 트랜잭션 중 A와 B가 함께 구매된 비율 (분모: 전체)
- **신뢰도**: A를 구매한 사람 중 B도 구매한 비율 (분모: A 구매 건수)
- **로킹 단위**: 클수록 동시 수행↓·오버헤드↓·충돌↑·일관성↑, 작을수록 그 반대
- **S락(공유 락)**: 여러 트랜잭션이 동시에 보유 가능 (읽기 전용)
- **X락(배타 락)**: 하나의 트랜잭션만 보유 가능 (읽기·쓰기)
- **2단계 로킹(2PL)**: 확장 단계(lock만) → 수축 단계(unlock만), unlock 후 재lock 불가
- **교착상태 해결책**: 예방(Prevention), 탐지(Detection), 회복(Recovery) 3가지 (비선점은 독립 해결책 아님)
- **트랜잭션 상태**: 활동→부분완료→완료, 또는 활동→실패→철회 (실패→부분완료는 불가)
- **체크포인트 이전 Commit**: 아무 작업 없음 / 이후 Commit: REDO / 미완료: UNDO
- **ARIES**: UNDO 시에도 CLR(보상 로그 레코드)를 반드시 남김 (UNDO 시 로깅 안 한다는 말은 틀림)
- **구문 분석(파싱)**: 고급 질의어 처리 1단계, SQL 문법 오류 검사
- **경험적 최적화**: 제한적인 SELECT는 하위 노드에 배치 (상위가 아님)
- **추론(Inference)**: 공개 정보 조합으로 기밀 정보를 간접적으로 알아내는 보안 위협
- **DDoS**: 수천 대의 좀비 PC로 동시 공격하여 서비스 마비 (D = Distributed)
- **SQL 인젝션**: 입력창에 SQL 코드를 삽입하여 DB를 조작하는 공격
- **GRANT ... TO**: 권한 부여 / **REVOKE ... FROM**: 권한 회수
- **trigger condition**: 무결성 규정에서 검사 시기를 나타내는 구성요소
- **CHECK**: 무결성 규정에서 제약 조건을 기술하는 명령어
- **정규화 이상현상 원인**: 여러 종속 관계(사실)를 하나의 릴레이션에 표현했기 때문
- **정규화 단점**: JOIN 증가로 응답 속도 저하 가능 (빠른 응답을 보장하지 않음)
- **스키마 변환 3원리**: 무손실, 최소 중복, 분리의 원칙 (통합의 원칙은 포함 안 됨)
- **실체화된 뷰(Materialized View)**: 쿼리 결과를 미리 저장하여 성능 향상 (저하가 아님)
