# 🔐 동시성 제어와 로킹: 교착상태, 로크 단위, 2단계 로킹

데이터베이스를 공부하다 보면 반드시 마주치는 개념들이 있습니다. 바로 **동시성 제어(Concurrency Control)**, **로킹(Locking)**, **교착상태(Deadlock)**, 그리고 **2단계 로킹 규약(2PL)**입니다. 이 문서에서는 시험에 자주 출제되는 핵심 개념들을 초보자도 쉽게 이해할 수 있도록 실생활 비유와 함께 아주 자세하게 설명합니다. 중간에 포기하지 마세요 — 읽고 나면 반드시 "아, 이거였구나!" 하는 순간이 옵니다.

---

## 🗂️ 1. 개념 소개: 이 문서에서 다루는 것들

이 문서는 데이터베이스 시험 기출문제를 바탕으로 다음 개념들을 다룹니다.

- **버퍼링(Buffering)**: I/O 성능을 높이기 위한 메모리 기법
- **구문 분석(Parsing)**: SQL 처리 단계 중 첫 번째 관문
- **추론(Inference)**: 공개 정보로 비밀을 유추하는 보안 위협
- **DDoS 공격**: 분산 서비스 거부 공격
- **정규형(Normal Form)**: 릴레이션의 품질을 나타내는 등급
- **연관 규칙(Association Rule)**: 지지도와 신뢰도 계산
- **교착상태(Deadlock)**: 트랜잭션이 서로를 기다리며 멈추는 현상
- **로킹(Locking)과 2단계 로킹(2PL)**: 동시성 제어의 핵심 메커니즘
- **체크포인트(Checkpoint)와 ARIES**: 데이터베이스 회복 기법
- **트랜잭션 상태 변화**: 활동 → 완료 또는 철회까지의 흐름
- **무결성 규정**: CHECK 제약 조건과 도메인 무결성
- **정규화(Normalization)와 역정규화**: 이상현상의 원인과 해결책

:::info 학습 목표
이 문서를 끝까지 읽고 나면 위의 모든 개념을 개념 설명, 실제 예시, 시험 출제 포인트까지 함께 이해할 수 있습니다.
:::

## 📦 2. 버퍼링, SQL 처리 단계, 보안 개념 (주관식 1~4번)

### 🔵 버퍼링(Buffering) — 왜 매번 저장하면 느릴까?

**왜 이게 필요한가?** 컴퓨터에서 데이터를 저장장치(하드디스크, SSD)에 저장하는 작업은 생각보다 훨씬 느립니다. CPU나 메모리에 비해 저장장치 입출력(I/O) 속도는 수백 배 이상 차이가 납니다. 만약 `write` 연산이 일어날 때마다 저장장치에 직접 저장한다면, 매번 느린 I/O 작업이 발생하여 전체 데이터베이스 성능이 심각하게 저하됩니다.

> **버퍼링(Buffering)**: 레코드(데이터 한 줄)를 메인 메모리의 버퍼 블록(임시 저장 공간)에 모아두었다가, 일정량이 쌓이면 한꺼번에 저장장치에 저장하는 기법.

**실생활 비유**: 마치 택배 기사님이 배달물을 하나씩 따로따로 배달하는 것이 아니라, 같은 아파트 단지 물건을 트럭에 다 모아서 한 번에 배달하는 것과 같습니다. 한 건씩 배달하면 연료와 시간이 엄청나게 낭비되지만, 묶어서 한 번에 가면 훨씬 효율적이죠.

:::tip 시험 포인트
"write 연산 시마다 저장장치에 저장하면 성능이 저하된다 → 해결책은?" 이라는 문제가 나오면 **버퍼링(Buffering)** 이 정답입니다.
:::

---

### 🔵 고급 질의어 처리 단계 — SQL은 어떻게 처리되나?

**왜 이게 필요한가?** 우리가 SQL을 입력하면, 데이터베이스는 그냥 바로 실행하지 않습니다. 마치 요리사가 레시피(SQL)를 받아서 재료를 확인하고, 손질하고, 조리하는 순서가 있듯이, SQL도 여러 단계를 거쳐 최종 실행됩니다. 이 과정을 **질의 처리(Query Processing)** 라고 합니다.

처리 단계는 다음과 같습니다:

| 단계 | 명칭 | 설명 |
|------|------|------|
| 1단계 | **파싱(Parsing, 구문 분석)** | SQL 문법 오류 검사. "이 문장이 올바른 SQL인가?" 확인 |
| 2단계 | **내부 표현(쿼리 트리)으로 변환** | SQL을 DB 내부 자료구조(트리 형태)로 변환 |
| 3단계 | **후보 프로시저 선택** | 실행 가능한 여러 실행 계획(후보)을 만들어냄 |
| 4단계 | **경험적 최적화 규칙 적용** | 가장 효율적인 실행 계획을 선택 |

**1단계 파싱(Parsing)**은 마치 학교에서 선생님이 학생의 답안지를 받아서 "이 글자가 맞나? 문장이 제대로 쓰여 있나?"를 먼저 확인하는 것과 같습니다. 문법이 틀렸으면 그 즉시 오류 메시지를 돌려보냅니다.

:::info 2단계 내부 표현 변환이란?
2단계는 SQL 문장을 트리(나무 구조) 형태로 변환하는 단계입니다. 이렇게 하면 DB 내부에서 최적화하기 쉬운 형태가 됩니다. 시험에서 "두 번째 단계는?" 이라고 물으면 **내부 형태(쿼리 트리)로 변환** 이 정답입니다.
:::

**경험적 최적화 규칙**과 관련된 시험 포인트도 중요합니다. 최적화의 핵심 규칙들을 정리하면:

- `SELECT` 연산은 **가장 아래(하위 노드)**에 배치하여 먼저 실행 → 데이터양을 빨리 줄임
- 가장 결과를 많이 줄이는(제한적인) `SELECT`는 **하위 노드에** 배치 (상위 노드 ❌)
- 카티션 프로덕트(Cartesian Product)는 리프 노드에서 일어나지 않도록 배치
- 카티션 프로덕트 + `SELECT` 조합은 **조인(JOIN)**으로 변환하여 최적화

:::warning 자주 틀리는 포인트
"가장 제한적인 SELECT는 상위 노드로" → ❌ 틀린 설명입니다!  
가장 제한적인 SELECT는 **하위 노드(아래)**에 배치해야 합니다. 아래에서 먼저 데이터를 걸러줘야 위쪽 연산에서 처리할 데이터량이 줄어들기 때문입니다.
:::

### 🔵 추론(Inference) — 공개 정보로 비밀을 알아낸다?

**왜 이게 필요한가?** 데이터베이스 보안에서는 단순히 "비밀 데이터에 접근하지 못하게 막는 것"만으로는 충분하지 않습니다. 보안 등급이 없는 일반 사용자도 공개된 정보들을 교묘하게 조합해서 기밀 정보를 알아낼 수 있기 때문입니다.

> **추론(Inference)**: 보안으로 분류되지 않은 공개 정보를 조합하여 기밀 정보를 유추(추론)해 내는 행위.

**실생활 예시**: 회사 전체 직원 수가 10명이고, 인사팀을 제외한 9명의 평균 월급이 공개되어 있다고 합시다. 또한 전체 10명의 평균 월급도 공개되어 있다면? 이 두 공개 정보를 계산하면 인사팀 직원의 월급을 정확히 알아낼 수 있습니다. 어떤 비밀 데이터에도 접근하지 않았는데도 말이죠!

이것이 바로 **추론 공격**입니다. 데이터베이스 보안 설계 시 반드시 고려해야 할 위협입니다.

---

### 🔵 DDoS 공격 — 좀비 PC 군단의 공격

**DDoS(Distributed Denial of Service, 분산 서비스 거부 공격)**는 악성코드에 감염된 수많은 좀비 PC들을 이용해 특정 서버나 웹사이트에 엄청난 양의 트래픽(접속 요청)을 한꺼번에 보내어 정상적인 서비스가 불가능하게 만드는 해킹 방식입니다.

| 구분 | 공격 주체 | 특징 |
|------|-----------|------|
| **DoS** | 1대의 컴퓨터 | 단순 서비스 거부 공격 |
| **DDoS** | 수천~수만 대의 좀비 PC | 분산(Distributed) 공격, 추적 어려움 |

**비유**: DoS는 한 명이 가게 문 앞에 서서 다른 손님이 못 들어오게 막는 것이고, DDoS는 수천 명이 동시에 몰려와 가게를 마비시키는 것입니다. 수천 명이 동시에 오면 어디서 왔는지 파악하기도 어렵습니다.

## 📐 3. 정규형과 연관 규칙 (주관식 5번, 객관식 6번)

### 🔵 함수적 종속성과 정규형 판단

**왜 이게 필요한가?** 데이터베이스 설계에서 릴레이션(테이블)이 얼마나 잘 설계되었는지를 **정규형(Normal Form)**으로 평가합니다. 정규형이 높을수록 이상현상(삽입/삭제/갱신 이상)이 줄어드는 잘 설계된 테이블입니다.

**문제 예시**: R(A, B, C), 기본키: (A, B), 함수적 종속성: {A, B} → C, C → B 일 때 몇 정규형인가?

**단계별 분석**:

**1단계 - 2NF(제2정규형) 확인**:
- 2NF는 기본키에 대한 **완전 함수 종속**이 만족되어야 합니다.
- 완전 함수 종속(Complete Functional Dependency): 기본키 전체에 종속되고, 기본키의 일부분에는 종속되지 않는 것.
- `{A, B} → C`: 기본키 (A, B) 전체에 C가 종속됨 → ✅ **완전 함수 종속 만족**
- 따라서 2NF는 **만족**합니다.

**2단계 - 3NF(제3정규형) 확인**:
- 3NF는 **이행적 종속(Transitive Dependency)**이 없어야 합니다.
- 이행적 종속: A → B이고 B → C이면 A → C가 되는 것처럼, 기본키가 아닌 속성을 거쳐서 간접적으로 종속되는 관계.
- `C → B`: C가 B를 결정합니다. 그런데 C는 기본키가 아닙니다.
- 기본키가 아닌 C가 다른 속성 B를 결정하고 있으므로 → ❌ **이행적 종속 존재, 3NF 위반**

**결론**: R은 **제2정규형(2NF)**입니다.

:::note 정규형 정리
- **1NF**: 모든 속성이 원자값(더 이상 쪼갤 수 없는 값)을 가짐
- **2NF**: 1NF + 완전 함수 종속 (부분 종속 제거)
- **3NF**: 2NF + 이행적 종속 제거
- **BCNF**: 3NF + 모든 결정자가 후보키
:::

### 🔵 연관 규칙: 지지도와 신뢰도 계산

**왜 이게 필요한가?** 마트에서 "우유를 사는 사람은 주스도 산다"와 같은 구매 패턴을 분석하는 것을 **연관 규칙 분석(Association Rule Mining)**이라고 합니다. 이때 규칙의 강도를 나타내는 지표가 **지지도(Support)**와 **신뢰도(Confidence)**입니다.

**문제 예시**: 연관규칙 `Milk ⇒ Juice` — 총 1,000개 트랜잭션, Milk 구매 700개, Juice 구매 500개, 동시 구매 350개

**지지도(Support) 계산**:
```
지지도 = (Milk와 Juice 동시 구매 수) / (전체 트랜잭션 수)
       = 350 / 1000
       = 0.35
       = 35%
```

**신뢰도(Confidence) 계산**:
```
신뢰도 = (Milk와 Juice 동시 구매 수) / (Milk 구매 수)
       = 350 / 700
       = 0.5
       = 50%
```

> **지지도(Support)** = 전체 중 두 상품을 함께 산 비율. "이 규칙이 얼마나 자주 나타나는가?"  
> **신뢰도(Confidence)** = Milk를 산 사람 중 Juice도 산 비율. "Milk를 사면 Juice도 살 확률이 얼마나 되는가?"

:::tip 암기 팁
- 지지도 = 동시 구매 / **전체** (분모: 전체)
- 신뢰도 = 동시 구매 / **원인(Milk)** (분모: 원인 항목)
:::

## 🔒 4. 교착상태와 로킹의 핵심 원리 (객관식 7~10번)

### 🔵 교착상태(Deadlock) — 서로 기다리다 아무것도 못 하는 상황

**왜 이게 필요한가?** 여러 트랜잭션(작업 묶음)이 동시에 실행될 때, 서로 상대방이 점유한 자원을 기다리며 아무도 진행하지 못하는 상황이 발생할 수 있습니다. 이것이 **교착상태(Deadlock)**입니다.

**실생활 비유**: 좁은 골목에서 두 차가 마주쳤는데, 서로 "네가 먼저 비켜" 하며 기다리는 상황입니다. 양쪽 다 움직이지 않으면 영원히 막히게 됩니다.

교착상태의 **해결 방법 3가지**:

| 해결 방법 | 설명 | 예시 |
|-----------|------|------|
| **예방(Prevention)** | 교착상태가 발생하지 않도록 사전에 조건 차단 | 트랜잭션 시작 시 필요한 모든 자원을 미리 잠금 |
| **탐지(Detection)** | 교착상태 발생을 감지하는 알고리즘 실행 | 대기 그래프(Wait-for Graph)에서 사이클 탐지 |
| **회복(Recovery)** | 교착상태 발생 후 한 트랜잭션을 강제 종료(철회) | 가장 적은 비용의 트랜잭션을 선택해 철회 |

:::warning 시험 함정 주의!
**"비선점(Non-preemption)"은 교착상태의 독립적인 해결책이 아닙니다!**  
비선점은 교착상태 예방(Prevention) 전략의 **하위 조건** 중 하나입니다. 문제에서 "교착상태 해결책으로 틀린 것은?"이라고 물으면 **비선점**이 정답입니다. 예방, 탐지, 회복 — 이 세 가지가 공식적인 해결 전략입니다.
:::

### 🔵 로킹(Locking) 단위와 특성

**왜 이게 필요한가?** 여러 사용자가 동시에 같은 데이터에 접근할 때 데이터가 꼬이지 않도록 **잠금(Lock)**을 사용합니다. 마치 화장실에 들어갈 때 문을 잠그는 것처럼, 내가 사용하는 데이터에 잠금을 걸어두는 것입니다.

**로킹 단위(Lock Granularity)**란 잠금을 거는 데이터의 크기를 의미합니다. 데이터베이스 전체를 잠글 수도 있고, 테이블 하나만 잠글 수도 있고, 특정 레코드(행) 하나만 잠글 수도 있습니다.

| 로킹 단위 크기 | 동시 수행 가능성 | 오버헤드(관리 비용) | 충돌 가능성 | 일관성 |
|--------------|----------------|-------------------|------------|--------|
| **크면 (DB 전체)** | ↓ 낮아짐 | ↓ 낮아짐 | ↑ 높아짐 | ↑ 높아짐 |
| **작으면 (레코드 단위)** | ↑ 높아짐 | ↑ 높아짐 | ↓ 낮아짐 | ↓ 낮아짐 |

**이해하기 쉬운 비유**:
- **큰 단위 잠금** = 도서관 전체를 내가 독점. 아무도 못 들어오지만, 내가 쓰는 책만 잠그는 것보다 훨씬 간단(오버헤드 낮음).
- **작은 단위 잠금** = 내가 쓰는 책 하나만 잠금. 다른 사람은 다른 책을 쓸 수 있어 동시 이용 가능(동시 수행 높음). 하지만 어떤 책이 잠겨있는지 일일이 추적해야 해서 관리가 복잡(오버헤드 높음).

:::warning 시험 함정!
"로킹 단위가 크면 일관성이 증가한다" — 이것은 **맞는 말**입니다!  
하지만 시험에서 **틀린 것을 고르라**는 문제에서 이 선택지가 나오면 당황할 수 있습니다. 문제의 다른 선택지들과 함께 잘 확인하세요. 실제 시험(객관식 8번)에서 정답은 ③번이었습니다 — 문제 맥락을 잘 읽어야 합니다.
:::

### 🔵 공유 락(S-Lock)과 배타 락(X-Lock)의 양립 관계

로킹에는 두 가지 종류가 있습니다:

- **공유 락(Shared Lock, S-Lock)**: 읽기(Read) 전용 잠금. 여러 트랜잭션이 동시에 보유 가능. 읽기는 여러 명이 동시에 해도 괜찮기 때문입니다.
- **배타 락(Exclusive Lock, X-Lock)**: 읽기 + 쓰기 잠금. 혼자만 보유 가능. 쓰기 중에는 다른 사람이 읽거나 쓰면 안 됩니다.

**양립 관계 표** (두 락이 동시에 존재할 수 있는지):

|  | **S락 (공유)** | **X락 (배타)** |
|--|---------------|---------------|
| **S락 (공유)** | ✅ 양립 가능 | ❌ 양립 불가 |
| **X락 (배타)** | ❌ 양립 불가 | ❌ 양립 불가 |

**비유**: S락은 도서관 책을 여러 명이 동시에 **읽는** 것(가능). X락은 책에 낙서(쓰기)를 하는데 다른 사람이 보거나 같이 낙서하면 안 되는 것(불가능).

:::tip 시험 포인트
서로 양립할 수 있는 락의 조합은 **S락 + S락** 뿐입니다!  
X-X, S-X, X-S 조합은 모두 양립 불가입니다.
:::

### 🔵 2단계 로킹 규약(2PL: Two-Phase Locking Protocol)

**왜 이게 필요한가?** 로킹만 사용한다고 해서 무조건 직렬 가능성(Serializability, 동시 실행 결과가 순서대로 실행한 것과 동일함)이 보장되지는 않습니다. 락을 걸고 해제하는 **순서와 방식**이 중요합니다. 2단계 로킹 규약은 이 문제를 해결하기 위해 만들어졌습니다.

> **2단계 로킹 규약(2PL)**: 트랜잭션의 락 연산을 두 단계로 엄격히 나누는 규칙.  
> - **1단계 - 확장 단계(Growing Phase)**: 락(lock)만 걸 수 있고, 언락(unlock)은 절대 불가  
> - **2단계 - 수축 단계(Shrinking Phase)**: 언락(unlock)만 가능하고, 새로운 락은 절대 불가

**핵심**: 한 번 언락을 시작하면 그 트랜잭션은 다시는 새로운 락을 걸 수 없습니다.

**비유**: 마트에서 장보기를 생각해보세요.
- **확장 단계** = 카트에 물건을 계속 담는 단계. 이미 담은 물건을 다시 내려놓으면 안 됩니다.
- **수축 단계** = 계산대에서 물건을 꺼내 계산하는 단계. 이 시점에는 새로운 물건을 카트에 추가할 수 없습니다.

```
확장 단계:  lock(A), lock(B), lock(C)  ← 락만 가능
          ---[이 경계를 넘으면 다시 lock 불가]---
수축 단계:  unlock(A), unlock(B), unlock(C)  ← 언락만 가능
```

**잘못된 예시 ❌**:
```
lock(A) → lock(B) → unlock(A) → lock(C)
                              ↑
                   unlock 후 다시 lock → 2PL 위반!
```

**올바른 예시 ✅**:
```
lock(A) → lock(B) → lock(C) → unlock(A) → unlock(B) → unlock(C)
←────── 확장 단계 ──────→←──────── 수축 단계 ────────────→
```

:::danger 2PL에서 절대 하면 안 되는 것
unlock이 한 번 이라도 발생한 후에는 **절대 새로운 lock을 걸 수 없습니다!**  
이를 위반하면 직렬 가능성이 보장되지 않아 데이터 일관성이 깨집니다.
:::

**2PL의 특성 정리**:

- ✅ 모든 트랜잭션이 2PL을 준수하면 해당 스케줄은 **직렬 가능(Serializable)**합니다.
- ✅ 트랜잭션 종료 전에 모든 lock을 unlock해야 합니다.
- ✅ 다른 트랜잭션이 X-lock을 보유한 데이터에는 새로운 lock을 걸 수 없습니다.
- ❌ 2PL을 사용해도 **교착상태는 여전히 발생할 수 있습니다**. (2PL은 교착상태 해결책이 아닙니다!)

## 🔄 5. 체크포인트, ARIES, 트랜잭션 상태 (객관식 11~13번)

### 🔵 체크포인트(Checkpoint) 기법 — 어디서부터 복구할까?

**왜 이게 필요한가?** 데이터베이스 시스템이 갑자기 장애(예: 정전, 시스템 오류)로 멈췄을 때, 처음부터 모든 로그를 검토하며 복구하면 엄청난 시간이 걸립니다. **체크포인트**는 "여기까지는 안전하게 저장됐다"는 표시를 주기적으로 남겨두어, 장애 발생 시 체크포인트 이후부터만 복구하면 되도록 하는 기법입니다.

**체크포인트 기준으로 트랜잭션 처리 방식**:

| 상황 | 처리 방법 | 이유 |
|------|-----------|------|
| **체크포인트 이전에 Commit된 트랜잭션** | 아무것도 안 함 (No Action) | 이미 안전하게 저장 완료됨 |
| **체크포인트 이후에 Commit된 트랜잭션** | REDO (재실행) | 메모리에는 있었지만 저장장치에 반영 안 됐을 수 있음 |
| **Commit이 안 된 트랜잭션** | UNDO (취소) | 완료되지 않은 작업이므로 이전 상태로 되돌려야 함 |

**비유**: 자동차 여행 중 중간 지점(체크포인트)에서 "여기까지는 안전하게 도착했다"고 표시해둔다고 생각하세요. 나중에 길을 잃으면 처음부터 다시 출발하는 게 아니라, 마지막 체크포인트부터 다시 시작하면 됩니다.

:::tip 시험 포인트
**체크포인트 이전에 Commit된 트랜잭션 → 아무 작업도 수행하지 않음!**  
이것이 체크포인트 기법의 핵심입니다. 이미 안전하게 저장된 것은 건드릴 필요가 없습니다.
:::

### 🔵 ARIES 회복 기법 — 최신 DB 회복의 표준

**ARIES(Algorithm for Recovery and Isolation Exploiting Semantics)**는 현대 데이터베이스 회복 알고리즘의 표준입니다. 세 단계로 동작합니다:

1. **분석(Analysis) 단계**: 로그를 분석하여 어떤 트랜잭션을 REDO/UNDO할지 결정
2. **REDO 단계**: 로그에 기록된 모든 변경 사항을 재실행 (Commit 여부 관계없이)
3. **UNDO 단계**: Commit이 안 된 트랜잭션의 변경 사항을 취소

**ARIES의 핵심 특성**:

- Commit된 트랜잭션은 **REDO** 수행
- Commit이 안 된 트랜잭션은 **UNDO** 수행
- **UNDO 시에도 CLR(Compensation Log Record, 보상 로그 레코드)을 기록합니다!**

:::danger 절대 틀리면 안 되는 포인트
"ARIES는 UNDO 시에는 로깅을 하지 않는다" → ❌ **완전히 틀린 설명!**  
ARIES는 UNDO 시에도 **CLR(보상 로그 레코드)**를 남깁니다.  
이는 UNDO 중에 또 장애가 발생할 경우를 대비한 것입니다.
:::

### 🔵 트랜잭션 상태 변화 — 활동에서 완료 또는 철회까지

**트랜잭션(Transaction)**은 데이터베이스의 논리적 작업 단위입니다. 은행에서 A 계좌에서 B 계좌로 돈을 이체하는 전체 과정이 하나의 트랜잭션입니다.

트랜잭션의 상태 변화는 다음과 같습니다:

```
활동(Active) ─────────────────→ 부분 완료(Partially Committed) ──→ 완료(Committed)
     │                                                                    
     └──────────────────────────→ 실패(Failed) ──────────────────→ 철회(Aborted)
```

- **활동(Active)**: 트랜잭션이 실행 중인 상태
- **부분 완료(Partially Committed)**: 마지막 연산까지 실행했지만 아직 DB에 완전히 반영되지 않은 상태
- **완료(Committed)**: 모든 변경 사항이 DB에 안전하게 반영된 최종 완료 상태
- **실패(Failed)**: 오류가 발생하여 더 이상 진행할 수 없는 상태
- **철회(Aborted)**: 실패 후 모든 변경 사항을 취소(UNDO)하고 원래 상태로 돌아간 상태

:::warning 절대 존재하지 않는 경로
**실패(Failed) → 부분 완료(Partially Committed)** 라는 경로는 존재하지 않습니다!  
실패는 반드시 철회(Aborted)로만 이어집니다. 시험에서 "틀린 상태 변화는?"이라고 물으면 이 경로를 고르면 됩니다.
:::

## 🛡️ 6. DB 성능, 보안, 무결성 (객관식 14~21번)

### 🔵 DB 튜닝 기법 — 더 빠른 데이터베이스 만들기

**왜 이게 필요한가?** 데이터베이스가 커질수록 쿼리 실행 속도가 느려집니다. **튜닝(Tuning)**은 이를 개선하는 작업입니다.

주요 튜닝 기법들:

- **역정규화(De-normalization)**: 정규화로 분리된 테이블을 다시 합쳐서 JOIN을 줄이고 조회 속도를 높이는 기법
- **실체화된 뷰(Materialized View)**: 복잡한 쿼리의 결과를 미리 저장해두어 빠르게 조회하는 기법 → **성능 향상에 도움!** (성능 저하가 아닙니다!)
- **UNION → UNION ALL 대체**: UNION은 중복을 제거하는 추가 작업이 있지만, UNION ALL은 그냥 합치기만 하므로 더 빠름
- **불필요한 COUNT 제거**: `COUNT(*)`는 전체 스캔이 필요하므로 불필요한 경우 제거

:::danger 절대 틀리면 안 됩니다
"실체화된 뷰(Materialized View)는 성능을 저하시킨다" → ❌ **완전히 틀린 설명!**  
실체화된 뷰는 쿼리 결과를 **미리 저장**해두기 때문에 반복 조회 시 훨씬 빠릅니다. 성능 **향상** 기법입니다.
:::

### 🔵 GRANT와 REVOKE — 권한 부여와 회수

데이터베이스에서 사용자에게 특정 테이블에 대한 권한을 부여하거나 회수할 때 사용하는 SQL 명령어입니다.

```sql
-- ✅ 올바른 권한 부여 예시
-- U1 사용자에게 STUDENT 테이블 검색(SELECT) 권한 부여
GRANT SELECT ON STUDENT TO U1;

-- ✅ 올바른 권한 회수 예시
-- U1 사용자로부터 STUDENT 테이블 검색(SELECT) 권한 회수
REVOKE SELECT ON STUDENT FROM U1;
```

| 명령어 | 방향 | 전치사 |
|--------|------|--------|
| **GRANT** | 권한 **부여** | **TO** (누구에게) |
| **REVOKE** | 권한 **회수** | **FROM** (누구로부터) |

**비유**: GRANT는 "열쇠를 줬다(TO)", REVOKE는 "열쇠를 돌려받았다(FROM)"로 기억하세요!

### 🔵 SQL 인젝션(SQL Injection) — 입력창을 통한 DB 공격

**SQL 인젝션**은 사용자 입력창에 악의적인 SQL 코드를 삽입하여 데이터베이스를 조작하는 공격입니다. SQL 변형, 코드 투입, 함수 투입을 통해 DB를 공격하는 기술입니다.

**예시**: 로그인 화면에서 비밀번호 입력란에 아래와 같이 입력하면:
```sql
-- ❌ 악의적인 SQL 인젝션 입력 예시
' OR '1'='1
```
이것이 쿼리에 삽입되면 `WHERE password='' OR '1'='1'`가 되어 항상 참(True)이 되어 비밀번호 없이 로그인이 가능해집니다.

### 🔵 무결성 규정(Integrity Constraint) — 데이터의 품질 보장

**무결성 규정**은 데이터베이스에 저장되는 데이터가 올바른 값을 갖도록 보장하는 규칙입니다.

무결성 규정의 구성 요소:

| 구성 요소 | 의미 | 예시 |
|-----------|------|------|
| **rule name** | 규정의 이름 | CONST-YEAR |
| **trigger condition** | 검사 시기 (언제 이 규칙을 확인할 것인가) | AFTER INSERT, BEFORE UPDATE |
| **constraints** | 만족해야 할 조건 | CHECK (VALUE IN (1,2,3,4)) |
| **violation action** | 위반 시 취할 행동 | ROLLBACK, REJECT |

**도메인 무결성 예시**:
```sql
-- ✅ 도메인 무결성 규정 예시
-- DYEAR 도메인을 SMALLINT로 정의하고,
-- 값은 반드시 1, 2, 3, 4 중 하나여야 함 (학년 제한)
CREATE DOMAIN DYEAR SMALLINT
    CONSTRAINT CONST-YEAR
    CHECK (VALUE IN (1, 2, 3, 4));
    -- VALUE: 해당 도메인에 입력되는 값을 의미
    -- IN (1,2,3,4): 1, 2, 3, 4 중 하나여야 한다는 조건
```

- `CREATE DOMAIN DYEAR SMALLINT`: DYEAR라는 이름의 도메인을 SMALLINT(작은 정수) 타입으로 만들겠다
- `CONSTRAINT CONST-YEAR`: 이 제약 조건의 이름은 CONST-YEAR
- `CHECK (VALUE IN (1,2,3,4))`: 값은 반드시 1, 2, 3, 4 중 하나여야 함

:::info 도메인 무결성이란?
**도메인 무결성(Domain Integrity)**: 특정 속성(컬럼)에 입력될 수 있는 값의 범위나 형식을 제한하는 무결성. 예를 들어 학년(YEAR)은 1~4학년만 가능하므로 1, 2, 3, 4만 허용하는 것입니다.
:::

무결성 제약 조건을 기술할 때는 **CHECK** 명령어를 사용합니다:

```sql
-- ✅ CHECK 제약 조건 예시
-- 나이는 0 이상 150 이하여야 한다
CHECK (age >= 0 AND age <= 150)

-- ✅ 성별은 'M' 또는 'F'만 가능
CHECK (gender IN ('M', 'F'))
```

## 📊 7. 정규화, 이상현상, 스키마 변환 (객관식 22~25번)

### 🔵 이상현상(Anomaly) — 잘못 설계된 테이블의 문제점

**왜 이게 필요한가?** 데이터베이스를 설계할 때 여러 개의 서로 다른 사실(종속 관계)을 억지로 하나의 테이블에 담으면, 데이터를 삽입/삭제/수정할 때 문제가 생깁니다. 이를 **이상현상(Anomaly)**이라고 합니다.

이상현상의 근본 원인은: **여러 가지 사실(종속 관계)을 하나의 릴레이션으로 표현했기 때문**입니다.

이상현상의 종류:
- **삽입 이상(Insertion Anomaly)**: 새 데이터를 삽입할 때 불필요한 데이터도 함께 입력해야 하는 문제
- **삭제 이상(Deletion Anomaly)**: 데이터를 삭제할 때 의도하지 않은 다른 데이터도 함께 삭제되는 문제
- **갱신 이상(Update Anomaly)**: 데이터를 수정할 때 일부만 수정되어 데이터 불일치가 생기는 문제

이상현상의 해결책은 바로 **정규화(Normalization)**입니다!

---

### 🔵 정규화(Normalization) — 이상현상의 해결사

**정규화(Normalization)**는 릴레이션을 올바른 형태로 분해하여 데이터의 중복을 줄이고 이상현상을 제거하는 과정입니다. 마치 복잡하게 엉킨 서랍을 종류별로 정리하는 것과 같습니다.

정규화의 특성:

| 특성 | 설명 |
|------|------|
| ✅ 중복 제거 | 반복되는 데이터를 없애 저장 공간 절약 |
| ✅ 변경 이상 제거 | 삽입/삭제/갱신 이상 현상 해결 |
| ✅ 종속성 분리 | 하나의 종속성 → 하나의 릴레이션으로 분해 |
| ✅ 정규형 검사 | 릴레이션이 몇 정규형인지 테스트 가능 |
| ❌ 빠른 응답 보장 안 함 | 테이블이 분리되어 JOIN이 많아지므로 오히려 느려질 수 있음 |

:::danger 정규화와 성능에 대한 오해
"정규화된 릴레이션은 빠른 응답을 보인다" → ❌ **틀린 설명!**  
정규화를 하면 테이블이 여러 개로 분리되어 데이터 조회 시 JOIN 연산이 많아집니다. 이는 오히려 **응답 속도를 저하**시킬 수 있습니다.  
빠른 응답이 필요할 때는 **역정규화(De-normalization)**를 사용합니다.
:::

### 🔵 스키마 변환의 3원리

**스키마 변환(Schema Transformation)**은 개념적 스키마를 논리적 스키마로 변환하는 과정에서 지켜야 할 원리들이 있습니다.

스키마 변환의 **3가지 원리**:

1. **정보 표현의 무손실(Lossless Representation)**: 변환 후에도 원래의 정보가 손실 없이 모두 표현되어야 합니다. 변환 과정에서 데이터가 사라지면 안 됩니다.
2. **최소의 데이터 중복(Minimal Redundancy)**: 같은 데이터가 여러 곳에 중복 저장되는 것을 최소화합니다. 중복이 많으면 갱신 이상 등의 문제가 생깁니다.
3. **분리의 원칙(Separation Principle)**: 독립적인 관계(종속성)는 독립된 릴레이션으로 분리해야 합니다.

:::warning 스키마 변환 원리에 해당하지 않는 것
**통합의 원칙**은 스키마 변환의 3원리에 **포함되지 않습니다!**  
무손실, 최소 중복, 분리 — 이 세 가지만 스키마 변환의 원리입니다. 시험에서 "스키마 변환 원리와 가장 거리가 먼 것"을 물으면 **통합의 원칙**을 고르세요.
:::

## 📋 핵심 정리

이 문서에서 학습한 핵심 내용을 정리합니다.

- **버퍼링(Buffering)**: 레코드를 메인 메모리에 모아두었다가 한 번에 저장장치에 쓰는 I/O 최적화 기법
- **SQL 처리 4단계**: ①파싱 → ②내부 표현 변환 → ③후보 프로시저 선택 → ④경험적 최적화
- **추론(Inference)**: 공개 정보를 조합해 기밀 정보를 유추하는 보안 위협
- **DDoS**: 수천 대의 좀비 PC가 동시에 공격하는 분산 서비스 거부 공격 (DoS의 분산형)
- **연관 규칙**: 지지도 = 동시구매/전체, 신뢰도 = 동시구매/원인항목
- **교착상태 해결책 3가지**: 예방(Prevention), 탐지(Detection), 회복(Recovery) — 비선점은 독립 해결책 아님
- **로킹 단위**: 클수록 동시성↓, 오버헤드↓, 충돌↑, 일관성↑ / 작을수록 그 반대
- **S락 vs X락**: S-S만 양립 가능, X가 포함된 조합은 모두 양립 불가
- **2단계 로킹(2PL)**: 확장 단계(lock만) → 수축 단계(unlock만), unlock 후 lock 불가, 직렬 가능성 보장
- **체크포인트**: 이전 Commit → 아무것도 안 함, 이후 Commit → REDO, 미완료 → UNDO
- **ARIES**: UNDO 시에도 CLR(보상 로그 레코드) 기록 — "UNDO 시 로깅 안 함"은 틀린 설명
- **트랜잭션 상태**: 활동 → 부분완료 → 완료 또는 활동 → 실패 → 철회, 실패→부분완료 경로는 없음
- **실체화된 뷰**: 쿼리 결과 미리 저장 → 성능 향상 도구 (저하 아님)
- **GRANT TO / REVOKE FROM**: 권한 부여는 TO, 권한 회수는 FROM
- **SQL 인젝션**: 입력창에 SQL 코드 삽입으로 DB 조작하는 공격
- **무결성 규정 구성**: rule name / trigger condition(검사 시기) / constraints(CHECK) / violation action
- **도메인 무결성**: 속성값의 범위/형식을 제한하는 무결성 규정
- **이상현상 원인**: 여러 가지 사실을 하나의 릴레이션으로 표현했기 때문
- **정규화**: 이상현상 해결, 종속성 분리. 단, 정규화 후 JOIN 증가로 응답 느려질 수 있음
- **역정규화(De-normalization)**: 빠른 응답을 위해 정규화된 테이블을 다시 합치는 기법
- **스키마 변환 3원리**: 무손실, 최소 중복, 분리의 원칙 — 통합의 원칙은 포함 안 됨
- **2NF vs 3NF**: 완전 함수 종속 만족 = 2NF, 이행적 종속 없음 = 3NF
