# 🗄️ 연관규칙 분석과 DB 성능 튜닝 — 시험 완전 정복 가이드

데이터베이스 시험에서 자주 등장하는 **연관규칙 분석**, **DB 성능 튜닝**, **트랜잭션 관리**, **보안**, **정규화** 개념을 한 번에 정리합니다. 단순 암기가 아니라 **"왜 이 개념이 필요한가?"** 부터 출발해서 실제 시험 문제 풀이까지 연결하는 것이 이 문서의 목표입니다. 초보자도 충분히 이해할 수 있도록 비유와 예시를 풍부하게 담았으니, 천천히 따라오세요.

---

## 🚀 1. DB 성능 향상 기법 — 버퍼링과 질의 처리

### 🔎 버퍼링(Buffering)이란 무엇인가?

**왜 버퍼링이 필요한가?**부터 생각해 봅시다. 컴퓨터에서 데이터를 저장장치(하드디스크, SSD)에 쓰는 작업은 **메인 메모리(RAM)에서 계산하는 것보다 수백 배 느립니다.** 만약 데이터를 변경할 때마다 매번 저장장치에 기록한다면, 수천 건의 데이터를 처리하는 동안 수천 번의 느린 I/O(입출력) 작업이 발생합니다. 이것은 마치 편의점 알바생이 손님이 물건을 하나 살 때마다 창고에 들어가서 재고를 기록하는 것과 같습니다. 너무 비효율적이죠.

> **버퍼링(Buffering)**: write 연산이 발생할 때마다 저장장치에 저장하지 않고, 메인 메모리의 **버퍼 블록(임시 저장 공간)**에 여러 레코드를 모아두었다가 **한꺼번에** 저장장치에 기록하는 기법입니다.

이 원리는 우리 일상에서도 쉽게 찾아볼 수 있습니다. **택배 기사님**이 집집마다 하나씩 배달하러 왔다 갔다 하지 않고, 한 트럭에 여러 건의 택배를 가득 실어서 한 번 출발할 때 여러 집을 연속으로 배달하는 것과 똑같은 원리입니다. 한 번의 이동으로 여러 건의 배달을 처리하니 훨씬 효율적입니다.

:::tip 버퍼링의 핵심 효과
- **I/O 횟수 감소**: 저장장치 접근 횟수를 대폭 줄여 전체 처리 속도가 향상됩니다.
- **CPU 활용 극대화**: 저장 작업을 기다리는 시간 없이 다음 연산을 바로 처리할 수 있습니다.
- **배치 처리**: 여러 변경 사항을 묶어서 한 번에 처리하므로 디스크 쓰기 효율이 높아집니다.
:::

### 🔎 고급 질의어(SQL) 처리 4단계

SQL 문장을 우리가 입력하면 데이터베이스는 그냥 바로 실행하는 것이 아닙니다. 마치 우리가 외국어 문장을 번역할 때 문법을 먼저 확인하고, 의미를 파악하고, 가장 효율적인 번역 방법을 찾는 것처럼 **4단계의 처리 과정**을 거칩니다.

| 단계 | 이름 | 설명 |
|------|------|------|
| 1단계 | **구문 분석 (파싱, Parsing)** | SQL 문장의 문법 오류를 검사합니다 |
| 2단계 | **내부 표현 변환** | SQL을 내부적으로 처리하기 쉬운 쿼리 트리 형태로 변환합니다 |
| 3단계 | **후보 프로시저 선택** | 실행 가능한 여러 처리 방법 중 후보를 선택합니다 |
| 4단계 | **경험적 최적화 규칙 적용** | 가장 효율적인 실행 계획을 선택합니다 |

**1단계: 구문 분석(파싱)**은 마치 영어 선생님이 학생의 작문을 받아서 문법 오류가 있는지 먼저 빨간 펜으로 체크하는 것과 같습니다. `SELCT * FORM users` 처럼 오타가 있다면 이 단계에서 오류가 잡힙니다. 시험에서 "질의문을 분석하고 문법적으로 이상이 있는지 체크하는 단계"는 **구문 분석(파싱)** 이라는 것을 기억하세요.

**2단계: 내부 표현 변환**은 SQL 문장을 데이터베이스가 처리하기 쉬운 내부 형식인 **쿼리 트리(Query Tree)** 로 바꾸는 단계입니다. 이것은 마치 악보를 읽기 좋은 형태로 다시 정리하는 것과 같습니다.

**4단계: 경험적 최적화 규칙**에는 중요한 원칙이 있습니다. **SELECT(선택) 연산은 가능한 한 가장 아래(하위 노드)에** 배치해야 합니다. 왜냐하면 SELECT 연산은 데이터를 필터링하여 줄여주기 때문에, 먼저 실행할수록 이후 연산이 처리해야 할 데이터 양이 줄어들기 때문입니다. 또한 **카티션 프로덕트(두 테이블의 모든 조합)**는 데이터 양을 폭발적으로 늘리므로, SELECT와 함께 **조인(JOIN)으로 변환**하는 것이 좋습니다.

:::warning 자주 나오는 함정
경험적 최적화에서 "가장 제한적인 SELECT는 **상위** 노드에 배치"라는 선택지는 **틀린 설명**입니다. 제한적인 SELECT일수록 더 **아래(하위)** 에 배치해야 먼저 실행되어 데이터를 빨리 줄일 수 있습니다.
:::

## 📊 2. 연관규칙 분석 — 지지도와 신뢰도 완전 정복

### 🔎 연관규칙이란 무엇인가?

**왜 연관규칙 분석이 필요할까요?** 마트에서 장을 보는 고객들의 구매 데이터를 분석한다고 생각해 봅시다. "우유를 산 사람은 주스도 많이 사더라"는 패턴을 발견하면, 우유 옆에 주스를 진열하거나 우유 구매자에게 주스 할인 쿠폰을 주는 전략을 세울 수 있습니다. 이처럼 **데이터 속에서 함께 자주 발생하는 항목들의 관계를 찾아내는 것**이 연관규칙 분석입니다.

연관규칙은 `A ⇒ B` 형태로 표현합니다. "A를 구매하면 B도 구매한다"는 의미입니다. 이 규칙의 강도를 측정하는 두 가지 핵심 지표가 **지지도**와 **신뢰도**입니다.

### 📐 지지도(Support) 계산법

> **지지도 = (A와 B를 동시에 구매한 트랜잭션 수) ÷ (전체 트랜잭션 수) × 100%**

지지도는 "전체 거래 중에서 A와 B가 함께 등장하는 비율"을 나타냅니다. 지지도가 낮으면 그 연관규칙은 아주 드문 경우에만 해당하므로 실용성이 낮습니다.

### 📐 신뢰도(Confidence) 계산법

> **신뢰도 = (A와 B를 동시에 구매한 트랜잭션 수) ÷ (A를 구매한 트랜잭션 수) × 100%**

신뢰도는 "A를 구매한 사람 중에서 B도 구매한 비율"을 나타냅니다. 신뢰도가 높을수록 "A를 사면 B도 산다"는 규칙이 더 믿을 만합니다.

### 🧮 실제 시험 문제 풀기

**문제**: `Milk ⇒ Juice` 규칙에서 총 1,000개 트랜잭션, Milk 구매 700개, Juice 구매 500개, 동시 구매 350개일 때 지지도와 신뢰도는?

```
# 연관규칙 계산 예시

전체 트랜잭션 = 1,000개
Milk 구매 횟수 = 700개
Juice 구매 횟수 = 500개
동시 구매(Milk AND Juice) = 350개

# ① 지지도(Support) 계산
지지도 = 동시구매 / 전체 = 350 / 1000 = 0.35 = 35%

# ② 신뢰도(Confidence) 계산
신뢰도 = 동시구매 / Milk구매 = 350 / 700 = 0.50 = 50%
# → "우유를 산 사람 중 50%가 주스도 샀다"는 의미

# 정답: 지지도 35%, 신뢰도 50%
```

:::info 지지도 vs 신뢰도 핵심 차이
- **지지도**: 분모가 **전체 트랜잭션** → 이 규칙이 얼마나 자주 등장하는가
- **신뢰도**: 분모가 **조건(A) 트랜잭션** → A를 샀을 때 B도 살 확률이 얼마나 되는가
:::

신뢰도 계산에서 분모를 헷갈리는 경우가 많습니다. `Milk ⇒ Juice`에서 신뢰도의 분모는 화살표 **왼쪽**인 Milk의 구매 횟수입니다. 마치 조건부 확률처럼 "Milk를 산 것을 전제로(조건으로)"라는 뜻입니다.

## 🔒 3. 트랜잭션 관리 — 로킹, 교착상태, 회복

### 🔎 로킹(Locking)이란 무엇인가?

여러 사람이 동시에 같은 데이터를 수정하면 어떻게 될까요? 예를 들어, 두 사람이 동시에 은행 계좌에서 돈을 인출하려 한다면 잔액이 이상하게 처리될 수 있습니다. **로킹(Locking)**은 이런 문제를 막기 위해 한 트랜잭션(작업 묶음)이 데이터를 사용하는 동안 **다른 트랜잭션이 접근하지 못하도록 잠그는 기법**입니다. 마치 화장실 칸에 들어가서 문을 잠그는 것과 같습니다. 안에 있는 동안은 다른 사람이 들어올 수 없죠.

### 🔑 락(Lock)의 종류

| 락 종류 | 기호 | 의미 | 다른 S락과 양립? | 다른 X락과 양립? |
|---------|------|------|----------------|----------------|
| **공유락 (Shared Lock)** | S | 읽기 전용 잠금 | ✅ 가능 | ❌ 불가 |
| **배타락 (Exclusive Lock)** | X | 읽기+쓰기 잠금 | ❌ 불가 | ❌ 불가 |

**공유락(S락)**은 "이 데이터를 읽겠습니다"라고 선언하는 것입니다. 여러 사람이 동시에 같은 책을 읽는 것은 가능하기 때문에, S락은 여러 트랜잭션이 동시에 보유할 수 있습니다. 반면 **배타락(X락)**은 "이 데이터를 수정하겠습니다"라는 선언으로, 수정 중에는 누구도 접근할 수 없어야 하므로 혼자만 보유할 수 있습니다.

### 📏 로킹 단위(Granularity)와 트레이드오프

로킹 단위란 잠금을 거는 데이터의 크기를 말합니다. 데이터베이스 전체를 잠글 수도 있고, 테이블 하나, 레코드 하나, 필드 하나를 잠글 수도 있습니다.

| 로킹 단위 | 동시 수행 가능성 | 오버헤드(부하) | 충돌 가능성 |
|-----------|---------------|--------------|------------|
| **크면** (예: DB 전체) | ↓ 낮아짐 | ↓ 낮아짐 | ↑ 높아짐 |
| **작으면** (예: 레코드 단위) | ↑ 높아짐 | ↑ 높아짐 | ↓ 낮아짐 |

로킹 단위가 크면 관리해야 할 락의 수가 줄어들어 오버헤드는 낮아지지만, 한 번에 큰 범위를 잠그기 때문에 다른 트랜잭션이 기다려야 하는 경우가 많아져 동시 수행 가능성이 낮아집니다. 반대로 단위가 작으면 세밀하게 잠그기 때문에 동시성은 높아지지만 관리해야 할 락이 많아져 오버헤드가 증가합니다.

### 🔄 2단계 로킹 규약 (2-Phase Locking Protocol)

**2단계 로킹 규약**은 트랜잭션의 직렬 가능성(Serializability, 동시 실행해도 순서대로 실행한 것과 같은 결과를 보장하는 성질)을 보장하는 중요한 규칙입니다.

- **확장 단계 (Growing Phase)**: 트랜잭션이 필요한 락을 **획득(lock)만** 할 수 있습니다. 아직 락을 해제할 수 없습니다.
- **수축 단계 (Shrinking Phase)**: 트랜잭션이 락을 **해제(unlock)만** 할 수 있습니다. 더 이상 새로운 락을 획득할 수 없습니다.

핵심은 **한 번 unlock을 시작하면 다시 lock을 걸 수 없다**는 것입니다. 마치 짐을 정리할 때 한 번 내려놓기 시작하면 다시 집어드는 행동 없이 계속 내려놓기만 해야 하는 것과 같습니다.

```
# 2단계 로킹 규약 예시

# ✅ 올바른 2단계 로킹
# 확장 단계: lock만 수행
lock(A)  # A에 락 획득
lock(B)  # B에 락 획득
lock(C)  # C에 락 획득

# 수축 단계: unlock만 수행
unlock(A)  # A 락 해제
unlock(B)  # B 락 해제
unlock(C)  # C 락 해제

# ❌ 2단계 로킹 위반 예시
lock(A)    # A 락 획득
unlock(A)  # A 락 해제 (수축 단계 시작!)
lock(B)    # ❌ 오류! 수축 단계에서 새로운 lock 불가
```

:::danger 절대 하면 안 되는 실수
2단계 로킹에서 **unlock 이후 다시 lock을 거는 행동은 규약 위반**입니다. 이렇게 하면 직렬 가능성이 깨져 데이터 불일치가 발생할 수 있습니다.
:::

### 💀 교착상태(Deadlock)와 해결 방법

**교착상태**란 두 트랜잭션이 서로가 보유한 락을 기다리며 영원히 멈춰있는 상태입니다. 마치 좁은 골목에서 두 차가 서로 마주보고 "네가 먼저 비켜라"고 하며 아무도 물러서지 않는 상황과 같습니다.

교착상태의 해결책은 크게 세 가지입니다:

- **예방(Prevention)**: 교착상태가 발생하지 않도록 처음부터 막는 방법. 모든 락을 한꺼번에 획득하거나, 순서를 정해서 락을 획득합니다.
- **탐지(Detection)**: 교착상태가 발생했는지 주기적으로 확인하는 방법. 대기 그래프(Wait-for Graph)를 분석하여 사이클이 생기면 교착상태로 판단합니다.
- **회복(Recovery)**: 교착상태가 감지되면 관련 트랜잭션 중 하나를 강제로 철회(Rollback)하여 나머지가 진행될 수 있게 합니다.

:::warning 시험 함정 주의
**비선점(Non-preemption)**은 교착상태 **예방 전략의 한 가지 방법**(이미 획득된 자원을 강제로 빼앗지 않는 조건)이지, 교착상태의 독립적인 해결책이 아닙니다. "교착상태 해결책"을 묻는 문제에서 비선점은 **틀린 선택지**입니다.
:::

### ♻️ 트랜잭션 상태 변화

트랜잭션은 다음과 같은 상태를 거칩니다:

```
[활동(Active)]
    ↓ 정상 실행
[부분 완료(Partially Committed)]
    ↓ 최종 저장 성공
[완료(Committed)]

[활동(Active)]
    ↓ 오류 발생
[실패(Failed)]
    ↓ 롤백
[철회(Aborted)]
```

**절대 존재하지 않는 경로**: `실패(Failed) → 부분완료(Partially Committed)`. 한번 실패한 트랜잭션은 부분완료로 돌아갈 수 없습니다. 반드시 **철회(Aborted)** 로만 갑니다.

---

## 💾 4. 회복 기법 — 체크포인트와 ARIES

### 🔎 체크포인트(Checkpoint) 기법

데이터베이스가 갑자기 고장났을 때, 로그(변경 기록)를 처음부터 전부 다시 검토하면 시간이 너무 오래 걸립니다. **체크포인트**는 "여기까지는 확실히 저장이 완료됐다"는 표시를 주기적으로 남겨두는 기법입니다. 마치 비디오 게임의 **세이브 포인트**와 같습니다. 중간 세이브 포인트부터 다시 시작하면 되니 처음부터 할 필요가 없죠.

체크포인트 발생 시점을 기준으로 각 트랜잭션에 대해 어떤 처리를 해야 하는지 정리하면 다음과 같습니다:

| 트랜잭션 상태 | 처리 방법 | 이유 |
|-------------|---------|------|
| 체크포인트 **이전에 완료(Commit)** | **아무것도 하지 않음** | 이미 안전하게 저장 완료됨 |
| 체크포인트 **이후에 완료(Commit)** | **REDO (재실행)** | 저장 안 됐을 수 있으므로 다시 적용 |
| **완료되지 않은** 트랜잭션 | **UNDO (취소)** | 미완성 작업은 원상복구 |

**체크포인트 이전에 이미 Commit된 트랜잭션**은 체크포인트 당시 이미 디스크에 안전하게 기록된 상태이므로 **아무런 작업도 수행하지 않습니다.** 이것이 시험에서 가장 자주 묻는 포인트입니다.

### 🔬 ARIES 기법

ARIES(Algorithms for Recovery and Isolation Exploiting Semantics)는 현대 데이터베이스 시스템에서 사용되는 고급 회복 기법입니다. 가장 중요한 특징은 **UNDO를 수행할 때도 CLR(Compensation Log Record, 보상 로그 레코드)을 남긴다**는 것입니다.

:::warning ARIES에서 자주 틀리는 내용
"ARIES는 REDO 시 로깅하고, UNDO 시에는 로깅하지 않는다"는 설명은 **틀렸습니다**. ARIES는 UNDO 작업을 할 때에도 반드시 **CLR(보상 로그)**를 기록합니다. 이는 UNDO 중에 다시 장애가 발생하더라도 안전하게 회복할 수 있도록 하기 위함입니다.
:::

ARIES의 주요 특징:
- **Commit된 트랜잭션**: REDO (작업 재실행)
- **Commit되지 않은 트랜잭션**: UNDO (작업 취소)
- **UNDO 시에도 CLR 로그 기록** → 이중 장애에도 안전한 회복 보장

## 🛡️ 5. 데이터베이스 보안 — 추론, DDoS, SQL 인젝션, 권한 관리

### 🔎 추론(Inference) 공격

**왜 추론이 보안 문제가 될까요?** 보안 등급이 없는 일반 사용자는 기밀 정보에 직접 접근할 수 없습니다. 그런데 공개된 정보 여러 개를 조합하면 기밀 정보를 **간접적으로 알아낼 수 있는 경우**가 있습니다. 이것이 바로 **추론(Inference)** 공격입니다.

예를 들어, "A팀의 평균 월급"과 "A팀의 팀원 수", "A팀 전체 총 월급"이 모두 공개된 정보라면, 이를 조합하여 특정 팀원의 개인 월급을 계산해낼 수 있습니다. 누구도 개인 월급을 직접 알려준 적이 없지만 수학적 계산으로 유추가 가능한 것이죠.

### 💣 DDoS (분산 서비스 거부 공격)

**DoS(Denial of Service)**는 한 대의 컴퓨터가 특정 서버에 엄청난 양의 요청을 보내 서버를 마비시키는 공격입니다. **DDoS(Distributed DoS)**는 여기서 D = Distributed(분산)이 추가된 것으로, **악성코드에 감염된 수천 대의 좀비 PC들이 동시에** 특정 서버를 공격하는 방식입니다.

마치 한 명이 가게 문을 막고 있는 것(DoS)과, 수천 명의 사람들이 동시에 몰려와 가게 문을 막아서 진짜 손님이 들어오지 못하게 하는 것(DDoS)의 차이입니다. 규모가 훨씬 크고 방어하기 어렵습니다.

### 💉 SQL 인젝션(SQL Injection)

**SQL 인젝션**은 입력 창에 SQL 코드를 삽입하여 데이터베이스를 불법으로 조작하는 공격입니다. "SQL 변형, 코드 투입, 함수 투입을 통해 DB를 공격하는 기술"이라고 표현되기도 합니다.

```sql
-- ❌ 취약한 SQL (SQL 인젝션에 당할 수 있는 예시)
-- 사용자 입력: ' OR '1'='1
-- 생성되는 쿼리:
SELECT * FROM users 
WHERE username = '' OR '1'='1'  -- 항상 참이 되어 모든 데이터 반환!
AND password = '';

-- ✅ 안전한 SQL (파라미터 바인딩 사용)
-- 입력값을 직접 SQL에 넣지 않고 바인딩 변수를 사용
SELECT * FROM users 
WHERE username = ?   -- ? 자리에 입력값이 텍스트로만 처리됨
AND password = ?;    -- SQL 코드로 해석되지 않아 안전
```

### 🔑 권한 관리 — GRANT와 REVOKE

데이터베이스에서 사용자에게 특정 작업 권한을 주거나 회수하는 명령어입니다.

```sql
-- ✅ 권한 부여 (GRANT): TO 사용
-- U1 사용자에게 STUDENT 테이블 검색(SELECT) 권한 부여
GRANT SELECT ON STUDENT TO U1;

-- ✅ 권한 회수 (REVOKE): FROM 사용
-- U1 사용자에게서 STUDENT 테이블 검색(SELECT) 권한 회수
REVOKE SELECT ON STUDENT FROM U1;

-- 핵심 암기법:
-- GRANT → 주다 → TO (에게)
-- REVOKE → 빼앗다 → FROM (로부터)
```

:::tip GRANT vs REVOKE 헷갈리지 않는 법
- **GRANT**는 선물을 **"TO(에게)"** 줍니다 → `GRANT 권한 ON 테이블 TO 사용자`
- **REVOKE**는 권한을 **"FROM(으로부터)"** 빼앗습니다 → `REVOKE 권한 ON 테이블 FROM 사용자`
:::

## 📋 6. 무결성 규정과 스키마 변환

### 🔎 무결성 규정(Integrity Rule)의 구성 요소

**무결성(Integrity)**이란 데이터가 정확하고 일관성 있게 유지되는 성질입니다. 데이터베이스에서 "학년은 반드시 1~4 사이여야 한다"처럼 데이터가 지켜야 할 규칙을 **무결성 규정**이라 합니다.

무결성 규정은 다음 네 가지 구성 요소로 이루어집니다:

| 구성 요소 | 의미 | 예시 |
|----------|------|------|
| **rule name** | 규정의 이름 | CONST-YEAR |
| **trigger condition** | 검사 **시기** | AFTER UPDATE (업데이트 후) |
| **constraints** | 만족해야 할 **조건** | CHECK (VALUE IN (1,2,3,4)) |
| **violation action** | 위반 시 **처리** | ROLLBACK |

시험에서 "검사 시기를 나타내는 것"을 묻는다면 정답은 **trigger condition**입니다.

### 🏗️ 도메인 무결성 규정

```sql
-- 도메인 무결성 규정 예시
-- DYEAR라는 도메인을 만들고, 값이 반드시 1, 2, 3, 4 중 하나여야 함을 지정
CREATE DOMAIN DYEAR SMALLINT          -- DYEAR라는 이름의 도메인 생성, 타입은 SMALLINT(작은 정수)
  CONSTRAINT CONST-YEAR               -- 제약 조건의 이름: CONST-YEAR
  CHECK (VALUE IN (1, 2, 3, 4));      -- 조건: 값이 1, 2, 3, 4 중 하나여야 함
  
-- CHECK 키워드가 제약 조건(constraints)을 기술하는 명령어
-- VALUE는 해당 도메인의 값을 가리키는 특별 키워드
```

이 SQL은 "학년(DYEAR)"이라는 도메인을 정의하면서 반드시 1, 2, 3, 4 중 하나의 값만 허용하도록 제약을 거는 **도메인 무결성 규정**의 예시입니다. `CHECK` 키워드가 제약 조건을 기술하는 역할을 합니다.

### 🔄 스키마 변환의 3원리

데이터베이스 스키마(구조)를 변환할 때는 세 가지 원리를 따라야 합니다:

1. **정보 표현의 무손실**: 변환 전후로 정보가 손실되어서는 안 됩니다.
2. **최소의 데이터 중복 허용**: 중복을 최대한 줄입니다.
3. **분리의 원칙**: 독립적인 사실들은 분리된 릴레이션으로 표현합니다.

**"통합의 원칙"은 스키마 변환 원리에 해당하지 않습니다.** 시험에서 "스키마 변환 원리와 가장 거리가 먼 것"을 묻는다면 **통합의 원칙**이 정답입니다.

## 🗃️ 7. 정규화 — 이상현상과 정규형

### 🔎 이상현상(Anomaly)이란 무엇인가?

정규화되지 않은 테이블(릴레이션)에서는 **삽입 이상**, **삭제 이상**, **갱신 이상**이 발생합니다. 이 문제의 **근본적인 원인**은 "여러 가지 서로 다른 사실(종속 관계)을 하나의 릴레이션에 억지로 몰아넣었기 때문"입니다.

예를 들어, 학생 정보와 수강 정보와 교수 정보를 한 테이블에 넣으면:
- 수강 신청 없이는 학생 정보를 추가할 수 없는 **삽입 이상**
- 수강 취소하면 학생 정보도 사라지는 **삭제 이상**
- 교수 연락처를 바꾸면 관련된 모든 행을 수정해야 하는 **갱신 이상**

이런 이상현상의 해결책은 **정규화(Normalization)**, 즉 릴레이션을 올바른 형태로 **분해**하는 것입니다.

### 📊 정규형 판별하기 — 실제 시험 문제 풀이

**문제**: R(A, B, C), 기본키: (A, B), 함수적 종속성: {A, B} → C, C → B 일 때 R은 몇 정규형인가?

```
# 정규형 판별 과정

기본키: (A, B)
함수적 종속성:
  {A, B} → C  : A와 B가 함께 있어야 C가 결정됨 (완전 함수 종속)
  C → B        : C가 B를 결정함 (이행적 종속)

# 1NF 검사: 모든 속성이 원자값? → ✅ 만족

# 2NF 검사: 기본키의 일부에 종속되는 비주요 속성이 없는가?
  (= 완전 함수 종속인가?)
  {A, B} → C : A와 B 모두 있어야 C가 결정 → 완전 함수 종속 ✅
  → 2NF 만족

# 3NF 검사: 이행적 종속이 없는가?
  C → B 존재: C가 B를 결정하는데 C는 기본키가 아님
  → 이행적 종속 발생 ❌
  → 3NF 불만족

# 결론: 제2정규형(2NF)
```

**이행적 종속(Transitive Dependency)**이란 "A → B → C" 처럼 간접적으로 종속되는 관계입니다. 이 문제에서는 `기본키(A,B) → C → B` 처럼 C를 거쳐서 B가 결정되는 이행적 종속이 존재합니다. 3NF가 되려면 이행적 종속이 없어야 하므로, 이 릴레이션은 **제2정규형**에 머무릅니다.

### ⚠️ 정규화의 함정 — 속도 문제

정규화를 하면 테이블이 여러 개로 분해됩니다. 테이블이 많아지면 데이터를 조회할 때 **JOIN(여러 테이블을 연결하는 연산)을 많이 써야** 하고, 이는 오히려 **응답 속도를 느리게** 만들 수 있습니다.

:::warning 정규화와 성능의 역설
"정규화된 릴레이션은 빠른 응답을 보인다"는 설명은 **틀렸습니다**. 정규화를 하면 무결성과 일관성은 높아지지만, 조회 성능은 오히려 저하될 수 있습니다.

성능을 위해서는 오히려 **역정규화(De-normalization)** 를 사용합니다. 역정규화는 의도적으로 중복을 허용하여 JOIN을 줄이고 조회 속도를 높이는 기법입니다.
:::

### 🚀 DB 성능 튜닝 기법 총정리

시험에 자주 나오는 성능 튜닝 기법들을 정리합니다:

| 튜닝 기법 | 효과 | 설명 |
|----------|------|------|
| **역정규화** | 성능 향상 | JOIN 감소로 조회 속도 향상 |
| **실체화된 뷰 (Materialized View)** | 성능 향상 | 쿼리 결과를 미리 저장해 빠른 응답 |
| **UNION → UNION ALL 대체** | 성능 향상 | UNION은 중복 제거 과정이 있어 느림, UNION ALL은 중복 제거 없이 빠름 |
| **불필요한 COUNT 제거** | 성능 향상 | 존재 여부 확인 시 COUNT 대신 EXISTS 사용 |

:::info 실체화된 뷰(Materialized View)
일반 뷰(View)는 조회할 때마다 쿼리를 실행하는 **가상 테이블**입니다. 반면 **실체화된 뷰**는 쿼리 결과를 실제로 저장해 두고, 다음 조회 시 저장된 결과를 그대로 반환합니다. 미리 요리를 해서 냉장고에 넣어둔 것처럼, 조회할 때마다 다시 요리할 필요가 없어 빠릅니다. **"실체화된 뷰가 성능을 저하시킨다"는 설명은 틀렸습니다.**
:::

## ✅ 핵심 정리

- **버퍼링(Buffering)**: write 연산 시 메인 메모리에 모아두었다가 한꺼번에 저장하여 I/O 횟수를 줄이는 DB 성능 향상 기법
- **질의 처리 4단계**: 파싱 → 내부 표현 변환 → 후보 프로시저 선택 → 경험적 최적화 규칙 적용
- **경험적 최적화**: SELECT 연산은 가능한 가장 아래(하위) 노드에 배치하여 데이터를 먼저 줄여야 함
- **지지도**: 동시 구매 수 ÷ 전체 트랜잭션 수, **신뢰도**: 동시 구매 수 ÷ 조건(A) 트랜잭션 수
- **공유락(S)**: 읽기 전용, 여러 트랜잭션 동시 보유 가능 / **배타락(X)**: 읽기+쓰기, 혼자만 보유 가능
- **2단계 로킹**: 확장 단계(lock만) → 수축 단계(unlock만), unlock 후 재lock 불가
- **교착상태 해결책**: 예방(Prevention), 탐지(Detection), 회복(Recovery) 3가지 — 비선점은 독립 해결책이 아님
- **트랜잭션 상태**: 활동 → 부분완료 → 완료 / 활동 → 실패 → 철회 (실패에서 부분완료로 가는 경로 없음)
- **체크포인트 이전 Commit 트랜잭션**: 아무 처리도 하지 않음 (이미 안전하게 저장됨)
- **ARIES**: UNDO 시에도 CLR(보상 로그 레코드)을 반드시 기록함
- **추론(Inference)**: 공개된 정보 조합으로 기밀 정보를 간접적으로 알아내는 보안 위협
- **DDoS**: 수천 대의 좀비 PC가 동시에 서버를 공격하는 분산 서비스 거부 공격
- **SQL 인젝션**: 입력창에 SQL 코드 삽입으로 DB를 불법 조작하는 공격
- **GRANT**: 권한 부여 (TO 사용) / **REVOKE**: 권한 회수 (FROM 사용)
- **무결성 규정의 trigger condition**: 검사 **시기**를 나타내는 구성 요소
- **CHECK**: 제약 조건(constraints)을 기술하는 SQL 명령어
- **스키마 변환 3원리**: 무손실, 최소 중복, 분리의 원칙 (통합의 원칙은 포함되지 않음)
- **이상현상 원인**: 여러 가지 사실(종속 관계)을 하나의 릴레이션으로 표현했기 때문
- **2NF**: 완전 함수 종속 만족 / **3NF**: 이행적 종속 없어야 함
- **정규화**: 이상현상 제거에 유리하지만 JOIN 증가로 응답 속도는 느려질 수 있음
- **역정규화(De-normalization)**: 의도적 중복 허용으로 조회 성능을 높이는 기법
- **실체화된 뷰(Materialized View)**: 쿼리 결과 사전 저장으로 성능을 **향상**시킴 (저하가 아님)
