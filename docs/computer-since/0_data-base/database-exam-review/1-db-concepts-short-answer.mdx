### 1번 🗄️ 버퍼링(Buffering) — 모아서 한 번에!

**문제:** write 연산 시마다 저장장치에 저장하면 성능이 저하된다. I/O 시간을 줄이기 위해 레코드를 메인 메모리의 버퍼 블록에 모아 한꺼번에 저장하여 DB 성능을 향상시키는 기법은?

> **정답: 버퍼링(Buffering)**

#### 왜 버퍼링이 필요한가?

컴퓨터에서 데이터를 처리하는 속도와 저장장치(하드디스크, SSD 등)에 저장하는 속도는 엄청난 차이가 납니다. CPU와 메인 메모리(RAM)는 나노초(ns, 10억분의 1초) 단위로 동작하지만, 디스크에 데이터를 쓰는 작업(I/O, Input/Output)은 밀리초(ms, 1000분의 1초) 단위로 훨씬 느립니다.

만약 데이터를 변경할 때마다 즉시 디스크에 저장한다면, 매번 느린 디스크 I/O가 발생해서 전체 시스템 성능이 크게 떨어집니다. 마치 편의점 아르바이트생이 손님이 물건을 하나 집을 때마다 창고에 들어가서 재고를 채우는 것과 같습니다 — 매우 비효율적이죠.

#### 버퍼링의 핵심 원리

**버퍼링(Buffering)** 은 변경된 데이터를 즉시 디스크에 쓰지 않고, 일단 **메인 메모리(RAM)에 있는 버퍼 블록(buffer block, 임시 저장 공간)** 에 모아두었다가 충분히 쌓이거나 특정 시점이 되면 한꺼번에 디스크에 저장하는 방식입니다.

배달 앱의 비유로 이해해 봅시다. 배달 기사가 주문 한 건이 들어올 때마다 한 건씩 배달하러 가면 비효율적입니다. 대신 같은 방향의 배달 여러 건을 묶어서 한 번에 나가면 시간과 연료를 절약할 수 있습니다. 버퍼링이 바로 이 원리입니다.

```
메인 메모리(빠름)          디스크(느림)
┌──────────────┐         ┌──────────────┐
│  버퍼 블록   │  ──→    │  저장장치    │
│  [데이터1]   │  한번에  │  (실제저장)  │
│  [데이터2]   │  저장    │              │
│  [데이터3]   │         │              │
└──────────────┘         └──────────────┘
```

:::tip 시험 포인트
버퍼링은 **"I/O 횟수를 줄여서 성능을 향상"** 시키는 기법입니다. "메인 메모리의 버퍼 블록에 모아서 한꺼번에 저장"이라는 표현이 핵심 키워드입니다.
:::

:::info 배경 지식
버퍼(Buffer)라는 단어는 원래 "완충재"를 뜻합니다. 빠른 CPU와 느린 디스크 사이의 속도 차이를 완충해주는 역할을 한다고 해서 버퍼라고 부릅니다. 유튜브 동영상을 볼 때 "버퍼링 중..."이라고 나오는 것도 같은 원리입니다 — 데이터를 미리 메모리에 모아두는 중이라는 뜻입니다.
:::

### 2번 🔍 구문 분석(파싱, Parsing) — SQL 문법 검사관

**문제:** 고급 질의어 처리 단계 중 질의문을 분석하고 문법적으로 이상이 있는지 체크하는 것은?

> **정답: 구문 분석(파싱, Parsing)**

#### 왜 파싱이 필요한가?

사용자가 데이터베이스에 SQL 쿼리를 입력하면, DB 시스템은 그 쿼리를 바로 실행할 수 없습니다. 마치 외국어로 쓴 편지를 받았을 때 번역하고 내용을 이해한 다음에야 답장을 쓸 수 있는 것처럼, DB도 SQL 문장을 여러 단계를 거쳐 처리합니다.

그 첫 번째 단계가 바로 **파싱(Parsing, 구문 분석)** 입니다. 사람이 쓴 SQL 문장이 문법적으로 올바른지 먼저 확인하는 것입니다. 마치 학교에서 선생님이 학생의 작문을 받아서 맞춤법과 문법부터 검사하는 것과 같습니다.

#### 고급 질의어 처리 4단계

SQL 쿼리가 실행되기까지의 전체 과정은 다음 4단계를 거칩니다:

| 단계 | 이름 | 하는 일 |
|------|------|---------|
| ① | **파싱(Parsing)** | SQL 문법 오류 검사, 테이블/컬럼 존재 여부 확인 |
| ② | **내부 표현 변환** | SQL을 DB 내부에서 처리할 수 있는 형태(쿼리 트리)로 변환 |
| ③ | **후보 프로시저 선택** | 실행 가능한 여러 방법(실행 계획) 중 후보 선정 |
| ④ | **경험적 최적화** | 가장 효율적인 실행 계획으로 최종 결정 |

```sql
-- 사용자가 입력한 SQL (파싱 대상)
SELECT name, age
FROM student
WHERE grade = 'A';

-- 파싱 단계에서 검사하는 것들:
-- 1. SELECT, FROM, WHERE 문법이 맞는가?
-- 2. 'student' 테이블이 실제로 존재하는가?
-- 3. 'name', 'age', 'grade' 컬럼이 student 테이블에 있는가?
-- 4. 따옴표, 괄호 등이 올바르게 닫혀 있는가?
```

:::warning 혼동 주의
파싱은 **문법 검사** 단계이지, 실제로 데이터를 검색하거나 최적화하는 단계가 아닙니다. 파싱에서 오류가 발견되면 그 즉시 오류 메시지를 반환하고 이후 단계는 실행되지 않습니다.
:::

### 3번 🕵️ 추론(Inference) — 공개 정보로 비밀 캐기

**문제:** 보안 등급이 없는 일반 사용자가 보안으로 분류되지 않는 정보에 정당하게 접근하여 기밀 정보를 유추하는 것은?

> **정답: 추론(Inference)**

#### 왜 추론 공격이 위험한가?

데이터베이스 보안에서 가장 막기 어려운 공격 중 하나가 바로 **추론(Inference)** 입니다. 해킹처럼 불법적인 방법을 쓰지 않고, 자신이 합법적으로 접근할 수 있는 공개된 정보들을 조합해서 접근이 금지된 비밀 정보를 알아내는 것입니다.

마치 퍼즐 조각처럼, 각각의 조각은 아무 의미 없어 보이지만 여러 조각을 맞추면 전체 그림이 드러나는 것과 같습니다.

#### 실생활 예시로 이해하기

회사 직원 데이터베이스를 예로 들어봅시다:

- 일반 직원이 접근 가능한 정보: **팀 평균 월급**, **팀 총 인원 수**
- 접근이 금지된 정보: **개인별 월급**

만약 어떤 직원이 다음 두 가지 정보를 조합하면:
1. "개발팀 평균 월급은 500만원이다" (공개 정보)
2. "개발팀 인원은 5명이다" (공개 정보)
3. "4명의 월급이 각각 450만원이다" (공개 정보)

→ 나머지 1명의 월급 = (500만 × 5) - (450만 × 4) = 2500만 - 1800만 = **700만원** 이라고 유추할 수 있습니다!

이처럼 개인 월급이라는 기밀 정보에 직접 접근하지 않았지만, 공개된 정보들을 조합해서 알아낸 것이 바로 **추론 공격**입니다.

:::info 추론 vs 해킹의 차이
- **해킹**: 허가받지 않은 방법으로 시스템에 침입
- **추론**: 허가된 방법으로 접근한 공개 정보를 *조합*하여 비밀 정보를 알아냄

추론은 "불법 침입"이 없기 때문에 탐지하고 막기가 훨씬 어렵습니다.
:::

### 4번 💣 DDoS — 좀비 PC 부대의 총공격

**문제:** 악성코드에 감염된 좀비 PC를 통해 특정 사이트에 트래픽을 증가시켜 서비스를 못 하게 하는 해킹 방식은?

> **정답: DDoS (분산 서비스 거부 공격, Distributed Denial of Service)**

#### DoS vs DDoS — 뭐가 다른가?

먼저 **DoS(Denial of Service, 서비스 거부 공격)** 를 이해해야 합니다. DoS는 1대의 공격자 컴퓨터가 목표 서버에 엄청난 양의 요청을 보내서 서버가 정상적인 서비스를 못 하게 만드는 공격입니다. 마치 한 사람이 가게 전화기에 쉬지 않고 전화를 걸어서 다른 손님들이 전화를 못 하게 막는 것과 같습니다.

**DDoS**의 D는 **Distributed(분산된)** 를 의미합니다. 1대가 아니라 수천, 수만 대의 컴퓨터가 동시에 공격합니다. 단, 이 컴퓨터들은 사용자도 모르게 악성코드에 감염된 **좀비 PC(Botnet, 봇넷)** 들입니다.

```
[공격자 1명]
      ↓ 명령
[좀비 PC 수천 대] → → → → [목표 서버]
[좀비 PC 수천 대] → → → →  "서버 다운!"
[좀비 PC 수천 대] → → → →
```

| 구분 | DoS | DDoS |
|------|-----|------|
| 공격 주체 | 1대의 컴퓨터 | 수천~수만 대의 좀비 PC |
| 탐지 난이도 | 상대적으로 쉬움 | 매우 어려움 |
| 공격 규모 | 소규모 | 대규모 |
| 차단 방법 | IP 차단 | IP 차단만으로 불가능 |

:::danger 주의
DDoS 공격의 피해자는 서버뿐만이 아닙니다. 나도 모르게 내 컴퓨터가 악성코드에 감염되어 좀비 PC가 될 수 있습니다. 백신 프로그램을 항상 최신 상태로 유지하세요.
:::

### 5번 📊 정규형 판별 — 2NF인가, 3NF인가?

**문제:** R(A, B, C), 기본키: (A, B), 함수적 종속성: {A, B} → C, C → B 일 때 릴레이션 R은 몇 정규형인가?

> **정답: 제2정규형(2NF)**

#### 정규형을 판별하는 방법

이 문제는 차근차근 단계별로 검사해야 합니다. 정규형 판별은 **낮은 정규형부터 높은 정규형 순서로** 만족 여부를 확인합니다.

**주어진 조건 정리:**
- 릴레이션: R(A, B, C)
- 기본키(Primary Key): (A, B) — A와 B가 합쳐져야 기본키
- 함수적 종속성 1: {A, B} → C
- 함수적 종속성 2: C → B

**① 1NF(제1정규형) 검사: 원자값 여부**

1NF는 모든 속성의 값이 원자값(더 이상 쪼갤 수 없는 값)이어야 합니다. 이 문제에서는 1NF를 만족한다고 가정합니다. ✅

**② 2NF(제2정규형) 검사: 완전 함수 종속**

2NF는 기본키가 복합키(여러 속성으로 이루어진 키)일 때, 기본키의 **일부**에만 종속되는 속성(부분 함수 종속)이 없어야 합니다.

- {A, B} → C : A와 B **전체**가 C를 결정합니다. 이것은 **완전 함수 종속**입니다. ✅
- C가 A에만 종속되거나, B에만 종속되는 경우가 없으므로 부분 함수 종속이 없습니다.

따라서 **2NF를 만족**합니다. ✅

**③ 3NF(제3정규형) 검사: 이행적 종속**

3NF는 **이행적 종속(transitive dependency)** 이 없어야 합니다. 이행적 종속이란 "A → B이고 B → C이면 A → C가 성립"하는 간접적인 종속 관계입니다.

- C → B 가 존재합니다.
- B는 기본키 (A, B)의 일부입니다.
- C는 기본키가 아닌 일반 속성입니다.
- **기본키가 아닌 C가 기본키의 일부인 B를 결정**하고 있습니다.

이것은 3NF의 조건을 위반합니다. 3NF에서는 **결정자(Determinant, 다른 속성을 결정하는 속성)는 반드시 후보키여야** 합니다. 하지만 C는 후보키가 아니므로 3NF를 만족하지 않습니다. ❌

```
기본키: (A, B)
         ↓
         C  ← {A,B} → C (완전 함수 종속 → 2NF ✅)
         ↓
         B  ← C → B (이행적 종속 → 3NF ❌)

결론: 2NF는 만족, 3NF는 불만족 → 제2정규형
```

:::tip 정규형 판별 공식 암기법
- **2NF**: 기본키 **전체**에 종속 (부분 종속 없음)
- **3NF**: 기본키가 아닌 일반 속성이 다른 일반 속성을 결정하면 안 됨 (이행적 종속 없음)
- **BCNF**: 모든 결정자가 후보키여야 함
:::

### 6번 📈 연관규칙: 지지도와 신뢰도 계산

**문제:** 연관규칙 'Milk ⇒ Juice' — 총 1,000개 트랜잭션, Milk 700개, Juice 500개, 동시구매 350개. 지지도와 신뢰도는?

> **정답: ① 지지도 35%, 신뢰도 50%**

연관규칙(Association Rule)은 장바구니 분석에 많이 사용되는 데이터 마이닝 기법입니다. "우유를 사는 사람은 주스도 함께 산다"처럼 항목들 사이의 연관성을 수치로 표현합니다.

**지지도(Support)** 는 전체 거래 중에서 두 품목이 함께 구매된 비율입니다:

```
지지도 = 동시구매 횟수 / 전체 트랜잭션 수
       = 350 / 1000
       = 0.35 = 35%
```

**신뢰도(Confidence)** 는 Milk를 구매한 사람 중에서 Juice도 함께 구매한 비율입니다:

```
신뢰도 = 동시구매 횟수 / Milk 구매 횟수
       = 350 / 700
       = 0.5 = 50%
```

| 지표 | 공식 | 계산 | 결과 |
|------|------|------|------|
| **지지도** | 동시구매 / 전체 | 350 / 1000 | **35%** |
| **신뢰도** | 동시구매 / Milk구매 | 350 / 700 | **50%** |

:::note 해석
신뢰도 50%의 의미: "우유를 산 사람 중 절반이 주스도 함께 샀다"는 뜻입니다. 이 수치가 높을수록 연관성이 강하다고 볼 수 있습니다.
:::

### 7번 🔒 교착상태(Deadlock) 해결 전략

**문제:** 트랜잭션들의 교착상태 문제에 대한 해결책으로 틀린 것은?

> **정답: ② 비선점**

**교착상태(Deadlock, 데드락)** 란 두 개 이상의 트랜잭션이 서로 상대방이 가진 자원을 기다리며 영원히 진행되지 못하는 상황입니다. 마치 좁은 골목에서 두 자동차가 마주쳤을 때 서로 비켜주기를 기다리며 아무도 못 가는 것과 같습니다.

교착상태의 해결 전략은 **3가지**입니다:

1. **예방(Prevention)**: 교착상태가 애초에 발생하지 않도록 조건을 막음
2. **탐지(Detection)**: 교착상태 발생을 감지하고 확인함
3. **회복(Recovery)**: 교착상태 발생 후 정상 상태로 되돌림

**비선점(Non-preemption)** 은 "이미 할당된 자원을 강제로 빼앗지 않는다"는 원칙으로, 교착상태 **예방 전략의 하위 개념**입니다. 독립적인 해결책이 아니라 예방 전략의 일부이므로, 독립적인 해결책으로 나열된 보기에서는 틀린 것이 됩니다.

### 8번 🔑 로킹 단위와 성능의 관계

**문제:** 로킹(locking)을 사용하여 동시성 제어를 하는 시스템에 대한 설명으로 틀린 것은?

> **정답: ③ 로킹 단위가 크면 데이터베이스의 일관성이 증가한다**

**로킹(Locking)** 은 여러 트랜잭션이 동시에 같은 데이터에 접근할 때 충돌을 방지하기 위해 데이터에 자물쇠를 거는 메커니즘입니다. 로킹 **단위(Granularity)** 는 자물쇠를 거는 범위를 말하며, 테이블 전체, 페이지, 레코드, 필드 등 다양한 크기가 있습니다.

| 로킹 단위 | 동시 수행 | 오버헤드(관리 부담) | 충돌 가능성 | 일관성 |
|-----------|-----------|---------------------|-------------|--------|
| **크면** | ↓ 감소 | ↓ 감소 | ↑ 증가 | ↑ 유지/증가 |
| **작으면** | ↑ 증가 | ↑ 증가 | ↓ 감소 | ↓ 감소 |

보기 ③ "로킹 단위가 크면 일관성이 증가한다"는 **실제로 맞는 말**처럼 보일 수 있지만, 시험 출제 의도는 이 표현이 잘못된 보기임을 묻는 것입니다. 로킹 단위가 크면 동시 수행이 줄어들고 충돌 가능성이 높아지며, 일관성이 무조건 "증가"한다고 단순화하기 어렵습니다. 시험에서는 이 보기를 틀린 것으로 처리합니다.

### 9번 🤝 로크의 양립 가능성

**문제:** 서로 양립할 수 있는 로크로 짝지어진 것은?

> **정답: ③ shared-shared**

로크(Lock)의 종류는 크게 두 가지입니다:
- **S락(Shared Lock, 공유 락)**: 읽기 전용 락. 여러 트랜잭션이 동시에 읽을 수 있음
- **X락(Exclusive Lock, 배타 락)**: 쓰기 락. 하나의 트랜잭션만 사용 가능

| | **S락 요청** | **X락 요청** |
|---|---|---|
| **S락 보유 중** | ✅ 양립 가능 | ❌ 불가 |
| **X락 보유 중** | ❌ 불가 | ❌ 불가 |

S락끼리는 "나는 읽기만 할게, 너도 읽어도 돼"라고 서로 허용하기 때문에 양립이 가능합니다. 하지만 X락이 포함되면 "나 혼자 쓸 거야, 아무도 건드리지 마"이기 때문에 다른 어떤 락과도 함께 사용할 수 없습니다.

### 10번 🔄 2단계 로킹 규약(2PL)

**문제:** 로킹(locking)에 대한 설명으로 옳지 않은 것은?

> **정답: ④ 2단계 로킹 규약에서는 한 트랜잭션이 unlock 후 다른 데이터 아이템을 lock할 수 있다**

**2단계 로킹 규약(Two-Phase Locking, 2PL)** 은 트랜잭션의 락 획득과 해제를 두 단계로 엄격히 나눈 규칙입니다:

- **1단계 - 확장 단계(Growing Phase)**: lock만 할 수 있고, unlock은 할 수 없음
- **2단계 - 수축 단계(Shrinking Phase)**: unlock만 할 수 있고, 새로운 lock은 절대 불가

```
시간 →
[lock A] [lock B] [lock C] | [unlock A] [unlock B] [unlock C]
←── 확장 단계 ──────────→ | ←───── 수축 단계 ──────────────→
                           ↑
                     이 지점 이후로는 lock 절대 금지!
```

보기 ④는 "unlock 후에도 다른 아이템을 lock할 수 있다"고 하는데, 이것은 2PL 규약의 핵심을 위반합니다. **한번 unlock을 시작하면 다시는 lock을 걸 수 없습니다.**

:::tip 2PL을 사용하는 이유
2PL을 모든 트랜잭션이 준수하면, 그 스케줄은 반드시 **직렬 가능(serializable)** 합니다. 즉, 동시에 실행해도 순서대로 하나씩 실행한 것과 동일한 결과를 보장합니다.
:::

### 11번 ♻️ 체크포인트 기법

**문제:** 체크포인트 기법에서 아무런 작업도 수행하지 않는 경우는?

> **정답: ① 체크포인트 이전에 Commit된 트랜잭션**

**체크포인트(Checkpoint)** 는 데이터베이스가 주기적으로 현재의 안전한 상태를 디스크에 기록하는 지점입니다. 시스템이 장애로 다운됐다가 재시작될 때, 체크포인트를 기준으로 회복 작업의 범위를 결정합니다.

| 상황 | 처리 방법 | 이유 |
|------|-----------|------|
| 체크포인트 **이전**에 Commit ✅ | **아무것도 안 함** | 이미 안전하게 디스크에 저장됨 |
| 체크포인트 **이후**에 Commit ✅ | **REDO(재실행)** | 커밋은 됐지만 디스크에 아직 반영 안 됐을 수 있음 |
| Commit이 **안 된** 트랜잭션 | **UNDO(취소)** | 미완성 작업이므로 원래 상태로 되돌려야 함 |

체크포인트 이전에 이미 커밋된 트랜잭션은 체크포인트 시점에 이미 안전하게 저장되었기 때문에 장애가 발생해도 걱정할 것이 없습니다. 따라서 **아무런 회복 작업도 필요하지 않습니다.**

### 12번 📜 ARIES 기법

**문제:** ARIES 기법에 대한 설명으로 틀린 것은?

> **정답: ② REDO 시 로깅O, UNDO 시 로깅X**

**ARIES(Algorithm for Recovery and Isolation Exploiting Semantics)** 는 데이터베이스 회복을 위한 대표적인 알고리즘입니다. ARIES의 핵심 특징 중 하나는 **UNDO 작업을 수행할 때도 반드시 로그를 남긴다**는 것입니다.

UNDO 시 남기는 로그를 **CLR(Compensation Log Record, 보상 로그 레코드)** 라고 합니다. CLR을 남기는 이유는 UNDO 작업 도중 또 다른 장애가 발생했을 때, 이미 수행한 UNDO 작업을 중복으로 다시 하지 않도록 하기 위해서입니다.

보기 ② "UNDO 시 로깅X"는 틀린 설명입니다. ARIES는 모든 작업에 로그를 남기는 **"Write-Ahead Logging(WAL)"** 원칙을 철저히 따릅니다.

:::info ARIES의 3단계 회복 과정
1. **분석(Analysis)**: 로그를 분석하여 REDO와 UNDO가 필요한 트랜잭션 목록 작성
2. **REDO(재실행)**: Commit된 트랜잭션의 변경사항을 다시 적용
3. **UNDO(취소)**: Commit되지 않은 트랜잭션의 변경사항을 원상복구
:::

### 13번 🔄 트랜잭션 상태 전이도

**문제:** 트랜잭션 상태변화로 틀린 것은?

> **정답: ③ 활동→실패→부분완료**

트랜잭션은 생명주기 동안 다음과 같은 상태를 거칩니다:

```
[활동(Active)]
      ↓            ↓
[부분완료]      [실패(Failed)]
      ↓            ↓
[완료(Committed)] [철회(Aborted)]
```

- **활동(Active)**: 트랜잭션이 실행 중인 상태
- **부분완료(Partially Committed)**: 마지막 연산까지 실행은 완료했지만, 아직 디스크에 최종 저장(commit)이 안 된 상태
- **완료(Committed)**: 성공적으로 완료되어 디스크에 영구 저장된 상태
- **실패(Failed)**: 오류가 발생하여 더 이상 진행할 수 없는 상태
- **철회(Aborted)**: 실패 후 변경사항이 모두 취소(UNDO)되어 원래 상태로 돌아간 상태

**실패 → 부분완료** 경로는 존재하지 않습니다. 한번 실패 상태가 되면 철회(취소)로만 갈 수 있습니다.

### 14번 ⚡ DB 성능 튜닝 기법

**문제:** DB 성능 향상을 위한 튜닝 기법으로 틀린 것은?

> **정답: ② 실체화된 뷰는 성능을 저하시키므로 자제**

**실체화된 뷰(Materialized View)** 는 일반 뷰(View)와 달리 쿼리 결과를 실제로 미리 계산하여 디스크에 저장해두는 특별한 뷰입니다. 마치 자주 쓰는 보고서를 미리 만들어두고 필요할 때 꺼내 보는 것과 같습니다.

일반 뷰는 쿼리할 때마다 실시간으로 계산하지만, 실체화된 뷰는 이미 계산된 결과를 저장해두기 때문에 **복잡한 집계 쿼리의 응답 속도를 대폭 향상**시킬 수 있습니다.

따라서 "성능을 저하시키므로 자제"는 완전히 틀린 설명입니다.

| 튜닝 기법 | 설명 | 효과 |
|-----------|------|------|
| 역정규화(De-normalization) | 분해된 테이블을 다시 합침 | JOIN 감소로 조회 속도 향상 |
| 실체화된 뷰 | 쿼리 결과를 미리 저장 | 복잡한 집계 쿼리 속도 향상 |
| UNION → UNION ALL | 중복 제거 과정 생략 | 불필요한 정렬/비교 작업 감소 |
| 불필요한 COUNT 제거 | 필요 없는 집계 제거 | 불필요한 연산 감소 |

### 16번 🌿 경험적 최적화 규칙

**문제:** 초기 질의 트리를 최적화된 트리로 변환하기 위한 경험적 최적화 규칙으로 틀린 것은?

> **정답: ② 가장 제한적인 SELECT는 상위 노드로**

경험적 최적화(Heuristic Optimization)는 쿼리 트리를 변환할 때 일반적으로 효과가 좋다고 알려진 규칙들을 적용하는 과정입니다. 마치 요리할 때 "센 불에서 먼저 볶다가 중불로 줄인다"는 경험적 요리 순서를 따르는 것과 비슷합니다.

핵심 규칙들:

1. **SELECT 연산을 가능한 한 아래(리프 노드 방향)로** 내린다 → 데이터를 빨리 줄여서 이후 처리 부담 감소
2. **가장 제한적인(결과 행 수가 가장 적은) SELECT를 하위 노드에** 배치 → 가장 먼저 실행되어 데이터 양을 최대로 줄임
3. **카티션 프로덕트(Cartesian Product, 두 테이블의 모든 조합)** 뒤에 SELECT가 오면 JOIN으로 변환

보기 ②는 "상위 노드로"라고 했는데, 올바른 규칙은 "**하위 노드(아래)**에 배치"입니다. 제한적인 SELECT를 위쪽에 배치하면 먼저 실행되지 않아서 최적화 효과가 없습니다.

### 17번 🛡️ 권한 회수(REVOKE)

**문제:** 사용자 U1에게 STUDENT 테이블 검색 권한을 회수하는 명령어는?

> **정답: ③ REVOKE SELECT ON STUDENT FROM U1;**

SQL의 권한 관리 명령어는 두 가지입니다:

```sql
-- ✅ 권한 부여 (TO 사용)
GRANT SELECT ON STUDENT TO U1;
-- "U1에게 STUDENT 테이블 SELECT 권한을 준다"

-- ✅ 권한 회수 (FROM 사용)
REVOKE SELECT ON STUDENT FROM U1;
-- "U1으로부터 STUDENT 테이블 SELECT 권한을 회수한다"
```

```sql
-- ❌ 잘못된 예시 (FROM/TO 혼용)
REVOKE SELECT ON STUDENT TO U1;   -- TO는 GRANT에 사용
GRANT SELECT ON STUDENT FROM U1;  -- FROM은 REVOKE에 사용
```

| 명령어 | 전치사 | 방향 |
|--------|--------|------|
| **GRANT** | **TO** | 시스템 → 사용자 (권한 부여) |
| **REVOKE** | **FROM** | 사용자 → 시스템 (권한 회수) |

:::tip 암기법
- **GRANT TO**: "TO 너에게 줄게" (TO = ~에게)
- **REVOKE FROM**: "FROM 너한테서 빼앗아" (FROM = ~로부터)
:::

### 18번 💉 SQL 인젝션(SQL Injection)

**문제:** SQL 변형, 코드 투입, 함수 투입을 통해 DB를 공격하는 해킹 기술은?

> **정답: ④ SQL 인젝션(SQL Injection)**

**SQL 인젝션** 은 사용자 입력창에 악의적인 SQL 코드를 삽입(Inject)하여 데이터베이스를 조작하는 공격 방법입니다. 웹 애플리케이션이 사용자 입력값을 제대로 검증하지 않고 그대로 SQL 쿼리에 포함시킬 때 발생합니다.

```sql
-- ✅ 정상적인 로그인 쿼리
SELECT * FROM users 
WHERE username = '입력한_아이디' AND password = '입력한_비밀번호';

-- ❌ SQL 인젝션 공격 예시
-- 사용자가 아이디 입력창에 다음을 입력: admin' --
-- 그러면 쿼리가 이렇게 변함:
SELECT * FROM users 
WHERE username = 'admin' --' AND password = '아무거나';
-- '--' 이후는 주석 처리됨 → 비밀번호 검사 무력화!
-- 'admin' 계정으로 비밀번호 없이 로그인 성공
```

:::danger SQL 인젝션 방어법
SQL 인젝션을 막으려면 **Prepared Statement(준비된 문장)** 또는 **파라미터 바인딩** 방식을 사용해야 합니다. 사용자 입력값을 직접 SQL 문자열에 이어붙이는 방식은 절대 사용하지 마세요.
:::

### 19번 📋 무결성 규정의 구성 요소

**문제:** 무결성 규정 중 검사 시기를 나타내는 것은?

> **정답: ② trigger condition**

**무결성 규정(Integrity Constraint)** 은 데이터베이스에 저장된 데이터가 항상 올바른 상태를 유지하도록 강제하는 규칙입니다. 무결성 규정은 4가지 구성 요소로 이루어집니다:

| 구성 요소 | 역할 | 예시 |
|-----------|------|------|
| **rule name** | 규정의 이름 | CONST-AGE |
| **trigger condition** | 검사할 시기 (언제 검사?) | AFTER INSERT, BEFORE UPDATE |
| **constraints** | 만족해야 할 조건 (무엇을 검사?) | CHECK (age >= 0 AND age <= 150) |
| **violation action** | 위반 시 처리 방법 (어떻게 할?) | REJECT, NOTIFY |

**trigger condition** 은 "이 무결성 검사를 언제 수행할 것인가"를 정의합니다. 예를 들어 "INSERT 후에 검사", "UPDATE 전에 검사" 등의 시기를 지정합니다.

### 20번 🏷️ 도메인 무결성 규정

**문제:** `CREATE DOMAIN DYEAR SMALLINT CONSTRAINT CONST-YEAR CHECK (VALUE IN (1,2,3,4));` 는 어떤 무결성 규정인가?

> **정답: ③ 도메인 무결성 규정**

```sql
CREATE DOMAIN DYEAR        -- 'DYEAR'라는 이름의 도메인 생성
    SMALLINT               -- 기본 타입은 SMALLINT(작은 정수)
    CONSTRAINT CONST-YEAR  -- 제약 조건 이름: CONST-YEAR
    CHECK (VALUE IN (1,2,3,4));  -- 값은 반드시 1, 2, 3, 4 중 하나여야 함
```

**도메인 무결성(Domain Integrity)** 은 특정 속성이 가질 수 있는 값의 범위나 종류를 제한하는 무결성입니다. 위 예시는 학년(DYEAR) 값이 반드시 1, 2, 3, 4 중 하나여야 한다고 정의한 것입니다. 5학년이나 0학년 같은 유효하지 않은 값이 들어오는 것을 막습니다.

### 21번 ✅ CHECK 제약 조건

**문제:** 무결성 규정에서 제약 조건(constraints)을 기술하는 명령어는?

> **정답: ③ CHECK**

```sql
-- ✅ CHECK를 사용한 제약 조건 예시
CREATE TABLE STUDENT (
    student_id INT PRIMARY KEY,
    grade CHAR(1) CHECK (grade IN ('A', 'B', 'C', 'D', 'F')),  -- 성적은 A~F만 허용
    age INT CHECK (age >= 0 AND age <= 150),  -- 나이는 0~150 사이만 허용
    gpa DECIMAL(3,2) CHECK (gpa >= 0.0 AND gpa <= 4.5)  -- 학점은 0.0~4.5 사이만
);
```

**CHECK** 키워드 다음에 괄호 안에 반드시 참이어야 하는 조건식을 작성합니다. 데이터를 삽입하거나 수정할 때 이 조건을 만족하지 않으면 작업이 거부됩니다.

### 22번 🔀 스키마 변환의 원리

**문제:** 스키마 변환의 원리와 가장 거리가 먼 것은?

> **정답: ④ 통합의 원칙**

**스키마 변환(Schema Transformation)** 은 ER 다이어그램이나 개념적 스키마를 관계형 데이터베이스의 테이블 구조로 변환하는 과정입니다. 이 변환 과정에서 지켜야 할 **3가지 원리**가 있습니다:

1. **정보 표현의 무손실(Lossless Representation)**: 변환 과정에서 어떤 정보도 잃어버리면 안 됨
2. **최소의 데이터 중복(Minimum Redundancy)**: 같은 데이터가 여러 곳에 중복 저장되지 않도록
3. **분리의 원칙(Principle of Separation)**: 서로 독립적인 관계는 별도의 테이블로 분리

**통합의 원칙**은 이 3가지 원리에 포함되지 않으므로 정답이 됩니다.

### 23번 ⚠️ 이상현상의 근본 원인

**문제:** 정규화되지 못한 릴레이션 조작 시 이상현상의 근본적인 원인은?

> **정답: ④ 여러 가지 사실을 하나의 릴레이션으로 표현**

데이터베이스에서 **이상현상(Anomaly)** 이란 정규화되지 않은 테이블에서 데이터를 삽입, 삭제, 수정할 때 발생하는 비정상적인 현상입니다:

- **삽입 이상(Insertion Anomaly)**: 특정 데이터를 삽입하려면 원하지 않는 다른 데이터도 함께 넣어야 하는 상황
- **삭제 이상(Deletion Anomaly)**: 특정 데이터를 삭제하면 의도치 않게 다른 필요한 데이터도 함께 삭제되는 상황
- **갱신 이상(Update Anomaly)**: 중복된 데이터 중 일부만 수정되어 데이터 불일치가 발생하는 상황

이 모든 이상현상의 근본 원인은 **서로 다른 종류의 사실(독립적인 개체나 관계)들을 하나의 테이블에 억지로 합쳐놓았기 때문**입니다.

예를 들어 "학생 정보"와 "수강 정보"를 하나의 테이블에 합치면, 학생 한 명이 여러 과목을 수강할 때 학생 정보가 여러 번 중복되고, 수강 취소 시 학생 정보가 함께 사라질 수 있습니다.

### 24번 🔧 이상현상의 해결책: 정규화

**문제:** 데이터 종속 관계를 무리하게 하나의 릴레이션으로 표현했을 때 이상현상 해결 방법은?

> **정답: ③ 정규화(Normalization)**

**정규화(Normalization)** 는 이상현상이 발생하는 릴레이션을 올바른 형태로 분해하여 이상현상을 제거하는 과정입니다. 마치 복잡하게 얽힌 물건들을 용도에 맞게 서랍에 나눠 정리하는 것과 같습니다.

정규화를 통해:
- 각 테이블은 하나의 주제(사실)만 담게 됩니다
- 데이터 중복이 줄어들어 저장 공간이 절약됩니다
- 삽입, 삭제, 갱신 이상현상이 해소됩니다

### 25번 🐢 정규화의 단점: 성능 저하 가능성

**문제:** 정규화에 대한 설명 중 틀린 것은?

> **정답: ④ 정규화된 릴레이션은 빠른 응답을 보인다**

정규화는 분명히 많은 장점이 있지만, **한 가지 중요한 단점**이 있습니다. 테이블이 여러 개로 분해되면, 데이터를 조회할 때 **JOIN(조인, 여러 테이블을 연결하는 연산)** 이 많아집니다. JOIN은 비용이 많이 드는 연산이기 때문에 정규화가 많이 될수록 오히려 쿼리 응답 속도가 느려질 수 있습니다.

이 때문에 실제 서비스에서는 성능을 위해 **역정규화(De-normalization)** 를 일부 적용하기도 합니다. 역정규화는 의도적으로 중복을 허용하여 JOIN 없이도 필요한 데이터를 한 테이블에서 바로 가져올 수 있게 합니다.

| | **정규화(Normalization)** | **역정규화(De-normalization)** |
|---|---|---|
| **목적** | 이상현상 제거, 중복 최소화 | 조회 성능 향상 |
| **테이블 수** | 많아짐 | 적어짐 |
| **JOIN** | 많이 필요 | 적게 필요 |
| **쓰기 성능** | 향상 | 저하 가능 |
| **읽기 성능** | 저하 가능 | 향상 |

:::tip 실무 팁
실무에서는 정규화와 역정규화 사이의 균형을 찾는 것이 중요합니다. OLTP(실시간 거래 처리) 시스템은 정규화를, OLAP(대용량 분석) 시스템은 역정규화를 더 많이 활용합니다.
:::

## ✅ 핵심 정리

이 문서에서 다룬 핵심 개념들을 한눈에 정리합니다.

- **버퍼링(Buffering)**: 데이터를 메인 메모리에 모아두었다가 한꺼번에 디스크에 저장하여 I/O 횟수를 줄이는 성능 향상 기법
- **파싱(Parsing)**: 고급 질의어 처리 4단계 중 첫 번째 단계로, SQL 문장의 문법적 오류를 검사하는 과정
- **추론(Inference)**: 공개된 정보를 조합하여 비밀 정보를 알아내는 보안 위협으로, 직접 침입 없이 이루어짐
- **DDoS**: 악성코드에 감염된 수천 대의 좀비 PC가 동시에 특정 서버를 공격하여 서비스를 마비시키는 분산 서비스 거부 공격
- **2NF(제2정규형)**: 기본키 전체에 완전 함수 종속하며 부분 함수 종속이 없는 정규형
- **3NF(제3정규형)**: 2NF를 만족하면서 이행적 종속(기본키가 아닌 속성이 다른 속성을 결정)이 없는 정규형
- **지지도**: 전체 거래 중 두 품목이 함께 구매된 비율 (동시구매/전체)
- **신뢰도**: 선행 품목 구매자 중 후행 품목도 구매한 비율 (동시구매/선행품목)
- **교착상태 해결**: 예방(Prevention), 탐지(Detection), 회복(Recovery) 3가지가 핵심
- **2단계 로킹(2PL)**: 확장 단계(lock만)와 수축 단계(unlock만)로 나뉘며, unlock 후 재lock 절대 불가
- **체크포인트 이전 Commit**: 이미 안전하게 저장되어 있으므로 회복 시 아무 작업도 필요 없음
- **ARIES**: UNDO 시에도 CLR(보상 로그)을 반드시 남기는 회복 알고리즘
- **실체화된 뷰(Materialized View)**: 쿼리 결과를 미리 저장하여 성능을 향상시키는 기법 (저하가 아님!)
- **GRANT/REVOKE**: 권한 부여는 `GRANT ... TO`, 권한 회수는 `REVOKE ... FROM`
- **SQL 인젝션**: 입력창에 악의적인 SQL 코드를 삽입하여 DB를 조작하는 공격
- **경험적 최적화**: 제한적인 SELECT는 상위가 아닌 **하위 노드**에 배치해야 최적화 효과가 있음
- **정규화의 단점**: 테이블 분해로 JOIN이 많아져 오히려 조회 성능이 저하될 수 있음
- **역정규화**: 성능 향상을 위해 의도적으로 중복을 허용하는 기법으로, 정규화와 반대 방향
