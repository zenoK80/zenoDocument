# 🔍 다중행 서브쿼리와 IN·EXISTS·ALL·ANY 연산자

SQL을 처음 배울 때는 `SELECT ... FROM ... WHERE ...` 같은 단순한 형태로 데이터를 조회합니다. 그런데 실무에서는 "영업부에 속한 직원 중 평균 급여보다 많이 받는 사람을 찾아라"처럼 **여러 조건이 겹치고, 다른 테이블의 정보가 조건으로 필요한 복잡한 질문**을 자주 마주치게 됩니다. 이럴 때 등장하는 강력한 도구가 바로 **서브쿼리(Subquery, 부속 질의문)** 입니다.

이번 문서에서는 서브쿼리의 개념부터 시작해서, 결과가 여러 행(Row)으로 나오는 **다중행 서브쿼리**를 다룰 때 사용하는 `IN`, `EXISTS`, `ALL`, `ANY` 연산자를 하나씩 깊이 있게 살펴보겠습니다. 처음 SQL을 접하는 분도 충분히 이해할 수 있도록 비유와 예시를 최대한 풍부하게 담았습니다.

---

## 📌 서브쿼리(부속 질의문)란 무엇인가?

### 서브쿼리의 개념

**서브쿼리(Subquery)** 란 하나의 SQL 문장 안에 또 다른 SQL 문장이 포함된 형태를 말합니다. 쉽게 말하면 **SQL 안의 SQL**입니다.

마치 수학 문제를 풀 때 "먼저 괄호 안을 계산하고, 그 결과로 바깥 계산을 하는 것"과 같습니다. 괄호 안의 계산이 바로 서브쿼리이고, 바깥 계산이 메인 쿼리(Main Query)입니다.

구조를 도식으로 표현하면 이렇습니다:

```
SELECT 컬럼명
FROM 테이블명
WHERE 컬럼명 = (SELECT 컬럼명     ← 이 부분이 서브쿼리
                FROM 테이블명
                WHERE 조건)
↑
이 전체가 메인 쿼리
```

바깥에 있는 큰 SQL을 **메인 쿼리(Main Query)** 라고 부르고, `WHERE` 절의 조건 안에 들어있는 작은 SQL을 **서브쿼리(Subquery)** 또는 **부속 질의문**이라고 부릅니다.

:::info 서브쿼리는 왜 필요한가?
일상생활에서 비유를 들어보겠습니다. 회사에서 "박영권 씨와 같은 직급을 가진 사원들의 명단을 뽑아주세요"라는 요청을 받았다고 상상해보세요.

이 작업을 하려면 두 단계가 필요합니다:
1. 먼저 박영권 씨의 직급이 무엇인지 찾는다.
2. 그 직급과 동일한 사람들을 전부 조회한다.

서브쿼리가 없다면 이 두 단계를 별도의 SQL로 두 번 실행해야 합니다. 하지만 서브쿼리를 사용하면 이 두 단계를 **하나의 SQL로 한 번에** 처리할 수 있습니다. 바로 이것이 서브쿼리가 필요한 이유입니다.
:::

### 서브쿼리 vs 조인(JOIN)

서브쿼리와 비슷한 역할을 하는 것이 **조인(JOIN)** 입니다. 둘 다 여러 테이블의 정보를 함께 사용할 수 있습니다. 하지만 결정적인 차이점이 있습니다.

| 구분 | 서브쿼리 | 조인(JOIN) |
|------|----------|------------|
| 결과 출력 | **메인 쿼리의 테이블** 컬럼만 출력 | 연결된 **두 테이블 모두**의 컬럼 출력 가능 |
| 사용 목적 | 다른 테이블의 값을 **조건**으로만 활용 | 두 테이블의 데이터를 **함께 보여줄 때** 활용 |
| 성능 | 경우에 따라 조인보다 효율적일 수 있음 | 대용량 데이터에서 최적화가 잘 되어 있음 |
| 직관성 | 단계별 로직이 명확해 읽기 쉬움 | 복잡한 연결 관계를 한눈에 표현 가능 |

> **핵심 차이:** 서브쿼리는 "다른 테이블에서 조건만 빌려오는 것"이고, 조인은 "두 테이블을 하나로 합쳐서 보는 것"입니다.

### 서브쿼리 작성 시 기본 규칙

서브쿼리를 작성할 때 반드시 지켜야 하는 기본 규칙이 있습니다:

1. **서브쿼리는 반드시 괄호 `( )`로 감싸야 합니다.** 괄호가 없으면 SQL 엔진이 어디까지가 서브쿼리인지 구분할 수 없습니다.
2. **서브쿼리 안에는 `ORDER BY`(정렬)를 사용할 수 없습니다.** 서브쿼리의 역할은 메인 쿼리에 값을 넘겨주는 것이므로, 굳이 정렬할 필요가 없습니다.
3. **서브쿼리의 결과 행 수에 따라 사용하는 연산자가 달라집니다.** 이것이 이 문서에서 가장 중요한 핵심입니다.

## 📊 단일행 서브쿼리 vs 다중행 서브쿼리

### 왜 구분해야 하는가?

서브쿼리의 결과가 몇 행(Row)인지에 따라 사용할 수 있는 비교 연산자가 완전히 달라집니다. 이것은 마치 택배를 받을 때와 비슷합니다. 소포 하나를 받을 때는 "이 택배가 내 것이냐?"라고 단순하게 확인하면 됩니다. 하지만 여러 개의 택배가 한꺼번에 왔다면 "이 중에 내 것이 하나라도 있냐?", "이것들이 모두 내 것이냐?" 같은 방식으로 확인해야 하죠.

SQL도 마찬가지입니다. 서브쿼리 결과가 하나면 단순 비교 연산자를 쓰면 되지만, 결과가 여러 개라면 그에 맞는 **다중행 비교 연산자**를 써야 합니다.

### 단일행 서브쿼리

**단일행 서브쿼리(Single-Row Subquery)** 는 서브쿼리의 실행 결과가 **단 한 행(또는 0행)** 으로 반환될 때를 말합니다. 이때는 우리가 익숙한 일반 비교 연산자를 그대로 사용할 수 있습니다.

사용 가능한 연산자: `=`, `>`, `<`, `>=`, `<=`, `<>` (같지 않다)

```sql
-- ✅ 올바른 예시: 박영권과 같은 직급의 사원 이름과 직급 검색
SELECT emp_name, title          -- 사원 이름과 직급을 출력
FROM employee                   -- employee(직원) 테이블에서
WHERE title = (                 -- 직급이 서브쿼리 결과와 같은 사원
    SELECT title                -- 직급을 조회
    FROM employee               -- employee 테이블에서
    WHERE emp_name = '박영권'   -- 이름이 박영권인 사람의
);
```

이 예시에서 서브쿼리 `(SELECT title FROM employee WHERE emp_name = '박영권')`는 박영권이라는 사람이 단 한 명이므로 결과가 항상 1행입니다. 따라서 `=` 연산자를 사용해도 안전합니다.

### 다중행 서브쿼리

**다중행 서브쿼리(Multi-Row Subquery)** 는 서브쿼리의 실행 결과가 **여러 행**으로 반환될 때를 말합니다. 이때 `=` 같은 단일 비교 연산자를 사용하면 오류가 발생합니다.

:::danger 절대 하면 안 되는 것
서브쿼리 결과가 여러 행인데 `=` 같은 단일행 연산자를 사용하면 **오류(Error)** 가 발생합니다.

```sql
-- ❌ 잘못된 예시: 서브쿼리 결과가 여러 행인데 = 사용
SELECT emp_name
FROM employee
WHERE dno = (                      -- 오류! 서브쿼리 결과가 여러 행일 수 있음
    SELECT dno
    FROM department
    WHERE dname IN ('영업부', '개발부')  -- 영업부, 개발부 → 여러 행 반환
);
-- 오류 메시지: single-row subquery returns more than one row
```

이 경우 반드시 다중행 연산자인 `IN`을 사용해야 합니다.
:::

다중행 서브쿼리에서 사용하는 연산자들을 표로 먼저 정리한 뒤 하나씩 자세히 살펴보겠습니다.

| 연산자 | 의미 | 독립 사용 여부 |
|--------|------|--------------|
| `IN` | 목록 중 **하나라도 일치**하면 참 | 독립 사용 가능 |
| `NOT IN` | 목록 중 **하나도 일치하지 않으면** 참 | 독립 사용 가능 |
| `EXISTS` | 서브쿼리 결과가 **존재하면** 참 | 독립 사용 가능 |
| `NOT EXISTS` | 서브쿼리 결과가 **존재하지 않으면** 참 | 독립 사용 가능 |
| `ANY` / `SOME` | 하나라도 조건을 만족하면 참 | **비교 연산자와 함께 사용** |
| `ALL` | 모두 조건을 만족해야 참 | **비교 연산자와 함께 사용** |

## 🔑 다중행 연산자 핵심 원리

### IN 연산자 — "이 목록 중에 있느냐?"

**`IN` 연산자**는 서브쿼리(또는 직접 나열한 값 목록)에서 반환된 결과 중 **하나라도 일치하는 값이 있으면 참(TRUE)** 을 반환합니다.

마치 "이 명단에 네 이름이 있니?"라고 묻는 것과 같습니다. 명단에 이름이 한 번이라도 들어가 있으면 "있다(참)"는 것이죠.

**동작 원리 이해하기:**

서브쿼리 결과가 `(3, 4, 2, 6)`이라고 가정해봅시다.

- 비교값이 `3`이면? → `3`이 `(3, 4, 2, 6)` 안에 있음 → **참(TRUE)** ✅
- 비교값이 `1`이면? → `1`이 `(3, 4, 2, 6)` 안에 없음 → **거짓(FALSE)** ❌

```sql
-- ✅ 올바른 예시: 영업부 또는 개발부에 근무하는 사원의 이름 검색
SELECT emp_name                  -- 사원 이름을 출력
FROM employee                    -- employee(직원) 테이블에서
WHERE dno IN (                   -- 부서번호가 서브쿼리 결과 목록 안에 있으면
    SELECT dno                   -- 부서번호를 조회
    FROM department              -- department(부서) 테이블에서
    WHERE dname = '영업부'       -- 부서명이 영업부이거나
       OR dname = '개발부'       -- 개발부인 부서의 번호
);
```

**코드 한 줄씩 설명:**
- `SELECT emp_name` : 최종적으로 사원 이름만 출력합니다.
- `FROM employee` : 사원 테이블에서 데이터를 가져옵니다.
- `WHERE dno IN (...)` : 사원의 부서번호(dno)가 괄호 안 서브쿼리 결과 목록에 포함되어 있는지 확인합니다.
- 서브쿼리 `SELECT dno FROM department WHERE dname = '영업부' OR dname = '개발부'` : 부서 테이블에서 영업부와 개발부의 부서번호를 모두 가져옵니다. 이 결과가 여러 행이므로 `IN`을 사용하는 것이 적절합니다.

### NOT IN 연산자 — "이 목록에 없는 것을 찾아라"

**`NOT IN`** 은 `IN`의 반대입니다. 서브쿼리 결과 목록에 **하나도 포함되지 않을 때** 참이 됩니다.

비유: "이 블랙리스트에 없는 사람만 입장 가능합니다"와 같습니다.

서브쿼리 결과가 `(3, 4, 2, 6)`이라고 가정:

- 비교값이 `1`이면? → `1`이 `(3, 4, 2, 6)`에 없음 → **참(TRUE)** ✅
- 비교값이 `3`이면? → `3`이 `(3, 4, 2, 6)`에 있음 → **거짓(FALSE)** ❌

```sql
-- ✅ 올바른 예시: 영업부나 개발부에 속하지 않는 사원의 이름 검색
SELECT emp_name                  -- 사원 이름을 출력
FROM employee                    -- employee(직원) 테이블에서
WHERE dno NOT IN (               -- 부서번호가 서브쿼리 결과 목록에 없으면
    SELECT dno                   -- 부서번호를 조회
    FROM department              -- department(부서) 테이블에서
    WHERE dname = '영업부'       -- 영업부이거나
       OR dname = '개발부'       -- 개발부인 부서의 번호
);
```

:::warning NOT IN과 NULL 주의사항
`NOT IN`을 사용할 때 서브쿼리 결과에 `NULL` 값이 하나라도 포함되면, 전체 결과가 아무것도 반환되지 않을 수 있습니다. `NULL`은 "알 수 없는 값"이기 때문에 어떤 값과 비교해도 참/거짓을 판단할 수 없기 때문입니다. 실무에서 `NOT IN`을 사용할 때는 `WHERE 컬럼 IS NOT NULL` 조건을 서브쿼리 안에 추가하는 것이 안전합니다.
:::

### EXISTS 연산자 — "결과가 존재하느냐?"

**`EXISTS` 연산자**는 서브쿼리의 실행 결과가 **한 행이라도 존재하면 참**, 하나도 없으면 거짓을 반환합니다. `IN`과 달리 **특정 값을 비교하는 것이 아니라, 결과 자체가 있느냐 없느냐**만을 확인합니다.

`EXISTS`는 특히 서브쿼리에서 반환해야 할 컬럼이 여러 개일 때 유용합니다. `IN`은 단일 컬럼의 값을 목록과 비교하지만, `EXISTS`는 `SELECT *`처럼 여러 컬럼을 한꺼번에 반환하는 경우에도 사용할 수 있습니다.

비유: "냉장고에 음식이 남아 있느냐?"라고 묻는 것과 같습니다. 어떤 음식인지, 몇 개인지는 중요하지 않고 오직 "있다/없다"만 확인하는 것입니다.

```sql
-- ✅ 올바른 예시: 영업부 또는 개발부에 근무하는 사원의 이름 검색 (EXISTS 사용)
SELECT emp_name                  -- 사원 이름을 출력
FROM employee E                  -- employee 테이블을 E라는 별명으로 사용
WHERE EXISTS (                   -- 서브쿼리 결과가 존재하면
    SELECT *                     -- 모든 컬럼 (어떤 값인지보다 존재 여부가 중요)
    FROM department D            -- department 테이블을 D라는 별명으로 사용
    WHERE E.dno = D.dno          -- 직원의 부서번호와 부서 테이블의 번호가 같고
      AND (D.dname = '영업부'    -- 부서명이 영업부이거나
        OR D.dname = '개발부')   -- 개발부인 경우
);
```

**코드 한 줄씩 설명:**
- `FROM employee E` : employee 테이블에 `E`라는 별명(Alias)을 붙입니다. 서브쿼리 안에서 이 테이블을 참조할 때 `E.컬럼명`으로 사용합니다.
- `WHERE EXISTS (...)` : 괄호 안 서브쿼리가 한 행이라도 결과를 반환하면 조건을 만족합니다.
- `SELECT *` : EXISTS는 값의 내용보다 존재 여부만 확인하므로 `*`(모든 컬럼)를 사용해도 됩니다.
- `WHERE E.dno = D.dno` : 바깥(메인) 쿼리의 직원 부서번호와 안쪽(서브) 쿼리의 부서 테이블 번호가 일치하는지 확인합니다. 이 조건이 `IN`에서의 값 비교 역할을 대신합니다.

**IN vs EXISTS 비교:**

| 구분 | IN | EXISTS |
|------|-----|--------|
| 반환 컬럼 수 | 단일 컬럼만 비교 | 여러 컬럼도 가능 (`SELECT *`) |
| WHERE절 형태 | `WHERE 컬럼 IN (서브쿼리)` | `WHERE EXISTS (서브쿼리)` |
| 동작 방식 | 서브쿼리 결과 목록과 값 비교 | 서브쿼리 결과 존재 여부만 확인 |
| 속성명 위치 | WHERE 절 앞에 컬럼명 명시 | WHERE 절 앞에 컬럼명 생략 |

### ANY / SOME 연산자 — "하나라도 만족하면 참"

**`ANY`(또는 `SOME`)** 는 서브쿼리 결과 중 **하나라도 조건을 만족하면 참**을 반환합니다. 반드시 `=`, `>`, `<`, `>=`, `<=`, `<>` 같은 **비교 연산자와 함께** 사용해야 합니다.

비유: "이 선수들 중 한 명이라도 나보다 키가 크냐?"라고 묻는 것과 같습니다. 단 한 명이라도 나보다 크면 "있다(참)"입니다.

**동작 원리 이해하기:**

서브쿼리 결과(급여 목록)가 `(250만원, 300만원, 400만원)`이라고 가정합니다.

- `> ANY (250만원, 300만원, 400만원)` 에서 비교값이 `300만원`이면?
  - 250만원보다 크냐? → 참 ✅ → **하나라도 만족하므로 전체 결과 참**
- `> ANY (250만원, 300만원, 400만원)` 에서 비교값이 `400만원`이면?
  - 250만원보다 크냐? → 참 ✅ → **하나라도 만족하므로 전체 결과 참**
- `> ANY (250만원, 300만원, 400만원)` 에서 비교값이 `200만원`이면?
  - 250만원보다 크냐? → 거짓, 300만원보다 크냐? → 거짓, 400만원보다 크냐? → 거짓 → **모두 불만족이므로 거짓**

```sql
-- ✅ 올바른 예시: 개발부 직원 중 한 명이라도보다 급여가 높은 영업부 직원 조회
SELECT emp_name, salary          -- 사원 이름과 급여를 출력
FROM employee                    -- employee(직원) 테이블에서
WHERE salary > ANY (             -- 급여가 서브쿼리 결과 중 하나라도 초과하면
    SELECT salary                -- 급여를 조회
    FROM employee                -- employee 테이블에서
    WHERE dno = (                -- 부서번호가
        SELECT dno               -- (개발부 부서번호를
        FROM department          -- department 테이블에서
        WHERE dname = '개발부'   -- 개발부로 조회)
    )
);
```

:::tip ANY와 IN의 관계
`= ANY`는 사실상 `IN`과 동일한 의미입니다.
- `WHERE salary = ANY (300, 400, 500)` ← 이것은
- `WHERE salary IN (300, 400, 500)` ← 이것과 같습니다.

하지만 `ANY`는 `=` 외에도 `>`, `<`, `>=`, `<=` 등과 함께 쓸 수 있다는 점이 `IN`과의 차이입니다.
:::

### ALL 연산자 — "모두 만족해야 참"

**`ALL` 연산자**는 서브쿼리 결과의 **모든 값이 조건을 만족해야 참**을 반환합니다. `ANY`와 마찬가지로 반드시 **비교 연산자와 함께** 사용해야 합니다.

비유: "이 반의 **모든** 학생들보다 내 점수가 높아야 수석이다"와 같습니다. 단 한 명이라도 나보다 점수가 높으면 수석이 아니죠.

**동작 원리 이해하기:**

서브쿼리 결과(급여 목록)가 `(250만원, 300만원, 400만원)`이라고 가정합니다.

- `> ALL (250만원, 300만원, 400만원)` 에서 비교값이 `500만원`이면?
  - 250만원보다 크냐? → 참, 300만원보다 크냐? → 참, 400만원보다 크냐? → 참 → **모두 만족하므로 참** ✅
- `> ALL (250만원, 300만원, 400만원)` 에서 비교값이 `300만원`이면?
  - 250만원보다 크냐? → 참, 300만원보다 크냐? → 거짓(같은 값이므로 초과 아님) → **하나라도 불만족하므로 거짓** ❌
- `> ALL (250만원, 300만원, 400만원)` 에서 비교값이 `150만원`이면?
  - 250만원보다 크냐? → 거짓 → **즉시 거짓** ❌

아래 표로 `ALL`의 다양한 사용 예를 정리합니다:

| 조건식 | 서브쿼리 결과 | 비교값 | 결과 |
|--------|-------------|-------|------|
| `> ALL (250, 300, 400)` | 250, 300, 400 | 500 | **참** (500 > 250, 300, 400 모두 만족) |
| `> ALL (250, 300, 400)` | 250, 300, 400 | 300 | **거짓** (300 > 300 불만족) |
| `= ALL (250, 300, 400)` | 250, 300, 400 | 300 | **거짓** (300 = 250 불만족) |
| `> ALL (150, 200, 250)` | 150, 200, 250 | 300 | **참** (300 > 150, 200, 250 모두 만족) |
| `<> ALL (250, 300, 400)` | 250, 300, 400 | 150 | **참** (150 ≠ 250, ≠ 300, ≠ 400 모두 만족) |

```sql
-- ✅ 올바른 예시: 모든 직원의 급여보다 높은 급여를 받는 직원 조회
SELECT emp_name, salary          -- 사원 이름과 급여를 출력
FROM employee                    -- employee(직원) 테이블에서
WHERE salary > ALL (             -- 급여가 서브쿼리의 모든 결과보다 높으면
    SELECT salary                -- 급여를 조회
    FROM employee                -- employee 테이블에서
    WHERE dno = 'D002'           -- 특정 부서(D002)에 속한 직원들의 급여 목록
);
```

## 💡 실제 예시로 완전히 이해하기

### 예시 1 — 박영권과 같은 직급의 사원 검색

> "박영권이라는 사람과 같은 직급을 가진 사원들의 이름과 직급을 검색하라."

이 문제를 풀기 위한 논리 흐름은 다음과 같습니다:

1. 먼저 박영권의 직급이 무엇인지 알아야 합니다 → 서브쿼리
2. 그 직급과 동일한 모든 사원을 찾습니다 → 메인 쿼리

```sql
-- ✅ 올바른 예시: 박영권과 같은 직급의 사원 이름과 직급 검색
SELECT emp_name, title           -- 사원 이름과 직급을 출력 (메인 쿼리 결과)
FROM employee                    -- employee(직원) 테이블에서
WHERE title = (                  -- 직급이 서브쿼리 결과와 같은 사람
    SELECT title                 -- 직급(title)을 조회 (서브쿼리 시작)
    FROM employee                -- employee 테이블에서
    WHERE emp_name = '박영권'    -- 이름이 박영권인 사람의 직급
);                               -- 서브쿼리 끝
```

이 쿼리가 동작하는 순서:
1. 서브쿼리 `(SELECT title FROM employee WHERE emp_name = '박영권')`가 먼저 실행됩니다. 예를 들어 결과가 `'과장'`이라고 하면,
2. 메인 쿼리가 `WHERE title = '과장'`으로 바뀌어 실행됩니다.
3. 최종 결과: 직급이 '과장'인 모든 사원의 이름과 직급이 출력됩니다.

### 예시 2 — 영업부·개발부 근무 사원 검색 (IN 사용)

> "영업부 또는 개발부에 근무하는 사원의 이름을 검색하라."

```sql
-- ✅ 올바른 예시: IN을 이용한 다중행 서브쿼리
SELECT emp_name                  -- 사원 이름만 출력 (메인 쿼리는 employee 테이블 결과만)
FROM employee                    -- employee(직원) 테이블에서
WHERE dno IN (                   -- 부서번호가 서브쿼리 결과 목록에 포함되면
    SELECT dno                   -- 부서번호를 조회 (서브쿼리)
    FROM department              -- department(부서) 테이블에서
    WHERE dname = '영업부'       -- 영업부이거나
       OR dname = '개발부'       -- 개발부인 부서의 번호 (여러 행 반환)
);
```

같은 결과를 조인으로 작성하면:

```sql
-- ✅ 비교: 동일 결과를 조인(JOIN)으로 작성한 예시
SELECT E.emp_name                -- 사원 이름을 출력 (E는 employee의 별명)
FROM employee E, department D    -- employee(E)와 department(D) 두 테이블 사용
WHERE E.dno = D.dno              -- 조인 조건: 두 테이블의 부서번호가 같아야 함
  AND (D.dname = '영업부'        -- 그리고 부서명이 영업부이거나
    OR D.dname = '개발부');      -- 개발부인 경우
```

두 쿼리의 결과는 동일합니다. 하지만 서브쿼리 방식은 최종 결과가 `employee` 테이블의 컬럼만 출력되는 반면, 조인 방식은 `department` 테이블의 컬럼도 함께 출력할 수 있다는 차이가 있습니다.

### 예시 3 — A동 위치 부서 직원 검색 (LIKE + 서브쿼리)

> "A동에 위치하는 부서의 직원 이름을 검색하라."

```sql
-- ✅ 올바른 예시: LIKE 패턴 매칭을 서브쿼리 조건으로 활용
SELECT emp_name                  -- 사원 이름을 출력
FROM employee                    -- employee(직원) 테이블에서
WHERE dno IN (                   -- 부서번호가 서브쿼리 결과 목록에 있으면
    SELECT dno                   -- 부서번호를 조회
    FROM department              -- department(부서) 테이블에서
    WHERE location LIKE 'A%'     -- 부서 위치(location)가 A로 시작하는 경우
);                               -- 예: A동, A관, A빌딩 모두 포함됨
```

여기서 `LIKE 'A%'`의 `%`는 **와일드카드(Wildcard)** 로, 'A' 뒤에 어떤 문자가 오더라도 일치합니다. 예를 들어 `A동`, `A관`, `A빌딩`이 모두 해당됩니다. `LIKE` 연산자를 사용할 때는 반드시 `=`가 아닌 `LIKE` 키워드를 써야 한다는 점을 기억하세요.

### 예시 4 — EXISTS를 이용한 다중 속성 비교

> "영업부 또는 개발부에 근무하는 사원의 이름을 검색하라." (EXISTS 버전)

```sql
-- ✅ 올바른 예시: EXISTS를 이용해 여러 속성을 한꺼번에 비교
SELECT emp_name                  -- 사원 이름을 출력
FROM employee E                  -- employee 테이블을 E라고 별명 지정
WHERE EXISTS (                   -- 서브쿼리 결과가 존재하면 (있으면 참)
    SELECT *                     -- 모든 컬럼 반환 (값 내용보다 존재 여부가 핵심)
    FROM department D            -- department 테이블을 D라고 별명 지정
    WHERE E.dno = D.dno          -- 직원의 부서번호와 부서 테이블 번호 일치 (조인 조건)
      AND (D.dname = '영업부'    -- 부서명이 영업부이거나
        OR D.dname = '개발부')   -- 개발부인 경우
);
```

`EXISTS`에서는 `WHERE` 절에 컬럼명을 따로 쓰지 않고 바로 `EXISTS (...)`를 사용합니다. `IN`처럼 값을 하나하나 비교하는 것이 아니라, 조건에 맞는 행이 하나라도 존재하면 참이 됩니다. 특히 `SELECT *`처럼 여러 컬럼을 한꺼번에 넘겨줘야 할 때 `IN` 대신 `EXISTS`를 사용하는 것이 적절합니다.

## ⚠️ 주의사항과 실전 팁

### LIKE 연산자와 와일드카드 완벽 정리

`LIKE`는 문자열 패턴을 비교할 때 사용하는 특별한 연산자입니다. 두 가지 와일드카드(Wildcard, 특수 기호)를 지원합니다:

| 기호 | 이름 | 의미 | 예시 |
|------|------|------|------|
| `%` | 퍼센트 | 길이 제한 없이 모든 문자열 | `LIKE 'A%'` → A로 시작하는 모든 값 |
| `_` | 밑줄(언더바) | 정확히 한 글자 | `LIKE 'K_O'` → K로 시작하고 O로 끝나는 3글자 |

```sql
-- ❌ 잘못된 예시: LIKE 대신 = 사용
SELECT emp_name
FROM employee
WHERE phone = '777%';   -- 오류! % 와일드카드는 = 연산자와 사용 불가

-- ✅ 올바른 예시: LIKE 연산자 사용
SELECT emp_name                  -- 사원 이름을 출력
FROM employee                    -- employee(직원) 테이블에서
WHERE phone LIKE '777%';         -- 전화번호가 777로 시작하는 사원
```

**실전 예시: 제품 코드가 총 6자리이고, 첫 글자는 A, 마지막 두 글자는 89인 제품 검색**

```sql
-- ✅ 올바른 예시: LIKE와 언더바(_)를 조합한 패턴 검색
SELECT product_name              -- 제품명을 출력
FROM product                     -- product(제품) 테이블에서
WHERE product_code LIKE 'A___89';-- 총 6자리: A + 모르는 3글자(___)+ 89
-- A _ _ _ 8 9 → A로 시작, 중간 3글자 무관, 89로 끝, 총 6글자
```

### NULL 값 검색 — IS NULL / IS NOT NULL

`NULL`은 "값이 없음" 또는 "아직 결정되지 않음"을 의미하는 특별한 상태입니다. `NULL`은 0도 아니고 빈 문자열도 아닙니다. 그래서 일반 비교 연산자(`=`, `<>`)로는 `NULL` 값을 비교할 수 없습니다.

:::danger NULL 비교에서 절대 하면 안 되는 것
```sql
-- ❌ 절대 사용 금지: NULL을 = 로 비교
SELECT emp_name
FROM employee
WHERE dno = NULL;   -- 이것은 항상 결과가 없음! NULL은 = 로 비교 불가
```

`NULL = NULL`은 참이 아닙니다. `NULL`은 "모른다"는 의미이기 때문에 "모른다 = 모른다"는 성립하지 않습니다.
:::

```sql
-- ✅ 올바른 예시: IS NULL로 NULL 값 검색
SELECT emp_no, emp_name          -- 직원번호와 이름을 출력
FROM employee                    -- employee(직원) 테이블에서
WHERE dno IS NULL;               -- 부서번호가 NULL인 직원 (아직 부서 미배정)

-- ✅ 올바른 예시: IS NOT NULL로 NULL이 아닌 값 검색
SELECT emp_no, emp_name          -- 직원번호와 이름을 출력
FROM employee                    -- employee(직원) 테이블에서
WHERE dno IS NOT NULL;           -- 부서번호가 이미 확정된 직원
```

### 서브쿼리 결과 유형별 연산자 선택 가이드

```
서브쿼리 결과가 몇 행?
├── 0행 또는 1행 (단일행)
│   └── 사용 가능 연산자: =, >, <, >=, <=, <>
│
└── 여러 행 (다중행)
    ├── 값 목록과 비교할 때 → IN / NOT IN
    ├── 존재 여부만 확인할 때 → EXISTS / NOT EXISTS
    ├── 하나라도 만족하면 될 때 → 비교연산자 + ANY (또는 SOME)
    └── 모두 만족해야 할 때 → 비교연산자 + ALL
```

:::tip 실무에서 연산자를 선택하는 방법
- **값 목록이 명확하고 컬럼이 하나**일 때 → `IN`
- **여러 컬럼을 조건으로 사용하거나, 존재 여부만 체크**할 때 → `EXISTS`
- **"어느 것보다 크다/작다"처럼 범위를 비교**할 때 → `ANY` 또는 `ALL`
- **서브쿼리 결과에 NULL이 포함될 수 있는 상황**에서는 `NOT IN` 대신 `NOT EXISTS` 사용을 권장
:::

### 서브쿼리 안에서 ORDER BY를 쓰면 안 되는 이유

서브쿼리는 메인 쿼리에 **조건 값을 넘겨주는 역할**만 합니다. 메인 쿼리에 값을 넘겨줄 때 정렬 순서는 전혀 의미가 없습니다. 정렬은 최종 결과를 사용자에게 보여줄 때 메인 쿼리에서 하면 됩니다.

```sql
-- ❌ 잘못된 예시: 서브쿼리 안에 ORDER BY 사용
SELECT emp_name
FROM employee
WHERE dno IN (
    SELECT dno
    FROM department
    WHERE dname = '영업부'
    ORDER BY dno    -- 오류! 서브쿼리 안에서 ORDER BY 사용 불가
);

-- ✅ 올바른 예시: ORDER BY는 메인 쿼리에서만 사용
SELECT emp_name                  -- 사원 이름을 출력
FROM employee                    -- employee(직원) 테이블에서
WHERE dno IN (                   -- IN으로 다중행 서브쿼리 처리
    SELECT dno                   -- 서브쿼리: 부서번호 조회
    FROM department
    WHERE dname = '영업부'
)
ORDER BY emp_name;               -- 정렬은 메인 쿼리 마지막에 적용
```

## 📋 핵심 정리

이 문서에서 학습한 핵심 개념을 정리합니다.

- **서브쿼리(부속 질의문)** 는 SQL 안에 또 다른 SQL이 포함된 구조로, 반드시 괄호 `( )`로 감싸야 합니다.
- **메인 쿼리**는 바깥의 큰 SQL이고, **서브쿼리**는 `WHERE` 절 조건 안에 들어가는 안쪽의 작은 SQL입니다.
- 서브쿼리 결과가 **1행 이하**이면 단일행 서브쿼리로 `=`, `>`, `<` 등 일반 비교 연산자를 사용합니다.
- 서브쿼리 결과가 **여러 행**이면 다중행 서브쿼리로 `IN`, `EXISTS`, `ANY`, `ALL` 등 다중행 연산자를 사용합니다.
- **`IN`**: 목록 중 하나라도 일치하면 참. 단일 컬럼 비교에 사용하며 독립적으로 사용 가능합니다.
- **`NOT IN`**: 목록에 하나도 없어야 참. 단, 서브쿼리 결과에 `NULL`이 있으면 예상치 못한 동작을 할 수 있습니다.
- **`EXISTS`**: 서브쿼리 결과가 한 행이라도 존재하면 참. 여러 컬럼을 반환할 때 `IN` 대신 사용합니다.
- **`ANY`/`SOME`**: 하나라도 조건을 만족하면 참. 반드시 비교 연산자(`>`, `<` 등)와 함께 사용해야 합니다.
- **`ALL`**: 서브쿼리의 모든 결과가 조건을 만족해야 참. 반드시 비교 연산자와 함께 사용해야 합니다.
- **서브쿼리 안에서는 `ORDER BY`를 사용할 수 없습니다.** 정렬은 메인 쿼리에서만 합니다.
- 서브쿼리와 조인의 차이: 서브쿼리는 최종 결과가 **메인 쿼리 테이블의 컬럼만** 출력되고, 조인은 **두 테이블 모두**의 컬럼을 출력할 수 있습니다.
- **`LIKE`** 는 문자열 패턴 검색 연산자로, `%`(모든 문자열)와 `_`(한 글자)를 와일드카드로 사용합니다. `=` 대신 반드시 `LIKE` 키워드를 사용해야 합니다.
- **`NULL` 비교**는 `= NULL`이 아닌 `IS NULL`, `IS NOT NULL`로만 가능합니다.
