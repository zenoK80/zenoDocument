# 🔍 단일행 서브쿼리와 비교 연산자

SQL을 어느 정도 배우다 보면 "단순히 한 테이블에서 데이터를 꺼내는 것 이상"이 필요해지는 순간이 찾아옵니다. 예를 들어 "박영권 씨와 같은 직급을 가진 직원을 모두 찾아줘"라는 요청을 받았을 때, 우리는 먼저 박영권 씨의 직급이 무엇인지 알아야 하고, 그다음에 그 직급에 해당하는 직원들을 찾아야 합니다. 이 두 단계를 **SQL 한 문장으로 해결**할 수 있게 해주는 것이 바로 **서브쿼리(Subquery, 부속 질의문)** 입니다.

이 문서에서는 서브쿼리의 개념부터 시작해서, 단일행 서브쿼리와 다중행 서브쿼리의 차이, 각각에서 사용하는 비교 연산자, 그리고 `LIKE`, `NULL` 검색, 집합 연산, 데이터 조작어(DML)까지 초보자도 이해할 수 있도록 하나씩 풀어서 설명합니다.

---

## 📌 서브쿼리(부속 질의문)란 무엇인가?

### 개념 소개

**서브쿼리(Subquery)** 는 하나의 SQL 문장 **안에** 또 다른 SQL 문장이 들어가 있는 구조를 말합니다. 우리말로는 **부속 질의문** 이라고 부릅니다.

마치 러시아의 마트료시카 인형처럼, 큰 인형 안에 작은 인형이 들어 있는 구조입니다. 바깥쪽의 큰 SQL 문을 **메인 쿼리(Main Query)** 라고 하고, 안쪽에 들어가 있는 작은 SQL 문을 **서브쿼리(Subquery)** 라고 합니다.

```sql
-- 기본 서브쿼리 구조
SELECT 컬럼명           -- 메인 쿼리 시작
FROM 테이블명
WHERE 컬럼명 = (        -- 서브쿼리는 괄호로 감싼다
    SELECT 컬럼명       -- 서브쿼리 시작
    FROM 테이블명
    WHERE 조건
);                      -- 메인 쿼리 끝
```

위 구조에서 안쪽 `SELECT` 문이 먼저 실행되어 결과값을 만들어내고, 그 결과값이 바깥쪽 메인 쿼리의 `WHERE` 절 조건으로 들어가게 됩니다.

### 왜 서브쿼리가 필요한가?

이전에 배운 `SELECT`, `GROUP BY`, `HAVING`, `ORDER BY` 같은 명령어들은 **하나의 SQL 문장**으로 데이터를 검색하는 방식이었습니다. 하지만 현실 세계에서는 "다른 데이터를 먼저 알아야 검색 조건을 만들 수 있는" 상황이 매우 자주 발생합니다.

예를 들어 쇼핑몰 데이터베이스에서 "평균 주문 금액보다 많이 주문한 고객을 찾아라"라는 요청이 있다고 가정해 보겠습니다. 이 경우:

1. 먼저 전체 주문의 **평균 금액**을 계산해야 합니다.
2. 그 평균 금액보다 높은 주문을 한 고객을 찾아야 합니다.

이 두 단계를 별도의 쿼리로 나눠서 실행할 수도 있지만, 서브쿼리를 사용하면 **단 한 번의 SQL 문장**으로 해결할 수 있습니다.

### 서브쿼리 vs 조인(JOIN)의 차이

서브쿼리와 조인은 둘 다 여러 테이블의 데이터를 연결해서 사용하지만, 결정적인 차이가 있습니다.

| 구분 | 조인(JOIN) | 서브쿼리(Subquery) |
|------|-----------|-----------------|
| 결과 출력 | 연결된 **두 테이블 모두**의 컬럼 출력 가능 | **메인 쿼리 테이블**의 컬럼만 출력 |
| 성능 | 경우에 따라 다름 | 상황에 따라 더 효율적일 수 있음 |
| 코드 가독성 | 테이블 관계가 명확할 때 직관적 | 조건이 복잡할 때 더 읽기 쉬울 수 있음 |
| 사용 목적 | 두 테이블 데이터를 합쳐서 보고 싶을 때 | 조건 계산에 다른 테이블이 필요할 때 |

> **핵심 차이**: 조인은 두 테이블을 합쳐서 양쪽 데이터를 모두 볼 수 있지만, 서브쿼리는 최종 결과가 **항상 메인 쿼리 테이블의 데이터만** 출력됩니다. 두 번째 테이블(서브쿼리 테이블)은 조건을 비교하기 위한 도구로만 사용됩니다.

### 서브쿼리 사용 시 주의사항

:::warning 서브쿼리 기본 규칙
1. **서브쿼리는 반드시 괄호 `( )` 로 감싸야 합니다.**
2. **서브쿼리 안에서는 `ORDER BY` (정렬) 를 사용할 수 없습니다.** 서브쿼리는 조건값을 메인 쿼리에 넘겨주는 역할만 하기 때문에, 결과를 정렬할 필요가 없습니다.
3. **서브쿼리의 결과 건수에 따라 사용하는 비교 연산자가 달라집니다.** 이 부분이 서브쿼리에서 가장 중요한 핵심입니다.
:::

## 📊 단일행 서브쿼리 vs 다중행 서브쿼리

### 왜 구분이 필요한가?

서브쿼리가 반환하는 결과의 **행(Row, 레코드) 수**에 따라 사용할 수 있는 비교 연산자가 완전히 달라집니다. 이걸 구분하지 않으면 SQL 오류가 발생하거나 잘못된 결과가 나옵니다.

마치 수학에서 숫자 하나와 숫자 여러 개를 비교하는 방식이 다른 것과 같습니다. "내 점수(80점)가 70점보다 크냐?"는 숫자 하나와 비교하는 것이지만, "내 점수가 반 친구들 점수 중 하나보다도 크냐?"는 여러 숫자와 비교하는 방식입니다.

### 단일행 서브쿼리 (Single-Row Subquery)

**단일행 서브쿼리**는 서브쿼리의 실행 결과가 **0건 또는 1건** 인 경우를 말합니다. 결과가 딱 하나이거나 아예 없는 경우입니다.

이 경우에는 우리가 평소에 알고 있는 **일반 비교 연산자**를 사용할 수 있습니다.

| 연산자 | 의미 | 예시 |
|--------|------|------|
| `=` | 같다 | `salary = (서브쿼리)` |
| `>` | 크다 | `salary > (서브쿼리)` |
| `<` | 작다 | `salary < (서브쿼리)` |
| `>=` | 크거나 같다 | `salary >= (서브쿼리)` |
| `<=` | 작거나 같다 | `salary <= (서브쿼리)` |
| `<>` 또는 `!=` | 같지 않다 | `salary <> (서브쿼리)` |

```sql
-- ✅ 올바른 단일행 서브쿼리 예시
-- "박영권"과 같은 직급을 가진 사원의 이름과 직급 검색
SELECT emp_name, title          -- 직원 이름과 직급을 출력한다
FROM employee                   -- employee(직원) 테이블에서
WHERE title = (                 -- 직급이 다음 서브쿼리 결과와 같은 경우
    SELECT title                -- 직급을 가져온다
    FROM employee               -- employee 테이블에서
    WHERE emp_name = '박영권'   -- 이름이 '박영권'인 사람의
);
```

위 쿼리의 실행 순서를 단계별로 살펴보면:

1. **서브쿼리 실행**: `employee` 테이블에서 이름이 `'박영권'`인 사람의 `title`(직급)을 찾습니다. 예를 들어 결과가 `'과장'`이라고 가정합니다.
2. **메인 쿼리 실행**: 서브쿼리 결과인 `'과장'`이 `WHERE title = '과장'` 조건으로 들어가서, 직급이 과장인 모든 직원의 이름과 직급을 출력합니다.

:::danger 단일행 서브쿼리에 여러 행이 반환될 때
단일행 비교 연산자(`=`, `>`, `<` 등)를 사용했는데 서브쿼리가 여러 건의 결과를 반환하면 **SQL 오류가 발생**합니다. 예를 들어 이름이 같은 사람이 두 명 있다면 서브쿼리가 직급을 두 개 반환하게 되고, `=` 연산자는 하나의 값만 비교할 수 있으므로 오류가 납니다. 이런 경우에는 다중행 비교 연산자를 사용해야 합니다.
:::

### 다중행 서브쿼리 (Multi-Row Subquery)

**다중행 서브쿼리**는 서브쿼리의 실행 결과가 **여러 건** 인 경우를 말합니다. 이 경우에는 일반 비교 연산자(`=`, `>` 등)를 사용할 수 없고, **다중행 비교 연산자**를 사용해야 합니다.

| 연산자 | 의미 | 독립 사용 여부 |
|--------|------|--------------|
| `IN` | 목록 중 하나라도 일치하면 참 | 독립 사용 가능 |
| `NOT IN` | 목록 중 어느 것과도 일치하지 않으면 참 | 독립 사용 가능 |
| `EXISTS` | 서브쿼리 결과가 존재하면 참 | 독립 사용 가능 |
| `NOT EXISTS` | 서브쿼리 결과가 존재하지 않으면 참 | 독립 사용 가능 |
| `ANY` / `SOME` | 하나 이상의 값과 조건을 만족하면 참 | **비교 연산자와 함께 사용** |
| `ALL` | 모든 값과 조건을 만족해야 참 | **비교 연산자와 함께 사용** |

:::info ANY와 SOME의 관계
`ANY`와 `SOME`은 완전히 **동일한 연산자**입니다. 두 키워드 중 어느 것을 사용해도 결과는 같습니다. SQL 표준에서 같은 기능을 두 가지 이름으로 지원하는 것입니다.
:::

## 🔬 다중행 비교 연산자 상세 설명

### IN과 NOT IN

**`IN`** 은 서브쿼리(또는 목록)의 값들 중 **하나라도 일치하면** 참(`TRUE`)이 됩니다. 마치 "이 명단에 내 이름이 있나요?"를 확인하는 것과 같습니다.

```sql
-- 서브쿼리 결과가 (3, 4, 2, 6) 이라고 가정할 때
-- IN 예시 판단:
-- 조건값이 3이라면 → 3이 (3, 4, 2, 6) 안에 있음 → 참(TRUE)
-- 조건값이 5라면 → 5가 (3, 4, 2, 6) 안에 없음 → 거짓(FALSE)

-- ✅ 올바른 IN 사용 예시
-- 영업부 또는 개발부에 근무하는 직원의 이름 검색
SELECT emp_name                      -- 직원 이름을 출력한다
FROM employee                        -- employee(직원) 테이블에서
WHERE dno IN (                       -- 부서번호가 서브쿼리 결과 중 하나에 포함되면
    SELECT dno                       -- 부서번호를 가져온다
    FROM department                  -- department(부서) 테이블에서
    WHERE dept_name = '영업'         -- 부서명이 '영업'이거나
       OR dept_name = '개발'         -- 부서명이 '개발'인 경우
);
```

**`NOT IN`** 은 `IN`의 반대로, 서브쿼리 결과 중 **어느 것과도 일치하지 않을 때** 참이 됩니다.

```sql
-- ✅ NOT IN 사용 예시
-- 영업부, 개발부를 제외한 다른 부서 직원의 이름 검색
SELECT emp_name
FROM employee
WHERE dno NOT IN (
    SELECT dno
    FROM department
    WHERE dept_name = '영업'
       OR dept_name = '개발'
);
```

### ANY / SOME

**`ANY`** (또는 `SOME`)는 서브쿼리 결과 중 **하나 이상**의 값이 조건을 만족하면 참이 됩니다. 반드시 `>`, `<`, `=` 등의 **비교 연산자와 함께** 사용해야 합니다.

```sql
-- 서브쿼리 결과가 (250만원, 300만원, 400만원) 이라고 가정

-- "> ANY" 예시 판단:
-- 조건값이 300만원이라면 → (250만원, 300만원, 400만원) 중 250만원보다 크므로 → 참(TRUE)
-- 조건값이 400만원이라면 → (250만원, 300만원, 400만원) 중 하나라도 400만원보다 작은 것이 없음
--                        → 250만원, 300만원이 400만원보다 작으므로 참(TRUE)
-- 조건값이 450만원이라면 → 모두 450만원보다 작으므로 → 참(TRUE)

-- ✅ ANY 사용 예시
-- 영업팀 직원 급여 중 하나보다라도 적게 받는 직원 검색
SELECT emp_name, salary
FROM employee
WHERE salary < ANY (             -- 급여가 서브쿼리 결과 중 하나보다라도 작으면
    SELECT salary
    FROM employee
    WHERE dept_name = '영업팀'   -- 영업팀 직원들의 급여
);
```

:::tip ANY 연산자 핵심 이해
`> ANY (서브쿼리)` 는 "서브쿼리 결과 중 **가장 작은 값**보다 크면 참"이 됩니다.
`< ANY (서브쿼리)` 는 "서브쿼리 결과 중 **가장 큰 값**보다 작으면 참"이 됩니다.
즉, ANY는 **"하나라도 만족하면 OK"** 라는 관대한 조건입니다.
:::

### ALL

**`ALL`** 은 서브쿼리의 **모든 결과값**이 조건을 만족해야 참이 됩니다. ANY와 마찬가지로 반드시 비교 연산자와 함께 사용해야 합니다.

```sql
-- 서브쿼리 결과가 (250만원, 300만원, 400만원) 이라고 가정

-- "> ALL" 예시 판단:
-- 조건값이 300만원 → 300 > 250 (참), 300 > 300 (거짓) → 전체 거짓(FALSE)
-- 조건값이 150만원 → 150 > 250 (거짓) → 전체 거짓(FALSE)
-- 조건값이 450만원 → 450 > 250 (참), 450 > 300 (참), 450 > 400 (참) → 전체 참(TRUE)

-- ✅ ALL 사용 예시
-- 모든 영업팀 직원의 급여보다 많이 받는 직원 검색 (최고 급여보다도 많이 받는 사람)
SELECT emp_name, salary
FROM employee
WHERE salary > ALL (             -- 급여가 서브쿼리 결과 모두보다 크면
    SELECT salary
    FROM employee
    WHERE dept_name = '영업팀'   -- 영업팀 직원들의 급여
);
```

:::tip ALL 연산자 핵심 이해
`> ALL (서브쿼리)` 는 "서브쿼리 결과 중 **가장 큰 값**보다도 크면 참"이 됩니다.
`< ALL (서브쿼리)` 는 "서브쿼리 결과 중 **가장 작은 값**보다도 작으면 참"이 됩니다.
즉, ALL은 **"모두 만족해야 OK"** 라는 까다로운 조건입니다.
:::

### EXISTS와 NOT EXISTS

**`EXISTS`** 는 서브쿼리의 결과가 **한 건이라도 존재하면** 참이 됩니다. 특히 서브쿼리에서 여러 컬럼을 반환해야 할 때(`SELECT *` 를 쓸 때) 유용합니다.

```sql
-- ❌ 잘못된 예시: 여러 컬럼을 반환할 때 IN을 쓰면 오류 발생
SELECT emp_name
FROM employee E
WHERE dno IN (
    SELECT *                     -- 여러 컬럼을 반환 → IN과 함께 사용 불가
    FROM department D
    WHERE D.dno = E.dno
      AND (dept_name = '영업' OR dept_name = '개발')
);

-- ✅ 올바른 예시: 여러 컬럼 반환 시 EXISTS 사용
SELECT emp_name
FROM employee E
WHERE EXISTS (                   -- 서브쿼리 결과가 존재하면 참
    SELECT *                     -- 모든 컬럼 반환 (EXISTS는 이걸 허용)
    FROM department D
    WHERE D.dno = E.dno          -- 직원 테이블과 부서 테이블의 부서번호가 같고
      AND (dept_name = '영업'    -- 부서명이 '영업'이거나
           OR dept_name = '개발')-- 부서명이 '개발'인 경우
);
```

`EXISTS`와 `IN`의 중요한 차이점은 다음과 같습니다.

| 구분 | IN | EXISTS |
|------|-----|--------|
| 반환 컬럼 | 단일 컬럼 결과와 비교 | `SELECT *` 처럼 여러 컬럼 가능 |
| WHERE 절 속성 | 컬럼명을 명시해야 함 | 컬럼명 생략 가능 |
| 사용 시점 | 단일 속성 비교 시 | 여러 속성을 한꺼번에 반환할 때 |

**`NOT EXISTS`** 는 서브쿼리의 결과가 **존재하지 않을 때** 참이 됩니다.

### IN vs ANY vs ALL 비교 정리

다음 예시를 통해 세 연산자의 차이를 한눈에 비교해 보겠습니다. 서브쿼리 결과가 `(250, 300, 400)` (단위: 만원)이라고 가정합니다.

| 조건 | 연산자 | 조건값 300 | 조건값 450 | 조건값 150 |
|------|--------|-----------|-----------|-----------|
| `IN (250, 300, 400)` | IN | 참 (300이 목록에 있음) | 거짓 (450이 목록에 없음) | 거짓 (150이 목록에 없음) |
| `> ANY (250, 300, 400)` | ANY | 참 (250보다는 큼) | 참 (400보다도 큼) | 거짓 (250보다 작음) |
| `> ALL (250, 300, 400)` | ALL | 거짓 (400보다 작음) | 참 (400보다 큼) | 거짓 (250보다 작음) |
| `= ALL (250, 300, 400)` | ALL | 거짓 (모두 300이 아님) | 거짓 | 거짓 |

## 💡 서브쿼리 실제 예시 모음

### 예시 1: 단일행 서브쿼리 - 같은 직급 직원 검색

```sql
-- 박영권과 같은 직급(title)을 가진 사원들의 이름과 직급 검색

-- ✅ 서브쿼리를 이용한 방법
SELECT emp_name, title           -- 직원 이름과 직급을 출력
FROM employee                    -- employee(직원) 테이블에서
WHERE title = (                  -- 직급이 아래 서브쿼리 결과와 같으면
    SELECT title                 -- 직급을 가져온다
    FROM employee                -- employee 테이블에서
    WHERE emp_name = '박영권'    -- 이름이 '박영권'인 사람의
);
-- 실행 순서:
-- 1단계: 서브쿼리 실행 → 박영권의 직급(예: '과장') 반환
-- 2단계: 메인쿼리 실행 → title = '과장'인 모든 직원 출력
```

### 예시 2: 다중행 서브쿼리 - IN을 이용한 부서 직원 검색

```sql
-- 영업부 또는 개발부에 근무하는 직원의 이름을 검색

-- ✅ 서브쿼리(IN)를 이용한 방법
SELECT emp_name                      -- 직원 이름을 출력
FROM employee                        -- employee(직원) 테이블에서
WHERE dno IN (                       -- 부서번호가 서브쿼리 결과 목록에 포함되면
    SELECT dno                       -- 부서번호를 가져온다
    FROM department                  -- department(부서) 테이블에서
    WHERE dept_name = '영업'         -- 부서명이 '영업'이거나
       OR dept_name = '개발'         -- 부서명이 '개발'인 경우
);

-- ✅ 조인(JOIN)을 이용한 동일한 결과
SELECT E.emp_name                    -- 직원 이름을 출력
FROM employee E, department D        -- 두 테이블 사용 (조인)
WHERE E.dno = D.dno                  -- 조인 조건: 두 테이블의 부서번호가 같고
  AND (D.dept_name = '영업'          -- 부서명이 '영업'이거나
       OR D.dept_name = '개발');     -- 부서명이 '개발'인 경우
-- 두 방법의 결과는 동일하지만,
-- 서브쿼리 방법은 최종 결과가 employee 테이블 컬럼만 출력됨
-- 조인 방법은 양쪽 테이블 컬럼 모두 출력 가능
```

### 예시 3: LIKE를 사용한 부서 직원 검색

```sql
-- A동에 위치하는 부서의 직원 이름을 검색

-- ✅ LIKE와 서브쿼리를 결합한 방법
SELECT emp_name                      -- 직원 이름을 출력
FROM employee                        -- employee(직원) 테이블에서
WHERE dno IN (                       -- 부서번호가 서브쿼리 결과에 포함되면
    SELECT dno                       -- 부서번호를 가져온다
    FROM department                  -- department(부서) 테이블에서
    WHERE dept_location LIKE 'A%'   -- 부서 위치가 'A'로 시작하는 경우
    -- 'A%': A1, A동, A관 등 A로 시작하는 모든 값 포함
);
```

## 🔤 LIKE 연산자와 NULL 검색

### LIKE 연산자 - 문자열 패턴 검색

**`LIKE`** 연산자는 문자열을 **패턴으로** 비교 검색할 때 사용합니다. 정확한 값을 모르거나, 특정 패턴으로 시작하거나 끝나는 데이터를 찾을 때 사용합니다.

`LIKE`에서 사용하는 두 가지 와일드카드(특수 문자):

| 와일드카드 | 의미 | 예시 |
|-----------|------|------|
| `%` | **0개 이상**의 임의 문자 | `'A%'` → A, A1, ABC, A동 모두 해당 |
| `_` (밑줄) | **정확히 1개**의 임의 문자 | `'K_O'` → KAO, KBO, KXO 등 3글자만 해당 |

:::danger = 대신 반드시 LIKE 사용
`%`나 `_`를 이용한 패턴 검색 시 `=` 연산자를 사용하면 **안 됩니다**. 반드시 `LIKE` 키워드를 사용해야 합니다.

```sql
-- ❌ 잘못된 예시 (오류 발생 또는 잘못된 결과)
WHERE phone = '777%'        -- = 연산자로 패턴 검색 불가

-- ✅ 올바른 예시
WHERE phone LIKE '777%'     -- LIKE 연산자로 패턴 검색
```
:::

```sql
-- 예시 1: 전화번호 국번이 777로 시작하는 직원 이름 검색
SELECT emp_name                      -- 직원 이름을 출력
FROM employee                        -- employee(직원) 테이블에서
WHERE phone LIKE '777%';             -- 전화번호가 777로 시작하는 경우
-- '777%': 777로 시작하고 뒤에 어떤 문자가 와도 됨
-- 예: 777-1234, 7771234, 777ABCD 모두 해당

-- 예시 2: 제품 코드가 총 6자리이고, 첫 글자는 A, 마지막 두 글자는 89인 제품 검색
SELECT product_name                  -- 제품명을 출력
FROM product                         -- product(제품) 테이블에서
WHERE product_code LIKE 'A___89';    -- 첫글자 A, 중간 3글자 임의, 끝 두글자 89
-- 'A___89': 총 6자리 (A + _ + _ + _ + 8 + 9)
-- 예: A12389, AXYZ89, AABC89 등

-- 예시 3: NOT LIKE - 특정 패턴을 제외하고 검색
SELECT emp_name
FROM employee
WHERE phone NOT LIKE '777%';         -- 전화번호가 777로 시작하지 않는 직원
```

### NULL 값 검색

**`NULL`** 은 데이터베이스에서 "값이 없음" 또는 "알 수 없음"을 나타내는 특별한 값입니다. `NULL`은 숫자 `0`도 아니고, 빈 문자열(`''`)도 아닙니다. 완전히 다른 개념입니다.

마치 "답변 없음"과 "0점"은 다른 것처럼, `NULL`은 "해당 없음" 또는 "미정"을 의미합니다.

:::danger NULL 비교 시 = 연산자 절대 사용 금지
`NULL` 값을 비교할 때 `= NULL` 또는 `<> NULL` 같은 일반 비교 연산자를 사용하면 **항상 거짓(FALSE)** 이 반환되어 원하는 결과를 얻을 수 없습니다. NULL은 반드시 `IS NULL` 또는 `IS NOT NULL`로만 비교해야 합니다.

```sql
-- ❌ 절대 이렇게 하면 안 됩니다
WHERE dno = NULL      -- 항상 FALSE, 원하는 결과 안 나옴
WHERE dno != NULL     -- 항상 FALSE, 원하는 결과 안 나옴

-- ✅ 반드시 이렇게 사용해야 합니다
WHERE dno IS NULL     -- 부서번호가 NULL인 경우
WHERE dno IS NOT NULL -- 부서번호가 NULL이 아닌 경우
```
:::

```sql
-- 예시 1: 아직 부서가 결정되지 않은 직원(부서번호가 NULL인) 번호와 이름 검색
SELECT emp_no, emp_name              -- 직원번호와 이름을 출력
FROM employee                        -- employee(직원) 테이블에서
WHERE dno IS NULL;                   -- 부서번호(dno)가 NULL인 경우
-- 부서가 배정되지 않아서 dno 값이 아예 없는 직원을 찾는다

-- 예시 2: 이미 부서가 확정된 직원(부서번호가 NULL이 아닌) 번호와 이름 검색
SELECT emp_no, emp_name              -- 직원번호와 이름을 출력
FROM employee                        -- employee(직원) 테이블에서
WHERE dno IS NOT NULL;               -- 부서번호(dno)가 NULL이 아닌 경우
-- 부서가 이미 배정되어 dno 값이 있는 직원을 찾는다
```

## 🗂️ 집합 연산자 (UNION, INTERSECT, EXCEPT)

### 집합 연산이란?

데이터베이스에서 **집합 연산**은 두 개의 SELECT 결과를 수학의 집합처럼 합치거나 교차하거나 빼는 작업입니다. 관계 대수에서 배운 합집합, 교집합, 차집합을 SQL로 구현하는 방법입니다.

집합 연산을 사용하려면 두 테이블이 **합병 가능(Union-Compatible)** 해야 합니다. 즉, 두 SELECT 결과의 **컬럼 수**와 **데이터 타입**이 서로 같아야 합니다.

> **기본 원칙**: 집합 연산의 결과에서는 중복되는 행(튜플)이 자동으로 제거됩니다. 만약 중복을 포함시키고 싶다면 `ALL`을 붙여야 합니다.

### 집합 연산자 종류

| 연산자 | 의미 | 중복 처리 |
|--------|------|----------|
| `UNION` | 합집합 (두 결과를 합침) | 중복 제거 |
| `UNION ALL` | 합집합 (두 결과를 합침) | 중복 포함 |
| `INTERSECT` | 교집합 (공통으로 있는 것) | 중복 제거 |
| `INTERSECT ALL` | 교집합 | 중복 포함 |
| `EXCEPT` | 차집합 (첫 번째에만 있는 것) | 중복 제거 |
| `EXCEPT ALL` | 차집합 | 중복 포함 |

### 집합 연산 예시

R1 = {1, 2, 3, 5}, R2 = {1, 2, 3, 4} 라고 가정할 때:

- **R1 UNION R2** = {1, 2, 3, 4, 5} (합집합, 중복 제거)
- **R1 UNION ALL R2** = {1, 1, 2, 2, 3, 3, 4, 5} (합집합, 중복 포함)
- **R1 INTERSECT R2** = {1, 2, 3} (교집합, 공통 원소)
- **R1 EXCEPT R2** = {5} (R1에만 있는 것)
- **R2 EXCEPT R1** = {4} (R2에만 있는 것)

```sql
-- ✅ UNION 예시
-- 직원번호가 100이거나, 부서가 기획과인 직원의 부서번호 검색 (OR 조건과 유사)
SELECT dno                           -- 직원번호가 100인 직원의 부서번호
FROM employee
WHERE emp_no = 100

UNION                                -- 합집합 (중복 제거)

SELECT dno                           -- 부서명이 기획과인 부서의 부서번호
FROM department
WHERE dept_name = '기획과';

-- ✅ UNION ALL 예시 (중복 포함)
SELECT dno FROM employee WHERE emp_no = 100
UNION ALL                            -- 합집합 (중복 포함)
SELECT dno FROM department WHERE dept_name = '기획과';

-- ✅ INTERSECT 예시
-- 두 조건 모두 만족하는 부서번호 (교집합)
SELECT dno FROM employee WHERE emp_no = 100
INTERSECT
SELECT dno FROM department WHERE dept_name = '기획과';

-- ✅ EXCEPT 예시
-- 첫 번째 결과에만 있고 두 번째 결과에는 없는 부서번호 (차집합)
SELECT dno FROM employee WHERE emp_no = 100
EXCEPT
SELECT dno FROM department WHERE dept_name = '기획과';
```

## ✏️ 데이터 조작어(DML): INSERT, DELETE, UPDATE

### INSERT - 데이터 삽입

**`INSERT`** 는 테이블에 새로운 데이터(행, 레코드)를 추가하는 명령어입니다. 마치 엑셀 표에 새 행을 추가하는 것과 같습니다.

```sql
-- INSERT 기본 문법
INSERT INTO 테이블명 (컬럼1, 컬럼2, ...)  -- 어느 컬럼에 값을 넣을지 지정
VALUES (값1, 값2, ...);                    -- 넣을 값들

-- 모든 컬럼에 값을 넣을 때는 컬럼명 생략 가능
INSERT INTO 테이블명
VALUES (값1, 값2, ...);                    -- 테이블의 모든 컬럼에 순서대로 값 입력
```

INSERT 사용 시 주요 규칙:

- `NULL` 값도 삽입 가능합니다. (`VALUES (..., NULL, ...)`)
- 모든 컬럼에 값을 입력할 때는 컬럼명을 **생략 가능**합니다.
- 일부 컬럼에만 값을 넣을 때는 반드시 **컬럼명을 명시**해야 합니다.
- 하나의 `INSERT` 문으로 여러 행을 동시에 삽입할 수 있습니다.
- **한 번에 하나의 테이블에만** 삽입할 수 있습니다.
- `INSERT` 문 안에서 정렬(`ORDER BY`)은 사용할 수 없습니다.
- 기본키(Primary Key) 컬럼에는 `NULL` 값이나 중복 값을 넣을 수 없습니다.

```sql
-- ✅ 예시 1: 부서 테이블에 새 부서 삽입 (모든 컬럼 입력, 컬럼명 생략)
-- department 테이블 구조: dno, dept_name, location
INSERT INTO department
VALUES ('D5', '연구', NULL);             -- 부서번호, 부서명, 위치(NULL=미정)

-- ✅ 예시 2: 일부 컬럼만 입력 (컬럼명 명시 필수)
INSERT INTO department (dno, dept_name)  -- 부서번호와 부서명만 입력
VALUES ('D6', '홍보');                   -- 위치(location)는 자동으로 NULL

-- ✅ 예시 3: SELECT 결과를 다른 테이블에 복사 삽입
-- 급여 300만원 이상인 직원 정보를 'high_salary' 테이블에 복사
INSERT INTO high_salary (name, title, salary)  -- high_salary 테이블의 세 컬럼에
SELECT emp_name, title, salary                 -- 직원 이름, 직급, 급여를 가져와서
FROM employee                                  -- employee 테이블에서
WHERE salary >= 300;                           -- 급여가 300만원 이상인 직원만

-- ✅ 예시 4: 부서번호가 D1인 직원 정보를 인사과 직원 테이블에 삽입
INSERT INTO hr_employee (emp_no, emp_name, base_salary)  -- 삽입할 컬럼 지정
SELECT emp_no, emp_name, base_salary                     -- 검색할 데이터
FROM employee                                            -- employee 테이블에서
WHERE dno = 'D1';                                        -- 부서번호가 D1인 직원만
```

### DELETE - 데이터 삭제

**`DELETE`** 는 테이블에서 특정 조건에 맞는 행(레코드)을 삭제하는 명령어입니다. **테이블 구조 자체는 남아있고**, 데이터(행)만 삭제됩니다. 이 점이 테이블 전체를 삭제하는 `DROP TABLE`과의 차이점입니다.

```sql
-- DELETE 기본 문법
DELETE FROM 테이블명                     -- 어떤 테이블에서 삭제할지
WHERE 조건;                              -- 어떤 조건의 행을 삭제할지

-- WHERE 절이 없으면 테이블의 모든 행이 삭제됨 (주의!)
DELETE FROM 테이블명;                    -- 테이블의 모든 데이터 삭제
```

:::danger DELETE 사용 시 주의사항
- `WHERE` 절 없이 `DELETE FROM 테이블명;` 을 실행하면 **테이블의 모든 데이터가 삭제**됩니다. 한 번 삭제된 데이터는 복구가 어려우므로 반드시 `WHERE` 조건을 확인하고 실행하세요.
- 데이터 삭제 시 **참조 무결성(Referential Integrity)** 을 반드시 고려해야 합니다. 다른 테이블에서 참조하고 있는 데이터를 삭제하려고 하면 오류가 발생할 수 있습니다.
:::

```sql
-- ✅ 예시 1: 특정 조건의 데이터 삭제
-- 부서번호가 4번인 부서 삭제
DELETE FROM department                   -- department(부서) 테이블에서
WHERE dno = 4;                           -- 부서번호가 4인 행을 삭제

-- ✅ 예시 2: 사원번호로 특정 직원 삭제
DELETE FROM emp                          -- emp(직원) 테이블에서
WHERE emp_no = 7902;                     -- 사원번호가 7902인 행을 삭제

-- ✅ 예시 3: 서브쿼리를 이용한 삭제
-- 평균 급여보다 적게 받는 직원 삭제
DELETE FROM emp                          -- emp 테이블에서
WHERE salary < (                         -- 급여가 아래 서브쿼리 결과보다 작으면
    SELECT AVG(salary)                   -- 전체 평균 급여를 계산
    FROM emp                             -- emp 테이블에서
);

-- ✅ 예시 4: 테이블의 모든 데이터 삭제 (구조는 유지)
DELETE FROM emp;                         -- emp 테이블의 모든 행 삭제 (테이블은 남음)

-- ✅ 예시 5: 서브쿼리를 이용한 관련 데이터 삭제
-- 홍길동과 같은 부서에 근무하는 모든 직원 삭제
DELETE FROM employee                     -- employee 테이블에서
WHERE dno IN (                           -- 부서번호가 서브쿼리 결과에 포함되면
    SELECT dno                           -- 부서번호를 가져온다
    FROM employee                        -- employee 테이블에서
    WHERE emp_name = '홍길동'            -- 이름이 '홍길동'인 직원의 부서번호
);
```

### UPDATE - 데이터 갱신

**`UPDATE`** 는 테이블에 이미 있는 데이터를 수정(갱신)하는 명령어입니다. `SET` 절을 이용해 수정할 컬럼과 새로운 값을 지정합니다.

```sql
-- UPDATE 기본 문법
UPDATE 테이블명                          -- 수정할 테이블
SET 컬럼1 = 새값1,                       -- 수정할 컬럼과 새로운 값 (콤마로 구분하여 여러 컬럼 동시 수정 가능)
    컬럼2 = 새값2
WHERE 조건;                              -- 어떤 행을 수정할지 조건
```

```sql
-- ✅ 예시 1: 사원번호 2106인 직원의 부서를 3번으로 이동하고 급여를 5% 인상
UPDATE employee                          -- employee(직원) 테이블을 수정
SET dno = 3,                             -- 부서번호를 3번으로 변경
    salary = salary * 1.05              -- 급여를 현재의 1.05배로 변경 (5% 인상)
WHERE emp_no = 2106;                     -- 사원번호가 2106인 직원만

-- ✅ 예시 2: 급여 10% 인상
UPDATE employee
SET salary = salary * 1.1              -- 급여를 현재의 1.1배로 변경 (10% 인상)
WHERE emp_no = 7902;                   -- 사원번호가 7902인 직원

-- ✅ 예시 3: 부서번호가 20인 직원의 급여를 10% 인상
UPDATE emp                             -- emp 테이블을 수정
SET salary = salary * 1.1             -- 급여 10% 인상
WHERE dno = 20;                        -- 부서번호가 20인 직원들

-- ✅ 예시 4: 모든 직원의 입사일을 오늘 날짜로 변경 (WHERE 없음 = 전체 적용)
UPDATE emp                             -- emp 테이블을 수정
SET hire_date = SYSDATE;               -- 입사일(hire_date)을 시스템 현재 날짜로 변경
-- SYSDATE: 데이터베이스 시스템의 현재 날짜 및 시간 반환 함수

-- ✅ 예시 5: 서브쿼리를 이용한 업데이트
-- 컴퓨터과 학생의 기말고사 성적을 5점 감점
UPDATE enroll                          -- enroll(수강) 테이블을 수정
SET final_score = final_score - 5     -- 기말고사 점수를 5점 감점
WHERE sno IN (                         -- 학번이 서브쿼리 결과에 포함되면
    SELECT sno                         -- 학번을 가져온다
    FROM student                       -- student(학생) 테이블에서
    WHERE department = '컴퓨터'        -- 학과가 '컴퓨터'인 학생의 학번
);
-- 실행 과정:
-- 1단계: 서브쿼리 → student 테이블에서 컴퓨터과 학생들의 학번 목록 조회
-- 2단계: 메인쿼리 → 해당 학번을 가진 학생들의 enroll 테이블 기말점수 5점 감점
```

:::note GROUP BY와 COUNT를 활용한 서브쿼리
```sql
-- 소속 직원이 3명 이상인 부서의 부서번호 검색
SELECT dno                             -- 부서번호를 출력
FROM employee                          -- employee(직원) 테이블에서
GROUP BY dno                           -- 부서번호별로 그룹화
HAVING COUNT(*) >= 3;                  -- 그룹 내 직원 수가 3명 이상인 부서만
-- COUNT(*): 각 그룹(부서)의 레코드(직원) 수를 센다
-- HAVING: 그룹에 대한 조건을 걸 때 사용 (WHERE는 개별 행에 대한 조건)
```
:::

## 🛡️ 데이터 무결성 유지의 중요성

### 왜 무결성이 중요한가?

데이터베이스는 처음에 데이터를 입력할 때는 정확하고 일관된 상태를 유지합니다. 마치 새 컴퓨터를 사서 처음 켰을 때 모든 것이 깔끔하게 동작하는 것처럼요. 하지만 데이터를 계속 **삽입하고, 삭제하고, 수정하는 작업**을 반복하다 보면 데이터 간의 일관성이 깨질 위험이 생깁니다.

예를 들어, 어떤 부서를 삭제했는데 그 부서에 소속된 직원 데이터는 여전히 남아있다면? 그 직원 데이터는 존재하지 않는 부서를 참조하게 되어 **참조 무결성 위반**이 발생합니다.

### 세 가지 무결성 규칙

**1. 개체 무결성 (Entity Integrity)**

기본키(Primary Key)에 관한 규칙입니다.
- 기본키는 **절대 `NULL` 값**을 가질 수 없습니다.
- 기본키는 **중복 값**이 있어서는 안 됩니다.
- 모든 행은 기본키를 통해 유일하게 식별되어야 합니다.

**2. 참조 무결성 (Referential Integrity)**

외래키(Foreign Key)에 관한 규칙입니다.
- 외래키는 **참조하는 테이블에 존재하는 값**만 가질 수 있습니다.
- 존재하지 않는 부서 번호를 직원의 `dno`(외래키)에 넣으면 안 됩니다.
- 참조 무결성을 유지하기 위해 `RESTRICT`, `CASCADE`, `SET NULL`, `SET DEFAULT` 같은 옵션을 사용합니다.

**3. 도메인 무결성 (Domain Integrity)**

속성(컬럼) 값의 **허용 범위**에 관한 규칙입니다.
- 점수 컬럼은 0~100 사이의 값만 허용
- 급여 컬럼은 100만원~300만원 사이만 허용
- 지역 컬럼은 '서울', '부산', '대구' 중 하나만 허용
- 이런 도메인 제약이 항상 지켜져야 데이터를 안전하게 사용할 수 있습니다.

> **데이터 무결성이 지켜질 때**: 데이터베이스의 성능이 유지되고, 데이터를 신뢰할 수 있으며, 프로그램이 예상대로 동작합니다. 삽입, 삭제, 수정 등 어떤 작업을 하더라도 이 세 가지 무결성 규칙은 반드시 지켜져야 합니다.

## ⚠️ 주의사항 및 핵심 팁

### 서브쿼리 사용 시 자주 하는 실수

:::warning 서브쿼리 결과 건수 확인 필수
서브쿼리를 작성하기 전에 **해당 서브쿼리가 몇 건의 결과를 반환하는지** 먼저 파악해야 합니다. 결과가 항상 1건이면 단일행 연산자(`=`)를, 여러 건이 나올 수 있으면 다중행 연산자(`IN`, `ANY`, `ALL` 등)를 사용해야 합니다. 이 판단을 잘못하면 오류가 발생하거나 잘못된 결과가 나옵니다.
:::

:::warning ORDER BY는 서브쿼리에서 사용 불가
서브쿼리 안에서는 `ORDER BY` 정렬 절을 사용할 수 없습니다. 서브쿼리는 조건값을 메인 쿼리에 넘겨주는 역할만 하므로 정렬이 의미가 없습니다.
:::

:::tip IN과 EXISTS의 선택 기준
- 서브쿼리에서 **단일 컬럼**을 반환할 때 → `IN` 사용
- 서브쿼리에서 **여러 컬럼**(`SELECT *`)을 반환할 때 → `EXISTS` 사용
- 서브쿼리 결과가 **매우 클 때** → 일반적으로 `EXISTS`가 더 효율적
:::

:::tip LIKE 패턴 작성 팁
`%`(퍼센트)는 길이에 관계없이 **0개 이상의 임의 문자**를 의미합니다.
`_`(밑줄)는 **정확히 1개의 임의 문자**를 의미합니다.
두 가지를 조합해서 복잡한 패턴도 표현할 수 있습니다.

예: `'A___89'` = A로 시작하고 정확히 3자리 임의 문자 후 89로 끝나는 총 6글자
:::

:::danger NULL 비교는 반드시 IS NULL / IS NOT NULL
NULL은 "값이 없는 상태"이므로 어떤 값과 비교해도 항상 `UNKNOWN`(알 수 없음)이 반환됩니다. `= NULL`, `<> NULL`, `> NULL` 같은 표현은 모두 올바른 결과를 반환하지 못합니다. 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용하세요.
:::

:::tip SQL 기본기가 중요한 이유
지금 배우는 표준 SQL의 기본 개념은 Oracle, MySQL, MS-SQL, PostgreSQL 등 어떤 데이터베이스를 사용하더라도 동일하게 적용됩니다. 각 데이터베이스마다 문법이 조금씩 다르더라도, 기본 개념(서브쿼리, 조인, 집합 연산, DML)을 완벽히 이해하고 있으면 새로운 데이터베이스도 빠르게 배울 수 있습니다.
:::

## 📝 핵심 정리

- **서브쿼리(부속 질의문)**: SQL 문 안에 또 다른 SQL 문이 포함된 구조이며, 안쪽이 서브쿼리, 바깥쪽이 메인 쿼리이다.
- **서브쿼리 결과가 0~1건**: 단일행 서브쿼리 → `=`, `>`, `<`, `>=`, `<=`, `<>` 등 일반 비교 연산자 사용.
- **서브쿼리 결과가 여러 건**: 다중행 서브쿼리 → `IN`, `NOT IN`, `ANY`, `ALL`, `EXISTS`, `NOT EXISTS` 사용.
- **IN**: 목록 중 하나라도 일치하면 참 (단일 컬럼 비교).
- **ANY / SOME**: 결과 중 하나라도 조건을 만족하면 참, 반드시 비교 연산자와 함께 사용.
- **ALL**: 결과 전부가 조건을 만족해야 참, 반드시 비교 연산자와 함께 사용.
- **EXISTS**: 서브쿼리 결과가 존재하면 참, 여러 컬럼 반환 시(`SELECT *`) 적합.
- **서브쿼리 안에서 ORDER BY 사용 불가**: 서브쿼리는 조건값을 넘겨주는 역할만 한다.
- **LIKE 연산자**: 문자열 패턴 검색, `%`는 0개 이상 임의 문자, `_`는 정확히 1개 임의 문자.
- **LIKE 사용 시 `=` 사용 불가**: 반드시 `LIKE` 키워드를 써야 한다.
- **NULL 비교**: `= NULL`은 절대 사용 불가, 반드시 `IS NULL` 또는 `IS NOT NULL` 사용.
- **집합 연산**: `UNION`(합집합), `INTERSECT`(교집합), `EXCEPT`(차집합), `ALL` 붙이면 중복 포함.
- **INSERT**: 데이터 삽입, 모든 컬럼 입력 시 컬럼명 생략 가능, 일부 입력 시 컬럼명 필수.
- **DELETE**: 조건을 만족하는 행만 삭제, `WHERE` 없으면 전체 삭제, 테이블 구조는 유지.
- **UPDATE ... SET**: 조건을 만족하는 행의 컬럼 값 수정, 콤마로 여러 컬럼 동시 수정 가능.
- **데이터 무결성**: 개체 무결성(기본키 유일+NOT NULL), 참조 무결성(외래키 참조 유효), 도메인 무결성(값 범위 준수)를 항상 유지해야 한다.
