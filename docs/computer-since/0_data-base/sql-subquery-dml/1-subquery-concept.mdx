# 🔍 부속질의문(서브쿼리)의 개념과 특징

SQL을 배우다 보면 "단순한 SELECT 문 하나로는 원하는 데이터를 꺼내기 어렵다"는 상황을 반드시 마주치게 됩니다. 예를 들어 "박영권 씨와 같은 직급을 가진 직원들을 모두 찾아라"라는 요청이 들어왔을 때, 박영권 씨의 직급을 먼저 조회한 다음, 그 결과를 이용해 다시 한 번 조회해야 하죠. 이럴 때 등장하는 개념이 바로 **부속질의문(서브쿼리, Subquery)** 입니다. 이번 문서에서는 서브쿼리의 개념부터 종류, 연산자, 실전 예제, 그리고 데이터 조작어(DML)와의 연계까지 초보자도 이해할 수 있도록 아주 자세하게 살펴보겠습니다.

---

## 📌 서브쿼리(부속질의문)란 무엇인가?

### 개념 소개

**서브쿼리(Subquery)** 란 하나의 SQL 명령문 **안에 또 다른 SQL 명령문이 포함된 구조**를 말합니다. 우리말로는 **부속질의문(附屬質疑文)** 이라고도 부릅니다.

비유를 들어볼까요? 마치 **러시아 마트료시카 인형**처럼, 큰 인형 안에 작은 인형이 들어 있는 구조입니다. 바깥의 큰 인형이 **메인 쿼리(Main Query)** 이고, 안에 들어 있는 작은 인형이 **서브쿼리(Sub Query)** 입니다.

구체적으로는 아래와 같은 구조입니다.

```sql
-- 메인 쿼리: 직원 이름과 직급을 조회
SELECT emp_name, title
FROM employee
WHERE title = (
    -- 서브쿼리: 박영권의 직급을 먼저 조회
    SELECT title
    FROM employee
    WHERE emp_name = '박영권'
);
```

위 코드에서 `WHERE title = (...)` 안에 들어 있는 `SELECT title FROM employee WHERE emp_name = '박영권'` 부분이 바로 **서브쿼리**입니다. 서브쿼리가 먼저 실행되어 결과를 만들어내면, 그 결과가 바깥 메인 쿼리의 조건으로 들어가게 됩니다.

> **핵심 개념 요약:** 서브쿼리는 메인 쿼리의 WHERE 절(조건 절) 안에 포함된 또 다른 SELECT 문입니다. 서브쿼리가 먼저 실행되고, 그 결과가 메인 쿼리의 조건값으로 사용됩니다.

### 메인 쿼리 vs 서브쿼리 구분

| 구분 | 위치 | 역할 |
|------|------|------|
| **메인 쿼리 (Main Query)** | 가장 바깥쪽 SQL 문 | 최종 결과를 출력하는 주체 |
| **서브쿼리 (Sub Query)** | WHERE 절 안쪽 괄호 내부 | 메인 쿼리의 조건값을 만들어서 넘겨주는 역할 |

## 🤔 왜 서브쿼리가 필요한가? — 배경과 필요성

### 기존 방식의 한계

지난 시간에 배운 `SELECT`, `GROUP BY`, `HAVING`, `ORDER BY` 같은 SQL 명령어들은 **하나의 SQL 문장으로 데이터를 검색하고 처리**하는 방식이었습니다. 이 방식은 단순한 조회에는 충분하지만, **두 단계 이상의 조회가 필요한 경우**에는 한계가 생깁니다.

예를 들어 "박영권 씨와 같은 직급을 가진 직원 목록을 가져와라"라는 요청을 처리하려면 어떻게 해야 할까요?

1. **1단계:** 박영권 씨의 직급을 먼저 조회한다.
2. **2단계:** 그 직급과 동일한 직원들을 조회한다.

기존 방식으로는 이 두 단계를 **따로따로 SQL을 실행**해야 합니다. 즉, 첫 번째 SQL로 박영권의 직급을 조회해서 눈으로 확인한 뒤, 그 값을 두 번째 SQL에 직접 입력해야 하죠. 이는 매우 번거롭고 **자동화가 불가능**합니다.

서브쿼리를 사용하면 이 두 단계를 **하나의 SQL 문장으로** 해결할 수 있습니다.

### 서브쿼리 vs 조인(JOIN) — 무엇이 다른가?

서브쿼리와 비슷한 역할을 하는 것으로 **조인(JOIN)** 이 있습니다. 둘 다 두 개 이상의 테이블을 연결해서 데이터를 가져오는 방식이지만, 결과 출력 방식에서 큰 차이가 있습니다.

| 비교 항목 | 서브쿼리 (Subquery) | 조인 (JOIN) |
|-----------|---------------------|-------------|
| **결과 출력** | 메인 쿼리 테이블의 컬럼만 출력 | 두 테이블 모두의 컬럼 출력 가능 |
| **서브 테이블 역할** | 조건 비교용으로만 사용 | 결과 출력에도 참여 |
| **가독성** | 논리적으로 이해하기 쉬움 | 복잡한 관계 표현에 유리 |
| **성능** | 단순 조건 비교 시 효율적일 수 있음 | 복잡한 다중 테이블 조회에 유리 |

:::info 중요한 차이점
조인은 두 개의 테이블을 연결해서 **양쪽 테이블의 컬럼 모두**를 결과로 볼 수 있습니다. 반면 서브쿼리는 두 개의 테이블을 사용하더라도 **최종 결과는 메인 쿼리의 테이블 컬럼만** 출력됩니다. 서브 쿼리에서 사용한 테이블은 오직 조건 비교를 위해서만 활용될 뿐, 그 테이블의 데이터는 최종 결과에 나타나지 않습니다.
:::

## ⚙️ 서브쿼리의 핵심 원리와 규칙

### 서브쿼리 작성 시 반드시 지켜야 할 규칙

서브쿼리를 처음 접하는 분들이 실수하기 쉬운 규칙들이 있습니다. 하나씩 꼼꼼히 살펴보겠습니다.

**규칙 1: 서브쿼리는 반드시 괄호( )로 감싼다**

서브쿼리는 메인 쿼리의 WHERE 절 안에서 하나의 값 또는 값의 집합처럼 동작해야 하기 때문에, 반드시 괄호로 묶어서 사용해야 합니다. 괄호가 없으면 SQL 파서(SQL 문장을 해석하는 도구)가 어디까지가 서브쿼리인지 구분할 수 없습니다.

```sql
-- ❌ 잘못된 예시: 괄호 없이 서브쿼리 사용
SELECT emp_name
FROM employee
WHERE title = SELECT title FROM employee WHERE emp_name = '박영권';
-- 오류 발생! 괄호로 감싸야 합니다.

-- ✅ 올바른 예시: 괄호로 서브쿼리 감싸기
SELECT emp_name
FROM employee
WHERE title = (SELECT title FROM employee WHERE emp_name = '박영권');
-- 정상 동작
```

**규칙 2: 서브쿼리 안에서 ORDER BY(정렬)를 사용할 수 없다**

서브쿼리의 목적은 메인 쿼리에게 **조건값을 넘겨주는 것**입니다. 결과를 정렬해서 보여주는 것이 아니라, 단순히 값을 반환하는 역할이기 때문에 `ORDER BY`는 사용할 수 없습니다.

```sql
-- ❌ 잘못된 예시: 서브쿼리 안에 ORDER BY 사용
SELECT emp_name
FROM employee
WHERE title = (
    SELECT title
    FROM employee
    WHERE emp_name = '박영권'
    ORDER BY title  -- 오류! 서브쿼리 안에서 정렬 불가
);

-- ✅ 올바른 예시: ORDER BY 제거
SELECT emp_name
FROM employee
WHERE title = (
    SELECT title
    FROM employee
    WHERE emp_name = '박영권'
    -- ORDER BY 없이 사용
);
```

:::warning 주의사항
서브쿼리 안에서는 `ORDER BY`를 사용할 수 없습니다. 정렬이 필요하다면 메인 쿼리의 마지막에 `ORDER BY`를 붙이세요.
:::

### 단일 행 서브쿼리 vs 다중 행 서브쿼리

서브쿼리를 이해하는 데 있어 **가장 중요한 개념**이 바로 이것입니다. 서브쿼리가 반환하는 결과의 **행(row) 수**에 따라 사용해야 하는 비교 연산자가 달라집니다.

**단일 행 서브쿼리 (Single-Row Subquery)**

서브쿼리의 실행 결과가 **1건 이하(0건 또는 1건)** 일 때를 말합니다. 이 경우에는 우리가 흔히 아는 일반 비교 연산자를 사용합니다.

| 연산자 | 의미 |
|--------|------|
| `=` | 같다 |
| `>` | 크다 |
| `<` | 작다 |
| `>=` | 크거나 같다 |
| `<=` | 작거나 같다 |
| `<>` 또는 `!=` | 같지 않다 |

**다중 행 서브쿼리 (Multi-Row Subquery)**

서브쿼리의 실행 결과가 **여러 건(2건 이상)** 일 때를 말합니다. 이 경우에는 **다중 행 비교 연산자**를 사용해야 합니다. 여러 개의 값과 단순히 `=`로 비교하면 오류가 발생하기 때문입니다.

```
마치 택배 배송처럼 생각해보세요.
단일 행 서브쿼리 = 받는 사람이 딱 한 명인 택배 → 정확한 주소(=) 하나로 비교
다중 행 서브쿼리 = 받는 사람이 여러 명인 택배 목록 → 목록(IN) 안에 포함되는지 비교
```

다중 행 비교 연산자는 다음 섹션에서 자세히 설명합니다.

## 🔢 다중 행 비교 연산자 완전 정복

다중 행 서브쿼리에서 사용하는 연산자들을 하나씩 살펴보겠습니다. 각 연산자마다 실생활 예시와 함께 설명하겠습니다.

### IN — "이 목록 안에 있으면 참"

**IN**은 서브쿼리가 반환한 여러 값들 중 **하나라도 일치하면 참(TRUE)** 이 되는 연산자입니다. 마치 "이 명단에 내 이름이 있는지 확인하는 것"과 같습니다.

```sql
-- 예시: 영업부 또는 개발부에 근무하는 직원의 이름 검색
SELECT emp_name          -- 직원 이름을 출력
FROM employee            -- 직원 테이블에서
WHERE dno IN (           -- 부서 번호가 아래 목록 안에 포함되면
    SELECT dno           -- 부서 번호를 가져옴
    FROM department      -- 부서 테이블에서
    WHERE dept_name = '영업부'   -- 부서명이 영업부이거나
       OR dept_name = '개발부'   -- 개발부인 경우
);
```

**IN 연산자 동작 예시:**

서브쿼리 결과가 `(3, 4, 2, 6)` 이라면:
- 비교값이 `3`이면 → 목록에 있으므로 **참(TRUE)**
- 비교값이 `7`이면 → 목록에 없으므로 **거짓(FALSE)**

### NOT IN — "이 목록 안에 없어야 참"

**NOT IN**은 IN의 반대입니다. 서브쿼리가 반환한 값들 중 **하나도 일치하지 않아야 참(TRUE)** 이 됩니다.

```sql
-- 예시: 영업부와 개발부에 속하지 않는 직원의 이름 검색
SELECT emp_name
FROM employee
WHERE dno NOT IN (        -- 부서 번호가 목록 안에 없으면
    SELECT dno
    FROM department
    WHERE dept_name = '영업부'
       OR dept_name = '개발부'
);
```

### ANY / SOME — "하나라도 만족하면 참"

**ANY**와 **SOME**은 같은 의미입니다. 서브쿼리 결과 중 **하나 이상이 조건을 만족하면 참(TRUE)** 이 됩니다. 반드시 `>`, `<`, `=` 등의 비교 연산자와 함께 사용합니다.

:::note ANY와 SOME
`ANY`와 `SOME`은 완전히 동일한 기능을 합니다. SQL 표준에서는 둘 다 허용되며, 어떤 것을 써도 결과는 같습니다.
:::

```sql
-- 예시: 급여가 서브쿼리 결과 중 하나라도 300만원보다 큰 직원 검색
-- 서브쿼리 결과: (250만원, 300만원, 400만원)
SELECT emp_name
FROM employee
WHERE salary > ANY (     -- 서브쿼리 결과 중 하나라도 salary보다 작으면 참
    SELECT salary
    FROM employee
    WHERE dept_name = '영업부'
);
-- salary가 400만원 초과라면: 250, 300, 400 중 하나라도 salary보다 작으면 참
-- salary가 300만원이라면: 250이 300보다 작으니까 참
-- salary가 200만원이라면: 250, 300, 400 모두 200보다 크니까 거짓
```

**ANY 연산자 동작 예시 (서브쿼리 결과: 250만원, 300만원, 400만원):**

| 비교 조건 | 판단 | 이유 |
|-----------|------|------|
| `salary > ANY (250, 300, 400)` where salary = 400만원 | ✅ 참 | 400 > 250 하나라도 만족 |
| `salary > ANY (250, 300, 400)` where salary = 300만원 | ✅ 참 | 300 > 250 하나라도 만족 |
| `salary > ANY (250, 300, 400)` where salary = 200만원 | ❌ 거짓 | 200은 250, 300, 400 모두보다 작음 |

### ALL — "모든 값을 만족해야 참"

**ALL**은 서브쿼리가 반환한 **모든 값에 대해 조건이 참이어야** 최종적으로 참(TRUE)이 됩니다. 역시 비교 연산자와 함께 사용합니다.

```sql
-- 예시: 서브쿼리 결과의 모든 값보다 급여가 큰 직원 검색
-- 서브쿼리 결과: (150만원, 250만원, 300만원)
SELECT emp_name
FROM employee
WHERE salary > ALL (    -- 서브쿼리의 모든 값보다 커야 참
    SELECT salary
    FROM employee
    WHERE dno = 'd2'
);
-- salary가 400만원이라면: 150, 250, 300 모두보다 크므로 참
-- salary가 280만원이라면: 150보다 크지만 300보다는 작으므로 거짓
```

**ALL 연산자 동작 예시 (서브쿼리 결과: 250만원, 300만원, 400만원):**

| 비교 조건 | 판단 | 이유 |
|-----------|------|------|
| `salary > ALL (250, 300, 400)` where salary = 500만원 | ✅ 참 | 500 > 250, 300, 400 모두 만족 |
| `salary > ALL (250, 300, 400)` where salary = 300만원 | ❌ 거짓 | 300은 300보다 크지 않음(같음) |
| `= ALL (250, 300, 400)` where salary = 300만원 | ❌ 거짓 | 300은 250, 400과 같지 않음 |
| `> ALL (150, 250, 300)` where salary = 400만원 | ✅ 참 | 400 > 150, 250, 300 모두 만족 |

:::tip ALL 사용 팁
`> ALL (서브쿼리)` 는 "서브쿼리 결과 중 최댓값보다 크다"와 동일한 의미입니다. `< ALL (서브쿼리)` 는 "서브쿼리 결과 중 최솟값보다 작다"와 동일합니다.
:::

### EXISTS — "결과가 존재하면 참"

**EXISTS**는 서브쿼리의 결과가 **한 건이라도 존재하면 참(TRUE)**, 아무것도 없으면 거짓(FALSE)이 됩니다. 특히 서브쿼리에서 **여러 개의 컬럼**을 반환해야 할 때 유용합니다.

IN과 EXISTS의 가장 큰 차이는, IN은 반환 컬럼이 하나일 때 주로 쓰고, **EXISTS는 여러 컬럼을 한꺼번에 반환하는 서브쿼리**에서 활용된다는 점입니다.

```sql
-- ❌ 여러 컬럼 반환 시 IN 사용 불가
SELECT emp_name
FROM employee E
WHERE (E.dno, E.emp_name) IN (   -- 여러 컬럼을 IN으로 비교하기 어려움
    SELECT *
    FROM department D
    WHERE D.dno = E.dno
    AND D.dept_name IN ('영업부', '개발부')
);

-- ✅ EXISTS 사용: 여러 컬럼 결과도 자연스럽게 처리
SELECT emp_name           -- 직원 이름 출력 (메인 쿼리)
FROM employee E           -- 직원 테이블 (별명: E)
WHERE EXISTS (            -- 서브쿼리 결과가 존재하면 참
    SELECT *              -- 모든 컬럼 반환 (존재 여부만 확인)
    FROM department D     -- 부서 테이블 (별명: D)
    WHERE D.dno = E.dno   -- 조인 조건: 부서 번호가 같고
    AND D.dept_name IN ('영업부', '개발부')  -- 영업부 또는 개발부인 경우
);
```

위 코드를 한 줄씩 설명하면:
- `SELECT emp_name FROM employee E` : 직원 테이블에서 직원 이름을 출력하되, 테이블에 'E'라는 별명을 붙임
- `WHERE EXISTS (...)` : 괄호 안 서브쿼리 결과가 한 건이라도 있으면 해당 직원을 결과에 포함
- `SELECT * FROM department D` : 부서 테이블 전체를 조회 (EXISTS는 값 자체보다 존재 여부만 따지므로 `*` 사용)
- `WHERE D.dno = E.dno` : 직원 테이블의 부서 번호와 부서 테이블의 부서 번호가 일치하는 조건 (조인 조건)
- `AND D.dept_name IN ('영업부', '개발부')` : 그 중 부서명이 영업부 또는 개발부인 것만 필터링

**다중 행 비교 연산자 전체 정리:**

| 연산자 | 비교 연산자 필요 여부 | 의미 | 결과가 참이 되는 조건 |
|--------|----------------------|------|----------------------|
| `IN` | 불필요 (독자 사용) | 목록 안에 포함 | 하나라도 일치하면 참 |
| `NOT IN` | 불필요 (독자 사용) | 목록 안에 미포함 | 하나도 일치하지 않으면 참 |
| `ANY` / `SOME` | **필수** (예: `> ANY`) | 하나 이상 만족 | 하나라도 조건 충족 시 참 |
| `ALL` | **필수** (예: `> ALL`) | 모두 만족 | 모든 값이 조건 충족 시 참 |
| `EXISTS` | 불필요 (독자 사용) | 존재 여부 확인 | 결과가 1건이라도 있으면 참 |
| `NOT EXISTS` | 불필요 (독자 사용) | 미존재 여부 확인 | 결과가 0건이면 참 |

:::danger 절대 하면 안 되는 것
다중 행 서브쿼리(결과가 여러 건)에 단일 행 비교 연산자(`=`, `>`, `<` 등)를 단독으로 사용하면 **오류가 발생**합니다. 서브쿼리 결과가 여러 행이면 반드시 `IN`, `ANY`, `ALL`, `EXISTS` 등의 다중 행 비교 연산자를 사용하세요.

```sql
-- ❌ 절대 금지: 다중 행 결과에 = 사용
SELECT emp_name
FROM employee
WHERE dno = (              -- 서브쿼리 결과가 여러 건이면 오류 발생!
    SELECT dno
    FROM department
    WHERE dept_name IN ('영업부', '개발부')  -- 결과가 2건 이상
);

-- ✅ 올바른 방법: IN 사용
SELECT emp_name
FROM employee
WHERE dno IN (
    SELECT dno
    FROM department
    WHERE dept_name IN ('영업부', '개발부')
);
```
:::

## 💡 서브쿼리 실전 예제

### 예제 1: 단일 행 서브쿼리 — 특정 사람과 같은 직급 조회

"박영권 씨와 같은 직급을 가진 직원들의 이름과 직급을 검색하라"

```sql
-- 박영권과 동일 직급의 직원 이름과 직급 검색
SELECT emp_name,   -- 직원 이름 출력
       title       -- 직급 출력
FROM employee      -- 직원 테이블에서
WHERE title = (    -- 직급이 서브쿼리 결과와 같으면
    SELECT title   -- 직급 조회
    FROM employee  -- 직원 테이블에서
    WHERE emp_name = '박영권'  -- 박영권의 직급을 찾아 메인 쿼리의 조건으로 전달
);
```

**처리 순서:**
1. 서브쿼리 실행: `SELECT title FROM employee WHERE emp_name = '박영권'` → 예: `'과장'` 반환
2. 메인 쿼리 실행: `SELECT emp_name, title FROM employee WHERE title = '과장'`
3. 최종 결과: 직급이 '과장'인 모든 직원의 이름과 직급 출력

### 예제 2: 다중 행 서브쿼리(IN) — 특정 부서 직원 조회

"영업부 또는 개발부에 근무하는 직원의 이름을 검색하라"

```sql
-- 영업부 또는 개발부 소속 직원 이름 검색
SELECT emp_name        -- 직원 이름 출력 (메인 쿼리 테이블의 컬럼만 출력됨)
FROM employee          -- 직원 테이블이 메인 쿼리의 테이블
WHERE dno IN (         -- 부서 번호가 아래 서브쿼리 결과 목록에 포함되면
    SELECT dno         -- 부서 번호 반환
    FROM department    -- 부서 테이블에서 (서브쿼리 테이블 - 조건 비교용으로만 사용)
    WHERE dept_name = '영업부'   -- 영업부이거나
       OR dept_name = '개발부'   -- 개발부인 부서의 번호만 선택
);
```

이 예제에서 **서브쿼리의 결과가 여러 건**(영업부 번호, 개발부 번호)이므로 `IN` 연산자를 사용합니다. 최종 결과에는 `department` 테이블의 컬럼은 나타나지 않고, `employee` 테이블의 `emp_name`만 출력됩니다. 이것이 조인과 다른 서브쿼리의 핵심 특징입니다.

### 예제 3: LIKE 연산자를 활용한 서브쿼리 — A동 부서 직원 조회

"A동에 위치하는 부서의 직원 이름을 검색하라"

```sql
-- A동에 위치한 부서의 직원 이름 검색
SELECT emp_name          -- 직원 이름 출력
FROM employee            -- 직원 테이블에서
WHERE dno IN (           -- 부서 번호가 아래 결과에 포함되면
    SELECT dno           -- 부서 번호 반환
    FROM department      -- 부서 테이블에서
    WHERE dept_location LIKE 'A%'  -- 부서 위치가 A로 시작하는 경우 (A동, A관 등 모두 포함)
);
-- LIKE 'A%' 의미: A로 시작하는 모든 문자열 (A1, A동, A100 등 가능)
```

### 예제 4: EXISTS를 활용한 서브쿼리

"영업부 또는 개발부에 근무하는 직원의 이름을 EXISTS로 검색하라"

```sql
-- EXISTS를 사용한 부서별 직원 검색
SELECT emp_name           -- 직원 이름 출력
FROM employee E           -- 직원 테이블 (별명: E)
WHERE EXISTS (            -- 서브쿼리 결과가 존재하면 (1건이라도 있으면 참)
    SELECT *              -- 모든 컬럼 반환 (존재 여부 확인만 하므로 * 사용)
    FROM department D     -- 부서 테이블 (별명: D)
    WHERE D.dno = E.dno   -- 직원의 부서번호와 부서테이블의 부서번호가 같고 (조인 조건)
      AND D.dept_name IN ('영업부', '개발부')  -- 영업부 또는 개발부인 경우
);
```

:::tip EXISTS vs IN 선택 기준
- 서브쿼리 결과가 **단일 컬럼**이면 → `IN` 사용
- 서브쿼리 결과가 **여러 컬럼**이거나, 조인 조건을 서브쿼리 안에 포함시키고 싶을 때 → `EXISTS` 사용
:::

## 🔤 LIKE 연산자와 NULL 검색

### LIKE — 문자열 패턴 검색

`LIKE`는 **문자열을 패턴으로 검색**할 때 사용하는 연산자입니다. 정확한 값을 모를 때, 특정 문자로 시작하거나 끝나는 데이터를 찾을 때 활용됩니다.

**와일드카드 문자 두 가지:**

| 와일드카드 | 의미 | 예시 |
|-----------|------|------|
| `%` | 길이 제한 없이 아무 문자 | `'A%'` → A로 시작하는 모든 문자열 (A, A1, Abc, A동 등) |
| `_` (언더바) | 정확히 한 글자 | `'K_O'` → K로 시작하고 O로 끝나는 3글자 단어 (KAO, KBO 등) |

```sql
-- ✅ 예시 1: 전화번호 국번이 777인 직원 검색
SELECT emp_name           -- 직원 이름 출력
FROM employee             -- 직원 테이블에서
WHERE phone LIKE '777%';  -- 전화번호가 777로 시작하면 (뒤에 몇 자리든 상관없음)
-- LIKE를 쓰는 이유: % 와일드카드 사용 시 반드시 LIKE 연산자 사용 (= 불가)

-- ✅ 예시 2: 제품 코드가 총 6자리, 첫 글자 A, 마지막 두 글자 89인 제품 검색
SELECT product_name       -- 제품명 출력
FROM product              -- 제품 테이블에서
WHERE product_code LIKE 'A___89';
-- A: 첫 글자는 A
-- ___: 가운데 3자리는 아무 문자나 가능 (언더바 3개 = 3글자)
-- 89: 마지막 두 글자는 반드시 89
-- 총 글자 수: 1(A) + 3(_) + 2(89) = 6자리
```

:::danger 절대 하면 안 되는 것
`%`나 `_`를 사용할 때 `=` 연산자를 사용하면 안 됩니다. 반드시 `LIKE` 키워드를 사용해야 합니다.

```sql
-- ❌ 잘못된 예시
WHERE phone = '777%';   -- 오류! % 와일드카드는 LIKE와만 사용 가능

-- ✅ 올바른 예시
WHERE phone LIKE '777%';
```
:::

### IS NULL / IS NOT NULL — NULL 값 검색

**NULL**은 "값이 없음"을 나타내는 특별한 값입니다. 0도 아니고, 빈 문자열도 아닙니다. NULL은 "아직 결정되지 않은 값"이라고 이해하면 됩니다.

NULL 값을 찾을 때는 절대 `= NULL`을 사용해서는 안 됩니다. 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용해야 합니다.

```sql
-- ❌ 잘못된 예시: = NULL 사용 (허용되지 않음)
SELECT emp_no, emp_name
FROM employee
WHERE dno = NULL;   -- 오류! NULL은 = 로 비교 불가

-- ✅ 올바른 예시: IS NULL 사용
-- 부서가 아직 결정되지 않은 직원(부서번호가 NULL인 직원)의 번호와 이름 검색
SELECT emp_no,       -- 직원 번호 출력
       emp_name      -- 직원 이름 출력
FROM employee        -- 직원 테이블에서
WHERE dno IS NULL;   -- 부서 번호가 NULL인 경우 (부서 미배정 직원)

-- ✅ 올바른 예시: IS NOT NULL 사용
-- 부서가 이미 결정된 직원(부서번호가 NULL이 아닌 직원)의 번호와 이름 검색
SELECT emp_no,
       emp_name
FROM employee
WHERE dno IS NOT NULL;  -- 부서 번호가 NULL이 아닌 경우 (부서 배정 완료 직원)
```

| 조건 | 문법 | 의미 |
|------|------|------|
| NULL 검색 | `IS NULL` | 값이 없는(미입력) 행 검색 |
| NOT NULL 검색 | `IS NOT NULL` | 값이 있는 행 검색 |
| ❌ 사용 금지 | `= NULL` | SQL에서 허용되지 않음 |

## 🔗 집합 연산자 — UNION, INTERSECT, EXCEPT

SQL에서는 관계대수(Relational Algebra)의 집합 연산을 SQL 명령어로 구현할 수 있습니다. **합집합(UNION)**, **교집합(INTERSECT)**, **차집합(EXCEPT)** 이 그것입니다.

### 집합 연산의 기본 조건 — 호환성(Union Compatibility)

집합 연산을 수행하려면 두 테이블이 **합병 가능(Union Compatible)** 해야 합니다. 즉, 두 테이블의 **컬럼 수(차수)와 각 컬럼의 데이터 타입이 동일**해야 합니다. 마치 두 개의 바구니를 합치려면, 두 바구니가 같은 종류의 물건을 담고 있어야 한다는 것과 같습니다.

또한, 기본적으로 SQL 집합 연산은 **중복 튜플을 자동으로 제거**합니다. 뒤에 `ALL`을 붙이면 중복을 허용할 수 있습니다.

### UNION — 합집합

두 쿼리의 결과를 **모두 합쳐서** 출력합니다. 기본적으로 중복 제거.

```sql
-- R1: (1,2,3,5), R2: (1,2,3,4) 데이터가 있을 때

-- UNION (중복 제거): 결과 → 1,2,3,4,5
SELECT dno FROM employee WHERE emp_no = 100   -- 직원번호 100의 부서번호
UNION                                          -- 두 결과를 합집합 (중복 제거)
SELECT dno FROM department WHERE dept_name = '기획과';  -- 기획과의 부서번호

-- UNION ALL (중복 포함): 결과 → 1,1,2,2,3,3,4,5
SELECT dno FROM employee WHERE emp_no = 100
UNION ALL                                      -- 두 결과를 합집합 (중복 포함)
SELECT dno FROM department WHERE dept_name = '기획과';
```

### INTERSECT — 교집합

두 쿼리 결과에서 **공통으로 존재하는 행만** 출력합니다.

```sql
-- 교집합: 두 결과에 모두 있는 데이터만 출력
-- R1: (1,2,3,5), R2: (1,2,3,4) → 결과: 1,2,3
SELECT dno FROM employee
INTERSECT   -- 교집합: 두 결과 모두에 존재하는 부서번호만 출력
SELECT dno FROM department;
```

### EXCEPT — 차집합

첫 번째 쿼리 결과에서 두 번째 쿼리 결과를 **빼고 남은 것**을 출력합니다.

```sql
-- R1-R2 차집합: R1에만 있는 데이터 → 결과: 5
SELECT dno FROM employee       -- R1
EXCEPT                          -- R1에서 R2를 뺀 차집합
SELECT dno FROM department;    -- R2

-- R2-R1 차집합: R2에만 있는 데이터 → 결과: 4
SELECT dno FROM department     -- R2
EXCEPT
SELECT dno FROM employee;      -- R1
```

| 연산 | SQL 명령어 | ALL 옵션 | 의미 |
|------|-----------|----------|------|
| 합집합 | `UNION` | `UNION ALL` | 두 결과 합치기 (ALL은 중복 포함) |
| 교집합 | `INTERSECT` | `INTERSECT ALL` | 공통 데이터만 |
| 차집합 | `EXCEPT` | `EXCEPT ALL` | 첫 번째에만 있는 데이터 |

## 🛠️ 데이터 조작어(DML) — INSERT, DELETE, UPDATE

SQL의 데이터 조작어(DML, Data Manipulation Language)는 **데이터를 삽입(INSERT), 삭제(DELETE), 수정(UPDATE)** 하는 명령어입니다. 서브쿼리와 결합하면 훨씬 강력하게 활용할 수 있습니다.

### INSERT — 데이터 삽입

`INSERT`는 테이블에 새로운 행(레코드)을 추가하는 명령입니다. 마치 빈 서랍에 물건을 새로 넣는 것과 같습니다.

**기본 문법:**

```sql
-- 기본 삽입: 모든 컬럼에 값 입력 (컬럼명 생략 가능)
INSERT INTO department        -- department 테이블에 삽입
VALUES ('d5', '연구', NULL);  -- 부서번호, 부서명, 위치 (위치는 아직 미정이라 NULL)
-- 모든 컬럼에 값을 입력할 때는 컬럼명 생략 가능

-- 일부 컬럼만 삽입: 컬럼명 반드시 명시
INSERT INTO department (dno, dept_name)  -- dno와 dept_name 컬럼만 입력
VALUES ('d6', '홍보부');                 -- 위치(dept_location)는 생략 → 자동으로 NULL
-- 일부 컬럼만 입력할 때는 어떤 컬럼에 넣는지 반드시 명시해야 함
```

**SELECT를 이용한 대량 삽입:**

```sql
-- SELECT 결과를 다른 테이블에 한꺼번에 복사하여 삽입
INSERT INTO high_salary (name, title, salary)  -- high_salary 테이블의 name, title, salary 컬럼에
SELECT emp_name,   -- 직원 이름
       title,      -- 직급
       salary      -- 급여
FROM employee      -- 직원 테이블에서
WHERE salary >= 3000000;  -- 급여가 300만원 이상인 직원 데이터만 복사하여 삽입
```

**INSERT 주요 특징:**

```sql
-- 한 번에 여러 행 삽입 (콤마로 구분)
INSERT INTO department
VALUES ('d7', '법무', 'B동'),   -- 첫 번째 행
       ('d8', '총무', 'C동'),   -- 두 번째 행
       ('d9', '인사', 'A동');   -- 세 번째 행
```

:::warning INSERT 주의사항
- **기본키(Primary Key)** 컬럼에는 NULL 값 입력 불가
- **기본키** 컬럼에는 중복 값 입력 불가
- `NOT NULL` 제약이 있는 컬럼은 반드시 값을 입력해야 함
- 하나의 INSERT 문으로 **하나의 테이블에만** 삽입 가능 (여러 테이블 동시 삽입 불가)
- INSERT 시 ORDER BY(정렬)는 사용 불가
:::

### DELETE — 데이터 삭제

`DELETE`는 테이블에서 행(레코드)을 삭제하는 명령입니다. `DROP`이 테이블 자체를 없애는 것이라면, `DELETE`는 테이블 구조는 그대로 두고 **데이터(내용)만 삭제**하는 것입니다.

```sql
-- 특정 조건의 행 삭제
DELETE FROM department        -- department 테이블에서 삭제
WHERE dno = 'd4';             -- 부서번호가 d4인 행만 삭제 (테이블 구조는 유지)

-- 특정 직원번호의 데이터 삭제
DELETE FROM emp               -- emp 테이블에서
WHERE emp_no = 7902;          -- 직원번호가 7902인 행 삭제

-- 평균급여보다 적게 받는 직원 모두 삭제
DELETE FROM emp               -- emp 테이블에서
WHERE salary < (              -- 급여가 서브쿼리 결과보다 작은 행 삭제
    SELECT AVG(salary)        -- 평균 급여 계산 (서브쿼리)
    FROM emp
);

-- 테이블의 모든 행 삭제 (조건 없음 = 전체 삭제)
DELETE FROM emp;              -- WHERE 절이 없으면 모든 행 삭제! (테이블 구조는 남음)

-- 서브쿼리를 이용한 삭제: 홍길동과 같은 부서의 모든 직원 삭제
DELETE FROM employee          -- 직원 테이블에서
WHERE dno IN (                -- 부서 번호가 서브쿼리 결과 목록에 있으면
    SELECT dno                -- 홍길동의 부서 번호를 조회
    FROM employee
    WHERE emp_name = '홍길동' -- 홍길동의 부서번호를 찾아 삭제 조건으로 사용
);
```

:::danger 절대 하면 안 되는 것
`WHERE` 절 없이 `DELETE FROM 테이블명;`을 실행하면 해당 테이블의 **모든 데이터가 삭제**됩니다. 실행 전 반드시 조건을 확인하세요. 또한 참조 무결성(외래키 제약)으로 인해 다른 테이블에서 참조 중인 데이터는 삭제 시 오류가 발생할 수 있으니 주의하세요.
:::

### UPDATE — 데이터 수정

`UPDATE`는 기존 데이터를 새로운 값으로 **수정(갱신)** 하는 명령입니다. `SET` 키워드로 변경할 컬럼과 값을 지정하고, `WHERE` 절로 수정 대상을 한정합니다.

```sql
-- 특정 사원의 부서와 급여 동시 수정
UPDATE employee            -- employee 테이블을 수정
SET dno = 3,               -- 부서 번호를 3으로 변경
    salary = salary * 1.05 -- 급여를 현재의 5% 인상 (현재 급여 × 1.05)
WHERE emp_no = 2106;       -- 사원번호가 2106인 사원만 수정

-- 특정 부서 직원 급여 10% 인상
UPDATE employee            -- employee 테이블 수정
SET salary = salary * 1.1  -- 급여 10% 인상
WHERE dno = 20;            -- 부서번호가 20인 직원만 대상

-- 모든 직원의 입사일을 오늘 날짜로 변경
UPDATE emp                 -- emp 테이블 수정
SET hire_date = SYSDATE;   -- 입사일을 시스템 현재 날짜(SYSDATE)로 변경
-- WHERE 절 없음 = 모든 행 수정

-- 서브쿼리를 이용한 UPDATE: 컴퓨터과 학생의 기말고사 5점 감점
UPDATE enroll              -- 수강(enroll) 테이블 수정
SET final = final - 5      -- 기말고사 점수를 5점 감점
WHERE sno IN (             -- 학번이 서브쿼리 결과에 포함된 학생만
    SELECT sno             -- 컴퓨터과 학생의 학번 조회
    FROM student           -- 학생 테이블에서
    WHERE department = '컴퓨터'  -- 학과가 컴퓨터인 학생
);
```

위 서브쿼리 UPDATE 예시를 한 줄씩 설명하면:
- `UPDATE enroll` : 수강 테이블을 수정 대상으로 지정
- `SET final = final - 5` : 기말고사(final) 점수에서 5점을 뺀 값으로 업데이트
- `WHERE sno IN (...)` : 학번이 서브쿼리 결과 목록에 있는 행만 수정
- `SELECT sno FROM student WHERE department = '컴퓨터'` : 학과가 컴퓨터인 학생들의 학번 목록을 반환

:::tip UPDATE 활용 팁
- `SET` 절에 콤마(`,`)로 구분하여 여러 컬럼을 동시에 수정할 수 있습니다.
- 급여 인상은 `salary = salary * 1.1` (10% 인상), 감점은 `final = final - 5` 처럼 현재 값을 기준으로 연산합니다.
- 서브쿼리를 `WHERE` 절에 활용하면 다른 테이블의 조건을 기반으로 수정할 수 있어 매우 강력합니다.
:::

## ⚠️ 데이터 무결성 — DML 수행 시 반드시 지켜야 할 것

데이터를 삽입·삭제·수정하는 작업은 데이터베이스의 **무결성(Integrity, 데이터의 정확성과 일관성)** 을 위협할 수 있습니다. 처음에 완벽하게 설계된 데이터베이스도, 삽입·삭제·수정이 반복되다 보면 데이터가 엉키거나 모순이 생길 수 있습니다. 마치 새 컴퓨터는 잘 돌아가지만, 프로그램을 깔았다 지웠다를 반복하면 성능이 저하되는 것처럼요.

따라서 DML 작업 시 아래 세 가지 무결성이 반드시 유지되어야 합니다.

### 개체 무결성 (Entity Integrity)

**기본키(Primary Key)** 에 관한 규칙입니다.
- 기본키는 **NULL 값을 가질 수 없습니다.** 기본키는 각 행을 유일하게 식별하는 값인데, NULL은 "값 없음"을 의미하므로 식별이 불가능해집니다.
- 기본키는 **중복 값을 가질 수 없습니다.** 동일한 기본키 값을 가진 두 행이 존재하면 어느 행을 가리키는지 알 수 없게 됩니다.

### 참조 무결성 (Referential Integrity)

**외래키(Foreign Key)** 에 관한 규칙입니다.
- 외래키는 **참조하는 테이블에 존재하지 않는 값을 가질 수 없습니다.** 예를 들어, 직원 테이블의 부서번호(외래키)가 부서 테이블에 없는 번호를 가리키면 안 됩니다.
- 참조 무결성 보장을 위해 삭제·수정 시 `RESTRICT`, `CASCADE`, `SET NULL`, `SET DEFAULT` 등의 옵션을 활용합니다.

### 도메인 무결성 (Domain Integrity)

**속성(컬럼)의 값 범위**에 관한 규칙입니다.
- 각 속성이 정해진 범위와 타입의 값만 가져야 합니다.
- 예: 점수는 0~100, 급여는 100만원~300만원, 성별은 'M' 또는 'F' 등

> **무결성이 왜 중요한가?** 데이터베이스는 단순히 데이터를 저장하는 창고가 아닙니다. 기업의 중요한 의사결정 자료, 고객 정보, 금융 거래 내역 등 핵심 데이터를 담고 있습니다. 무결성이 깨지면 잘못된 데이터를 기반으로 분석이 이루어지고, 이는 심각한 비즈니스 문제로 이어질 수 있습니다. 따라서 DML 작업 시 무결성 유지는 선택이 아닌 필수입니다.

## 📝 주의사항과 실전 팁

### 서브쿼리 사용 시 흔한 실수 모음

**실수 1: 다중 행 결과에 단일 행 연산자 사용**

```sql
-- ❌ 오류: 서브쿼리 결과가 여러 행인데 = 사용
SELECT emp_name FROM employee
WHERE dno = (SELECT dno FROM department WHERE dept_name LIKE 'A%');
-- dept_name이 A로 시작하는 부서가 여러 개면 오류 발생

-- ✅ 올바른 방법: IN 사용
SELECT emp_name FROM employee
WHERE dno IN (SELECT dno FROM department WHERE dept_name LIKE 'A%');
```

**실수 2: 서브쿼리에 ORDER BY 사용**

```sql
-- ❌ 오류: 서브쿼리 안에 ORDER BY 사용
SELECT emp_name FROM employee
WHERE dno = (
    SELECT dno FROM department
    WHERE dept_name = '영업부'
    ORDER BY dno  -- 오류! 서브쿼리에 ORDER BY 불가
);

-- ✅ 올바른 방법: ORDER BY는 메인 쿼리 맨 마지막에
SELECT emp_name FROM employee
WHERE dno = (SELECT dno FROM department WHERE dept_name = '영업부')
ORDER BY emp_name;  -- 정렬은 메인 쿼리 마지막에
```

**실수 3: NULL 비교에 = 사용**

```sql
-- ❌ 오류: NULL 비교에 = 사용
SELECT * FROM employee WHERE dno = NULL;   -- 결과 없음 (오류는 아니지만 항상 거짓)

-- ✅ 올바른 방법: IS NULL 사용
SELECT * FROM employee WHERE dno IS NULL;  -- 정상 동작
```

**실수 4: 서브쿼리 괄호 누락**

```sql
-- ❌ 오류: 괄호 없음
SELECT emp_name FROM employee
WHERE dno = SELECT dno FROM department WHERE dept_name = '영업부';
-- 문법 오류 발생

-- ✅ 올바른 방법: 반드시 괄호로 감싸기
SELECT emp_name FROM employee
WHERE dno = (SELECT dno FROM department WHERE dept_name = '영업부');
```

### GROUP BY + HAVING + 서브쿼리 활용

```sql
-- 소속 직원이 3명 이상인 부서의 부서번호 검색
SELECT dno,           -- 부서 번호 출력
       COUNT(*) AS 직원수  -- 각 부서의 직원 수 계산
FROM employee         -- 직원 테이블에서
GROUP BY dno          -- 부서 번호별로 그룹 묶기
HAVING COUNT(*) >= 3; -- 직원 수가 3명 이상인 그룹만 필터링
-- GROUP BY로 부서별로 묶고, HAVING으로 3명 이상인 부서만 출력
```

## 🎯 핵심 정리

이번 문서에서 학습한 핵심 개념들을 정리합니다.

- **서브쿼리(부속질의문)** 는 SQL 문 안에 또 다른 SQL 문을 포함하는 구조로, 메인 쿼리의 WHERE 절 조건값을 만들어주는 역할을 한다.
- 서브쿼리는 반드시 **괄호( )로 감싸야** 하며, 내부에 `ORDER BY`를 사용할 수 없다.
- **단일 행 서브쿼리**는 결과가 1건 이하일 때 `=`, `>`, `<` 등 일반 비교 연산자를 사용한다.
- **다중 행 서브쿼리**는 결과가 여러 건일 때 `IN`, `NOT IN`, `ANY`, `ALL`, `EXISTS` 등 다중 행 연산자를 사용한다.
- **IN**은 목록 중 하나라도 일치하면 참, **NOT IN**은 하나도 일치하지 않아야 참이다.
- **ANY/SOME**은 비교 연산자와 함께 사용하며 하나 이상 조건 충족 시 참, **ALL**은 모든 값이 조건 충족 시 참이다.
- **EXISTS**는 서브쿼리 결과가 1건이라도 있으면 참이며, 여러 컬럼을 반환하는 경우에 주로 활용한다.
- 서브쿼리와 조인의 차이: 서브쿼리는 **메인 쿼리 테이블의 컬럼만** 최종 결과로 출력된다.
- **LIKE**는 문자열 패턴 검색에 사용하며, `%`(길이 무제한 와일드카드)와 `_`(한 글자 와일드카드)를 활용한다.
- **NULL 검색**은 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용해야 하며, `= NULL`은 허용되지 않는다.
- **UNION**(합집합), **INTERSECT**(교집합), **EXCEPT**(차집합)는 두 쿼리 결과를 집합 연산으로 처리하며, `ALL`을 붙이면 중복을 포함한다.
- **INSERT**는 데이터 삽입, **DELETE**는 데이터 삭제, **UPDATE**는 데이터 수정 명령어이며, 모두 서브쿼리와 결합하여 강력하게 활용 가능하다.
- DML 수행 시 **개체 무결성**(기본키 규칙), **참조 무결성**(외래키 규칙), **도메인 무결성**(값 범위 규칙)을 반드시 유지해야 한다.
- SQL의 기본 원리와 개념을 잘 이해하면, Oracle·MySQL·MSSQL 등 다양한 DBMS에도 빠르게 적응할 수 있다.
