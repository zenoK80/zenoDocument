# 🗄️ SQL의 발전 역사와 NoSQL 비교

데이터베이스를 공부하다 보면 가장 먼저 만나게 되는 개념이 바로 **SQL**입니다. SQL은 단순한 명령어 집합이 아니라, 수십 년에 걸쳐 발전해 온 하나의 "언어 표준"입니다. 이 문서에서는 SQL이 무엇인지, 어떻게 발전해 왔는지, 그리고 최근 빅데이터 시대에 함께 등장한 **NoSQL**과는 어떤 차이가 있는지를 초보자도 완전히 이해할 수 있도록 아주 자세하게 설명합니다.

---

## 1️⃣ SQL이란 무엇인가? — 개념 소개

### SQL의 정의

**SQL**(Structured Query Language, 구조화된 질의 언어)은 **관계형 데이터베이스(테이블 형태로 데이터를 저장하는 데이터베이스)**를 대상으로 데이터를 다루기 위해 사용하는 언어입니다. 실무에서는 "시퀄(sequel)"이라고 부르기도 합니다.

마치 우리가 영어로 외국인에게 말을 걸 듯, **컴퓨터(데이터베이스 서버)에게 데이터를 달라고 요청하거나, 저장하거나, 수정하거나, 삭제할 때 사용하는 언어**가 바로 SQL입니다.

> 💡 **SQL은 Oracle, MS SQL Server, MySQL 등 거의 모든 관계형 데이터베이스에서 공통으로 사용하는 표준 언어입니다.** 어떤 데이터베이스를 쓰더라도 SQL의 핵심 문법은 동일하게 적용됩니다.

### SQL의 주요 특징

SQL은 다음 세 가지 중요한 특성을 가집니다.

| 특징 | 설명 |
|------|------|
| **고급 언어** | 사람이 이해하기 쉬운 형태로 작성할 수 있습니다 |
| **비절차적 언어** | "어떻게 처리할지"가 아니라 "무엇을 원하는지"만 말하면 됩니다 |
| **집합 지향 언어** | 결과가 한 건씩이 아니라 테이블(집합) 형태로 반환됩니다 |

**비절차적(Non-Procedural)** 이라는 말이 어렵게 느껴질 수 있는데, 쉽게 말하면 이렇습니다. 식당에서 음식을 주문할 때 "이 재료를 몇 도로 몇 분 동안 볶아주세요"라고 조리 과정을 설명하지 않아도 됩니다. 그냥 "비빔밥 하나 주세요"라고 말하면 됩니다. SQL도 마찬가지입니다. "직원 테이블에서 과장 직급인 사람들을 가져와줘"라고 원하는 것만 말하면, 데이터베이스가 알아서 처리해 줍니다.

### SQL의 사용 방법 두 가지

SQL은 두 가지 방식으로 사용할 수 있습니다.

1. **독자적 사용**: Oracle, MySQL 등의 데이터베이스 툴에 직접 접속해서 SQL 명령어를 직접 입력하고 결과를 확인하는 방식입니다. 마치 계산기에 직접 숫자를 입력하듯이 사용하는 것입니다.

2. **내장(삽입) 사용**: Java, Python 같은 프로그래밍 언어 코드 안에 SQL을 넣어서 사용하는 방식입니다. 예를 들어 Java 웹 애플리케이션 코드 안에 SQL 문장을 삽입해서 데이터베이스와 연동하는 방식입니다.

:::info SQL이 고급 언어이기 때문에 두 방식 모두 가능합니다
SQL이 고수준(High-Level) 언어이기 때문에 독자적으로도, 다른 프로그래밍 언어 안에 내장해서도 사용할 수 있습니다. 이것이 SQL이 강력한 이유 중 하나입니다.
:::

## 2️⃣ SQL 명령어의 세 가지 분류 — 왜 이렇게 나누는가?

### DDL, DML, DCL의 구분

SQL 명령어는 크게 **세 가지 종류**로 나뉩니다. 이 구분을 이해하면 SQL 전체 구조가 한눈에 들어옵니다.

마치 건물을 짓는 과정과 비슷합니다. 먼저 설계도를 그리고(정의), 그 안에 가구와 물건을 채우고 관리하고(조작), 출입 권한을 설정하는(제어) 것처럼요.

#### 📐 DDL — 데이터 정의어 (Data Definition Language)

테이블(데이터를 담는 그릇)을 만들고, 수정하고, 삭제하는 명령어입니다. 건물의 뼈대를 세우는 작업과 같습니다.

| 명령어 | 역할 |
|--------|------|
| `CREATE` | 테이블 생성 |
| `ALTER` | 테이블 수정 |
| `DROP` | 테이블 삭제 |

#### 🔧 DML — 데이터 조작어 (Data Manipulation Language)

이미 만들어진 테이블 안에서 데이터를 삽입하고, 수정하고, 삭제하고, 검색하는 명령어입니다. 건물 안의 가구를 배치하고 관리하는 것과 같습니다.

| 명령어 | 역할 |
|--------|------|
| `INSERT` | 데이터 삽입 |
| `UPDATE` | 데이터 수정 |
| `DELETE` | 데이터 삭제 |
| `SELECT` | 데이터 검색 |

#### 🔐 DCL — 데이터 제어어 (Data Control Language)

데이터베이스에 접근할 수 있는 권한을 부여하거나 회수하고, 데이터의 무결성(오류 없이 정확한 상태)을 지키기 위한 명령어입니다. 건물의 출입 카드를 관리하는 것과 같습니다.

| 명령어 | 역할 |
|--------|------|
| `GRANT` | 권한 부여 |
| `REVOKE` | 권한 회수 |
| `COMMIT` | 작업 확정(저장) |
| `ROLLBACK` | 작업 취소(되돌리기) |

> 📌 **실제 작업 순서**: 테이블을 먼저 만들어야(DDL) 그 안에 데이터를 넣고 다룰 수 있고(DML), 권한도 설정할 수 있습니다(DCL). 그래서 보통 DDL → DML → DCL 순서로 진행됩니다.

:::note COMMIT과 ROLLBACK이란?
`COMMIT`은 내가 수행한 작업(삽입, 수정, 삭제 등)을 확정해서 데이터베이스에 영구 저장하는 것이고, `ROLLBACK`은 작업을 취소해서 이전 상태로 되돌리는 것입니다. 이 개념은 이후 **트랜잭션(작업 묶음)** 단원에서 자세히 다룹니다.
:::

## 3️⃣ SQL의 발전 역사 — 표준이 어떻게 진화해 왔는가?

### SQL은 하루아침에 만들어지지 않았습니다

SQL은 1970년대에 처음 개념이 등장한 이후, 수십 년에 걸쳐 꾸준히 발전해 왔습니다. 마치 스마트폰이 처음에는 전화만 되다가, 점점 카메라, 인터넷, 앱 등의 기능이 추가되어 진화한 것처럼 SQL도 버전이 올라가면서 점점 더 많은 기능을 갖추게 되었습니다.

### 📅 SQL 발전 연표

#### SQL-86 (1986년) — 최초의 표준화

1986년, **SQL의 첫 번째 국제 표준**이 만들어졌습니다. ANSI(미국 국립 표준 협회)와 ISO(국제 표준화 기구)가 SQL을 공식 표준으로 인정한 것입니다. 이 시기는 "SQL이라는 언어가 공식적으로 세상에 등장한 시점"이라고 볼 수 있습니다.

#### SQL-89 (SQL1, 1989년) — 참조 무결성 추가

1989년에 발표된 **SQL1**에서는 **참조 무결성(Referential Integrity)** 기능이 추가되었습니다. 참조 무결성이란 테이블 간의 관계에서 데이터의 일관성을 지켜주는 규칙입니다. 예를 들어, 존재하지 않는 부서 번호를 직원 테이블에 입력하지 못하도록 막아주는 것입니다. 이 개념은 **외래 키(Foreign Key)** 와 함께 작동하며, 뒤에서 더 자세히 설명합니다.

#### SQL-92 (SQL2, 1992년) — 응용 프로그램 내장 및 다양한 데이터 타입

1992년에 발표된 **SQL2**에서는 두 가지 중요한 기능이 추가되었습니다.

- **삽입 SQL(Embedded SQL)**: Java, C 같은 프로그래밍 언어 안에 SQL을 직접 삽입해서 사용하는 기능이 표준화되었습니다.
- **다양한 데이터 타입**: 날짜, 시간, 문자열 등 더 다양한 형태의 데이터를 저장할 수 있는 타입들이 추가되었습니다.

SQL2는 오늘날에도 가장 널리 사용되는 SQL 표준의 기반이 되는 중요한 버전입니다.

#### SQL-99 (SQL3, 1999년) — 객체 지향 개념 도입

1999년에 발표된 **SQL3**에서는 **객체 지향(Object-Oriented) 데이터베이스** 개념이 도입되었습니다. 이 버전부터 SQL은 단순한 테이블 처리를 넘어서, 더 복잡한 데이터 구조도 다룰 수 있게 되었습니다.

:::info SQL 버전 정리표

| 버전 | 연도 | 주요 특징 |
|------|------|-----------|
| SQL-86 | 1986년 | 최초 국제 표준 제정 |
| SQL-89 (SQL1) | 1989년 | 참조 무결성, 외래 키 기능 추가 |
| SQL-92 (SQL2) | 1992년 | 삽입 SQL, 다양한 데이터 타입 추가 |
| SQL-99 (SQL3) | 1999년 | 객체 지향 데이터베이스 개념 도입 |

:::

## 4️⃣ NoSQL이란? — SQL과의 비교

### 빅데이터 시대가 만들어낸 새로운 패러다임

SQL이 아무리 훌륭해도, 세상에는 SQL만으로 처리하기 어려운 데이터들이 있습니다. 예를 들어볼까요?

- 소셜 미디어에 매초 수백만 건씩 올라오는 **게시글과 댓글**
- 사용자가 앱을 사용할 때 발생하는 **위치 정보 로그**
- 온라인 쇼핑몰의 **상품 리뷰 텍스트**
- 동영상, 이미지와 같은 **비정형(형태가 정해지지 않은) 데이터**

이런 데이터들은 Excel의 표처럼 **행과 열로 깔끔하게 정리되어 있지 않습니다.** 텍스트가 어떤 건 짧고 어떤 건 길고, 구조 자체가 제각각입니다. 이런 데이터를 **비관계형 데이터(Non-Relational Data)** 라고 합니다.

**NoSQL**은 바로 이런 비관계형 데이터를 처리하기 위해 등장한 데이터베이스 시스템과 그것을 다루는 방법의 총칭입니다.

:::tip NoSQL의 이름 뜻
NoSQL은 "No SQL(SQL이 아니다)"이라는 의미도 있지만, 더 정확하게는 **"Not Only SQL(SQL만이 전부가 아니다)"** 의 약자로 해석됩니다. SQL을 완전히 버린 것이 아니라, SQL 외에도 다양한 방법으로 데이터를 처리할 수 있다는 의미입니다.
:::

### SQL(관계형 DB)의 ACID 특성

관계형 데이터베이스는 **ACID**라는 네 가지 성질을 반드시 지켜야 합니다. 이 네 가지는 데이터베이스가 믿을 수 있는 데이터를 보장하기 위한 핵심 원칙입니다.

| 약자 | 용어 | 의미 | 비유 |
|------|------|------|------|
| **A** | 원자성 (Atomicity) | 작업은 전부 성공하거나 전부 실패해야 함 | 계좌 이체 시 출금과 입금이 둘 다 되거나 둘 다 안 되어야 함 |
| **C** | 일관성 (Consistency) | 작업 전후에 데이터가 항상 올바른 상태여야 함 | 이체 전후 잔액 합계가 같아야 함 |
| **I** | 고립성 (Isolation) | 내 작업이 진행 중일 때 다른 사람이 간섭할 수 없음 | 은행 창구에서 업무 처리 중 다른 사람이 끼어들지 못함 |
| **D** | 지속성 (Durability) | 완료된 작업 결과는 영구적으로 저장됨 | 저장 버튼을 누르면 꺼도 데이터가 남아있음 |

:::note ACID는 나중에 다시 배웁니다
ACID 특성은 **트랜잭션(작업 묶음)과 병행 제어** 단원에서 훨씬 더 자세하게 다룹니다. 지금은 "관계형 데이터베이스는 이 네 가지 원칙을 지킨다"는 정도만 기억해 두세요.
:::

### NoSQL의 BASE 특성

반면 NoSQL은 ACID 대신 **BASE**라는 특성을 가집니다. 빅데이터처럼 방대한 양의 데이터를 빠르게 처리하려면 ACID처럼 엄격한 규칙을 지키는 것이 오히려 성능을 떨어뜨릴 수 있기 때문입니다.

| 약자 | 용어 | 의미 |
|------|------|------|
| **BA** | 기본 가용성 (Basically Available) | 시스템은 항상 사용 가능한 상태를 유지함 |
| **S** | 소프트 상태 (Soft State) | 사용자가 관리하지 않으면 데이터가 영속성(계속 남아있는 성질)을 갖지 못할 수도 있음. 더 이상 필요 없는 데이터는 만료(expire)될 수 있음 |
| **E** | 최종 일관성 (Eventually Consistent) | 지금 당장 완전한 일관성이 아니어도, 언젠가는 일관된 상태가 됨 |

**왜 이렇게 할까요?** 예를 들어, 실시간으로 수백만 명의 사용자가 남기는 상품 리뷰를 분석해서 "이 제품 좋아요?" 같은 답변을 내놓아야 한다고 생각해 봅시다. 100% 완벽한 일관성을 지키면서 모든 데이터를 처리하려면 몇 시간, 아니 며칠이 걸릴 수도 있습니다. 하지만 사용자는 지금 당장 "이 제품 괜찮은 것 같아요" 정도의 빠른 답변을 원합니다. 이처럼 **속도와 확장성이 완벽한 일관성보다 더 중요한 상황**에서 NoSQL이 사용됩니다.

### SQL vs NoSQL 핵심 비교

| 항목 | SQL (관계형 DB) | NoSQL (비관계형 DB) |
|------|----------------|---------------------|
| **데이터 구조** | 테이블(행과 열) | 문서, 키-값, 그래프 등 다양한 형태 |
| **스키마(구조 정의)** | 엄격한 스키마 필요 | 유연한 구조, 스키마 없이도 가능 |
| **핵심 특성** | ACID (정확성 중심) | BASE (가용성·확장성 중심) |
| **정규화** | 반드시 필요 | 어렵거나 불필요 |
| **적합한 데이터** | 정형 데이터 (표 형태) | 비정형 데이터 (텍스트, 이미지 등) |
| **주요 사용 사례** | 회계, 은행, ERP 등 | 빅데이터, SNS, 실시간 분석 등 |
| **설계 중심** | 테이블을 잘 설계하는 것 | 무엇을 질의할지를 먼저 설계하는 것 |

:::warning NoSQL = 빅데이터? 오해하지 마세요!
NoSQL이 빅데이터에 사용되는 것은 맞지만, **빅데이터를 처리하기 위해 반드시 NoSQL이 필요한 것은 아닙니다.** 빅데이터를 처리하는 방법은 NoSQL 외에도 다양하게 존재합니다. "NoSQL = 빅데이터 전용"이라는 등호는 틀립니다.
:::

## 5️⃣ SQL DDL 명령어 실전 — CREATE, ALTER, DROP

이제 실제 SQL 명령어를 배워봅시다. 먼저 테이블을 만들고(CREATE), 수정하고(ALTER), 삭제하는(DROP) **DDL(데이터 정의어)** 부터 시작합니다.

### 📌 CREATE TABLE — 테이블 만들기

`CREATE TABLE`은 새로운 테이블(데이터를 저장할 그릇)을 만드는 명령어입니다. 마치 엑셀에서 새 시트를 만들고 열 이름을 정하는 것과 비슷합니다.

#### 기본 문법 구조

```sql
CREATE TABLE 테이블명 (
  속성명1 데이터타입 [NOT NULL],   -- NOT NULL: 반드시 값을 입력해야 함
  속성명2 데이터타입 [DEFAULT 기본값], -- DEFAULT: 값을 안 넣으면 자동으로 들어갈 값
  ...
  [PRIMARY KEY (속성명)],          -- 기본키 설정
  [UNIQUE (속성명)],               -- 대체키(유일한 값, NULL 허용) 설정
  [FOREIGN KEY (속성명) REFERENCES 참조테이블(속성명)
    [ON DELETE 옵션] [ON UPDATE 옵션]], -- 외래키 설정
  [CONSTRAINT 제약조건명 CHECK (조건)] -- 체크 제약조건 설정
);
```

#### 주요 옵션 설명

- **NOT NULL**: "이 열은 반드시 값이 있어야 한다"는 뜻입니다. 예를 들어 이름이 없는 고객은 존재할 수 없으니, 이름 열에 `NOT NULL`을 붙입니다.
- **DEFAULT 값**: 데이터를 입력할 때 이 열에 값을 넣지 않으면 자동으로 지정한 값이 들어갑니다. 예를 들어 적립금의 기본값을 0으로 설정할 수 있습니다.
- **PRIMARY KEY**: 기본 키(테이블에서 각 행을 유일하게 식별하는 열)를 설정합니다. 기본 키는 절대로 중복되거나 NULL이 될 수 없습니다. 학생 테이블의 학번, 직원 테이블의 사원번호가 대표적인 기본 키입니다.
- **UNIQUE**: 대체 키를 설정합니다. 기본 키처럼 중복 값은 허용하지 않지만, NULL 값은 허용합니다.
- **FOREIGN KEY ... REFERENCES**: 외래 키(다른 테이블의 기본 키를 참조하는 열)를 설정합니다. 참조 무결성을 지키기 위해 사용합니다.
- **CHECK**: 특정 범위나 조건 내의 값만 입력되도록 제한합니다.

#### ✅ 실전 예제 1 — 고객 테이블 만들기

```sql
-- 고객 테이블 생성
-- 조건: 기본키는 고객아이디, 고객이름과 등급은 반드시 입력, 적립금 기본값 0
CREATE TABLE 고객 (
  고객아이디  VARCHAR(20)  NOT NULL,   -- 고객 ID: 문자 20자, 필수 입력
  고객이름    VARCHAR(10)  NOT NULL,   -- 고객 이름: 문자 10자, 필수 입력
  나이        INTEGER,                 -- 나이: 정수형, 선택 입력
  등급        VARCHAR(10)  NOT NULL,   -- 등급: 문자 10자, 필수 입력
  직업        VARCHAR(20),             -- 직업: 문자 20자, 선택 입력
  적립금      INTEGER      DEFAULT 0,  -- 적립금: 정수형, 미입력 시 자동으로 0 저장
  PRIMARY KEY (고객아이디)             -- 고객아이디를 기본 키로 설정
);
```

**코드 한 줄씩 설명:**
- `CREATE TABLE 고객` → "고객"이라는 이름의 테이블을 만들기 시작합니다.
- `고객아이디 VARCHAR(20) NOT NULL` → "고객아이디"라는 열을 만들고, 최대 20글자 문자열을 저장하며, 반드시 값을 입력해야 합니다.
- `고객이름 VARCHAR(10) NOT NULL` → "고객이름" 열은 최대 10글자 문자열이며, 반드시 입력해야 합니다.
- `나이 INTEGER` → "나이" 열은 정수(소수점 없는 숫자)를 저장합니다. `NOT NULL`이 없으므로 입력하지 않아도 됩니다.
- `등급 VARCHAR(10) NOT NULL` → "등급" 열은 문자열이고, 반드시 입력해야 합니다.
- `직업 VARCHAR(20)` → "직업" 열은 선택 입력입니다.
- `적립금 INTEGER DEFAULT 0` → "적립금" 열은 값을 넣지 않으면 자동으로 0이 저장됩니다.
- `PRIMARY KEY (고객아이디)` → 고객아이디를 이 테이블의 기본 키로 설정합니다.

#### ✅ 실전 예제 2 — 제품 테이블 (CHECK 제약조건 포함)

```sql
-- 제품 테이블 생성
-- 조건: 기본키는 제품번호, 재고량은 0 이상 10000 이하로 제한
CREATE TABLE 제품 (
  제품번호    CHAR(3)      NOT NULL,    -- 제품번호: 고정 3자리 문자, 필수 입력
  제품명      VARCHAR(20),             -- 제품명: 가변 20자 문자열
  재고량      INTEGER,                 -- 재고량: 정수형
  단가        INTEGER,                 -- 단가: 정수형
  제조업체    VARCHAR(20),             -- 제조업체: 가변 20자 문자열
  PRIMARY KEY (제품번호),             -- 제품번호를 기본 키로 설정
  CONSTRAINT 재고량체크               -- 제약조건의 이름을 "재고량체크"로 지정
    CHECK (재고량 >= 0 AND 재고량 <= 10000)  -- 재고량은 0 이상 10000 이하여야 함
);
```

**코드 한 줄씩 설명:**
- `CHAR(3)` → 고정 길이 3자리 문자입니다. 예를 들어 "P01", "A12" 같은 형태입니다.
- `CONSTRAINT 재고량체크` → 이 제약조건에 "재고량체크"라는 이름을 붙입니다. 나중에 이 이름으로 제약조건을 삭제할 수 있습니다.
- `CHECK (재고량 >= 0 AND 재고량 <= 10000)` → 재고량은 반드시 0 이상이고 10,000 이하여야 합니다. 만약 -1이나 50,000이 입력되면 오류가 발생합니다.

#### ✅ 실전 예제 3 — 주문 테이블 (외래키 2개 포함)

```sql
-- 주문 테이블 생성
-- 기본키: 주문번호
-- 외래키1: 주문고객 → 고객 테이블의 고객아이디 참조
-- 외래키2: 주문제품 → 제품 테이블의 제품번호 참조
CREATE TABLE 주문 (
  주문번호    INTEGER      NOT NULL,   -- 주문번호: 정수형, 필수 입력
  주문고객    VARCHAR(20),             -- 주문고객: 고객 테이블의 고객아이디를 참조할 열
  주문제품    CHAR(3),                 -- 주문제품: 제품 테이블의 제품번호를 참조할 열
  수량        INTEGER,                 -- 주문 수량: 정수형
  PRIMARY KEY (주문번호),             -- 주문번호를 기본 키로 설정
  FOREIGN KEY (주문고객)              -- 주문고객 열을 외래 키로 설정
    REFERENCES 고객(고객아이디),      -- 고객 테이블의 고객아이디를 참조
  FOREIGN KEY (주문제품)              -- 주문제품 열을 외래 키로 설정
    REFERENCES 제품(제품번호)         -- 제품 테이블의 제품번호를 참조
);
```

**코드 한 줄씩 설명:**
- `FOREIGN KEY (주문고객) REFERENCES 고객(고객아이디)` → "주문고객" 열은 "고객" 테이블의 "고객아이디" 열을 외래 키로 참조합니다. 즉, 주문에 입력되는 고객 아이디는 반드시 고객 테이블에 존재하는 값이어야 합니다.
- `FOREIGN KEY (주문제품) REFERENCES 제품(제품번호)` → "주문제품" 열은 "제품" 테이블의 "제품번호"를 참조합니다. 외래 키는 여러 개 설정할 수 있습니다.

### 🔑 외래 키 삭제 옵션 4가지

외래 키로 연결된 데이터를 삭제하거나 수정할 때, 참조 무결성을 어떻게 처리할지 결정하는 옵션이 4가지 있습니다.

| 옵션 | 동작 방식 | 비유 |
|------|-----------|------|
| `NO ACTION` / `RESTRICT` | 참조하는 데이터가 있으면 삭제/수정 **거부** | 아직 이사 안 간 세입자가 있으면 집 철거 불가 |
| `CASCADE` | 참조하는 데이터도 **함께 삭제/수정** | 건물 철거 시 내부 가구도 함께 폐기 |
| `SET NULL` | 참조하는 데이터를 **NULL로 변경** | 부서가 없어지면 해당 직원의 부서 정보를 공백으로 처리 |
| `SET DEFAULT` | 참조하는 데이터를 **미리 정한 기본값으로 변경** | 부서가 없어지면 직원을 기본 부서로 이동 |

:::warning 기본값(DEFAULT)은 RESTRICT입니다
옵션을 명시하지 않으면 기본적으로 `RESTRICT`가 적용됩니다. 참조하고 있는 데이터가 있으면 삭제나 수정이 거부됩니다. 이것이 데이터 무결성을 보호하는 안전한 기본 동작입니다.
:::

### 📌 ALTER TABLE — 기존 테이블 수정하기

`ALTER TABLE`은 이미 만들어진 테이블에 열을 추가하거나, 삭제하거나, 제약조건을 변경할 때 사용합니다. 마치 이미 완성된 서류 양식에 새로운 항목을 추가하거나 불필요한 항목을 지우는 것과 같습니다.

#### ✅ 열 추가 — ADD

```sql
-- 고객 테이블에 '가입날짜' 열 추가하기
ALTER TABLE 고객
  ADD 가입날짜 DATETIME;   -- DATETIME: 날짜와 시간을 저장하는 데이터 타입
```

- `ALTER TABLE 고객` → "고객" 테이블을 수정하겠다.
- `ADD 가입날짜 DATETIME` → "가입날짜"라는 이름의 날짜/시간형 열을 새로 추가한다.

#### ✅ 열 삭제 — DROP COLUMN

```sql
-- 고객 테이블에서 '등급' 열 삭제하기
-- CASCADE: 다른 테이블에서 이 열을 참조하고 있다면 함께 삭제
ALTER TABLE 고객
  DROP 등급 CASCADE;
```

- `DROP 등급` → "등급" 열을 삭제한다.
- `CASCADE` → 만약 다른 테이블이 이 열을 참조하고 있다면, 그쪽도 함께 삭제한다.

#### ✅ 제약조건 추가 — ADD CONSTRAINT

```sql
-- 고객 테이블에 '나이는 20세 이상' 제약조건 추가
ALTER TABLE 고객
  ADD CONSTRAINT 나이체크   -- 제약조건 이름: "나이체크"
    CHECK (나이 >= 20);      -- 나이는 반드시 20 이상이어야 함
```

#### ✅ 제약조건 삭제 — DROP CONSTRAINT

```sql
-- 앞서 추가한 '나이체크' 제약조건 삭제
ALTER TABLE 고객
  DROP CONSTRAINT 나이체크;  -- 제약조건 이름으로 삭제
```

- 제약조건을 추가할 때는 조건의 내용을 상세히 써야 하지만, 삭제할 때는 제약조건 이름만 써주면 됩니다.

### 📌 DROP TABLE — 테이블 삭제하기

`DROP TABLE`은 테이블 자체를 완전히 삭제합니다. `DELETE`가 테이블 안의 데이터만 지우는 것이라면, `DROP TABLE`은 테이블 구조까지 통째로 없애버리는 것입니다. 마치 파일 내용을 지우는 것(DELETE)과 파일 자체를 휴지통에 버리는 것(DROP)의 차이입니다.

```sql
-- 고객 테이블 삭제 (다른 테이블이 참조하고 있으면 삭제 거부)
DROP TABLE 고객 RESTRICT;

-- 고객 테이블 삭제 (다른 테이블이 참조하고 있어도 함께 삭제)
DROP TABLE 고객 CASCADE;
```

:::danger DROP TABLE은 되돌릴 수 없습니다
`DROP TABLE`을 실행하면 테이블과 그 안의 모든 데이터가 완전히 삭제됩니다. `ROLLBACK`으로도 복구가 되지 않는 경우가 많습니다. 실무에서는 반드시 백업 후 실행하세요.
:::

## 6️⃣ SELECT 명령어 — 데이터 검색의 핵심

### SELECT가 가장 많이 쓰이는 이유

개발자가 데이터베이스를 다루는 작업 중 가장 많이 하는 것이 바로 데이터를 **조회(검색)** 하는 것입니다. 상품 목록을 보여주거나, 사용자 정보를 가져오거나, 통계 보고서를 만드는 모든 작업이 `SELECT` 명령어로 이루어집니다.

### SELECT 기본 문법 구조

```sql
SELECT [DISTINCT] 속성명1, 속성명2, ...   -- 출력할 열 지정 (DISTINCT: 중복 제거)
FROM 테이블명                              -- 어느 테이블에서 가져올지
[WHERE 조건]                               -- 어떤 조건의 행만 가져올지
[GROUP BY 속성명]                          -- 어떤 기준으로 그룹 지을지
[HAVING 그룹조건]                          -- 그룹에 대한 조건
[ORDER BY 속성명 [ASC|DESC]];             -- 결과를 어떤 순서로 정렬할지
```

각 절(Clause)의 역할을 정리하면 이렇습니다.

| 절 | 역할 | 비유 |
|----|------|------|
| `SELECT` | 출력할 열 선택 | 마트 영수증에서 어떤 항목을 볼지 선택 |
| `FROM` | 데이터를 가져올 테이블 지정 | 어느 마트에서 살지 결정 |
| `WHERE` | 특정 조건의 행만 필터링 | "1만 원 이상 상품만 보여줘" |
| `GROUP BY` | 특정 기준으로 묶어서 집계 | "카테고리별로 묶어서 보여줘" |
| `HAVING` | 그룹에 대한 조건 | "카테고리별 합계가 5만 원 이상인 것만" |
| `ORDER BY` | 결과 정렬 | "가격 오름차순으로 정렬" |

- **ASC(오름차순)**: 1, 2, 3, 4 ... 작은 값에서 큰 값으로 올라가는 정렬
- **DESC(내림차순)**: 100, 90, 80 ... 큰 값에서 작은 값으로 내려가는 정렬

### ✅ SELECT 기본 예제들

아래 예제에서는 `사원(employee)` 테이블과 `부서(department)` 테이블을 사용합니다.

**사원 테이블 (employee)**

| 사원번호 | 사원명 | 직급 | 상사번호 | 급여 | 부서번호 |
|----------|--------|------|----------|------|----------|
| 1 | 김철수 | 부장 | NULL | 600 | 1 |
| 2 | 이영희 | 과장 | 1 | 450 | 1 |
| 3 | 박민준 | 과장 | 1 | 430 | 2 |
| 4 | 최지우 | 사원 | 2 | 280 | 2 |
| 5 | 정하늘 | 사원 | 3 | 300 | 3 |

#### 예제 1 — 모든 열 출력 (`*` 사용)

```sql
-- 부서 테이블의 모든 정보를 전부 출력
SELECT *          -- * (별표): 모든 열을 출력하겠다는 의미
FROM 부서;        -- 부서 테이블에서 가져온다
```

#### 예제 2 — 특정 열만 출력

```sql
-- 사원 테이블에서 사원명과 직급만 출력
SELECT 사원명, 직급   -- 보고 싶은 열만 골라서 나열
FROM 사원;
```

#### 예제 3 — 중복 제거 (DISTINCT)

```sql
-- 사원 테이블에서 직급 목록을 출력하되, 중복은 제거
SELECT DISTINCT 직급   -- DISTINCT: 동일한 값이 여러 번 나와도 한 번만 출력
FROM 사원;
-- 결과: 부장, 과장, 사원 (각각 한 번씩만 출력)
```

#### 예제 4 — 조건 검색 (WHERE)

```sql
-- 부서번호가 2번인 사원의 모든 정보 출력
SELECT *
FROM 사원
WHERE 부서번호 = 2;   -- 부서번호가 2인 행만 필터링
```

#### 예제 5 — 문자열 패턴 검색 (LIKE)

```sql
-- 이름이 '이'로 시작하는 사원의 이름, 직급, 부서번호 출력
SELECT 사원명, 직급, 부서번호
FROM 사원
WHERE 사원명 LIKE '이%';   -- '이'로 시작하는 모든 이름 (%: 이후 어떤 문자든 상관없음)
```

```sql
-- 이름이 'kim'으로 끝나는 사원 검색 (영문 이름 예시)
SELECT *
FROM 사원
WHERE 사원명 LIKE '%kim';   -- '%kim': 앞에 어떤 문자가 와도 되고, 마지막이 kim이면 됨
```

:::tip LIKE 패턴 문자 두 가지
- `%` (퍼센트): 0개 이상의 어떤 문자도 대체 가능. 예) `'이%'` → 이영희, 이민수, 이 등
- `_` (언더바): 정확히 1개의 문자를 대체. 예) `'이__'` → 이름이 정확히 3글자이고 '이'로 시작하는 것
:::

#### 예제 6 — 복합 조건 (AND)

```sql
-- 직급이 과장이면서 부서번호가 1번인 사원의 이름과 급여 출력
SELECT 사원명, 급여
FROM 사원
WHERE 직급 = '과장'    -- 조건 1: 직급이 과장이어야 하고
  AND 부서번호 = 1;    -- 조건 2: 동시에 부서번호도 1이어야 함
```

#### 예제 7 — 부정 조건 (NOT EQUAL)

```sql
-- 직급이 과장이지만 부서번호가 1번이 아닌 사원의 이름과 급여 출력
SELECT 사원명, 급여
FROM 사원
WHERE 직급 = '과장'
  AND 부서번호 <> 1;   -- <> 또는 != : "같지 않다"를 표현 (= 이렇게 쓰면 안 됨)
```

❌ **잘못된 예시** (SQL에서 불가):
```sql
-- 이런 형태는 SQL에서 사용할 수 없습니다
WHERE 부서번호 =! 1   -- 잘못된 문법
```

✅ **올바른 예시**:
```sql
WHERE 부서번호 <> 1   -- 올바른 "같지 않다" 표현
-- 또는
WHERE 부서번호 != 1   -- 이것도 대부분의 DBMS에서 사용 가능
```

#### 예제 8 — 범위 검색 (BETWEEN)

```sql
-- 급여가 300만원 이상 450만원 이하인 사원의 이름, 직급, 급여 출력
SELECT 사원명, 직급, 급여
FROM 사원
WHERE 급여 BETWEEN 300 AND 450;   -- 300 이상 450 이하 (양 끝값 포함)

-- 위와 동일한 결과를 내는 방법 (AND 사용)
SELECT 사원명, 직급, 급여
FROM 사원
WHERE 급여 >= 300    -- 급여가 300 이상이고
  AND 급여 <= 450;   -- 급여가 450 이하인 것
```

#### 예제 9 — 목록 검색 (IN)

```sql
-- 부서번호가 1번이거나 3번인 사원의 모든 정보 출력
SELECT *
FROM 사원
WHERE 부서번호 IN (1, 3);   -- IN: 괄호 안의 값 중 하나라도 일치하면 됨
-- OR 조건과 같은 의미: WHERE 부서번호 = 1 OR 부서번호 = 3
```

#### 예제 10 — 열에 수식 사용 (SELECT에서 계산)

```sql
-- 과장 직급 사원들의 이름, 현재 급여, 10% 인상 후 급여 출력
SELECT 사원명,                  -- 사원 이름 출력
       급여,                    -- 현재 급여 출력
       급여 * 1.1 AS 인상후급여 -- 급여에 1.1을 곱해서 10% 인상된 값을 계산하여 출력
                                 -- AS: 출력되는 열의 이름을 '인상후급여'로 지정
FROM 사원
WHERE 직급 = '과장';
```

:::warning SELECT 수식은 실제 데이터를 변경하지 않습니다
`SELECT 급여 * 1.1`처럼 계산식을 써도, 데이터베이스에 저장된 급여 값은 전혀 바뀌지 않습니다. 그냥 화면에 계산된 값을 보여주는 것뿐입니다. 실제 데이터를 변경하려면 `UPDATE` 명령어를 사용해야 합니다.
:::

### WHERE 절에서 사용하는 연산자 정리

```sql
-- 비교 연산자 예시
WHERE 급여 = 300     -- 급여가 정확히 300인 것
WHERE 급여 > 300     -- 급여가 300보다 큰 것
WHERE 급여 >= 300    -- 급여가 300 이상인 것
WHERE 급여 < 300     -- 급여가 300보다 작은 것
WHERE 급여 <= 300    -- 급여가 300 이하인 것
WHERE 급여 <> 300    -- 급여가 300이 아닌 것 (같지 않다)

-- SQL 전용 연산자 예시
WHERE 급여 BETWEEN 300 AND 450       -- 급여가 300 이상 450 이하
WHERE 부서번호 IN (1, 2, 3)           -- 부서번호가 1, 2, 3 중 하나
WHERE 사원명 LIKE '김%'               -- 이름이 '김'으로 시작
WHERE 사원명 LIKE '%수'               -- 이름이 '수'로 끝남
WHERE 사원명 LIKE '김__'              -- 이름이 '김'으로 시작하고 총 3글자
WHERE 상사번호 IS NULL                -- 상사번호가 NULL인 것 (최고 직급 등)
WHERE 상사번호 IS NOT NULL            -- 상사번호가 NULL이 아닌 것

-- 논리 연산자 예시
WHERE 직급 = '과장' AND 부서번호 = 1  -- 두 조건 모두 만족
WHERE 직급 = '과장' OR 직급 = '부장'  -- 둘 중 하나만 만족해도 됨
WHERE NOT 직급 = '사원'               -- '사원'이 아닌 것
```

### SQL 문법 작성 규칙 정리

```sql
-- 규칙 1: 문장의 끝은 반드시 세미콜론(;)으로 마침
SELECT * FROM 사원;   -- ✅ 올바른 마침

-- 규칙 2: 명령어 대소문자 구분 없음 (대문자 권장)
select * from 사원;   -- ✅ 소문자도 동작함
SELECT * FROM 사원;   -- ✅ 대문자도 동작함 (관례상 대문자 사용)

-- 규칙 3: 문자열 데이터는 대소문자 구분함!
WHERE 사원명 = 'Korea'   -- 'Korea'와 'korea'는 다른 값으로 처리됨
WHERE 사원명 = 'korea'   -- 위와 다른 결과가 나올 수 있음
```

:::tip 가독성을 위해 대문자를 쓰세요
SQL 명령어 자체(`SELECT`, `FROM`, `WHERE` 등)는 대소문자를 구분하지 않지만, **관례상 명령어는 대문자로, 테이블명과 열명은 소문자로 쓰는 것이 가독성에 좋습니다.** 팀 프로젝트에서는 코딩 컨벤션(작성 규칙)을 미리 정하고 통일하는 것이 중요합니다.
:::

## 7️⃣ 주의사항과 실전 팁 🔍

### 기본 키(PRIMARY KEY) 설계 팁

기본 키는 테이블에서 각 행을 유일하게 식별하는 핵심 요소입니다. 현실 세계의 예를 들면:

- 🏫 **학생 테이블** → 학번이 기본 키 (같은 학번을 가진 학생은 없음)
- 🏢 **직원 테이블** → 사원번호가 기본 키
- 🏛️ **국민 데이터** → 주민등록번호가 기본 키 (최근에는 아이핀 등 대체 수단 사용)
- 🛒 **상품 테이블** → 상품코드가 기본 키
- 📦 **주문 테이블** → 주문번호가 기본 키

:::tip 이름이나 전화번호를 기본 키로 쓰면 안 되는 이유
이름은 동명이인이 있을 수 있고, 전화번호는 바뀔 수 있습니다. 기본 키는 **절대 바뀌지 않고, 절대 중복되지 않는 값**이어야 합니다. 그래서 보통 인위적으로 생성한 번호(학번, 사원번호, 주문번호 등)를 기본 키로 사용합니다.
:::

### NULL 처리 주의사항

`NULL`은 "값이 없다" 또는 "아직 알 수 없다"는 의미입니다. 일반적인 비교 연산자로는 NULL을 올바르게 비교할 수 없습니다.

```sql
-- ❌ 잘못된 NULL 비교 (결과가 항상 FALSE)
WHERE 상사번호 = NULL    -- 이렇게 쓰면 아무것도 검색되지 않음

-- ✅ 올바른 NULL 비교
WHERE 상사번호 IS NULL     -- NULL인 것을 찾을 때
WHERE 상사번호 IS NOT NULL -- NULL이 아닌 것을 찾을 때
```

:::danger NULL은 = 으로 비교하면 절대 안 됩니다
`NULL = NULL`은 SQL에서 **FALSE**입니다. NULL은 "값이 없는 상태"이기 때문에, 두 개의 "없음"이 서로 같다고 할 수 없습니다. 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용하세요.
:::

### 연산자 우선순위

SQL에서 여러 조건을 한꺼번에 쓸 때는 우선순위를 주의해야 합니다.

1. **괄호 `()`** — 가장 먼저 처리
2. **NOT** — 부정 연산자
3. **비교 연산자** (`=`, `<>`, `>`, `<`, `>=`, `<=`, `BETWEEN`, `IN`, `LIKE`, `IS NULL`)
4. **AND** — 논리곱
5. **OR** — 논리합

```sql
-- 의도: 직급이 과장이거나, (직급이 사원이면서 급여가 300 이상)인 사원
-- ❌ 잘못된 예 (OR가 AND보다 우선순위가 낮아서 의도와 다르게 동작할 수 있음)
WHERE 직급 = '과장' OR 직급 = '사원' AND 급여 >= 300

-- ✅ 올바른 예 (괄호로 명확하게 의도 표현)
WHERE 직급 = '과장' OR (직급 = '사원' AND 급여 >= 300)
```

:::tip 헷갈릴 때는 괄호를 사용하세요
복잡한 조건을 쓸 때는 우선순위를 외우는 것보다 **괄호를 사용해서 의도를 명확하게 표현**하는 것이 실수를 줄이는 가장 좋은 방법입니다. 코드는 컴퓨터보다 사람이 읽기 쉽게 써야 합니다.
:::

## 📚 핵심 정리

이 문서에서 배운 핵심 개념을 정리합니다.

- **SQL(Structured Query Language)** 은 관계형 데이터베이스를 다루는 표준 질의 언어로, 고급·비절차적·집합 지향 언어입니다.
- SQL은 **독자적 사용**(직접 접속)과 **내장 사용**(Java 등 프로그램 안에 삽입) 두 가지 방식으로 활용할 수 있습니다.
- SQL 명령어는 **DDL(정의어: CREATE, ALTER, DROP)**, **DML(조작어: INSERT, UPDATE, DELETE, SELECT)**, **DCL(제어어: GRANT, REVOKE, COMMIT, ROLLBACK)** 세 가지로 분류됩니다.
- SQL은 **1986년 표준화** 이후 1989년(SQL1, 참조 무결성), 1992년(SQL2, 내장 SQL), 1999년(SQL3, 객체 지향) 순으로 발전했습니다.
- **NoSQL(Not Only SQL)** 은 비관계형 데이터를 처리하기 위한 시스템으로, 빅데이터·비정형 데이터에 적합합니다.
- 관계형 DB는 **ACID**(원자성·일관성·고립성·지속성)를, NoSQL은 **BASE**(기본 가용성·소프트 상태·최종 일관성)를 특성으로 합니다.
- NoSQL은 ACID의 엄격한 일관성 대신 **가용성과 확장성**을 우선시합니다.
- **CREATE TABLE**은 테이블을 생성하며, NOT NULL, DEFAULT, PRIMARY KEY, UNIQUE, FOREIGN KEY, CHECK 옵션을 함께 설정할 수 있습니다.
- 외래 키의 삭제/수정 옵션은 **RESTRICT(거부), CASCADE(연쇄 삭제), SET NULL(NULL로 변경), SET DEFAULT(기본값으로 변경)** 네 가지입니다.
- **ALTER TABLE**은 기존 테이블에 열을 추가(ADD)하거나 삭제(DROP)하고, 제약조건을 추가·삭제할 수 있습니다.
- **DROP TABLE**은 테이블 구조와 데이터를 완전히 삭제하며, RESTRICT(참조 시 거부) 또는 CASCADE(연쇄 삭제) 옵션을 사용합니다.
- **SELECT** 명령어는 SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY 구조를 가집니다.
- WHERE 절에서는 비교 연산자, BETWEEN, IN, LIKE, IS NULL 등 다양한 SQL 연산자를 활용합니다.
- NULL 비교는 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용해야 합니다. `= NULL`은 절대 사용하면 안 됩니다.
- SQL 명령어 키워드는 대소문자를 구분하지 않지만, **문자열 데이터 값은 대소문자를 구분합니다.**
