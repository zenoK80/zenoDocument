# 🗄️ SQL 명령어 완전 정복 – DDL, DML, DCL부터 SELECT까지

데이터베이스를 공부할 때 가장 먼저, 그리고 가장 많이 만나게 되는 것이 바로 **SQL(Structured Query Language, 구조화된 질의 언어)** 입니다. SQL은 단순한 도구가 아니라 데이터베이스와 대화하는 언어 그 자체입니다. Oracle이든, MySQL이든, MS SQL이든 어떤 관계형 데이터베이스를 사용하든 공통으로 쓰이는 이 언어를 제대로 이해하면, 어떤 데이터베이스 환경에서도 자신 있게 작업할 수 있게 됩니다.

이 문서에서는 SQL이 무엇인지 기초 개념부터 시작해서, SQL 명령어가 어떤 기준으로 분류되는지, 그리고 테이블을 만들고(DDL), 데이터를 다루고(DML), 데이터를 조회하는(SELECT) 실제 예제까지 아주 자세하게 살펴보겠습니다.

---

## 1️⃣ SQL이란 무엇인가? – 개념 소개

### SQL의 정의

**SQL**은 **Structured Query Language(구조화된 질의 언어)** 의 약자입니다. 실무에서는 "시퀄(Sequel)"이라고 읽는 경우가 많습니다. 이름에서 알 수 있듯이, SQL은 데이터베이스에 저장된 데이터를 다루기 위한 **표준화된 언어**입니다.

마치 우리가 도서관 사서에게 "컴퓨터 관련 책 중에서 2020년 이후에 출판된 것만 보여주세요"라고 말하듯이, SQL은 데이터베이스에게 "이 조건을 만족하는 데이터를 찾아서 보여줘"라고 요청하는 언어입니다.

### SQL의 주요 특징

SQL은 단순히 데이터를 검색하는 것을 넘어, **테이블을 만들고, 데이터를 삽입하고, 수정하고, 삭제하고, 권한을 관리하는** 모든 작업을 포함합니다. 아래는 SQL의 핵심 특징입니다.

| 특징 | 설명 |
|------|------|
| **고급 언어** | 사람이 이해하기 쉬운 형태로 작성 가능 |
| **비절차적 언어** | "어떻게 가져올지"가 아니라 "무엇을 가져올지"만 명시 |
| **집합 기반 처리** | 결과가 한 건씩이 아니라 테이블(집합) 형태로 반환 |
| **독립적 사용 가능** | DB에 직접 접속해서 명령어 입력 가능 |
| **내장(Embedded) 사용 가능** | Java, Python 등 다른 프로그래밍 언어 안에 삽입해서 사용 가능 |

:::info SQL의 비절차적 특성이란?
절차적 언어(C, Java 등)는 "1번 항목 꺼내고, 2번 항목 꺼내고, 비교하고, 반환하고..."처럼 **과정(How)**을 하나하나 명시해야 합니다. 반면 SQL은 "점수가 80점 이상인 학생 목록 줘"처럼 **원하는 결과(What)**만 말하면, 데이터베이스가 알아서 최적의 방법으로 데이터를 찾아줍니다. 마치 음식 배달 앱에서 "짜장면 하나 주문"이라고 하면, 배달 경로는 배달원이 알아서 찾는 것과 같습니다.
:::

### SQL의 발전 역사

SQL은 오랜 역사를 가진 표준 언어입니다. 버전이 올라갈수록 더 강력하고 다양한 기능이 추가되었습니다.

- **1986년**: SQL 최초 표준화 (ANSI 표준)
- **1989년 (SQL 1)**: **참조 무결성(외래키 관계에서 데이터의 일관성을 보장하는 규칙)** 기능 추가
- **1992년 (SQL 2 / SQL-92)**: 응용 프로그램 내장 SQL, 다양한 데이터 타입 추가
- **1999년 (SQL 3 / SQL-99)**: **객체 지향(Object-Oriented)** 데이터베이스 개념 통합

## 2️⃣ SQL vs NoSQL – 왜 두 가지가 존재하는가?

### 관계형 DB와 SQL

SQL은 기본적으로 **관계형 데이터베이스(Relational Database)**, 즉 데이터를 깔끔한 표(테이블) 형태로 저장하는 데이터베이스를 위한 언어입니다. 학생 정보를 저장할 때 학번, 이름, 성적 같은 칼럼을 가진 표에 저장하는 방식이죠.

관계형 데이터베이스는 **ACID 성질**을 반드시 지켜야 합니다.

| ACID 성질 | 한국어 이름 | 의미 |
|-----------|-----------|------|
| **A**tomicity | 원자성 | 작업이 전부 완료되거나 전부 취소되거나, 중간은 없음 |
| **C**onsistency | 일관성 | 트랜잭션(작업 묶음) 전후로 데이터의 규칙이 유지됨 |
| **I**solation | 고립성 | 내 데이터를 처리하는 동안 다른 사람이 간섭 불가 |
| **D**urability | 지속성 | 한 번 저장된 데이터는 영구적으로 유지됨 |

:::note ACID는 나중에 더 자세히
ACID 성질은 이후 **트랜잭션(Transaction)** 과 **병행 제어(Concurrency Control)** 수업에서 아주 자세하게 다룹니다. 지금은 "관계형 데이터베이스는 이 4가지 성질을 철저하게 지킨다"는 정도만 기억해 두세요.
:::

### NoSQL – 비관계형 데이터베이스

**NoSQL**은 **Not Only SQL**의 약자로, 관계형 데이터베이스 방식이 아닌 다른 방식으로 데이터를 저장하고 조회하는 모든 데이터베이스를 통칭합니다.

예를 들어, 쇼핑몰 사용자들이 남긴 수천만 개의 상품 리뷰를 분석하고 싶다고 해봅시다. 이 리뷰 데이터는 자연어 문장이고, 길이도 제각각이며, 이모지도 포함되어 있습니다. 이런 데이터를 깔끔한 표(테이블)에 담기는 매우 어렵습니다. 이런 **비정형(Non-structured) 데이터**를 빠르게 처리하기 위해 NoSQL이 등장했습니다.

SQL과 NoSQL의 차이를 표로 정리하면 다음과 같습니다.

| 비교 항목 | SQL (관계형 DB) | NoSQL (비관계형 DB) |
|-----------|----------------|---------------------|
| **데이터 형태** | 정형 데이터 (표/테이블) | 비정형 데이터 (텍스트, 이미지, JSON 등) |
| **중점 가치** | 일관성(Consistency) + 무결성 | 가용성(Availability) + 확장성(Scalability) |
| **정규화** | 정규화 필수 | 정규화 어려움/불필요 |
| **설계 핵심** | 테이블 구조를 잘 설계 | 어떤 질의를 할지가 더 중요 |
| **일관성** | 즉시 일관성 (즉각 정확) | 최종적 일관성 (언젠간 정확해짐) |
| **활용 사례** | 은행, 쇼핑몰 재고 관리 등 | 빅데이터 분석, 실시간 로그 처리 등 |

:::tip NoSQL과 빅데이터의 관계
NoSQL은 빅데이터를 처리하는 데 자주 사용되지만, 빅데이터를 처리하기 위해 반드시 NoSQL만 써야 하는 것은 아닙니다. 빅데이터 처리에는 NoSQL 외에도 다양한 방법이 존재합니다. NoSQL은 "빠른 분석"과 "대용량 처리"에 강점이 있는 도구라고 이해하면 됩니다.
:::

## 3️⃣ SQL 명령어 분류 – DDL, DML, DCL

SQL 명령어는 크게 세 가지로 분류됩니다. 마치 요리를 할 때 **주방 준비(그릇/재료 세팅) → 요리(재료 손질, 조리) → 관리(위생 점검, 허가)** 의 순서로 진행되듯이, SQL도 순서와 역할이 있습니다.

### 📌 DDL (Data Definition Language) – 데이터 정의어

**DDL**은 데이터베이스의 구조, 즉 **테이블 자체를 만들고, 수정하고, 삭제하는** 명령어입니다. 데이터를 담을 그릇(테이블)을 만드는 작업이라고 이해하면 됩니다.

| 명령어 | 역할 |
|--------|------|
| `CREATE` | 테이블 생성 |
| `ALTER` | 테이블 구조 수정 (컬럼 추가/삭제/변경) |
| `DROP` | 테이블 삭제 |

### 📌 DML (Data Manipulation Language) – 데이터 조작어

**DML**은 이미 만들어진 테이블 안에 들어있는 **데이터를 삽입하고, 수정하고, 삭제하고, 검색하는** 명령어입니다. 그릇에 음식(데이터)을 담거나 꺼내거나 바꾸는 작업입니다.

| 명령어 | 역할 |
|--------|------|
| `INSERT` | 데이터 삽입 |
| `UPDATE` | 데이터 수정 |
| `DELETE` | 데이터 삭제 |
| `SELECT` | 데이터 검색/조회 |

### 📌 DCL (Data Control Language) – 데이터 제어어

**DCL**은 데이터베이스의 **보안과 무결성(데이터가 올바른 상태를 유지하는 것)을 관리하는** 명령어입니다. 권한을 주고 빼앗거나, 작업을 확정하거나 취소하는 역할을 합니다.

| 명령어 | 역할 |
|--------|------|
| `GRANT` | 특정 사용자에게 권한 부여 |
| `REVOKE` | 부여된 권한 회수 |
| `COMMIT` | 작업 결과를 데이터베이스에 영구 반영 |
| `ROLLBACK` | 작업 내용을 이전 상태로 되돌림 |

> 💡 **순서가 중요합니다!** 반드시 DDL → DML → DCL 순서로 이해해야 합니다. 테이블이 먼저 만들어져야(DDL), 그 안에 데이터를 넣고 조작할 수 있고(DML), 그 이후에 권한 관리나 트랜잭션 제어가 의미 있기 때문입니다.

## 4️⃣ DDL 완전 정복 – CREATE, ALTER, DROP

### 🔨 CREATE TABLE – 테이블 만들기

`CREATE TABLE`은 데이터를 저장할 테이블(표)을 처음 만드는 명령어입니다. 엑셀에서 새 시트를 만들고 첫 번째 행에 열 이름(컬럼명)을 적는 것과 비슷합니다. 단, SQL에서는 각 컬럼에 어떤 종류의 데이터가 들어갈지(데이터 타입)와 어떤 규칙을 적용할지(제약 조건)도 함께 정의합니다.

**기본 문법 형식:**

```sql
CREATE TABLE 테이블이름 (
    컬럼명1 데이터타입 [제약조건],
    컬럼명2 데이터타입 [제약조건],
    ...
    [PRIMARY KEY (컬럼명)],
    [UNIQUE (컬럼명)],
    [FOREIGN KEY (컬럼명) REFERENCES 참조테이블(참조컬럼)],
    [CHECK (조건식)]
);
```

대괄호 `[ ]`로 감싸진 부분은 **생략 가능한 옵션**입니다.

#### 제약 조건(Constraint) 정리

제약 조건은 테이블에 잘못된 데이터가 들어오지 못하도록 막는 규칙입니다. 마치 회원 가입 폼에서 "이메일 형식이 맞지 않으면 가입 불가"처럼, 데이터베이스 레벨에서 데이터의 품질을 보장합니다.

| 제약 조건 | 의미 | 비유 |
|-----------|------|------|
| `NOT NULL` | 반드시 값이 있어야 함 (빈칸 불가) | 이름 없이는 회원 가입 불가 |
| `PRIMARY KEY` | 기본키 – 유일하고 NULL 불가 | 학생증의 학번 |
| `UNIQUE` | 대체키 – 유일하지만 NULL 허용 | 이메일 주소 (유일하지만 처음엔 미입력 가능) |
| `FOREIGN KEY` | 외래키 – 다른 테이블의 기본키를 참조 | 주문서의 고객 ID가 고객 테이블을 가리킴 |
| `CHECK` | 특정 조건을 만족해야 함 | 나이는 0 이상이어야 함 |
| `DEFAULT` | 값이 입력되지 않으면 기본값 사용 | 가입일을 안 넣으면 오늘 날짜로 자동 입력 |

#### 실제 예제 1: 고객 테이블 생성

**요구사항:**
- 테이블 이름: `고객`
- 컬럼: 고객아이디, 고객이름, 나이, 등급, 직업, 적립금
- 기본키: 고객아이디
- 반드시 입력해야 하는 컬럼: 고객이름, 등급
- 적립금은 입력하지 않으면 기본값 0

```sql
-- 고객 테이블 생성
CREATE TABLE 고객 (
    고객아이디   VARCHAR(20)   NOT NULL,   -- 고객 ID, 최대 20자 문자열, 반드시 입력
    고객이름     VARCHAR(10)   NOT NULL,   -- 고객 이름, 최대 10자, 반드시 입력
    나이         INT,                      -- 나이, 정수형, 선택 입력
    등급         VARCHAR(10)   NOT NULL,   -- 등급, 최대 10자, 반드시 입력
    직업         VARCHAR(20),              -- 직업, 최대 20자, 선택 입력
    적립금       INT           DEFAULT 0,  -- 적립금, 정수형, 미입력 시 0으로 자동 설정
    PRIMARY KEY (고객아이디)               -- 고객아이디를 기본키로 설정
);
```

**코드 한 줄씩 설명:**
- `VARCHAR(20)`: 최대 20글자의 가변 길이 문자열 타입입니다. 실제 입력된 만큼만 공간을 사용합니다.
- `NOT NULL`: 이 컬럼은 반드시 값이 있어야 합니다. 비워두면 오류가 발생합니다.
- `INT`: 정수(Integer) 타입으로, 소수점 없는 숫자를 저장합니다.
- `DEFAULT 0`: 적립금을 입력하지 않고 레코드를 추가하면 자동으로 0이 들어갑니다.
- `PRIMARY KEY (고객아이디)`: 고객아이디를 기본키로 지정합니다. 중복 불가, NULL 불가입니다.

#### 실제 예제 2: 제품 테이블 생성 (CHECK 제약 조건 포함)

**요구사항:**
- 기본키: 제품번호
- 재고량은 항상 0개 이상 10,000개 이하 유지

```sql
-- 제품 테이블 생성 (CHECK 제약 조건 포함)
CREATE TABLE 제품 (
    제품번호     CHAR(3)       NOT NULL,    -- 제품번호, 고정 3자리 문자, 반드시 입력
    제품명       VARCHAR(20),               -- 제품명, 최대 20바이트 (한글 10자)
    재고량       INT,                       -- 재고량, 정수형
    단가         INT,                       -- 단가, 정수형
    제조업체     VARCHAR(20),               -- 제조업체명, 최대 20바이트
    PRIMARY KEY (제품번호),                 -- 제품번호를 기본키로 설정
    CHECK (재고량 >= 0 AND 재고량 <= 10000) -- 재고량은 0 이상 10000 이하만 허용
);
```

**CHECK 제약 조건 설명:**
- `CHECK (재고량 >= 0 AND 재고량 <= 10000)`: 재고량 컬럼에 -1이나 10001 같은 값이 들어오려고 하면 데이터베이스가 자동으로 거부합니다. 마치 입구 경비원이 규격에 맞지 않는 상품의 반입을 막는 것과 같습니다.

:::note CHAR vs VARCHAR 차이
- `CHAR(3)`: 항상 3자리를 고정으로 사용합니다. "AB"를 넣어도 "AB "처럼 공백으로 채워 3자리를 유지합니다.
- `VARCHAR(20)`: 최대 20자까지 쓸 수 있고, 실제 입력된 글자 수만큼만 저장합니다. 공간 효율적입니다.
:::

#### 실제 예제 3: 주문 테이블 생성 (FOREIGN KEY 포함)

**요구사항:**
- 기본키: 주문번호
- 외래키 1: 주문고객 → 고객 테이블의 고객아이디 참조
- 외래키 2: 주문제품 → 제품 테이블의 제품번호 참조

```sql
-- 주문 테이블 생성 (외래키 2개 포함)
CREATE TABLE 주문 (
    주문번호     INT           NOT NULL,    -- 주문번호, 정수형, 반드시 입력
    주문고객     VARCHAR(20),               -- 주문한 고객의 ID (외래키)
    주문제품     CHAR(3),                   -- 주문한 제품의 번호 (외래키)
    주문수량     INT,                       -- 주문 수량
    주문일자     DATE,                      -- 주문 날짜
    PRIMARY KEY (주문번호),                 -- 주문번호를 기본키로 설정
    FOREIGN KEY (주문고객) REFERENCES 고객(고객아이디),  -- 고객 테이블의 고객아이디를 참조
    FOREIGN KEY (주문제품) REFERENCES 제품(제품번호)     -- 제품 테이블의 제품번호를 참조
);
```

**FOREIGN KEY 설명:**
- `FOREIGN KEY (주문고객) REFERENCES 고객(고객아이디)`: 주문 테이블의 `주문고객` 컬럼은 고객 테이블의 `고객아이디` 컬럼을 가리킵니다. 즉, 존재하지 않는 고객 ID로는 주문을 넣을 수 없습니다. 이것이 바로 **참조 무결성(Referential Integrity)** 입니다.

#### 외래키 옵션 – 삭제/수정 시 어떻게 할 것인가?

참조 무결성을 관리하기 위해 외래키에 옵션을 붙일 수 있습니다. 예를 들어, 고객이 탈퇴해서 고객 테이블에서 삭제될 때, 그 고객의 주문 기록은 어떻게 할까요?

| 옵션 | 한국어 의미 | 동작 방식 |
|------|-----------|----------|
| `RESTRICT` (또는 `NO ACTION`) | 거부 | 참조하는 데이터가 있으면 삭제/수정 불가 |
| `CASCADE` | 연쇄 삭제/수정 | 부모 데이터 삭제 시 자식 데이터도 함께 삭제 |
| `SET NULL` | NULL로 변경 | 부모 데이터 삭제 시 자식의 해당 컬럼을 NULL로 설정 |
| `SET DEFAULT` | 기본값으로 변경 | 부모 데이터 삭제 시 자식의 해당 컬럼을 기본값으로 설정 |

```sql
-- 외래키에 CASCADE 옵션 적용 예시
FOREIGN KEY (주문고객)
    REFERENCES 고객(고객아이디)
    ON DELETE CASCADE   -- 고객 삭제 시 해당 고객의 주문도 함께 삭제
    ON UPDATE CASCADE   -- 고객 ID 변경 시 주문 테이블의 ID도 함께 변경
```

:::warning RESTRICT vs CASCADE 선택 주의
`CASCADE`는 편리하지만 위험합니다. 실수로 부모 데이터를 삭제하면 관련된 모든 자식 데이터가 사라집니다. 중요한 업무 데이터베이스에서는 대부분 `RESTRICT`를 기본으로 사용하여 실수로 인한 데이터 손실을 방지합니다.
:::

### 🔧 ALTER TABLE – 테이블 수정하기

`ALTER TABLE`은 이미 만들어진 테이블의 구조를 변경할 때 사용합니다. 처음 설계했을 때 미처 생각하지 못한 컬럼을 나중에 추가하거나, 더 이상 필요 없는 컬럼을 제거하거나, 제약 조건을 추가/삭제할 때 씁니다.

#### 컬럼 추가 – ADD

```sql
-- 고객 테이블에 '가입날짜' 컬럼 추가
ALTER TABLE 고객
ADD 가입날짜 DATETIME;  -- DATETIME 타입의 가입날짜 컬럼을 새로 추가

-- 고객 테이블에 '전화번호' 컬럼을 NOT NULL 조건과 함께 추가
ALTER TABLE 고객
ADD 전화번호 VARCHAR(15) NOT NULL;  -- 전화번호 컬럼, 최대 15자, 반드시 입력 필요
```

#### 컬럼 삭제 – DROP COLUMN

```sql
-- 고객 테이블에서 '등급' 컬럼 삭제 (CASCADE: 이 컬럼을 참조하는 것도 함께 삭제)
ALTER TABLE 고객
DROP COLUMN 등급 CASCADE;

-- RESTRICT: 이 컬럼을 참조하는 것이 있으면 삭제 거부
ALTER TABLE 고객
DROP COLUMN 등급 RESTRICT;
```

#### 제약 조건 추가 – ADD CONSTRAINT

```sql
-- 고객 테이블에 '나이는 20세 이상이어야 한다'는 제약 조건 추가
ALTER TABLE 고객
ADD CONSTRAINT check_age CHECK (나이 >= 20);
-- check_age: 제약 조건의 이름 (나중에 삭제할 때 이 이름으로 참조함)
```

#### 제약 조건 삭제 – DROP CONSTRAINT

```sql
-- 위에서 추가한 check_age 제약 조건 삭제
ALTER TABLE 고객
DROP CONSTRAINT check_age;  -- 제약 조건 이름만 알면 바로 삭제 가능
```

### 🗑️ DROP TABLE – 테이블 삭제하기

`DROP TABLE`은 테이블 자체를 완전히 없애는 명령어입니다. 테이블 안의 데이터뿐만 아니라 테이블 구조(스키마)까지 모두 삭제됩니다. 마치 엑셀에서 시트 자체를 삭제하는 것과 같습니다.

```sql
-- ❌ 잘못된 예 – 외래키로 참조당하고 있는 테이블을 그냥 삭제하려 할 때
-- (다른 테이블이 참조 중이면 오류 발생 가능)
DROP TABLE 고객;

-- ✅ 올바른 예 1 – RESTRICT 옵션으로 안전하게 (참조하는 테이블이 있으면 삭제 거부)
DROP TABLE 고객 RESTRICT;

-- ✅ 올바른 예 2 – CASCADE 옵션 (함께 참조하는 다른 테이블도 연쇄적으로 삭제)
DROP TABLE 고객 CASCADE;
```

:::danger DROP TABLE은 되돌릴 수 없습니다
`DROP TABLE`을 실행하면 **해당 테이블과 모든 데이터가 영구적으로 삭제**됩니다. `ROLLBACK`으로도 되돌릴 수 없는 DDL 명령어이므로, 실행 전에 반드시 백업을 확인하고 신중하게 사용해야 합니다.
:::

### SQL 작성 시 기본 규칙

```sql
-- ✅ 대소문자 구분 없음 (명령어)
SELECT * FROM 고객;   -- 이것과
select * from 고객;   -- 이것은 완전히 동일합니다

-- ⚠️ 문자열 데이터는 대소문자 구분 있음
-- 'Korea'와 'korea'는 다른 값입니다 (ASCII 코드가 다르기 때문)

-- ✅ 문장의 끝은 반드시 세미콜론(;)으로 마무리
SELECT * FROM 고객;  -- 세미콜론 필수!
```

## 5️⃣ DML의 꽃 – SELECT 완전 정복

### SELECT의 기본 구조

`SELECT`는 SQL에서 가장 자주 사용하는 명령어입니다. 데이터베이스에서 원하는 데이터를 **조회(검색)** 할 때 사용합니다. 실무에서 개발자가 하루에 수십 번씩 쓰는 명령어이므로 완벽하게 익혀두는 것이 중요합니다.

**SELECT의 완전한 기본 문법:**

```sql
SELECT [DISTINCT] 컬럼명1, 컬럼명2, ...  -- 출력할 컬럼 지정 (* 은 전체)
FROM 테이블명                              -- 데이터를 가져올 테이블
WHERE 조건식                              -- 필터링 조건
GROUP BY 컬럼명                           -- 그룹화 기준
HAVING 그룹조건식                         -- 그룹에 대한 필터링 조건
ORDER BY 컬럼명 [ASC|DESC];              -- 정렬 기준 및 방향
```

각 절의 역할을 정리하면 다음과 같습니다.

| 절(Clause) | 역할 | 비유 |
|-----------|------|------|
| `SELECT` | 어떤 컬럼을 출력할지 | 카탈로그에서 어떤 정보를 볼지 선택 |
| `DISTINCT` | 중복 제거 | 똑같은 항목 하나로 합치기 |
| `FROM` | 어떤 테이블에서 가져올지 | 어느 창고에서 꺼낼지 |
| `WHERE` | 어떤 조건에 맞는 데이터만 | 창고에서 특정 조건 상품만 꺼내기 |
| `GROUP BY` | 특정 기준으로 묶기 | 지역별, 부서별로 묶어서 보기 |
| `HAVING` | 그룹에 대한 조건 | 묶인 그룹 중에서 또 필터링 |
| `ORDER BY` | 정렬 | 가나다 순, 가격 순으로 정렬 |

### 예제 데이터 소개

아래 두 테이블을 기준으로 예제를 살펴보겠습니다.

**EMPLOYEE (직원) 테이블:**

| EmpNo | EmpName | Title  | Salary | DNo |
|-------|---------|--------|--------|-----|
| 1     | 김철수  | 사원   | 250    | 1   |
| 2     | 이영희  | 과장   | 400    | 1   |
| 3     | 박민수  | 대리   | 300    | 2   |
| 4     | 최지영  | 과장   | 450    | 2   |
| 5     | 이성진  | 사원   | 200    | 3   |
| 6     | 이민경  | 과장   | 380    | 1   |

**DEPARTMENT (부서) 테이블:**

| DNo | DName  | Floor |
|-----|--------|-------|
| 1   | 영업부 | 8     |
| 2   | 개발부 | 10    |
| 3   | 인사부 | 7     |

### 기본 SELECT 예제

#### 예제 1: 모든 데이터 조회

```sql
-- DEPARTMENT 테이블의 모든 데이터를 전부 출력
-- * (별표)는 모든 컬럼을 의미합니다
SELECT *
FROM DEPARTMENT;

-- 결과: DNo, DName, Floor 컬럼이 모두 출력됩니다
```

#### 예제 2: 특정 컬럼만 조회

```sql
-- DEPARTMENT 테이블에서 부서번호와 부서명만 출력 (층수는 제외)
SELECT DNo, DName
FROM DEPARTMENT;

-- 컬럼을 쉼표로 구분하여 원하는 것만 지정합니다
```

#### 예제 3: DISTINCT – 중복 제거

```sql
-- ❌ 중복 포함: 모든 직원의 직급을 조회 (같은 직급이 여러 번 나옴)
SELECT Title
FROM EMPLOYEE;
-- 결과: 사원, 과장, 대리, 과장, 사원, 과장 (중복 있음)

-- ✅ 중복 제거: 직급의 종류만 조회
SELECT DISTINCT Title
FROM EMPLOYEE;
-- 결과: 사원, 과장, 대리 (중복 제거)
```

#### 예제 4: WHERE 절 – 조건 필터링

```sql
-- 부서번호(DNo)가 2인 직원의 모든 정보를 조회
SELECT *
FROM EMPLOYEE
WHERE DNo = 2;
-- 결과: 박민수(대리, 300), 최지영(과장, 450)만 출력

-- 특정 컬럼만 조회하면서 조건 적용
SELECT EmpName, Title, Salary
FROM EMPLOYEE
WHERE DNo = 2;
```

#### 예제 5: LIKE – 문자열 패턴 검색

```sql
-- 이름이 '이'로 시작하는 직원 검색 (% = 뒤에 어떤 글자든 OK)
SELECT EmpName, Title, DNo
FROM EMPLOYEE
WHERE EmpName LIKE '이%';
-- 결과: 이영희, 이성진, 이민경

-- 이름이 '수'로 끝나는 직원 검색
SELECT EmpName
FROM EMPLOYEE
WHERE EmpName LIKE '%수';
-- 결과: 김철수, 박민수

-- 이름이 정확히 3글자이고 가운데 글자가 '민'인 직원
-- _ (언더바) 하나 = 정확히 한 글자
SELECT EmpName
FROM EMPLOYEE
WHERE EmpName LIKE '_민_';
-- 결과: 박민수, 이민경
```

:::tip % 와 _ 의 차이
- `%`: 글자 수 관계없이 어떤 문자든 허용합니다. 0글자도 됩니다.
- `_` (언더바): 정확히 한 글자를 나타냅니다. `'이__'`이면 이로 시작하는 정확히 3글자를 의미합니다.
:::

#### 예제 6: AND / OR – 복합 조건

```sql
-- ✅ AND: 직급이 '과장'이면서(AND) 부서번호가 1번인 직원의 이름과 급여
SELECT EmpName, Salary
FROM EMPLOYEE
WHERE Title = '과장' AND DNo = 1;
-- 결과: 이영희(400), 이민경(380)

-- ✅ OR: 부서번호가 1번이거나(OR) 3번인 직원 전체 정보
SELECT *
FROM EMPLOYEE
WHERE DNo = 1 OR DNo = 3;
-- 결과: DNo가 1 또는 3인 모든 직원
```

#### 예제 7: BETWEEN – 범위 조건

```sql
-- 급여가 300만원 이상 450만원 이하인 직원의 이름, 직급, 급여 조회

-- ✅ 방법 1: BETWEEN 연산자 사용 (300과 450 모두 포함)
SELECT EmpName, Title, Salary
FROM EMPLOYEE
WHERE Salary BETWEEN 300 AND 450;

-- ✅ 방법 2: 비교 연산자 직접 사용 (결과 동일)
SELECT EmpName, Title, Salary
FROM EMPLOYEE
WHERE Salary >= 300 AND Salary <= 450;

-- 두 방법의 결과는 완전히 동일합니다
-- 결과: 이영희(400), 박민수(300), 최지영(450), 이민경(380)
```

#### 예제 8: IN – 목록 중 하나와 일치

```sql
-- 부서번호가 1번이거나 3번인 직원 전체 정보 조회

-- ✅ IN 사용: 목록 안의 값 중 하나라도 일치하면 선택
SELECT *
FROM EMPLOYEE
WHERE DNo IN (1, 3);

-- 위와 동일한 결과를 OR로 표현
SELECT *
FROM EMPLOYEE
WHERE DNo = 1 OR DNo = 3;

-- IN은 OR의 축약형이라고 이해하면 됩니다
-- IN (1, 2, 3, 4) = DNo=1 OR DNo=2 OR DNo=3 OR DNo=4
```

#### 예제 9: SELECT 절에 수식 사용하기

```sql
-- 과장 직급 직원들의 이름, 현재 급여, 10% 인상 후 급여를 조회
-- (실제 데이터를 변경하는 것이 아니라, 계산 결과만 화면에 보여주는 것)
SELECT
    EmpName,                    -- 직원 이름
    Salary,                     -- 현재 급여
    Salary * 1.1 AS 인상후급여  -- 10% 인상된 급여 (AS로 컬럼 별칭 지정)
FROM EMPLOYEE
WHERE Title = '과장';

-- * 1.1은 현재 급여의 100% + 10% = 110% = 1.1을 곱하는 것
-- 이 쿼리는 데이터를 변경하지 않습니다. 변경은 UPDATE 명령어를 써야 합니다.
```

#### 예제 10: 부정 연산자 – 같지 않음

```sql
-- ❌ SQL에서 != 기호는 표준 문법이 아닌 경우가 있습니다
-- (일부 DBMS에서는 지원하지 않을 수 있음)

-- ✅ 올바른 방법 1: <> 사용 (SQL 표준)
SELECT EmpName, Salary
FROM EMPLOYEE
WHERE Title = '과장' AND DNo <> 1;
-- 직급이 과장이면서 부서번호가 1번이 아닌 직원

-- ✅ 올바른 방법 2: NOT 사용
SELECT EmpName, Salary
FROM EMPLOYEE
WHERE Title = '과장' AND NOT DNo = 1;
```

#### 예제 11: IS NULL / IS NOT NULL – 빈 값 확인

```sql
-- 나이 정보가 없는(NULL인) 직원 조회
SELECT EmpName
FROM EMPLOYEE
WHERE 나이 IS NULL;

-- ❌ 잘못된 방법 – NULL은 = 으로 비교할 수 없습니다
SELECT EmpName
FROM EMPLOYEE
WHERE 나이 = NULL;  -- 이렇게 하면 아무 결과도 나오지 않습니다!

-- ✅ 나이 정보가 있는 직원만 조회
SELECT EmpName
FROM EMPLOYEE
WHERE 나이 IS NOT NULL;
```

:::danger NULL 비교 시 절대 = 사용 금지
NULL은 "값이 없음"을 의미합니다. `NULL = NULL`은 SQL에서 **참(TRUE)이 아닙니다**. 마치 "알 수 없음 = 알 수 없음"이 같다고 할 수 없는 것처럼요. NULL과의 비교는 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용해야 합니다.
:::

### ORDER BY – 정렬하기

```sql
-- 직원 전체를 급여 오름차순(낮은 것부터)으로 정렬
SELECT EmpName, Salary
FROM EMPLOYEE
ORDER BY Salary ASC;  -- ASC = Ascending (오름차순) = 1, 2, 3, 4...

-- 직원 전체를 급여 내림차순(높은 것부터)으로 정렬
SELECT EmpName, Salary
FROM EMPLOYEE
ORDER BY Salary DESC;  -- DESC = Descending (내림차순) = 9, 8, 7, 6...

-- ASC는 기본값이므로 생략 가능 (ORDER BY Salary 라고만 써도 오름차순)
```

### 연산자 우선순위

여러 연산자를 같이 쓸 때는 실행 순서가 있습니다. 수학에서 곱셈이 덧셈보다 먼저 계산되는 것처럼요.

1. **괄호** `( )` - 무조건 최우선
2. **NOT** - 부정 연산
3. **비교 연산자** `=, <>, <, >, <=, >=, BETWEEN, IN, LIKE, IS NULL`
4. **AND** - 논리 곱
5. **OR** - 논리 합

```sql
-- 예: 아래 두 쿼리는 결과가 다릅니다!

-- ❌ 의도와 다른 결과: AND가 OR보다 먼저 실행됨
-- (Title='과장' AND DNo=1) OR DNo=3 으로 해석됨
SELECT * FROM EMPLOYEE
WHERE Title = '과장' AND DNo = 1 OR DNo = 3;

-- ✅ 의도한 결과: 괄호로 명확하게 우선순위 지정
-- Title='과장' 이면서 (DNo=1 OR DNo=3) 으로 해석
SELECT * FROM EMPLOYEE
WHERE Title = '과장' AND (DNo = 1 OR DNo = 3);
```

## 6️⃣ 주의사항 및 핵심 팁

### ⚠️ 초보자가 자주 하는 실수들

#### 실수 1: PRIMARY KEY와 UNIQUE의 혼동

```sql
-- PRIMARY KEY: 기본키 - NULL 불가, 유일해야 함, 테이블당 하나만
-- UNIQUE: 대체키 - NULL 허용, 유일해야 함, 여러 개 가능

-- ✅ 올바른 예
CREATE TABLE 회원 (
    회원ID      VARCHAR(20) PRIMARY KEY,  -- 기본키: NULL 불가, 반드시 유일
    이메일      VARCHAR(50) UNIQUE,       -- 대체키: 유일하지만 처음엔 NULL 가능
    이름        VARCHAR(10) NOT NULL
);
```

#### 실수 2: SELECT에서 존재하지 않는 컬럼 지정

```sql
-- ❌ 오류: 컬럼명 오타 또는 존재하지 않는 컬럼 조회 시 오류 발생
SELECT EmpNmae  -- Name을 Nmae로 오타
FROM EMPLOYEE;

-- ✅ 정확한 컬럼명 사용
SELECT EmpName
FROM EMPLOYEE;
```

#### 실수 3: WHERE 절에서 문자열 비교 시 따옴표 누락

```sql
-- ❌ 오류: 문자열 값에 따옴표 없이 사용
SELECT * FROM EMPLOYEE WHERE Title = 과장;

-- ✅ 문자열은 반드시 작은따옴표(')로 감쌉니다
SELECT * FROM EMPLOYEE WHERE Title = '과장';
```

:::tip SQL 작성 습관
처음부터 좋은 습관을 들이면 나중에 복잡한 쿼리를 작성할 때 훨씬 쉬워집니다.
- **명령어는 대문자**: `SELECT`, `FROM`, `WHERE` 등을 대문자로 써서 가독성을 높입니다.
- **들여쓰기 활용**: 복잡한 쿼리일수록 절마다 줄을 바꾸고 들여쓰기를 합니다.
- **별칭(AS) 활용**: 컬럼명이 복잡하거나 수식이 있을 때 `AS 별칭`으로 이름을 붙여줍니다.
- **주석 달기**: `--` 또는 `/* */`로 코드에 설명을 달아두면 나중에 이해하기 쉽습니다.
:::

### SQL 문 작성 순서 vs 실행 순서

많은 초보자들이 헷갈리는 부분입니다. SQL은 **작성 순서**와 **실행 순서**가 다릅니다.

| 작성 순서 | 실행 순서 | 절 이름 |
|----------|----------|---------|
| 1번 | 3번 | SELECT |
| 2번 | 1번 | FROM |
| 3번 | 2번 | WHERE |
| 4번 | 4번 | GROUP BY |
| 5번 | 5번 | HAVING |
| 6번 | 6번 | ORDER BY |

> 💡 데이터베이스 엔진은 내부적으로 **FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY** 순서로 처리합니다. 이를 이해하면 "왜 WHERE 절에서 SELECT에서 만든 별칭을 쓸 수 없는지" 같은 의문이 해결됩니다.

## 📋 핵심 정리

이 문서에서 배운 핵심 개념을 한눈에 정리합니다.

- **SQL(Structured Query Language)**: 관계형 데이터베이스를 다루기 위한 표준화된 구조적 질의 언어
- **비절차적 언어**: "어떻게"가 아닌 "무엇을"만 명시하면 DB가 알아서 처리
- **SQL 명령어 3분류**: DDL(정의어), DML(조작어), DCL(제어어)
- **DDL 핵심 명령어**: `CREATE TABLE`(생성), `ALTER TABLE`(수정), `DROP TABLE`(삭제)
- **NOT NULL**: 해당 컬럼은 반드시 값이 입력되어야 함 (빈값 불허)
- **PRIMARY KEY**: 기본키 – 유일하고 NULL이 불가능한 컬럼, 테이블당 하나만 지정 가능
- **UNIQUE**: 대체키 – 유일하지만 NULL은 허용, 여러 개 지정 가능
- **FOREIGN KEY ~ REFERENCES**: 외래키 – 다른 테이블의 기본키를 참조하여 참조 무결성 유지
- **CHECK**: 컬럼에 허용할 값의 범위나 조건을 제한하는 제약 조건
- **DEFAULT**: 값이 입력되지 않을 때 자동으로 채워지는 기본값
- **외래키 옵션**: RESTRICT(삭제 거부), CASCADE(연쇄 삭제), SET NULL(NULL로 변경), SET DEFAULT(기본값으로 변경)
- **SELECT 기본 구조**: `SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY`
- **DISTINCT**: SELECT 결과에서 중복된 행을 제거
- **LIKE 연산자**: `%`는 임의의 글자 수, `_`는 정확히 한 글자를 나타내는 패턴 매칭
- **BETWEEN A AND B**: A 이상 B 이하의 범위를 나타내며, A와 B 모두 포함
- **IN (값1, 값2, ...)**: 목록 안의 값 중 하나라도 일치하면 선택 (OR의 축약형)
- **IS NULL / IS NOT NULL**: NULL 비교 시 반드시 사용, `= NULL`은 사용 불가
- **ORDER BY**: ASC(오름차순, 기본값), DESC(내림차순)로 결과를 정렬
- **SQL vs NoSQL**: SQL은 정형 데이터의 일관성 중심, NoSQL은 비정형 데이터의 가용성·확장성 중심
- **ACID 성질**: 관계형 데이터베이스가 보장해야 하는 원자성, 일관성, 고립성, 지속성
