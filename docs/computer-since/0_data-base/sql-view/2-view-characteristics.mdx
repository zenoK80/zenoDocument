# 🗂️ 뷰(View)의 특성과 동작 방식 완전 정복

데이터베이스를 공부하다 보면 "뷰(View)"라는 개념을 만나게 됩니다. 처음에는 테이블과 비슷해 보여서 헷갈리기 쉽지만, 뷰는 테이블과는 근본적으로 다른 특성을 가지고 있습니다. 이번 문서에서는 뷰가 무엇인지부터 시작해, 왜 필요한지, 어떻게 만들고 사용하는지, 그리고 어떤 제약이 있는지까지 차근차근 살펴보겠습니다.

---

## 🧩 뷰(View)란 무엇인가?

### 뷰의 기본 개념

**뷰(View)** 란 기존에 존재하는 테이블로부터 유도되어 논리적으로 만들어진 **가상의 테이블(Virtual Table)** 입니다. "가상의 테이블"이라는 말이 핵심입니다. 실제로 데이터가 저장된 공간이 아니라, 마치 창문처럼 기존 테이블을 특정 방식으로 바라보는 틀이라고 생각하면 됩니다.

예를 들어, 학교 건물 창문을 생각해 봅시다. 창문 자체가 운동장은 아니지만, 창문을 통해 운동장을 볼 수 있습니다. 창문의 크기나 위치에 따라 운동장의 특정 부분만 보이기도 하죠. 뷰도 마찬가지입니다. 실제 데이터는 기본 테이블(Base Table)에 있고, 뷰는 그 데이터를 특정 방식으로 보여주는 창문 역할을 합니다.

> **핵심 요약:** 기본 테이블은 실제 데이터가 물리적으로 저장된 공간이고, 뷰는 그 데이터를 논리적으로 재구성해서 보여주는 가상의 테이블입니다.

### 물리적 테이블 vs 논리적 뷰

기존의 테이블은 **물리적(Physical)** 으로 구현되어 있습니다. 즉, 하드디스크 같은 저장 장치에 실제 데이터가 쓰여 있습니다. 반면, 뷰는 **논리적(Logical)** 으로만 존재합니다. 뷰 안에는 데이터가 없고, 어떤 테이블에서 어떤 데이터를 가져와서 어떻게 보여줄지에 대한 **정의(Definition)** 만 저장되어 있습니다.

냉장고(기본 테이블)에 식재료(데이터)가 있고, 요리 레시피(뷰의 정의)는 냉장고와 별도로 종이에 적혀 있는 것과 같습니다. 레시피 종이 자체에는 식재료가 없지만, 레시피를 따라 조리하면 냉장고에서 재료를 꺼내 요리를 만들 수 있습니다.

:::info 뷰의 정의는 어디에 저장되나요?
뷰에 대한 정의, 즉 "어느 테이블에서 어떤 속성을 어떤 조건으로 가져온다"는 정보는 **시스템 카탈로그(System Catalog)** 또는 **데이터 사전(Data Dictionary)** 이라고 불리는 곳에 **메타데이터(Metadata, 데이터에 대한 데이터)** 형태로 저장됩니다. 이 메타데이터는 `SELECT ... FROM ... WHERE ...` 형태의 SQL 구문으로 저장됩니다.
:::

## 🔍 왜 뷰가 필요한가? — 뷰의 등장 배경

### 다양한 사용자의 다양한 요구

실제 현업 데이터베이스에는 수십 개, 수백 개의 테이블이 있고, 수많은 사람들이 데이터베이스를 사용합니다. 은행 시스템을 예로 들어봅시다. 은행 계좌 테이블 하나에 수많은 고객 정보가 들어있습니다. 은행 업무를 처리할 때는 입금 처리, 출금 처리, 이체 처리, 잔액 조회 등 다양한 작업이 이루어집니다.

이 모든 작업마다 기본 테이블 전체를 다 보여주는 것은 비효율적이고 위험합니다. 잔액 조회 담당자가 모든 고객의 개인정보까지 볼 필요는 없잖아요. 이처럼 **사용자마다 필요한 데이터가 다르고, 보안 수준도 달라야 하는 상황**에서 뷰가 등장했습니다.

### 보안과 데이터 보호

뷰는 **보안(Security)** 목적으로도 매우 중요하게 사용됩니다. 기본 테이블에는 민감한 정보가 가득할 수 있습니다. 예를 들어 직원 테이블에 이름, 부서, 급여, 주민등록번호가 모두 들어있다면, 일반 직원이 동료의 급여나 주민등록번호를 보아서는 안 됩니다.

이럴 때 이름과 부서만 보여주는 뷰를 만들어 일반 직원에게는 그 뷰만 접근하도록 권한을 설정하면, 민감한 정보를 완전히 감출 수 있습니다. **뷰를 사용하는 일반 사용자는 자신이 보는 데이터가 어느 테이블의 어떤 속성에서 왔는지 알 수 없습니다.** 개발자(뷰를 만든 사람)는 내부 구조를 알고 있지만, 사용자는 그냥 데이터를 보기만 하는 것이지요. 이것이 뷰의 강력한 보안 기능입니다.

### 논리적 독립성 제공

뷰는 **논리적 독립성(Logical Independence)** 을 제공합니다. 기본 테이블의 구조가 변경되더라도 뷰는 영향을 받지 않습니다. 예를 들어 학생 테이블에 새로운 속성 D를 추가해도, 기존에 만들어놓은 뷰는 수정 없이 그대로 사용할 수 있습니다. 테이블 내 데이터의 물리적 위치가 바뀌어도 마찬가지입니다. 이처럼 기본 테이블의 **확장(속성 추가)** 이나 **재구성(물리적 위치 변경)** 이 일어나도 뷰는 영향받지 않는다는 것이 논리적 독립성의 핵심입니다.

## ⚙️ 뷰의 핵심 원리 — 어떻게 동작하는가?

### 논리적 뷰 vs 실체화된 뷰

뷰는 크게 두 종류로 나눌 수 있습니다.

| 구분 | 논리적 뷰 (일반 뷰) | 실체화된 뷰 (Materialized View) |
|---|---|---|
| 데이터 저장 여부 | ❌ 데이터 없음 | ✅ 실제 데이터 저장 |
| 처리 속도 | 느림 (기본 테이블 접근 필요) | 빠름 (직접 데이터 보유) |
| 데이터 동기화 | 항상 최신 (기본 테이블 참조) | 기본 테이블과 동기화 필요 |
| 사용 목적 | 보안, 논리적 독립성 | 성능 향상 |
| 영어 표현 | View (Virtual Table) | Materialized View |

**논리적 뷰**는 데이터를 가지고 있지 않기 때문에, 누군가 뷰를 실행(조회)하면 그 시점에 기본 테이블로 가서 데이터를 가져옵니다. 마치 주머니에 돈이 없는 사람이 물건값을 치를 때 은행(기본 테이블)에 달려가서 돈을 찾아오는 것과 같습니다. 당연히 한 단계를 더 거쳐야 하므로 **성능이 저하**될 수 있습니다.

**실체화된 뷰**는 이 성능 문제를 해결하기 위해 도입되었습니다. 자주 조회되는 데이터를 아예 뷰 자체에 물리적으로 저장해두는 방식입니다. 주머니에 미리 현금을 넣어두는 것처럼, 데이터를 즉시 꺼내줄 수 있으니 빠릅니다. 단, 기본 테이블의 데이터가 바뀌면 뷰의 데이터도 함께 동기화(Synchronize)해 주어야 한다는 관리 부담이 생깁니다.

### 뷰의 실행 시점

논리적 뷰는 **실행 시간(Runtime)에만 구체화**되는 특수한 테이블입니다. 평상시에는 시스템 카탈로그에 정의만 저장된 채 조용히 대기하고 있습니다. 사용자가 뷰에 접근하는 순간, 저장된 SELECT 구문이 실행되어 기본 테이블에서 데이터를 가져와 화면에 보여줍니다.

### 뷰의 확장 가능성

뷰는 반드시 테이블로부터만 만들어지는 것은 아닙니다. **테이블 → 뷰 → 또 다른 뷰** 형태로 뷰를 기반으로 새로운 뷰를 만들 수도 있고, **뷰와 뷰를 조인(Join)** 해서 또 다른 뷰를 만들 수도 있습니다. 이처럼 뷰는 얼마든지 확장하여 다양하게 활용할 수 있습니다.

:::warning 기본 테이블이 삭제되면?
기본 테이블이 삭제되면 그 테이블을 기반으로 만들어진 **모든 뷰도 자동으로 제거**됩니다. 원본 데이터를 가져올 공간이 사라졌기 때문입니다. 이를 **전파 소멸(Propagate Destroy)** 이라고 합니다. 기본 테이블을 삭제할 때는 반드시 그 테이블을 참조하는 뷰가 있는지 먼저 확인하세요.
:::

### 외부 스키마로서의 뷰

데이터베이스 스키마(Schema, 구조 정의)는 세 가지 단계로 나뉩니다: 외부 스키마, 개념 스키마, 내부 스키마. 이 중 **외부 스키마(External Schema)** 가 바로 뷰에 해당합니다. 사용자마다 서로 다른 관점에서 데이터베이스를 바라보는 틀이 외부 스키마이며, 이것이 곧 뷰입니다.

## 🛠️ 뷰 만들기 — CREATE VIEW 문법과 종류

### 기본 문법

뷰를 만들 때는 `CREATE VIEW` 명령어를 사용합니다. 테이블을 만들 때 `CREATE TABLE`을 쓰는 것처럼요.

```sql
-- 뷰 기본 생성 문법
CREATE VIEW 뷰이름 (속성1, 속성2, ...)
AS SELECT 속성1, 속성2, ...
   FROM 기본테이블명
   WHERE 조건
   [WITH CHECK OPTION];
```

- `CREATE VIEW 뷰이름`: 만들 뷰의 이름을 지정합니다.
- `(속성1, 속성2, ...)`: 이 뷰에서 보여줄 속성(컬럼)의 이름을 나열합니다. 생략하면 SELECT 절의 컬럼명을 그대로 사용합니다.
- `AS SELECT ...`: 어떤 테이블에서 어떤 데이터를 가져올지 정의하는 SELECT 구문입니다.
- `WITH CHECK OPTION`: 뷰를 통한 데이터 변경 시 뷰의 정의를 위반하면 실행을 거부합니다.

:::warning 뷰의 AS SELECT에서 사용 불가한 기능
일반 SELECT 문에서는 `UNION`(합집합), `ORDER BY`(정렬) 등을 사용할 수 있지만, **뷰를 정의하는 AS SELECT 절에서는 이러한 기능을 사용할 수 없습니다.** 뷰는 데이터를 정렬하거나 합치는 등의 추가 처리를 자체적으로 수행할 수 없다는 점을 기억하세요.
:::

### WITH CHECK OPTION 이란?

`WITH CHECK OPTION`은 뷰의 정의를 지키기 위한 안전장치입니다. 예를 들어, 컴퓨터공학과 학생들만 포함하는 뷰를 만들었다면, 그 뷰를 통해 다른 학과 데이터를 삽입하거나 수정하려 할 때 실행을 거부합니다.

```sql
-- ✅ WITH CHECK OPTION을 사용한 뷰 생성 예시
CREATE VIEW CS_Student (SNO, SNAME, YEAR)
AS SELECT SNO, SNAME, YEAR
   FROM Student
   WHERE Department = '컴퓨터'
WITH CHECK OPTION;
-- WITH CHECK OPTION: 이 뷰를 통해 데이터를 변경할 때,
-- Department가 '컴퓨터'인 조건을 위반하면 실행을 거부함
```

```sql
-- ❌ 아래 UPDATE는 WITH CHECK OPTION에 의해 거부됨
UPDATE CS_Student
SET Department = '경영학과'  -- 컴퓨터가 아닌 학과로 변경 시도
WHERE SNO = 100;
-- 이 뷰는 Department = '컴퓨터' 조건이 있으므로 경영학과로의 변경 거부!
```

### 뷰의 4가지 생성 방법

뷰는 원본 데이터를 어떻게 선택하느냐에 따라 네 가지 방식으로 만들 수 있습니다.

#### 1️⃣ 열 부분집합 뷰 (Projection View)

필요한 **열(Column, 속성)만 선택**해서 만드는 뷰입니다. 관계 대수에서 **프로젝션(Projection)** 연산에 해당합니다. 전체 속성 중 내가 원하는 속성만 골라서 새로운 뷰를 구성합니다.

```sql
-- ✅ 열 부분집합 뷰 예시 1: 학번과 학과만 보여주는 뷰
CREATE VIEW SVIEW1 (SNO, Department)
AS SELECT SNO, Department
   FROM Student;
-- Student 테이블에서 학번(SNO)과 학과(Department) 두 속성만 선택

-- ✅ 열 부분집합 뷰 예시 2: 이름과 학과만 보여주는 뷰
CREATE VIEW SVIEW2 (SNAME, Department)
AS SELECT SNAME, Department
   FROM Student;
-- Student 테이블에서 이름(SNAME)과 학과(Department) 두 속성만 선택
```

#### 2️⃣ 행 부분집합 뷰 (Selection View)

모든 속성을 포함하되, **특정 조건을 만족하는 행(Row)만 선택**해서 만드는 뷰입니다. 관계 대수에서 **셀렉션(Selection)** 연산에 해당합니다.

```sql
-- ✅ 행 부분집합 뷰 예시: 4학년 학생들만 보여주는 뷰
CREATE VIEW Senior_Student (SNO, SNAME, YEAR, Department)
AS SELECT SNO, SNAME, YEAR, Department
   FROM Student
   WHERE YEAR = 4;
-- 모든 속성을 가져오되, 학년(YEAR)이 4학년인 학생들의 행만 선택
-- 이것이 행 부분집합 뷰 = 셀렉션(Selection)을 통해 만들어진 뷰
```

#### 3️⃣ 통계적 요약 뷰 (Statistical Summary View)

**집계 함수(Aggregate Function)** 를 이용하여 총점, 평균 등 **요약 정보**를 보여주는 뷰입니다. `GROUP BY`와 함께 사용되는 경우가 많습니다.

```sql
-- ✅ 통계적 요약 뷰 예시: 강좌별 중간고사 평균을 보여주는 뷰
CREATE VIEW CourseStats (CNO, AvgMidterm)
AS SELECT CNO, AVG(Midterm)
   FROM Enroll
   GROUP BY CNO;
-- Enroll 테이블에서 강좌번호(CNO)를 기준으로 그룹을 묶고
-- 각 강좌의 중간고사(Midterm) 평균값을 계산하여 뷰로 만듦
-- 통계 데이터도 뷰로 만들 수 있다는 것이 핵심!
```

#### 4️⃣ 조인 뷰 (Join View)

**두 개 이상의 테이블을 조인(Join)** 하여 만드는 뷰입니다. 여러 테이블에 흩어진 정보를 하나의 뷰로 통합해서 볼 수 있습니다.

```sql
-- ✅ 조인 뷰 예시: 장학생 목록 뷰 (Student 테이블 + Enroll 테이블 조인)
CREATE VIEW Scholarship_Student (SNO, SNAME, Department, Final)
AS SELECT S.SNO, S.SNAME, S.Department, E.Final
   FROM Student S, Enroll E          -- 두 테이블을 조인
   WHERE S.SNO = E.SNO               -- 조인 조건: 두 테이블의 학번이 같은 것
     AND E.Final > 95;               -- 추가 조건: 기말고사 점수가 95점 초과 (장학생)
-- Student 테이블의 SNO와 Enroll 테이블의 SNO가 같은 레코드를 조인하고
-- 기말고사 점수가 95점 초과인 학생들을 장학생 뷰로 만듦
```

### 컴퓨터공학과 학생 뷰 생성 전체 예시

```sql
-- ✅ 실제 뷰 생성 전체 예시: 컴퓨터공학과 학생 뷰
CREATE VIEW CS_Student (SNO, SNAME, YEAR)
-- CS_Student: 뷰의 이름
-- (SNO, SNAME, YEAR): 이 뷰에서 보여줄 속성 이름 (학번, 이름, 학년)
AS SELECT SNO, SNAME, YEAR
-- SELECT: 가져올 데이터 항목
   FROM Student
-- FROM: 데이터를 가져올 기본 테이블 이름
   WHERE Department = '컴퓨터'
-- WHERE: 조건 = 학과가 '컴퓨터'인 학생만
WITH CHECK OPTION;
-- WITH CHECK OPTION: 이 뷰를 통해 변경 시 Department='컴퓨터' 조건 위반 불가

-- 이 뷰를 실행하면 Department가 '컴퓨터'인 학생들의
-- 학번, 이름, 학년 정보만 화면에 출력됨
-- 예) 100번, 300번, 400번 학생 3명이 출력
```

## 🗑️ 뷰 변경과 삭제 — ALTER VIEW와 DROP VIEW

### ALTER VIEW는 사용하지 않는다

뷰는 `ALTER VIEW` 명령어로 변경하지 않습니다. 왜냐하면 뷰는 자체 데이터를 가지고 있지 않고 논리적인 정의만 가지고 있기 때문에, 굳이 수정할 필요 없이 **삭제하고 다시 만들면 그만**이기 때문입니다. 실무에서도 뷰를 수정할 때는 기존 뷰를 DROP하고 CREATE VIEW로 새로 만드는 것이 일반적입니다.

```sql
-- ❌ 잘못된 방법: ALTER VIEW는 뷰에 사용하지 않음
ALTER VIEW CS_Student ...;  -- 권장하지 않음

-- ✅ 올바른 방법: 삭제 후 재생성
DROP VIEW CS_Student;       -- 1단계: 기존 뷰 삭제
CREATE VIEW CS_Student ...;  -- 2단계: 새로운 정의로 뷰 재생성
```

### DROP VIEW — 뷰 삭제하기

뷰를 삭제할 때는 `DROP VIEW` 명령어를 사용합니다. 이때 중요한 옵션이 두 가지 있습니다: `RESTRICT`와 `CASCADE`.

| 옵션 | 설명 | 상황 |
|---|---|---|
| `RESTRICT` (기본값) | 다른 뷰가 이 뷰를 참조하고 있으면 삭제를 **거부** | 안전 우선 |
| `CASCADE` | 이 뷰를 참조하는 다른 뷰들도 **연쇄적으로 모두 삭제** | 강제 삭제 |

```sql
-- ✅ RESTRICT 옵션 사용 예시
DROP VIEW Department_Size RESTRICT;
-- Department_Size 뷰를 삭제하려는데,
-- 만약 이 뷰를 참조하는 다른 뷰가 있다면 삭제를 거부함
-- RESTRICT는 기본값이므로 생략해도 동일하게 동작함

-- ✅ CASCADE 옵션 사용 예시
DROP VIEW ANA CASCADE;
-- ANA 뷰를 삭제할 때, 이 뷰를 참조하는 SS1 등의 뷰들도
-- 연쇄적으로 함께 삭제됨 (주의해서 사용해야 함!)
```

:::danger CASCADE 사용 시 주의사항
`CASCADE` 옵션을 사용하면 해당 뷰를 직접·간접으로 참조하는 **모든 뷰가 연쇄 삭제**됩니다. 삭제된 뷰는 복구할 수 없으므로 CASCADE를 사용하기 전에 반드시 참조 관계를 확인하세요.
:::

## 🚫 뷰의 데이터 조작 제약 — 변경이 안 되는 경우

### 뷰를 통한 데이터 변경의 원칙

이론적으로 뷰를 통해 삽입(INSERT), 삭제(DELETE), 갱신(UPDATE), 검색(SELECT) 모두 가능합니다. 하지만 실제로는 매우 제한적입니다. 뷰는 자체 데이터를 가지지 않는 가상 테이블이기 때문에, 뷰를 통한 변경은 실제로 기본 테이블을 변경하는 것이어야 하는데, 이것이 항상 가능하지는 않습니다.

### 변경이 거부되는 6가지 상황

다음과 같은 경우에는 뷰를 통한 데이터 변경(삽입·삭제·갱신)이 **거부**됩니다.

**① 산술식이나 집계 함수가 포함된 경우**

뷰 정의에 `SUM()`, `AVG()`, `COUNT()` 같은 집계 함수나 `A + B` 같은 수식이 포함되어 있으면 변경 불가입니다. 계산된 결과 값을 어떻게 원래 테이블에 반영해야 할지 알 수 없기 때문입니다.

```sql
-- ❌ 집계 함수가 포함된 뷰를 통한 데이터 변경 불가
CREATE VIEW CourseStats AS
  SELECT CNO, AVG(Midterm) AS AvgMidterm FROM Enroll GROUP BY CNO;

-- 아래 UPDATE는 거부됨
UPDATE CourseStats SET AvgMidterm = 90 WHERE CNO = 'C001';
-- AVG() 계산 결과를 역으로 원본 테이블에 반영하는 것은 불가능
```

**② GROUP BY가 사용된 경우**

뷰 정의에 `GROUP BY`로 데이터가 그룹으로 묶여 있는 경우 변경이 불가합니다. 그룹화된 결과를 다시 개별 레코드로 분리해서 원본 테이블에 반영하는 것이 불가능하기 때문입니다.

**③ DISTINCT로 중복이 제거된 경우**

`DISTINCT`를 사용해 중복 값을 제거했다면, 제거된 중복 데이터를 어떤 원본 레코드에 매핑할지 알 수 없어 변경이 불가합니다.

**④ 두 개 이상의 테이블을 조인한 경우 (조인 뷰)**

여러 테이블을 조인해서 만든 뷰는 데이터 변경이 원칙적으로 제한됩니다. 변경된 값이 어느 테이블에 반영되어야 하는지 모호하기 때문입니다.

**⑤ WITH CHECK OPTION으로 정의 위반이 감지된 경우**

앞서 설명한 대로, `WITH CHECK OPTION`이 설정된 뷰에서 뷰의 정의를 위반하는 변경 시도는 거부됩니다.

**⑥ NOT NULL 속성이 뷰에 포함되지 않은 경우**

이 경우가 특히 중요합니다. 기본 테이블의 속성 A, B, C가 모두 `NOT NULL`(반드시 값을 입력해야 함) 제약이 걸려 있는데, 뷰를 A와 B만으로 만들었다면 어떻게 될까요?

```sql
-- 기본 테이블 구조 (A, B, C 모두 NOT NULL)
CREATE TABLE Base (
  A INT NOT NULL,   -- 반드시 값 필요
  B INT NOT NULL,   -- 반드시 값 필요
  C INT NOT NULL    -- 반드시 값 필요
);

-- ❌ 문제가 생기는 뷰: C 속성이 NOT NULL인데 뷰에 포함되지 않음
CREATE VIEW AB_View (A, B)
AS SELECT A, B FROM Base;

-- 이 뷰를 통해 INSERT 시도
INSERT INTO AB_View VALUES (1, 2);
-- A=1, B=2는 입력 가능하지만,
-- C는 뷰에 없으므로 값을 입력할 수 없음 → C에 NULL이 들어가려 함
-- 하지만 C는 NOT NULL이므로 무결성(데이터 정확성) 위반!
-- 따라서 이 삽입은 거부됨
```

이처럼 NOT NULL 속성이 뷰에서 빠져 있으면 삽입 시 해당 속성에 NULL이 들어가려 하여 **무결성(Integrity, 데이터의 정확성·완전성)** 이 깨지기 때문에 변경이 거부됩니다.

## ✅ 뷰의 장점과 단점 정리

### 뷰의 장점 🌟

**① 논리적 독립성 제공**

기본 테이블 구조가 바뀌어도 뷰는 영향받지 않습니다. 새 속성이 추가되거나 데이터 위치가 재구성되어도 뷰를 사용하는 애플리케이션은 수정하지 않아도 됩니다.

**② 강력한 보안 기능**

민감한 데이터를 숨기고 필요한 데이터만 사용자에게 노출할 수 있습니다. 뷰의 정의가 사용자에게 공개되지 않으므로 내부 테이블 구조도 보호됩니다.

**③ 데이터 관리의 단순화**

필요한 데이터만 뷰로 정의해두면, 복잡한 SQL을 매번 작성하지 않아도 뷰 이름만으로 원하는 데이터에 접근할 수 있습니다.

**④ 사용자별 맞춤형 데이터 제공**

사용자의 역할과 필요에 따라 다양한 뷰를 만들어 각자에게 맞는 데이터 화면을 제공할 수 있습니다.

### 뷰의 단점 ⚠️

**① ALTER VIEW 사용 불가**

뷰는 수정이 안 되므로 삭제 후 재생성해야 합니다. 번거로울 수 있습니다.

**② 삽입·삭제·갱신에 많은 제약**

앞서 설명한 여러 가지 이유로 인해 뷰를 통한 데이터 변경에는 제약이 많습니다.

**③ 인덱스 사용 불가**

논리적 가상 테이블이므로 **인덱스(Index, 빠른 검색을 위한 색인)** 를 적용할 수 없습니다. 검색 성능 최적화에 한계가 있습니다.

**④ 성능 저하 가능성**

뷰를 통해 데이터를 조회할 때마다 기본 테이블에 접근해야 하므로, 뷰를 과도하게 사용하면 성능이 저하될 수 있습니다. 이 문제의 해결책이 바로 **실체화된 뷰(Materialized View)** 입니다.

:::tip 성능 저하 해결책: 실체화된 뷰
자주 조회되는 뷰가 있다면 **실체화된 뷰(Materialized View)** 를 고려해보세요. 실체화된 뷰는 실제 데이터를 물리적으로 저장하므로 조회 속도가 훨씬 빠릅니다. 단, 기본 테이블과의 **데이터 동기화(Synchronization)** 를 항상 유지해야 한다는 관리 부담이 생깁니다.
:::

## 🔄 뷰 생성 방식의 두 가지 분류

뷰를 만드는 방식은 크게 두 가지로 분류됩니다.

### 질의 수정 방식 (Query Modification)

기본 테이블을 가지고 **논리적으로만** 만들어지는 일반 뷰입니다. 물리적인 저장 공간을 사용하지 않습니다. 뷰를 실행할 때마다 기본 테이블에 접근하여 데이터를 가져옵니다. 우리가 일반적으로 "뷰"라고 부르는 것이 이 방식입니다.

- **특징:** 항상 최신 데이터 반영, 별도 저장 공간 불필요
- **단점:** 매번 기본 테이블 접근으로 성능 저하 가능

### 뷰 형성 방식 (View Materialization)

아예 물리적인 테이블처럼 뷰를 만들어 **실제 데이터를 저장**하는 방식입니다. **실체화된 뷰**가 이에 해당합니다.

- **특징:** 빠른 조회 속도
- **단점:** 기본 테이블과 뷰의 데이터를 항상 동기화 유지 필요. 기본 테이블 변경 시 뷰도 업데이트, 뷰 변경 시 기본 테이블도 업데이트해야 함

## 📋 뷰 관련 전체 SQL 명령어 요약

```sql
-- ✅ 뷰 생성 기본 형태
CREATE VIEW 뷰이름 (속성목록)
AS SELECT ...
   FROM 테이블명
   WHERE 조건
   [WITH CHECK OPTION];

-- ✅ 단순 뷰 생성 (속성 목록 생략 가능)
CREATE VIEW MyView
AS SELECT SNO, SNAME FROM Student WHERE Department = '컴퓨터';

-- ✅ WITH CHECK OPTION 포함 뷰 생성
CREATE VIEW CS_View (SNO, SNAME, YEAR)
AS SELECT SNO, SNAME, YEAR FROM Student
   WHERE Department = '컴퓨터'
WITH CHECK OPTION;
-- 이 뷰를 통해 Department를 '컴퓨터' 외의 값으로 변경하면 자동 거부

-- ✅ 뷰 삭제 (RESTRICT: 참조하는 뷰가 있으면 삭제 거부 - 기본값)
DROP VIEW CS_View RESTRICT;

-- ✅ 뷰 삭제 (CASCADE: 참조하는 모든 뷰 연쇄 삭제)
DROP VIEW CS_View CASCADE;

-- ✅ 뷰를 통한 데이터 조회 (일반 테이블처럼 사용)
SELECT * FROM CS_View;
SELECT SNO, SNAME FROM CS_View WHERE YEAR = 2;

-- ✅ 뷰를 통한 데이터 삽입 (제약 없을 경우)
INSERT INTO CS_View VALUES (500, '홍길동', 1);
-- WITH CHECK OPTION이 있는 경우, Department가 '컴퓨터'여야만 가능

-- ✅ 뷰를 통한 데이터 갱신 (제약 없을 경우)
UPDATE CS_View SET YEAR = 3 WHERE SNO = 100;

-- ❌ ALTER VIEW는 사용하지 않음
-- ALTER VIEW CS_View ...;  -- 권장하지 않음. DROP 후 CREATE로 재작성

-- ✅ 뷰 수정이 필요할 때는 삭제 후 재생성
DROP VIEW CS_View;
CREATE VIEW CS_View (SNO, SNAME, YEAR)
AS SELECT SNO, SNAME, YEAR FROM Student
   WHERE Department = '컴퓨터';
```

## 💡 주의사항과 실전 팁

### 뷰 설계 시 반드시 고려할 것

:::tip 뷰 이름 규칙
뷰 이름은 그 뷰가 무슨 데이터를 보여주는지 한눈에 알 수 있도록 명확하게 지어주세요. 예: `CS_Student_View`, `Senior_Student_View`, `Scholarship_View` 등. 나중에 뷰가 많아지면 이름만 보고도 무엇인지 알 수 있어야 유지보수가 편합니다.
:::

:::note NOT NULL 속성과 뷰
뷰를 통해 삽입 작업을 할 계획이라면, 기본 테이블에서 `NOT NULL` 제약이 걸린 속성은 반드시 뷰에 포함시켜야 합니다. 그렇지 않으면 삽입 시 무결성 위반으로 거부됩니다.
:::

:::warning 조인 뷰의 데이터 변경
두 개 이상의 테이블을 조인하여 만든 뷰를 통해서는 데이터 변경(삽입·삭제·갱신)이 일반적으로 제한됩니다. 조인 뷰는 주로 **읽기 전용(Read-Only)** 용도로 사용하세요.
:::

:::danger CASCADE 삭제 남용 금지
`DROP VIEW ... CASCADE`는 연쇄 삭제를 일으키므로, 삭제할 뷰를 참조하는 모든 뷰가 함께 사라집니다. 운영 환경에서는 반드시 참조 관계를 확인한 후 신중하게 사용하세요. 잘못하면 여러 서비스가 한꺼번에 오류를 낼 수 있습니다.
:::

### 개발자와 사용자의 역할 구분

뷰에는 두 가지 원칙이 있습니다.

1. **뷰를 만든 개발자**는 뷰의 정의(어떤 테이블에서 어떤 데이터를 가져오는지)를 정확하게 알고 있어야 합니다.
2. **뷰를 사용하는 일반 사용자**는 뷰의 내부 정의를 알아서는 안 됩니다. 보안을 유지해야 하기 때문입니다.

이 원칙은 현실에서도 그대로 적용됩니다. 앱 사용자가 앱 내부 코드를 볼 필요가 없듯이, 데이터베이스 사용자가 뷰의 내부 구조를 알 필요는 없습니다.

## 📌 핵심 정리

- **뷰(View)** 는 기존 테이블로부터 유도된 논리적 가상 테이블(Virtual Table)로, 자체 데이터를 가지지 않는다.
- **논리적 뷰**는 실제 데이터 없이 정의만 존재하며, 실행 시간에만 구체화된다.
- **실체화된 뷰(Materialized View)** 는 성능 향상을 위해 실제 데이터를 물리적으로 저장하며, 기본 테이블과 데이터 동기화가 필요하다.
- 뷰의 정의는 **시스템 카탈로그(데이터 사전)** 에 SELECT 구문 형태로 메타데이터로 저장된다.
- 뷰는 **외부 스키마**에 해당하며, 사용자마다 다른 관점의 데이터 접근을 가능하게 한다.
- 뷰를 만들 때는 `CREATE VIEW ... AS SELECT ...` 문법을 사용한다.
- `WITH CHECK OPTION`은 뷰의 정의를 위반하는 데이터 변경을 자동으로 거부하는 안전장치이다.
- 뷰는 `ALTER VIEW`로 변경하지 않고, `DROP VIEW` 후 `CREATE VIEW`로 재생성한다.
- `DROP VIEW RESTRICT`는 참조 중인 뷰가 있으면 삭제를 거부하고(기본값), `CASCADE`는 연쇄 삭제한다.
- 집계 함수, GROUP BY, DISTINCT, 조인, WITH CHECK OPTION 위반, NOT NULL 속성 누락 등의 경우 뷰를 통한 데이터 변경이 거부된다.
- 뷰는 **논리적 독립성**, **보안**, **데이터 관리 간소화**, **다양한 응용 지원**이라는 장점을 가진다.
- 뷰의 단점은 **ALTER 불가**, **삽입·삭제·갱신 제약**, **인덱스 사용 불가**, **성능 저하** 가능성이다.
- 뷰는 테이블에서 만들 수 있고, 뷰에서 또 다른 뷰를 만들 수 있으며, 기본 테이블 삭제 시 연관 뷰도 자동 제거된다 (전파 소멸).
- 뷰 생성 방식은 **질의 수정 방식**(논리적 뷰)과 **뷰 형성 방식**(실체화된 뷰)으로 나뉜다.
