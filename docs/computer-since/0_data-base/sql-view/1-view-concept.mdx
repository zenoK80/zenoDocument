# 🗂️ 뷰(View)의 개념과 종류

데이터베이스를 배우다 보면 "뷰(View)"라는 단어를 자주 마주치게 됩니다. 처음 들으면 "뷰? 그냥 테이블 아닌가?"라고 생각할 수 있지만, 뷰는 일반 테이블과는 본질적으로 다른 개념입니다. 이번 문서에서는 뷰가 무엇인지, 왜 필요한지, 어떻게 만들고 관리하는지를 아주 꼼꼼하게, 처음 배우는 분들도 이해할 수 있도록 설명합니다.

---

## 🧐 뷰(View)란 무엇인가?

### 뷰의 기본 개념

**뷰(View)** 는 **기존에 존재하는 테이블로부터 유도되어 논리적으로 만들어지는 가상의 테이블**입니다. 영어로는 **Virtual Table(가상 테이블)** 이라고도 부릅니다.

"가상의 테이블"이라는 말이 조금 낯설 수 있는데, 쉽게 비유하자면 이렇습니다. 실제 도서관에는 수천 권의 책이 꽂혀 있습니다(기본 테이블). 그런데 사서 선생님이 "컴퓨터 관련 도서 목록"이라고 적힌 안내문을 따로 만들어 붙여 두었습니다(뷰). 그 안내문 자체에는 책이 들어 있지 않습니다. 안내문에 적힌 목록을 보고 진짜 책이 꽂힌 서가에 가서 책을 꺼내 오는 것이죠.

이처럼 **뷰는 데이터 자체를 가지고 있지 않고**, 실제 데이터는 항상 기본 테이블(Base Table)에 저장되어 있습니다. 뷰는 단지 "어느 테이블에서, 어떤 조건으로, 어떤 열(컬럼)을 가져와서 보여줄 것인지"에 대한 **정의(Definition)** 만을 가지고 있습니다.

> 🔑 **핵심 요약**: 뷰는 테이블처럼 생겼지만, 데이터가 없는 **가상의 테이블**입니다. 실제 데이터는 기본 테이블에 있고, 뷰는 그 데이터를 어떻게 보여줄지에 대한 **설명서**입니다.

### 뷰의 정의는 어디에 저장될까?

뷰는 데이터를 직접 가지고 있지 않는 대신, 뷰에 대한 **정의 정보**는 **시스템 카탈로그(System Catalog)** 또는 **데이터 사전(Data Dictionary)** 이라고 불리는 곳에 **메타데이터(Meta Data, 데이터에 대한 데이터)** 형태로 저장됩니다.

저장되는 내용은 다음과 같습니다:
- 이 뷰의 이름은 무엇인가?
- 어느 기본 테이블에서 데이터를 가져오는가?
- 어떤 열(속성)을 가져오는가?
- 어떤 조건(WHERE절)으로 필터링하는가?

즉, `SELECT FROM WHERE` 형태의 검색 명령어가 메타데이터로 저장되어 있다가, 누군가 그 뷰를 실행(호출)하면 그때서야 기본 테이블에 접근해 데이터를 가져와 화면에 보여줍니다. 이처럼 뷰는 **실행 시간(Run Time)에만 구체화되는 특수한 테이블**이라고 표현합니다.

## 🤔 뷰는 왜 필요한가? — 뷰의 등장 배경

### 다양한 사용자 요구를 처리하기 위해

실제 데이터베이스 시스템을 생각해 봅시다. 하나의 은행에는 수많은 업무가 있습니다. 입금, 출금, 이체, 잔액 조회 등 각각의 업무에 필요한 데이터는 조금씩 다릅니다.

만약 뷰가 없다면 업무마다 별도의 테이블을 만들어야 할까요? 그렇게 되면 동일한 데이터가 여러 테이블에 중복으로 저장되어 관리가 매우 복잡해집니다. 데이터가 바뀌면 모든 테이블을 일일이 수정해야 하는 문제도 생깁니다.

뷰를 사용하면 **기본 테이블 하나에 실제 데이터를 두고**, 각 업무(입금, 출금, 이체, 잔액 조회)에 맞는 뷰를 여러 개 만들어서, 각각의 사용자가 필요한 데이터만 볼 수 있도록 설정할 수 있습니다. 데이터는 한 곳에서 관리되고, 보여주는 방식만 다르게 하는 것입니다.

### 보안(Security)을 위해

뷰는 **보안(Security) 도구**로도 매우 유용하게 사용됩니다. 예를 들어 직원 테이블에는 이름, 사번, 부서, 연봉, 주민등록번호 등 민감한 정보들이 들어 있다고 합시다. 일반 직원에게는 연봉이나 주민등록번호 같은 민감한 정보를 보여주면 안 됩니다.

이때 뷰를 사용하면 일반 직원용 뷰에는 이름과 부서 정보만 포함시키고, 인사 담당자용 뷰에는 연봉 정보까지 포함시키는 방식으로 **권한별로 다른 데이터 접근**을 설정할 수 있습니다.

:::info 외부 스키마(External Schema)로서의 뷰
데이터베이스 이론에서 스키마(Schema)를 세 단계로 나눌 때, 사용자 관점의 스키마를 **외부 스키마(External Schema)** 라고 합니다. 뷰는 바로 이 외부 스키마에 해당합니다. 즉, 사용자마다 자신에게 필요한 데이터만 보이도록 하는 개인화된 데이터베이스 뷰를 의미합니다.
:::

## 🔍 뷰의 핵심 원리 — 논리적 뷰 vs 물리적 뷰

### 논리적 뷰 (Logical View) — 일반적인 뷰

우리가 일반적으로 "뷰"라고 부르는 것은 **논리적 뷰**입니다. 논리적 뷰는 **데이터를 직접 저장하지 않습니다**. 뷰의 정의(어떤 테이블에서, 어떤 조건으로 데이터를 가져올 것인가)만 시스템 카탈로그에 저장해 두었다가, 뷰가 호출될 때마다 기본 테이블에서 데이터를 가져옵니다.

이를 비유로 설명하면, 내 지갑에 돈이 없는 상황과 같습니다. 누군가 돈을 달라고 하면 직접 내 주머니에서 꺼내줄 수 없고, 은행(기본 테이블)에 가서 돈을 인출해 와서 줘야 합니다. 한 단계를 더 거쳐야 하니 당연히 시간이 더 걸립니다.

이 때문에 논리적 뷰는 **논리적 독립성(Logical Independence)** 을 제공하는 대신, **성능 저하(Performance Degradation)** 라는 단점을 가지고 있습니다.

| 구분 | 설명 |
|---|---|
| **데이터 저장 여부** | ❌ 데이터 없음 (정의만 저장) |
| **처리 방식** | 실행 시마다 기본 테이블에서 데이터를 가져옴 |
| **장점** | 논리적 독립성 제공, 보안 강화 |
| **단점** | 기본 테이블 접근 횟수 증가로 성능 저하 가능 |

### 실체화된 뷰 (Materialized View) — 물리적 뷰

논리적 뷰의 성능 문제를 해결하기 위해 등장한 것이 **실체화된 뷰(Materialized View)** 입니다. 영어로는 "Virtualized View"라고도 부릅니다.

실체화된 뷰는 논리적 뷰와 달리 **실제 데이터를 물리적으로 저장**합니다. 자주 검색되는 데이터를 미리 계산하고 저장해 두기 때문에, 뷰가 호출될 때 기본 테이블에 매번 접근할 필요 없이 **저장된 데이터를 바로 반환**합니다. 마치 내 지갑에 미리 돈을 준비해 두는 것과 같습니다. 돈을 달라고 하면 바로 꺼내줄 수 있으니 훨씬 빠릅니다.

다만 실체화된 뷰는 한 가지 중요한 과제를 수반합니다. **기본 테이블과 뷰의 데이터를 항상 동기화(Synchronize)** 해야 한다는 점입니다. 기본 테이블의 데이터가 바뀌면 실체화된 뷰의 데이터도 함께 업데이트되어야 하고, 반대로 실체화된 뷰의 데이터가 바뀌면 기본 테이블에도 반영이 되어야 합니다. 두 곳의 데이터가 항상 일치해야 한다는 운영 부담이 생기는 것입니다.

| 구분 | 논리적 뷰 | 실체화된 뷰 |
|---|---|---|
| **데이터 저장** | ❌ 없음 | ✅ 있음 |
| **처리 속도** | 느림 (기본 테이블 접근 필요) | 빠름 (데이터 직접 보유) |
| **데이터 동기화** | 불필요 (항상 최신 데이터) | 필요 (기본 테이블과 동기화) |
| **사용 시기** | 일반적인 경우 | 자주 조회되는 데이터 |

:::tip 실체화된 뷰는 언제 쓸까?
실체화된 뷰는 집계 쿼리(전체 합계, 평균 등)처럼 계산 비용이 크고 자주 조회되는 경우에 특히 효과적입니다. 예를 들어 "이번 달 전체 매출 합계"를 매번 수백만 건의 데이터를 합산해서 계산하는 대신, 미리 계산해 놓은 실체화된 뷰에서 바로 가져오는 방식입니다.
:::

### 뷰의 두 가지 원칙 — 아는 사람과 모르는 사람

뷰에는 중요한 두 가지 원칙이 있습니다.

1. **뷰를 만든 개발자(DBA)** 는 뷰의 정의가 어떻게 되어 있는지 정확히 알고 있어야 합니다. 어느 테이블에서 어떤 조건으로 데이터를 가져오는지 모두 알아야 뷰를 올바르게 관리할 수 있습니다.

2. **뷰를 사용하는 일반 사용자** 는 뷰의 내부 정의를 알 필요가 없고, 알아서도 안 됩니다. 사용자는 그저 뷰를 통해 자신에게 허용된 데이터만 볼 수 있을 뿐이고, 그 데이터가 어느 테이블의 어떤 컬럼에서 왔는지는 감춰져 있습니다.

이것이 바로 뷰가 **보안(Security)** 도구로 활용되는 핵심 원리입니다.

## 🏗️ 뷰의 종류 — 4가지 생성 방법

뷰는 기본 테이블에서 데이터를 가져오는 방식에 따라 크게 4가지 종류로 나눌 수 있습니다. 각각의 종류와 그 예시를 자세히 살펴보겠습니다.

### 1️⃣ 열 부분 집합 뷰 (Column Subset View) — 프로젝션 뷰

**열 부분 집합 뷰**는 기본 테이블의 전체 열(컬럼, 속성) 중에서 **필요한 열만 선택해서 만드는 뷰**입니다. 관계 대수(Relational Algebra)에서 **프로젝션(Projection)** 연산에 해당합니다.

예를 들어 학생 테이블(Student Table)에 학번(SNO), 이름(SNAME), 학년(YEAR), 학과(Department) 네 개의 열이 있을 때, 그 중에서 학번과 학과만 필요하다면 그 두 열만으로 뷰를 만드는 것입니다.

```sql
-- 열 부분 집합 뷰 예시 1: 학번과 학과만 보여주는 뷰
CREATE VIEW SVIEW1 (SNO, DEPARTMENT)
AS SELECT SNO, DEPARTMENT   -- 학번과 학과 두 열만 선택
FROM STUDENT;               -- 기본 테이블은 STUDENT

-- 열 부분 집합 뷰 예시 2: 이름과 학과만 보여주는 뷰
CREATE VIEW SVIEW2 (SNAME, DEPARTMENT)
AS SELECT SNAME, DEPARTMENT -- 이름과 학과 두 열만 선택
FROM STUDENT;               -- 동일한 기본 테이블에서
```

위 코드에서 `SVIEW1`과 `SVIEW2`는 모두 같은 `STUDENT` 테이블을 기반으로 하지만, 보여주는 열이 다릅니다. 이렇게 하나의 테이블로부터 여러 개의 뷰를 만들 수 있습니다.

### 2️⃣ 행 부분 집합 뷰 (Row Subset View) — 셀렉션 뷰

**행 부분 집합 뷰**는 기본 테이블의 모든 열을 포함하되, **특정 조건을 만족하는 행(로우, 레코드)만 골라서 만드는 뷰**입니다. 관계 대수에서 **셀렉션(Selection)** 연산에 해당합니다.

예를 들어 학생 테이블에서 4학년 학생들의 데이터만 보고 싶다면, WHERE 조건으로 학년이 4인 행들만 필터링하여 뷰를 만드는 것입니다.

```sql
-- 행 부분 집합 뷰 예시: 4학년 학생들만 보여주는 뷰
CREATE VIEW SENIOR_STUDENTS (SNO, SNAME, YEAR, DEPARTMENT)
AS SELECT SNO, SNAME, YEAR, DEPARTMENT  -- 모든 열을 다 포함
FROM STUDENT
WHERE YEAR = 4;  -- 단, 4학년 학생들의 행만 선택
```

이 뷰를 실행하면 학생 테이블에서 학년이 4인 학생들의 데이터만 화면에 출력됩니다. 열은 모두 포함하되 행을 조건으로 걸러내는 방식입니다.

### 3️⃣ 통계적 요약 뷰 (Statistical Summary View) — 집단 함수 뷰

**통계적 요약 뷰**는 **집단 함수(Aggregate Function, 집계 함수)** 를 이용해서 합계, 평균, 최댓값 등의 **요약 통계 정보를 보여주는 뷰**입니다.

예를 들어 수강(Enrollment) 테이블에서 과목별 중간고사 평균 점수를 구하는 뷰를 만들 수 있습니다. 이때 `GROUP BY`를 사용해 과목 번호(CNO)별로 묶은 뒤, 각 과목의 중간고사(Midterm) 평균을 계산합니다.

```sql
-- 통계적 요약 뷰: 과목별 중간고사 평균 점수 뷰
CREATE VIEW COSTAT (CNO, AVG_MIDTERM)
AS SELECT CNO,                  -- 과목 번호
          AVG(MIDTERM)          -- 중간고사 점수의 평균 계산
FROM ENROLL                     -- 수강 테이블에서
GROUP BY CNO;                   -- 과목 번호별로 그룹화
```

이 뷰를 실행하면 각 과목의 번호와 해당 과목의 중간고사 평균 점수가 표 형태로 출력됩니다. 실제 수강 테이블의 개별 점수 데이터를 모두 보여주는 것이 아니라, 과목별로 요약된 통계 정보만 보여주는 것이죠.

### 4️⃣ 조인 뷰 (Join View) — 여러 테이블 결합 뷰

**조인 뷰**는 **두 개 이상의 테이블을 조인(Join, 결합)해서 만든 뷰**입니다. 서로 다른 테이블에 흩어져 있는 관련 데이터를 하나의 뷰로 연결해 볼 수 있습니다.

아래 예시는 학생 테이블(STUDENT)과 수강 테이블(ENROLL)을 조인하여, 기말고사(FINAL) 점수가 95점보다 높은 우수 학생(장학생 후보)들을 보여주는 뷰입니다.

```sql
-- 조인 뷰: 기말고사 95점 초과 우수 학생 뷰
CREATE VIEW HONOR_STUDENT
AS SELECT S.SNO, S.SNAME, E.CNO, E.FINAL  -- 필요한 열들 선택
FROM STUDENT S, ENROLL E                   -- 두 개의 테이블을 대상으로
WHERE S.SNO = E.SNO                        -- 조인 조건: 학번이 같은 것끼리 연결
AND E.FINAL > 95;                          -- 추가 조건: 기말 점수 95점 초과
```

이 뷰에서 `WHERE S.SNO = E.SNO`는 조인 조건입니다. 학생 테이블의 학번(SNO)과 수강 테이블의 학번(SNO)이 같은 레코드끼리 연결합니다. 마치 두 장의 명부를 학번 기준으로 맞춰서 붙이는 것처럼요. 그 중에서 기말고사 점수가 95점을 초과하는 학생들만 뷰에 포함됩니다.

:::note 뷰 위에 뷰를 만들 수도 있다
뷰는 반드시 기본 테이블을 기반으로만 만들어지는 것이 아닙니다. **이미 만들어진 뷰를 기반으로 또 다른 뷰를 만들 수 있고**, 뷰와 뷰를 조인해서 새로운 뷰를 만들 수도 있습니다. 단, 기본 테이블이 삭제되면 그 위에 만들어진 모든 뷰도 자동으로 제거됩니다.
:::

## 🛠️ 뷰의 생성, 변경, 삭제 — SQL 명령어

### 뷰 생성 — CREATE VIEW

뷰를 만들 때는 `CREATE VIEW` 명령어를 사용합니다. 테이블을 만들 때 `CREATE TABLE`을 사용하는 것처럼, 뷰는 `CREATE VIEW`로 만듭니다.

기본 문법은 다음과 같습니다:

```sql
-- 뷰 생성 기본 문법
CREATE VIEW 뷰이름 (속성1, 속성2, ...)
AS SELECT 열1, 열2, ...
FROM 기본테이블
WHERE 조건
[WITH CHECK OPTION];  -- 선택사항: 뷰 정의 위반 시 실행 거부
```

아래는 컴퓨터공학과 학생들만 포함하는 뷰를 만드는 실제 예시입니다:

```sql
-- ✅ 올바른 뷰 생성 예시
-- 컴퓨터공학과 학생들로만 구성된 CS_STUDENT 뷰 생성
CREATE VIEW CS_STUDENT (SNO, SNAME, YEAR)
-- CS_STUDENT: 뷰의 이름
-- (SNO, SNAME, YEAR): 뷰에서 보여줄 속성들 (학번, 이름, 학년)
AS SELECT SNO, SNAME, YEAR
-- 기본 테이블에서 학번, 이름, 학년을 선택
FROM STUDENT
-- 기본 테이블은 STUDENT (학생 테이블)
WHERE DEPARTMENT = '컴퓨터'
-- 조건: 학과가 '컴퓨터'인 학생들만
WITH CHECK OPTION;
-- 이 뷰를 통해 데이터를 변경할 때,
-- 학과가 '컴퓨터'가 아닌 데이터는 거부
```

위 코드를 한 줄씩 살펴보겠습니다:

- **`CREATE VIEW CS_STUDENT`**: `CS_STUDENT`라는 이름의 뷰를 새로 만들겠다는 선언입니다.
- **`(SNO, SNAME, YEAR)`**: 이 뷰에서 보여줄 열(속성)의 이름입니다. 학번, 이름, 학년 세 가지입니다.
- **`AS SELECT SNO, SNAME, YEAR FROM STUDENT`**: 기본 테이블인 `STUDENT`에서 학번, 이름, 학년 데이터를 가져옵니다.
- **`WHERE DEPARTMENT = '컴퓨터'`**: 전체 학생 중 학과가 '컴퓨터'인 학생들만 가져옵니다.
- **`WITH CHECK OPTION`**: 이 뷰를 통해 데이터를 삽입하거나 수정할 때, 학과가 '컴퓨터'가 아닌 데이터는 실행을 거부합니다.

### WITH CHECK OPTION 이란?

**`WITH CHECK OPTION`** 은 뷰의 정의(조건)를 위반하는 데이터 변경 시도를 **자동으로 거부**하는 옵션입니다.

예를 들어 위의 `CS_STUDENT` 뷰는 학과가 '컴퓨터'인 학생들만을 위한 뷰입니다. 누군가 이 뷰를 통해 학생의 학과를 '컴퓨터'에서 '경영학'으로 변경하려 한다면?

```sql
-- ❌ WITH CHECK OPTION으로 인해 거부되는 예시
-- CS_STUDENT 뷰는 '컴퓨터' 학과만 허용하는데
-- 다른 학과로 변경하려 시도
UPDATE CS_STUDENT
SET DEPARTMENT = '경영학'  -- 컴퓨터 → 경영학으로 변경 시도
WHERE SNO = '100';
-- → WITH CHECK OPTION에 의해 실행 거부!
-- 이 뷰는 '컴퓨터' 학과 데이터만 다룰 수 있습니다.
```

`WITH CHECK OPTION`이 설정되어 있으면 위 UPDATE 문은 실행 자체가 거부됩니다. 뷰를 만들 때 설정한 조건(`DEPARTMENT = '컴퓨터'`)에 위배되기 때문입니다.

:::warning WITH CHECK OPTION을 반드시 이해하자
WITH CHECK OPTION을 설정하지 않으면 뷰의 정의 조건과 맞지 않는 데이터도 수정이 가능합니다. 이 경우 뷰에서는 그 데이터가 보이지 않게 됩니다(뷰 조건에 맞지 않으니까요). 데이터는 변경됐지만 뷰에서는 확인이 안 되는 혼란스러운 상황이 발생할 수 있습니다. 뷰의 무결성을 지키려면 WITH CHECK OPTION 사용을 권장합니다.
:::

### 뷰 변경 — ALTER VIEW는 없다!

테이블을 변경할 때 `ALTER TABLE`을 사용하는 것과 달리, **뷰는 `ALTER VIEW` 명령어를 사용하지 않습니다**. 왜냐하면 뷰는 자체적으로 데이터를 가지고 있지 않고 논리적인 정의만 있기 때문에, 굳이 수정할 필요 없이 삭제 후 재생성하면 되기 때문입니다.

```sql
-- ❌ 잘못된 방법: ALTER VIEW는 사용하지 않음
-- ALTER VIEW CS_STUDENT ... (사용 금지)

-- ✅ 올바른 방법: 기존 뷰를 삭제하고 새로 만들기
DROP VIEW CS_STUDENT;            -- 기존 뷰 삭제
CREATE VIEW CS_STUDENT (...)     -- 수정된 내용으로 새로 생성
AS SELECT ...;
```

### 뷰 삭제 — DROP VIEW

뷰를 삭제할 때는 `DROP VIEW` 명령어를 사용합니다. 이때 중요한 두 가지 옵션이 있습니다.

| 옵션 | 동작 | 기본값 |
|---|---|---|
| **RESTRICT(제한)** | 다른 뷰가 이 뷰를 참조하고 있으면 삭제 거부 | ✅ 기본값 |
| **CASCADE(연쇄)** | 이 뷰를 참조하는 다른 뷰도 함께 삭제 | - |

```sql
-- ✅ RESTRICT 옵션으로 삭제 (기본값)
-- DEPARTMENT_SIZE 뷰를 참조하는 다른 뷰가 있으면 삭제 거부
DROP VIEW DEPARTMENT_SIZE RESTRICT;

-- ✅ CASCADE 옵션으로 삭제
-- ANA 뷰와 그것을 참조하는 SS1 뷰도 함께 삭제
DROP VIEW ANA CASCADE;
-- → ANA 뷰가 삭제됨
-- → ANA를 참조하던 SS1 뷰도 연쇄적으로 함께 삭제됨
```

이 개념은 외래키(Foreign Key)에서 참조 무결성을 다룰 때 사용하는 RESTRICT/CASCADE 옵션과 동일한 원리입니다. 참조되고 있으면 삭제를 막는 것이 RESTRICT, 함께 삭제하는 것이 CASCADE입니다.

:::danger 기본 테이블이 삭제되면?
기본 테이블이 삭제되면 그 테이블을 기반으로 만들어진 **모든 뷰가 자동으로 제거**됩니다. 이를 전파 소멸(Propagate Destroy)이라고 합니다. 기본 테이블은 뷰가 데이터를 가져오는 원천이므로, 원천이 사라지면 뷰도 존재 의미가 없어지기 때문입니다.
:::

## ⚠️ 뷰의 데이터 조작 제약 — 변경이 안 되는 경우

뷰를 통해서도 이론적으로는 데이터를 삽입(INSERT), 삭제(DELETE), 갱신(UPDATE)할 수 있습니다. 그러나 실제로는 **매우 많은 제약**이 존재합니다. 뷰 자체는 데이터를 가지고 있지 않은 가상의 테이블이기 때문에 데이터 변경 작업에 많은 제한이 따릅니다.

아래의 경우에는 뷰를 통한 데이터 변경이 **거부**됩니다:

### 변경이 거부되는 대표적인 경우들

**① 산술식(수식)이나 집계 함수가 포함된 경우**

```sql
-- ❌ 변경 불가 예시: 뷰 생성 시 계산식 포함
CREATE VIEW SCORE_VIEW (SNO, TOTAL)
AS SELECT SNO, MIDTERM + FINAL  -- 더하기 연산이 포함됨
FROM ENROLL;
-- → 이 뷰를 통해 TOTAL을 수정하려 해도 거부됨
-- 이유: TOTAL은 실제 컬럼이 아닌 계산된 값이므로
-- 어떤 기본 컬럼 값을 수정해야 할지 데이터베이스가 판단 불가
```

**② GROUP BY나 DISTINCT(중복 제거)가 포함된 경우**

```sql
-- ❌ 변경 불가 예시: GROUP BY 사용
CREATE VIEW DEPT_AVG (DEPARTMENT, AVG_YEAR)
AS SELECT DEPARTMENT, AVG(YEAR)  -- 집계 함수 사용
FROM STUDENT
GROUP BY DEPARTMENT;             -- 그룹으로 묶임
-- → 이 뷰를 통한 데이터 변경 거부
-- 이유: 그룹화된 결과는 여러 행을 합친 것이므로
-- 특정 원본 행을 특정할 수 없음
```

**③ 조인(JOIN)으로 만들어진 뷰인 경우**

두 개 이상의 테이블을 조인해서 만든 뷰는 데이터 변경 시 어느 테이블의 데이터를 수정해야 할지 모호해지는 경우가 많기 때문에 변경이 제한됩니다.

**④ NOT NULL 속성이 뷰에 포함되지 않은 경우**

이 경우는 특히 중요합니다. 예를 들어 기본 테이블에 A, B, C 세 개의 열이 있고, 세 열 모두 NOT NULL(반드시 값이 있어야 함) 조건이 설정되어 있다고 합시다. 그런데 A와 B만으로 뷰를 만들었다면?

```sql
-- 기본 테이블 구조 (A, B, C 모두 NOT NULL)
-- CREATE TABLE BASE_TABLE (
--     A VARCHAR(10) NOT NULL,  -- 반드시 입력 필요
--     B INT NOT NULL,          -- 반드시 입력 필요
--     C DATE NOT NULL          -- 반드시 입력 필요
-- );

-- ❌ 문제가 되는 뷰: C 열이 빠져 있음
CREATE VIEW AB_VIEW (A, B)
AS SELECT A, B  -- C는 포함하지 않음
FROM BASE_TABLE;

-- ❌ 이 뷰를 통해 데이터 삽입 시도
INSERT INTO AB_VIEW VALUES ('홍길동', 30);
-- → 거부됨!
-- 이유: A='홍길동', B=30을 넣을 수는 있지만
-- NOT NULL인 C에는 값을 넣을 방법이 없어서
-- C가 NULL이 되어버려 무결성(Integrity) 위반 발생
```

이처럼 NOT NULL 제약이 있는 열이 뷰에 포함되지 않으면, 그 뷰를 통해 데이터를 삽입할 때 해당 열에 NULL 값이 들어가게 되어 **무결성(Integrity, 데이터의 정확성과 일관성)** 이 깨집니다. 이런 상황을 방지하기 위해 데이터베이스는 해당 삽입 작업을 거부합니다.

**⑤ WITH CHECK OPTION의 정의를 위반하는 경우**

앞서 설명한 것처럼, WITH CHECK OPTION이 설정된 뷰에 뷰의 정의 조건에 맞지 않는 데이터를 삽입하거나 수정하려 할 때도 거부됩니다.

## 💡 뷰의 장점과 단점 — 정리

### 뷰의 장점 ✅

**① 논리적 독립성(Logical Independence) 제공**

기본 테이블의 구조가 변경되더라도 뷰에는 영향이 없습니다. 예를 들어 기본 테이블에 새로운 열 D를 추가하더라도, 기존에 A, B, C로 만들어둔 뷰는 수정 없이 그대로 사용할 수 있습니다. 또한 기본 테이블의 물리적 위치가 바뀌어도(재구성, Reorganization) 뷰는 영향을 받지 않습니다.

> 🔑 **논리적 독립성**: 기본 테이블이 확장(속성 추가)되거나 재구성(물리적 위치 변경)되어도 뷰는 전혀 영향을 받지 않는 특성

**② 보안(Security) 제공**

뷰는 기본 테이블의 일부분만 보여주기 때문에, 사용자 권한에 따라 접근할 수 있는 데이터를 세밀하게 제어할 수 있습니다. 민감한 데이터(연봉, 주민등록번호 등)가 포함된 열은 일반 사용자용 뷰에서 제외함으로써 데이터를 보호합니다.

**③ 데이터 관리 간편화**

필요한 데이터만 보여주는 뷰를 사용하면, 복잡한 조인이나 조건문을 매번 작성하지 않아도 됩니다. 자주 사용하는 복잡한 쿼리를 뷰로 저장해 두면, 간단하게 뷰 이름만 호출해서 사용할 수 있습니다.

**④ 사용자 다양성 지원**

각기 다른 업무를 담당하는 사용자들에게 각자의 역할에 맞는 뷰를 제공함으로써, 동일한 기본 테이블에서 다양한 응용을 처리할 수 있습니다.

### 뷰의 단점 ❌

**① ALTER VIEW 불가능**

뷰의 내용을 수정하려면 삭제 후 재생성해야 합니다. `ALTER VIEW`로 수정하는 것은 지원되지 않습니다.

**② 삽입·삭제·갱신 제한 많음**

앞서 설명한 것처럼 뷰를 통한 데이터 변경에는 많은 제약이 있습니다.

**③ 인덱스(Index, 색인) 사용 불가**

인덱스는 실제 데이터가 저장된 테이블에만 적용할 수 있습니다. 뷰는 논리적인 가상 테이블이기 때문에 인덱스를 생성하거나 사용할 수 없습니다.

**④ 성능 저하 가능성**

뷰를 많이 사용하면 매번 기본 테이블에 접근해서 데이터를 가져와야 하므로 성능이 저하될 수 있습니다. 이 문제는 실체화된 뷰(Materialized View)를 사용해 해결할 수 있습니다.

| 구분 | 내용 |
|---|---|
| ✅ 논리적 독립성 | 기본 테이블 변경에도 뷰는 영향 없음 |
| ✅ 보안 제공 | 일부 데이터만 노출하여 민감 정보 보호 |
| ✅ 관리 편의성 | 복잡한 쿼리를 뷰로 저장해 간단하게 호출 |
| ✅ 다양한 활용 | 사용자별 맞춤 데이터 제공 가능 |
| ❌ ALTER 불가 | 수정 시 삭제 후 재생성 필요 |
| ❌ 변경 제한 | 삽입·삭제·갱신에 많은 제약 |
| ❌ 인덱스 불가 | 성능 최적화에 인덱스 활용 불가 |
| ❌ 성능 저하 | 매번 기본 테이블 접근 필요 |

## 🚨 주의사항 및 실전 팁

### 뷰를 생성할 때 주의할 점

:::warning ORDER BY와 UNION은 뷰에서 사용 불가
일반 SELECT 문에서는 ORDER BY(정렬)와 UNION(합집합) 등을 자유롭게 사용할 수 있습니다. 그러나 **뷰를 생성하는 SELECT 문에서는 ORDER BY와 UNION을 사용할 수 없습니다**. 뷰 자체는 정렬된 결과를 저장하는 개념이 아니며, UNION은 뷰의 단순한 논리적 구조와 맞지 않기 때문입니다. 뷰를 조회할 때 ORDER BY를 적용하는 것은 가능합니다.
:::

### NOT NULL 열은 반드시 뷰에 포함시키기

삽입 연산이 필요한 뷰를 만들 때는, 기본 테이블에서 NOT NULL 조건이 있는 모든 열을 반드시 뷰에 포함시켜야 합니다. 그렇지 않으면 뷰를 통한 INSERT 시 무결성 오류가 발생합니다.

```sql
-- 기본 테이블
-- PRODUCT (P_ID NOT NULL, P_NAME NOT NULL, PRICE NOT NULL, STOCK INT)

-- ❌ 잘못된 뷰 (삽입 불가 - NOT NULL인 PRICE가 빠짐)
CREATE VIEW PRODUCT_INFO (P_ID, P_NAME)
AS SELECT P_ID, P_NAME
FROM PRODUCT;
-- 이 뷰로 INSERT 시도하면 PRICE가 NULL이 되어 오류 발생

-- ✅ 올바른 뷰 (삽입 가능 - NOT NULL 열 모두 포함)
CREATE VIEW PRODUCT_FULL (P_ID, P_NAME, PRICE)
AS SELECT P_ID, P_NAME, PRICE  -- NOT NULL 열 모두 포함
FROM PRODUCT;
```

### 보안 목적으로 뷰를 적극 활용하기

민감한 정보가 포함된 테이블에 직접 접근 권한을 부여하는 것보다, 필요한 정보만 담은 뷰를 만들고 그 뷰에만 접근 권한을 부여하는 것이 훨씬 안전합니다.

```sql
-- ✅ 보안을 위한 뷰 활용 예시
-- 직원 테이블에는 연봉, 주민번호 등 민감 정보 포함
-- 일반 직원용 뷰: 이름과 부서만 보여줌
CREATE VIEW EMP_PUBLIC (EMP_NAME, DEPARTMENT)
AS SELECT EMP_NAME, DEPARTMENT
FROM EMPLOYEE;  -- 연봉, 주민번호 등은 제외

-- 인사 담당자용 뷰: 연봉 정보까지 포함
CREATE VIEW EMP_HR (EMP_NAME, DEPARTMENT, SALARY)
AS SELECT EMP_NAME, DEPARTMENT, SALARY
FROM EMPLOYEE;  -- 주민번호 등 최고 민감 정보는 여전히 제외
```

:::tip 뷰 사용의 황금률
- 자주 쓰는 복잡한 JOIN 쿼리가 있다면 → **뷰로 만들어 두기**
- 테이블의 일부 데이터만 특정 사용자에게 보여줘야 한다면 → **뷰 + WITH CHECK OPTION**
- 성능이 중요하고 자주 조회되는 집계 데이터라면 → **실체화된 뷰 고려**
- 뷰를 수정해야 한다면 → **DROP VIEW 후 CREATE VIEW로 재생성**
:::

## 📝 핵심 정리

이 문서에서 배운 핵심 개념들을 정리합니다.

- **뷰(View)** 는 기본 테이블로부터 논리적으로 유도된 가상의 테이블이며, 데이터를 직접 저장하지 않는다
- **논리적 뷰** 는 데이터를 저장하지 않고 정의만 가지며, 실행 시마다 기본 테이블에서 데이터를 가져온다
- **실체화된 뷰(Materialized View)** 는 데이터를 물리적으로 저장해 성능을 높이지만, 기본 테이블과 동기화가 필요하다
- 뷰의 정의는 **시스템 카탈로그(데이터 사전)** 에 메타데이터 형태로 저장된다
- 뷰는 **외부 스키마(External Schema)** 에 해당하며, 사용자별 데이터 접근 제어에 활용된다
- 뷰는 **열 부분 집합 뷰, 행 부분 집합 뷰, 통계적 요약 뷰, 조인 뷰** 네 가지 종류로 나뉜다
- 뷰 생성은 **CREATE VIEW**, 삭제는 **DROP VIEW** 명령어를 사용하며, ALTER VIEW는 존재하지 않는다
- **WITH CHECK OPTION** 은 뷰의 정의를 위반하는 데이터 변경 시도를 자동으로 거부한다
- 뷰 삭제 시 **RESTRICT** 는 참조 중이면 삭제 거부, **CASCADE** 는 연쇄 삭제를 수행한다
- 뷰를 통한 데이터 변경은 산술식, 집계 함수, GROUP BY, 조인 뷰 등 다양한 경우에 제한된다
- NOT NULL 속성이 있는 열이 뷰에서 빠져 있으면 해당 뷰를 통한 삽입 작업은 거부된다
- 뷰의 주요 장점은 **논리적 독립성, 보안 제공, 데이터 관리 편의성**이다
- 뷰의 주요 단점은 **성능 저하, 인덱스 사용 불가, 데이터 변경 제한**이다
- 기본 테이블이 삭제되면 해당 테이블을 기반으로 한 모든 뷰도 **자동으로 제거(전파 소멸)** 된다
