# 🔐 뷰(View)의 보안 기능과 제약 조건

데이터베이스를 공부하다 보면 반드시 마주치게 되는 개념이 바로 **뷰(View)** 입니다. 뷰는 단순히 "테이블을 예쁘게 보여주는 도구"가 아닙니다. 올바르게 이해하고 사용하면 **보안(Security)**, **논리적 독립성(Logical Independence)**, **데이터 관리 편의성** 등 다양한 이점을 얻을 수 있는 강력한 SQL 고급 기능입니다.

이 문서에서는 뷰가 무엇인지부터 시작해서, 왜 보안 도구로 활용되는지, 어떻게 만들고 삭제하는지, 그리고 뷰를 통해 데이터를 조작할 때 어떤 제약이 존재하는지를 초보자도 이해할 수 있도록 아주 자세하게 설명합니다.

---

## 📌 뷰(View)란 무엇인가?

### 가상의 테이블, 뷰

뷰(View)는 **기존에 존재하는 테이블로부터 유도되어 논리적으로 만들어지는 가상의 테이블(Virtual Table)** 입니다. 여기서 핵심 키워드는 **"가상"** 입니다.

일반 테이블은 **물리적으로 구현**되어 있어서 실제 데이터가 디스크(저장 장치) 안에 저장되어 있습니다. 마치 실제 창고에 물건이 가득 쌓여 있는 것처럼요. 반면 뷰는 **데이터 자체를 가지고 있지 않습니다.** 뷰는 "어느 창고의 어떤 선반에서, 어떤 조건의 물건을 꺼내오겠다"는 **청사진(설계도)** 만 가지고 있는 것입니다.

누군가가 뷰에 접근하면 그때서야 뷰에 정의된 대로 기본 테이블(Base Table)에서 데이터를 가져와서 화면에 보여줍니다. 평소에는 조용히 있다가, 누군가 호출하면 그때 그때 기본 테이블에서 데이터를 가져오는 방식입니다.

> **핵심 개념:** 뷰는 데이터를 직접 저장하지 않는 가상의 테이블이며, 실행 시간에만 구체화(Materialization)되는 특수한 테이블입니다.

### 논리적 뷰 vs 물리적 뷰(실체화된 뷰)

뷰는 크게 두 가지 종류로 나눌 수 있습니다.

| 구분 | 논리적 뷰 (Logical View) | 물리적 뷰 / 실체화된 뷰 (Materialized View) |
|---|---|---|
| 데이터 저장 여부 | ❌ 저장하지 않음 | ✅ 실제 데이터 저장 |
| 데이터 접근 방식 | 기본 테이블에서 매번 가져옴 | 뷰 자체에 저장된 데이터 사용 |
| 성능 | 상대적으로 느림 | 빠름 |
| 데이터 동기화 필요 | 없음 (항상 최신 데이터) | 필요 (기본 테이블과 동기화) |
| 일반적인 사용 | 보안, 논리적 독립성 목적 | 성능 향상 목적 |

**논리적 뷰**는 우리가 일반적으로 "뷰"라고 부르는 것입니다. 데이터가 없으니, 기본 테이블에 매번 접근해야 합니다. 마치 내 지갑에 돈이 없어서 은행에 가서 돈을 꺼내와야 하는 것처럼, 한 단계를 더 거쳐야 하므로 **성능이 떨어지는 단점**이 있습니다.

**실체화된 뷰(Materialized View)** 는 이 성능 문제를 해결하기 위해 등장했습니다. 자주 검색되는 데이터는 아예 뷰에도 물리적으로 저장해 두자는 아이디어입니다. 마치 자주 쓰는 재료를 냉장고 앞칸에 꺼내 두는 것처럼, 필요할 때 바로 꺼낼 수 있어 훨씬 빠릅니다. 단, 기본 테이블의 데이터가 바뀌면 실체화된 뷰의 데이터도 함께 업데이트(동기화, Synchronization)해야 한다는 관리 부담이 생깁니다.

:::info 외부 스키마(External Schema)와 뷰
데이터베이스 3단계 스키마 구조에서 **외부 스키마(External Schema)** 가 바로 뷰를 의미합니다. 사용자마다 다른 뷰를 제공함으로써 각 사용자에게 필요한 데이터만 보여주는 개념입니다. 뷰는 사용자의 관점에서 데이터베이스를 바라보는 창문(Window) 역할을 합니다.
:::

### 뷰의 정의는 어디에 저장될까?

뷰 자체는 데이터를 저장하지 않지만, **뷰의 정의(Definition)** 는 어딘가에 저장되어야 합니다. 뷰가 호출될 때 "어떤 테이블에서, 어떤 조건으로, 어떤 컬럼을 가져올지"를 알아야 하기 때문입니다.

뷰의 정의는 **시스템 카탈로그(System Catalog)** 또는 **데이터 사전(Data Dictionary)** 이라고 불리는 곳에 **메타데이터(Metadata, 데이터에 대한 데이터)** 형태로 저장됩니다. 이 정보는 `SELECT ... FROM ... WHERE ...` 형태로 기록되어 있어, 뷰가 실행될 때마다 이 정의를 읽어서 기본 테이블에서 데이터를 가져오게 됩니다.

## 🔒 뷰를 왜 보안 도구로 사용하는가?

### 데이터 접근 제한의 필요성

회사의 인사 데이터베이스를 생각해봅시다. 이 테이블에는 직원의 이름, 부서, 연봉, 주민등록번호, 계좌번호 등이 모두 저장되어 있습니다. 그런데 영업팀 직원이 동료의 이름과 부서 정보는 알아야 하지만, 연봉이나 계좌번호는 절대 알아서는 안 됩니다.

이럴 때 뷰가 강력한 보안 도구가 됩니다. **이름과 부서만 보여주는 뷰를 만들어서** 영업팀 직원에게는 그 뷰에만 접근 권한을 주면 됩니다. 원본 테이블에는 아예 접근할 수 없도록 막아두는 것이죠.

### 뷰의 보안 원칙: 알아야 할 사람과 몰라야 할 사람

뷰의 보안에는 두 가지 중요한 원칙이 있습니다.

**첫 번째 원칙:** 뷰를 만든 개발자(DBA, 데이터베이스 관리자)는 뷰의 정의를 정확하게 알고 있어야 합니다. 어떤 테이블에서 어떤 데이터를 가져와서 어떻게 구성했는지를 완벽하게 이해하고 관리해야 합니다.

**두 번째 원칙:** 뷰를 사용하는 일반 사용자는 그 뷰의 정의를 알아서는 **안 됩니다.** 사용자는 자신이 보고 있는 데이터가 어느 테이블의 어떤 컬럼에서 온 건지, 원본 테이블에 어떤 민감한 정보가 더 있는지 알 수 없습니다. 이것이 바로 뷰가 제공하는 **데이터 은닉(Data Hiding)** 기능입니다.

> **비유:** 은행 창구 직원은 고객에게 "잔액이 얼마입니다"라고 알려줍니다. 하지만 고객은 은행 내부의 전산 시스템이 어떤 구조로 되어 있는지, 어느 서버에 어떤 테이블로 저장되어 있는지 알 수 없습니다. 뷰가 바로 이 은행 창구 역할을 하는 것입니다.

:::tip 보안을 위한 뷰 활용 전략
- 민감한 컬럼(연봉, 주민번호, 계좌번호 등)을 제외한 뷰를 생성하여 일반 사용자에게 제공
- 특정 부서나 조건의 데이터만 보이는 뷰를 만들어 부서별 데이터 접근 제한
- 원본 테이블에는 DBA만 접근 가능하도록 권한 설정
:::

## 🏗️ 뷰 생성하기: CREATE VIEW

### 기본 문법

뷰를 만들 때는 `CREATE VIEW` 명령어를 사용합니다. 테이블을 만들 때 `CREATE TABLE`을 쓰는 것처럼요.

```sql
-- 뷰 생성 기본 문법
CREATE VIEW 뷰이름 (속성1, 속성2, ...)
AS
SELECT 컬럼1, 컬럼2, ...
FROM 기본테이블
WHERE 조건;
```

:::warning 뷰에서 사용할 수 없는 기능
뷰의 `AS SELECT` 문은 일반 SELECT문과 거의 같지만, 다음 기능은 사용할 수 없습니다:
- `ORDER BY` (정렬): 뷰 내부에서 데이터 정렬 불가
- `UNION` (합집합): 뷰 내부에서 집합 연산 불가
:::

### 뷰의 4가지 생성 유형

뷰는 만드는 방법에 따라 크게 4가지 유형으로 분류됩니다.

#### 1️⃣ 열 부분 집합 뷰 (Column Subset View)

**왜 필요한가?** 전체 테이블에서 일부 컬럼(열)만 보여줘야 할 때 사용합니다. 예를 들어 직원 테이블에서 이름과 부서만 공개해야 할 때 유용합니다. 관계 대수(Relational Algebra)의 **프로젝션(Projection, 열 선택)** 연산에 해당합니다.

```sql
-- ✅ 올바른 예시: 학생 테이블에서 학번과 학과 정보만 가진 뷰 생성
CREATE VIEW SVIEW1 (SNO, DEPARTMENT)
AS
SELECT SNO, DEPARTMENT   -- 전체 컬럼 중 학번, 학과만 선택
FROM STUDENT;            -- 기본 테이블은 STUDENT

-- ✅ 또 다른 열 부분 집합 뷰: 이름과 학과만 포함
CREATE VIEW SVIEW2 (SNAME, DEPARTMENT)
AS
SELECT SNAME, DEPARTMENT  -- 이름, 학과만 선택
FROM STUDENT;             -- 민감한 정보(학년 등)는 제외
```

이 뷰를 사용하면 사용자는 학번과 학과 정보만 볼 수 있고, 학생의 다른 정보(예: 성적, 개인정보 등)는 접근할 수 없습니다.

#### 2️⃣ 행 부분 집합 뷰 (Row Subset View)

**왜 필요한가?** 특정 조건을 만족하는 행(Row)만 보여줘야 할 때 사용합니다. 예를 들어 4학년 학생 데이터만 담당하는 담당자에게는 4학년 행만 보이는 뷰를 제공하면 됩니다. 관계 대수의 **셀렉션(Selection, 행 선택)** 연산에 해당합니다.

```sql
-- ✅ 올바른 예시: 4학년 학생 데이터만 보여주는 뷰 생성
CREATE VIEW SENIOR_STUDENT (SNO, SNAME, YEAR, DEPARTMENT)
AS
SELECT SNO, SNAME, YEAR, DEPARTMENT  -- 모든 컬럼을 가져오되
FROM STUDENT
WHERE YEAR = 4;                       -- 4학년인 행만 필터링
```

이 뷰를 조회하면 전체 학생이 아닌, 4학년 학생 데이터만 결과로 나타납니다.

#### 3️⃣ 통계적 요약 뷰 (Statistical Summary View)

**왜 필요한가?** 집계 함수(COUNT, AVG, SUM 등)를 이용해서 요약 통계 정보를 뷰로 만들 때 사용합니다. 예를 들어 과목별 중간고사 평균 점수를 항상 조회해야 하는 상황에서 유용합니다.

```sql
-- ✅ 올바른 예시: 과목별 중간고사 평균 점수를 보여주는 뷰 생성
CREATE VIEW COSTAT (CNO, AVG_MIDTERM)
AS
SELECT CNO, AVG(MIDTERM)   -- 과목번호와 중간고사 평균을 계산
FROM ENROLL                 -- 수강 테이블에서
GROUP BY CNO;               -- 과목번호(CNO)별로 그룹화하여 평균 계산
```

`CNO`가 과목 번호라면, 이 뷰를 조회하면 과목별로 중간고사 평균이 계산되어 보입니다. 복잡한 집계 쿼리를 매번 작성하지 않고 뷰 이름만 호출하면 됩니다.

#### 4️⃣ 조인 뷰 (Join View)

**왜 필요한가?** 두 개 이상의 테이블을 조인(JOIN, 합치기)하여 뷰를 만들 때 사용합니다. 예를 들어 학생 정보와 수강 정보를 합쳐서, 기말고사 성적이 95점 이상인 우수 학생 목록을 뷰로 만들 수 있습니다.

```sql
-- ✅ 올바른 예시: 기말고사 95점 이상 우수 학생 뷰 (조인 뷰)
CREATE VIEW HONOR_STUDENT (SNO, SNAME, CNO, FINAL)
AS
SELECT S.SNO, S.SNAME, E.CNO, E.FINAL  -- 두 테이블의 컬럼을 함께 선택
FROM STUDENT S, ENROLL E               -- 학생 테이블과 수강 테이블
WHERE S.SNO = E.SNO                    -- 조인 조건: 학번이 같은 것끼리 연결
  AND E.FINAL > 95;                    -- 기말고사 95점 초과인 학생만 필터링
```

이 뷰를 조회하면 두 테이블이 자동으로 조인된 결과에서 우수 학생 목록만 나타납니다. 사용자는 조인이라는 복잡한 개념을 몰라도 뷰 이름만 알면 결과를 얻을 수 있습니다.

:::note 뷰의 계층 구조
뷰는 테이블로부터만 만들 수 있는 것이 아닙니다. **뷰로부터 또 다른 뷰를 만들 수도 있고, 뷰와 뷰를 조인해서 새로운 뷰를 만들 수도 있습니다.** 하지만 모든 뷰의 출발점은 반드시 기본 테이블이어야 합니다. 기본 테이블이 사라지면 그 위에 만들어진 모든 뷰는 자동으로 제거됩니다.
:::

## 🛡️ WITH CHECK OPTION: 뷰의 무결성 보호 장치

### WITH CHECK OPTION이 왜 필요한가?

뷰를 통해 데이터를 삽입하거나 수정할 수 있는 경우, 한 가지 위험한 상황이 발생할 수 있습니다. 예를 들어 **컴퓨터공학과 학생만 보여주는 뷰**를 만들었는데, 그 뷰를 통해서 경영학과 학생 데이터를 삽입하거나 수정하는 일이 생길 수 있습니다.

이렇게 되면 **뷰의 생성 조건(컴퓨터공학과 학생만)** 에 위배되는 데이터가 기본 테이블에 들어가게 됩니다. 물론 그 데이터는 뷰에는 보이지 않겠지만(조건에 맞지 않으니까), 기본 테이블이 뷰의 의도와 다른 방향으로 오염될 수 있습니다. 이 문제를 방지하기 위해 `WITH CHECK OPTION`을 사용합니다.

### WITH CHECK OPTION 사용 예시

```sql
-- ✅ WITH CHECK OPTION을 포함한 뷰 생성 예시
CREATE VIEW CS_STUDENT (SNO, SNAME, YEAR)
AS
SELECT SNO, SNAME, YEAR          -- 학번, 이름, 학년 컬럼 선택
FROM STUDENT
WHERE DEPARTMENT = '컴퓨터'      -- 컴퓨터공학과 학생만 필터링
WITH CHECK OPTION;               -- 뷰의 정의를 위반하는 변경 거부
```

위의 `CS_STUDENT` 뷰에는 컴퓨터공학과 학생만 포함됩니다.

```sql
-- ❌ 잘못된 예시: 뷰를 통해 다른 학과로 변경 시도 (WITH CHECK OPTION에 의해 거부됨)
UPDATE CS_STUDENT
SET DEPARTMENT = '경영'          -- 컴퓨터 → 경영으로 변경 시도
WHERE SNO = '100';               -- 이 명령은 실행 거부됨!

-- ✅ 올바른 예시: 뷰의 조건에 맞는 변경 (허용됨)
UPDATE CS_STUDENT
SET YEAR = 3                     -- 학년 변경은 뷰 정의를 위반하지 않으므로 허용
WHERE SNO = '100';
```

`WITH CHECK OPTION`을 사용하면, 뷰를 통한 모든 데이터 변경 작업이 **뷰의 WHERE 조건을 만족하는지** 자동으로 검사합니다. 조건을 위반하는 변경은 즉시 거부(Reject)됩니다. 마치 컴퓨터공학과 동아리방의 문지기가 "컴공 학생만 입장 가능합니다"라고 지키고 서 있는 것과 같습니다.

:::warning WITH CHECK OPTION 없이 뷰를 통한 데이터 변경
`WITH CHECK OPTION`을 지정하지 않으면, 뷰의 조건에 맞지 않는 데이터도 뷰를 통해 삽입하거나 수정할 수 있습니다. 이 경우 해당 데이터는 뷰에서는 보이지 않지만 기본 테이블에는 저장됩니다. 보안이나 데이터 무결성이 중요한 뷰라면 반드시 `WITH CHECK OPTION`을 사용하세요.
:::

## 🗑️ 뷰 삭제하기: DROP VIEW와 삭제 옵션

### 뷰는 ALTER로 수정할 수 없다

뷰는 **`ALTER VIEW` 명령어로 수정하는 것이 불가능합니다.** 이유는 간단합니다. 뷰는 실제 데이터를 가지고 있지 않고 논리적인 정의만 저장되어 있기 때문에, 굳이 수정할 필요 없이 삭제하고 다시 만들면 됩니다. 집에 있는 청사진(설계도)이 마음에 들지 않으면 찢고 새로 그리면 되는 것처럼요.

```sql
-- ❌ 잘못된 예시: 뷰는 ALTER로 변경 불가
ALTER VIEW CS_STUDENT ...;  -- 이 문법은 사용하지 않음

-- ✅ 올바른 방법: 기존 뷰 삭제 후 새로운 뷰 생성
DROP VIEW CS_STUDENT;       -- 기존 뷰 삭제
CREATE VIEW CS_STUDENT (...) AS SELECT ...;  -- 새로운 정의로 다시 생성
```

### DROP VIEW의 두 가지 옵션: RESTRICT vs CASCADE

뷰를 삭제할 때 주의해야 할 점이 있습니다. 한 뷰를 기반으로 다른 뷰가 만들어져 있을 수 있기 때문입니다. 예를 들어 `VIEW1`을 기반으로 `VIEW2`가 만들어진 상황에서 `VIEW1`을 삭제하려고 하면 `VIEW2`는 어떻게 될까요?

이 상황을 처리하는 방식이 두 가지 있습니다.

| 옵션 | 동작 방식 | 설명 |
|---|---|---|
| `RESTRICT` (기본값) | 삭제 거부 | 다른 뷰나 객체가 참조 중이면 삭제 자체를 막음 |
| `CASCADE` | 연쇄 삭제 | 참조하는 모든 뷰도 함께 삭제됨 |

```sql
-- ✅ RESTRICT 옵션: 다른 곳에서 참조 중이면 삭제 거부 (기본값)
DROP VIEW DEPARTMENT_SIZE RESTRICT;
-- DEPARTMENT_SIZE를 참조하는 다른 뷰가 있으면 삭제 거부됨

-- ✅ CASCADE 옵션: 참조하는 모든 뷰까지 연쇄 삭제
DROP VIEW ANA CASCADE;
-- ANA를 참조하는 SS1 같은 뷰도 함께 자동으로 삭제됨
```

**RESTRICT**는 "안전 제일" 모드입니다. 다른 곳에서 이 뷰를 참조하고 있으면 "삭제할 수 없습니다"라는 경고를 보내고 작업을 멈춥니다. 옵션을 명시하지 않으면 자동으로 RESTRICT가 적용됩니다.

**CASCADE**는 "도미노 삭제" 모드입니다. 삭제하려는 뷰를 기반으로 만들어진 모든 뷰가 연쇄적으로 함께 삭제됩니다. 편리하지만 의도치 않게 많은 뷰가 사라질 수 있으니 주의해야 합니다.

:::danger CASCADE 사용 시 주의사항
`DROP VIEW ... CASCADE`를 실행하면 해당 뷰를 직접, 간접적으로 참조하는 **모든 뷰가 연쇄 삭제**됩니다. 운영 중인 시스템에서 실수로 실행하면 수많은 뷰가 사라져 큰 장애가 발생할 수 있습니다. 반드시 삭제 대상과 참조 관계를 먼저 확인한 후 실행하세요.
:::

### 기본 테이블이 삭제되면?

기본 테이블(Base Table)이 삭제(`DROP TABLE`)되면, 그 테이블을 기반으로 만들어진 **모든 인덱스와 뷰는 자동으로 의미를 잃게 됩니다.** 원본 데이터가 사라졌으니 뷰가 아무리 정의되어 있어도 가져올 데이터가 없기 때문입니다. 이 현상을 **Propagate Destroy(파괴의 전파)** 라고 표현합니다.

## ⚠️ 뷰의 제약 조건: 변경이 허용되지 않는 경우

### 뷰를 통한 데이터 조작의 한계

이론적으로는 뷰를 통해 `SELECT`, `INSERT`, `UPDATE`, `DELETE` 등 모든 데이터 조작이 가능합니다. 하지만 실제로는 **매우 제한적입니다.** 뷰는 가상의 테이블이기 때문에, 변경 작업을 허용하면 기본 테이블의 데이터 무결성이 깨질 수 있는 상황이 많습니다. 그래서 다음과 같은 경우에는 뷰를 통한 데이터 변경이 **거부(Rejected)** 됩니다.

### 변경이 거부되는 6가지 상황

#### 1. 산술식이나 집계 함수가 포함된 뷰

뷰의 컬럼이 `AVG(MIDTERM)`, `SUM(SCORE)`, `COUNT(*)` 같은 **집계 함수(Aggregate Function)** 나 산술 계산식으로 정의된 경우, 그 컬럼의 값을 역으로 기본 테이블에 반영하는 것이 불가능합니다. 평균값을 직접 수정한다는 것이 논리적으로 성립하지 않기 때문입니다.

```sql
-- 이 뷰를 통한 AVG_MIDTERM 값 변경은 거부됨
CREATE VIEW COSTAT (CNO, AVG_MIDTERM)
AS SELECT CNO, AVG(MIDTERM) FROM ENROLL GROUP BY CNO;

-- ❌ 거부됨: 집계값을 직접 수정할 수 없음
UPDATE COSTAT SET AVG_MIDTERM = 90 WHERE CNO = 'C001';
```

#### 2. GROUP BY 또는 DISTINCT가 포함된 뷰

뷰 정의에서 `GROUP BY`(그룹화)나 `DISTINCT`(중복 제거)가 사용된 경우도 변경이 불가능합니다. 여러 행을 하나로 합친 결과나 중복을 제거한 결과를 다시 원래 행으로 되돌려서 수정하는 것은 의미가 없기 때문입니다.

#### 3. 조인(JOIN)으로 만들어진 뷰

두 개 이상의 테이블을 조인하여 만든 **조인 뷰(Join View)** 는 기본적으로 수정이 제한됩니다. 수정된 값이 어느 테이블에 반영되어야 하는지 모호하기 때문입니다.

```sql
-- 조인 뷰: 두 테이블 결합
CREATE VIEW STUDENT_COURSE (SNO, SNAME, CNO, FINAL)
AS SELECT S.SNO, S.SNAME, E.CNO, E.FINAL
FROM STUDENT S, ENROLL E
WHERE S.SNO = E.SNO;

-- ❌ 거부될 수 있음: 어느 테이블을 수정해야 하는지 불명확
UPDATE STUDENT_COURSE SET SNAME = '홍길동' WHERE SNO = '100';
```

#### 4. WITH CHECK OPTION 위반

앞서 설명한 것처럼, `WITH CHECK OPTION`이 지정된 뷰에서 뷰의 WHERE 조건을 위반하는 변경은 거부됩니다.

#### 5. 셀프 조인(Self Join)으로 만들어진 뷰

자기 자신의 테이블을 가지고 조인하는 셀프 조인으로 만들어진 뷰도 변경이 제한됩니다.

#### 6. NOT NULL 컬럼이 뷰에 포함되지 않은 경우 삽입 불가

이 경우는 특히 중요하고 실수하기 쉽습니다. 기본 테이블에 `NOT NULL`(반드시 값 입력 필수) 조건이 있는 컬럼이 있는데, 그 컬럼이 뷰에 포함되지 않은 경우를 생각해 봅시다.

예를 들어 기본 테이블에 `A`, `B`, `C` 컬럼이 있고, 세 컬럼 모두 `NOT NULL`이라고 가정합시다. 그런데 뷰를 `A`와 `B`만으로 만들었다면, 이 뷰를 통해 데이터를 삽입하면 `C`에는 값을 넣을 수 없게 됩니다. 결국 `C`에 `NULL`이 들어가게 되어 `NOT NULL` 제약 조건을 위반하게 됩니다. 이런 상황에서는 뷰를 통한 삽입이 거부됩니다.

```sql
-- 기본 테이블: A, B, C 모두 NOT NULL
CREATE TABLE BASE_TABLE (
    A INT NOT NULL,
    B VARCHAR(10) NOT NULL,
    C VARCHAR(10) NOT NULL   -- C도 반드시 값 필요
);

-- ❌ 문제가 있는 뷰: NOT NULL 컬럼 C가 포함되지 않음
CREATE VIEW PARTIAL_VIEW (A, B)
AS SELECT A, B FROM BASE_TABLE;

-- ❌ 거부됨: 뷰를 통한 삽입 시 C에 NULL이 들어가 NOT NULL 위반
INSERT INTO PARTIAL_VIEW VALUES (1, '홍길동');
-- C가 NULL이 되어 기본 테이블의 NOT NULL 제약 위반 → 삽입 거부

-- ✅ 올바른 방법: NOT NULL 컬럼 C를 포함하여 뷰 생성
CREATE VIEW FULL_VIEW (A, B, C)
AS SELECT A, B, C FROM BASE_TABLE;
```

:::warning NOT NULL 컬럼과 뷰 설계 시 주의사항
뷰를 통해 데이터 삽입도 허용하려면, 기본 테이블에서 `NOT NULL` 제약이 있는 컬럼은 반드시 뷰에 포함시켜야 합니다. 그렇지 않으면 뷰를 통한 삽입 작업이 모두 거부됩니다.
:::

## ✨ 뷰의 장점과 단점 총정리

### 🟢 뷰의 장점

#### 1. 논리적 독립성 (Logical Independence) 제공

기본 테이블에 새로운 컬럼이 추가되거나 테이블 구조가 변경되어도, 기존에 만들어진 뷰는 아무런 영향을 받지 않습니다. 마치 카탈로그를 보고 물건을 주문하는 고객은 창고 내부 구조가 바뀌어도 주문 방법이 바뀌지 않는 것처럼요.

```sql
-- 기본 테이블에 새 컬럼 D 추가
ALTER TABLE STUDENT ADD COLUMN D VARCHAR(20);

-- 뷰는 변경 없이 그대로 사용 가능
SELECT * FROM CS_STUDENT;  -- 뷰 정의가 그대로 유지됨
```

**확장(Extension):** 기본 테이블에 컬럼이나 행이 추가되어도 기존 뷰는 영향받지 않음
**재구성(Restructuring):** 테이블의 물리적 저장 위치나 순서가 바뀌어도 뷰는 그대로 동작

#### 2. 보안성 (Security) 제공

일부 데이터만 보여주는 뷰를 통해 민감한 정보를 숨길 수 있습니다. 사용자는 자신에게 허용된 뷰를 통해서만 데이터에 접근하므로, 원본 테이블의 민감한 컬럼이나 행에는 접근할 수 없습니다.

#### 3. 데이터 관리 편의성

자주 사용하는 복잡한 쿼리(JOIN, 집계 등)를 뷰로 만들어 두면, 매번 복잡한 SQL을 작성하지 않고 뷰 이름만 호출하면 됩니다. 코드가 간결해지고 실수할 가능성도 줄어듭니다.

#### 4. 사용자별 다양한 활용

같은 기본 테이블에서 사용자의 역할에 따라 다른 뷰를 제공할 수 있습니다. 영업팀, 개발팀, 인사팀 각각에게 맞는 뷰를 설계하면 사용자 경험도 향상됩니다.

### 🔴 뷰의 단점

#### 1. ALTER VIEW 불가

뷰는 생성 후 수정이 불가능합니다. 변경이 필요하면 삭제 후 재생성해야 합니다.

#### 2. 삽입·삭제·갱신에 많은 제한

앞서 설명한 것처럼, 뷰를 통한 데이터 조작에는 많은 제약이 있습니다. 집계 함수, 조인, GROUP BY 등이 포함된 뷰는 사실상 읽기 전용(Read-Only)으로 사용해야 합니다.

#### 3. 인덱스(Index) 사용 불가

**인덱스(Index, 색인)**는 데이터 검색 속도를 높이는 도구입니다. 하지만 뷰는 논리적인 가상 테이블이기 때문에 인덱스를 생성할 수 없습니다.

#### 4. 성능 저하

뷰를 실행할 때마다 기본 테이블에서 데이터를 가져오는 과정이 반복되므로, 직접 테이블에 접근하는 것보다 성능이 떨어질 수 있습니다. 이를 해결하려면 **실체화된 뷰(Materialized View)** 를 사용해야 합니다.

| 구분 | 일반 뷰 | 실체화된 뷰 |
|---|---|---|
| 성능 | 느림 (매번 기본 테이블 접근) | 빠름 (저장된 데이터 직접 접근) |
| 데이터 신선도 | 항상 최신 | 동기화 필요 |
| 저장 공간 | 불필요 | 추가 저장 공간 필요 |
| 관리 복잡도 | 단순 | 동기화 관리 필요 |

:::tip 실체화된 뷰의 동기화
실체화된 뷰(Materialized View)를 사용하면 기본 테이블의 데이터가 변경될 때마다 뷰의 데이터도 함께 업데이트해야 합니다. 이 작업을 **동기화(Synchronization)** 라고 하며, 기본 테이블과 실체화된 뷰가 항상 같은 데이터를 가지도록 유지해야 합니다. 이를 위해 뷰 형성 방법(View Maintenance)이라는 운영 기법이 활용됩니다.
:::

## 🎯 주의사항 및 실무 팁

### 뷰 설계 시 체크리스트

뷰를 설계할 때 다음 항목들을 미리 점검하면 나중에 발생할 수 있는 문제를 예방할 수 있습니다.

**✅ 보안 목적의 뷰라면:**
- 민감한 컬럼이 완전히 제외되었는지 확인
- `WITH CHECK OPTION` 적용 여부 검토
- 원본 테이블 접근 권한이 일반 사용자에게 차단되어 있는지 확인

**✅ 데이터 조작(INSERT/UPDATE/DELETE)이 필요한 뷰라면:**
- `NOT NULL` 컬럼이 모두 포함되어 있는지 확인
- 집계 함수나 GROUP BY, DISTINCT가 없는지 확인
- 조인 뷰라면 변경 가능 여부를 미리 테스트

**✅ 성능이 중요한 뷰라면:**
- 자주 호출되는 뷰라면 실체화된 뷰(Materialized View) 검토
- 뷰를 통한 복잡한 조인이나 집계가 반복되면 성능 이슈 발생 가능

### 뷰 이름 명명 규칙 팁

뷰 이름은 테이블과 구분되도록 명확하게 짓는 것이 좋습니다. 실무에서는 `V_` 접두사를 붙이거나, `_VIEW`를 뒤에 붙이는 관행을 많이 사용합니다.

```sql
-- ✅ 추천하는 뷰 이름 예시
CREATE VIEW V_CS_STUDENT ...;       -- V_ 접두사 사용
CREATE VIEW CS_STUDENT_VIEW ...;    -- _VIEW 접미사 사용

-- ❌ 추천하지 않는 뷰 이름 예시
CREATE VIEW STUDENT ...;            -- 테이블 이름과 혼동될 수 있음
CREATE VIEW A ...;                  -- 의미를 알 수 없는 이름
```

:::tip 뷰 vs 테이블 구분 전략
대규모 프로젝트에서는 뷰와 테이블이 수백 개가 넘을 수도 있습니다. 명명 규칙(Naming Convention)을 팀 내에서 통일하면 유지보수와 협업이 훨씬 수월해집니다. 예를 들어 테이블은 `TB_`, 뷰는 `VW_`, 저장 프로시저는 `SP_`와 같이 접두사를 통일하는 것이 좋은 방법입니다.
:::

## 📚 핵심 정리

이 문서에서 학습한 뷰(View)의 보안 기능과 제약 조건에 대한 핵심 내용을 정리합니다.

- **뷰(View)** 는 기본 테이블로부터 유도된 가상의 테이블로, 데이터를 직접 저장하지 않는다.
- **논리적 뷰**는 실행 시점에 기본 테이블에서 데이터를 가져오며, **실체화된 뷰**는 데이터를 직접 저장하여 성능을 높인다.
- 뷰의 정의는 **시스템 카탈로그(데이터 사전)** 에 메타데이터 형태로 저장된다.
- 뷰는 **일부 데이터만 노출**함으로써 강력한 보안 기능을 제공하며, 뷰를 만든 사람은 정의를 알지만 사용자는 알 수 없다.
- **열 부분 집합 뷰**는 프로젝션, **행 부분 집합 뷰**는 셀렉션, **통계적 요약 뷰**는 집계 함수, **조인 뷰**는 JOIN으로 생성된다.
- **`WITH CHECK OPTION`** 은 뷰의 조건(WHERE 절)을 위반하는 데이터 변경 시 자동으로 거부하는 무결성 보호 장치이다.
- 뷰 수정은 `ALTER VIEW`로 불가능하며, **DROP 후 재생성**이 올바른 방법이다.
- **`DROP VIEW ... RESTRICT`** (기본값)는 참조 중인 뷰가 있으면 삭제를 거부하고, **`CASCADE`** 는 참조하는 모든 뷰를 연쇄 삭제한다.
- 집계 함수 포함, GROUP BY, DISTINCT, 조인, WITH CHECK OPTION 위반, NOT NULL 컬럼 미포함 등의 경우 **뷰를 통한 데이터 변경이 거부**된다.
- 뷰의 장점은 **논리적 독립성, 보안성, 관리 편의성, 다양한 활용**이며, 단점은 **ALTER 불가, 조작 제한, 인덱스 불가, 성능 저하**이다.
- 뷰는 **`인덱스(Index)`를 사용할 수 없으며**, 성능 개선이 필요할 때는 실체화된 뷰를 활용한다.
- 기본 테이블이 삭제되면 그를 기반으로 한 모든 뷰는 **자동으로 의미를 잃는다(Propagate Destroy)**.
