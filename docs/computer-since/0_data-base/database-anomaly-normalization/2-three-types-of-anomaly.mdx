# 삽입·삭제·갱신 이상현상의 종류와 예시 🗄️

데이터베이스를 배우다 보면 "이상현상(Anomaly)"이라는 단어를 자주 만나게 됩니다. 처음 들으면 굉장히 어렵게 느껴지지만, 사실 우리 일상에서 충분히 공감할 수 있는 문제들입니다. 이 문서에서는 데이터베이스 이상현상이 정확히 무엇인지, 왜 발생하는지, 어떤 종류가 있는지를 아주 쉽고 자세하게 살펴보겠습니다. 그리고 이를 해결하기 위한 **정규화(Normalization)** 와 그 기반이 되는 **함수적 종속성(Functional Dependency)** 개념까지 단계적으로 이해해 봅시다.

---

## 🤔 이상현상이란 무엇인가?

### 이상현상의 정의

**이상현상(Anomaly)** 이란 말 그대로 "정상이 아닌 상태", 즉 데이터베이스에 문제가 있는 상태를 의미합니다. 영어로는 **Anomaly(어노멀리)** 라고 하며, 정상적인 상태의 반대 개념입니다.

좀 더 구체적으로 정의하면, **관계 스키마(릴레이션의 구조, 즉 테이블의 설계 청사진)** 가 잘못 설계되었을 때 데이터가 불필요하게 중복되고, 그 중복으로 인해 데이터를 삽입·삭제·갱신하는 과정에서 예상치 못한 오류가 발생하는 현상입니다.

> 🏠 **비유로 이해하기**: 마치 한 집에 여러 사람의 이름과 소지품을 뒤섞어서 기록해 놓은 노트와 같습니다. "홍길동의 우산을 버렸더니 노트 전체가 날아가 버리는" 황당한 상황이 바로 이상현상입니다.

### 이상현상은 왜 생기는가?

이상현상의 근본 원인은 **서로 다른 개체(Entity)의 정보를 하나의 테이블에 무리하게 합쳐 놓았기 때문**입니다.

예를 들어, 학생 정보와 수강 정보는 서로 다른 개체입니다. 학생은 학교에 존재하는 존재이고, 수강은 학생이 특정 과목을 듣는 행위입니다. 이 둘을 하나의 테이블에 합쳐 놓으면, 학생에 대한 정보(학년 등)가 수강한 과목 수만큼 반복해서 저장됩니다. 이런 **불필요한 데이터 중복**이 이상현상의 씨앗이 됩니다.

:::info 핵심 배경 지식
데이터베이스는 설계(Design) → 구축(Build) → 운영(Operation) 의 단계를 거칩니다. 이상현상은 주로 **설계 단계에서 스키마가 잘못 설계될 때** 발생하며, 운영 단계에서 사용자나 개발자가 데이터를 조작할 때 그 문제가 드러납니다.
:::

## 📋 예제 테이블: 수강 릴레이션 이해하기

### 수강 테이블 구조 살펴보기

이상현상을 이해하기 위해 아래와 같은 **수강(수강 릴레이션)** 테이블을 예시로 사용하겠습니다. 이 테이블 하나에 학생 정보와 수강 정보가 모두 담겨 있는 상황입니다.

| 학번 | 과목번호 | 성적 | 학년 |
|------|----------|------|------|
| 100  | C413     | A    | 4    |
| 100  | E412     | A    | 4    |
| 200  | C123     | B    | 3    |
| 300  | C312     | A    | 1    |
| 300  | C324     | A    | 1    |
| 300  | C413     | A    | 1    |
| 400  | C312     | A    | 4    |
| 400  | C324     | A    | 4    |
| 400  | C413     | B    | 4    |
| 400  | E412     | C    | 4    |

### 기본키(Primary Key)는 무엇인가?

이 테이블에서 기본키를 어떻게 설정할까요?

- **학번만으로 기본키가 될 수 있을까?** → ❌ 불가능합니다. 100번 학생이 2개 행, 300번 학생이 3개 행으로 **중복**되어 있기 때문에, 학번 하나만으로는 각 행을 고유하게 식별할 수 없습니다.
- **과목번호만으로 기본키가 될 수 있을까?** → ❌ 불가능합니다. C413이라는 과목이 여러 행에 중복되어 있기 때문입니다.
- **학번 + 과목번호 복합키(Composite Key)** → ✅ 가능합니다. 학번과 과목번호를 함께 묶으면 각 행을 고유하게 식별할 수 있습니다.

따라서 이 테이블의 **기본키는 (학번, 과목번호) 복합키**입니다.

:::note 복합키란?
복합키(Composite Key)란 두 개 이상의 컬럼을 조합하여 하나의 기본키로 사용하는 것입니다. 마치 아파트 주소가 "동 + 호수"로 구성되어야 정확한 위치를 특정할 수 있는 것처럼, 두 속성을 합쳐야 비로소 유일한 식별자가 되는 경우에 사용합니다.
:::

## ❌ 삭제 이상(Deletion Anomaly)

### 삭제 이상이란?

**삭제 이상(Deletion Anomaly)** 이란, 어떤 데이터를 삭제하려고 했더니 **삭제해서는 안 되는 다른 정보까지 함께 사라지는 현상**입니다. 이를 **연쇄 삭제(Cascade Deletion)** 라고도 부릅니다.

> 📮 **비유로 이해하기**: 택배 주소가 적힌 종이에 이름, 연락처, 주소, 주문 상품 정보가 모두 적혀 있다고 상상해 보세요. 주문을 취소했다고 해서 그 종이 전체를 불태워버리면, 이 사람의 연락처와 주소 정보도 함께 사라져 버립니다. 상품 주문 정보만 지우고 싶었는데 말이죠.

### 삭제 이상 예시

위의 수강 테이블에서, **학번 200번 학생이 C123 과목을 수강 취소**했다고 가정해 봅시다.

학생 200번이 C123 과목을 철회했으니, 이 행(Row, 튜플)을 삭제해야 합니다.

```sql
-- 200번 학생의 C123 수강 기록을 삭제하는 SQL
DELETE FROM 수강
WHERE 학번 = 200 AND 과목번호 = 'C123';
-- ❌ 문제: 이 한 줄 삭제로 인해 200번 학생이 "3학년"이라는 정보도 함께 사라집니다!
```

삭제 전후를 비교해 보면:

| 상태 | 학번 | 과목번호 | 성적 | 학년 |
|------|------|----------|------|------|
| 삭제 전 | 200  | C123     | B    | **3**    |
| 삭제 후 | (행 자체가 사라짐) | — | — | **— (3학년 정보 소실!)** |

200번 학생은 **수강을 취소한 것이지, 학교를 그만둔 게 아닙니다.** 따라서 이 학생이 **3학년**이라는 정보는 반드시 데이터베이스 어딘가에 남아있어야 합니다. 하지만 이 테이블 구조에서는 학생 정보와 수강 정보가 뒤섞여 있기 때문에, 수강 기록을 담고 있는 유일한 행을 삭제하면 학년 정보도 함께 사라집니다.

:::warning 삭제 이상의 핵심 포인트
데이터베이스에서 삭제(DELETE)는 **행(튜플) 단위**로 이루어집니다. 행 안의 특정 컬럼 값만 골라서 삭제할 수 없습니다. (NULL로 바꿀 수는 있지만, 그것은 삭제가 아닙니다.) 따라서 학생의 수강 기록과 학생의 학년 정보가 같은 행에 있으면, 수강 기록을 지울 때 학년 정보도 피해를 입게 됩니다.
:::

## ➕ 삽입 이상(Insertion Anomaly)

### 삽입 이상이란?

**삽입 이상(Insertion Anomaly)** 이란, 어떤 데이터를 삽입하려고 할 때 **원하지 않는 불필요한 데이터까지 함께 입력해야만 삽입이 가능한 현상**입니다. 즉, 꼭 필요하지 않은 정보를 억지로 채워 넣어야 하는 문제입니다.

> 🏫 **비유로 이해하기**: 학교 원서 접수를 하는데 "지원 과목"란을 반드시 채워야만 이름과 학년을 등록할 수 있다고 상상해 보세요. 아직 어떤 과목을 들을지 결정하지 않은 신입생은 등록 자체를 할 수 없게 됩니다. 이것이 삽입 이상입니다.

### 삽입 이상 예시

**학번 600번, 2학년** 신입생이 학교에 새로 왔습니다. 이 학생의 학번과 학년 정보를 수강 테이블에 등록하려고 합니다.

그런데 앞서 설명했듯이, 이 테이블의 **기본키는 (학번, 과목번호) 복합키**입니다. 기본키의 중요한 특성 중 하나는 **NULL 값을 가질 수 없다**는 것입니다.

```sql
-- 600번 학생(2학년) 정보 삽입 시도
INSERT INTO 수강 (학번, 과목번호, 성적, 학년)
VALUES (600, NULL, NULL, 2);
-- ❌ 오류 발생! 기본키(과목번호)에 NULL을 넣을 수 없습니다.
-- 과목을 아직 수강 신청하지 않았는데 과목번호가 필요합니다.
```

이 학생은 아직 아무 과목도 수강 신청을 하지 않았습니다. 그런데도 과목번호 없이는 이 테이블에 데이터를 삽입할 수 없습니다. 결국 어쩔 수 없이 가짜 값을 넣어야 하는 상황이 됩니다.

```sql
-- ❌ 잘못된 해결 방법: 가짜(위조) 과목번호를 억지로 입력
INSERT INTO 수강 (학번, 과목번호, 성적, 학년)
VALUES (600, 'DUMMY001', NULL, 2);
-- 이렇게 넣은 'DUMMY001'은 실제 존재하지 않는 가짜 데이터입니다.
-- 이것을 "위조 튜플(Spurious Tuple)" 또는 "가짜 튜플"이라고 합니다.
```

이처럼 실제로는 존재하지 않는 임의의 쓰레기 값을 **위조 튜플(Spurious Tuple)** 이라고 합니다. 실제 데이터가 아닌 가짜 값이 테이블에 섞이면 데이터의 신뢰성이 크게 떨어집니다.

:::danger 위조 튜플은 절대 허용하면 안 됩니다
위조 튜플(가짜 데이터)이 테이블에 삽입되면 나중에 어떤 데이터가 진짜이고 가짜인지 구분이 불가능해집니다. 데이터베이스의 **무결성(Integrity, 데이터의 정확성과 일관성)** 이 근본적으로 훼손됩니다. 올바른 설계를 통해 위조 튜플이 발생하지 않도록 예방해야 합니다.
:::

## 🔄 갱신 이상(Update Anomaly)

### 갱신 이상이란?

**갱신 이상(Update Anomaly)** 이란, 중복 저장된 데이터 중 **일부만 갱신(수정)되고 나머지는 갱신되지 않아 데이터 간에 모순(불일치)이 발생하는 현상**입니다. 같은 사람에 대한 정보가 여러 행에 반복 저장되어 있을 때, 한 곳만 수정하면 나머지는 옛날 값을 그대로 갖게 됩니다.

> 📒 **비유로 이해하기**: 같은 사람의 전화번호가 주소록에 5군데 적혀 있다고 해봅시다. 번호가 바뀌어서 3군데만 고쳤다면, 나머지 2군데는 여전히 옛날 번호입니다. 이제 어떤 번호가 맞는지 아무도 모릅니다. 이것이 갱신 이상입니다.

### 갱신 이상 예시

수강 테이블에서 **학번 400번 학생의 학년이 잘못 입력**되어 있습니다. 4학년이 아니라 3학년으로 수정해야 합니다.

400번 학생의 데이터를 확인해 보면:

| 학번 | 과목번호 | 성적 | 학년 |
|------|----------|------|------|
| 400  | C312     | A    | **4** |
| 400  | C324     | A    | **4** |
| 400  | C413     | B    | **4** |
| 400  | E412     | C    | **4** |

400번 학생이 4개의 과목을 수강하고 있으므로, **학년 정보가 4개 행에 걸쳐 중복** 저장되어 있습니다. 이를 모두 3학년으로 바꾸려면 4번의 UPDATE를 수행해야 합니다.

```sql
-- 400번 학생의 학년을 3학년으로 수정하는 SQL
-- ❌ 문제가 있는 상황: 4번이나 반복해서 같은 내용을 수정해야 합니다
UPDATE 수강 SET 학년 = 3 WHERE 학번 = 400 AND 과목번호 = 'C312';
UPDATE 수강 SET 학년 = 3 WHERE 학번 = 400 AND 과목번호 = 'C324';
UPDATE 수강 SET 학년 = 3 WHERE 학번 = 400 AND 과목번호 = 'C413';
UPDATE 수강 SET 학년 = 3 WHERE 학번 = 400 AND 과목번호 = 'E412';
-- 만약 이 중 하나라도 빠뜨리면? 갱신 이상 발생!
```

만약 실수로 일부 행만 수정했다고 가정해 봅시다.

| 학번 | 과목번호 | 성적 | 학년 |
|------|----------|------|------|
| 400  | C312     | A    | **3** (수정됨) |
| 400  | C324     | A    | **3** (수정됨) |
| 400  | C413     | B    | **4** (수정 안 됨 ❌) |
| 400  | E412     | C    | **4** (수정 안 됨 ❌) |

이제 400번 학생의 학년이 어떤 행에서는 3학년, 어떤 행에서는 4학년으로 **서로 다른 값**을 가집니다. 이것을 **데이터 모순(Inconsistency)** 또는 **데이터 불일치**라고 합니다. 이런 데이터베이스는 신뢰할 수 없습니다.

:::warning 갱신 이상의 핵심 포인트
갱신 이상은 **데이터 중복**에서 비롯됩니다. 같은 정보가 여러 곳에 저장되어 있으면, 수정 시 모든 곳을 동시에 바꿔야 합니다. 사람이 하는 일이기 때문에 실수가 생길 수밖에 없고, 그 결과 동일한 사실에 대해 서로 다른 값이 공존하는 모순이 발생합니다.
:::

### 세 가지 이상현상 비교 요약

| 이상현상 종류 | 발생 상황 | 문제점 |
|--------------|-----------|--------|
| **삭제 이상** | 데이터를 삭제할 때 | 원하지 않는 다른 정보까지 함께 삭제됨 (정보 손실) |
| **삽입 이상** | 데이터를 삽입할 때 | 불필요한 데이터(위조 튜플)까지 함께 삽입해야만 원하는 데이터를 삽입 가능 |
| **갱신 이상** | 데이터를 수정할 때 | 중복 데이터 중 일부만 수정되어 데이터 불일치(모순) 발생 |

## 🛠️ 이상현상의 원인과 해결책: 정규화

### 이상현상의 근본 원인

앞서 살펴본 세 가지 이상현상은 모두 같은 뿌리에서 출발합니다. 바로 **서로 다른 종속 관계를 가진 속성들을 하나의 테이블에 무리하게 합쳐 놓았기 때문**입니다.

수강 테이블을 분석해 보면:
- **학번 → 학년**: 학번이 학년을 결정합니다. (학생 개인 정보)
- **(학번, 과목번호) → 성적**: 학번과 과목번호가 함께 성적을 결정합니다. (수강 정보)

이 두 가지 종속 관계가 하나의 테이블에 섞여 있어서 문제가 생깁니다.

:::info 이상현상의 원인을 한 문장으로
**속성(컬럼)들 간에 존재하는 서로 다른 종속 관계를 무리하게 하나의 테이블로 표현했기 때문에** 이상현상이 발생합니다.
:::

### 해결책: 정규화(Normalization)

**정규화(Normalization)** 란 이상현상을 제거하기 위해, 하나의 테이블을 **속성들 간의 종속 관계를 분석**하여 **여러 개의 테이블로 분해하는 과정**입니다.

수강 테이블을 두 개의 테이블로 분리해 봅시다.

**분리 전 (문제 있는 테이블):**

```
수강 테이블 (학번, 과목번호, 성적, 학년)
- 학번과 학년(학생 정보)이 수강 정보와 뒤섞여 있음
```

**분리 후 (정규화된 테이블):**

```
학생 테이블 (학번, 학년)
- 학생 개인 정보만 담당

수강 테이블 (학번, 과목번호, 성적)
- 수강 관련 정보만 담당
```

```sql
-- ✅ 정규화 후: 올바르게 분리된 테이블 생성 예시

-- 학생 테이블: 학생 고유 정보만 저장
CREATE TABLE 학생 (
    학번 INT PRIMARY KEY,     -- 학번이 기본키 (학생을 고유하게 식별)
    학년 INT NOT NULL         -- 학년 정보
);

-- 수강 테이블: 수강 정보만 저장
CREATE TABLE 수강 (
    학번 INT,                          -- 학생 테이블을 참조하는 외래키
    과목번호 VARCHAR(10),              -- 과목 식별자
    성적 CHAR(2),                      -- 해당 과목의 성적
    PRIMARY KEY (학번, 과목번호),      -- 학번+과목번호 복합키
    FOREIGN KEY (학번) REFERENCES 학생(학번)  -- 외래키로 학생 테이블과 연결
);
```

이렇게 분리하면:
- **삭제 이상 해결**: 200번 학생이 수강을 취소해도 학생 테이블의 학년 정보는 그대로 남아있습니다.
- **삽입 이상 해결**: 600번 신입생을 학생 테이블에 바로 등록할 수 있습니다. 수강 신청은 나중에 하면 됩니다.
- **갱신 이상 해결**: 400번 학생의 학년을 수정할 때 학생 테이블에서 딱 한 번만 수정하면 됩니다.

### 정규화 시 반드시 지켜야 할 원칙

정규화는 그냥 마음대로 테이블을 쪼개는 것이 아닙니다. 아래 세 가지 원칙을 반드시 지켜야 합니다.

**① 정보 표현의 무손실 원칙**
테이블을 분해하더라도 원래 가지고 있던 모든 정보가 분해된 테이블들 안에 그대로 보존되어야 합니다. 분해 과정에서 어떤 데이터도 사라져서는 안 됩니다.

**② 최소한의 중복만 허용**
중복을 100% 제거하는 것은 현실적으로 불가능합니다. 테이블 간 연결을 위해 **외래키(Foreign Key)** 처럼 꼭 필요한 중복은 허용하되, 불필요한 중복은 모두 제거해야 합니다.

**③ 분리의 원칙**
하나의 독립된 관계(종속성)는 하나의 별도 릴레이션(테이블)으로 분리해서 표현해야 합니다. 학생 정보는 학생 테이블에, 수강 정보는 수강 테이블에 독립적으로 관리해야 합니다.

### 데이터베이스 설계 가이드라인

정규화를 올바르게 수행하기 위한 설계 가이드라인을 정리하면 다음과 같습니다.

| 가이드라인 | 내용 |
|-----------|------|
| **다른 개체 타입은 합치지 말 것** | 학생 테이블, 교수 테이블, 과목 테이블은 각각 별도로 만들어야 합니다. |
| **중복·갱신 이상이 없도록 설계** | 같은 정보가 여러 테이블에 중복 저장되지 않도록 주의합니다. |
| **NULL 값 최소화** | NULL이 많이 발생하는 구조는 설계가 잘못된 신호입니다. |
| **위조 튜플 방지** | 기본키 제약 때문에 억지로 가짜 값을 넣어야 하는 상황을 만들면 안 됩니다. |

## 🔬 함수적 종속성(Functional Dependency): 정규화의 수학적 기반

### 함수적 종속성이란?

**함수적 종속성(Functional Dependency)** 이란 정규화의 수학적 이론적 근거입니다. 고등학교 수학에서 배운 **함수(Function)** 개념을 데이터베이스에 적용한 것입니다.

쉽게 말하면, **"X가 Y를 결정한다(X → Y)"** 는 관계입니다. 이때 X를 **결정자(Determinant)**, Y를 **종속자(Dependent)** 라고 부릅니다.

> 🪪 **비유로 이해하기**: 주민등록번호(X)가 이름(Y)을 결정합니다. 주민등록번호가 같으면 이름도 반드시 같습니다. 동일한 주민등록번호를 가진 두 사람이 서로 다른 이름을 가질 수는 없습니다. 이것이 함수적 종속성입니다.

### 함수적 종속성의 조건

X가 Y를 함수적으로 결정한다(X → Y)고 하려면:

> **테이블 안의 모든 행(튜플)에서, X 값이 같으면 Y 값도 반드시 같아야 합니다.**

다음 예시로 확인해 봅시다.

**✅ 함수 종속이 성립하는 경우:**

| 학번(X) | 학년(Y) |
|---------|---------|
| 100     | 4       |
| 200     | 3       |
| 100     | 4       |

학번이 100인 행이 두 개 있는데, 둘 다 학년이 4입니다. X 값(학번)이 같으면 Y 값(학년)도 같습니다. → **학번 → 학년** 성립 ✅

**❌ 함수 종속이 성립하지 않는 경우:**

| 학번(X) | 이름(Y) |
|---------|---------|
| A100    | 김철수  |
| A100    | 이영희  |

학번이 A100인데 이름이 김철수도 되고 이영희도 됩니다. X 값이 같은데 Y 값이 다릅니다. → **학번 → 이름** 성립 안 함 ❌ (실제 데이터베이스에서는 이런 상황이 있어서는 안 됩니다)

### 수강 테이블의 함수 종속 다이어그램

수강 테이블의 속성들 간 종속 관계를 **함수 종속 다이어그램(FD Diagram)** 으로 표현하면 다음과 같습니다.

```
학번 ──────────────────── → 학년
  │
  └──────────────┐
과목번호 ─────────┘──────── → 성적
```

- **(학번, 과목번호) → 성적**: 학번과 과목번호가 함께 성적을 결정합니다.
- **학번 → 학년**: 학번만으로 학년을 결정합니다.

이처럼 학번이 성적도 결정하는 것처럼 보이지만, 실제로는 학번만으로는 성적을 결정할 수 없고, 과목번호와 함께해야 성적을 결정할 수 있습니다. 이렇게 복합키의 일부만으로 다른 속성이 결정되는 관계를 **부분 함수 종속(Partial Functional Dependency)** 이라고 하며, 이것이 이상현상의 직접적인 원인이 됩니다.

## 📐 암스트롱 공리(Armstrong's Axioms): 함수 종속성 추론 규칙

### 암스트롱 공리란?

**암스트롱 공리(Armstrong's Axioms)** 란 기존에 알고 있는 함수 종속 관계로부터 새로운 함수 종속 관계를 논리적으로 추론(유추)할 수 있게 해주는 규칙들입니다. 수학의 공리(axiom)처럼, 이 규칙들은 항상 참입니다.

### 기본 규칙 세 가지

**① 재귀 규칙(Reflexivity Rule)**

X가 Y를 결정한다면(X ⊇ Y), X는 스스로를 결정합니다(X → X). 자기 자신은 항상 자기 자신을 결정합니다.

```
예: X → X 는 항상 성립
    학번 → 학번 (자기 자신은 항상 자신을 결정)
```

**② 증가 규칙(Augmentation Rule)**

X → Y 이면, WX → WY 도 성립합니다. 양쪽에 동일한 속성 W를 추가해도 함수 종속 관계는 유지됩니다.

```
예: 학번 → 학년 이면
    (학번, 과목번호) → (학년, 과목번호) 도 성립
    (양쪽에 과목번호를 동시에 추가했으므로)
```

**③ 이행 규칙(Transitivity Rule)**

X → Y 이고 Y → Z 이면, X → Z 도 성립합니다. 마치 삼단논법처럼 이행됩니다.

```
예: "나는 사람이고, 사람은 언젠가 죽는다, 따라서 나도 언젠가 죽는다"
    A → B 이고 B → C 이면, A → C 도 성립
```

### 추가 규칙 세 가지

기본 규칙 세 가지로부터 파생되는 추가 규칙들입니다.

**④ 결합 규칙(Union Rule)**

X → Y 이고 X → Z 이면, X → YZ 도 성립합니다.

```
예: 학번 → 이름 이고 학번 → 연락처 이면
    학번 → (이름, 연락처) 도 성립
```

**⑤ 분해 규칙(Decomposition Rule)**

X → YZ 이면, X → Y 이고 X → Z 도 성립합니다. 결합 규칙의 반대입니다.

```
예: 학번 → (이름, 연락처) 이면
    학번 → 이름 이고 학번 → 연락처 도 각각 성립
```

**⑥ 가이행 규칙(Pseudotransitivity Rule)**

W → X 이고 XY → Z 이면, WY → Z 가 성립합니다.

```
증명 과정:
1) W → X 양쪽에 Y를 증가시키면 → WY → XY (증가 규칙)
2) XY → Z 가 주어졌으므로
3) WY → XY → Z (이행 규칙 적용)
4) 따라서 WY → Z 성립
```

| 규칙 이름 | 표현 | 설명 |
|----------|------|------|
| 재귀 규칙 | X ⊇ Y → X → Y | 자기 자신을 포함하면 종속 성립 |
| 증가 규칙 | X → Y → WX → WY | 양쪽에 같은 속성 추가 시 성립 |
| 이행 규칙 | X→Y, Y→Z → X→Z | 삼단논법처럼 이행 적용 |
| 결합 규칙 | X→Y, X→Z → X→YZ | 두 종속을 하나로 결합 |
| 분해 규칙 | X→YZ → X→Y, X→Z | 하나의 종속을 두 개로 분해 |
| 가이행 규칙 | W→X, XY→Z → WY→Z | 증가+이행 규칙의 복합 |

## 🗜️ 최소 커버(Minimal Cover): 중복 종속성 제거하기

### 최소 커버란?

**최소 커버(Minimal Cover)** 란 함수 종속 관계들의 집합에서 **중복되는 종속 관계를 제거하여 꼭 필요한 최소한의 종속 관계만 남긴 집합**입니다.

> 🎒 **비유로 이해하기**: 여행 가방에 짐을 쌀 때, 중복되는 물건은 하나만 가져가는 것입니다. 우산이 두 개 있으면 하나는 집에 두고 하나만 챙기는 것처럼, 다른 종속 관계로부터 유추할 수 있는 종속 관계는 굳이 따로 보관할 필요가 없습니다.

### 최소 커버 예시

다음 함수 종속 집합이 있다고 가정합니다.

```
F = { B → A, D → A, AB → D }
```

**1단계: 오른쪽 속성이 모두 단일 속성인지 확인**

이미 오른쪽이 모두 단일 속성(A, A, D)이므로 1단계 통과 ✅

**2단계: 왼쪽에서 불필요한 속성 제거 (최소화)**

`AB → D` 에서 A 없이 `B → D`만으로도 성립하는지 확인합니다.

```
B → A 가 있고, 양쪽에 B를 증가시키면:
BB → AB (증가 규칙)
BB = B 이므로 → B → AB
AB → D 가 있으므로 (이행 규칙):
B → D 성립!
```

따라서 `AB → D`는 `B → D`로 대체 가능합니다.

변환된 집합: `F = { B → A, D → A, B → D }`

**3단계: 이행 종속으로 제거 가능한 종속성 확인**

```
B → D 이고, D → A 이므로 (이행 규칙):
B → A 성립!

따라서 B → A 는 이행 종속으로 유추 가능하므로 제거합니다.
```

**최소 커버 결과:**

```
F_min = { D → A, B → D }
```

이 두 가지 종속 관계만 있으면, 원래 세 가지 종속 관계를 모두 표현할 수 있습니다.

:::tip 최소 커버를 구하는 3단계
1. **오른쪽 단일화**: 오른쪽 속성이 여러 개라면 분해 규칙으로 단일 속성으로 만들기
2. **왼쪽 최소화**: 왼쪽에서 없어도 되는 속성 제거 (다른 종속으로 유추 가능한지 확인)
3. **중복 제거**: 이행 종속 등으로 유추 가능한 종속 관계 제거
:::

## ⚖️ 정규화의 목적과 한계

### 정규화는 이상현상 제거가 목적

정규화의 주목적은 명확합니다. **삽입 이상, 삭제 이상, 갱신 이상을 제거하여 데이터의 논리적 품질을 높이는 것**입니다. 물리적 성능(속도)을 향상시키기 위한 것이 아닙니다.

데이터베이스 설계는 크게 세 단계로 나뉩니다.

| 설계 단계 | 수행 내용 |
|----------|-----------|
| **개념적 설계** | ER 다이어그램으로 개체와 관계 모델링 |
| **논리적 설계** | 테이블 구조 설계, **정규화 수행** |
| **물리적 설계** | 인덱스, 저장 구조 최적화, 역정규화 고려 |

정규화는 **논리적 설계 단계**에서 수행합니다.

### 정규화의 단점: 성능 저하

정규화를 하면 테이블이 여러 개로 쪼개집니다. 원하는 정보를 조회할 때 여러 테이블을 **조인(JOIN)** 연산으로 합쳐야 합니다. 조인 연산이 많아질수록 쿼리 속도가 느려질 수 있습니다.

```
정규화 수준이 높아질수록:
✅ 이상현상 감소 (데이터 품질 향상)
❌ 조인 연산 증가 (쿼리 성능 저하)
```

### 역정규화(Denormalization)란?

이러한 이유로, 실제 현장에서는 성능이 너무 떨어질 경우 일부러 테이블을 다시 합치는 **역정규화(Denormalization)** 를 수행하기도 합니다. 역정규화는 **물리적 설계 단계**에서 수행합니다.

```
정규화(논리적 설계 단계): 이상현상 제거를 위해 테이블을 분리
역정규화(물리적 설계 단계): 성능 향상을 위해 분리된 테이블을 다시 합침
```

### 실무에서의 정규화 수준

정규화에는 단계가 있습니다. **1정규형, 2정규형, 3정규형, BCNF, 4정규형, 5정규형**이 순서대로 존재하며, 단계가 높을수록 이상현상이 더 잘 제거됩니다.

실무에서는 보통 **3정규형(3NF) 또는 BCNF 수준**까지 정규화를 수행합니다. 이 수준에서 이상현상 제거와 성능 유지의 균형을 가장 잘 맞출 수 있기 때문입니다.

:::tip 정규화 수준 선택의 균형
너무 낮은 단계 → 이상현상 잔존, 데이터 품질 저하
너무 높은 단계 → 테이블이 너무 많이 쪼개져서 조인 과부하, 성능 저하
실무 권장 → **3정규형(3NF) 또는 BCNF** 수준
:::

## 📝 주의사항과 실수하기 쉬운 것들

### 이상현상 관련 흔한 오해들

**오해 1: "데이터가 많아지면 이상현상이 생긴다"**
→ 이상현상은 데이터 양과 관계없이 **설계의 문제**입니다. 데이터가 1건이어도 잘못 설계된 테이블은 이상현상을 가질 수 있습니다.

**오해 2: "정규화를 하면 성능이 좋아진다"**
→ 정규화는 **논리적 품질(이상현상 제거)** 을 위한 것이지, 성능 향상을 위한 것이 아닙니다. 오히려 조인 연산이 늘어나 성능이 저하될 수 있습니다.

**오해 3: "중복은 무조건 나쁜 것이다"**
→ 테이블 간 연결을 위한 **외래키처럼 꼭 필요한 중복은 허용**해야 합니다. 중복이 완전히 없는 데이터베이스는 만들 수 없습니다. **불필요한 중복**을 최소화하는 것이 목표입니다.

:::warning 기본키(Primary Key) 설계 시 주의사항
- 기본키는 **NULL 값을 가질 수 없습니다**. 기본키가 NULL이면 행을 고유하게 식별할 수 없습니다.
- 복합키를 사용할 때, 구성하는 모든 컬럼이 NULL이 아닌 값을 가져야 합니다.
- 복합키가 필요한 상황이 자주 생긴다면, 설계를 재검토해야 합니다. 자연스럽게 복합키가 기본키가 되는 경우도 있지만, 인위적으로 `AUTO_INCREMENT` 같은 대리키(Surrogate Key)를 도입하는 것도 좋은 방법입니다.
:::

## ✅ 핵심 정리

- **이상현상(Anomaly)** 은 관계 스키마가 잘못 설계되어 데이터 중복이 발생하고, 이로 인해 삽입·삭제·갱신 시 예상치 못한 문제가 생기는 현상이다.
- **삭제 이상**: 데이터를 삭제할 때 원치 않는 정보까지 함께 삭제되어 정보가 손실되는 현상이다.
- **삽입 이상**: 데이터를 삽입할 때 불필요한 가짜(위조) 데이터까지 함께 삽입해야만 원하는 데이터를 저장할 수 있는 현상이다.
- **갱신 이상**: 중복된 데이터 중 일부만 갱신되고 나머지는 갱신되지 않아 데이터 불일치(모순)가 발생하는 현상이다.
- 이상현상의 근본 원인은 **서로 다른 종속 관계를 가진 속성들을 하나의 테이블에 무리하게 합쳐 놓은 것**이다.
- **정규화(Normalization)** 는 이상현상 제거를 위해 테이블을 속성 간 종속 관계에 따라 분해하는 과정이며, **논리적 설계 단계**에서 수행한다.
- 정규화의 3원칙: **정보 무손실 원칙**, **최소 중복 허용**, **분리의 원칙**이다.
- **함수적 종속성(X → Y)** 이란 "X 값이 같으면 Y 값도 반드시 같다"는 관계로, X는 결정자, Y는 종속자이다.
- **암스트롱 공리**의 기본 규칙은 **재귀 규칙, 증가 규칙, 이행 규칙** 세 가지이며, 이로부터 결합·분해·가이행 규칙을 유도할 수 있다.
- **최소 커버(Minimal Cover)** 는 중복되는 함수 종속 관계를 제거하여 꼭 필요한 최소한의 종속 관계 집합을 만드는 것이다.
- 정규화는 **이상현상 제거**가 목적이며, 성능 향상 목적이 아니다. 오히려 조인 연산 증가로 성능이 저하될 수 있다.
- 실무에서는 **3정규형(3NF) 또는 BCNF 수준**까지 정규화를 수행하는 것이 이상현상 제거와 성능 유지의 균형점이다.
- **역정규화(Denormalization)** 는 성능 향상을 위해 정규화된 테이블을 다시 합치는 것으로, **물리적 설계 단계**에서 수행한다.
