# 🗄️ 데이터베이스 이상현상의 개념과 원인

데이터베이스를 처음 배우는 분들이 가장 많이 당황하는 순간 중 하나는, "분명히 테이블을 만들었는데 왜 데이터가 이상하게 사라지거나, 뭔가를 저장하려면 억지로 다른 값을 넣어야 하지?"라는 상황을 맞닥뜨렸을 때입니다. 이 문서는 바로 그 의문을 해소하기 위한 것입니다. 데이터베이스 이상현상(Anomaly)이 무엇인지, 왜 발생하는지, 그리고 어떻게 해결하는지를 초보자도 이해할 수 있도록 아주 자세하게 풀어 설명합니다.

---

## 🤔 이상현상(Anomaly)이란 무엇인가?

### 정상과 비정상의 차이

**이상현상(Anomaly)**이라는 단어는 영어로 "Anomaly"라고 합니다. 뜻 그대로 **"정상적이지 않은 상태"**, 즉 **문제가 있는 상태**를 말합니다. 반대말은 "정상(Normal)"이겠죠.

데이터베이스에서 이상현상이란, **관계 스키마(테이블 구조)가 잘못 설계되었을 때 발생하는 예상치 못한 문제들**을 말합니다. 여기서 "관계 스키마(Schema)"란 테이블의 구조, 즉 어떤 컬럼이 있고 어떤 테이블끼리 연결되어 있는지를 정의한 설계도라고 생각하면 됩니다.

마치 집을 지을 때 설계도가 잘못되면 문이 안 열리거나, 벽에 물이 새거나, 창문이 바닥에 달리는 등 이상한 일이 생기는 것처럼, 데이터베이스 설계가 잘못되면 데이터를 저장하고 수정하고 삭제하는 과정에서 엉뚱한 문제들이 터져 나오게 됩니다.

### 이상현상은 왜 생기는가?

이상현상은 **데이터의 중복(Redundancy)**에서 시작됩니다. 예를 들어, 학생 정보를 저장하는 테이블이 있고, 수강 정보를 저장하는 테이블이 따로 있어야 하는데, 이 두 가지를 **하나의 테이블에 무리하게 합쳐놓으면** 문제가 생깁니다.

학생 테이블에는 학생 정보(학번, 학년, 이름 등)가 있어야 하고, 수강 테이블에는 수강 정보(학번, 과목번호, 성적)가 있어야 합니다. 그런데 이 두 개를 하나로 합쳐버리면, 학생의 학년 정보가 수강한 과목 수만큼 **반복해서 중복 저장**되는 문제가 생깁니다. 이렇게 중복된 데이터가 있는 상황에서 데이터를 삽입하거나, 삭제하거나, 수정하면 **원치 않는 부작용**이 발생하는데, 이것이 바로 이상현상입니다.

> **핵심 정의**: 이상현상이란, 관계 스키마가 잘못 설계되어 데이터 중복이 발생하고, 그 중복으로 인해 삽입·삭제·갱신 작업 시 의도치 않은 문제가 생기는 현상입니다.

:::info 이상현상이 발생하는 흐름
1. 설계자가 여러 개의 관계를 하나의 테이블에 무리하게 합침
2. 불필요한 데이터 중복 발생
3. 삽입, 삭제, 갱신 작업 시 예상치 못한 부작용 발생
4. 데이터의 일관성과 무결성이 깨짐
:::

## 📋 이상현상의 세 가지 종류

이상현상에는 크게 세 가지 종류가 있습니다. **삭제 이상(Deletion Anomaly)**, **삽입 이상(Insertion Anomaly)**, **갱신 이상(Update Anomaly)**입니다. 이 세 가지를 하나씩 자세히 살펴보겠습니다.

먼저, 예시로 사용할 **수강(受講) 테이블**을 살펴봅시다. 아래 테이블 하나로 학생 정보와 수강 정보를 모두 저장하고 있는 상황입니다.

| 학번 | 과목번호 | 성적 | 학년 |
|------|----------|------|------|
| 100  | C413     | A    | 4    |
| 100  | E412     | A    | 4    |
| 200  | C123     | B    | 3    |
| 300  | C312     | A    | 1    |
| 300  | C324     | A    | 1    |
| 300  | C413     | A    | 1    |
| 400  | C312     | C    | 4    |
| 400  | C324     | A    | 4    |
| 400  | C413     | B    | 4    |
| 400  | E412     | A    | 4    |

이 테이블에서 **기본키(Primary Key)**는 무엇일까요? 학번만으로는 기본키가 될 수 없습니다. 왜냐하면 학번 100번이 두 번 등장하고, 300번이 세 번, 400번이 네 번 등장하기 때문에 중복이 발생해서 행(Row)을 유일하게 식별할 수 없습니다. 과목번호도 마찬가지로 C413이 여러 번 등장합니다. 따라서 **학번과 과목번호를 합친 복합키(Composite Key)**를 기본키로 설정해야 합니다.

:::note 복합키(Composite Key)란?
두 개 이상의 컬럼을 묶어서 기본키로 사용하는 것입니다. 학번 혼자, 과목번호 혼자는 중복이 있지만, 학번 + 과목번호의 조합은 테이블에서 유일합니다. 마치 이름만으로는 같은 사람이 있을 수 있지만, 이름 + 생년월일의 조합은 훨씬 더 유일하게 식별할 수 있는 것과 같습니다.
:::

이제 이 수강 테이블을 가지고 세 가지 이상현상이 어떻게 발생하는지 살펴보겠습니다.

---

### 🗑️ 삭제 이상(Deletion Anomaly)

**삭제 이상**이란, 하나의 행(Row, 튜플)을 삭제했을 때 **원치 않는 다른 정보까지 함께 삭제되어 정보가 손실되는 현상**입니다. 쉽게 말해, "과목만 취소했을 뿐인데 내 학적 정보까지 사라져버리는" 황당한 상황입니다.

**예시:**
학번 200번 학생이 과목 C123을 수강 취소했습니다. 이 학생은 학교를 그만둔 게 아니라 단순히 과목 하나를 취소한 것이므로, **"200번 학생이 3학년이다"라는 정보는 데이터베이스에 남아있어야** 합니다.

그런데 현재 테이블에서 200번 학생의 행은 딱 하나뿐입니다. 데이터베이스에서 삭제 작업은 행(튜플) 전체를 삭제하는 것이기 때문에, C123 과목 수강 정보를 삭제하면 **200번 학생이 3학년이라는 정보도 함께 사라져버립니다**.

```sql
-- ❌ 잘못된 상황: 과목 취소를 위해 행을 삭제하면 학생 정보도 함께 사라짐
DELETE FROM 수강
WHERE 학번 = 200 AND 과목번호 = 'C123';
-- 이 명령 실행 후 200번 학생이 3학년이라는 정보가 영원히 사라짐!
```

```sql
-- ✅ 올바른 설계: 학생 테이블과 수강 테이블이 분리되어 있다면
-- 수강 정보만 삭제하고, 학생 정보는 별도 테이블에서 보존됨
DELETE FROM 수강
WHERE 학번 = 200 AND 과목번호 = 'C123';
-- 학생 테이블의 200번 학생, 3학년 정보는 그대로 유지됨
```

이렇게 **삭제 이상**은 "연쇄 삭제(Cascade Delete)"처럼 내가 지우려고 했던 것 이상의 정보가 사라지는 현상입니다. 마치 냉장고에서 우유만 꺼내려고 했는데 냉장고 전체가 쏟아져 나오는 것과 같습니다.

---

### ➕ 삽입 이상(Insertion Anomaly)

**삽입 이상**이란, 새로운 데이터를 삽입할 때 **원하지 않는 불필요한 데이터까지 함께 입력해야만 삽입이 가능한 현상**입니다. 즉, 내가 저장하고 싶은 정보를 저장하기 위해 **가짜 데이터(위조 튜플)**를 억지로 넣어야 하는 상황입니다.

**예시:**
학번 600번인 신입생이 입학해서 "600번 학생은 2학년이다"라는 정보를 테이블에 저장하려고 합니다. 이 학생은 아직 아무 과목도 수강 신청을 하지 않은 상태입니다.

그런데 현재 수강 테이블의 기본키는 **(학번, 과목번호)** 복합키입니다. 기본키의 중요한 특성 중 하나는 **NULL 값을 가질 수 없다**는 것입니다. 따라서 과목번호 없이는 행을 삽입할 수 없습니다.

결국 600번 학생의 학년 정보를 저장하기 위해서는, 실제로 수강 신청도 하지 않은 **가짜 과목번호(위조 튜플, Spurious Tuple)**를 억지로 넣어야만 합니다.

```sql
-- ❌ 잘못된 상황: 학생 정보만 넣으려 해도 과목번호가 없으면 삽입 불가
INSERT INTO 수강 (학번, 학년)
VALUES (600, 2);
-- 오류 발생! 기본키인 과목번호가 NULL이 될 수 없음

-- ❌ 억지로 가짜 값을 넣는 상황 (위조 튜플)
INSERT INTO 수강 (학번, 과목번호, 성적, 학년)
VALUES (600, '####', NULL, 2);
-- 존재하지 않는 과목번호 '####'를 억지로 넣어야 삽입 가능
-- 이것이 위조 튜플(Spurious Tuple)임
```

```sql
-- ✅ 올바른 설계: 학생 테이블이 분리되어 있다면
-- 학생 테이블에 자유롭게 삽입 가능
INSERT INTO 학생 (학번, 학년)
VALUES (600, 2);
-- 과목 수강 신청 전이라도 아무 문제 없이 저장됨
```

이처럼 **삽입 이상**은 쓸데없이 다른 정보를 끼워 넣어야만 원하는 정보를 저장할 수 있는 문제입니다. 마치 도서관에 책을 반납하려면 반드시 새 책도 한 권 사와야 한다는 황당한 규칙처럼 말이죠.

---

### ✏️ 갱신 이상(Update Anomaly)

**갱신 이상**이란, 중복된 데이터 중 **일부만 수정되고 나머지는 수정되지 않아서 데이터 간 모순(불일치)이 발생하는 현상**입니다. 같은 사람의 정보인데 어떤 행에서는 A, 다른 행에서는 B로 서로 다른 값이 저장되어 데이터베이스를 신뢰할 수 없게 되는 상황입니다.

**예시:**
학번 400번 학생의 학년이 실제로는 3학년인데 잘못 입력되어 4학년으로 저장되어 있습니다. 이를 3학년으로 수정해야 합니다.

위 테이블을 보면 400번 학생은 **4개의 과목**을 수강하고 있어서, 400번 학생의 행이 총 4개 있습니다. 4학년을 3학년으로 바꾸려면 이 4개의 행을 **모두 수정**해야 합니다.

```sql
-- ❌ 잘못된 상황: 400번 학생의 학년을 하나씩 모두 수정해야 함
UPDATE 수강 SET 학년 = 3 WHERE 학번 = 400 AND 과목번호 = 'C312';
UPDATE 수강 SET 학년 = 3 WHERE 학번 = 400 AND 과목번호 = 'C324';
UPDATE 수강 SET 학년 = 3 WHERE 학번 = 400 AND 과목번호 = 'C413';
UPDATE 수강 SET 학년 = 3 WHERE 학번 = 400 AND 과목번호 = 'E412';
-- 만약 마지막 줄을 빠뜨리면? 400번 학생이 어떤 행에서는 3학년, 어떤 행에서는 4학년!
```

```sql
-- ✅ 올바른 설계: 학생 테이블이 분리되어 있다면
-- 단 한 번의 수정으로 끝남
UPDATE 학생 SET 학년 = 3 WHERE 학번 = 400;
-- 400번 학생의 학년 정보가 한 곳에만 저장되어 있으므로 단 한 번만 수정하면 됨
```

만약 4개의 행 중 3개만 수정하고 1개를 빠뜨리면, 400번 학생의 학년이 어떤 행에서는 3학년, 어떤 행에서는 4학년으로 저장되어 **정보의 모순(Inconsistency)**이 생깁니다. 이런 데이터베이스는 신뢰할 수 없고 사용할 수 없게 됩니다.

이 세 가지 이상현상을 표로 정리하면 다음과 같습니다.

| 이상현상 종류 | 발생 상황 | 문제점 | 비유 |
|---|---|---|---|
| **삭제 이상** | 데이터 일부를 삭제할 때 | 삭제하면 안 되는 정보까지 함께 삭제됨 | 우유만 꺼내려다 냉장고 전체가 쏟아짐 |
| **삽입 이상** | 새 데이터를 삽입할 때 | 불필요한 가짜 데이터(위조 튜플)까지 입력해야 함 | 책 반납 시 새 책 구매를 강요당함 |
| **갱신 이상** | 기존 데이터를 수정할 때 | 일부만 수정되어 데이터 불일치(모순) 발생 | 주소를 바꿨는데 서류 중 일부에만 반영됨 |

:::warning 이상현상의 공통 원인
세 가지 이상현상의 공통 원인은 **불필요한 데이터 중복**입니다. 중복이 많을수록 이상현상이 많이 발생합니다. 중복의 근본 원인은 서로 다른 관계(학생 정보, 수강 정보)를 무리하게 하나의 테이블에 합친 것입니다.
:::

## 🔍 이상현상의 원인: 함수적 종속성(Functional Dependency)

### 왜 함수적 종속성을 알아야 하는가?

이상현상이 발생하는 근본 원인은, **속성(컬럼)들 사이에 존재하는 종속 관계를 무시하고 여러 가지 관계를 하나의 테이블로 표현했기 때문**입니다. 이 종속 관계를 제대로 이해하고 분석해야만 이상현상을 해결할 수 있습니다.

그 핵심 도구가 바로 **함수적 종속성(Functional Dependency)**입니다. 이것은 수학의 함수 개념에서 유래한 이론입니다. 고등학교 때 배운 함수 `f(x) = y`, 즉 x를 넣으면 y가 하나 결정되는 그 개념입니다.

### 함수적 종속성이란?

**함수적 종속성**을 한마디로 표현하면 **"X가 Y를 결정한다"**는 것입니다.

- **X**: 결정자(Determinant) — 결정하는 쪽
- **Y**: 종속자(Dependent) — 결정받는 쪽
- **표기법**: `X → Y` (X가 Y를 결정한다)

예를 들어, 학교에서 학생 한 명에게 고유한 **학번**이 부여됩니다. 학번을 알면 그 학생의 이름, 주소, 연락처, 학년, 전공 등 모든 정보를 알 수 있습니다. 이런 관계를 다음과 같이 표현합니다.

```
학번 → 이름
학번 → 학년
학번 → 전공
학번 → 연락처
```

이것이 바로 함수적 종속성입니다. 마치 **주민등록번호**가 대한민국 국민을 유일하게 식별하는 것처럼, 결정자(주민등록번호)가 같으면 종속자(이름, 주소 등)도 반드시 같아야 합니다.

> **핵심 원칙**: X 값이 같으면, 그에 대응하는 Y 값도 반드시 같아야 한다. 하나의 X 값에 대해 Y 값은 오직 하나만 연관되어야 한다.

### 함수적 종속성 판별 예시

아래 예시로 함수 종속이 성립하는지 판별해봅시다.

```
예시 테이블 (속성 B가 속성 A를 결정하는가?)

B    A
2 → 1   (B=2이면 A=1)
4 → 1   (B=4이면 A=1)
2 → 1   (B=2이면 A=1 — B 값이 같고, A 값도 같음 ✅)

→ B=2일 때 A가 항상 1이므로 함수 종속 성립! (B → A)
```

```
반례 (함수 종속이 성립하지 않는 경우)

D    결과
4 → 5   (D=4이면 결과=5)
4 → 5   (D=4이면 결과=5)
4 → 1   (D=4인데 결과=1? 같은 D 값에 다른 결과!)

→ D=4일 때 결과가 5도 되고 1도 되므로 함수 종속 성립 안 함! ❌
```

:::tip 함수 종속 판별 핵심 규칙
X 값이 동일한 두 행(튜플)을 찾았을 때, Y 값도 동일하면 함수 종속 성립. Y 값이 다르면 함수 종속 불성립.
:::

### 수강 테이블의 함수 종속 관계

앞서 살펴본 수강 테이블에서 속성들 간의 함수 종속 관계를 분석해보면 다음과 같습니다.

```
(학번, 과목번호) → 성적   : 학번과 과목번호를 알면 성적이 결정됨
학번 → 학년              : 학번만 알아도 학년이 결정됨
```

이것을 **함수 종속 다이어그램**으로 표현하면 한눈에 관계를 파악할 수 있습니다.

```
┌─────────────────────────────────────────┐
│  학번 ─────────────────────────→ 학년  │
│    ↘                                    │
│      (학번, 과목번호) ──────→ 성적     │
│    ↗                                    │
│  과목번호                               │
└─────────────────────────────────────────┘
```

여기서 중요한 점은, **학번은 학년을 결정**하고, **(학번, 과목번호) 복합키는 성적을 결정**하는, 서로 다른 두 가지 종속 관계가 **하나의 테이블 안에 섞여 있다**는 것입니다. 이 두 관계를 억지로 하나의 테이블에 합쳐놓았기 때문에 이상현상이 발생했던 것입니다.

## 📐 암스트롱 공리(Armstrong's Axioms): 함수 종속성 추론 규칙

### 왜 추론 규칙이 필요한가?

데이터베이스를 설계할 때 테이블의 모든 함수 종속 관계를 직접 하나하나 나열하기가 어렵습니다. 이미 알고 있는 종속 관계로부터 **새로운 종속 관계를 논리적으로 유추**할 수 있다면 훨씬 효율적입니다. 이를 위한 기본 규칙이 **암스트롱 공리(Armstrong's Axioms)**입니다.

암스트롱 공리에는 3가지 기본 규칙과 3가지 추가 규칙이 있습니다.

### 기본 3가지 규칙

**① 재귀 규칙(Reflexivity Rule)**

자기 자신은 자기 자신을 결정한다는 규칙입니다. X → Y이면 X → X가 성립합니다. 즉, Y는 항상 Y를 결정합니다.

```
예: 학번 → 학번  (학번이 같으면 학번이 같다 — 당연한 말이죠!)
    성적 → 성적  (성적이 같으면 성적이 같다)
```

이것은 너무 당연한 말처럼 보이지만, 수학적으로 함수 종속의 기초를 이루는 규칙입니다.

**② 증가 규칙(Augmentation Rule)**

X → Y이면, 양쪽에 동일한 속성 W를 추가해도 종속 관계가 성립합니다. 즉, `WX → WY`가 됩니다.

```
예: 학번 → 학년  이면
    (학번, 과목번호) → (학년, 과목번호)  도 성립
```

마치 양쪽 저울에 동일한 무게를 추가하면 균형이 유지되는 것과 같습니다. 양쪽에 같은 것을 더해도 관계는 그대로 성립합니다.

**③ 이행 규칙(Transitivity Rule)**

X → Y이고 Y → Z이면, X → Z가 성립합니다. 이것은 우리가 일상에서도 자주 쓰는 논리입니다.

```
예: "나는 사람이고, 사람은 언젠가 죽는다. 따라서 나도 언젠가 죽는다."

학번 → 학과코드  이고
학과코드 → 학과명  이면
학번 → 학과명  도 성립!
```

### 추가 3가지 규칙

기본 3가지 규칙에서 유도할 수 있는 추가 규칙들입니다.

**④ 결합 규칙(Union Rule)**

X → Y이고 X → Z이면, X → YZ가 성립합니다. 하나의 결정자가 여러 종속자를 결정하면 묶어서 표현할 수 있습니다.

```
학번 → 이름  이고
학번 → 학년  이면
학번 → (이름, 학년)  이 성립
```

**⑤ 분해 규칙(Decomposition Rule)**

결합 규칙의 반대입니다. X → YZ이면, X → Y이고 X → Z로 분해할 수 있습니다.

```
학번 → (이름, 학년)  이면
학번 → 이름  이고
학번 → 학년  으로 분해 가능
```

**⑥ 가이행 규칙(Pseudotransitivity Rule)**

W → X이고 XY → Z이면, WY → Z가 성립합니다.

```
풀이 과정:
1. W → X  (이미 알고 있는 종속 관계)
2. 양쪽에 Y를 증가(Augmentation): WY → XY
3. XY → Z  (이미 알고 있는 종속 관계)
4. WY → XY → Z  이므로 이행 규칙에 의해 WY → Z 성립!
```

이 규칙들을 표로 정리하면 다음과 같습니다.

| 규칙 이름 | 조건 | 결론 | 설명 |
|---|---|---|---|
| 재귀 규칙 | X → Y | X → X | 자기 자신은 자신을 결정 |
| 증가 규칙 | X → Y | WX → WY | 양쪽에 같은 속성 추가 |
| 이행 규칙 | X → Y, Y → Z | X → Z | 연쇄적 종속 관계 |
| 결합 규칙 | X → Y, X → Z | X → YZ | 여러 종속자를 묶음 |
| 분해 규칙 | X → YZ | X → Y, X → Z | 묶인 종속자를 분리 |
| 가이행 규칙 | W → X, XY → Z | WY → Z | 증가+이행 규칙 결합 |

:::tip 암스트롱 공리 활용법
이미 알고 있는 종속 관계 집합에서 출발해서, 위의 6가지 규칙을 적용하면 새로운 종속 관계를 논리적으로 유추할 수 있습니다. 이것이 데이터베이스 정규화의 수학적 기반입니다.
:::

## 🧹 최소 커버(Minimal Cover): 중복 종속성 제거

### 왜 최소 커버가 필요한가?

함수 종속성을 분석하다 보면 **중복되는 종속 관계**가 생길 수 있습니다. 예를 들어 X → Y이고 Y → Z이면 X → Z를 이미 유추할 수 있는데, 굳이 X → Z를 별도로 저장할 필요가 없습니다. 이렇게 **유추 가능한 중복 종속 관계를 제거하여 최소한으로 필요한 종속 관계 집합만 남기는 것**을 **최소 커버(Minimal Cover)**라고 합니다.

마치 지도에서 불필요한 경로를 제거하고 꼭 필요한 경로만 남기는 것과 같습니다.

### 최소 커버 찾는 방법

최소 커버를 구하는 규칙은 다음 세 가지입니다.

1. **오른쪽(종속자)에는 속성이 하나만** 있어야 한다. 만약 `X → YZ`처럼 오른쪽에 여러 속성이 있으면 분해 규칙으로 `X → Y`, `X → Z`로 분리한다.
2. **왼쪽(결정자)을 최소화**한다. 왼쪽에서 속성 하나를 제거해도 같은 종속 관계를 유추할 수 있다면, 그 속성은 불필요하므로 제거한다.
3. **중복 종속성을 제거**한다. 다른 종속 관계로 유추할 수 있는 종속 관계는 제거한다.

### 최소 커버 예시

다음 함수 종속 집합에서 최소 커버를 구해봅시다.

```
B → A
D → A
AB → D
```

**1단계**: 오른쪽 속성 단일화 확인
모두 오른쪽에 속성이 하나씩이므로 조건 1 만족. 통과.

**2단계**: 왼쪽 결정자 최소화
`AB → D`에서 A만으로 D를 결정할 수 있는지, B만으로 D를 결정할 수 있는지 확인.

```
B → A  (이미 알고 있음)
B → A에 양쪽에 B를 증가: BB → AB, BB는 B이므로 B → AB
B → AB이고 AB → D이므로 이행 규칙에 의해 B → D 성립!
따라서 AB → D는 B → D로 대체 가능
```

따라서 종속 집합은 다음과 같이 바뀝니다.

```
B → A
D → A
B → D
```

**3단계**: 중복 종속성 제거
`B → D`이고 `D → A`이므로 이행 규칙에 의해 `B → A`를 유추할 수 있습니다. 따라서 `B → A`는 중복이므로 제거합니다.

```
최소 커버 결과:
D → A
B → D
```

이 두 가지 종속 관계만 있으면 원래의 세 가지 종속 관계를 모두 표현할 수 있습니다!

:::info 최소 커버가 여러 개일 수 있음
최소 커버는 반드시 하나가 아닐 수 있습니다. 어떤 순서로, 어떤 방식으로 중복을 제거하느냐에 따라 다른 최소 커버가 나올 수 있습니다. 중요한 것은 모든 원래 종속 관계를 유추할 수 있어야 하고, 더 이상 중복을 제거할 수 없는 최소 상태여야 한다는 것입니다.
:::

## 🛠️ 이상현상의 해결책: 정규화(Normalization)

### 정규화란 무엇인가?

지금까지 이상현상이 무엇인지, 왜 생기는지를 살펴봤습니다. 그렇다면 이 문제를 어떻게 해결할까요? 해답은 바로 **정규화(Normalization)**입니다.

**정규화**란, 하나의 테이블(릴레이션)을 **속성들 간의 종속 관계를 분석하여** 여러 개의 작은 테이블로 분해하는 과정입니다. 하나의 종속 관계가 하나의 테이블에 표현되도록 구조를 개선하는 것이죠.

앞서 살펴본 수강 테이블에는 두 가지 종속 관계가 섞여 있었습니다.

```
관계 1: 학번 → 학년         (학생 정보)
관계 2: (학번, 과목번호) → 성적  (수강 정보)
```

이 두 관계를 분리해서 **학생 테이블**과 **수강 테이블** 두 개로 나눕니다.

```
학생 테이블: (학번, 학년)
수강 테이블: (학번, 과목번호, 성적)
```

이렇게 분리하면?
- **삭제 이상 해결**: 200번 학생의 수강 정보를 삭제해도 학생 테이블의 학년 정보는 그대로 유지됩니다.
- **삽입 이상 해결**: 600번 학생의 학년 정보를 학생 테이블에 자유롭게 삽입할 수 있습니다. 과목 정보가 없어도 됩니다.
- **갱신 이상 해결**: 400번 학생의 학년을 수정할 때 학생 테이블에서 단 한 번만 수정하면 됩니다.

### 정규화의 세 가지 원칙

정규화를 올바르게 수행하기 위해 반드시 지켜야 할 원칙이 있습니다.

**원칙 1: 정보 표현의 무손실(Lossless Decomposition)**

테이블을 분해해도 **원래 있던 데이터가 손실되어서는 안 됩니다**. 분해 전과 분해 후에 표현할 수 있는 정보의 양이 동일해야 합니다. 마치 레고 블록을 분리했다가 다시 조립해도 원래 모양이 나와야 하는 것처럼요.

**원칙 2: 최소한의 중복만 허용**

중복을 완전히 100% 제거하는 것은 현실적으로 불가능합니다. 테이블을 연결하는 외래키(Foreign Key)처럼 꼭 필요한 중복은 허용하지만, **불필요한 중복은 최대한 제거**해야 합니다. 중복이 줄어들수록 이상현상이 줄어들기 때문입니다.

**원칙 3: 분리의 원칙(Separation Principle)**

**하나의 독립된 관계는 하나의 별도 테이블로 분리**해서 표현해야 합니다. 학생 정보(학번-학년 관계)와 수강 정보(학번-과목번호-성적 관계)는 서로 독립적인 관계이므로, 각각 별도의 테이블로 분리해야 한다는 원칙입니다.

### 데이터베이스 설계 가이드라인

정규화를 포함한 좋은 데이터베이스 설계를 위한 가이드라인을 정리하면 다음과 같습니다.

- ✅ **다른 개체 타입은 하나로 합치지 마라**: 학생 테이블, 교수 테이블, 과목 테이블은 각각 따로 설계해야 합니다.
- ✅ **중복이나 갱신 이상이 발생하지 않도록 설계하라**: 테이블 구조를 설계할 때부터 이상현상을 예방해야 합니다.
- ✅ **NULL 값이 가능하면 발생하지 않도록 하라**: NULL이 많은 설계는 데이터 품질 문제를 일으킵니다.
- ✅ **위조 튜플(Spurious Tuple)이 발생하지 않도록 하라**: 가짜 데이터를 억지로 넣어야 하는 상황이 생기지 않도록 설계해야 합니다.

### 정규화의 단계

정규화는 여러 단계로 나뉩니다. 각 단계를 거칠수록 이상현상이 줄어듭니다.

| 정규화 단계 | 명칭 | 특징 |
|---|---|---|
| 1NF | 제1정규형 | 모든 속성 값이 원자(더 쪼갤 수 없는) 값 |
| 2NF | 제2정규형 | 부분 함수 종속 제거 |
| 3NF | 제3정규형 | 이행 함수 종속 제거 |
| BCNF | 보이스-코드 정규형 | 모든 결정자가 후보키 |
| 4NF | 제4정규형 | 다치 종속 제거 |
| 5NF | 제5정규형 | 조인 종속 제거 |

:::tip 실무에서의 정규화 수준
실무에서는 대부분 **3정규형(3NF) 또는 BCNF 수준**까지 정규화를 수행합니다. 4NF, 5NF까지 가면 무결성은 완벽해지지만 성능 문제가 심각해질 수 있기 때문입니다.
:::

### 정규화와 성능의 트레이드오프

정규화를 하면 테이블이 여러 개로 나뉩니다. 원하는 데이터를 조회할 때 여러 테이블을 **조인(JOIN)**해야 하는 경우가 많아지고, 조인 연산이 많아질수록 **데이터베이스 성능이 떨어질 수 있습니다**.

:::warning 정규화는 성능 향상이 목적이 아님!
정규화의 목적은 **이상현상 제거**와 **데이터 무결성 유지**입니다. 성능을 향상시키기 위한 것이 아닙니다. 오히려 지나친 정규화는 성능을 저하시킬 수 있습니다.

반대로 성능을 위해 정규화된 테이블을 다시 합치는 것을 **역정규화(Denormalization)**라고 하며, 이는 물리적 설계 단계에서 고려합니다.
:::

정규화는 **논리적 설계 단계**에서 수행하고, 역정규화는 **물리적 설계 단계**에서 성능을 고려하여 필요한 경우에만 수행합니다.

```
데이터베이스 설계 단계:
1. 개념적 설계 → ER 다이어그램 작성
2. 논리적 설계 → 정규화(이상현상 제거) ← 여기서 정규화!
3. 물리적 설계 → 역정규화(성능 최적화) ← 여기서 필요시 역정규화
```

## ⚠️ 주의사항 및 실전 팁

### 자주 하는 실수들

:::danger 절대 하지 말아야 할 것
서로 다른 개체(학생, 교수, 과목 등)의 정보를 하나의 테이블에 무리하게 합치지 마세요. 이것이 모든 이상현상의 출발점입니다. 테이블 수가 줄어든다고 좋은 설계가 아닙니다.
:::

:::danger NULL 값을 기본키에 허용하지 말 것
기본키는 절대로 NULL 값을 가질 수 없습니다. 만약 복합키를 사용한다면 복합키를 구성하는 모든 컬럼에 NULL이 들어가면 안 됩니다. 삽입 이상의 직접적인 원인이 됩니다.
:::

### 정규화 진행 시 체크리스트

정규화를 수행할 때 다음 사항들을 체크하세요.

- [ ] 각 테이블의 기본키가 명확하게 정의되어 있는가?
- [ ] 하나의 테이블에 서로 다른 종속 관계가 섞여 있지는 않은가?
- [ ] 데이터 중복이 불필요하게 발생하고 있지는 않은가?
- [ ] 삽입, 삭제, 갱신 시 의도치 않은 부작용이 발생하지는 않는가?
- [ ] 테이블 분해 후에도 원래 정보를 모두 복원할 수 있는가?
- [ ] 분해된 테이블들이 외래키(Foreign Key)로 적절히 연결되어 있는가?

:::tip 함수 종속 다이어그램 먼저 그리기
정규화를 시작하기 전에 먼저 속성들 간의 함수 종속 관계를 다이어그램으로 그려보세요. 어떤 속성이 무엇을 결정하는지 시각적으로 파악하면 어떻게 테이블을 분리해야 할지 훨씬 명확하게 보입니다.
:::

### 이상현상 점검 예시 코드

실제 업무에서 이상현상이 있는 테이블인지 확인하는 간단한 SQL 쿼리 예시입니다.

```sql
-- 갱신 이상 점검: 동일한 학번에 서로 다른 학년 값이 존재하는지 확인
-- ❌ 아래 쿼리 결과가 존재하면 갱신 이상 발생 가능성!
SELECT 학번, COUNT(DISTINCT 학년) AS 학년_종류_수
FROM 수강
GROUP BY 학번
HAVING COUNT(DISTINCT 학년) > 1;
-- 같은 학번인데 학년이 2가지 이상이면 데이터 불일치 상태!

-- ✅ 올바른 설계에서는 위 쿼리의 결과가 항상 0건이어야 함
-- (학생 테이블에 학년이 별도로 관리되므로)
```

```sql
-- 데이터 중복 확인: 학번별로 학년 정보가 몇 번 중복 저장되어 있는지 확인
-- ❌ 아래 쿼리에서 학년_중복_수가 크면 클수록 갱신 이상 위험 증가
SELECT 학번, 학년, COUNT(*) AS 학년_중복_수
FROM 수강
GROUP BY 학번, 학년
ORDER BY 학년_중복_수 DESC;
-- 중복이 많을수록 갱신 이상 발생 가능성 높음
```

```sql
-- 정규화 후 올바른 테이블 구조 예시

-- 학생 테이블: 학생 정보만 저장 (학번 → 학년 관계)
CREATE TABLE 학생 (
    학번 VARCHAR(10) PRIMARY KEY,  -- 기본키, NULL 불허
    학년 INT NOT NULL               -- 학생의 학년 정보
);

-- 수강 테이블: 수강 정보만 저장 ((학번, 과목번호) → 성적 관계)
CREATE TABLE 수강 (
    학번     VARCHAR(10),            -- 학생 테이블 참조 (외래키)
    과목번호 VARCHAR(10),            -- 과목 정보 참조
    성적     CHAR(2),                -- 해당 과목의 성적
    PRIMARY KEY (학번, 과목번호),   -- 복합키로 기본키 설정
    FOREIGN KEY (학번) REFERENCES 학생(학번)  -- 학생 테이블과 연결
);
-- 이렇게 분리하면 삽입/삭제/갱신 이상현상 모두 해결됨!
```

## 📝 핵심 정리

이 문서에서 배운 핵심 개념을 정리합니다.

- **이상현상(Anomaly)**은 관계 스키마가 잘못 설계되어 데이터 중복이 발생하고, 이로 인해 삽입·삭제·갱신 작업 시 의도치 않은 문제가 생기는 현상이다.
- **삭제 이상**은 튜플 삭제 시 삭제하면 안 되는 정보까지 함께 삭제되어 정보가 손실되는 현상이다.
- **삽입 이상**은 데이터 삽입 시 불필요한 가짜 데이터(위조 튜플)까지 강제로 입력해야 하는 현상이다.
- **갱신 이상**은 중복된 데이터의 일부만 수정되어 데이터 간 불일치(모순)가 발생하는 현상이다.
- **이상현상의 근본 원인**은 서로 다른 종속 관계를 하나의 테이블에 무리하게 합쳐서 발생하는 불필요한 데이터 중복이다.
- **함수적 종속성(Functional Dependency)**은 "X → Y", 즉 X가 Y를 결정하는 관계로, X 값이 같으면 Y 값도 반드시 같아야 한다.
- **결정자(X)**는 결정하는 속성이고, **종속자(Y)**는 결정받는 속성이다.
- **암스트롱 공리**는 기존 종속 관계로부터 새로운 종속 관계를 논리적으로 유추하는 6가지 규칙(재귀, 증가, 이행, 결합, 분해, 가이행)이다.
- **최소 커버(Minimal Cover)**는 중복되는 종속 관계를 제거하여 최소한으로 필요한 종속 관계 집합만 남긴 것이다.
- **정규화(Normalization)**는 속성들 간의 종속성을 분석하여 하나의 테이블을 여러 테이블로 분해함으로써 이상현상을 제거하는 과정이다.
- 정규화는 **이상현상 제거**가 목적이며, 성능 향상이 목적이 아니다. 오히려 조인 연산 증가로 성능이 저하될 수 있다.
- 정규화는 **논리적 설계 단계**에서, 역정규화는 **물리적 설계 단계**에서 수행한다.
- 실무에서는 일반적으로 **3정규형(3NF) 또는 BCNF 수준**의 정규화를 적용한다.
