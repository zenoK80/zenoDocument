# 🗄️ 함수적 종속성과 스키마 변환 원리

데이터베이스를 처음 배울 때 가장 많이 마주치는 개념 중 하나가 바로 **이상현상(Anomaly)** 과 **정규화(Normalization)** 입니다. 테이블을 만들었는데 데이터를 추가하거나 삭제할 때 뭔가 이상하게 동작한다면, 십중팔구 설계 단계에서 문제가 생긴 것입니다. 이 문서에서는 왜 이런 문제가 발생하는지, 어떻게 해결할 수 있는지를 처음 배우는 분들도 충분히 이해할 수 있도록 아주 자세하게 풀어보겠습니다.

---

## 📌 이상현상이란 무엇인가?

### 이상현상의 정의

**이상현상(Anomaly)** 이란 말 그대로 "정상적이지 않은 상태", 즉 **데이터베이스를 사용하는 과정에서 예상치 못하게 발생하는 문제**를 의미합니다. 영어로는 'Anomaly(어노멀리)'라고 부릅니다.

데이터베이스를 설계하고 구축한 다음, 실제로 사용하는 단계에서 데이터를 추가(삽입)하거나, 수정(갱신)하거나, 제거(삭제)할 때 원래 의도하지 않았던 부작용이 생기는 경우가 있습니다. 이런 현상을 우리는 이상현상이라고 부릅니다.

### 왜 이상현상이 발생하는가?

이상현상이 발생하는 근본적인 원인은 **관계 스키마(테이블 구조)가 잘못 설계**되어 있기 때문입니다. 쉽게 비유하자면, 학생 정보를 담는 서랍과 교수 정보를 담는 서랍이 따로 있어야 하는데, 두 가지 정보를 한 서랍에 뒤섞어 넣은 것과 같습니다. 나중에 특정 정보를 꺼내거나 바꾸려 할 때 의도치 않게 다른 정보까지 함께 영향을 받게 되는 것이죠.

> 🔑 **핵심:** 관계 스키마가 잘못 설계되면 불필요한 데이터 중복이 발생하고, 이 중복이 삽입·삭제·갱신 시 이상현상의 원인이 됩니다.

### 예시 테이블: 수강 릴레이션

이상현상을 설명하기 위해 아래와 같은 **수강 테이블(릴레이션)** 을 예시로 사용합니다.

| 학번 | 과목번호 | 성적 | 학년 |
|------|----------|------|------|
| 100  | C413     | A    | 4    |
| 100  | E412     | A    | 4    |
| 200  | C123     | B    | 3    |
| 300  | C312     | A    | 1    |
| 300  | C324     | A    | 1    |
| 300  | C413     | A    | 1    |
| 400  | C312     | A    | 4    |
| 400  | C324     | A    | 4    |
| 400  | C413     | A    | 4    |
| 400  | E412     | A    | 4    |

이 테이블에서 **기본키(Primary Key)** 는 무엇일까요?

- `학번` 만으로는 기본키가 될 수 없습니다. 왜냐하면 100번, 300번, 400번 학생이 여러 행에 중복되어 나타나기 때문입니다.
- `과목번호` 만으로도 기본키가 될 수 없습니다. C413이라는 과목이 여러 학생에게 중복됩니다.
- 따라서 **`학번 + 과목번호`의 복합키(Composite Key)** 가 기본키가 됩니다.

:::info 복합키란?
복합키(Composite Key)는 두 개 이상의 컬럼을 합쳐서 하나의 기본키로 사용하는 방식입니다. 마치 이름과 생년월일을 함께 써야 특정 사람을 정확히 가리킬 수 있는 것처럼, 단일 컬럼으로는 행을 유일하게 식별할 수 없을 때 사용합니다.
:::

## ⚠️ 이상현상의 세 가지 종류

이상현상에는 크게 세 가지 종류가 있습니다. 각각 **삭제 이상**, **삽입 이상**, **갱신 이상**입니다. 하나씩 자세히 살펴보겠습니다.

### 🗑️ 삭제 이상 (Deletion Anomaly)

**삭제 이상**이란, 어떤 데이터를 삭제하려고 했는데 **지워서는 안 되는 정보까지 함께 사라져 버리는 현상**입니다. 이를 '연쇄 삭제'라고도 합니다.

실생활로 비유하면 이렇습니다. 학생이 특정 과목의 수강을 취소했습니다. 수강 취소는 "과목 정보만 없애는 것"이어야 합니다. 학생이 학교를 그만둔 것이 아니니까요. 그런데 테이블 구조 때문에 과목 정보를 지우면 그 학생의 학년 정보까지 함께 사라지는 문제가 발생합니다.

**예시:**
- 학번 200번 학생이 과목 `C123`을 수강 취소했습니다.
- 수강 테이블에서 해당 행(튜플 전체)을 삭제하면, `200번 학생이 3학년`이라는 정보도 함께 사라집니다.
- 그 학생은 단지 과목만 취소했을 뿐인데, 마치 학교를 그만둔 것처럼 학생 정보 자체가 사라져 버립니다.

```sql
-- ❌ 잘못된 설계: 수강 취소 시 학생 정보까지 삭제됨
-- 200번 학생의 C123 수강 취소를 위해 행 전체를 삭제
DELETE FROM 수강 WHERE 학번 = 200 AND 과목번호 = 'C123';
-- 결과: 200번 학생이 3학년이라는 정보도 함께 사라짐 → 삭제 이상 발생!

-- ✅ 올바른 설계: 학생 테이블과 수강 테이블을 분리
-- 학생 테이블에는 학생 고유 정보만 저장
DELETE FROM 수강 WHERE 학번 = 200 AND 과목번호 = 'C123';
-- 결과: 수강 정보만 삭제되고, 학생 테이블의 200번 학생 정보는 그대로 유지됨
```

:::warning 삭제 이상의 핵심
행(튜플) 단위로만 삭제할 수 있기 때문에, 행 안의 일부 데이터만 선택적으로 제거하는 것이 불가능합니다. 이 때문에 원치 않는 정보 손실(정보 손실)이 발생합니다.
:::

### ➕ 삽입 이상 (Insertion Anomaly)

**삽입 이상**이란, 새로운 데이터를 삽입하려고 할 때 **원하지 않는 불필요한 데이터까지 함께 입력해야만 삽입이 가능한 현상**입니다.

실생활로 비유하면, 새 학생이 입학해서 학생 정보를 등록하려는데 "수강 과목을 반드시 입력해야 한다"는 규칙 때문에 아직 수강 신청도 하지 않은 과목을 억지로 넣어야 하는 상황입니다.

**예시:**
- 학번 600번의 2학년 학생이 새로 입학했습니다.
- 이 학생의 정보를 수강 테이블에 입력하려고 합니다.
- 그런데 기본키가 `학번 + 과목번호` 복합키이기 때문에, **기본키는 NULL 값을 가질 수 없다**는 규칙에 의해 과목번호를 반드시 입력해야 합니다.
- 아직 수강 신청도 하지 않은 학생인데, 과목번호를 입력하지 않으면 학생 정보 자체를 등록할 수 없습니다.
- 결국 **가짜 값(스퓨리어스 튜플, Spurious Tuple)** 을 억지로 넣어야 합니다.

```sql
-- ❌ 잘못된 설계: 600번 학생(2학년) 정보 삽입 시 과목번호가 없어서 실패
INSERT INTO 수강 (학번, 과목번호, 성적, 학년)
VALUES (600, NULL, NULL, 2);
-- 결과: 기본키(학번 + 과목번호)에 NULL이 포함될 수 없어 오류 발생!

-- ❌ 임시방편으로 가짜 데이터 입력 (스퓨리어스 튜플)
INSERT INTO 수강 (학번, 과목번호, 성적, 학년)
VALUES (600, 'DUMMY', NULL, 2);
-- 결과: 데이터가 들어가긴 하지만 'DUMMY'라는 가짜 과목 정보가 생김 → 삽입 이상 발생!

-- ✅ 올바른 설계: 학생 테이블과 수강 테이블을 분리하면 문제없이 학생 정보 삽입 가능
INSERT INTO 학생 (학번, 학년) VALUES (600, 2);
-- 결과: 수강 신청 여부와 무관하게 학생 정보 등록 가능
```

:::danger 스퓨리어스 튜플(Spurious Tuple)
스퓨리어스 튜플이란 '위조 튜플', '가짜 튜플'이라고도 합니다. 정상적인 데이터를 삽입하기 위해 의미 없는 가짜 데이터를 억지로 넣는 것으로, 데이터의 무결성(정확성)을 심각하게 해칩니다. 절대로 이런 방식으로 데이터를 관리해서는 안 됩니다.
:::

### 🔄 갱신 이상 (Update Anomaly)

**갱신 이상**이란, 중복된 데이터 중 일부만 갱신(수정)되고 나머지는 수정되지 않아서 **데이터 간에 불일치(모순)가 발생하는 현상**입니다.

실생활로 비유하면, 회사 직원의 부서 이름이 여러 문서에 중복 기재되어 있는데, 부서 이름이 바뀌었을 때 일부 문서만 수정하고 나머지를 빠뜨린 상황과 같습니다. 어떤 문서에는 '영업팀', 다른 문서에는 '판매팀'이라고 나와 있어 혼란이 생기는 것이죠.

**예시:**
- 학번 400번 학생의 학년이 4학년으로 잘못 입력되어 3학년으로 수정해야 합니다.
- 수강 테이블을 보면 400번 학생이 4개의 과목을 수강하고 있어 학년 정보가 **4개 행에 중복 저장**되어 있습니다.
- 3학년으로 고치려면 4번 모두 수정해야 합니다.
- 만약 실수로 일부만 수정하면, 같은 400번 학생인데 어떤 행에는 '3학년', 다른 행에는 '4학년'이 남아있는 **정보 모순(데이터 불일치)** 이 발생합니다.

```sql
-- ❌ 잘못된 설계: 갱신 시 모든 중복 행을 수정해야 함
-- 400번 학생의 학년을 4학년에서 3학년으로 수정
UPDATE 수강 SET 학년 = 3 WHERE 학번 = 400;
-- 만약 일부만 수정하면 같은 학생의 학년 정보가 3학년과 4학년으로 뒤섞임 → 갱신 이상!

-- ✅ 올바른 설계: 학생 테이블과 수강 테이블을 분리하면 단 한 번의 수정으로 해결
UPDATE 학생 SET 학년 = 3 WHERE 학번 = 400;
-- 결과: 학생 테이블에서 한 번만 수정하면 모든 수강 정보에 자동으로 반영됨
```

| 이상현상 종류 | 발생 상황 | 결과 |
|---|---|---|
| **삭제 이상** | 데이터 삭제 시 | 원하지 않는 정보까지 함께 삭제됨 (정보 손실) |
| **삽입 이상** | 데이터 삽입 시 | 불필요한 가짜 데이터를 함께 입력해야 함 |
| **갱신 이상** | 데이터 수정 시 | 중복 데이터 중 일부만 수정되어 불일치 발생 |

## 🔍 함수적 종속성(Functional Dependency) 이란?

### 왜 함수적 종속성이 필요한가?

앞에서 살펴본 이상현상의 근본 원인은 **"서로 다른 개념의 데이터를 하나의 테이블에 무리하게 합쳐놓은 것"** 이었습니다. 그렇다면 어떤 기준으로 테이블을 나눠야 할까요?

바로 여기서 **함수적 종속성(Functional Dependency)** 이라는 개념이 등장합니다. 이것은 수학의 함수 이론에 근거한 개념으로, 쉽게 말하면 **"어떤 속성(컬럼)이 다른 속성을 결정하는 관계"** 를 의미합니다. 이 관계를 분석해서 테이블을 어떻게 나눌지 결정하는 것이 정규화의 출발점입니다.

### 함수적 종속성의 기본 개념

> 📐 **"X가 Y를 결정한다"** = X → Y = Y는 X에 함수적으로 종속된다

함수적 종속성에서는 두 가지 용어가 등장합니다.

- **결정자(Determinant)**: 다른 속성을 결정하는 쪽. 위 표현에서 `X`에 해당합니다.
- **종속자(Dependent)**: 결정되는 쪽. 위 표현에서 `Y`에 해당합니다.

**쉬운 비유:**
마치 **주민등록번호**와 같습니다. 주민등록번호(X)가 같으면 그 사람의 이름(Y)은 반드시 하나입니다. 주민등록번호 `900101-1234567`에 해당하는 이름이 '김철수'이기도 하고 '이영희'이기도 한 경우는 절대 있을 수 없습니다. 이처럼 **결정자 값이 같으면 종속자 값도 반드시 같아야** 합니다.

또 다른 예를 들어보겠습니다. 학번(X)은 학생의 이름, 주소, 연락처, 학년, 전공(Y) 등을 결정합니다. 학번이 `A100`인 학생의 이름은 오직 하나여야 합니다. 학번이 같은데 이름이 다른 두 사람이 존재하는 것은 불가능합니다.

```
학번(X) → 이름, 주소, 연락처, 학년, 전공(Y)
  └─ 학번이 같으면 → 이름도 반드시 같다 ✅
  └─ 학번이 같은데 → 이름이 다르다 ❌ (함수 종속 아님)
```

### 함수적 종속성의 판별 방법

함수적 종속성이 성립하는지 판별하는 방법은 다음과 같습니다.

> **두 개의 튜플(행)에서 X 값이 같다면, Y 값도 반드시 같아야 한다.**

```
테이블 예시:

A(결정자) | B(종속자)
----------|----------
   100    |    Kim       ← A=100 → B=Kim
   200    |    Lee
   100    |    Kim       ← A=100 → B=Kim (같은 A값에 같은 B값 ✅)

→ A가 B를 결정한다 (함수 종속 성립)

---

A(결정자) | B(?)
----------|-----
   100    |    Kim       ← A=100 → B=Kim
   200    |    Lee
   100    |    Park      ← A=100 → B=Park (같은 A값인데 B가 다름! ❌)

→ A가 B를 결정할 수 없다 (함수 종속 불성립)
```

### 수강 테이블의 함수적 종속 관계

앞서 살펴본 수강 테이블에서 속성들 간의 함수적 종속 관계를 분석해 보면 다음과 같습니다.

```
학번 + 과목번호 → 성적   (복합키가 성적을 결정)
학번           → 학년   (학번만으로 학년을 결정)
```

이것을 **함수 종속 다이어그램(Functional Dependency Diagram)** 으로 표현하면 속성들 간의 관계를 시각적으로 쉽게 파악할 수 있습니다.

:::tip 함수 종속 다이어그램
함수 종속 다이어그램은 테이블 내 속성들이 서로 어떤 결정 관계를 가지는지를 화살표(→)로 표현한 그림입니다. 이 다이어그램을 보면 어떤 속성이 결정자이고 어떤 속성이 종속자인지 한눈에 알 수 있습니다. 나중에 테이블을 어떻게 분리해야 하는지 결정할 때 매우 유용하게 활용됩니다.
:::

## 📐 암스트롱 공리(Armstrong's Axioms): 함수 종속성 추론 규칙

### 왜 추론 규칙이 필요한가?

실제 데이터베이스 설계에서는 직접 명시되지 않은 함수 종속 관계도 기존의 관계로부터 **논리적으로 새로운 종속 관계를 유추**해낼 필요가 있습니다. 이때 사용하는 것이 바로 **암스트롱 공리(Armstrong's Axioms)** 입니다. 마치 수학에서 공식을 이용해 새로운 결과를 유도하듯이, 알려진 함수 종속 관계로부터 새로운 관계를 도출하는 규칙입니다.

암스트롱 공리는 **기본 규칙 3가지**와 **추가 규칙 3가지**로 구성됩니다.

### 기본 규칙 3가지

#### 1️⃣ 재귀 규칙 (Reflexivity Rule)

**"X가 Y를 포함하면, X는 Y를 결정한다."**

가장 직관적인 규칙입니다. 자기 자신은 항상 자기 자신을 결정합니다. 예를 들어, 학번은 학번을 결정합니다(당연하죠!). 즉, 어떤 속성 집합은 자신의 부분집합을 항상 결정할 수 있습니다.

```
X → Y (X가 Y를 포함할 때)
예: {학번, 학년} → {학번}  ← 학번은 {학번, 학년} 안에 포함되므로 성립
```

#### 2️⃣ 증가 규칙 (Augmentation Rule)

**"X → Y 이면, WX → WY 이다."**

X가 Y를 결정한다면, 양쪽에 같은 속성 W를 추가해도 여전히 종속 관계가 성립합니다. 비유하자면, A와 B의 관계가 있을 때 양쪽에 똑같이 100을 더해도 두 값의 관계는 변하지 않는 것과 같습니다.

```
X → Y  이면
WX → WY 도 성립

예: 학번 → 학년  이면
    (학번, 과목번호) → (학년, 과목번호) 도 성립
```

#### 3️⃣ 이행 규칙 (Transitivity Rule)

**"X → Y 이고 Y → Z 이면, X → Z 이다."**

우리가 일상에서도 많이 쓰는 논리입니다. "나는 사람이고, 사람은 언젠가 죽는다. 따라서 나도 언젠가 죽는다."와 같은 삼단논법과 같습니다.

```
X → Y  이고
Y → Z  이면
X → Z  성립

예: 학번 → 주민번호  이고
    주민번호 → 이름  이면
    학번 → 이름  성립
```

### 추가 규칙 3가지

기본 규칙 3가지에서 파생된 추가 규칙들도 있습니다.

#### 4️⃣ 결합 규칙 (Union Rule)

**"X → Y 이고 X → Z 이면, X → YZ 이다."**

X가 Y도 결정하고 Z도 결정한다면, X는 Y와 Z를 동시에 결정할 수 있습니다.

```
학번 → 이름  이고
학번 → 연락처  이면
학번 → (이름, 연락처)  성립
```

#### 5️⃣ 분해 규칙 (Decomposition Rule)

**"X → YZ 이면, X → Y 이고 X → Z 이다."**

결합 규칙의 반대입니다. X가 Y와 Z를 동시에 결정한다면, 각각 따로따로도 결정할 수 있습니다.

```
학번 → (이름, 연락처)  이면
학번 → 이름  이고
학번 → 연락처  도 각각 성립
```

#### 6️⃣ 가이행 규칙 (Pseudotransitivity Rule)

**"W → X 이고 XY → Z 이면, WY → Z 이다."**

이 규칙은 증가 규칙과 이행 규칙을 결합한 것입니다.

```
W → X 이면, 양쪽에 Y를 추가(증가 규칙):
WY → XY

XY → Z 가 있으므로 이행 규칙 적용:
WY → Z 성립
```

| 규칙 이름 | 수식 | 설명 |
|---|---|---|
| **재귀 규칙** | Y ⊆ X이면 X → Y | 자신의 부분집합은 항상 결정 가능 |
| **증가 규칙** | X → Y이면 WX → WY | 양쪽에 같은 속성 추가해도 성립 |
| **이행 규칙** | X → Y, Y → Z이면 X → Z | 삼단논법과 같은 논리 |
| **결합 규칙** | X → Y, X → Z이면 X → YZ | 두 종속을 하나로 합침 |
| **분해 규칙** | X → YZ이면 X → Y, X → Z | 하나를 두 개로 분리 |
| **가이행 규칙** | W → X, XY → Z이면 WY → Z | 증가 + 이행의 결합 |

### 암스트롱 공리 활용 예시

다음과 같은 함수 종속 관계가 주어졌을 때, 새로운 종속 관계를 유추해 봅시다.

```
주어진 종속 관계:
A → B
B → H
C → BD
B → F
AF → Z
CG → I
```

**① A → H 를 유추할 수 있는가?**

```
A → B  (주어진 관계)
B → H  (주어진 관계)
∴ A → H  (이행 규칙 적용) ✅
```

**② CG → HI 를 유추할 수 있는가?**

```
C → BD  →  분해 규칙 적용  →  C → B, C → D
CG → I  (주어진 관계)

C → B 이고 B → H 이므로 이행 규칙:  C → H
∴ CG → H  (C가 H를 결정하므로, CG도 H를 결정 가능)
   CG → I  (주어진 관계)
결합 규칙: CG → HI ✅
```

**③ AG → I 를 유추할 수 있는가?**

```
A → C  (A → BD에서 이행 등으로 유추 가능하다고 가정)
CG → I  (주어진 관계)
∴ 가이행 규칙: AG → I ✅
  (A → C 이고 CG → I 이면, AG → I)
```

## 🧹 최소 커버(Minimal Cover): 중복 종속 관계 제거

### 최소 커버란 무엇인가?

함수 종속 관계를 모두 나열하다 보면 **이미 다른 종속 관계로부터 유추 가능한 중복된 종속 관계**가 포함되어 있는 경우가 있습니다. 이런 불필요한 중복 종속을 제거하고 **꼭 필요한 최소한의 함수 종속 집합만 남긴 것**을 **최소 커버(Minimal Cover)** 라고 합니다.

쉽게 비유하면, 서울에서 부산까지 가는 방법이 여러 가지 있을 때, 다른 경로들을 이미 알고 있다면 굳이 모든 경로를 기억하지 않아도 되는 것과 같습니다. 핵심적인 경로만 알고 있어도 나머지를 유추할 수 있습니다.

### 최소 커버를 구하는 규칙

최소 커버를 구하기 위해서는 세 가지 조건을 확인해야 합니다.

1. **오른쪽(종속자) 속성은 반드시 하나여야 한다.** 예: `X → AB` 형태라면 `X → A`, `X → B`로 분해합니다.
2. **왼쪽(결정자)에서 불필요한 속성을 제거할 수 있는지 확인한다.** `AB → C`를 `A → C` 또는 `B → C`로 단순화할 수 있는지 검토합니다.
3. **전체 집합에서 이행 종속 등으로 유추 가능한 중복 종속은 제거한다.**

### 최소 커버 예시

**주어진 함수 종속 집합:**
```
B → A
D → A
AB → D
```

**Step 1: 오른쪽 속성이 모두 하나인지 확인**

모두 오른쪽에 속성이 하나씩이므로 조건 1 만족 ✅

**Step 2: 왼쪽 결정자 단순화 가능 여부 확인**

`AB → D`에서 A를 제거해 `B → D`로 단순화할 수 있는지 확인합니다.

```
B → A  (주어진 관계)
양쪽에 B 증가:  BB → AB  →  B → AB (BB는 결국 B)
AB → D  (주어진 관계)
이행 규칙:  B → D  ✅

따라서 AB → D 는 B → D 로 대체 가능!
```

**변환 후 종속 집합:**
```
B → A
D → A
B → D
```

**Step 3: 이행 종속으로 유추 가능한 중복 제거**

```
B → D  이고
D → A  이면
이행 규칙:  B → A  ✅

B → A 는 B → D 와 D → A 로부터 유추 가능하므로 제거!
```

**최종 최소 커버:**
```
D → A
B → D
```

이 두 가지 종속 관계만으로 원래의 세 가지 종속 관계를 모두 표현할 수 있습니다.

:::tip 최소 커버의 의의
최소 커버를 구하는 것은 데이터베이스 정규화에서 중요한 단계입니다. 불필요한 중복 종속을 제거함으로써 테이블 분해 시 꼭 필요한 구조만 남길 수 있고, 이를 통해 이상현상을 최소화하면서도 데이터 중복을 줄일 수 있습니다.
:::

## 🔨 정규화(Normalization): 이상현상 해결을 위한 스키마 변환

### 정규화란 무엇인가?

**정규화(Normalization)** 란 **하나의 릴레이션(테이블)을 속성들 간의 함수적 종속 관계를 기준으로 여러 개의 릴레이션으로 분해하는 과정**입니다. 쉽게 말해, 뒤섞여 있는 정보를 종류별로 깔끔하게 분리해서 각각의 서랍에 담는 작업입니다.

이 과정에서 가장 중요한 원칙은 **"하나의 독립된 종속 관계는 하나의 릴레이션으로"** 입니다. 수강 테이블 예시에서 보면, 학생의 학번이 학년을 결정하는 관계와, 학번+과목번호가 성적을 결정하는 관계는 서로 다른 독립적인 종속 관계입니다. 따라서 이 둘을 별도의 테이블로 분리해야 합니다.

```sql
-- ❌ 잘못된 설계: 학생 정보와 수강 정보를 하나의 테이블에 합침
CREATE TABLE 수강 (
    학번      INT,       -- 학생을 식별하는 번호
    과목번호  VARCHAR(10), -- 수강 과목 번호
    성적      CHAR(2),   -- 해당 과목의 성적
    학년      INT,       -- 학생의 학년 (학생 고유 정보)
    PRIMARY KEY (학번, 과목번호)  -- 복합키를 기본키로 사용
);
-- 문제: 학년은 학번에만 종속되는데, 과목 수강 정보와 함께 저장되어 중복 발생

-- ✅ 올바른 설계: 학생 테이블과 수강 테이블로 분리
CREATE TABLE 학생 (
    학번  INT PRIMARY KEY,  -- 학생 고유 식별자
    학년  INT               -- 학번에만 종속되는 학년 정보
);

CREATE TABLE 수강 (
    학번      INT,           -- 학생 테이블과 연결하는 외래키
    과목번호  VARCHAR(10),   -- 수강 과목 번호
    성적      CHAR(2),       -- 학번+과목번호에 종속되는 성적 정보
    PRIMARY KEY (학번, 과목번호),    -- 복합키 기본키
    FOREIGN KEY (학번) REFERENCES 학생(학번)  -- 외래키로 학생 테이블 참조
);
```

### 스키마 변환(정규화)의 세 가지 원칙

정규화를 수행할 때 반드시 지켜야 할 세 가지 원칙이 있습니다.

#### ① 정보 표현의 무손실 (Lossless Decomposition)

하나의 테이블을 여러 개로 분해하더라도 **원래 있던 정보가 단 하나도 사라져서는 안 됩니다.** 분해 전에 존재하던 모든 데이터는 분해 후에도 어딘가에 완전히 보존되어 있어야 합니다. 마치 종이를 가위로 자를 때 조각들을 다시 합치면 원래 종이가 되어야 하는 것처럼요.

#### ② 최소한의 중복만 허용

중복을 100% 제거하는 것은 현실적으로 불가능합니다. 테이블을 연결하기 위한 **외래키(Foreign Key)** 는 어쩔 수 없이 중복될 수밖에 없습니다. 중요한 것은 **불필요한 중복을 제거**하고 **꼭 필요한 최소한의 중복만 허용**하는 것입니다. 중복이 많을수록 이상현상이 많이 발생하기 때문입니다.

#### ③ 분리의 원칙 (Separation Principle)

**하나의 독립된 관계는 별도의 릴레이션으로 분리**해서 표현해야 합니다. 학생 정보와 수강 정보는 서로 독립된 관계이므로, 이 둘은 별개의 테이블로 분리되어야 합니다. 이를 통해 **종속성이 보존**되고, 각 테이블 내에서는 하나의 명확한 종속 관계만 존재하게 됩니다.

### 데이터베이스 설계 가이드라인

이상현상을 방지하기 위한 데이터베이스 설계 가이드라인을 정리하면 다음과 같습니다.

1. **서로 다른 개체 타입을 하나의 테이블에 합치지 마라.** 학생, 교수, 과목은 각각 별도의 테이블로 관리합니다.
2. **중복이나 갱신 이상현상이 발생하지 않도록 설계하라.** 함수 종속 관계를 분석해서 적절히 테이블을 분리합니다.
3. **NULL 값이 가능하면 발생하지 않도록 설계하라.** NULL이 많아지면 데이터 조회 시 오류나 예외 처리가 복잡해집니다.
4. **스퓨리어스 튜플(가짜 데이터)이 발생하지 않도록 설계하라.** 정상적인 데이터 삽입을 위해 가짜 데이터를 넣어야 하는 구조를 피합니다.

### 정규화의 단계

정규화에는 여러 단계가 있습니다.

| 정규형 단계 | 명칭 | 설명 |
|---|---|---|
| **1NF** | 제1정규형 | 모든 속성값이 원자값(더 이상 나눌 수 없는 단일 값)을 가져야 함 |
| **2NF** | 제2정규형 | 부분 함수 종속 제거 |
| **3NF** | 제3정규형 | 이행 함수 종속 제거 |
| **BCNF** | 보이스-코드 정규형 | 모든 결정자가 후보키여야 함 |
| **4NF** | 제4정규형 | 다치 종속 제거 |
| **5NF** | 제5정규형 | 조인 종속 제거 |

:::info 실무에서는 어디까지 정규화할까?
실제 현장에서는 주로 **3정규형(3NF)** 또는 **BCNF** 수준까지 정규화를 수행합니다. 정규화 단계가 높아질수록 이상현상은 완벽하게 제거되지만, 테이블이 많아져서 조인 연산이 증가하고 성능이 저하됩니다. 반대로 정규화를 너무 적게 하면 이상현상이 남아있게 됩니다. 실무에서는 이 **성능과 무결성 사이의 균형점**이 3NF~BCNF 수준입니다.
:::

### 정규화와 역정규화의 관계

정규화가 진행되는 설계 단계와 그 목적에 대해 정리하면 다음과 같습니다.

```
데이터베이스 설계 단계:
┌─────────────┐
│  개념적 설계  │  ← 엔티티, 관계 등 개념 수준 설계
└──────┬──────┘
       │
┌──────▼──────┐
│  논리적 설계  │  ← ✅ 정규화(Normalization) 수행 단계
└──────┬──────┘       이상현상 제거, 함수 종속 분석
       │
┌──────▼──────┐
│  물리적 설계  │  ← 역정규화(Denormalization) 수행 단계
└─────────────┘       성능 최적화를 위해 일부 테이블 다시 합침
```

- **정규화**는 **논리적 설계 단계**에서 수행합니다. 목적은 **이상현상 제거와 데이터 무결성 보장**입니다.
- **역정규화(Denormalization)** 는 **물리적 설계 단계**에서 수행합니다. 정규화로 테이블이 너무 많아져 성능이 떨어질 때, 의도적으로 일부 테이블을 다시 합치는 작업입니다.

:::warning 정규화는 성능 향상이 목적이 아닙니다!
정규화를 수행하면 테이블이 여러 개로 나뉘기 때문에 원하는 데이터를 얻기 위해 **조인(JOIN) 연산**을 많이 사용해야 합니다. 조인 연산이 많아질수록 **성능은 오히려 저하**될 수 있습니다. 정규화의 목적은 순수하게 **이상현상을 없애고 데이터 무결성을 유지**하는 것입니다. 성능 최적화는 역정규화 단계에서 별도로 고려합니다.
:::

## ⚡ 주의사항 및 실전 팁

### 이상현상을 막기 위한 설계 원칙 요약

데이터베이스를 처음 설계할 때부터 이상현상을 예방하는 것이 사후에 고치는 것보다 훨씬 효율적입니다. 아래의 체크리스트를 설계 단계에서 항상 확인하세요.

```
✅ 체크리스트:
□ 서로 다른 개념의 데이터(학생 정보 / 수강 정보)를 하나의 테이블에 합치지 않았는가?
□ 함수 종속 관계를 분석해서 각 테이블의 기본키와 속성 간의 관계가 명확한가?
□ 기본키에 NULL 값이 들어올 가능성이 없는가?
□ 데이터를 삽입할 때 가짜(더미) 데이터를 입력해야 하는 상황이 생기지는 않는가?
□ 하나의 데이터를 수정할 때 여러 행을 동시에 수정해야 하는 중복이 없는가?
□ 데이터 삭제 시 원치 않는 정보가 함께 사라지지 않는가?
```

### 외래키를 통한 테이블 연결

정규화로 테이블을 분리한 후에는 **외래키(Foreign Key)** 를 통해 테이블들을 연결합니다. 이것이 앞서 언급한 "꼭 필요한 최소한의 중복"입니다.

```sql
-- 학생 테이블 (학번이 학년을 결정하는 종속 관계)
CREATE TABLE 학생 (
    학번  INT PRIMARY KEY,  -- 기본키: 학생 고유 식별자
    학년  INT NOT NULL      -- 학번에 함수 종속된 속성
);

-- 수강 테이블 (학번+과목번호가 성적을 결정하는 종속 관계)
CREATE TABLE 수강 (
    학번      INT NOT NULL,           -- 외래키: 학생 테이블 참조 (최소한의 중복 허용)
    과목번호  VARCHAR(10) NOT NULL,   -- 과목 식별자
    성적      CHAR(2),                -- 학번+과목번호에 함수 종속된 속성
    PRIMARY KEY (학번, 과목번호),     -- 복합 기본키
    FOREIGN KEY (학번) REFERENCES 학생(학번)  -- 학생 테이블과 연결
);

-- ✅ 이제 이상현상 없이 정상 동작:
-- 삭제 이상 해결: 수강 취소해도 학생 테이블의 학년 정보 보존
-- 삽입 이상 해결: 수강 신청 전에도 학생 테이블에 학생 정보 삽입 가능
-- 갱신 이상 해결: 학년 변경 시 학생 테이블에서 한 번만 수정하면 됨
```

:::note 함수 종속성 분석의 순서
1. 테이블의 모든 속성(컬럼)을 나열합니다.
2. 각 속성 간의 결정 관계를 화살표(→)로 표시합니다.
3. 암스트롱 공리를 이용해 숨겨진 종속 관계를 추가로 유추합니다.
4. 최소 커버를 구해 불필요한 중복 종속을 제거합니다.
5. 각 종속 관계를 기준으로 별도의 테이블을 생성합니다.
:::

## 📝 핵심 정리

이 문서에서 배운 핵심 개념을 정리합니다.

- **이상현상(Anomaly)** 은 잘못 설계된 테이블 구조로 인해 데이터 삽입·삭제·갱신 시 발생하는 비정상적인 부작용이다.
- **삭제 이상**: 데이터 삭제 시 원하지 않는 정보까지 함께 사라지는 현상(연쇄 삭제).
- **삽입 이상**: 데이터 삽입 시 불필요한 가짜 데이터(스퓨리어스 튜플)까지 입력해야 하는 현상.
- **갱신 이상**: 중복 데이터 중 일부만 수정되어 데이터 불일치(모순)가 발생하는 현상.
- **이상현상의 원인**은 서로 다른 개념의 데이터를 하나의 테이블에 무리하게 합쳐놓아 불필요한 중복이 생기는 것이다.
- **함수적 종속성(X → Y)** 은 결정자(X)의 값이 같으면 종속자(Y)의 값도 반드시 같아야 한다는 관계를 의미한다.
- **결정자**는 다른 속성을 결정하는 속성, **종속자**는 결정되는 속성이다.
- **암스트롱 공리**는 기존 함수 종속 관계로부터 새로운 종속 관계를 추론하는 규칙(재귀, 증가, 이행, 결합, 분해, 가이행).
- **최소 커버**는 중복된 함수 종속 관계를 제거하여 최소한의 필요한 종속 집합만 남긴 것이다.
- **정규화(Normalization)** 는 함수 종속 관계를 기준으로 하나의 테이블을 여러 테이블로 분해하여 이상현상을 제거하는 과정이다.
- 정규화의 세 가지 원칙은 **정보 무손실**, **최소한의 중복 허용**, **분리의 원칙**이다.
- 정규화는 **논리적 설계 단계**에서 수행하며, 이상현상 제거가 목적이지 성능 향상이 목적이 아니다.
- **역정규화(Denormalization)** 는 성능 최적화를 위해 물리적 설계 단계에서 정규화된 테이블을 다시 합치는 과정이다.
- 실무에서는 주로 **3정규형(3NF) 또는 BCNF** 수준까지 정규화를 수행한다.
