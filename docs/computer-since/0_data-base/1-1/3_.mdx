# 데이터베이스 구조

데이터베이스를 구성하는 핵심 요소인 개체(Entity), 속성(Attribute), 관계(Relationship)와 논리적/물리적 구조를 학습합니다.

---

## 1. 개체 (Entity)

### 개념
데이터베이스에서 관리하고자 하는 정보의 대상이 되는 실체입니다.

### 개체의 예시

**대학교 시스템:**
```
학생, 교수, 과목, 학과, 강의실
```

**쇼핑몰 시스템:**
```
고객, 상품, 주문, 카테고리, 배송
```

**병원 시스템:**
```
환자, 의사, 진료, 처방, 병실
```

### 개체의 특징
- 독립적으로 존재할 수 있는 대상
- 다른 개체와 구별 가능
- 여러 개의 속성으로 구성
- 테이블로 표현됨

---

## 2. 속성 (Attribute)

### 개념
개체가 가지고 있는 특성이나 성질을 나타내는 항목입니다.

### 속성의 예시

**학생 개체의 속성:**
```
학번, 이름, 학과, 학년, 생년월일, 주소, 전화번호, 이메일
```

**상품 개체의 속성:**
```
상품코드, 상품명, 가격, 재고수량, 제조사, 카테고리
```

### 속성의 구분

**키 속성 (Key Attribute):**
- 개체를 유일하게 식별하는 속성
- 예: 학번, 주민번호, 상품코드

**일반 속성 (Non-Key Attribute):**
- 개체의 특성을 나타내는 속성
- 예: 이름, 주소, 가격

### 테이블 표현

```sql
학생 테이블
┌────────┬──────┬────────┬──────┬────────────┐
│ 학번*  │ 이름 │ 학과   │ 학년 │ 전화번호   │
├────────┼──────┼────────┼──────┼────────────┤
│2024001 │홍길동│컴퓨터  │  3   │010-1234-5678│
│2024002 │김철수│전자    │  2   │010-2345-6789│
│2024003 │이영희│기계    │  4   │010-3456-7890│
└────────┴──────┴────────┴──────┴────────────┘
* = 기본키 (Primary Key)
```

## 3. 관계 (Relationship)

### 개념
개체와 개체 간의 연관성을 나타냅니다.

### 관계의 종류

**1) 개체 간 관계**

```
학생 ─(수강)─ 과목
교수 ─(담당)─ 과목
학생 ─(소속)─ 학과
```

**2) 속성 간 관계 (주종 관계)**

```
학생 테이블
├─ 학번 (주인 - Primary Key)
│   ├─ 이름 (종속)
│   ├─ 학과 (종속)
│   └─ 학년 (종속)
```

### 관계의 표현

**ER 다이어그램 표기:**
```
┌────────┐         ┌────────┐
│  학생  │─────지도─────│  교수  │
└────────┘         └────────┘
```

**관계형 모델:**
```sql
지도 관계 테이블
┌────────┬──────────┬──────────┐
│ 학번   │ 교수번호 │ 지도시작일│
├────────┼──────────┼──────────┤
│2024001 │   P001   │2024-03-01│
│2024002 │   P002   │2024-03-01│
└────────┴──────────┴──────────┘
```

## 4. 기본키 (Primary Key)

### 개념
개체를 유일하게 식별할 수 있는 속성 또는 속성의 집합입니다.

### 기본키의 특징

**유일성 (Uniqueness):**
- 모든 레코드에서 값이 중복되지 않음
- 예: 학번, 주민번호, 상품코드

**최소성 (Minimality):**
- 필요 최소한의 속성으로 구성
- 예: (학번) ○, (학번+이름) ✗

**NOT NULL:**
- 반드시 값이 존재해야 함
- NULL 값 허용 안 됨

### 기본키 예시

```sql
-- 학생 테이블
학번 → 기본키 (전 세계에서 유일)

-- 국민 정보
주민번호 → 기본키 (전 국민 중 유일)

-- 주문 테이블
주문번호 → 기본키 (모든 주문 중 유일)
```

### 복합키 (Composite Key)

```sql
수강 테이블
┌────────┬──────────┬────────┐
│ 학번*  │ 과목코드* │ 성적   │
├────────┼──────────┼────────┤
│2024001 │  CS101   │  A+    │
│2024001 │  CS102   │  B+    │
│2024002 │  CS101   │  A0    │
└────────┴──────────┴────────┘
기본키 = (학번 + 과목코드)
```

## 5. 개체 타입과 개체 집합

### 용어 정리

**개체 타입 (Entity Type):**
- 데이터베이스의 **구조** (틀, 스키마)
- 속성들의 정의

**개체 집합 (Entity Set):**
- 실제 저장된 **값** (데이터, 인스턴스)
- 레코드들의 모음

### 비교 예시

```sql
개체 타입 (구조):
학생(학번, 이름, 학과, 학년)

개체 집합 (값):
(2024001, 홍길동, 컴퓨터, 3)
(2024002, 김철수, 전자, 2)
(2024003, 이영희, 기계, 4)
```

### 동일 개념의 다양한 용어

| 구조 (Schema) | 값 (Instance) |
|---------------|---------------|
| 개체 타입 | 개체 집합 |
| 스키마 (Schema) | 인스턴스 (Instance) |
| 내포 (Intension) | 외연 (Extension) |
| 래퍼 (Wrapper) | 튜플 집합 (Tuple Set) |

## 6. 논리적 구조 vs 물리적 구조

### 개념

**논리적 구조:**
- 사용자가 보는 데이터의 관점
- 테이블, 뷰, 관계로 표현

**물리적 구조:**
- 저장 장치(디스크)가 보는 데이터의 관점
- 파일, 블록, 페이지로 저장

### 구조 비교

```
논리적 구조 (사용자 관점)
┌──────────────────┐
│  학생 테이블     │
│ ┌────┬────┬────┐│
│ │학번│이름│학과││
│ ├────┼────┼────┤│
│ │001 │홍길동│컴공││
│ └────┴────┴────┘│
└──────────────────┘
         ↕
   논리-물리 매핑
         ↕
물리적 구조 (디스크 관점)
┌──────────────────┐
│ 하드디스크       │
│ ┌──────────────┐ │
│ │ 데이터 블록  │ │
│ │ 001홍길동컴공│ │
│ └──────────────┘ │
└──────────────────┘
```

### 특징

**1대1 대응 관계:**
- 하나의 논리적 레코드 ↔ 하나의 물리적 레코드
- 같은 데이터, 다른 관점

**데이터 독립성:**
- 물리적 구조 변경 시 논리적 구조 영향 없음
- 사용자는 물리적 구조를 몰라도 됨

### 레코드 종류

**논리적 레코드:**
- 사용자가 다루는 데이터 단위
- 테이블의 한 행 (Row)

**물리적 레코드 (저장 레코드):**
- 디스크에 실제 저장되는 데이터 단위
- 블록, 페이지 단위로 관리

## 7. 관계의 주종 개념

### 속성 간 주종 관계

**기본키 = 주인 (Master)**
- 다른 속성들을 결정함
- 예: 학번이 결정되면 이름, 학과, 학년이 결정됨

**일반 속성 = 종속 (Dependent)**
- 기본키에 의해 결정됨
- 예: 이름, 학과, 학년은 학번에 종속

### 함수적 종속성

```
학번 → 이름
학번 → 학과
학번 → 학년

즉, 학번이 결정되면 나머지가 자동으로 결정됨
```

### 예시

```sql
학생(학번, 이름, 학과, 학년)

학번이 2024001이면
  → 이름은 "홍길동"
  → 학과는 "컴퓨터"
  → 학년은 "3"

학번 하나로 모든 속성이 결정됨!
```

## 8. 개체-관계 모델 (ER Model)

### ER 다이어그램 기호

**개체 (Entity):**
```
┌────────────┐
│   학생     │
└────────────┘
```

**속성 (Attribute):**
```
    학번
     ↓
┌────────────┐
│   학생     │
└────────────┘
    ↓
   이름
```

**관계 (Relationship):**
```
┌────────┐         ┌────────┐
│  학생  │────┬────│  과목  │
└────────┘  수강   └────────┘
```

### 관계의 카디널리티

**1:1 관계 (일대일):**
```
학생 ─── 학생증
(한 학생은 하나의 학생증만 가짐)
```

**1:N 관계 (일대다):**
```
학과 ──< 학생
(한 학과에 여러 학생이 소속)
```

**M:N 관계 (다대다):**
```
학생 >──< 과목
(한 학생이 여러 과목 수강, 한 과목에 여러 학생)
```

## 9. 실전 예제

### 예제 1: 대학교 데이터베이스

**개체:**
```
학생, 교수, 과목, 학과, 강의실
```

**속성:**
```
학생: 학번, 이름, 학과, 학년, 주소
교수: 교수번호, 이름, 학과, 직급
과목: 과목코드, 과목명, 학점, 시간
```

**관계:**
```
학생 ─(수강)─ 과목
교수 ─(강의)─ 과목
학생 ─(소속)─ 학과
교수 ─(소속)─ 학과
```

**기본키:**
```
학생: 학번
교수: 교수번호
과목: 과목코드
학과: 학과코드
```

### 예제 2: 온라인 쇼핑몰

**개체:**
```
고객, 상품, 주문, 카테고리, 배송
```

**속성:**
```
고객: 고객ID, 이름, 이메일, 전화번호, 주소
상품: 상품코드, 상품명, 가격, 재고, 카테고리
주문: 주문번호, 고객ID, 주문일, 총금액
```

**관계:**
```
고객 ─(주문)─ 상품
상품 ─(속함)─ 카테고리
주문 ─(포함)─ 배송
```

**기본키:**
```
고객: 고객ID
상품: 상품코드
주문: 주문번호
```

## 핵심 정리

### 데이터베이스 3대 구성 요소

```
1. 개체 (Entity)      - "무엇을" 저장할 것인가
2. 속성 (Attribute)   - "어떤 정보를" 저장할 것인가
3. 관계 (Relationship) - "어떻게 연결되어" 있는가
```

### 구조의 이중성

```
논리적 구조 (사용자) ↔ 물리적 구조 (시스템)
개체 타입 (구조)     ↔ 개체 집합 (값)
스키마 (Schema)      ↔ 인스턴스 (Instance)
```

### 기본키의 3대 조건

```
1. 유일성 (Unique)     - 중복 불가
2. 최소성 (Minimal)    - 최소 속성으로 구성
3. NOT NULL           - 반드시 값 존재
```

### 관계의 이해

```
개체 간 관계: 학생 ─(수강)─ 과목
속성 간 관계: 학번 → 이름, 학과, 학년
```

### 암기 팁

```
개속관 (개체-속성-관계)
논물 (논리적-물리적)
타집 (타입-집합)
```

이 구조를 이해하면 데이터베이스 설계의 기본을 마스터한 것입니다!
