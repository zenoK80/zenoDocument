# 성능 평가 및 품질 관리

## 📊 성능 평가의 필요성

```
"성능이 좋다" vs "성능이 나쁘다"

이는 주관적 판단입니다.
객관적 기준이 필요합니다!

그 기준이 바로 TPC(표준)입니다.
```

---

## 🎯 TPC (Transaction Processing Performance Council)

국제적으로 인정된 성능 평가 표준입니다.

### 기본 개념

```
TPC 점수 = 1초에 처리하는 트랜잭션 개수

예:
- TPS(Transactions Per Second) = 100
  → 1초에 100개 트랜잭션 처리

- TPS = 1000
  → 1초에 1000개 트랜잭션 처리
  → 더 높을수록 성능 좋음
```

### Price/TPS (가격 대비 성능)

```
실제 비즈니스에서는 성능 자체보다
"비용 대비 성능"이 중요합니다.

CPU 가격: 25만원
- TPS 100 기준: 2,500원/TPS
- TPS 200 기준: 1,250원/TPS

→ TPS가 높을수록 가성비 우수
```

---

## 📋 TPC 종류

### TPC-A
```
은행 입출금 시스템 평가
- ATM 시뮬레이션
- 입출금 트랜잭션
- 높은 안전성 요구
```

### TPC-B
```
운영체제(OS) 성능 평가
- 데이터 갱신 작업
- 로깅, 저널링
- 시스템 신뢰성 중시
```

### TPC-C ⭐ (가장 널리 사용)
```
전자상거래 시스템 평가
- 주문 처리
- 상품 검색
- 결제 처리
- 일반 비즈니스에 가장 적합
```

### TPC-D
```
의사결정 시스템(OLAP) 평가
- 복잡한 쿼리
- 데이터 분석
- 보고서 생성
- 대용량 데이터 처리
```

### TPC-H
```
대규모 데이터 분석 평가
- 예측 불가능한 쿼리
- 다양한 작업 패턴
- 사용자 행동 불확실성
- 복잡한 조인과 집계
```

### TPC-R
```
보고서 생성 평가
- 레포트 작성
- 요약 데이터 생성
- 이미지화
- 통계 계산
```

### TPC-W
```
웹 E-Commerce 시스템 평가
- 인터넷 환경
- 웹 브라우징
- 온라인 쇼핑
- 네트워크 지연 포함
```

---

## 🎓 평가 기준 선택

```
시스템 유형에 따라 다른 TPC 사용:

은행 시스템 → TPC-A
----
의사결정 시스템 → TPC-D
----
온라인 쇼핑몰 → TPC-C
----
웹 서비스 → TPC-W
----
데이터 분석 시스템 → TPC-H
```

## 📌 품질 관리 (Data Quality Management)

좋은 성능만으로는 부족합니다. 품질도 중요합니다.

```
성능 높음 + 품질 낮음 = 쓸모없는 시스템

예: 슈퍼카
- 성능: 시속 600km
- 품질: 자주 고장
→ 실제로는 사용 불가능
```

### 품질의 정의

```
품질 = 사용자를 만족시키는 수준

높은 품질:
+ 정확한 데이터
+ 신뢰할 수 있는 시스템
+ 안정적인 서비스
+ 예측 가능한 동작
```

---

## 🔍 3가지 품질 진단

### 1) 값 진단 (Value Diagnosis)

데이터 자체의 정확성을 확인합니다.

```
확인 대상:
- 테이블: 구조 문제
- 속성: 데이터 타입
- 코드: 코드값 유효성
- 관계: 외래키 무결성
- 비즈니스 규칙: 규칙 준수

예시:
┌───────────────────────┐
│ 값 검증               │
├───────────────────────┤
│ ✓ 학번은 6자리 숫자? │
│ ✓ 성별은 남/여 만?   │
│ ✓ 점수는 0-100?      │
│ ✓ NULL 금지 컬럼?    │
└───────────────────────┘
```

---

### 2) 구조 진단 (Structure Diagnosis)

데이터베이스 설계의 정확성을 확인합니다.

```
확인 대상:
- 정규화 수준
- 스키마 설계
- 테이블 관계
- 키 설정

리버스 모델링:
완성된 DB ← (역분석) ← ER 다이어그램 복원

예시: 수입차 역분석
완성된 자동차 → 분해 → 설계도 재구성
```

---

### 3) 프로세스 진단 (Process Diagnosis)

데이터 관리 전체 과정이 체계적인지 확인합니다.

```
확인 대상:
- 데이터 수집 절차
- 처리 규칙 준수
- 갱신 주기
- 검증 절차
- 모니터링 방식

점검:
┌────────────────────────┐
│ 1. 데이터 수집 프로세스│
│ 2. 데이터 처리 절차    │
│ 3. 데이터 저장 방식    │
│ 4. 데이터 검증 규칙    │
│ 5. 지속적 모니터링     │
└────────────────────────┘
```

```
Step 1: 측정

```

## 🎯 효과적인 품질 관리

### 전사적 접근

```
품질 관리는 IT부서만의 책임이 아닙니다.

조직 전체 참여:
┌───────────────────────────────┐
│ 데이터 수집 부서              │
│ (정확한 입력)                  │
│         ↓                      │
│ 처리 부서                      │
│ (올바른 처리)                  │
│         ↓                      │
│ IT부서                        │
│ (시스템 운영)                  │
│         ↓                      │
│ 사용자 부서                    │
│ (피드백)                       │
└───────────────────────────────┘
```

### 지속적 개선

```
음식 요리에 비유:
1. 조리 중 맛 확인
2. 중간에 맛 조정
3. 완성 후 재점검
4. 먹는 사람 피드백 반영

DB도 마찬가지:
1. 데이터 입력 시 검증
2. 처리 중 모니터링
3. 저장 후 확인
4. 사용자 피드백 반영
```

## 📊 성능과 품질의 균형

```
선택이 아니라 동시 달성이 필요합니다.

┌──────────────────────────────┐
│ 성능 좋음 + 품질 높음        │
│ = 최고의 시스템              │
├──────────────────────────────┤
│ 성능 좋음 + 품질 낮음        │
│ = 쓸 수 없는 시스템          │
├──────────────────────────────┤
│ 성능 낮음 + 품질 높음        │
│ = 느리지만 믿을 수 있는 시스템
├──────────────────────────────┤
│ 성능 낮음 + 품질 낮음        │
│ = 최악의 시스템              │
└──────────────────────────────┘
```

## 💡 실무 적용

### 성능 개선 프로세스

```
1. 현재 성능 측정 (TPC 기준)
   ↓
2. 병목 지점 파악
   ↓
3. 원인 분석
   - 하드웨어? 설계? SQL?
   ↓
4. 개선 방안 수립
   - 우선순위 결정
   ↓
5. 개선 실행
   - 단계적 적용
   ↓
6. 효과 검증
   - 성능 재측정
   ↓
7. 지속적 모니터링
```

### 품질 관리 활동

```
매일:
- 에러 로그 확인
- 이상 데이터 점검

주별:
- 데이터 정합성 확인
- 통계 비교

월별:
- 전체 품질 평가
- 개선안 수립

분기별:
- 프로세스 검토
- 시스템 최적화
```
