# 🗂️ 애트리뷰트와 도메인의 정의 및 역할

관계형 데이터베이스를 처음 공부하다 보면 "애트리뷰트", "도메인", "릴레이션", "튜플"과 같은 낯선 용어들이 쏟아져 나옵니다. 처음엔 영어 단어도 생소하고, 테이블이라고 부르면 될 것 같은데 왜 굳이 릴레이션이라고 부르는지 헷갈리기도 하죠. 이 문서에서는 그 혼란을 완전히 해소할 수 있도록, 관계 데이터의 기본 구성 요소인 **애트리뷰트(속성)** 와 **도메인(값의 범위)** 을 아주 쉽고 자세하게 설명합니다. 비유와 실제 예시를 가득 담았으니, 천천히 따라오시면 됩니다.

---

## 🏗️ 관계 데이터와 릴레이션이란 무엇인가

### 관계 데이터의 기본 개념

**관계 데이터(Relational Data)** 란, 데이터를 **테이블 형태로 표현하고, 테이블과 테이블 사이의 관계를 맺어 정보를 관리하는 방식**을 말합니다. 마치 엑셀 스프레드시트처럼 행과 열로 이루어진 표를 떠올리시면 됩니다. 그런데 단순히 표 하나만 있는 것이 아니라, 여러 표들이 서로 연결되어 있다는 점이 핵심입니다.

예를 들어, 아래처럼 학생 테이블과 수강 테이블이 있다고 생각해보겠습니다.

| 학번 (기본키) | 이름 | 학년 |
|---|---|---|
| 100 | 홍길동 | 1 |
| 200 | 김철수 | 2 |
| 300 | 이영희 | 3 |

| 수강번호 | 과목명 | 학번 (외래키) |
|---|---|---|
| A001 | 데이터베이스 | 100 |
| A002 | 알고리즘 | 200 |
| A003 | 운영체제 | 100 |

위 두 테이블에서 **학번 100** 이라는 값이 양쪽 테이블에 모두 존재합니다. 학생 테이블의 학번 100은 **기본키(Primary Key)** 이고, 수강 테이블의 학번 100은 **외래키(Foreign Key)** 입니다. 이 두 값이 연결 고리 역할을 해서 "홍길동 학생이 데이터베이스와 운영체제를 수강한다"는 정보를 표현할 수 있게 됩니다.

> 💡 **핵심 요약:** 관계 데이터는 **기본키와 외래키를 통해 테이블과 테이블이 연결**되는 구조입니다.

### 테이블과 릴레이션의 차이

같은 것을 두 가지 이름으로 부르는 이유가 뭘까요? 바로 **사용 맥락이 다르기 때문**입니다.

| 구분 | 테이블 (Table) | 릴레이션 (Relation) |
|---|---|---|
| 사용 맥락 | 실무, 현장에서 사용 | 이론, 학문적 맥락에서 사용 |
| 행(Row) | 행 (Row) 또는 레코드 | **튜플 (Tuple)** |
| 열(Column) | 열 (Column) | **애트리뷰트 (Attribute)** |
| 개념의 성격 | 구체적, 실용적 | 수학적, 추상적 |
| 중복 허용 | 같은 행 중복 가능 | 튜플 중복 불가 (유일성) |
| 순서 | 열 순서 존재, 행 정렬 가능 | 튜플·애트리뷰트 모두 순서 없음 |

쉽게 비유하자면, **릴레이션은 설계 도면**이고 **테이블은 실제로 지어진 건물**이라고 생각하면 됩니다. 설계 도면(릴레이션)에는 수학적으로 엄격한 규칙이 적용되지만, 실제 건물(테이블)은 현실의 필요에 따라 조금 더 유연하게 활용됩니다.

:::info 용어 정리
데이터베이스 이론 공부에서는 **릴레이션, 튜플, 애트리뷰트**라는 용어를 씁니다.
실무(SQL 작성 등)에서는 **테이블, 행(레코드), 열(컬럼)** 이라는 용어를 씁니다.
두 용어 세트가 서로 대응된다는 것을 기억해 두세요!
:::

### 릴레이션의 결과는 레코드 집합

릴레이션의 또 다른 중요한 특징은, **연산의 결과가 항상 레코드 집합(테이블 형태)으로 나온다**는 것입니다. 예를 들어 "점수가 80점 이상인 학생들의 학번과 이름을 출력해라"라는 명령을 내리면, 결과가 단순히 텍스트 한 줄로 나오는 게 아니라 아래처럼 **또 하나의 테이블 형태**로 반환됩니다.

```sql
-- 점수가 80점 이상인 학생의 학번과 이름을 조회하는 SQL 예시
SELECT 학번, 이름
FROM 학생
WHERE 점수 >= 80;
```

| 학번 | 이름 |
|---|---|
| 100 | 홍길동 |
| 300 | 이영희 |

이처럼 입력도 테이블, 출력도 테이블인 것이 관계형 데이터베이스의 핵심 특성입니다.

## 🔑 애트리뷰트(Attribute, 속성)란 무엇인가

### 왜 애트리뷰트라는 개념이 필요한가

데이터베이스에서 어떤 정보를 저장할 때, 우리는 그 정보를 구성하는 **가장 기본적인 단위**가 무엇인지 정의해야 합니다. 예를 들어 "학생"에 대한 정보를 저장한다면, 학생을 설명하는 요소들—학번, 이름, 학년, 학과—이 각각 하나의 단위가 됩니다. 이 단위들을 **애트리뷰트(Attribute, 속성)** 라고 부릅니다.

마치 사람을 설명할 때 "키", "몸무게", "나이", "이름"처럼 각각의 특성으로 나눠 설명하듯이, 데이터베이스도 개체(사람, 상품, 주문 등)를 여러 개의 애트리뷰트로 나눠서 표현합니다.

### 애트리뷰트의 정의와 특징

**애트리뷰트(Attribute)** 는 릴레이션(테이블)에서 **열(Column)에 해당하며, 더 이상 쪼갤 수 없는 가장 작은 논리적 단위**입니다.

> 🧩 **애트리뷰트 = 릴레이션의 가장 작은 논리적 단위 = 테이블의 열(Column)**

예를 들어 학생 테이블이 아래와 같다면:

| **학번** | **이름** | **학년** | **학과** |
|---|---|---|---|
| 100 | 홍길동 | 1 | 컴퓨터공학 |
| 200 | 김철수 | 2 | 전자공학 |

여기서 **학번, 이름, 학년, 학과** 각각이 하나의 애트리뷰트입니다. "이름"이라는 애트리뷰트를 더 잘게 쪼개서 "성"과 "이름"으로 나눌 수도 있지만, 그렇게 하지 않고 "이름" 전체를 하나의 원자적 단위로 취급하는 것이 바로 애트리뷰트의 개념입니다.

애트리뷰트가 갖는 가장 중요한 규칙 두 가지를 기억해 두세요:

**첫째, 속성의 명칭은 유일해야 합니다.** 같은 릴레이션 안에서 "이름"이라는 애트리뷰트가 두 개 존재할 수 없습니다. 마치 한 학교 안에서 반(班)의 이름이 "1반"으로 두 개일 수 없는 것과 같습니다.

**둘째, 각 애트리뷰트의 값은 원자 값(Atomic Value), 즉 더 이상 나눌 수 없는 단 하나의 값이어야 합니다.** 하나의 칸에 여러 값이 들어가면 안 됩니다. 예를 들어 점수 칸에 "80, 90"처럼 두 개의 값이 들어가면 안 됩니다.

:::warning 원자 값 위반 예시
아래처럼 하나의 셀에 여러 값이 들어가는 것은 **관계형 데이터베이스의 규칙 위반**입니다!

❌ **잘못된 예시:**

| 학번 | 이름 | 점수 |
|---|---|---|
| 100 | 홍길동 | 80, 90, 75 |

✅ **올바른 예시:**

| 학번 | 이름 | 국어점수 | 영어점수 | 수학점수 |
|---|---|---|---|---|
| 100 | 홍길동 | 80 | 90 | 75 |

각 칸(애트리뷰트)에는 반드시 **하나의 값**만 들어가야 합니다.
:::

```sql
-- ❌ 잘못된 테이블 설계: 하나의 컬럼에 여러 값이 들어가는 구조
-- 이렇게 설계하면 WHERE 조건으로 특정 값을 찾기가 매우 어려워집니다
CREATE TABLE 학생_잘못된예 (
    학번 INT,
    이름 VARCHAR(10),
    점수 VARCHAR(50)  -- "80, 90, 75" 처럼 여러 값을 텍스트로 저장 → 절대 금지!
);

-- ✅ 올바른 테이블 설계: 각 애트리뷰트에 원자 값 하나만 저장
CREATE TABLE 학생_올바른예 (
    학번 INT PRIMARY KEY,          -- 기본키: 학생을 유일하게 식별
    이름 VARCHAR(10) NOT NULL,     -- 이름: 문자열 10자 이내
    국어점수 INT,                   -- 국어 점수: 정수 하나만 저장
    영어점수 INT,                   -- 영어 점수: 정수 하나만 저장
    수학점수 INT                    -- 수학 점수: 정수 하나만 저장
);
```

왜 원자 값이 중요할까요? 만약 점수 칸에 "80, 90, 75"처럼 여러 값이 뭉쳐 들어가 있으면, "점수가 80점 이상인 학생을 찾아라"라고 했을 때 데이터베이스가 그 값을 어떻게 비교해야 할지 알 수가 없습니다. **데이터는 비교하고, 검색하고, 연산할 수 있어야 의미가 있는데**, 뭉쳐있는 값은 이런 작업을 불가능하게 만듭니다.

## 🎯 도메인(Domain)이란 무엇인가

### 왜 도메인 개념이 필요한가

데이터베이스에서 "비교"는 매우 중요한 작업입니다. "이 학생의 점수가 80점 이상인가?", "이 직원의 나이가 30세 이상인가?"처럼 값을 비교해야 할 일이 많습니다. 그런데 **비교가 의미 있으려면, 비교되는 두 값이 같은 종류의 값이어야 합니다.** 국어 점수(100점 만점)와 수학 점수(300점 만점)를 단순히 숫자만 보고 비교하면 어떻게 될까요? 이 문제를 해결하기 위해 **도메인(Domain)** 이라는 개념이 필요합니다.

### 도메인의 정의

> 📌 **도메인(Domain)** = 하나의 애트리뷰트(속성)가 취할 수 있는, **같은 타입의 원자 값들의 집합**

쉽게 말하면, **어떤 속성에 들어올 수 있는 값의 범위와 종류를 미리 정해놓은 것**입니다. 마치 "이 칸에는 0부터 100 사이의 정수만 넣을 수 있어!"라고 규칙을 정해두는 것과 같습니다.

예를 들어 살펴봅시다:

| 애트리뷰트 | 도메인 (값의 범위) | 예시 |
|---|---|---|
| 국어 점수 | 0 이상 100 이하의 정수 | 75, 80, 100 |
| 수학 점수 | 0 이상 300 이하의 정수 | 150, 200, 300 |
| 학년 | 1, 2, 3, 4 중 하나의 정수 | 1, 2, 3, 4 |
| 학과명 | 10자 이하의 문자열 | "컴퓨터공학", "전자공학" |
| 성별 | '남' 또는 '여' | '남', '여' |

### 도메인이 다르면 비교가 무의미하다

이 개념이 왜 중요한지 구체적인 예로 설명해 보겠습니다.

어떤 학교에 두 가지 시험이 있다고 가정합니다:
- **국어 시험**: 0점 ~ 100점 만점
- **수학 시험**: 0점 ~ 300점 만점

홍길동 학생이 국어 80점, 수학 80점을 받았습니다. 두 과목 다 80점인데, 이 두 점수를 단순히 "둘 다 80점으로 같다"고 비교하는 것이 올바를까요? 절대 아닙니다!

- 국어 80점 = **100점 만점에서 80%** (꽤 잘한 것)
- 수학 80점 = **300점 만점에서 약 26.7%** (잘 못한 것)

이처럼 **도메인(값의 범위)이 다른 두 값을 단순히 숫자만 보고 비교하면 전혀 의미 없는 결과, 혹은 잘못된 결과**를 얻게 됩니다. 따라서 데이터베이스는 **같은 도메인을 가진 값들끼리만 비교가 허용**되어야 합니다.

:::danger 도메인이 다른 값의 비교
도메인이 서로 다른 두 애트리뷰트의 값을 비교하는 것은 **데이터베이스의 무결성(데이터의 정확성과 일관성)을 해칩니다.** 설계 단계에서 각 속성의 도메인을 반드시 명확하게 정의해야 합니다. 나중에 잘못 설계된 도메인 때문에 처음부터 다시 설계해야 하는 상황이 올 수 있습니다!
:::

```sql
-- ✅ 올바른 예시: 도메인을 명확하게 제약 조건으로 설정하는 방법
CREATE TABLE 시험성적 (
    학번     INT           PRIMARY KEY,                  -- 학번: 기본키
    이름     VARCHAR(10)   NOT NULL,                     -- 이름: 필수값
    국어점수 INT           CHECK (국어점수 BETWEEN 0 AND 100),   -- 도메인: 0~100
    수학점수 INT           CHECK (수학점수 BETWEEN 0 AND 300),   -- 도메인: 0~300
    학년     INT           CHECK (학년 BETWEEN 1 AND 4),        -- 도메인: 1~4
    성별     CHAR(1)       CHECK (성별 IN ('남', '여'))           -- 도메인: '남' 또는 '여'
);

-- ❌ 잘못된 예시: 도메인 제약 없이 아무 값이나 들어올 수 있는 설계
CREATE TABLE 시험성적_잘못된예 (
    학번     INT,          -- 기본키 설정도 없음!
    이름     VARCHAR(10),
    국어점수 INT,          -- -999 나 9999 같은 이상한 값이 들어올 수 있음!
    수학점수 INT,          -- 범위 제한이 없어서 데이터 신뢰성 하락
    학년     INT,          -- 학년이 100이 들어와도 막을 수 없음
    성별     CHAR(1)       -- '가', 'Z' 같은 잘못된 값도 허용됨
);
```

**도메인 설정이 왜 중요한지** 한 줄로 정리하면: **도메인을 잘 설정해야 데이터의 무결성(정확성)을 보장할 수 있고, 나중에 예상치 못한 오류를 방지할 수 있습니다.**

:::tip 도메인 설계 팁
데이터베이스를 설계할 때, 각 속성에 대해 다음 질문을 스스로 해보세요:
- 이 속성에 들어올 수 있는 **값의 종류**는 무엇인가? (정수, 문자, 날짜 등)
- 이 속성의 **최솟값과 최댓값**은 얼마인가?
- 이 속성에 **NULL(빈 값)** 이 허용되는가?
- 이 속성에 들어올 수 있는 **특정 값의 목록**이 있는가? (예: '남', '여')

이 네 가지를 명확히 정의하면 도메인을 올바르게 설정할 수 있습니다.
:::

## 📐 릴레이션의 구성: 스키마와 인스턴스

### 릴레이션은 두 가지로 이루어진다

릴레이션(Relation)을 완전하게 이해하려면, 릴레이션이 **두 가지 요소로 구성**된다는 것을 알아야 합니다. 바로 **스키마(Schema)** 와 **인스턴스(Instance)** 입니다.

건물을 다시 비유로 들어보겠습니다. 건물을 짓기 전에 **설계 도면(스키마)** 이 있고, 실제로 완성된 **건물 자체(인스턴스)** 가 있습니다. 설계 도면은 방이 몇 개인지, 어디에 문이 있는지 등 **구조**를 정의합니다. 실제 건물은 사람들이 살면서 가구도 들어오고 바뀌는 **실제 내용**입니다.

데이터베이스도 마찬가지입니다:
- **스키마 = 설계 도면** (구조 정의)
- **인스턴스 = 실제 건물 내부** (실제 저장된 데이터)

### 릴레이션 스키마(Schema) — 구조와 규칙

**릴레이션 스키마(Relation Schema)** 는 릴레이션의 **구조를 정의하는 것**입니다. 스키마는 다음 세 가지 요소로 구성됩니다:

1. **릴레이션의 이름**: 예) `학생`
2. **애트리뷰트(속성)의 이름**: 예) `학번, 이름, 학년, 학과`
3. **제약 조건(Constraint)**: 기본키, NULL 허용 여부, 도메인 범위 등

스키마는 한 번 정해지면 자주 바뀌지 않는 **정적(Static)인 특성**을 가집니다. 마치 건물의 구조(방 개수, 벽의 위치)가 쉽게 바뀌지 않는 것처럼요. 이런 정적인 특성 때문에 스키마를 **인텐션(Intension)** 이라고도 부릅니다.

```sql
-- 학생 릴레이션의 스키마 정의 예시
-- 이것이 바로 릴레이션의 '구조'를 만드는 것입니다
CREATE TABLE 학생 (
    학번  INT          PRIMARY KEY,         -- 기본키: 중복 불가, NULL 불가
    이름  VARCHAR(6)   NOT NULL,            -- 문자 6자 이내, NULL 불가
    학년  INT          CHECK (학년 BETWEEN 1 AND 4),  -- 1~4 사이 값만 허용
    학과  VARCHAR(10)                        -- 문자 10자 이내
    -- 이것들이 모두 합쳐져서 '학생' 릴레이션의 스키마가 됩니다
);
```

스키마는 보통 이렇게 표기합니다: **학생(학번, 이름, 학년, 학과)**

### 릴레이션 인스턴스(Instance) — 실제 저장된 데이터

**릴레이션 인스턴스(Relation Instance)** 는 릴레이션에 **실제로 저장된 데이터**, 즉 **튜플(행)들의 집합**입니다. 인스턴스는 **익스텐션(Extension)** 또는 **익스텐트(Extent)** 라고도 부릅니다.

인스턴스는 데이터를 삽입(INSERT), 삭제(DELETE), 수정(UPDATE)할 때마다 내용이 달라집니다. 따라서 **동적(Dynamic)인 특성**을 가집니다. 건물에 사는 사람이 바뀌거나 가구 배치가 달라지는 것처럼, 실제 데이터는 계속 변화합니다.

```sql
-- 학생 릴레이션의 인스턴스 (실제 데이터) 삽입 예시
INSERT INTO 학생 VALUES (100, '홍길동', 1, '컴퓨터공학');  -- 1번째 튜플 삽입
INSERT INTO 학생 VALUES (200, '김철수', 2, '전자공학');    -- 2번째 튜플 삽입
INSERT INTO 학생 VALUES (300, '이영희', 3, '컴퓨터공학'); -- 3번째 튜플 삽입
INSERT INTO 학생 VALUES (400, '박민준', 1, '기계공학');    -- 4번째 튜플 삽입
INSERT INTO 학생 VALUES (500, '최지아', 4, '화학공학');    -- 5번째 튜플 삽입

-- 이 5개의 행(튜플)이 현재 시점에서 '학생' 릴레이션의 인스턴스입니다
-- 나중에 INSERT나 DELETE가 발생하면 인스턴스가 바뀝니다
```

이를 표로 나타내면:

| 학번 | 이름 | 학년 | 학과 |
|---|---|---|---|
| 100 | 홍길동 | 1 | 컴퓨터공학 |
| 200 | 김철수 | 2 | 전자공학 |
| 300 | 이영희 | 3 | 컴퓨터공학 |
| 400 | 박민준 | 1 | 기계공학 |
| 500 | 최지아 | 4 | 화학공학 |

위 표에서 **학번, 이름, 학년, 학과**라는 열 이름이 스키마이고, **실제로 들어있는 5개의 행 전체**가 인스턴스입니다.

> 🔑 **전체 릴레이션 = 스키마(구조) + 인스턴스(실제 데이터)**
> 구조만 있고 데이터가 없어도, 데이터만 있고 구조가 없어도 완전한 릴레이션이 아닙니다. 두 가지가 합쳐져야 비로소 완전한 릴레이션이 됩니다.

### 차수(Degree)와 카디널리티(Cardinality)

릴레이션을 설명할 때 꼭 알아야 하는 두 가지 숫자가 있습니다.

**차수(Degree)** 는 **애트리뷰트(속성)의 개수**입니다. 위 학생 테이블은 학번, 이름, 학년, 학과 — 총 4개의 속성을 가지고 있으므로 차수는 **4**입니다. 차수는 스키마(구조)와 관련된 값이므로, 한 번 정해지면 잘 변하지 않는 **정적인 특성**을 가집니다.

**카디널리티(Cardinality)** 는 **튜플(행)의 개수**입니다. 위 학생 테이블에는 현재 5개의 행이 있으므로 카디널리티는 **5**입니다. 카디널리티는 인스턴스(실제 데이터)와 관련된 값이므로, 데이터를 삽입하거나 삭제할 때마다 바뀌는 **동적인 특성**을 가집니다.

| 용어 | 의미 | 특성 | 예시 |
|---|---|---|---|
| **차수 (Degree)** | 애트리뷰트(열)의 개수 | 정적 (잘 안 바뀜) | 학번·이름·학년·학과 → 4 |
| **카디널리티 (Cardinality)** | 튜플(행)의 개수 | 동적 (자주 바뀜) | 5개 행 → 5 |

:::note 암기 팁
- **차수(Degree)** → **D**egree → **D**ata Structure(구조) → **정적**
- **카디널리티(Cardinality)** → **C**ardinality → **C**hanging(변화) → **동적**
:::

## ⚙️ 릴레이션의 네 가지 특성

릴레이션은 단순한 표가 아니라, 수학적으로 엄밀하게 정의된 구조입니다. 릴레이션이 갖는 네 가지 중요한 특성을 이해하면 왜 관계형 데이터베이스가 이런 방식으로 동작하는지 납득할 수 있게 됩니다.

### 1️⃣ 튜플의 유일성 (Uniqueness of Tuples)

릴레이션 안에 있는 **모든 튜플(행)은 서로 달라야 합니다.** 완전히 똑같은 행이 두 개 이상 존재하면 안 됩니다. 이를 **튜플의 유일성**이라고 합니다.

❌ **잘못된 예시 (튜플 중복 — 릴레이션 정의 위반):**

| 학번 | 이름 | 학년 |
|---|---|---|
| 100 | 홍길동 | 1 |
| 100 | 홍길동 | 1 |
| 200 | 김철수 | 2 |

위처럼 완전히 동일한 행(100, 홍길동, 1)이 두 번 나오면 **릴레이션의 정의를 위반**합니다. 이 때문에 실제 데이터베이스에서는 **기본키(Primary Key)** 를 설정해서 중복을 원천적으로 방지합니다.

✅ **올바른 예시:**

| 학번 (기본키) | 이름 | 학년 |
|---|---|---|
| 100 | 홍길동 | 1 |
| 200 | 김철수 | 2 |
| 300 | 이영희 | 3 |

### 2️⃣ 튜플의 무순서성 (No Order of Tuples)

릴레이션에서는 **튜플들 사이에 순서가 없습니다.** 학번 100번 행이 첫 번째에 있든 세 번째에 있든, 릴레이션의 의미는 동일합니다.

실제 테이블에서는 "학번 순으로 정렬", "점수 높은 순으로 정렬" 같은 작업을 하지만, 이것은 **조회(SELECT) 시점에서의 편의를 위한 것**일 뿐, 릴레이션 자체에 순서가 있다는 의미가 아닙니다. 릴레이션은 수학의 **집합(Set)** 개념에 기반하고 있는데, 집합은 원소들 사이에 순서가 없습니다.

### 3️⃣ 애트리뷰트의 무순서성 (No Order of Attributes)

**열(Column, 애트리뷰트)들 사이에도 순서가 없습니다.** 학번이 첫 번째 열이든 세 번째 열이든 릴레이션의 의미는 변하지 않습니다.

실제로 테이블을 만들 때는 왼쪽에서 오른쪽으로 순서가 있는 것처럼 보이지만, 릴레이션 이론에서는 각 속성이 독립적인 것이지 순서를 가지는 것이 아닙니다.

### 4️⃣ 애트리뷰트 값의 원자성 (Atomicity of Attribute Values)

앞서 도메인 설명에서도 언급했듯이, **모든 애트리뷰트의 값은 더 이상 나눌 수 없는 원자 값(Atomic Value)이어야 합니다.** 하나의 셀에 여러 값이 들어가면 안 됩니다.

```sql
-- 네 가지 특성을 모두 반영한 릴레이션 설계 예시

CREATE TABLE 학생 (
    학번  INT          PRIMARY KEY,    -- 튜플 유일성 보장: 중복 학번 불가
    이름  VARCHAR(6)   NOT NULL,       -- 원자 값: 이름 하나만 저장
    학년  INT          CHECK (학년 BETWEEN 1 AND 4),  -- 도메인 제약
    학과  VARCHAR(10)                   -- 원자 값: 학과 하나만 저장
    -- 열의 순서는 릴레이션 이론상 의미 없음 (무순서성)
    -- 행의 순서도 릴레이션 이론상 의미 없음 (무순서성)
);
```

네 가지 특성을 한눈에 정리하면:

| 특성 | 내용 | 실질적 의미 |
|---|---|---|
| **튜플의 유일성** | 중복 행 불허 | 기본키로 보장 |
| **튜플의 무순서성** | 행 순서 없음 | 집합 개념 기반 |
| **애트리뷰트의 무순서성** | 열 순서 없음 | 각 속성은 독립적 |
| **애트리뷰트 값의 원자성** | 셀에 값 하나만 | 다중값 금지 |

## ⚠️ 주의사항 및 실전 팁

### 테이블과 릴레이션을 혼동하지 말 것

이 문서에서 여러 번 강조했지만, **테이블과 릴레이션은 비슷하지만 엄밀하게는 다릅니다.** 시험이나 이론 공부에서는 릴레이션의 엄격한 특성(튜플 유일성, 무순서성 등)을 기준으로 답해야 합니다. 실무 SQL 작업에서는 테이블의 유연한 특성(행 순서, 중복 등)을 활용하는 경우도 있음을 구분해서 이해하세요.

### 도메인 설정은 설계 초기에 반드시 확정하라

데이터베이스 설계에서 도메인을 처음에 잘못 정하면, 나중에 데이터가 수만 건 쌓인 뒤에 수정해야 하는 대참사가 발생할 수 있습니다. 예를 들어, 점수를 0~100으로 설정해 두었다가 나중에 0~300으로 바꿔야 한다면 기존 데이터와의 정합성(맞아떨어지는 정도)을 다 확인해야 합니다. **설계 단계에서 꼼꼼하게 도메인을 정의하는 것이 최선입니다.**

:::warning 스키마 변경의 위험성
이미 운영 중인 데이터베이스의 스키마(구조)를 바꾸는 것은 매우 위험한 작업입니다. 기존 데이터와 새로운 스키마가 충돌할 수 있고, 이를 복구하는 데 엄청난 시간과 비용이 소모됩니다. **처음 설계를 잘 하는 것이 가장 중요합니다.**
:::

### 애트리뷰트 이름은 직관적이고 유일하게

같은 릴레이션 안에서 애트리뷰트 이름은 반드시 유일해야 합니다. 그리고 이름만 봐도 어떤 데이터를 저장하는지 알 수 있도록 **직관적인 이름**을 사용하는 것이 좋습니다.

```sql
-- ❌ 나쁜 예시: 이름만으로 의미를 알기 어려움
CREATE TABLE T1 (
    a1 INT,
    a2 VARCHAR(10),
    a3 INT
);

-- ✅ 좋은 예시: 이름만으로도 어떤 데이터인지 명확함
CREATE TABLE 학생 (
    학번  INT          PRIMARY KEY,
    이름  VARCHAR(10)  NOT NULL,
    학년  INT
);
```

### NULL 값과 도메인

도메인을 설정할 때 **NULL 값 허용 여부**도 반드시 고려해야 합니다. NULL은 "값이 없음"을 나타내는 특별한 표시인데, NULL이 너무 많으면 데이터베이스 성능이 저하될 수 있고, 비교 연산에서도 예상치 못한 결과가 나올 수 있습니다.

```sql
-- NULL 허용과 불허의 차이
CREATE TABLE 직원 (
    직원번호  INT          PRIMARY KEY,       -- NULL 불가 (기본키)
    이름      VARCHAR(10)  NOT NULL,          -- NULL 불가 (명시적 설정)
    이메일    VARCHAR(50),                    -- NULL 허용 (이메일 없는 직원 가능)
    부서번호  INT          NOT NULL           -- NULL 불가 (반드시 소속 부서 있어야 함)
);
```

:::tip NULL 관리 팁
NULL이 많을수록 데이터베이스 성능이 떨어집니다. 설계 단계에서 어떤 속성이 NULL을 허용해야 하는지 신중하게 결정하고, 가능하면 NULL 발생을 최소화하는 방향으로 설계하세요.
:::

## 📋 핵심 정리

이 문서에서 배운 핵심 개념들을 정리합니다.

- **관계 데이터(Relational Data)** 는 테이블 형태로 데이터를 표현하며, 기본키와 외래키로 테이블 간 관계를 맺는다.
- **릴레이션(Relation)** 은 테이블의 수학적·이론적 표현이며, **테이블**은 실무에서 사용하는 구체적 표현이다.
- **튜플(Tuple)** 은 릴레이션의 행(Row), **애트리뷰트(Attribute)** 는 릴레이션의 열(Column)에 해당한다.
- **애트리뷰트** 는 릴레이션의 가장 작은 논리적 단위이며, 명칭은 유일해야 하고 원자 값만을 가져야 한다.
- **도메인(Domain)** 은 하나의 애트리뷰트가 취할 수 있는 같은 타입의 원자 값들의 집합(범위)이다.
- **도메인이 다른 값끼리의 비교는 무의미하며**, 잘못된 결과를 초래할 수 있어 데이터 무결성을 해친다.
- **릴레이션 스키마(Schema)** 는 릴레이션의 이름, 속성 이름, 제약 조건으로 구성된 구조 정의이며 **정적**이다.
- **릴레이션 인스턴스(Instance)** 는 특정 시점에 릴레이션에 저장된 실제 튜플들의 집합이며 **동적**이다.
- **차수(Degree)** 는 애트리뷰트(열)의 개수로 정적 특성을, **카디널리티(Cardinality)** 는 튜플(행)의 개수로 동적 특성을 나타낸다.
- 릴레이션의 네 가지 특성: **튜플의 유일성**, **튜플의 무순서성**, **애트리뷰트의 무순서성**, **애트리뷰트 값의 원자성**.
- **전체 릴레이션 = 스키마 + 인스턴스**이며, 두 가지가 합쳐져야 완전한 릴레이션이 된다.
- 도메인 설정은 설계 초기에 명확히 해야 하며, 나중에 변경하면 큰 비용이 발생할 수 있다.
