# 🗂️ 릴레이션과 테이블의 개념 및 구성 요소

데이터베이스를 공부하다 보면 "테이블"이라는 말은 자주 들었는데, 갑자기 "릴레이션", "튜플", "애트리뷰트", "카디널리티"처럼 생소한 용어들이 등장해서 당황한 경험이 있으실 겁니다. 이 문서에서는 그 낯선 개념들을 하나하나 쉽게 풀어서, 처음 데이터베이스를 공부하는 분들도 확실히 이해할 수 있도록 정리해 드립니다. 단순한 용어 암기가 아니라, **왜 이런 개념이 필요한지**부터 시작해서 실제 예시와 함께 살펴보겠습니다.

---

## 🧩 관계 데이터(Relational Data)란 무엇인가?

### 관계 데이터의 기본 아이디어

데이터베이스에는 수많은 종류가 있지만, 오늘날 가장 널리 쓰이는 방식이 바로 **관계형 데이터베이스(Relational Database)** 입니다. 그렇다면 "관계형"이라는 말은 무슨 뜻일까요?

쉽게 말하면, 데이터를 **테이블(표) 형태**로 저장하고, 테이블과 테이블 사이를 **관계(Relation)로 연결**하는 방식입니다. 마치 학교에서 학생 명단표와 성적표가 따로 있지만, 학번이라는 공통된 값으로 두 표를 연결하는 것처럼요.

> **관계 데이터 구조** = 테이블과 테이블을 연결하여 데이터 간의 관계를 표현하는 구조

### 기본키와 외래키: 테이블을 이어주는 다리

테이블과 테이블을 어떻게 연결할까요? 바로 **기본키(Primary Key)** 와 **외래키(Foreign Key)** 를 통해서입니다.

예를 들어 두 개의 테이블이 있다고 가정해봅시다.

| 학생 테이블 | 성적 테이블 |
|---|---|
| **학번: 1, 2, 3, 4** | 성적ID: 4, 5, 6, 7 |
| 이름: 홍길동, 김철수 ... | **학번(외래키): 4** |

여기서 학생 테이블의 **학번 4**와 성적 테이블의 **학번 4**가 서로 연결되어 있습니다. 학생 테이블에서의 학번은 **기본키(Primary Key)**, 성적 테이블에서 참조하는 학번은 **외래키(Foreign Key)** 라고 부릅니다.

:::info 기본키와 외래키 한 줄 정리
- **기본키**: 테이블에서 각 행(레코드)을 유일하게 식별하는 값. 중복 불가, 빈 값(NULL) 불가
- **외래키**: 다른 테이블의 기본키를 참조하는 값. 두 테이블을 연결하는 다리 역할
:::

이렇게 기본키와 외래키를 통해 **개체(Entity, 데이터의 대상)와 개체 사이의 관계를 테이블로 표현**하는 것이 관계형 데이터베이스의 가장 핵심적인 특징입니다.

## 📐 릴레이션(Relation)과 테이블(Table)의 차이

### 왜 같은 것에 이름이 두 개인가?

처음 데이터베이스를 배울 때 가장 헷갈리는 것 중 하나가 바로 "테이블이랑 릴레이션이 같은 말 아닌가요?"라는 질문입니다. 반은 맞고 반은 틀립니다.

**테이블(Table)** 은 우리가 실무에서 실제로 사용하는 개념입니다. 엑셀 표처럼 행과 열로 이루어진 데이터 저장 구조를 말하죠. **릴레이션(Relation)** 은 그 테이블을 수학적으로 표현한 개념입니다. 쉽게 말해, 릴레이션은 테이블의 수학적 이론 버전입니다.

> 마치 "사과"와 "Malus domestica(사과의 학명)"의 관계처럼, 같은 대상을 일상 언어(테이블)와 학문적 언어(릴레이션)로 부르는 것입니다.

### 용어 비교: 테이블 vs 릴레이션

실무에서는 테이블 용어를, 이론에서는 릴레이션 용어를 사용합니다. 아래 표를 꼭 기억해두세요.

| 개념 | 테이블 용어 | 릴레이션 용어 |
|---|---|---|
| 가로줄 (한 행의 데이터) | **행(Row)** / 레코드(Record) | **튜플(Tuple)** |
| 세로줄 (하나의 속성) | **열(Column)** | **애트리뷰트(Attribute)** / 속성 |
| 전체 구조 | **테이블(Table)** | **릴레이션(Relation)** |

:::tip 시험이나 강의에서 헷갈릴 때
"튜플", "애트리뷰트"라는 단어가 나오면 → **릴레이션(수학적/이론적 개념)**  
"행", "열", "레코드"라는 단어가 나오면 → **테이블(실무적 개념)**  
이렇게 구분하시면 됩니다!
:::

### 테이블과 릴레이션의 실질적인 차이점

같은 개념 같아 보이지만, 엄밀하게는 다른 점이 있습니다. 이 차이가 나중에 설계할 때 굉장히 중요합니다.

| 비교 항목 | 릴레이션(Relation) | 테이블(Table) |
|---|---|---|
| 중복 행 허용 여부 | ❌ 절대 중복 불가 (튜플 유일성) | ✅ 필요에 따라 중복 허용 |
| 행의 순서 | ❌ 순서 없음 (무순서성) | ✅ 정렬 가능 (예: 학번순) |
| 열의 순서 | ❌ 순서 없음 (무순서성) | ✅ 왼쪽→오른쪽 순서 있음 |
| 사용 맥락 | 이론, 학문적 설명 | 실무, 실제 DB 시스템 |

예를 들어, 테이블에서는 성적표를 학번 1번부터 순서대로 정렬해서 보여주는 것이 일반적이고 왼쪽에 학번, 오른쪽에 이름, 그 다음 점수 순으로 열의 순서가 있습니다. 하지만 릴레이션은 수학적 집합(Set) 개념이라 순서가 의미 없습니다.

또한 릴레이션에서는 `(A1, B1)` 이라는 튜플이 두 번 들어가면 그것은 이미 릴레이션이 아닙니다. 하지만 테이블에서는 업무 처리 목적에 따라 중복된 행을 허용하기도 합니다.

## 🔬 릴레이션의 핵심 구성 요소

### 애트리뷰트(Attribute, 속성) — 더 이상 쪼갤 수 없는 최소 단위

**왜 애트리뷰트가 필요한가?** 데이터를 저장할 때 어떤 정보를 어떤 단위로 나눠서 저장할지 정해야 합니다. 그 최소 단위가 바로 애트리뷰트입니다.

마치 사람을 설명할 때 "이름", "나이", "주소"처럼 더 이상 쪼갤 수 없는 정보 항목 하나하나를 애트리뷰트라고 합니다.

- **릴레이션의 가장 작은 논리적 단위**입니다.
- 각 애트리뷰트의 명칭은 **유일(Unique)** 해야 합니다. 같은 릴레이션 안에 "이름"이라는 애트리뷰트가 두 개 있으면 안 되죠.
- 각 애트리뷰트는 **원자 값(Atomic Value)**, 즉 더 이상 쪼갤 수 없는 단 하나의 값만 가져야 합니다.

:::warning 원자 값 위반 예시
하나의 "점수" 애트리뷰트에 `80점, 50점` 처럼 두 개의 값을 넣으면 안 됩니다!  
데이터베이스가 "50점 이상인 학생 검색"을 할 때 셀 안에 값이 여러 개면 처리 자체가 불가능합니다.
:::

```sql
-- ❌ 잘못된 예시: 하나의 셀에 여러 점수를 넣음 (원자 값 위반)
-- 학번  | 이름   | 점수
-- 1001  | 홍길동 | 80, 90, 75  <- 이렇게 하면 안 됩니다

-- ✅ 올바른 예시: 각 과목별로 별도 행이나 별도 테이블로 분리
-- 학번  | 이름   | 국어점수 | 수학점수 | 영어점수
-- 1001  | 홍길동 | 80       | 90       | 75
```

### 도메인(Domain) — 속성이 가질 수 있는 값의 범위

**왜 도메인이 필요한가?** 데이터의 정확성을 보장하기 위해서입니다. 예를 들어 나이를 저장하는 애트리뷰트에 "안녕하세요"라는 문자열이 들어가면 안 되겠죠. 도메인은 바로 이런 상황을 막아주는 안전장치입니다.

> **도메인(Domain)** = 하나의 애트리뷰트(속성)가 취할 수 있는, 같은 타입의 원자 값들의 집합

쉽게 말하면, "이 칸에는 이런 값만 들어올 수 있어"라고 정해놓은 규칙입니다. 예를 들어:

- 학년 속성의 도메인: `{1, 2, 3, 4}` (대학교 기준)
- 성별 속성의 도메인: `{'남', '여'}`
- 국어 점수 속성의 도메인: `0 ~ 100 사이의 정수`

**도메인이 다르면 비교 자체가 의미 없습니다.** 이것이 아주 중요한 포인트입니다.

예를 들어 국어 점수(0~100점 만점)와 수학 점수(0~300점 만점)가 있다고 합시다. 둘 다 80점이지만 의미가 완전히 다릅니다. 국어 80점은 100점 만점에서 꽤 높은 점수지만, 수학 80점은 300점 만점에서 아주 낮은 점수입니다. 서로 도메인이 다른 값을 비교하면 잘못된 결론이 나옵니다.

:::danger 도메인 설계를 잘못하면?
처음 설계할 때 도메인을 잘못 정해놓으면 나중에 데이터를 비교하거나 처리할 때 예상치 못한 오류가 발생합니다.  
그렇게 되면 처음부터 설계를 다시 해야 하는 대참사가 일어날 수 있습니다.  
**도메인은 처음 설계 단계에서 꼼꼼하게 정해야 합니다!**
:::

### 차수(Degree)와 카디널리티(Cardinality) — 릴레이션의 크기를 표현하는 두 가지 방법

릴레이션의 크기를 표현할 때 두 가지 중요한 개념이 있습니다. 이 두 개념은 정반대의 성질을 가지고 있어서 함께 이해하면 더 쉽습니다.

#### 차수(Degree) — 정적인 특성

**차수(Degree)** 는 릴레이션에 있는 **애트리뷰트(열)의 개수**입니다.

마치 표의 열(컬럼) 개수라고 생각하면 됩니다. 학번, 이름, 학년, 학과라는 4개의 열이 있다면 차수는 4입니다.

차수는 **정적인(Static) 특성**을 가집니다. 한 번 테이블 구조를 만들면 자주 바뀌지 않기 때문입니다. 이것을 **스키마(Schema, 구조)** 라고도 부릅니다. 학생 테이블의 학번, 이름, 학년, 학과라는 구조는 학기가 바뀌어도 변하지 않죠.

#### 카디널리티(Cardinality) — 동적인 특성

**카디널리티(Cardinality)** 는 릴레이션에 있는 **튜플(행)의 개수**입니다.

학생이 100명이면 카디널리티는 100, 신입생이 입학하면 카디널리티가 늘어나고, 졸업생이 떠나면 줄어듭니다.

카디널리티는 **동적인(Dynamic) 특성**을 가집니다. 데이터를 삽입(INSERT), 삭제(DELETE), 수정(UPDATE)할 때마다 계속 변하기 때문입니다.

| 구분 | 차수(Degree) | 카디널리티(Cardinality) |
|---|---|---|
| 대상 | 애트리뷰트(열)의 수 | 튜플(행)의 수 |
| 특성 | **정적(Static)** — 잘 변하지 않음 | **동적(Dynamic)** — 자주 변함 |
| 다른 이름 | 스키마(Schema) | 인스턴스(Instance) |
| 예시 | 학번, 이름, 학년, 학과 → 4 | 현재 등록된 학생 수 → 500 |

## 🏗️ 릴레이션의 스키마(Schema)와 인스턴스(Instance)

### 릴레이션 = 스키마 + 인스턴스

릴레이션을 완전히 이해하려면 **스키마(Schema)** 와 **인스턴스(Instance)** 라는 두 가지 개념을 반드시 알아야 합니다.

집을 예시로 들어볼까요? 건축 설계도는 스키마고, 그 설계도를 바탕으로 실제로 지어진 집은 인스턴스입니다. 설계도(스키마)만 있어도 집이 아니고, 설계도 없이 집(인스턴스)만 있어도 완전한 건물이라 할 수 없습니다. 두 가지가 합쳐져야 완전한 릴레이션입니다.

### 릴레이션 스키마(Schema) — 구조를 정의하는 설계도

**릴레이션 스키마**는 릴레이션의 이름, 애트리뷰트 이름, 그리고 각종 제약 조건들로 구성됩니다. 다른 말로 **인텐션(Intension)** 이라고도 합니다.

스키마에 포함되는 것들:
- **릴레이션 이름**: 예) `학생`
- **애트리뷰트 이름**: 예) `학번, 이름, 학년, 학과`
- **제약 조건**: 예) "학번은 중복될 수 없고 NULL 불가", "점수는 0~100 사이"

스키마는 **정적인 특성**을 가집니다. 학생 테이블의 구조(학번, 이름, 학년, 학과)는 한 학기가 지나도 바뀌지 않습니다.

```sql
-- 스키마 정의 예시: CREATE TABLE이 곧 스키마를 만드는 것
CREATE TABLE 학생 (
    학번   INTEGER     PRIMARY KEY,    -- 기본키, NULL 불가, 중복 불가
    이름   CHAR(6)     NOT NULL,       -- 6자리 문자, 비어있으면 안 됨
    학년   INTEGER,                    -- 정수형 학년
    학과   CHAR(10)                    -- 10자리 문자로 학과명 저장
);
```

### 릴레이션 인스턴스(Instance) — 실제로 저장된 데이터

**릴레이션 인스턴스**는 특정 시점에 릴레이션에 실제로 저장되어 있는 데이터 값들의 집합입니다. 다른 말로 **익스텐션(Extension)** 또는 **상태(State)** 라고도 합니다.

인스턴스는 곧 **튜플들의 집합**입니다. 학생 테이블에 실제로 들어있는 100번 홍길동, 200번 김철수 같은 데이터 행들이 인스턴스입니다.

인스턴스는 **동적인 특성**을 가집니다. 새 학생이 등록(INSERT)되거나 졸업(DELETE)하면 인스턴스는 변합니다.

아래 표가 하나의 릴레이션을 스키마와 인스턴스로 구분한 예입니다.

```
[릴레이션 이름] 학생
[스키마]  학번(INTEGER) | 이름(CHAR 6) | 학년(INTEGER) | 학과(CHAR 10)
          ← 이 구조 전체가 스키마(Schema) →

[인스턴스]
100 | 홍길동 | 1 | 컴퓨터공학
200 | 김철수 | 2 | 정보시스템
300 | 이영희 | 3 | 소프트웨어
400 | 박민준 | 4 | 컴퓨터공학
500 | 최지수 | 1 | 정보시스템
← 이 튜플들 전체가 인스턴스(Instance) →

[전체 릴레이션] = 스키마 + 인스턴스
```

:::note 스키마 vs 인스턴스 요약
| 구분 | 스키마(Schema) | 인스턴스(Instance) |
|---|---|---|
| 다른 이름 | 인텐션(Intension) | 익스텐션(Extension) |
| 내용 | 릴레이션 이름 + 애트리뷰트 + 제약 조건 | 실제 저장된 튜플(행) 데이터 |
| 변화 | 정적 (잘 안 바뀜) | 동적 (자주 바뀜) |
| 비유 | 건물 설계도 | 실제 지어진 건물 |
:::

## 📜 릴레이션의 특성(Properties of Relation)

릴레이션은 수학적 집합 이론을 기반으로 하기 때문에 일반 테이블과는 다른 엄격한 특성이 있습니다. 이 특성들을 잘 이해해야 나중에 데이터베이스를 설계하고 조작할 때 실수를 피할 수 있습니다.

### 1️⃣ 튜플의 유일성(Uniqueness of Tuples)

**왜 필요한가?** 만약 완전히 똑같은 두 행이 존재한다면, 데이터베이스는 어떤 행을 수정하거나 삭제해야 할지 알 수 없습니다. 마치 주민등록번호 없이 동명이인을 구별하려는 것과 같죠.

> **한 릴레이션 안에 있는 모든 튜플은 서로 다른 값을 가져야 합니다. 중복 튜플은 허용되지 않습니다.**

```sql
-- ❌ 릴레이션 위반 예시: 완전히 동일한 튜플이 두 개 존재
-- 학번 | 이름
-- A1   | 홍길동
-- A1   | 홍길동   ← 이 행은 릴레이션에서 허용되지 않음

-- ✅ 올바른 예시: 모든 튜플이 서로 다름
-- 학번 | 이름
-- A1   | 홍길동
-- A2   | 김철수   ← 학번이 달라 유일하게 구별 가능
```

### 2️⃣ 튜플의 무순서성(No Ordering of Tuples)

**왜 중요한가?** 릴레이션은 수학적 집합(Set)이기 때문에, 집합의 원소들 사이에는 순서가 없습니다. 따라서 튜플들 사이에 "첫 번째 튜플", "두 번째 튜플"이라는 개념이 없습니다.

실무에서 테이블을 보면 보통 학번 순서대로 정렬되어 있는 것처럼 보이지만, 그것은 우리가 편의상 정렬(ORDER BY)한 것이지, 릴레이션 자체가 순서를 가지는 것이 아닙니다.

> 냉장고에 여러 재료가 들어있을 때, 어떤 순서로 들어있든 재료 자체는 변하지 않는 것처럼, 릴레이션의 튜플들은 어떤 순서로 저장되어 있든 릴레이션의 의미는 동일합니다.

### 3️⃣ 애트리뷰트의 무순서성(No Ordering of Attributes)

**왜 중요한가?** 마찬가지로 릴레이션의 열(애트리뷰트)들 사이에도 순서가 없습니다. "학번, 이름, 학년, 학과" 순서나 "학과, 학년, 이름, 학번" 순서나 릴레이션의 의미는 동일합니다.

일반 테이블에서는 왼쪽에서 오른쪽으로 순서가 있고(학번 → 이름 → 학년 → 학과), 그 순서가 중요하게 느껴지지만, 릴레이션 이론에서는 열의 순서는 의미가 없습니다. 각 애트리뷰트는 이름으로 구별되기 때문입니다.

### 4️⃣ 애트리뷰트 값의 원자성(Atomicity of Attribute Values)

**왜 필요한가?** 데이터베이스가 각 셀에서 하나의 값을 꺼내 비교하고 연산하려면, 그 값이 단 하나의 원자 값이어야 합니다. 셀 안에 여러 값이 들어있으면 "이 값이 50점 이상인가?"라는 질문에 대답할 수 없습니다.

> **모든 애트리뷰트 값은 더 이상 쪼갤 수 없는 원자 값(Atomic Value) 하나여야 합니다.**

```sql
-- ❌ 잘못된 예시: 하나의 애트리뷰트에 여러 값 저장
-- 학번 | 취미
-- 1001 | 독서, 등산, 수영   ← 원자성 위반! 값이 여러 개

-- ✅ 올바른 예시 방법 1: 별도 테이블로 분리
-- [취미 테이블]
-- 학번 | 취미
-- 1001 | 독서
-- 1001 | 등산
-- 1001 | 수영

-- ✅ 올바른 예시 방법 2: 다중값 속성을 별도 릴레이션으로 분리 (권장)
-- 학번과 취미를 복합키로 하는 별도 릴레이션 생성
```

## ⚠️ 릴레이션을 테이블로 변환할 때 주의사항과 팁

### ER 다이어그램을 릴레이션으로 변환하는 원칙

데이터베이스를 설계할 때는 먼저 **ER 다이어그램(Entity-Relationship Diagram)** 으로 개체와 관계를 표현한 뒤, 이를 실제 테이블(릴레이션)로 변환합니다. 이때 변환 방법은 개체 간의 관계 유형에 따라 달라집니다.

변환 시 반드시 고려해야 할 요소들:

1. **관계 유형**: 1:1인지, 1:N인지, M:N인지
2. **참여 유형**: 전체 참여(의무적, Mandatory/Total)인지, 부분 참여(선택적, Optional/Partial)인지
3. **기본키 선택**: 개체 무결성을 지킬 수 있는 기본키를 선택
4. **NULL 값 최소화**: NULL이 많아지면 데이터베이스 성능이 저하됨
5. **조인 최소화**: 테이블이 너무 많아지면 조인 연산이 많아져 성능 저하

:::warning NULL과 조인에 대한 성능 경고
- **NULL이 많아지면**: 저장 공간 낭비 + 비교 연산 시 오류 가능성
- **조인이 많아지면**: 여러 테이블을 결합하는 연산이 늘어나 쿼리 속도가 느려짐  
두 가지 모두 데이터베이스 성능을 저하시키는 주요 원인입니다. 설계 단계에서 최소화해야 합니다.
:::

### 1:1 관계 변환 방법

**1:1 관계에서 양쪽 모두 선택적 참여(Optional)**

예) 남자와 여자의 결혼 관계: 남자도 결혼 안 할 수 있고, 여자도 결혼 안 할 수 있음.

이 경우에는 **양쪽 테이블에 상대방의 기본키를 외래키로 각각 추가**합니다. 결혼한 사람은 상대방 주민번호가 채워지고, 결혼하지 않은 사람은 NULL이 들어갑니다.

```sql
-- ✅ 1:1 양쪽 선택적 참여 변환 예시
CREATE TABLE 남자 (
    남자주민번호  CHAR(14)  PRIMARY KEY,     -- 기본키
    남자이름      CHAR(10),
    여자주민번호  CHAR(14),                  -- 외래키 (결혼 안 하면 NULL)
    결혼날짜      DATE                       -- 설명 속성
);

CREATE TABLE 여자 (
    여자주민번호  CHAR(14)  PRIMARY KEY,     -- 기본키
    여자이름      CHAR(10),
    남자주민번호  CHAR(14),                  -- 외래키 (결혼 안 하면 NULL)
    결혼날짜      DATE                       -- 설명 속성
);
```

**1:1 관계에서 한쪽만 선택적, 한쪽은 의무적 참여**

예) 손님과 카드: 손님은 카드를 발급받을 수도 있고 안 받을 수도 있음(선택적). 하지만 카드는 반드시 주인인 손님이 있어야 함(의무적).

이 경우에는 **선택적 참여 쪽의 기본키를, 의무적(전체) 참여 쪽의 외래키로 넣습니다.** 이렇게 해야 NULL이 줄어들기 때문입니다.

```sql
-- ✅ 1:1 한쪽 의무적 참여 변환 예시
CREATE TABLE 손님 (
    손님번호  INTEGER   PRIMARY KEY,    -- 기본키
    손님이름  CHAR(10)
    -- 손님 쪽에는 카드 정보를 넣지 않음 (NULL이 많아질 수 있기 때문)
);

CREATE TABLE 카드 (
    카드번호    CHAR(16)  PRIMARY KEY,  -- 기본키
    카드종류    CHAR(10),
    신용한도    INTEGER,
    손님번호    INTEGER   NOT NULL      -- 외래키: 카드는 반드시 손님이 있어야 함
);
-- 카드 쪽에 손님번호(외래키)를 넣으면 NULL이 발생하지 않음!
```

**1:1 관계에서 양쪽 모두 의무적 참여**

양쪽 다 반드시 관계에 참여해야 하므로 테이블을 굳이 두 개로 나눌 필요 없이 **하나의 테이블로 합칩니다.** 이것이 훨씬 효율적입니다.

```sql
-- ✅ 1:1 양쪽 의무적 참여: 하나의 테이블로 합치기
CREATE TABLE 손님_선적정보 (
    손님번호    INTEGER   PRIMARY KEY,  -- 양쪽 기본키를 복합키로
    우편번호    CHAR(5),
    선적주소    VARCHAR(100),
    손님이름    CHAR(10)
);
```

### 1:N 관계 변환 방법

**1:N(일대다) 관계**에서는 반드시 **"1" 쪽의 기본키를 "N(다)" 쪽의 외래키로 넣습니다.**

예) 한 명의 판매원이 여러 개의 주문을 기재. 판매원(1) ↔ 주문(N).

```sql
-- ❌ 잘못된 예시: 다(N) 쪽의 기본키를 1 쪽에 외래키로 넣음
-- 이렇게 하면 기본키가 중복되어 개체 무결성 위반!
CREATE TABLE 판매원_잘못된예시 (
    판매원이름  CHAR(10)  PRIMARY KEY,  -- 기본키인데
    주문번호    CHAR(10)                -- 주문번호가 여러 개면 판매원이름이 중복됨 ← 위반!
);

-- ✅ 올바른 예시: 1 쪽의 기본키를 N 쪽의 외래키로 넣음
CREATE TABLE 판매원 (
    판매원이름  CHAR(10)  PRIMARY KEY   -- 1 쪽 기본키
);

CREATE TABLE 주문 (
    주문번호    CHAR(10)  PRIMARY KEY,  -- N 쪽 기본키
    주문내용    VARCHAR(200),
    판매원이름  CHAR(10)               -- 1 쪽 기본키를 외래키로 참조
    -- FOREIGN KEY (판매원이름) REFERENCES 판매원(판매원이름)
);
```

:::danger 1:N 관계에서 절대 하면 안 되는 실수
다(N) 쪽의 기본키를 1 쪽의 외래키로 넣으면 절대 안 됩니다!  
한 판매원이 여러 주문을 담당하는데, 판매원 테이블에 주문번호를 넣으면 같은 판매원 행이 주문 수만큼 중복되어 기본키가 중복됩니다.  
이는 **개체 무결성(Entity Integrity)** 위반입니다. **반드시 1 쪽의 기본키 → N 쪽의 외래키** 방향을 지키세요.
:::

### M:N 관계 변환 방법

**M:N(다대다) 관계**는 어느 쪽에도 외래키를 직접 넣을 수 없습니다. 왜냐하면 양쪽 모두 여러 개와 연결되기 때문입니다. 이럴 때는 **별도의 관계 테이블을 만들고, 양쪽의 기본키를 복합키로 사용**합니다.

예) 학생(M) ↔ 과목(N): 학생은 여러 과목을 수강하고, 과목도 여러 학생이 수강.

```sql
-- ✅ M:N 관계 변환: 별도 관계 테이블 생성
CREATE TABLE 학생 (
    학생번호  INTEGER  PRIMARY KEY,   -- 학생 기본키
    학과      CHAR(20)
);

CREATE TABLE 과목 (
    과목번호  CHAR(10)  PRIMARY KEY,  -- 과목 기본키
    과목이름  CHAR(50)
);

-- 별도 관계 테이블: 양쪽 기본키를 가져와 복합키로 사용
CREATE TABLE 등록 (
    학생번호  INTEGER,               -- 학생 테이블 기본키 참조
    과목번호  CHAR(10),              -- 과목 테이블 기본키 참조
    학점      CHAR(2),               -- 관계 자체의 속성(설명 속성)
    PRIMARY KEY (학생번호, 과목번호) -- 두 컬럼을 합쳐 복합키로 기본키 설정
);
```

### 다중값 속성, 약한 개체, 복합 속성 변환

**다중값 속성(Multi-valued Attribute)**: 하나의 속성에 여러 값이 올 수 있는 경우(예: 부서의 위치가 서울, 부산, 대전 여러 곳). 이 경우 **별도의 테이블로 반드시 분리**합니다.

```sql
-- ✅ 다중값 속성 변환 예시
CREATE TABLE 부서정보 (
    부서번호  INTEGER   PRIMARY KEY,  -- 기본키
    부서이름  CHAR(20),
    매니저    CHAR(10)
    -- 위치(다중값)는 이 테이블에 넣지 않음!
);

-- 다중값 속성 별도 테이블 생성
CREATE TABLE 부서위치 (
    부서번호  INTEGER,               -- 부서정보의 기본키 참조
    위치      CHAR(20),              -- 다중값 속성
    PRIMARY KEY (부서번호, 위치)     -- 복합키로 기본키
);
```

**약한 개체 타입(Weak Entity Type)**: 강한 개체의 기본키 없이는 스스로를 식별할 수 없는 개체(예: 사원의 부양가족). 이 경우 **강한 개체의 기본키를 외래키로 가져와** 함께 복합키로 사용합니다.

```sql
-- ✅ 약한 개체 변환 예시
CREATE TABLE 사원 (
    사원번호  INTEGER  PRIMARY KEY,  -- 강한 개체의 기본키
    사원이름  CHAR(10)
);

CREATE TABLE 부양가족 (
    사원번호  INTEGER,              -- 강한 개체의 기본키를 외래키로 가져옴
    가족이름  CHAR(10),             -- 약한 개체의 부분키
    관계      CHAR(5),
    PRIMARY KEY (사원번호, 가족이름) -- 두 개를 합쳐야 유일하게 식별 가능
);
```

**유도 속성(Derived Attribute)**: 다른 값으로부터 계산될 수 있는 속성(예: 나이 → 생년월일로부터 계산). 유도 속성은 **테이블에 저장하지 않습니다.** ER 다이어그램에서 점선으로 표시되며, 실제 릴레이션에는 포함시키지 않습니다.

**복합 속성(Composite Attribute)**: "주소"를 "시, 군, 동"으로 세분화하는 것처럼 더 작은 속성으로 나눌 수 있는 속성. 필요에 따라 그대로 두거나 세부 속성으로 분리할 수 있습니다.

## 🔗 순환 관계와 3진 이상 관계 변환

### 순환 관계성(Recursive Relationship)

순환 관계성이란 **같은 개체 타입이 자기 자신과 관계를 맺는 것**입니다. 마치 회사 조직도에서 사원이 상사를 참조하는데, 상사도 동일한 사원 테이블에 있는 경우입니다.

**1:N 순환 관계**: 한 명의 상사가 여러 명의 부하직원을 감독.

이 경우 테이블 하나만 생성되고, 1:N 규칙대로 **1 쪽(상사)의 기본키를 N 쪽(부하)의 외래키로** 같은 테이블 안에 추가합니다.

```sql
-- ✅ 1:N 순환 관계 변환 예시
CREATE TABLE 직원 (
    직원번호      INTEGER   PRIMARY KEY,   -- 기본키
    직원이름      CHAR(10),
    상사직원번호  INTEGER                  -- 자기 테이블의 기본키를 외래키로 참조
    -- FOREIGN KEY (상사직원번호) REFERENCES 직원(직원번호)
);
-- 상사가 없는 최고 책임자는 상사직원번호에 NULL이 들어감
```

**M:N 순환 관계**: 한 직원이 여러 직원을 감독하고, 한 직원이 여러 직원에게 감독받는 경우.

M:N 규칙대로 **별도의 관계 테이블을 추가**합니다. 이 경우 테이블이 2개가 됩니다(직원 테이블 + 감독 관계 테이블).

```sql
-- ✅ M:N 순환 관계 변환 예시
CREATE TABLE 직원 (
    직원번호  INTEGER  PRIMARY KEY,       -- 기본키
    직원이름  CHAR(10)
);

CREATE TABLE 감독 (
    상사직원번호  INTEGER,                 -- 상사 역할의 직원번호
    부하직원번호  INTEGER,                 -- 부하 역할의 직원번호
    PRIMARY KEY (상사직원번호, 부하직원번호) -- 복합키
);
```

### 3진 이상 관계(Ternary and Higher Relationship)

개체가 3개 이상이 관계를 맺는 경우를 **3진 관계(Ternary Relationship)** 라고 합니다. 예) 사원 + 부품 + 프로젝트가 "공급"이라는 관계로 연결.

이 경우 **무조건 별도의 관계 테이블을 생성**하고, 관련된 모든 개체의 기본키를 가져와 복합키로 사용합니다. 3개의 개체 테이블 + 1개의 관계 테이블 = 총 4개 테이블이 만들어집니다.

```sql
-- ✅ 3진 관계 변환 예시
CREATE TABLE 사원 (
    공급자번호  INTEGER  PRIMARY KEY     -- 사원 기본키
);

CREATE TABLE 부품 (
    부품번호  INTEGER  PRIMARY KEY       -- 부품 기본키
);

CREATE TABLE 프로젝트 (
    프로젝트번호  INTEGER  PRIMARY KEY   -- 프로젝트 기본키
);

-- 별도 관계 테이블: 3개 개체의 기본키를 모두 가져와 복합키 구성
CREATE TABLE 공급 (
    공급자번호    INTEGER,               -- 사원 기본키 참조
    부품번호      INTEGER,               -- 부품 기본키 참조
    프로젝트번호  INTEGER,               -- 프로젝트 기본키 참조
    물량          INTEGER,               -- 관계의 설명 속성
    PRIMARY KEY (공급자번호, 부품번호, 프로젝트번호) -- 세 개 복합키
);
```

## 📋 핵심 정리

이 문서에서 배운 핵심 개념들을 정리합니다.

- **관계 데이터**: 테이블 형태로 개체와 개체 간의 관계를 표현하며, 기본키와 외래키로 테이블을 연결한다.
- **릴레이션(Relation)**: 테이블의 수학적 개념으로, 이론에서는 릴레이션, 실무에서는 테이블이라는 용어를 사용한다.
- **튜플(Tuple)**: 릴레이션에서 행(Row)을 지칭하는 수학적 용어이다.
- **애트리뷰트(Attribute)**: 릴레이션에서 열(Column)을 지칭하며, 더 이상 쪼갤 수 없는 원자 값만 가져야 한다.
- **도메인(Domain)**: 애트리뷰트가 가질 수 있는 값의 범위이며, 서로 다른 도메인의 값을 비교하는 것은 의미가 없다.
- **차수(Degree)**: 릴레이션의 애트리뷰트(열) 수이며, 스키마에 해당하는 정적인 특성을 가진다.
- **카디널리티(Cardinality)**: 릴레이션의 튜플(행) 수이며, 삽입·삭제·수정에 따라 변하는 동적인 특성을 가진다.
- **스키마(Schema)**: 릴레이션의 이름, 애트리뷰트, 제약 조건으로 이루어진 구조(설계도)이며 정적이다.
- **인스턴스(Instance)**: 특정 시점에 릴레이션에 실제로 저장된 튜플들의 집합이며 동적이다.
- **튜플의 유일성**: 릴레이션 안의 모든 튜플은 서로 중복될 수 없다.
- **튜플·애트리뷰트의 무순서성**: 릴레이션에서 행과 열은 순서 개념이 없다.
- **1:1 관계 변환**: 참여 유형(선택적/의무적)에 따라 외래키 위치와 테이블 병합 방법이 달라진다.
- **1:N 관계 변환**: 반드시 "1" 쪽 기본키를 "N" 쪽 외래키로 삽입한다. 반대로 하면 기본키 중복이 발생한다.
- **M:N 관계 변환**: 반드시 별도의 관계 테이블을 생성하고, 양쪽 기본키를 복합키로 사용한다.
- **다중값 속성**: 별도의 테이블로 분리하고 원래 기본키와 함께 복합키로 구성한다.
- **약한 개체**: 강한 개체의 기본키를 외래키로 가져와 복합키로 유일성을 확보한다.
- **유도 속성**: 다른 값으로부터 계산되는 속성이므로 실제 테이블에는 저장하지 않는다.
- **NULL 최소화와 조인 최소화**: 두 가지 모두 데이터베이스 성능에 직결되므로 설계 단계에서 반드시 고려해야 한다.
