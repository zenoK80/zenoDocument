# 🗂️ 관계 데이터 구조와 기본키·외래키

데이터베이스를 처음 공부하다 보면 "릴레이션", "튜플", "애트리뷰트", "기본키", "외래키" 같은 낯선 용어들이 쏟아집니다. 마치 외국어처럼 느껴지죠. 하지만 이 개념들은 사실 우리가 일상에서 늘 사용하는 **표(테이블)** 구조를 수학적으로 정리한 것에 불과합니다. 이번 문서에서는 관계 데이터가 무엇인지, 테이블과 릴레이션의 차이는 무엇인지, 그리고 ER 다이어그램(개체-관계 설계도)을 실제 테이블로 어떻게 변환하는지를 처음 배우는 분도 완벽하게 이해할 수 있도록 하나씩 차근차근 설명합니다.

---

## 📌 관계 데이터란 무엇인가?

### 관계 데이터의 기본 개념

**관계 데이터(Relational Data)** 란, 데이터를 **테이블(표) 형태**로 표현하고, 테이블과 테이블 사이의 **관계(Relation)** 를 통해 데이터를 연결하는 방식입니다.

쉽게 말하면, 엑셀 시트처럼 행과 열로 구성된 표를 여러 개 만들고, 그 표들을 특정 값으로 서로 연결해 주는 것입니다. 우리가 학교에서 성적표를 받을 때 "학번 - 이름 - 국어점수 - 수학점수" 처럼 정리된 표를 떠올리면 됩니다.

> 핵심 정의: **관계 데이터 구조**란, 테이블과 테이블 간의 관계를 **기본키(Primary Key)** 와 **외래키(Foreign Key)** 를 통해 맺어주는 구조입니다.

### 기본키와 외래키의 첫 번째 이해

두 개의 테이블을 예로 들어 봅시다. **학생 테이블**에 학번 1, 2, 3, 4가 있고, **수강신청 테이블**에 4, 5, 6, 7이 있다고 하면, **숫자 4**가 양쪽 테이블에 모두 등장합니다.

- 학생 테이블의 **4**는 **기본키(Primary Key)**: 이 테이블에서 각 행을 유일하게 식별하는 값
- 수강신청 테이블의 **4**는 **외래키(Foreign Key)**: 다른 테이블의 기본키를 참조하는 값

마치 **주민등록번호**처럼, 기본키는 각 사람(행)을 유일하게 식별하는 번호입니다. 그리고 외래키는 "이 데이터는 저 테이블의 이 사람과 연결되어 있어요"라고 알려주는 **참조 번호** 역할을 합니다.

:::info 관계형 데이터베이스의 핵심
관계형 데이터베이스(RDBMS)의 가장 중요한 특징은 **기본키와 외래키를 통해 테이블 간 관계를 표현**한다는 것입니다. MySQL, PostgreSQL, Oracle 등 우리가 흔히 사용하는 데이터베이스가 모두 이 방식으로 동작합니다.
:::

## 🧮 테이블과 릴레이션: 같은 듯 다른 두 개념

### 왜 릴레이션이라는 말을 따로 쓸까?

처음 데이터베이스 이론을 공부하면 "테이블(Table)"과 "릴레이션(Relation)"이 혼용되어 등장합니다. 둘 다 같은 표 구조를 가리키는 것 같은데, 왜 두 가지 말을 쓰는 걸까요?

- **테이블**: 실무에서 실제로 사용하는 개념. "학생 테이블을 만들어라", "주문 테이블에서 조회해라"처럼 현장에서 쓰는 말
- **릴레이션(Relation)**: 테이블을 **수학적(이론적)으로 표현**한 개념. 데이터베이스 이론을 설명할 때 사용하는 학술 용어

> 릴레이션(Relation)은 테이블을 수학 집합론의 관점에서 바라본 개념입니다. "관계(Relation)"라는 뜻 그대로, 수학적으로 엄밀하게 정의된 테이블이라고 생각하시면 됩니다.

### 용어 대응표: 테이블 vs 릴레이션

테이블 용어와 릴레이션 용어는 다음과 같이 대응됩니다.

| 테이블 용어 | 릴레이션 용어 | 설명 |
|---|---|---|
| 테이블 (Table) | 릴레이션 (Relation) | 표 전체 |
| 행 (Row) | 튜플 (Tuple) | 표의 한 줄 (한 건의 데이터) |
| 열 (Column) | 애트리뷰트 (Attribute) | 표의 한 칸 항목 (속성) |
| 값의 개수 (행 수) | 카디널리티 (Cardinality) | 튜플의 수 |
| 열의 개수 (속성 수) | 차수 (Degree) | 애트리뷰트의 수 |

예를 들어, 학생 테이블에 학번·이름·학년·학과 4개 열이 있고, 학생이 500명이라면 **차수(Degree) = 4**, **카디널리티(Cardinality) = 500**이 됩니다.

### 차수(Degree)와 카디널리티(Cardinality): 정적 vs 동적

이 두 개념은 매우 중요합니다. 차이를 이해하지 못하면 데이터베이스 구조를 설명하기가 어렵습니다.

**차수(Degree, 속성의 수)** 는 **정적(Static)** 입니다. 마치 아파트 평면도처럼, 한번 설계되면 잘 바뀌지 않습니다. 학생 테이블에 학번, 이름, 학년, 학과가 있다면, 이 4개의 열은 테이블이 존재하는 한 거의 변하지 않습니다. 이것을 **스키마(Schema, 구조)** 라고도 합니다.

**카디널리티(Cardinality, 튜플의 수)** 는 **동적(Dynamic)** 입니다. 마치 아파트에 사는 입주민 수처럼, 새로운 학생이 입학하면 늘어나고(INSERT), 졸업하거나 자퇴하면 줄어듭니다(DELETE). 데이터의 삽입·삭제·갱신 작업이 일어날 때마다 값이 변합니다.

:::tip 시험에 자주 나오는 포인트
- **차수(Degree)** = 열(Column) 수 = 속성(Attribute) 수 = **정적**
- **카디널리티(Cardinality)** = 행(Row) 수 = 튜플(Tuple) 수 = **동적**
:::

## 🏗️ 릴레이션의 핵심 구성 요소

### 애트리뷰트(Attribute): 데이터의 최소 단위

**애트리뷰트(Attribute, 속성)** 는 릴레이션을 구성하는 **가장 작은 논리적 단위**입니다. 더 이상 쪼갤 수 없는 최소 단위라고 생각하면 됩니다. 마치 원자(Atom)처럼요.

예를 들어 학생 테이블에서 "학번", "이름", "학년", "학과"는 각각 하나의 애트리뷰트입니다. 이것들을 더 작게 쪼갤 수 없죠. 또한, **각 애트리뷰트의 이름은 테이블 안에서 반드시 유일**해야 합니다. 같은 이름의 열이 두 개 존재하면 데이터를 구분할 수 없으니까요.

### 도메인(Domain): 속성이 가질 수 있는 값의 범위

**도메인(Domain)** 이란, 하나의 애트리뷰트가 취할 수 있는 **같은 타입의 원자 값들의 집합**입니다. 쉽게 말하면, "이 칸에 들어올 수 있는 값의 범위"입니다.

예를 들어,
- **학년** 애트리뷰트의 도메인: 1, 2, 3, 4 (정수)
- **이름** 애트리뷰트의 도메인: 문자열 최대 10자
- **국어 점수** 애트리뷰트의 도메인: 0 ~ 100 (정수)
- **수학 점수** 애트리뷰트의 도메인: 0 ~ 300 (정수)

여기서 매우 중요한 개념이 등장합니다. **국어 80점**과 **수학 80점**은 같은 값처럼 보이지만, 도메인이 다르기 때문에 직접 비교하는 것은 의미가 없습니다. 국어는 100점 만점에 80점이고, 수학은 300점 만점에 80점이니까요. 마치 **키(cm)와 몸무게(kg)를 같은 숫자라고 비교하는 것**처럼 잘못된 비교입니다.

:::warning 도메인 설계는 신중하게
도메인을 잘못 설계하면 나중에 데이터 비교나 처리 시 **예상치 못한 오류**가 발생합니다. 처음 설계 단계에서 각 속성의 타입, 범위, 제약 조건을 꼼꼼하게 정의해 두어야 합니다. 잘못 설계하면 나중에 처음부터 다시 설계해야 하는 최악의 상황이 생길 수 있습니다.
:::

### 릴레이션 스키마(Schema)와 인스턴스(Instance)

릴레이션 전체는 두 부분으로 나눌 수 있습니다.

**릴레이션 스키마(Schema, 구조)** 는 릴레이션의 **뼈대**입니다. 다음 요소들로 구성됩니다.
- 릴레이션의 이름 (예: 학생)
- 애트리뷰트의 이름 (예: 학번, 이름, 학년, 학과)
- 각 속성의 도메인과 제약 조건 (예: 학번은 정수형, 중복 불가, NULL 불가)

스키마는 **정적(Static)** 입니다. 건물의 설계도처럼 잘 변하지 않습니다. 스키마를 **내연(Intension)** 이라고도 합니다.

**릴레이션 인스턴스(Instance, 값)** 는 릴레이션에 실제로 저장된 **데이터 값의 집합**입니다. 어느 한 시점에서의 릴레이션 내용을 의미하며, 튜플들의 집합입니다.

인스턴스는 **동적(Dynamic)** 입니다. 학생이 입학하거나 졸업할 때마다 값이 바뀝니다. 인스턴스를 **외연(Extension)** 이라고도 합니다.

> **전체 릴레이션 = 스키마(구조) + 인스턴스(값)**
> 구조만 있어도 안 되고, 값만 있어도 안 됩니다. 두 개가 합쳐져야 비로소 하나의 완전한 릴레이션이 됩니다.

아래 예시를 보면 직관적으로 이해됩니다.

```sql
-- 학생 릴레이션의 스키마 (구조 정의)
-- 릴레이션 이름: 학생
-- 애트리뷰트: 학번(INTEGER), 이름(CHAR(6)), 학년(INTEGER), 학과(CHAR(10))
-- 제약조건: 학번은 기본키 (중복 불가, NULL 불가)

CREATE TABLE 학생 (
    학번   INTEGER PRIMARY KEY,  -- 기본키: 유일하고 NULL 불가
    이름   CHAR(6),              -- 최대 6자리 문자
    학년   INTEGER,              -- 정수형 학년
    학과   CHAR(10)              -- 최대 10자리 문자
);

-- 릴레이션 인스턴스 (실제 데이터 삽입)
INSERT INTO 학생 VALUES (100, '홍길동', 1, '컴퓨터공학');  -- 튜플 1
INSERT INTO 학생 VALUES (200, '이순신', 2, '소프트웨어');  -- 튜플 2
INSERT INTO 학생 VALUES (300, '강감찬', 3, '정보보안');    -- 튜플 3
INSERT INTO 학생 VALUES (400, '유관순', 4, '데이터공학');  -- 튜플 4
INSERT INTO 학생 VALUES (500, '장보고', 2, '컴퓨터공학'); -- 튜플 5
```

위 코드에서 `CREATE TABLE` 부분이 **스키마**, `INSERT INTO` 로 들어간 실제 값들이 **인스턴스**입니다.

## 📐 릴레이션의 4가지 특성

릴레이션은 수학적 집합 개념에서 나왔기 때문에, 일반 테이블과 다른 엄격한 특성을 가집니다. 총 4가지를 반드시 기억하세요.

### 1️⃣ 튜플의 유일성

한 릴레이션에 포함된 **모든 튜플은 서로 다른 값**을 가져야 합니다. 완전히 동일한 행이 두 번 존재할 수 없습니다. 마치 같은 주민등록번호를 가진 사람이 두 명 있을 수 없는 것처럼요.

❌ **잘못된 예시 (튜플 중복)**

| 학번 | 이름 |
|---|---|
| 100 | 홍길동 |
| 100 | 홍길동 |
| 200 | 이순신 |

✅ **올바른 예시 (튜플 유일)**

| 학번 | 이름 |
|---|---|
| 100 | 홍길동 |
| 200 | 이순신 |
| 300 | 강감찬 |

위 잘못된 예시처럼 완전히 동일한 행이 존재하면 **릴레이션의 정의를 위반**한 것입니다.

### 2️⃣ 튜플의 무순서성(無順序性)

릴레이션에서 튜플들 사이에는 **순서가 없습니다**. 학번이 100번인 행이 반드시 200번 행보다 먼저 나와야 한다는 법이 없습니다. 릴레이션은 수학적 집합(Set)이기 때문에, 집합의 원소들 사이에는 순서가 없는 것과 같은 이치입니다.

:::note 실무와의 차이
실제 테이블에서는 `ORDER BY` 구문으로 정렬된 결과를 얻을 수 있습니다. 하지만 이것은 **조회 시 임시로 정렬한 것**이지, 릴레이션 자체에 순서가 있다는 뜻이 아닙니다.
:::

### 3️⃣ 애트리뷰트의 무순서성(無順序性)

튜플뿐만 아니라 **애트리뷰트(열)들 사이에도 순서가 없습니다**. 학번 열이 반드시 이름 열 앞에 나와야 한다는 규칙은 수학적 릴레이션에서는 존재하지 않습니다. 각 애트리뷰트는 독립적인 존재입니다.

다만, 실무 테이블에서는 왼쪽에서 오른쪽으로 "학번 → 이름 → 학년 → 학과" 순으로 의미 있는 순서가 있는 것이 일반적입니다. 이것이 이론적 릴레이션과 실제 테이블의 차이점 중 하나입니다.

### 4️⃣ 애트리뷰트의 원자성(原子性)

모든 애트리뷰트 값은 **원자 값(Atomic Value)**, 즉 **더 이상 쪼갤 수 없는 하나의 단일 값**이어야 합니다.

❌ **잘못된 예시 (하나의 칸에 여러 값)**

| 학번 | 이름 | 점수 |
|---|---|---|
| 100 | 홍길동 | 80, 90, 70 |

✅ **올바른 예시 (하나의 칸에 하나의 값)**

| 학번 | 이름 | 점수 |
|---|---|---|
| 100 | 홍길동 | 80 |

하나의 셀에 여러 값이 있으면, "점수가 80 이상인 학생을 찾아라"라는 조건을 처리할 수 없습니다. 데이터베이스가 그 셀의 값을 어떻게 비교해야 할지 모르기 때문입니다. 따라서 각 셀에는 반드시 하나의 원자 값만 들어가야 합니다.

## 🔀 테이블 vs 릴레이션: 미묘하지만 중요한 차이

앞서 테이블과 릴레이션이 비슷하지만 다르다고 했습니다. 정확히 어떻게 다른지 정리해 봅니다.

| 구분 | 릴레이션 (이론) | 테이블 (실무) |
|---|---|---|
| 중복 행 허용 여부 | ❌ 절대 불가 | ✅ 가능 (필요시) |
| 행의 순서 | 순서 없음 | 일반적으로 순서 있음 |
| 열의 순서 | 순서 없음 | 왼쪽→오른쪽 순서 있음 |
| 개념 출처 | 수학적(집합론) | 실용적(DBMS 구현) |

예를 들어, 릴레이션에서는 튜플의 유일성 규칙 때문에 동일한 행이 두 번 존재할 수 없습니다. 하지만 실제 테이블에서는 특정 목적을 위해 중복된 행이 존재할 수도 있습니다. 또한 실제 성적표 테이블은 보통 학번 순서로 정렬되어 있지만, 릴레이션 이론상으로는 그런 정렬 순서가 의미 없습니다.

> 정리하자면, **릴레이션은 추상적인 수학적 개념**이고, **테이블은 그 릴레이션을 구체적으로 구현한 실무 도구**입니다. 테이블은 릴레이션보다 좀 더 유연합니다.

## 🔗 ER 다이어그램을 릴레이션(테이블)으로 변환하기

### 변환의 필요성과 고려 사항

ER 다이어그램(Entity-Relationship Diagram, 개체-관계 설계도)은 데이터베이스의 **설계 청사진**입니다. 집을 지을 때 설계도가 있듯이, 데이터베이스를 만들기 전에 ER 다이어그램을 먼저 그립니다. 그런데 이 설계도를 실제 테이블로 변환할 때 여러 가지를 고려해야 합니다.

**변환 시 반드시 고려해야 할 사항들:**

- **관계의 종류**: 1대1, 1대다, 다대다 중 어느 것인가?
- **참여 방식**: 전체 참여(의무적, Mandatory/Total)인가, 부분 참여(선택적, Optional/Partial)인가?
- **기본키 선택**: 개체무결성(각 행이 유일하게 식별되는 성질)을 지킬 수 있는가?
- **NULL 값 최소화**: NULL이 많아지면 데이터베이스 성능이 떨어집니다.
- **조인 연산 최소화**: 테이블이 너무 많아지면 테이블을 합치는 조인(Join) 연산이 많아져 성능이 저하됩니다.

:::warning NULL과 조인이 많으면 성능이 떨어집니다
NULL 값이 많거나 테이블이 너무 많아 조인이 빈번하게 발생하면 데이터베이스 성능이 심각하게 저하될 수 있습니다. 테이블 설계 단계에서부터 이를 최소화할 수 있도록 신중하게 설계해야 합니다.
:::

### 이진 관계(두 개체 간 관계) 변환 규칙

#### 1대1 관계 변환

1대1 관계는 참여 방식에 따라 3가지 경우로 나뉩니다.

**① 양쪽 모두 선택적 참여(Optional-Optional)**

예: 남자와 여자가 결혼하는데, 남자 중에도 결혼 안 하는 사람이 있고, 여자 중에도 결혼 안 하는 사람이 있는 경우.

이 경우 **서로의 기본키를 상대방 테이블의 외래키로 추가**합니다. 둘 다 관계에 참여 안 할 수도 있으므로, 결혼하지 않은 쪽에는 NULL이 들어갑니다.

```sql
-- 남자 테이블: 기본키는 남자주민번호
CREATE TABLE 남자 (
    남자주민번호  CHAR(14) PRIMARY KEY,  -- 기본키
    남자이름      VARCHAR(20),
    여자주민번호  CHAR(14),              -- 외래키 (결혼 안 했으면 NULL)
    결혼날짜      DATE                   -- 결혼 관계의 설명 속성
);

-- 여자 테이블: 기본키는 여자주민번호
CREATE TABLE 여자 (
    여자주민번호  CHAR(14) PRIMARY KEY,  -- 기본키
    여자이름      VARCHAR(20),
    남자주민번호  CHAR(14),              -- 외래키 (결혼 안 했으면 NULL)
    결혼날짜      DATE                   -- 결혼 관계의 설명 속성
);
```

**② 한쪽만 의무적 참여(Optional-Mandatory)**

예: 손님과 카드의 관계. 손님은 카드를 발급받을 수도 있고 안 받을 수도 있습니다(선택적 참여). 하지만 카드가 존재한다면 반드시 카드를 발급받은 손님이 있어야 합니다(의무적 참여).

이 경우 **선택적 참여 쪽의 기본키를 의무적 참여 쪽의 외래키로 넣습니다.** 이렇게 해야 NULL 값을 줄일 수 있습니다.

왜 그럴까요? 만약 반대로 카드 번호를 손님 테이블에 넣으면, 카드를 발급받지 않은 손님들의 카드 번호 칸에 모두 NULL이 들어갑니다. 반대로 손님 번호를 카드 테이블에 넣으면, 카드는 반드시 손님이 있어야 발급되므로 NULL이 전혀 생기지 않습니다.

```sql
-- 손님 테이블 (선택적 참여)
CREATE TABLE 손님 (
    손님번호  INTEGER PRIMARY KEY,  -- 기본키
    손님이름  VARCHAR(20)
);

-- 카드 테이블 (의무적 참여: 카드가 있으면 반드시 손님이 있어야 함)
CREATE TABLE 카드 (
    카드번호    CHAR(16) PRIMARY KEY,  -- 기본키
    카드종류    VARCHAR(20),
    신용한도    INTEGER,
    손님번호    INTEGER NOT NULL,      -- 외래키 (손님 테이블의 기본키 참조, NULL 불가)
    FOREIGN KEY (손님번호) REFERENCES 손님(손님번호)
);
-- 손님번호를 카드 쪽에 넣어야 NULL이 생기지 않습니다!
```

**③ 양쪽 모두 의무적 참여(Mandatory-Mandatory)**

예: 손님과 선적 정보가 무조건 1대1로 대응되어야 하는 경우(손님이 있으면 반드시 선적 정보가 있고, 선적 정보가 있으면 반드시 손님이 있는 경우).

이 경우 **두 테이블을 하나로 합칩니다.** 어차피 항상 같이 있어야 하므로, 하나의 테이블로 만드는 것이 더 효율적입니다.

```sql
-- 손님 + 선적정보를 하나의 테이블로 합침
-- 둘 다 의무적 참여이므로 테이블을 합치는 것이 효율적
CREATE TABLE 손님_선적정보 (
    손님번호    INTEGER PRIMARY KEY,   -- 손님의 기본키
    우편번호    CHAR(5),               -- 선적정보의 속성
    선적주소    VARCHAR(100),          -- 선적정보의 속성
    손님이름    VARCHAR(20)            -- 손님의 속성
);
```

#### 1대다(1:N) 관계 변환

**1대다 관계**는 실무에서 가장 자주 등장하는 관계입니다. 예를 들어, 한 명의 판매원이 여러 개의 주문을 기재하는 경우입니다.

규칙: **1측의 기본키를 N측(다 측)의 외래키로 넣습니다.**

왜 반대로 하면 안 될까요? 만약 N측의 기본키(주문번호)를 1측(판매원) 테이블에 넣으면, 한 명의 판매원이 여러 주문을 처리하므로 판매원의 행이 여러 번 중복됩니다. 그러면 판매원의 기본키가 중복되어 **기본키 유일성 원칙을 위반**하게 됩니다.

❌ **잘못된 예시 (기본키 중복 발생)**

```sql
-- 이렇게 하면 절대 안 됩니다!
-- 판매원 '김영희'가 A1, A2, A3 세 개 주문을 처리했다면...
-- 기본키인 '판매원이름'이 중복됩니다 → 기본키 유일성 위반!

-- 잘못된 구조:
-- 판매원이름(PK) | 판매원전화 | 주문번호(FK)
-- 김영희         | 010-1234   | A1          ← 기본키 중복!
-- 김영희         | 010-1234   | A2          ← 기본키 중복!
-- 김영희         | 010-1234   | A3          ← 기본키 중복!
```

✅ **올바른 예시 (1측 기본키 → N측 외래키)**

```sql
-- 판매원 테이블 (1측)
CREATE TABLE 판매원 (
    판매원이름  VARCHAR(20) PRIMARY KEY,  -- 1측 기본키
    판매원전화  CHAR(13)
);

-- 주문 테이블 (N측): 1측의 기본키를 외래키로 가져옴
CREATE TABLE 주문 (
    주문번호    CHAR(10) PRIMARY KEY,     -- N측 기본키
    주문날짜    DATE,
    주문금액    INTEGER,
    판매원이름  VARCHAR(20),              -- 1측의 기본키를 외래키로 추가 ← 핵심!
    FOREIGN KEY (판매원이름) REFERENCES 판매원(판매원이름)
);
-- 이렇게 하면 주문 테이블에서 '이 주문을 기재한 판매원이 누구인지' 바로 알 수 있습니다.
```

#### 다대다(M:N) 관계 변환

**다대다 관계**는 양쪽 어느 테이블에도 외래키를 직접 넣을 수 없습니다. 예를 들어, 한 학생이 여러 과목을 수강하고, 한 과목에 여러 학생이 등록하는 경우입니다.

규칙: **별도의 새로운 테이블을 만들고, 양쪽의 기본키를 가져와 복합키(Composite Key, 두 개 이상의 속성으로 이루어진 기본키)로 만듭니다.**

```sql
-- 학생 테이블 (M측)
CREATE TABLE 학생 (
    학생번호  INTEGER PRIMARY KEY,  -- 기본키
    학과      VARCHAR(20)
);

-- 과목 테이블 (N측)
CREATE TABLE 과목 (
    과목번호  CHAR(10) PRIMARY KEY,  -- 기본키
    과목이름  VARCHAR(50)
);

-- 등록 테이블 (다대다 관계를 위한 별도 테이블)
-- 양쪽의 기본키를 가져와 복합키로 구성
CREATE TABLE 등록 (
    학생번호  INTEGER,               -- 학생 테이블의 기본키를 외래키로 참조
    과목번호  CHAR(10),              -- 과목 테이블의 기본키를 외래키로 참조
    학점      CHAR(2),               -- 등록 관계의 설명 속성 (예: A+, B0 등)
    PRIMARY KEY (학생번호, 과목번호), -- 두 개를 합쳐 복합키로 기본키 구성
    FOREIGN KEY (학생번호) REFERENCES 학생(학생번호),
    FOREIGN KEY (과목번호) REFERENCES 과목(과목번호)
);
```

### 순환 관계성(Recursive Relationship) 변환

**순환 관계성**이란 **같은 개체(동일한 테이블)** 가 자기 자신과 관계를 맺는 경우입니다. 예를 들어, 직원 테이블에서 "상사가 부하직원을 감독한다"는 관계처럼 직원끼리 관계를 맺는 경우입니다. 이때 양쪽 모두 같은 속성을 가진 동일한 개체입니다.

방법은 이진 관계와 완전히 동일합니다. 차이는 테이블이 1개 덜 만들어진다는 것입니다.

**1대다 순환 관계:**

```sql
-- 직원 테이블 하나만 만들어집니다
CREATE TABLE 직원 (
    직원번호    INTEGER PRIMARY KEY,  -- 기본키
    직원이름    VARCHAR(20),
    부서        VARCHAR(20),
    상사직원번호 INTEGER,             -- 외래키: 1측(상사)의 기본키를 N측(부하)에 넣음
                                      -- 상사가 없는 최상위 직원은 NULL
    FOREIGN KEY (상사직원번호) REFERENCES 직원(직원번호)
    -- 자기 자신을 참조하는 외래키 (셀프 조인용)
);
```

**다대다 순환 관계:**

```sql
-- 직원 테이블
CREATE TABLE 직원 (
    직원번호  INTEGER PRIMARY KEY,
    직원이름  VARCHAR(20)
);

-- 감독 관계 테이블 (별도 생성)
CREATE TABLE 감독 (
    상사직원번호  INTEGER,    -- 상사 측의 직원번호
    부하직원번호  INTEGER,    -- 부하 측의 직원번호
    PRIMARY KEY (상사직원번호, 부하직원번호),  -- 복합키
    FOREIGN KEY (상사직원번호) REFERENCES 직원(직원번호),
    FOREIGN KEY (부하직원번호) REFERENCES 직원(직원번호)
);
```

### 3진 이상 관계(Ternary Relationship) 변환

**3진 관계**란 세 개의 개체가 동시에 하나의 관계에 참여하는 경우입니다. 예를 들어, 사원이 프로젝트에 부품을 공급하는 관계처럼 세 개체가 동시에 얽히는 경우입니다.

규칙: **무조건 별도의 테이블을 만들고, 세 개체의 기본키를 모두 가져와 복합키로 구성합니다.** 이 경우 총 4개의 테이블이 만들어집니다 (개체 3개 + 관계 테이블 1개).

```sql
-- 사원 테이블
CREATE TABLE 사원 (
    공급자번호  INTEGER PRIMARY KEY,
    사원이름    VARCHAR(20)
);

-- 부품 테이블
CREATE TABLE 부품 (
    부품번호  INTEGER PRIMARY KEY,
    부품이름  VARCHAR(50)
);

-- 프로젝트 테이블
CREATE TABLE 프로젝트 (
    프로젝트번호  INTEGER PRIMARY KEY,
    프로젝트명    VARCHAR(100)
);

-- 공급 관계 테이블 (3진 관계를 위한 별도 테이블)
-- 세 개체의 기본키를 모두 가져와 복합키 구성
CREATE TABLE 공급 (
    공급자번호    INTEGER,    -- 사원 테이블 기본키 참조
    부품번호      INTEGER,    -- 부품 테이블 기본키 참조
    프로젝트번호  INTEGER,    -- 프로젝트 테이블 기본키 참조
    물량          INTEGER,    -- 공급 관계의 설명 속성
    PRIMARY KEY (공급자번호, 부품번호, 프로젝트번호), -- 세 개 복합키
    FOREIGN KEY (공급자번호) REFERENCES 사원(공급자번호),
    FOREIGN KEY (부품번호) REFERENCES 부품(부품번호),
    FOREIGN KEY (프로젝트번호) REFERENCES 프로젝트(프로젝트번호)
);
```

### 특수한 경우의 변환 규칙

#### 일반화(Generalization): 슈퍼클래스와 서브클래스

객체지향 프로그래밍의 상속 개념과 비슷합니다. 예를 들어 **직원** 테이블이 있고, 그 하위에 **생산 직원**과 **영업 직원**이 있는 경우입니다.

규칙: **슈퍼클래스(상위)의 기본키를 서브클래스(하위)에서 그대로 상속받아 사용합니다.**

```sql
-- 슈퍼클래스: 직원 테이블
CREATE TABLE 직원 (
    직원번호  INTEGER PRIMARY KEY,  -- 슈퍼클래스 기본키
    직원이름  VARCHAR(20),
    호봉      INTEGER
);

-- 서브클래스: 생산직원 (슈퍼클래스의 기본키를 상속)
CREATE TABLE 생산직원 (
    직원번호    INTEGER PRIMARY KEY,  -- 슈퍼클래스의 기본키를 그대로 사용
    기술분야    VARCHAR(50),
    FOREIGN KEY (직원번호) REFERENCES 직원(직원번호)
);

-- 서브클래스: 영업직원 (슈퍼클래스의 기본키를 상속)
CREATE TABLE 영업직원 (
    직원번호    INTEGER PRIMARY KEY,  -- 슈퍼클래스의 기본키를 그대로 사용
    최대할인율  DECIMAL(5,2),
    FOREIGN KEY (직원번호) REFERENCES 직원(직원번호)
);
-- 총 3개의 테이블이 만들어집니다
```

#### 약한 개체 타입(Weak Entity Type)

**약한 개체**란 자기 자신만으로는 유일하게 식별될 수 없고, **강한 개체(Strong Entity)의 기본키에 의존**해야 하는 개체입니다. 예를 들어, 사원의 부양가족을 생각해 보세요. 부양가족의 이름은 "민준"일 수 있는데, 다른 사원의 부양가족도 "민준"일 수 있습니다. 따라서 부양가족 이름만으로는 누구의 부양가족인지 알 수 없습니다.

규칙: **강한 개체의 기본키를 약한 개체 테이블에 외래키로 가져와, 약한 개체의 부분키(Partial Key)와 함께 복합키로 만듭니다.**

```sql
-- 강한 개체: 사원 테이블
CREATE TABLE 사원 (
    사원번호  INTEGER PRIMARY KEY,
    사원이름  VARCHAR(20)
);

-- 약한 개체: 부양가족 테이블
-- 사원번호 없이는 유일하게 식별 불가능
CREATE TABLE 부양가족 (
    사원번호    INTEGER,         -- 강한 개체의 기본키를 외래키로 가져옴
    부양가족이름 VARCHAR(20),    -- 약한 개체의 부분키 (단독으로는 유일성 보장 불가)
    관계        VARCHAR(10),
    PRIMARY KEY (사원번호, 부양가족이름),  -- 복합키로 유일성 보장
    FOREIGN KEY (사원번호) REFERENCES 사원(사원번호)
);
```

#### 다중값 속성(Multi-valued Attribute)

**다중값 속성**이란 하나의 속성이 여러 값을 가질 수 있는 경우입니다. ER 다이어그램에서 이중 타원(동그라미 두 개)으로 표시됩니다. 예를 들어, 부서의 위치는 서울, 부산, 대전 등 여러 곳일 수 있습니다.

릴레이션의 **원자성 원칙** 때문에 하나의 셀에 여러 값을 넣을 수 없으므로, 다중값 속성은 **반드시 별도 테이블**로 분리해야 합니다.

```sql
-- 부서 테이블 (다중값 속성인 '위치'는 제외)
CREATE TABLE 부서정보 (
    부서번호  INTEGER PRIMARY KEY,
    부서이름  VARCHAR(50),
    매니저    VARCHAR(20)
    -- 위치 속성은 여기에 넣지 않습니다!
);

-- 부서 위치 테이블 (다중값 속성을 별도 테이블로 분리)
CREATE TABLE 부서위치 (
    부서번호  INTEGER,        -- 부서정보 테이블의 기본키 참조
    위치      VARCHAR(50),    -- 다중값 속성 값
    PRIMARY KEY (부서번호, 위치),  -- 복합키
    FOREIGN KEY (부서번호) REFERENCES 부서정보(부서번호)
);

-- 예: 1번 부서가 서울, 부산, 대전에 있다면
-- INSERT INTO 부서위치 VALUES (1, '서울');
-- INSERT INTO 부서위치 VALUES (1, '부산');
-- INSERT INTO 부서위치 VALUES (1, '대전');
```

:::danger 유도 속성은 테이블에 저장하지 마세요
ER 다이어그램에서 **점선 타원**으로 표시된 **유도 속성(Derived Attribute)** 은 다른 속성으로부터 계산해 낼 수 있는 값입니다. 예를 들어 나이는 생년월일로부터 계산할 수 있습니다. 유도 속성은 **실제 테이블에 저장하지 않습니다.** 저장하면 중복 데이터가 생기고 데이터 불일치 문제가 발생할 수 있습니다.
:::

## 📊 ER → 릴레이션 변환 규칙 종합 요약표

지금까지 배운 모든 변환 규칙을 한 눈에 정리해 봅니다.

| 상황 | 변환 방법 | 기본키 구성 | 테이블 수 |
|---|---|---|---|
| 1:1 (양쪽 선택적) | 서로의 기본키를 외래키로 교환 | 각자의 기본키 유지 | 2개 |
| 1:1 (한쪽 의무적) | 선택적 측 기본키 → 의무적 측 외래키 | 각자의 기본키 유지 | 2개 |
| 1:1 (양쪽 의무적) | 두 테이블 합침 | 양쪽 기본키를 모두 포함 | 1개 |
| 1:N (이진) | 1측 기본키 → N측 외래키 | 각자의 기본키 유지 | 2개 |
| M:N (이진) | 별도 테이블 생성, 양쪽 기본키 → 복합키 | 양쪽 기본키의 복합키 | 3개 |
| 1:N (순환) | 1측 기본키 → 같은 테이블에 외래키 | 기존 기본키 유지 | 1개 |
| M:N (순환) | 별도 관계 테이블 생성 | 복합키 | 2개 |
| 3진 이상 관계 | 무조건 별도 테이블 생성 | 참여 개체의 복합키 | 개체 수+1개 |
| 일반화 (상속) | 슈퍼클래스 기본키 → 서브클래스 외래키 | 상속받은 기본키 | 1+서브클래스 수 |
| 약한 개체 | 강한 개체 기본키 → 약한 개체 외래키 | 복합키 | 2개 |
| 다중값 속성 | 별도 테이블 분리 | 상위 기본키+속성값 복합키 | 기존+1개 |
| 유도 속성 | 테이블에 저장하지 않음 | 해당 없음 | 변화 없음 |

## ⚠️ 주의사항과 실전 팁

### NULL 값 최소화 전략

NULL 값은 "값이 없음"을 의미하는데, NULL이 많으면 여러 문제가 생깁니다.

1. **저장 공간 낭비**: NULL도 저장 공간을 차지합니다.
2. **처리 복잡도 증가**: NULL과의 비교는 특별한 처리가 필요합니다 (IS NULL, IS NOT NULL).
3. **성능 저하**: NULL이 많은 컬럼은 인덱스 효율이 떨어집니다.
4. **데이터 무결성 위협**: NULL의 의미가 모호해질 수 있습니다.

따라서 1대1 관계에서 선택적-의무적 참여가 있을 때, 항상 **선택적 참여 쪽의 기본키를 의무적 참여 쪽의 외래키로** 넣어서 NULL을 줄여야 합니다.

### 조인 연산 최소화 전략

테이블이 너무 많아지면 원하는 정보를 얻기 위해 여러 테이블을 합치는 **조인(Join)** 연산이 많아집니다. 조인은 비용이 큰 연산이므로, 불필요한 테이블 분리를 피하는 것이 좋습니다.

> 실무에서는 정규화(데이터 중복을 없애기 위한 테이블 분리)와 성능(조인 최소화를 위한 테이블 통합) 사이에서 균형을 잡는 것이 매우 중요한 설계 역량입니다.

### 기본키 선택 원칙

:::tip 좋은 기본키의 조건
1. **유일성**: 모든 행을 유일하게 식별할 수 있어야 합니다.
2. **최소성**: 기본키를 구성하는 속성은 가능한 한 적어야 합니다.
3. **불변성**: 기본키 값은 자주 바뀌지 않아야 합니다. (이름, 전화번호보다는 학번, 사원번호 권장)
4. **비NULL**: 기본키에는 NULL이 들어갈 수 없습니다 (개체 무결성).
:::

## 🎯 핵심 정리

- **관계 데이터**는 테이블 형태로 개체와 개체 간의 관계를 표현하며, **기본키(Primary Key)** 와 **외래키(Foreign Key)** 로 테이블 간 관계를 맺는다.
- **릴레이션(Relation)** 은 테이블의 수학적 개념이며, 이론에서는 릴레이션, 실무에서는 테이블이라는 용어를 주로 사용한다.
- **튜플(Tuple)** 은 행(Row), **애트리뷰트(Attribute)** 는 열(Column)에 해당하는 릴레이션 용어이다.
- **차수(Degree)** 는 속성의 수로 정적(Static)이며, **카디널리티(Cardinality)** 는 튜플의 수로 동적(Dynamic)이다.
- **도메인(Domain)** 은 속성이 취할 수 있는 값의 범위이며, 도메인이 다른 속성끼리 비교하는 것은 의미가 없다.
- **릴레이션 스키마** 는 구조(정적), **릴레이션 인스턴스** 는 실제 값(동적)이며, 둘을 합쳐야 전체 릴레이션이 된다.
- 릴레이션의 4대 특성: **튜플의 유일성**, **튜플의 무순서성**, **애트리뷰트의 무순서성**, **애트리뷰트의 원자성(단일 값)**
- **1대1 관계** 변환 시, 참여 방식(선택/의무)에 따라 외래키 방향이 달라지며 NULL 최소화가 기준이다.
- **1대다 관계** 에서는 반드시 **1측의 기본키 → N측의 외래키**로 넣어야 기본키 유일성을 지킬 수 있다.
- **다대다 관계** 는 반드시 **별도 테이블을 생성**하고 양쪽 기본키를 복합키로 만들어야 한다.
- **순환 관계** 는 이진 관계와 방법은 같지만, 개체가 하나이므로 테이블 수가 하나 줄어든다.
- **3진 이상 관계** 는 무조건 별도 관계 테이블을 생성하고, 참여 개체들의 기본키를 복합키로 구성한다.
- **약한 개체** 는 강한 개체의 기본키를 가져와 복합키로 유일성을 보장해야 한다.
- **다중값 속성** 은 원자성 원칙에 따라 반드시 별도 테이블로 분리해야 한다.
- **유도 속성** 은 다른 값으로 계산 가능하므로 테이블에 저장하지 않는다.
- NULL 값과 조인 연산이 많아질수록 데이터베이스 성능이 저하되므로, 설계 단계에서 최소화 전략을 세워야 한다.
