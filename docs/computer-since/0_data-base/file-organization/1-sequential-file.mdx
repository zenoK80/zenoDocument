# 📂 파일 조직 방법: 순차 파일 구조부터 해싱까지 완전 정복

데이터베이스나 운영체제를 공부하다 보면 "파일을 어떻게 저장할까?"라는 질문을 반드시 마주치게 됩니다. 단순히 데이터를 저장하는 것처럼 보여도, 실제로는 **어떤 방식으로 저장하느냐에 따라 검색 속도, 저장 효율, 유지 관리 비용이 완전히 달라집니다.** 이 문서에서는 파일을 조직하는 세 가지 핵심 방법인 **순차 구조**, **인덱스 구조**, **해싱 구조**를 처음 배우는 분도 이해할 수 있도록 차근차근 설명합니다.

- B-Tree, B\*-Tree, B+-Tree의 개념과 삽입/삭제 원리

## 🗂️ 파일 조직이란 무엇인가?

### 왜 파일 조직 방법이 중요한가?

파일 조직(File Organization)이란 **데이터(레코드)를 저장 매체에 어떤 순서와 방식으로 배치할 것인가**를 결정하는 방법입니다. 마치 도서관에서 책을 정리하는 방식을 생각해 보세요. 책을 아무 자리에나 꽂아두면 나중에 찾을 때 처음부터 끝까지 모두 살펴봐야 합니다. 하지만 장르별, 저자 이름순으로 정리해 두면 훨씬 빠르게 원하는 책을 찾을 수 있죠. 데이터베이스도 똑같습니다.

저장 방식에 따라 크게 **세 가지**로 나눌 수 있습니다.

| 조직 방법 | 핵심 개념 | 특징 |
|---|---|---|
| **순차 구조** | 레코드를 순서대로 저장 | 단순하지만 검색이 느릴 수 있음 |
| **인덱스 구조** | 목차(인덱스)를 통해 접근 | 빠른 검색, 책의 목차와 유사 |
| **해싱 구조** | 수식(함수)으로 주소를 직접 계산 | 가장 빠른 직접 접근 가능 |

이 세 가지 방법은 각각 장단점이 있으며, 상황에 따라 적합한 방법을 선택해야 합니다. 지금부터 하나씩 자세하게 살펴보겠습니다.

## 📋 순차 파일 구조: 엔트리 순차와 키 순차

### 순차 파일이란?

순차 파일(Sequential File)은 이름 그대로 **레코드를 일렬로 순서대로 저장하는 방식**입니다. 마치 줄 서기처럼, 먼저 들어온 데이터가 앞쪽에 위치하거나, 특정 기준으로 정렬하여 저장합니다. 순차 파일은 다시 두 가지 유형으로 나뉩니다.

### 엔트리 순차 파일 (Entry-Sequenced File)

**엔트리 순차 파일**은 레코드가 도착하는 순서 그대로 저장하는 방식입니다. 어떤 분석, 분류, 정렬 과정도 거치지 않습니다. 마치 택배가 창고에 도착하는 순서대로 그냥 쌓아두는 것과 같습니다. 먼저 도착한 택배가 먼저 쌓이고, 나중에 도착한 택배가 그 위에 쌓이는 것처럼요.

예를 들어 학번이 `1번, 3번, 2번, 5번, 6번, 4번` 순서로 데이터가 입력된다면, **엔트리 순차 파일**에는 다음과 같이 저장됩니다.

```
저장 순서: [1번] → [3번] → [2번] → [5번] → [6번] → [4번]
```

입력된 순서 그대로 저장됩니다. 데이터를 정렬하거나 재배치하는 과정이 없으므로 **저장 속도는 빠르지만**, 특정 데이터를 찾으려면 처음부터 끝까지 모두 살펴봐야 하는 단점이 있습니다.

:::note 엔트리 순차 파일 정리
- 레코드를 **입력된 순서 그대로** 저장
- 정렬, 분류, 분석 과정 **없음**
- 저장은 빠르지만, 검색 시 처음부터 끝까지 탐색 필요
:::

### 키 순차 파일 (Key-Sequenced File)

**키 순차 파일**은 데이터가 들어올 때 특정 필드(이를 **키 필드(Key Field)**라고 합니다)를 기준으로 **정렬하여 저장**하는 방식입니다. 마치 도서관 사서가 책이 반납될 때마다 청구기호 순서에 맞게 꽂아두는 것처럼요. 항상 정렬된 상태를 유지합니다.

같은 예시로, 학번이 `1번, 3번, 2번, 5번, 6번, 4번` 순서로 데이터가 입력된다면, **키 순차 파일**에는 다음과 같이 저장됩니다.

```
입력: 1번 → 저장: [1번]
입력: 3번 → 저장: [1번] → [3번]
입력: 2번 → 2번 < 3번이므로 위치 교환 → 저장: [1번] → [2번] → [3번]
입력: 5번 → 저장: [1번] → [2번] → [3번] → [5번]
입력: 6번 → 저장: [1번] → [2번] → [3번] → [5번] → [6번]
입력: 4번 → 4번은 3번보다 크고 5번보다 작으므로 중간에 삽입
       → 저장: [1번] → [2번] → [3번] → [4번] → [5번] → [6번]
```

결과적으로 항상 오름차순으로 정렬된 상태가 유지됩니다.

> 💡 **핵심 차이점**: 엔트리 순차는 "들어온 순서대로 그냥 저장", 키 순차는 "들어올 때마다 키 값 기준으로 정렬하여 저장"입니다.

| 구분 | 엔트리 순차 | 키 순차 |
|---|---|---|
| 저장 방식 | 입력 순서대로 | 키 필드 기준 정렬 |
| 정렬 여부 | ❌ 정렬 없음 | ✅ 항상 정렬됨 |
| 저장 속도 | 빠름 | 상대적으로 느림 (정렬 필요) |
| 검색 효율 | 낮음 | 비교적 높음 |

```python
# 엔트리 순차 파일 방식 (파이썬 예시)
entry_sequential = []

# 학번 데이터가 무작위로 들어옴
incoming_data = [1, 3, 2, 5, 6, 4]

for record in incoming_data:
    entry_sequential.append(record)  # 그냥 뒤에 추가 (정렬 없음)

print("엔트리 순차:", entry_sequential)
# 출력: [1, 3, 2, 5, 6, 4] → 입력 순서 그대로

# 키 순차 파일 방식
key_sequential = []

for record in incoming_data:
    key_sequential.append(record)       # 추가 후
    key_sequential.sort()               # 키 값 기준으로 정렬

print("키 순차:", key_sequential)
# 출력: [1, 2, 3, 4, 5, 6] → 항상 정렬된 상태
```

:::tip 언제 어떤 방식을 쓸까?
- **엔트리 순차**: 데이터 저장 순서가 중요하거나, 로그(log)처럼 시간 순서대로 기록해야 할 때
- **키 순차**: 자주 검색이 필요하고, 특정 기준으로 정렬된 데이터가 필요할 때
:::

## 📑 인덱스 구조: 목차로 데이터 찾기

### 인덱스란 무엇인가?

인덱스(Index)는 우리가 책에서 보는 **목차(Table of Contents)**와 완전히 동일한 개념입니다. 책이 1,000페이지라고 생각해 보세요. 특정 내용을 찾으려면 두 가지 방법이 있습니다.

1. **처음부터 끝까지 읽기**: 1페이지부터 1,000페이지까지 모두 확인 → 시간이 엄청나게 걸림
2. **목차를 먼저 확인**: 목차에서 해당 내용의 페이지 번호를 찾고, 바로 그 페이지로 이동 → 훨씬 빠름

데이터베이스의 인덱스도 마찬가지입니다. **인덱스는 `키 값(Key)`과 `주소(Address)`의 쌍으로 구성**되어 있습니다. 원하는 데이터를 찾으려면 먼저 인덱스로 가서 해당 키 값의 주소를 확인하고, 그 주소로 바로 이동합니다.

```
[인덱스 파일]          [데이터 파일]
키 값  | 주소          주소  | 데이터
-------|------          ------|--------
 k1   |  →  ─────────→  주소A | k1 데이터
 k2   |  →  ─────────→  주소B | k2 데이터
 k3   |  →  ─────────→  주소C | k3 데이터
```

> 🔑 **인덱스의 핵심 원리**: 데이터를 찾을 때 **반드시 인덱스를 먼저 거쳐서** 실제 데이터 위치로 이동합니다.

### 밀집 인덱스 (Dense Index)

밀집 인덱스는 **파일에 있는 모든 레코드의 키 값에 대해 인덱스 항목을 빠짐없이 가지고 있는** 인덱스입니다. 즉, 데이터 레코드가 100개라면 인덱스 항목도 100개가 있어야 합니다.

도서관 비유로 설명하면, 도서관에 책이 1,000권 있을 때 그 1,000권 모두에 대한 목록 카드가 빠짐없이 정리되어 있는 카드 목록함이 바로 밀집 인덱스입니다.

```
[밀집 인덱스 예시]

인덱스 파일:              데이터 파일:
Brighton  → ──────────→  Brighton 레코드
Downtown  → ──────────→  Downtown 레코드
Mianus    → ──────────→  Mianus 레코드
Perryridge→ ──────────→  Perryridge 레코드
Redwood   → ──────────→  Redwood 레코드
Roundhill → ──────────→  Roundhill 레코드
```

모든 데이터(Brighton, Downtown, Mianus, Perryridge, Redwood, Roundhill)에 대한 인덱스가 빠짐없이 존재합니다.

**Perryridge를 찾는 과정:**
1. 인덱스 파일에서 `Perryridge` 항목을 찾음
2. 해당 항목이 가리키는 주소로 이동
3. 해당 주소의 레코드부터 순차적으로 탐색하여 원하는 데이터 발견

:::note 밀집 인덱스의 특징
- 모든 검색 키 값에 대해 인덱스 항목 존재
- 어떤 데이터든 반드시 인덱스에서 바로 찾을 수 있음
- 인덱스 파일의 크기가 데이터 양에 비례하여 커짐
:::

### 희소 인덱스 (Sparse Index)

희소 인덱스는 밀집 인덱스와 달리 **모든 레코드가 아닌, 일부 레코드에 대해서만 인덱스 항목을 가지고 있는** 방식입니다. 예를 들어 데이터가 6개인데 인덱스는 3개만 있는 것입니다.

```
[희소 인덱스 예시]

인덱스 파일:              데이터 파일:
Brighton  → ──────────→  Brighton 레코드
Mianus    → ──────────→  Downtown 레코드  (인덱스 없음)
Redwood   → ──────────→  Mianus 레코드
                          Perryridge 레코드 (인덱스 없음)
                          Redwood 레코드
                          Roundhill 레코드  (인덱스 없음)
```

데이터는 6개지만 인덱스는 3개(Brighton, Mianus, Redwood)만 존재합니다.

**Perryridge를 희소 인덱스에서 찾는 방법:**

인덱스는 항상 정렬되어 있습니다(책의 목차가 정렬되어 있듯이). Perryridge에 대한 인덱스 항목이 없을 때는 다음 규칙을 적용합니다.

> **"찾으려는 키 값보다 작거나 같은 값 중에서 가장 큰 인덱스 항목을 찾아 그 위치에서부터 순차 탐색"**

알파벳 순서로 `B(Brighton) < D(Downtown) < M(Mianus) < P(Perryridge)` 이므로, Perryridge보다 작거나 같은 인덱스 중 가장 큰 것은 **Mianus**입니다. 따라서 Mianus 인덱스가 가리키는 위치로 이동한 뒤, 그곳부터 순차적으로 탐색하여 Perryridge를 찾습니다.

| 구분 | 밀집 인덱스 | 희소 인덱스 |
|---|---|---|
| 인덱스 항목 수 | 모든 레코드에 대응 | 일부 레코드만 대응 |
| 인덱스 파일 크기 | 큼 | 작음 |
| 검색 방법 | 인덱스에서 바로 찾음 | 근접 항목 찾고 순차 탐색 |
| 검색 속도 | 빠름 | 상대적으로 느릴 수 있음 |

### 다단계 인덱스 (Multi-level Index)

다단계 인덱스는 **인덱스를 여러 계층으로 구성**하는 방식입니다. 영어 사전에서 단어를 찾는 방법을 생각해 보세요.

- 1단계: 알파벳 첫 글자 `K` 탭으로 이동
- 2단계: `Ko`로 시작하는 페이지로 이동
- 3단계: `Kor`로 시작하는 단어들 사이에서 `Korea` 탐색

이처럼 범위를 점점 좁혀가면서 원하는 데이터를 찾는 것이 다단계 인덱스입니다.

```
[다단계 인덱스 구조]

1단계 인덱스 (최상위)
  ├── A~M 블록 → 2단계 인덱스 블록1
  └── N~Z 블록 → 2단계 인덱스 블록2

2단계 인덱스 블록1
  ├── A~F 블록 → 3단계 인덱스 (실제 데이터 근처)
  └── G~M 블록 → 3단계 인덱스 (실제 데이터 근처)

3단계 인덱스
  └── 실제 데이터 파일
```

> 💡 인덱스 단계가 늘어날수록 한 번에 건너뛸 수 있는 범위가 넓어지므로, **검색 단계 수(높이)를 줄여 성능을 높일 수 있습니다.**

### 정적 인덱스 vs 동적 인덱스

인덱스 구조는 변화 여부에 따라 정적과 동적으로도 나눌 수 있습니다.

#### 정적 인덱스 (Static Index)

정적 인덱스는 **인덱스의 구조 자체는 변경되지 않는** 방식입니다. 인덱스 안의 내용(키 값, 주소)은 레코드가 삽입되고 삭제될 때마다 바뀔 수 있지만, 인덱스의 틀(구조) 자체는 고정되어 있습니다. 대표적인 예로 **ISAM(Indexed Sequential Access Method)** 파일이 있습니다.

구조가 고정되어 있으니, 저장 공간이 꽉 차버리면 **오버플로우(Overflow, 저장 공간 초과)** 가 발생합니다. 이를 처리하기 위해 별도의 오버플로우 영역이 필요합니다. 오버플로우 영역도 꽉 차면 **주기적으로 파일 재조직(File Reorganization)**을 통해 공간을 확보해야 합니다.

**팬 아웃(Fan-out)과 인덱스 높이(Height)**

정적 인덱스의 성능을 나타내는 두 가지 지표가 있습니다.

- **팬 아웃(Fan-out)**: 인덱스에서 옆으로 뻗어나가는 가지의 수 (가로 방향)
- **인덱스 높이(Height)**: 인덱스의 레벨 수 (세로 방향)

우유 배달을 예시로 설명하면, 한 번에 여러 집에 배달할 수 있는 배달부(팬 아웃이 큰 경우)가, 한 집씩 순서대로 찾아가는 배달부(높이가 큰 경우)보다 훨씬 효율적입니다.

> **팬 아웃이 크면 → 인덱스 높이가 낮아짐 → 검색 성능이 좋아짐**
> **팬 아웃이 작으면 → 인덱스 높이가 높아짐 → 더 많은 단계를 거쳐야 하므로 성능이 떨어짐**

#### 동적 인덱스 (Dynamic Index)

동적 인덱스는 **데이터가 삽입되거나 삭제될 때 인덱스 구조 자체가 자동으로 변하는** 방식입니다. 마치 세포 분열처럼, 공간이 꽉 차면 인덱스가 두 개로 분열(Split)되고, 공간이 너무 비면 다시 합쳐(Merge)집니다. 대표적인 예로 **B+-Tree** 가 있습니다.

동적 인덱스는 오버플로우 영역이 별도로 필요 없고, 구조가 자동으로 조정되므로 유지 관리가 편리합니다.

| 구분 | 정적 인덱스 | 동적 인덱스 |
|---|---|---|
| 구조 변경 | 변경 안 됨 | 자동 변경 |
| 오버플로우 영역 | 필요함 | 불필요 |
| 파일 재조직 | 주기적으로 필요 | 불필요 |
| 대표 예 | ISAM | B+-Tree |

## 🌳 B-Tree, B\*-Tree, B+-Tree: 동적 인덱스의 핵심

### B-Tree란 무엇인가?

B-Tree(Balanced Tree, 균형 트리)는 **모든 리프 노드(Leaf Node, 가장 말단의 노드)가 동일한 높이(레벨)를 가지는 트리 구조**입니다. 여기서 'B'는 'Balanced(균형)'의 약자입니다.

일반 트리는 데이터가 한쪽으로 치우쳐 높이가 불균형해질 수 있습니다. 하지만 B-Tree는 항상 균형을 유지하기 때문에, 루트(Root, 최상단 노드)에서 임의의 리프 노드까지의 거리가 모두 동일합니다. 이는 **어떤 데이터를 검색해도 같은 시간이 걸린다**는 것을 의미합니다.

```
[균형이 깨진 트리 - B-Tree 아님 ❌]
        1
       /
      2
     /
    3
   /
  4

[균형 잡힌 B-Tree ✅]
        2
       / \
      1   3
```

**B-Tree의 차수(Order)**

B-Tree에서 **차수(Order)**는 하나의 노드가 가질 수 있는 자식 노드의 최대 수를 의미합니다. 차수가 5인 B-Tree라면:
- 하나의 노드가 최대 5개의 자식 노드를 가질 수 있음
- 하나의 노드에 저장할 수 있는 키 값은 최대 4개 (차수 - 1)

**B-Tree의 핵심 특징:**

- 모든 리프 노드는 같은 레벨에 있음
- 키 값은 오름차순으로 정렬됨
- 키 값이 중복 저장되지 않음
- 루트는 리프가 아닌 이상 적어도 두 개의 서브 트리를 가짐
- **노드는 반 이상(1/2 이상) 채워져야** 다음 단계로 트리가 내려감

:::tip 노드를 반 이상 채워야 하는 이유
B-Tree는 트리의 높이가 낮을수록 검색 성능이 좋습니다. 반 이상을 채워서 트리를 확장하도록 강제함으로써, 불필요하게 트리 높이가 높아지는 것을 방지합니다. 이것이 B-Tree가 우수한 검색 성능을 유지하는 비결입니다.
:::

**차수가 3인 B-Tree 삽입 예시:**

```
[차수 3 B-Tree에 43, 69, 138, 19 순서로 삽입]

Step 1: 43 삽입
  [43]

Step 2: 69 삽입 (오름차순 정렬)
  [43 | 69]

Step 3: 138 삽입 → 노드가 꽉 참 (최대 키 2개) → 분열(Split) 발생!
  43, 69, 138 중 가운데 값 69가 루트로 올라감
  왼쪽: [43], 루트: [69], 오른쪽: [138]

        [69]
       /    \
    [43]   [138]

Step 4: 19 삽입
  19 < 69이므로 왼쪽으로 이동
  43이 있는 노드에 19 삽입 (오름차순 정렬)

        [69]
       /    \
  [19|43]  [138]
```

```python
# B-Tree 개념 이해를 위한 간단한 파이썬 예시
# (실제 B-Tree 구현은 복잡하므로 개념 설명용)

class BTreeNode:
    def __init__(self, order):
        self.order = order          # 차수 (최대 자식 수)
        self.keys = []              # 이 노드에 저장된 키 값들
        self.children = []          # 자식 노드들
        self.is_leaf = True         # 리프 노드 여부

    def is_full(self):
        # 노드가 가득 찼는지 확인 (최대 키 수 = 차수 - 1)
        return len(self.keys) >= self.order - 1

# 차수 3인 노드 생성
node = BTreeNode(order=3)
print(f"차수 3 B-Tree: 최대 키 수 = {node.order - 1}개")  # 최대 2개
print(f"차수 3 B-Tree: 최대 자식 수 = {node.order}개")    # 최대 3개
```

### B\*-Tree

B\*-Tree는 B-Tree를 개선한 버전으로, B-Tree가 노드를 **1/2 이상** 채워야 한다면, B\*-Tree는 **2/3 이상** 채워야 합니다. 더 많이 채워서 트리의 높이를 더 낮추므로, B-Tree보다 검색 성능이 더 좋습니다.

> B\*-Tree = B-Tree보다 노드를 **더 꽉 채워서** 트리 높이를 더 낮춘 버전

### B+-Tree: 인덱스 세트 + 순차 세트

B+-Tree는 현대 데이터베이스에서 **가장 널리 사용되는 인덱스 구조**입니다. B-Tree와 비슷하지만 두 가지 중요한 차이점이 있습니다.

**B-Tree와 B+-Tree의 차이점:**

| 구분 | B-Tree | B+-Tree |
|---|---|---|
| 키 값 저장 위치 | 모든 노드 | 리프 노드(순차 세트)에 모두 저장 |
| 키 값 중복 | 중복 없음 | 리프와 내부 노드에 중복 가능 |
| 리프 노드 연결 | 연결 없음 | 링크(포인터)로 순서대로 연결 |
| 구성 | 단일 구조 | 인덱스 세트 + 순차 세트 |

**B+-Tree의 두 가지 구성 요소:**

1. **인덱스 세트(Index Set)**: 루트부터 리프 바로 위까지의 내부 노드들. 실제 데이터로 가는 경로만 제공
2. **순차 세트(Sequence Set)**: 리프 노드들. 실제 키 값이 모두 저장되며, 링크(포인터)로 서로 연결됨

```
[B+-Tree 구조 예시]

        [인덱스 세트]
              [50]
             /    \
          [25]   [75]
         /   \   /   \
        ↓     ↓ ↓     ↓
[순차 세트 - 리프 노드들 링크로 연결]
[10|20] → [25|30] → [50|60] → [75|90]
```

**B+-Tree의 강점:**

B+-Tree는 두 가지 접근 방식을 모두 효율적으로 지원합니다.

- **특정 데이터 검색**: 인덱스 세트를 통해 원하는 리프 노드로 빠르게 이동
- **전체 데이터 순차 처리**: 순차 세트(리프 노드 링크)를 따라가며 모든 데이터를 순서대로 처리

마치 하나의 자료구조에서 "빠른 검색"과 "순차 처리" 두 마리 토끼를 모두 잡은 것입니다!

**B+-Tree 삽입 예시 (차수 3):**

```
[차수 3 B+-Tree에 13, 8, 4, 11, 6 순서로 삽입]

Step 1: 13 삽입
  [13]  ← 리프 노드

Step 2: 8 삽입 (오름차순 정렬)
  [8 | 13]

Step 3: 4 삽입 → 노드 꽉 참 → 분열!
  4, 8, 13 중 가운데 8이 올라감
  B+-Tree는 키 값 중복이 허용되므로 8이 인덱스에도 남음

  인덱스 세트:   [8]
                /   \
  순차 세트:  [4|8] → [13]  ← 링크로 연결

Step 4: 11 삽입
  11 > 8이므로 오른쪽 리프로 이동

  인덱스 세트:   [8]
                /   \
  순차 세트:  [4|8] → [11|13]

Step 5: 6 삽입
  6은 4와 8 사이 → 왼쪽 리프에 삽입 시도하나 꽉 참
  → 분열 발생! 4, 6, 8 중 6이 올라감

  인덱스 세트:   [6 | 8]
               /   |    \
  순차 세트: [4|6]→[8]→[11|13]  ← 링크로 연결
```

**B+-Tree 삭제 예시:**

삭제할 때는 언더플로우(노드에 키가 너무 적어지는 현상)가 발생하면 인접 노드에서 빌려오거나 병합(Merge)합니다.

```
[60을 삭제하는 예시 (60, 50, 62, 65가 있을 때)]

삭제 전:
  [60]
 /    \
[50]  [62|65]

60 삭제 → 루트가 없어짐 → 남은 키 50, 62, 65 중 가운데 값 62가 새 루트로 올라감

삭제 후:
  [62]
 /    \
[50]  [65]
```

:::warning B-Tree vs B+-Tree 헷갈리지 않기
- **B-Tree**: 모든 노드에 키 값 저장, 키 값 중복 없음
- **B+-Tree**: 리프 노드에 모든 키 값 저장, 내부 노드 키 값이 리프에 중복될 수 있음, 리프 노드끼리 링크로 연결
:::

## 🗺️ Bitmap 인덱스와 R-Tree: 특수 목적 인덱스

### Bitmap 인덱스

Bitmap 인덱스는 **테이블이 매우 크고, 인덱스 컬럼의 값 종류가 적을 때(낮은 분포도)** 매우 유용한 인덱스 방식입니다.

**낮은 분포도(Low Cardinality)**란 해당 컬럼이 가질 수 있는 값의 종류가 적다는 의미입니다. 예를 들어:
- **성별**: 남(0), 여(1) → 2가지 값만 존재 (낮은 분포도)
- **지역**: 서울, 부산, 대구 등 몇 가지 값 (낮은 분포도)
- **주민등록번호**: 모든 사람이 다 다름 (높은 분포도)

Bitmap 인덱스는 키 값을 **0과 1의 비트(Bit)**로 변환하여 저장합니다.

```
[Bitmap 인덱스 예시 - 성별 컬럼]

원본 데이터:          Bitmap 인덱스:
ID | 성별            남자(0) | 여자(1)
1  | 남              1       | 0
2  | 여              0       | 1
3  | 남              1       | 0
4  | 여              0       | 1
5  | 남              1       | 0

"남자인 사람 수 계산" → 남자 비트맵의 1 개수만 세면 됨 → 3명
```

**Bitmap 인덱스의 장단점:**

| 특성 | 설명 |
|---|---|
| **저장 공간** | 0과 1로 표현하므로 매우 작음 |
| **처리 속도** | 비트 연산(AND, OR)이 빠르므로 수행 속도가 빠름 |
| **갱신 비용** | 데이터가 삽입/수정되면 전체 비트맵을 다시 계산해야 하므로 높음 |
| **적합한 용도** | 데이터 분석, 의사결정, 빅데이터 분석 |
| **부적합한 용도** | 삽입/수정이 빈번한 온라인 업무 처리 시스템 |

| 구분 | B-Tree 인덱스 | Bitmap 인덱스 |
|---|---|---|
| 데이터 양 | 소량 ~ 중간 | 대량 |
| 인덱스 크기 | 큼 | 매우 작음 |
| 갱신 비용 | 적음 | 많음 |
| 분포도 | 높음 (주민번호 등) | 낮음 (성별, 지역 등) |
| 주요 용도 | 온라인 업무 처리(OLTP) | 데이터 분석(OLAP), 빅데이터 |

:::tip Bitmap 인덱스 활용 예시
빅데이터 분석 시스템, 데이터 마이닝, 의사결정 지원 시스템(DSS)처럼 대량의 데이터를 분석하는 환경에서 특히 유용합니다. 반면, 쇼핑몰처럼 주문이 실시간으로 들어오고 수정되는 환경에서는 B-Tree 인덱스가 더 적합합니다.
:::

### R-Tree: 지리 정보를 위한 인덱스

R-Tree에서 'R'은 **Region(지역)**의 약자입니다. R-Tree는 **지리 정보, 공간 데이터(위치 좌표, CAD 데이터 등)를 저장하고 검색하기 위한 인덱스 구조**입니다.

R-Tree는 **MBR(Minimum Bounding Rectangle, 최소 경계 사각형)**이라는 개념을 계층적으로 구성합니다. MBR은 해당 지역에 속하는 모든 객체를 포함하는 가장 작은 사각형 박스입니다.

```
[R-Tree 공간 인덱스 예시]

지도 위의 객체들:
  A 영역: 객체 1, 2, 5, 6 포함
  B 영역: 객체 3, 4, 7 포함
  C 영역: 객체 8, 9 포함 (일부 D와 겹침)
  D 영역: 객체 8, 10 포함 (일부 C와 겹침)

8번 객체는 C와 D 두 영역에 걸쳐 있음 (겹침 관계)
```

R-Tree에서 중요한 두 가지 관계:

- **포함 관계(Containment)**: 객체가 특정 MBR 안에 완전히 포함됨 (예: 객체 1, 2, 5, 6이 A 영역 안에 있음)
- **겹침 관계(Overlap)**: 객체가 두 개 이상의 MBR에 걸쳐 있음 (예: 객체 8이 C와 D에 걸쳐 있음)

> ⚠️ 겹침 관계가 있으면 하나의 데이터를 찾기 위해 여러 영역을 탐색해야 하므로 **성능이 저하**됩니다. 따라서 **포함 관계와 겹침 관계를 모두 최소화할 때 R-Tree의 성능이 가장 좋습니다.**

| R-Tree 종류 | 특징 |
|---|---|
| **R-Tree** | 기본 공간 인덱스 |
| **R\*-Tree** | 사각형 면적 최소화, 겹침 최소화, 둘레 길이 최소화, 기억장소 이용률 최적화 |
| **R+-Tree** | 겹침 관계를 완전히 제거 |

## 🔑 해싱: 주소를 직접 계산하여 찾기

### 해싱이란 무엇인가?

해싱(Hashing)은 **다른 레코드를 참조하지 않고 해싱 함수(Hashing Function)를 통해 목표 레코드의 저장 주소를 직접 계산하여 찾아가는 방법**입니다. 마치 물건에 GPS를 붙여놓은 것처럼, 어디에 있는지 바로 알 수 있습니다.

```
해싱 함수: F(키 값) = 저장 주소

예시:
  F("Database") = 100  →  100번 주소에 바로 이동하면 "Database" 레코드가 있음
```

순차 방법이나 인덱스 방법과 달리, 해싱은 중간 과정 없이 원하는 위치로 바로 도달할 수 있어서 이론적으로 가장 빠릅니다.

**해싱 관련 핵심 용어:**

- **버켓(Bucket)**: 하나의 주소에 저장할 수 있는 공간 (강의실처럼 일정 수의 레코드를 수용)
- **적재율(Load Factor)**: 버켓에 실제로 채워진 비율 (예: 100명 수용 가능한 버켓에 10개만 있으면 적재율 10%)
- **해싱 함수(Hash Function)**: 키 값을 주소로 변환하는 수식

### 해싱 함수의 종류

**1. 나머지 연산 해싱 (Division Method)**

키 값을 특정 수로 나눈 나머지를 주소로 사용합니다. 프로그래밍에서 `%` 연산자와 동일합니다.

```python
# 나머지 연산 해싱 예시
def division_hash(key, table_size):
    return key % table_size  # 키를 테이블 크기로 나눈 나머지

table_size = 100
key = 512
address = division_hash(key, table_size)
print(f"키 값: {key}, 해시 주소: {address}")
# 512 % 100 = 12 → 주소 12
```

**2. 중간 제곱 해싱 (Mid-Square Method)**

키 값을 제곱한 후 중간 자리 숫자를 주소로 사용합니다.

```python
# 중간 제곱 해싱 예시
def mid_square_hash(key, digits):
    """
    key: 키 값
    digits: 취할 자릿수 (해시 테이블 크기에 따라 결정)
    """
    squared = key ** 2          # 512^2 = 262144
    squared_str = str(squared)  # "262144"
    mid = len(squared_str) // 2 # 중간 위치 계산
    # 중간에서 digits 자리를 추출
    start = mid - digits // 2
    address = int(squared_str[start:start + digits])
    return address

key = 512
# 512^2 = 262144 → 중간 2자리 → "21" → 주소 21
print(f"키 값: {key}, 제곱: {key**2}, 해시 주소: 21")
```

**3. 중첩 해싱 / 폴딩 (Folding Method)**

키 값을 일정 자릿수로 나누어 각 부분을 더한 후 주소로 사용합니다.

```python
# 중첩 해싱 예시 (폴딩)
# 키 값: 1234567890123, 해시 테이블 크기 1000 (세 자리)
key = "1234567890123"
chunk_size = 3  # 세 자리씩 나눔
table_size = 1000

total = 0
chunks = []

# 세 자리씩 나누되, 짝수 번째는 뒤집어서(접어서) 더함
for i in range(0, len(key), chunk_size):
    chunk = key[i:i + chunk_size].ljust(chunk_size, '0')
    if (i // chunk_size) % 2 == 1:
        chunk = chunk[::-1]  # 홀수 번째 조각은 뒤집기 (접기)
    chunks.append(chunk)
    total += int(chunk)

address = total % table_size  # 세 자리만 취하기 위해 1000으로 나눈 나머지
print(f"조각들: {chunks}")
print(f"합계: {total}, 해시 주소: {address}")
```

**4. 숫자 이동 변환 (Digit Transformation)**

폴딩과 비슷하지만, 조각을 뒤집지 않고 그대로 더합니다.

**5. 진수 변환 (Radix Transformation)**

키 값을 다른 진수(예: 11진수)로 변환하여 주소를 계산합니다.

```python
# 진수 변환 해싱 예시
# 키 172를 11진수로 변환
key = 172
base = 11

result = 0
position = 0
temp_key = key

while temp_key > 0:
    digit = temp_key % 10       # 각 자리 숫자 추출
    result += digit * (base ** position)  # 해당 자리 × 11^자리수
    temp_key //= 10
    position += 1

print(f"키 값 {key}의 11진수 변환 결과(해시 주소): {result}")
# 1×11² + 7×11¹ + 2×11⁰ = 121 + 77 + 2 = 200 → 주소 200
```

### 충돌(Collision)과 해결 방법

해싱에서 **충돌(Collision)**이란 **서로 다른 키 값이 같은 해시 주소를 가리키는 현상**입니다. 예를 들어 `F("Database") = 100`이고 `F("PL") = 100`이면, 두 레코드가 모두 100번 주소를 사용하려 하므로 충돌이 발생합니다.

:::note 용어 정리
- **충돌(Collision)**: 두 개 이상의 레코드가 동일한 버켓(주소)으로 해싱되는 현상
- **동거자(Synonyms)**: 충돌이 난 레코드들 (같은 주소를 공유하는 키 값들)
:::

충돌을 해결하는 방법은 크게 **개방 해싱**과 **폐쇄 해싱**으로 나뉩니다.

#### 개방 해싱 (Open Hashing) - 체이닝 (Chaining)

충돌이 발생하면 **해시 테이블 밖의 새로운 공간에 저장하고 링크드 리스트(Linked List)로 연결**하는 방법입니다.

```
[개방 해싱 - 체이닝 예시]

해시 테이블:
주소 100 → [Database] → [PL] → NULL
주소 101 → [OS] → NULL
주소 102 → NULL

Database와 PL이 같은 주소(100)로 해싱됨 → 링크로 연결하여 해결
```

마치 아파트 방이 꽉 찼을 때, 옆방에 들어가서 "나는 원래 저 방에 있어야 해"라고 표시해두는 것과 같습니다.

#### 폐쇄 해싱 (Closed Hashing) - 개방 주소법

충돌이 발생하면 **해시 테이블 내부에서 빈 공간을 찾아 저장**하는 방법입니다. 표 밖으로 나가지 않고 표 안에서 해결합니다.

| 방법 | 설명 |
|---|---|
| **선형 탐사(Linear Probing)** | 충돌 발생 시 홈 주소(원래 주소)부터 순서대로 빈 공간을 탐색 |
| **제곱 탐사(Quadratic Probing)** | 충돌 발생 시 제곱 거리만큼 뛰어넘으며 빈 공간 탐색 (1, 4, 9, 16...) |
| **이중 해싱(Double Hashing)** | 충돌 발생 시 다른 해싱 함수를 사용하여 새 주소 계산 |
| **재해싱(Rehashing)** | 해시 테이블 크기 자체를 늘려서 새로 해싱 |

```python
# 선형 탐사 예시
def linear_probing_insert(hash_table, key, value, hash_func):
    table_size = len(hash_table)
    address = hash_func(key)  # 원래 해시 주소 계산

    # 빈 공간을 찾을 때까지 순서대로 탐색
    i = 0
    while hash_table[(address + i) % table_size] is not None:
        i += 1  # 다음 칸으로 이동
        if i == table_size:
            print("해시 테이블이 꽉 찼습니다!")
            return

    # 빈 공간 발견 → 저장
    actual_address = (address + i) % table_size
    hash_table[actual_address] = (key, value)
    print(f"키 '{key}' → 원래 주소: {address}, 실제 저장 주소: {actual_address}")

# 해시 테이블 초기화 (크기 10)
hash_table = [None] * 10

def simple_hash(key_str):
    # 간단한 해시: 키 문자열의 길이를 10으로 나눈 나머지
    return len(key_str) % 10

# 충돌 상황 시뮬레이션
linear_probing_insert(hash_table, "DB", "Database", simple_hash)
linear_probing_insert(hash_table, "OS", "Operating System", simple_hash)  # 충돌 가능
```

#### 확장성 해싱 (Extensible Hashing)

**디렉터리(Directory)**라는 별도의 구조를 이용하여 충돌을 회피하는 방법입니다. 1단계에서 디렉터리를 통해 먼저 걸러내어 충돌 가능성을 줄입니다.

:::tip 충돌 해결 방법 선택 기준
- **삽입/삭제가 빈번하고 크기 예측이 어려울 때**: 체이닝(개방 해싱) 추천
- **메모리가 제한적이고 해시 테이블 크기가 고정될 때**: 폐쇄 해싱 추천
- **충돌이 반복적으로 발생할 때**: 이중 해싱 또는 재해싱 고려
:::

:::danger 해싱 설계 시 절대 주의사항
해싱 함수가 잘못 설계되면 대부분의 키 값이 같은 주소로 해싱되는 **클러스터링(Clustering)** 현상이 발생합니다. 이 경우 선형 탐사를 사용하면 특정 주소 주변만 계속 꽉 차는 현상이 심화됩니다. 반드시 **균등한 분포를 보장하는 해싱 함수**를 선택해야 합니다.
:::

## ⚠️ 주의사항 및 실무 팁

### 인덱스 사용 시 주의사항

:::warning 인덱스는 만능이 아닙니다
인덱스는 검색 속도를 높여주지만, **삽입/수정/삭제 연산의 속도는 오히려 느려집니다.** 데이터가 변경될 때마다 인덱스도 함께 업데이트해야 하기 때문입니다. 또한 인덱스 자체도 저장 공간을 차지합니다. 모든 컬럼에 인덱스를 걸면 저장 공간 낭비와 쓰기 성능 저하가 발생합니다.
:::

**인덱스를 걸어야 할 때:**
- 자주 검색(조회)되는 컬럼
- WHERE 조건절에 자주 사용되는 컬럼
- 데이터 변경(삽입/수정/삭제)이 적은 컬럼
- 분포도(카디널리티)가 높은 컬럼 (고유한 값이 많을수록 효과적)

**인덱스를 걸지 말아야 할 때:**
- 값의 종류가 매우 적은 컬럼 (예: 성별 → Bitmap Index 고려)
- 데이터 삽입/수정이 매우 빈번한 컬럼
- 데이터 양이 매우 적은 테이블

### 해싱 사용 시 주의사항

:::warning 해싱의 한계
해싱은 **정확한 키 값으로 단일 레코드를 찾을 때** 매우 빠릅니다. 하지만 **범위 검색(예: 10~20 사이의 값 찾기)이나 정렬된 결과가 필요한 경우에는 적합하지 않습니다.** 또한 해싱 함수와 버켓 크기 설계가 잘못되면 충돌이 빈번하게 발생하여 오히려 성능이 나빠질 수 있습니다.
:::

### B-Tree 계열 사용 팁

:::tip B+-Tree가 데이터베이스에서 선택받는 이유
실무 데이터베이스(MySQL InnoDB, PostgreSQL 등)에서는 B+-Tree를 기본 인덱스 구조로 사용합니다. 그 이유는:
1. **균형 유지**: 항상 모든 리프가 같은 높이 → 어떤 데이터든 동일한 시간에 검색
2. **순차 처리 지원**: 리프 노드가 링크로 연결 → 범위 검색, 정렬된 결과 처리가 효율적
3. **공간 효율**: 노드를 반 이상 채워 트리 높이를 최소화
:::

## 📌 핵심 정리

이 문서에서 학습한 핵심 개념들을 정리합니다.

- **파일 조직 방법**은 순차 구조, 인덱스 구조, 해싱 구조 세 가지로 나뉜다
- **엔트리 순차 파일**은 레코드를 입력된 순서 그대로 저장하며, 정렬 과정이 없다
- **키 순차 파일**은 키 필드를 기준으로 정렬하여 저장하며, 항상 정렬된 상태를 유지한다
- **인덱스**는 책의 목차처럼 키 값과 주소의 쌍으로 구성되며, 데이터 접근 시 반드시 인덱스를 먼저 거친다
- **밀집 인덱스**는 모든 레코드에 대한 인덱스 항목을 가지며, **희소 인덱스**는 일부만 가진다
- **다단계 인덱스**는 인덱스를 여러 계층으로 구성하여 범위를 점점 좁혀가며 검색한다
- **정적 인덱스**는 인덱스 구조가 고정되어 오버플로우 영역이 필요하고, **동적 인덱스**는 구조가 자동으로 변경된다
- **팬 아웃이 클수록 인덱스 높이가 낮아져** 검색 성능이 향상된다
- **B-Tree**는 모든 리프가 동일한 높이를 가지는 균형 트리이며, 노드를 1/2 이상 채워야 한다
- **B\*-Tree**는 B-Tree보다 노드를 2/3 이상 채워 트리 높이를 더 낮춘다
- **B+-Tree**는 인덱스 세트와 순차 세트로 구성되며, 리프 노드가 링크로 연결되어 특정 검색과 순차 처리를 모두 효율적으로 지원한다
- **Bitmap 인덱스**는 낮은 분포도(성별, 지역 등)의 대량 데이터를 0과 1로 변환하여 분석 목적에 사용한다
- **R-Tree**는 지리/공간 데이터를 MBR 계층 구조로 저장하며, 포함 관계와 겹침 관계를 최소화할수록 성능이 좋다
- **해싱**은 해싱 함수로 저장 주소를 직접 계산하여 레코드에 즉시 접근하는 방법이다
- 해싱의 **충돌**은 개방 해싱(체이닝)과 폐쇄 해싱(선형 탐사, 제곱 탐사, 이중 해싱, 재해싱)으로 해결한다
