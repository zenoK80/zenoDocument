# 📚 인덱스의 종류: 밀집 인덱스와 희소 인덱스, 정적과 동적

데이터베이스를 공부하다 보면 "인덱스(Index)"라는 단어를 정말 자주 만나게 됩니다. 그런데 인덱스에도 여러 종류가 있다는 사실, 알고 계셨나요? 이번 문서에서는 파일 조직 방법 전체 맥락 속에서 인덱스가 무엇인지 살펴보고, **밀집 인덱스(Dense Index)와 희소 인덱스(Sparse Index)**, **정적 인덱스(Static Index)와 동적 인덱스(Dynamic Index)**, 그리고 B-Tree, B+Tree, 비트맵 인덱스, R-Tree 등 실무에서도 자주 등장하는 개념들을 초보자도 이해할 수 있는 수준으로 자세하게 설명합니다.

---

## 🗂️ 파일 조직 방법의 전체 그림

### 왜 파일 조직 방법을 알아야 할까요?

컴퓨터에 저장된 수많은 데이터를 빠르게 찾으려면, 처음부터 데이터를 **어떻게 저장하느냐**가 매우 중요합니다. 마치 도서관에서 책을 찾으려 할 때, 책이 아무렇게나 쌓여 있다면 원하는 책을 찾는 데 엄청나게 오래 걸리겠죠. 반대로 분류 기준과 목차가 잘 갖춰진 도서관이라면 빠르게 원하는 책을 찾을 수 있습니다. 데이터베이스도 마찬가지입니다. **저장 방식(파일 조직 방법)** 을 잘 설계해야 나중에 데이터를 빠르고 효율적으로 검색할 수 있습니다.

### 파일 조직의 세 가지 접근 방법

파일에 저장된 레코드(record, 데이터 한 줄 한 줄)에 접근하는 방법은 크게 세 가지로 나눌 수 있습니다.

| 방법 | 설명 | 비유 |
|------|------|------|
| **순차 구조(Sequential)** | 레코드를 순서대로 하나씩 차례로 처리 | 책을 1페이지부터 끝까지 순서대로 읽기 |
| **인덱스 구조(Index)** | 인덱스(목차)를 먼저 참조한 뒤 해당 위치로 이동 | 책의 목차에서 원하는 챕터를 찾은 뒤 해당 페이지로 이동 |
| **해싱 구조(Hashing)** | 해시 함수(주소 계산 공식)를 통해 곧바로 해당 위치로 점프 | 물건 보관함 번호를 계산해서 바로 해당 칸으로 이동 |

이 세 가지 중에서 특히 **인덱스 구조**는 실제 데이터베이스에서 가장 많이 활용되는 방식입니다. 이번 문서의 핵심 주제이기도 하므로, 이 개념을 집중적으로 파고들어 보겠습니다.

## 📖 순차 파일 구조: 기본 개념 정리

인덱스를 본격적으로 설명하기 전에, 순차 파일 구조를 간단히 살펴보겠습니다.

### 엔트리 순차 파일 (Entry-Sequenced File)

**엔트리 순차 파일**은 레코드가 도착하는 순서 그대로 저장하는 방식입니다. 정렬도 없고, 분류도 없습니다. 마치 받은 편지함에 도착한 순서대로 메일이 쌓이는 것과 같습니다. 1번, 3번, 2번, 5번, 6번, 4번 순으로 데이터가 들어온다면, 저장도 그 순서 그대로 1번 → 3번 → 2번 → 5번 → 6번 → 4번으로 기록됩니다. 가장 단순한 방식이지만, 특정 데이터를 검색할 때는 처음부터 끝까지 모두 뒤져야 한다는 단점이 있습니다.

### 키 순차 파일 (Key-Sequenced File)

**키 순차 파일**은 데이터가 들어올 때마다 특정 필드(키 필드)를 기준으로 **정렬하면서 저장**하는 방식입니다. 예를 들어 학번이 1번, 3번, 2번, 5번, 6번, 4번 순으로 들어온다고 가정해봅시다. 키 순차 파일에서는 3번이 들어오면 이미 있는 1번과 비교해서 순서를 맞추고, 2번이 들어오면 다시 정렬 위치를 찾아서 삽입합니다. 결과적으로 1번 → 2번 → 3번 → 4번 → 5번 → 6번 순으로 정렬된 상태로 저장됩니다. 검색 속도가 엔트리 순차 파일보다 훨씬 유리합니다.

:::info 순차 파일 요약
- **엔트리 순차**: 도착 순서 그대로 저장, 정렬 없음
- **키 순차**: 키 필드를 기준으로 정렬하면서 저장
:::

## 🔍 인덱스(Index)란 무엇인가?

### 책의 목차가 바로 인덱스입니다

**인덱스(Index)** 라는 단어는 영어로 '색인', '목차'를 의미합니다. 실생활에서도 쉽게 찾아볼 수 있습니다. 두꺼운 교재를 생각해봅시다. 책 앞부분에는 반드시 **목차(Index)**가 있습니다. 내가 원하는 챕터나 키워드가 몇 페이지에 있는지 알려주는 역할을 하죠.

만약 목차가 없다면 어떨까요? 원하는 내용을 찾으려면 1페이지부터 차례로 넘기며 읽어야 합니다. 하지만 목차가 있다면, 먼저 목차에서 원하는 항목을 찾고 → 해당 페이지로 바로 이동할 수 있습니다. 이것이 바로 인덱스의 핵심 아이디어입니다.

데이터베이스에서도 마찬가지입니다. 인덱스는 **키 값(Key Value)과 주소(Address) 쌍**으로 구성되어 있습니다. 데이터를 찾을 때는 반드시 인덱스 파일을 먼저 방문하고, 인덱스가 알려주는 주소로 이동해서 실제 데이터를 읽어오는 방식입니다.

> **인덱스 구조 = 인덱스 파일(키 값 + 주소) → 데이터 파일(실제 레코드)**
> 데이터 검색 시 반드시 인덱스를 먼저 거쳐서 데이터 파일로 이동합니다.

### 인덱스의 기본 구조

```
[ 인덱스 파일 ]          [ 데이터 파일 ]
 키 값  |  주소           실제 레코드들
--------|--------         ----------------
  k1   | → 주소1    →    k1 데이터 ...
  k2   | → 주소2    →    k2 데이터 ...
  k3   | → 주소3    →    k3 데이터 ...
```

예를 들어 k3 데이터를 찾고 싶다면:
1. 인덱스 파일로 이동
2. 인덱스 파일에서 k3의 주소를 확인
3. 그 주소의 데이터 파일로 이동
4. k3 데이터를 읽음

이처럼 **항상 두 단계**(인덱스 → 데이터)를 거치는 것이 인덱스 구조의 특징입니다.

## 🗃️ 밀집 인덱스 vs 희소 인덱스

이제 인덱스의 첫 번째 분류인 **밀집 인덱스(Dense Index)** 와 **희소 인덱스(Sparse Index)** 를 살펴보겠습니다.

### 밀집 인덱스 (Dense Index) — 모든 키 값을 다 기억하는 인덱스

**밀집 인덱스(Dense Index)** 란, 데이터 파일에 존재하는 **모든 레코드의 키 값**에 대해 인덱스 항목을 빠짐없이 보유하는 방식입니다. 마치 100페이지 분량의 책에서 1페이지부터 100페이지까지 **모든 페이지**에 대한 목차 항목을 가지고 있는 것과 같습니다.

예를 들어, 데이터 파일에 다음과 같은 키 값이 있다고 가정합시다:
- Brighton, Downtown, Mianus, Perryridge, Redwood, Roundhill

**밀집 인덱스**는 이 6개의 키 값 모두에 대해 인덱스 항목을 보유합니다. 따라서 어떤 키 값을 검색하더라도, 반드시 인덱스에서 해당 키를 직접 찾을 수 있고, 그 인덱스가 가리키는 주소로 바로 이동할 수 있습니다.

```
[ 밀집 인덱스 파일 ]            [ 데이터 파일 ]
 Brighton  → 주소1         →   Brighton 레코드
 Downtown  → 주소2         →   Downtown 레코드
 Mianus    → 주소3         →   Mianus 레코드
 Perryridge→ 주소4         →   Perryridge 레코드
 Redwood   → 주소5         →   Redwood 레코드
 Roundhill → 주소6         →   Roundhill 레코드
```

**Perryridge를 찾는다면?**
1. 인덱스 파일에서 Perryridge 항목을 찾음
2. 인덱스가 가리키는 주소로 이동
3. Perryridge 레코드 확인

:::tip 밀집 인덱스의 장점
모든 키 값에 대한 인덱스가 있으므로 **어떤 레코드도 바로 찾아갈 수 있습니다**. 검색이 직관적이고 빠릅니다.
:::

:::warning 밀집 인덱스의 단점
데이터 파일에 레코드가 많을수록 인덱스 파일 자체의 크기도 커집니다. 인덱스 파일을 저장하는 공간이 많이 필요합니다.
:::

### 희소 인덱스 (Sparse Index) — 일부 키 값만 기억하는 인덱스

**희소 인덱스(Sparse Index)** 는 **모든** 키 값이 아닌, **일부 키 값**에 대해서만 인덱스 항목을 갖는 방식입니다. 예를 들어 위의 6개 키 값 중 Brighton, Mianus, Redwood — 이렇게 3개만 인덱스에 포함시키는 것입니다. 나머지 키 값(Downtown, Perryridge, Roundhill)은 인덱스에 직접 항목이 없습니다.

```
[ 희소 인덱스 파일 ]            [ 데이터 파일 (정렬됨) ]
 Brighton  → 주소1         →   Brighton 레코드
 Mianus    → 주소3         →   Downtown 레코드
 Redwood   → 주소5         →   Mianus 레코드
                               Perryridge 레코드
                               Redwood 레코드
                               Roundhill 레코드
```

:::info 희소 인덱스의 전제 조건
희소 인덱스는 **반드시 데이터 파일이 정렬된 상태**여야 합니다. 인덱스에 없는 키 값을 찾을 때, 인덱스의 가장 가까운 항목부터 순차적으로 탐색하기 때문입니다.
:::

**Perryridge를 희소 인덱스로 찾는다면?**

Perryridge는 인덱스에 직접 항목이 없습니다. 이 경우 다음 방법을 사용합니다:

> **Perryridge보다 작거나 같은 키 값 중 가장 큰 것**을 인덱스에서 찾습니다.

인덱스에는 Brighton(B), Mianus(M), Redwood(R)가 있습니다. 알파벳 순서로 P(Perryridge)는 M(Mianus)보다 크고 R(Redwood)보다 작습니다. 따라서 **Mianus**를 시작점으로 삼아, 거기서부터 순차적으로 탐색하면 Perryridge를 찾을 수 있습니다.

이렇게 하면 처음(Brighton)부터 찾지 않아도 되므로, 순차 방법보다 훨씬 빠르게 원하는 레코드를 검색할 수 있습니다.

### 밀집 인덱스 vs 희소 인덱스 비교

| 구분 | 밀집 인덱스 (Dense) | 희소 인덱스 (Sparse) |
|------|---------------------|----------------------|
| **인덱스 항목 수** | 모든 레코드마다 1개 | 일부 레코드에만 존재 |
| **인덱스 파일 크기** | 큼 | 작음 |
| **검색 방식** | 인덱스에서 직접 찾아 이동 | 가장 가까운 인덱스 항목 → 순차 탐색 |
| **데이터 파일 정렬 필요** | 필수 아님 | **반드시 필요** |
| **검색 속도** | 빠름 (직접 접근) | 상대적으로 느릴 수 있음 |
| **공간 효율** | 낮음 | 높음 |

## 🏗️ 다단계 인덱스 (Multi-Level Index)

### 왜 다단계 인덱스가 필요한가?

데이터가 엄청나게 많아지면, 인덱스 파일도 거대해집니다. 인덱스 파일이 너무 크면, 인덱스에서 원하는 항목을 찾는 것도 오래 걸립니다. 이를 해결하기 위해 **인덱스 위에 또 인덱스를 두는** 방식이 바로 **다단계 인덱스(Multi-Level Index)** 입니다.

영어 사전에서 "Korea"를 찾는 과정을 생각해봅시다:
1. 먼저 **K** 구역으로 이동합니다. (1단계 인덱스)
2. K 구역에서 **Ko** 로 시작하는 부분을 찾습니다. (2단계 인덱스)
3. Ko 부분에서 **Kor**로 시작하는 단어를 찾습니다. (3단계 인덱스)
4. 마지막으로 순차적으로 "Korea"를 찾습니다.

이처럼 인덱스를 여러 단계(레벨)로 나누어, **단계를 거칠수록 탐색 범위를 좁혀가는 방식**이 다단계 인덱스입니다. 각 단계마다 탐색 범위가 줄어들기 때문에 매우 효율적입니다.

## ⚙️ 정적 인덱스 vs 동적 인덱스

인덱스의 두 번째 분류는 **정적(Static)이냐 동적(Dynamic)이냐** 입니다. 이는 인덱스의 **구조 자체가 변하느냐 변하지 않느냐**를 기준으로 나눕니다.

### 정적 인덱스 (Static Index)

**정적 인덱스(Static Index)** 는 인덱스의 구조가 고정되어 있는 방식입니다. 레코드가 삽입되거나 삭제될 때 인덱스 안의 **내용(키 값, 주소)**은 갱신되지만, 인덱스 파일의 **구조(틀) 자체는 변하지 않습니다**. 마치 고정된 크기의 책장처럼, 책의 내용은 바뀔 수 있지만 책장의 칸 수와 모양은 그대로입니다.

대표적인 정적 인덱스 파일 구조로는 **ISAM(Indexed Sequential Access Method)** 이 있습니다.

**정적 인덱스의 구조:**

```
[ 인덱스 영역 ] ← 구조가 고정됨
[ 데이터 영역 ] ← 데이터가 꽉 차면 오버플로우 발생
[ 오버플로우 영역 ] ← 넘치는 데이터를 임시로 저장
```

데이터가 계속 추가되어 데이터 영역이 꽉 차게 되면 **오버플로우(overflow, 용량 초과)**가 발생합니다. 이 경우 별도의 **오버플로우 영역**에 데이터를 임시 저장합니다. 오버플로우 영역까지 꽉 차면 더 이상 데이터를 넣을 수 없으므로, **주기적으로 파일 재조직(reorganization)** 을 통해 공간을 정리해야 합니다.

:::warning 정적 인덱스의 주의사항
오버플로우 영역이 과도하게 늘어나면 검색 성능이 급격히 떨어집니다. 주기적인 파일 재조직이 반드시 필요합니다.
:::

### 팬 아웃(Fan-out)과 인덱스 높이(Height)

정적 인덱스의 성능을 평가할 때 두 가지 지표를 사용합니다.

- **팬 아웃(Fan-out)**: 하나의 인덱스 노드에서 가지를 뻗는 방향의 수. 즉, **옆으로 얼마나 넓게 퍼지느냐**를 나타냅니다.
- **인덱스 높이(Height)**: 루트 노드에서 리프 노드까지 몇 단계를 거치느냐. 즉, **위에서 아래로 얼마나 깊은가**를 나타냅니다.

우유 배달을 예로 들어보겠습니다. 집이 4채 있을 때:
- **팬 아웃이 4, 높이가 1**: 한 번에 4집을 동시에 배달 → 매우 빠름 ✅
- **팬 아웃이 1, 높이가 4**: 1집씩 차례로 이동하며 4번 배달 → 느림 ❌

> **팬 아웃이 크면 인덱스 높이는 낮아지고, 검색 성능은 좋아집니다.**
> 반대로 팬 아웃이 작으면 인덱스 높이가 높아지고, 처리 단계가 많아져 성능이 떨어집니다.

### 동적 인덱스 (Dynamic Index)

**동적 인덱스(Dynamic Index)** 는 인덱스의 **구조 자체가 자동으로 변하는** 방식입니다. 데이터가 추가되어 공간이 꽉 차면 자동으로 분할(split)되고, 데이터가 줄어들면 다시 병합(merge)됩니다. 마치 생물의 **세포 분열**과 같습니다. 하나의 세포가 꽉 차면 둘로 나뉘고, 반대로 너무 작아지면 다시 합쳐지는 것이죠.

대표적인 동적 인덱스 파일 구조로는 **B+Tree(비플러스 트리)** 가 있습니다.

**정적 인덱스 vs 동적 인덱스 비교:**

| 구분 | 정적 인덱스 | 동적 인덱스 |
|------|-------------|-------------|
| **구조 변화** | 변하지 않음 (고정) | 삽입/삭제 시 자동으로 변함 |
| **오버플로우 영역** | 반드시 필요 | 필요 없음 |
| **공간 관리** | 주기적 파일 재조직 필요 | 자동 분할/병합 |
| **대표 구조** | ISAM | B+Tree |

## 🌳 B-Tree와 B+Tree: 동적 인덱스의 핵심

### B-Tree (B트리) — 균형잡힌 트리

**B-Tree(Balanced Tree, 균형 트리)** 는 동적 인덱스의 대표적인 구조입니다. 이름에서 **B(Balanced)** 는 '균형이 잡혀 있다'는 뜻입니다.

B-Tree의 가장 큰 특징은 **모든 리프 노드(leaf node, 가장 아래의 단말 노드)가 동일한 높이(레벨)**에 있다는 것입니다. 즉, 루트(root, 최상위 노드)에서 어떤 리프 노드까지 가더라도 항상 같은 단계 수를 거칩니다. 이를 통해 어떤 데이터를 검색하든 일정한 시간(성능)을 보장합니다.

```
     [ 루트 노드 ]
    /              \
[ 내부 노드 ]   [ 내부 노드 ]
  /     \           /      \
[리프] [리프]   [리프]   [리프]
← 모든 리프 노드가 같은 레벨에 있음 ✅
```

:::info B-Tree의 차수(Order)
**차수(Order)** 란 하나의 노드가 가질 수 있는 **자식 노드의 최대 수**를 말합니다.
- 차수가 5인 B-Tree: 하나의 노드에서 최대 5개의 자식으로 이동 가능
- 이 경우 하나의 노드에 저장할 수 있는 키 값은 최대 4개

**트리를 반 이상 채워야 다음 단계(레벨)로 내려갈 수 있습니다.** 이 규칙 덕분에 트리의 높이가 필요 이상으로 높아지는 것을 방지하고, 검색 성능을 좋게 유지합니다.
:::

### B-Tree 삽입 예시

차수가 3인 B-Tree에 43, 69, 138, 19 순서로 삽입하는 과정을 살펴보겠습니다.

```
Step 1: 43 삽입
[ 43 ]

Step 2: 69 삽입 (오름차순 정렬)
[ 43 | 69 ]

Step 3: 138 삽입 → 노드가 꽉 참(차수 3이므로 키 값 최대 2개) → 분열(Split)!
43, 69, 138 중 가운데 값(69)이 루트로 올라감
      [ 69 ]
     /      \
  [ 43 ]  [ 138 ]

Step 4: 19 삽입
19 < 69 → 왼쪽 노드로 이동 → 19 < 43 → 43 왼쪽에 삽입
      [ 69 ]
     /      \
 [ 19 | 43 ]  [ 138 ]
```

:::note B-Tree 삽입 규칙
- 오름차순으로 정렬하여 삽입
- 노드가 꽉 차면(오버플로우) → 가운데 값이 부모로 올라가고, 나머지는 좌우로 분열
- 왼쪽 자식은 부모보다 작은 값, 오른쪽 자식은 부모보다 큰 값
:::

### B-Tree 삭제 예시

B-Tree에서 키 값을 삭제하면 경우에 따라 노드가 합쳐지거나 재구성될 수 있습니다.

**예시 1: 단순 삭제**
```
삭제 전: [ 50 | 58 ]
58 삭제 후: [ 50 ]
```

**예시 2: 루트 삭제 후 재구성**
```
삭제 전:
      [ 60 ]
     /      \
  [ 50 ]  [ 62 | 65 ]

60(루트) 삭제 →  남은 키: 50, 62, 65 중 가운데 값 62가 새 루트로
      [ 62 ]
     /      \
  [ 50 ]  [ 65 ]
```

### B*Tree (비스타 트리) — B-Tree의 업그레이드 버전

**B*Tree** 는 B-Tree를 개선한 구조입니다. B-Tree가 노드를 **1/2 이상** 채워야 한다면, B*Tree는 **2/3 이상** 채워야 합니다. 채우는 비율이 높아지므로 트리의 높이가 더 낮아지고, 결과적으로 B-Tree보다 검색 성능이 더 좋습니다.

:::tip B*Tree 핵심 요약
**B-Tree**: 노드를 1/2 이상 채움
**B*Tree**: 노드를 2/3 이상 채움 → 높이가 더 낮아짐 → 성능 향상
:::

### B+Tree (비플러스 트리) — 가장 많이 사용되는 인덱스 구조

**B+Tree** 는 실제 데이터베이스에서 인덱스를 구성할 때 **가장 많이 사용되는 구조**입니다. B-Tree와 비슷하지만 중요한 차이점이 있습니다.

**B+Tree의 구성:**
- **인덱스 세트(Index Set)**: 상위 노드들. 실제 데이터를 저장하지 않고 **경로(주소)만 제공**합니다.
- **순차 세트(Sequence Set)**: 리프 노드들. **실제 키 값과 데이터**가 모두 저장되어 있으며, 포인터 링크로 서로 연결됩니다.

```
[ 인덱스 세트 ]
      [ 8 ]
     /     \
  [ 4 ]   [ 11 | 13 ]
    ↓          ↓
[ 순차 세트 ] (리프 노드들이 링크로 연결)
[4] ↔ [8] ↔ [11] ↔ [13] ↔ ...
```

**B+Tree의 핵심 특징:**

1. **모든 리프 노드가 같은 레벨** — 어떤 데이터도 동일한 단계로 검색 가능
2. **키 값의 중복 저장** — 인덱스 세트에 있는 키 값이 순차 세트(리프)에도 다시 등장합니다. 이것이 B-Tree와의 가장 큰 차이점입니다.
3. **순차 탐색 가능** — 리프 노드들이 링크로 연결되어 있어, 전체 데이터를 순서대로 처리할 때 순차 세트를 따라가면 됩니다.
4. **오름차순 정렬** — 항상 정렬된 상태를 유지합니다.

**B+Tree의 강력한 장점:**
- **특정 데이터 검색** → 인덱스 세트를 통해 빠르게 리프로 이동
- **전체 데이터 순차 처리** → 순차 세트(리프 링크)를 따라 순서대로 처리

> 하나의 자료 구조에서 **랜덤 검색(특정 값 찾기)** 과 **순차 처리(전체 데이터 스캔)** 모두 효율적으로 처리할 수 있는 것이 B+Tree의 가장 큰 강점입니다.

### B+Tree 삽입 예시 (차수 3)

차수 3 → 자식 최대 3개, 키 값 최대 2개

```
Step 1: 13 삽입
[ 13 ]

Step 2: 8 삽입 (8 < 13 → 왼쪽)
B+Tree의 키 값 2개 채워짐 — 인덱스 세트 없이 리프 하나:
[ 8 | 13 ]

Step 3: 4 삽입 → 오버플로우 → 분열!
4, 8, 13 중 가운데 8이 인덱스 세트(루트)로 올라감
B+Tree이므로 8은 리프에도 중복 저장됨:
     [ 8 ]         ← 인덱스 세트
    /     \
[ 4 | 8 ] [ 13 ]  ← 순차 세트 (리프, 링크 연결)

Step 4: 11 삽입
11 > 8 → 오른쪽 노드 [ 13 ] 에 삽입
오름차순 정렬: 11 < 13
     [ 8 ]
    /     \
[ 4 | 8 ] [ 11 | 13 ]

Step 5: 6 삽입
6 > 4이고 6 < 8 → 왼쪽 노드 [ 4 | 8 ] 에 삽입 시도
이미 꽉 찼으므로 4, 6, 8 분열!
가운데 6이 루트로 올라가야 하지만 루트의 오른쪽에 빈 자리 있음(차수 3)
→ 8이 인덱스 세트로 올라가며 재구성:
        [ 6 | 8 ]
       /    |    \
[ 4|6 ] [8] [ 11|13 ]
(순차 세트 링크: [4|6] ↔ [8] ↔ [11|13])
```

:::note B-Tree vs B+Tree 핵심 차이
- **B-Tree**: 키 값이 인덱스 노드에 한 번만 저장됨 (중복 없음)
- **B+Tree**: 키 값이 인덱스 세트와 순차 세트(리프) 양쪽에 **중복 저장**됨
- **B+Tree**: 리프 노드들이 **링크로 연결**되어 순차 탐색 가능
:::

## 🗺️ 비트맵 인덱스 (Bitmap Index)

### 언제 비트맵 인덱스를 사용하나요?

**비트맵 인덱스(Bitmap Index)** 는 테이블이 매우 크고, 인덱스로 사용할 컬럼의 **값의 종류가 매우 적을 때(낮은 분포도)** 사용합니다. 예를 들어:
- **성별**: 남(0) 또는 여(1) — 딱 2가지
- **지역**: 서울, 부산, 대구 등 — 비교적 적은 종류
- **구매 여부**: 구매(1) 또는 미구매(0)

이처럼 값의 종류가 적은 컬럼은 각 값에 대해 **0과 1의 비트(bit, 0 또는 1로만 표현되는 가장 작은 데이터 단위)** 로 인덱스를 구성합니다. 1억 명의 회원 데이터에서 "남성 회원 중 구매자"를 찾는다면, 비트맵 인덱스끼리 **AND 연산**만으로 초고속으로 결과를 구할 수 있습니다.

**비트맵 인덱스의 장점:**
- 인덱스를 0과 1로만 표현하므로 **저장 공간이 매우 작습니다**
- 비트 연산(AND, OR, NOT)으로 **처리 속도가 매우 빠릅니다**
- 대량의 데이터를 분석하는 **데이터 웨어하우스, 의사결정 시스템**에 매우 유용합니다

**비트맵 인덱스의 단점:**
- 데이터가 삽입되거나 변경될 때마다 **0과 1의 비트 배열 전체를 다시 계산**해야 합니다
- 따라서 **삽입/갱신이 빈번한 테이블에는 적합하지 않습니다**

### B-Tree 인덱스 vs 비트맵 인덱스 비교

| 구분 | B-Tree 인덱스 | 비트맵 인덱스 |
|------|---------------|---------------|
| **적합한 데이터 규모** | 소량 ~ 중간 | 대량 |
| **인덱스 크기** | 상대적으로 큼 | 매우 작음 (비트로 표현) |
| **갱신 비용** | 적음 | 많음 (전체 재계산) |
| **적합한 분포도** | 큰 분포도 (주민등록번호 등) | 작은 분포도 (성별, 지역 등) |
| **주요 사용처** | 일반 온라인 업무 처리(OLTP) | 빅데이터 분석, 데이터 마이닝(OLAP) |

## 🌐 R-Tree: 공간 데이터를 위한 인덱스

### R-Tree란 무엇인가요?

**R-Tree** 는 지리 정보, 공간 데이터를 저장하고 검색하기 위해 설계된 인덱스 구조입니다. 여기서 **R**은 **Region(영역, 지역)** 의 약자입니다. CAD 데이터, 지도 데이터, 위치 기반 서비스 등 **2차원 또는 3차원 공간 데이터**를 다룰 때 사용합니다.

R-Tree는 **MBR(Minimum Bounding Rectangle, 최소 경계 사각형)** 이라는 개념을 계층 형태로 표현합니다. MBR이란, 어떤 객체나 객체들의 묶음을 완전히 둘러싸는 **가장 작은 직사각형**을 의미합니다.

### 포함 관계와 겹침 관계

R-Tree에서 가장 중요한 두 가지 개념이 바로 **포함 관계**와 **겹침 관계**입니다.

- **포함 관계**: 하나의 MBR 안에 여러 객체가 완전히 포함된 경우. 예를 들어 사각형 A 안에 1, 2, 5, 6번 객체가 포함되어 있다면, 이것이 포함 관계입니다.
- **겹침 관계**: 하나의 객체가 두 MBR에 걸쳐 있는 경우. 예를 들어 8번 객체가 사각형 C와 D 양쪽에 걸쳐 있다면 겹침 관계가 발생합니다.

**겹침 관계의 문제점:**
- 8번 객체를 찾으려면 C로도 가고, D로도 가야 합니다.
- 인덱스를 한 번만 방문해서 원하는 데이터를 찾아야 효율적인데, 겹침이 있으면 **두 군데 이상을 방문**해야 합니다.
- 따라서 겹침 관계가 많을수록 성능이 나빠집니다.

> **포함 관계와 겹침 관계가 모두 최소일 때 R-Tree의 효율이 가장 좋습니다.**

### R-Tree의 종류

| 종류 | 특징 |
|------|------|
| **R-Tree** | 기본 형태. MBR로 공간을 계층적으로 표현 |
| **R\*Tree** | MBR 면적 최소화 + 겹침 최소화 + 둘레 길이 최소화 + 기억장소 이용률 최적화 |
| **R+Tree** | 겹침 관계를 완전히 제거한 구조 |

## 🔑 해싱(Hashing): 직접 주소로 찾아가는 방법

### 해싱이란?

**해싱(Hashing)** 은 다른 레코드를 참조하지 않고, 키 값에 **해시 함수(Hash Function, 주소 계산 공식)**를 적용해서 **목표 레코드의 주소를 직접 계산**하는 방법입니다. 마치 물건 보관함 번호를 공식으로 계산해서 바로 해당 칸으로 이동하는 것처럼요.

```
키 값 → [해시 함수 F(x)] → 주소 → 데이터 직접 접근
예: F("database") = 100 → 100번지로 바로 이동
```

### 해시 함수의 종류

여러 가지 해시 함수 방식이 있습니다:

**1. 제산(나머지) 해싱**
```
키 값 512, 해시 테이블 크기 100
512 % 100 = 12  →  주소: 12
```
키 값을 해시 테이블 크기로 나눈 나머지를 주소로 사용합니다.

**2. 중간 제곱 해싱**
```
키 값 512
512² = 262144
해시 테이블 크기 100 (두 자리) → 가운데 두 자리: 21  →  주소: 21
```
키 값을 제곱한 뒤, 가운데 자릿수를 주소로 사용합니다.

**3. 중첩(폴딩) 해싱**
```
키 값: 1234567890123, 해시 테이블 크기 1000 (세 자리)
세 자리씩 나눠 접어서 더함:
123 + 456(↔654) + 789 + 012(↔210) + 3 = 1779
1779 % 1000 = 779  →  주소: 779
```
키 값을 일정 자릿수로 나눈 후 접어서(교차하며) 더한 뒤 해시 테이블 크기로 나눈 나머지를 주소로 사용합니다.

**4. 숫자 이동(Digit Shifting) 해싱**
```
키 값: 1234567890123, 해시 테이블 크기 1000 (세 자리)
세 자리씩 나눠 그냥 더함 (접지 않음):
123 + 456 + 789 + 012 + 3 = 1383
1383 % 1000 = 383  →  주소: 383
```

**5. 진수 변환 해싱**
```
키 값: 172 → 11진수로 변환
1×11² + 7×11¹ + 2×11⁰ = 121 + 77 + 2 = 200  →  주소: 200
```

### 충돌(Collision)과 해결 방법

**충돌(Collision)** 이란, 서로 다른 키 값이 동일한 주소로 계산되는 경우입니다. 예를 들어 F("database") = 100이고 F("PL") = 100이면, 두 레코드가 같은 주소(100번지)를 가리키게 됩니다. 이때 같은 버켓(bucket, 하나의 주소 공간)에 들어가려는 레코드들을 **동거자(Synonym)** 라고 합니다.

:::danger 충돌을 방치하면?
충돌이 해결되지 않으면 데이터가 덮어써지거나 저장 자체가 실패합니다. 반드시 충돌 해결 방법을 사용해야 합니다.
:::

**충돌 해결 방법:**

| 방법 | 분류 | 설명 |
|------|------|------|
| **체이닝(Chaining)** | 개방 해싱 | 충돌 시 버켓 밖의 새로운 공간에 저장하고 링크로 연결 |
| **선형 탐사(Linear Probing)** | 폐쇄 해싱 | 충돌 시 가장 가까운 빈 공간을 순서대로 찾아 저장 |
| **제곱 탐사(Quadratic Probing)** | 폐쇄 해싱 | 충돌 시 제곱 거리만큼 이동하며 빈 공간 탐색 |
| **이중 해싱(Double Hashing)** | 폐쇄 해싱 | 충돌 시 다른 해시 함수를 사용해 새 주소 계산 |
| **재해싱(Rehashing)** | 폐쇄 해싱 | 해시 테이블 크기를 늘리고 전체 재해싱 |
| **확장성 해싱(Extendible Hashing)** | 별도 분류 | 디렉터리를 사용해 충돌 사전 회피 |

- **개방 해싱(Open Hashing)**: 충돌 발생 시 **버켓 외부의 공간**을 사용. 대표적인 방법이 체이닝입니다.
- **폐쇄 해싱(Closed Hashing)**: 충돌 발생 시 **버켓 내부의 다른 공간**에서 해결. 선형 탐사, 제곱 탐사, 이중 해싱, 재해싱이 해당합니다.

## ⚠️ 주의사항 및 실전 팁

### 인덱스 관련 주의사항

:::warning 인덱스가 많을수록 항상 좋은 건 아닙니다
인덱스가 많으면 검색(SELECT)은 빨라지지만, 데이터 삽입(INSERT)/수정(UPDATE)/삭제(DELETE) 시 인덱스도 함께 갱신해야 하므로 **쓰기 성능이 떨어집니다**. 적절한 수의 인덱스를 유지하는 것이 중요합니다.
:::

:::tip 비트맵 인덱스 사용 시
삽입/갱신이 많이 발생하는 온라인 트랜잭션 처리(OLTP) 환경에서는 비트맵 인덱스 사용을 피하세요. 분석 전용 데이터 웨어하우스(OLAP)에 적합합니다.
:::

:::tip B+Tree 인덱스의 선택
실제 관계형 데이터베이스(MySQL의 InnoDB, PostgreSQL 등)에서는 기본적으로 **B+Tree 인덱스**를 사용합니다. 범위 검색(BETWEEN, >, &lt; 등)과 정렬(ORDER BY)에도 강력하며, 순차 처리와 랜덤 접근 모두 효율적입니다.
:::

:::danger 정적 인덱스 오버플로우 방치 금지
ISAM과 같은 정적 인덱스 구조에서 오버플로우 영역이 과도하게 커지도록 방치하면, 검색 성능이 급격히 저하됩니다. **주기적인 파일 재조직**을 반드시 수행해야 합니다.
:::

### 잘못된 예시와 올바른 예시

**❌ 잘못된 예시: 비트맵 인덱스를 OLTP 환경에 남용**

```sql
-- 잘못된 예시: 매 초마다 수천 건의 INSERT/UPDATE가 발생하는 주문 테이블에
-- 비트맵 인덱스를 생성하면 갱신 비용이 폭발적으로 증가함
CREATE BITMAP INDEX idx_order_status ON orders(status);
-- 주문 상태가 빈번하게 변경될 때마다 비트 배열 전체를 재계산해야 함 → 성능 저하
```

**✅ 올바른 예시: 분석 전용 데이터 웨어하우스에 비트맵 인덱스 적용**

```sql
-- 올바른 예시: 거의 변경되지 않는 대용량 분석 테이블에 비트맵 인덱스 적용
-- 성별(M/F 두 가지 값), 지역(소수의 구분), 구매여부(Y/N) 등
CREATE BITMAP INDEX idx_gender ON customer_analysis(gender);
-- → 0과 1로만 구성되어 공간 절약 + AND/OR 비트 연산으로 초고속 분석 가능
```

**❌ 잘못된 예시: 팬 아웃이 낮은 인덱스 구조 선택**

```
-- 팬 아웃 1, 높이 4인 구조 (비효율적)
루트 → A → B → C → 리프
: 데이터 하나를 찾기 위해 4단계를 거쳐야 함 → 검색이 느림
```

**✅ 올바른 예시: 팬 아웃이 높은 인덱스 구조 선택**

```
-- 팬 아웃 4, 높이 1인 구조 (효율적)
루트 → [리프1, 리프2, 리프3, 리프4]
: 루트에서 바로 원하는 리프로 이동 → 빠른 검색
```

## 📋 핵심 정리

- **파일 조직 방법**은 순차 구조, 인덱스 구조, 해싱 구조 세 가지로 나뉜다.
- **순차 파일**은 엔트리 순차(도착 순서대로 저장)와 키 순차(키 필드 기준 정렬 저장)로 구분된다.
- **인덱스 구조**는 키 값과 주소 쌍으로 구성된 인덱스 파일을 먼저 참조한 후 데이터 파일로 이동한다.
- **밀집 인덱스**는 모든 레코드의 키 값에 대해 인덱스 항목을 보유한다 (공간 큼, 검색 빠름).
- **희소 인덱스**는 일부 키 값에만 인덱스 항목을 두며, 데이터 파일이 반드시 정렬되어 있어야 한다.
- **다단계 인덱스**는 인덱스 위에 인덱스를 두어 탐색 범위를 단계적으로 좁혀간다.
- **정적 인덱스**는 구조가 고정되어 오버플로우 영역이 필요하며, 주기적 파일 재조직이 필요하다 (대표: ISAM).
- **동적 인덱스**는 삽입/삭제 시 자동 분할·병합되며 오버플로우 영역이 불필요하다 (대표: B+Tree).
- **팬 아웃이 크면 인덱스 높이가 낮아지고** 검색 성능이 향상된다.
- **B-Tree**는 모든 리프 노드가 같은 레벨에 있는 균형 트리로, 노드를 1/2 이상 채워야 한다.
- **B*Tree**는 노드를 2/3 이상 채워 B-Tree보다 높이가 낮고 성능이 더 좋다.
- **B+Tree**는 인덱스 세트(경로 제공)와 순차 세트(실제 데이터 저장, 링크 연결)로 구성되며 키 값이 중복 저장된다.
- **비트맵 인덱스**는 값의 종류가 적은 컬럼에 적합하며, 0/1 비트로 표현해 공간 절약과 빠른 분석이 가능하다.
- **R-Tree**는 공간/지리 데이터를 MBR로 계층 표현하며, 포함·겹침 관계가 최소일 때 가장 효율적이다.
- **해싱**은 해시 함수로 주소를 직접 계산해 레코드에 접근하는 방식이다.
- **충돌 해결 방법**은 개방 해싱(체이닝)과 폐쇄 해싱(선형 탐사, 제곱 탐사, 이중 해싱, 재해싱)으로 나뉜다.
