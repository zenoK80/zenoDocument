# 📚 인덱스 구조의 개념과 동작 원리

데이터베이스를 공부하다 보면 "인덱스(Index)"라는 말을 정말 자주 듣게 됩니다. 그런데 인덱스가 정확히 무엇인지, 왜 필요한지, 어떤 종류가 있는지 헷갈리는 분들이 많습니다. 이번 문서에서는 파일 조직 방법 전체를 체계적으로 살펴보고, 그 중심인 인덱스 구조의 개념과 동작 원리를 초보자도 이해할 수 있도록 아주 쉽고 자세하게 설명합니다.

---

## 🗂️ 파일 조직 방법의 전체 그림

### 파일 조직이란 무엇인가?

데이터베이스나 파일 시스템에서 **레코드(record, 하나의 데이터 행)**를 저장할 때, 어떤 방식으로 저장하고 어떻게 꺼내올 것인지를 결정하는 것이 바로 **파일 조직 방법**입니다. 마치 도서관에서 책을 어떤 방식으로 배열하느냐에 따라 책을 찾는 속도가 달라지는 것처럼, 데이터를 저장하는 방식에 따라 검색 속도와 효율이 크게 달라집니다.

파일 조직 방법은 **접근 방법(access method)**에 따라 크게 세 가지로 나뉩니다.

| 방법 | 설명 | 특징 |
|------|------|------|
| **순차 구조** | 레코드를 순서대로 저장하고 처음부터 순서대로 접근 | 단순하지만 검색이 느림 |
| **인덱스 구조** | 인덱스(목차)를 이용해 원하는 레코드에 빠르게 접근 | 검색 속도가 빠름 |
| **해싱 구조** | 해싱 함수(수식)로 주소를 직접 계산해서 바로 접근 | 가장 빠른 직접 접근 |

이 세 가지 방법 중에서 데이터베이스에서 가장 많이 사용되고 중요한 것이 바로 **인덱스 구조**입니다. 이 문서에서는 순차 구조에서 출발해 인덱스 구조의 모든 것을 설명하고, 마지막으로 해싱 구조까지 다룹니다.

---

## 📋 순차 파일 구조 — 가장 단순한 저장 방법

### 엔트리 순차 파일 (Entry-Sequenced File)

**엔트리 순차 파일**은 레코드가 도착하는 순서 그대로 저장하는 방식입니다. 정렬도 없고, 분류도 없고, 어떤 가공도 없습니다. 말 그대로 "들어오는 순서대로 그냥 집어넣는" 방식입니다.

예를 들어, 학생 레코드가 학번 순서 없이 다음과 같이 도착한다고 가정해봅시다.

```
도착 순서: 1번 → 3번 → 2번 → 5번 → 6번 → 4번
저장 결과: [1번] [3번] [2번] [5번] [6번] [4번]
```

보이는 것처럼, 도착한 순서 그대로 저장됩니다. 마치 택배가 도착한 순서대로 창고에 쌓아두는 것처럼요. 나중에 특정 택배를 찾으려면 처음부터 하나씩 뒤져야 합니다.

### 키 순차 파일 (Key-Sequenced File)

**키 순차 파일**은 데이터가 들어올 때 **특정 필드(키 필드)**를 기준으로 정렬하면서 저장하는 방식입니다. 엔트리 순차와 달리, 들어오는 데이터마다 기존 데이터와 비교해서 올바른 위치에 끼워 넣습니다.

같은 학번 예시로 보면:

```
도착 순서: 1번 → 3번 → 2번 → 5번 → 6번 → 4번

[1번 도착]     → [1번]
[3번 도착]     → [1번] [3번]
[2번 도착]     → [1번] [2번] [3번]   ← 2번이 3번보다 작으므로 위치 교환
[5번 도착]     → [1번] [2번] [3번] [5번]
[6번 도착]     → [1번] [2번] [3번] [5번] [6번]
[4번 도착]     → [1번] [2번] [3번] [4번] [5번] [6번]  ← 4번 정렬 삽입

최종 저장 결과: [1번] [2번] [3번] [4번] [5번] [6번]
```

마치 카드를 한 장씩 받으면서 계속 정렬된 상태로 손에 쥐고 있는 것과 같습니다. 저장된 결과가 항상 정렬되어 있기 때문에, 나중에 이진 탐색 등을 활용해 더 빠르게 검색할 수 있습니다.

:::info 순차 파일 정리
- **엔트리 순차**: 도착 순서 그대로 저장 (정렬 없음)
- **키 순차**: 키 필드 기준으로 정렬하면서 저장 (항상 정렬된 상태 유지)
:::

## 🔍 인덱스 구조 — 목차로 빠르게 찾아가는 방법

### 인덱스란 무엇인가?

> **인덱스(Index)는 책의 목차(Table of Contents)와 완전히 같은 개념입니다.**

여러분이 두꺼운 책에서 "데이터베이스"라는 내용이 있는 페이지를 찾는다고 해봅시다. 두 가지 방법이 있습니다.

1. **순차 방법**: 1페이지부터 한 장씩 넘기면서 찾는다 → 느리고 비효율적
2. **인덱스 방법**: 책 앞의 목차(인덱스)를 펼쳐서 "데이터베이스 → 253페이지"를 확인하고, 253페이지로 바로 이동한다 → 빠르고 효율적

데이터베이스의 인덱스도 마찬가지입니다. **인덱스 파일**은 **키 값(Key Value)**과 **주소(Address)**의 쌍으로 구성되어 있습니다.

```
인덱스 파일:
+----------+----------+
|  키 값   |   주소   |
+----------+----------+
|   k1     |  0x0010  |
|   k2     |  0x0020  |
|   k3     |  0x0030  |  ← k3를 찾으려면 이 주소로 이동
|   k4     |  0x0040  |
+----------+----------+

데이터 파일:
주소 0x0030에 k3 레코드가 실제로 저장되어 있음
```

**데이터를 찾는 과정:**
1. 인덱스 파일로 먼저 이동
2. 찾고 싶은 키 값(k3)을 인덱스에서 검색
3. k3에 해당하는 주소(0x0030)를 확인
4. 그 주소로 바로 이동해서 실제 데이터를 읽음

이처럼 **무조건 인덱스를 먼저 거쳐서 실제 데이터로 이동**하는 것이 인덱스 구조의 핵심입니다.

## 🗃️ 인덱스의 종류 1 — 밀집 인덱스 vs 희소 인덱스

### 밀집 인덱스 (Dense Index)

**밀집 인덱스**는 데이터 파일에 있는 **모든 레코드**에 대한 인덱스 항목을 빠짐없이 가지고 있는 방식입니다. 마치 책의 목차가 모든 페이지에 대해 하나씩 항목을 가지고 있는 것과 같습니다.

> **밀집 인덱스 정의**: 인덱스가 파일에 있는 모든 검색 키 값에 대해 빠짐없이 인덱스 항목(키 값 + 주소)을 보유하는 방식

예를 들어, 데이터 파일에 다음 6개의 레코드가 있다면:

```
데이터 파일 키 값: Brighton, Downtown, Mianus, Perryridge, Redwood, Roundhill

밀집 인덱스:
+-------------+----------+
|    키 값    |   주소   |
+-------------+----------+
|  Brighton   |  주소 1  |
|  Downtown   |  주소 2  |
|  Mianus     |  주소 3  |  ← 6개 모두 포함
|  Perryridge |  주소 4  |
|  Redwood    |  주소 5  |
|  Roundhill  |  주소 6  |
+-------------+----------+
```

모든 키 값에 대한 인덱스가 존재하기 때문에, 어떤 레코드를 찾더라도 반드시 인덱스에서 해당 키 값을 찾을 수 있습니다. 예를 들어 `Perryridge`를 찾는다면, 인덱스에서 `Perryridge`를 찾고 → 주소 4로 바로 이동하면 됩니다.

:::tip 밀집 인덱스의 장점
모든 레코드에 대한 인덱스가 있으므로, 어떤 데이터든 인덱스를 통해 정확하게 바로 찾아갈 수 있습니다.
:::

### 희소 인덱스 (Sparse Index)

**희소 인덱스**는 전체 레코드 중 **일부 키 값에 대해서만** 인덱스 항목을 가지는 방식입니다. 밀집 인덱스와 달리, 인덱스가 모든 레코드를 커버하지 않습니다.

같은 6개의 레코드 예시에서 희소 인덱스는 이렇게 구성될 수 있습니다:

```
데이터 파일 키 값: Brighton, Downtown, Mianus, Perryridge, Redwood, Roundhill

희소 인덱스 (3개만 보유):
+-------------+----------+
|    키 값    |   주소   |
+-------------+----------+
|  Brighton   |  주소 1  |
|  Mianus     |  주소 3  |  ← 6개 중 3개만 포함
|  Redwood    |  주소 5  |
+-------------+----------+
```

그렇다면 희소 인덱스에서 `Perryridge`를 어떻게 찾을까요? 인덱스에 `Perryridge`가 없으니까 못 찾는 걸까요? 그렇지 않습니다!

**희소 인덱스의 검색 방법:**
1. 인덱스는 항상 **정렬**되어 있습니다 (책의 목차가 순서대로 되어있는 것처럼)
2. 찾고 싶은 키 값(`Perryridge`)보다 **작거나 같은 키 값 중에서 가장 큰 것**을 인덱스에서 찾습니다
3. `Brighton(B) < Mianus(M) < Perryridge(P)`이므로, `Mianus`가 해당됩니다
4. `Mianus`의 주소로 이동해서 → 그 위치부터 **순차적으로** `Perryridge`를 찾습니다

처음부터 찾는 것보다 훨씬 빠르게 시작 위치를 좁혀줍니다. 마치 전화번호부에서 'P'로 시작하는 섹션으로 바로 이동해서 거기부터 찾는 것과 같습니다.

| 구분 | 밀집 인덱스 | 희소 인덱스 |
|------|------------|------------|
| 인덱스 항목 수 | 모든 레코드에 대해 1:1 | 일부 레코드에 대해서만 |
| 저장 공간 | 많이 필요 | 적게 필요 |
| 검색 방법 | 정확한 주소로 바로 이동 | 근접 위치로 이동 후 순차 탐색 |
| 검색 속도 | 더 빠름 | 상대적으로 조금 느림 |

### 다단계 인덱스 (Multi-Level Index)

**다단계 인덱스**는 인덱스를 여러 단계로 구성하는 방식입니다. 인덱스가 너무 커지면 인덱스 자체를 검색하는 데도 시간이 걸리기 때문에, 인덱스 위에 또 다른 인덱스를 두는 개념입니다.

영어 사전에서 `Korea`라는 단어를 찾는 과정을 생각해봅시다:

```
1단계: 'K'로 시작하는 섹션으로 이동
2단계: 'Ko'로 시작하는 부분으로 이동
3단계: 'Kor'로 시작하는 부분에서 Korea 찾기
```

이처럼 **범위를 단계적으로 좁혀가면서** 찾는 방식이 다단계 인덱스입니다. 각 단계의 인덱스가 다음 단계의 인덱스를 가리키고, 마지막 단계의 인덱스가 실제 데이터를 가리킵니다.

```
[1단계 인덱스]     [2단계 인덱스]     [데이터]
  인덱스 블록 1 → 인덱스 블록 A → 실제 레코드들
  인덱스 블록 2 → 인덱스 블록 B → 실제 레코드들
  인덱스 블록 3 → 인덱스 블록 C → 실제 레코드들
```

## 🌳 인덱스의 종류 2 — 정적 인덱스 vs 동적 인덱스

### 정적 인덱스 (Static Index)

**정적 인덱스**는 인덱스의 **구조(틀) 자체는 변경되지 않는** 방식입니다. 물론 데이터가 삽입되거나 삭제되면 인덱스 안의 **내용(키 값과 주소)**은 바뀝니다. 하지만 인덱스의 **형태와 구조**는 처음 만들어진 그대로 고정됩니다.

대표적인 예시가 **ISAM(Indexed Sequential Access Method, 인덱스 순차 접근 방법)** 파일 구조입니다.

**정적 인덱스의 구성:**
- **인덱스 영역**: 키 값과 주소를 저장하는 고정된 공간
- **데이터 영역**: 실제 레코드를 저장하는 고정된 공간
- **오버플로우 영역**: 데이터 영역이 꽉 찼을 때 추가로 사용하는 공간

```
[인덱스 영역] → [데이터 영역] → [오버플로우 영역]
  (고정된 구조)   (꽉 차면)       (추가 공간 사용)
```

구조가 고정되어 있기 때문에 데이터가 계속 추가되어 데이터 영역이 꽉 차면 **오버플로우(overflow, 넘침)**가 발생합니다. 이때는 오버플로우 영역에 데이터를 저장합니다. 하지만 오버플로우 영역마저 꽉 차면 **주기적으로 파일 재조직(reorganization)**을 해서 공간을 확보해야 합니다.

:::warning 정적 인덱스의 단점
오버플로우가 지속되면 검색 성능이 저하됩니다. 주기적인 파일 재조직이 필요하며, 이 과정에서 시스템이 잠시 중단될 수 있습니다.
:::

### 팬 아웃(Fan-Out)과 인덱스 높이

정적 인덱스의 성능을 나타내는 두 가지 중요한 지표가 있습니다.

- **팬 아웃(Fan-Out)**: 하나의 인덱스 노드(블록)에서 옆으로 뻗어나가는 자식 수 (가로 방향)
- **인덱스 높이(Height)**: 인덱스의 레벨 수 (세로 방향)

우유 배달에 비유해서 생각해보겠습니다:

```
팬 아웃이 큰 경우 (넓고 낮은 구조):
층   → [집A] [집B] [집C] [집D]   ← 한 번에 4집 배달 (빠름!)
높이=1, 팬아웃=4

팬 아웃이 작은 경우 (좁고 높은 구조):
4층  → [집D]
3층  → [집C]    ← 한 층씩 올라가며 배달 (느림!)
2층  → [집B]
1층  → [집A]
높이=4, 팬아웃=1
```

> **팬 아웃이 크면 → 인덱스 높이가 낮아짐 → 검색 성능이 좋아짐**
> **팬 아웃이 작으면 → 인덱스 높이가 높아짐 → 검색 성능이 나빠짐**

### 동적 인덱스 (Dynamic Index)

**동적 인덱스**는 데이터가 추가되거나 삭제될 때 인덱스의 **구조 자체가 자동으로 변경**되는 방식입니다. 마치 살아있는 세포처럼, 공간이 부족하면 스스로 분열(split)하고, 공간이 남으면 합치는(merge) 방식으로 동작합니다.

동적 인덱스의 대표적인 구조는 **B⁺트리(B+ Tree)**이며, **인덱스 세트(Index Set)**와 **순차 세트(Sequence Set)**로 구성됩니다.

- **오버플로우 영역이 필요 없습니다** (구조가 자동으로 늘어나고 줄어드니까)
- 항상 균형(Balance)이 맞는 상태를 유지합니다

## 🌲 B트리 계열 — 동적 인덱스의 핵심

### B트리 (B-Tree)의 기본 개념

**B트리(Balanced Tree, 균형 트리)**는 모든 리프 노드(leaf node, 가장 아래쪽 말단 노드)가 동일한 높이(레벨)에 위치하는 트리 구조입니다. **B의 약자가 Balanced(균형)**을 의미합니다.

```
❌ B트리가 아닌 경우 (높이가 불균형):
        [루트]
       /      \
    [A]        [B]
   /   \
 [C]   [D]
         \
          [E]    ← 리프 노드들의 높이가 다름

✅ B트리인 경우 (모든 리프 노드가 같은 레벨):
           [루트]
          /      \
       [A]        [B]
      /   \      /   \
   [C]   [D]  [E]   [F]   ← 모든 리프 노드가 같은 레벨
```

B트리의 중요한 특징 중 하나는 **차수(Order)**입니다. **차수**란 하나의 노드가 가질 수 있는 자식 노드의 최대 수를 의미합니다.

```
5차 B트리의 노드 구조:
+------+------+------+------+   ← 키 값 최대 4개 저장 가능
|  k1  |  k2  |  k3  |  k4  |
+------+------+------+------+
↓      ↓      ↓      ↓      ↓   ← 자식 노드 최대 5개 (차수=5)
자1   자2    자3    자4    자5
```

**B트리의 핵심 규칙**: 트리를 **절반 이상 채워야** 다음 레벨로 분기가 생깁니다. 이 규칙 덕분에 트리의 높이가 필요 이상으로 커지는 것을 방지하고, 검색 성능을 높게 유지합니다.

**B트리의 주요 특징:**
- 모든 리프 노드는 같은 레벨에 존재
- 키 값은 오름차순으로 정렬됨
- 키 값은 중복 저장되지 않음
- 루트 노드는 (리프 노드가 아닌 이상) 최소 두 개의 서브트리를 가짐

### B트리 삽입 예시

차수가 3인 B트리에 43, 69, 138, 19를 순서대로 삽입하는 과정을 살펴봅시다. 차수가 3이므로 키 값은 최대 2개까지 저장할 수 있습니다.

```python
# B트리 삽입 과정 시각화 (차수 = 3)

# 1단계: 43 삽입
# [43]

# 2단계: 69 삽입 (오름차순 정렬)
# [43 | 69]

# 3단계: 138 삽입 → 노드가 가득 참 → 스플릿(분열) 발생!
# 43, 69, 138 중 가운데 값인 69가 루트로 올라감
#       [69]          ← 루트
#      /    \
#   [43]   [138]      ← 작은 값 왼쪽, 큰 값 오른쪽

# 4단계: 19 삽입
# 19 < 69 이므로 왼쪽 [43]으로 이동
# 19 < 43 이므로 43 앞에 삽입
#       [69]
#      /    \
#  [19|43]  [138]     ← 19가 43 왼쪽에 삽입됨
```

:::tip 중위 순회 원칙
B트리는 중위 순회(In-order traversal) 방식으로 동작합니다. 왼쪽 자식 < 부모 < 오른쪽 자식 순서가 항상 유지됩니다. 스플릿 시에는 가운데 값이 부모 노드로 올라갑니다.
:::

### B트리 삭제 예시

```python
# B트리 삭제 과정 시각화

# 현재 트리 상태:
#       [60]
#      /    \
#   [50]   [62|65]

# 케이스 1: 58 삭제 (존재하지 않는 값이므로 생략)
# [50|58]에서 58을 삭제하면 [50]만 남음

# 케이스 2: 60(루트) 삭제
# 루트가 삭제되면 남은 값 50, 62, 65 중 가운데 값인 62가 루트로 승격
#       [62]          ← 62가 새 루트
#      /    \
#   [50]   [65]       ← 50은 왼쪽, 65는 오른쪽
```

### B*트리 (B*-Tree)

**B\*트리**는 B트리를 개선한 버전입니다. B트리는 트리를 **1/2 이상** 채우는 반면, B\*트리는 **2/3 이상** 채우도록 요구합니다. 더 촘촘하게 채우기 때문에 트리의 높이가 더 낮아지고, 결과적으로 검색 성능이 B트리보다 향상됩니다.

> **B트리**: 노드를 1/2 이상 채움
> **B\*트리**: 노드를 2/3 이상 채움 → 더 낮은 트리 높이 → 더 좋은 성능

### B⁺트리 (B+ Tree) — 가장 많이 사용되는 인덱스 구조

**B⁺트리**는 데이터베이스에서 **가장 널리 사용되는 인덱스 구조**입니다. B트리와 비슷하지만, 두 가지 중요한 차이점이 있습니다.

**B⁺트리의 구성:**
- **인덱스 세트(Index Set)**: 경로를 안내하는 역할만 함. 실제 데이터는 없고 키 값만 있음
- **순차 세트(Sequence Set)**: 실제 데이터(레코드)를 저장하는 리프 노드들. 포인터 링크로 서로 연결되어 있음

```
B⁺트리 구조:

[인덱스 세트]      ← 경로 안내만 담당
    [50 | 75]
    /    |    \
  [25] [62]  [88]

[순차 세트 (리프 노드들)]    ← 실제 데이터 저장
[19|25] ↔ [50|62] ↔ [75|88]  ← 링크로 서로 연결!
```

**B⁺트리의 핵심 장점:**

1. **특정 데이터 검색**: 인덱스 세트를 이용해서 원하는 리프 노드로 빠르게 이동
2. **전체 데이터 순차 처리**: 리프 노드들이 링크로 연결되어 있어서 순차 세트를 따라 쭉 처리 가능

하나의 자료 구조에서 **랜덤 접근**과 **순차 접근** 모두에 효율적으로 대응할 수 있다는 것이 B⁺트리의 가장 큰 강점입니다.

**B트리와 B⁺트리의 차이점:**

| 구분 | B트리 | B⁺트리 |
|------|-------|--------|
| 키 값 중복 | 중복 없음 | 인덱스 세트와 순차 세트에 중복 저장됨 |
| 데이터 위치 | 모든 노드에 데이터 있음 | 리프 노드(순차 세트)에만 데이터 있음 |
| 순차 처리 | 트리 전체 순회 필요 | 리프 노드 링크로 쉽게 순차 처리 |
| 주요 용도 | 일반적인 트리 탐색 | 데이터베이스 인덱스 |

### B⁺트리 삽입 예시

차수가 3인 B⁺트리에 13, 8, 4, 11, 6을 순서대로 삽입하는 과정입니다.

```python
# B⁺트리 삽입 과정 (차수 = 3, 키 값 최대 2개)

# 1단계: 13 삽입
# 순차세트: [13]

# 2단계: 8 삽입 (오름차순 정렬)
# 순차세트: [8 | 13]

# 3단계: 4 삽입 → 노드 가득 참 → 스플릿!
# B트리: 가운데 값(8)이 루트로 올라가고 사라짐
# B⁺트리: 가운데 값(8)이 루트로 올라가지만, 리프에도 남아있음! (키 값 중복)
#
#       [8]                ← 인덱스 세트 (경로 안내)
#      /   \
#   [4]   [8|13]           ← 순차 세트 (8이 중복 저장됨!)
#   ↔ 링크로 연결 ↔

# 4단계: 11 삽입
# 11 > 8이므로 오른쪽 리프 [8|13]으로 이동
#       [8]
#      /   \
#   [4]   [8|11|13]  → 가득 참!
# → 스플릿 발생, 11이 위로 올라감
#       [8 | 11]
#      /    |    \
#   [4]   [8]   [11|13]

# 5단계: 6 삽입
# 6 < 8이므로 왼쪽 리프 [4]로 이동
#       [8 | 11]
#      /    |    \
#  [4|6]  [8]   [11|13]    ← 6이 4 옆에 삽입
```

:::note B⁺트리의 키 값 중복
B⁺트리에서는 인덱스 세트의 키 값이 순차 세트(리프 노드)에도 그대로 남아있습니다. 이것이 B트리와의 핵심 차이점입니다. 덕분에 모든 실제 데이터는 반드시 리프 노드에 존재하게 됩니다.
:::

## 🗺️ 특수 목적 인덱스 — 비트맵 인덱스와 R트리

### 비트맵 인덱스 (Bitmap Index)

**비트맵 인덱스**는 테이블이 매우 크고, 인덱스 컬럼의 값의 종류가 적을 때(낮은 분포도) 사용하는 특별한 인덱스입니다.

예를 들어, 수백만 명의 고객 데이터에서 "성별" 컬럼은 '남자' 아니면 '여자', 딱 두 가지 값만 가집니다. 이런 경우에 비트맵 인덱스가 매우 유용합니다.

**비트맵 인덱스의 동작 원리:**
기존 키 값을 0과 1의 비트(bit)로 변환해서 저장합니다.

```
일반 데이터:
+--------+--------+
| 고객ID |  성별  |
+--------+--------+
|   001  |  남자  |
|   002  |  여자  |
|   003  |  남자  |
|   004  |  여자  |
|   005  |  남자  |
+--------+--------+

비트맵 인덱스로 변환:
         남자  여자
001  →    1     0
002  →    0     1
003  →    1     0
004  →    0     1
005  →    1     0
```

**비트맵 인덱스의 장점:**
- 비트(0과 1)로 표현되어 **저장 공간을 크게 절약**할 수 있음
- 0이냐 1이냐만 비교하면 되므로 **수행 속도가 빠름**
- 빅데이터 분석, 의사결정 지원 시스템(DSS), 데이터 마이닝에 유용

**비트맵 인덱스의 단점:**
- 데이터가 삽입되거나 갱신될 때마다 **전체 비트맵을 다시 계산**해야 함
- 따라서 **삽입/갱신이 자주 발생하는 테이블에는 부적합**

| 구분 | B트리 인덱스 | 비트맵 인덱스 |
|------|------------|-------------|
| 적합한 데이터 규모 | 소량 ~ 중간 | 대량 |
| 인덱스 크기 | 상대적으로 큼 | 매우 작음 (0과 1로 압축) |
| 갱신 비용 | 적음 | 많음 (전체 재계산) |
| 분포도 | 높은 분포도 적합 (주민번호 등) | 낮은 분포도 적합 (성별, 지역 등) |
| 주요 용도 | 일반 온라인 트랜잭션 처리(OLTP) | 데이터 분석, 의사결정 지원(OLAP) |

### R트리 (R-Tree) — 지리 정보를 위한 인덱스

**R트리**는 지리 정보(공간 데이터)를 저장하고 검색하기 위한 특수 인덱스 구조입니다. 여기서 **R은 Region(지역)**의 약자입니다. CAD 데이터, 지도 데이터, 위치 정보 등을 다룰 때 사용합니다.

R트리는 **MBR(Minimum Bounding Rectangle, 최소 경계 사각형)**이라는 개념을 사용합니다. 여러 공간 객체들을 하나의 최소 크기의 직사각형으로 감싸서 계층적으로 표현합니다.

```
지도 위의 데이터:
         ┌─────────────┐
         │    A 영역   │
         │  ┌──┐  ┌──┐│
         │  │1 │  │2 ││   ← A가 1, 2, 5, 6을 포함
         │  └──┘  └──┘│
         │  ┌──┐  ┌──┐│
         │  │5 │  │6 ││
         │  └──┘  └──┘│
         └─────────────┘

         ┌─────┐
         │  B  │          ← B 영역
         │ 3,4 │
         └─────┘

    ┌──────────┐
    │   C  ┌──────────┐
    │      │    D     │   ← C와 D가 겹쳐 있음 (8번 객체가 양쪽에 걸침)
    │  7   │  8  │  9 │
    └──────┘──────────┘
```

위 그림에서 **8번 객체는 C와 D에 모두 걸쳐** 있습니다. 이것이 바로 **겹침 관계(Overlap)**입니다. 8번을 검색하려면 C와 D 두 곳을 모두 확인해야 하므로 성능이 떨어집니다.

반면 **포함 관계(Containment)**는 A 안에 1, 2, 5, 6이 완전히 포함되어 있는 것처럼, 하나의 MBR 안에 완전히 속한 경우입니다.

**R트리 계열 비교:**

| 구분 | 설명 |
|------|------|
| **R트리** | 기본 구조. 포함 관계와 겹침 관계를 모두 허용 |
| **R\*트리** | MBR 면적 최소화 + 겹침 최소화 + 둘레 길이 최소화로 성능 개선 |
| **R+트리** | 겹침 관계를 완전히 제거. 하나의 객체는 하나의 MBR에만 속함 |

> **포함과 겹침이 모두 최소일 때 R트리의 검색 효율이 가장 좋습니다.**

## #️⃣ 해싱 구조 — 주소를 직접 계산하는 방법

### 해싱이란 무엇인가?

**해싱(Hashing)**은 키 값을 **해싱 함수(Hashing Function)**에 넣으면, 해당 레코드가 저장된 **주소가 바로 계산되어 나오는** 방식입니다. 다른 레코드를 참조하거나 인덱스를 거칠 필요 없이 직접 목표 레코드에 접근합니다.

```
해싱의 동작 원리:

키 값 "Database" → [해싱 함수 F] → 주소 100
                                        ↓
                              주소 100번지에 바로 접근
                              → "Database" 레코드 발견!
```

마치 창고에 물건을 넣을 때 물건 이름으로 번호를 계산해서, 그 번호의 선반에 바로 놓는 것과 같습니다. 나중에 꺼낼 때도 이름으로 번호를 계산해서 그 선반으로 직접 가면 됩니다.

**해싱 관련 주요 용어:**
- **버켓(Bucket)**: 하나의 주소에 저장할 수 있는 레코드 묶음 (예: 101호 강의실 = 100명 수용)
- **적재율(Load Factor)**: 버켓에 실제로 채워진 비율 (예: 100명 중 10명 → 10% 적재율)
- **해싱 함수(Hashing Function)**: 키 값을 주소로 변환하는 수식

### 해싱 함수의 종류

**1. 제산 나머지 해싱 (Division-Remainder Hashing)**

키 값을 테이블 크기로 나눈 나머지를 주소로 사용합니다.

```python
# 제산 나머지 해싱 예시
key = 512          # 키 값
table_size = 100   # 해시 테이블 크기

# 512를 100으로 나눈 나머지 계산
address = key % table_size
# 512 % 100 = 12
# → 주소 12번지에 저장

print(f"키 값 {key}의 해싱 주소: {address}")  # 출력: 12
```

**2. 중간 제곱 해싱 (Mid-Square Hashing)**

키 값을 제곱한 후, 결과의 중간 자리 수를 주소로 사용합니다.

```python
# 중간 제곱 해싱 예시
key = 512          # 키 값
table_size = 100   # 테이블 크기 = 100 → 주소는 2자리 (00~99)

squared = key ** 2        # 512² = 262144
squared_str = str(squared)  # "262144"

# 중간 2자리 추출 (6자리 숫자에서 가운데 2자리 = 21)
mid_start = len(squared_str) // 2 - 1  # 인덱스 계산
address = int(squared_str[mid_start : mid_start + 2])

# 결과: 262144 → 가운데 2자리 "21" → 주소 21
print(f"키 값 {key}의 해싱 주소: {address}")  # 출력: 21
```

**3. 중첩 해싱 (Folding Hash)**

키 값을 일정 자리 수로 나눈 후, 겹쳐서(접어서) 더한 결과를 주소로 사용합니다.

```
키 값: 1234567890123
테이블 크기: 1000 → 3자리 주소 사용

세 자리씩 나누기: 123 | 456 | 789 | 012 | 3

접기(Folding):
  123
+ 654  (456을 뒤집음)
+ 789
+ 210  (012를 뒤집음)
+   3
-----
 1779

1779 % 1000 = 779  → 최종 주소: 779
```

**4. 숫자 이동 변환 (Digit Transformation)**

세 자리씩 나누되, 뒤집지 않고 그대로 더합니다.

```
키 값: 1234567890123
테이블 크기: 1000 → 3자리 주소 사용

세 자리씩 나누기: 123 | 456 | 789 | 012 | 3

그대로 더하기:
  123
+ 456
+ 789
+ 012
+   3
-----
 1383

1383 % 1000 = 383  → 최종 주소: 383
```

**5. 진수 변환 해싱 (Radix Transformation)**

키 값을 다른 진수로 변환한 값을 주소로 사용합니다.

```
키 값: 172 (10진수)
11진수로 변환:

172 = 1 × 11² + 7 × 11¹ + 2 × 11⁰
    = 1 × 121 + 7 × 11  + 2 × 1
    = 121 + 77 + 2
    = 200

→ 최종 주소: 200
```

### 충돌(Collision)과 해결 방법

해싱의 가장 큰 문제는 **충돌(Collision)**입니다. 서로 다른 키 값이 해싱 함수를 통해 **같은 주소**를 가리키게 되는 현상입니다.

```
F("Database") → 주소 100
F("PL")       → 주소 100  ← 충돌! 같은 주소
```

충돌이 난 두 레코드를 **동거자(Synonym)**라고 부릅니다. 충돌을 해결하는 방법은 크게 두 가지로 나뉩니다.

**1. 개방 해싱 (Open Hashing) — 체이닝(Chaining)**

충돌이 발생하면 해시 테이블 **외부**에 새로운 공간을 만들어 저장하고, 링크(포인터)로 연결합니다. 마치 방이 꽉 차면 옆방에 두고 문패를 달아두는 것처럼요.

```
주소 100: [Database] → [PL] → NULL
                         ↑
                  충돌 발생 시 링크드 리스트로 연결
```

**2. 폐쇄 해싱 (Closed Hashing) — 내부에서 해결**

충돌이 발생하면 해시 테이블 **내부**의 빈 공간을 찾아 저장합니다.

- **선형 탐사(Linear Probing)**: 충돌 발생 지점부터 순서대로 빈 자리를 찾음
- **제곱 탐사(Quadratic Probing)**: 처음 지점부터 순서대로가 아니라 제곱 거리만큼 건너뛰며 빈 자리를 찾음
- **이중 해싱(Double Hashing)**: 충돌 발생 시 다른 해싱 함수를 적용해 새 주소 계산
- **재해싱(Rehashing)**: 해시 테이블 크기 자체를 키워서 공간을 확보하고 전체 재해싱

```
선형 탐사 예시:
주소 100에 충돌 발생
→ 101 확인 (비어있으면 저장)
→ 101도 차있으면 102 확인
→ 빈 자리 나올 때까지 순서대로 탐색

제곱 탐사 예시:
주소 100에 충돌 발생
→ 100 + 1² = 101 확인
→ 비어있으면 저장, 아니면 100 + 2² = 104 확인
→ 아니면 100 + 3² = 109 확인... (제곱 거리로 건너뜀)
```

**3. 확장성 해싱 (Extendible Hashing)**

**디렉터리(Directory)**라는 개념을 사용해서 충돌을 미리 회피하는 방법입니다. 디렉터리를 통해 한 번 더 걸러주는 방식으로 충돌 가능성을 줄입니다.

:::warning 해싱 충돌 처리 방법 요약
- **개방 해싱**: 체이닝 (외부 링크드 리스트 활용)
- **폐쇄 해싱**: 선형 탐사, 제곱 탐사, 이중 해싱, 재해싱 (내부 공간 활용)
- **확장성 해싱**: 디렉터리를 통한 사전 회피
:::

## ⚠️ 주의사항과 실전 팁

### 인덱스를 설계할 때 고려할 점

인덱스는 검색 속도를 높여주지만, 무조건 많이 만든다고 좋은 것이 아닙니다.

:::danger 인덱스 남용 금지
인덱스는 데이터 삽입/수정/삭제(DML) 시 인덱스도 함께 갱신해야 하므로, 인덱스가 너무 많으면 오히려 쓰기 성능이 저하됩니다. 꼭 필요한 컬럼에만 인덱스를 생성하세요.
:::

:::tip 인덱스 설계 실전 팁
1. **자주 검색되는 컬럼**에 인덱스를 생성하세요
2. **분포도가 높은 컬럼**(주민번호, 이메일 등)에는 B트리 인덱스가 효과적입니다
3. **분포도가 낮은 컬럼**(성별, 지역코드 등)에는 비트맵 인덱스가 효과적입니다
4. **삽입/갱신이 빈번한 테이블**에는 비트맵 인덱스를 피하세요
5. **B⁺트리**는 범위 검색과 순차 처리 모두에 강점이 있어 가장 범용적으로 사용됩니다
:::

### 트리 구조 선택 기준

```
일반 온라인 트랜잭션 처리(OLTP, 일반 업무 시스템):
→ B트리 또는 B⁺트리 사용

대용량 데이터 분석, 의사결정 지원(OLAP, 빅데이터 분석):
→ 비트맵 인덱스 사용

지리 정보, 공간 데이터 처리:
→ R트리 계열 사용

데이터 삽입/갱신/삭제가 많고 동적으로 변하는 환경:
→ 동적 인덱스(B⁺트리) 사용 (오버플로우 영역 불필요)

구조가 고정되고 안정적인 환경:
→ 정적 인덱스(ISAM) 사용 가능 (단, 파일 재조직 계획 필요)
```

:::note 해싱 vs 인덱스 선택 기준
- **등호 검색(=)만 필요하다면**: 해싱이 가장 빠릅니다
- **범위 검색(<, >, BETWEEN)이 필요하다면**: 인덱스(B⁺트리)를 사용하세요. 해싱은 범위 검색에 적합하지 않습니다
:::

## 📌 핵심 정리

이 문서에서 배운 핵심 개념들을 정리합니다.

- **파일 조직 방법**은 순차 구조, 인덱스 구조, 해싱 구조 세 가지로 나뉜다
- **엔트리 순차 파일**은 도착 순서 그대로 저장하고, **키 순차 파일**은 키 필드 기준으로 정렬하며 저장한다
- **인덱스 구조**는 책의 목차처럼 키 값과 주소 쌍으로 구성되며, 무조건 인덱스를 먼저 거쳐 데이터에 접근한다
- **밀집 인덱스**는 모든 레코드에 대한 인덱스를 보유하고, **희소 인덱스**는 일부 레코드에 대해서만 인덱스를 보유한다
- **희소 인덱스**에서 찾는 키가 없으면, 찾는 키보다 작거나 같은 가장 큰 인덱스 항목으로 이동 후 순차 탐색한다
- **다단계 인덱스**는 인덱스 위에 인덱스를 두어 범위를 단계적으로 좁혀가는 방식이다
- **정적 인덱스(ISAM)**는 인덱스 구조가 고정되며, 오버플로우 시 별도 오버플로우 영역이 필요하다
- **팬 아웃이 크면** 인덱스 높이가 낮아져 검색 성능이 좋아진다
- **동적 인덱스(B⁺트리)**는 데이터 변화에 따라 구조가 자동으로 분열/합병되며 오버플로우 영역이 필요 없다
- **B트리**는 모든 리프 노드가 같은 레벨이며, 1/2 이상 채워야 하고, 키 값이 중복되지 않는다
- **B\*트리**는 B트리보다 더 촘촘하게(2/3 이상) 채워 성능을 높인 구조다
- **B⁺트리**는 인덱스 세트와 순차 세트로 구성되며, 리프 노드가 링크로 연결되어 순차 처리와 랜덤 접근 모두에 효율적이다
- **B⁺트리**는 키 값이 인덱스 세트와 순차 세트에 중복 저장되는 것이 B트리와의 핵심 차이다
- **비트맵 인덱스**는 키 값을 0과 1로 변환해 저장 공간을 절약하며, 낮은 분포도 컬럼과 데이터 분석에 적합하다
- **R트리**는 MBR(최소 경계 사각형) 개념으로 지리/공간 데이터를 저장하며, 포함 관계와 겹침 관계가 핵심 개념이다
- **해싱**은 해싱 함수로 키 값에서 주소를 직접 계산하여 레코드에 바로 접근하는 방식이다
- **충돌(Collision)**은 서로 다른 키 값이 같은 주소를 가리키는 현상이며, 개방 해싱(체이닝)과 폐쇄 해싱(선형·제곱 탐사, 이중 해싱, 재해싱)으로 해결한다
- **해싱**은 등호 검색에 강하고, **B⁺트리 인덱스**는 범위 검색에 강하다
