# 💾 물리적 저장 매체의 종류와 특징 — 데이터베이스는 어디에, 어떻게 저장될까?

데이터베이스를 배울 때 우리는 흔히 SQL 쿼리나 테이블 설계에만 집중하기 쉽습니다. 하지만 데이터베이스가 실제로 **어디에**, **어떤 방식으로** 저장되고, **어떤 과정을 거쳐** 우리에게 전달되는지를 이해하는 것은 성능 최적화와 시스템 설계에 있어 매우 중요한 기초 지식입니다.

이번 문서에서는 데이터베이스의 **물리적 저장 매체의 종류**, **하드디스크의 구조와 데이터 접근 시간**, **내부 처리 과정**, **클러스터링**, **RAID**, 그리고 **블로킹**까지 초보자도 이해할 수 있도록 단계적으로 설명합니다.

---

## 🗂️ 1. 물리적 저장 매체란 무엇인가?

### 저장 매체를 알아야 하는 이유

데이터베이스에 저장된 데이터는 결국 어떤 **물리적인 장치** 위에 올라가 있습니다. 마치 우리가 메모를 종이에 쓰든, 화이트보드에 쓰든, 스마트폰에 저장하든 — 정보를 담는 **그릇이 다르면 특성도 다릅니다**. 저장 매체의 특성을 이해하면 왜 데이터베이스 성능이 느려지는지, 어떻게 개선할 수 있는지를 파악할 수 있습니다.

현대 데이터베이스 시스템에서 사용되는 물리적 저장 매체는 크게 세 가지로 나눌 수 있습니다.

| 저장 매체 | 대표 예시 | 특징 요약 |
|---|---|---|
| 전통적 하드디스크 (HDD) | 일반 데스크탑 HDD | 대용량, 저렴, 속도 느림 |
| 플래시 메모리 (Flash Memory) | USB, SD카드 | 비휘발성, 읽기 빠름, 충격에 강함 |
| 솔리드 스테이트 드라이브 (SSD) | 노트북 내장 SSD | 반도체 기반, 빠름, 소음 없음, 비쌈 |

---

### 📌 플래시 메모리 (Flash Memory)

**플래시 메모리**는 이름 그대로 **메모리 칩(chip) 안에 정보를 보관하는 방식**입니다. 우리가 흔히 쓰는 USB 드라이브나 SD카드가 바로 플래시 메모리 기반입니다.

일반적으로 우리가 말하는 "메모리"는 **RAM(램)**을 가리킵니다. RAM은 **휘발성(Volatile)** 메모리로, 컴퓨터 전원을 끄면 저장된 내용이 모두 사라집니다. 마치 화이트보드에 적어둔 내용을 지우개로 싹 지우는 것처럼요. 반면 플래시 메모리는 **비휘발성(Non-Volatile)** 메모리입니다. 전원이 꺼져도 데이터가 그대로 유지됩니다. 공책에 볼펜으로 적어두는 것에 비유할 수 있죠.

**플래시 메모리의 주요 특징:**

- **읽기 속도 > 쓰기 속도**: 플래시 메모리는 데이터를 읽는 속도가 쓰는 속도보다 훨씬 빠릅니다. 책을 읽는 것보다 책에 새로운 내용을 기록하는 것이 더 오래 걸리는 것과 비슷합니다.
- **충격에 강함**: 기계적으로 움직이는 부품이 없기 때문에 물리적 충격에 훨씬 강합니다. 전통적 HDD는 충격을 받으면 헤드가 손상될 수 있지만, 플래시 메모리는 그런 위험이 적습니다.
- **전력 소모가 적음**: 기계적 부품이 없으므로 소비 전력이 낮습니다.

:::info 같은 용량인데 가격이 다른 이유
마트에서 64GB USB를 보면 1만 원짜리도 있고 10만 원짜리도 있습니다. 용량은 같은데 왜 가격이 이렇게 다를까요? 바로 **읽기/쓰기 속도의 차이** 때문입니다. 더 빠른 플래시 메모리일수록 더 고급 반도체 칩을 사용하기 때문에 가격이 높아집니다. 구매 시에는 용량뿐만 아니라 속도 규격도 확인하는 것이 좋습니다.
:::

---

### 📌 솔리드 스테이트 드라이브 (SSD, Solid State Drive)

**SSD**는 **반도체(Semiconductor)**를 이용해 데이터를 저장하는 장치입니다. SSD를 분해해 보면 내부에 수많은 플래시 메모리 칩들이 회로 기판(보드) 위에 촘촘히 연결되어 있는 것을 볼 수 있습니다.

SSD는 기존의 전통적 HDD와 비교했을 때 다음과 같은 장단점을 가집니다.

| 비교 항목 | HDD (전통 하드디스크) | SSD |
|---|---|---|
| 속도 | 느림 | 매우 빠름 |
| 기계적 지연 | 있음 (헤드, 플래터 회전) | 없음 |
| 소음 | 있음 | 거의 없음 |
| 발열 | 상대적으로 높음 | 낮음 |
| 크기/무게 | 크고 무거움 | 작고 가벼움 |
| 용량 대비 가격 | 저렴 | 비쌈 |
| 수명 | 상대적으로 김 | 상대적으로 짧음 |

SSD는 노트북이나 소형 디바이스에 주로 사용됩니다. **단점은 단위 용량당 가격이 HDD보다 비싸고, 쓰기 횟수에 따른 수명이 HDD보다 짧다**는 점입니다.

:::tip SSD와 HDD 선택 기준
- **속도가 중요하고 용량이 크지 않아도 된다면**: SSD 선택
- **대용량 저장이 필요하고 비용을 낮추고 싶다면**: HDD 선택
- **서버 환경**: 보통 SSD + RAID 조합을 많이 사용합니다 (RAID는 뒤에서 설명합니다)
:::

## 🔩 2. 하드디스크의 구조와 데이터 접근 시간

### 왜 하드디스크 구조를 알아야 할까?

데이터베이스의 성능을 결정하는 핵심 요소 중 하나는 바로 **하드디스크에서 데이터를 얼마나 빠르게 가져올 수 있느냐**입니다. 이것을 이해하려면 먼저 하드디스크가 어떻게 생겼고, 어떻게 동작하는지를 알아야 합니다.

### 하드디스크의 구조

하드디스크는 겉보기에 단순한 금속 상자처럼 보이지만, 내부는 매우 정교한 구조로 이루어져 있습니다.

**주요 구성 요소:**

- **플래터(Platter)**: 실제 데이터가 자기적으로 기록되는 동그란 원판입니다. 마치 CD처럼 생겼고, 여러 장이 겹쳐서 구성됩니다.
- **트랙(Track)**: 플래터 위의 동심원(같은 중심을 가진 여러 개의 원)입니다. 데이터는 이 트랙 위에 저장됩니다.
- **섹터(Sector)**: 트랙을 여러 조각으로 나눈 것입니다. 피자를 조각으로 나눈 것처럼, 트랙을 부채꼴 모양으로 나눈 단위입니다.
- **실린더(Cylinder)**: 여러 플래터에서 **지름이 같은 트랙들의 집합**입니다. 여러 장의 플래터가 쌓여 있을 때, 동일한 위치(같은 지름)에 있는 모든 트랙을 묶어 실린더라고 합니다.
- **읽기/쓰기 헤드(Read/Write Head)**: 실제로 데이터를 읽고 쓰는 장치입니다. 과거 LP 레코드판(턴테이블)에서 음악을 읽어내는 바늘 침과 비슷한 역할을 합니다. 레코드판이 회전하면 바늘이 홈을 읽어 음악이 나오듯이, 플래터가 회전하면 헤드가 데이터를 읽어옵니다.
- **액세스 암(Access Arm)**: 읽기/쓰기 헤드가 부착된 팔(Arm)입니다. 안쪽과 바깥쪽으로 이동하며 원하는 트랙을 찾아갑니다.

> **정리**: 하드디스크는 마치 레코드판(플래터)이 회전하고, 바늘(헤드)이 달린 팔(액세스 암)이 움직이면서 원하는 위치의 데이터를 읽어내는 구조입니다.

---

### ⏱️ 데이터 접근 시간 (Access Time)

**접근 시간(Access Time)**이란 하드디스크에서 원하는 데이터를 찾아 **메인 메모리(RAM)로 전송하는 데 걸리는 총 시간**입니다.

여기서 중요한 점이 있습니다. 하드디스크에서 읽은 데이터는 **CPU로 직접 가지 않습니다**. 반드시 **메모리(RAM)**를 거쳐야 합니다. 그 흐름을 정리하면 다음과 같습니다.

```
하드디스크 → 메모리(RAM) → CPU (처리)
CPU → 메모리(RAM) → 하드디스크 (저장)
```

따라서 **Access Time = 하드디스크에서 메모리까지 데이터를 가져오는 시간**이며, 이것은 다음 세 가지 시간의 합입니다.

```
Access Time = Seek Time + Rotational Delay Time + Transfer Time
              (탐색 시간)  (회전 지연 시간)         (데이터 전송 시간)
```

---

#### 1️⃣ 탐색 시간 (Seek Time)

**탐색 시간**이란 액세스 암이 현재 위치에서 **원하는 트랙으로 이동하는 데 걸리는 시간**입니다.

예를 들어, 현재 헤드가 5번 트랙에 있는데 데이터가 20번 트랙에 있다면, 액세스 암이 20번 트랙까지 이동하는 시간이 바로 탐색 시간입니다. 서랍장에서 3번 서랍에 있는 파일을 꺼내러 가는 시간과 같습니다.

#### 2️⃣ 회전 지연 시간 (Rotational Delay Time)

**회전 지연 시간**이란 원하는 트랙을 찾은 다음, 플래터가 회전하면서 **원하는 섹터가 헤드 위치로 오기까지 기다리는 시간**입니다.

액세스 암은 트랙을 찾은 후 고정됩니다. 이제 플래터 자체가 회전하면서 원하는 섹터가 헤드 밑으로 올 때까지 기다려야 합니다. 마치 레코드판이 돌아서 내가 듣고 싶은 노래 시작 부분이 바늘 아래로 올 때까지 기다리는 시간과 같습니다.

#### 3️⃣ 데이터 전송 시간 (Transfer Time)

**데이터 전송 시간**이란 헤드가 데이터를 읽어 **메모리로 전송하는 데 걸리는 시간**입니다.

이 시간은 **전송하는 데이터의 양에 비례**합니다. 읽어와야 할 데이터가 많으면 많을수록 전송 시간이 길어집니다. 택배로 물건을 보낼 때, 물건이 많을수록 배송 시간이 오래 걸리는 것과 같습니다.

```
데이터 전송 시간 ∝ 전송 데이터 양 (비례 관계)
```

:::warning 데이터베이스 성능의 핵심
데이터베이스의 성능을 향상시킨다는 것은 결국 **I/O(Input/Output) 횟수를 줄이는 것**입니다. 하드디스크에서 메모리로 데이터를 읽고 쓰는 I/O 작업은 CPU 연산보다 압도적으로 느리기 때문에, I/O를 얼마나 효율적으로 줄이느냐가 DBMS 성능을 좌우합니다.
:::

## 🏗️ 3. 데이터베이스 내부 처리 과정

### 사용자의 요청이 어떻게 처리되는가?

사용자가 데이터베이스에 "학생 정보를 보여줘"라고 요청했을 때, 그 데이터는 어떤 경로로 사용자에게 전달될까요? 단순히 하드디스크에서 바로 튀어나오는 게 아닙니다. 다음과 같은 순서로 처리됩니다.

```
사용자
  ↓ 레코드 요청
DBMS (데이터베이스 관리 시스템)
  ↓ 물리적 레코드 요청
운영체제 (OS)
  ↓ 페이지 요청
파일 관리자 (File Manager)
  ↓ 해당 페이지 읽기 요청
디스크 관리자 (Disk Manager)
  ↓ 블록 형태로 페이지 반환
파일 관리자
  ↓ 페이지에서 레코드 추출 후 반환
DBMS
  ↓ 최종 결과 반환
사용자
```

이 흐름을 단계별로 자세히 살펴보겠습니다.

---

### 📌 디스크 관리자 (Disk Manager)

**디스크 관리자**는 운영체제의 한 구성 요소로, **모든 물리적 I/O 연산에 대한 책임**을 집니다. 쉽게 말해, 하드디스크에서 데이터를 실제로 꺼내오는 역할을 합니다.

디스크 관리자가 하는 일은 크게 두 가지입니다.

**① 페이지(블록) 단위로 데이터를 읽어옴**

하드디스크에서 데이터를 가져올 때는 레코드 하나씩 가져오는 것이 아니라 **페이지(Page)** 또는 **블록(Block)** 단위로 가져옵니다. 이것이 I/O의 기본 단위입니다.

**② 디스크 디렉터리(Disk Directory)를 통해 빠르게 위치 파악**

디스크 디렉터리는 마치 **도서관의 목차(index)**와 같습니다. `0번 실린더, 0번 트랙`에 위치하며, 각 데이터(테이블)가 하드디스크의 어느 위치에 저장되어 있는지 주소 정보를 가지고 있습니다.

예를 들어 학생 테이블, 교수 테이블, 과목 테이블이 있다고 할 때, 교수 테이블을 가져오려면 처음부터 하드디스크 전체를 뒤지는 것이 아니라 **디스크 디렉터리에서 교수 테이블의 시작 주소를 찾아 바로 그 위치부터 읽기 시작**합니다. 이렇게 하면 탐색 시간을 크게 줄일 수 있습니다.

---

### 📌 파일 관리자 (File Manager)

**파일 관리자**는 디스크 관리자가 가져온 페이지(블록) 안에서 **원하는 레코드를 정확히 찾아 DBMS에게 전달**하는 역할을 합니다.

파일 관리자는 **RID(Record ID, 레코드 식별자)**를 활용합니다. RID는 `페이지 번호 + 페이지 내 순서`로 구성됩니다.

예를 들어 `RID = #5, 8`이라면 **5번 페이지의 8번째 레코드**를 의미합니다. 파일 관리자는 이 RID를 알고 있으므로 페이지 안에서 처음부터 하나씩 뒤지지 않고, 해당 위치로 **바로 점프**해서 레코드를 가져올 수 있습니다.

> **정리**: 디스크 관리자는 **디스크 디렉터리**로 올바른 페이지를 빠르게 찾고, 파일 관리자는 **RID**로 페이지 안에서 올바른 레코드를 빠르게 찾습니다.

---

## 🗃️ 4. 클러스터링 (Clustering)

### 왜 클러스터링이 필요한가?

여러분이 윈도우를 오래 사용하다 보면 컴퓨터가 점점 느려지는 경험을 하셨을 겁니다. 그때 **"디스크 조각 모음"**을 실행하면 성능이 좋아지는 경우가 있죠. 이것이 바로 클러스터링의 개념과 연결됩니다.

처음에 프로그램을 설치할 때는 데이터가 하드디스크에 순서대로 저장됩니다.

```
[한글] → [엑셀] → [게임A]
```

하지만 엑셀을 삭제하고 그 자리에 용량이 큰 게임B를 설치하면, 게임B가 엑셀이 있던 자리와 그 뒤까지 이어서 저장됩니다.

```
[한글] → [게임B 일부] → [게임A] → [게임B 나머지]
```

이렇게 되면 게임B를 읽으려면 중간에 게임A를 건너뛰었다가 다시 뒤로 가야 하는 **비효율**이 생깁니다. 사용자 눈에 보이는 **논리적 순서**와 하드디스크에 실제로 저장된 **물리적 순서**가 달라진 것입니다.

**클러스터링(Clustering)**이란 이처럼 흩어진 데이터를 **논리적 순서와 물리적 순서가 일치하도록 재정렬**하는 작업입니다. 특정 **기준 키(Clustering Key)**를 정해서 그 키를 기준으로 데이터를 정렬하여 저장합니다.

---

### 클러스터링의 효과와 장점

```
❌ 클러스터링 없이 - 데이터가 여기저기 흩어진 경우:
[남학생A] → [여학생B] → [남학생C] → [여학생D] → [남학생E]

→ "남학생만 조회"하려면 전체를 다 읽으며 골라내야 함 (비효율)

✅ 클러스터링 적용 - 성별을 기준 키로 정렬한 경우:
[남학생A] → [남학생C] → [남학생E] → [여학생B] → [여학생D]

→ "남학생만 조회"하면 앞부분만 읽으면 됨 (효율적)
```

**클러스터링의 두 가지 핵심 장점:**

**① 같은 페이지에 연속 레코드가 모임 → I/O 횟수 감소**

연속으로 읽어야 할 레코드 A와 B가 같은 페이지에 들어있다면, 디스크에서 한 번만 읽어오면 됩니다. 만약 A는 10번 페이지에, B는 11번 페이지에 있다면 I/O가 두 번 발생하지만, 둘 다 같은 10번 페이지에 있다면 I/O는 한 번만 발생합니다.

**② 인접한 트랙 내에서 탐색 → 탐색 시간(Seek Time) 절약**

연속 레코드가 같은 트랙 위에 있다면, 액세스 암이 처음에 한 번만 이동하면 그 다음부터는 회전 지연 시간(Rotational Delay)만 발생합니다. 탐색 시간이 반복해서 발생하지 않으니 성능이 크게 향상됩니다.

---

### 클러스터링 키 선택 기준

클러스터링 키는 아무 컬럼이나 선택하면 안 됩니다. 다음 기준을 고려해야 합니다.

- **범위 검색이 자주 일어나는 컬럼**: 가격대 범위 조회, 지역별 조회, 성별 조회 등 **일부 데이터만 가져오는 경우**에 효과적입니다. 전체 데이터를 다 가져오는 경우에는 클러스터링 효과가 없습니다.
- **값이 여러 레코드에 걸쳐 같은 컬럼**: 학번이나 주민번호처럼 유일한 값보다는 성별, 지역, 가격대처럼 **같은 값이 여러 레코드에 분포**하는 컬럼이 적합합니다.
- **수정이 자주 발생하지 않는 컬럼**: 클러스터링 키 값이 변경되면 재정렬이 필요하므로, **잘 바뀌지 않는 값**(성별, 지역 등)을 키로 사용해야 효율적입니다.

:::warning 클러스터링의 단점
윈도우 디스크 조각 모음을 수행하면 그 동안 컴퓨터가 거의 사용 불가 상태가 됩니다. 클러스터링도 마찬가지로 **재정렬하는 작업 자체에 시간과 비용이 많이 듭니다**. 전체 데이터를 모두 가져오는 Full Scan 작업에는 오히려 클러스터링이 없는 것보다 성능이 떨어질 수 있습니다. 따라서 **부분 범위 검색이 많은 상황**에서 클러스터링을 적용해야 합니다.
:::

## ⚡ 5. RAID (Redundant Array of Independent Disks)

### RAID란 무엇이고 왜 필요한가?

**RAID**란 **여러 개의 하드디스크를 하나의 가상 디스크처럼 묶어서 사용하는 기술**입니다. 과거에는 하드디스크 하나의 용량이 작아서 여러 개를 묶어 대용량을 만들기 위해 사용했지만, 오늘날에는 두 가지 목적으로 주로 활용됩니다.

1. **병렬 처리로 속도 향상**: 여러 디스크에 데이터를 분산 저장하고 동시에 읽고 쓰면 속도가 빨라집니다.
2. **중복 저장으로 데이터 보호**: 디스크 하나가 고장나더라도 다른 디스크의 데이터로 복구할 수 있습니다.

> 서버를 운영할 때는 대부분 RAID 시스템을 사용합니다. 하드디스크 하나만 믿다가 고장나면 모든 데이터를 잃을 수 있기 때문입니다.

**RAID의 주요 장점:**

- **가용성(Availability) 향상**: 디스크가 고장나도 서비스가 중단되지 않습니다.
- **데이터 보호**: 중복 저장으로 데이터 손실을 방지합니다.
- **드라이브 접속 수 제한 우회**: 운영체제가 인식할 수 있는 물리적 디스크 수의 제한을 여러 디스크를 하나로 묶어서 우회합니다.
- **저렴한 비용으로 대용량 구현**: 작은 디스크 여러 개를 묶어 대용량 저장소를 만들 수 있습니다.
- **데이터 분산 처리**: I/O를 여러 디스크에 나눠 처리해 효율을 높입니다.

---

### RAID의 레벨별 종류

#### 🔹 RAID 0 — 스트라이핑 (Striping)

RAID 0은 데이터를 **여러 디스크에 나누어(분산) 저장**하는 방식입니다.

```
예시: 처리해야 할 데이터 [1, 2, 3]

❌ 디스크 1개 사용:
디스크1: [1] → [2] → [3]  (순차 처리, 30분 소요)

✅ RAID 0 - 디스크 3개 병렬 사용:
디스크1: [1]  ↗
디스크2: [2]  → 동시 처리 (10분으로 단축!)
디스크3: [3]  ↘
```

- **장점**: 처리 속도가 매우 빠릅니다. 디스크 수에 비례하여 속도가 향상됩니다.
- **단점**: 디스크 3개 중 하나라도 고장나면 **전체 데이터를 사용할 수 없습니다**. 원래 하나였던 데이터를 3개로 나눠놨기 때문에 부분 복구가 불가능합니다.

:::danger RAID 0의 치명적 단점
RAID 0은 데이터 복구 기능이 전혀 없습니다. 속도는 가장 빠르지만 디스크 하나라도 고장나면 전체 데이터가 손실됩니다. 중요한 데이터를 RAID 0에만 보관하는 것은 절대 피해야 합니다.
:::

---

#### 🔹 RAID 1 — 미러링 (Mirroring)

RAID 1은 **똑같은 데이터를 두 개의 디스크에 동시에 저장**하는 방식입니다.

```
디스크1 (주): [A, B, C, D]
디스크2 (복사본): [A, B, C, D]  ← 디스크1과 항상 동기화
```

- 디스크1이 고장나면 → 즉시 디스크2로 전환하여 서비스를 이어갑니다.
- **장점**: 완벽한 데이터 복구가 가능합니다.
- **단점**: 저장 공간과 비용이 2배로 필요합니다. 쓰기 작업도 두 곳에 동시에 해야 하므로 비용이 가장 높습니다.

---

#### 🔹 RAID 2 — 해밍 코드 방식 (현재 사용 안 함)

RAID 2는 스트라이핑을 하면서 **해밍 코드(Hamming Code)**를 이용해 에러를 감지하고 복구하는 방식입니다. 현재는 **사장된(더 이상 사용하지 않는) 방식**이므로 개념만 가볍게 알고 넘어가면 됩니다.

---

#### 🔹 RAID 3 — 바이트 단위 스트라이핑 + 전용 패리티 드라이브

RAID 3은 데이터를 **바이트(Byte) 단위**로 여러 디스크에 나누어 저장하고, 별도의 **패리티 드라이브(Parity Drive)**를 하나 두어 오류 복구에 활용합니다.

```
디스크1: [데이터 1의 일부]
디스크2: [데이터 2의 일부]
디스크3: [데이터 3의 일부]
패리티 드라이브: [모든 변경사항의 패리티 정보]
```

- **패리티(Parity)**란 데이터의 오류를 감지하고 복구하기 위한 추가 정보입니다.
- **장점**: RAID 0처럼 병렬 처리가 가능하고, 디스크 고장 시 복구할 수 있습니다.
- **단점**: 모든 쓰기 작업의 패리티 정보가 패리티 드라이브 하나에 집중되어 **병목 현상(Bottleneck)**이 발생합니다.

---

#### 🔹 RAID 4 — 블록 단위 스트라이핑 + 전용 패리티 드라이브

RAID 3과 구조는 같지만, 데이터를 **블록(Block) 단위**로 나누어 저장합니다. RAID 3보다 한 번에 전송하는 데이터 양이 커서 CAD 데이터, 이미지 등 대용량 파일 처리에 유리합니다.

- **단점**: RAID 3과 마찬가지로 패리티 드라이브에 병목 현상이 발생합니다.

---

#### 🔹 RAID 5 — 분산 패리티 (현재 가장 널리 사용)

RAID 5는 RAID 3과 RAID 4의 단점인 **패리티 드라이브 병목 현상을 해결**한 방식입니다. 패리티 정보를 별도 드라이브에 모아두지 않고, **모든 드라이브에 분산하여 저장**합니다.

```
디스크1: [데이터A] [데이터B] [패리티C]
디스크2: [데이터D] [패리티A] [데이터E]
디스크3: [패리티D] [데이터F] [데이터G]
```

- **장점**: 패리티 병목 현상이 없고, 성능도 우수하며, 디스크 하나 고장 시 복구 가능합니다.
- 현재 서버 환경에서 **가장 많이 사용되는 RAID 방식**입니다.

---

#### 🔹 RAID 6 — 이중 패리티

RAID 6은 **패리티 드라이브 자체가 고장나는 경우를 대비**한 방식입니다. 패리티 정보를 **2중으로 분산 저장**하여 디스크 2개가 동시에 고장나도 복구가 가능합니다.

- **장점**: 매우 높은 안정성을 제공합니다.
- **단점**: RAID 5보다 더 많은 저장 공간이 패리티에 사용됩니다.

---

#### 🔹 RAID 0+1 과 RAID 1+0

- **RAID 0+1**: 스트라이핑(RAID 0)한 것을 미러링(RAID 1)하는 방식입니다. 즉, 빠르게 나눈 것을 복사해 둡니다.
- **RAID 1+0**: 미러링(RAID 1)한 것을 스트라이핑(RAID 0)하는 방식입니다. 복사해 둔 것을 빠르게 나눕니다.

두 방식 모두 속도와 안정성을 동시에 얻을 수 있지만, 구조적 차이가 있으며 RAID 1+0이 복구 측면에서 더 유리합니다.

| RAID 레벨 | 속도 | 복구 기능 | 사용 예 |
|---|---|---|---|
| RAID 0 | ⭐⭐⭐⭐⭐ | ❌ | 임시 작업, 고속 처리 |
| RAID 1 | ⭐⭐ | ⭐⭐⭐⭐⭐ | 중요 데이터 보관 |
| RAID 5 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 일반 서버 (가장 보편적) |
| RAID 6 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 고안정성 서버 |
| RAID 1+0 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 고성능 + 고안정성 |

---

## 📦 6. 블로킹 (Blocking)과 블로킹 인수 (BFR)

### 블로킹이 왜 필요한가?

데이터베이스의 데이터는 하드디스크에서 메모리로 이동할 때, 레코드 하나씩 따로 가져오는 것이 아니라 **블록(Block)** 단위로 묶어서 가져옵니다. 이처럼 여러 레코드를 하나의 블록에 묶어서 전송하는 것을 **블로킹(Blocking)**이라고 합니다.

왜 이렇게 해야 할까요? 다음 예시를 보겠습니다.

```python
# 예시: 보내야 할 레코드가 3개 (레코드1, 레코드2, 레코드3)

# ❌ 레코드를 하나씩 따로 전송하는 경우:
# I/O 발생 횟수 = 3번
io_count_individual = 3  # 비효율적

# ✅ 하나의 블록에 3개를 묶어서 전송하는 경우:
# I/O 발생 횟수 = 1번
io_count_blocked = 1  # 효율적!
```

I/O 횟수가 3번에서 1번으로 줄어들었습니다. 데이터베이스 성능의 핵심이 I/O 횟수를 줄이는 것이라고 했으니, 블로킹이 얼마나 중요한지 알 수 있습니다.

---

### 블로킹 인수 (BFR, Blocking Factor)

**블로킹 인수(BFR)**란 **하나의 블록(페이지)에 저장될 수 있는 레코드의 개수**를 의미합니다.

```
블로킹 인수(BFR) = 블록 크기 ÷ 레코드 크기  (소수점은 버림)
```

:::note 블로킹 인수 계산 예시

- **레코드 크기**: 100 바이트
- **레코드 수**: 10,000개
- **블록(페이지) 크기**: 1,024 바이트

**① 블로킹 인수(BFR) 계산:**

```
BFR = 1024 ÷ 100 = 10.24 → 소수점 버림 → BFR = 10
```

한 블록에 레코드가 10개 들어갑니다. (11개를 넣으면 1,100바이트가 되어 블록 크기를 초과)

**② 필요한 블록 수 계산:**

```
필요한 블록 수 = 전체 레코드 수 ÷ BFR
             = 10,000 ÷ 10
             = 1,000개 블록 필요
```

:::

---

### 적절한 블록 크기 선택의 중요성

블록 크기는 너무 작아도, 너무 커도 문제입니다.

```
❌ 블록 크기가 너무 작은 경우:
- 한 블록에 레코드가 1~2개밖에 못 들어감
- I/O 횟수가 많아져 성능 저하

❌ 블록 크기가 너무 큰 경우:
- 레코드가 1개만 필요해도 블록 전체를 가져와야 함
- 나머지 공간은 낭비 → 메모리 비효율

✅ 적절한 블록 크기:
- 파일 크기와 접근 패턴을 고려하여 최적값 선택
- I/O 횟수와 메모리 낭비 사이의 균형을 맞춤
```

:::tip 파일 용어와 데이터베이스 용어 비교
데이터베이스에서 쓰는 용어와 파일 시스템에서 쓰는 용어가 다르지만 같은 개념을 가리킵니다.

| 파일 시스템 용어 | 데이터베이스 용어 |
|---|---|
| 필드 (Field) | 속성 (Attribute) |
| 레코드 (Record) | 튜플 (Tuple) |
| 파일 (File) | 릴레이션 (Relation, 테이블) |

필드들이 모여 레코드가 되고, 레코드가 모여 파일이 되고, 파일이 모여 데이터베이스가 됩니다.
:::

### 데이터 처리의 전체 흐름 요약

지금까지 배운 내용을 전체 흐름으로 정리하면 다음과 같습니다.

```
[하드디스크]
    ↓ (느림 — I/O 발생, Access Time 소요)
[메모리 (RAM)]   ← 블로킹으로 블록 단위 전송, I/O 횟수 최소화
    ↓ (빠름)
[CPU]            ← 실제 데이터 처리 수행
    ↓ (빠름)
[메모리 (RAM)]
    ↓ (느림 — I/O 발생)
[하드디스크]     ← 처리된 데이터를 다시 저장
```

- **하드디스크**: 대용량, 비휘발성, 접근 속도 매우 느림
- **메모리**: 중간 속도, 휘발성 (전원 끄면 사라짐)
- **CPU**: 매우 빠름, 처리 장치

데이터베이스 성능을 높이려면 결국 **하드디스크와 메모리 사이의 I/O 횟수를 최대한 줄이는 것**이 핵심입니다. 이를 위해 클러스터링, RAID, 블로킹 같은 개념이 사용됩니다.

## ✅ 핵심 정리

- **플래시 메모리**는 비휘발성 메모리로, 읽기 속도가 쓰기 속도보다 빠르고 충격에 강하다.
- **SSD**는 반도체 기반으로 HDD보다 빠르고 소음이 적지만, 가격이 비싸고 수명이 짧다.
- **하드디스크 구조**는 플래터, 트랙, 섹터, 실린더, 읽기/쓰기 헤드, 액세스 암으로 구성된다.
- **Access Time** = Seek Time(탐색 시간) + Rotational Delay Time(회전 지연 시간) + Transfer Time(데이터 전송 시간)이다.
- 데이터 처리 흐름은 **하드디스크 → 메모리 → CPU → 메모리 → 하드디스크** 순서이며, CPU가 하드디스크를 직접 접근하지 않는다.
- **디스크 관리자**는 디스크 디렉터리를 통해 원하는 페이지의 시작 주소를 빠르게 찾는다.
- **파일 관리자**는 RID(레코드 ID)를 통해 페이지 내 원하는 레코드를 빠르게 찾는다.
- **클러스터링**은 논리적 순서와 물리적 순서를 일치시켜 범위 검색 성능을 높이는 기법이다.
- **클러스터링 키**는 범위 검색이 많고, 수정이 드물고, 같은 값이 여러 레코드에 분포하는 컬럼으로 선택해야 한다.
- **RAID 0**은 스트라이핑으로 속도는 빠르지만 데이터 복구가 불가능하다.
- **RAID 1**은 미러링으로 완벽한 복구가 가능하지만 비용이 2배이다.
- **RAID 5**는 분산 패리티를 사용하여 현재 가장 널리 쓰이는 RAID 방식이다.
- **RAID 6**은 이중 패리티로 RAID 5보다 높은 안정성을 제공한다.
- **블로킹(Blocking)**은 여러 레코드를 하나의 블록으로 묶어 I/O 횟수를 줄이는 기법이다.
- **블로킹 인수(BFR)** = 블록 크기 ÷ 레코드 크기이며, 필요한 블록 수 = 전체 레코드 수 ÷ BFR이다.
- **데이터베이스 성능의 핵심**은 하드디스크와 메모리 사이의 I/O 횟수를 최소화하는 것이다.
