# 💾 하드디스크 구조와 데이터 접근 방식

데이터베이스가 실제로 어디에 저장되고, 어떻게 꺼내 쓰는지 궁금하신 적 있나요? SQL 쿼리 한 줄을 입력했을 때 내부에서 무슨 일이 일어나는지, 왜 어떤 쿼리는 빠르고 어떤 쿼리는 느린지 — 이 모든 질문의 답이 바로 이번 주제에 담겨 있습니다. 이번 글에서는 **데이터베이스의 물리적 저장 방식**, 즉 하드디스크의 구조와 데이터가 접근되는 과정을 처음 배우는 분도 이해할 수 있도록 아주 자세히 풀어드리겠습니다.

---

## 🗂️ 물리적 저장 매체의 종류

### 왜 저장 매체를 알아야 할까?

데이터베이스는 결국 어딘가에 **물리적으로 저장**됩니다. 그 저장 공간이 어떤 방식으로 작동하느냐에 따라 데이터를 읽고 쓰는 속도가 완전히 달라집니다. 마치 책을 도서관에서 찾을 때, 도서관이 잘 정리되어 있으면 빨리 찾고 엉망이면 오래 걸리는 것처럼, 저장 매체의 특성을 이해해야 데이터베이스 성능을 제대로 이해할 수 있습니다.

현재 우리가 주로 사용하는 물리적 저장 매체는 크게 세 가지입니다.

### 전통적인 하드디스크 (HDD)

우리가 오랫동안 사용해 온 가장 대표적인 저장 장치입니다. **자기(magnetic) 방식**으로 데이터를 저장하며, 내부에 실제로 돌아가는 원판(플래터)이 있습니다. 용량 대비 가격이 저렴하고 대용량 저장에 유리하지만, 물리적으로 움직이는 부품이 있기 때문에 속도가 느리고 충격에 약하다는 단점이 있습니다. 이번 글에서 가장 집중적으로 다룰 저장 매체입니다.

### 플래시 메모리 (Flash Memory)

플래시 메모리는 **메모리 칩 안에 정보를 유지하는 방식**을 사용합니다. 우리가 흔히 사용하는 USB 드라이브가 바로 플래시 메모리의 대표적인 예입니다.

플래시 메모리의 핵심 특징을 정리하면 다음과 같습니다.

| 특징 | 설명 |
|------|------|
| 비휘발성 | 전원이 꺼져도 데이터가 사라지지 않음 |
| 읽기/쓰기 속도 차이 | 읽는 속도가 쓰는 속도보다 훨씬 빠름 |
| 충격 내성 | 기계적 부품이 없어 충격에 강함 |
| 낮은 전력 소모 | 기존 HDD보다 전력 소모가 적음 |

> **비휘발성(Non-volatile)**이란, 전원이 꺼져도 데이터가 사라지지 않는 성질을 말합니다. 반대로 **휘발성(Volatile)**은 전원이 꺼지면 데이터가 사라지는 성질로, 우리가 흔히 쓰는 RAM(램)이 대표적인 휘발성 메모리입니다.

여기서 재미있는 사실이 하나 있습니다. 같은 용량(예: 64GB)의 USB 메모리를 구매할 때 만 원짜리도 있고 오만 원짜리도 있습니다. 이 가격 차이의 핵심은 바로 **읽기/쓰기 속도**입니다. 용량은 같아도 데이터를 읽고 쓰는 속도가 빠를수록 가격이 높아집니다.

### SSD (Solid State Drive, 솔리드 스테이트 드라이브)

SSD는 **반도체를 이용해서 데이터를 저장하는 방식**입니다. SSD를 분해해보면 내부에 무수히 많은 메모리 칩들이 반도체 기판에 연결되어 있습니다. 플래시 메모리를 여러 개 묶어서 하드디스크처럼 사용하는 방식이라고 생각하면 됩니다.

**SSD의 장점:**
- HDD보다 데이터 읽기/쓰기 속도가 훨씬 빠름
- 기계적 부품이 없어 지연(딜레이)이 거의 없음
- 소음과 발열이 적음
- 가볍고 소형화가 가능해 노트북, 소형 디바이스에 적합

**SSD의 단점:**
- 단위 용량당 가격이 HDD보다 비쌈
- 수명이 HDD보다 짧음 (쓰기 횟수에 제한이 있음)

:::info 저장 매체 비교 요약
현재 가장 널리 쓰이는 저장 매체를 정리하면 다음과 같습니다.

| 구분 | HDD | SSD | 플래시 메모리 |
|------|-----|-----|--------------|
| 속도 | 느림 | 빠름 | 보통 |
| 가격 | 저렴 | 비쌈 | 중간 |
| 충격 내성 | 약함 | 강함 | 강함 |
| 용량 | 대용량 가능 | 중간 | 소용량 |
| 소음 | 있음 | 없음 | 없음 |
:::

## 🔩 하드디스크의 내부 구조

### 왜 구조를 알아야 할까?

하드디스크가 어떻게 생겼는지 알아야, 데이터를 찾는 데 왜 시간이 걸리는지, 그 시간을 어떻게 줄일 수 있는지를 이해할 수 있습니다. 마치 도서관 책장이 어떻게 배치되어 있는지 알아야 책을 빨리 찾는 방법을 알 수 있는 것처럼요.

### 하드디스크의 핵심 구성 요소

하드디스크의 내부는 다음과 같이 구성되어 있습니다.

**① 플래터(Platter) — 데이터가 저장되는 원판**

자기(magnetic) 물질로 코팅된 둥근 원판입니다. 마치 오래된 LP 레코드판처럼 생겼습니다. 하드디스크 내부에는 이런 원판이 여러 장 겹쳐져 있습니다.

**② 트랙(Track) — 플래터 위의 동심원**

플래터 표면에는 **동심원(같은 중심을 가진 크기가 다른 원들)** 형태로 데이터 저장 공간이 나뉘어 있습니다. 이 각각의 원을 **트랙(Track)** 이라고 합니다. 마치 레코드판의 홈처럼, 데이터는 이 트랙을 따라 저장됩니다.

**③ 섹터(Sector) — 트랙을 나눈 조각**

각각의 트랙은 다시 여러 개의 **섹터(Sector)** 로 나뉩니다. 피자를 조각으로 자른 것처럼, 트랙을 부채꼴 모양으로 나눈 것이 섹터입니다. 실제 데이터는 이 섹터 단위로 저장됩니다.

**④ 실린더(Cylinder) — 같은 위치의 트랙 묶음**

하드디스크 안에는 플래터가 여러 장 있습니다. 이 여러 장의 플래터에서 **같은 지름을 가진 트랙들을 묶은 것**을 **실린더(Cylinder)** 라고 합니다. 위에서 내려다보면 원통(실린더) 모양이 되기 때문에 이런 이름이 붙었습니다.

**⑤ 읽기/쓰기 헤드(Read/Write Head) — 데이터를 읽고 쓰는 장치**

데이터를 실제로 읽고 쓰는 장치입니다. 옛날 LP 레코드플레이어를 생각해보세요. 레코드판이 빙글빙글 돌면, 옆에서 바늘이 달린 팔(암)이 판 위에 내려와 소리를 읽어내죠. 하드디스크의 읽기/쓰기 헤드도 똑같습니다. **액세스 암(Access Arm)** 이라는 팔 끝에 달려 있으며, 플래터가 회전하는 동안 원하는 데이터 위치로 이동해 데이터를 읽거나 씁니다.

:::note 하드디스크 작동 원리 요약
1. 플래터가 고속으로 회전합니다 (보통 분당 5,400~7,200번)
2. 액세스 암이 원하는 트랙 위치로 이동합니다
3. 플래터가 계속 회전하면서 원하는 섹터가 헤드 아래로 옵니다
4. 헤드가 해당 섹터의 데이터를 읽습니다
:::

## ⏱️ 데이터 접근 시간 (Access Time)

### 왜 접근 시간이 중요한가?

데이터베이스 성능의 핵심은 결국 **얼마나 빨리 데이터를 가져오느냐**에 달려 있습니다. 그리고 그 속도는 하드디스크에서 데이터를 읽어 메모리로 가져오는 데 걸리는 시간이 결정합니다. 이 시간을 **접근 시간(Access Time)** 이라고 하며, 세 가지 요소의 합으로 구성됩니다.

> **접근 시간(Access Time) = 탐색 시간 + 회전 지연 시간 + 데이터 전송 시간**

### 데이터가 이동하는 경로

하드디스크에서 데이터를 읽어와서 처리하는 경로는 다음과 같습니다.

```
하드디스크 → 메모리(RAM) → CPU(처리) → 메모리(RAM) → 하드디스크(저장)
```

중요한 포인트가 있습니다. CPU는 하드디스크에서 직접 데이터를 가져오지 못합니다. **반드시 메모리(RAM)를 거쳐야** 합니다. 하드디스크의 데이터가 메모리로 올라와야 비로소 CPU가 처리할 수 있습니다. 따라서 **하드디스크에서 메모리로 데이터를 얼마나 효율적으로 빠르게 가져오느냐**가 데이터베이스 성능을 결정하는 핵심입니다.

이때 하드디스크와 메모리 사이의 데이터 이동을 **I/O(Input/Output, 입출력)** 라고 하며, **I/O 횟수가 적을수록 성능이 좋습니다.**

### ① 탐색 시간 (Seek Time)

**탐색 시간**은 액세스 암(Access Arm)이 현재 위치에서 **원하는 트랙까지 이동하는 데 걸리는 시간**입니다.

레코드플레이어 바늘 달린 팔을 떠올려보세요. 1번 트랙에서 데이터를 읽다가 갑자기 10번 트랙의 데이터를 읽어야 한다면, 팔이 10번 트랙 위치로 이동해야 합니다. 이 이동 시간이 바로 탐색 시간입니다.

- 헤드가 멀리 이동할수록 탐색 시간이 길어집니다
- 헤드가 가까이 이동할수록 탐색 시간이 짧아집니다
- **가장 많은 시간이 걸리는 단계**입니다

### ② 회전 지연 시간 (Rotational Delay Time)

원하는 트랙을 찾았다면, 이제 그 트랙에서 **원하는 섹터가 헤드 아래로 돌아올 때까지 기다리는 시간**이 필요합니다. 이것이 **회전 지연 시간(Rotational Delay Time)** 입니다.

액세스 암은 원하는 트랙 위에 멈춰 있습니다. 하지만 원하는 섹터가 지금 헤드 반대편에 있다면, 플래터가 반바퀴 돌아올 때까지 기다려야 합니다. 이 기다리는 시간이 회전 지연 시간입니다.

- 최악의 경우: 원하는 섹터가 방금 헤드를 지나쳐서 거의 한 바퀴를 기다려야 하는 경우
- 최선의 경우: 원하는 섹터가 지금 바로 헤드 아래에 있는 경우 (지연 시간 = 0)
- **평균적으로 디스크가 반 바퀴 도는 시간**이 걸립니다

### ③ 데이터 전송 시간 (Transfer Time)

마침내 원하는 섹터가 헤드 아래에 왔습니다. 이제 헤드가 데이터를 읽어서 메모리로 전송합니다. 이 시간이 **데이터 전송 시간(Transfer Time)** 입니다.

- 전송할 데이터 양이 많을수록 전송 시간이 길어집니다
- 전송할 데이터 양이 적을수록 전송 시간이 짧아집니다
- **전송 시간은 전송 용량에 비례합니다**

### 접근 시간 정리

```
Access Time = Seek Time + Rotational Delay Time + Transfer Time
접근 시간  = 탐색 시간 + 회전 지연 시간       + 데이터 전송 시간
```

| 단계 | 역할 | 특징 |
|------|------|------|
| 탐색 시간 (Seek Time) | 원하는 트랙 찾기 | 이동 거리에 따라 가변, 가장 오래 걸림 |
| 회전 지연 시간 (Rotational Delay) | 원하는 섹터 찾기 | 최대 한 바퀴 회전 시간 |
| 전송 시간 (Transfer Time) | 데이터를 메모리로 전송 | 데이터 용량에 비례 |

:::tip 핵심 포인트
데이터베이스 성능을 높이려면 결국 **I/O 횟수를 줄여야** 합니다. I/O 횟수가 줄면 전체 접근 시간이 줄고, 시스템 성능이 향상됩니다. 이후 나오는 클러스터링, 블로킹, RAID 등의 개념은 모두 이 목표를 달성하기 위한 방법들입니다.
:::

## 🏗️ 데이터베이스의 내부 처리 구조

### 데이터가 처리되는 전체 흐름

사용자가 데이터를 요청했을 때 실제로 어떤 과정이 일어나는지 살펴보겠습니다. 전체 구조는 다음과 같습니다.

```
사용자 → DBMS → 파일 관리자 → 디스크 관리자 → 하드디스크
```

이 흐름을 단계별로 자세히 살펴보겠습니다.

**1단계: 사용자가 DBMS에 레코드 요청**

사용자가 "교수 테이블 데이터를 보여줘"라고 요청합니다.

**2단계: DBMS가 운영체제에 저장 레코드 요청**

DBMS는 내부 처리 후, 운영체제에 물리적인 저장 레코드를 달라고 요청합니다.

**3단계: 파일 관리자가 디스크 관리자에게 페이지 요청**

운영체제의 파일 관리자가 디스크 관리자에게 해당 데이터가 있는 페이지(블록)를 요청합니다.

**4단계: 디스크 관리자가 하드디스크에서 페이지를 가져옴**

디스크 관리자가 하드디스크에서 해당 페이지를 찾아 파일 관리자에게 전달합니다.

**5단계: 파일 관리자가 레코드를 찾아 DBMS에 반환**

파일 관리자는 가져온 페이지에서 필요한 레코드를 찾아 DBMS에 반환합니다.

**6단계: DBMS가 최종 결과를 사용자에게 전달**

DBMS가 처리 결과를 사용자에게 보여줍니다.

### 디스크 관리자 (Disk Manager)

**디스크 관리자**는 운영체제의 구성 요소 중 하나로, **모든 물리적 I/O 연산에 대한 책임**을 집니다. 쉽게 말해, 하드디스크에서 실제로 데이터(페이지/블록)를 꺼내오는 일을 합니다.

디스크 관리자에는 **디스크 디렉터리(Disk Directory)** 라는 것이 있습니다. 이것은 각 페이지(데이터 묶음)가 하드디스크의 어느 위치(몇 번 실린더, 몇 번 트랙)에 있는지 **위치 정보를 저장한 일종의 목차**입니다.

예를 들어 설명해봅시다. 도서관에 가서 "데이터베이스 책을 찾아주세요"라고 하면, 사서가 목차(카탈로그)를 보고 "3층 B-7 서가에 있습니다"라고 알려줍니다. 처음부터 책을 한 권씩 전부 확인하는 것이 아니라, 목차를 통해 바로 위치를 파악하는 거죠.

디스크 디렉터리도 마찬가지입니다. 교수 테이블이 5번 페이지부터 시작된다는 정보가 있으면, 처음부터 순서대로 읽지 않고 바로 5번 페이지로 이동해서 읽어올 수 있습니다. **이것이 검색 시간을 크게 단축시킵니다.**

### 파일 관리자 (File Manager)

**파일 관리자**는 디스크 관리자가 가져온 페이지 안에서 **특정 레코드의 위치를 찾아 관리하는 역할**을 합니다.

페이지는 마치 책의 한 페이지와 같습니다. 한 페이지에는 여러 줄의 내용이 있고, 내가 원하는 내용은 특정 줄에 있습니다. 파일 관리자는 그 특정 줄(레코드)을 빠르게 찾아주는 역할을 합니다.

이를 위해 **RID(Record ID, 레코드 식별자)** 라는 개념을 사용합니다. RID는 레코드의 정확한 주소로, **"몇 번 페이지의 몇 번째 레코드"** 형식으로 표현됩니다.

```
RID = #페이지번호 + 레코드번호
예시: RID = #5, 8  → 5번 페이지의 8번째 레코드
```

RID가 있으면 파일 관리자는 처음부터 하나씩 레코드를 확인할 필요 없이 **바로 원하는 레코드로 이동**할 수 있습니다.

:::info 디스크 관리자 vs 파일 관리자 역할 비교
| 구분 | 디스크 관리자 | 파일 관리자 |
|------|-------------|------------|
| 역할 | 하드디스크에서 페이지(블록) 단위로 데이터 가져오기 | 페이지 안에서 특정 레코드 찾기 |
| 사용 도구 | 디스크 디렉터리 (페이지 위치 정보) | RID (레코드 위치 정보) |
| 접근 단위 | 페이지(블록) | 레코드 |
:::

## 🧩 클러스터링 (Clustering)

### 클러스터링이 필요한 이유

여러분, 윈도우 컴퓨터를 오래 쓰다 보면 성능이 떨어지는 걸 경험해보셨죠? 그래서 **"조각 모음(Defragmentation)"** 을 해본 분들도 있을 겁니다. 조각 모음을 하고 나면 컴퓨터 성능이 좋아집니다. 클러스터링은 바로 이 조각 모음과 비슷한 개념입니다.

왜 조각 모음이 필요한지부터 이해해 봅시다. 처음에 하드디스크에 파일을 저장하면 순서대로 잘 저장됩니다.

```
[한글] [엑셀] [게임A]
```

그런데 엑셀을 삭제하고 새로운 대용량 게임B를 설치하면 이렇게 됩니다.

```
[한글] [게임B_1부분] [게임A] [게임B_2부분]
```

이제 게임B를 실행하려면 [게임B_1부분]을 읽다가 중간에 건너뛰고 [게임B_2부분]을 읽어야 합니다. 이렇게 데이터가 **논리적 순서와 물리적 저장 순서가 달라지면** 성능이 떨어집니다.

### 클러스터링의 개념

**클러스터링(Clustering)** 이란, **사용자가 논리적으로 보는 데이터 순서와 실제 하드디스크에 물리적으로 저장된 순서를 일치시키는 작업**입니다.

데이터를 특정 기준(클러스터링 키)으로 정렬해서 저장하면, 관련된 데이터들이 하드디스크 상에서 가까이 모여 있게 됩니다. 그러면 해당 데이터를 읽어올 때 헤드가 멀리 이동할 필요 없이 연속해서 빠르게 읽을 수 있습니다.

### 클러스터링의 장점

**① 같은 페이지에 연속 데이터가 있는 경우**

만약 내가 읽어야 할 레코드 A와 B가 같은 페이지에 있다면, **I/O가 딱 한 번만 발생**합니다. 한 번 페이지를 읽어오면 A와 B 모두 가져올 수 있기 때문입니다. 클러스터링이 없으면 A는 10번 페이지에, B는 20번 페이지에 따로 있어서 I/O가 두 번 발생합니다.

**② 인접한 페이지에 연속 데이터가 있는 경우**

A와 B가 같은 페이지는 아니지만 같은 트랙 안에 있다면, **탐색 시간(Seek Time)은 한 번만 발생**하고 이후로는 회전 지연 시간만 발생합니다. 트랙을 이동하지 않아도 되기 때문에 성능이 크게 향상됩니다.

### 클러스터링 키 선택 기준

클러스터링은 아무 기준(키)으로나 하는 게 아닙니다. **적절한 클러스터링 키를 선택하는 것이 중요합니다.**

좋은 클러스터링 키의 조건:

- **범위 검색이 자주 발생하는 컬럼**: 성별(남/여), 지역, 가격 범위 등 "남자 데이터만", "서울 데이터만", "1만원~2만원 사이 데이터만" 같은 검색에 효과적입니다.
- **수정이 자주 일어나지 않는 컬럼**: 성별은 거의 바뀌지 않습니다. 반면 가격은 자주 바뀔 수 있습니다. 클러스터링 키가 자주 바뀌면 클러스터링을 계속 다시 해야 해서 오히려 성능이 떨어집니다.
- **인덱스를 활용하기 어려운 컬럼**: 학번, 주민등록번호처럼 모두 다른 값(고유값)은 인덱스가 효과적이지만, 성별·지역처럼 동일한 값이 많은 컬럼은 인덱스보다 클러스터링이 더 유리합니다.

### 클러스터링의 주의사항

:::warning 전체 스캔 시 클러스터링은 역효과
클러스터링은 **부분 데이터를 검색할 때** 효과적입니다. 만약 테이블의 모든 데이터를 가져와야 한다면, 굳이 클러스터링 작업에 시간을 쓸 필요가 없습니다.

마치 조각 모음(Defragmentation)처럼, 클러스터링 작업 자체가 시간이 오래 걸립니다. 조각 모음을 하는 동안 컴퓨터가 매우 느려지듯이, 클러스터링을 수행하는 동안에는 다른 작업에 지장이 생길 수 있습니다. 전체 데이터를 스캔하는 경우에는 오히려 속도가 떨어질 수 있으니 주의하세요.
:::

## 🛡️ RAID (레이드)

### RAID가 등장한 배경

예전에는 하드디스크 한 개의 용량이 매우 작았습니다. 수십 GB도 큰 용량이던 시절이 있었죠. 그래서 용량이 작은 하드디스크 여러 개를 묶어서 하나의 대용량 저장 장치처럼 사용하는 방법이 필요했습니다. 이것이 **RAID(Redundant Array of Independent Disks, 독립 디스크들의 중복 배열)** 의 시작입니다.

지금은 하드디스크 하나의 용량이 충분히 크기 때문에, 용량 확장보다는 다음 두 가지 목적으로 RAID를 사용합니다.

1. **속도 향상**: 여러 디스크에 데이터를 나눠 동시에 처리 (병렬 처리)
2. **데이터 보호**: 디스크 하나가 고장나도 복구 가능

> RAID는 여러 개의 하드디스크를 하나의 **가상 디스크(Virtual Disk)** 로 묶어서 대용량 고성능 저장 장치로 활용하는 기술입니다.

**RAID의 주요 장점:**
- **가용성 향상**: 한 드라이브가 고장나도 서비스 중단 없음
- **데이터 보호**: 중복 저장으로 데이터 손실 방지
- **드라이브 접속 제한 우회**: 운영체제의 물리적 드라이브 연결 개수 제한을 피할 수 있음
- **대용량 구현**: 저렴한 비용으로 대용량 스토리지 구축
- **성능 향상**: 데이터 분산 처리로 I/O 속도 향상

### RAID 0 — 스트라이핑 (Striping)

**RAID 0**은 데이터를 여러 개의 디스크에 **나눠서(분산해서) 저장하는 방식**입니다. 이 분산 방식을 **스트라이핑(Striping)** 이라고 합니다.

```
[데이터 전체] → 분할 → [디스크1: 데이터1/3] [디스크2: 데이터2/3] [디스크3: 데이터3/3]
```

예를 들어 30분짜리 작업이 있다고 합시다.

- **RAID 없이 디스크 1개 사용**: 30분 소요
- **RAID 0으로 디스크 3개 사용**: 각 디스크에 1/3씩 나누어 동시에 처리 → **10분**으로 단축!

**장점:** 속도가 매우 빠름 (분산 처리)

**단점:** 디스크 3개 중 **1개라도 고장나면 전체 데이터를 잃어버림**. 원래 하나였던 것을 3개로 나눠놨기 때문에 하나라도 없으면 전체가 망가집니다.

:::danger RAID 0의 치명적 단점
RAID 0은 **백업이나 복구 기능이 전혀 없습니다.** 단 하나의 디스크 장애가 전체 데이터 손실로 이어집니다. 중요한 데이터를 저장하는 용도로는 절대 사용하지 마세요. 속도가 중요하고 데이터 손실을 감수할 수 있는 임시 작업용으로만 사용하세요.
:::

### RAID 1 — 미러링 (Mirroring)

**RAID 1**은 데이터를 **두 개의 디스크에 동일하게 복사**하는 방식입니다. 마치 거울처럼 똑같은 내용이 두 곳에 저장되기 때문에 **미러링(Mirroring)** 이라고 합니다.

```
[데이터] → 동시 저장 → [디스크1: 데이터] [디스크2: 데이터(완전 복사본)]
```

디스크 1이 고장나면 즉시 디스크 2로 전환해서 서비스를 계속할 수 있습니다. 데이터가 100% 동기화되어 있기 때문에 **완벽한 복구**가 가능합니다.

**장점:** 하나가 고장나도 완벽하게 복구 가능

**단점:** 저장도 두 군데, 처리도 두 군데 해야 하므로 **비용이 가장 높은 방식**입니다.

### RAID 2 — 해밍 코드 방식

RAID 0처럼 스트라이핑 방식으로 데이터를 나누되, **해밍 코드(Hamming Code)** 라는 오류 수정 방식을 사용해 고장 시 복구가 가능합니다. 속도도 빠르고 복구도 되는 좋은 방식이었지만, **현재는 사용하지 않는 방식**입니다. 참고로만 알아두세요.

### RAID 3 — 바이트 단위 스트라이핑 + 전용 패리티 디스크

RAID 0의 스트라이핑 방식을 사용하되, 별도로 **패리티 드라이브(Parity Drive)** 를 하나 더 추가하는 방식입니다.

```
[디스크1: 데이터1] [디스크2: 데이터2] [디스크3: 데이터3] [패리티 디스크: 오류 복구 정보]
```

**패리티(Parity)** 란 오류 검출 및 수정을 위한 추가 정보입니다. 데이터 디스크 중 하나가 고장나면, 패리티 디스크의 정보를 이용해 **손실된 데이터를 복구**할 수 있습니다.

**데이터 전송 단위:** 바이트(Byte) 단위

**장점:** RAID 0의 속도 + 장애 복구 가능

**단점:** 모든 디스크의 변경 사항이 패리티 드라이브로 집중되어 **패리티 드라이브에 병목 현상(Bottleneck)** 이 발생합니다. 패리티 디스크가 하나뿐이니, 모든 쓰기 작업마다 패리티 디스크도 업데이트해야 해서 그 드라이브만 과부하가 걸립니다.

### RAID 4 — 블록 단위 스트라이핑 + 전용 패리티 디스크

RAID 3과 구조는 동일하지만, **데이터 전송 단위가 블록(Block) 단위**로 더 큽니다. 대량의 데이터(이미지, CAD 파일 등)를 저장하고 관리할 때 유리합니다.

**단점:** RAID 3과 마찬가지로 **패리티 드라이브의 병목 현상** 문제가 있습니다.

### RAID 5 — 분산 패리티 (현재 가장 널리 사용되는 방식)

RAID 3과 4의 **병목 현상 문제를 해결**한 방식이 바로 **RAID 5**입니다. 현재 서버 환경에서 가장 많이 사용됩니다.

핵심 아이디어는 간단합니다. 패리티 정보를 하나의 전용 디스크에 몰아넣는 대신, **모든 디스크에 패리티 정보를 나누어 분산 저장**합니다.

```
[디스크1: 데이터A + 패리티B] [디스크2: 데이터B + 패리티C] [디스크3: 데이터C + 패리티A]
```

이렇게 하면 특정 디스크 하나에 패리티가 집중되지 않아 병목 현상이 해소됩니다.

**장점:** 스트라이핑의 속도 + 패리티를 통한 복구 가능 + 병목 현상 없음

### RAID 6 — 이중 분산 패리티

RAID 5에서 **패리티 드라이브 자체가 고장나는 경우**를 대비한 방식입니다. 패리티 정보를 **2중으로** 저장합니다. 즉, 디스크 2개가 동시에 고장나도 데이터를 복구할 수 있습니다.

**장점:** RAID 5보다 훨씬 높은 안정성

**단점:** 저장 공간 효율이 조금 떨어지고, 구성 비용이 더 높음

### RAID 0+1 vs RAID 1+0

- **RAID 0+1**: RAID 0(스트라이핑)으로 묶은 것을 RAID 1(미러링)으로 복제
- **RAID 1+0**: RAID 1(미러링)으로 복제한 것을 RAID 0(스트라이핑)으로 분산

두 방식 모두 속도와 안정성을 동시에 추구하지만, 구조적 차이로 인해 장애 허용 범위가 다릅니다.

:::info RAID 레벨 비교 요약
| RAID 레벨 | 방식 | 속도 | 복구 가능 | 사용 여부 |
|-----------|------|------|-----------|----------|
| RAID 0 | 스트라이핑 | 매우 빠름 | ❌ | 사용 중 |
| RAID 1 | 미러링 | 보통 | ✅ (완벽) | 사용 중 |
| RAID 2 | 해밍 코드 | 빠름 | ✅ | ❌ 미사용 |
| RAID 3 | 바이트 스트라이핑 + 전용 패리티 | 빠름 | ✅ | 제한적 |
| RAID 4 | 블록 스트라이핑 + 전용 패리티 | 빠름 | ✅ | 제한적 |
| RAID 5 | 분산 패리티 | 빠름 | ✅ | ✅ 가장 널리 사용 |
| RAID 6 | 이중 분산 패리티 | 빠름 | ✅ (2개 고장 허용) | 고가용성 환경 |
:::

## 📦 파일 구조와 블로킹 (Blocking)

### 파일 구조의 기본 개념

데이터베이스에서 사용하는 용어와 파일 시스템에서 사용하는 용어가 조금 다릅니다. 다음 표를 참고하세요.

| 파일 시스템 개념 | 데이터베이스 개념 |
|----------------|----------------|
| 필드 (Field) | 속성 (Attribute) |
| 레코드 (Record) | 튜플 (Tuple) |
| 파일 (File) | 릴레이션/테이블 (Relation) |
| 파일들의 집합 | 데이터베이스 (Database) |

- **필드(Field)**: 데이터의 가장 작은 단위. 예: 이름, 나이, 주소
- **레코드(Record)**: 필드들이 모여서 만들어진 한 행. 예: 홍길동/25세/서울
- **파일(File)**: 레코드들이 모인 집합
- **데이터베이스**: 파일들이 모인 집합

### 하드디스크에서의 데이터 처리 구조

데이터베이스는 하드디스크에 저장됩니다. 하드디스크의 특성을 다시 정리하면:

- **장점**: 용량이 크고, 전원이 꺼져도 데이터가 살아있음 (비휘발성)
- **단점**: 접근 속도가 매우 느림

CPU는 하드디스크 데이터를 직접 처리할 수 없고, 반드시 메모리를 거쳐야 합니다.

```
하드디스크 → 메모리 → CPU(처리) → 메모리 → 하드디스크
```

### 블로킹(Blocking)이란?

**블로킹(Blocking)** 이란, 여러 개의 레코드를 **하나의 블록(Block = 페이지)에 묶어서 한 번에 이동시키는 방식**입니다.

왜 필요할까요? 다음 예시를 보겠습니다.

- **레코드 개별 전송**: 레코드가 3개라면 I/O가 3번 발생
- **블록 단위 전송**: 레코드 3개를 하나의 블록에 담아서 I/O가 1번만 발생

```
❌ 레코드 개별 전송 (비효율적):
[레코드1] → I/O 1번
[레코드2] → I/O 1번
[레코드3] → I/O 1번
= 총 I/O 3번 발생

✅ 블록 단위 전송 (효율적):
[레코드1 + 레코드2 + 레코드3] → I/O 1번
= 총 I/O 1번 발생
```

I/O 횟수가 적을수록 성능이 좋아진다고 했죠? 블로킹은 I/O 횟수를 줄이는 핵심 방법입니다.

### 블로킹 인수 (Blocking Factor, BFR)

**블로킹 인수(BFR)** 는 **한 블록에 저장되는 레코드의 개수**를 의미합니다.

```
블로킹 인수(BFR) = 블록 크기 ÷ 레코드 크기 (소수점 이하 버림)
```

블로킹 인수를 크게 하면 I/O 횟수는 줄어들지만, 필요 이상으로 많은 데이터를 메모리로 불러와 **메모리 낭비**가 생깁니다. 반대로 너무 작게 하면 I/O 횟수가 많아집니다. **적절한 블록 크기를 선택하는 것이 중요합니다.**

### 블로킹 계산 예제

```python
# 블로킹 인수와 필요 블록 수 계산 예제

# 주어진 조건
record_size = 100      # 레코드 하나의 크기: 100바이트
total_records = 10000  # 전체 레코드 수: 10,000개
block_size = 1024      # 블록 하나의 크기: 1,024바이트

# 1단계: 블로킹 인수(BFR) 계산
# 한 블록에 100바이트짜리 레코드가 몇 개 들어가는지 계산
bfr = block_size // record_size  # 정수 나눗셈 (소수점 버림)
print(f"블로킹 인수(BFR): {bfr}개")
# 결과: 1024 ÷ 100 = 10개 (11개가 되면 1,100바이트가 되어 블록 크기 초과)

# 2단계: 필요한 블록 수 계산
# 전체 레코드 10,000개를 저장하려면 몇 개의 블록이 필요한지 계산
import math
required_blocks = math.ceil(total_records / bfr)  # 올림 처리 (레코드가 남으면 블록 하나 더 필요)
print(f"필요한 블록 수: {required_blocks}개")
# 결과: 10,000 ÷ 10 = 1,000개 블록 필요
```

**계산 결과 풀이:**

1. **블로킹 인수(BFR)**: `1,024 ÷ 100 = 10.24` → 소수점 버림 → **BFR = 10**
   - 한 블록(1,024바이트)에 100바이트짜리 레코드가 **10개**만 들어갑니다
   - 11개를 넣으면 1,100바이트가 되어 블록 크기(1,024바이트) 초과

2. **필요한 블록 수**: `10,000 ÷ 10 = 1,000`
   - 레코드 10,000개를 저장하려면 블록이 **1,000개** 필요합니다

```
❌ 블로킹 없이 레코드 단위 I/O:
레코드 10,000개 → I/O 10,000번 발생

✅ 블로킹 적용 후 블록 단위 I/O:
블록 1,000개 → I/O 1,000번만 발생
(무려 I/O 횟수가 1/10로 줄어듦!)
```

:::tip 블로킹 핵심 정리
- 블로킹 인수가 크면 → I/O 횟수 감소 ✅, 메모리 낭비 증가 ⚠️
- 블로킹 인수가 작으면 → I/O 횟수 증가 ⚠️, 메모리 효율 좋음 ✅
- **파일 크기와 사용 패턴을 고려해서 적절한 블록 크기를 결정**하는 것이 중요합니다
:::

## ⚠️ 주의사항 및 실무 팁

### 클러스터링 사용 시 주의할 점

클러스터링을 무조건 좋은 것이라고 생각하면 안 됩니다. 다음 상황을 잘 구분해서 사용하세요.

**✅ 클러스터링이 효과적인 경우:**
- 전체 데이터 중 **일부만 조회**하는 경우 (예: 남자 데이터만, 서울 데이터만)
- **범위 검색**이 자주 발생하는 경우 (예: 가격이 1만원~5만원 사이)
- 클러스터링 키 컬럼의 **수정이 드문 경우** (예: 성별, 지역)

**❌ 클러스터링이 비효율적인 경우:**
- **전체 데이터를 모두 조회**해야 하는 경우
- 클러스터링 키 컬럼이 **자주 변경**되는 경우
- 데이터 변경이 매우 빈번한 시스템

### RAID 선택 시 고려사항

:::warning RAID 선택 기준
- **속도만 중요하고 데이터 손실을 감수할 수 있다면**: RAID 0
- **데이터 보호가 최우선이고 비용을 감수할 수 있다면**: RAID 1
- **속도와 보호의 균형이 필요한 일반적인 서버 환경**: RAID 5 (가장 일반적)
- **최고 수준의 안정성이 필요한 중요 시스템**: RAID 6
:::

### I/O 최소화 전략

데이터베이스 성능 향상의 핵심은 **I/O 횟수를 줄이는 것**입니다. 이를 위한 주요 전략을 정리하면:

1. **블로킹**: 여러 레코드를 블록 단위로 묶어 한 번에 전송
2. **클러스터링**: 관련 데이터를 물리적으로 가까이 배치해 불필요한 헤드 이동 감소
3. **RAID**: 여러 디스크에 분산 처리로 병렬 I/O 실현
4. **버퍼링**: 자주 사용하는 데이터를 메모리에 캐시해 디스크 접근 최소화

:::danger 절대 하지 말아야 할 것
- **중요한 데이터를 RAID 0만으로 관리**: 디스크 하나가 고장나면 모든 데이터가 사라집니다. 백업 전략을 반드시 병행하세요.
- **클러스터링 키를 자주 변경되는 컬럼으로 설정**: 데이터 변경 시마다 클러스터링을 다시 해야 해서 오히려 성능이 크게 저하됩니다.
- **블록 크기를 무조건 크게 설정**: 불필요하게 많은 데이터가 메모리로 올라와 메모리 부족 현상이 발생할 수 있습니다.
:::

## 📋 핵심 정리

이번 글에서 배운 내용을 핵심만 간추려 정리합니다.

- **물리적 저장 매체**는 HDD, SSD, 플래시 메모리로 나뉘며 각각 속도·비용·특성이 다르다
- **하드디스크 구조**는 트랙(동심원), 섹터(트랙의 조각), 실린더(같은 지름의 트랙 묶음), 읽기/쓰기 헤드로 구성된다
- **접근 시간(Access Time)** = 탐색 시간(Seek Time) + 회전 지연 시간(Rotational Delay) + 전송 시간(Transfer Time)
- **데이터 처리 경로**는 하드디스크 → 메모리 → CPU 순이며, CPU는 하드디스크를 직접 처리하지 못한다
- **I/O 횟수**가 DBMS 성능을 결정하는 가장 중요한 요소다 (횟수가 적을수록 성능이 좋다)
- **디스크 관리자**는 디스크 디렉터리를 통해 페이지 위치를 파악하고, 물리적 I/O를 담당한다
- **파일 관리자**는 RID(레코드 ID)를 통해 페이지 내 특정 레코드를 빠르게 찾는다
- **클러스터링**은 논리적 순서와 물리적 저장 순서를 일치시켜 I/O를 줄이는 방법이다
- **클러스터링**은 부분 데이터 범위 검색에 효과적이며, 전체 스캔 시에는 오히려 비효율적이다
- **RAID 0**: 스트라이핑(속도↑, 복구 불가), **RAID 1**: 미러링(복구 가능, 비용↑)
- **RAID 5**는 분산 패리티 방식으로 현재 가장 널리 사용되는 RAID 레벨이다
- **블로킹(Blocking)** 은 여러 레코드를 블록 단위로 묶어 I/O 횟수를 줄이는 기법이다
- **블로킹 인수(BFR)** = 블록 크기 ÷ 레코드 크기로 계산하며, 한 블록에 저장되는 레코드 수를 의미한다
- **필요한 블록 수** = 전체 레코드 수 ÷ 블로킹 인수로 계산한다
