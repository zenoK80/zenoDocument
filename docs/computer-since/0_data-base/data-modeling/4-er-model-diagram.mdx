# 🗺️ ER 모델과 ER 다이어그램 표기법

데이터베이스를 설계할 때 가장 먼저 해야 할 일은 무엇일까요? 바로 **현실 세계를 체계적으로 표현하는 것**입니다. 학교에는 학생이 있고 교수가 있고, 학생은 과목을 수강하고, 교수는 학생을 지도합니다. 이런 복잡한 현실 세계의 관계를 컴퓨터가 이해할 수 있는 데이터베이스로 어떻게 옮길 수 있을까요? 이때 등장하는 것이 바로 **ER 모델(Entity-Relationship Model, 개체-관계 모델)** 입니다.

이 문서는 ER 모델과 ER 다이어그램의 기본 개념부터 표기법, 개체·속성·관계의 종류, 연결함정, 그리고 확장 ER 모델까지 초보자도 이해할 수 있도록 아주 자세하게 설명합니다. 끝까지 함께 따라오세요!

---

## 🏗️ 데이터 모델링이란 무엇인가?

### 현실 세계를 데이터베이스로 옮기는 과정

우리가 살고 있는 현실 세계에는 수많은 사물과 사람, 그리고 그들 사이의 관계가 존재합니다. 예를 들어 학교라는 세계에는 학생, 교수, 과목이 존재하고, 학생은 교수로부터 지도를 받으며, 과목을 수강합니다. 이런 현실 세계의 복잡한 상황을 그대로 컴퓨터 데이터베이스에 저장하려면 **체계적인 설계 과정**이 필요합니다. 그 설계 과정 전체를 우리는 **데이터 모델링**이라고 부릅니다.

데이터 모델링은 크게 세 단계로 나뉩니다. 이 세 단계는 이름이 비슷해서 헷갈릴 수 있으니 차이점을 명확히 이해하는 것이 중요합니다.

### 세 가지 모델링 단계

| 단계 | 이름 | 결과물 | 설명 |
|------|------|--------|------|
| 1단계 | **개념적 모델링** | 개념적 구조 (ER 다이어그램) | 현실 세계를 추상적 개념(도형)으로 표현 |
| 2단계 | **데이터 모델링** | 논리적 구조 (테이블) | 개념적 구조를 테이블 형태로 변환 |
| 3단계 | **데이터 구조화** | 물리적 구조 (실제 파일) | 테이블을 컴퓨터 디스크에 저장 가능한 형태로 변환 |

**개념적 모델링**은 마치 건축가가 집을 짓기 전에 도면을 그리는 것과 같습니다. "이 건물에는 거실, 침실, 화장실이 필요하고, 각 방들은 이런 관계로 연결된다"는 것을 도면에 표시하듯이, 데이터베이스 설계자는 "학생, 교수, 과목이라는 개체가 필요하고, 그들 사이에는 이런 관계가 있다"는 것을 ER 다이어그램으로 표현합니다.

**데이터 모델링**은 그 도면을 실제 시공 도면으로 변환하는 것입니다. 추상적인 ER 다이어그램을 실제 데이터베이스에서 사용할 수 있는 테이블(표) 형태로 바꿉니다.

**데이터 구조화**는 시공 도면을 바탕으로 실제 건물을 짓는 것입니다. 테이블 설계를 실제 컴퓨터의 하드디스크에 물리적으로 저장할 수 있는 형태로 최종 변환합니다.

> 💡 **흐름 요약**: 현실 세계 → (개념적 모델링) → ER 다이어그램 → (데이터 모델링) → 테이블 → (데이터 구조화) → 물리적 저장

### 데이터베이스 설계 단계 복습

데이터베이스를 설계하는 전체 단계를 다시 한번 정리해보겠습니다.

1. **요구사항 수집 및 분석**: 사용자가 원하는 것이 무엇인지 파악합니다. "학생 정보가 필요하다", "교수와 학생의 지도 관계를 저장해야 한다" 같은 요구사항을 모읍니다.

2. **개념적 설계**: ER 다이어그램을 만드는 단계입니다. 이 단계에서 중요한 특징은 **DBMS(데이터베이스 관리 시스템)와 하드웨어 모두에 독립적**이라는 점입니다. 즉, Oracle을 쓸 것인지 MySQL을 쓸 것인지, 서버가 어떤 종류인지 전혀 신경 쓰지 않고 순수하게 개념적으로만 설계합니다.

3. **논리적 설계**: 개념적 설계 결과를 관계형, 계층형, 네트워크형 등의 논리적 구조로 변환합니다. 이 단계는 **DBMS에는 종속적이지만 하드웨어에는 독립적**입니다. 어떤 종류의 데이터베이스를 쓸지는 결정하지만, 서버 사양은 신경 쓰지 않습니다.

4. **물리적 설계**: 실제 디스크에 데이터를 어떻게 저장할지 설계합니다. 이 단계는 **DBMS와 하드웨어 모두에 종속적**입니다. 실제 저장 공간, 인덱스 구조, 파일 배치 등 구체적인 사항을 결정합니다.

## 📦 데이터 모델의 구성 요소 (SOC)

### 데이터 모델이란?

데이터 모델은 단순히 "데이터를 어떻게 저장할까?"만을 다루는 것이 아닙니다. 데이터 모델은 **세 가지 핵심 요소**로 구성됩니다. 이 세 가지를 영어 첫 글자를 따서 **SOC**라고 부릅니다.

### S - 구조 (Structure, 스트럭처)

**구조**란 데이터베이스에서 표현해야 할 대상(개체)과 그 관계를 명세한 것입니다. 마치 집의 설계도에서 "방이 몇 개이고, 각 방의 크기는 얼마이며, 방들이 어떻게 연결되어 있는가"를 표시하는 것처럼, 구조는 "어떤 개체가 있고, 그 개체는 어떤 속성을 가지며, 개체들이 어떤 관계로 연결되어 있는가"를 정의합니다.

예를 들어, 학생이라는 개체는 학번, 이름, 주소, 학년이라는 속성을 가지고 있고, 교수라는 개체와는 "지도" 관계로 연결되어 있다는 것 — 이 모든 것이 구조에 해당합니다.

### O - 연산 (Operation, 오퍼레이션)

**연산**이란 실제 데이터(인스턴스, 즉 실제 값)에 대해 수행하는 작업들입니다. 마치 냉장고에 재료(데이터)를 넣고, 꺼내고, 상한 재료를 버리고, 무엇이 있는지 확인하는 것처럼, 데이터베이스에서도 데이터를 **삽입(INSERT)**, **삭제(DELETE)**, **갱신(UPDATE)**, **검색(SELECT)** 하는 작업이 있습니다. 이 네 가지 작업이 바로 연산입니다.

### C - 제약 조건 (Constraint, 컨스트레인트)

**제약 조건**이란 데이터가 갖춰야 할 규칙들입니다. 마치 도로 교통법처럼, 아무리 자동차가 달리고 싶어도 지켜야 할 규칙이 있듯이, 데이터베이스에도 지켜야 할 규칙이 있습니다. 학번은 중복될 수 없다(개체 무결성), 학생 테이블의 교수 번호는 반드시 교수 테이블에 존재하는 값이어야 한다(참조 무결성) 같은 것들이 제약 조건입니다.

> 📌 **SOC 한 줄 요약**
> - **S**tructure(구조): 무엇이 있고 어떻게 연결되어 있는가
> - **O**peration(연산): 데이터를 어떻게 다루는가
> - **C**onstraint(제약 조건): 데이터가 따라야 할 규칙은 무엇인가

## 🔷 ER 모델과 ER 다이어그램 표기법

### ER 모델이란?

**ER 모델(Entity-Relationship Model)**은 데이터 모델 중 가장 보편적으로 많이 사용되는 모델입니다. 1976년 **피터 첸(Peter Chen)**이라는 학자가 제안한 것으로, 현실 세계의 개체(Entity)와 개체 간의 관계(Relationship)를 **그래프 형태**로 표현하는 개념적 모델입니다.

ER 모델의 가장 큰 장점은 DBMS와 하드웨어에 독립적이라는 점입니다. Oracle이든 MySQL이든 상관없이 같은 ER 다이어그램으로 설계할 수 있습니다.

### ER 다이어그램의 기호 총정리

ER 다이어그램은 특정 도형(기호)을 사용해서 개체, 관계, 속성을 표현합니다. 각 기호의 의미를 정확히 이해하는 것이 ER 다이어그램을 읽고 쓰는 첫 번째 단계입니다.

| 기호 | 모양 | 의미 |
|------|------|------|
| 사각형(□) | 네모 | **개체 타입(Entity Type)** — 강한 개체 |
| 이중 사각형(□□) | 이중 네모 | **약한 개체 타입(Weak Entity Type)** |
| 다이아몬드(◇) | 마름모 | **관계 타입(Relationship Type)** |
| 이중 다이아몬드(◇◇) | 이중 마름모 | **식별 관계 타입** — 약한 개체의 관계 |
| 타원/원(○) | 동그라미 | **속성(Attribute)** |
| 이중 타원(○○) | 이중 동그라미 | **다중값 속성(Multi-valued Attribute)** |
| 점선 타원 | 점선 동그라미 | **유도 속성(Derived Attribute)** |
| 밑줄 속성 | 속성명에 밑줄 | **기본키(Primary Key) 속성** |
| 점선 밑줄 속성 | 점선 밑줄 | **부분키(Partial Key) 속성** |
| 단일 선(—) | 선 하나 | **부분 참여(Partial Participation)** |
| 이중 선(══) | 선 두 개 | **전체 참여(Total Participation)** |
| 계층형 타원 | 타원 안에 타원 | **복합 속성(Composite Attribute)** |

:::info ER 다이어그램 기호 암기 팁
- **이중**으로 표시된 것은 항상 "특별한" 의미를 가집니다. 이중 사각형=약한 개체, 이중 다이아몬드=식별 관계, 이중 타원=다중값 속성
- **점선**은 "실제로 저장되지 않는" 것을 의미합니다. 점선 타원=유도 속성(계산으로 구하는 값)
- **밑줄**은 "유일하게 식별"하는 키를 의미합니다.
:::

### ER 다이어그램 예시 — 대학교 데이터베이스

실제 예시를 통해 ER 다이어그램이 어떻게 구성되는지 살펴보겠습니다. 대학교에는 교수, 학생, 과목이 있고 다음과 같은 관계가 있습니다.

- 교수는 학생을 **지도**한다 (1명의 교수 : 여러 명의 학생 = **1:N 관계**)
- 교수는 과목을 **강의**한다 (1명의 교수 : 여러 개의 과목 = **1:N 관계**)
- 학생은 과목을 **등록**한다 (여러 명의 학생 : 여러 개의 과목 = **M:N 관계**)

각 개체의 속성은 다음과 같습니다.

- **교수**: 교수번호(기본키), 이름, 전공, 학과
- **학생**: 학번(기본키), 이름, 주소, 학년
- **과목**: 과목번호(기본키), 과목명, 학점

그리고 학생과 과목의 **등록** 관계에는 **성적**이라는 속성이 있습니다. 이처럼 관계 자체가 속성을 가질 때, 그 속성을 **설명 속성(Descriptive Attribute)**이라고 합니다.

:::note 왜 성적은 등록 관계의 속성인가?
성적이라는 값은 학생 혼자만의 정보도 아니고, 과목 혼자만의 정보도 아닙니다. "어떤 학생이 어떤 과목을 수강했을 때의 성적"이기 때문에, 학생과 과목의 등록 관계에서만 의미가 생깁니다. 이렇게 관계가 생겼을 때 비로소 의미 있는 속성을 **설명 속성**이라고 합니다.
:::

## 🔑 개체 타입 — 강한 개체와 약한 개체

### 강한 개체 (Strong Entity)

**강한 개체**는 자기 자신의 속성만으로도 유일하게 식별할 수 있는 개체입니다. 쉽게 말해, 혼자서도 충분히 존재할 수 있는 독립적인 개체입니다. 앞서 예를 든 교수, 학생, 과목 모두 강한 개체입니다. 교수번호만 알면 교수를 유일하게 찾을 수 있고, 학번만 알면 학생을 찾을 수 있습니다. ER 다이어그램에서는 **단일 사각형(□)**으로 표시합니다.

### 약한 개체 (Weak Entity)

**약한 개체**는 자기 자신의 속성만으로는 유일하게 식별할 수 없어서, 반드시 다른 강한 개체와 연결해야만 존재할 수 있는 개체입니다. ER 다이어그램에서는 **이중 사각형(□□)**으로 표시합니다.

이 개념이 어렵게 느껴진다면 다음 예시를 보세요.

**예시: 직원과 부양가족**

회사에는 **직원** 테이블과 **부양가족** 테이블이 있습니다. 직원은 직원번호(기본키)로 유일하게 식별할 수 있으므로 강한 개체입니다. 그런데 부양가족은 어떨까요?

삼성전자에는 수만 명의 직원이 있고, 각 직원마다 부양가족이 있을 것입니다. 부양가족의 이름이 "김민준"인 사람이 수백 명일 수 있습니다. 이름만으로는 어느 직원의 부양가족인지 구별할 수 없습니다. 따라서 부양가족을 식별하려면 반드시 **"누구의 부양가족인가"**, 즉 직원번호와 함께 사용해야 합니다.

또한 부양가족은 직원이 회사에 없으면 존재할 수 없습니다. 회사를 떠난 직원의 부양가족은 우리 회사 부양가족 테이블에 더 이상 필요 없습니다. 이처럼 강한 개체 없이는 존재할 수 없는 것이 **약한 개체**입니다.

```
[직원]                    <<부양>>                   [[부양가족]]
직원번호(기본키)  ════════════════════════  가족이름(부분키)
이름                                                  관계
부서
```

약한 개체의 특징을 정리하면 다음과 같습니다.

- **자기 자신의 속성만으로는 키를 구성할 수 없습니다.**
- **강한 개체와 연결해야만 유일하게 식별이 가능합니다.**
- **강한 개체가 삭제되면 약한 개체도 함께 삭제됩니다.** (존재 종속)
- **약한 개체의 관계는 이중 다이아몬드(◇◇)로 표시합니다.** (식별 관계)
- **약한 개체의 부분적 식별자를 부분키(Partial Key)**라고 하며, 점선 밑줄로 표시합니다.
- **약한 개체는 항상 강한 개체에 전체 참여(이중 선)**해야 합니다.
- **강한 개체와 약한 개체의 관계는 항상 1:N입니다.**

:::warning 약한 개체 주의사항
약한 개체의 부분키는 강한 개체의 기본키와 **합쳐져야** 비로소 완전한 식별자가 됩니다. 예를 들어, 부양가족의 "가족이름"은 그 자체로는 부분키이지만, "직원번호 + 가족이름"이 합쳐지면 완전한 기본키 역할을 합니다.
:::

## 🏷️ 속성의 종류

### 속성이란?

**속성(Attribute)**은 개체나 관계가 가지는 세부 정보입니다. 학생이라는 개체가 학번, 이름, 주소, 학년이라는 정보를 가지듯이, 속성은 개체의 특성을 나타냅니다. ER 다이어그램에서는 **타원(○)**으로 표시합니다.

속성에는 여러 종류가 있습니다.

### 단순 속성 vs 복합 속성

**단순 속성(Simple Attribute)**은 더 이상 작은 단위로 쪼갤 수 없는 속성입니다. 예를 들어 학번, 나이, 성별 같은 것들입니다. 이름을 "홍길동"이라고 하면 더 이상 의미 있게 분해하기 어렵습니다.

**복합 속성(Composite Attribute)**은 여러 하위 속성으로 구성된 속성입니다. 예를 들어 **주소**는 시, 구, 동, 번지로 나눌 수 있고, **이름**도 성과 이름으로 나눌 수 있습니다. ER 다이어그램에서는 계층 형태의 타원으로 표시합니다. 마치 책의 목차처럼 큰 항목 아래에 세부 항목이 붙어 있는 형태입니다.

```
       [주소]          ← 복합 속성 (타원)
      /   |   \
   (시) (구)  (동)    ← 하위 단순 속성들
```

### 단일값 속성 vs 다중값 속성

**단일값 속성(Single-valued Attribute)**은 하나의 값만 가지는 속성입니다. 사람의 나이는 하나밖에 없고, 주민번호도 하나입니다. 일반 타원(○)으로 표시합니다.

**다중값 속성(Multi-valued Attribute)**은 여러 개의 값을 가질 수 있는 속성입니다. 취미는 독서, 등산, 수영처럼 여러 개가 될 수 있고, 학위도 학사, 석사, 박사를 동시에 보유할 수 있습니다. **이중 타원(○○)**으로 표시합니다.

### 저장 속성 vs 유도 속성

**저장 속성(Stored Attribute)**은 실제로 하드디스크에 값이 저장되는 속성입니다. 이름, 학번, 주민번호 등이 이에 해당합니다. 일반 타원(○)으로 표시합니다.

**유도 속성(Derived Attribute)**은 다른 속성의 값으로부터 계산해낼 수 있는 속성입니다. 실제 데이터베이스에 저장되지 않고 필요할 때 계산으로 구합니다. **점선 타원**으로 표시합니다.

예를 들어 주민번호로부터 나이를 계산할 수 있습니다. 그러면 나이는 굳이 저장하지 않아도 주민번호에서 유도(계산)할 수 있으므로 유도 속성이 됩니다.

```
학생
├── 주민번호     ← 저장 속성 (실선 타원)
└── 나이         ← 유도 속성 (점선 타원, 주민번호로부터 계산)
```

:::tip 유도 속성을 사용하는 이유
나이는 매년 달라집니다. 만약 나이를 직접 저장한다면 매년 모든 사람의 나이를 업데이트해야 합니다. 하지만 주민번호로부터 나이를 계산하는 유도 속성으로 처리하면 주민번호만 저장해도 항상 정확한 나이를 구할 수 있습니다.
:::

### 속성 종류 한눈에 보기

| 속성 종류 | 설명 | ER 다이어그램 표기 | 예시 |
|-----------|------|-------------------|------|
| 단순 속성 | 더 이상 분해 불가 | 일반 타원 | 학번, 성별 |
| 복합 속성 | 하위 속성으로 분해 가능 | 계층형 타원 | 주소(시+구+동) |
| 단일값 속성 | 하나의 값만 가짐 | 일반 타원 | 나이, 주민번호 |
| 다중값 속성 | 여러 값을 가질 수 있음 | 이중 타원 | 취미, 학위 |
| 저장 속성 | 실제 DB에 저장됨 | 실선 타원 | 이름, 학번 |
| 유도 속성 | 계산으로 구함, 저장 안 됨 | 점선 타원 | 나이(주민번호에서 계산) |

## ❓ NULL(널)의 의미

### NULL이란?

데이터베이스에서 **NULL(널)**은 매우 중요한 개념입니다. NULL은 흔히 "값이 없다"고 생각하지만, 정확히는 세 가지 의미를 가집니다.

:::warning NULL은 0이 아닙니다!
NULL은 숫자 0이나 공백 문자("")와 전혀 다른 의미입니다. 0은 "0이라는 값이 있다"는 것이고, ""는 "빈 문자열이라는 값이 있다"는 것입니다. 하지만 NULL은 "값이 무엇인지 알 수 없다"는 완전히 다른 의미입니다. 이 차이를 반드시 이해하세요!
:::

### NULL의 세 가지 의미

**첫 번째: 해당 사항 없음**

어떤 속성이 특정 데이터에 적용되지 않는 경우입니다. 예를 들어 학생 테이블에 병역 컬럼이 있다고 합시다. 남학생은 군필, 미필, 면제 등의 값이 들어갈 수 있지만, 여학생에게는 병역이 해당 사항이 없습니다. 이런 경우 NULL을 사용합니다. 또는 단독주택에 사는 사람의 경우 아파트 동호수 정보는 해당 사항이 없으므로 NULL이 됩니다.

**두 번째: 값은 있지만 아직 알 수 없음**

분명히 값이 존재하지만 현재 그 값이 무엇인지 알 수 없는 경우입니다. 예를 들어 어제 이사를 가서 새 주소를 아직 파악하지 못한 경우, 주소가 없는 것이 아니라 분명히 주소는 있지만 아직 확인을 못 한 것이므로 NULL을 사용합니다.

**세 번째: 값이 존재하는지 여부조차 알 수 없음**

어떤 사람에게 집 전화번호가 있는지 없는지조차 모르는 경우입니다. 요즘에는 집 전화를 사용하지 않는 사람도 많은데, 그 사람에게 집 전화가 있는지 없는지 확인을 아예 못 한 경우 NULL을 사용합니다.

```sql
-- NULL 사용 예시

-- ❌ 잘못된 생각: NULL을 0이나 빈 문자열로 처리
SELECT * FROM 학생 WHERE 병역 = '';   -- 이렇게 하면 안 됩니다
SELECT * FROM 학생 WHERE 병역 = 0;    -- 이렇게 해도 안 됩니다

-- ✅ 올바른 방법: NULL은 IS NULL / IS NOT NULL로 확인
SELECT * FROM 학생 WHERE 병역 IS NULL;     -- 병역 해당 없는 학생 조회
SELECT * FROM 학생 WHERE 병역 IS NOT NULL; -- 병역 정보가 있는 학생 조회
```

## 🔗 관계 타입과 참여 제약 조건

### 관계란?

**관계(Relationship)**는 개체와 개체 사이의 연결고리입니다. 학생과 교수가 "지도"라는 관계로 연결되고, 학생과 과목이 "등록"이라는 관계로 연결됩니다. 관계는 ER 다이어그램에서 **다이아몬드(◇)**로 표시합니다.

관계도 속성을 가질 수 있습니다. 앞서 말씀드린 것처럼, 학생과 과목의 등록 관계에서 "성적"이라는 속성은 등록이라는 관계에 속합니다. 이를 **설명 속성(Descriptive Attribute)**이라고 합니다.

### 관계의 카디널리티 (1:1, 1:N, M:N)

관계에서 가장 중요한 제약 조건은 **카디널리티(Cardinality)**, 즉 몇 대 몇의 관계인지를 표현하는 것입니다.

**1:1 (일대일) 관계**는 한쪽 개체 하나가 반대쪽 개체 하나와만 연결되는 관계입니다. 마치 한 나라에 대통령이 한 명이고, 한 명의 대통령이 한 나라를 대표하는 것처럼, 또는 남자 한 명과 여자 한 명이 결혼하는 것처럼 일대일로 대응합니다.

**1:N (일대다) 관계**는 한쪽 개체 하나가 반대쪽 개체 여러 개와 연결되는 관계입니다. 1명의 교수가 여러 명의 학생을 지도하는 것, 1명의 부모로부터 여러 자녀가 태어나는 것이 1:N 관계입니다.

**M:N (다대다) 관계**는 양쪽 개체 모두 여러 개씩 연결될 수 있는 관계입니다. 학생은 여러 과목을 수강하고, 과목도 여러 학생이 수강할 수 있으므로 학생과 과목의 관계는 M:N입니다.

| 관계 유형 | 설명 | 실생활 예시 | 학교 DB 예시 |
|-----------|------|-------------|-------------|
| 1:1 | 하나 ↔ 하나 | 남자 ↔ 여자 (결혼) | 교수 ↔ 연구실 |
| 1:N | 하나 ↔ 여러 개 | 부모 ↔ 자녀들 | 교수 → 지도학생들 |
| N:1 | 여러 개 ↔ 하나 | 자녀들 ↔ 부모 | 지도학생들 → 교수 |
| M:N | 여러 개 ↔ 여러 개 | 학생들 ↔ 동아리들 | 학생들 ↔ 과목들 |

:::note 1:N과 N:1의 차이
1:N과 N:1은 같은 관계를 **방향에 따라 다르게 표현**한 것입니다. "교수가 학생을 지도한다"는 관점에서 보면 1:N이고, "학생이 교수로부터 지도받는다"는 관점에서 보면 N:1입니다. 관계는 항상 방향성이 있으므로, 어떤 방향에서 보느냐에 따라 표현이 달라집니다.
:::

### 전체 참여 vs 부분 참여

관계의 또 다른 중요한 제약 조건은 **참여 제약 조건**입니다. 개체의 모든 인스턴스(실제 데이터)가 반드시 관계에 참여해야 하는지, 아니면 참여하지 않아도 되는지를 나타냅니다.

**전체 참여(Total Participation, 의무적 참여)**는 해당 개체의 모든 인스턴스가 반드시 관계에 참여해야 한다는 것입니다. ER 다이어그램에서 **이중 선(══)**으로 표시합니다. 쉽게 말해 "반드시 해야 한다"는 의미입니다.

예를 들어, 우리 회사에서 모든 사원은 반드시 어떤 부서에 소속되어야 합니다. 부서에 소속되지 않은 사원은 존재할 수 없습니다. 이런 경우 사원은 부서 소속 관계에 전체 참여합니다.

**부분 참여(Partial Participation, 선택적 참여)**는 관계에 참여할 수도 있고 안 할 수도 있다는 것입니다. ER 다이어그램에서 **단일 선(—)**으로 표시합니다. "해도 되고 안 해도 된다"는 의미입니다.

예를 들어, 새로 신설된 부서는 아직 소속 사원이 없을 수 있습니다. 부서는 있지만 소속 사원이 0명인 것도 가능합니다. 이런 경우 부서는 사원 소속 관계에 부분 참여합니다.

```
[부서] —————◇소속◇═════ [사원]
 (단일 선)         (이중 선)
 부분 참여          전체 참여
 = 사원이 없는      = 반드시 부서에
   부서도 OK          소속되어야 함
```

:::tip 참여 제약 조건을 실제 DB 설계에 적용하기
전체 참여는 데이터베이스에서 **NOT NULL 제약 조건**으로 구현됩니다. "사원은 반드시 부서에 소속되어야 한다"는 전체 참여 조건은, 사원 테이블의 부서번호 컬럼에 NOT NULL을 설정하는 것으로 표현됩니다.
:::

## ⚠️ 연결함정 — 잘못된 설계의 함정들

### 연결함정이란?

**연결함정(Connection Trap)**이란 ER 다이어그램을 잘못 설계해서 기존에 표현할 수 있었던 정보가 누락되거나 추적 불가능해지는 현상입니다. 마치 지도를 잘못 그려서 길을 찾지 못하게 되는 것과 같습니다.

연결함정에는 두 가지 종류가 있습니다.

### 부채꼴 함정 (Fan Trap)

**부채꼴 함정**은 관계 모델을 잘못 설계해서 특정 정보를 추적할 수 없게 되는 문제입니다. 주로 1:N 관계를 잘못 배치했을 때 발생합니다.

예를 들어, "교수는 대학(단과대학)에 재직하고, 대학은 여러 학과를 포함한다"고 설계했다고 합시다. 이 경우:

- 교수가 어느 대학에 속하는지: 알 수 있음 ✅
- 대학에 어떤 학과들이 있는지: 알 수 있음 ✅
- **교수가 어느 학과에 속하는지: 알 수 없음** ❌

이처럼 교수 → 대학 → 학과라는 경로로는 "교수가 어느 학과인지"를 알 수 없습니다. 이것이 부채꼴 함정입니다.

**해결 방법**: 모델의 위치를 변환합니다. 즉, 관계의 연결 구조를 바꿔서 정보를 추적할 수 있도록 만들어줍니다.

### 균열 함정 (Chasm Trap)

**균열 함정**은 반드시 필요한 관계가 누락되어서 정보를 연결할 수 없게 되는 문제입니다.

예를 들어, "학과는 교수를 채용하고, 교수는 학생을 지도한다"고만 설계했다고 합시다. 이 경우:

- 학과에 어떤 교수들이 재직하는지: 알 수 있음 ✅
- 교수가 어떤 학생들을 지도하는지: 알 수 있음 ✅
- **학생이 어느 학과 학생인지: 알 수 없음** ❌

학생 → 교수 → 학과라는 경로로는 "이 학생이 어느 학과 학생인지"를 알 수 없습니다.

**해결 방법**: 부채꼴 함정과 달리, 단순히 위치를 바꾸는 것으로는 해결되지 않습니다. **새로운 관계를 추가**해야 합니다. 예를 들어 "학생은 학과에 소속된다"는 관계를 새로 추가해야 합니다.

| 함정 유형 | 원인 | 해결 방법 |
|-----------|------|-----------|
| 부채꼴 함정 | 관계의 위치(배치)가 잘못됨 | 관계 모델의 위치를 변환 |
| 균열 함정 | 필요한 관계가 누락됨 | 새로운 관계를 추가 |

## 🌿 확장 ER 모델 (EER 모델)

### 왜 확장 ER 모델이 필요한가?

기존의 ER 모델은 단순한 개체, 관계, 속성으로 현실 세계를 표현합니다. 하지만 현대 데이터베이스에서는 CAD(컴퓨터 설계) 데이터, 지리 정보 데이터, 멀티미디어 데이터처럼 훨씬 복잡한 데이터를 다뤄야 합니다. 또한 객체지향 프로그래밍에서처럼 개체 간의 **상속(Inheritance)** 관계나 **재사용** 개념도 표현해야 합니다.

기존 ER 모델만으로는 이런 복잡한 개념을 표현하기 어렵기 때문에, **전문화(Specialization)**, **일반화(Generalization)**, **집단화(Aggregation)**, **연관화(Association)** 같은 개념을 추가한 **확장 ER 모델(EER: Extended Entity-Relationship Model)**이 등장했습니다.

### 슈퍼클래스와 서브클래스

확장 ER 모델의 핵심 개념은 **슈퍼클래스(Superclass)**와 **서브클래스(Subclass)**입니다. 이는 객체지향 프로그래밍의 상속 개념과 유사합니다.

예를 들어, 학생이라는 개체는 학부생과 대학원생으로 나뉩니다.

- **슈퍼클래스**: 학생 (상위 개체)
  - 속성: 학번(기본키), 이름, 학과
- **서브클래스 1**: 학부생 (하위 개체)
  - 추가 속성: 학년
- **서브클래스 2**: 대학원생 (하위 개체)
  - 추가 속성: 학위과정

학부생도 학생이고, 대학원생도 학생이므로 학번, 이름, 학과는 공통적으로 필요합니다. 이런 공통 속성은 슈퍼클래스인 학생에 모아두고, 각각에만 필요한 속성은 서브클래스에 둡니다. 슈퍼클래스와 서브클래스의 관계를 **Is-a 관계**라고 합니다. "학부생 Is-a 학생" (학부생은 학생이다)라고 읽습니다.

### Disjoint(D) vs Overlap(O)

서브클래스가 여러 개 있을 때, 하나의 인스턴스가 여러 서브클래스에 동시에 속할 수 있는지 없는지를 나타냅니다.

**D (Disjoint, 분리)**는 하나의 개체가 서브클래스 중 하나에만 속할 수 있다는 의미입니다. 학부생이면서 동시에 대학원생인 학생은 존재할 수 없으므로 D(분리)로 표시합니다.

**O (Overlap, 중첩)**는 하나의 개체가 여러 서브클래스에 동시에 속할 수 있다는 의미입니다. 만약 특수한 제도로 학부생이면서 동시에 대학원 과정도 밟을 수 있다면 O(중첩)으로 표시합니다.

```
           [학생]              ← 슈퍼클래스
          학번(기본키)
          이름, 학과
              |
             (D)               ← Disjoint: 중복 불가
           /     \
    [학부생]     [대학원생]    ← 서브클래스들
      학년         학위과정
```

### 일반화 (Generalization) vs 세분화 (Specialization)

**일반화**는 여러 개체의 공통점을 찾아 상위 개체(슈퍼클래스)를 만드는 과정입니다. 방향이 **아래에서 위로(상향식)**입니다. 학부생과 대학원생의 공통 속성을 묶어서 학생이라는 슈퍼클래스를 만드는 것이 일반화입니다. **추상화**, **개념화**라고도 합니다.

**세분화(전문화)**는 반대로 상위 개체에서 구체적인 하위 개체(서브클래스)를 만드는 과정입니다. 방향이 **위에서 아래로(하향식)**입니다. 학생을 학부생과 대학원생으로 나누는 것이 세분화입니다. **전문화**, **구체화**라고도 합니다.

```
일반화 (상향식, ↑):
승용차 + 트럭 → (공통점 묶기) → 자동차(슈퍼클래스)

세분화 (하향식, ↓):
자동차(슈퍼클래스) → (차이점으로 나누기) → 승용차 + 트럭
```

실제 예시로 살펴보면, 승용차와 트럭이라는 개체가 있을 때:

- **공통 속성**: 등록번호, 차대번호, 가격 → 슈퍼클래스 **자동차**로 올림
- **승용차만의 속성**: 최고속도, 승객수 → 서브클래스 **승용차**에 남김
- **트럭만의 속성**: 용적량 → 서브클래스 **트럭**에 남김
- 자동차가 승용차이면서 동시에 트럭일 수는 없으므로 → **D (Disjoint)**

### 집단화 (Aggregation)

**집단화(Aggregation)**는 관계와 관계 사이에 또 다른 관계를 표현해야 할 때 사용합니다. 원래 관계(다이아몬드)는 개체와 개체 사이를 연결하는 것인데, 관계와 관계를 다시 연결할 수는 없습니다. 이럴 때 관계 전체를 하나의 개체처럼 묶어서 새로운 관계를 만드는 것이 집단화입니다.

**Is-a Part Of 관계**라고도 합니다.

예를 들어, "종업원"이 "프로젝트"에 "작업"한다는 관계가 있습니다. 그런데 "종업원이 프로젝트에 작업할 때 특정 기계를 사용한다"는 것을 표현하고 싶다면, "작업" 관계 자체와 "기계" 개체 사이에 "사용" 관계를 만들어야 합니다. 그런데 관계(다이아몬드)끼리는 직접 연결할 수 없으므로, "종업원 + 프로젝트 + 작업" 전체를 하나의 집합인 "작업들"이라는 개체로 묶어서 처리합니다.

### 연관화 (Association)

**연관화(Association)**는 개체와 개체 사이의 연관성 여부를 판단하는 개념입니다. **Is Associated With 관계**라고 표현하며, 두 개체 간에 어떤 의미 있는 연관이 있는지를 모델에 명시적으로 표현합니다.

## 📊 논리적 데이터 모델의 종류

### ER 다이어그램을 테이블로 변환하면?

ER 다이어그램(개념적 설계)이 완성되면, 이를 실제 데이터베이스에서 사용할 수 있는 **논리적 데이터 모델**로 변환해야 합니다. 논리적 데이터 모델에는 크게 세 가지 종류가 있습니다.

### 관계 데이터 모델 (Relational Data Model)

가장 널리 사용되는 모델입니다. 데이터를 **테이블(표)** 형태로 표현합니다. 테이블은 행(Row)과 열(Column)로 구성됩니다. 우리가 일반적으로 알고 있는 Oracle, MS-SQL, MySQL이 모두 관계 데이터 모델을 기반으로 합니다.

관계 데이터 모델은 네트워크나 계층 모델에 비해 **추상적**입니다. 학생과 교수의 관계를 그림으로 그리면 한눈에 보이지만, 테이블로 표현하면 숫자와 텍스트의 나열로 보입니다. 그래도 가장 단순하고 강력하며 유연하기 때문에 오늘날 대부분의 데이터베이스가 이 모델을 사용합니다.

### 네트워크 데이터 모델 (Network Data Model)

데이터를 **그래프(Graph)** 형태로 표현합니다. 오너(Owner)와 멤버(Member)의 관계를 가지며, **사이클(Cycle, 순환)을 허용**합니다. 즉, A→B→C→A처럼 순환 관계가 가능합니다. 대표적인 DBMS로는 CODASYL DB, TOTAL이 있습니다.

### 계층 데이터 모델 (Hierarchical Data Model)

데이터를 **트리(Tree)** 구조로 표현합니다. 부모-자식 관계(상하 관계)만 표현 가능하며, **사이클을 허용하지 않습니다**. 자식이 부모가 되는 순환은 불가능합니다. 대표적인 DBMS로는 IMS(Information Management System)가 있습니다.

| 모델 | 표현 방식 | 사이클 허용 | 대표 DBMS |
|------|-----------|-------------|-----------|
| 관계 데이터 모델 | 테이블(표) | — | Oracle, MySQL, MS-SQL |
| 네트워크 데이터 모델 | 그래프 | 허용 | CODASYL DB, TOTAL |
| 계층 데이터 모델 | 트리 | 불허 | IMS |

## 🛠️ ER 다이어그램 작성 예시 코드

아래는 대학교 데이터베이스를 SQL로 표현한 예시입니다. ER 다이어그램을 설계한 후 실제 테이블로 변환하는 과정을 코드로 살펴보겠습니다.

```sql
-- ✅ 올바른 예시: ER 다이어그램 설계를 반영한 테이블 생성

-- 1. 강한 개체: 교수 테이블 생성
CREATE TABLE 교수 (
    교수번호 VARCHAR(10) PRIMARY KEY,  -- 기본키 (밑줄 속성)
    이름     VARCHAR(50) NOT NULL,      -- 단순 속성, 저장 속성
    전공     VARCHAR(50),               -- 단순 속성
    학과     VARCHAR(50)                -- 단순 속성
);

-- 2. 강한 개체: 학생 테이블 생성
CREATE TABLE 학생 (
    학번     VARCHAR(10) PRIMARY KEY,  -- 기본키 (밑줄 속성)
    이름     VARCHAR(50) NOT NULL,      -- 단순 속성, 저장 속성
    주소     VARCHAR(100),              -- 복합 속성을 단순화한 것
    학년     INT,                        -- 단순 속성
    -- 나이는 유도 속성이므로 컬럼에 저장하지 않음 (계산으로 구함)
    지도교수번호 VARCHAR(10),           -- 외래키 (교수와 지도 관계, 부분 참여이므로 NULL 허용)
    FOREIGN KEY (지도교수번호) REFERENCES 교수(교수번호)
);

-- 3. 강한 개체: 과목 테이블 생성
CREATE TABLE 과목 (
    과목번호 VARCHAR(10) PRIMARY KEY,  -- 기본키
    과목명   VARCHAR(100) NOT NULL,    -- 단순 속성
    학점     INT                         -- 단순 속성
);

-- 4. M:N 관계 테이블: 학생-과목 등록 (설명 속성 '성적' 포함)
CREATE TABLE 등록 (
    학번     VARCHAR(10),               -- 학생 기본키 참조
    과목번호 VARCHAR(10),               -- 과목 기본키 참조
    성적     VARCHAR(2),                -- 설명 속성 (관계가 가지는 속성)
    PRIMARY KEY (학번, 과목번호),       -- 복합 기본키
    FOREIGN KEY (학번) REFERENCES 학생(학번),
    FOREIGN KEY (과목번호) REFERENCES 과목(과목번호)
);

-- 5. 약한 개체: 부양가족 테이블 생성
CREATE TABLE 부양가족 (
    직원번호  VARCHAR(10),              -- 강한 개체(직원)의 기본키 참조
    가족이름  VARCHAR(50),              -- 부분키 (점선 밑줄 속성)
    관계      VARCHAR(20),              -- 일반 속성
    PRIMARY KEY (직원번호, 가족이름),   -- 직원번호 + 가족이름 = 완전한 기본키
    FOREIGN KEY (직원번호) REFERENCES 직원(직원번호)
    -- 약한 개체이므로 직원이 삭제되면 부양가족도 삭제 (CASCADE)
    ON DELETE CASCADE
);
```

```sql
-- ❌ 잘못된 예시: 약한 개체를 강한 개체처럼 설계

CREATE TABLE 부양가족_잘못된예시 (
    가족ID   INT PRIMARY KEY AUTO_INCREMENT,  -- 임의의 기본키를 만들어버림
    가족이름 VARCHAR(50),
    -- 직원번호가 없으면 어느 직원의 부양가족인지 알 수 없음!
    관계     VARCHAR(20)
    -- 외래키 없음 → 직원과의 연결 고리가 없음
);
-- 이렇게 하면 부양가족이 어느 직원의 부양가족인지 알 수 없게 됩니다.
-- 약한 개체는 반드시 강한 개체의 기본키를 포함해야 합니다!
```

## 💡 주의사항과 실전 팁

### ER 다이어그램 작성 시 자주 하는 실수

:::danger 절대 하지 말아야 할 것
**약한 개체의 부분키를 기본키로 착각하지 마세요!** 부분키는 그 자체로 유일한 식별이 불가능합니다. 반드시 강한 개체의 기본키와 합쳐서 복합 기본키로 사용해야 합니다.
:::

:::danger 관계 누락 주의
균열 함정처럼, 필요한 관계를 빠뜨리면 나중에 원하는 정보를 조회할 수 없게 됩니다. ER 다이어그램을 작성한 후 "A에서 B의 정보를 알 수 있는가?"를 반드시 검증하세요.
:::

:::warning 설명 속성 위치 주의
속성이 어떤 개체에 속하는지 헷갈릴 때, "이 값이 A 혼자, 아니면 B 혼자 결정하는가, 아니면 A와 B가 만났을 때 결정되는가?"를 생각해보세요. A와 B가 만났을 때 결정되는 속성은 두 개체의 관계에 붙이는 설명 속성입니다.
:::

:::tip 슈퍼클래스 결정 팁
확장 ER 모델에서 슈퍼클래스를 결정할 때는 "여러 하위 개체들이 공통으로 가지는 속성은 무엇인가?"를 먼저 찾으세요. 공통 속성들을 슈퍼클래스로 올리고, 차별화되는 속성들은 각 서브클래스에 남깁니다. 이것이 일반화(추상화)의 핵심입니다.
:::

:::tip NULL 처리 팁
NULL은 0이나 빈 문자열로 대체하지 마세요. NULL을 포함한 컬럼에서 검색하거나 연산할 때는 반드시 `IS NULL`, `IS NOT NULL`, `COALESCE()` 등의 NULL 처리 함수를 사용하세요.
:::

### 연결함정 방지 체크리스트

ER 다이어그램을 그린 후 다음 항목을 확인하세요.

1. **모든 필요한 관계가 포함되어 있는가?** (균열 함정 방지)
2. **특정 개체에서 다른 개체까지 정보를 추적할 수 있는가?** (부채꼴 함정 방지)
3. **삼항 관계(세 개체 사이의 관계)를 이항 관계(두 개체 사이의 관계) 여러 개로 분해했을 때 정보 손실이 없는가?**
4. **약한 개체는 강한 개체와 반드시 연결되어 있는가?**
5. **M:N 관계는 관계 테이블로 분리될 준비가 되어 있는가?**

## 📝 핵심 정리

- **데이터 모델링의 3단계**: 개념적 모델링(ER 다이어그램) → 데이터 모델링(테이블) → 데이터 구조화(물리적 저장)
- **데이터 모델의 구성 요소(SOC)**: 구조(Structure), 연산(Operation), 제약 조건(Constraint)
- **ER 모델**은 피터 첸이 제안한 개념적 모델로, 개체와 관계를 그래프 형태로 표현한다
- **ER 다이어그램 기호**: 사각형=개체, 다이아몬드=관계, 타원=속성, 이중 표시=특별한 의미
- **강한 개체**는 자신의 속성만으로 유일하게 식별 가능하고, **약한 개체**는 강한 개체에 종속되어야만 식별 가능하다
- **약한 개체의 부분키**는 강한 개체의 기본키와 합쳐야 완전한 식별자가 된다
- **속성 종류**: 단순/복합, 단일값/다중값, 저장/유도 속성으로 나뉜다
- **유도 속성**은 다른 속성으로 계산 가능하므로 DB에 저장하지 않으며 점선 타원으로 표시한다
- **NULL**은 0이나 빈 문자열이 아니며, 해당 없음 / 값 모름 / 존재 여부 불명의 세 가지 의미를 가진다
- **관계 카디널리티**: 1:1, 1:N, N:1, M:N 네 가지 유형이 있다
- **전체 참여(이중 선)**는 반드시 관계에 참여해야 하고, **부분 참여(단일 선)**는 선택적으로 참여할 수 있다
- **연결함정**: 부채꼴 함정(모델 위치 잘못)은 모델 변환으로, 균열 함정(관계 누락)은 관계 추가로 해결
- **확장 ER 모델(EER)**은 슈퍼클래스/서브클래스, Disjoint(D)/Overlap(O), 일반화/세분화, 집단화, 연관화 개념을 추가한 모델이다
- **일반화(상향식)**는 공통점을 찾아 슈퍼클래스를 만들고, **세분화(하향식)**는 차이점으로 서브클래스를 만든다
- **집단화**는 관계와 관계 간의 관계를 표현할 때 사용하는 Is-a Part of 개념이다
- **논리적 데이터 모델**: 관계형(테이블), 네트워크형(그래프), 계층형(트리) 세 가지가 있다
