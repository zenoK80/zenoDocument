# 🏗️ 데이터베이스 설계 단계: 개념적·논리적·물리적 설계

데이터베이스는 그냥 만들어지는 게 아닙니다. 마치 건물을 짓기 전에 설계도를 먼저 그리듯이, 데이터베이스도 **체계적인 설계 과정**을 거쳐야 합니다. 이 문서에서는 현실 세계의 정보를 어떻게 데이터베이스로 옮기는지, 그 전체 흐름을 처음부터 끝까지 아주 자세하게 살펴보겠습니다.

:::info 이 문서에서 배울 내용
- 데이터 모델링의 3단계 (개념적 → 논리적 → 물리적)
- 데이터 모델의 구성 요소 (구조, 연산, 제약 조건)
- ER 다이어그램의 핵심 요소 (개체, 속성, 관계)
- 강한 개체와 약한 개체의 차이
- 속성의 다양한 종류와 NULL의 의미
- 관계의 종류 (1:1, 1:N, N:M)
- 확장 ER 모델 (일반화, 세분화, 집단화, 연관화)
- 연결함정 (부채골 함정, 균열 함정)
:::

## 🌍 데이터 모델링이란 무엇인가?

### 현실 세계를 데이터베이스로 옮기는 과정

우리 주변에는 수많은 정보들이 존재합니다. 학교를 예로 들면, 학생이 있고, 교수가 있고, 과목이 있습니다. 학생은 교수로부터 지도를 받고, 과목을 수강하며, 교수는 과목을 강의합니다. 이런 **현실 세계의 복잡한 관계들**을 컴퓨터가 저장하고 관리할 수 있도록 바꾸는 작업이 바로 **데이터 모델링(Data Modeling)** 입니다.

마치 지도를 만드는 것과 비슷합니다. 실제 지형(현실 세계)을 그대로 옮길 수는 없으니, 축적과 기호를 사용해서 지도(데이터베이스 설계도)로 표현하는 것이죠. 데이터 모델링도 현실의 복잡한 상황을 **추상적인 기호와 도형**으로 표현해서 데이터베이스를 만들기 위한 설계도를 그리는 과정입니다.

### 세 가지 핵심 용어의 차이 이해하기

데이터 모델링을 공부하다 보면 비슷하게 생긴 용어들이 등장해서 헷갈리기 쉽습니다. 먼저 이 세 가지 용어의 차이를 명확히 이해해야 합니다.

| 용어 | 의미 | 결과물 |
|---|---|---|
| **개념적 모델링** | 현실 세계를 추상적 개념(도형)으로 표현하는 과정 | 개념적 구조 (ER 다이어그램) |
| **데이터 모델링** | 개념적 구조를 논리적 구조(테이블)로 변환하는 과정 | 논리적 구조 (테이블) |
| **데이터 구조화** | 논리적 구조를 컴퓨터가 접근할 수 있는 물리적 구조로 변환하는 과정 | 물리적 구조 (디스크 저장 구조) |

**개념적 모델링**은 "학생이 과목을 수강한다", "교수가 학생을 지도한다"처럼 현실의 관계를 **도형(직사각형, 타원, 마름모 등)** 으로 표현하는 단계입니다. 이 과정의 결과물이 바로 **개념적 구조**, 즉 **ER 다이어그램(Entity-Relationship Diagram)** 입니다.

**데이터 모델링**은 그렇게 만들어진 ER 다이어그램을 실제 데이터베이스에서 사용하는 **테이블 형태**로 변환하는 작업입니다. 예를 들어 "학생"이라는 개체를 학번, 이름, 학과, 학년 컬럼을 가진 테이블로 바꾸는 것이죠.

**데이터 구조화**는 그 테이블을 실제로 하드 디스크에 저장하는 방법, 인덱스(색인) 구조, 파일 형태 등을 결정하는 가장 마지막 단계입니다.

> 🏠 비유하자면 이렇습니다. 집을 지을 때 먼저 건축가가 아이디어 스케치(개념적 모델링)를 하고, 그 스케치를 정밀한 설계도(데이터 모델링)로 바꾼 다음, 실제로 벽돌과 콘크리트로 집을 짓는(데이터 구조화) 순서와 똑같습니다.

## 🗺️ 데이터베이스 설계의 3단계

### 전체 흐름 한눈에 보기

데이터베이스를 설계하는 과정은 크게 세 단계로 나뉩니다. 각 단계가 어떤 특성을 가지고 있는지 표로 먼저 정리해 봅시다.

| 설계 단계 | 주요 작업 | 결과물 | DBMS 종속성 | 하드웨어 종속성 |
|---|---|---|---|---|
| **개념적 설계** | ER 다이어그램 작성 | ER 다이어그램 | 독립적 ✅ | 독립적 ✅ |
| **논리적 설계** | 테이블 구조 설계 | 관계형/계층형/네트워크형 테이블 | 종속적 ❌ | 독립적 ✅ |
| **물리적 설계** | 디스크 저장 구조 결정 | 물리적 저장 구조 | 종속적 ❌ | 종속적 ❌ |

### 1단계: 개념적 설계 (Conceptual Design)

개념적 설계는 **사용자의 요구사항을 수집하고 분석**하는 것에서 시작합니다. "우리 학교 데이터베이스에 어떤 정보가 필요한가?"를 먼저 파악하는 것이죠. 그런 다음 그 요구사항을 바탕으로 **ER 다이어그램**을 그립니다.

이 단계의 가장 큰 특징은 **DBMS(데이터베이스 관리 시스템)와 하드웨어 모두에 독립적**이라는 점입니다. 즉, Oracle을 쓸지 MySQL을 쓸지, 어떤 서버를 사용할지 전혀 신경 쓰지 않아도 됩니다. 오직 "어떤 데이터가 있고, 그 데이터들이 어떻게 연결되는가"에만 집중하면 됩니다.

:::tip 개념적 설계의 핵심
개념적 설계는 특정 기술에 얽매이지 않고 **현실 세계의 정보 구조**를 순수하게 표현하는 단계입니다. 어떤 데이터베이스를 사용할지 결정되지 않은 상태에서도 설계를 시작할 수 있습니다.
:::

### 2단계: 논리적 설계 (Logical Design)

논리적 설계는 개념적 설계에서 나온 ER 다이어그램을 **실제 데이터베이스에서 사용할 수 있는 구조**로 변환하는 단계입니다. 이 단계에서 나오는 것이 바로 **테이블(Table)** 입니다.

이 단계에서는 어떤 종류의 데이터베이스 모델을 사용할지 결정합니다. 관계형(테이블 형태), 계층형(트리 형태), 네트워크형(그래프 형태) 중 하나를 선택하게 됩니다. 따라서 논리적 설계는 **DBMS에는 종속적**이지만 **하드웨어에는 여전히 독립적**입니다. Oracle용 테이블 구조를 설계했다면 그것은 Oracle이라는 DBMS에 맞춰진 것이지만, 어떤 서버 컴퓨터를 사용할지와는 무관합니다.

### 3단계: 물리적 설계 (Physical Design)

물리적 설계는 논리적 설계에서 만든 테이블을 **실제 디스크(저장 장치)에 어떻게 저장할 것인가**를 결정하는 마지막 단계입니다. 인덱스(빠른 검색을 위한 색인)를 어떻게 만들지, 파일을 어떤 형식으로 저장할지 등을 결정합니다.

이 단계는 **DBMS와 하드웨어 모두에 종속적**입니다. Oracle 데이터베이스와 MySQL 데이터베이스의 물리적 저장 방식이 다르고, 같은 DBMS라도 어떤 하드웨어(SSD인지 HDD인지, 메모리가 얼마인지 등)를 사용하는지에 따라 최적화 방법이 달라집니다.

## 🧩 데이터 모델의 구성 요소: SOC

### 데이터 모델은 세 가지로 이루어진다

데이터 모델이란 무엇인지 한마디로 정의하면, **데이터를 표현하고 관리하는 방법의 틀**이라고 할 수 있습니다. 이 틀은 세 가지 요소로 구성됩니다. 영어 첫 글자를 따서 **SOC**라고 기억하면 좋습니다.

- **S** - Structure (구조, 스트럭처): 데이터의 형태
- **O** - Operation (연산, 오퍼레이션): 데이터에 할 수 있는 작업
- **C** - Constraint (제약 조건, 컨스트레인트): 데이터가 지켜야 할 규칙

### S - 구조 (Structure)

**구조**는 데이터베이스에서 표현해야 할 대상(개체)과 그 개체들 사이의 관계를 정의하는 것입니다. 마치 집의 평면도처럼, "어떤 방이 있고 방들이 어떻게 연결되어 있는가"를 보여주는 것이라고 생각하면 됩니다.

예를 들어 학교 데이터베이스를 설계한다면, 학생이라는 개체는 학번, 이름, 학과, 학년이라는 속성(특징)을 가지고 있습니다. 교수라는 개체는 교수번호, 이름, 전공, 학과를 가집니다. 그리고 학생과 교수 사이에는 "지도"라는 관계가 있습니다. 이런 개체들과 그 관계 전체가 바로 **구조**입니다.

### O - 연산 (Operation)

**연산**은 데이터베이스에 저장된 실제 값(인스턴스)에 대해 수행하는 작업들입니다. 인스턴스란 쉽게 말해 **실제로 저장된 데이터 값**을 의미합니다. 예를 들어 학생 테이블에 "홍길동, 학번 20240001, 컴퓨터공학과 1학년"이라는 실제 데이터가 들어가 있다면, 이것이 인스턴스입니다.

연산에는 네 가지 기본 작업이 있습니다.
- **삽입(Insert)**: 새로운 데이터를 추가하는 것 (새 학생 등록)
- **삭제(Delete)**: 기존 데이터를 지우는 것 (졸업생 데이터 삭제)
- **갱신(Update)**: 기존 데이터를 수정하는 것 (주소 변경)
- **검색(Select)**: 원하는 데이터를 조회하는 것 (특정 학생 정보 확인)

### C - 제약 조건 (Constraint)

**제약 조건**은 데이터가 반드시 지켜야 하는 규칙들입니다. 마치 교통법규처럼, 이 규칙을 어기면 데이터를 저장할 수 없습니다. 대표적인 제약 조건으로는 다음과 같은 것들이 있습니다.

- **개체 무결성(Entity Integrity)**: 기본키(Primary Key)는 NULL이 될 수 없고 중복될 수 없다
- **참조 무결성(Referential Integrity)**: 외래키(Foreign Key)는 참조하는 테이블에 존재하는 값만 가질 수 있다
- **도메인 무결성(Domain Integrity)**: 각 속성의 값은 정해진 범위와 형식을 따라야 한다

> 💡 SOC를 기억하는 팁: "데이터 모델은 **구조**(어떻게 생겼나), **연산**(뭘 할 수 있나), **제약 조건**(무엇을 지켜야 하나)의 세 가지로 이루어진다"고 정리하세요.

## 📐 ER 다이어그램: 개념적 설계의 핵심 도구

### ER 모델이란?

**ER 모델(Entity-Relationship Model)** 은 데이터베이스를 개념적으로 설계할 때 가장 많이 사용되는 방법입니다. **Entity(개체)** 와 **Relationship(관계)** 의 앞 글자를 따서 ER이라고 부릅니다. 1976년 피터 첸(Peter Chen)이 제안하여 만들어진 이 모델은, 현재까지도 데이터베이스 설계의 표준처럼 사용되고 있습니다.

ER 모델의 핵심은 **그래프(도형) 형태로 데이터의 구조를 표현**한다는 것입니다. 특정 DBMS나 하드웨어에 종속되지 않기 때문에, Oracle을 쓸지 MySQL을 쓸지 결정하기 전에도 설계를 시작할 수 있습니다.

### ER 다이어그램의 기본 기호들

ER 다이어그램에서 사용하는 도형들은 각각 특정한 의미를 가집니다. 마치 지도에서 삼각형은 산, 물결은 강을 나타내듯이 말이죠.

| 도형 | 이름 | 의미 |
|---|---|---|
| ⬜ 직사각형 (사각형) | 개체 타입 (Entity Type) | 데이터를 저장할 대상 (학생, 교수, 과목 등) |
| 🔷 마름모 (다이아몬드) | 관계 타입 (Relationship Type) | 개체들 사이의 연결 (지도, 수강, 강의 등) |
| ⭕ 타원 (동그라미) | 속성 (Attribute) | 개체나 관계의 특징 (이름, 학번, 성적 등) |
| ⭕ 밑줄 있는 타원 | 기본키 속성 (Primary Key) | 개체를 유일하게 식별하는 속성 |
| ⭕ 점선 타원 | 부분키 속성 (Partial Key) | 약한 개체의 식별자 |
| ⭕ 점선 타원 (파선) | 유도 속성 (Derived Attribute) | 다른 속성으로부터 계산되는 속성 |
| ⭕⭕ 이중 타원 | 다중값 속성 (Multi-valued Attribute) | 여러 값을 가질 수 있는 속성 |
| ⭕ 계층형 타원 | 복합 속성 (Composite Attribute) | 더 작은 속성들로 나뉘는 속성 |
| ⬜⬜ 이중 직사각형 | 약한 개체 타입 (Weak Entity Type) | 단독으로 존재할 수 없는 개체 |
| 🔷🔷 이중 마름모 | 식별 관계 (Identifying Relationship) | 약한 개체와 강한 개체의 관계 |
| ─── 단일 선 | 부분 참여 (Partial Participation) | 선택적으로 관계에 참여 |
| ══ 이중 선 | 전체 참여 (Total Participation) | 반드시 관계에 참여해야 함 |

### ER 다이어그램 실제 예시: 학교 데이터베이스

실제 학교 데이터베이스를 ER 다이어그램으로 표현한다고 가정해 봅시다.

**개체(Entity):**
- **교수**: 교수번호(기본키), 이름, 전공, 학과
- **학생**: 학번(기본키), 이름, 주소, 학년
- **과목**: 과목번호(기본키), 과목명, 학점

**관계(Relationship):**
- 교수 - **지도** - 학생: 1명의 교수가 여러 명의 학생을 지도 → **1:N 관계**
- 교수 - **강의** - 과목: 1명의 교수가 여러 과목을 강의 → **1:N 관계**
- 학생 - **등록** - 과목: 여러 학생이 여러 과목을 수강 → **N:M 관계**

특히 **등록** 관계에는 **성적**이라는 **설명속성(Descriptive Attribute)** 이 붙어 있습니다. 설명속성은 관계 자체가 가지는 속성입니다. 성적은 학생에게만 속하는 것도 아니고 과목에만 속하는 것도 아닙니다. "이 학생이 이 과목을 수강했을 때 받은 성적"이라는, 두 개체의 관계에서 발생하는 정보이기 때문에 관계의 속성(설명속성)으로 표현합니다.

```
[교수] ──1──── ◇지도◇ ────N── [학생]
  |                               |
  |                               |
  1                               N
  |                               |
  ◇강의◇ ──────────────── ◇등록◇
  |                         |
  N                      (성적: 설명속성)
  |
[과목]
```

:::note 설명속성이 왜 필요한가?
성적을 학생 테이블에 넣으면 "어떤 과목의 성적인지" 알 수 없고, 과목 테이블에 넣으면 "누구의 성적인지" 알 수 없습니다. 그래서 학생과 과목의 **관계(등록)** 에 성적을 붙여서 "홍길동이 데이터베이스 과목을 수강해서 받은 성적"을 표현합니다.
:::

## 💪 강한 개체와 약한 개체

### 개체에도 강함과 약함이 있다?

모든 개체가 다 동등한 것은 아닙니다. 어떤 개체는 **혼자서도 독립적으로 존재**할 수 있지만, 어떤 개체는 **다른 개체 없이는 존재할 수 없습니다**. 이 차이를 구분하기 위해 **강한 개체(Strong Entity)** 와 **약한 개체(Weak Entity)** 라는 개념이 있습니다.

### 강한 개체 (Strong Entity)

**강한 개체**는 자기 자신의 속성만으로 개체를 **유일하게 식별**할 수 있는 개체입니다. 즉, 기본키(Primary Key)를 자체적으로 가질 수 있는 개체입니다. ER 다이어그램에서 **단일 직사각형**으로 표현합니다.

예를 들어 **직원** 개체는 직원번호라는 속성이 있고, 이 직원번호는 전 직원 중에서 유일합니다. 따라서 직원 개체는 혼자서도 독립적으로 존재할 수 있는 강한 개체입니다.

### 약한 개체 (Weak Entity)

**약한 개체**는 자기 자신의 속성만으로는 개체를 유일하게 식별할 수 없는 개체입니다. 반드시 **다른 강한 개체의 도움을 받아야** 식별이 가능합니다. ER 다이어그램에서 **이중 직사각형**으로 표현합니다.

가장 좋은 예시가 **부양가족** 개체입니다. 회사에서는 직원의 부양가족 정보를 관리합니다. 그런데 대기업에는 수만 명의 직원이 있고, 각 직원의 부양가족들 중에는 이름이 같은 사람이 얼마든지 있을 수 있습니다. 예를 들어 "김지수"라는 이름을 가진 부양가족이 수백 명이 될 수도 있죠.

따라서 **부양가족의 이름만으로는 어느 직원의 부양가족인지 구분할 수 없습니다**. "사번 10001번 직원의 부양가족 김지수"처럼, **직원번호와 함께 사용해야만** 유일하게 식별할 수 있습니다.

:::warning 약한 개체의 핵심 특성
1. **단독으로 존재 불가**: 부양가족은 직원 없이 데이터베이스에 존재할 수 없습니다.
2. **강한 개체에 종속**: 강한 개체(직원)가 삭제되면 약한 개체(부양가족)도 함께 삭제되어야 합니다.
3. **부분키 사용**: 약한 개체의 식별자는 기본키가 아닌 **부분키(Partial Key)** 라고 합니다 (ER 다이어그램에서 점선 밑줄로 표시).
4. **전체 참여 필수**: 약한 개체는 반드시 강한 개체와의 관계에 참여해야 합니다.
5. **1:N 관계**: 강한 개체와 약한 개체는 항상 1:N 관계입니다 (1명의 직원이 여러 부양가족을 가질 수 있음).
:::

**약한 개체와 강한 개체의 관계**를 **식별 관계(Identifying Relationship)** 라고 하며, ER 다이어그램에서 **이중 마름모**로 표현합니다.

```
[직원]══════════《부양◇》════════[부양가족]
  |                                   |
 직원번호(기본키)              이름(부분키, 점선 밑줄)
  이름                           나이
  부서                           관계
```

> 🏢 실생활 비유: 회사 건물에 비유하면, **직원**은 건물 자체이고 **부양가족**은 직원이라는 건물에 딸려 있는 창고 같은 존재입니다. 본 건물이 없으면 창고도 의미가 없죠.

## 🏷️ 속성(Attribute)의 다양한 종류

### 속성이란 무엇인가?

**속성(Attribute)** 은 개체가 가지는 **특징이나 성질**입니다. 학생이라는 개체의 속성은 학번, 이름, 학과, 학년처럼 그 학생을 설명하는 정보들입니다. ER 다이어그램에서 속성은 **타원(동그라미)** 으로 표현됩니다.

속성에는 여러 종류가 있으며, 각각의 특성이 다릅니다.

### 단순 속성 vs 복합 속성

**단순 속성(Simple Attribute)** 은 더 이상 쪼갤 수 없는 속성입니다. 예를 들어 나이, 학번, 이름 같은 것들입니다. 나이를 더 작은 단위로 나눌 수는 없죠.

**복합 속성(Composite Attribute)** 은 더 작은 속성들로 나눌 수 있는 속성입니다. 예를 들어 **주소**는 시/도, 시/군/구, 상세주소, 동/호수로 나눌 수 있습니다. 이렇게 계층적으로 분해되는 속성을 복합 속성이라고 합니다. ER 다이어그램에서는 타원 아래에 더 작은 타원들이 달린 형태로 표현합니다.

### 단일값 속성 vs 다중값 속성

**단일값 속성(Single-valued Attribute)** 은 하나의 값만 가질 수 있는 속성입니다. 예를 들어 나이는 한 사람이 동시에 두 개의 나이를 가질 수 없으므로 단일값 속성입니다.

**다중값 속성(Multi-valued Attribute)** 은 여러 개의 값을 동시에 가질 수 있는 속성입니다. 예를 들어 **취미**는 "독서, 등산, 요리"처럼 여러 개일 수 있습니다. 또는 **학위**는 학사 학위를 받은 뒤 방통대를 통해 또 다른 학사 학위를 취득할 수 있으므로 여러 개가 될 수 있습니다. ER 다이어그램에서는 **이중 타원(동그라미 두 겹)** 으로 표현합니다.

### 저장 속성 vs 유도 속성

**저장 속성(Stored Attribute)** 은 실제로 데이터베이스의 디스크에 저장되는 속성입니다. 이름, 주민번호, 학번 같은 것들이 저장 속성입니다.

**유도 속성(Derived Attribute)** 은 다른 속성의 값으로부터 **계산되어 나오는 속성**입니다. 실제 디스크에 저장하지 않고, 필요할 때마다 계산해서 만들어냅니다. ER 다이어그램에서는 **점선 타원**으로 표현합니다.

가장 좋은 예시가 **나이**입니다. 주민번호를 알면 나이를 계산할 수 있습니다. 그래서 주민번호는 저장 속성, 나이는 유도 속성이 됩니다.

:::tip 유도 속성을 왜 쓸까요?
나이를 직접 저장하면 매년 값을 업데이트해야 하는 번거로움이 생깁니다. 하지만 주민번호에서 나이를 계산하면 항상 정확한 나이를 얻을 수 있습니다. 저장 공간도 절약되고 데이터의 일관성도 유지됩니다.
:::

```sql
-- 저장 속성 예시: 주민번호를 저장하고
SELECT resident_number FROM student WHERE student_id = '20240001';

-- 유도 속성 예시: 나이는 주민번호로부터 계산 (저장하지 않음)
SELECT YEAR(NOW()) - YEAR(birth_date) AS age  -- 태어난 연도로부터 나이 계산
FROM student 
WHERE student_id = '20240001';
```

## ❓ NULL의 의미: 0도 아니고 공백도 아니다

### NULL이란 무엇인가?

프로그래밍이나 데이터베이스를 공부하다 보면 **NULL**이라는 개념이 자주 등장합니다. NULL은 **0도 아니고, 빈 문자열("")도 아닌** 완전히 별개의 개념입니다.

NULL이 사용되는 세 가지 상황을 잘 기억해 두세요.

### 상황 1: 해당 사항 없음

어떤 속성이 특정 개체에 **아예 해당되지 않을 때** NULL을 사용합니다.

예를 들어 학생 테이블에 **병역** 컬럼이 있다고 합시다. 남학생은 "군필", "미필", "면제" 중 하나가 들어가겠지만, 여학생은 병역 자체가 해당 사항이 없습니다. 이런 경우 NULL을 사용합니다.

또 다른 예로, 주소 테이블에 **동호수** 컬럼이 있다고 합시다. 아파트에 사는 사람은 "101동 303호" 같은 값이 들어가지만, 단독주택에 사는 사람은 동호수가 없으므로 NULL이 들어갑니다.

### 상황 2: 값이 존재하지만 아직 알 수 없을 때

값은 분명히 있는데, **아직 그 값이 무엇인지 모를 때**도 NULL을 사용합니다.

예를 들어 어제 이사를 해서 새 주소를 아직 확인하지 못한 학생이 있다면, 주소 컬럼에 NULL을 넣습니다. 주소가 없는 것이 아니라, 주소는 분명히 있지만 아직 파악이 안 된 것이죠.

### 상황 3: 값이 존재하는지조차 알 수 없을 때

값이 있는지 없는지 **그 여부 자체를 모를 때**도 NULL을 사용합니다.

예를 들어 어떤 학생이 집 전화번호가 있는지 없는지조차 모를 경우, 집전화 컬럼에 NULL을 사용합니다. 집 전화가 있을 수도 있고 없을 수도 있는데, 그 여부 자체를 알 수 없는 상황입니다.

| NULL의 사용 상황 | 예시 |
|---|---|
| 해당 사항 없음 | 여학생의 병역, 단독주택 거주자의 동호수 |
| 값은 있지만 아직 알 수 없음 | 이사 직후 확인되지 않은 주소 |
| 값이 있는지조차 알 수 없음 | 집 전화 보유 여부가 불명확한 경우 |

:::danger NULL 관련 절대 혼동하지 말 것
- `NULL ≠ 0`: 0은 숫자값이지만 NULL은 "없음"입니다
- `NULL ≠ ""` (빈 문자열): 빈 문자열은 값이 있되 내용이 없는 것이고, NULL은 값 자체가 없음을 의미합니다
- **기본키(Primary Key)** 는 절대 NULL이 될 수 없습니다
- **외래키(Foreign Key)** 는 NULL이 될 수 있습니다
:::

## 🔗 관계(Relationship)의 종류와 참여 방식

### 관계란 무엇인가?

**관계(Relationship)** 는 개체와 개체 사이를 연결하는 **연결고리**입니다. 학생과 과목이라는 두 개체 사이에는 "수강"이라는 관계가 있고, 교수와 학생 사이에는 "지도"라는 관계가 있습니다. ER 다이어그램에서 관계는 **마름모(다이아몬드)** 로 표현됩니다.

### 관계의 4가지 유형 (카디날리티)

관계에는 **카디날리티(Cardinality, 대응 수)** 라는 개념이 있습니다. 이는 "한 개체가 다른 개체와 몇 개씩 대응되는가"를 나타냅니다.

**1:1 (일대일) 관계**
- 한 쪽 개체 하나가 다른 쪽 개체 하나와 대응됩니다
- 예시: 남자 1명과 여자 1명의 결혼 관계
- 예시: 국가 1개와 수도 1개의 관계

**1:N (일대다) 관계**
- 한 쪽 개체 하나가 다른 쪽 개체 여러 개와 대응됩니다
- 예시: 1명의 교수가 여러 명의 학생을 지도
- 예시: 부모 1쌍에서 자녀 여러 명

**N:1 (다대일) 관계**
- 여러 개체가 다른 쪽 개체 하나와 대응됩니다
- 1:N의 방향을 반대로 본 것입니다
- 예시: 여러 명의 학생이 1명의 교수로부터 지도를 받음

**N:M (다대다) 관계**
- 양쪽 모두 여러 개체와 대응됩니다
- 예시: 학생(여러 명)이 과목(여러 개)을 수강
  - 학생 홍길동: 데이터베이스, 운영체제, 알고리즘 수강
  - 데이터베이스 과목: 홍길동, 이영희, 박철수 수강

:::info 관계의 방향성
관계는 항상 방향성이 있습니다. "1명의 교수가 여러 명의 학생을 지도한다"고 보면 1:N이고, "여러 명의 학생이 1명의 교수로부터 지도를 받는다"고 보면 N:1입니다. 어느 방향에서 보는지에 따라 표현이 달라지지만, 관계의 본질은 같습니다.
:::

### 전체 참여 vs 부분 참여

관계에 참여하는 방식도 두 가지로 나뉩니다. 이것은 "반드시 참여해야 하는가, 참여하지 않아도 되는가"의 문제입니다.

**전체 참여 (Total Participation) = 의무적 참여**
- 개체의 **모든 인스턴스**가 반드시 관계에 참여해야 합니다
- ER 다이어그램에서 **이중 선(══)** 으로 표현
- 예시: **사원**은 반드시 어느 부서에 소속되어야 합니다. 부서 없는 사원은 존재할 수 없죠.

**부분 참여 (Partial Participation) = 선택적 참여**
- 개체의 인스턴스가 관계에 참여할 수도 있고, 참여하지 않을 수도 있습니다
- ER 다이어그램에서 **단일 선(─)** 으로 표현
- 예시: **부서**는 사원이 없어도 존재할 수 있습니다. 새로 신설된 부서는 아직 배치된 사원이 없을 수 있습니다.

```
[부서] ──── ◇소속◇ ════ [사원]

부서: 단일 선 → 부분 참여 (사원 없는 부서도 존재 가능)
사원: 이중 선 → 전체 참여 (모든 사원은 반드시 부서에 소속)
```

## ⚠️ 연결함정: 잘못된 설계가 만드는 문제

### 연결함정이란?

아무리 좋은 의도로 설계를 해도 잘못 설계하면 **정보를 잃어버리거나 추적할 수 없게 되는** 문제가 발생합니다. 이를 **연결함정(Connection Trap)** 이라고 합니다.

연결함정에는 두 가지 종류가 있습니다: **부채골 함정**과 **균열 함정**입니다.

### 부채골 함정 (Fan Trap)

**부채골 함정**은 관계 모델의 **위치나 구조가 잘못 설계**되어 특정 정보를 추적할 수 없게 되는 문제입니다.

예를 들어 대학교 데이터베이스를 설계한다고 합시다. 교수, 대학(단과대학), 학과가 있습니다. 설계를 잘못하면 이런 상황이 생길 수 있습니다.

**잘못된 설계 ❌:**
```
[교수] ─── ◇재직◇ ─── [대학]
                         |
                      ◇소속◇
                         |
                       [학과]
```

이 설계에서는:
- 교수가 어느 대학에 있는지는 알 수 있습니다 (교수 → 재직 → 대학)
- 대학에 어떤 학과가 있는지는 알 수 있습니다 (대학 → 소속 → 학과)
- 하지만 **교수가 어느 학과에 속하는지는 알 수 없습니다!** 이것이 부채골 함정입니다.

**올바른 설계 ✅:**
단과대학과 학과의 위치를 바꿔서 모델을 변환하면 해결할 수 있습니다.

```
[교수] ─── ◇재직◇ ─── [학과]
                         |
                      ◇소속◇
                         |
                       [대학]
```

이렇게 하면 교수 → 재직 → 학과를 통해 교수가 어느 학과에 속하는지 알 수 있습니다.

### 균열 함정 (Chasm Trap)

**균열 함정**은 **관계가 누락**되어 특정 정보를 알 수 없게 되는 문제입니다. 부채골 함정은 모델의 위치를 바꿔서 해결할 수 있지만, 균열 함정은 **새로운 관계를 추가**해야 해결됩니다.

**잘못된 설계 ❌:**
```
[학과] ─── ◇재직◇ ─── [교수]
   |
◇지도◇
   |
[학생]
```

이 설계에서는:
- 학과에 어떤 교수가 재직하는지는 알 수 있습니다
- 교수가 어떤 학생을 지도하는지는 알 수 있습니다
- 하지만 **학생이 어느 학과에 속하는지는 알 수 없습니다!** 이것이 균열 함정입니다.

**올바른 설계 ✅:**
학과와 학생 사이에 **"소속"이라는 관계를 새롭게 추가**해서 해결합니다.

```
[학과] ─── ◇재직◇ ─── [교수]
  |                       |
◇소속◇ (새로 추가)       ◇지도◇
  |                       |
[학생] ─────────────────────
```

:::warning 연결함정 요약
| 함정 종류 | 원인 | 해결 방법 |
|---|---|---|
| **부채골 함정** | 관계 모델의 위치/구조가 잘못됨 | 개체의 위치를 변경하여 모델 변환 |
| **균열 함정** | 필요한 관계가 누락됨 | 새로운 관계를 추가 |
:::

## 🚀 확장 ER 모델: 더 복잡한 세계 표현하기

### 왜 확장 ER 모델이 필요한가?

기존의 ER 모델은 단순한 개체, 관계, 속성만으로 데이터를 표현했습니다. 그런데 CAD(컴퓨터 설계) 데이터, 지리정보 데이터, 멀티미디어 데이터 등 **복잡한 현실 세계의 정보**를 표현하려다 보니 기존 ER 모델만으로는 한계가 생겼습니다. 특히 **객체 지향 개념(상속, 재사용 등)** 을 표현하기 어려웠습니다.

그래서 기존 ER 모델에 **일반화, 세분화, 집단화, 연관화** 등의 개념을 추가한 것이 바로 **확장 ER 모델(Extended ER Model)** 입니다.

### 슈퍼클래스와 서브클래스

확장 ER 모델에서는 개체들 사이에 **계층 관계**를 표현할 수 있습니다.

- **슈퍼클래스(Superclass)**: 공통 속성을 가지는 상위 개체 (예: 학생)
- **서브클래스(Subclass)**: 슈퍼클래스의 특수한 형태 (예: 학부생, 대학원생)
- **Is-A 관계**: 슈퍼클래스와 서브클래스의 관계. "학부생은 학생이다(학부생 Is-A 학생)"

예시:
- **학생(슈퍼클래스)**: 학번(기본키), 이름, 학과 → 모든 학생에게 공통
- **학부생(서브클래스)**: 학년 → 학부생에게만 있는 속성
- **대학원생(서브클래스)**: 학위과정 → 대학원생에게만 있는 속성

### D와 O: 중복 허용 여부

서브클래스들 사이에 중복이 허용되는지 여부를 두 가지 기호로 표현합니다.

- **D (Disjoint, 분리)**: 서브클래스들이 서로 **겹치지 않음**. 한 개체는 하나의 서브클래스에만 속할 수 있습니다.
  - 예: 학부생이면서 동시에 대학원생인 학생은 존재할 수 없음
- **O (Overlap, 중첩)**: 서브클래스들이 **겹칠 수 있음**. 한 개체가 여러 서브클래스에 동시에 속할 수 있습니다.
  - 예: 어떤 특수한 학교에서 학부 과정과 대학원 과정을 동시에 밟을 수 있다면 O를 사용

### 일반화 (Generalization): 위로 올라가기

**일반화**는 여러 개체의 **공통점을 찾아 상위 개체(슈퍼클래스)를 만드는** 과정입니다. **하향식에서 상향식**으로, 구체적인 것에서 추상적인 것으로 올라가는 방향입니다.

예시: 승용차와 트럭이 있습니다.
- 승용차 속성: 등록번호, 차대번호, 가격, 최고속도, 승객수
- 트럭 속성: 등록번호, 차대번호, 가격, 적재용량

공통 속성인 **등록번호, 차대번호, 가격**을 위로 올려 **자동차(슈퍼클래스)** 를 만들고, 승용차만의 속성(최고속도, 승객수)과 트럭만의 속성(적재용량)은 각각의 서브클래스로 내립니다.

관련 용어들: **상향식**, **추상화**, **개념화** - 모두 같은 방향의 과정을 가리킵니다.

### 세분화 (Specialization): 아래로 내려가기

**세분화**는 반대로 상위 개체(슈퍼클래스)를 **더 구체적인 하위 개체(서브클래스)로 나누는** 과정입니다. **하향식**으로 내려가는 방향입니다.

예시: 학생이라는 개체를 학부생과 대학원생으로 나누는 것.

관련 용어들: **하향식**, **전문화**, **구체화** - 모두 같은 방향의 과정을 가리킵니다.

```
일반화: 학부생 + 대학원생 ──↑── 학생 (공통점을 찾아 올라감)
세분화: 학생 ──↓── 학부생, 대학원생 (차이점으로 나누어 내려감)
```

### 집단화 (Aggregation): 관계를 개체처럼 다루기

**집단화(Aggregation)** 는 **관계와 관계를 연결해야 할 때** 사용하는 개념입니다. ER 다이어그램에서 기본적으로 관계는 개체와 개체를 연결하는 역할만 합니다. 관계와 관계를 직접 연결하는 것은 불가능합니다.

이런 한계를 극복하기 위해, **여러 개체와 그 관계를 묶어서 하나의 새로운 개체처럼 취급**하는 것이 집단화입니다. 이 묶음을 **어그리게이션(Aggregation)** 이라고도 합니다. Is-a-part-of(~의 일부이다) 관계라고도 표현합니다.

예시: 종업원과 프로젝트가 "작업" 관계로 연결되어 있습니다. 그 작업에서 기계를 "사용"한다는 관계를 추가하고 싶은데, 기계는 관계(작업)에 연결되어야 합니다. 이때 종업원-작업-프로젝트 전체를 묶어서 **"작업들"이라는 새로운 개체**로 만들고, 그 개체와 기계를 "사용" 관계로 연결합니다.

### 연관화 (Association)

**연관화(Association)** 는 개체와 개체 사이의 **연관성 여부를 판단**하는 개념입니다. Is-associated-with(~와 연관된다) 관계를 표현합니다. 두 개체가 서로 어떤 연관을 맺고 있는지를 명시적으로 표현할 때 사용합니다.

## 📊 논리적 데이터 모델의 종류

### ER 다이어그램을 실제 구조로 변환하기

개념적 설계(ER 다이어그램)를 마쳤으면, 이제 이것을 **실제 데이터베이스에서 쓸 수 있는 논리적 구조**로 바꿔야 합니다. 논리적 데이터 모델에는 크게 세 가지 종류가 있습니다.

### 관계 데이터 모델 (Relational Data Model)

가장 널리 사용되는 데이터 모델로, 데이터를 **테이블(행과 열로 이루어진 표)** 형태로 표현합니다. 우리가 흔히 알고 있는 Oracle, MySQL, MSSQL, PostgreSQL 등이 모두 관계형 데이터베이스입니다.

- **행(Row)**: 튜플(tuple)이라고도 하며, 하나의 데이터 레코드
- **열(Column)**: 속성(attribute)이라고도 하며, 데이터의 종류

예를 들어 학생 테이블:

| 학번 | 이름 | 학과 | 학년 |
|---|---|---|---|
| 20240001 | 홍길동 | 컴퓨터공학 | 1 |
| 20240002 | 이영희 | 경영학 | 2 |

:::note 관계 데이터 모델의 특징
관계 데이터 모델은 데이터를 테이블로 표현하기 때문에 **매우 추상적**입니다. 계층형이나 네트워크형처럼 그림으로 관계를 보여주지 않고, 테이블의 키(key)를 통해 관계를 표현합니다. 그래서 처음에는 관계를 파악하기 어려울 수 있지만, 일관성 있고 강력한 구조를 제공합니다.
:::

### 네트워크 데이터 모델 (Network Data Model)

데이터를 **그래프(Graph) 형태**로 표현하는 모델입니다. 오너(Owner)와 멤버(Member)의 관계를 가지며, **사이클(Cycle, 순환 구조)** 이 허용됩니다. 대표적인 DBMS로는 **IDMS**, **Total**이 있습니다.

### 계층 데이터 모델 (Hierarchical Data Model)

데이터를 **트리(Tree) 구조**로 표현하는 모델입니다. 부모-자식 관계로 데이터를 연결하며, 트리 구조의 특성상 **사이클이 허용되지 않습니다**. 자식은 부모가 될 수 없고, 한 개의 자식 노드는 오직 하나의 부모만 가집니다. 대표적인 DBMS로는 **IMS(Information Management System)** 가 있습니다.

| 특성 | 관계형 | 네트워크형 | 계층형 |
|---|---|---|---|
| **표현 방식** | 테이블 | 그래프 | 트리 |
| **사이클 허용** | N/A | ✅ 허용 | ❌ 불허 |
| **대표 DBMS** | Oracle, MySQL | IDMS, Total | IMS |
| **주요 관계** | 키(Key) 연결 | 오너-멤버 | 부모-자식 |

## 🎓 핵심 정리

이 문서에서 학습한 핵심 개념들을 정리합니다.

- **데이터 모델링 3단계**: 개념적 모델링(ER 다이어그램 작성) → 데이터 모델링(테이블 변환) → 데이터 구조화(물리적 저장 구조 결정)
- **DB 설계 3단계**: 개념적 설계(DBMS·HW 독립) → 논리적 설계(DBMS 종속, HW 독립) → 물리적 설계(DBMS·HW 모두 종속)
- **데이터 모델의 구성 요소(SOC)**: 구조(Structure), 연산(Operation), 제약 조건(Constraint)
- **ER 다이어그램 기호**: 직사각형(개체), 마름모(관계), 타원(속성), 이중 직사각형(약한 개체), 이중 마름모(식별 관계)
- **강한 개체 vs 약한 개체**: 강한 개체는 단독 존재 가능, 약한 개체는 강한 개체 없이 존재 불가
- **부분키**: 약한 개체가 가지는 식별자로, 강한 개체의 기본키와 결합해야 유일한 식별이 가능
- **속성의 종류**: 단순/복합, 단일값/다중값, 저장/유도 속성
- **NULL의 의미**: 해당 사항 없음 / 값은 있으나 알 수 없음 / 값의 존재 여부조차 알 수 없음 (0이나 빈 문자열과 다름)
- **설명속성**: 관계(다이아몬드)가 가지는 속성으로, 관계 자체의 특성을 표현
- **관계 유형**: 1:1, 1:N, N:1, N:M
- **참여 방식**: 전체 참여(이중 선, 의무적), 부분 참여(단일 선, 선택적)
- **연결함정**: 부채골 함정(모델 위치 오류 → 위치 변환으로 해결), 균열 함정(관계 누락 → 관계 추가로 해결)
- **확장 ER 모델**: 일반화(상향식, 공통점 추출), 세분화(하향식, 차이점 구분), 집단화(관계를 개체처럼 묶기), 연관화(연관성 표현)
- **슈퍼클래스/서브클래스**: D(Disjoint, 중복 불허), O(Overlap, 중복 허용)
- **논리적 데이터 모델 종류**: 관계형(테이블), 네트워크형(그래프, 사이클 허용), 계층형(트리, 사이클 불허)
