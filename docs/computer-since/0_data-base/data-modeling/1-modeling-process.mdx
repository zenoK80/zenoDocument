# 데이터 모델링 프로세스: 개념적 모델링부터 데이터 구조화까지 🗺️

데이터베이스를 공부하다 보면 "데이터 모델링"이라는 말을 자주 듣게 됩니다. 그런데 막상 이게 무슨 뜻인지, 왜 필요한지, 어떤 순서로 이루어지는지 제대로 설명해 주는 곳이 많지 않습니다. 이 문서에서는 데이터 모델링의 전체 프로세스를 처음부터 끝까지, 비전공자도 이해할 수 있도록 아주 친절하게 풀어 설명합니다.

:::info 이 문서에서 배울 내용
- 데이터 모델링이란 무엇인가 (개념적 모델링 → 데이터 모델링 → 데이터 구조화)
- 데이터 모델의 세 가지 구성 요소 (구조, 연산, 제약 조건)
- ER 다이어그램의 구성 요소 (개체, 관계, 속성)
- 약한 개체와 강한 개체의 차이
- 속성의 종류 (단순, 복합, 단일값, 다중값, 저장, 유도 속성)
- NULL의 의미와 올바른 이해
- 관계 타입과 참여 제약 조건
- 연결함정 (부채골 함정, 균열 함정)
- 확장 ER 모델 (일반화, 세분화, 집단화, 연관화)
- 관계·네트워크·계층 데이터 모델 비교
:::

## 🏗️ 데이터 모델링이란 무엇인가?

### 현실 세계를 데이터베이스로 옮기는 과정

우리가 살고 있는 세상에는 수많은 것들이 존재합니다. 학생이 있고, 교수가 있고, 과목이 있습니다. 학생은 과목을 수강하고, 교수는 학생을 지도하고, 교수는 과목을 가르칩니다. 이런 현실 세계의 복잡한 상황을 컴퓨터의 데이터베이스 안에 담으려면 어떻게 해야 할까요?

마치 건물을 짓기 전에 **설계도(blueprint)**를 그리듯이, 데이터베이스를 만들기 전에도 설계 과정이 필요합니다. 이 설계 과정 전체를 가리켜 **데이터 모델링**이라고 합니다.

> 데이터 모델링이란 현실 세계에 존재하는 유형·무형의 다양한 환경들을 실제 데이터베이스로 구현하기 위해 모델링해 나가는 과정입니다.

### 세 가지 핵심 용어: 개념적 모델링 → 데이터 모델링 → 데이터 구조화

많은 분들이 헷갈려 하는 부분이 바로 이 세 가지 용어입니다. 비슷해 보이지만 각각 다른 단계를 가리킵니다. 아래 표로 한눈에 정리해 봅니다.

| 단계 | 용어 | 하는 일 | 결과물 |
|---|---|---|---|
| 1단계 | **개념적 모델링** | 현실 세계를 추상적 개념으로 표현 | 개념적 구조 (ER 다이어그램) |
| 2단계 | **데이터 모델링** | 개념적 구조를 논리적 구조로 변환 | 논리적 구조 (테이블) |
| 3단계 | **데이터 구조화** | 논리적 구조를 물리적 구조로 변환 | 물리적 구조 (디스크 저장 형태) |

**1단계 — 개념적 모델링**: "학생이 과목을 수강한다", "교수가 학생을 지도한다"는 현실 세계의 상황을 **추상적인 도형(그래프)**으로 표현하는 과정입니다. 여기서 만들어지는 결과물이 바로 **ER 다이어그램(Entity-Relationship Diagram)**입니다. ER 다이어그램은 개체(학생, 교수, 과목)와 그 관계(수강, 지도, 강의)를 도형으로 그린 그림이라고 생각하면 됩니다.

**2단계 — 데이터 모델링**: ER 다이어그램으로 표현된 개념적 구조를 실제 **테이블(표)** 형태로 변환합니다. 우리가 흔히 엑셀처럼 행과 열로 이루어진 표를 생각하면 됩니다. 이 단계의 결과물이 **논리적 구조**입니다.

**3단계 — 데이터 구조화**: 테이블로 표현된 논리적 구조를 실제 컴퓨터의 **디스크(하드드라이브)**에 저장할 수 있는 물리적인 형태로 변환합니다. 이 단계의 결과물이 **물리적 구조**입니다.

:::tip 쉬운 비유로 이해하기
건물 건축에 비유하면 이렇습니다.
- **개념적 모델링** = 건축가가 손으로 스케치하는 초기 아이디어 그림
- **데이터 모델링** = 정식 설계 도면 (CAD로 그린 도면)
- **데이터 구조화** = 실제 공사 (벽돌을 쌓고 기둥을 세우는 것)
:::

### 데이터베이스 설계의 전체 단계 복습

데이터베이스를 설계할 때는 크게 세 가지 설계 단계를 거칩니다. 이 단계들은 위에서 설명한 모델링 프로세스와 맞닿아 있습니다.

| 설계 단계 | 주요 작업 | DBMS 종속성 | 하드웨어 종속성 |
|---|---|---|---|
| **요구사항 수집·분석** | 사용자 요구사항 파악 | 독립 | 독립 |
| **개념적 설계** | ER 다이어그램 작성 | 독립 | 독립 |
| **논리적 설계** | 테이블(관계형/계층형/네트워크형) 변환 | **종속** | 독립 |
| **물리적 설계** | 디스크 물리 구조로 변환 | **종속** | **종속** |

- **개념적 설계**는 어떤 DBMS(Oracle이든 MySQL이든)를 쓸지, 어떤 하드웨어를 쓸지와 **전혀 상관없이** 설계할 수 있습니다. 즉, 독립적입니다.
- **논리적 설계**는 어떤 DBMS를 사용하느냐에 따라 설계 방식이 달라집니다. 관계형 DBMS라면 테이블로, 계층형 DBMS라면 트리 구조로 만들어야 하기 때문에 DBMS에 종속됩니다.
- **물리적 설계**는 DBMS와 하드웨어 모두에 종속됩니다.

## 🔧 데이터 모델의 구성 요소: SOC

### 데이터 모델은 세 가지로 이루어진다

**데이터 모델(Data Model)**이란 데이터를 어떻게 구성하고, 어떤 작업을 할 수 있으며, 어떤 규칙을 지켜야 하는지를 정의한 틀입니다. 이 데이터 모델은 세 가지 요소로 구성됩니다. 영어 첫 글자를 따서 **SOC**라고 기억하면 편합니다.

> **S**tructure(구조) + **O**peration(연산) + **C**onstraint(제약 조건) = 데이터 모델

### S — 구조 (Structure)

**구조**란 데이터베이스에서 표현해야 할 대상(개체)과 개체 간의 관계를 정의한 것입니다. 마치 레고 블록의 설계도처럼, "어떤 블록(개체)이 있고, 블록들이 어떻게 연결되는가(관계)"를 정해 놓은 것입니다.

예를 들어, 학생(개체)이 있고 학번·이름·성적·학년(속성)을 가지며, 교수(개체)와 '지도'라는 관계를 맺는다는 것이 모두 구조에 해당합니다.

### O — 연산 (Operation)

**연산**이란 데이터베이스에 실제 값(인스턴스)을 가지고 수행하는 작업을 말합니다. 여기서 **인스턴스(instance)**란 실제 저장된 데이터 값을 가리킵니다. 예를 들어, "학번 20230001, 이름 홍길동, 학년 2"처럼 실제로 저장된 한 행(row)의 데이터가 인스턴스입니다.

연산의 종류에는 다음 네 가지가 있습니다.
- **삽입(Insert)**: 새로운 데이터를 추가하는 것
- **삭제(Delete)**: 기존 데이터를 지우는 것
- **갱신(Update)**: 기존 데이터를 수정하는 것
- **검색(Select)**: 저장된 데이터를 조회하는 것

### C — 제약 조건 (Constraint)

**제약 조건**이란 구조와 연산이 이루어지는 과정에서 반드시 지켜야 하는 규칙들입니다. 마치 교통 법규처럼, 데이터가 지켜야 할 최소한의 규칙입니다.

대표적인 제약 조건으로는 아래 세 가지가 있습니다.
- **개체 무결성**: 기본키(Primary Key)는 NULL 값을 가질 수 없고, 중복될 수 없습니다.
- **참조 무결성**: 외래키(Foreign Key)는 참조하는 테이블에 존재하는 값이어야 합니다.
- **도메인 무결성**: 각 속성의 값은 정해진 범위(도메인) 안에 있어야 합니다.

## 📐 ER 모델과 ER 다이어그램의 구성 요소

### ER 모델이란?

**ER 모델(Entity-Relationship Model)**은 개념적 데이터 모델 중 가장 널리 사용되는 모델입니다. **Entity(개체)**와 **Relationship(관계)**의 앞 글자를 따서 ER 모델이라고 부릅니다. 1976년 피터 첸(Peter Chen)이라는 연구자가 제안한 모델로, 현실 세계의 데이터를 **그래프(도형) 형태**로 표현합니다.

ER 모델의 가장 큰 장점은 **DBMS와 하드웨어에 독립적**이라는 것입니다. 어떤 데이터베이스 시스템을 사용하더라도 동일한 ER 다이어그램으로 설계할 수 있습니다.

### ER 다이어그램의 도형 요소

ER 다이어그램에서는 각각 다른 의미를 가진 도형들을 사용합니다.

| 도형 | 이름 | 의미 |
|---|---|---|
| 사각형 (□) | 개체(Entity) | 독립적으로 존재하는 대상 (학생, 교수, 과목) |
| 이중 사각형 (□□) | 약한 개체(Weak Entity) | 강한 개체 없이 단독 존재 불가능한 개체 |
| 다이아몬드 (◇) | 관계(Relationship) | 개체 간의 연결 (수강, 지도, 강의) |
| 이중 다이아몬드 (◇◇) | 식별 관계 | 약한 개체가 갖는 관계 |
| 원 (○) | 속성(Attribute) | 개체나 관계가 가진 특성 (이름, 학번) |
| 밑줄 있는 원 | 기본키 속성 | 개체를 유일하게 식별하는 속성 |
| 점선 밑줄 원 | 부분키 속성 | 약한 개체의 부분적 식별자 |
| 점선 원 | 유도 속성 | 다른 속성으로부터 계산되는 속성 |
| 이중 원 | 다중값 속성 | 여러 개의 값을 가질 수 있는 속성 |
| 계층 원 | 복합 속성 | 더 작은 속성으로 분해 가능한 속성 |
| 단일 선 | 부분 참여 | 관계에 참여해도 되고 안 해도 됨 |
| 이중 선 | 전체 참여 | 반드시 관계에 참여해야 함 |

### ER 다이어그램 실제 예시

아래는 교수·학생·과목의 관계를 ER 다이어그램으로 표현한 예입니다.

```
[교수] ─── <지도> ─── [학생]
  │    1       N    │
  │                 │
<강의>           <등록>
  │    N       M    │
  │                 │
[과목] ────────────┘
          (성적: 설명속성)
```

- **교수와 학생**: **1:N 관계** — 한 명의 교수가 여러 명의 학생을 지도합니다.
- **교수와 과목**: **1:N 관계** — 한 명의 교수가 여러 과목을 강의합니다.
- **학생과 과목**: **N:M 관계** — 한 학생이 여러 과목을 수강하고, 한 과목을 여러 학생이 수강합니다.

각 개체의 속성은 다음과 같습니다.
- **교수**: 교수번호(기본키), 이름, 전공, 학과
- **학생**: 학번(기본키), 이름, 주소, 학년
- **과목**: 과목번호(기본키), 과목명

그리고 학생과 과목의 **등록** 관계에는 **성적**이라는 **설명속성(Descriptive Attribute)**이 붙어 있습니다. 성적은 특정 학생이 특정 과목을 수강했을 때만 생기는 값이므로, 학생 개체에도 과목 개체에도 넣기 애매하고, **등록이라는 관계에 붙이는 것이 가장 자연스럽습니다.**

:::note 설명속성이란?
관계(다이아몬드)가 직접 소유하는 속성을 **설명속성**이라고 합니다. 이 속성은 두 개체 중 어느 한 곳에 넣기 어려울 때, 관계 자체에 달아 줍니다. 성적처럼 "어떤 학생이 어떤 과목을 등록했을 때"라는 관계가 존재해야만 의미가 생기는 속성이 대표적입니다.
:::

## 💪 강한 개체와 약한 개체

### 왜 개체에 '강하다', '약하다'는 구분이 필요한가?

현실 세계에는 **혼자서 존재할 수 있는 것**이 있는가 하면, **다른 무언가에 의존해야만 존재할 수 있는 것**도 있습니다. 예를 들어, 직원은 회사에서 독립적으로 존재할 수 있습니다. 하지만 직원의 **부양가족**은 어떨까요? 부양가족은 직원이 있어야만 의미가 생깁니다. 직원이 회사를 떠나면, 그 부양가족 정보도 더 이상 관리할 필요가 없어집니다.

이처럼 **혼자서 독립적으로 존재할 수 있는 개체**를 **강한 개체(Strong Entity)**, **다른 개체에 종속되어야만 존재할 수 있는 개체**를 **약한 개체(Weak Entity)**라고 합니다.

### 약한 개체의 특징

약한 개체는 다음과 같은 특징을 가집니다.

1. **자기 자신의 속성만으로는 유일하게 식별할 수 없습니다.**
   - 예를 들어, 삼성전자에는 수천 명의 부양가족이 있을 것이고, 그 중에 "김민준"이라는 이름을 가진 사람이 여러 명일 수 있습니다. 이름만으로는 어떤 직원의 부양가족인지 식별이 불가능합니다.

2. **반드시 강한 개체와 연결되어야 식별됩니다.**
   - "직원번호 1001번 직원의 부양가족 김민준"처럼, 강한 개체의 기본키(직원번호)를 함께 사용해야 유일하게 식별할 수 있습니다.

3. **약한 개체가 가지는 부분적인 식별자를 부분키(Partial Key)라고 합니다.**
   - 부분키는 ER 다이어그램에서 **점선 밑줄**로 표시합니다.

4. **약한 개체는 강한 개체에 반드시 전체 참여합니다.**
   - 직원 없는 부양가족은 존재할 수 없으므로, 부양가족은 항상 직원과의 관계에 참여해야 합니다.

5. **약한 개체와 강한 개체의 관계는 항상 1:N 관계이며, 이를 식별 관계(Identifying Relationship)라고 합니다.**
   - 이중 다이아몬드(◇◇)로 표시합니다.

```
[직원] ════ <<부양>> ════ [[부양가족]]
  │     1          N
직원번호        이름(부분키)
(기본키)        나이
이름           관계
```

:::warning 약한 개체 vs 강한 개체 구분 주의!
- 약한 개체는 ER 다이어그램에서 **이중 사각형**으로 표시합니다.
- 약한 개체가 갖는 관계는 **이중 다이아몬드**로 표시합니다.
- 약한 개체의 식별자는 **기본키가 아닌 부분키**입니다.
- 약한 개체는 강한 개체 없이는 **단독으로 존재할 수 없습니다.**
:::

## 🏷️ 속성의 종류

### 왜 속성을 여러 종류로 나누는가?

데이터베이스에 저장되는 정보의 특성이 모두 같지 않습니다. 어떤 정보는 하나의 값만 가지고, 어떤 정보는 여러 값을 가집니다. 어떤 정보는 직접 저장되고, 어떤 정보는 다른 값으로부터 계산해서 얻습니다. 이런 다양한 특성을 정확하게 표현하기 위해 속성을 여러 종류로 나눕니다.

### 단순 속성 vs 복합 속성

**단순 속성(Simple Attribute)**은 더 이상 작은 단위로 나눌 수 없는 속성입니다. 예를 들어, 학생의 '이름'은 더 이상 분해할 이유가 없으므로 단순 속성입니다.

**복합 속성(Composite Attribute)**은 더 작은 여러 속성으로 분해할 수 있는 속성입니다. 예를 들어, '주소'는 '시/도', '구/군', '동/읍면', '상세주소'로 나눌 수 있습니다. ER 다이어그램에서는 계층 형태(부모 원 아래 자식 원들)로 표현합니다.

```
         주소 (복합 속성)
        /    |    \
      시/도  구/군  상세주소
```

### 단일값 속성 vs 다중값 속성

**단일값 속성(Single-valued Attribute)**은 하나의 값만 가지는 속성입니다. 사람의 '나이'는 하나뿐이므로 단일값 속성입니다.

**다중값 속성(Multi-valued Attribute)**은 여러 개의 값을 동시에 가질 수 있는 속성입니다. 예를 들어, '취미'는 독서, 운동, 요리처럼 여러 개를 가질 수 있습니다. '학위'도 학사, 석사처럼 여러 개를 가질 수 있습니다. ER 다이어그램에서는 **이중 원**으로 표시합니다.

### 저장 속성 vs 유도 속성

**저장 속성(Stored Attribute)**은 실제로 하드디스크에 저장되는 속성입니다. 학생의 '이름', '주민번호' 같은 것들이 저장 속성입니다.

**유도 속성(Derived Attribute)**은 다른 속성으로부터 계산해서 얻을 수 있는 속성으로, 실제로 데이터베이스에 저장하지 않습니다. 예를 들어, '주민번호'가 저장되어 있다면, '나이'는 주민번호에서 생년월일을 추출해 현재 날짜와 계산하면 구할 수 있습니다. 따라서 '나이'는 유도 속성입니다. ER 다이어그램에서는 **점선 원**으로 표시합니다.

```
  학생
   │
   ├── 학번 (저장 속성, 기본키)
   ├── 이름 (저장 속성)
   ├── 주민번호 (저장 속성)
   └── 나이 (유도 속성 — 주민번호로부터 계산)
```

:::tip 유도 속성을 사용하는 이유
나이를 직접 저장하면, 매년 나이가 바뀔 때마다 데이터를 업데이트해야 합니다. 하지만 주민번호(저장 속성)로부터 나이를 계산(유도)하면, 주민번호는 변하지 않으므로 항상 정확한 나이를 얻을 수 있습니다. 이처럼 유도 속성은 **데이터 관리의 효율성**을 위해 사용합니다.
:::

## 🚫 NULL의 진짜 의미

### NULL은 0도 아니고 공백도 아니다

데이터베이스를 배우는 분들이 자주 오해하는 개념 중 하나가 바로 **NULL**입니다. NULL은 숫자 0과도 다르고, 빈 문자열("")과도 전혀 다릅니다. NULL은 **NULL 그 자체만의 의미**를 가집니다.

NULL이 사용되는 상황은 크게 세 가지입니다.

### 상황 1: 해당 사항이 없는 경우

테이블에 학생 정보를 저장할 때 '병역' 컬럼이 있다고 가정해 봅니다. 남학생은 '군필', '미필', '면제' 중 하나의 값이 들어갑니다. 하지만 **여학생은 병역과 관련이 없습니다.** 해당 사항 자체가 없는 것이죠. 이때 병역 컬럼에 **NULL**을 입력합니다.

또 다른 예로, 아파트에 거주하는 사람에게는 '동호수' 정보가 있지만, **단독주택에 거주하는 사람에게는 동호수가 필요 없습니다.** 이때도 동호수 컬럼에 NULL을 씁니다.

### 상황 2: 값은 존재하지만 아직 알 수 없는 경우

주소는 모든 사람이 반드시 가지고 있습니다. 하지만 어제 이사를 해서 아직 새 주소를 정확히 모를 수 있습니다. 이때 주소 컬럼에 NULL을 씁니다. **값이 반드시 존재하지만, 지금 당장 그 값이 무엇인지 알 수 없는 상태**입니다.

### 상황 3: 값의 존재 여부조차 알 수 없는 경우

집전화번호를 예로 들면, 어떤 사람은 집전화가 있고 어떤 사람은 없습니다. 더 나아가 있는지 없는지조차 파악이 안 된 경우가 있습니다. 이때도 NULL을 사용합니다.

```sql
-- NULL 사용 예시
-- 학생 테이블에 데이터 삽입

-- ✅ 올바른 예: 여학생의 병역란에 NULL 삽입
INSERT INTO 학생(학번, 이름, 성별, 병역)
VALUES ('20230001', '김지수', '여', NULL);
-- NULL은 '해당 없음'을 의미합니다.

-- ❌ 잘못된 예: NULL 대신 0이나 빈 문자열을 쓰는 경우
INSERT INTO 학생(학번, 이름, 성별, 병역)
VALUES ('20230002', '이수진', '여', '');
-- 빈 문자열('')은 NULL이 아닙니다. 의미가 다릅니다!

INSERT INTO 학생(학번, 이름, 성별, 병역)
VALUES ('20230003', '박민서', '여', 0);
-- 0은 NULL이 아닙니다. 0은 숫자 값입니다!
```

:::danger NULL과 기본키
기본키(Primary Key)는 절대로 NULL 값을 가질 수 없습니다. 기본키는 각 행을 유일하게 식별하는 역할을 하는데, NULL은 "알 수 없음"을 의미하므로 식별이 불가능해집니다.

반면 외래키(Foreign Key)는 NULL 값을 가질 수 있습니다. 예를 들어, 아직 부서가 배정되지 않은 신입사원의 부서 코드는 NULL일 수 있습니다.
:::

## 🔗 관계 타입과 참여 제약 조건

### 관계의 네 가지 타입

개체와 개체 사이의 관계는 **몇 대 몇**으로 연결되느냐에 따라 네 가지로 나뉩니다. 이를 **카디널리티(Cardinality)**라고도 합니다.

| 관계 타입 | 설명 | 실생활 예시 |
|---|---|---|
| **1:1 (일대일)** | 양쪽 각각 하나씩 연결 | 남자 1명 ↔ 여자 1명 결혼 관계 |
| **1:N (일대다)** | 한쪽 하나가 다른 쪽 여러 개와 연결 | 교수 1명 → 학생 여러 명 지도 |
| **N:1 (다대일)** | 한쪽 여러 개가 다른 쪽 하나와 연결 | 학생 여러 명 → 교수 1명에게 지도 받음 |
| **N:M (다대다)** | 양쪽 모두 여러 개씩 연결 | 학생 여러 명 ↔ 과목 여러 개 수강 |

:::note 1:N과 N:1의 차이
1:N과 N:1은 **방향성**의 차이입니다. "교수가 학생을 지도한다"고 보면 1:N이고, "학생이 교수에게 지도를 받는다"고 보면 N:1입니다. 관계는 항상 방향성이 있기 때문에, 어느 방향에서 바라보느냐에 따라 1:N이 될 수도 있고 N:1이 될 수도 있습니다.
:::

### 전체 참여 vs 부분 참여

관계에 참여할 때 "반드시 참여해야 하는가, 아니면 선택적으로 참여할 수 있는가"에 따라 **전체 참여**와 **부분 참여**로 나뉩니다.

**부분 참여(Partial Participation, 선택적 참여)**는 관계에 참여해도 되고 안 해도 됩니다. ER 다이어그램에서 **단일 선**으로 표시하며, **옵셔널(Optional)**이라고도 합니다.

예를 들어, 부서와 사원의 '소속' 관계에서 **부서**는 부분 참여입니다. 새로 신설된 부서에는 아직 소속 사원이 없을 수 있기 때문입니다. 부서가 소속 관계에 참여하지 않아도 (사원이 없어도) 부서는 존재할 수 있습니다.

**전체 참여(Total Participation, 의무적 참여)**는 반드시 관계에 참여해야 합니다. ER 다이어그램에서 **이중 선**으로 표시하며, **토탈(Total)**이라고도 합니다.

위의 예에서 **사원**은 전체 참여입니다. 우리 회사에 어느 부서에도 소속되지 않은 사원이 존재할 수는 없기 때문입니다. 사원은 반드시 어떤 부서에 소속되어야 합니다.

```
[부서] ─── <소속> ════ [사원]
 (부분참여)              (전체참여)
  단일선                 이중선
```

## ⚠️ 연결함정: 관계 설계의 함정

### 연결함정이란 무엇인가?

**연결함정(Connection Trap)**이란 ER 다이어그램에서 개체와 관계 타입을 잘못 설계하여, **기존에 파악할 수 있었던 정보가 누락되거나 추적이 불가능해지는 현상**입니다. 잘못된 모델 설계로 인해 "관계의 구멍"이 생기는 것입니다.

연결함정에는 두 가지 종류가 있습니다.

### 부채골 함정 (Fan Trap)

**부채골 함정**은 관계 모델을 잘못 설계하여 **정보를 추적할 수 없게 되는 문제**입니다. 이름처럼 마치 부채처럼 퍼져나가다가 길을 잃는 상황입니다.

**예시**: 교수, 대학, 학과의 관계를 생각해 봅시다. 우리가 알고 싶은 정보는 "어떤 교수가 어느 학과에 소속되어 있는가?"입니다.

그런데 잘못된 모델을 아래처럼 설계했다고 가정합니다.

```
[교수] ─── <재직> ─── [대학] ─── <소속> ─── [학과]
```

이 모델에서는 "교수가 어느 대학에 재직하는지"와 "대학에 어떤 학과가 있는지"는 알 수 있습니다. 하지만 **"교수가 어느 학과에 속하는지"는 알 수 없습니다!** 대학 → 여러 학과로 퍼져나가는 1:N 관계 때문에 교수와 특정 학과를 연결할 방법이 없어지는 것입니다.

이를 해결하려면 **모델을 변환**하여 교수와 학과가 직접 연결될 수 있도록 설계해야 합니다.

```
-- ✅ 올바른 설계
[교수] ─── <재직> ─── [학과] ─── <소속> ─── [대학]
```

이렇게 하면 교수 → 학과 → 대학의 순서로 정보를 추적할 수 있습니다.

### 균열 함정 (Chasm Trap)

**균열 함정**은 부채골 함정과 다르게, **관계 자체가 누락되어** 정보를 파악할 수 없게 되는 문제입니다.

**예시**: 학과, 교수, 학생의 관계를 생각해 봅시다. 아래 모델을 보겠습니다.

```
[학과] ─── <재직> ─── [교수] ─── <지도> ─── [학생]
```

이 모델에서는 "어떤 학과에 어떤 교수가 재직하는지"와 "어떤 교수가 어떤 학생을 지도하는지"는 알 수 있습니다. 하지만 **"어떤 학생이 어느 학과 소속인지"는 알 수 없습니다!** 학과와 학생을 직접 연결하는 관계가 없기 때문입니다.

부채골 함정과 달리, 균열 함정은 모델의 위치를 바꾼다고 해결되지 않습니다. **새로운 관계를 추가**해야 합니다.

```
-- ✅ 올바른 설계: 학과와 학생 사이에 '등록' 관계 추가
[학과] ─── <재직> ─── [교수] ─── <지도> ─── [학생]
  └──────────────── <등록> ──────────────────┘
```

| 함정 종류 | 문제 원인 | 해결 방법 |
|---|---|---|
| **부채골 함정** | 관계 모델의 위치가 잘못됨 | 개체나 관계의 위치를 변환 |
| **균열 함정** | 필요한 관계가 누락됨 | 새로운 관계를 추가 |

:::warning 연결함정 주의
연결함정은 실제 데이터베이스 설계에서 자주 발생하는 실수입니다. ER 다이어그램을 그린 후에는 반드시 "필요한 모든 정보를 추적할 수 있는가?"를 검토해야 합니다.
:::

## 🌐 확장 ER 모델 (Extended ER Model)

### 왜 확장 ER 모델이 필요한가?

전통적인 ER 모델은 개체, 관계, 속성이라는 세 가지 요소만으로 현실 세계를 표현합니다. 하지만 현대의 데이터베이스에서는 **CAD 데이터, 지리정보 데이터, 멀티미디어 데이터** 등 훨씬 복잡한 데이터를 다루어야 합니다. 또한 객체지향 프로그래밍 개념인 **상속(inheritance)**이나 **재사용** 같은 개념도 표현해야 할 필요가 생겼습니다.

기존 ER 모델로는 이런 복잡한 요구사항을 모두 커버할 수 없어서, **전문화, 일반화, 상속, 집단화, 연관화** 같은 개념들을 추가한 것이 바로 **확장 ER 모델(Extended ER Model, EER)**입니다.

### 슈퍼클래스와 서브클래스

확장 ER 모델의 핵심 개념은 **슈퍼클래스(Superclass)**와 **서브클래스(Subclass)**입니다.

예를 들어, '학생'이라는 개체는 **학부생**과 **대학원생**으로 나눌 수 있습니다.
- 공통 속성: 학번(기본키), 이름, 학과 → **슈퍼클래스 '학생'**에 위치
- 학부생만의 속성: 학년 → **서브클래스 '학부생'**에 위치
- 대학원생만의 속성: 학위과정 → **서브클래스 '대학원생'**에 위치

```
         [학생] ← 슈퍼클래스
       학번, 이름, 학과
             │
            (D)  ← Disjoint (중복 불허)
           /   \
    [학부생]   [대학원생] ← 서브클래스
      학년       학위과정
```

슈퍼클래스와 서브클래스의 관계를 **Is-A 관계**라고 합니다. "학부생은 학생이다(학부생 Is-A 학생)", "대학원생은 학생이다(대학원생 Is-A 학생)"처럼 표현합니다.

**D (Disjoint, 분리)**: 학부생과 대학원생을 동시에 할 수 없다면 D를 표시합니다. 두 서브클래스가 **중복을 허용하지 않습니다.**

**O (Overlap, 중첩)**: 특수한 경우, 학부생이면서 동시에 대학원생인 경우가 가능하다면 O를 표시합니다. 두 서브클래스가 **중복을 허용합니다.**

### 일반화 vs 세분화

같은 슈퍼클래스-서브클래스 구조를 어느 방향으로 바라보느냐에 따라 이름이 달라집니다.

**일반화(Generalization)**는 **아래에서 위로** 올라가는 방향입니다. 학부생과 대학원생의 **공통점을 찾아** 상위 개념인 '학생'으로 묶는 것입니다. 즉, 차이점은 무시하고 공통점을 찾아 일반화하는 것입니다.

- 방향: **상향식(Bottom-up)**
- 다른 말: 추상화, 개념화

**세분화(Specialization)**는 **위에서 아래로** 내려가는 방향입니다. '학생'을 더 구체적으로 나누어 학부생과 대학원생으로 세분하는 것입니다.

- 방향: **하향식(Top-down)**
- 다른 말: 전문화, 구체화
- 대상: 항상 슈퍼클래스를 세분화합니다.

```
          학부생, 대학원생 ──(일반화: 공통점 추출)──▶ 학생
          학생 ──(세분화: 차이점 강조)──▶ 학부생, 대학원생
```

### 자동차 예시로 이해하는 일반화

승용차와 트럭의 속성을 살펴봅니다.
- **공통 속성**: 등록번호, 차대번호, 가격
- **승용차만의 속성**: 최고속도, 승객수
- **트럭만의 속성**: 용적량

이를 일반화하면:

```
          [자동차] ← 슈퍼클래스
       등록번호, 차대번호, 가격
                │
               (D)
              /   \
       [승용차]   [트럭] ← 서브클래스
     최고속도,승객수  용적량
```

### 집단화 (Aggregation)

**집단화(Aggregation)**는 **개체와 관계, 그리고 또 다른 관계를 표현**할 때 필요합니다. ER 모델에서 관계는 개체와 개체 사이에만 존재할 수 있고, 관계와 관계 사이에는 직접 관계를 맺을 수 없습니다.

예를 들어, "종업원이 프로젝트에서 작업한다"는 관계가 있고, 이 "작업"이라는 관계에서 "기계를 사용한다"는 또 다른 관계를 표현하고 싶을 때 문제가 생깁니다.

이를 해결하기 위해 **종업원 + 프로젝트 + 작업 관계 전체**를 하나의 단위(개체처럼)로 묶어버립니다. 이렇게 묶인 단위를 **집단(Aggregation)**이라고 하고, 이 집단과 '기계' 개체 사이에 '사용' 관계를 맺을 수 있게 됩니다.

- 집단화의 관계 표현: **Is-a-part-of (부분 관계)**
- 마치 레고 블록 여러 개를 하나의 조립 완성품으로 묶는 것과 같습니다.

### 연관화 (Association)

**연관화(Association)**는 개체와 개체 사이의 **연관성 여부를 판단**하는 개념입니다. Is-associated-with 관계를 표현하며, 두 개체가 서로 관련이 있는지 없는지를 나타냅니다.

## 📊 관계·네트워크·계층 데이터 모델 비교

### 논리적 데이터 모델의 세 가지 종류

ER 다이어그램(개념적 구조)을 논리적 구조로 변환할 때, 어떤 방식으로 표현하느냐에 따라 세 가지 데이터 모델로 나뉩니다.

| 구분 | 관계 데이터 모델 | 네트워크 데이터 모델 | 계층 데이터 모델 |
|---|---|---|---|
| **표현 방식** | 테이블 (행/열) | 그래프 (오너-멤버) | 트리 구조 |
| **사이클 허용** | - | ✅ 허용 | ❌ 불허 |
| **대표 DBMS** | Oracle, MySQL, MS SQL, DB2 | IDS, TOTAL | IMS |
| **특징** | 추상적, 유연성 높음 | 복잡한 관계 표현 | 부모-자식 관계 |

### 관계 데이터 모델 (Relational Data Model)

**관계 데이터 모델**은 데이터를 **테이블(표)** 형태로 표현합니다. 우리가 흔히 알고 있는 Oracle, MySQL, MS SQL Server 등이 모두 관계 데이터 모델을 기반으로 합니다.

테이블은 **행(Row, 튜플)**과 **열(Column, 속성)**로 구성됩니다. 네트워크나 계층 모델에 비해 **추상적**으로 표현되지만, 그만큼 유연하고 강력합니다.

```sql
-- 관계 데이터 모델의 예: 테이블로 표현된 학생 데이터
-- 학생 테이블
SELECT * FROM 학생;

/*
학번       | 이름    | 학년 | 학과
---------- | ------- | ---- | ----
20230001   | 홍길동  |  2   | 컴퓨터공학
20230002   | 김철수  |  1   | 전자공학
20230003   | 이영희  |  3   | 경영학
*/
```

### 네트워크 데이터 모델 (Network Data Model)

**네트워크 데이터 모델**은 데이터를 **그래프(Graph)** 형태로 표현합니다. **오너(Owner)**와 **멤버(Member)**의 관계로 데이터를 연결합니다. **사이클(순환 구조)**을 허용하기 때문에 복잡한 관계를 표현할 수 있습니다. 대표적인 DBMS로는 TOTAL 등이 있습니다.

### 계층 데이터 모델 (Hierarchical Data Model)

**계층 데이터 모델**은 데이터를 **트리(Tree)** 구조로 표현합니다. 나무를 거꾸로 뒤집어 놓은 것처럼, 루트(뿌리)가 위에 있고 아래로 내려갈수록 자식 노드가 붙습니다. 자식이 부모가 될 수 없기 때문에 **사이클을 허용하지 않으며**, 항상 **부모-자식(상화) 관계**가 성립합니다. 대표적인 DBMS로는 IMS가 있습니다.

## 📝 핵심 정리

이 문서에서 배운 내용을 한눈에 정리합니다.

- **데이터 모델링 3단계**: 개념적 모델링(→ER 다이어그램) → 데이터 모델링(→테이블) → 데이터 구조화(→물리적 저장)
- **데이터베이스 설계 단계**: 요구사항 수집·분석 → 개념적 설계 → 논리적 설계 → 물리적 설계
- **데이터 모델의 구성 요소(SOC)**: 구조(Structure) + 연산(Operation) + 제약 조건(Constraint)
- **ER 모델**: 피터 첸이 제안한 개념적 데이터 모델로, 개체·관계·속성을 도형으로 표현
- **강한 개체**: 단독으로 존재 가능한 개체 (단일 사각형)
- **약한 개체**: 강한 개체 없이 존재 불가, 부분키를 가짐 (이중 사각형)
- **속성의 종류**: 단순/복합, 단일값/다중값, 저장/유도 속성
- **NULL의 의미**: 해당 사항 없음 / 값을 알 수 없음 / 존재 여부 불명확 (0이나 공백과 다름)
- **관계 타입**: 1:1, 1:N, N:1, N:M 네 가지
- **참여 제약**: 전체 참여(반드시 참여, 이중선) / 부분 참여(선택적 참여, 단일선)
- **연결함정**: 부채골 함정(위치 잘못 설계 → 모델 변환으로 해결) / 균열 함정(관계 누락 → 관계 추가로 해결)
- **확장 ER 모델**: 슈퍼클래스/서브클래스, Is-A 관계, D(분리)/O(중첩) 표기
- **일반화**: 하위 → 상위 (상향식, 공통점 추출) / **세분화**: 상위 → 하위 (하향식, 차이점 강조)
- **집단화**: 관계를 개체처럼 묶어 또 다른 관계를 맺게 하는 기법 (Is-a-part-of)
- **논리적 데이터 모델**: 관계(테이블) / 네트워크(그래프) / 계층(트리) 세 가지 종류
