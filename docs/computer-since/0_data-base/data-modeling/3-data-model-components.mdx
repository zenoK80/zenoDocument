## 🗺️ 데이터 모델이란 무엇인가?

우리가 살고 있는 현실 세계에는 수많은 것들이 존재합니다. 학생이 있고, 교수가 있고, 과목이 있고, 회사에는 직원이 있고, 부서가 있죠. 이러한 현실 세계의 복잡한 상황들을 컴퓨터 데이터베이스 안에 정확하게 담아내려면 어떻게 해야 할까요? 그냥 무작정 컴퓨터에 저장하면 될까요?

그렇지 않습니다. 마치 건축가가 집을 짓기 전에 설계 도면을 그리듯, 데이터베이스를 만들기 전에도 반드시 **설계 과정**이 필요합니다. 이 설계 과정에서 사용하는 도구와 방법론을 우리는 **데이터 모델(Data Model)** 이라고 부릅니다.

> **데이터 모델이란**, 현실 세계에 존재하는 유형·무형의 다양한 환경들을 실제 데이터베이스로 구현하기 위해 **모델링(설계)해 나가는 과정과 그 결과물**을 의미합니다.

### 🔍 헷갈리기 쉬운 용어들 정리

데이터 모델을 처음 공부하면 비슷한 용어들이 많이 나와서 혼란스럽습니다. 실제로 강의에서도 이 용어들의 차이를 명확히 구분하는 것이 중요하다고 강조합니다. 아래 표로 한눈에 정리해 보겠습니다.

| 용어 | 의미 | 결과물 | 비유 |
|---|---|---|---|
| **개념적 모델링** | 현실 세계를 추상적 개념으로 표현하는 과정 | 개념적 구조 (ER 다이어그램) | 건축 설계 도면 초안 그리기 |
| **데이터 모델링** | 개념적 구조를 논리적 구조로 변환하는 과정 | 논리적 구조 (테이블) | 설계 도면을 시공 도면으로 바꾸기 |
| **데이터 구조화** | 논리적 구조를 실제 컴퓨터가 접근할 수 있는 물리적 구조로 변환하는 과정 | 물리적 구조 (디스크 저장 구조) | 실제 건물 시공하기 |

쉽게 말하면, 이 세 단계는 **현실 세계 → 그림(ER 다이어그램) → 테이블 → 컴퓨터 저장** 순서로 점점 구체화되는 과정입니다.

### 📐 데이터베이스 설계 단계 한눈에 보기

데이터베이스를 설계할 때는 크게 세 단계를 거칩니다. 이 단계들은 각각 어느 정도로 구체적인지, 그리고 어떤 환경에 종속되는지에 따라 구분됩니다.

**1단계 - 개념적 설계(Conceptual Design)**
사용자의 요구사항을 수집하고 분석한 뒤, ER 다이어그램을 만들어내는 단계입니다. 이 단계에서는 어떤 DBMS(데이터베이스 관리 시스템)를 쓸지, 어떤 하드웨어를 쓸지 전혀 신경 쓰지 않아도 됩니다. **DBMS 독립적, 하드웨어 독립적**입니다.

**2단계 - 논리적 설계(Logical Design)**
개념적 설계의 결과물(ER 다이어그램)을 관계형, 계층형, 네트워크형 등의 **논리적 구조(테이블 등)** 로 변환하는 단계입니다. 이 단계에서는 어떤 DBMS를 쓸지 결정해야 하므로 **DBMS 종속적, 하드웨어 독립적**입니다.

**3단계 - 물리적 설계(Physical Design)**
실제 디스크의 물리적인 구조로 변환하는 단계입니다. 어떤 DBMS를 쓰는지, 어떤 하드웨어를 쓰는지 모두 관련이 있으므로 **DBMS 종속적, 하드웨어 종속적**입니다.

## 🧱 데이터 모델의 3가지 구성 요소: SOC

이제 본격적으로 이번 강의의 핵심 주제로 들어가겠습니다. **데이터 모델은 딱 3가지 요소로 구성**됩니다. 영어 첫 글자를 따서 **SOC**라고 부릅니다.

> **S** = Structure (구조)
> **O** = Operation (연산)
> **C** = Constraint (제약 조건)

마치 게임을 설계할 때 "어떤 캐릭터와 맵이 있는가(구조)", "어떤 행동을 할 수 있는가(연산)", "어떤 규칙을 지켜야 하는가(제약 조건)"를 정하는 것과 비슷합니다. 이 세 가지가 갖춰져야 비로소 완전한 데이터 모델이 됩니다.

### 📦 S - 구조(Structure): 데이터베이스의 뼈대

**구조(Structure)** 란, 데이터베이스에서 표현해야 할 **대상(개체)의 타입**과 **개체들 간의 관계**를 명세한 것입니다. 쉽게 말하면, "우리 데이터베이스 안에 어떤 종류의 데이터가 있고, 그것들이 어떤 모습을 하고 있는가?"를 정의한 것입니다.

예를 들어, 학교 데이터베이스를 만든다고 생각해 봅시다.
- **학생**이라는 개체는 학번, 이름, 주소, 학년이라는 속성을 가집니다.
- **교수**라는 개체는 교수번호, 이름, 전공, 학과라는 속성을 가집니다.
- **학생과 교수** 사이에는 "지도" 관계가 있습니다.

이렇게 개체의 종류, 각 개체가 가진 속성, 개체들 간의 관계를 모두 정의한 것이 바로 **구조**입니다. 마치 회사 조직도처럼, 누가 있고 서로 어떤 관계인지를 한눈에 보여주는 설계도라고 생각하면 됩니다.

### ⚙️ O - 연산(Operation): 데이터베이스에서 할 수 있는 행동

**연산(Operation)** 이란, 실제 데이터(인스턴스, 즉 개체의 실제 값)에 대해 수행할 수 있는 작업들을 의미합니다. 여기서 **인스턴스(instance)** 란 실제로 저장된 값을 말합니다. 예를 들어, "학생"이라는 개체 타입에서 "홍길동, 학번 2024001, 3학년"은 하나의 인스턴스(실제 값)입니다.

연산에는 크게 4가지가 있습니다.

- **삽입(Insert)**: 새로운 데이터를 데이터베이스에 집어넣는 것. (예: 신입생 정보를 학생 테이블에 추가)
- **삭제(Delete)**: 기존 데이터를 데이터베이스에서 지우는 것. (예: 졸업한 학생 정보를 삭제)
- **갱신(Update)**: 기존 데이터를 수정하는 것. (예: 학생 주소가 바뀌어 정보를 업데이트)
- **검색(Select/Query)**: 원하는 데이터를 조회하는 것. (예: 3학년 학생 목록을 조회)

이 네 가지 연산은 데이터베이스 분야에서 워낙 중요해서 **CRUD(Create, Read, Update, Delete)** 라는 약어로도 불립니다.

### 🔒 C - 제약 조건(Constraint): 데이터베이스의 규칙

**제약 조건(Constraint)** 이란, 개체의 구조에 연산이 이루어지는 과정에서 **반드시 지켜야 하는 규칙들**입니다. 마치 게임의 규칙처럼, 이 규칙을 어기는 데이터는 데이터베이스에 들어올 수 없습니다.

대표적인 제약 조건에는 다음과 같은 것들이 있습니다.

- **개체 무결성(Entity Integrity)**: 기본키(Primary Key)는 절대 NULL(빈 값)이 될 수 없고, 중복된 값이 있으면 안 됩니다. 학번이 없는 학생은 존재할 수 없는 것과 같습니다.
- **참조 무결성(Referential Integrity)**: 외래키(Foreign Key)가 참조하는 값은 실제로 존재해야 합니다. 존재하지 않는 교수번호를 지도교수로 등록할 수 없는 것과 같습니다.
- **도메인 무결성(Domain Integrity)**: 각 속성의 값은 정해진 범위(도메인) 안에 있어야 합니다. 학점이 0~4.5 사이여야 한다는 규칙이 도메인 무결성의 예입니다.

:::info SOC 요약
- **S (Structure, 구조)**: 데이터베이스에 어떤 개체와 관계가 있는지 정의 → "무엇이 있는가?"
- **O (Operation, 연산)**: 데이터에 어떤 작업을 할 수 있는지 정의 → "무엇을 할 수 있는가?"
- **C (Constraint, 제약 조건)**: 데이터가 지켜야 할 규칙 정의 → "어떤 규칙이 있는가?"
:::

## 🗂️ ER 모델과 ER 다이어그램의 핵심 요소

### 📌 ER 모델이란?

**ER 모델(Entity-Relationship Model)** 은 데이터 모델 중에서 가장 보편적으로 많이 사용되는 개념적 데이터 모델입니다. **Entity(개체)** 와 **Relationship(관계)** 의 앞 글자를 따서 ER 모델이라고 부릅니다. 1976년 피터 첸(Peter Chen)이라는 학자가 제안한 방법으로, 개체와 개체 간의 관계를 **그래프 형태(다이어그램)** 로 표현합니다.

ER 모델의 가장 큰 장점은 **DBMS와 하드웨어에 독립적**이라는 점입니다. 즉, 어떤 데이터베이스 시스템을 쓸지, 어떤 컴퓨터를 쓸지 결정하기 전에 먼저 데이터의 구조를 설계할 수 있습니다.

### 🖊️ ER 다이어그램의 기호 총정리

ER 다이어그램에서는 각 요소를 특정 도형으로 표현합니다. 아래 표로 한눈에 정리해 보겠습니다.

| 기호 | 모양 | 의미 |
|---|---|---|
| **사각형** (□) | 직사각형 | 개체(Entity) |
| **이중 사각형** (□□) | 테두리가 두 개인 직사각형 | 약한 개체(Weak Entity) |
| **다이아몬드** (◇) | 마름모 | 관계(Relationship) |
| **이중 다이아몬드** (◇◇) | 테두리가 두 개인 마름모 | 식별 관계(Identifying Relationship) |
| **타원** (○) | 원/타원 | 속성(Attribute) |
| **이중 타원** (○○) | 테두리가 두 개인 타원 | 다중값 속성(Multi-valued Attribute) |
| **점선 타원** (⊙) | 점선 타원 | 유도 속성(Derived Attribute) |
| **계층형 타원** | 타원 안에 타원 | 복합 속성(Composite Attribute) |
| **밑줄 속성** | 속성 이름에 실선 밑줄 | 기본키(Primary Key) |
| **점선 밑줄 속성** | 속성 이름에 점선 밑줄 | 부분키(Partial Key) |
| **단일 선** (─) | 개체-관계 연결 선 하나 | 부분 참여(Partial Participation) |
| **이중 선** (═) | 개체-관계 연결 선 두 개 | 전체 참여(Total Participation) |

### 🏫 ER 다이어그램 예시: 학교 데이터베이스

실제 예시를 통해 ER 다이어그램이 어떻게 구성되는지 살펴보겠습니다.

학교 데이터베이스에는 다음과 같은 개체와 관계가 있습니다.

- **교수** 개체: 교수번호(기본키), 이름, 전공, 학과
- **학생** 개체: 학번(기본키), 이름, 주소, 학년
- **과목** 개체: 과목번호(기본키), 과목명
- **지도** 관계: 교수와 학생 사이 (1:N - 한 교수가 여러 학생을 지도)
- **강의** 관계: 교수와 과목 사이 (1:N - 한 교수가 여러 과목을 강의)
- **등록** 관계: 학생과 과목 사이 (N:M - 여러 학생이 여러 과목을 수강)

여기서 흥미로운 점이 있습니다. **등록** 관계에는 **성적**이라는 속성이 붙어 있습니다. 이것을 **설명 속성(Descriptive Attribute)** 이라고 합니다.

> 💡 **설명 속성이 왜 필요한가?**
> 성적은 "학생"에만 속한다고 볼 수도 없고, "과목"에만 속한다고 볼 수도 없습니다. 성적은 **특정 학생이 특정 과목을 수강했을 때만** 생기는 값이기 때문입니다. 따라서 "등록"이라는 관계 자체의 속성으로 관리하는 것이 가장 자연스럽습니다. 이런 관계가 가지는 속성을 **설명 속성**이라고 합니다.

## 🔑 개체의 종류: 강한 개체와 약한 개체

### 💪 강한 개체(Strong Entity)

**강한 개체**란, 자기 자신의 속성만으로도 고유하게 식별할 수 있는 개체입니다. 예를 들어, "직원" 개체는 직원번호라는 속성만으로 각 직원을 구별할 수 있습니다. 직원번호 1001번은 전 회사에서 단 한 명뿐이니까요. 이런 개체를 강한 개체라고 하며, ER 다이어그램에서 **단일 사각형**으로 표현합니다.

### 🤝 약한 개체(Weak Entity): 혼자서는 살 수 없는 개체

**약한 개체**란, 자기 자신의 속성만으로는 고유하게 식별할 수 없어서, **반드시 다른 강한 개체에 의존해야 하는 개체**입니다. ER 다이어그램에서 **이중 사각형**으로 표현합니다.

가장 쉬운 예시가 바로 **"부양가족"** 입니다.

회사에는 직원 테이블과 부양가족 테이블이 따로 있을 수 있습니다. 직원 테이블은 단독으로 존재할 수 있습니다. 하지만 부양가족은 어떨까요?

**"우리 회사의 부양가족"** 이라는 말 자체가 이상합니다. 부양가족은 반드시 **어떤 직원의** 부양가족이어야 합니다. 직원이 없는 부양가족은 회사 데이터베이스에 존재할 수 없습니다.

또 다른 문제가 있습니다. 삼성전자처럼 큰 회사에는 부양가족이 수만 명이 될 수 있는데, 그 중에 "김민준"이라는 이름을 가진 부양가족이 여러 명일 수 있습니다. 이름만으로는 누구의 부양가족인지 구별할 수가 없습니다. 그래서 부양가족을 식별하려면 반드시 **"직원번호 + 부양가족 이름"** 처럼 강한 개체의 키를 함께 사용해야 합니다.

:::warning 약한 개체의 핵심 특징
1. **단독으로 존재할 수 없다**: 강한 개체가 삭제되면 약한 개체도 함께 삭제됩니다.
2. **부분키(Partial Key)를 가진다**: 자신만으로는 완전히 식별할 수 없기 때문에 부분키라고 합니다. ER 다이어그램에서 **점선 밑줄**로 표현합니다.
3. **식별 관계(Identifying Relationship)**: 약한 개체와 강한 개체를 연결하는 관계를 식별 관계라고 하며, **이중 다이아몬드**로 표현합니다.
4. **항상 전체 참여(Total Participation)**: 약한 개체는 반드시 강한 개체와 관계를 맺어야 합니다.
5. **1:N 관계**: 강한 개체 1개에 약한 개체 여러 개가 연결됩니다.
:::

```
[직원]══════《부양》══════[부양가족]
직원번호(PK)    식별관계     이름(부분키, 점선밑줄)
이름                       관계
...

// 직원번호 1001 홍길동 → 부양가족: 홍아들(이름), 홍딸(이름)
// 직원번호 1002 김철수 → 부양가족: 김아들(이름)
// → "홍아들"과 "김아들"을 구별하려면 직원번호(1001, 1002)가 반드시 필요!
```

## 🏷️ 속성(Attribute)의 다양한 종류

속성은 개체나 관계가 가지고 있는 특성(데이터)입니다. 속성에도 여러 종류가 있으며, ER 다이어그램에서 각각 다른 모양으로 표현됩니다.

### 1️⃣ 단순 속성 vs 복합 속성

**단순 속성(Simple Attribute)** 은 더 이상 쪼갤 수 없는 속성입니다. 예를 들어, "이름"은 더 이상 세분화하기 어렵습니다(물론 성과 이름으로 나눌 수는 있지만, 단일한 값으로 사용한다면 단순 속성입니다). ER 다이어그램에서 **일반 타원**으로 표현합니다.

**복합 속성(Composite Attribute)** 은 하나의 속성을 더 세분화할 수 있는 속성입니다. 예를 들어, "주소"라는 속성은 "시", "구", "동", "상세주소"로 나눌 수 있습니다. ER 다이어그램에서 **계층적 타원 구조**로 표현합니다. 마치 폴더 안에 하위 폴더가 있는 것처럼 계층 구조를 이룹니다.

### 2️⃣ 단일값 속성 vs 다중값 속성

**단일값 속성(Single-valued Attribute)** 은 하나의 개체에 대해 오직 하나의 값만 가질 수 있는 속성입니다. 예를 들어, 사람의 "나이"는 동시에 두 개일 수 없습니다. **일반 타원**으로 표현합니다.

**다중값 속성(Multi-valued Attribute)** 은 하나의 개체에 대해 여러 개의 값을 가질 수 있는 속성입니다. 예를 들어, "취미"는 독서, 운동, 요리 등 여러 개를 동시에 가질 수 있습니다. 학위도 학사, 석사, 박사를 동시에 가질 수 있죠. **이중 타원**으로 표현합니다.

### 3️⃣ 저장 속성 vs 유도 속성

**저장 속성(Stored Attribute)** 은 실제로 디스크(하드디스크)에 저장되는 속성입니다. 예를 들어, 주민번호는 데이터베이스에 실제로 저장됩니다. **일반 타원**으로 표현합니다.

**유도 속성(Derived Attribute)** 은 다른 속성으로부터 계산해서 얻을 수 있는 속성으로, **실제로는 데이터베이스에 저장되지 않습니다**. 예를 들어, "나이"는 주민번호(또는 생년월일)로부터 계산해낼 수 있습니다. 굳이 나이를 따로 저장하지 않아도 되죠. **점선 타원**으로 표현합니다.

마치 계산기처럼, 주민번호라는 재료를 넣으면 나이라는 결과가 나오는 것과 같습니다. 계산기 자체를 저장하는 게 아니라, 재료(주민번호)만 저장해두고 필요할 때마다 계산하는 방식입니다.

```sql
-- 저장 속성 예시: 주민번호는 테이블에 직접 저장
-- 유도 속성 예시: 나이는 주민번호로부터 계산 (저장 X)

-- 학생 테이블 예시
CREATE TABLE 학생 (
    학번 VARCHAR(10) PRIMARY KEY,  -- 기본키, 저장 속성
    이름 VARCHAR(50) NOT NULL,      -- 저장 속성
    주민번호 VARCHAR(14),           -- 저장 속성 (나이를 계산하는 데 사용)
    -- 나이는 별도로 저장하지 않음! 주민번호로 계산 가능 (유도 속성)
    학년 INT                        -- 저장 속성
);

-- 나이가 필요할 때는 이렇게 계산해서 가져옴 (유도 속성의 개념)
SELECT 이름,
       YEAR(NOW()) - SUBSTRING(주민번호, 1, 2) AS 나이  -- 주민번호에서 계산
FROM 학생;
```

## 🔗 NULL의 개념: 0이 아닌 '해당 없음'

데이터베이스를 공부하면서 반드시 제대로 이해해야 하는 개념이 바로 **NULL**입니다. 많은 초보자들이 NULL을 0이나 빈 문자열("")과 혼동하는데, 이것은 완전히 다른 개념입니다.

> **NULL은 0도 아니고, 빈 문자열도 아닙니다. NULL은 그 자체로 독립적인 의미를 가집니다.**

NULL이 사용되는 상황은 크게 세 가지입니다.

**상황 1: 해당 사항 없음**
학생 테이블에 "병역" 컬럼이 있다고 합시다. 남학생은 군필, 미필, 면제 등의 값이 들어갈 수 있습니다. 하지만 여학생은 군필도 아니고, 미필도 아니고, 면제도 아닙니다. 여학생에게는 병역이 **해당 사항 없음**입니다. 이럴 때 NULL을 사용합니다.

또 다른 예로, 아파트 주소에는 동호수가 있지만, 단독주택에는 동호수가 없습니다. 단독주택에 사는 사람의 동호수 컬럼은 NULL이 됩니다.

**상황 2: 값은 있지만 아직 알 수 없음**
어제 이사를 가서 새 주소를 아직 정확히 모른다고 합시다. 주소가 없는 것이 아니라, 주소는 분명히 있지만 그 값이 무엇인지 아직 모르는 상황입니다. 이럴 때도 NULL을 사용합니다.

**상황 3: 값이 존재하는지조차 알 수 없음**
"집 전화번호" 컬럼이 있는데, 해당 사람이 집 전화기를 가지고 있는지 없는지조차 모를 때입니다. 있는데 모르는 게 아니라, 아예 있는지 없는지조차 불분명한 상황입니다. 이럴 때도 NULL을 사용합니다.

| 상황 | 의미 | NULL 사용 여부 |
|---|---|---|
| 여학생의 병역 | 해당 사항 자체가 없음 | ✅ NULL |
| 단독주택의 동호수 | 해당 개념 자체가 없음 | ✅ NULL |
| 이사 후 새 주소 미확인 | 값은 있지만 아직 모름 | ✅ NULL |
| 집 전화 유무 불분명 | 있는지 없는지 불분명 | ✅ NULL |
| 시험 점수 0점 | 명확히 0점이라는 값 존재 | ❌ 0을 사용 |
| 이름 미입력 | 빈 문자열로 처리 가능 | ❌ 경우에 따라 다름 |

:::danger NULL과 0, 빈 문자열을 절대 혼동하지 마세요!
- `NULL`은 "알 수 없음" 또는 "해당 없음"
- `0`은 명확한 숫자 값 **영(zero)**
- `""`(빈 문자열)은 내용이 없는 문자열

`NULL = 0`이나 `NULL = ""`은 **거짓(FALSE)** 입니다. NULL과의 비교는 항상 `IS NULL` 또는 `IS NOT NULL`을 사용해야 합니다.

```sql
-- ❌ 잘못된 예시: NULL을 = 으로 비교하면 결과가 항상 알 수 없음(UNKNOWN)
SELECT * FROM 학생 WHERE 병역 = NULL;  -- 결과: 아무것도 안 나옴!

-- ✅ 올바른 예시: IS NULL로 비교해야 함
SELECT * FROM 학생 WHERE 병역 IS NULL;  -- 결과: 병역이 NULL인 모든 학생
```
:::

## 🔄 관계(Relationship)의 종류와 특성

### 📊 관계의 카디널리티(대응수): 1:1, 1:N, N:M

개체와 개체 사이의 관계에서 **몇 대 몇의 관계**인지를 나타내는 것을 **카디널리티(Cardinality, 대응수)** 라고 합니다. 4가지 유형이 있습니다.

**1:1 (일대일) 관계**
한 개체 인스턴스가 다른 개체 인스턴스 딱 하나와만 관계를 맺는 경우입니다. 예를 들어, 우리나라에서 한 남성과 한 여성이 법적으로 결혼하는 관계를 생각해 보세요. 한 남성은 오직 한 명의 여성과 결혼하고, 한 여성도 오직 한 명의 남성과 결혼합니다.

**1:N (일대다) 관계**
한 개체 인스턴스가 여러 개의 다른 개체 인스턴스와 관계를 맺는 경우입니다. 예를 들어, 한 명의 교수가 여러 명의 학생을 지도합니다. 교수 입장에서는 학생이 여러 명이지만, 학생 입장에서는 지도교수가 한 명입니다.

**N:1 (다대일) 관계**
여러 개체 인스턴스가 하나의 개체 인스턴스와 관계를 맺는 경우입니다. 1:N의 반대 방향으로 바라보는 것입니다. 여러 명의 학생이 한 명의 교수에게 지도를 받는 것처럼요. 관계는 **방향성**이 있기 때문에 같은 관계도 어느 방향에서 보느냐에 따라 1:N이 되거나 N:1이 됩니다.

**N:M (다대다) 관계**
여러 개체 인스턴스가 여러 개의 다른 개체 인스턴스와 관계를 맺는 경우입니다. 예를 들어, 한 명의 학생은 여러 과목을 수강할 수 있고, 한 과목도 여러 명의 학생이 수강할 수 있습니다. 이것이 N:M(다대다) 관계입니다.

### 🎯 전체 참여 vs 부분 참여

관계에 참여하는 방식에도 두 가지가 있습니다. ER 다이어그램에서 **선의 개수**로 구분합니다.

**부분 참여(Partial Participation) = 선택적 참여**
개체가 관계에 참여할 수도 있고, 참여하지 않을 수도 있습니다. ER 다이어그램에서 **단일 선(─)** 으로 표현합니다. 예를 들어, 새로 신설된 부서는 아직 소속 사원이 없을 수도 있습니다. 부서는 사원 없이도 존재할 수 있으므로 부서 측은 **부분 참여**입니다.

**전체 참여(Total Participation) = 의무적 참여**
개체가 반드시 관계에 참여해야 합니다. ER 다이어그램에서 **이중 선(═)** 으로 표현합니다. 예를 들어, 사원은 반드시 어떤 부서에 소속되어야 합니다. 어떤 부서에도 속하지 않는 사원은 있을 수 없습니다. 사원 측은 **전체 참여**입니다.

:::tip 실생활 비유로 이해하기
- **부분 참여**: 동아리 회원이 꼭 모든 행사에 참여할 의무는 없는 것 → 참여해도 되고 안 해도 됨
- **전체 참여**: 학교 학생은 반드시 어딘가의 학과에 소속되어야 하는 것 → 반드시 참여해야 함
:::

## ⚠️ 연결 함정(Connection Trap): 관계 설계 시 주의할 오류

### 🕳️ 연결 함정이란?

**연결 함정(Connection Trap)** 이란, ER 다이어그램에서 개체와 관계 타입을 잘못 설계하여 **기존에 알 수 있던 정보가 누락되거나 추적 불가능해지는 현상**입니다. 마치 지도에서 특정 도로를 잘못 그려서 목적지를 찾지 못하게 되는 것과 같습니다.

연결 함정에는 크게 두 가지 종류가 있습니다.

### 🌊 부채골 함정(Fan Trap)

**부채골 함정**이란, 관계 모델을 잘못 설계하여 **정보를 추적할 수 없게 되는 문제**입니다. 모델 자체는 논리적으로 어색하지 않아 보이지만, 실제로 필요한 정보를 얻을 수 없는 상황이 발생합니다.

예시로 살펴보겠습니다.

- 교수가 대학에 재직하고 있습니다.
- 대학에는 여러 학과가 소속되어 있습니다.

만약 이것을 **교수 → 재직 → 대학 → 소속 → 학과** 순서로 1:N, 1:N으로 연결한다면, 다음 정보는 알 수 있습니다.
- 어떤 교수가 어느 대학에 재직하는지 ✅
- 그 대학에 어떤 학과들이 있는지 ✅

그러나 정작 중요한 **"교수가 어느 학과에 소속되어 있는지"** 는 알 수 없게 됩니다. 교수가 대학에 재직하는 건 알고, 그 대학에 학과들이 있는 건 알지만, 그 교수가 구체적으로 어떤 학과 소속인지는 모르는 것이죠.

이 문제의 해결 방법은 **관계 모델의 위치를 바꾸는 것**입니다. **교수 → 소속 → 학과 → 속함 → 대학** 으로 순서를 바꾸면, 교수가 어느 학과 소속인지를 직접 알 수 있게 됩니다.

### 💥 균열 함정(Chasm Trap)

**균열 함정**이란, 필요한 **관계 자체가 누락**되어 정보를 추적할 수 없게 되는 문제입니다. 부채골 함정은 모델의 위치를 바꿔서 해결할 수 있었지만, 균열 함정은 **새로운 관계를 추가**해야만 해결할 수 있습니다.

예시로 살펴보겠습니다.

- 교수가 학과에 재직하고 있습니다.
- 학생들이 교수로부터 지도를 받습니다.

이 모델로는 다음 정보를 알 수 있습니다.
- 어떤 교수가 어느 학과에 재직하는지 ✅
- 어떤 학생이 어떤 교수에게 지도받는지 ✅

그러나 **"학생이 어느 학과 소속인지"** 는 알 수 없습니다. 교수를 통해서 간접적으로 유추해보려 해도, 한 학과에 여러 교수가 있고 학생이 그 중 어느 교수에게 지도받는지는 알지만 학생이 속한 학과를 직접 알 방법이 없습니다.

이 문제를 해결하려면 **학생과 학과 사이에 직접적인 "소속" 관계를 새로 추가**해야 합니다.

| | 부채골 함정 | 균열 함정 |
|---|---|---|
| **원인** | 관계 모델의 위치가 잘못됨 | 필요한 관계가 누락됨 |
| **해결책** | 모델의 위치를 변경 | 새로운 관계를 추가 |
| **특징** | 구조는 있으나 경로가 잘못됨 | 연결 자체가 없음 |

## 🌟 확장 ER 모델(Extended ER Model): 더 복잡한 세상을 표현하다

### 🆕 확장 ER 모델이 필요한 이유

기존의 ER 모델은 개체, 관계, 속성으로 현실 세계를 표현합니다. 하지만 현대에는 CAD 데이터, 지리정보 데이터, 멀티미디어 데이터 등 훨씬 복잡한 데이터들이 등장했습니다. 또한 **객체 지향 프로그래밍**에서 사용하는 상속, 재사용 등의 개념도 표현해야 할 필요가 생겼습니다. 기존 ER 모델만으로는 이런 복잡한 상황을 표현하기에 한계가 있어서 등장한 것이 **확장 ER 모델(Extended ER Model, EER)**입니다.

확장 ER 모델에는 **전문화(Specialization)**, **일반화(Generalization)**, **집단화(Aggregation)**, **연관화(Association)** 등의 개념이 추가됩니다.

### 🏗️ 슈퍼클래스와 서브클래스

확장 ER 모델에서는 개체를 **슈퍼클래스(Superclass)** 와 **서브클래스(Subclass)** 로 나눌 수 있습니다.

- **슈퍼클래스**: 여러 하위 개체들의 **공통 속성**을 묶어놓은 상위 개체입니다.
- **서브클래스**: 슈퍼클래스에서 파생된 하위 개체로, **고유한 속성**을 추가로 가집니다.
- **Is-a 관계**: 슈퍼클래스와 서브클래스 사이의 관계를 **Is-a 관계**라고 합니다. "학부생은 학생이다(학부생 Is-a 학생)"처럼요.

예시를 봅시다. "학생"이라는 슈퍼클래스가 있고, "학부생"과 "대학원생"이라는 서브클래스가 있습니다.

- 슈퍼클래스 **학생**: 학번(기본키), 이름, 학과 → 학부생과 대학원생 모두에게 공통으로 필요한 속성
- 서브클래스 **학부생**: 학년 → 학부생에게만 있는 속성
- 서브클래스 **대학원생**: 학위과정 → 대학원생에게만 있는 속성

공통 속성은 위(슈퍼클래스)로 올리고, 개별 속성은 아래(서브클래스)로 내리는 구조입니다. 마치 회사에서 직원들의 공통 정보는 "직원" 테이블에, 개발자만의 정보는 "개발자" 테이블에 따로 관리하는 것처럼요.

### 🔀 Disjoint(D) vs Overlap(O)

서브클래스들 사이의 관계에는 두 가지 옵션이 있습니다.

**D (Disjoint, 분리)**: 하나의 개체 인스턴스가 서브클래스 중 **오직 하나**에만 속할 수 있습니다. 학부생이면서 동시에 대학원생은 될 수 없는 것처럼요. ER 다이어그램에서 서브클래스 연결 부분에 **"D"** 로 표시합니다.

**O (Overlap, 중첩)**: 하나의 개체 인스턴스가 서브클래스 중 **둘 이상**에 동시에 속할 수 있습니다. 특수한 경우(예: 특정 학교에서 학부와 대학원을 동시에 다닐 수 있는 경우)에 사용합니다. ER 다이어그램에서 **"O"** 로 표시합니다.

### ⬆️ 일반화(Generalization) vs ⬇️ 세분화(Specialization)

**일반화(Generalization)**: 여러 개체들의 차이점은 무시하고 **공통점을 찾아서 상위 개념(슈퍼클래스)을 만드는 과정**입니다. "학부생"과 "대학원생"의 공통점인 학번, 이름, 학과를 찾아서 "학생"이라는 슈퍼클래스를 만드는 것이죠. **아래에서 위로(상향식)** 방향입니다. 추상화, 개념화라고도 부릅니다.

**세분화(Specialization)**: 하나의 개체를 더 구체적인 **하위 개념(서브클래스)으로 나누는 과정**입니다. "학생"을 "학부생"과 "대학원생"으로 나누는 것이죠. **위에서 아래로(하향식)** 방향입니다. 전문화, 구체화라고도 부릅니다. 세분화의 대상은 항상 슈퍼클래스입니다.

:::note 일반화와 세분화: 자동차 예시
- **자동차** (슈퍼클래스): 등록번호, 차대번호, 가격 (공통 속성)
  - **승용차** (서브클래스): 최고속도, 승객수 (승용차만의 속성)
  - **트럭** (서브클래스): 적재용량 (트럭만의 속성)

승용차와 트럭의 공통 속성을 찾아서 "자동차"를 만드는 것 → **일반화** (↑)
"자동차"를 "승용차"와 "트럭"으로 나누는 것 → **세분화** (↓)
승용차이면서 트럭일 수는 없으니 → **D (Disjoint)**
:::

### 🧩 집단화(Aggregation)

**집단화(Aggregation)** 란, 관계와 관계를 직접 연결할 수 없는 한계를 극복하기 위해, **여러 개체와 그들의 관계를 하나의 더 큰 개체로 묶는 것**입니다. Is-a-part-of 관계라고도 합니다.

예를 들어, "종업원"과 "프로젝트" 사이에 "작업" 관계가 있습니다. 그리고 이 "작업"이라는 관계에 "기계"를 연결하고 싶은데, 관계와 관계를 직접 연결할 수는 없습니다.

이때 집단화를 사용합니다. **"종업원 + 작업 + 프로젝트"** 전체를 **"작업들"** 이라는 하나의 새로운 집단 개체로 묶어버립니다. 그러면 이 집단 개체는 개체로 취급되므로, "기계"와 "사용"이라는 관계로 연결할 수 있게 됩니다.

### 🤝 연관화(Association)

**연관화(Association)** 는 개체와 개체 사이의 연관성 여부를 판단하고 표현하는 것으로, Is-associated 관계라고 합니다. 개체들이 서로 연관이 있는지 없는지, 그 연관의 성격이 무엇인지를 명확히 하는 데 사용됩니다.

## 📋 논리적 데이터 모델: 관계형, 네트워크형, 계층형

### 🗃️ 관계 데이터 모델(Relational Data Model)

**관계 데이터 모델**은 데이터를 **테이블(표)** 형태로 표현하는 논리적 모델입니다. 우리가 일상적으로 가장 많이 접하는 형태로, 행(Row)과 열(Column)로 구성된 테이블로 모든 데이터를 표현합니다.

네트워크 모델이나 계층 모델에 비해 훨씬 **추상적**입니다. 학생과 교수의 관계를 그림이 아닌 테이블로 표현하니, 눈으로 직관적으로 관계를 파악하기는 더 어렵습니다. 하지만 표준화되어 있고 SQL이라는 강력한 언어로 조작할 수 있어서 현재 가장 널리 사용됩니다.

대표적인 DBMS: **Oracle, MS SQL Server, MySQL, PostgreSQL, Sybase, DB2, Informix**

### 🕸️ 네트워크 데이터 모델(Network Data Model)

**네트워크 데이터 모델**은 데이터를 **그래프(Graph)** 형태로 표현합니다. 오너(Owner)와 멤버(Member)의 관계를 가지며, **사이클(Cycle)을 허용**합니다(A→B→C→A처럼 순환 관계가 가능). 관계가 시각적으로 잘 드러나지만 복잡한 데이터를 다루기 어렵습니다.

대표적인 DBMS: **CODASYL DBTG, TOTAL**

### 🌳 계층 데이터 모델(Hierarchical Data Model)

**계층 데이터 모델**은 데이터를 **트리(Tree)** 구조로 표현합니다. 부모-자식 관계(상하 관계)로 데이터를 표현하며, **사이클을 허용하지 않습니다**(자식이 부모가 될 수 없음). 단순하고 직관적이지만 다대다(N:M) 관계를 표현하기 어렵다는 단점이 있습니다.

대표적인 DBMS: **IMS(IBM Information Management System)**

| 모델 | 표현 방식 | 사이클 | 대표 DBMS |
|---|---|---|---|
| **관계형** | 테이블(행·열) | - | Oracle, MySQL, MS SQL |
| **네트워크형** | 그래프 | 허용 | CODASYL DBTG, TOTAL |
| **계층형** | 트리 | 불허용 | IMS |

## 📝 핵심 정리

이번 문서에서 배운 내용을 핵심만 추려서 정리합니다.

- **데이터 모델**은 현실 세계를 데이터베이스로 구현하기 위한 설계 방법론이며, **개념적 모델링 → 데이터 모델링 → 데이터 구조화**의 순서로 진행된다.
- **데이터 모델의 3요소(SOC)**: **S**tructure(구조, 개체와 관계의 정의) + **O**peration(연산, 삽입·삭제·갱신·검색) + **C**onstraint(제약 조건, 무결성 규칙)
- **ER 다이어그램**은 개체(사각형), 관계(다이아몬드), 속성(타원)으로 현실 세계를 그래프 형태로 표현하는 개념적 데이터 모델이다.
- **강한 개체**는 자신의 속성만으로 식별 가능하고, **약한 개체**는 강한 개체에 의존해야만 존재 및 식별이 가능하다.
- 약한 개체의 식별자는 **부분키(Partial Key, 점선 밑줄)**, 연결 관계는 **식별 관계(이중 다이아몬드)** 로 표현한다.
- **속성의 종류**: 단순/복합, 단일값/다중값, 저장/유도 속성으로 구분되며 ER 다이어그램에서 각기 다른 기호로 표현된다.
- **NULL**은 0도 빈 문자열도 아니며, "해당 없음", "값을 알 수 없음", "존재 여부 불분명"의 세 가지 의미로 사용된다.
- 관계의 **카디널리티(대응수)**: 1:1, 1:N, N:1, N:M의 4가지로 구분된다.
- **전체 참여(이중 선)**는 반드시 관계에 참여해야 함을 의미하고, **부분 참여(단일 선)**는 선택적 참여를 의미한다.
- **연결 함정**에는 모델 위치 오류인 **부채골 함정**(모델 위치 변경으로 해결)과 관계 누락인 **균열 함정**(관계 추가로 해결)이 있다.
- **확장 ER 모델**은 슈퍼클래스/서브클래스, 일반화(상향식), 세분화(하향식), 집단화, 연관화 등의 개념을 추가하여 복잡한 현실을 표현한다.
- **D(Disjoint)** 는 서브클래스 간 중복 불허, **O(Overlap)** 는 중복 허용을 의미한다.
- 논리적 데이터 모델에는 **관계형(테이블)**, **네트워크형(그래프)**, **계층형(트리)** 의 세 가지가 있다.
