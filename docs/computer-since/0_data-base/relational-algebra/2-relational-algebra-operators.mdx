# 🧮 관계대수 연산자 분류: 일반 집합 연산자와 순수 관계 연산자

데이터베이스를 공부하다 보면 반드시 마주치게 되는 개념이 있습니다. 바로 **관계대수(Relational Algebra)** 입니다. 처음 접하면 수학 용어처럼 느껴져서 막막할 수 있지만, 실제로는 우리가 중학교 때 배운 집합 개념과 데이터베이스의 기본 조작 방법을 합쳐 놓은 것에 불과합니다. 이 문서에서는 관계대수가 무엇인지부터 시작해서, 두 가지 핵심 연산자 분류인 **일반 집합 연산자**와 **순수 관계 연산자**를 아주 쉽고 자세하게 설명합니다.

---

## 🌱 관계대수란 무엇인가?

### 관계대수의 정의

**관계대수(Relational Algebra)** 란 한마디로 "**릴레이션(테이블)을 처리하기 위한 연산의 집합**"입니다. 여기서 릴레이션(Relation)이란 수학적인 용어로, 실제 데이터베이스에서는 우리가 흔히 부르는 **테이블(Table)** 과 같은 개념입니다. 행(Row)과 열(Column)로 구성된 표 형태의 데이터 구조를 말합니다.

마치 요리를 할 때 재료(릴레이션)를 가지고 다양한 조리 도구(연산자)를 써서 원하는 요리(결과)를 만들어 내듯이, 관계대수는 테이블이라는 재료를 여러 연산자로 조작해서 원하는 데이터를 추출해 내는 방법입니다.

> 💡 **핵심 포인트**: 관계대수에서 연산의 대상도 릴레이션(테이블)이고, 연산의 결과로 나오는 것도 릴레이션(테이블)입니다. 입력도 테이블, 출력도 테이블!

### 왜 관계대수를 배워야 하는가?

우리가 데이터베이스에서 사용하는 SQL 명령어들(SELECT, FROM, WHERE, JOIN 등)은 사실 이 관계대수라는 수학적 이론을 기반으로 만들어진 것입니다. 즉, 관계대수는 SQL의 이론적 토대입니다.

마치 건물을 지을 때 설계도면이 있어야 하듯이, SQL이라는 건물은 관계대수라는 설계도면 위에 세워진 것입니다. 설계도면을 이해하면 건물의 구조를 훨씬 잘 이해할 수 있듯이, 관계대수를 이해하면 SQL의 동작 원리를 훨씬 깊이 이해할 수 있습니다.

### 관계대수는 절차적(Procedural) 방법이다

관계대수는 **절차적인 방법**입니다. 이 말은, 원하는 정보가 무엇인지뿐만 아니라, 그 정보를 **어떤 순서와 방법으로 유도해 낼 것인지**도 함께 기술해야 한다는 의미입니다.

예를 들어, "성적이 80점 이상인 남학생의 이름을 가져와라"라는 요청이 있다면, 관계대수에서는 ① 학생 테이블에서 남학생을 먼저 찾고 → ② 그 중에서 성적이 80점 이상인 학생을 찾고 → ③ 이름 열만 추출하라는 구체적인 순서와 방법까지 명시해야 합니다.

## 📚 관계대수 연산자의 두 가지 분류

관계대수의 연산자는 크게 두 가지로 구분됩니다.

| 분류 | 연산자 종류 | 기호 | 설명 |
|------|-------------|------|------|
| **일반 집합 연산자** | 합집합 (Union) | ∪ | 두 릴레이션의 합 |
| **일반 집합 연산자** | 교집합 (Intersection) | ∩ | 두 릴레이션의 공통 부분 |
| **일반 집합 연산자** | 차집합 (Difference) | − | 한 릴레이션에만 있는 부분 |
| **일반 집합 연산자** | 카티션 프로덕트 (Cartesian Product) | × | 두 릴레이션의 모든 조합 |
| **순수 관계 연산자** | 셀렉트 (Select) | σ (시그마) | 조건에 맞는 행 추출 |
| **순수 관계 연산자** | 프로젝트 (Project) | π (파이) | 원하는 열 추출 |
| **순수 관계 연산자** | 조인 (Join) | ⋈ | 두 릴레이션 연결 |
| **순수 관계 연산자** | 디비전 (Division) | ÷ | 나누기 연산 |

:::info 일반 집합 연산자 vs 순수 관계 연산자
- **일반 집합 연산자**: 수학의 집합 이론에서 가져온 연산자입니다. 중학교, 고등학교 때 배운 합집합, 교집합, 차집합, 곱집합 개념이 그대로 적용됩니다.
- **순수 관계 연산자**: 데이터베이스에서만 사용하기 위해 새롭게 개발된 연산자입니다. 릴레이션(테이블)의 특성에 맞게 설계된 연산자들입니다.
:::

## 📦 일반 집합 연산자 (General Set Operators)

### 🔵 합집합 (Union) — ∪

**합집합**은 두 릴레이션 R과 S에 있는 모든 튜플(행)을 합쳐서 하나의 릴레이션으로 만드는 연산입니다. 마치 두 학급의 학생 명단을 하나로 합치는 것과 같습니다. 1반 명단과 2반 명단을 하나의 전체 명단으로 합치면 합집합이 됩니다.

#### 합집합의 조건: 합병 가능(Union Compatible)

합집합을 수행하려면 두 릴레이션이 **합병 가능(Union Compatible)** 해야 합니다. 합병 가능하다는 말은 두 가지 조건을 모두 만족해야 한다는 뜻입니다.

1. **차수(Degree)가 같아야 한다**: 차수란 속성(열, Column)의 개수를 말합니다. R이 속성 3개, S도 속성 3개여야 합집합이 가능합니다.
2. **대응되는 속성의 도메인(Domain)이 같아야 한다**: 도메인이란 속성이 가질 수 있는 값의 종류입니다. 예를 들어 R의 첫 번째 속성이 숫자 타입이면, S의 첫 번째 속성도 숫자 타입이어야 합니다. 이름은 달라도 됩니다.

:::note 속성 이름은 달라도 괜찮아요!
합집합을 할 때 R의 속성 이름이 `학번`이고, S의 속성 이름이 `수험번호`라도 서로 대응되는 위치에 있고, 데이터 타입(도메인)이 같다면 합집합이 가능합니다. 이름 자체는 합병 가능성의 조건이 아닙니다.
:::

#### 합집합의 결과 특성

- **결과의 차수(속성 수)**: R 또는 S의 차수와 동일합니다. (같으니까 어느 쪽이든 같음)
- **결과의 카디널리티(Cardinality, 튜플 수)**: R의 튜플 수 + S의 튜플 수입니다. 단, **중복된 튜플은 하나만 표시**합니다. 릴레이션은 튜플의 유일성이 보장되어야 하기 때문입니다.

#### 합집합 예시

아래 두 테이블 **학생1**과 **학생2**를 합집합 해봅시다.

**학생1 테이블** (차수: 3, 카디널리티: 4)

| 이름 | 과목 | 점수 |
|------|------|------|
| 오태오 | 자료구조 | 90 |
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |
| 김민준 | 운영체제 | 75 |

**학생2 테이블** (차수: 3, 카디널리티: 4)

| 이름 | 과목 | 점수 |
|------|------|------|
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |
| 박서연 | 알고리즘 | 85 |
| 최지우 | 데이터베이스 | 95 |

**학생1 ∪ 학생2 결과** (차수: 3, 카디널리티: 6 — 중복 2개 제거)

| 이름 | 과목 | 점수 |
|------|------|------|
| 오태오 | 자료구조 | 90 |
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |
| 김민준 | 운영체제 | 75 |
| 박서연 | 알고리즘 | 85 |
| 최지우 | 데이터베이스 | 95 |

원래라면 4 + 4 = 8개가 나와야 하지만, `이재민-컴퓨터구조-90`과 `조남선-자료구조-80`이 양쪽 테이블에 중복 존재하므로 제거하여 **총 6개**가 결과로 나옵니다.

### 🟡 교집합 (Intersection) — ∩

**교집합**은 두 릴레이션 R과 S에서 **공통으로 존재하는 튜플**만을 추출하는 연산입니다. 마치 1반과 2반에서 동아리 활동을 같이 하는 학생만 찾는 것처럼, 양쪽 모두에 속하는 데이터만 뽑아냅니다.

교집합도 합집합과 마찬가지로 **합병 가능** 조건을 만족해야 합니다. 즉, 두 릴레이션의 차수(속성 수)가 같고, 대응되는 속성의 도메인이 같아야 합니다.

#### 교집합의 결과 특성

- **결과의 차수**: R 또는 S의 차수와 동일
- **결과의 카디널리티**: R과 S 중 더 작은 카디널리티보다 작거나 같습니다. 공통인 것만 찾으니 당연히 작은 쪽보다 많이 나올 수 없습니다.

**학생1 ∩ 학생2 결과** (공통 튜플만 추출)

| 이름 | 과목 | 점수 |
|------|------|------|
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |

양쪽 테이블에 공통으로 존재하는 두 행만 결과로 나옵니다.

---

### 🔴 차집합 (Difference) — −

**차집합**은 R − S라고 쓸 때, R에는 있지만 S에는 없는 튜플만을 추출하는 연산입니다. 예를 들어 1반 학생 명단에서 동아리 회원 명단을 빼면, 동아리에 가입하지 않은 1반 학생들만 남는 것과 같습니다.

:::warning 차집합은 순서가 중요합니다!
R − S와 S − R은 완전히 다른 결과가 나옵니다. 5 − 2 = 3이지만, 2 − 5 = −3인 것처럼, 차집합은 어떤 것을 기준으로 빼느냐에 따라 결과가 완전히 달라집니다. **교환 법칙이 성립하지 않습니다.**
:::

#### 차집합의 결과 특성

- **결과의 차수**: R 또는 S의 차수와 동일
- **결과의 카디널리티**: R의 카디널리티보다 작거나 같습니다. R에서 일부를 빼는 것이므로 R보다 많이 나올 수는 없습니다.

**학생1 − 학생2 결과** (학생1에만 있고 학생2에는 없는 튜플)

| 이름 | 과목 | 점수 |
|------|------|------|
| 오태오 | 자료구조 | 90 |
| 김민준 | 운영체제 | 75 |

`이재민-컴퓨터구조-90`과 `조남선-자료구조-80`은 학생2에도 있으므로 제거되고, 학생1에만 있는 두 행만 남습니다.

---

### 🟣 카티션 프로덕트 (Cartesian Product) — ×

**카티션 프로덕트(곱집합)** 는 두 릴레이션 R과 S의 모든 튜플을 순서대로 하나씩 짝지어 새로운 릴레이션을 만드는 연산입니다. 마치 메뉴판에서 음식 2가지와 음료 3가지가 있을 때, 가능한 모든 세트 조합을 나열하면 2 × 3 = 6가지가 되는 것과 같습니다.

**카티션 프로덕트의 가장 큰 특징**은 **합병 가능하지 않아도 된다**는 점입니다. 앞서 본 합집합, 교집합, 차집합은 모두 두 릴레이션의 속성 수가 같아야 했지만, 카티션 프로덕트는 속성 수가 달라도 수행할 수 있습니다.

#### 카티션 프로덕트의 결과 특성

- **결과의 차수**: R의 차수 + S의 차수 (두 릴레이션의 속성을 모두 합침)
- **결과의 카디널리티**: R의 카디널리티 × S의 카디널리티 (모든 조합을 만들기 때문에 곱함)

#### 카티션 프로덕트 예시

R (차수: 3, 카디널리티: 4)와 S (차수: 1, 카디널리티: 3)를 카티션 프로덕트 하면:

- **결과 차수**: 3 + 1 = **4**
- **결과 카디널리티**: 4 × 3 = **12**

R의 각 튜플이 S의 모든 튜플과 한 번씩 짝을 이루기 때문에, 총 12개의 튜플이 결과로 생성됩니다.

:::tip 카티션 프로덕트와 조인의 관계
카티션 프로덕트는 그 자체로는 실용성이 크지 않지만, 뒤에서 배울 **조인(Join)** 연산의 기초가 됩니다. 조인은 카티션 프로덕트 결과에서 특정 조건을 만족하는 튜플만 선택한 것이라고 볼 수 있습니다.
:::

### 📊 교환 법칙과 결합 법칙 정리

일반 집합 연산자들에 대해 교환 법칙과 결합 법칙이 성립하는지 정리해 보겠습니다.

**교환 법칙(R OP S = S OP R):**

| 연산자 | 교환 법칙 성립 여부 | 설명 |
|--------|--------------------|----|
| 합집합 (∪) | ✅ 성립 | R∪S = S∪R (1반+2반 = 2반+1반) |
| 교집합 (∩) | ✅ 성립 | R∩S = S∩R (공통이니 순서 무관) |
| 카티션 프로덕트 (×) | ✅ 성립 | R×S = S×R (곱하기는 순서 무관) |
| **차집합 (−)** | ❌ 불성립 | R−S ≠ S−R (5−2≠2−5) |
| **디비전 (÷)** | ❌ 불성립 | R÷S ≠ S÷R (순서 바뀌면 다른 결과) |
| **세미조인 (⋉)** | ❌ 불성립 | 순서에 따라 다른 결과 |

**결합 법칙((R OP S) OP T = R OP (S OP T)):**

| 연산자 | 결합 법칙 성립 여부 |
|--------|-------------------|
| 합집합, 교집합, 카티션 프로덕트, (일반)조인 | ✅ 성립 |
| 차집합, 디비전, 세미조인 | ❌ 불성립 |

:::danger 차집합·디비전·세미조인 사용 시 주의!
차집합, 디비전, 세미조인은 **교환 법칙도, 결합 법칙도 모두 성립하지 않습니다.** 이 연산자들은 순서가 바뀌면 전혀 다른 결과가 나올 수 있으므로, 사용 시 연산 순서에 반드시 주의해야 합니다.
:::

## 🔬 순수 관계 연산자 (Pure Relational Operators)

순수 관계 연산자는 수학의 집합 이론에서 빌려온 것이 아니라, **데이터베이스에서 릴레이션을 조작하기 위해 새롭게 개발된 연산자**들입니다. 이 연산자들은 SQL 명령어와 직접적으로 연결됩니다.

---

### 🔽 셀렉트 (Select) — σ (시그마)

**셀렉트 연산자**는 릴레이션에서 **주어진 조건을 만족하는 튜플(행, Row)을 추출**하는 연산자입니다. 마치 도서관에서 수많은 책 중에서 "2020년 이후에 출판된 책"이라는 조건을 만족하는 책들만 골라내는 것과 같습니다.

기호: **σ조건(R)** — "릴레이션 R에서 조건을 만족하는 튜플을 추출하라"

#### 셀렉트는 수평적 부분집합

셀렉트 연산은 **수평적 부분집합(Horizontal Subset)** 이라고 부릅니다. 테이블에서 행(가로 방향)을 기준으로 조건에 맞는 것만 골라내기 때문입니다. 열(속성)의 수는 변하지 않고 그대로 유지되며, 오직 행의 수만 줄어듭니다.

```
전체 테이블 (행 여러 개)
┌────────────────────┐
│ 오태오 │ 자료구조 │ 90 │  ← 조건 만족 ✅ (선택됨)
│ 이재민 │ 컴구조   │ 70 │  ← 조건 불만족 ❌ (제외)
│ 조남선 │ 자료구조 │ 80 │  ← 조건 불만족 ❌ (제외)
│ 김민준 │ 운영체제 │ 85 │  ← 조건 만족 ✅ (선택됨)
└────────────────────┘
        ↓ σ 점수 ≥ 80 (학생)
┌────────────────────┐
│ 오태오 │ 자료구조 │ 90 │
│ 김민준 │ 운영체제 │ 85 │
└────────────────────┘
```

#### 셀렉트와 SQL의 관계

SQL에서 `SELECT ... FROM ... WHERE 조건`이라고 쓸 때, **WHERE절의 조건에 해당하는 부분**이 바로 관계대수의 셀렉트 연산자입니다.

```sql
-- 학생 테이블에서 점수가 80점 이상인 학생을 조회
-- σ 점수≥80 (학생) 에 해당하는 SQL
SELECT *
FROM 학생              -- 릴레이션 R = 학생 테이블
WHERE 점수 >= 80;      -- 이 부분이 관계대수의 셀렉트 연산자!
```

#### 셀렉트의 결과 특성

- **결과의 차수**: 원래 릴레이션과 동일 (열은 그대로 유지)
- **결과의 카디널리티**: 원래 릴레이션의 튜플 수보다 작거나 같음. 학생이 50명이면 조건을 만족하는 학생이 50명을 초과할 수는 없습니다.
- **교환적(Commutative)**: 여러 조건이 있을 때 조건 적용 순서를 바꿔도 결과가 동일합니다.

#### 선택도(Selectivity)란?

**선택도(Selectivity)** 란 전체 튜플 수 중에서 조건에 의해 선택된 튜플의 비율입니다.

> **선택도 = 선택된 튜플 수 ÷ 전체 튜플 수**

예를 들어, 학생 100명 중 남학생이 5명이면 "성별 = 남자" 조건의 선택도는 **5%**입니다. 서울 거주자가 60명이면 "지역 = 서울" 조건의 선택도는 **60%**입니다.

선택도는 **쿼리 최적화(Query Optimization)** 에서 매우 중요합니다. **선택도가 낮은 조건(더 적은 결과를 반환하는 조건)을 먼저 수행**하면 성능이 훨씬 좋아집니다.

:::tip 선택도와 쿼리 최적화 — 실전 예시
100명의 학생 중 남학생이 5명, 서울 거주자가 60명일 때, "남학생이면서 서울 거주자"를 찾는 경우:

- **방법 A (비효율)**: 먼저 서울 거주자 60명을 찾고 → 그 중 남학생 찾기 (60명 처리 후 필터링)
- **방법 B (효율)**: 먼저 남학생 5명을 찾고 → 그 중 서울 거주자 찾기 (5명만 처리하면 됨)

**방법 B가 훨씬 빠릅니다!** 선택도가 낮은(결과가 적게 나오는) 조건을 먼저 적용하는 것이 일반적인 데이터베이스 성능 최적화 방법입니다.
:::

### 📐 프로젝트 (Project) — π (파이)

**프로젝트 연산자**는 릴레이션에서 원하는 **속성(열, Column)만을 추출**하는 연산자입니다. 마치 학생 정보 전체 중에서 "이름과 전공만 보여줘"라고 요청하는 것과 같습니다.

기호: **π속성목록(R)** — "릴레이션 R에서 해당 속성들만 추출하라"

#### 프로젝트는 수직적 부분집합

프로젝트 연산은 **수직적 부분집합(Vertical Subset)** 이라고 부릅니다. 테이블에서 열(세로 방향)을 기준으로 원하는 열만 선택하기 때문입니다. 행(튜플)의 수는 유지되지만(중복 제거 후), 열의 수가 줄어듭니다.

```
전체 테이블 (열 여러 개)
┌──────┬─────────┬──────┬──────┬──────┐
│ 학번 │  이름   │ 전공 │ 학년 │ 점수 │
│ 1001 │ 오태오  │ 컴공 │  3   │  90  │
│ 1002 │ 이재민  │ 전기 │  2   │  70  │
│ 1003 │ 조남선  │ 컴공 │  1   │  80  │
└──────┴─────────┴──────┴──────┴──────┘
        ↓ π 이름, 전공 (학생)
┌─────────┬──────┐
│  이름   │ 전공 │
│ 오태오  │ 컴공 │
│ 이재민  │ 전기 │
│ 조남선  │ 컴공 │
└─────────┴──────┘
```

#### 프로젝트와 SQL의 관계

SQL에서 `SELECT 속성명 FROM 테이블`이라고 쓸 때, **SELECT 바로 뒤에 오는 속성(열) 목록 부분**이 관계대수의 프로젝트 연산자에 해당합니다.

```sql
-- 학생 테이블에서 이름과 전공만 조회
-- π 이름, 전공 (학생) 에 해당하는 SQL
SELECT 이름, 전공    -- 이 부분이 관계대수의 프로젝트 연산자!
FROM 학생;
```

#### 프로젝트의 중복 제거

프로젝트 결과에서도 릴레이션의 성질에 따라 **중복 튜플은 제거**됩니다. 예를 들어, 학생 테이블에서 `전공`만 추출하면, 같은 전공을 가진 학생이 여러 명 있더라도 동일한 전공명은 한 번만 나타납니다.

:::note 셀렉트(σ)와 프로젝트(π)의 차이를 명확히!
- **셀렉트(σ)**: **행(가로)** 방향으로 조건을 만족하는 튜플만 골라냄 → 수평적 부분집합
- **프로젝트(π)**: **열(세로)** 방향으로 원하는 속성만 골라냄 → 수직적 부분집합

셀렉트는 "어떤 학생들을 볼 것인가(행 필터)", 프로젝트는 "그 학생들의 어떤 정보를 볼 것인가(열 선택)"라고 이해하면 쉽습니다.
:::

### 🔗 조인 (Join) — ⋈

**조인(Join)** 은 두 개 이상의 릴레이션에서 **연관된 튜플들을 연결하여 하나의 릴레이션으로 합치는** 연산자입니다. 관계 데이터베이스에서 가장 핵심적이고 자주 사용되는 연산자 중 하나입니다.

#### 왜 조인이 필요한가?

현실 세계의 데이터는 여러 테이블에 분산되어 저장됩니다. 예를 들어 학생 정보는 학생 테이블에, 교수 정보는 교수 테이블에, 강의 정보는 강의 테이블에 각각 저장됩니다. 이렇게 나눠 저장하는 이유는 **데이터의 중복을 줄이고 일관성을 유지**하기 위해서입니다.

그런데 "내 지도교수의 이름이 뭐야?"라는 질문에 답하려면 학생 테이블과 교수 테이블의 데이터를 **연결(조인)** 해야 합니다. 학생 테이블에는 지도교수 번호만 있고, 교수 이름은 교수 테이블에 있기 때문입니다.

마치 우편번호부(지역번호가 실제 주소와 연결되는)처럼, 조인은 서로 다른 테이블에 흩어진 정보를 연결고리(공통 속성)를 통해 합쳐주는 역할을 합니다.

#### 조인의 수식 표현

조인은 수식으로 다음과 같이 표현됩니다:

> **R ⋈θ S = σθ(R × S)**

즉, **카티션 프로덕트(R × S)** 를 먼저 수행한 다음, 그 결과에서 **특정 조건(θ)을 만족하는 튜플만 셀렉트**한 것이 바로 조인입니다.

#### 조인의 종류

**① 세타 조인 (Theta Join)**

세타 조인은 두 릴레이션의 속성을 비교할 때 **모든 비교 연산자(=, ≠, &lt;, >, ≤, ≥)** 를 사용할 수 있는 조인입니다. 비교 연산자를 θ(세타)로 표기하기 때문에 세타 조인이라고 부릅니다.

```
R ⋈ (R.학번 = S.학번) S   -- 같을 때 조인
R ⋈ (R.성적 > S.최저점)S  -- 클 때 조인
```

**② 동일 조인 (Equi Join)**

동일 조인은 세타 조인에서 비교 조건이 **등호(=)** 인 경우, 즉 두 속성의 값이 **같은 경우에만** 연결하는 조인입니다.

동일 조인의 특징은 **동일 속성(같은 값을 가진 열)이 결과에 중복되어 나타난다**는 점입니다.

아래 예시를 보겠습니다.

**학생 테이블:**

| 학번 | 성명 | 학과코드 | 학년 |
|------|------|----------|------|
| 9801 | 홍길동 | CS | 1 |
| 9802 | 김영희 | EE | 2 |
| 9803 | 이철수 | CS | 3 |
| 9804 | 박민아 | ME | 4 |
| 9805 | 최준혁 | CS | 1 |

**성적 테이블:**

| 학번 | 과목 | 성적 |
|------|------|------|
| 9801 | 자료구조 | A |
| 9801 | 운영체제 | B |
| 9802 | 회로이론 | A |
| 9803 | 알고리즘 | A |
| 9804 | 열역학 | B |

**동일 조인 결과 (학생.학번 = 성적.학번):**

| 학번 | 성명 | 학과코드 | 학년 | **학번** | 과목 | 성적 |
|------|------|----------|------|----------|------|------|
| 9801 | 홍길동 | CS | 1 | **9801** | 자료구조 | A |
| 9801 | 홍길동 | CS | 1 | **9801** | 운영체제 | B |
| 9802 | 김영희 | EE | 2 | **9802** | 회로이론 | A |
| 9803 | 이철수 | CS | 3 | **9803** | 알고리즘 | A |
| 9804 | 박민아 | ME | 4 | **9804** | 열역학 | B |

9805번 학생은 성적 테이블에 해당 학번이 없으므로 조인 결과에 포함되지 않습니다. 또한 학번 열이 **두 번 중복되어 나타나는 것**을 확인할 수 있습니다. 이것이 동일 조인의 특징입니다.

**③ 자연 조인 (Natural Join)**

자연 조인은 동일 조인에서 **중복되는 속성을 하나만 남기고 제거**한 것입니다. 위의 동일 조인 결과에서 중복된 `학번` 열 중 하나를 제거하면 자연 조인이 됩니다.

**자연 조인 결과:**

| 학번 | 성명 | 학과코드 | 학년 | 과목 | 성적 |
|------|------|----------|------|------|------|
| 9801 | 홍길동 | CS | 1 | 자료구조 | A |
| 9801 | 홍길동 | CS | 1 | 운영체제 | B |
| 9802 | 김영희 | EE | 2 | 회로이론 | A |
| 9803 | 이철수 | CS | 3 | 알고리즘 | A |
| 9804 | 박민아 | ME | 4 | 열역학 | B |

중복된 `학번` 열이 하나로 합쳐져서 훨씬 깔끔한 결과가 나옵니다. 실무에서 가장 많이 사용하는 조인 형태가 바로 자연 조인(또는 그에 준하는 방식)입니다.

:::warning 조인 횟수가 많아지면 성능이 저하됩니다!
조인은 두 테이블을 연결하는 강력한 기능이지만, **조인 횟수가 늘어날수록 쿼리 성능은 저하**됩니다. 하나의 테이블에서 바로 결과를 뽑는 것이 가장 빠르고, 다른 테이블과 연결할수록 처리해야 할 데이터가 늘어나 느려집니다. 따라서 테이블 설계 단계에서 조인 횟수를 최소화하는 방향으로 설계하는 것이 중요합니다.
:::

### ➗ 디비전 (Division) — ÷

**디비전(나누기)** 은 S에 연관되어 있는 R의 튜플을 선택하는 연산으로, 조금 복잡하게 느껴질 수 있지만 예시를 보면 직관적으로 이해할 수 있습니다.

기호: **R ÷ S**

수식으로 표현하면: R의 차수가 m+n이고, S의 차수가 n일 때, 결과의 차수는 m입니다.

예를 들어, R이 속성 (A, B, C)를 가지고, S가 속성 (C)만 가진다면, R ÷ S의 결과는 속성 (A, B)가 됩니다.

#### 디비전 예시로 이해하기

아래의 릴레이션 R이 있다고 가정합니다.

**R 테이블:**

| A | B |
|---|---|
| a1 | b1 |
| a1 | b2 |
| a1 | b3 |
| a2 | b1 |
| a2 | b2 |
| a3 | b1 |

**S1 = {b1}** 로 나누면: b1을 가지고 있는 모든 A값 → **{a1, a2, a3}**

**S2 = {b1, b2}** 로 나누면: b1과 b2를 **모두** 가지고 있는 A값 → **{a1, a2}** (a3는 b2가 없으므로 제외)

**S3 = {b1, b2, b3}** 으로 나누면: b1, b2, b3을 **모두** 가지고 있는 A값 → **{a1}** (a1만 세 값 모두 보유)

> 💡 디비전의 핵심: S에 있는 **모든 값을 가지고 있는** R의 튜플을 찾는 것입니다. "이 조건들을 전부 만족하는 것만 골라라"는 의미입니다.

### 🏷️ 개명 연산 (Rename) — ρ (로)

**개명 연산**은 릴레이션이나 속성에 **새로운 이름을 부여**하는 연산자입니다. 기호는 ρ(로, Rho)를 사용합니다.

이 연산이 필요한 이유는, 연산의 중간 결과를 임시로 저장하고 다시 활용해야 할 때 그 결과에 이름을 붙여야 하기 때문입니다.

예를 들어, 전체 학생 중에서 컴퓨터공학과 학생만 뽑아서 그 결과를 `컴학생`이라는 임시 릴레이션으로 저장해 두고, 이후 연산에서 `컴학생`을 다시 활용하는 방식입니다.

```
ρ컴학생 ← σ 학과='컴퓨터공학' (학생)
-- "학생 테이블에서 학과가 컴퓨터공학인 튜플을 선택하여 '컴학생'이라는 이름을 붙인다"
```

이렇게 이름을 부여한 중간 결과는 이후 연산에서 하나의 독립된 릴레이션처럼 사용할 수 있습니다.

## ⚙️ 기본 연산자와 복합 연산자

### 기본 연산자 (Primitive Operators)

**기본 연산자**는 **다른 연산자로는 표현할 수 없는 독립적인 연산자**들입니다. 다음 다섯 가지가 기본 연산자에 해당합니다.

| 기본 연산자 | 설명 |
|------------|------|
| 합집합 (∪) | 다른 연산자로 표현 불가 |
| 차집합 (−) | 다른 연산자로 표현 불가 |
| 카티션 프로덕트 (×) | 다른 연산자로 표현 불가 |
| 프로젝트 (π) | 다른 연산자로 표현 불가 |
| 셀렉트 (σ) | 다른 연산자로 표현 불가 |

이 연산자들은 각각 고유한 기능을 수행하며, 서로 대체할 수 없습니다.

### 복합 연산자 (Derived Operators)

**복합 연산자**는 기본 연산자들을 조합해서 표현할 수 있는 연산자들입니다.

**① 교집합 (∩) → 차집합으로 표현 가능**

```
R ∩ S = R − (R − S)
```

R에서 "R에만 있는 것"을 빼면 "R과 S 모두에 있는 것", 즉 교집합이 남습니다.

**② 조인 (⋈) → 카티션 프로덕트 + 셀렉트로 표현 가능**

```
R ⋈θ S = σθ(R × S)
```

R과 S의 모든 조합(카티션 프로덕트)에서 조건θ를 만족하는 것만 선택(셀렉트)하면 조인과 같습니다.

**③ 디비전 (÷) → 차집합 + 카티션 프로덕트로 표현 가능**

디비전도 차집합과 카티션 프로덕트를 조합하여 표현할 수 있습니다.

:::info 왜 기본/복합 연산자 구분이 중요한가?
기본 연산자와 복합 연산자의 구분은 데이터베이스 이론을 이해하는 데 중요합니다. 복합 연산자는 편의를 위해 사용하는 것이며, 실제로는 기본 연산자들로 모두 구현 가능합니다. 이는 데이터베이스 엔진이 내부적으로 쿼리를 최적화하는 방식과도 관련이 있습니다.
:::

## 📋 관계대수 연산자 우선순위

여러 연산자를 괄호 없이 함께 사용할 때는 **우선순위**에 따라 처리 순서가 결정됩니다. 마치 수학에서 곱하기를 더하기보다 먼저 계산하는 것처럼, 관계대수에도 연산 우선순위가 있습니다.

| 우선순위 | 연산자 |
|----------|--------|
| 1 (가장 높음) | 프로젝트 (π) |
| 2 | 셀렉트 (σ) |
| 3 | 카티션 프로덕트 (×), 조인 (⋈), 디비전 (÷) |
| 4 | 차집합 (−) |
| 5 | 합집합 (∪), 교집합 (∩) |

예를 들어 `π이름(σ점수≥80(학생) × 성적)` 이라는 식이 있다면, 먼저 σ(셀렉트)가 실행되고, 그 결과와 성적 테이블의 ×(카티션 프로덕트)가 실행되고, 마지막으로 π(프로젝트)가 실행됩니다.

:::tip 괄호를 적극 활용하세요!
우선순위가 헷갈릴 때는 괄호를 사용하여 의도한 연산 순서를 명확히 표현하는 것이 좋습니다. 괄호는 항상 우선순위보다 먼저 처리됩니다.
:::

## ⚠️ 주의사항 및 실전 팁

### 합병 가능성 조건을 항상 확인하라

합집합, 교집합, 차집합을 수행하기 전에는 반드시 두 릴레이션이 **합병 가능한지** 확인해야 합니다.

```sql
-- ❌ 잘못된 예: 차수가 다른 두 테이블을 합집합 시도
SELECT 학번, 이름 FROM 학생
UNION
SELECT 학번, 이름, 학과 FROM 대학원생;  -- 열 수가 달라서 오류!

-- ✅ 올바른 예: 같은 차수로 맞춰서 합집합
SELECT 학번, 이름 FROM 학생
UNION
SELECT 학번, 이름 FROM 대학원생;  -- 열 수 동일 (합병 가능)
```

### 조인 시 조인 조건을 반드시 명시하라

카티션 프로덕트(CROSS JOIN)와 일반 조인은 결과가 완전히 다릅니다. 조인 조건 없이 두 테이블을 연결하면 카티션 프로덕트가 수행되어 원하지 않는 엄청난 양의 데이터가 생성될 수 있습니다.

```sql
-- ❌ 잘못된 예: 조인 조건 없음 → 카티션 프로덕트 발생
-- 학생 100명 × 교수 50명 = 5,000개 행이 생성됨!
SELECT *
FROM 학생, 교수;

-- ✅ 올바른 예: 조인 조건 명시
SELECT 학생.이름, 교수.교수명
FROM 학생
JOIN 교수 ON 학생.지도교수번호 = 교수.교수번호;
-- 실제 연관된 데이터만 연결됨
```

### 셀렉트 조건의 순서를 최적화하라

여러 조건이 있을 때 선택도가 낮은 조건(결과를 더 많이 줄이는 조건)을 앞에 배치하면 성능이 향상됩니다.

```sql
-- ❌ 비효율적인 예: 선택도 높은 조건 먼저
-- 서울 거주자 60명을 먼저 찾고, 그 중 남학생을 찾음
SELECT * FROM 학생
WHERE 지역 = '서울'       -- 100명 중 60명 선택 (선택도 60%)
  AND 성별 = '남';        -- 그 중 남학생 선택

-- ✅ 효율적인 예: 선택도 낮은 조건 먼저
-- 남학생 5명을 먼저 찾고, 그 중 서울 거주자를 찾음
SELECT * FROM 학생
WHERE 성별 = '남'         -- 100명 중 5명 선택 (선택도 5%)
  AND 지역 = '서울';      -- 그 중 서울 거주자 선택
```

## 📌 핵심 정리

- **관계대수**는 릴레이션(테이블)을 처리하기 위한 연산의 집합으로, SQL의 이론적 토대이며 절차적 방법이다.
- **릴레이션(Relation)** 은 테이블을 수학적으로 표현한 것이며, 관계대수의 입력과 출력 모두 릴레이션이다.
- **일반 집합 연산자**는 합집합(∪), 교집합(∩), 차집합(−), 카티션 프로덕트(×) 네 가지로, 수학 집합 이론에서 유래했다.
- **순수 관계 연산자**는 셀렉트(σ), 프로젝트(π), 조인(⋈), 디비전(÷)으로, 데이터베이스를 위해 개발된 연산자이다.
- **합병 가능(Union Compatible)** 조건: 합집합·교집합·차집합을 하려면 두 릴레이션의 차수(속성 수)가 같고 대응되는 도메인이 같아야 한다.
- **카티션 프로덕트**는 합병 가능 조건 없이 사용 가능하며, 결과 차수는 두 릴레이션 차수의 합, 카디널리티는 곱이다.
- **셀렉트(σ)** 는 조건을 만족하는 행(튜플)을 추출하는 수평적 부분집합으로, SQL의 WHERE절에 해당한다.
- **프로젝트(π)** 는 원하는 열(속성)만 추출하는 수직적 부분집합으로, SQL의 SELECT 열 목록에 해당한다.
- **선택도(Selectivity)** 는 전체 튜플 중 선택된 튜플의 비율이며, 선택도가 낮은 조건을 먼저 적용하면 성능이 향상된다.
- **동일 조인**은 조건이 등호(=)인 조인으로 결과에 중복 속성이 나타나고, **자연 조인**은 중복 속성을 제거한 형태다.
- **교환 법칙 성립**: 합집합, 교집합, (일반)조인, 카티션 프로덕트. **불성립**: 차집합, 디비전, 세미조인.
- **기본 연산자**: 합집합, 차집합, 카티션 프로덕트, 프로젝트, 셀렉트 — 다른 연산자로 대체 불가.
- **복합 연산자**: 교집합, 조인, 디비전 — 기본 연산자들로 표현 가능.
- **연산자 우선순위**: 프로젝트 > 셀렉트 > 카티션 프로덕트/조인/디비전 > 차집합 > 합집합/교집합 순이다.
