# 🧮 관계대수의 개념과 특징

데이터베이스를 공부하다 보면 어느 순간 **"관계대수"** 라는 단어를 마주치게 됩니다. 처음 보면 "대수(代數)? 수학 얘기인가?" 싶어서 당황할 수도 있습니다. 맞습니다. 관계대수는 수학적 이론에서 출발한 개념입니다. 하지만 너무 겁먹지 않아도 됩니다. 우리가 중학교, 고등학교 때 배웠던 **합집합, 교집합, 차집합** 같은 개념들이 이미 그 기초를 이루고 있으니까요.

이 문서에서는 관계대수가 무엇인지, 왜 필요한지, 어떤 연산자들로 구성되어 있는지를 **초보자도 쉽게 이해할 수 있도록** 차근차근 풀어서 설명하겠습니다.

---

## 📌 관계대수란 무엇인가?

### 🤔 왜 관계대수를 배워야 할까?

우리가 데이터베이스에서 데이터를 꺼낼 때 사용하는 **SQL(Structured Query Language, 구조화된 질의 언어)** 같은 명령어들은 사실 허공에서 갑자기 만들어진 것이 아닙니다. 마치 프로그래밍 언어가 수학의 논리 이론에 기반하는 것처럼, 데이터베이스 명령어들도 수학적 이론을 근거로 설계되었습니다.

그 수학적 이론의 뿌리가 바로 **관계대수(Relational Algebra)** 와 **관계 해석(Relational Calculus)** 입니다. 즉, 우리가 매일 쓰는 `SELECT * FROM 학생 WHERE 성적 >= 80` 같은 SQL 문장은 관계대수의 이론을 기반으로 구체화된 표현입니다. 이 뿌리를 이해해두면 SQL을 훨씬 더 깊게, 그리고 올바르게 이해할 수 있게 됩니다.

### 📖 관계대수의 정의

> **관계대수(Relational Algebra)** 란, 릴레이션(테이블)을 처리하기 위한 연산의 집합이다.

여기서 **릴레이션(Relation)** 이라는 단어가 등장합니다. 수학적으로는 "릴레이션"이라고 부르지만, 우리가 실제 데이터베이스에서 보는 것은 **테이블(Table)** 입니다. 행(Row)과 열(Column)로 이루어진 바로 그 표 형태의 구조를 말합니다.

마치 엑셀 스프레드시트에서 데이터가 표 형식으로 정리되어 있는 것처럼, 릴레이션도 **행(튜플, Tuple)** 과 **열(속성, Attribute)** 로 구성되어 있습니다.

관계대수는 이 릴레이션을 대상으로 여러 가지 **연산자(Operator)** 를 사용해 원하는 데이터를 추출하는 방법입니다.

### 🔁 입력도 릴레이션, 출력도 릴레이션

관계대수의 중요한 특징 중 하나는 **"연산의 대상(입력)도 릴레이션이고, 연산의 결과(출력)도 릴레이션"** 이라는 점입니다.

마치 과일 주스 기계에 사과를 넣으면 사과 주스가 나오듯이, 관계대수 연산에 릴레이션(테이블)을 넣으면 릴레이션(테이블)이 나옵니다. 이 성질 덕분에 연산의 결과를 또 다른 연산의 입력으로 사용할 수 있어서, 여러 연산을 **연속적으로 이어 붙이는 것**이 가능합니다.

### 🚶 절차적인 방법이란?

관계대수는 **절차적(Procedural)** 방법이라고 합니다. 이게 무슨 말이냐면, 단순히 "어떤 데이터를 원하는가"만 기술하는 것이 아니라, **"어떤 순서와 방법으로 데이터를 유도해 낼 것인가"** 까지 함께 기술해야 한다는 뜻입니다.

예를 들어, 요리 레시피에 비유하자면 "김치찌개를 만들어라"고만 말하는 것이 아니라, "먼저 돼지고기를 볶고, 그다음 김치를 넣고, 물을 붓고, 20분 끓인다"처럼 **구체적인 절차를 명시하는 것**이 관계대수의 방식입니다.

## 🗂️ 관계대수의 분류: 두 가지 연산자 그룹

관계대수의 연산자는 크게 두 가지 그룹으로 나뉩니다.

| 구분 | 이름 | 포함 연산자 | 기원 |
|---|---|---|---|
| 첫 번째 | **일반 집합 연산자** | 합집합, 교집합, 차집합, 카티션 프로덕트 | 수학의 집합이론 |
| 두 번째 | **순수 관계 연산자** | 셀렉트, 프로젝트, 조인, 디비전 | 데이터베이스 전용 |

**일반 집합 연산자(General Set Operators)** 는 우리가 수학 시간에 이미 배운 합집합(∪), 교집합(∩), 차집합(-), 곱집합(×) 개념을 데이터베이스에 적용한 것입니다.

**순수 관계 연산자(Pure Relational Operators)** 는 데이터베이스의 특성에 맞게 새롭게 만들어진 연산자들로, 릴레이션에서 특정 조건의 데이터를 꺼내거나, 두 테이블을 연결하거나, 나누는 연산을 수행합니다.

각 연산자의 기호는 아래와 같습니다.

| 연산자 이름 | 영어명 | 기호 |
|---|---|---|
| 합집합 | Union | ∪ |
| 교집합 | Intersection | ∩ |
| 차집합 | Difference | - |
| 카티션 프로덕트 | Cartesian Product | × |
| 셀렉트 | Select | σ (시그마) |
| 프로젝트 | Project | π (파이) |
| 조인 | Join | ⋈ |
| 디비전 | Division | ÷ |

:::info 수학 기호가 낯설어도 괜찮아요
관계대수의 기호(σ, π, ⋈, ÷ 등)는 처음 보면 생소하게 느껴질 수 있습니다. 하지만 이 기호들은 각 연산의 의미를 압축해서 표현한 것일 뿐, 핵심은 **"어떤 데이터를 어떻게 꺼내느냐"** 입니다. 기호보다 개념을 먼저 이해하는 데 집중하세요.
:::

## 🔢 일반 집합 연산자 (General Set Operators)

### 🟡 합집합 (Union, ∪)

**합집합이 왜 필요할까요?** 예를 들어 학교에 두 개의 학생 명단 테이블이 있다고 가정해봅시다. 하나는 1학년 학생 명단이고, 다른 하나는 2학년 학생 명단입니다. 이 두 명단을 합쳐서 전체 학생 명단을 만들고 싶을 때 합집합을 사용합니다. 마치 두 개의 출석부를 하나로 합치는 것과 같습니다.

**합집합의 조건: 합병 가능(Union Compatible)**

두 릴레이션을 합집합하려면 반드시 **합병 가능(Union Compatible)** 해야 합니다. 합병 가능하다는 것은 두 가지 조건을 만족해야 한다는 의미입니다.

1. **차수(Degree, 속성의 개수)가 같아야 합니다.** 릴레이션 R의 열 개수와 릴레이션 S의 열 개수가 같아야 합니다. 열이 3개인 테이블과 열이 5개인 테이블은 합집합을 할 수 없습니다.
2. **대응되는 속성별로 도메인(Domain, 데이터 타입)이 같아야 합니다.** 예를 들어 R의 첫 번째 열이 숫자(정수형)라면, S의 첫 번째 열도 숫자여야 합니다. 이름은 달라도 괜찮지만 타입은 같아야 합니다.

**합집합의 결과 특성**

- **결과의 차수(속성 개수)**: R 또는 S의 차수와 동일합니다. (어차피 같으니까요)
- **결과의 카디널리티(Cardinality, 튜플 수)**: R의 튜플 수 + S의 튜플 수. 단, 중복된 튜플은 제거합니다.

:::note 카디널리티(Cardinality)란?
카디널리티는 릴레이션(테이블)에 들어있는 **행(튜플, 레코드)의 개수**를 말합니다. 학생 테이블에 학생이 30명 들어있다면 카디널리티는 30입니다.
:::

**왜 중복을 제거할까요?** 릴레이션의 기본 성질 중 하나가 **튜플의 유일성**입니다. 즉, 릴레이션 안에는 완전히 똑같은 행이 두 개 이상 존재할 수 없습니다. 따라서 합집합 결과에서 두 테이블에 공통으로 존재하는 행은 하나만 남기고 나머지는 제거됩니다.

**실제 예시**

아래와 같이 학생1 테이블과 학생2 테이블이 있다고 가정합니다.

| 학생1 테이블 | | |
|---|---|---|
| **이름** | **과목** | **점수** |
| 오태오 | 자료구조 | 90 |
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |
| 홍길동 | 운영체제 | 75 |

| 학생2 테이블 | | |
|---|---|---|
| **이름** | **과목** | **점수** |
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |
| 김민수 | 알고리즘 | 85 |
| 박지현 | 네트워크 | 70 |

두 테이블 모두 속성이 3개(차수=3)이고 도메인도 같으므로 **합병 가능**합니다.

합집합 결과 = 학생1 ∪ 학생2:

| **이름** | **과목** | **점수** |
|---|---|---|
| 오태오 | 자료구조 | 90 |
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |
| 홍길동 | 운영체제 | 75 |
| 김민수 | 알고리즘 | 85 |
| 박지현 | 네트워크 | 70 |

원래 4 + 4 = 8개가 나와야 하지만, "이재민 컴퓨터구조 90"과 "조남선 자료구조 80"이 양쪽에 중복되어 있으므로 제거되어 총 **6개의 튜플**이 결과로 나옵니다.

---

### 🔵 교집합 (Intersection, ∩)

**교집합이 왜 필요할까요?** 앞의 예시에서 학생1 명단과 학생2 명단 **양쪽 모두에 등록된 학생**을 찾고 싶을 때 교집합을 사용합니다. 마치 두 동아리에 동시에 가입한 학생을 찾는 것처럼요.

**교집합의 조건**: 합집합과 마찬가지로 **합병 가능**해야 합니다. 즉, 차수가 같고 대응되는 도메인도 같아야 합니다.

**교집합의 결과 특성**

- **결과의 차수**: R 또는 S의 차수와 동일
- **결과의 카디널리티**: 최대 min(R의 카디널리티, S의 카디널리티)

왜 카디널리티가 두 테이블 중 **작은 쪽**을 넘을 수 없을까요? 교집합은 양쪽에 공통으로 있는 것만 뽑는데, 공통된 것이 작은 쪽보다 더 많을 수는 없기 때문입니다. 예를 들어 R에 10개, S에 3개가 있다면 공통은 아무리 많아도 3개를 넘을 수 없습니다.

위 학생1, 학생2 예시로 교집합을 구하면:

- 학생1 ∩ 학생2 = 이재민 컴퓨터구조 90, 조남선 자료구조 80 → **2개의 튜플**

---

### 🔴 차집합 (Difference, -)

**차집합이 왜 필요할까요?** "학생1 명단에는 있지만 학생2 명단에는 없는 학생"을 찾고 싶을 때 차집합을 사용합니다. 마치 A 동아리 명단에서 B 동아리 명단에 있는 사람을 모두 제외하고 남은 사람만 찾는 것처럼요.

**차집합의 조건**: 마찬가지로 **합병 가능**해야 합니다.

**차집합의 결과 특성**

- **결과의 차수**: R 또는 S의 차수와 동일
- **결과의 카디널리티**: 최대 R의 카디널리티 (R - S는 R보다 많아질 수 없음)

위 학생1, 학생2 예시로 차집합을 구하면:

- 학생1 - 학생2 = 오태오 자료구조 90, 홍길동 운영체제 75 → **2개의 튜플**
- 학생2 - 학생1 = 김민수 알고리즘 85, 박지현 네트워크 70 → **2개의 튜플**

:::warning 차집합은 순서가 중요합니다!
**R - S** 와 **S - R** 의 결과는 완전히 다릅니다. 마치 5 - 2 = 3이지만 2 - 5 = -3인 것처럼요. 차집합은 연산 순서를 바꾸면 결과가 달라지므로 항상 주의해야 합니다.
:::

### 🟣 카티션 프로덕트 (Cartesian Product, ×)

**카티션 프로덕트가 왜 필요할까요?** 앞의 세 연산자(합집합, 교집합, 차집합)는 모두 합병 가능한 두 테이블에만 적용할 수 있었습니다. 즉, 속성 개수가 같아야만 했죠. 하지만 카티션 프로덕트는 **속성 개수가 달라도** 연산을 수행할 수 있습니다. 그리고 이 연산은 뒤에서 배울 **조인(Join)** 의 기반이 됩니다.

**카티션 프로덕트는 합병 조건이 필요 없습니다.** 서로 다른 구조의 두 테이블도 곱하기 연산을 적용할 수 있습니다.

**카티션 프로덕트의 결과 특성**

- **결과의 차수(속성 개수)**: R의 차수 + S의 차수
- **결과의 카디널리티(행 수)**: R의 카디널리티 × S의 카디널리티

마치 옷 가게에서 상의 3벌과 하의 4벌이 있을 때 가능한 모든 코디 조합이 3 × 4 = 12가지인 것처럼, 카티션 프로덕트는 두 테이블의 **모든 행의 조합**을 만들어냅니다.

**실제 예시**

R 테이블(차수=3, 카디널리티=4)과 S 테이블(차수=1, 카디널리티=3)을 카티션 프로덕트하면:

- 결과의 차수 = 3 + 1 = **4**
- 결과의 카디널리티 = 4 × 3 = **12**

R의 첫 번째 행과 S의 모든 행을 조합하고, 그다음 R의 두 번째 행과 S의 모든 행을 조합하는 방식으로 총 12개의 행이 생성됩니다.

:::note 카티션 프로덕트와 조인의 관계
카티션 프로덕트는 두 테이블의 모든 조합을 만들어냅니다. 이후 이 결과에서 특정 조건을 만족하는 행만 선택하면 그것이 바로 **조인(Join)** 이 됩니다. 즉, 조인 = 카티션 프로덕트 + 셀렉트로 표현할 수 있습니다.
:::

## ⚙️ 순수 관계 연산자 (Pure Relational Operators)

순수 관계 연산자는 수학의 집합이론을 그대로 가져온 것이 아니라, **데이터베이스의 특성에 맞게 새롭게 개발된 연산자들**입니다. 실제 데이터를 다루는 데 훨씬 직접적으로 활용됩니다.

### 🔍 셀렉트 (Select, σ - 시그마)

**셀렉트 연산자가 왜 필요할까요?** 학생 테이블에 100명의 학생이 있을 때, 그 중에서 **점수가 80점 이상인 학생만** 골라서 보고 싶다면 어떻게 해야 할까요? 셀렉트 연산자가 바로 이 역할을 합니다. 조건을 만족하는 행(튜플)만 골라내는 연산자입니다.

**셀렉트 연산자의 특징**

셀렉트(σ)는 릴레이션에서 **주어진 조건을 만족하는 튜플(행)** 을 찾아내는 연산자입니다. 이것을 **"수평적 부분집합(Horizontal Subset)"** 이라고 표현합니다. 왜냐하면 테이블에서 가로 방향, 즉 행을 기준으로 데이터를 잘라내기 때문입니다.

```
표현 방식: σ [조건] (릴레이션)
SQL 대응: SELECT * FROM 릴레이션 WHERE 조건
```

- **결과의 차수**: 원래 릴레이션과 동일 (속성의 개수는 변하지 않음)
- **결과의 카디널리티**: 원래 릴레이션의 카디널리티보다 작거나 같음

하나의 릴레이션만을 대상으로 연산을 수행하며, 30명의 학생 중 조건에 맞는 학생을 찾을 때 결과가 30명을 초과할 수는 없습니다.

**셀렉트 연산은 교환적(Commutative)입니다**

여러 조건이 있을 때 어떤 순서로 적용해도 결과가 같습니다.

예를 들어, "남학생이면서 서울에 사는 학생"을 찾는다면:
- 방법 1: 먼저 남학생을 찾고 → 그 중에서 서울인 사람을 찾는다
- 방법 2: 먼저 서울인 사람을 찾고 → 그 중에서 남학생을 찾는다
- 방법 3: 남학생이면서 서울인 사람을 동시에 찾는다

이 세 가지 방법 모두 최종 결과는 동일합니다. 이를 **교환적(Commutative)** 이라고 합니다.

**선택도(Selectivity)란 무엇인가?**

셀렉트 연산에서 중요한 개념으로 **선택도(Selectivity)** 가 있습니다. 선택도란 **전체 튜플 수 중에서 조건을 만족하는 튜플의 비율**입니다.

> 선택도 = (조건을 만족하는 튜플 수) ÷ (전체 튜플 수) × 100%

선택도는 **쿼리 최적화(Query Optimization)** 에서 매우 중요한 척도가 됩니다. **선택도가 낮은 조건(결과가 적게 나오는 조건)을 먼저 수행하는 것이 성능 향상에 유리합니다.**

예를 들어, 100명의 학생 중에서:
- 남학생은 5명 → 선택도 5%
- 서울 거주자는 60명 → 선택도 60%

이때 "남학생이면서 서울 거주자"를 찾으려면, **선택도가 낮은 "남학생" 조건을 먼저 적용**하는 것이 훨씬 효율적입니다. 왜냐하면:
- 남학생 먼저: 5명을 뽑고 → 그 중에서 서울인 사람 찾기 (5명만 검사)
- 서울 먼저: 60명을 뽑고 → 그 중에서 남학생 찾기 (60명을 메모리에 올려야 함)

첫 번째 방법이 훨씬 적은 연산을 수행하므로 성능이 좋습니다.

**실제 예시**

```sql
-- σ (점수 >= 80) (학생)
-- SQL로 표현하면:
SELECT * FROM 학생 WHERE 점수 >= 80;
```

학생 테이블 전체 4개 튜플 중 점수 80 이상인 2개 튜플이 결과로 나오면, 선택도 = 2/4 × 100% = **50%**

### 📋 프로젝트 (Project, π - 파이)

**프로젝트 연산자가 왜 필요할까요?** 학생 테이블에 학번, 이름, 전공, 학년, 점수 등 5개의 속성이 있을 때, 그 중에서 **이름과 전공만** 보고 싶을 때 사용합니다. 특정 열(속성)만 추출하는 연산자입니다.

**프로젝트 연산자의 특징**

프로젝트(π)는 릴레이션에서 **지정된 속성(열)만** 추출하는 연산자입니다. 이것을 **"수직적 부분집합(Vertical Subset)"** 이라고 표현합니다. 왜냐하면 테이블에서 세로 방향, 즉 열을 기준으로 데이터를 잘라내기 때문입니다.

셀렉트(σ)가 행을 기준으로 자른다면, 프로젝트(π)는 열을 기준으로 자릅니다.

```
표현 방식: π [속성 목록] (릴레이션)
SQL 대응: SELECT 속성1, 속성2 FROM 릴레이션
```

- **결과의 차수**: 지정한 속성의 개수 (원래보다 작거나 같음)
- **결과의 카디널리티**: 원래 릴레이션의 카디널리티보다 작거나 같음 (중복 제거로 인해)

중복 제거가 발생하는 이유는, 여러 속성 중 일부만 남기면 원래 구분되던 행들이 동일해질 수 있기 때문입니다. 예를 들어 학생 테이블에서 "학년"만 추출하면, 같은 학년의 학생들이 모두 중복되어 나타나게 됩니다. 릴레이션의 튜플 유일성 원칙에 따라 중복은 제거되고 하나만 남깁니다.

**실제 예시**

```sql
-- π (이름, 전공) (학생)
-- SQL로 표현하면:
SELECT 이름, 전공 FROM 학생;
```

학생 테이블의 5개 속성 중 이름과 전공 2개 속성만 추출됩니다.

**셀렉트(σ)와 프로젝트(π) 비교**

| 구분 | 셀렉트 (σ) | 프로젝트 (π) |
|---|---|---|
| **역할** | 조건에 맞는 행 추출 | 원하는 열 추출 |
| **방향** | 수평적 (가로) | 수직적 (세로) |
| **SQL 대응** | WHERE 절 | SELECT 절의 컬럼 지정 |
| **결과 차수** | 원본과 동일 | 원본보다 작거나 같음 |

### 🔗 조인 (Join, ⋈)

**조인이 왜 필요할까요?** 데이터베이스에서는 정보를 여러 테이블에 나누어 저장합니다. 예를 들어, 학생 테이블에는 학번과 지도교수번호가 있고, 교수 테이블에는 교수번호와 교수이름이 있다고 가정해봅시다. "나의 지도교수 이름이 무엇인가?"를 알고 싶다면, 학생 테이블만 봐서는 알 수 없습니다. 교수 이름은 교수 테이블에 있으니까요. 이때 두 테이블을 **연결해서** 정보를 가져오는 연산이 바로 **조인(Join)** 입니다.

마치 학생증에 적힌 지도교수 번호를 들고 교수 명단에서 그 번호를 찾아 교수 이름을 확인하는 것과 같습니다.

**조인의 기본 개념**

조인은 두 릴레이션에서 **특정 속성의 값을 비교하여 조건을 만족하는 튜플들을 연결**하는 연산입니다. 공식으로 표현하면 다음과 같습니다.

> 조인(R ⋈ S) = σ [조건] (R × S)

즉, **카티션 프로덕트(×)로 모든 조합을 만들고, 거기서 셀렉트(σ)로 조건에 맞는 것만 뽑는 것**이 조인입니다.

:::warning 조인은 성능에 영향을 줍니다
조인 횟수가 많아질수록 성능은 그만큼 떨어집니다. 하나의 테이블에서 바로 결과를 얻는 것이 가장 빠르고, 여러 테이블을 연결할수록 처리 비용이 증가합니다. 실무에서는 조인 횟수를 최소화하는 것이 데이터베이스 성능 최적화의 핵심입니다.
:::

**조인의 종류**

조인에는 여러 종류가 있습니다. 각각 어떤 상황에서 어떻게 다른지 살펴보겠습니다.

**① 세타 조인 (Theta Join)**

세타 조인은 **모든 비교 연산자(=, ≠, &lt;, >, ≤, ≥)를 사용하는 조인**입니다. 가장 일반적인 조인의 형태입니다. 세타(θ)는 임의의 비교 연산자를 의미합니다.

**② 동일 조인 (Equi Join)**

동일 조인은 세타 조인 중에서 **비교 조건이 등호(=)인 경우**입니다. 즉, 두 속성의 값이 같은 튜플만 연결합니다. 가장 많이 사용되는 조인 방식입니다.

동일 조인의 특징은 결과에 **동일한 속성이 중복되어 나타난다**는 점입니다. 예를 들어 학생 테이블의 "학번"과 성적 테이블의 "학번"을 동일 조인하면, 결과에 "학번" 속성이 **두 번** 나타납니다. 이것이 동일 조인의 특징입니다.

**③ 자연 조인 (Natural Join)**

자연 조인은 동일 조인에서 **중복되는 속성 하나를 제거**한 것입니다. 동일 조인이 같은 속성 값을 두 번 보여준다면, 자연 조인은 그 중복을 없애서 더 깔끔한 결과를 제공합니다.

> 자연 조인 = 동일 조인 - 중복 속성 제거

실무에서 가장 많이 사용하는 조인이 바로 이 **자연 조인**입니다.

**실제 예시로 보는 동일 조인 vs 자연 조인**

학생 테이블과 성적 테이블이 있다고 합시다.

| 학번 | 성명 | 학과코드 |
|---|---|---|
| 9801 | 김철수 | CS |
| 9802 | 이영희 | EE |
| 9803 | 박민준 | CS |
| 9804 | 최수진 | ME |
| 9805 | 정태양 | CS |

| 학번 | 과목 | 성적 |
|---|---|---|
| 9801 | 자료구조 | 90 |
| 9801 | 알고리즘 | 85 |
| 9802 | 회로이론 | 80 |
| 9803 | 자료구조 | 75 |
| 9804 | 역학 | 88 |

동일 조인 결과 (학생.학번 = 성적.학번):

| 학번 | 성명 | 학과코드 | **학번** | 과목 | 성적 |
|---|---|---|---|---|---|
| 9801 | 김철수 | CS | 9801 | 자료구조 | 90 |
| 9801 | 김철수 | CS | 9801 | 알고리즘 | 85 |
| ... | ... | ... | ... | ... | ... |

학번 속성이 **두 번 중복**되어 나타납니다.

자연 조인 결과 (중복 속성 제거):

| **학번** | 성명 | 학과코드 | 과목 | 성적 |
|---|---|---|---|---|
| 9801 | 김철수 | CS | 자료구조 | 90 |
| 9801 | 김철수 | CS | 알고리즘 | 85 |
| ... | ... | ... | ... | ... |

학번이 하나만 남고, 9805(정태양)는 성적 테이블에 없으므로 **조인에 참여하지 못합니다**.

### ➗ 디비전 (Division, ÷)

**디비전이 왜 필요할까요?** "모든 필수 과목을 수강한 학생"처럼, **S 릴레이션의 모든 값에 대응하는 R 릴레이션의 튜플**을 찾을 때 사용합니다. 마치 "모든 조건을 완벽히 충족하는 것을 찾아라"와 같은 연산입니다.

**디비전의 개념**

디비전은 R ÷ S 로 표현하며, "S에 연관되어 있는 R의 튜플을 선택"합니다.

수식으로 표현하면:
- R의 차수가 m+n이고, S의 차수가 n이면, 결과의 차수는 m입니다.

쉽게 말해, R의 속성에서 S의 속성 부분을 제외한 나머지 속성들 중, **S의 모든 값과 대응되는 것들**을 찾아냅니다.

**실제 예시**

R 테이블(A, B 두 속성):

| A | B |
|---|---|
| a1 | b1 |
| a1 | b2 |
| a2 | b1 |
| a2 | b2 |
| a3 | b1 |

S1 = {b1} 로 디비전 (R ÷ S1):
- b1을 가진 A를 찾으면 → a1, a2, a3 모두 b1을 갖고 있으므로 결과 = **{a1, a2, a3}**

S2 = {b1, b2} 로 디비전 (R ÷ S2):
- b1과 b2 **둘 다** 가진 A를 찾으면 → a1(b1, b2 보유), a2(b1, b2 보유), a3(b1만 있고 b2 없음)
- 결과 = **{a1, a2}**

S3 = {b1, b2, b3} 로 디비전 (R ÷ S3):
- b1, b2, b3 **모두** 가진 A를 찾으면 → a1(b3 없음), a2(b3 없음), a3(b2, b3 없음)
- 결과 = **없음 (공집합)**

디비전은 "모든 조건을 충족하는 것"만을 찾기 때문에, 조건이 많아질수록 결과가 줄어드는 경향이 있습니다.

### 🏷️ 개명 연산 (Rename, ρ - 로우)

**개명 연산이 왜 필요할까요?** 여러 연산을 순서대로 수행하다 보면 중간 결과가 생기게 됩니다. 이 중간 결과는 화면에 보여주고 나면 사라지는 임시 데이터입니다. 하지만 이 중간 결과를 **이후 연산에서 다시 사용**하고 싶다면 어떻게 해야 할까요? 개명 연산을 사용해서 이름을 붙여주면 됩니다.

마치 임시 메모장에 중요한 계산 결과를 적어두고 나중에 꺼내 쓰는 것처럼, 개명 연산은 중간 결과에 이름(임시 테이블명)을 부여해서 다음 연산에서 활용할 수 있게 합니다.

예를 들어, 전체 학생 중 학과가 "컴퓨터공학"인 학생들만 추출해서 `컴학생`이라는 이름의 임시 테이블로 저장해두면, 이후 연산에서 `컴학생`이라는 이름으로 그 결과를 참조할 수 있습니다.

## 📐 교환 법칙과 결합 법칙

### ↔️ 교환 법칙 (Commutative Law)

교환 법칙이란 **연산에서 피연산자의 순서를 바꿔도 결과가 같다**는 성질입니다. 덧셈에서 5 + 2 = 2 + 5인 것처럼요.

관계대수에서 **교환 법칙이 성립하는 연산자**:

| 연산자 | 교환 법칙 | 예시 |
|---|---|---|
| **합집합 (∪)** | R ∪ S = S ∪ R | 두 집합을 합치는 순서는 무관 |
| **교집합 (∩)** | R ∩ S = S ∩ R | 공통 원소를 찾는 순서는 무관 |
| **조인 (⋈)** | R ⋈ S = S ⋈ R | 두 테이블을 연결하는 순서는 무관 |
| **카티션 프로덕트 (×)** | R × S = S × R | 모든 조합을 만드는 순서는 무관 |

관계대수에서 **교환 법칙이 성립하지 않는 연산자**:

| 연산자 | 이유 |
|---|---|
| **차집합 (-)** | R - S ≠ S - R (5-2 ≠ 2-5) |
| **디비전 (÷)** | R ÷ S ≠ S ÷ R (10÷2 ≠ 2÷10) |
| **세미조인 (⋉)** | 한쪽 방향으로만 연결하는 조인이므로 순서가 중요 |

### 🔄 결합 법칙 (Associative Law)

결합 법칙이란 **세 개 이상의 연산에서 어느 것을 먼저 계산해도 결과가 같다**는 성질입니다. 덧셈에서 (5 + 2) + 7 = 5 + (2 + 7)인 것처럼요.

관계대수에서 **결합 법칙이 성립하는 연산자**: 합집합(∪), 교집합(∩), 조인(⋈), 카티션 프로덕트(×)

**결합 법칙이 성립하지 않는 연산자**: 차집합(-), 디비전(÷), 세미조인(⋉)

:::danger 차집합, 디비전, 세미조인 주의!
차집합, 디비전, 세미조인은 **교환 법칙도 결합 법칙도 모두 성립하지 않습니다.** 연산 순서가 바뀌면 완전히 다른 결과가 나올 수 있으므로, 이 세 연산자를 사용할 때는 반드시 **연산 순서를 신중하게 결정**해야 합니다.
:::

## 🏗️ 기본 연산자와 복합 연산자

### 🧱 기본 연산자 (Primitive Operators)

기본 연산자는 **다른 연산자로 표현할 수 없는, 그 자체로 독립적인 연산자**입니다. 마치 레고 블록의 기본 조각처럼, 이 연산자들은 더 이상 분해할 수 없는 기본 단위입니다.

기본 연산자 목록:
- **합집합 (∪)**
- **차집합 (-)**
- **카티션 프로덕트 (×)**
- **프로젝트 (π)**
- **셀렉트 (σ)**

이 다섯 가지 연산자는 다른 어떤 연산자로도 대체할 수 없습니다. 이 기본 연산자들이 모여 다른 복합 연산자를 만들어낼 수 있습니다.

### 🔀 복합 연산자 (Derived Operators)

복합 연산자는 **기본 연산자들의 조합으로 표현할 수 있는 연산자**입니다.

**교집합 (∩)은 차집합(-)으로 표현 가능합니다:**

```
R ∩ S = R - (R - S) - (S - R)
```

그림으로 이해하면, R에서 "R에만 있는 부분(R-S)"를 빼면 R과 S의 공통 부분만 남게 됩니다.

**조인 (⋈)은 카티션 프로덕트(×)와 셀렉트(σ)로 표현 가능합니다:**

```
R ⋈[조건] S = σ[조건] (R × S)
```

R과 S의 모든 조합을 만든 다음(×), 그 중에서 조건에 맞는 것만 선택(σ)하면 조인이 됩니다.

**디비전 (÷)은 차집합(-)과 카티션 프로덕트(×)로 표현 가능합니다.**

| 연산자 종류 | 연산자 이름 | 특징 |
|---|---|---|
| **기본 연산자** | 합집합, 차집합, 카티션 프로덕트, 프로젝트, 셀렉트 | 다른 연산자로 대체 불가 |
| **복합 연산자** | 교집합, 조인, 디비전 | 기본 연산자의 조합으로 표현 가능 |

## ⚡ 연산자 우선순위

관계대수에서도 수학처럼 **연산자마다 우선순위**가 있습니다. 수학에서 2 + 3 × 4를 계산할 때 곱하기를 먼저 하듯이, 관계대수에서도 우선순위에 따라 연산을 수행합니다.

관계대수 연산자의 우선순위 (높은 것부터 낮은 순서):

| 우선순위 | 연산자 |
|---|---|
| 1 (가장 높음) | 프로젝트 (π) |
| 2 | 셀렉트 (σ) |
| 3 | 카티션 프로덕트 (×), 조인 (⋈), 디비전 (÷) |
| 4 | 차집합 (-) |
| 5 | 합집합 (∪) |
| 6 (가장 낮음) | 교집합 (∩) |

예를 들어 괄호 없이 여러 연산자를 함께 쓴다면, **우선순위가 높은 연산자부터 먼저 처리**합니다. 마치 수식에서 곱셈을 덧셈보다 먼저 계산하는 것처럼요.

:::tip 괄호를 적극 활용하세요
복잡한 관계대수 식을 작성할 때는 우선순위를 외우는 것보다 **괄호를 명시적으로 사용**하는 것이 훨씬 안전하고 가독성도 좋습니다. 괄호를 사용하면 연산 순서가 명확해져서 실수를 방지할 수 있습니다.
:::

## ⚠️ 주의사항 및 실전 팁

### ✅ 올바른 합집합 사용 예시와 잘못된 예시

```sql
-- ✅ 올바른 예시: 두 테이블의 속성 개수와 도메인이 같을 때
-- 학생1 테이블: (이름 VARCHAR, 과목 VARCHAR, 점수 INT)
-- 학생2 테이블: (성명 VARCHAR, 교과목 VARCHAR, 성적 INT)
-- 이름은 달라도 되지만, 속성 수(3개)와 데이터 타입이 같으므로 합집합 가능
학생1 ∪ 학생2  -- 합병 가능 조건 충족

-- ❌ 잘못된 예시: 속성 개수가 다를 때
-- 학생3 테이블: (이름 VARCHAR, 과목 VARCHAR, 점수 INT, 학년 INT) -- 속성 4개
-- 학생1 테이블: (이름 VARCHAR, 과목 VARCHAR, 점수 INT) -- 속성 3개
학생1 ∪ 학생3  -- ❌ 속성 개수 불일치! 합병 불가
```

### ✅ 셀렉트와 프로젝트 혼합 사용

```sql
-- ✅ 올바른 예시: 셀렉트 후 프로젝트 적용
-- "점수가 80 이상인 학생의 이름과 전공만 보고 싶을 때"

-- 1단계: 셀렉트(σ)로 점수 80 이상인 행 추출
-- σ(점수 >= 80)(학생)

-- 2단계: 프로젝트(π)로 이름, 전공 열만 추출
-- π(이름, 전공)(σ(점수 >= 80)(학생))

-- SQL로 표현하면:
SELECT 이름, 전공     -- 프로젝트(π)에 해당
FROM 학생             -- 대상 릴레이션
WHERE 점수 >= 80;     -- 셀렉트(σ)에 해당
```

이 예시에서 알 수 있듯이, **관계대수의 연산이 SQL 명령어와 정확히 대응**됩니다. 관계대수를 이해하면 SQL의 구조가 왜 그렇게 설계되었는지 자연스럽게 이해됩니다.

### ✅ 조인 사용 시 주의사항

```sql
-- ✅ 올바른 조인 예시: 공통 속성(학번)으로 연결
-- 자연 조인: 학생 ⋈ 성적
-- SQL로 표현하면:
SELECT 학생.학번, 학생.성명, 성적.과목, 성적.점수
FROM 학생
JOIN 성적 ON 학생.학번 = 성적.학번;

-- ❌ 잘못된 예시: 관련 없는 속성으로 조인 (카티션 프로덕트와 동일한 결과)
-- 공통 의미가 없는 속성으로 조인하면 의미 없는 데이터 조합이 생성됨
SELECT * FROM 학생 JOIN 성적 ON 학생.이름 = 성적.점수;
-- ❌ 이름(문자열)과 점수(숫자)를 비교하는 것은 의미 없음
```

:::tip 선택도를 활용한 쿼리 최적화
실제 데이터베이스에서 여러 조건을 AND로 연결할 때, **결과 행이 적게 나오는 조건(선택도가 낮은 조건)을 앞에 배치**하면 성능이 향상될 수 있습니다. 데이터베이스 옵티마이저가 자동으로 처리해주기도 하지만, 원리를 이해하고 있으면 훨씬 효율적인 쿼리를 작성할 수 있습니다.
:::

:::warning 합병 가능 조건을 꼭 확인하세요
합집합(∪), 교집합(∩), 차집합(-)을 수행하기 전에는 반드시:
1. **차수(속성 개수)가 같은지** 확인
2. **대응되는 속성의 도메인(데이터 타입)이 같은지** 확인

이 두 조건 중 하나라도 만족하지 않으면 연산을 수행할 수 없습니다.
:::

## 📚 핵심 정리

이 문서에서 배운 핵심 개념들을 정리합니다.

- **관계대수**는 릴레이션(테이블)을 처리하기 위한 연산의 집합이며, 연산의 대상과 결과 모두 릴레이션이다
- 관계대수는 **절차적 방법**으로, 원하는 데이터와 그것을 유도하는 방법을 모두 기술한다
- **일반 집합 연산자**: 합집합(∪), 교집합(∩), 차집합(-), 카티션 프로덕트(×)
- **순수 관계 연산자**: 셀렉트(σ), 프로젝트(π), 조인(⋈), 디비전(÷)
- **합병 가능 조건**: 차수(속성 수)가 같고 대응 속성의 도메인(데이터 타입)이 같아야 합집합·교집합·차집합이 가능하다
- **카티션 프로덕트**는 합병 조건 없이 두 테이블의 모든 조합을 생성하며, 결과 차수는 합, 카디널리티는 곱이다
- **셀렉트(σ)** 는 조건에 맞는 행을 추출하는 수평적 부분집합 연산이며, SQL의 WHERE 절에 대응된다
- **프로젝트(π)** 는 원하는 열을 추출하는 수직적 부분집합 연산이며, SQL의 SELECT 컬럼 지정에 대응된다
- **선택도**는 전체 튜플 중 조건을 만족하는 비율로, 선택도가 낮은 조건을 먼저 수행하면 성능이 향상된다
- **자연 조인**은 동일 조인에서 중복 속성을 제거한 것으로, 실무에서 가장 많이 사용된다
- **교환 법칙**은 합집합, 교집합, 조인, 카티션 프로덕트에서 성립하고, 차집합, 디비전, 세미조인에서 성립하지 않는다
- **결합 법칙**도 교환 법칙과 동일한 연산자들에서 성립/불성립한다
- **기본 연산자**(합집합, 차집합, 카티션 프로덕트, 프로젝트, 셀렉트)는 다른 연산자로 대체 불가하다
- **복합 연산자**(교집합, 조인, 디비전)는 기본 연산자의 조합으로 표현할 수 있다
- 연산자 **우선순위**: 프로젝트 > 셀렉트 > 카티션 프로덕트/조인/디비전 > 차집합 > 합집합 > 교집합
