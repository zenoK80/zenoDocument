# 🧮 일반 집합 연산자: 합집합(Union) 연산

데이터베이스를 공부하다 보면 "관계대수"라는 생소한 용어를 마주치게 됩니다. 수학 시간에 배운 합집합, 교집합, 차집합이 데이터베이스 세계에서 어떻게 쓰이는지, 처음 접하는 분들도 확실하게 이해할 수 있도록 차근차근 설명해 드리겠습니다.

---

## 🗂️ 관계대수란 무엇인가?

### 관계대수의 정의

**관계대수(Relational Algebra)** 란 한마디로 말하면 **"릴레이션(테이블)을 처리하기 위한 연산의 집합"** 입니다. 조금 어렵게 느껴질 수 있으니, 아주 쉽게 풀어서 설명해 드리겠습니다.

우리가 데이터베이스에서 사용하는 `SELECT`, `JOIN`, `WHERE` 같은 명령어들은 사실 하늘에서 뚝 떨어진 것이 아닙니다. 이 명령어들은 수십 년 전에 수학자들이 정립한 **수학적 이론**을 기반으로 만들어진 것입니다. 그 수학적 이론의 기반이 바로 **관계대수**와 **관계 해석(Relational Calculus)** 입니다.

> 💡 관계대수는 데이터베이스 명령어들의 수학적 뿌리입니다. 마치 건물의 설계도처럼, SQL이라는 언어가 지어지기 전에 먼저 이론적으로 정립된 개념입니다.

### 릴레이션(Relation)이란?

관계대수에서 핵심이 되는 개념이 바로 **릴레이션(Relation)** 입니다. 우리가 실제 데이터베이스에서 "테이블(Table)"이라고 부르는 것을 수학적으로는 "릴레이션"이라고 부릅니다. 행(Row)과 열(Column)로 구성된 표 형태의 데이터 구조라고 생각하면 됩니다.

마치 엑셀 스프레드시트처럼 생긴 표가 바로 릴레이션입니다.

### 관계대수의 중요한 특징: 절차적 방법

관계대수는 **절차적인 방법(Procedural Method)** 입니다. 즉, "어떤 데이터를 원하는가?"뿐만 아니라 "그 데이터를 어떤 순서로 추출할 것인가?"까지 함께 기술해야 합니다. 이것은 마치 요리 레시피와 같습니다. 단순히 "김치찌개를 만들어라"가 아니라 "먼저 돼지고기를 볶고, 그 다음 김치를 넣고, 물을 붓고, 끓인다"처럼 순서가 명확히 정해져 있습니다.

또한 관계대수의 또 다른 중요한 특징은 **입력도 릴레이션, 출력도 릴레이션**이라는 점입니다. 테이블에서 연산을 수행하면, 그 결과도 테이블 형태로 나옵니다. 이것을 **"닫힘 성질(Closure Property)"** 이라고 합니다.

## 📚 관계대수 연산자의 분류

관계대수 연산자는 크게 두 가지로 나뉩니다. 이 분류를 먼저 이해해야 나머지 내용을 체계적으로 공부할 수 있습니다.

### 일반 집합 연산자 (General Set Operations)

수학의 집합 이론에서 가져온 연산자들입니다. 중학교, 고등학교 수학 시간에 배웠던 바로 그 개념들입니다.

| 연산자 이름 | 영문 표기 | 수학 기호 | 설명 |
|---|---|---|---|
| **합집합** | Union | ∪ | 두 테이블의 모든 데이터를 합침 |
| **교집합** | Intersection | ∩ | 두 테이블에 공통으로 있는 데이터만 추출 |
| **차집합** | Difference | − | 한 테이블에만 있는 데이터를 추출 |
| **곱집합** | Cartesian Product | × | 두 테이블의 모든 조합을 생성 |

### 순수 관계 연산자 (Pure Relational Operations)

데이터베이스에 특화되어 새롭게 개발된 연산자들입니다. 일반 수학에는 없고, 관계형 데이터베이스를 위해 만들어진 연산자입니다.

| 연산자 이름 | 영문 표기 | 수학 기호 | SQL 대응 |
|---|---|---|---|
| **셀렉트** | Select | σ (시그마) | WHERE 절 |
| **프로젝트** | Project | π (파이) | SELECT 절의 컬럼 선택 |
| **조인** | Join | ⋈ | JOIN |
| **디비전** | Division | ÷ | 특정 조건을 모두 만족하는 데이터 |

:::info 왜 두 가지로 나누는가?
일반 집합 연산자는 수학에서 빌려온 것이고, 순수 관계 연산자는 데이터베이스를 위해 새로 만든 것입니다. 이렇게 나누어 이해하면 각 연산자의 뿌리를 파악하는 데 도움이 됩니다.
:::

## 🔗 합병 가능성(Union Compatibility)이란?

### 왜 합병 가능성이 필요한가?

일반 집합 연산자 중 **합집합, 교집합, 차집합** 세 가지를 사용하려면 반드시 지켜야 할 조건이 있습니다. 바로 **"합병 가능(Union Compatible)"** 해야 한다는 것입니다.

왜 이런 조건이 필요할까요? 생각해 보세요. 만약 학생 이름 목록과 강좌 시간표를 합치려고 하면 어떻게 될까요? 이름 목록은 "이름, 학번" 두 개의 열을 가지고 있고, 강좌 시간표는 "강좌명, 요일, 시간" 세 개의 열을 가지고 있다면, 이 둘을 합치는 것은 의미가 없겠죠. 마치 사과와 자동차를 더하려는 것처럼요.

### 합병 가능 조건 두 가지

**합병 가능**하려면 다음 두 가지 조건을 모두 만족해야 합니다.

**조건 1️⃣: 차수(Degree)가 같아야 한다**

**차수(Degree)** 란 릴레이션(테이블)의 **속성(열, Column)의 개수**를 말합니다. 마치 서랍장의 칸 수처럼, 두 테이블의 칸 수가 같아야 합니다. R 테이블에 속성이 3개 있으면, S 테이블에도 속성이 3개 있어야 합니다.

**조건 2️⃣: 대응되는 속성의 도메인이 같아야 한다**

**도메인(Domain)** 이란 각 속성이 가질 수 있는 값의 범위, 즉 **데이터 타입**을 말합니다. R 테이블의 첫 번째 열이 숫자(정수형)이면, S 테이블의 첫 번째 열도 숫자(정수형)이어야 합니다. 한쪽은 숫자, 다른 쪽은 문자열이면 합칠 수 없습니다.

:::note 이름은 달라도 된다!
중요한 것은 속성의 **이름은 달라도 된다**는 점입니다. R 테이블의 첫 번째 속성이 "학번"이고, S 테이블의 첫 번째 속성이 "직원번호"이더라도, 둘 다 숫자형이고 두 테이블의 속성 개수가 같다면 합병 가능합니다.
:::

```
[합병 가능한 예시]
테이블 R: (학번:정수, 이름:문자, 학과:문자)  → 차수 3
테이블 S: (사번:정수, 성명:문자, 부서:문자)  → 차수 3
✅ 차수 동일(3=3), 대응 도메인 동일(정수,문자,문자) → 합병 가능!

[합병 불가능한 예시]
테이블 R: (학번:정수, 이름:문자)             → 차수 2
테이블 S: (사번:정수, 성명:문자, 부서:문자)  → 차수 3
❌ 차수 불일치(2≠3) → 합병 불가능!
```

## ✅ 합집합(Union) 연산 완벽 이해

### 합집합이란?

**합집합(Union)** 은 두 릴레이션 R과 S에 있는 **모든 튜플(행)을 합쳐서** 하나의 결과 릴레이션을 만드는 연산입니다. 기호로는 **R ∪ S** 로 표현합니다.

비유하자면, 두 반의 학생 명단을 하나로 합치는 것과 같습니다. A반 명단과 B반 명단을 합치면 전체 학생 명단이 되는 것이죠. 단, 두 반에 모두 이름이 올라가 있는 학생이 있다면(중복), 한 번만 적습니다.

> 📌 **합집합의 핵심 규칙**: 두 테이블에 같은 행이 있으면 중복을 제거하고 한 번만 표시합니다. 이는 릴레이션의 기본 성질인 **"튜플의 유일성"** 때문입니다.

### 합집합의 결과 특성

합집합을 수행했을 때 결과 릴레이션이 어떤 모양이 될지 미리 예측할 수 있습니다.

**📐 결과의 차수(속성 개수)**

결과 릴레이션의 차수는 R의 차수 또는 S의 차수와 동일합니다. 합병 가능 조건에 의해 두 릴레이션의 차수가 같으므로, 결과도 같은 차수를 갖습니다. 예를 들어, R의 차수가 3이고 S의 차수도 3이면, 결과의 차수도 3입니다.

**📊 결과의 카디널리티(튜플 수)**

**카디널리티(Cardinality)** 란 릴레이션의 **튜플(행)의 수**를 말합니다. 합집합의 결과 카디널리티는 다음과 같이 계산됩니다.

```
결과 카디널리티 ≤ R의 카디널리티 + S의 카디널리티
```

왜 등호가 아니라 "작거나 같다(≤)"일까요? 중복이 제거될 수 있기 때문입니다. 만약 R에 5개의 행이 있고 S에도 5개의 행이 있을 때, 겹치는 행이 없다면 결과는 10개, 모두 겹친다면 결과는 5개가 됩니다.

| 상황 | R의 행 수 | S의 행 수 | 결과 행 수 |
|---|---|---|---|
| 중복 없음 | 5 | 5 | 10 |
| 일부 중복 (3개 겹침) | 5 | 5 | 7 |
| 완전 중복 | 5 | 5 | 5 |

### 실제 예시: 학생1 테이블과 학생2 테이블의 합집합

이제 실제 데이터를 가지고 합집합을 직접 수행해 봅시다.

**학생1 테이블 (R)**

| 이름 | 과목 | 점수 |
|---|---|---|
| 오태오 | 자료구조 | 90 |
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |
| 박지훈 | 운영체제 | 75 |

**학생2 테이블 (S)**

| 이름 | 과목 | 점수 |
|---|---|---|
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |
| 김수현 | 알고리즘 | 95 |
| 최민준 | 데이터베이스 | 88 |

두 테이블 모두 차수가 3(이름, 과목, 점수)으로 동일하고, 각 속성의 도메인도 같으니 **합병 가능**합니다.

**합집합 수행 과정:**

1. R의 모든 튜플을 결과에 넣습니다: 오태오/자료구조/90, 이재민/컴퓨터구조/90, 조남선/자료구조/80, 박지훈/운영체제/75
2. S의 튜플을 하나씩 확인하면서, 이미 결과에 없는 것만 추가합니다.
   - 이재민/컴퓨터구조/90 → 이미 있음 → **제거**
   - 조남선/자료구조/80 → 이미 있음 → **제거**
   - 김수현/알고리즘/95 → 없음 → **추가**
   - 최민준/데이터베이스/88 → 없음 → **추가**

**합집합 결과 (R ∪ S)**

| 이름 | 과목 | 점수 |
|---|---|---|
| 오태오 | 자료구조 | 90 |
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |
| 박지훈 | 운영체제 | 75 |
| 김수현 | 알고리즘 | 95 |
| 최민준 | 데이터베이스 | 88 |

- **결과 차수**: 3 (변화 없음)
- **결과 카디널리티**: 6 (4 + 4 - 2개 중복 제거 = 6)

```sql
-- SQL로 표현하면 다음과 같습니다
-- UNION 키워드가 수학의 합집합(∪)에 해당하며, 자동으로 중복을 제거합니다

SELECT 이름, 과목, 점수   -- 학생1 테이블의 모든 열을 선택
FROM 학생1
UNION                     -- 합집합 연산 수행 (중복 자동 제거)
SELECT 이름, 과목, 점수   -- 학생2 테이블의 모든 열을 선택
FROM 학생2;
```

:::tip UNION vs UNION ALL
SQL에서 `UNION`은 중복을 자동으로 제거하지만, `UNION ALL`은 중복을 제거하지 않고 모두 포함합니다. 관계대수의 합집합은 릴레이션의 튜플 유일성 때문에 `UNION`(중복 제거)에 해당합니다.
:::

## 📐 교집합(Intersection), 차집합(Difference), 곱집합(Cartesian Product)

합집합만큼 중요한 나머지 일반 집합 연산자들도 함께 이해해 봅시다.

### 교집합(Intersection) 연산 — R ∩ S

**교집합**은 두 릴레이션 R과 S **모두에 공통으로 존재하는 튜플만** 추출하는 연산입니다. 기호로는 **R ∩ S** 로 표현합니다.

비유하자면, A반과 B반에 동시에 수강 신청을 한 학생들만 찾는 것과 같습니다. 양쪽 명단에 모두 이름이 있는 학생만 결과에 포함됩니다.

**교집합의 결과 특성:**

- **차수**: R 또는 S의 차수와 동일
- **카디널리티**: `결과 카디널리티 ≤ MIN(R의 카디널리티, S의 카디널리티)`
  - 공통된 것만 뽑으니, 작은 테이블의 행 수보다 많이 나올 수 없습니다
- 마찬가지로 **합병 가능** 조건을 만족해야 합니다

**앞선 학생1, 학생2 테이블의 교집합:**

| 이름 | 과목 | 점수 |
|---|---|---|
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |

두 테이블 모두에 존재하는 2개의 행만 결과로 나옵니다.

```sql
-- SQL로 교집합 표현 (일부 DBMS에서 지원)
SELECT 이름, 과목, 점수
FROM 학생1
INTERSECT   -- 교집합 연산: 두 테이블에 모두 있는 행만 선택
SELECT 이름, 과목, 점수
FROM 학생2;
```

### 차집합(Difference) 연산 — R − S

**차집합**은 릴레이션 R에는 있지만 S에는 **없는** 튜플만 추출하는 연산입니다. 기호로는 **R − S** 로 표현합니다.

비유하자면, A반 학생 명단에서 B반 학생과 겹치는 사람을 빼고 A반에만 있는 학생을 찾는 것입니다. 여기서 중요한 점은 **R − S와 S − R은 완전히 다른 결과**를 낸다는 것입니다.

**차집합의 결과 특성:**

- **차수**: R 또는 S의 차수와 동일
- **카디널리티**: `결과 카디널리티 ≤ R의 카디널리티`
  - R에서 일부를 빼는 것이니, R의 행 수보다 많이 나올 수 없습니다

**학생1 − 학생2 (학생1에만 있고 학생2에는 없는 것):**

| 이름 | 과목 | 점수 |
|---|---|---|
| 오태오 | 자료구조 | 90 |
| 박지훈 | 운영체제 | 75 |

**학생2 − 학생1 (학생2에만 있고 학생1에는 없는 것):**

| 이름 | 과목 | 점수 |
|---|---|---|
| 김수현 | 알고리즘 | 95 |
| 최민준 | 데이터베이스 | 88 |

보시다시피 R − S와 S − R은 완전히 다른 결과가 나옵니다.

```sql
-- SQL로 차집합 표현
SELECT 이름, 과목, 점수
FROM 학생1
EXCEPT      -- 차집합 연산: 학생1에 있고 학생2에 없는 행만 선택
SELECT 이름, 과목, 점수
FROM 학생2;
```

:::warning 차집합의 순서 주의!
차집합은 **순서가 매우 중요**합니다. `R − S`와 `S − R`은 전혀 다른 결과를 냅니다. 마치 `5 - 2 = 3`과 `2 - 5 = -3`이 다른 것처럼요. 이것을 "교환 법칙이 성립하지 않는다"고 합니다. 반드시 어느 테이블에서 어느 테이블을 빼는지 명확히 하세요.
:::

### 곱집합(Cartesian Product) 연산 — R × S

**곱집합(카티션 프로덕트, Cartesian Product)** 은 합집합, 교집합, 차집합과 달리 **합병 가능하지 않아도** 수행할 수 있는 연산입니다. R의 **모든 튜플과 S의 모든 튜플을 빠짐없이 모든 조합으로 연결**한 결과를 만들어냅니다. 기호로는 **R × S** 로 표현합니다.

비유하자면, 음식 메뉴와 음료 메뉴를 완전히 조합하는 것과 같습니다. 음식이 3가지(피자, 파스타, 샐러드), 음료가 2가지(콜라, 주스)라면, 가능한 세트 메뉴는 3 × 2 = 6가지가 됩니다.

**곱집합의 결과 특성:**

- **차수**: `결과 차수 = R의 차수 + S의 차수`
  - 두 테이블의 열을 모두 옆으로 붙이므로, 속성 개수가 더해집니다
- **카디널리티**: `결과 카디널리티 = R의 카디널리티 × S의 카디널리티`
  - 모든 조합이 만들어지므로, 행 수가 곱해집니다

**예시:**

테이블 R (차수 3, 카디널리티 4):

| 이름 | 과목 | 점수 |
|---|---|---|
| 오태오 | 자료구조 | 90 |
| 이재민 | 컴퓨터구조 | 90 |
| 조남선 | 자료구조 | 80 |
| 박지훈 | 운영체제 | 75 |

테이블 S (차수 1, 카디널리티 3):

| 학기 |
|---|
| 1학기 |
| 2학기 |
| 여름학기 |

**곱집합 결과 R × S (차수 4, 카디널리티 12)**

| 이름 | 과목 | 점수 | 학기 |
|---|---|---|---|
| 오태오 | 자료구조 | 90 | 1학기 |
| 오태오 | 자료구조 | 90 | 2학기 |
| 오태오 | 자료구조 | 90 | 여름학기 |
| 이재민 | 컴퓨터구조 | 90 | 1학기 |
| 이재민 | 컴퓨터구조 | 90 | 2학기 |
| 이재민 | 컴퓨터구조 | 90 | 여름학기 |
| ... | ... | ... | ... |

이처럼 R의 각 행과 S의 모든 행이 조합되어 4 × 3 = **12개의 행**이 만들어집니다. 차수는 3 + 1 = **4개의 속성**이 됩니다.

:::info 곱집합과 조인의 관계
**조인(Join) = 곱집합(Cartesian Product) + 셀렉트(Select)** 입니다. 즉, 두 테이블의 모든 조합을 만든(곱집합) 다음, 특정 조건을 만족하는 행만 골라내는(셀렉트) 것이 조인입니다. 이것이 조인이 "복합 연산자"인 이유입니다.
:::

## ⚖️ 교환 법칙과 결합 법칙

### 교환 법칙(Commutative Law)

**교환 법칙**이란 피연산자의 순서를 바꾸어도 결과가 같은 성질입니다. 덧셈에서 `2 + 5 = 5 + 2`가 성립하는 것처럼요.

**✅ 교환 법칙이 성립하는 연산자:**

- **합집합**: R ∪ S = S ∪ R
  - R과 S를 합치나 S와 R을 합치나 결과는 동일합니다.
- **교집합**: R ∩ S = S ∩ R
  - 공통 부분을 찾는 것은 순서와 무관합니다.
- **조인(Join)**: R ⋈ S = S ⋈ R
  - 일반적인 내부 조인은 순서에 무관합니다.
- **곱집합**: R × S = S × R
  - 5 × 3 = 3 × 5 = 15, 곱하기는 순서가 달라도 결과가 같습니다.

**❌ 교환 법칙이 성립하지 않는 연산자:**

- **차집합**: R − S ≠ S − R
  - 5 − 2 = 3 이지만, 2 − 5 = -3 으로 다릅니다. 데이터베이스에서도 마찬가지입니다.
- **디비전**: R ÷ S ≠ S ÷ R
  - 나눗셈도 순서가 바뀌면 결과가 달라집니다.
- **세미조인(Semi-Join)**: 한쪽에만 열린 형태의 조인으로, 방향에 따라 결과가 달라집니다.

### 결합 법칙(Associative Law)

**결합 법칙**이란 세 개 이상의 연산에서 어느 것을 먼저 묶어서 계산해도 결과가 같은 성질입니다. 덧셈에서 `(1 + 2) + 3 = 1 + (2 + 3) = 6`이 성립하는 것처럼요.

**✅ 결합 법칙이 성립하는 연산자:** 합집합, 교집합, 조인, 곱집합

**❌ 결합 법칙이 성립하지 않는 연산자:** 차집합, 디비전, 세미조인

:::danger 차집합, 디비전, 세미조인 사용 시 주의!
이 세 연산자는 **교환 법칙도, 결합 법칙도 모두 성립하지 않습니다.** 연산 순서가 바뀌면 완전히 다른 결과가 나올 수 있습니다. 여러 연산을 조합할 때 이 세 연산자가 포함되어 있다면, 반드시 연산 순서를 명확하게 지정해야 합니다.
:::

| 연산자 | 교환 법칙 | 결합 법칙 |
|---|---|---|
| 합집합 (∪) | ✅ 성립 | ✅ 성립 |
| 교집합 (∩) | ✅ 성립 | ✅ 성립 |
| 조인 (⋈) | ✅ 성립 | ✅ 성립 |
| 곱집합 (×) | ✅ 성립 | ✅ 성립 |
| 차집합 (−) | ❌ 불성립 | ❌ 불성립 |
| 디비전 (÷) | ❌ 불성립 | ❌ 불성립 |
| 세미조인 | ❌ 불성립 | ❌ 불성립 |

## 🔬 순수 관계 연산자 살펴보기

일반 집합 연산자 외에도 데이터베이스를 위해 특별히 만들어진 순수 관계 연산자들이 있습니다. 이들은 관계형 데이터베이스의 핵심 기능을 담당합니다.

### 셀렉트(Select) 연산자 — σ (시그마)

**셀렉트(Select)** 연산자는 릴레이션에서 **특정 조건을 만족하는 튜플(행)을 추출**하는 연산입니다. 수학 기호로는 **σ (시그마)** 를 사용합니다.

냉장고에서 재료를 꺼낼 때, "신선한 것만 꺼내라"는 조건을 적용하는 것과 같습니다. 냉장고(테이블) 전체에서 조건을 만족하는 것(신선한 재료 = 조건에 맞는 행)만 골라냅니다.

- **수평적 부분 집합**: 행(가로) 방향으로 데이터를 잘라냅니다
- **SQL 대응**: `WHERE` 절에 해당합니다
- **결과 카디널리티**: `결과 ≤ 원본 릴레이션의 카디널리티`

```sql
-- 관계대수: σ 점수≥80 (학생)
-- SQL 표현: 학생 테이블에서 점수가 80점 이상인 행만 선택

SELECT *            -- 모든 열을 선택
FROM 학생            -- 학생 테이블에서
WHERE 점수 >= 80;   -- 점수가 80 이상인 행만 (이것이 σ 연산에 해당)
```

**셀렉트 연산은 교환적(Commutative)입니다:**

여러 조건을 적용할 때, 어떤 조건을 먼저 적용해도 최종 결과는 동일합니다.

예를 들어 "남학생이면서 서울에 사는 학생"을 찾을 때:
- 방법 1: 남학생 먼저 필터링 → 그 중 서울 거주자 필터링
- 방법 2: 서울 거주자 먼저 필터링 → 그 중 남학생 필터링
- 방법 3: 남학생이면서 서울 조건을 동시에 적용

세 방법 모두 **동일한 결과**를 냅니다.

### 선택도(Selectivity)와 성능 최적화

**선택도(Selectivity)** 란 전체 튜플 중에서 조건을 만족하는 튜플의 비율입니다.

```
선택도 = (조건을 만족하는 튜플 수) / (전체 튜플 수) × 100%
```

예를 들어 학생이 100명인 테이블에서 80점 이상인 학생이 2명이라면, 선택도는 2%입니다.

**선택도가 왜 중요한가?** 쿼리 성능 최적화에 핵심적인 역할을 합니다.

> 🚀 **성능 최적화 원칙**: 선택도가 **작은** 조건(데이터를 많이 걸러내는 조건)을 **먼저** 실행하세요!

**실제 예시로 이해하기:**

100명의 학생 중에서 "남학생이면서 서울에 사는" 학생을 찾는다고 가정합니다.
- 남학생: 5명 (선택도 5%)
- 서울 거주: 60명 (선택도 60%)

❌ **비효율적인 방법**: 서울 거주자 60명 먼저 추출 → 60명 중에서 남학생 5명 찾기
- 60명의 데이터를 메모리에 올린 후 다시 필터링 → 불필요한 메모리와 CPU 사용

✅ **효율적인 방법**: 남학생 5명 먼저 추출 → 5명 중에서 서울 거주자 찾기
- 딱 5명만 다음 필터에 전달되므로 훨씬 빠릅니다

이것이 데이터베이스 엔진이 내부적으로 수행하는 **쿼리 최적화(Query Optimization)** 의 핵심 원리입니다.

### 프로젝트(Project) 연산자 — π (파이)

**프로젝트(Project)** 연산자는 릴레이션에서 **특정 속성(열)만 추출**하는 연산입니다. 수학 기호로는 **π (파이)** 를 사용합니다.

셀렉트가 "어떤 행을 가져올까?"를 결정한다면, 프로젝트는 "어떤 열을 가져올까?"를 결정합니다.

마치 주민등록증에서 이름과 생년월일만 보이게 가리개로 덮는 것과 같습니다. 원본 데이터(주민등록증)는 그대로 있지만, 필요한 정보(이름, 생년월일)만 보여주는 것입니다.

- **수직적 부분 집합**: 열(세로) 방향으로 데이터를 잘라냅니다
- **SQL 대응**: `SELECT` 절에서 특정 컬럼명을 나열하는 것에 해당합니다
- **중복 제거**: 결과에서 중복되는 튜플은 하나만 표시합니다

```sql
-- 관계대수: π 이름,전공 (학생)
-- SQL 표현: 학생 테이블에서 이름과 전공 열만 선택

SELECT 이름, 전공     -- 이름과 전공 열만 선택 (이것이 π 연산에 해당)
FROM 학생;            -- 학생 테이블에서
```

**셀렉트(σ)와 프로젝트(π) 비교:**

| 구분 | 셀렉트(σ) | 프로젝트(π) |
|---|---|---|
| 추출 방향 | 수평(행 기준) | 수직(열 기준) |
| 제어 대상 | 어떤 행을 가져올지 | 어떤 열을 보여줄지 |
| SQL 대응 | WHERE 절 | SELECT 절의 컬럼 목록 |
| 중복 제거 | 기본 적용 | 기본 적용 |

## 🔗 조인(Join)과 디비전(Division) 연산

### 조인(Join) 연산 — ⋈

**조인(Join)** 은 두 개 이상의 릴레이션을 **공통된 속성 값을 기준으로 연결**하여 하나의 결과 릴레이션을 만드는 연산입니다.

현실 세계에서의 비유: 학교에서 학생 카드와 성적 카드가 따로 존재할 때, 학생의 성적을 보려면 두 카드를 학번으로 연결해서 봐야 합니다. 이 연결 작업이 바로 조인입니다.

:::info 조인이 필요한 이유
데이터베이스에서는 데이터 중복을 최소화하기 위해 관련 데이터를 여러 테이블로 나누어 저장합니다(이를 정규화라고 합니다). 그러다 보니 실제 필요한 정보가 여러 테이블에 흩어져 있고, 이것들을 하나로 합쳐서 보기 위해 조인이 필요합니다.
:::

**조인의 종류:**

**① 세타 조인(Theta Join)**: 모든 비교 연산자(=, ≠, >, &lt;, ≥, ≤)를 사용할 수 있는 가장 일반적인 조인입니다. 예를 들어 "A 테이블의 점수가 B 테이블의 기준점수보다 큰 경우"도 조인으로 표현할 수 있습니다.

**② 동일 조인(Equi-Join)**: 세타 조인 중에서 **같다(=)는 조건**만 사용하는 조인입니다. 가장 많이 사용되는 조인 형태이며, 결과에 **같은 속성이 두 번 중복되어** 나타나는 특징이 있습니다.

```
동일 조인 결과 예시:
학번(학생테이블) | 학번(성적테이블) | 이름 | 과목 | 점수
9801            | 9801            | 홍길동 | 수학 | 90
```

두 테이블 모두에서 학번이 나타나기 때문에 중복이 발생합니다.

**③ 자연 조인(Natural Join)**: 동일 조인에서 **중복되는 속성을 제거**하고 하나만 남긴 조인입니다. 실무에서 가장 많이 사용되는 조인 형태입니다.

```
자연 조인 결과 예시:
학번  | 이름   | 과목 | 점수
9801  | 홍길동 | 수학 | 90
```

중복되던 학번 속성이 하나로 합쳐집니다.

**조인과 성능:**

조인은 편리하지만 **성능 비용이 큽니다.** 조인을 수행할 때마다 두 테이블을 서로 비교하는 작업이 필요하기 때문입니다. 조인 횟수가 늘어날수록 성능은 저하됩니다. 따라서 데이터베이스 설계 시 필요한 경우에만 조인을 최소화하도록 설계하는 것이 좋습니다.

### 디비전(Division) 연산 — ÷

**디비전(Division)** 은 R ÷ S 형태로 표현하며, **S의 모든 튜플과 연관된 R의 튜플**을 선택하는 연산입니다. 기호로는 **R ÷ S** 로 표현합니다.

직관적으로 이해하기 어려운 연산이므로 예시로 설명하겠습니다.

**디비전 예시:**

테이블 R(수강 현황):

| 학생 | 과목 |
|---|---|
| A학생 | 수학 |
| A학생 | 영어 |
| A학생 | 과학 |
| B학생 | 수학 |
| B학생 | 영어 |
| C학생 | 수학 |

테이블 S(필수 과목):

| 과목 |
|---|
| 수학 |
| 영어 |

**R ÷ S 질문**: "S의 모든 과목(수학, 영어)을 다 수강한 학생은?"

- A학생: 수학 ✅, 영어 ✅ → **포함**
- B학생: 수학 ✅, 영어 ✅ → **포함**
- C학생: 수학 ✅, 영어 ❌ → **제외**

**결과:**

| 학생 |
|---|
| A학생 |
| B학생 |

이처럼 디비전은 "모든 조건을 전부 만족하는" 데이터를 찾을 때 사용합니다.

**디비전의 차수 계산:**
- R의 차수가 m+n이고, S의 차수가 n이면, 결과의 차수는 m입니다
- 위 예시: R 차수 2(학생+과목), S 차수 1(과목), 결과 차수 1(학생) → 2-1=1

## 🏗️ 기본 연산자 vs 복합 연산자

### 기본 연산자(Primitive Operators)

**기본 연산자**란 다른 연산자로는 표현할 수 없는, 그 자체로 독립적인 연산자를 말합니다. 이것들은 관계대수의 기본 블록(Building Block)입니다.

**기본 연산자 목록:**
- **합집합 (∪)**: 다른 연산자의 조합으로 표현 불가
- **차집합 (−)**: 다른 연산자의 조합으로 표현 불가
- **곱집합 (×)**: 다른 연산자의 조합으로 표현 불가
- **셀렉트 (σ)**: 다른 연산자의 조합으로 표현 불가
- **프로젝트 (π)**: 다른 연산자의 조합으로 표현 불가

### 복합 연산자(Derived Operators)

**복합 연산자**란 기본 연산자들의 조합으로 표현할 수 있는 연산자입니다.

- **조인 (⋈)** = **곱집합(×) + 셀렉트(σ)**
  - 두 테이블의 모든 조합(곱집합)에서 조건을 만족하는 것(셀렉트)만 추출
  
- **교집합 (∩)** = 차집합(−) 연산자로 표현 가능
  - `R ∩ S = R − (R − S)` 또는 `R ∩ S = S − (S − R)`
  
- **디비전 (÷)** = 차집합(−) + 곱집합(×)으로 표현 가능

| 구분 | 연산자 | 설명 |
|---|---|---|
| **기본 연산자** | ∪, −, ×, σ, π | 다른 것으로 대체 불가 |
| **복합 연산자** | ⋈, ∩, ÷ | 기본 연산자로 표현 가능 |

## 🎯 관계대수 연산자 우선순위

여러 연산자를 괄호 없이 연달아 사용할 때, 어떤 것을 먼저 계산해야 할지 우선순위가 정해져 있습니다. 마치 수식에서 곱하기를 더하기보다 먼저 계산하는 것처럼요.

**우선순위 (높음 → 낮음):**

1. **π (프로젝트)** — 가장 높은 우선순위
2. **σ (셀렉트)** — 두 번째
3. **× (곱집합), ⋈ (조인), ÷ (디비전)** — 세 번째
4. **− (차집합)** — 네 번째
5. **∪ (합집합), ∩ (교집합)** — 가장 낮은 우선순위

예를 들어 `π 이름 σ 점수≥80 학생 × 성적` 이라는 표현이 있다면:

1. 먼저 `학생 × 성적` (곱집합)
2. 그 결과에 `σ 점수≥80` (셀렉트) 적용
3. 마지막으로 `π 이름` (프로젝트) 적용

:::tip 괄호를 적극 활용하세요
연산자 우선순위가 헷갈릴 때는 괄호를 명시적으로 사용하면 실수를 줄일 수 있습니다. 우선순위를 암기하는 것보다 괄호로 명확히 표현하는 습관이 더 중요합니다.
:::

## ⚠️ 주의사항 및 핵심 팁

### 합병 가능 조건 잊지 말기

합집합, 교집합, 차집합을 사용할 때는 반드시 두 테이블이 합병 가능한지 먼저 확인하세요.

❌ **잘못된 예시:**
```
테이블 A: (학번, 이름, 학과)           -- 차수 3
테이블 B: (과목코드, 과목명)           -- 차수 2

A ∪ B  →  오류! 차수가 다름 (3 ≠ 2)
```

✅ **올바른 예시:**
```
테이블 A: (학번, 이름, 학과)           -- 차수 3
테이블 B: (직원번호, 성명, 부서)       -- 차수 3

A ∪ B  →  가능! 차수 동일, 도메인 동일
```

### 중복 제거는 자동으로

합집합, 교집합, 프로젝트 결과에서 중복 튜플은 자동으로 하나만 남겨집니다. 이것은 릴레이션의 기본 성질이므로 별도로 처리할 필요가 없습니다.

### 차집합과 디비전의 순서 주의

이 두 연산자는 교환 법칙과 결합 법칙이 모두 성립하지 않습니다. 연산 순서가 바뀌면 완전히 다른 결과가 나오므로, 항상 어느 테이블을 기준으로 연산하는지 명확히 해야 합니다.

### 조인은 성능 비용이 크다

조인은 매우 강력하지만, 조인 횟수가 늘어날수록 데이터베이스의 성능이 저하됩니다. 실무에서는 필요한 경우에만 조인을 사용하고, 가능하면 단일 테이블에서 결과를 얻을 수 있도록 설계하는 것이 좋습니다.

:::warning 선택도를 항상 고려하세요
여러 조건이 있을 때는 **선택도가 낮은(더 많이 걸러내는) 조건을 먼저** 적용하세요. 이것만으로도 쿼리 성능을 크게 향상시킬 수 있습니다. 데이터베이스 옵티마이저(최적화 엔진)가 자동으로 해주기도 하지만, 개발자가 이 원리를 이해하고 있으면 더 효율적인 쿼리를 작성할 수 있습니다.
:::

## 📝 핵심 정리

이 문서에서 배운 내용을 한눈에 정리합니다.

- **관계대수**란 릴레이션(테이블)을 대상으로 데이터를 추출하는 절차적인 연산의 집합이다.
- **릴레이션(Relation)** 은 수학적 용어로, 실제 데이터베이스의 테이블(Table)과 같은 개념이다.
- 관계대수 연산자는 **일반 집합 연산자**(합집합, 교집합, 차집합, 곱집합)와 **순수 관계 연산자**(셀렉트, 프로젝트, 조인, 디비전)로 나뉜다.
- **합병 가능(Union Compatible)** 조건: 차수(속성 개수)가 같고, 대응 속성의 도메인(데이터 타입)이 같아야 하며, 이름은 달라도 된다.
- **합집합(∪)**: 두 릴레이션의 모든 튜플을 합치며, 중복 튜플은 자동 제거된다.
- **결과 카디널리티**: 합집합 ≤ R+S, 교집합 ≤ MIN(R,S), 차집합 ≤ R, 곱집합 = R×S.
- **교환 법칙**은 합집합, 교집합, 조인, 곱집합에서 성립하고, **차집합, 디비전, 세미조인**에서는 성립하지 않는다.
- **결합 법칙** 역시 합집합, 교집합, 조인, 곱집합에서 성립하고, **차집합, 디비전, 세미조인**에서는 성립하지 않는다.
- **셀렉트(σ)** 는 수평적 부분 집합(행 추출)이며 SQL의 WHERE 절에 해당한다.
- **프로젝트(π)** 는 수직적 부분 집합(열 추출)이며 SQL의 SELECT 절 컬럼 목록에 해당한다.
- **선택도(Selectivity)** 가 낮은 조건을 먼저 실행하면 쿼리 성능을 최적화할 수 있다.
- **조인(⋈)** = 곱집합(×) + 셀렉트(σ)이며, 자연 조인은 동일 조인에서 중복 속성을 제거한 것이다.
- **기본 연산자**(∪, −, ×, σ, π)는 다른 연산자로 대체 불가능하고, **복합 연산자**(⋈, ∩, ÷)는 기본 연산자로 표현 가능하다.
- **연산자 우선순위**: π > σ > ×, ⋈, ÷ > − > ∪, ∩ 순서로 처리된다.
