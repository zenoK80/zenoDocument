# 💰 소프트웨어 개발 비용과 규모 측정 (LOC, MM)

소프트웨어를 개발하려고 마음먹었을 때 가장 먼저 마주치는 현실적인 질문이 있습니다.

> **"이 프로그램 만드는 데 얼마나 걸려요? 비용은 얼마나 들어요?"**

이 질문에 답하기 위해서는 소프트웨어의 **규모**를 먼저 측정할 수 있어야 합니다. 집을 지을 때 평수와 층수를 먼저 파악해야 건축 비용을 계산할 수 있듯이, 소프트웨어도 그 크기와 복잡도를 측정하는 기준이 필요합니다. 이 문서에서는 소프트웨어 공학에서 가장 널리 쓰이는 두 가지 규모 측정 단위인 **LOC(Lines of Code, 코드 라인 수)** 와 **MM(Man-Month, 맨 먼스)** 를 중심으로, 소프트웨어 개발 비용이 어떻게 결정되는지 아주 자세히 살펴보겠습니다.

---

## 🏗️ 소프트웨어 개발, 왜 이렇게 비쌀까?

### 소프트웨어는 보이지 않는 제품이다

우리가 일상에서 사용하는 물건들, 예를 들어 냉장고, 자동차, 스마트폰 같은 하드웨어 제품은 눈에 보이고 손으로 만질 수 있습니다. 그래서 "이 냉장고는 500리터짜리니까 이 정도 가격이겠구나"라는 판단이 어느 정도 가능하죠.

그런데 소프트웨어는 **눈에 보이지 않는 제품**입니다. 완성된 쇼핑몰 앱을 보면 그냥 화면이 예쁘게 잘 돌아가는 것처럼 보이지만, 그 뒤에 수십만 줄의 코드와 수십 명의 개발자의 노력이 숨어 있습니다. 이처럼 소프트웨어의 규모를 외부에서 직접 가늠하기 어렵기 때문에, 개발 비용을 산정하는 것이 대단히 어렵고 복잡한 작업이 됩니다.

### 소프트웨어 개발 비용의 현실

소프트웨어 개발 비용은 여러분이 생각하는 것보다 훨씬 큽니다. 실무에서 프로젝트를 수행해 보면, 규모에 따라 다음과 같은 비용이 발생합니다.

| 프로젝트 규모 | 예상 비용 범위 | 예시 |
|---|---|---|
| 소규모 | 3천만 원 ~ 1억 원 | 소형 쇼핑몰, 사내 업무 시스템 |
| 중규모 | 1억 원 ~ 5억 원 | 중형 플랫폼, 통합 ERP 시스템 |
| 대규모 | 5억 원 ~ 수백억 원 | 정부 공공 시스템, 국가 인프라 |

이렇게 비용이 비싸지는 이유는 결국 **사람의 시간과 노력**이 들어가기 때문입니다. 소프트웨어 개발은 자동화된 공장에서 물건을 찍어내는 방식이 아니라, 숙련된 개발자들이 머리를 써서 한 줄 한 줄 코드를 작성하는 지식 집약적인 작업입니다.

:::info 소프트웨어 비용이 비싼 이유
소프트웨어 개발에서 가장 큰 비중을 차지하는 비용은 바로 **인건비(人件費)** 입니다. 숙련된 개발자 한 명의 연봉은 수천만 원에서 억대에 달하며, 대형 프로젝트에는 수십 명의 개발자가 수개월에서 수년간 투입됩니다. 여기에 테스트, 기획, 설계, 유지보수 비용까지 더해지면 최종 비용은 상당히 커집니다.
:::

## 📏 LOC (Lines of Code) — 코드 라인 수로 규모 측정하기

### LOC란 무엇인가?

**LOC(Lines of Code)** 는 말 그대로 **소프트웨어를 구성하는 코드의 총 줄 수**를 의미합니다. 소프트웨어의 규모를 측정하는 가장 전통적이고 가장 널리 사용되는 방법입니다.

마치 소설의 분량을 "총 400페이지 분량"이라고 표현하듯이, 소프트웨어의 규모를 "총 5만 라인(줄)짜리 프로그램"이라고 표현하는 것이 LOC 방식입니다.

> **LOC = 소프트웨어 전체 소스 코드의 총 줄 수**

### LOC를 왜 써야 하는가?

소프트웨어 개발 비용을 계산하려면 먼저 그 소프트웨어가 얼마나 큰지 알아야 합니다. LOC는 이 '크기'를 숫자로 표현할 수 있게 해주는 매우 실용적인 도구입니다.

예를 들어, 클라이언트가 "쇼핑몰 하나 만들고 싶어요"라고 하면 개발자는 막연히 어렵다고만 할 수 없습니다. 기능 목록을 분석한 뒤 "이 쇼핑몰의 규모는 대략 5만 라인 정도 될 것 같습니다"라고 구체적인 수치를 제시할 수 있어야, 그다음 단계인 비용과 일정 산정이 가능해집니다.

### LOC 코드 예시로 이해하기

아래는 Python으로 작성한 간단한 코드 예시입니다. 줄 수가 어떻게 세어지는지 직접 확인해 보세요.

```python
# 학생 성적 관리 시스템 - 간단한 예시
# 이 파일의 LOC: 실행 코드 기준으로 세면 약 15줄

# 학생 정보를 저장하는 딕셔너리 (LOC 1줄)
students = {}

def add_student(name, score):
    """학생 이름과 점수를 등록하는 함수"""
    # 학생 이름을 키로, 점수를 값으로 저장 (LOC 1줄)
    students[name] = score
    # 등록 완료 메시지 출력 (LOC 1줄)
    print(f"{name} 학생이 등록되었습니다. 점수: {score}")

def get_average():
    """전체 학생의 평균 점수를 계산하는 함수"""
    # 학생이 없는 경우 예외 처리 (LOC 1줄)
    if not students:
        return 0
    # 전체 점수의 합을 학생 수로 나눠 평균 계산 (LOC 1줄)
    return sum(students.values()) / len(students)

# 학생 데이터 추가 (각 1줄씩)
add_student("홍길동", 85)
add_student("김영희", 92)
add_student("이철수", 78)

# 평균 출력 (LOC 1줄)
print(f"전체 평균 점수: {get_average():.1f}점")
```

이 코드는 약 15~20줄 수준의 아주 작은 프로그램입니다. 그런데 실제 학교에서 사용하는 학생 관리 시스템은 **로그인, 권한 관리, 데이터베이스 연동, 통계 분석, 학부모 알림, 출석 관리** 등 수백 가지 기능이 포함되어 수만 ~ 수십만 줄에 달하게 됩니다.

### LOC 측정의 현실적인 예시

```python
# ❌ 잘못된 LOC 이해: "코드 줄이 많으면 무조건 좋은 프로그램이다"
# 아래처럼 불필요하게 줄을 늘리는 것은 LOC를 부풀릴 뿐 품질과 무관합니다

def add(a, b):

    result = a + b

    return result  # 빈 줄을 넣어 LOC를 늘렸지만 실제 가치는 없음


# ✅ 올바른 LOC 이해: 의미 있는 기능 코드의 줄 수를 세는 것
# 실제로는 주석, 빈 줄을 제외한 실행 코드(소스 라인)만 세거나
# 전체 줄 수를 세는 방식 중 프로젝트 기준에 따라 선택합니다

def add(a, b):
    # 두 숫자를 더해서 반환하는 함수
    return a + b  # 간결하고 명확한 코드가 품질이 높은 코드
```

:::warning LOC는 품질 지표가 아닙니다
LOC는 **규모(크기)를 측정**하는 지표일 뿐, **코드의 품질이나 우수성**을 나타내지 않습니다. 코드가 1만 줄이라도 엉망으로 작성된 것보다, 5천 줄이라도 효율적으로 작성된 코드가 훨씬 우수할 수 있습니다. LOC는 어디까지나 규모 산정과 비용 계산을 위한 도구입니다.
:::

## 👷 MM (Man-Month) — 인력과 시간으로 규모 측정하기

### MM(맨 먼스)란 무엇인가?

**MM(Man-Month, 맨 먼스)** 는 소프트웨어를 개발하는 데 **얼마나 많은 인력이 얼마나 오랫동안 필요한지**를 나타내는 단위입니다.

공식으로 표현하면 다음과 같습니다.

> **MM (맨 먼스) = 투입 인원 수 × 개발 기간(월)**

쉽게 말해서, 1 MM은 **개발자 1명이 1개월 동안 할 수 있는 작업량**입니다. 마치 노동 현장에서 "이 공사는 인부 10명이 한 달 동안 해야 끝난다"라고 말하는 것처럼, 소프트웨어 개발도 "이 프로젝트는 10 MM 규모다"라고 표현할 수 있습니다.

### MM 계산 예시

```
예시 1) MM = 1
→ 개발자 1명이 1개월 동안 완성할 수 있는 규모의 소프트웨어

예시 2) MM = 10
→ 개발자 10명이 1개월 동안 완성할 수 있는 규모
  또는 개발자 1명이 10개월 동안 완성할 수 있는 규모
  또는 개발자 5명이 2개월 동안 완성할 수 있는 규모

예시 3) MM = 30
→ 개발자 5명이 6개월 동안 완성할 수 있는 규모
  또는 개발자 10명이 3개월 동안 완성할 수 있는 규모
```

이처럼 MM은 인원과 기간을 곱한 값이기 때문에, 같은 MM이라도 팀 구성을 어떻게 하느냐에 따라 다양한 방식으로 프로젝트를 운영할 수 있습니다.

### MM을 실제 비용으로 환산하기

MM이 왜 중요하냐면, 바로 **인건비와 직결**되기 때문입니다. 개발자 한 명의 월급을 알면 MM으로부터 인건비를 바로 계산할 수 있습니다.

```
[비용 계산 예시]

- 프로젝트 규모: 30 MM
- 개발자 평균 월 인건비: 500만 원

→ 순수 인건비 = 30 MM × 500만 원 = 1억 5천만 원

여기에 추가로:
  + 기획/분석 비용
  + 테스트 비용
  + 유지보수 비용
  + 서버/인프라 비용
  + 관리/행정 비용

= 최종 프로젝트 총 비용
```

:::tip 비용 계산은 인건비만으로 끝이 아닙니다
실무에서 프로젝트 예산을 산정할 때, 순수 인건비는 전체 비용의 60~70% 수준이며, 나머지 30~40%는 기획 문서 작성, 테스트, 유지보수, 서버 운영, 교육, 행정 비용 등으로 채워집니다. 따라서 LOC와 MM으로 계산한 인건비는 최종 예산의 출발점이지 종착점이 아닙니다.
:::

## 📊 소프트웨어 생산성 — LOC와 MM의 연결고리

### 생산성(Productivity)이란 무엇인가?

**소프트웨어 생산성**은 **1 MM(개발자 1명이 한 달 동안) 동안 생산하는 코드의 LOC**를 의미합니다.

> **생산성 = LOC ÷ MM**

예를 들어, 개발자 A가 한 달 동안 10,000줄의 코드를 생산했다면 생산성은 10,000 LOC/MM입니다. 개발자 B가 한 달 동안 2,000줄의 코드를 생산했다면 생산성은 2,000 LOC/MM이 됩니다.

### 생산성 차이가 비용에 미치는 영향

생산성이 높은 개발자는 같은 시간에 더 많은 코드를 작성합니다. 이는 곧 같은 규모의 프로젝트를 더 빠르게, 더 적은 인원으로 완성할 수 있다는 뜻입니다. 반대로 생산성이 낮은 개발자들로 팀을 구성하면 더 많은 시간과 인원이 필요해져 비용이 크게 늘어납니다.

```
[생산성에 따른 비용 비교 예시]

프로젝트 규모: 총 10만 LOC (100,000줄)

📌 시나리오 A - 고급 개발자 팀
생산성: 10,000 LOC/MM (한 명이 한 달에 1만 줄)
필요 MM: 100,000 ÷ 10,000 = 10 MM
→ 5명이서 2개월 완성 가능
→ 인건비: 10 MM × 500만 원 = 5,000만 원

📌 시나리오 B - 초급 개발자 팀
생산성: 2,000 LOC/MM (한 명이 한 달에 2천 줄)
필요 MM: 100,000 ÷ 2,000 = 50 MM
→ 5명이서 10개월 걸림
→ 인건비: 50 MM × 300만 원 = 1억 5,000만 원
```

이처럼 생산성 차이 하나가 **비용을 3배 이상** 차이나게 만들 수 있습니다. 이 때문에 숙련된 고급 개발자는 초급 개발자보다 훨씬 높은 연봉을 받으며, 결과적으로 기업 입장에서도 오히려 비용 효율이 좋을 수 있습니다.

### 초급 · 중급 · 고급 개발자의 생산성 차이

| 개발자 등급 | 평균 생산성 (LOC/MM) | 특징 |
|---|---|---|
| 초급 개발자 | 1,000 ~ 3,000 | 기본 기능 구현 중심, 코드 리뷰 필요 |
| 중급 개발자 | 3,000 ~ 7,000 | 설계부터 구현까지 독립적으로 가능 |
| 고급 개발자 | 7,000 ~ 15,000+ | 아키텍처 설계, 성능 최적화, 팀 리딩 가능 |

:::note 생산성 수치는 절대적인 기준이 아닙니다
위 표의 수치는 프로젝트 성격, 사용하는 프로그래밍 언어, 개발 환경, 팀 문화에 따라 크게 달라질 수 있습니다. 예를 들어, 단순 반복적인 화면 개발은 LOC가 높게 나오지만, 알고리즘을 최적화하는 작업은 몇 줄 안 되는 코드로 엄청난 가치를 만들어내기도 합니다. 따라서 생산성 지표는 참고용으로 활용해야 합니다.
:::

## 🔢 실제 프로젝트에서 LOC와 MM 적용하기

### 쇼핑몰 개발 비용 산정 실습

실제 현장에서 클라이언트가 개발을 의뢰할 때 어떤 과정으로 비용을 산정하는지 단계별로 살펴보겠습니다.

**[클라이언트 요구사항]**
- 상품 목록 및 상세 페이지
- 장바구니 및 주문 기능
- 재고 관리 시스템
- 관리자 페이지 (상품 등록/수정/삭제)
- 주문 시 SMS 자동 발송
- 배송 추적 기능
- 회원 가입 및 로그인

**[개발자의 규모 분석]**

```
각 기능별 예상 LOC 산정:

1. 회원 가입/로그인/권한 관리         → 약 5,000 LOC
2. 상품 목록/상세/검색 기능          → 약 8,000 LOC
3. 장바구니 및 주문 처리             → 약 10,000 LOC
4. 재고 관리 시스템                 → 약 7,000 LOC
5. 관리자 페이지 전체               → 약 12,000 LOC
6. SMS 자동 발송 연동               → 약 3,000 LOC
7. 배송 추적 API 연동               → 약 5,000 LOC
─────────────────────────────────────
총 예상 LOC                        ≈ 50,000 LOC (5만 라인)
```

**[비용 산정]**

```
총 LOC: 50,000줄
팀 평균 생산성: 5,000 LOC/MM (중급 개발자 기준)

필요 MM = 50,000 ÷ 5,000 = 10 MM

팀 구성 예시:
  → 개발자 5명 × 2개월 = 10 MM

개발자 평균 월 인건비: 600만 원
순수 인건비 = 10 MM × 600만 원 = 6,000만 원

기타 비용 (테스트, 기획, 설계, 인프라 등) 약 40% 추가:
= 6,000만 원 × 1.4 ≈ 8,400만 원

최종 견적: 약 8,000만 원 ~ 1억 원 수준
```

이처럼 LOC와 MM을 활용하면 단순히 "얼마예요?"라는 질문에 **근거 있는 숫자**로 답할 수 있게 됩니다. 물론 이는 예상 수치이며, 실제 개발 과정에서 요구사항 변경, 기술적 난이도, 일정 지연 등으로 인해 달라질 수 있습니다.

### 5만 LOC 프로젝트의 실제 비용 범위

현업에서 약 5만 라인 규모의 프로젝트를 진행한다고 하면 일반적으로 다음과 같은 비용 범위가 나타납니다.

| 구성 | 비용 |
|---|---|
| 최소 인건비 기준 | 약 4,000만 원 |
| 일반적인 중간값 | 약 8,000만 원 ~ 1억 원 |
| 고급 개발팀 + 완벽한 테스트/문서화 포함 | 약 1억 2,000만 원 이상 |

이 범위가 이렇게 넓게 형성되는 이유는 개발팀의 역량, 사용하는 기술 스택, 테스트 수준, 문서화 요구 정도, 프로젝트 관리 방식에 따라 비용이 크게 달라지기 때문입니다.

## ⚠️ 소프트웨어 개발 비용 산정 시 주의사항과 현실적인 문제들

### 지연과 낮은 신뢰도 — 예산 초과의 단골 원인

실제 현장에서 소프트웨어 프로젝트를 조사해 보면 **35% 이상이 예정된 기간을 초과**한다는 통계가 있습니다. "한 달이면 끝난다"고 했던 프로젝트가 두 달, 세 달로 늘어나는 일은 소프트웨어 업계에서 매우 흔한 일입니다.

이런 지연이 발생하는 주요 원인은 다음과 같습니다.

- **요구사항 변경**: 개발 도중 클라이언트가 기능을 추가하거나 바꾸는 경우
- **기술적 난제**: 예상보다 복잡한 로직 구현에 시간이 걸리는 경우
- **통합 문제**: 여러 모듈을 합치는 과정에서 예상치 못한 오류 발생
- **테스트 기간 과소평가**: 테스트에 충분한 시간을 할당하지 않은 경우
- **인력 변동**: 핵심 개발자가 중도에 이탈하는 경우

:::warning 예산 책정 시 반드시 여유분을 두세요
현업 경험에서 보면, 소프트웨어 프로젝트의 초기 예산은 **실제 소요되는 비용의 70~80% 수준**으로 책정되는 경우가 많습니다. 따라서 LOC와 MM으로 계산한 예산에서 **최소 20~30%의 여유 예산(버퍼)**을 추가로 확보하는 것이 현실적인 프로젝트 관리입니다. 일정도 마찬가지로 예상 기간의 1.2배 ~ 1.5배를 확보해 두는 것이 안전합니다.
:::

### 소프트웨어는 하드웨어처럼 닳지 않는다

소프트웨어에 대한 가장 흔한 오해 중 하나는 **"오래 쓰면 고장 난다"** 는 생각입니다. 자동차는 오래 타면 엔진이 마모되고, 스마트폰은 오래 쓰면 배터리가 소모됩니다. 이런 물리적 노후화 개념을 소프트웨어에 그대로 적용하면 안 됩니다.

소프트웨어는 오래 사용한다고 해서 **물리적으로 닳거나 마모되지 않습니다**. 아래 한글 프로그램을 10년 동안 사용했다고 해서 프로그램 파일 자체가 낡아지지는 않습니다. 소프트웨어의 오류는 대부분 다음 두 가지 이유로 발생합니다.

1. **설계나 개발 단계에서 심어진 잠재적 버그**: 처음부터 오류가 존재했지만, 특정 입력 조건이 발생하기 전까지는 발견되지 않았을 뿐입니다.
2. **환경 변화에 의한 부적합**: 운영 체제, 하드웨어, 네트워크 환경이 바뀌면서 기존 소프트웨어와 충돌이 생기는 경우입니다.

:::danger 신뢰도 문제는 생명과 재산에 직결됩니다
미사일 발사 시스템, 비행기 제어 소프트웨어, 자동차 급발진 제어 시스템 — 이런 분야에서 소프트웨어의 신뢰도 문제는 단순한 불편함을 넘어 인명 피해로 이어질 수 있습니다. 실제로 우주 로켓의 초기 발사 실패, 자동차 급발진 사고 등 많은 경우에서 하드웨어보다 소프트웨어의 신뢰성 문제가 원인으로 지목됩니다. 바로 이것이 소프트웨어 공학이 필요한 핵심 이유입니다.
:::

### 유지보수 비용이 개발 비용보다 크다

많은 분들이 "소프트웨어는 한 번 만들면 끝"이라고 생각하지만, 실제로는 **유지보수에 드는 총 비용이 초기 개발 비용을 상회하는 경우가 매우 많습니다.**

자동차가 고장 나면 해당 부품만 교체하면 되듯이, 소프트웨어도 문제 있는 부분만 딱 잘라내고 교체하면 좋겠지만 현실은 그렇지 않습니다. 소프트웨어는 수많은 모듈이 **서로 긴밀하게 연결**되어 있기 때문에, 한 부분을 수정하면 다른 부분에 예상치 못한 영향을 주는 경우가 빈번합니다. 이를 **사이드 이펙트(부작용)** 라고 부릅니다.

```python
# ❌ 잘못된 유지보수 접근: 한 부분만 수정했더니 다른 곳에서 오류 발생

def calculate_discount(price, member_type):
    # 기존 코드: 일반 회원은 5% 할인
    if member_type == "normal":
        return price * 0.95

# 나중에 VIP 회원 추가를 위해 수정했더니...
def calculate_discount(price, member_type):
    if member_type == "normal":
        return price * 0.95
    elif member_type == "vip":
        return price * 0.80
    # 문제: 기존에 "premium" 타입을 사용하던 다른 모듈이 있었는데
    # 그 모듈이 갑자기 할인 없이 원가로 계산되는 버그 발생!


# ✅ 올바른 유지보수 접근: 영향 범위를 먼저 파악하고 수정

# 1단계: 이 함수를 호출하는 모든 곳을 파악한다
# 2단계: 기존 동작이 바뀌지 않도록 하위 호환성을 유지한다
# 3단계: 테스트 케이스로 기존 기능이 정상 동작함을 검증한다
# 4단계: 그 후에 새 기능을 추가한다

def calculate_discount(price, member_type):
    """
    회원 유형에 따른 할인 가격 계산
    - normal: 5% 할인
    - premium: 10% 할인 (기존 타입 유지)
    - vip: 20% 할인 (신규 추가)
    """
    discount_rates = {
        "normal": 0.95,
        "premium": 0.90,   # 기존 타입을 삭제하지 않고 유지
        "vip": 0.80         # 새로운 타입 추가
    }
    # 정의되지 않은 타입은 할인 없이 원가 반환 (안전장치)
    rate = discount_rates.get(member_type, 1.0)
    return price * rate
```

이처럼 소프트웨어 유지보수는 단순히 "잘못된 줄만 고치면 되는 것"이 아니라, **전체 시스템에 미치는 영향을 철저히 분석한 뒤 신중하게 수정**해야 하는 복잡한 작업입니다. 이것이 유지보수 비용이 생각보다 훨씬 크게 드는 이유입니다.

### 과거 경험이 그대로 재활용되기 어렵다

요리를 잘하는 사람이 라면 끓이는 법을 한 번 제대로 배우면 그 다음부터는 눈 감고도 맛있게 끓일 수 있습니다. 노하우가 그대로 적용되기 때문입니다. 그런데 소프트웨어 개발은 그렇지 않습니다.

과거에 성공적으로 완성한 쇼핑몰 프로젝트의 경험이 있다고 해서, 새로운 쇼핑몰 프로젝트에 과거 코드를 그대로 가져다 쓰기가 어렵습니다. 그 이유는 다음과 같습니다.

- **사용자 요구사항이 달라진다**: 새 클라이언트는 다른 기능과 다른 디자인을 원합니다.
- **기술 환경이 변화한다**: 3년 전에 쓰던 프레임워크(개발 도구 묶음)는 이미 구식이 되어 있을 수 있습니다.
- **개발 방법론이 발전한다**: 더 효율적인 개발 방식이 계속 등장합니다.
- **법적/보안 요구사항이 강화된다**: 개인정보 보호법, 보안 규정 등이 계속 바뀝니다.

이 때문에 소프트웨어 개발자는 **끊임없는 자기 계발**이 필수입니다. 지금 잘 아는 기술도 3~5년 후에는 시대에 뒤떨어질 수 있기 때문에, 새로운 기술과 방법론을 지속적으로 학습하는 자세가 매우 중요합니다.

### 개발자 개인 역량이 프로젝트 성패를 좌우한다

같은 "10년 경력 자바 개발자"라도 역량 차이는 극과 극일 수 있습니다. 뛰어난 개발자는 문제를 빠르게 파악하고 효율적인 구조로 코드를 작성하는 반면, 그렇지 못한 개발자는 같은 문제를 해결하는 데 몇 배의 시간이 걸리고 유지보수하기 어려운 코드를 남기기도 합니다.

이 차이는 단순히 연봉 차이로만 끝나지 않습니다. 프로젝트 전체의 성공 여부에 영향을 미칩니다. 그래서 좋은 개발팀을 구성하는 것이 비용을 아끼는 것 못지않게 중요하며, 뛰어난 개발자에게 높은 연봉을 주는 것이 결과적으로 기업에 더 이득이 될 수 있습니다.

## 💡 LOC와 MM 활용 시 실전 팁

### 기능 목록으로 LOC를 먼저 추정하라

실무에서 LOC를 정확하게 미리 알기는 어렵습니다. 개발하기 전에는 실제로 몇 줄이 될지 모르기 때문입니다. 그래서 전문가들은 **기능 목록(Function List)** 을 기반으로 LOC를 추정합니다.

```
기능 하나당 예상 LOC 기준 (일반적인 웹 애플리케이션 기준):

- 로그인/로그아웃 기능        → 약 500 ~ 1,000 LOC
- 게시판 CRUD (등록/조회/수정/삭제) → 약 1,500 ~ 3,000 LOC
- 파일 업로드/다운로드 기능    → 약 500 ~ 1,500 LOC
- 결제 모듈 연동              → 약 2,000 ~ 5,000 LOC
- 이메일/SMS 발송 기능        → 약 300 ~ 1,000 LOC
- 통계/리포트 기능            → 약 2,000 ~ 4,000 LOC
```

이 수치들은 경험에 기반한 추정치이며, 시스템의 복잡도와 요구사항에 따라 크게 달라질 수 있습니다.

### MM 계산 시 버퍼(여유분)를 반드시 포함하라

앞서 언급했듯이, 소프트웨어 프로젝트는 지연되는 경우가 매우 많습니다. 따라서 순수 계산으로 나온 MM에 **반드시 여유분을 더해야** 합니다.

```
[MM 계산 시 권장 버퍼 비율]

기본 개발 MM: 10 MM

+ 테스트 및 디버깅 (기본 MM의 20~30%): +2 ~ 3 MM
+ 요구사항 변경 대응 (기본 MM의 10~20%): +1 ~ 2 MM
+ 문서화 및 교육 (기본 MM의 10%): +1 MM
+ 예비 버퍼 (기본 MM의 10~15%): +1 ~ 1.5 MM
─────────────────────────────────────────────
최종 권장 MM: 15 ~ 17.5 MM (기본의 1.5배 ~ 1.75배)
```

:::tip 일정 지연을 대비하는 전문가의 자세
경험 많은 PM(프로젝트 매니저)은 처음부터 "이 프로젝트는 반드시 지연이 생길 것"을 전제로 계획을 세웁니다. 이것은 패배주의가 아니라, 소프트웨어 개발의 복잡성을 현실적으로 인식하는 전문가적 태도입니다. "계획대로만 되면 정말 행복한 프로젝트"라는 말이 현장에서 자주 나오는 이유가 여기에 있습니다.
:::

### 생산성 데이터를 꾸준히 기록하라

개발팀이 지속적으로 발전하려면, 각 프로젝트마다 **실제 생산성 데이터를 기록하고 분석**하는 습관이 필요합니다.

```
[프로젝트 완료 후 기록해야 할 데이터 예시]

프로젝트명: ABC 쇼핑몰 개발
개발 기간: 3개월
투입 인원: 4명
총 투입 MM: 12 MM
최종 LOC: 48,000줄
실제 생산성: 48,000 ÷ 12 = 4,000 LOC/MM

예상 생산성 대비 실제 생산성: 5,000 → 4,000 (20% 낮음)
주요 지연 원인: 요구사항 변경 3회, 결제 모듈 연동 어려움

→ 다음 프로젝트 계획 시 생산성을 4,000 LOC/MM으로 기준점 수정
```

이렇게 데이터를 쌓아가면, 다음 프로젝트에서 훨씬 정확한 비용과 일정을 산정할 수 있게 됩니다.

## 📝 핵심 정리

- **LOC(Lines of Code)** 는 소프트웨어의 규모를 총 코드 줄 수로 측정하는 가장 전통적인 방법이다.
- **MM(Man-Month, 맨 먼스)** 는 개발자 1명이 1개월 동안 할 수 있는 작업량을 단위로 하여 프로젝트 규모를 인력과 시간으로 표현하는 방법이다.
- **소프트웨어 생산성**은 1 MM당 생산되는 LOC로 계산하며, 생산성이 높을수록 같은 규모의 프로젝트를 더 빠르고 저렴하게 완성할 수 있다.
- **개발 비용은 LOC와 MM을 기반으로 인건비를 산정한 후**, 테스트/기획/유지보수 비용 등을 추가하여 최종 예산을 결정한다.
- **소프트웨어는 하드웨어처럼 닳지 않는다.** 소프트웨어 오류는 물리적 노후화가 아닌, 설계/개발 단계에서 심어진 잠재적 버그나 환경 변화에서 비롯된다.
- 실제 프로젝트의 **35% 이상이 예정 기간을 초과**하므로, 예산과 일정 계획 시 반드시 여유 버퍼를 추가해야 한다.
- **유지보수 비용**은 초기 개발 비용을 초과할 수 있으며, 소프트웨어는 구조적 연결 때문에 부분 교체가 어렵다.
- **개발자의 역량 차이**가 프로젝트 성패를 좌우하므로, 생산성 높은 팀 구성이 비용 절감의 핵심이다.
- 과거 프로젝트 경험이 새 프로젝트에 그대로 적용되기 어렵기 때문에, 개발자는 **지속적인 자기 계발**을 게을리하면 안 된다.
- LOC와 MM은 소프트웨어 공학에서 비용 산정의 **출발점**이지, 최종 답이 아님을 항상 명심해야 한다.
