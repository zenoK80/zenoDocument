# 🏗️ 소프트웨어 공학이란 무엇인가

안녕하세요! 이번 시간부터 함께 **소프트웨어 공학(Software Engineering)** 이라는 과목을 배워나갑니다. 이 과목은 전산학을 공부하는 분, 정보처리기사 자격증을 준비하는 분, 또는 실무에서 개발·시스템 분석 업무를 하는 분이라면 반드시 알아야 할 핵심 과목입니다.

처음 접하면 "공학? 소프트웨어에 왜 공학이 필요하지?"라는 의문이 생길 수 있습니다. 이 문서를 읽고 나면 그 의문이 자연스럽게 풀릴 것입니다. 실무에서 어떻게 적용되는지 감을 잡으면서 따라오시면, 개발 현장에서 큰 도움이 될 것이라 확신합니다. 자, 시작해봅시다! 🚀

---

## 🧭 소프트웨어 공학이란? — 개념 소개

### 소프트웨어(Software)란 무엇인가

소프트웨어라는 단어는 일상적으로 자주 씁니다. 스마트폰 앱도 소프트웨어고, 엑셀도 소프트웨어이며, 지하철 운행을 통제하는 시스템도 소프트웨어입니다. 그런데 우리가 이 과목에서 다루는 소프트웨어는 **"아마추어가 취미로 대충 만든 프로그램"** 수준이 아닙니다.

산업 현장에서 실제로 사용되는 소프트웨어는 다음과 같은 곳에 쓰입니다:

- 🚀 **미사일 발사 제어 시스템** — GPS 위성 신호, 풍속, 기후 데이터를 종합해 정밀 제어
- ✈️ **항공기 이착륙 관제 시스템** — 비행기가 뜨고 내리는 모든 과정을 소프트웨어로 통제
- 🚇 **지하철·교통망 통제 시스템** — 수십만 명의 이동을 안전하게 관리
- 🤖 **산업용 로봇 제어** — 자동차 공장의 용접 로봇, 반도체 공정 자동화 등

만약 이런 소프트웨어에 심각한 오류가 있다면 어떻게 될까요? 단순히 불편한 수준이 아니라 **인명 피해와 막대한 재산 손실**로 이어질 수 있습니다. 그렇기 때문에 산업용 소프트웨어는 반드시 **견고(Robust)** 해야 합니다.

> **견고한 소프트웨어**란, 어떤 입력 조건이 들어와도 예상치 못한 오류 없이 안정적으로 동작하는 소프트웨어를 의미합니다.

### 그렇다면 "공학"이란 무엇인가

**공학(Engineering)** 이란, 어떤 문제를 해결하기 위해 과학적·수학적·체계적인 원리를 적용하는 학문입니다. 건축 공학자가 건물을 설계할 때 그냥 "대충 튼튼하게 지으면 되겠지"라고 하지 않죠. 내진 설계 기준, 하중 계산, 자재 강도 등 철저한 공학적 원리를 적용합니다.

소프트웨어도 마찬가지입니다. **"대충 짜봤는데 잘 돌아가더라"** 방식으로는 미사일을 제어할 수 없습니다. 체계적이고 과학적인 접근이 필요하죠.

### 소프트웨어 공학의 정의

여러 전문 학자들이 소프트웨어 공학을 어떻게 정의했는지 살펴봅시다:

| 학자/기관 | 정의 핵심 내용 |
|---|---|
| **보엠(Boehm)** | 신뢰성 있고 경제적인 소프트웨어를 만들기 위해 공학적 원리를 응용하고 확립한 것 |
| **보엠II** | 설계·개발·운용·유지보수 과정에서 필요한 과학적 지식을 문서화하고 관리하는 것 |
| **와츠 프레이(Watts Humphrey)** | 품질 좋은 소프트웨어를 경제적으로 개발하기 위해 과학·공학·수학적 방법을 총체적으로 적용하는 것 |
| **브루스(Bruce)** | 품질, 효율, 비용을 공학적으로 접근하는 것 |
| **IEEE** | 소프트웨어의 개발·운용·유지보수·폐기까지 체계적으로 접근하는 방법 |
| **페럴레이(Fairley)** | 전산학·경제학·경영과학·의사소통·문제 해결을 공학적 접근으로 통합한 새로운 기술 체계 |

모든 정의에서 공통적으로 등장하는 키워드가 있습니다: **신뢰성**, **경제성(비용 효율)**, **품질**, **체계적 접근**. 이 네 가지가 바로 소프트웨어 공학의 핵심입니다.

:::info 소프트웨어 공학은 컴퓨터만의 학문이 아닙니다
페럴레이의 정의처럼, 소프트웨어 공학에는 **경제학**(비용·예산 분석), **경영과학**(최적화), **의사소통**(팀 협업), **문제 해결** 등 다양한 학문이 융합됩니다. 단순히 코딩을 잘하는 것이 목적이 아니라, 전체 시스템을 체계적으로 관리하는 것이 핵심입니다.
:::

## ❓ 왜 소프트웨어 공학이 필요한가 — 배경과 문제점

소프트웨어를 개발하다 보면 수많은 문제들이 현실에서 발생합니다. 이 문제들을 하나씩 살펴보면, 왜 공학적 접근이 반드시 필요한지 자연스럽게 이해가 됩니다.

### 🔴 문제 1: 소프트웨어 개발 비용이 매우 비싸다

소프트웨어 개발 비용은 많은 분들이 생각하는 것보다 **훨씬 비쌉니다**. 실무에서 프로젝트 관리(PM) 경험을 바탕으로 보면, 작은 프로젝트도 최소 5천만 원, 중간 규모는 2~3억 원, 정부 대규모 프로젝트는 수십억~수백억 원에 달하기도 합니다.

그렇다면 비용은 어떻게 산정할까요? 여기서 두 가지 중요한 개념이 등장합니다.

**📏 LOC (Lines of Code, 코드 라인 수)**

LOC는 소프트웨어의 **규모를 측정하는 가장 일반적인 단위**입니다. 마치 건물을 지을 때 "연면적 몇 평"으로 규모를 표현하듯이, 소프트웨어는 "총 코드가 몇 라인"인지로 규모를 표현합니다.

```java
// 예시: 간단한 쇼핑몰 주문 처리 메서드 (약 15 LOC)
public void processOrder(Order order) {
    // 재고 확인
    if (!inventoryService.checkStock(order.getProductId())) {
        throw new OutOfStockException("재고 부족");
    }
    // 결제 처리
    paymentService.charge(order.getPaymentInfo(), order.getTotalPrice());
    // 주문 저장
    orderRepository.save(order);
    // SMS 발송
    smsService.sendOrderConfirmation(order.getUserPhone(), order.getOrderId());
}
```

실제 쇼핑몰 시스템 전체라면 이런 메서드가 수천 개 이상 모여 수만~수십만 LOC가 됩니다.

**👤 MM (Man-Month, 맨 먼스)**

MM은 **소프트웨어 개발에 필요한 인력 규모를 나타내는 단위**입니다. "맨 먼스 1"은 개발자 1명이 한 달 동안 처리할 수 있는 작업량을 의미합니다. 마치 "이 공사는 인부 10명이 한 달 일해야 할 분량"이라고 표현하듯이 말이죠.

- 맨 먼스 1 = 개발자 1명 × 1개월
- 맨 먼스 10 = 개발자 10명 × 1개월 (또는 개발자 5명 × 2개월)

**📈 소프트웨어 생산성**

생산성은 **맨 먼스당 생산하는 LOC**로 측정합니다. 개발자 A가 한 달에 10만 라인을 개발하고, 개발자 B가 한 달에 2만 라인을 개발한다면, A의 생산성은 B보다 5배 높습니다. 당연히 인건비도 차이가 납니다.

> **실무 예시**: 5만 LOC 규모의 프로젝트라면, 인건비만 최소 4천만 원에서 최대 1억 2천만 원 정도가 듭니다. 여기에 유지보수, 테스트, 기획 문서 작성 비용이 추가됩니다.

:::tip 비용 산정 방법
프로젝트 비용은 "이거 개발하는 데 얼마예요?"라고 물어봐서 답할 수 있는 게 아닙니다. **기능 목록 → LOC 추정 → 맨 먼스 계산 → 개발자 연봉 기반 인건비 산출 → 부가 비용 합산** 순서로 산정합니다. 이 상세한 계산 방법은 뒷 단원에서 배웁니다.
:::

### 🔴 문제 2: 일정 지연과 낮은 신뢰도

약 600개 회사를 대상으로 조사한 결과, **35% 이상의 프로젝트가 일정 지연**을 경험했습니다. "한 달이면 완성"이라고 했는데 두 달이 걸리고, 기능도 완전히 동작하지 않는 경우가 빈번하게 발생합니다.

**신뢰도(Reliability)** 문제도 심각합니다. 신뢰도가 낮다는 것은 소프트웨어가 **정상적으로 동작하지 않는다**는 의미입니다. 실제 사례들을 보면:

- 🚀 **로켓 발사 실패**: 아폴로 로켓 초기 발사 시 폭발 사고 — 방위 산업 분야에서 목표 미명중 사고의 70% 이상이 소프트웨어 신뢰성 문제
- 🚗 **자동차 급발진**: 하드웨어 오류가 아닌 소프트웨어 문제일 가능성이 매우 높음
- 🛰️ **인공위성 발사 실패**: 초기 발사 실패의 상당 부분이 소프트웨어 오류에서 비롯됨

소프트웨어는 매우 복잡하게 얽혀 있기 때문에, "어디에 문제가 있는가"를 찾아내는 것 자체가 매우 어렵습니다. 바로 이 복잡성 때문에 **공학적 접근**이 필요한 것입니다.

### 🔴 문제 3: 소프트웨어는 하드웨어와 근본적으로 다르다

자동차를 오래 타면 엔진이 닳고 부품이 마모됩니다. 스마트폰을 오래 쓰면 배터리가 부풀어 오릅니다. 이것은 **물리적 노후화**입니다.

그런데 소프트웨어는 다릅니다. 아래한글(HWP)을 10년 동안 매일 사용한다고 해서 소프트웨어 자체가 마모되거나 노후화되지 않습니다. 소프트웨어의 오류는 노후화 때문이 아니라, **처음 설계나 개발 과정에서 제대로 분석되지 못했거나 예상치 못한 오류가 이미 코드 안에 잠재되어 있었기 때문**입니다.

| 구분 | 하드웨어 | 소프트웨어 |
|---|---|---|
| 오류 원인 | 물리적 마모, 노후화 | 설계/개발 단계의 잠재적 버그 |
| 수명 | 시간이 지날수록 성능 저하 | 시간이 지나도 코드 자체는 변하지 않음 |
| 수리 방식 | 부품 교체로 해결 가능 | 전체 구조 파악 후 수정 필요 |
| 품질 보증 | 불량률 0 도전 가능 (정량적) | 정량적 기준 적용이 매우 어려움 |

:::warning 흔한 오해
"프로그램이 갑자기 오류 났어. 오래 써서 그런 거 아니야?" — 이것은 잘못된 이해입니다. 소프트웨어 오류는 노후화 때문이 아니라, **처음부터 잠재해 있던 버그가 특정 조건에서 발현된 것**입니다. 마치 시한폭탄처럼, 특정 입력 조건이 맞아야 터지는 오류가 존재할 수 있습니다.
:::

### 🔴 문제 4: 유지보수와 재작업이 어렵다

자동차 부품이 고장 나면 그 부품만 교체하면 됩니다. 컴퓨터 메모리가 고장 나면 메모리만 새것으로 교체하면 됩니다.

그런데 소프트웨어는 다릅니다. **A 기능만 수정하면 B, C 기능에도 영향이 갈 수 있습니다.** 소프트웨어는 구조적으로 서로 연결되어 있기 때문입니다. 마치 스파게티 면처럼 얽혀있는 코드에서 한 가닥만 잡아당기면 전체가 흐트러지듯이요.

그래서:

- 유지보수 비용이 **개발 비용보다 더 많이 드는 경우**가 자주 있습니다.
- 처음에는 아무 문제 없던 시스템도, 사용하다 보면 예상치 못한 오류들이 하나씩 발견됩니다. 이는 오래 사용해서가 아니라, 그 오류를 발현시키는 **특정 입력 조건을 처음으로 만났기 때문**입니다.
- 아래한글, 안드로이드 등의 소프트웨어가 자주 업데이트를 권장하는 것도 이런 이유입니다. 소프트웨어는 자주 변경되고 업그레이드됩니다.

### 🔴 문제 5: 개발 예산 초과와 경험 축적의 어려움

실무에서는 1억 원짜리 프로젝트가 1억 2천만 원, 1억 5천만 원으로 예산을 초과하는 일이 빈번합니다. 개발 기간도 "한 달"이라고 했다가 "열흘만 더", "일주일만 더"가 반복됩니다.

또한, **과거 경험이 새 프로젝트에 그대로 적용되지 않습니다.** 라면을 맛있게 끓이는 노하우는 한 번 터득하면 계속 써먹을 수 있습니다. 그러나 소프트웨어는 다릅니다:

- 사용자의 요구사항이 매번 조금씩 달라집니다.
- 개발 환경(프레임워크, 언어 버전 등)이 지속적으로 변화합니다.
- 최신 개발 방법론이 계속 등장합니다.

그래서 소프트웨어 개발자는 **끊임없는 자기개발**이 필수입니다. 10년 경력의 Java 개발자라도 억대 연봉을 받는 사람이 있는 반면, 3천만 원도 받기 어려운 사람이 있습니다. 그 차이는 바로 **지속적인 학습과 역량 강화**에서 비롯됩니다.

### 🔴 문제 6: 품질 보증의 어려움

공산품은 "불량"의 기준이 명확합니다. 단추가 하나 빠진 옷은 불량이고, 반품이 가능합니다. 그러나 소프트웨어는 "불량"이라는 개념을 적용하기가 매우 어렵습니다. 하드웨어처럼 "100만 시간 무결함"과 같은 정량적 기준을 소프트웨어에 그대로 적용할 수 없습니다.

> 이러한 모든 문제점들을 해결하기 위해, 전문 학자들이 **공학적 접근 방법**으로 소프트웨어를 관리하자고 제안한 것이 바로 **소프트웨어 공학**의 탄생 배경입니다.

## ⚙️ 소프트웨어 공학의 핵심 원리 — 프로세스, 방법, 도구

소프트웨어 공학은 크게 세 가지 관점으로 이해할 수 있습니다: **프로세스(Process)**, **방법(Method)**, **도구(Tool)**. 이 세 가지는 요리에 비유하면 각각 **조리 순서**, **조리 방법**, **조리 도구**에 해당합니다.

### 🔷 프로세스 (Process) — 개발 공정의 순서

**프로세스**란 어떤 목표를 달성하기 위해 **순차적으로 처리되는 단계**를 의미합니다. 마치 요리할 때 "재료 손질 → 볶기 → 간 맞추기 → 플레이팅"처럼 순서가 있는 것과 같습니다.

소프트웨어 개발의 프로세스를 쉽게 표현하면:

```
[입력(Input)] → [프로세싱(Processing)] → [출력(Output)]
     ↑                                          |
사용자 요구사항                          완성된 소프트웨어
```

각 단계에서는 **산출물(결과 문서)**이 생성됩니다:

| 프로세스 단계 | 산출물 |
|---|---|
| 요구조건 수집 및 분석 | 요구명세서 |
| 설계 | 설계 문서 (아키텍처 다이어그램 등) |
| 개발 정책 수립 | 개발 정책서 |
| 프로그래밍 | 소스 코드 |
| 테스트 | 테스트 수행 보고서 |
| 인수인계 | 인수인계 문서 |

프로세스는 규모에 따라 적용 방식이 달라집니다:

- **비정형화 프로세스**: 규모가 작거나 프로토타입(샘플) 수준의 개발일 때 — 세부 프로세스 모델 없이 기존 방식 적용
- **정형화 프로세스**: 대규모 프로젝트 (6개월 이상, 수십억 예산) — 각 단계별 산출물이 체계적으로 관리되어야 함
- **방법 기반 프로세스**: 역공학(기존 시스템을 분석), 재공학(기존 시스템을 개선) 등 특정 개발 방법론 적용 시
- **자동화 도구 기반 프로세스**: CASE(Computer-Aided Software Engineering) 도구로 프로세스 관리

:::note 프로토타입이란?
**프로토타입(Prototype)** 은 완성품이 아닌 **샘플 또는 시범 버전**을 의미합니다. 실제 시스템을 개발하기 전에 핵심 기능만 간단히 구현해보는 것입니다. 집을 짓기 전에 모형을 만들어보는 것과 같습니다.
:::

### 🔷 방법 (Method) — 어떻게 개발할 것인가

**방법**이란 원하는 결과를 만들기 위해 **적용되는 기법과 절차**를 의미합니다. 요리로 비유하면, 찜을 할 것인지 구이를 할 것인지와 같은 **조리 방법**입니다.

개발의 각 단계별로 방법이 존재합니다:

- **요구 분석 단계**: 사용자 인터뷰, 유스케이스 분석 등
- **설계 단계**: 구조적 설계, 객체지향 설계 등
- **프로그래밍 단계**: Java, Python, C++ 등의 언어와 코딩 컨벤션
- **테스팅 단계**: 단위 테스트, 통합 테스트, 인수 테스트 등

이러한 방법들을 하나의 체계로 묶은 것을 **방법론(Methodology)** 이라고 합니다. 방법론은 크게 세 가지로 나뉩니다:

#### 1️⃣ 구조적 방법론 (Structured Methodology) — 프로세스 중심

마치 폭포(Waterfall)처럼 위에서 아래로 순서대로 진행하는 방식입니다. **하향식(Top-Down)** 개발 방법론이라고도 합니다.

```
요구 분석 → 설계 → 구현 → 테스트 → 유지보수
```

특징:
- 단계가 명확하게 구분됨
- 개발자 개인의 능력이 프로그램 품질을 크게 좌우함
- 단순한 단위 시스템 개발에 적합

#### 2️⃣ 정보공학 방법론 (Information Engineering Methodology) — 데이터 중심

**데이터(자료)의 흐름과 구조**를 중심으로 시스템을 설계하는 방법론입니다. 여기서 핵심 도구는 **ER 다이어그램(Entity-Relationship Diagram)** 입니다.

ER 다이어그램을 쉽게 설명하면:

```
[학생] ─── 지도받음 ───▷ [교수]
  |                           |
학번, 이름, 학과          교수번호, 이름, 전공
```

- 학생(개체)과 교수(개체)가 "지도 관계"로 연결됨
- 한 명의 교수가 여러 명의 학생을 지도 (1:N 관계)
- 학생 정보와 교수 정보가 데이터베이스에서 서로 연결되어 관리됨

다른 예시로 성적 처리 흐름을 보면:

```
시험 응시 → 성적 데이터 생성 → 교수 점수 부여 → DB 저장
→ 학생 성적 조회 → 성적표 발송
```

이처럼 **데이터가 어떻게 흐르는지**를 중심으로 시스템을 설계합니다. 프로그래밍 자동화 도구를 활용할 수 있다는 특징이 있습니다.

#### 3️⃣ 객체지향 방법론 (Object-Oriented Methodology) — 현재 가장 널리 사용

**자료(데이터)와 프로세스(기능)를 하나의 객체(Object)로 묶어서** 설계하는 방법론입니다. 현재 Java, Python, C++ 등 대부분의 현대 프로그래밍 언어가 객체지향을 기반으로 합니다.

알약 비유로 이해해봅시다: 감기약 캡슐 안에는 다양한 성분(데이터)과 그 성분이 하는 역할(프로세스)이 들어있습니다. 캡슐 껍데기(객체)를 보면 내부 성분이 보이지 않습니다. 이것이 바로 **정보 은닉(Information Hiding)** 입니다.

```java
// ✅ 객체지향 방법론 예시: 학생 객체
public class Student {
    // 데이터(속성) — 캡슐 안에 숨겨짐 (private)
    private String studentId;   // 학번
    private String name;        // 이름
    private double gpa;         // 학점

    // 프로세스(메서드) — 외부에서 접근 가능한 기능
    public String getName() {
        return name; // 이름 반환
    }

    public void updateGpa(double newGpa) {
        // 학점 유효성 검사 후 업데이트
        if (newGpa >= 0.0 && newGpa <= 4.5) {
            this.gpa = newGpa;
        }
    }
}
```

```java
// ❌ 절차적(구조적) 방식 — 데이터와 기능이 분리되어 관리 어려움
String studentName = "홍길동";
double studentGpa = 3.5;
// 어디서든 직접 접근 가능 → 데이터 무결성 보장 어려움
studentGpa = -999; // 잘못된 값이 들어와도 막을 방법 없음
```

객체지향 방법론의 특징:

| 특징 | 설명 |
|---|---|
| **모듈화** | 자료와 프로세스를 하나의 객체로 묶음 |
| **정보 은닉** | 내부 데이터를 외부에서 직접 접근 불가 |
| **재사용성** | 만들어 놓은 객체를 다른 곳에서도 사용 가능 |
| **확장성** | 기존 객체를 상속(Inheritance)받아 새 기능 추가 용이 |
| **협업 용이** | 분석가·설계자·프로그래머가 협력 개발 가능 |
| **점증적 개발** | 하나씩 개발하고 살을 붙여가며 반복 개선 가능 |

### 🔷 도구 (Tool) — 자동화를 위한 수단

**도구**란 개발 과정의 자동화를 돕는 각종 소프트웨어 도구들입니다. 요리로 비유하면 냄비, 오븐, 믹서기 같은 **조리 도구**입니다.

- **설계 자동화 도구**: UML 다이어그램 작성 도구, ER 다이어그램 도구
- **개발 자동화 도구**: IDE(통합 개발 환경), 코드 자동 완성 도구
- **산출물 자동화 도구**: 문서 자동 생성, 테스트 케이스 자동 생성

100% 완벽한 자동화 도구는 아직 존재하지 않지만, 반자동화 수준의 도구들을 잘 활용하면 **생산성이 크게 향상**됩니다.

## 📊 실제 예시로 보는 소프트웨어 공학 — 쇼핑몰 개발 시나리오

말로만 설명하면 어렵습니다. 실제 쇼핑몰 개발 프로젝트를 예시로 소프트웨어 공학이 어떻게 적용되는지 살펴보겠습니다.

### 시나리오: 소규모 쇼핑몰 개발 요청

고객이 다음과 같이 요청합니다:

> "쇼핑몰을 만들어 주세요. 주문 기능, 재고 관리, 관리자 페이지에서 상품 등록, 주문 시 SMS 자동 발송, 배송 추적 기능이 필요합니다."

### Step 1: 요구 분석 → 요구명세서 작성

```
[요구명세서 예시]
기능 요구사항:
  - FR-001: 상품 목록 조회 및 검색
  - FR-002: 장바구니 추가/삭제
  - FR-003: 주문 처리 (결제 포함)
  - FR-004: 주문 시 SMS 자동 발송
  - FR-005: 재고 관리 (수량 자동 차감)
  - FR-006: 관리자 상품 등록/수정/삭제
  - FR-007: 배송 추적 연동

비기능 요구사항:
  - NFR-001: 동시 접속자 1,000명 이상 처리
  - NFR-002: 결제 오류율 0.01% 이하
  - NFR-003: 페이지 응답 속도 3초 이내
```

### Step 2: LOC 및 비용 추정

```python
# LOC 기반 비용 추정 예시 (개념 설명용 의사코드)

# 기능별 예상 LOC 추정
estimated_loc = {
    "상품 관리": 8000,      # 상품 CRUD, 이미지 처리 등
    "주문 처리": 12000,     # 결제, 재고 차감, 이메일 등
    "SMS 발송": 3000,       # 외부 API 연동
    "배송 추적": 5000,      # 외부 물류 API 연동
    "관리자 페이지": 7000,  # 대시보드, 통계 등
    "공통 모듈": 5000,      # 인증, 로깅, DB 연결 등
}

total_loc = sum(estimated_loc.values())  # 총 40,000 LOC

# 개발자 생산성 가정: 한 명이 한 달에 5,000 LOC
developer_productivity = 5000  # LOC/MM

# 필요한 맨 먼스 계산
required_mm = total_loc / developer_productivity  # 8 MM

# 개발자 월 인건비 가정: 500만원
monthly_salary = 5_000_000  # 원

# 순수 인건비 계산
labor_cost = required_mm * monthly_salary  # 4,000만원

# 부가 비용 (테스트, 기획, 유지보수 등) = 인건비의 약 50%
total_cost = labor_cost * 1.5  # 약 6,000만원

print(f"총 LOC: {total_loc:,} 라인")         # 40,000 라인
print(f"필요 맨 먼스: {required_mm} MM")       # 8 MM
print(f"예상 총 비용: {total_cost:,.0f} 원")   # 약 6,000만원
```

### Step 3: 프로세스 진행

```
요구 분석 완료
    ↓
설계 (DB 설계 + UI/UX + 아키텍처)
    ↓
개발 (Java/Spring Boot 기반)
    ↓
테스트 (단위 테스트 → 통합 테스트 → 인수 테스트)
    ↓
교육 (관리자·운영자 사용법 교육)
    ↓
오픈 및 유지보수
```

각 단계마다 **문서(산출물)** 가 생성되고, 이를 체계적으로 관리하는 것이 **소프트웨어 공학의 핵심**입니다.

:::warning 소프트웨어 개발에서 자주 하는 실수
```
❌ 잘못된 접근: "요구사항 분석 없이 바로 코딩 시작"
   → 개발 중간에 요구사항이 계속 바뀜
   → 이미 만든 코드를 전부 다시 짜야 하는 상황 발생
   → 비용과 시간이 2~3배로 늘어남

✅ 올바른 접근: "요구명세서 작성 → 설계 문서 확정 → 개발 시작"
   → 변경 사항이 생겨도 문서 기반으로 영향 범위 파악 가능
   → 비용과 시간 낭비 최소화
```
:::

## 🏛️ 소프트웨어 공학의 역사

소프트웨어 공학은 갑자기 생겨난 것이 아닙니다. 수십 년에 걸친 **소프트웨어 위기(Software Crisis)** 를 겪으면서 점진적으로 발전해왔습니다.

### 📅 1960년대 — 소프트웨어 공학의 탄생

- **NATO 회의**에서 처음으로 "소프트웨어 공학"이라는 단어가 등장
- 당시 소프트웨어 개발은 완전한 주먹구구식이었음
- **고투(GOTO)문** 남용이 큰 문제였습니다

고투문이 왜 문제일까요?

```c
// ❌ 고투문 남용 예시 — 스파게티 코드
start:
    printf("입력: ");
    // 처리 로직...
    goto process;  // A에서 갑자기 B로 점프

process:
    // 처리 중...
    goto validate; // B에서 C로 점프

validate:
    // 검증 중...
    goto start;    // C에서 다시 A로 — 무한 루프 가능성!
    // 이렇게 되면 사이클이 생겨 빠져나올 수 없음
```

```c
// ✅ 구조적 프로그래밍 — 명확한 흐름
void processInput() {
    // 순서대로 실행: 입력 → 처리 → 검증
    char input = getInput();       // 1. 입력 받기
    int result = processData(input); // 2. 처리
    validateResult(result);          // 3. 검증
}
// 흐름이 위에서 아래로 명확함
```

- 소프트웨어 위기 인식: "이대로는 안 된다! 공학적 접근이 필요하다!"

### 📅 1970년대 — 소프트웨어 공학의 학문적 정착

- 소프트웨어 공학이 **독립된 학문 분야**로 자리잡음
- 품질과 생산성 문제가 본격적으로 논의됨
- **소프트웨어 생명주기(Software Life Cycle)** 개념 도입: 요구 분석 → 설계 → 개발 → 테스트 → 유지보수 → 폐기
- 구조적 프로그래밍이 소프트웨어 위기 극복의 수단으로 각광받기 시작

### 📅 1980년대 전반 — 방법론의 폭발적 성장

- 구조적 방법론, Jackson 방법론, Yourdon 방법론 등 **다양한 분석·설계 방법론** 등장
- **테스트, 유지보수, 프로젝트 관리**의 중요성 부각
- **소프트웨어 공장(Software Factory)** 개념 탄생: 모듈을 공장에서 부품 찍어내듯 생산

### 📅 1980년대 후반 — 객체지향의 시대 개막

- **객체지향 분석 및 설계 방법론** 본격 사용
- Java, C++과 같은 객체지향 프로그래밍 언어 확산
- **재사용성, 자동화 도구, 정보공학, 품질 관리, 형상 관리(Configuration Management)** 등 새로운 기법들 활용
- "단순 생산보다 개발자의 역량이 중요하다"는 인식 확산
- 소프트웨어 공학이 본격적으로 성숙기에 접어듦

:::info 형상 관리(Configuration Management)란?
소프트웨어의 **변경 사항을 체계적으로 관리**하는 것입니다. 예를 들어, Git으로 코드 버전을 관리하는 것이 형상 관리의 대표적인 예입니다. 누가, 언제, 무엇을, 왜 변경했는지 추적할 수 있게 됩니다.
:::

## 💡 소프트웨어 공학의 특징, 목적, 주요 이슈

### 소프트웨어 공학의 적용 범위

소프트웨어 공학은 모든 프로그램에 적용하는 것이 아닙니다. "1부터 100까지 합을 구하는 프로그램"을 만드는 데 소프트웨어 공학 문서를 작성할 필요는 없습니다.

소프트웨어 공학이 필요한 경우:

- ✅ **팀 단위 개발** (혼자가 아닌 여러 명이 협업)
- ✅ **규모가 큰 프로젝트** (6개월 이상, 수천만 원 이상)
- ✅ **장기 유지보수가 필요한 시스템**
- ✅ **높은 신뢰성이 요구되는 시스템** (항공, 의료, 금융 등)

소프트웨어 공학이 필요 없는 경우:

- ❌ 간단한 취미 프로젝트 (개인 블로그 스크립트 등)
- ❌ 극히 단기간의 일회성 작업
- ❌ 프로토타입 수준의 샘플 개발

### 소프트웨어 공학의 목적

소프트웨어 공학의 핵심 목적을 정리하면:

1. **생산성 향상**: 같은 시간·인력으로 더 많고 좋은 소프트웨어 개발
2. **품질 향상**: 오류가 적고 신뢰성 높은 소프트웨어 개발
3. **유지보수 용이성**: 나중에 수정·개선이 쉬운 구조로 개발
4. **재사용성 확보**: 한번 만든 모듈을 다른 곳에서도 활용
5. **비용과 기간 최적화**: 정해진 예산과 일정 내에 최고 품질의 소프트웨어 생산

> 소프트웨어 공학의 핵심은 **"개발"이 목적이 아닙니다.** 개발은 개발자가 알아서 합니다. 소프트웨어 공학은 **사용자의 요구사항을 전략적으로 계획·분석하고, 전체 시스템을 하나의 관점에서 관리하여 최적의 성과를 이끌어내는 것**이 진정한 목적입니다.

### 소프트웨어 공학의 주요 이슈 (요리 비유로 정리)

| 소프트웨어 공학 요소 | 요리 비유 | 설명 |
|---|---|---|
| **방법(Method)** | 조리 방법 | 구이냐 찜이냐 — 구조적이냐 객체지향이냐 |
| **도구(Tool)** | 조리 도구 | 오븐, 냄비 — 자동화 개발 도구, IDE |
| **프로세스(Process)** | 조리 순서 | 물 끓이기 → 재료 넣기 → 간 맞추기 |
| **패러다임(Paradigm)** | 요리 트렌드 | 요즘 유행하는 스타일 — 최신 개발 방법론 |

**패러다임(Paradigm)** 은 특히 중요합니다. 개발된 소프트웨어를 보면 "이건 최신 패러다임으로 만들었네" 또는 "이건 구식이네"를 알 수 있습니다. 구형 패러다임으로 만든 시스템은 개선이 필요하며, 이를 소프트웨어 공학 관점에서 체계적으로 분석하고 설계해야 합니다.

## ⚠️ 주의사항 및 실무 팁

### 🔑 팁 1: 요구사항 분석에 충분한 시간을 투자하라

실무에서 가장 많은 비용 낭비가 발생하는 원인은 **요구사항이 불명확한 채로 개발을 시작하는 것**입니다. 요구명세서를 꼼꼼하게 작성하고 고객의 확인을 받은 후 개발을 시작하면, 중간에 "이게 아닌데요"로 인한 재작업을 크게 줄일 수 있습니다.

:::danger 절대 하면 안 되는 것
요구사항 확인 없이 바로 코딩 시작하기. 특히 대규모 프로젝트에서 "일단 만들고 나중에 요구사항 맞추자"는 접근은 프로젝트 실패의 지름길입니다. 개발이 60~70% 진행된 후 요구사항이 바뀌면, 처음부터 다시 만드는 것과 비슷한 비용이 발생합니다.
:::

### 🔑 팁 2: 유지보수 비용을 미리 예산에 반영하라

많은 프로젝트에서 **개발 비용만 예산에 포함**하고 유지보수 비용을 간과하는 실수를 합니다. 실제로 소프트웨어 생애 전체 비용에서 유지보수 비용이 개발 비용을 초과하는 경우도 있습니다. 처음부터 유지보수하기 쉬운 구조로 설계하고, 유지보수 예산을 반드시 계획에 포함시키세요.

### 🔑 팁 3: 일정 지연을 예측에 반영하라

조사 결과에서 알 수 있듯이, 35% 이상의 프로젝트가 일정 지연을 경험합니다. **버퍼(여유 시간)** 를 일정에 반드시 포함시키세요. "3개월이면 끝난다"면, 공식 일정은 3.5~4개월로 잡는 것이 현실적입니다.

### 🔑 팁 4: 지속적인 자기개발이 필수

소프트웨어 개발 환경은 빠르게 변합니다. 과거에 A 방식으로 개발했다고 해서 새 프로젝트에 그대로 적용할 수 없는 경우가 많습니다. 새로운 프레임워크, 새로운 방법론, 새로운 도구를 꾸준히 학습해야 시장 경쟁력을 유지할 수 있습니다.

:::tip 생산성을 높이는 현실적인 방법
- **코드 리뷰 습관화**: 다른 개발자의 코드를 보면서 더 효율적인 방법을 배울 수 있습니다.
- **설계 먼저, 코딩 나중**: 처음 30분을 설계에 투자하면 코딩 시간이 절반으로 줄어들 수 있습니다.
- **테스트 코드 작성**: 처음에는 시간이 더 걸리지만, 장기적으로 버그 수정 시간을 대폭 줄여줍니다.
- **자동화 도구 적극 활용**: IDE의 자동완성, 리팩토링 도구 등을 잘 활용하면 생산성이 크게 향상됩니다.
:::

### 🔑 팁 5: 방법론은 프로젝트 성격에 맞게 선택하라

모든 프로젝트에 객체지향 방법론이 최선인 것은 아닙니다. 프로젝트의 규모, 기간, 팀 구성, 요구사항의 안정성 등을 고려해 적합한 방법론을 선택하는 것이 중요합니다.

```
단기·소규모 프로젝트 → 구조적 방법론 (빠르고 단순)
데이터 중심 시스템 → 정보공학 방법론 (DB 설계 중심)
장기·대규모·확장 필요 → 객체지향 방법론 (재사용성·확장성 우수)
기존 시스템 개선 → 역공학/재공학 방법론
```

## 📝 핵심 정리

이 문서에서 배운 핵심 개념을 정리합니다.

- **소프트웨어 공학**은 신뢰성·경제성·품질을 공학적 원리로 달성하기 위한 체계적 학문이다.
- 소프트웨어는 하드웨어와 달리 **물리적 노후화가 없고**, 오류는 설계·개발 단계의 잠재적 버그에서 비롯된다.
- **LOC(코드 라인 수)** 는 소프트웨어 규모를, **MM(맨 먼스)** 는 인력 규모를 측정하는 단위다.
- 소프트웨어 개발의 주요 문제점은 **높은 비용, 일정 지연, 낮은 신뢰도, 유지보수 어려움, 품질 보증의 한계**이다.
- 소프트웨어 공학은 **프로세스(순서), 방법(기법), 도구(자동화)** 세 가지 관점으로 구성된다.
- **구조적 방법론**은 프로세스 중심, **정보공학 방법론**은 데이터 중심, **객체지향 방법론**은 객체(데이터+기능) 중심이다.
- 현재 가장 널리 쓰이는 방법론은 **객체지향 방법론**으로, 재사용성·확장성·협업이 뛰어나다.
- 소프트웨어 공학의 역사는 1960년대 NATO 회의에서 시작해 70년대 학문화, 80년대 방법론 발전, 80년대 후반 객체지향 시대로 발전해왔다.
- 소프트웨어 공학의 목적은 **개발 자체가 아니라**, 사용자 요구사항 분석부터 시스템 관리까지 전체를 최적화하는 것이다.
- 소프트웨어 개발자는 환경 변화에 맞는 **지속적인 자기개발**이 필수다.
