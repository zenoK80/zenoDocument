# 🚀 애자일 기법의 개념과 특징

소프트웨어 개발 세계에서 "애자일(Agile)"이라는 단어를 한 번쯤은 들어보셨을 겁니다. 취업 공고에도 자주 등장하고, 개발팀 회의에서도 빠지지 않는 단어죠. 그런데 막상 "애자일이 뭐야?"라고 물어보면 명쾌하게 대답하기 어려운 경우가 많습니다. 이번 문서에서는 애자일이 왜 등장했는지, 어떤 원리로 동작하는지, 그리고 어떤 구체적인 기법들이 있는지를 처음 배우는 분도 이해할 수 있도록 아주 자세하게 풀어서 설명드리겠습니다.

---

## 🌍 왜 신속한 소프트웨어 개발이 필요한가?

### 빠르게 변하는 비즈니스 환경

요즘 세상이 얼마나 빠르게 돌아가는지 한번 생각해 보세요. 스마트폰 하나로 전 세계 어디서든 쇼핑을 하고, 음식을 주문하고, 택시를 부르는 시대입니다. 소비자의 요구는 매일매일 새롭게 바뀌고, 경쟁사는 하루가 다르게 새로운 서비스를 출시합니다. 이런 환경을 **"초경쟁 시대"** 라고 부를 만큼, 기업이 살아남기 위해서는 빠른 대응이 필수입니다.

예를 들어, A라는 스타트업이 새로운 배달 앱을 기획했다고 합시다. 만약 이 앱을 개발하는 데 2년이 걸린다면, 2년 후 시장은 완전히 다른 모습일 수 있습니다. 경쟁사가 이미 시장을 장악했을 수도 있고, 소비자의 트렌드가 바뀌어 아무도 그 앱을 원하지 않을 수도 있죠. **소프트웨어를 얼마나 빠르게 개발하고 새로운 서비스를 즉시 제공하느냐**에 따라 기업의 성패가 갈리는 시대가 된 것입니다.

### 기존 개발 모형의 한계

"그럼 기존 개발 방식을 그냥 더 빠르게 하면 되는 거 아닌가요?"라는 질문이 자연스럽게 나올 수 있습니다. 폭포수(Waterfall) 모형, 프로토타이핑 모형, 나선형(Spiral) 모형 등 기존의 개발 방식들이 있는데, 이것들을 그냥 속도를 높여서 하면 되지 않느냐는 거죠.

하지만 기존 개발 모형들은 구조적으로 다음과 같은 전제 조건이 필요합니다.

- **요구사항이 완전하게 명세(문서화)** 되어 있어야 한다
- **각 단계의 산출물(결과 문서)** 이 나와야 다음 단계로 진행할 수 있다
- 요구조건 분석 → 설계 → 구현 → 테스트 → 유지보수 순서를 **반드시 순서대로** 진행해야 한다

마치 아파트를 짓는 것과 같습니다. 설계도면이 완성되지 않으면 공사를 시작할 수 없고, 기초공사가 끝나야 골조를 올릴 수 있죠. 이렇게 순차적으로 진행해야 하는 구조에서는 아무리 빠르게 하려고 해도 물리적인 한계가 있습니다.

> 💡 **핵심:** 기존 개발 모형은 "완벽한 문서가 먼저, 개발은 그 다음"이라는 원칙을 따르기 때문에, 빠르게 변하는 요구사항에 유연하게 대응하기 어렵습니다.

이런 한계를 극복하기 위해 등장한 것이 바로 **신속한 소프트웨어 개발 모형**, 그 중심에 있는 것이 **애자일(Agile) 기법**입니다.

## 🧠 애자일 기법이란 무엇인가?

### 애자일의 기본 개념

**애자일(Agile)** 이라는 단어 자체가 "민첩한", "신속한"이라는 뜻을 가지고 있습니다. 애자일 기법은 소프트웨어의 **요구조건 분석 → 개발 → 테스트 → 인도(고객에게 전달)** 과정을 **반복적으로** 빠르게 처리해 나가는 방식입니다.

기존의 폭포수 모형을 다시 한번 떠올려 보겠습니다.

```
[폭포수 모형]
요구사항 분석 → (명세서 완성) → 설계 → (설계서 완성) → 구현 → (소스코드 완성) → 테스트 → 유지보수
```

이 방식은 각 단계가 완전히 끝나야 다음 단계로 넘어갑니다. 마치 계단을 하나씩 밟고 내려가는 폭포처럼요.

반면 애자일 방식은 이렇게 생각해 볼 수 있습니다.

```
[애자일 방식]
A 분석 → A 설계 → A 구현 → A 테스트 → 고객 확인
B 분석 → B 설계 → B 구현 → B 테스트 → 고객 확인
C 분석 → C 설계 → C 구현 → C 테스트 → 고객 확인
(반복, 동시 진행)
```

즉, **전체를 한꺼번에 완성하는 것이 아니라, 작은 단위로 쪼개서 빠르게 완성하고 고객에게 전달하는 방식**입니다. 이것이 핵심입니다.

### 신속한 개발이 가능한 두 가지 열쇠

애자일이 빠른 개발을 가능하게 하는 핵심 원리는 두 가지입니다.

**첫 번째: 동시 진행(병렬 처리)**

요구 분석, 설계, 구현이 순차적으로 진행되는 것이 아니라 **동시에** 진행됩니다. 분석팀이 A를 분석하는 동안, 이미 분석이 끝난 이전 내용을 기반으로 설계팀이 설계를 하고, 개발팀이 구현을 시작합니다. 마치 자동차 공장에서 차체를 조립하는 팀, 엔진을 조립하는 팀, 내장재를 붙이는 팀이 동시에 각자 일을 하는 것처럼요.

**두 번째: 핵심 기능만 먼저 정의**

전체 요구사항을 완벽하게 문서화하는 데 시간을 낭비하지 않습니다. **가장 중요한 기능만 먼저 정의**하고, 그것을 기반으로 설계하고 개발을 시작합니다. 그리고 요구사항이 추가되면 설계와 소스코드도 동시에 업데이트합니다. 마치 레고 블록을 쌓듯이, 기본 틀을 먼저 만들고 나중에 조각들을 하나씩 붙여나가는 방식이죠.

:::info 애자일의 핵심 전제
애자일은 "시스템의 요구사항이 개발 프로세스 기간 동안 계속 변경된다"는 사실을 처음부터 인정하고 받아들이는 개발 방식입니다. 변경을 두려워하거나 막으려는 것이 아니라, 변경이 언제든지 일어날 수 있도록 처음부터 설계하는 것입니다.
:::

## ⚙️ 애자일 기법의 핵심 원리

### 원리 1: 고객의 지속적인 참여

애자일에서 고객은 단순히 처음에 요구사항을 제출하고 끝이 아닙니다. **개발 프로세스 전체에 걸쳐 긴밀하게 참여**하면서 요구사항의 우선순위를 계속 알려줘야 합니다.

왜 이게 중요하냐면, 실제 프로젝트 경험상 고객은 처음에 자신이 원하는 것을 완벽하게 설명하지 못하는 경우가 대부분이기 때문입니다. 예를 들어 성적 조회 시스템을 개발한다고 할 때, 고객이 처음에는 "그냥 조회만 되면 돼요"라고 했다가, 개발이 끝난 뒤에 "날짜별로도 조회되어야 하고, 학번으로도, 점수별로도 조회가 되어야 해요"라고 추가 요구를 쏟아내는 상황이 생깁니다.

이런 상황을 막으려면 처음부터 고객이 개발 과정에 적극적으로 참여해서, **개발 중간중간에 결과물을 확인하고 피드백을 주어야** 합니다.

### 원리 2: 점증적 인도 (Incremental Delivery)

**점증적 인도**란 한꺼번에 모든 것을 완성해서 주는 것이 아니라, **하나가 완성되면 바로 전달하고, 또 하나가 완성되면 바로 전달하는 방식**을 말합니다.

예를 들어 기능 A, B, C, D를 개발해야 한다면:

| 전통적 방식 | 점증적 인도 방식 |
|---|---|
| A, B, C, D를 모두 완성한 뒤 한 번에 전달 | A 완성 → A 전달 → 고객 확인 |
| 최종 결과물이 나오기 전까지 고객이 아무것도 볼 수 없음 | B 완성 → B 전달 → 고객 확인 |
| 중간에 요구사항이 바뀌면 대규모 수정 필요 | C 완성 → C 전달 → 고객 확인 |
| 개발 기간이 길어질수록 위험 증가 | 각 단계마다 빠른 피드백과 수정 가능 |

마치 짜장면집에서 짜장면, 짬뽕, 탕수육, 볶음밥을 한꺼번에 모두 완성해서 내오는 것이 아니라, 완성된 것부터 하나씩 빠르게 내오는 것과 같습니다.

### 원리 3: 사람이 프로세스보다 중요하다

"사람은 프로세스가 아니다"라는 말이 애자일의 중요한 철학 중 하나입니다. 이 말의 의미는, 팀 구성원들이 **경직된 규칙과 정해진 절차에만 얽매이지 말고**, 각자의 방식으로 유연하게 일하면서 서로 소통하라는 것입니다.

중요한 것은 "지금 내가 무엇을 하고 있는지", "언제 끝날 것인지"를 팀원들이 서로 알고 있어야 한다는 점입니다. 그래야 동시에 빠르게 작업이 진행될 수 있으니까요.

### 원리 4: 변경을 적극 수용한다

애자일에서는 **요구사항이 변경되는 것을 처음부터 당연한 일로 받아들입니다.** "변경이 생길 수 있다"는 것을 알고 있기 때문에, 변경이 들어왔을 때 즉시 설계와 개발에 반영할 수 있도록 유연한 구조를 갖춰야 합니다.

반면 폭포수 모형 방식에서는 요구사항이 변경되면 이미 완성된 명세서, 설계서, 소스코드를 처음부터 다시 작성해야 하는 경우가 생깁니다. 이것이 얼마나 큰 낭비인지는 개발을 해본 분이라면 잘 아실 겁니다.

### 원리 5: 단순함 유지

개발 프로세스와 소프트웨어 설계 자체가 **단순해야** 빠른 커뮤니케이션이 가능하고, 점증적 개발이 원활하게 이루어집니다. 불필요하게 복잡한 구조는 속도를 늦추고 오류를 늘립니다.

:::tip 단순함의 힘
복잡한 것을 복잡하게 만드는 것은 누구나 할 수 있습니다. 복잡한 것을 단순하게 만드는 것이 진짜 실력입니다. 애자일은 "지금 당장 필요한 것만 구현한다"는 원칙을 중요하게 여깁니다.
:::

## 🔧 애자일 기법의 종류와 실제 예시

### 1️⃣ XP (eXtreme Programming, 익스트림 프로그래밍)

**XP**는 애자일 기법 중 하나로, 점증적 개발 방법을 변형시킨 것입니다. "익스트림(Extreme)"이라는 말이 붙은 이유는, 좋은 개발 관행들을 **극한의 수준까지 끌어올린다**는 의미입니다. 그 중에서도 특히 **고객의 참여**를 최대한 끌어올리는 것을 강조합니다.

XP의 기반이 되는 가치는 다음과 같습니다.

| 가치 | 설명 |
|---|---|
| **커뮤니케이션** | 팀원 간, 고객과의 지속적인 소통 |
| **단순함** | 지금 필요한 것만, 단순하게 |
| **피드백** | 즉시 결과를 확인하고 반영 |
| **용기** | 고객에게 현실적인 이야기를 할 수 있는 용기, 팀원에게 격려 |
| **존중** | 팀원과 고객의 의견을 존중 |

XP에서 고객의 요구사항을 수집할 때는 **스토리보드(이야기 형태의 시나리오)** 를 만듭니다. "로그인을 하면 → 성적 조회 화면이 나오고 → 날짜별로 조회할 수 있고 → 학번으로도 조회할 수 있다"와 같이 고객이 실제로 시스템을 사용하는 흐름을 이야기 형태로 정리하는 것입니다. 그러면 개발팀은 그 시나리오를 기반으로 구현할 수 있습니다.

#### XP의 12가지 실천 사항

XP에는 구체적으로 지켜야 할 12가지 실천 사항이 있습니다. 하나씩 살펴보겠습니다.

1. **계획 세우기**: 우선순위와 기술적 고려사항을 바탕으로 개발 범위와 계획을 세웁니다.
2. **짧은 릴리즈(Release) 주기**: 아주 짧은 개발 사이클로 빠르게 테스트 가능한 결과물을 만들어냅니다.
3. **메타포(Metaphor) 활용**: 시스템 전체에 대한 비유적이고 직관적인 개념을 공유해, 팀 전체가 쉽게 이해할 수 있도록 합니다.
4. **단순한 설계**: 지금 당장 필요한 것만 설계합니다. 미래의 요구사항을 미리 예측해서 복잡하게 만들지 않습니다.
5. **테스트 우선 개발**: 코드를 짜기 전에 먼저 테스트 코드를 작성합니다. 이 방식을 **TDD(Test-Driven Development, 테스트 주도 개발)** 라고도 합니다.
6. **리팩터링(Refactoring)**: 빠르게 개발하다 보면 코드 품질이 낮아지는 경우가 많습니다. 주기적으로 코드 구조를 개선해서 품질을 유지합니다. 마치 글을 쓰고 나서 퇴고(다듬기)를 하는 것과 같습니다.
7. **페어 프로그래밍(Pair Programming)**: 두 명이 짝을 이루어 함께 코딩합니다. 이 부분은 아래에서 더 자세히 설명합니다.
8. **공동 코드 소유**: 코드는 특정 개인의 것이 아닌 팀 전체의 것입니다. 누구나 코드를 수정할 수 있습니다.
9. **지속적인 통합(Continuous Integration)**: 개발한 코드를 즉시 통합하고 빌드하여 테스트합니다. 일주일씩 쌓아두었다가 한꺼번에 통합하지 않습니다.
10. **40시간 근무 원칙**: 야근과 초과 근무를 줄이고 계획적으로 40시간 이내에 업무를 마칩니다. 지속 가능한 속도로 일하는 것이 장기적으로 더 효율적이기 때문입니다.
11. **고객 상주**: 고객이 개발팀 현장에 직접 상주하면서 지속적으로 피드백을 제공합니다.
12. **코딩 표준 준수**: 모든 팀원이 동일한 코딩 규칙을 따릅니다. 이렇게 해야 서로의 코드를 쉽게 이해하고 수정할 수 있습니다.

#### XP의 단점

XP가 효과적으로 작동하려면 몇 가지 전제 조건이 필요합니다.

- 빠른 구현을 지원하는 **컴포넌트(재사용 가능한 코드 단위)나 프레임워크가 미리 준비**되어 있어야 합니다.
- 프로그래머가 테스트를 소홀히 하거나 예외 상황을 제대로 처리하지 않으면 불안전한 시스템이 만들어질 수 있습니다.
- 고객이 초반에 스토리 작성에 많이 참여했다는 이유로 테스트 단계에서 참여를 꺼릴 수 있습니다.

:::warning XP 적용 시 주의사항
XP는 고객의 적극적인 참여 없이는 제대로 동작하지 않습니다. 고객이 처음 요구사항 작성에만 참여하고 테스트와 검증에 참여하지 않는다면, 결국 "제가 원한 게 이게 아닌데요"라는 상황이 발생할 수 있습니다.
:::

### 2️⃣ 페어 프로그래밍 (Pair Programming, 짝 프로그래밍)

페어 프로그래밍은 XP의 대표적인 실천 방법으로, **두 명의 프로그래머가 하나의 컴퓨터 앞에 앉아서 함께 코드를 작성**하는 방식입니다.

처음 들으면 이런 생각이 드실 겁니다. "혼자 하면 더 빠른 거 아닌가요? 두 명이 같은 일을 하면 비효율적이지 않나요?"

이것은 매우 합리적인 의문입니다. 하지만 실제로 연구해보면 흥미로운 결과가 나옵니다.

#### 왜 둘이 하면 더 효율적인가?

페어 프로그래밍의 핵심은 **코딩 시작 전에 두 사람이 함께 토론하는 과정**에 있습니다. "이 기능을 어떻게 구현할까?", "이 방식으로 하면 나중에 이런 문제가 생기지 않을까?", "이 방법이 더 나을 것 같은데 어떻게 생각해?"와 같은 토론을 미리 충분히 하기 때문에, 실제 코딩 단계에서 잘못된 방향으로 개발하거나 나중에 전부 다시 작성해야 하는 상황이 크게 줄어듭니다.

다음 표를 통해 혼자 개발할 때와 짝으로 개발할 때를 비교해 봅시다.

| 비교 항목 | 혼자 개발 | 페어 프로그래밍 |
|---|---|---|
| **코딩 속도** | 빠름 | 상대적으로 느림 |
| **오류 발생 가능성** | 높음 | 낮음 (두 사람이 서로 검증) |
| **재작업 빈도** | 높음 | 낮음 (사전 토론으로 방향 확인) |
| **최종 완성까지의 총 시간** | 비슷하거나 더 길 수 있음 | 비슷하거나 더 짧을 수 있음 |
| **코드 품질** | 낮을 수 있음 | 높음 |
| **지식 공유** | 개인에게만 집중 | 팀 전체로 확산 |

즉, **코딩 자체의 속도는 느릴 수 있지만, 디버깅(오류 수정)과 재작업까지 포함한 전체 개발 과정을 보면 오히려 더 효율적**인 경우가 많습니다.

#### 페어 프로그래밍의 장점

- **공동 코드 소유권**: "이건 내 코드야, 저건 네 코드야"가 아니라, 코드에 대한 책임을 함께 집니다. 특정 개인이 책임을 혼자 떠안는 부담이 줄어듭니다.
- **지식의 팀 전체 확산**: 짝을 서로 돌아가며 바꾸기 때문에 (로테이션), 시간이 지나면 팀의 모든 구성원이 전체 코드를 이해하게 됩니다.
- **개발자 역량 향상**: 다양한 사람과 짝을 이루어 일하다 보면 서로의 장점을 배우게 됩니다.

#### 페어 프로그래밍의 단점

- **실력 차이 문제**: 고수 개발자와 초보 개발자가 짝이 되면, 고수가 기다리다 지칠 수 있습니다. 반대로 초보자도 눈치가 보여 자유롭게 의견을 내기 어려울 수 있습니다.
- **성향 문제**: 혼자 집중해서 일하는 것을 선호하는 사람에게는 짝으로 일하는 것이 부담이 될 수 있습니다.
- **원격 환경의 어려움**: 재택근무나 다른 장소에서 일할 때는 같은 컴퓨터 앞에 앉는 것처럼 원활하게 소통하기 어렵습니다.

:::note 페어 프로그래밍 팁
페어 프로그래밍의 효과를 최대로 높이려면, 비슷한 실력의 사람들끼리 짝을 짓는 것이 중요합니다. 또한 짝을 주기적으로 바꾸어 팀 전체의 코드 이해도를 높이세요.
:::

### 3️⃣ 스크럼 (Scrum)

**스크럼**은 럭비나 미식축구에서 선수들이 어깨를 맞대고 밀집 대형을 이루는 장면에서 이름을 따왔습니다. 마치 팀이 하나로 뭉쳐 함께 목표를 향해 나아가는 모습처럼, **개발팀이 하나의 원팀으로 뭉쳐 빠르게 문제를 해결해나가는 방식**입니다.

스크럼의 핵심은 **스프린트(Sprint)** 라고 하는 짧은 개발 주기를 중심으로 운영된다는 점입니다. 보통 **한 달(30일)을 하나의 스프린트**로 삼고, 그 안에 동작 가능한 소프트웨어 기능을 완성해서 제공합니다.

#### 스크럼의 주요 개념

| 용어 | 설명 |
|---|---|
| **스프린트(Sprint)** | 보통 한 달 정도의 짧은 개발 주기. 이 기간 안에 특정 기능을 완성하는 것을 목표로 합니다. |
| **백로그(Backlog)** | 이번 스프린트에서 해야 할 일의 목록. "이번 달에 요거, 요거, 요거 기능 개선합시다"라고 정해놓은 리스트입니다. |
| **스크럼 마스터(Scrum Master)** | 팀장 역할을 하는 사람. 팀이 원활하게 일할 수 있도록 장애물을 제거하고 프로세스를 관리합니다. |
| **제품 소유권자(Product Owner)** | 제품의 전체 방향을 관리하고 백로그의 우선순위를 결정하는 사람. |
| **일일 스크럼(Daily Scrum)** | 매일 15분씩 진행하는 짧은 회의. 어제 한 일, 오늘 할 일, 현재 막히는 것을 공유합니다. |
| **속도(Velocity)** | 한 번의 스프린트 동안 팀이 완성할 수 있는 작업량. |

#### 스크럼의 흐름

스크럼이 실제로 어떻게 진행되는지 간단한 예시를 통해 살펴보겠습니다.

```
1. 제품 소유권자가 백로그를 작성한다.
   → "이번 달에 로그인 기능, 성적 조회 기능, 알림 기능을 완성하자"

2. 스크럼 마스터가 팀(7명 이하)을 구성하고 스프린트를 시작한다.

3. 매일 아침 15분, 일일 스크럼 회의를 진행한다.
   → "나는 어제 로그인 UI를 완성했고, 오늘은 API 연동을 할 예정이야"
   → "나는 현재 데이터베이스 연결 부분에서 막혔어, 도움이 필요해"

4. 회의에서 나온 문제점을 빠르게 반영하고 해결한다.

5. 스프린트가 끝나면 완성된 기능을 리뷰하고, 다음 스프린트를 계획한다.
```

:::tip 스크럼 팀의 규모
스크럼에서는 팀의 인원이 **7명을 넘지 않도록** 합니다. 팀이 너무 크면 의사소통이 복잡해지고 회의 시간이 길어져서 "빠른 개발"이라는 목표가 흐려지기 때문입니다.
:::

### 4️⃣ RAD (Rapid Application Development, 신속 응용 시스템 개발)

**RAD**는 매우 짧은 주기, 보통 **60일에서 90일 이내**에 소프트웨어를 개발하기 위한 방법론입니다. 컴포넌트(미리 만들어진 재사용 가능한 부품) 기반으로 빠르게 소프트웨어를 조립하듯 개발하는 방식이라고 이해하시면 됩니다.

#### RAD의 진행 과정

```
비즈니스 모델링 → 데이터 모델링 → 프로세스 모델링 → 응용 프로그램 개발 → 테스트 및 인수
```

- **비즈니스 모델링**: 어떤 비즈니스 기능과 정보가 필요한지 정의합니다.
- **데이터 모델링**: 시스템이 처리해야 할 데이터 구조를 정의합니다.
- **프로세스 모델링**: 데이터를 처리하는 절차와 흐름을 정의합니다.
- **응용 프로그램 개발**: 기존 컴포넌트를 재사용하거나 새로 만들어 빠르게 개발합니다.
- **테스트 및 인수**: 완성된 시스템을 테스트하고 고객에게 전달합니다.

#### RAD의 특징과 한계

RAD는 짧은 시간 안에 시스템을 개발할 수 있다는 강점이 있습니다. 하지만 다음과 같은 경우에는 적합하지 않습니다.

- 기술적으로 **위험도가 높거나 새로운 기술**을 적용해야 하는 경우
- **고성능이나 고안정성**이 요구되는 경우 (예: 금융 시스템, 군사 시스템)
- **대규모 프로젝트**의 경우 (대규모 프로젝트에 RAD를 적용하면 엄청난 인력이 필요합니다)
- 개발 기간이 60~90일보다 훨씬 더 필요한 복잡한 시스템

:::warning RAD의 한계
RAD는 빠른 개발이 가능한 대신, 테스트 기간이 짧아질 수 있습니다. 또한 대규모 프로젝트에 RAD를 적용하면, 짧은 기간 안에 완성하기 위해 막대한 인력과 비용이 투입되어야 합니다.
:::

### 5️⃣ 기타 애자일 기법들

#### ASD (Adaptive Software Development, 적응적 소프트웨어 개발)

복잡한 소프트웨어 시스템을 개발하기 위해 나온 기법으로, "**추측하고, 협동하고, 학습한다**"는 철학을 기반으로 합니다. 사람들 간의 협동과 팀 조직화를 통해 빠르게 개발하는 방식입니다. 처음부터 완벽한 계획을 세우는 것이 아니라, 불확실한 상황에서 추측하며 시작하고, 협동을 통해 문제를 풀어나가고, 그 과정에서 배운 것을 다음에 적용하는 순환 구조입니다.

#### DSDM (Dynamic Systems Development Method, 동적 시스템 개발 방법)

통제된 프로젝트 환경에서 점증적 프로토타이핑 방식을 활용하는 기법입니다. **미리 프레임워크(개발의 틀)를 제공**해서, 개발팀이 그 틀 안에서 빠르게 개발할 수 있도록 지원합니다. 틀이 없으면 빠르게 시작하기 어렵기 때문에, 기본 구조를 미리 만들어두는 것이 핵심입니다.

#### FDD (Feature-Driven Development, 특징 주도 개발)

**6단계로 구성**되며, 앞의 3단계는 한 번만 수행하고 뒤의 3단계는 반복적으로 수행하는 방식입니다.

```
[한 번만 수행]
1단계: 전체 모델 개발 (전문가들이 검토 회의를 통해 전체 시스템 구조 설계)
2단계: 기능 리스트 구축 (개발해야 할 기능 목록 작성)
3단계: 기능 단위 계획 (일정과 우선순위 결정)

[반복 수행]
4단계: 기능 단위 설계
5단계: 기능 단위 구축
6단계: 설치(배포)
```

#### 린 소프트웨어 개발 (Lean Software Development)

**제조업의 린(Lean) 생산 방식**을 소프트웨어 개발에 적용한 것입니다. 린 제조업에서는 자동차 부품을 미리 창고에 쌓아두지 않고, 필요한 때에 필요한 부품이 들어오도록 합니다. 이를 **JIT(Just-In-Time, 적시 생산)** 이라고 하는데, 이 원리를 소프트웨어 개발에 적용해서 **낭비를 제거하고, 꼭 필요한 것만 개발하고, 빠르게 가치를 전달**하는 방식입니다.

#### 애자일 UP (Agile Unified Process)

기존에 소프트웨어 개발 프로세스 모형 중 하나였던 **UP(Unified Process, 도입-정년-구축-전이 통합 방식)** 을 애자일의 원칙과 접목한 것입니다. 팀을 통해 UP 방식을 빠르고 유연하게 처리해 나가는 구조입니다.

## ⚠️ 애자일 기법의 문제점과 적용이 어려운 상황

### 애자일이 모든 상황에 맞는 건 아니다

애자일은 매우 유용한 방법론이지만, 모든 상황에 적합한 것은 아닙니다. 다음과 같은 경우에는 애자일 적용이 어렵거나 적합하지 않습니다.

**1. 팀이 서로 다른 장소에서 작업하는 경우**

애자일의 핵심은 **빠른 커뮤니케이션**입니다. 팀원들이 각자 다른 장소에서 재택근무를 하거나, 심지어 시차가 다른 나라에 있는 경우라면 실시간으로 소통하고 함께 개발하는 것이 어려워집니다. 화상 통화, 메신저 등 협업 도구가 많이 발전했지만, 같은 공간에서 함께 일하는 것만큼 원활하지는 않습니다.

**2. 복잡하고 대규모 시스템 개발**

하드웨어와 소프트웨어가 복잡하게 얽혀 있거나, 시스템 자체가 매우 복잡한 경우, 단순함을 유지하기 어렵습니다. 복잡한 시스템은 빠른 커뮤니케이션과 점증적 개발이 어렵습니다.

**3. 안정성과 보안이 최우선인 시스템**

항공기 제어 시스템, 원자력 발전소 제어 시스템, 금융 거래 시스템처럼 **안정성과 보안이 절대적으로 중요한 시스템**에서는 애자일처럼 빠르고 유연하게 변경을 수용하는 방식이 위험할 수 있습니다. 이런 시스템은 철저한 명세와 검증을 거쳐야 하기 때문입니다.

:::danger 절대 간과하지 말 것
안전이 생명과 직결되는 시스템(의료기기 소프트웨어, 항공 제어 시스템, 원자력 제어 시스템 등)에 성급하게 애자일을 적용하면 치명적인 결과를 초래할 수 있습니다. 이러한 시스템은 반드시 엄격한 명세와 단계별 검증을 거쳐야 합니다.
:::

| 상황 | 애자일 적합 여부 |
|---|---|
| 빠르게 변화하는 요구사항의 웹/앱 서비스 | ✅ 적합 |
| 스타트업의 MVP(최소 기능 제품) 개발 | ✅ 적합 |
| 소규모 팀이 한 공간에서 일하는 경우 | ✅ 적합 |
| 항공기/원자력 등 안전이 최우선인 시스템 | ❌ 부적합 |
| 팀이 전 세계에 분산된 대규모 프로젝트 | ⚠️ 주의 필요 |
| 요구사항이 처음부터 완벽하게 고정된 경우 | ⚠️ 굳이 애자일 필요 없음 |

## 💡 코드로 보는 애자일의 흔적 — 리팩터링과 테스트 주도 개발

애자일의 원칙 중 "단순함 유지"와 "리팩터링"을 코드 예시로 살펴보겠습니다.

### ❌ 잘못된 예시: 빠르게 개발하느라 복잡해진 코드

```python
# 학생 성적 처리 함수 - 처음에 급하게 짠 코드 (리팩터링 전)
def process(data, t, f):
    # t와 f가 무엇을 의미하는지 알 수 없음
    # 변수명이 불분명하고, 한 함수가 너무 많은 역할을 함
    result = []
    for d in data:
        if d['score'] >= t:
            result.append({'name': d['name'], 'grade': 'pass', 'final': d['score'] * f})
        else:
            result.append({'name': d['name'], 'grade': 'fail', 'final': d['score'] * f})
    return result
```

이 코드는 작동은 하지만, `t`와 `f`가 무엇을 의미하는지, 이 함수가 정확히 무슨 일을 하는지 다른 사람이 보면 이해하기 어렵습니다.

### ✅ 올바른 예시: 리팩터링을 통해 단순하고 명확하게 만든 코드

```python
# 학생 성적 처리 함수 - 리팩터링 후 (의미가 명확한 코드)

def calculate_grade(score, passing_score):
    """
    점수가 합격 기준 이상이면 'pass', 미만이면 'fail'을 반환합니다.
    
    Args:
        score: 학생의 점수
        passing_score: 합격 기준 점수
    
    Returns:
        'pass' 또는 'fail' 문자열
    """
    # 합격 여부를 판단하는 단순한 함수
    if score >= passing_score:
        return 'pass'  # 합격
    return 'fail'  # 불합격


def apply_score_multiplier(score, multiplier):
    """
    점수에 가중치를 곱하여 최종 점수를 계산합니다.
    
    Args:
        score: 원래 점수
        multiplier: 가중치 (예: 1.1이면 10% 가산)
    
    Returns:
        최종 점수 (소수점 이하 반올림)
    """
    # 가중치를 적용한 최종 점수 계산
    return round(score * multiplier, 2)


def process_student_scores(students, passing_score, score_multiplier):
    """
    학생 목록을 받아 합격 여부와 최종 점수를 계산하여 반환합니다.
    
    Args:
        students: 학생 정보가 담긴 딕셔너리 리스트 [{'name': '홍길동', 'score': 85}, ...]
        passing_score: 합격 기준 점수 (예: 60)
        score_multiplier: 점수 가중치 (예: 1.1)
    
    Returns:
        처리된 학생 정보 리스트
    """
    processed_results = []  # 처리 결과를 담을 리스트

    for student in students:  # 학생 목록을 하나씩 순회
        student_name = student['name']       # 학생 이름 추출
        student_score = student['score']     # 학생 점수 추출

        # 합격 여부 계산 (별도 함수 호출 - 단일 책임 원칙)
        grade = calculate_grade(student_score, passing_score)

        # 최종 점수 계산 (별도 함수 호출 - 단일 책임 원칙)
        final_score = apply_score_multiplier(student_score, score_multiplier)

        # 결과를 딕셔너리 형태로 결과 리스트에 추가
        processed_results.append({
            'name': student_name,    # 학생 이름
            'grade': grade,          # 합격/불합격
            'final_score': final_score  # 최종 점수
        })

    return processed_results  # 전체 처리 결과 반환


# ===== 사용 예시 =====
students = [
    {'name': '홍길동', 'score': 85},   # 홍길동: 85점
    {'name': '김철수', 'score': 55},   # 김철수: 55점
    {'name': '이영희', 'score': 72},   # 이영희: 72점
]

# 합격 기준 60점, 가중치 1.1(10% 가산) 적용
results = process_student_scores(
    students=students,
    passing_score=60,
    score_multiplier=1.1
)

# 결과 출력
for result in results:
    print(f"이름: {result['name']}, 합격여부: {result['grade']}, 최종점수: {result['final_score']}")

# 출력 결과:
# 이름: 홍길동, 합격여부: pass, 최종점수: 93.5
# 이름: 김철수, 합격여부: fail, 최종점수: 60.5
# 이름: 이영희, 합격여부: pass, 최종점수: 79.2
```

**코드 설명:**
- `calculate_grade` 함수: 합격 여부 판단만을 담당합니다. **단 하나의 역할**만 합니다.
- `apply_score_multiplier` 함수: 점수에 가중치를 적용하는 역할만 합니다. **단 하나의 역할**만 합니다.
- `process_student_scores` 함수: 위 두 함수를 조합해서 전체 처리를 합니다. 각 기능이 분리되어 있어 **수정이 필요할 때 해당 함수만 수정**하면 됩니다.
- 모든 변수명이 의미를 명확하게 전달합니다. (`t` → `passing_score`, `f` → `score_multiplier`)

이것이 바로 애자일에서 강조하는 **리팩터링의 결과**입니다. 빠르게 개발하다 보면 처음에는 지저분한 코드가 나올 수 있지만, 주기적으로 코드를 정리하고 개선해서 품질을 높여나가는 것이 중요합니다.

## 📋 주의사항과 실무 팁

### 애자일을 도입할 때 흔히 하는 실수들

**실수 1: 애자일을 "문서 없이 개발하는 것"으로 오해한다**

애자일은 "문서를 만들지 말라"는 것이 아닙니다. **과도한 문서화보다 실제 동작하는 소프트웨어를 더 중시한다**는 것입니다. 꼭 필요한 문서는 여전히 작성해야 합니다.

**실수 2: 일일 스크럼 회의를 길게 진행한다**

일일 스크럼은 **15분** 이내로 진행해야 합니다. "어제 뭐 했어요?", "오늘 뭐 할 거예요?", "막히는 거 있어요?" 세 가지만 공유하면 됩니다. 문제 해결 논의는 회의가 끝난 뒤 관련된 사람들끼리 따로 합니다.

**실수 3: 스프린트 중간에 계속 새로운 요구사항을 추가한다**

스프린트가 시작되면 그 기간 동안의 목표(백로그)는 지켜야 합니다. 새로운 요구사항은 다음 스프린트의 백로그에 추가합니다. 스프린트 중간에 계속 새로운 것이 추가되면 아무것도 제대로 완성할 수 없습니다.

**실수 4: 팀이 너무 크다**

스크럼 팀은 7명 이하가 이상적입니다. 팀이 크면 커뮤니케이션 비용이 기하급수적으로 늘어납니다.

:::warning 가장 많이 하는 실수
"애자일이니까 계획 없이 바로 개발 시작해도 돼"라고 생각하는 것입니다. 애자일도 **계획을 세웁니다.** 다만 그 계획이 고정된 것이 아니라 변경에 유연하게 대응할 수 있도록 설계되어 있을 뿐입니다.
:::

### 실무에서 애자일을 성공적으로 적용하기 위한 팁

1. **고객(또는 제품 소유권자)의 적극적인 참여를 확보하는 것이 최우선**입니다. 개발팀만 열심히 해서는 성공할 수 없습니다.
2. **팀 전체가 애자일의 원칙을 이해하고 동의해야** 합니다. 일부 팀원만 애자일 방식으로 일하고 나머지는 기존 방식으로 일하면 혼란이 생깁니다.
3. **회고(Retrospective) 시간을 반드시 갖습니다.** 스프린트가 끝날 때마다 "이번에 잘한 것은 무엇인지, 개선할 것은 무엇인지"를 팀 전체가 함께 돌아보는 시간이 필요합니다.
4. **작은 것부터 시작합니다.** 처음부터 완벽한 애자일을 적용하려 하지 말고, 일일 스크럼 회의나 2주 스프린트처럼 하나씩 도입해보는 것이 현실적입니다.

## 📌 핵심 정리

이번 문서에서 배운 핵심 개념들을 정리합니다.

- **애자일(Agile)** 은 빠르게 변하는 비즈니스 환경에 대응하기 위해 등장한 신속한 소프트웨어 개발 방법론이다.
- 기존 폭포수 모형은 단계별 산출물이 완성되어야 다음 단계로 진행하지만, 애자일은 **동시 진행과 작은 단위의 점증적 개발**을 추구한다.
- 애자일의 핵심 원리는 **고객 참여, 점증적 인도, 사람 중심, 변경 수용, 단순함 유지**다.
- **XP(익스트림 프로그래밍)** 는 고객 참여를 극한으로 끌어올리고, 스토리보드 기반 개발과 페어 프로그래밍을 실천한다.
- **페어 프로그래밍** 은 두 명이 짝을 이뤄 개발하여 오류를 줄이고 코드 품질을 높이는 방식이다.
- **스크럼** 은 30일 스프린트 단위로 개발하며, 매일 15분 일일 회의로 빠른 문제 해결을 추구한다.
- **RAD** 는 60~90일 내 빠른 개발을 목표로 컴포넌트 재사용을 통해 소프트웨어를 신속하게 조립하는 방식이다.
- 애자일은 **안정성·보안이 최우선인 시스템, 분산된 팀, 복잡한 대규모 시스템**에는 적합하지 않을 수 있다.
- **리팩터링** 은 빠른 개발 후 코드 품질을 지속적으로 개선하는 과정으로, 애자일에서 매우 중요하다.
- 애자일은 단순한 개발 절차가 아니라, **팀이 일하는 방식과 문화 전반에 걸친 철학**으로 이해해야 한다.
