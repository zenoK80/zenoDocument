# 🏗️ 클래스 구조: 속성과 메서드, 그리고 리팩토링

프로그래밍을 처음 배우다 보면 "클래스(class)가 뭐야?", "속성이랑 메서드가 무슨 차이야?"라는 질문이 자연스럽게 생깁니다. 이 문서에서는 **UML(설계 도구)로 표현된 클래스 구조가 실제 Java 코드로 어떻게 구현되는지**를 차근차근 살펴보고, 나아가 이미 만들어진 코드를 더 좋게 다듬는 **리팩토링(refactoring)**까지 함께 이해해 봅니다.

:::info 이 문서를 읽기 전에
이 문서는 Java나 C 같은 프로그래밍 언어를 깊이 공부한 분이 아니어도 괜찮습니다. "아, UML로 설계한 것이 실제 코드로 이렇게 만들어지는구나"라는 **큰 그림(방법론적 구조)**을 이해하는 것이 목표입니다. 코드의 세세한 문법보다는 **개념과 흐름**에 집중해서 읽어주세요.
:::

## 📐 UML과 클래스: 설계도가 코드가 되는 순간

### UML이란 무엇인가?

**UML(Unified Modeling Language, 통합형 설계 언어)**은 소프트웨어를 만들기 전에 "이렇게 만들겠다"라고 약속하는 **표준화된 설계 도구**입니다. 마치 건물을 짓기 전에 건축가가 설계 도면을 그리듯이, 개발자들도 코드를 작성하기 전에 UML로 전체 구조를 먼저 그립니다.

UML로 그린 설계도는 특정 프로그래밍 언어에 종속되지 않습니다. Java 개발자든, Python 개발자든, C++ 개발자든 **동일한 UML 설계도를 보고 각자의 언어로 구현**할 수 있다는 점이 가장 큰 장점입니다. 마치 악보가 피아노로도, 바이올린으로도 연주될 수 있는 것처럼요.

### UML 클래스 다이어그램의 구조

UML에서 클래스는 보통 아래와 같이 세 칸으로 나뉜 직사각형으로 표현됩니다.

| 구역 | 내용 | 예시 |
|------|------|------|
| **1번째 칸** | 클래스 이름 | `User` |
| **2번째 칸** | 속성 (변수, 데이터) | `age: int`, `name: String` |
| **3번째 칸** | 메서드 (기능, 동작) | `getSchedule()`, `introduce()` |

이 설계도를 Java 개발자가 받으면, 그것을 보고 바로 코드를 작성합니다. "설계도 → 코드"로 이어지는 이 흐름을 이해하는 것이 이 섹션의 핵심입니다.

## 🧱 클래스의 핵심 구성 요소: 속성과 메서드

### 클래스란 무엇인가?

**클래스(Class)**란 **속성(attribute)**과 **메서드(method)**를 하나로 묶어놓은 것입니다. 마치 "사람"이라는 개념을 설명할 때 "이름, 나이, 키"처럼 **특징(속성)**이 있고, "걷다, 말하다, 먹다"처럼 **행동(메서드)**이 있는 것과 같습니다.

> 클래스 = 속성(데이터를 담는 틀) + 메서드(그 데이터로 실제 행동을 수행하는 기능)

### 속성(Attribute)이란?

**속성**은 클래스가 가지고 있는 **값(데이터)**을 저장하는 공간입니다. 냉장고에 비유하면, 냉장고 안에 있는 **재료들의 목록**이 바로 속성입니다. "이 클래스에는 어떤 정보가 들어가 있나요?"를 대답해주는 부분입니다.

예를 들어 `User`(사용자) 클래스에는 다음과 같은 속성이 있을 수 있습니다:
- `age` (나이) → 정수(int) 타입
- `name` (이름) → 문자열(String) 타입

### 메서드(Method)란?

**메서드**는 클래스가 수행할 수 있는 **기능, 동작, 행동**입니다. 냉장고 비유를 계속하면, 냉장고에서 **재료를 꺼내는 행위** 자체가 메서드입니다. "이 클래스는 무엇을 할 수 있나요?"를 대답해주는 부분입니다.

예를 들어 `User` 클래스에는 다음과 같은 메서드가 있을 수 있습니다:
- `getSchedule()` → 스케줄을 가져오는 기능
- `introduce()` → 자기 소개를 하는 기능

### UML → Java 코드 변환 예시

UML에서 설계한 `User` 클래스를 실제 Java 코드로 옮기면 아래와 같습니다.

```java
// ✅ UML 설계를 Java 코드로 구현한 User 클래스
public class User {

    // ===== 속성(Attribute) 영역 =====
    private int age;        // 나이: 정수형 (private = 이 클래스 내부에서만 사용)
    private String name;    // 이름: 문자열형 (private = 외부에서 직접 접근 불가)

    // ===== 메서드(Method) 영역 =====

    // getSchedule(): 스케줄을 가져오는 기능
    public void getSchedule() {
        // 스케줄 조회 로직이 이곳에 들어감
    }

    // introduce(): 자기소개를 하는 기능
    public void introduce() {
        // 자기소개 출력 로직이 이곳에 들어감
    }
}
```

이 코드를 한 줄씩 설명하면:

- `public class User` → **"User라는 이름의 클래스를 선언한다. 누구나 이 클래스를 사용할 수 있다(public)."**
- `private int age` → **"age라는 정수형 속성을 선언한다. 이 클래스 내부에서만 사용 가능하다(private)."**
- `private String name` → **"name이라는 문자열 속성을 선언한다. 역시 내부에서만 사용 가능하다."**
- `public void getSchedule()` → **"getSchedule이라는 메서드를 선언한다. 누구나 이 기능을 호출할 수 있다(public)."**
- `public void introduce()` → **"introduce라는 메서드를 선언한다. 자기소개 기능을 수행한다."**

:::tip public vs private 쉽게 이해하기
- **public(공용)**: 마치 편의점처럼 누구나 들어와서 사용할 수 있는 것
- **private(사적)**: 마치 내 방처럼 나(해당 클래스) 외에는 아무도 들어올 수 없는 것

속성은 보통 `private`으로 보호하고, 메서드를 통해서만 접근하게 만드는 것이 좋은 코딩 습관입니다.
:::

## 🔷 클래스의 종류: 일반 클래스, 추상 클래스, 인터페이스

### 일반 클래스 vs 추상 클래스

클래스에는 여러 종류가 있습니다. 가장 기본이 되는 **일반 클래스**부터 시작해서 **추상 클래스(abstract class)**, **인터페이스(interface)**까지 단계별로 살펴봅니다.

**추상 클래스(Abstract Class)**란 "한 개 이상의 메서드가 껍데기만 있고 실제 내용이 없는 클래스"입니다. 마치 요리 레시피에서 "재료를 볶는다"라고만 써놓고 구체적인 방법은 요리사가 직접 채워 넣도록 한 것과 같습니다.

| 구분 | 일반 클래스 | 추상 클래스 |
|------|------------|------------|
| 메서드 구현 | 내부에 실제 동작 코드 있음 | 껍데기(선언)만 있고 내용 없음 |
| 인스턴스 생성 | 직접 객체 생성 가능 | 직접 객체 생성 불가 |
| 사용 목적 | 바로 사용 가능한 완성된 클래스 | 다른 클래스가 상속받아 완성 |
| Java 키워드 | `class` | `abstract class` |

```java
// ✅ 추상 클래스(Abstract Class) 예시
abstract class User {

    private String name;  // 속성은 그대로 있음

    // 추상 메서드: 껍데기만 있고 실제 내용이 없음
    // 이 클래스를 상속받는 자식 클래스에서 반드시 구현해야 함
    public abstract void work();
}

// ✅ 추상 클래스를 상속받아 실제 기능을 구현하는 자식 클래스
class Employee extends User {

    // 부모의 추상 메서드 work()를 실제로 구현
    @Override
    public void work() {
        System.out.println("직원이 업무를 수행합니다.");  // 실제 내용을 여기서 채움
    }
}
```

:::note 추상 클래스가 왜 필요한가요?
마치 "자동차"라는 개념에는 반드시 "달린다"는 기능이 있어야 하지만, 그 달리는 방식은 승용차냐 트럭이냐에 따라 다를 수 있습니다. 추상 클래스는 이런 상황에서 **"이 기능은 반드시 있어야 해!"라는 규칙을 강제**하면서도, 구체적인 구현은 자식 클래스에 맡기는 방식입니다.
:::

### 일반화(Generalization): 상속 관계 이해하기

**일반화(Generalization)**란 **부모 클래스(superclass)**와 **자식 클래스(subclass)** 사이의 상속 관계를 의미합니다. 쇼핑몰을 예로 들면 이렇습니다:

- 쇼핑몰에는 **고객(Customer)**도 있고 **관리자(Admin)**도 있습니다.
- 이 둘을 합쳐보면 결국 모두 **사용자(User)**입니다.
- 따라서 `User`가 **부모 클래스(superclass)**, `Customer`와 `Admin`이 **자식 클래스(subclass)**가 됩니다.

자식 클래스는 부모 클래스의 속성과 메서드를 **상속(inheritance)**받아서 사용할 수 있습니다. 마치 부모님으로부터 유전자를 물려받듯이요.

```java
// ✅ 부모 클래스(superclass): User
public class User {

    private String id;        // 아이디 속성
    private String password;  // 패스워드 속성

    // 로그인 메서드: 자식 클래스들이 상속받아 사용
    public void login() {
        System.out.println("로그인 처리 중...");
    }
}

// ✅ 자식 클래스(subclass): Customer (User를 extends로 상속받음)
public class Customer extends User {

    // User의 login() 메서드를 상속받아 고객 전용 로그인으로 재정의
    @Override
    public void login() {
        System.out.println("고객 로그인 처리 중...");  // 고객만의 로그인 방식
    }
}

// ✅ 자식 클래스(subclass): Admin (User를 extends로 상속받음)
public class Admin extends User {

    // User의 login() 메서드를 상속받아 관리자 전용 로그인으로 재정의
    @Override
    public void login() {
        System.out.println("관리자 로그인 처리 중...");  // 관리자만의 로그인 방식
    }
}
```

> **상속(inheritance)**을 사용하면 공통된 코드를 부모 클래스에 한 번만 작성하고, 자식 클래스에서는 달라지는 부분만 추가하거나 수정하면 됩니다. 코드 중복이 줄어들고, 유지보수도 훨씬 쉬워집니다.

### 실체화(Realization): 인터페이스와 오버라이딩

**인터페이스(Interface)**는 추상 클래스보다 더 나아가서, **오직 명세(껍데기)만 가지고 있는 설계 규약**입니다. "이 기능이 반드시 있어야 한다"는 약속만 정의해두고, 실제 구현은 전혀 없습니다.

그리고 이 껍데기를 실제 기능으로 채우는 행위를 **오버라이딩(overriding, 재정의)**이라고 합니다. 추상 클래스를 상속할 때는 `extends`를 쓰지만, 인터페이스를 구현할 때는 **`implements`**를 사용합니다.

```java
// ✅ 인터페이스(Interface): CheckLog - 껍데기만 있음
public interface CheckLog {

    // 메서드 선언만 있고 내용(구현)이 없음
    // checkLog()는 "로그를 확인해야 한다"는 약속만 정의
    boolean checkLog();  // true(정상) 또는 false(비정상)를 반환한다는 규약
}

// ✅ 인터페이스를 implements로 구현한 실제 클래스
public class DateCheckLog implements CheckLog {

    // checkLog() 메서드를 오버라이딩(overriding)하여 실제 내용을 채움
    @Override
    public boolean checkLog() {
        // 실제 로그 확인 로직
        System.out.println("날짜 기반 로그를 확인합니다.");
        return true;  // 정상이면 true 반환
    }
}
```

:::info 인터페이스 vs 추상 클래스 한 눈에 비교

| 구분 | 추상 클래스 (`abstract class`) | 인터페이스 (`interface`) |
|------|-------------------------------|------------------------|
| 상속 키워드 | `extends` | `implements` |
| 속성(변수) 보유 | 가능 | 불가능 (상수만 가능) |
| 일반 메서드 포함 | 가능 | 불가능 (기본적으로) |
| 다중 상속 | 불가능 (단일 상속) | 가능 (여러 인터페이스 구현 가능) |
| 목적 | 공통 기능을 공유하면서 일부만 추상화 | 기능의 규약(약속)만 정의 |

:::

## 🔗 클래스 간의 관계: 의존, 연관, 집합

### 의존(Dependency): 다른 클래스를 참조하기

**의존(Dependency)**이란 어떤 클래스가 **다른 클래스를 일시적으로 참조(사용)**하는 관계입니다. 예를 들어, `User` 클래스가 `Schedule`(스케줄) 클래스를 사용할 때입니다.

객체 지향에서 중요한 규칙이 하나 있습니다. **클래스를 정의했다고 바로 쓸 수 있는 게 아니라**, 반드시 `new`라는 키워드로 **객체(instance)를 생성**한 뒤에 사용해야 합니다. 마치 집 설계도(클래스)가 있다고 바로 집에 살 수 있는 게 아니라, 설계도를 보고 집을 실제로 지어야(객체 생성) 살 수 있는 것과 같습니다.

```java
// ✅ 의존(Dependency) 관계 예시: User가 Schedule을 참조
public class User {

    // createSchedule(): 스케줄을 생성하는 메서드
    public void createSchedule() {

        // new 키워드로 Schedule 객체를 생성 (정의만 해서는 사용 불가!)
        Schedule schedule = new Schedule();

        // 생성된 schedule 객체를 실제로 사용
        schedule.save();  // 스케줄 저장 기능 호출
    }
}
```

### 연관(Association): 다른 클래스를 필드로 보유하기

**연관(Association)**은 의존보다 더 긴밀한 관계입니다. 다른 클래스의 객체를 **클래스의 속성(필드)으로 직접 가지고 있는** 관계입니다. 마치 사람(User)이 주소(Address)를 항상 가지고 다니는 것처럼요.

```java
// ✅ 연관(Association) 관계 예시: User가 Address를 필드로 보유
public class User {

    private String name;                    // 이름 속성
    private List<Address> addressList;      // 주소 목록(리스트)을 필드로 직접 보유

    // 주소를 추가하는 메서드
    public void addAddress(Address address) {
        this.addressList.add(address);       // 리스트에 주소 추가
    }
}
```

### 집합(Aggregation): 전체와 부분의 관계

**집합(Aggregation)**은 **전체와 부분의 관계**입니다. 연관과 비슷하지만, "전체가 없어져도 부분은 독립적으로 존재할 수 있다"는 차이가 있습니다. 예를 들어 학교(전체)가 없어져도 학생(부분)은 존재할 수 있는 것처럼요.

```java
// ✅ 집합(Aggregation) 관계 예시
public class AddressGroup {

    // User 객체들을 집합(그룹) 형태로 보유
    private List<User> userList;

    // 사용자를 그룹에 추가
    public void addUser(User user) {
        this.userList.add(user);  // 사용자 추가
    }
}
```

## ♻️ 리팩토링: 더 좋은 코드로 다시 태어나기

### 리팩토링이란 무엇인가?

**리팩토링(Refactoring)**이라는 단어를 분해해보면 이해가 쉽습니다. **"Factory(공장)"**에 **"Re(다시)"**가 붙은 것입니다. 즉, "코드를 만드는 공장을 다시 돌린다", 다시 말해 **이미 만들어진 코드를 다시 만드는 것**입니다.

그런데 왜 멀쩡히 동작하는 코드를 다시 만들까요? 바로 **더 좋은 코드**로 만들기 위해서입니다. 처음에 급하게 만든 코드는 기능은 되지만 읽기 어렵고, 유지보수하기 힘들고, 비효율적일 수 있습니다. 이것을 **결과(동작)는 똑같이 유지하면서 내부 코드 구조만 개선**하는 것이 리팩토링입니다.

> 🔑 **리팩토링의 핵심**: 외부에서 보이는 동작(결과)은 변하지 않고, **내부 코드 구조만 더 좋게 변경**하는 것

리팩토링을 통해 얻을 수 있는 이점:
- **가독성 향상**: 코드를 쉽게 읽고 이해할 수 있게 됨
- **유지보수 용이성**: 나중에 코드를 수정하거나 기능을 추가하기 쉬워짐
- **버그 발견 용이**: 깔끔한 코드에서는 버그도 더 쉽게 눈에 띔
- **개발 속도 향상**: 이해하기 쉬운 코드는 다음 기능도 빨리 만들 수 있게 해줌

### 리팩토링 과정은 어떻게 진행되나?

리팩토링은 **한꺼번에 모든 것을 바꾸는 게 아닙니다.** 마치 레고 블록을 하나씩 교체하듯이, **작은 단위로 조금씩 변경하고 테스트하는 과정**을 반복합니다.

1. **단일 리팩토링 수행** → 코드의 작은 한 부분만 개선
2. **테스트 실행** → 개선 후에도 기존과 동일하게 잘 동작하는지 확인
3. **정상 동작 확인** → 문제 없으면 다음 리팩토링으로 진행
4. **오류 발생 시** → 변경을 취소하고 원래 코드로 되돌린 뒤 다시 시도

이렇게 단계별로 진행하면 리팩토링 도중에 시스템이 망가지는 위험을 최소화할 수 있습니다.

## 🦨 코드 스멜: 나쁜 코드의 징후들

### 코드 스멜이란?

**코드 스멜(Code Smell)**은 단어 그대로 "코드에서 나는 나쁜 냄새"입니다. 음식이 상하면 냄새가 나듯이, 코드에도 뭔가 잘못됐다는 신호가 있습니다. 코드 스멜은 **리팩토링이 필요하다는 징조(신호)**입니다.

:::warning 코드 스멜을 방치하면?
코드 스멜을 그냥 놔두면 시간이 지날수록 코드는 점점 더 복잡해지고, 수정하기 어려워지며, 버그가 늘어납니다. 결국 "처음부터 다시 짜는 게 낫겠다"는 상황까지 올 수 있습니다. 조기에 발견하고 리팩토링으로 해결하는 것이 훨씬 효율적입니다.
:::

### 코드 스멜의 종류와 해결책

주요 코드 스멜의 종류와 그 해결 방법을 표로 정리하면 아래와 같습니다.

| 코드 스멜 종류 | 증상 | 해결 방법 |
|-------------|------|---------|
| **중복 코드(Duplicated Code)** | 같은 코드가 여러 곳에 반복됨 | 공통 메서드로 추출하거나 부모 클래스로 이동 |
| **긴 메서드(Long Method)** | 하나의 메서드가 너무 길고 복잡 | 메서드를 적절한 크기로 분리 |
| **거대한 클래스(Large Class)** | 속성과 메서드가 너무 많은 클래스 | 클래스를 작게 분리하고 상속 구조로 재편 |
| **긴 파라미터(Long Parameter List)** | 메서드에 입력값이 너무 많음 | 파라미터 수를 줄이거나 객체로 묶어서 전달 |
| **산탄총 수술(Shotgun Surgery)** | 하나를 수정하면 여러 클래스를 동시에 수정해야 함 | 관련 코드를 한 군데로 모음 |
| **기능의 욕심(Feature Envy)** | 다른 클래스 데이터를 지나치게 많이 사용 | 해당 데이터를 그 클래스로 이동 |
| **스위치문 남용(Switch Statements)** | switch문이 너무 많이 사용됨 | 상속과 오버라이딩으로 대체 |
| **병렬 상속 계층(Parallel Inheritance Hierarchies)** | 비슷한 계층 구조가 중복으로 존재 | 상속 구조를 통합하여 효율적으로 재편 |

### 실제 리팩토링 예시 코드

    public int getLevel(boolean hasPermission, int score) {
        int level;

        if (hasPermission) {           // 첫 번째 조건
            level = 2;
        } else {
            level = 1;
        }

        if (score > 5) {               // 두 번째 조건
            return level;
        }
        return level;
    }
}

// ✅ 좋은 예: 두 조건을 하나로 통합하여 간결하게
public class UserPermission {

    public int getLevel(boolean hasPermission, int score) {
        // 삼항 연산자로 두 조건을 하나로 합쳐 코드를 간소화
        // hasPermission이 true면 2, 아니면 1
        return (score > 5) ? (hasPermission ? 2 : 1) : 1;
    }
}

// ❌ 나쁜 예: 에러를 -1 같은 숫자 코드로 반환 (의미 파악 어려움)
public class BankAccount {

    public double withdraw(double amount) {

// ✅ 좋은 예: 예외(Exception)를 던져서 명확하게 에러를 처리
public class BankAccount {

    private double balance;

    public double withdraw(double amount) throws Exception {
        if (amount > balance) {
            // throw로 예외를 던져서 명확하게 에러 상황을 알림
            throw new Exception("잔액이 부족합니다. 현재 잔액: " + balance);
        }
        balance -= amount;
        return balance;
    }
}

## 🔍 코드 인스펙션과 정적 분석: 코드를 검사하다

### 코드 인스펙션이란?

**코드 인스펙션(Code Inspection, 코드 검사)**은 프로그램이 문제없이 컴파일(번역)된 후, **분석 도구나 동료 개발자가 코드를 꼼꼼하게 검토하는 과정**입니다. 마치 공장에서 제품이 생산된 후 품질 검사팀이 불량품을 걸러내는 것처럼요.

코드 인스펙션을 통해 확인하는 항목들:
- **코드 효율성**: 불필요하게 느리게 동작하는 코드가 없는가?
- **코딩 표준 준수**: 팀에서 정한 규칙을 따랐는가?
- **결합도 검사**: 클래스 간의 연결이 적절한가?
- **로직 오류**: 알고리즘에 논리적 실수가 없는가?
- **계산 오류**: 수식이나 연산에 실수가 없는가?
- **인터페이스 오류**: 데이터를 주고받을 때 문제가 없는가?
- **데이터 처리 오류**: 데이터를 저장하고 불러올 때 문제가 없는가?

### 정적 분석(Static Analysis)이란?

**정적 분석(Static Analysis)**은 프로그램을 **실제로 실행하지 않고**, 코드 텍스트(명령어)만 분석해서 결함을 찾아내는 방법입니다. 마치 책을 읽지 않고 문법 오류만 검사하는 것처럼, **코드를 돌려보지 않아도 문제를 미리 발견**할 수 있습니다.

정적 분석은 자동화된 도구를 사용하기 때문에, 사람이 눈으로 모든 코드를 검토하기 어려운 **대형 프로젝트**에서 특히 유용합니다.

## 👫 페어 프로그래밍: 함께하면 더 좋은 코드가 나온다

### 페어 프로그래밍이란?

**페어 프로그래밍(Pair Programming, 쌍 프로그래밍)**은 말 그대로 **두 명이 짝을 이뤄 함께 프로그래밍하는 방식**입니다. 혼자 개발할 때는 오류를 발견하거나 품질을 높이는 데 한계가 있지만, 팀원과 함께하면 서로의 실수를 잡아주고 더 나은 아이디어를 낼 수 있습니다.

페어 프로그래밍의 장점:
- **오류 조기 발견**: 옆에서 바로 피드백을 주기 때문에 실수를 즉시 수정 가능
- **지식 공유**: 서로의 기술과 노하우를 자연스럽게 배울 수 있음
- **코드 품질 향상**: 두 사람의 눈이 보는 코드가 한 사람의 코드보다 완성도가 높음
- **집중력 유지**: 혼자일 때보다 파트너가 있으면 더 집중하게 됨

### 핑퐁 프로그래밍: 효과적인 페어 프로그래밍 방법

페어 프로그래밍 중에서 특히 효과적인 방식이 **핑퐁 프로그래밍(Ping-Pong Programming)**입니다. 탁구에서 서로 공을 주고받듯이, 개발과 테스트를 두 사람이 번갈아 가며 수행합니다.

- **A가 기능을 개발** → **B가 테스트**
- **B가 기능을 개발** → **A가 테스트**
- 이 과정을 크로스(cross)로 계속 반복

이렇게 하면 개발자가 자신이 만든 코드를 스스로 테스트할 때 생기는 **확증 편향(자기가 만든 건 맞을 거라는 착각)**을 피할 수 있습니다.

:::warning 페어 프로그래밍 주의사항
- **대화에 너무 많은 시간을 쏟지 않도록** 주의하세요. 토론하다가 정작 개발 시간이 부족해질 수 있습니다.
- **같은 장소(또는 같은 화면)**에 있어야 효율적입니다. 서로 일정이 맞지 않거나 다른 장소에 있으면 소통이 어렵고 비효율적이 됩니다.
- 페어 프로그래밍은 **모든 상황에서 항상 좋은 것은 아닙니다.** 프로젝트 성격과 팀 상황에 맞게 적절히 활용해야 합니다.
:::

## 🚀 시퀀스 다이어그램과 상태 다이어그램의 코드 구현

### 시퀀스 다이어그램 → 코드

**시퀀스 다이어그램(Sequence Diagram)**은 객체들이 **시간 순서(순차적)**에 따라 어떻게 상호작용하는지를 보여주는 UML 다이어그램입니다. 이 시퀀스 다이어그램을 코드로 옮기면, 객체들이 순서대로 호출되는 흐름이 코드에 그대로 반영됩니다.

```java
// ✅ 시퀀스 다이어그램 기반 코드 예시
// 사용자 → 컨트롤러 → DB → 문서 반환의 순서를 코드로 표현
public class DocumentController {

    public String getDocument(String docId) {

        // 1단계: DB에서 문서 조회 (null일 수 있음)
        String document = database.find(docId);

        // 2단계: 문서가 존재하면 반환, 없으면 에러 메시지 반환
        if (document != null) {
            // 문서를 찾은 경우: 정상 반환
            return document;
        } else {
            // 문서를 찾지 못한 경우: "Document Not Found" 메시지 반환
            return "Document Not Found";
        }
    }
}
```

### 상태 다이어그램 → 코드

**상태 다이어그램(State Diagram)**은 객체가 **어떤 상태**에서 **어떤 이벤트**가 발생하면 **어떤 행동**을 취하고 **어떤 상태로 변화**하는지를 보여줍니다. 이것을 코드로 옮기면 주로 `switch-case` 문으로 구현됩니다.

```java
// ✅ 상태 다이어그램 기반 코드 예시: switch-case로 상태 처리
public class StateMachine {

    // 현재 상태를 저장하는 속성 (상태 다이어그램의 "상태 정보" 역할)
    private int state;

    // 이벤트 발생 시 호출되는 메서드 (상태 다이어그램의 "이벤트" 역할)
    public void handleEvent(int stateValue) {

        // switch-case: 상태 값에 따라 다른 처리를 수행
        switch (stateValue) {
            case 1:
                // 상태가 1일 때 수행할 액션
                System.out.println("상태 1: A 처리 실행");
                break;  // 이 case 처리 후 switch문 종료
            case 2:
                // 상태가 2일 때 수행할 액션
                System.out.println("상태 2: B 처리 실행");
                break;
            default:
                // 1도 2도 아닌 다른 상태일 때 기본 처리
                System.out.println("알 수 없는 상태: 기본 처리 실행");
                break;
        }
    }
}
```

:::

### 좋은 클래스 설계를 위한 팁

:::tip 클래스를 잘 설계하는 방법
1. **클래스 하나는 하나의 역할만** 담당하도록 설계하세요. "단일 책임 원칙"이라고 합니다.
2. **속성은 private으로**, 접근은 getter/setter 메서드를 통해서만 하세요.
3. **메서드는 짧게** 유지하세요. 한 메서드가 너무 길어진다면 분리할 신호입니다.
4. **공통 기능은 부모 클래스에** 두고 상속받아 사용하세요. 코드 중복을 줄일 수 있습니다.
5. **의미 있는 이름**을 사용하세요. `a`, `b`, `x` 같은 이름 대신 `userId`, `orderTotal` 같이 의미가 명확한 이름을 쓰세요.
:::

## 📝 핵심 정리

- **클래스(Class)**는 **속성(데이터 저장 공간)**과 **메서드(기능/동작)**를 하나로 묶어놓은 프로그래밍의 기본 단위이다.
- **UML 클래스 다이어그램**은 클래스 이름, 속성, 메서드를 시각적으로 표현한 설계 도구이며, 이를 보고 Java 코드로 구현한다.
- **public**은 누구나 사용 가능한 공개 접근, **private**은 클래스 내부에서만 사용 가능한 제한 접근이다.
- **추상 클래스(abstract class)**는 메서드의 껍데기(선언)만 있고 실제 구현이 없는 클래스로, 자식 클래스에서 완성된다.
- **일반화(Generalization)**는 부모 클래스(superclass)와 자식 클래스(subclass)의 상속 관계를 의미하며, `extends` 키워드로 구현된다.
- **인터페이스(Interface)**는 기능의 규약(명세)만 정의하며, `implements` 키워드로 실제 클래스에서 구현(오버라이딩)된다.
- **의존, 연관, 집합**은 클래스 간의 관계를 나타내며, 설계 목적에 따라 적절히 선택하여 사용한다.
- **리팩토링(Refactoring)**은 외부 동작(결과)은 그대로 유지하면서 내부 코드 구조를 개선하는 과정이다.
- **코드 스멜(Code Smell)**은 리팩토링이 필요하다는 신호로, 중복 코드, 긴 메서드, 거대한 클래스 등이 대표적인 예이다.
- **리팩토링은 작은 단위로 단계별**로 수행하며, 매 단계마다 테스트로 동작을 검증해야 한다.
- **코드 인스펙션(Code Inspection)**은 컴파일 후 분석 도구나 동료가 코드의 결함을 검사하는 품질 관리 활동이다.
- **정적 분석(Static Analysis)**은 프로그램을 실행하지 않고 코드 텍스트만 분석하여 결함을 찾는 방법이다.
- **페어 프로그래밍(Pair Programming)**은 두 개발자가 짝을 이뤄 개발과 테스트를 분담하며 코드 품질을 높이는 협업 방식이다.
- **핑퐁 프로그래밍**은 페어 프로그래밍의 효과적인 방식으로, 개발과 테스트를 번갈아 수행하며 서로 크로스 검증한다.
- 좋은 개발자는 코딩 실력뿐 아니라 **리팩토링, 코드 품질, 코드 스멜 최소화**를 항상 염두에 두어야 한다.
