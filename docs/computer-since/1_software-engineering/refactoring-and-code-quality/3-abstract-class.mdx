# 🏗️ 추상 클래스의 개념과 활용 — UML 설계에서 코드까지

소프트웨어를 개발한다는 것은 단순히 코드를 "작성"하는 행위가 아닙니다. 마치 건물을 짓기 전에 설계 도면을 그리듯이, 프로그램도 **설계 → 구현 → 개선**의 단계를 거칩니다. 이번 문서에서는 그 과정에서 핵심적으로 등장하는 **추상 클래스(Abstract Class)** 의 개념을 중심으로, UML 설계가 실제 코드로 어떻게 변환되는지, 그리고 만들어진 코드를 더 좋게 다듬는 **리팩토링(Refactoring)** 까지 함께 살펴보겠습니다.

프로그래밍 언어를 아직 깊이 배우지 않으셨더라도 괜찮습니다. "아, 이런 구조로 설계가 코드가 되는구나"라는 큰 그림을 잡는 것이 이 문서의 목표입니다. 천천히, 차근차근 따라와 보세요.

---

## 🗺️ UML 설계가 코드로 변환되는 방식

### UML이란 무엇인가?

**UML(Unified Modeling Language, 통합 모델링 언어)** 은 소프트웨어를 설계할 때 사용하는 표준화된 설계 도구입니다. 마치 건축가가 집을 짓기 전에 평면도를 그리듯이, 개발자들은 UML을 사용해서 프로그램의 구조를 그림으로 표현합니다. 이 설계 도면을 보고 개발자들이 실제 코드를 작성하는 것이죠.

UML은 전 세계 개발자들이 공통으로 사용하는 "언어"이기 때문에, 국적이나 사용하는 프로그래밍 언어가 달라도 UML 다이어그램 하나만 보면 서로 소통이 가능합니다.

### 클래스(Class)란 무엇인가?

**클래스(Class)** 는 프로그래밍에서 가장 기본이 되는 개념 중 하나입니다. 쉽게 말하면 **속성(Attribute, 값을 담는 그릇)** 과 **메서드(Method, 동작을 수행하는 기능)** 를 하나로 묶어 놓은 설계 틀입니다.

비유를 들어볼게요. 우리가 "사람"이라는 개념을 생각해보면:
- **속성**: 이름, 나이, 키, 몸무게 (사람이 가지고 있는 특성/값)
- **메서드**: 걷기, 말하기, 먹기 (사람이 할 수 있는 행동/기능)

이것을 프로그램으로 표현한 것이 바로 클래스입니다.

UML에서는 다음과 같은 표 형태로 클래스를 표현합니다.

| UML 구성 요소 | 설명 | 예시 |
|---|---|---|
| 클래스 이름 | 이 클래스가 무엇을 나타내는지 | `User` (사용자) |
| 속성(Attribute) | 클래스가 가지고 있는 데이터 | `age: int`, `name: String` |
| 메서드(Method) | 클래스가 수행하는 기능 | `getSchedule()`, `introduce()` |

이 UML 설계도를 받은 Java 개발자는 다음과 같이 코드를 작성합니다.

```java
// UML 설계도를 기반으로 작성한 User 클래스
public class User {

    // 속성(Attribute): 사용자의 데이터를 담는 변수들
    private int age;       // 나이 (정수형)
    private String name;   // 이름 (문자열)

    // 메서드(Method): 사용자가 수행하는 기능들
    public void getSchedule() {
        // 스케줄을 가져오는 기능 구현
    }

    public void introduce() {
        // 자기소개를 하는 기능 구현
    }
}
```

여기서 `public`은 **공용(외부에서 누구나 접근 가능)** 이라는 의미이고, `private`는 **내부 전용(이 클래스 안에서만 사용 가능)** 이라는 의미입니다. 마치 건물의 공용 공간과 직원 전용 공간처럼 접근 범위를 제한하는 것입니다.

:::info UML을 보는 관점
지금 당장 코드를 완벽히 이해하지 못해도 괜찮습니다. "UML이라는 설계 도면이 이런 형태의 코드로 만들어지는구나"라는 **구조적 흐름**을 이해하는 것이 이 단계에서의 목표입니다. 코드의 세부 문법은 Java나 C 언어를 공부하면서 자연스럽게 익히게 됩니다.
:::

## 🎭 추상 클래스(Abstract Class)란 무엇인가?

### 왜 추상 클래스가 필요한가?

일반 클래스는 속성과 메서드가 모두 구체적으로 구현되어 있습니다. 그런데 현실에서는 "아직 구체적인 내용은 모르지만, 이런 기능이 있어야 한다는 것은 알고 있을 때"가 있습니다.

예를 들어, 동물원 관리 시스템을 만든다고 가정해봅시다. 모든 동물은 "소리를 낸다"는 기능을 가지고 있습니다. 하지만 개는 "멍멍", 고양이는 "야옹", 사자는 "으르렁"처럼 소리를 내는 **방법**은 각기 다릅니다. 이때 "소리를 낸다"는 기능의 **틀(껍데기)** 만 먼저 만들어 두고, 구체적인 내용은 나중에 각 동물 클래스에서 채우는 방식을 사용합니다. 이것이 바로 **추상 클래스(Abstract Class)** 입니다.

> **추상 클래스란?** 한 개 이상의 메서드가 구체적인 구현 없이 껍데기(명세, 스펙)만 존재하는 클래스입니다. "이런 기능이 있어야 한다"는 약속만 해두고, 실제 내용은 이 클래스를 가져다 쓰는 쪽에서 채우도록 합니다.

### 일반 클래스 vs 추상 클래스 비교

| 구분 | 일반 클래스 | 추상 클래스 |
|---|---|---|
| 키워드 | `class` | `abstract class` |
| 메서드 구현 | 모든 메서드가 구현되어 있음 | 구현 없이 선언만 된 메서드 포함 가능 |
| 직접 객체 생성 | 가능 | 불가능 (반드시 상속받아서 사용) |
| 용도 | 완성된 설계 | 공통 틀(뼈대) 제공 |

### 추상 클래스 코드 예시

```java
// 추상 클래스 선언 - abstract 키워드를 class 앞에 붙임
public abstract class User {

    // 속성은 일반 클래스와 동일하게 가질 수 있음
    private String name; // 사용자 이름

    // 추상 메서드: 껍데기만 있고 내용(구현)이 없음
    // 이 클래스를 상속받는 곳에서 반드시 내용을 채워야 함
    public abstract void work();
}
```

```java
// 추상 클래스를 상속받아 실제 기능을 구현하는 클래스
public class Employee extends User {

    // 추상 메서드 work()를 실제 내용으로 채움 (오버라이딩)
    @Override
    public void work() {
        System.out.println("직원이 업무를 수행합니다."); // 실제 동작 구현
    }
}
```

위 코드에서 `User`라는 추상 클래스는 `work()`라는 메서드의 껍데기만 가지고 있습니다. 실제 내용은 `Employee`라는 클래스가 상속받아서 채워 넣습니다. 이렇게 하면 `User`를 기반으로 다양한 종류의 사용자 클래스를 효율적으로 만들 수 있습니다.

:::tip 추상 클래스를 레시피로 이해하기
추상 클래스는 마치 **요리 레시피의 목차** 같습니다. "1. 재료를 손질한다, 2. 볶는다, 3. 양념을 넣는다"라는 순서는 정해져 있지만, 각 요리마다 구체적으로 어떤 재료를 어떻게 손질할지는 레시피마다 다릅니다. 추상 클래스는 이 "순서와 틀"을 제공하고, 실제 구체적인 내용은 각각의 클래스가 채우는 것입니다.
:::

## 🌳 일반화(Generalization)와 상속(Inheritance)

### 일반화란 무엇이고 왜 필요한가?

**일반화(Generalization)** 란 슈퍼클래스(Superclass, 부모 클래스)와 서브클래스(Subclass, 자식 클래스) 사이의 **상속 관계**를 말합니다. 이것이 필요한 이유는 코드의 중복을 없애고, 공통 기능을 한 곳에서 관리하기 위해서입니다.

실생활 예시를 들어봅시다. 쇼핑몰 시스템을 만든다고 하면:
- **고객(Customer)**: 상품을 주문하고 결제하는 사람
- **관리자(Admin)**: 상품을 등록하고 시스템을 관리하는 사람

이 두 유형 모두 "사용자(User)"입니다. 둘 다 아이디와 패스워드가 있고, 로그인 기능이 필요합니다. 그렇다면 로그인 기능을 고객 클래스에도 따로 만들고, 관리자 클래스에도 따로 만드는 것은 매우 비효율적입니다. 공통 기능을 `User`라는 부모 클래스에 한번만 만들어 두고, `Customer`와 `Admin`이 그것을 **상속(Inheritance)** 받아서 사용하면 훨씬 효율적입니다.

```java
// 슈퍼클래스(부모 클래스): 공통 기능을 한 곳에 모아둠
public class User {

    // 공통 속성: 고객도 관리자도 모두 아이디와 패스워드를 가짐
    private String id;       // 아이디
    private String password; // 패스워드

    // 공통 메서드: 로그인 기능은 모든 사용자가 공통으로 사용
    public void login() {
        System.out.println("로그인을 수행합니다.");
    }
}
```

```java
// 서브클래스(자식 클래스): User를 상속받아 고객 클래스를 만듦
// extends 키워드로 부모 클래스를 상속받음
public class Customer extends User {

    // Customer만의 고유한 로그인 방식으로 오버라이딩(재정의)
    @Override
    public void login() {
        System.out.println("고객 전용 로그인을 수행합니다."); // 고객 맞춤 로그인
    }
}
```

```java
// 서브클래스: User를 상속받아 관리자 클래스를 만듦
public class Admin extends User {

    // Admin만의 고유한 로그인 방식으로 오버라이딩(재정의)
    @Override
    public void login() {
        System.out.println("관리자 전용 로그인을 수행합니다."); // 관리자 맞춤 로그인
    }
}
```

이처럼 `User`(부모)가 가진 `login()` 메서드를 `Customer`와 `Admin`(자식)이 각자의 방식으로 재정의하는 것을 **오버라이딩(Overriding, 재정의)** 이라고 합니다.

### 실체화(Realization)와 인터페이스(Interface)

**인터페이스(Interface)** 는 추상 클래스보다 한 단계 더 나아간 개념입니다. 추상 클래스는 일반 메서드와 추상 메서드를 함께 가질 수 있지만, 인터페이스는 **오직 껍데기만 있는 메서드(명세)** 로만 구성됩니다.

**실체화(Realization)** 는 인터페이스의 껍데기를 실제 기능으로 채워서 구현하는 것을 말합니다. 인터페이스는 `implements`라는 키워드를 사용해서 상속받습니다.

```java
// 인터페이스: 완전히 껍데기만 있는 기능 명세
// class 대신 interface 키워드 사용
public interface CheckLog {

    // 로그를 확인하는 기능의 껍데기만 선언
    // 실제 내용은 이 인터페이스를 구현하는 클래스에서 채움
    boolean checkLog(); // 성공이면 true, 실패면 false 반환
}
```

```java
// 인터페이스를 실제 클래스로 구현 (실체화)
// implements 키워드로 인터페이스를 상속받음
public class DateCheckLog implements CheckLog {

    // 인터페이스에 정의된 껍데기 메서드를 실제 내용으로 채움 (오버라이딩)
    @Override
    public boolean checkLog() {
        // 날짜 기반으로 로그를 확인하는 실제 로직
        System.out.println("날짜 기반 로그 확인 수행");
        return true; // 확인 성공 시 true 반환
    }
}
```

| 구분 | 추상 클래스 (Abstract Class) | 인터페이스 (Interface) |
|---|---|---|
| 키워드 | `abstract class` | `interface` |
| 상속 키워드 | `extends` | `implements` |
| 일반 메서드 포함 | 가능 | 불가능 (모두 껍데기) |
| 다중 상속 | 불가능 | 가능 |
| 목적 | 공통 기능 + 틀 제공 | 순수한 기능 명세만 제공 |

:::note 오버라이딩 vs 오버로딩
- **오버라이딩(Overriding)**: 부모 클래스나 인터페이스에서 물려받은 메서드를 자식 클래스에서 **새롭게 재정의**하는 것
- **오버로딩(Overloading)**: 같은 이름의 메서드를 **매개변수(입력값)의 종류나 개수**를 다르게 해서 여러 개 만드는 것

두 개념은 이름이 비슷해서 혼동하기 쉽지만 완전히 다른 개념입니다.
:::

## 🔧 리팩토링(Refactoring) — 좋은 코드로 다시 만들기

### 리팩토링이란 무엇이고 왜 필요한가?

**리팩토링(Refactoring)** 은 이미 만들어진 코드를 **결과는 그대로 유지하면서, 내부 구조를 더 좋게 개선하는 과정**입니다. "Re(다시) + Factory(만들다)"라는 의미에서 알 수 있듯이, 한 번 만든 것을 다시 더 잘 만드는 것입니다.

왜 이미 잘 돌아가는 코드를 다시 손봐야 할까요? 마치 처음 이사한 집을 정리할 때는 일단 짐을 아무 데나 쌓아두는 것처럼, 처음 개발할 때는 일단 동작하게 만드는 것이 우선입니다. 하지만 그 상태로 계속 살다 보면 어디에 무엇이 있는지 찾기도 힘들고, 나중에 가구를 추가하거나 이동시키기도 어렵습니다. **리팩토링은 바로 이 "집 정리"에 해당합니다.**

> **리팩토링의 핵심 원칙:** 외부에서 보이는 동작(결과)은 변경하지 않고, 내부 코드의 구조만 개선합니다. 화면에 표시되는 내용이나 기능 자체가 바뀌는 것이 아니라, 그 기능을 수행하는 코드의 품질이 좋아지는 것입니다.

리팩토링을 통해 얻을 수 있는 이점:
- **가독성(Readability)**: 코드를 읽고 이해하기 쉬워집니다
- **유지보수성(Maintainability)**: 나중에 수정하거나 기능을 추가하기 쉬워집니다
- **버그 발견 용이성**: 코드가 깔끔해지면 숨어있던 버그도 찾기 쉬워집니다
- **개발 속도 향상**: 코드가 정리되어 있으면 새 기능을 더 빠르게 추가할 수 있습니다

### 리팩토링 진행 방식

리팩토링은 절대 한꺼번에 모든 것을 바꾸지 않습니다. **작은 단위로 조금씩** 변경하고, 변경할 때마다 코드가 정상적으로 동작하는지 확인하는 방식으로 진행합니다.

```
1단계: 작은 부분 리팩토링
    ↓
2단계: 테스트 (잘 동작하는지 확인)
    ↓
3단계: 정상 동작 → 다음 리팩토링으로 이동
       비정상 동작 → 변경 취소, 원래대로 복구
    ↓
4단계: 반복
```

이 방식의 장점은 **언제든지 안전하게 되돌릴 수 있다**는 것입니다. 리팩토링 전에도 코드가 잘 동작했으므로, 문제가 생기면 이전 상태로 돌아가면 됩니다.

## 🦨 코드 스멜(Code Smell) — 나쁜 코드의 징후들

### 코드 스멜이란 무엇인가?

**코드 스멜(Code Smell)** 이란 코드에서 풍기는 "나쁜 냄새"입니다. 실제로 코드에서 냄새가 나는 것은 아니지만, 마치 음식이 상하면 냄새가 나듯이 코드에도 "이 코드는 문제가 있을 수 있다"는 신호를 보내는 특징들이 있습니다. 이런 특징을 코드 스멜이라고 부릅니다.

코드 스멜은 당장 프로그램이 작동하지 않는 버그는 아닙니다. 하지만 내버려 두면 유지보수가 점점 어려워지고, 결국 큰 문제로 이어집니다. 마치 방을 조금씩 어지르다 보면 나중에는 도저히 손댈 수 없는 상태가 되는 것처럼요.

### 코드 스멜의 종류와 해결 방법

**1. 중복 코드 (Duplicated Code)**

같은 코드가 여러 곳에 반복되어 있는 경우입니다. 나중에 수정이 필요할 때 모든 중복된 곳을 다 찾아서 수정해야 하므로 매우 위험합니다.

```java
// ❌ 나쁜 예시: 같은 출력 코드가 여러 곳에 중복됨
public void processOrder() {
    System.out.println("주문자: " + name);   // 중복!
    System.out.println("금액: " + price);
    // ... 주문 처리 로직
}

public void cancelOrder() {
    System.out.println("주문자: " + name);   // 중복!
    System.out.println("금액: " + price);
    // ... 취소 처리 로직
}
```

```java
// ✅ 좋은 예시: 중복 코드를 별도 메서드로 추출
// 출력 기능을 printDetail()이라는 메서드로 분리
public void printDetail() {
    System.out.println("주문자: " + name);   // 한 곳에서만 관리
    System.out.println("금액: " + price);
}

public void processOrder() {
    printDetail(); // 메서드를 호출해서 사용
    // ... 주문 처리 로직
}

public void cancelOrder() {
    printDetail(); // 동일하게 호출해서 사용
    // ... 취소 처리 로직
}
```

**2. 너무 긴 메서드 (Long Method)**

하나의 메서드가 지나치게 많은 일을 하고 있어서 코드가 길어지는 경우입니다. 메서드는 하나의 일만 잘 하도록 적절한 크기로 분리해야 합니다.

**3. 너무 큰 클래스 (Large Class)**

하나의 클래스에 속성과 메서드가 너무 많은 경우입니다. 클래스를 작게 나누고 상속을 활용해서 책임을 분산시키는 것이 좋습니다.

**4. 너무 많은 매개변수 (Long Parameter List)**

메서드가 입력으로 받는 매개변수(파라미터)가 지나치게 많은 경우입니다. 관련된 매개변수들은 별도의 객체로 묶어서 전달하는 것이 좋습니다.

```java
// ❌ 나쁜 예시: 매개변수가 너무 많음
public void createUser(String name, int age, String email,
                       String phone, String address, String city) {
    // 매개변수가 6개나 됨 → 호출할 때마다 순서를 기억해야 함
}
```

```java
// ✅ 좋은 예시: 관련 매개변수를 객체로 묶음
// UserInfo라는 객체에 관련 정보를 담아서 전달
public void createUser(UserInfo userInfo) {
    // 하나의 객체만 받으면 됨 → 훨씬 깔끔하고 확장성도 좋음
}
```

**5. 조건문 분기에 중복된 코드**

조건문의 모든 분기 안에 같은 코드 조각이 반복되는 경우, 그 코드를 조건문 밖으로 꺼내야 합니다.

```java
// ❌ 나쁜 예시: send()가 조건문 두 분기에 모두 중복됨
if (isSpecialDeal) {
    price = price * 0.95; // 할인 적용
    send();               // 중복!
} else {
    send();               // 중복!
}
```

```java
// ✅ 좋은 예시: 중복된 send()를 조건문 밖으로 꺼냄
if (isSpecialDeal) {
    price = price * 0.95; // 할인 적용만 조건문 안에
}
send(); // 공통 코드는 조건문 밖에서 한 번만 실행
```

**6. 에러 코드 대신 예외 처리 사용**

에러 상황을 `-1`이나 `0` 같은 특수한 숫자 값으로 표현하지 말고, `Exception(예외)` 처리를 활용해야 합니다.

```java
// ❌ 나쁜 예시: 에러 상황을 -1이라는 숫자값으로 표현
public int withdraw(int amount) {
    if (amount > balance) {
        return -1; // -1이 "잔액 부족"을 의미한다고 누가 알겠는가?
    }
    return balance - amount;
}
```

```java
// ✅ 좋은 예시: 에러 상황을 Exception(예외)으로 명확하게 표현
public int withdraw(int amount) {
    if (amount > balance) {
        // 잔액 부족 예외를 명확한 메시지와 함께 던짐
        throw new IllegalArgumentException("잔액이 부족합니다.");
    }
    return balance - amount; // 정상 처리 결과 반환
}
```

**7. 배열 대신 객체 사용 (Replace Array with Object)**

배열의 인덱스(위치 번호)에 직접 값을 넣는 방식 대신, 의미 있는 이름을 가진 객체를 만들어 사용합니다.

```java
// ❌ 나쁜 예시: 배열 인덱스로 데이터를 관리
String[] row = new String[2];
row[0] = "Liverpool"; // 0번이 팀 이름인지 어떻게 알지?
row[1] = "15";        // 1번이 순위인지 어떻게 알지?
```

```java
// ✅ 좋은 예시: 의미 있는 이름을 가진 객체로 대체
// TeamRecord 객체로 팀 기록을 명확하게 표현
TeamRecord record = new TeamRecord();
record.teamName = "Liverpool"; // 이름에서 의미가 바로 드러남
record.ranking = 15;           // 이름에서 의미가 바로 드러남
```

**8. public 필드를 private으로 변경**

클래스의 내부 데이터(필드)를 외부에서 마음대로 접근하게 두지 말고, `private`으로 감추고 접근자 메서드를 통해서만 접근하도록 해야 합니다.

```java
// ❌ 나쁜 예시: public 필드는 외부에서 마음대로 값을 바꿀 수 있어 위험
public class User {
    public String name; // 누구든 user.name = "해킹" 이런 식으로 바꿀 수 있음
}
```

```java
// ✅ 좋은 예시: private으로 감추고 접근자 메서드(getter/setter)를 통해 접근
public class User {
    private String name; // 외부에서 직접 접근 불가

    // 값을 가져오는 메서드 (getter)
    public String getName() {
        return name;
    }

    // 값을 설정하는 메서드 (setter): 검증 로직 추가도 가능
    public void setName(String name) {
        if (name != null && !name.isEmpty()) { // 빈 값 검증
            this.name = name;
        }
    }
}
```

:::warning 코드 스멜 방치의 위험성
코드 스멜은 당장 프로그램이 죽지는 않습니다. 하지만 방치하면 시간이 지날수록 코드는 점점 복잡해지고, 새로운 기능 추가나 버그 수정이 점점 어려워집니다. 나중에는 아무도 손대기 싫은 "레거시 코드(오래되고 복잡한 코드)"가 되어버립니다. 초기부터 코드 스멜을 인식하고 리팩토링 습관을 들이는 것이 중요합니다.
:::

## 🔍 코드 인스펙션과 정적 분석

### 코드 인스펙션이란?

**코드 인스펙션(Code Inspection, 코드 검사)** 은 프로그램이 컴파일(번역)은 잘 되었더라도, 분석 도구나 팀원들이 직접 코드를 꼼꼼히 검사해서 숨어있는 문제점을 찾아내는 과정입니다.

코드 인스펙션에서 확인하는 항목들:

| 검사 항목 | 설명 |
|---|---|
| 코딩 표준 준수 | 팀이나 회사에서 정한 코딩 규칙을 잘 따랐는가? |
| 결합 오류 | 여러 모듈을 연결하는 부분에 문제가 없는가? |
| 로직 오류 | 프로그램의 논리적 흐름에 문제가 없는가? |
| 계산 오류 | 숫자 계산 부분에 실수가 없는가? |
| 인터페이스 오류 | 모듈 간 데이터를 주고받는 부분에 문제가 없는가? |
| 데이터 처리 오류 | 데이터를 저장하고 불러오는 부분에 오류가 없는가? |
| 타이밍 오류 | 작업의 순서나 시간 처리에 문제가 없는가? |

코드 인스펙션을 정기적으로 수행하면 **코드의 품질이 전반적으로 향상**되고, 나중에 발생할 수 있는 큰 버그를 사전에 예방할 수 있습니다.

### 정적 분석(Static Analysis)이란?

**정적 분석(Static Analysis)** 은 프로그램을 실제로 실행하지 않고, **코드 텍스트(명령어 자체)를 분석해서 결함을 찾는 방법**입니다. 마치 책을 읽지 않고도 맞춤법 검사기가 오류를 찾아내는 것처럼, 프로그램이 돌아가지 않아도 코드 자체를 분석해서 문제를 미리 발견할 수 있습니다.

정적 분석은 다양한 자동화 도구를 통해 수행할 수 있으며, 코드를 작성하는 단계에서부터 즉각적인 피드백을 받을 수 있다는 장점이 있습니다.

## 👥 페어 프로그래밍(Pair Programming)

### 페어 프로그래밍이란 무엇이고 왜 유용한가?

**페어 프로그래밍(Pair Programming, 쌍 프로그래밍)** 은 두 명의 개발자가 함께 하나의 작업을 진행하는 방식입니다. 혼자 개발하면 내가 만든 코드의 문제점을 내가 직접 발견하기가 어렵습니다. 마치 내가 쓴 글의 오탈자를 내가 교정하기 어려운 것처럼요. 다른 사람의 눈으로 보면 금방 보이는 것이 자기 눈에는 보이지 않는 경우가 많습니다.

페어 프로그래밍은 이런 문제를 해결하기 위해:
- **토론**: 어떻게 구현할지 함께 의논
- **상호 검토**: 한 명이 코드를 작성하면 다른 한 명이 실시간으로 검토
- **협력**: 서로의 지식과 경험을 공유

하는 방식으로 진행됩니다.

### 핑퐁 프로그래밍(Ping-Pong Programming)

페어 프로그래밍의 효과적인 운영 방법 중 하나가 **핑퐁 프로그래밍**입니다. 마치 탁구처럼 서로 주고받는 방식으로 역할을 교대합니다.

```
A 개발자: 기능 개발 → B 개발자: 테스트
B 개발자: 기능 개발 → A 개발자: 테스트
A 개발자: 기능 개발 → B 개발자: 테스트
... (반복)
```

이렇게 하면 개발과 테스트가 분리되면서도 지속적으로 품질을 확인할 수 있습니다.

:::warning 페어 프로그래밍의 주의사항
페어 프로그래밍이 항상 효율적인 것은 아닙니다. 잘못 운영하면 오히려 비효율이 생길 수 있습니다.

- **대화에 너무 많은 시간을 쏟지 마세요**: 코딩보다 토론에 더 많은 시간이 걸리면 본말전도입니다
- **같은 장소에 있어야 효율적입니다**: 서로 다른 장소에 있으면 일정 조율만으로도 많은 시간이 낭비됩니다
- **일정 관리가 중요합니다**: 각자 언제 작업할 수 있는지 미리 조율하세요
:::

:::tip 혼자 개발할 때도 활용하는 팁
팀 프로젝트가 아니어서 페어 프로그래밍을 못 한다면, **코드 리뷰(Code Review)** 를 활용하세요. 내가 작성한 코드를 다른 팀원에게 검토 요청을 하거나, 아니면 하루 뒤에 내가 다시 내 코드를 읽어보는 것도 좋은 방법입니다. 시간이 지난 뒤 "타인의 눈"으로 코드를 보면 문제점이 더 잘 보입니다.
:::

## ⚠️ 주의사항 및 실전 팁

### 리팩토링할 때 절대 하면 안 되는 것들

:::danger 리팩토링 중 기능 변경 금지
리팩토링은 **외부 동작을 바꾸지 않으면서** 내부 구조를 개선하는 것입니다. 리팩토링과 기능 추가/변경을 동시에 하면 어떤 변경이 어떤 문제를 일으켰는지 추적하기가 매우 어려워집니다. 반드시 **리팩토링과 기능 개발은 분리해서 진행**하세요.
:::

:::danger 테스트 없는 리팩토링 금지
리팩토링을 하고 나서 테스트를 하지 않으면, 내가 의도치 않게 기존 기능을 망가뜨렸는지 알 수 없습니다. 모든 리팩토링 단계마다 반드시 테스트를 수행해서 코드가 여전히 정상 동작하는지 확인하세요.
:::

### 추상 클래스 설계 시 자주 하는 실수

**실수 1: 추상 클래스를 너무 많이 만들기**

모든 것을 추상화하려 하면 오히려 코드가 복잡해집니다. 진짜로 공통 기능이 필요하고, 다양한 구현이 필요한 경우에만 추상 클래스를 사용하세요.

**실수 2: 추상 클래스와 인터페이스의 혼동**

- 여러 클래스의 **공통 기능**을 묶고 싶다면 → **추상 클래스**
- 클래스가 **반드시 구현해야 할 기능의 약속(계약)** 을 정의하고 싶다면 → **인터페이스**

**실수 3: 상속을 너무 깊게 쌓기**

부모 → 자식 → 손자 → 증손자... 처럼 상속 계층이 너무 깊어지면 코드를 이해하기 매우 어려워집니다. 일반적으로 상속 계층은 2~3단계를 넘지 않도록 설계하는 것이 좋습니다.

## 📋 핵심 정리

- **UML(통합 모델링 언어)** 은 소프트웨어 설계를 그림으로 표현하는 표준 도구이며, 이 설계도가 실제 코드로 변환됩니다
- **클래스(Class)** 는 속성(데이터)과 메서드(기능)를 하나로 묶어 놓은 설계 틀입니다
- **추상 클래스(Abstract Class)** 는 메서드의 껍데기(명세)만 가지고 있는 클래스로, 반드시 상속받아서 내용을 채워 사용합니다
- **인터페이스(Interface)** 는 추상 클래스보다 더 순수하게 기능 명세만 담고 있으며, `implements`로 상속받아 구현합니다
- **오버라이딩(Overriding)** 은 부모로부터 물려받은 메서드를 자식 클래스에서 새롭게 재정의하는 것입니다
- **일반화(Generalization)** 는 공통 기능을 부모 클래스에 모아두고 자식 클래스가 상속받아 코드 중복을 줄이는 방식입니다
- **리팩토링(Refactoring)** 은 외부 동작은 유지하면서 내부 코드 구조를 더 좋게 개선하는 과정입니다
- **코드 스멜(Code Smell)** 은 나쁜 코드의 징후로, 중복 코드, 긴 메서드, 큰 클래스, 복잡한 조건문 등이 해당됩니다
- **리팩토링은 작은 단위로 진행**하고, 매 단계마다 테스트를 수행해야 안전합니다
- **코드 인스펙션(Code Inspection)** 은 코드를 꼼꼼히 검사해서 숨어있는 결함을 찾아내는 과정으로 코드 품질을 높입니다
- **정적 분석(Static Analysis)** 은 프로그램을 실행하지 않고 코드 텍스트를 분석해서 결함을 미리 찾는 방법입니다
- **페어 프로그래밍(Pair Programming)** 은 두 명이 함께 개발하며 서로 검토하고 협력해 코드 품질을 높이는 방법입니다
- **핑퐁 프로그래밍** 은 페어 프로그래밍에서 개발과 테스트 역할을 번갈아가며 수행하는 효율적인 운영 방식입니다
- 좋은 개발자는 코딩 실력만 아니라 리팩토링, 코드 품질 관리, 코드 스멜 최소화를 항상 염두에 두어야 합니다
