# 🏗️ UML 설계와 코드 구현의 연결 — 설계도가 코드가 되는 순간

소프트웨어 개발은 단순히 코드를 타이핑하는 작업이 아닙니다. 집을 지을 때 설계도 없이 벽돌을 쌓지 않듯, 좋은 소프트웨어는 반드시 **설계 → 구현** 의 순서를 따릅니다. 이 문서에서는 **UML(통합 모델링 언어)** 로 그려진 설계 도면이 실제 Java 코드로 어떻게 변환되는지, 그리고 완성된 코드를 더 좋게 다듬는 **리팩토링(Refactoring)** 이란 무엇인지를 아주 친절하게 살펴봅니다.

:::info 이 문서를 읽는 방법
프로그래밍 언어(Java, C 등)를 아직 배우지 않으셨다면, 코드 자체를 완벽히 이해하려 하지 않아도 됩니다. **"아, 설계한 내용이 이런 형태로 코드가 되는구나"** 라는 큰 그림을 잡는 것이 목표입니다. 나중에 Java를 공부할 때 이 문서를 다시 보면 훨씬 빠르게 이해가 될 것입니다.
:::

## 📐 UML이란 무엇인가? — 소프트웨어의 설계도

### UML의 개념

**UML(Unified Modeling Language, 통합 모델링 언어)** 은 소프트웨어를 개발하기 전에 그 구조와 동작을 **그림(도형과 화살표)으로 표현하는 표준화된 설계 언어**입니다. 마치 건축가가 건물을 짓기 전에 도면을 그리는 것처럼, 개발자들은 UML을 통해 소프트웨어의 설계를 시각적으로 표현하고 팀원들과 공유합니다.

UML은 단순한 그림이 아닙니다. **전 세계 개발자들이 공통으로 이해할 수 있도록 규칙이 정해진 표준 언어**이기 때문에, 한국 개발자가 그린 UML 다이어그램을 미국 개발자도 동일하게 해석할 수 있습니다. 이것이 "통합(Unified)"이라는 단어가 붙은 이유입니다.

### 왜 UML이 필요한가?

코드를 바로 작성하면 안 될까요? 물론 작은 프로그램은 설계 없이 바로 코딩할 수도 있습니다. 하지만 수십 명의 개발자가 수년에 걸쳐 만드는 대형 시스템이라면 얘기가 달라집니다. 설계도 없이 작업을 시작하면 각자 다른 방식으로 코드를 작성하게 되고, 나중에 이를 합칠 때 심각한 문제가 발생합니다.

> UML은 개발 팀 전체가 **같은 언어로 대화**할 수 있게 해주는 공통 설계 언어입니다. 설계 단계에서 문제를 발견하면 코드 한 줄 수정으로 끝나지만, 코딩이 끝난 후 문제를 발견하면 수백 줄을 고쳐야 할 수도 있습니다.

### UML에서 코드로 — 전체 흐름

UML 설계가 완료되면 개발자는 그 설계 문서를 받아 실제 프로그래밍 언어로 구현합니다. 이 문서에서는 **Java 언어**를 기준으로 UML의 각 요소가 어떻게 코드로 변환되는지 살펴볼 것입니다.

| UML 요소 | Java 코드 요소 | 설명 |
|---|---|---|
| 클래스 이름 | `class 이름` | 설계도의 클래스 이름 그대로 사용 |
| 속성(Attribute) | 클래스 내부 변수 | 데이터를 저장하는 틀 |
| 메서드(Method) | 클래스 내부 함수 | 실제 동작을 수행하는 기능 |
| 상속 관계 | `extends` 키워드 | 부모 클래스의 기능을 물려받음 |
| 인터페이스 | `interface` + `implements` | 껍데기 정의를 실제로 구현 |

## 🧱 클래스(Class) — 설계의 기본 단위

### 클래스란 무엇인가?

**클래스(Class)** 는 소프트웨어 설계의 가장 기본 단위입니다. 클래스는 **속성(Attribute, 데이터를 담는 틀)** 과 **메서드(Method, 실제 동작을 수행하는 기능)** 를 하나로 묶어 놓은 것입니다.

실생활로 비유하자면, 클래스는 **붕어빵 틀**과 같습니다. 붕어빵 틀 자체는 붕어빵이 아닙니다. 하지만 그 틀을 이용해 반죽을 넣으면(속성에 값을 넣으면) 실제 붕어빵(객체, Object)이 만들어집니다. 클래스는 설계도이고, 그 설계도로 만들어진 실제 결과물이 **객체(Object)** 입니다.

### UML 클래스 다이어그램 → Java 코드 변환

UML에서 `User`라는 클래스를 다음과 같이 설계했다고 가정해봅시다.

```
┌─────────────────┐
│      User       │  ← 클래스 이름
├─────────────────┤
│ - age : int     │  ← 속성(나이, 정수형)
│ - name : String │  ← 속성(이름, 문자열)
├─────────────────┤
│ + getSchedule() │  ← 메서드(스케줄 가져오기)
│ + introduce()   │  ← 메서드(자기소개)
└─────────────────┘
```

이 UML 설계도를 Java 개발자가 받으면 다음과 같이 코드를 작성합니다.

```java
// UML의 클래스 이름 'User'를 그대로 Java 클래스로 표현
public class User {

    // ── 속성(Attribute) 영역 ──────────────────────────
    // private: 이 클래스 내부에서만 사용 가능한 변수
    private int age;        // UML의 'age : int' → 나이(정수)
    private String name;    // UML의 'name : String' → 이름(문자열)

    // ── 메서드(Method) 영역 ──────────────────────────
    // public: 외부에서도 호출(불러다가 쓸 수) 있는 기능
    public void getSchedule() {
        // 스케줄을 가져오는 실제 기능 구현 자리
    }

    public void introduce() {
        // 자기소개를 수행하는 실제 기능 구현 자리
    }
}
```

### 코드 한 줄씩 뜯어보기

- `public class User` : **public**은 "공개"라는 뜻으로, 다른 곳에서도 이 클래스를 불러다 쓸 수 있습니다. **class**는 클래스를 선언한다는 Java 키워드이고, **User**는 UML 설계도에서 정의한 클래스 이름입니다.
- `private int age` : **private**는 "내부 전용"이라는 뜻으로, 이 User 클래스 안에서만 사용할 수 있는 변수입니다. **int**는 정수(1, 2, 3 같은 숫자)를 저장하는 데이터 타입이고, **age**는 나이를 담을 변수 이름입니다.
- `private String name` : **String**은 문자열(글자들의 묶음)을 저장하는 데이터 타입입니다. "홍길동" 같은 이름을 저장할 수 있습니다.
- `public void getSchedule()` : **void**는 "이 메서드는 결과값을 돌려주지 않는다"는 의미입니다. **getSchedule**은 메서드 이름이고, 뒤에 붙는 `()`는 "이것은 메서드(기능)입니다"라는 표시입니다.

:::tip public vs private 쉽게 이해하기
- **public** = 건물의 **로비**: 누구나 들어올 수 있음
- **private** = 건물의 **사무실 내부**: 직원(해당 클래스)만 들어올 수 있음

속성(데이터)은 보통 `private`으로 보호하고, 외부에서 필요할 때만 `public` 메서드를 통해 접근하도록 만드는 것이 좋은 설계 방식입니다.
:::

## 🎭 추상 클래스, 일반화, 실체화 — 클래스의 다양한 종류

### 추상 클래스(Abstract Class) — 껍데기만 있는 설계

**추상 클래스(Abstract Class)** 는 **하나 이상의 메서드가 실제 구현 내용 없이 이름(명세)만 정의되어 있는 클래스**입니다. 마치 "나중에 내용을 채워 넣을 빈 양식지" 같은 개념입니다.

왜 이런 것이 필요할까요? 예를 들어, "모든 동물은 소리를 낸다"는 것은 알지만, 개가 내는 소리와 고양이가 내는 소리는 다릅니다. 이때 `Animal`이라는 추상 클래스에 `makeSound()`라는 메서드의 이름만 정의해두고, 실제 소리는 `Dog`, `Cat` 각각의 클래스에서 구현하도록 하는 것이 추상 클래스의 역할입니다.

```java
// abstract 키워드가 붙으면 '추상 클래스'가 됩니다
// 일반 클래스(class User)와 다르게 abstract class User로 선언
public abstract class User {

    // 메서드의 이름과 형태만 있고, 내용({}) 이 없습니다
    // 이것을 '추상 메서드'라고 합니다
    public abstract void work(); // 실제 동작은 이 클래스를 상속받은 곳에서 구현

}
```

:::note 일반 클래스 vs 추상 클래스 비교

| 구분 | 일반 클래스 | 추상 클래스 |
|---|---|---|
| 선언 방법 | `class User` | `abstract class User` |
| 메서드 내용 | 반드시 내용이 있어야 함 | 내용 없는 추상 메서드 가능 |
| 직접 객체 생성 | 가능 (`new User()`) | 불가능 (상속받아서만 사용) |
| 용도 | 바로 사용 가능한 설계 | 공통 틀을 잡아주는 뼈대 설계 |

:::

### 일반화(Generalization) — 상속으로 공통점 묶기

**일반화(Generalization)** 는 **superclass(부모 클래스)와 subclass(자식 클래스)의 상속 관계**를 표현하는 UML 개념입니다. 여러 클래스에 공통으로 들어가는 속성과 메서드를 부모 클래스에 한 번만 정의하고, 자식 클래스들은 이를 **상속(extends)** 받아 재사용합니다.

예시로 쇼핑몰 시스템을 생각해봅시다. 쇼핑몰에는 일반 고객(`Customer`)과 관리자(`Admin`)가 있습니다. 둘 다 **아이디와 비밀번호로 로그인**한다는 공통점이 있습니다. 이 공통 기능을 `User`라는 부모 클래스에 정의하면 됩니다.

```java
// ── 부모 클래스(Superclass): User ──────────────────
// 고객과 관리자가 공통으로 가지는 속성과 메서드를 여기에 정의
public class User {

    private String id;        // 공통 속성: 아이디
    private String password;  // 공통 속성: 비밀번호

    // 공통 메서드: 로그인 기능 (모든 사용자에게 필요)
    public void login() {
        // 로그인 공통 로직 구현
    }
}
```

```java
// ── 자식 클래스(Subclass): Customer ────────────────
// 'extends User' = User 클래스를 상속받겠다는 선언
// Customer는 User의 id, password, login()을 모두 물려받습니다
public class Customer extends User {

    // Customer만의 고유 메서드: 고객 전용 로그인
    public void login() {
        // 고객 전용 로그인 로직 (예: 포인트 확인 등 추가 기능)
    }
}
```

```java
// ── 자식 클래스(Subclass): Admin ───────────────────
// 'extends User' = User 클래스를 상속받겠다는 선언
// Admin도 User의 모든 기능을 물려받습니다
public class Admin extends User {

    // Admin만의 고유 메서드: 관리자 전용 로그인
    public void login() {
        // 관리자 전용 로그인 로직 (예: 이중 인증, 접근 권한 확인 등)
    }
}
```

> **상속의 가장 큰 장점**: `id`, `password`, `login()` 같은 공통 코드를 한 번만 작성하면 됩니다. Customer와 Admin에 각각 똑같은 코드를 두 번 쓸 필요가 없어서 **코드 중복이 줄어들고 유지보수가 쉬워집니다.** 나중에 로그인 로직을 수정해야 할 때도 `User` 클래스 한 곳만 수정하면 됩니다.

### 실체화(Realization) — 인터페이스를 실제로 구현하기

**실체화(Realization)** 는 **인터페이스(Interface, 껍데기만 있는 설계)를 실제 클래스로 구현하는 관계**를 말합니다. 인터페이스는 메서드의 이름과 형태만 정의하고, 실제 동작 내용은 비어 있습니다. 이를 클래스가 `implements` 키워드로 받아서 실제 기능을 채워 넣는 것이 오버라이딩(Overriding, 덮어쓰기)입니다.

마치 **건축 법규(인터페이스)** 처럼 생각하면 됩니다. 법규에는 "건물에는 비상구가 있어야 한다"고 명시되어 있지만, 비상구의 구체적인 위치나 모양은 각 건물(클래스)이 직접 설계합니다.

```java
// ── 인터페이스(Interface): CheckLog ────────────────
// 'interface' 키워드로 선언 (class가 아님!)
// 메서드 이름과 반환 타입만 정의하고 내용은 없습니다 (껍데기)
public interface CheckLog {

    // boolean: true(참) 또는 false(거짓)를 반환하는 메서드
    boolean checkLog(); // 로그 확인이 성공하면 true, 실패하면 false 반환
}
```

```java
// ── 실제 구현 클래스: DateCheckLog ─────────────────
// 'implements CheckLog' = CheckLog 인터페이스를 구현하겠다는 선언
// 인터페이스와 달리 실제 동작 내용을 채워 넣습니다
public class DateCheckLog implements CheckLog {

    // @Override: 인터페이스에 정의된 메서드를 실제로 구현(덮어쓰기)
    @Override
    public boolean checkLog() {
        // 날짜 기반으로 로그를 실제로 확인하는 로직 구현
        // 성공하면 true, 실패하면 false를 돌려줌
        return true;
    }
}
```

:::tip 상속(extends) vs 구현(implements) 차이
- **extends(상속)**: 부모 클래스의 기능을 **물려받아** 사용. 부모가 이미 만들어 놓은 기능을 그대로 쓰거나 수정할 수 있음.
- **implements(구현)**: 인터페이스라는 **설계 계약서에 서명**하는 것. "이 인터페이스에 정의된 모든 메서드를 반드시 구현하겠다"는 약속.
:::

## 🔗 의존, 연관, 집합 — 클래스 사이의 관계

### 의존(Dependency) — 다른 클래스를 참조하기

**의존(Dependency)** 은 **한 클래스가 다른 클래스를 사용하는 관계**입니다. 예를 들어, `User` 클래스가 `Schedule` 클래스를 사용해 스케줄을 만드는 경우입니다. `Schedule`을 사용하려면 먼저 `new` 키워드로 **객체(실제 사용 가능한 인스턴스)를 생성**해야 합니다.

```java
// User 클래스가 Schedule 클래스에 의존하는 예시
public class User {

    // createSchedule: 스케줄을 생성하는 메서드
    public void createSchedule() {

        // 'new Schedule()' = Schedule 클래스로부터 실제 객체를 생성합니다
        // 클래스 정의만으로는 바로 쓸 수 없고, 반드시 'new'로 생성해야 합니다
        // 마치 붕어빵 틀(클래스)로 실제 붕어빵(객체)을 만드는 것과 같습니다
        Schedule schedule = new Schedule(); // 객체 생성

        // 생성된 객체(schedule)를 이용해 실제 기능 사용
        schedule.save(); // 스케줄 저장
    }
}
```

> **핵심**: 클래스는 **설계도(틀)**이고, `new`를 통해 만들어진 것이 **객체(실제 사용 가능한 것)**입니다. 설계도만 있다고 집에 살 수 없듯이, 클래스 정의만으로는 프로그램에서 실제로 사용할 수 없습니다.

### 연관(Association) — 필드로 다른 클래스 포함하기

**연관(Association)** 은 **한 클래스가 다른 클래스를 멤버 변수(필드)로 포함하는 관계**입니다. 의존과 달리 연관은 더 지속적인 관계입니다. 예를 들어, `User`가 여러 개의 주소(`Address`)를 가질 수 있는 경우를 표현합니다.

```java
import java.util.List; // 여러 개의 값을 담는 리스트 자료구조

// User 클래스가 Address를 '연관' 관계로 포함하는 예시
public class User {

    private String name; // 사용자 이름

    // List<Address>: 여러 개의 Address 객체를 담는 리스트
    // 한 사람이 집 주소, 회사 주소 등 여러 주소를 가질 수 있음
    private List<Address> addresses; // User가 Address 목록을 필드로 가짐

    // 주소 목록을 반환하는 메서드
    public List<Address> getAddresses() {
        return addresses;
    }
}
```

### 집합(Aggregation) — 전체와 부분의 관계

**집합(Aggregation)** 도 전체와 부분의 관계를 나타냅니다. `User`와 `Address`를 집합 형태로 표현하면, Address 클래스로 User 정보를 묶어서 관리하는 구조가 됩니다. 부품들을 모아 하나의 제품을 만드는 것처럼, 여러 객체를 집합 형태로 구성해 하나의 완성된 객체를 만드는 방식입니다.

## 📊 시퀀스 다이어그램과 상태 다이어그램 → 코드 구현

### 시퀀스 다이어그램(Sequence Diagram) — 동작 순서를 코드로

**시퀀스 다이어그램(Sequence Diagram)** 은 **객체들이 서로 메시지를 주고받는 순서**를 시간 흐름에 따라 표현한 UML 다이어그램입니다. 이를 코드로 구현하면 **순차적으로 실행되는 메서드 호출** 형태가 됩니다.

예를 들어, 사용자가 문서를 체크아웃하는 과정을 시퀀스 다이어그램으로 표현했다면, 코드는 다음과 같이 구현됩니다.

```java
// 시퀀스 다이어그램의 '문서 체크아웃' 흐름을 코드로 구현
public void checkout(String documentId) {

    // 1단계: 데이터베이스에서 문서(Document) 객체를 가져옴
    Document d = database.findDocument(documentId); // DB에서 문서 검색

    // 2단계: 문서가 존재하는지 확인 (null = 찾지 못한 경우)
    if (d != null) {
        // 3단계: 문서가 있으면 정상 처리 (론, 저장, 이용)
        d.load();    // 문서 불러오기
        d.save();    // 문서 저장하기
        d.use();     // 문서 이용하기

    } else {
        // 4단계: 문서가 없으면 "찾을 수 없음" 메시지 출력
        System.out.println("Document Not Found"); // 문서를 발견하지 못함
    }
}
```

이 코드는 시퀀스 다이어그램에서 표현한 "사용자 → 컨트롤러 → DB → 문서" 의 흐름을 그대로 순서대로 코드화한 것입니다. **다이어그램의 화살표 순서가 코드의 실행 순서**가 된다는 점이 핵심입니다.

### 상태 다이어그램(State Diagram) — 조건과 상태 변화를 코드로

**상태 다이어그램(State Diagram)** 은 **어떤 이벤트(사건)가 발생했을 때, 어떤 조건에서 어떤 상태로 변하는지**를 표현합니다. 이를 코드로 구현하면 주로 **switch-case문(여러 경우를 분기하는 구조)** 이나 조건문이 됩니다.

```java
// 상태 다이어그램을 Java 코드로 구현한 예시
public class OrderSystem {

    private int state; // 현재 상태를 저장하는 속성 (상태 다이어그램의 '상태 저장')

    // 이벤트를 처리하는 메서드 (상태 다이어그램의 '이벤트')
    public void handleEvent(int state) {

        // switch-case: 상태값에 따라 다른 동작 수행 (상태 다이어그램의 '조건 + 액션')
        switch (state) {

            case 1: // 상태가 1번인 경우 (예: 주문 접수)
                processOrderReceived(); // 주문 접수 처리 (액션 수행)
                break; // 이 케이스 처리 후 switch문 종료

            case 2: // 상태가 2번인 경우 (예: 배송 중)
                processShipping(); // 배송 처리 (액션 수행)
                break;

            default: // 1번도 2번도 아닌 경우
                // 처리할 케이스가 없으면 그냥 종료
                break;
        }
    }
}
```

:::note 상태 다이어그램 → 코드 매핑 정리

| 상태 다이어그램 요소 | Java 코드 요소 | 설명 |
|---|---|---|
| 상태(State) | 클래스의 속성(변수) | 현재 상태 값을 저장 |
| 이벤트(Event) | 메서드 | 상태를 변화시키는 기능 |
| 조건(Condition) | if문, switch-case | 어떤 상태일 때 어떤 처리를 할지 |
| 액션(Action) | 메서드 내부 명령어 | 실제로 수행되는 동작 |

:::

## 🔨 리팩토링(Refactoring) — 코드를 더 좋게 다듬기

### 리팩토링이란 무엇인가?

**리팩토링(Refactoring)** 은 **프로그램의 동작 결과는 그대로 유지하면서, 내부 코드 구조를 더 좋게 개선하는 작업**입니다. `Re(다시) + Factory(공장, 만들어내는 것)` = **이미 만든 것을 다시 더 좋게 만드는 것**이라고 이해하면 됩니다.

자동차로 비유하자면, 외관(기능, 결과)은 똑같이 시속 100km로 달리지만, 엔진 내부 구조를 더 효율적으로 개조해서 연비가 좋아지고 유지보수가 쉬워지는 것과 같습니다. **사용자 입장에서는 변한 게 없지만, 개발자 입장에서는 훨씬 다루기 좋은 코드가 됩니다.**

> 리팩토링 = 결과 변경 없이 더 좋은 코드로 구조를 개선하는 과정
> 다른 말로 **시스템 고도화**, **코드 개선** 이라고도 합니다.

### 리팩토링이 왜 필요한가?

처음 프로그래밍을 배울 때는 일단 "돌아가는 코드"를 만드는 데 집중하게 됩니다. 하지만 서비스가 커지고 팀원이 늘어날수록, **읽기 어렵고 수정하기 힘든 코드는 큰 문제**가 됩니다. 마치 대충 정리한 방은 처음엔 빨리 끝낼 수 있지만, 나중에 물건을 찾을 때 몇 배의 시간이 걸리는 것처럼요.

리팩토링을 통해 얻을 수 있는 이점은 다음과 같습니다.

- **가독성 향상**: 다른 개발자(혹은 미래의 나)가 코드를 쉽게 이해할 수 있게 됩니다.
- **유지보수 용이**: 버그를 찾고 수정하는 시간이 줄어듭니다.
- **버그 발견**: 코드를 정리하다 보면 숨어있던 버그를 발견하기도 합니다.
- **개발 속도 향상**: 장기적으로는 더 빠르게 새 기능을 추가할 수 있습니다.

### 리팩토링 과정 — 조금씩, 단계적으로

리팩토링은 한꺼번에 모든 것을 바꾸는 것이 아닙니다. **작은 단위로 하나씩 변경하고, 변경할 때마다 테스트해서 문제가 없는지 확인**하는 방식으로 진행합니다.

```
1. 작은 단위의 코드 하나를 리팩토링
        ↓
2. 테스트 실행 → 정상 작동 확인
        ↓
3. 정상 작동 ✅ → 다음 리팩토링으로 진행
   오류 발생 ❌ → 변경 취소 → 원래 코드로 되돌리기
        ↓
4. 반복
```

:::warning 리팩토링 시 주의사항
리팩토링은 반드시 **작은 단위로 나눠서** 진행해야 합니다. 한 번에 너무 많은 코드를 바꾸면, 오류가 발생했을 때 어디서 문제가 생겼는지 찾기가 매우 어려워집니다. "조금 바꾸고 → 테스트하고 → 조금 더 바꾸고 → 테스트하고"의 반복이 리팩토링의 핵심입니다.
:::

## 🦨 코드 스멜(Code Smell) — 나쁜 코드의 신호들

### 코드 스멜이란?

**코드 스멜(Code Smell)** 은 **코드에 문제가 있다는 신호(징후)**를 말합니다. "코드에서 냄새가 난다"는 표현처럼, 직접적인 오류(버그)는 아니지만 **리팩토링이 필요하다는 것을 알려주는 나쁜 코드의 특징들**입니다. 코드 스멜이 많을수록 프로그램은 읽기 어렵고, 수정하기 힘들고, 버그가 생기기 쉬워집니다.

### 주요 코드 스멜 종류와 해결 방법

#### 1️⃣ 코드 중복 (Duplicated Code)

같은 코드가 여러 곳에 반복해서 나타나는 경우입니다. 마치 같은 문장을 여러 페이지에 복사·붙여넣기 한 것처럼, 나중에 내용을 바꿔야 할 때 모든 곳을 찾아서 수정해야 하는 불편함이 생깁니다.

```java
// ❌ 나쁜 예시: 같은 출력 코드가 여러 곳에 중복
public class Report {
    public void printReport1() {
        System.out.println("이름: " + name); // 중복 코드
        System.out.println("나이: " + age);  // 중복 코드
        // ... 보고서 1 내용
    }

    public void printReport2() {
        System.out.println("이름: " + name); // 똑같은 코드 반복!
        System.out.println("나이: " + age);  // 똑같은 코드 반복!
        // ... 보고서 2 내용
    }
}
```

```java
// ✅ 좋은 예시: 중복 코드를 별도 메서드로 추출하여 재사용
public class Report {

    // 중복되던 코드를 하나의 메서드로 추출
    private void printDetail() {
        System.out.println("이름: " + name); // 한 곳에서만 정의
        System.out.println("나이: " + age);
    }

    public void printReport1() {
        printDetail(); // 메서드 호출로 대체 (한 줄로 끝!)
        // ... 보고서 1 내용
    }

    public void printReport2() {
        printDetail(); // 메서드 호출로 대체 (한 줄로 끝!)
        // ... 보고서 2 내용
    }
}
```

#### 2️⃣ 조건문 단순화 (Simplify Conditional)

복잡하게 얽혀있는 조건문을 간결하게 만드는 것입니다.

```java
// ❌ 나쁜 예시: 불필요하게 복잡한 조건문
int result;
boolean userRight = checkUserRight(); // 사용자 권한 확인
if (userRight == true) {
    result = 2; // 권한 있으면 2
} else {
    result = 1; // 권한 없으면 1
}

boolean compare = (score > 5); // 점수 비교
if (compare == true) {
    return result; // 5점 초과면 결과 반환
}
```

```java
// ✅ 좋은 예시: 두 조건을 하나의 간결한 표현으로 통합
// score가 5보다 크면 (권한 있으면 2, 없으면 1) 반환
// 삼항 연산자: 조건 ? 참일때값 : 거짓일때값
int result = (score > 5) ? (checkUserRight() ? 2 : 1) : 0;
// 코드가 훨씬 간결해짐!
```

#### 3️⃣ 배열을 객체로 교체 (Replace Array with Object)

배열의 각 인덱스(위치)에 특정 의미를 담아서 사용하면 코드를 읽기 어렵습니다. 이를 의미 있는 이름을 가진 객체로 바꾸면 훨씬 명확해집니다.

```java
// ❌ 나쁜 예시: 배열에 직접 값을 넣음 (인덱스 0이 뭔지, 1이 뭔지 알기 어려움)
String[] teamData = new String[2];
teamData[0] = "Liverpool"; // 0번이 팀 이름이라는 걸 코드만 보면 모름
teamData[1] = "15";        // 1번이 승리 횟수라는 걸 코드만 보면 모름
```

```java
// ✅ 좋은 예시: 의미 있는 이름을 가진 객체로 표현
public class Team {
    private String teamName;  // 팀 이름 (명확!)
    private int winCount;     // 승리 횟수 (명확!)

    // 생성자: 객체를 만들 때 초기값을 설정
    public Team(String teamName, int winCount) {
        this.teamName = teamName;
        this.winCount = winCount;
    }
}

// 사용: 코드만 봐도 무슨 데이터인지 바로 알 수 있음
Team liverpool = new Team("Liverpool", 15);
```

#### 4️⃣ public 필드를 private으로 변경

클래스의 속성(필드)을 외부에서 직접 수정할 수 있게 열어두면, 예상치 못한 곳에서 값이 바뀌어 버그가 발생할 수 있습니다.

```java
// ❌ 나쁜 예시: public 필드는 외부에서 직접 수정 가능 → 위험!
public class User {
    public String name; // 누구나 직접 바꿀 수 있음 → user.name = "해킹값"; 가능
}
```

```java
// ✅ 좋은 예시: private 필드 + 접근자 메서드(getter/setter) 사용
public class User {
    private String name; // 외부에서 직접 접근 불가

    // getter: 이름을 읽을 때만 사용하는 공개 메서드
    public String getName() {
        return name;
    }

    // setter: 이름을 설정할 때 유효성 검사를 거쳐서 안전하게 변경
    public void setName(String name) {
        if (name != null && !name.isEmpty()) { // 빈 이름은 허용하지 않음
            this.name = name;
        }
    }
}
```

#### 5️⃣ 조건문 공통 코드 밖으로 이동

if-else의 모든 분기에서 똑같은 코드가 실행된다면, 그 코드를 조건문 밖으로 꺼내는 것이 좋습니다.

```java
// ❌ 나쁜 예시: send() 호출이 if와 else 양쪽에 중복
if (isSpecialDeal) {
    price = price * 0.95; // 할인 적용
    send(order);          // 중복! ← 
} else {
    send(order);          // 중복! ←
}
```

```java
// ✅ 좋은 예시: 공통 코드(send)를 조건문 밖으로 이동
if (isSpecialDeal) {
    price = price * 0.95; // 할인 적용 (if 블록에만 있어야 하는 코드)
}
send(order); // 조건과 무관하게 항상 실행 → 한 번만 작성!
```

#### 6️⃣ 에러 코드 대신 예외 처리 사용

에러 상황을 `-1`, `0` 같은 숫자로 표현하면, 나중에 코드를 읽는 사람이 "이게 에러인지, 정상 값인지" 헷갈릴 수 있습니다. **예외(Exception) 처리**를 사용하면 에러 상황을 명확하게 표현할 수 있습니다.

```java
// ❌ 나쁜 예시: 에러를 숫자 코드로 표현 (-1이 에러라는 걸 알기 어려움)
public int withdraw(int amount) {
    if (amount > balance) {
        return -1; // -1이 "잔액 부족"을 의미하는 에러 코드 → 불명확!
    }
    balance -= amount;
    return 0; // 0이 성공을 의미 → 헷갈림
}
```

```java
// ✅ 좋은 예시: 에러 상황을 예외(Exception)로 명확하게 표현
public void withdraw(int amount) throws Exception {
    if (amount > balance) {
        // throw: 예외를 발생시켜 호출자에게 에러를 전달
        // "잔액이 부족합니다"라는 명확한 메시지와 함께
        throw new Exception("잔액이 부족합니다. 현재 잔액: " + balance);
    }
    balance -= amount; // 정상 처리
}
```

## 🔍 코드 인스펙션과 페어 프로그래밍 — 품질을 높이는 협업 방법

### 코드 인스펙션(Code Inspection) — 코드 검사

**코드 인스펙션(Code Inspection)** 은 **프로그램이 성공적으로 컴파일(실행 가능한 형태로 변환)된 후, 분석 도구나 동료 개발자가 코드를 꼼꼼히 검사하는 과정**입니다. 마치 건물 준공 검사처럼, 완성된 코드에 숨어있는 문제를 찾아내는 작업입니다.

코드 인스펙션을 통해 확인하는 항목들은 다음과 같습니다.

| 검사 항목 | 설명 |
|---|---|
| 효율성 | 코드가 불필요하게 느리거나 자원을 낭비하지는 않는가? |
| 코딩 표준 준수 | 팀에서 정한 코드 작성 규칙을 따르고 있는가? |
| 결합 여부 | 모듈들이 올바르게 연결되어 있는가? |
| 로직 오류 | 프로그램의 논리가 의도한 대로 작동하는가? |
| 계산 오류 | 수식이나 연산에 실수는 없는가? |
| 인터페이스 오류 | 다른 모듈과의 연결 부분에 문제는 없는가? |
| 데이터 처리 오류 | 데이터를 저장하고 불러오는 과정에 오류는 없는가? |

**정적 분석(Static Analysis)** 은 코드 인스펙션의 한 방법으로, **프로그램을 직접 실행하지 않고 코드의 텍스트(명령어들)를 분석해서 결함을 찾는 방법**입니다. 런타임 오류(실행 중 발생하는 오류)가 아닌, 코드 자체에 존재하는 잠재적 문제를 미리 찾아냅니다.

### 페어 프로그래밍(Pair Programming) — 함께 코딩하기

**페어 프로그래밍(Pair Programming)** 은 **두 명의 개발자가 함께 하나의 작업을 수행하는 협업 방식**입니다. 한 사람이 코드를 작성하면, 다른 사람이 실시간으로 검토하고 피드백을 주는 방식입니다. "쌍(pair) 프로그래밍"이라고도 합니다.

혼자 개발할 때의 문제점을 생각해봅시다. 내가 만들고, 내가 테스트하고, 내가 오류를 찾고, 내가 품질을 평가합니다. 이렇게 하면 자신의 실수를 스스로 잡아내기가 매우 어렵습니다. 마치 내가 쓴 글을 내가 교정하면 오탈자를 잘 못 찾는 것과 같습니다.

페어 프로그래밍의 효과적인 운영 방법으로 **핑퐁(Ping-Pong) 방식**이 있습니다.

```
개발자 A → 기능 개발
개발자 B → A가 개발한 것 테스트 & 검토
        ↓
개발자 B → 기능 개발
개발자 A → B가 개발한 것 테스트 & 검토
        ↓
(탁구처럼 서로 주고받으며 반복)
```

이렇게 하면 서로의 코드를 크로스로 검토하기 때문에, 한 사람이 놓친 버그를 다른 사람이 잡아낼 확률이 높아집니다.

:::warning 페어 프로그래밍 주의사항

페어 프로그래밍을 효과적으로 운영하려면 몇 가지를 주의해야 합니다.

1. **토론에만 너무 많은 시간을 쓰지 않기**: 코딩보다 대화가 많아지면 효율이 떨어집니다. 논의는 핵심 사항에 집중하고, 실제 구현 시간을 충분히 확보해야 합니다.
2. **같은 장소에 있는 것이 가장 효율적**: 실시간으로 소통하고 화면을 같이 보면서 작업해야 빠른 피드백이 가능합니다. 서로 다른 장소에 있으면 스케줄 조율 등으로 비효율이 생길 수 있습니다.
3. **역할을 명확히 나누기**: 누가 개발하고 누가 검토할지 역할을 명확히 하고, 주기적으로 역할을 교체합니다.

:::

## ⚠️ 더 많은 코드 스멜 종류와 해결 전략

앞에서 몇 가지 코드 스멜을 살펴봤지만, 실제 개발 현장에서 마주치는 코드 스멜은 더 다양합니다. 아래 표로 주요 코드 스멜과 해결 전략을 정리했습니다.

| 코드 스멜 | 증상 | 해결 방법 |
|---|---|---|
| 코드 중복 | 같은 코드가 여러 곳에 반복 | 메서드로 추출하여 한 곳에서 관리 |
| 긴 메서드 | 메서드 길이가 너무 길어 읽기 어려움 | 메서드를 적절한 크기로 분리 |
| 거대한 클래스 | 속성과 메서드가 너무 많음 | 클래스를 작게 분리하고 상속 구조 활용 |
| 긴 파라미터 목록 | 메서드에 매개변수가 너무 많음 | 관련 매개변수를 객체로 묶기 |
| 산탄총 수술 | 하나를 바꾸면 여러 클래스도 바꿔야 함 | 관련 코드를 한 곳으로 모으기 |
| 기능 편애 | 자신의 클래스보다 다른 클래스 데이터를 더 많이 사용 | 데이터와 관련 기능을 같은 클래스로 이동 |
| 데이터 뭉치 | 항상 함께 다니는 데이터 그룹 | 별도 클래스나 객체로 묶기 |
| 기본 타입 강박 | 객체 대신 기본 타입(int, String 등)만 사용 | 관련 데이터를 객체로 묶어 사용 |
| 스위치문 남용 | switch-case가 너무 많음 | 상속과 다형성(오버라이딩) 활용 |
| 병렬 상속 계층 | 비슷한 클래스 계층이 중복으로 존재 | 상속 구조를 재설계하여 통합 |

특히 **스위치문 남용**은 새로운 케이스가 추가될 때마다 switch 문을 찾아 수정해야 하는 불편함이 생깁니다. 이 경우 앞서 배운 상속과 오버라이딩을 활용하면, 새 클래스만 추가하면 되는 훨씬 유연한 구조로 개선할 수 있습니다.

:::danger 리팩토링하면 절대 하면 안 되는 것
리팩토링은 **기능(결과)을 변경하지 않는 것**이 원칙입니다. 리팩토링을 하면서 동시에 새 기능을 추가하거나 기존 기능을 수정하면, 나중에 문제가 생겼을 때 리팩토링 때문인지 기능 변경 때문인지 알 수 없게 됩니다. **리팩토링 = 코드 구조 개선만**, 기능 변경은 별도 작업으로 분리해야 합니다.
:::

## 💡 좋은 개발자가 되기 위한 마인드셋

리팩토링, 코드 스멜 제거, 코드 인스펙션, 페어 프로그래밍... 이 모든 것은 결국 하나의 목표를 향합니다. **"더 좋은 코드를 작성하는 개발자가 되는 것"** 입니다.

단순히 Java나 Python 같은 언어를 "문법적으로 틀리지 않게 작성하는 것"만으로는 좋은 개발자라고 할 수 없습니다. 다음과 같은 역량을 함께 키워야 합니다.

- **코드 가독성**: 6개월 뒤 내가 다시 봐도, 처음 보는 동료가 봐도 쉽게 이해할 수 있는 코드
- **코드 유지보수성**: 요구사항이 바뀌었을 때 최소한의 수정으로 대응할 수 있는 코드 구조
- **코드 재사용성**: 이미 만든 기능을 여러 곳에서 중복 없이 재활용할 수 있는 설계
- **협업 능력**: 팀원과 원활하게 소통하고 코드를 공유할 수 있는 커뮤니케이션 능력

이런 역량을 갖춘 개발자가 나중에 프로젝트 매니저(PM)로 성장하고, 팀을 이끄는 시니어 개발자가 됩니다. UML을 이해하고, 리팩토링을 실천하고, 코드 스멜을 줄이려는 노력이 쌓여서 여러분의 개발자로서의 가치가 높아집니다.

:::tip 입문자를 위한 학습 로드맵
1. **지금 단계**: UML의 각 요소가 어떤 코드 개념에 해당하는지 큰 그림 이해
2. **Java/Python 기초 학습 후**: 이 문서의 코드 예시를 다시 보며 구체적으로 이해
3. **프로젝트 경험 후**: 실제 코드에서 코드 스멜을 찾고 리팩토링 실습
4. **팀 프로젝트 시**: 페어 프로그래밍과 코드 인스펙션 직접 적용
:::

## 📝 핵심 정리

- **UML(통합 모델링 언어)** 은 소프트웨어 설계를 시각화하는 표준 설계 언어이며, 이 설계도가 실제 프로그래밍 코드로 구현된다.
- **클래스(Class)** 는 속성(데이터 틀)과 메서드(기능)를 하나로 묶은 설계 단위이며, Java에서 `class 이름 { }` 형태로 표현된다.
- **추상 클래스(Abstract Class)** 는 메서드의 내용 없이 이름(명세)만 정의된 클래스로, 자식 클래스에서 실제 내용을 구현한다.
- **일반화(Generalization)** 는 공통 기능을 부모 클래스(superclass)에 정의하고, 자식 클래스(subclass)가 `extends`로 상속받아 재사용하는 관계이다.
- **실체화(Realization)** 는 껍데기만 있는 인터페이스(interface)를 클래스가 `implements`로 받아 오버라이딩(overriding)으로 실제 기능을 구현하는 관계이다.
- **의존(Dependency)** 은 한 클래스가 다른 클래스를 사용하는 관계이며, `new` 키워드로 객체를 생성해야 실제 사용이 가능하다.
- **시퀀스 다이어그램** 은 객체 간 동작 순서를 표현하며, 코드에서 순차적 메서드 호출과 조건 분기(if-else)로 구현된다.
- **상태 다이어그램** 은 상태→이벤트→조건→액션의 흐름을 표현하며, 코드에서 switch-case문으로 주로 구현된다.
- **리팩토링(Refactoring)** 은 프로그램의 동작 결과는 유지하면서 내부 코드 구조를 더 좋게 개선하는 작업이다.
- **코드 스멜(Code Smell)** 은 리팩토링이 필요하다는 신호로, 코드 중복·긴 메서드·거대한 클래스·복잡한 조건문 등이 대표적이다.
- **코드 인스펙션(Code Inspection)** 은 완성된 코드를 분석 도구나 동료 개발자가 검사하여 숨겨진 결함을 찾아내는 과정이다.
- **페어 프로그래밍(Pair Programming)** 은 두 명의 개발자가 함께 개발·검토를 교대로 수행하여 코드 품질을 높이는 협업 방식이다.
- 좋은 개발자는 언어 문법뿐 아니라 리팩토링, 코드 품질 관리, 코드 스멜 최소화 등을 항상 염두에 두어야 한다.
