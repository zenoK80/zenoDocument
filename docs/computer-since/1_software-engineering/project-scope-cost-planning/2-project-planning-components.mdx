# 🗂️ 프로젝트 계획의 구성 요소: 누가·언제·무엇을·어떻게

소프트웨어 공학을 처음 접하는 분들은 "개발 = 코딩"이라고 생각하기 쉽습니다. 하지만 실제 현장에서는 코딩 이전에 **얼마나 치밀하게 계획을 세웠느냐**가 프로젝트의 성패를 좌우합니다. 마치 집을 지을 때 설계도 없이 벽돌을 쌓기 시작하면 안 되는 것처럼, 소프트웨어도 명확한 계획 없이 개발을 시작하면 반드시 문제가 생깁니다. 이번 문서에서는 프로젝트 계획이 무엇인지, 왜 필요한지, 어떤 구성 요소로 이루어지는지, 그리고 비용을 어떻게 추정하는지까지 하나씩 꼼꼼히 살펴보겠습니다.

---

## 🧭 프로젝트 계획이란 무엇인가?

### 계획이라는 개념부터 다시 생각해보기

우리는 일상 속에서 이미 수많은 계획을 세웁니다. 여행을 갈 때 숙소, 교통편, 일정을 미리 정하는 것도 계획이고, 시험 준비를 위해 과목별 공부 시간을 배분하는 것도 계획입니다. 계획을 세우는 이유는 단순합니다. **목적을 달성하기 위해서**입니다. 공부 계획은 좋은 학점을 받기 위해, 여행 계획은 즐거운 여행을 위해 세우는 것이죠.

소프트웨어 공학에서의 **프로젝트 계획**도 마찬가지입니다. 클라이언트(의뢰인)가 원하는 소프트웨어를 성공적으로 만들어내기 위해, 개발이 시작되기 **이전에** 사전에 모든 것을 결정하고 준비하는 과정을 말합니다.

> 💡 프로젝트 계획이란, **누가·언제·무엇을·어떻게** 할 것인지를 프로젝트가 시작되기 전에 미리 결정하는 작업입니다.

### 프로젝트 계획의 4가지 핵심 질문

프로젝트 계획을 구성하는 4가지 핵심 질문을 살펴보겠습니다. 이 네 가지는 어떤 규모의 프로젝트이든 반드시 사전에 답해야 하는 질문들입니다.

| 질문 | 의미 | 예시 |
|------|------|------|
| **누가 (Who)** | 프로젝트에 관련된 모든 이해관계자 | 클라이언트, PM, 개발자, 디자이너, 기획자, 사용자 |
| **언제 (When)** | 프로젝트의 시작과 종료 기간 | 다음 달부터 6개월간 |
| **무엇을 (What)** | 개발할 제품 또는 서비스의 내용 | 쇼핑몰 웹사이트, 모바일 앱, IoT 플랫폼 |
| **어떻게 (How)** | 개발 방법론, 팀 구성, 관리 방법 | 애자일 방법론, 팀 5명, 스크럼 방식 |

#### 👥 누가 (Who): 이해관계자 전체를 포함한다

"누가"라고 하면 단순히 개발자만 떠올리기 쉽지만, 실제로는 훨씬 더 넓은 범위의 사람들이 포함됩니다. 돈을 투자하는 **클라이언트(의뢰인)**, 프로젝트 전체를 총괄하는 **PM(프로젝트 매니저)**, 실제 코드를 작성하는 **프론트엔드/백엔드 개발자**, 화면을 디자인하는 **UI/UX 디자이너**, HTML·CSS를 작성하는 **퍼블리셔**, 서비스 흐름을 설계하는 **서비스 기획자**, 그리고 최종적으로 서비스를 사용하는 **일반 사용자**까지 모두 포함됩니다. 이 모든 사람들의 요구와 역할을 계획 단계에서 명확히 정의해야 합니다.

#### 📅 언제 (When): 기간 설정의 중요성

"언제"는 단순히 날짜를 정하는 것이 아닙니다. **현실적으로 달성 가능한 기간**을 설정하는 것이 핵심입니다. 6개월이면 충분한 프로젝트를 1년으로 잡으면 낭비이고, 반대로 무리하게 3개월로 잡으면 품질이 떨어지거나 프로젝트가 실패할 수 있습니다. 기간 설정은 뒤에서 배울 비용 추정과 직결되는 매우 중요한 요소입니다.

#### 💡 무엇을 (What): 개발 대상의 명확한 정의

"무엇을"은 최종 결과물이 무엇인지 명확히 하는 것입니다. "웹사이트 하나 만들어주세요"는 너무 모호합니다. 쇼핑몰인지, 정보 포털인지, IoT 기반 대시보드인지, 모바일 앱인지를 구체적으로 정의해야 합니다. 요즘처럼 빅데이터, 클라우드, 모바일 등 다양한 기술이 혼재하는 시대에는 더욱 명확한 정의가 필요합니다.

#### 🔧 어떻게 (How): 방법론과 팀 구성

"어떻게"는 어떤 개발 방법론(폭포수 모형, 애자일 등)을 사용할지, 팀을 어떻게 구성할지, 누가 어떤 역할을 맡을지를 결정하는 것입니다. 방법론 선택은 프로젝트의 유연성과 관리 방식에 큰 영향을 미치므로 신중하게 결정해야 합니다.

## ⚠️ 왜 프로젝트 계획이 이토록 중요한가?

### 계획 없이 개발하면 어떤 일이 생길까?

개발 현장에서는 클라이언트와 개발자 사이의 갈등이 자주 발생합니다. 그 원인의 대부분은 **불명확한 계획**에서 비롯됩니다.

클라이언트는 개발을 문서 수정처럼 쉽게 생각하는 경향이 있습니다. "버그 하나 고치는 게 그렇게 오래 걸려요?"라고 묻지만, 실제로 그 버그가 데이터베이스(데이터 저장소) 구조 문제이거나 핵심 알고리즘(문제를 해결하는 절차) 오류라면 며칠에서 한 달 이상 걸릴 수도 있습니다. 이런 상황이 반복되면 신뢰가 무너지고 프로젝트는 실패의 길로 접어듭니다.

:::warning 비현실적인 계획의 두 가지 함정
**함정 1 - 너무 길게 잡는 경우:** 3개월에 충분한 일을 6개월로 계획하면 "왜 이렇게 느리게 일하냐"는 불신을 받게 됩니다.

**함정 2 - 너무 짧게 잡는 경우:** 프로젝트를 수주하려는 욕심에 6개월 짜리 일을 4개월로 약속하면, 실제 개발 중에 기간이 부족해 프로젝트가 60%밖에 완성되지 못한 채 납기일을 맞게 됩니다.
:::

### 합리적 측정이 핵심이다

결국 프로젝트 계획의 핵심은 **자원(인력), 비용, 일정**이라는 세 가지 요소를 얼마나 합리적이고 객관적으로 측정하느냐에 있습니다.

- **자원(Resource)**: 몇 명의 인력이 필요한가?
- **비용(Cost)**: 얼마의 예산이 필요한가?
- **일정(Schedule)**: 얼마나 시간이 걸리는가?

예를 들어, 5억 원의 예산으로 6명이 6개월 동안 진행하는 프로젝트가 있다고 합시다. 과연 이 5억이 합리적인 금액인가? 6명이 6개월 동안 할 수 있는 업무량과 난이도에 맞는가? 이것을 **객관적인 데이터와 도구를 이용해 측정하고 검증**하는 것이 프로젝트 계획의 핵심 목표입니다.

> 🎯 프로젝트 계획 = 합리적인 측정을 위해 필요한 정보를 수집하고, 그 정보를 바탕으로 자원·비용·일정을 객관적으로 산출하는 과정

## 🔍 프로젝트 관리 대상: 3P와 4P

### 무엇을 관리해야 하는가?

프로젝트를 성공적으로 이끌기 위해서는 "무엇을 관리해야 하는가?"를 먼저 정의해야 합니다. 이를 **3P**와 **4P**로 나눌 수 있습니다.

### 3P: 사람(People), 프로세스(Process), 문제(Problem)

#### 👨‍👩‍👧‍👦 사람 (People)

프로젝트에 참여하는 모든 사람을 관리하는 것입니다. 백엔드 개발자, 프론트엔드 개발자, 퍼블리셔, 웹 디자이너, 서비스 기획자, 클라이언트 등 다양한 사람들이 협력합니다. 각자의 역할과 책임을 명확히 하고, 원활한 커뮤니케이션이 이루어지도록 관리해야 합니다.

#### ⚙️ 프로세스 (Process)

사람들이 어떤 절차와 방법론에 따라 일하는지를 관리하는 것입니다. 폭포수(Waterfall) 모형처럼 순서대로 진행할지, 애자일(Agile)처럼 반복적으로 개발할지를 결정하고 그 프로세스가 제대로 작동하고 있는지 지속적으로 점검합니다.

#### 🚨 문제 (Problem)

프로젝트를 진행하다 보면 예상치 못한 문제들이 반드시 발생합니다. 실제 현장에서 벌어지는 문제들을 예로 들면 다음과 같습니다.

- 팀원 5명 중 3명이 갑작스럽게 퇴사를 선언하는 경우
- 데이터베이스 설계 구조가 잘못된 것을 개발 중반에 발견하는 경우
- 안드로이드 OS(운영체제)의 새 버전이 출시되면서 기존 코드가 작동하지 않는 경우
- 고객의 요구사항이 개발 도중 갑자기 변경되는 경우

이런 문제들은 완전히 막을 수는 없지만, **사전에 리스크(위험 요소)를 파악하고 대응 방안을 마련**해두면 피해를 최소화할 수 있습니다.

### 4P: 3P + 제품(Product) & 프로젝트(Project)

4P는 3P에 **제품(Product)**과 **프로젝트(Project)** 관리를 추가한 더 넓은 개념입니다.

| 구분 | 3P | 4P |
|------|----|----|
| 사람 | ✅ | ✅ |
| 프로세스 | ✅ | ✅ |
| 문제 | ✅ | ✅ |
| 제품 | ❌ | ✅ |
| 프로젝트 전체 | ❌ | ✅ |

**제품(Product)**은 우리가 만들어내는 최종 결과물인 웹사이트, 앱, 솔루션 등을 관리하는 것입니다. **프로젝트(Project)**는 사람, 프로세스, 문제, 비용, 일정 등 모든 것을 총체적으로 묶어서 하나의 단위로 관리하는 것을 의미합니다. 결국 눈에 보이는 것, 진행되는 것, 만들어지는 것 모두를 통합적으로 관리하는 것이 4P의 핵심입니다.

## 📋 문제 정의: 개발의 첫 단추

### 왜 문제 정의부터 시작하는가?

개발자가 책상에 앉아 바로 코딩을 시작하면 안 되는 이유가 여기 있습니다. "무엇을 만들어야 하는가?"를 명확히 정의하지 않으면, 아무리 뛰어난 개발자라도 잘못된 방향으로 개발을 진행하게 됩니다. 이처럼 **개발하려는 것이 무엇인지 명확하게 정의하고, 그것을 바탕으로 개발 계획을 수립하는 과정**을 **문제 정의(Problem Definition)**라고 합니다.

:::info 자체 서비스 회사 vs 외주 개발 회사
개발자로 취업하면 크게 두 종류의 회사에 들어가게 됩니다.

- **자체 서비스 회사**: 네이버, 카카오, NC소프트처럼 자체 서비스를 직접 만들고 운영하는 회사
- **외주 개발 회사**: 클라이언트의 의뢰를 받아 개발해주고, 완성 후 다른 프로젝트를 받는 방식으로 운영하는 회사

두 경우 모두 문제 정의 단계가 필수입니다. 자체 서비스라면 내부 이해관계자들의 요구를, 외주라면 클라이언트의 요구를 명확히 정의해야 합니다.
:::

### 문제 정의는 어떻게 하는가?

문제 정의는 단순히 "쇼핑몰 만들어주세요"를 받아 적는 것이 아닙니다. 다음 세 가지 관점에서 깊이 있는 분석이 필요합니다.

**① 사용자 니즈(Needs) 분석**: 사용자가 현재 무엇이 필요하고, 어떤 불편함을 겪고 있으며, 트렌드는 어떻게 변화하고 있는지를 조사합니다.

**② 문제점 및 제약사항 파악**: 현재 운영 중인 시스템에서 발생하는 문제점과 제약사항을 파악합니다. 예를 들어 기존 쇼핑몰에서 장바구니 담기 후 결제 과정에서 오류가 자주 발생한다거나, 상품 검색이 느리다는 문제를 구체적으로 문서화합니다.

**③ 타당성 검토**: 발견된 문제를 해결하기 위한 방안이 실제로 실행 가능한지 세 가지 측면에서 검토합니다.

| 타당성 종류 | 질문 | 예시 |
|-------------|------|------|
| **경제적 타당성** | 이 개발에 투입하는 비용이 합리적인가? | 10억 원을 투자해서 사용자 니즈를 해결하는 것이 경제적으로 맞는가? |
| **기술적 타당성** | 요구하는 기능을 현재 기술로 구현할 수 있는가? | 음성 인식으로 자동 주문하는 AI 기능을 현재 기술 수준에서 구현 가능한가? |
| **법적 타당성** | 개발 내용이 법적으로 문제가 없는가? | 주민번호 전체를 회원가입에 사용하면 개인정보 보호법에 위배된다 |

:::danger 법적 타당성 간과 금지
개인정보 보호법이 강화된 지금, 회원가입 시 주민번호 전체를 수집하는 방식은 **법적으로 허용되지 않습니다**. 개발 전 반드시 법적 제약 사항을 검토해야 하며, 지적재산권·저작권·프로그램보호법 등도 꼼꼼히 확인해야 합니다. 개발을 다 완성하고 나서 법적 문제가 발견되면 처음부터 다시 해야 할 수도 있습니다.
:::

### 소프트웨어 범위 설정과 WBS

문제 정의를 통해 개발할 내용이 명확해지면, **소프트웨어 범위(Scope)**를 설정해야 합니다. 소프트웨어 범위는 기능, 성능, 제약 조건, 개발 인원, 일정을 모두 고려해서 결정합니다.

범위를 설정한 후에는 **WBS(Work Breakdown Structure, 작업 분류 체계)**를 만들어야 합니다. WBS는 마치 조직도처럼 프로젝트의 전체 작업을 계층적인 트리(나무) 구조로 분해한 것입니다.

예를 들어 학사 관리 시스템을 개발한다면 WBS는 다음과 같이 구성될 수 있습니다.

```
학사 관리 시스템
├── 수강 신청 모듈
│   ├── 강의 조회
│   ├── 수강 신청
│   └── 수강 취소
├── 성적 관리 모듈
│   ├── 성적 입력
│   ├── 성적 조회
│   └── 성적 증명서 출력
└── 강의 관리 모듈
    ├── 강의 개설
    ├── 강의 수정
    └── 강의 폐강
```

이렇게 WBS를 만들면 전체 작업량이 가시화되고, 각 작업의 담당자와 기간을 배정하기가 훨씬 쉬워집니다.

프로젝트 범위 관리는 다음 순서로 이루어집니다.

1. **요구사항 수집**: 이해관계자(클라이언트, 사용자 등)의 요구사항을 문서화
2. **범위 정의**: 현실적으로 개발 가능한 범위를 명확히 설정
3. **WBS 작성**: 범위를 계층적 구조로 분해
4. **범위 검증**: 설정된 범위가 올바른지 검증
5. **범위 통제**: 변경 사항 발생 시 통제 및 감시

## 💰 프로젝트 비용 추정: 얼마나 돈이 드는가?

### 비용 추정이 왜 어려운가?

프로젝트 비용 추정은 소프트웨어 공학에서 가장 어려운 과제 중 하나입니다. 비용을 너무 높게 잡으면 계약을 못 따고, 너무 낮게 잡으면 개발 도중 예산이 바닥나 프로젝트가 실패합니다. 게다가 소프트웨어 개발에는 다음과 같은 **알려지지 않은 요소들**이 많습니다.

- 개발 도중 예상치 못한 버그 발생 가능성
- 개발자가 중간에 퇴사하는 상황
- 새로운 OS나 프레임워크 버전 업데이트로 인한 호환성 문제
- 클라이언트의 요구사항 변경

:::tip 개발자 원가 계산의 현실
개발자 연봉이 5천만 원인 사람과 1억 원인 사람이 있다면, 실제 개발 능력이 두 배 차이가 날까요? 꼭 그렇지는 않습니다. 경험, 사용 언어 숙련도, 도메인(특정 분야) 지식에 따라 생산성은 천차만별입니다. 그래서 과거의 프로젝트 데이터와 경험이 비용 추정의 핵심 자산이 됩니다.
:::

### 비용의 구성 요소

프로젝트 예산은 단순히 개발자 월급만이 아닙니다. 크게 세 가지로 구성됩니다.

| 비용 종류 | 내용 | 예시 |
|-----------|------|------|
| **인건비** | 개발자, 기획자 등 직접 인력 비용 | 개발자 10명 × 연봉 1억 = 10억 |
| **경비** | 프로젝트 수행에 필요한 직접 비용 | 출장비, 장비 구매, 전기요금, 통신비, 소프트웨어 라이선스 |
| **간접경비** | 프로젝트를 간접 지원하는 비용 | 회계팀, 마케팅팀, 경영지원팀 등 오버헤드(간접 지원) 비용 |

### 비용에 영향을 미치는 요소들

비용은 다음과 같은 요소들에 의해 크게 달라집니다.

**① 프로젝트 규모**: 홈페이지 페이지 수가 100개냐 1,000개냐에 따라 비용이 기하급수적으로 증가합니다.

**② 프로그램 복잡도**: 단순 응용 프로그램을 1이라 하면, 복잡한 응용 프로그램은 3, 전체 시스템 규모는 9 정도의 비율로 비용이 증가합니다.

**③ 개발자의 역량**: 해당 언어에 얼마나 숙련되어 있는지, 디버깅(오류 수정) 능력이 얼마나 되는지가 비용과 기간에 직접 영향을 줍니다.

**④ 신뢰도 수준 (MTBF)**: MTBF(Mean Time Between Failure, 평균 고장 간격)는 에러가 얼마나 자주 발생하는지를 나타내는 지표입니다. 예를 들어 100시간 운용 중 10시간 동안 고장이 났다면 신뢰도는 90%입니다. 신뢰도가 높아야 할수록 개발에 더 많은 노력과 비용이 필요합니다.

**⑤ 기술 수준**: 어떤 장비, 도구, 개발 방법론을 사용하느냐에 따라서도 비용이 달라집니다.

### 비용 추정 방식: 상향식 vs 하향식

프로젝트 비용을 예측하는 방법에는 크게 두 가지가 있습니다.

**상향식(Bottom-Up) 방법**은 기간을 먼저 고정해놓고 인력을 계산하는 방식입니다. "5개월 안에 무조건 완성해야 해!"처럼 기간이 이미 정해진 경우에 사용합니다. 기간이 정해져 있으니 그 기간에 맞춰 몇 명이 필요한지를 역산하는 것입니다. 다만 개인의 주관적 판단에 의해 필요 인원이 달라질 수 있다는 단점이 있습니다.

**하향식(Top-Down) 방법**은 프로젝트의 규모를 먼저 예측하고, 그 규모에서 필요한 인력과 기간을 도출하는 방식입니다. 전체 페이지 수, 코드 라인 수, 난이도 등을 먼저 파악한 후 "이 규모면 몇 명이 몇 개월 하면 되겠다"를 계산합니다.

## 🧮 COCOMO: 비용 추정 공식

### COCOMO란 무엇인가?

**COCOMO(Constructive Cost Model, 구성적 비용 모델)**는 학자 보헴(Boehm)이 제안한 비용 추정 방법으로, **원시 프로그램의 규모(코드 라인 수)**를 기반으로 개발 기간과 필요 인원을 계산하는 방식입니다. 마치 건축에서 건물의 총 면적을 기준으로 공사 기간과 인력을 산출하는 것과 비슷한 개념입니다.

### 프로젝트 유형 분류

COCOMO는 프로젝트의 규모와 복잡도에 따라 세 가지 유형으로 분류합니다.

| 유형 | 영문명 | 규모 | 특징 |
|------|--------|------|------|
| **유기형** | Organic | 5만 라인 이하 | 소규모의 일반적인 응용 프로그램 |
| **반결합형** | Semi-detached | ~30만 라인 | 트랜잭션 처리, 운영체제, 데이터베이스 관련 |
| **내장형** | Embedded | 30만 라인 이상 | 실시간 시스템, 미사일 제어, 신호기 제어 등 하드웨어 포함 대규모 시스템 |

### COCOMO 기본 공식

COCOMO의 기본 공식은 다음과 같습니다. 여기서 **KDSI(Kilo Delivered Source Instructions)**는 천 라인 단위의 코드 수를 의미합니다. 예를 들어 코드가 1,000줄이면 1 KDSI입니다.

```
# COCOMO 기본 공식 (개념 설명용)

# 유기형 (Organic)
# MM: Man-Month (사람-달, 개발에 필요한 총 인력 × 기간)
# TDEV: Total Development Time (총 개발 기간, 단위: 개월)

MM_organic   = 2.4  × (KDSI ^ 1.05)   # 필요 맨먼스 계산
TDEV_organic = 2.5  × (MM ^ 0.38)     # 총 개발 기간 계산

# 반결합형 (Semi-detached)
MM_semi      = 3.0  × (KDSI ^ 1.12)
TDEV_semi    = 2.5  × (MM ^ 0.35)

# 내장형 (Embedded)
MM_embedded  = 3.6  × (KDSI ^ 1.20)
TDEV_embedded = 2.5 × (MM ^ 0.32)
```

:::note MM과 TDEV의 의미
- **MM (Man-Month, 맨먼스)**: "한 사람이 한 달 동안 일하는 양"을 1 MM이라고 합니다. 6 MM이라면 "1명이 6달" 또는 "6명이 1달" 동안 해야 하는 작업량입니다.
- **TDEV (Total Development Time)**: 프로젝트를 완성하는 데 필요한 총 기간(개월)입니다.
- **필요 인원** = MM ÷ TDEV
:::

### COCOMO 계산 예시 (반결합형)

33,360 라인짜리 반결합형 프로젝트를 예시로 계산해 보겠습니다.

```python
# COCOMO 반결합형 계산 예시

# KDSI 변환: 33,360 라인 → 33.36 KDSI (1KDSI = 1,000라인)
KDSI = 33360 / 1000  # = 33.36

# 맨먼스(MM) 계산: 반결합형 공식 적용
# MM = 3.0 × (KDSI ^ 1.12)
MM = 3.0 * (KDSI ** 1.12)  # ≈ 152 MM

# 총 개발 기간(TDEV) 계산
# TDEV = 2.5 × (MM ^ 0.35)
TDEV = 2.5 * (MM ** 0.35)  # ≈ 14.5개월

# 필요 총 인원 계산
# 인원 = MM ÷ TDEV
인원 = MM / TDEV  # ≈ 10.5 → 약 11명

print(f"필요 맨먼스: {MM:.0f} MM")          # 필요 맨먼스: 152 MM
print(f"총 개발 기간: {TDEV:.1f}개월")       # 총 개발 기간: 14.5개월
print(f"필요 개발 인원: 약 {인원:.0f}명")    # 필요 개발 인원: 약 11명
```

위 계산 결과를 해석하면 이렇습니다. 33,360줄짜리 반결합형 프로그램을 개발하려면, **총 152 맨먼스의 작업량**이 필요하고, **약 14.5개월**의 기간이 걸리며, **약 11명의 개발자**가 필요하다는 결론이 나옵니다.

### COCOMO 보정: 가중치를 통한 정밀화

기본 COCOMO 공식만으로는 현실을 정확히 반영하기 어렵습니다. 예를 들어 경험 많은 개발자로 구성된 팀이라면 실제로는 더 적은 인원과 시간이 필요할 것입니다. 이를 보완하기 위해 **노력 승수(Effort Multiplier)**, 즉 가중치를 적용합니다.

```python
# COCOMO 보정 예시

# 기본 MM 계산 (내장형, 128,000 라인)
KDSI = 128000 / 1000  # = 128 KDSI
MM_기본 = 3.6 * (KDSI ** 1.20)  # ≈ 1216 MM
TDEV_기본 = 2.5 * (MM_기본 ** 0.32)  # ≈ 24개월
인원_기본 = MM_기본 / TDEV_기본  # ≈ 51명

# 가중치 적용 (보정)
# 신뢰도 매우 높음 → 가중치 1.15 (신뢰도 높은 소프트웨어는 더 많은 노력 필요)
# 주기억 장치 제약 매우 높음 → 가중치 1.56
# 응용 프로그래머 경험 매우 풍부 → 가중치 0.82 (경험 많으면 노력 줄어듦)

가중치_종합 = 1.15 * 1.56 * 0.82  # ≈ 1.47

MM_보정 = MM_기본 * 가중치_종합  # 보정된 맨먼스
TDEV_보정 = 2.5 * (MM_보정 ** 0.32)  # 보정된 개발 기간
인원_보정 = MM_보정 / TDEV_보정  # 보정된 필요 인원

print(f"기본 MM: {MM_기본:.0f}, 보정 MM: {MM_보정:.0f}")
print(f"기본 기간: {TDEV_기본:.1f}개월, 보정 기간: {TDEV_보정:.1f}개월")
print(f"기본 인원: {인원_기본:.0f}명, 보정 인원: {인원_보정:.0f}명")
```

가중치는 다음과 같은 요소들에 따라 달라집니다.

| 분류 | 요소 | 낮을 때 | 높을 때 |
|------|------|---------|---------|
| **제품 특성** | 요구 신뢰도 | 가중치 낮음 | 가중치 높음 |
| **제품 특성** | 데이터베이스 크기 | 가중치 낮음 | 가중치 높음 |
| **개발자 특성** | 개발자 경험 | 가중치 높음 (경험 없으면 더 힘듦) | 가중치 낮음 |
| **개발자 특성** | 언어 숙련도 | 가중치 높음 | 가중치 낮음 |
| **프로젝트 특성** | 개발 일정 제약 | 가중치 낮음 | 가중치 높음 |

:::tip COCOMO 이렇게 이해하세요
COCOMO를 처음 보면 공식이 복잡해 보이지만 핵심은 단순합니다. **"코드가 몇 줄이면, 몇 명이 몇 달 필요하다"**를 계산하는 것입니다. 공식 자체를 외우는 것보다, **코드 규모로 인원과 기간을 객관적으로 산출한 뒤 현실적인 조건으로 보정한다**는 개념을 이해하는 것이 중요합니다.
:::

### COCOMO 2: 개발 단계를 반영한 진화된 모델

**COCOMO 2**는 기존 COCOMO의 한계를 극복한 발전된 모델로, 프로젝트 진행 단계에 따라 다른 계산 방법을 적용합니다.

- **1단계**: 프로토타입(시제품) 제작 단계
- **2단계**: 초기 설계 단계
- **3단계**: 구조 설계 이후 단계

왜 단계별로 다른 계산을 할까요? 프로젝트 초기에는 불확실한 요소가 많아 대략적인 추정만 가능하지만, 구조 설계가 완료된 이후에는 훨씬 정밀한 계산이 가능하기 때문입니다. 프로젝트가 진행될수록 더 정확한 비용 추정이 가능해집니다.

## 📐 기능 점수(FP) 방법: 라인 수를 모를 때의 대안

### 왜 기능 점수 방법이 필요한가?

COCOMO는 코드 라인 수를 기반으로 계산합니다. 그런데 현실에서 "이 프로젝트는 총 30만 라인짜리입니다"라고 미리 정확히 알 수 있을까요? 사실상 불가능합니다. 같은 기능을 어떤 언어로, 어떤 방식으로 개발하느냐에 따라 코드 라인 수는 천차만별이기 때문입니다.

이 문제를 해결하기 위해 등장한 것이 바로 **기능 점수(FP, Function Point)** 방법입니다. 마치 아파트 면적을 계산할 때 방 개수, 화장실 개수, 창문 개수를 세어서 면적을 산출하듯이, 소프트웨어의 기능 구성 요소를 세어서 전체 규모를 파악하는 방식입니다.

### 기능 점수의 5가지 구성 요소

기능 점수는 다음 5가지 항목을 기준으로 측정합니다.

| 항목 | 설명 | 예시 |
|------|------|------|
| **입력 (Input)** | 사용자가 시스템에 입력하는 데이터 | 회원가입 폼, 주문서 작성 |
| **출력 (Output)** | 시스템이 사용자에게 보여주는 화면 | 주문 결과 화면, 영수증 출력 |
| **질의 (Query)** | 사용자의 조회 요청 처리 | 상품 검색, 주문 내역 조회 |
| **파일 (File)** | 시스템이 관리하는 내부 데이터 파일 | 회원 정보 DB, 상품 목록 DB |
| **인터페이스 (Interface)** | 다른 시스템과의 연결 지점 | 결제 API 연동, 배송 시스템 연동 |

### 기능 점수 계산 예시

각 항목에는 난이도에 따라 가중치 점수가 다르게 부여됩니다. (단순: 3점, 보통: 4점, 복잡: 6점 등)

다음은 간단한 쇼핑몰 프로젝트의 기능 점수 계산 예시입니다.

```python
# 기능 점수(FP) 계산 예시

# 각 항목별 수량과 가중치 점수 정의
# 가중치: 단순=낮은값, 보통=중간값, 복잡=높은값

# (항목, 개수, 가중치점수) 형태
fp_항목 = {
    "입력":       (5, 4),   # 입력 5개, 보통 난이도(4점)
    "출력":       (6, 5),   # 출력 6개, 보통 난이도(5점)
    "질의":       (6, 3),   # 질의 6개, 단순(3점)
    "파일":       (8, 10),  # 파일 8개, 보통(10점)
    "인터페이스": (0, 7),   # 인터페이스 없음
}

# 기능 점수 합산
FP_합계 = 0
for 항목, (개수, 점수) in fp_항목.items():
    소계 = 개수 * 점수
    FP_합계 += 소계
    print(f"{항목}: {개수}개 × {점수}점 = {소계}점")

print(f"\n총 기능 점수(FP): {FP_합계}점")  # 148점

# LOC(라인 수) 변환: FP × 언어별 1점당 평균 라인 수
# 예: Python은 1 FP당 약 25 LOC
LOC_per_FP = 25
총_LOC = FP_합계 * LOC_per_FP  # 148 × 25 = 3,700 라인

print(f"예상 총 코드 라인 수: {총_LOC}줄")  # 3,700줄

# 맨먼스 계산: 총 LOC ÷ 팀 월간 생산성
월간_생산성 = 2000  # 한 명이 한 달에 2,000줄 개발 가능
MM = 총_LOC / 월간_생산성  # 3,700 ÷ 2,000 = 1.85 MM

print(f"필요 맨먼스: {MM:.2f} MM")   # 약 1.85 MM
print(f"→ 약 2명의 개발자가 1개월 필요")
```

이 예시에서 계산 결과는 다음과 같습니다.

- **총 기능 점수(FP)**: 148점
- **예상 총 코드 라인 수**: 148 × 25 = **3,700 라인**
- **필요 맨먼스**: 3,700 ÷ 2,000 = **약 1.9 MM** (즉, 약 2명의 개발자가 1개월 필요)

:::note 기능 점수 방법의 장점
기능 점수 방법은 코드를 한 줄도 작성하기 전, 즉 **개발 계획 단계에서도 프로젝트 규모를 예측**할 수 있다는 것이 가장 큰 장점입니다. 또한 어떤 프로그래밍 언어를 사용하든 동일한 기준으로 측정할 수 있어 언어 독립적인 규모 측정이 가능합니다.
:::

## 🚧 주의사항과 실전 팁

### 비용 추정 시 절대 하지 말아야 할 것들

:::danger 계약 수주를 위한 허위 낮은 견적 제시
프로젝트를 따내기 위해 실제 필요한 비용보다 훨씬 낮은 견적을 제시하는 것은 절대 하면 안 됩니다. 개발 도중 예산이 바닥나면 품질이 떨어지거나 프로젝트가 중단됩니다. 이는 클라이언트뿐 아니라 개발 회사 자신에게도 큰 손해입니다.
:::

:::danger 근거 없는 과도한 견적 부풀리기
반대로 합리적인 근거 없이 비용을 과도하게 부풀리면 신뢰를 잃고 계약 자체가 성립되지 않습니다. 모든 비용 추정에는 **객관적인 근거**가 있어야 합니다.
:::

### 실전에서 통하는 비용 추정 팁

:::tip 과거 프로젝트 데이터를 쌓아라
COCOMO든 기능 점수 방법이든, 가장 중요한 것은 **과거 프로젝트에서 축적된 실제 데이터**입니다. "이런 규모의 프로젝트를 이전에 3명이 4개월만에 완성했다"는 경험 데이터가 있으면 훨씬 정확한 예측이 가능합니다. 프로젝트가 끝날 때마다 반드시 실적 데이터를 기록하고 보관하세요.
:::

:::tip 버퍼(여유분)를 반드시 포함하라
아무리 정교하게 계산해도 예상치 못한 변수는 반드시 발생합니다. 기간과 비용에 **10~20%의 버퍼(여유분)**를 포함하는 것이 현장의 공통적인 지혜입니다. 이를 클라이언트에게 투명하게 설명하고 동의를 구하는 것이 장기적인 신뢰 관계에 도움이 됩니다.
:::

:::tip 단계별로 계획을 정밀화하라
COCOMO 2처럼 프로젝트 초기에는 큰 그림의 추정치를 제시하고, 설계가 구체화될수록 계획을 점점 더 정밀하게 다듬어 가는 방식이 현실적입니다. 처음부터 완벽한 계획을 세우려 하기보다, 단계별 검토와 수정을 반복하세요.
:::

### 잘못된 예 vs 올바른 예

❌ **잘못된 문제 정의**

```
클라이언트: "쇼핑몰 만들어주세요."
개발팀: "네, 알겠습니다. 시작할게요."
→ 기능 범위, 성능 요구사항, 법적 검토 없이 바로 개발 돌입
→ 결과: 개발 도중 요구사항 충돌, 기간 초과, 비용 초과
```

✅ **올바른 문제 정의**

```
클라이언트: "쇼핑몰 만들어주세요."
개발팀:
  1. 어떤 기능이 필요한지 요구사항 상세 인터뷰 진행
  2. 현재 운영 중인 시스템의 문제점 파악
  3. 경제적·기술적·법적 타당성 검토
  4. WBS 작성 및 범위 확정
  5. COCOMO 또는 기능 점수로 비용·기간·인원 산출
  6. 클라이언트와 계획서 검토 및 합의
→ 결과: 명확한 기준 하에 안정적인 프로젝트 진행 가능
```

## 📌 핵심 정리

이 문서에서 다룬 핵심 내용을 정리합니다.

- **프로젝트 계획**은 누가·언제·무엇을·어떻게 할 것인지를 프로젝트 시작 전에 미리 결정하는 작업이다
- 프로젝트 성패는 계획의 정밀도와 계획대로의 실행 여부에 달려 있다
- 프로젝트 관리의 핵심 3요소는 **자원(인력), 비용, 일정**이며 이를 합리적이고 객관적으로 측정하는 것이 핵심이다
- 관리 대상은 **3P(사람, 프로세스, 문제)**에서 발전한 **4P(사람, 프로세스, 제품, 프로젝트)**로 이해할 수 있다
- **문제 정의**는 개발의 첫 단추로, 사용자 니즈 분석 + 문제점 파악 + 경제적·기술적·법적 타당성 검토로 이루어진다
- **WBS(Work Breakdown Structure)**는 프로젝트 범위를 계층적 트리 구조로 분해한 작업 분류 체계이다
- 비용은 **인건비 + 경비 + 간접경비**로 구성되며, 프로젝트 규모·복잡도·개발자 역량·신뢰도 수준 등이 영향을 미친다
- **COCOMO**는 코드 라인 수(KDSI)를 기반으로 맨먼스·개발 기간·필요 인원을 계산하는 비용 추정 모델이다
- COCOMO는 유기형(5만 라인 이하), 반결합형(~30만 라인), 내장형(30만 라인 이상)으로 구분된다
- **COCOMO 보정**은 개발자 경험, 신뢰도 요구 수준, 하드웨어 제약 등에 따라 가중치를 적용해 추정값을 현실화하는 작업이다
- **COCOMO 2**는 프로젝트 진행 단계(프로토타입→초기 설계→구조 설계 이후)에 따라 다른 계산 방법을 적용하는 발전된 모델이다
- **기능 점수(FP)**는 입력·출력·질의·파일·인터페이스 개수를 기반으로 코드 라인 수를 예측하지 않고도 프로젝트 규모를 산출하는 방법이다
- 기능 점수 방법은 언어에 독립적이며 개발 계획 단계에서도 바로 적용 가능하다는 장점이 있다
- 모든 비용 추정에는 **과거 데이터와 경험**이 가장 중요한 자산이며, 현실적인 버퍼(여유분)를 반드시 포함해야 한다
