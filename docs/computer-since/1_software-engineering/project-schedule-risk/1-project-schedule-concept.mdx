# 🗓️ 프로젝트 일정의 개념과 구성 요소

소프트웨어 공학에서 코드를 잘 짜는 것만큼이나 중요한 것이 있습니다. 바로 **일정을 얼마나 잘 관리하느냐**입니다. 아무리 훌륭한 기능을 개발했더라도 정해진 기간 안에 납품하지 못하면 그 결과물은 의미를 잃을 수 있습니다. 이번 문서에서는 프로젝트 일정이란 무엇인지, 왜 중요한지, 그리고 어떻게 관리하는지를 초보자도 이해할 수 있도록 차근차근 살펴보겠습니다.

---

## 📌 프로젝트 일정이란 무엇인가?

### 일정(Schedule)의 기본 개념

여러분이 해외여행을 계획한다고 생각해 보세요. 첫째 날은 공항 도착 후 숙소 체크인, 둘째 날은 유명 관광지 방문, 셋째 날은 시장 투어, 마지막 날은 귀국. 이처럼 **여행의 각 날짜에 무엇을 할지를 순서대로 정리한 것**이 여행 일정입니다.

프로젝트 일정도 동일한 원리입니다. 소프트웨어 개발 프로젝트에는 혼자 뚝딱 만드는 개인 프로젝트가 아니라, 여러 명이 팀을 이루어 함께 진행하는 경우가 대부분입니다. 그리고 대부분의 프로젝트에는 **마감 기한(Deadline)**이 존재합니다. 3개월, 6개월, 때로는 1주일처럼 명확한 종료 시점이 있습니다.

> **프로젝트 일정**이란, 개발 프로젝트의 전체 프로세스(흐름)를 이루는 각각의 **작업(Task)**들을 파악하고, 그 작업들의 **순서(Order)**와 **소요 기간(Duration)**을 정리한 것입니다.

### 집 짓기 비유로 이해하는 프로젝트 일정 🏠

산속에 예쁜 별장을 짓는다고 상상해 봅시다. 재료를 사다가 바로 망치질부터 시작하지는 않겠죠? 실제 집을 지으려면 다음과 같은 순서로 진행됩니다.

| 단계 | 작업 내용 | 이유 |
|------|-----------|------|
| 1단계 | 부지 검토 (산림법, 건축법 확인) | 집을 지을 수 있는 땅인지 먼저 확인해야 함 |
| 2단계 | 설계도 작성 (건축사 의뢰) | 전체 구조를 계획해야 공사 가능 |
| 3단계 | 기초 공사 (터 닦기, 다지기) | 바닥이 튼튼해야 건물이 안전 |
| 4단계 | 골격 세우기 (뼈대 올리기) | 건물의 구조체 형성 |
| 5단계 | 외벽 및 지붕 공사 | 외부 마감 처리 |
| 6단계 | 내부 공사 (전기, 수도, 인테리어) | 실제 생활 가능한 공간 완성 |
| 7단계 | 준공 검사 및 허가 | 법규 기준 충족 여부 확인 |
| 8단계 | 입주 및 유지 보수 | 실생활 시작, 문제 발생 시 수리 |

이처럼 집을 짓는 것도 **각 단계별 작업**, **정해진 순서**, 그리고 각 작업에 필요한 **소요 기간**이 존재합니다. 소프트웨어 프로젝트도 마찬가지입니다. 요구 분석 → 설계 → 구현 → 테스트 → 유지 보수라는 단계가 있고, 각 단계마다 해야 할 작업과 기간이 존재하며, 이 모든 것을 묶어서 **프로젝트 일정**이라고 부릅니다.

## ⚠️ 왜 일정 관리가 중요한가?

### 기간 vs 품질, 무엇이 더 중요할까?

여기서 한 가지 질문을 드려볼게요.

- **케이스 A**: 납품 기한보다 한 달 늦었지만, 품질이 매우 훌륭한 소프트웨어
- **케이스 B**: 납품 기한을 정확히 지켰지만, 품질이 다소 아쉬운 소프트웨어

어느 것이 더 좋은 결과물일까요? 물론 **기한도 지키고 품질도 좋은 것**이 이상적입니다. 하지만 현실의 계약 관계에서, 실무에서는 **기간이 더 우선순위에 놓이는 경우가 많습니다.**

:::warning 기한을 넘기면 프로젝트가 쓰레기가 될 수 있습니다
예를 들어 대학교 수업을 생각해 보세요. 학기 중에 아무리 A+를 받아도, 졸업 요건을 충족하는 시기가 늦어져서 올해 안에 학위를 받지 못한다면 어떻게 될까요? 성적이 좋은 것보다 **기한 내에 학위를 취득하는 것** 자체가 더 중요해집니다. 프로젝트도 동일합니다. 납품 기한을 넘겨버리면 아무리 완성도가 높아도 계약 위반이 됩니다.
:::

### 개발자들이 야근하는 이유 🌙

왜 개발자들은 밤을 새우고 주말에도 출근하며 일할까요? 그것은 **프로젝트라는 개념 자체가 약속**이기 때문입니다. 고객과 계약서를 쓰고 특정 기일 내에 특정 기능을 납품하겠다고 약속한 것이 프로젝트입니다. 그 약속을 지키지 못하면 위약금이 발생하거나 신뢰가 무너집니다. 그러므로 **개발 기간이 넘어가지 않도록 일정을 철저하게 관리하는 것**이 프로젝트 관리의 핵심 목표입니다.

> 💡 **일정 관리(Schedule Management)**란, 소프트웨어 생명주기(요구 분석 → 설계 → 구현 → 테스트 → 유지 보수)의 각 개발 단계별로 정해진 일정 계획에 따라 작업이 제대로 진행되고 있는지를 **감시(Monitoring)**하고 **통제(Control)**하는 활동입니다.

## 🔧 프로젝트 일정 관리의 핵심 원리

프로젝트 일정을 관리하는 것은 단순히 달력에 날짜를 표시하는 것이 아닙니다. 체계적인 방법론을 따릅니다. 전체 흐름은 다음과 같습니다.

```
① 작업 분해 (WBS)
      ↓
② 작업 순서 및 의존 관계 파악 (CPM 네트워크)
      ↓
③ 최소/최대 소요 시간 계산 (PERT)
      ↓
④ 필요 인력 산정
      ↓
⑤ 일정표 작성 및 관리 (간트 차트)
```

각 단계를 하나씩 자세히 살펴보겠습니다.

## 📂 WBS: 작업 분해 구조

### WBS란 무엇인가?

**WBS(Work Breakdown Structure)**는 우리말로 **업무 분해 구조**입니다. 단어를 하나씩 풀어보면 이렇습니다.

- **Work**: 업무, 작업
- **Breakdown**: 분류하다, 쪼개다
- **Structure**: 구조, 체계

즉, **개발해야 할 전체 작업을 관리하기 쉬운 작은 단위로 쪼개서 계층 구조로 정리하는 방법**입니다. WBS를 만드는 이유는 명확합니다. 전체 프로젝트가 너무 크고 복잡하면 어디서부터 시작해야 할지 막막합니다. 이 큰 덩어리를 잘게 쪼개야 각 작업에 얼마나 많은 시간과 인력, 비용이 필요한지 계산할 수 있기 때문입니다.

:::info WBS의 핵심 특징
WBS는 작업의 **목록과 계층**을 나타낼 뿐, **작업의 순서는 나타내지 않습니다.** "재고 관리를 먼저 하고, 그 다음에 시설 관리를 한다"와 같은 순서 정보는 WBS에 포함되지 않습니다. 오직 "어떤 작업들이 있는가?"만을 정리하는 것이 WBS입니다.
:::

### ERP 시스템 개발 WBS 예시

회사 전체를 전산화하는 **ERP(Enterprise Resource Planning, 전사적 자원 관리)** 시스템을 개발한다고 가정해 봅시다. 전체 작업을 계층 구조로 분해하면 다음과 같습니다.

```
📦 ERP 시스템 개발 (최상위)
│
├── 💰 회계 관리 시스템
│   ├── 급여 관리 모듈
│   ├── 예산 관리 모듈
│   └── 자금 관리 모듈
│
├── 🏭 생산 관리 시스템
│   ├── 재고 관리 모듈
│   ├── 시설 관리 모듈
│   └── 공정 관리 모듈
│
└── 🛒 시장 관리 시스템
    ├── 고객 관리 모듈
    └── 판매 관리 모듈
```

이렇게 구조화하면 다음과 같은 이점이 생깁니다.

- **급여 관리 모듈** 개발에는 개발자 2명, 30일 필요 → 비용 산정 가능
- **재고 관리 모듈** 개발에는 개발자 3명, 45일 필요 → 인력 배치 가능
- 각 모듈을 합치면 상위 시스템이 완성되고, 모든 시스템이 합쳐지면 전체 ERP 완성

:::tip WBS를 만들 때의 팁
WBS를 처음 만들 때는 포스트잇이나 마인드맵 도구를 활용하면 편리합니다. 가장 큰 단위(전체 프로젝트)를 맨 위에 놓고, 점점 아래로 내려가며 세분화하세요. 더 이상 쪼갤 수 없을 정도로 작은 단위가 되면 그것이 실제 개발 가능한 **작업 패키지(Work Package)**가 됩니다.
:::

## 🔗 CPM: 임계 경로 분석

### CPM이란 무엇인가?

WBS로 "어떤 작업들이 있는지"는 파악했습니다. 이제 그 작업들을 **어떤 순서로 해야 하는지**, 그리고 **전체 프로젝트를 끝내는 데 최소 얼마나 걸리는지**를 분석해야 합니다. 이때 사용하는 기법이 **CPM(Critical Path Method, 임계 경로 기법)**입니다.

**임계 경로(Critical Path)**란 프로젝트 시작부터 끝까지 가장 긴 시간이 소요되는 작업 경로입니다. 마치 교통 체증이 심한 도로처럼, 이 경로가 지연되면 전체 프로젝트가 지연됩니다.

> 📌 예를 들어 성적 조회 기능을 개발하려면, **성적 입력 기능**과 **성적 처리 기능**이 먼저 완성되어야 합니다. 데이터 자체가 없는데 조회 화면만 만들어봤자 아무 의미가 없기 때문입니다. 이처럼 앞에 완료되어야 하는 작업을 **선행 작업(Predecessor)**이라고 합니다.

### CPM 네트워크 그리는 방법

CPM은 **방향성 그래프(Directed Graph)**로 표현합니다. 그래프를 그리는 요소는 세 가지입니다.

| 요소 | 모양 | 의미 |
|------|------|------|
| **원형 노드** | ⭕ | 소작업(이름 + 소요 기간 표시) |
| **박스 노드** | ⬜ | 중간 점검 지점(예상 종료율 표시) |
| **엣지(화살표)** | → | 작업의 전후 순서 관계 |

### CPM 네트워크 예시 분석

아래와 같은 작업 목록이 있다고 가정해 봅시다.

| 작업 | 선행 작업 | 소요 기간 |
|------|-----------|-----------|
| A | 없음 | 8일 |
| B | 없음 | 15일 |
| D | 없음 | 10일 |
| C | A | 15일 |
| E | B, D | 12일 |
| H | A | 7일 |
| I | A, C | 10일 |
| K | E, I | 8일 |
| L | H, K | 5일 |

이 표에서 읽어야 할 핵심 정보는 다음과 같습니다.

- **A, B, D**: 선행 작업이 없으므로 프로젝트 시작과 동시에 착수 가능
- **C**: A가 끝나야만 시작할 수 있음 (A → C)
- **E**: B와 D가 **모두** 끝나야 시작할 수 있음 (B, D → E)
- **K**: E와 I가 모두 끝나야 시작 가능 (E, I → K)
- **L**: H와 K가 모두 끝나야 시작 가능 (H, K → L)

```
[START]
   ├──→ (A: 8일) ──→ [M1] ──→ (C: 15일) ──→ (I: 10일) ──┐
   │              └──→ (H: 7일)  ───────────────────────────→ (L: 5일) → [END]
   ├──→ (B: 15일) ──┐                                      ↑
   └──→ (D: 10일) ──┴──→ (E: 12일) ──→ (K: 8일) ──────────┘
```

:::note 중간 점검 박스(M1)의 역할
CPM 그래프에서 박스 노드(⬜)는 **마일스톤(Milestone, 중간 점검 지점)**을 의미합니다. "A 작업이 정말로 완료되었는가?"를 검증한 후에 다음 작업(C, H 등)으로 넘어가도록 합니다. 실수나 오류가 다음 단계로 전파되는 것을 막아주는 안전장치 역할을 합니다.
:::

### 임계 경로(Critical Path) 찾기

CPM의 핵심은 **가장 시간이 많이 걸리는 경로**를 찾는 것입니다. 각 경로별 총 소요 시간을 계산해 봅시다.

| 경로 | 작업 순서 | 총 소요 일수 |
|------|-----------|-------------|
| 경로 1 | A → C → I → K → L | 8+15+10+8+5 = **46일** |
| 경로 2 | A → H → L | 8+7+5 = **20일** |
| 경로 3 | B → E → K → L | 15+12+8+5 = **40일** |
| 경로 4 | D → E → K → L | 10+12+8+5 = **35일** |

가장 긴 경로는 **경로 1(46일)**입니다. 따라서 임계 경로는 **A → C → I → K → L**이며, 이 프로젝트는 아무리 빠르게 진행해도 최소 **46일**이 걸립니다.

:::danger 임계 경로를 줄이지 않으면 아무 의미가 없습니다
경로 2(A → H → L)를 아무리 줄여도(예: 20일 → 10일) 전체 프로젝트 기간은 여전히 46일입니다. **임계 경로 위에 있는 작업들(A, C, I, K, L)을 줄여야만** 전체 프로젝트 기간이 단축됩니다. 프로젝트 관리자는 임계 경로 위의 작업들에 가장 집중적인 자원과 관심을 투입해야 합니다.
:::

## 📊 PERT: 불확실성을 고려한 일정 예측

### PERT란 무엇인가?

**PERT(Program Evaluation and Review Technique, 프로그램 평가 및 검토 기법)**는 개발 기간의 **불확실성(Uncertainty)**을 고려하여 일정을 예측하는 기법입니다. CPM이 확정된 소요 시간을 기반으로 한다면, PERT는 "이 작업이 얼마나 걸릴지 정확히 모를 때" 사용합니다.

PERT는 특히 다음과 같은 상황에서 유용합니다.

- **처음 해보는 신규 사업**: 경험이 없으므로 소요 시간을 정확히 알 수 없음
- **비반복 사업**: 한 번만 하는 프로젝트로 참고할 과거 데이터가 없음

### PERT의 세 가지 시간 추정값

PERT는 하나의 작업에 대해 세 가지 시나리오를 모두 고려합니다. 마치 날씨 예보처럼 "맑음/흐림/비" 세 가지 경우를 모두 준비하는 것과 같습니다.

| 구분 | 이름 | 설명 | 예시 |
|------|------|------|------|
| **낙관치(Optimistic)** | 가장 좋은 경우 | 아무 문제 없이 최대한 빠르게 완료 | 40일 |
| **기대치(Most Likely)** | 평균적인 경우 | 일반적인 상황에서 완료 | 60일 |
| **비관치(Pessimistic)** | 가장 나쁜 경우 | 퇴사, DB 장애 등 문제 발생 시 | 80일 |

예를 들어 어떤 작업이 "낙관적으로는 40일, 평균적으로는 60일, 비관적으로는 80일" 걸린다면, 이 세 값을 종합하여 현실적인 예측 기간을 구합니다. 이렇게 하면 "그냥 60일로 잡자"보다 훨씬 안정적인 일정 계획을 세울 수 있습니다.

:::tip PERT를 사용해야 하는 이유
현실에서는 중간에 팀원이 퇴사하거나, 데이터베이스에 예상치 못한 문제가 생기거나, 성능 테스트에서 실패하는 등 수많은 변수가 발생합니다. PERT는 이런 불확실성을 처음부터 계획에 반영하기 때문에, 예상보다 일정이 크게 초과되는 상황을 미연에 방지할 수 있습니다.
:::

## 📅 간트 차트(Gantt Chart): 일정을 눈으로 보다

### 간트 차트란 무엇인가?

WBS로 작업을 나누고, CPM/PERT로 순서와 기간을 파악했다면, 이제 이것을 **보기 쉬운 표 형태**로 만들어야 합니다. 이것이 **간트 차트(Gantt Chart)**입니다. 간트 차트는 프로젝트의 각 작업을 행(Row)으로, 시간(날짜/주/월)을 열(Column)로 표현하고, 각 작업의 **시작과 종료를 막대 그래프**로 나타낸 일정표입니다.

마치 학교 시간표처럼, 언제 어떤 수업(작업)이 있는지 한눈에 볼 수 있는 시각화 도구입니다.

### 간트 차트의 구성 요소

간트 차트에서 막대의 의미를 이해하는 것이 중요합니다.

| 표시 | 의미 |
|------|------|
| **흰색(밝은) 막대** | 해당 작업의 실제 소요 기간 |
| **회색 막대** | 슬랙 타임(Slack Time, 예비 시간) — 다음 작업에 영향을 주지 않고 연장 가능한 여유 시간 |
| **진행 표시** | 현재 기준으로 완료된 작업 비율 |

### 간트 차트 예시

```
작업명              | 1월 | 2월 | 3월 | 4월 | 5월 | 6월
--------------------|-----|-----|-----|-----|-----|-----
명세서 검토         |████|     |     |     |     |
예산 검토           |████|███  |     |     |     |
스케줄 검토         |     |████|     |     |     |
개발 계획 수립      |     |████|████|     |     |
계획 승인           |     |     |  ██|     |     |
설계                |     |     |████|████|     |
구현                |     |     |     |████|████|
테스트              |     |     |     |     |████|████
```

이 차트를 보면 "현재 4월 기준으로 설계는 완료되었고, 구현이 진행 중이며, 테스트는 아직 시작되지 않았다"는 것을 한눈에 파악할 수 있습니다.

### 인력 배치도와 자원 히스토그램

간트 차트는 작업 일정뿐 아니라 **인력 배치**에도 활용됩니다.

```
인원    | 1월~3월 | 4월~6월 | 7월~9월 | 10월~12월
--------|---------|---------|---------|----------
프레드  |   ████  |   ████  |         |
제인    |         |   ████  |   ████  |
민수    |   ████  |         |         |   ████
```

또한 **자원 히스토그램(Resource Histogram)**을 통해 특정 기간에 자원이 과부하(Overload) 상태인지, 여유(Underload) 상태인지를 시각적으로 확인할 수 있습니다. 예를 들어 6월~9월에 개발과 테스트가 동시에 진행되어 인력이 부족하다면, 미리 추가 인력을 확보하거나 일정을 조정할 수 있습니다.

:::tip 실무에서의 간트 차트
실무 현장에서는 Microsoft Project, Jira, Notion, Asana 등 다양한 도구로 간트 차트를 관리합니다. 처음 취업하면 팀장이 "간트 차트 업데이트해 주세요"라고 요청할 수 있으니, 기본 개념을 꼭 익혀두세요.
:::

## 👥 프로젝트 조직 구성

### 조직이 일정에 미치는 영향

일정을 잘 짜는 것도 중요하지만, **그 일정을 실행하는 사람들의 조직 구조**도 매우 중요합니다. 잘못된 조직 구성은 의사소통 문제를 일으키고, 결국 일정 지연으로 이어집니다. 소프트웨어 개발 생산성에 큰 영향을 미치는 프로젝트 조직 유형에는 세 가지가 있습니다.

### ① 책임 프로그래머 팀 조직 (Chief Programmer Team)

외과 수술팀을 생각해 보세요. 집도의(Chief Surgeon)가 모든 결정을 내리고, 보조 의사와 간호사가 그 지시에 따라 움직입니다. **책임 프로그래머 팀**도 동일한 구조입니다.

```
책임 프로그래머 (Chief Programmer)
       │
       ├── 프로그램 사서 (문서 관리, 테스트 계획 관리)
       ├── 프로그래머 (실제 개발 수행)
       └── 보조 프로그래머 (프로그래머 지원)
```

| 장점 | 단점 |
|------|------|
| 의사결정이 빠름 | 책임 프로그래머 한 명에게 성패가 달림 |
| 소규모 프로젝트에 적합 | 책임 프로그래머에게 업무 과중 |
| 초보자 훈련에 효과적 | 보조 프로그래머의 역할이 모호함 |

### ② 에고리스 팀 조직 (Egoless Team)

에고리스(Egoless)는 "자아가 없는", 즉 **민주주의식 의사결정**을 의미합니다. 팀원 모두가 동등한 권한과 책임을 가지고, 서로 토론하며 의사결정을 내립니다. 요즘 스타트업이나 수평적 문화를 지향하는 기업에서 많이 볼 수 있는 형태입니다. 직급 대신 닉네임으로 부르는 조직 문화가 이에 해당합니다.

| 장점 | 단점 |
|------|------|
| 팀 사기(Morale)가 높음 | 책임 소재가 불명확해질 수 있음 |
| 이직률이 낮음 | 대규모 프로젝트에 부적합 |
| 의사 교류가 활발함 | 회의가 너무 많아질 수 있음 |
| 장기 프로젝트에 적합 | 의사결정에 시간이 오래 걸림 |

### ③ 개층적 팀 조직 (Hierarchical Team)

앞의 두 가지 형태를 **적절히 결합한 혼합 형태**입니다. 상위에는 PM(프로젝트 매니저)과 고급 프로그래머가, 하위에는 중급/초급 프로그래머가 배치됩니다. 내부적으로는 협력하되, 의사결정 체계는 계층적으로 유지합니다.

```
PM (프로젝트 매니저)
       │
  고급 프로그래머 (A등급)
       │
  ┌────┴────┐
중급 개발자  중급 개발자
  │          │
초급 개발자  초급 개발자
```

| 장점 | 단점 |
|------|------|
| 권한과 책임이 명확 | 의사전달 경로가 길어짐 |
| 노하우 전수가 용이 | 하위 직원의 의견이 무시될 수 있음 |
| 대규모 프로젝트에 적합 | 계층이 많으면 의사소통 지연 |

:::warning 인원 추가 투입 = 기간 단축? ❌
많은 분들이 "인원을 더 투입하면 기간이 줄어들지 않을까?"라고 생각합니다. 하지만 현실은 다릅니다. 새로운 팀원이 합류하면 기존 코드를 파악하는 시간, 팀과의 인수인계, 새로운 커뮤니케이션 비용이 추가로 발생합니다. 이는 오히려 전체 생산성을 일시적으로 떨어뜨릴 수 있습니다. 처음부터 **적절한 규모의 조직을 효율적으로 구성**하는 것이 훨씬 중요합니다.
:::

## 🚨 위험 분석(Risk Analysis)과 위험 관리

### 위험(Risk)이란 무엇인가?

프로젝트를 진행하다 보면 예상치 못한 문제들이 발생합니다. 핵심 개발자의 갑작스러운 퇴사, 예산 초과, 요구 사항의 지속적인 변경 등이 대표적입니다. 이런 **프로젝트 성공을 위협하는 모든 불확실한 요소들**을 **위험(Risk)**이라고 합니다.

**위험 분석(Risk Analysis)**이란 프로젝트에 숨겨진 위험 요소를 미리 파악하고, 그것이 프로젝트에 미치는 영향을 분석하며, 그 위험을 관리하는 일련의 활동입니다.

### 대표적인 위험 요소들

| 위험 유형 | 구체적 사례 |
|-----------|-------------|
| **인력 위험** | 핵심 개발자 퇴사, 인력 부족 |
| **일정 위험** | 비현실적인 일정 수립, 작업 지연 |
| **예산 위험** | 예산 초과, 비용 급증 |
| **기술 위험** | 성능 저하, 기술적 취약점 발견 |
| **요구 사항 위험** | 요구 사항의 지속적 변경 |
| **과대 약속 위험** | 고객에게 실현 불가능한 기능을 약속 |

### 위험 관리 프로세스

위험 관리는 프로젝트 시작부터 종료까지 **지속적으로** 수행해야 하는 순환 활동입니다.

```
① 위험 관리 계획 수립
        ↓
② 위험 요소 파악 (식별)
        ↓
③ 위험 분석 (피해 정도, 발생 확률, 우선순위 결정)
        ↓
④ 위험 방지 및 해결
        ↓
⑤ 모니터링 (해결된 위험이 재발하지 않는지 지속 감시)
        ↓
    (다시 ②로 순환)
```

### 위험 우선순위 결정 방법

모든 위험을 동시에 해결할 수는 없습니다. 따라서 **우선순위**를 정해야 합니다. 일반적으로 **발생 확률**과 **피해 규모** 두 가지 기준으로 우선순위를 결정합니다.

| | 피해 규모 큼 | 피해 규모 작음 |
|--|-------------|---------------|
| **발생 확률 높음** | 🔴 최우선 해결 | 🟡 중간 우선순위 |
| **발생 확률 낮음** | 🟡 중간 우선순위 | 🟢 낮은 우선순위 |

예를 들어 "핵심 개발자 퇴사"는 발생 확률도 있고 피해 규모도 크기 때문에 최우선으로 관리해야 합니다. 반면 "사무실 인터넷 10분 장애"는 발생 확률은 있지만 피해 규모가 작으므로 낮은 우선순위로 관리합니다.

:::info 위험 관리는 PM(프로젝트 매니저)의 핵심 역할
위험 요소를 파악하고, 분석하고, 해결하고, 모니터링하는 전 과정이 **프로젝트 매니저(PM)**의 핵심 업무입니다. 좋은 PM은 코드를 잘 짜는 것이 아니라, 보이지 않는 위험을 미리 감지하고 팀을 보호하는 사람입니다.
:::

### 위험이 발생했을 때의 대처 예시

```
상황: 핵심 개발자 "김철수"가 퇴사 의사를 밝힘

① 위험 파악: 핵심 개발자 이탈 위험 발견
② 위험 분석: 피해 규모 '매우 큼' / 발생 확률 '높음' → 최우선 처리
③ 위험 해결:
   - 개인 면담을 통해 퇴사 원인 파악
   - 처우 개선 또는 역할 변경 제안
   - 프로젝트 완료 후 이직 협의
   - 동시에 대체 인력 풀(Pool) 확보 시작
④ 모니터링:
   - 주 1회 김철수와 면담 진행
   - 업무 만족도 지속 체크
   - 핵심 지식의 문서화 진행 (의존도 낮추기)
```

## 💡 주의사항 및 실전 팁

### ✅ 일정 관리에서 흔히 하는 실수들

**❌ 잘못된 예시: 너무 낙관적인 일정 수립**

```
// 나쁜 일정 계획의 예
로그인 기능 구현: 1일
회원가입 기능 구현: 1일
전체 DB 설계: 2일
API 전체 개발: 3일

// 현실: 위 작업들은 최소 2~3배 이상 시간이 필요합니다.
// 예상치 못한 버그, 설계 변경, 테스트 등이 모두 포함되어야 합니다.
```

**✅ 올바른 예시: 여유를 고려한 현실적 일정**

```
// 좋은 일정 계획의 예
로그인 기능 구현: 3일 (개발 1일 + 테스트 1일 + 버그 수정 여유 1일)
회원가입 기능 구현: 4일
전체 DB 설계: 5일 (리뷰 및 수정 포함)
API 전체 개발: 10일 (단위 테스트 포함)

// PERT 방식으로 낙관치/기대치/비관치를 모두 고려했습니다.
```

### 프로젝트 조직 구성 팁

```python
# 팀 구성 시 고려해야 할 요소 (의사 코드)

def compose_team(project_size, duration):
    team = []
    
    # 경험자와 초보자를 적절히 혼합
    # 경험 없는 사람들만 모으면 → 프로젝트 실패 위험
    # 경험자만 모으면 → 갈등 증가, 비용 과다
    team.append(senior_developer * 0.3)   # 시니어 30%
    team.append(mid_developer * 0.5)      # 미드 50%
    team.append(junior_developer * 0.2)   # 주니어 20%
    
    # 이직을 최소화하는 환경 조성
    # → 적절한 급여, 좋은 팀 분위기, 명확한 역할 부여
    
    # 프로젝트를 너무 잘게 쪼개지 말 것
    # → 팀이 너무 많아지면 의사소통 비용 폭발적 증가
    
    return team
```

### WBS, CPM, PERT, 간트 차트 비교 정리

| 기법 | 목적 | 핵심 질문 | 결과물 |
|------|------|-----------|--------|
| **WBS** | 작업 분해 | "어떤 작업들이 있는가?" | 계층 구조 작업 목록 |
| **CPM** | 경로 분석 | "가장 오래 걸리는 경로는?" | 방향성 그래프 + 임계 경로 |
| **PERT** | 기간 예측 | "불확실성을 고려하면 얼마나 걸리는가?" | 낙관치/기대치/비관치 |
| **간트 차트** | 일정 시각화 | "언제 무엇을 하는가?" | 막대 그래프 일정표 |

## ✅ 핵심 정리

- **프로젝트 일정**이란 개발 프로젝트의 각 작업(Task)에 대한 순서와 소요 기간을 정리한 계획이다.
- **일정 관리**는 개발 기간 지연을 방지하기 위해 반드시 필요하며, 계약 관계에서 기한 준수는 품질만큼이나 중요하다.
- **WBS(작업 분해 구조)**는 전체 프로젝트를 관리 가능한 작은 작업 단위로 분해하는 기법으로, 작업의 순서는 나타내지 않는다.
- **CPM(임계 경로 기법)**은 작업 간 의존 관계를 방향성 그래프로 표현하고, 가장 오래 걸리는 임계 경로를 찾아 관리한다.
- **임계 경로 위의 작업**을 줄여야만 전체 프로젝트 기간이 단축되며, 다른 경로를 아무리 줄여도 의미가 없다.
- **PERT**는 불확실성을 고려하여 낙관치/기대치/비관치 세 가지 시나리오로 일정을 예측하며, 경험 없는 신규/비반복 사업에 적합하다.
- **간트 차트**는 작업의 시작과 종료를 막대 그래프로 시각화한 일정표로, 인력 배치 및 자원 관리에도 활용된다.
- **프로젝트 조직 유형**에는 책임 프로그래머 팀(상명하달), 에고리스 팀(민주주의), 개층적 팀(혼합)이 있으며, 프로젝트 성격에 맞게 선택해야 한다.
- **인원 추가 투입이 기간 단축으로 이어지지 않는** 경우가 많으므로, 초기 조직 구성을 효율적으로 하는 것이 중요하다.
- **위험 관리(Risk Management)**는 프로젝트 시작부터 종료까지 파악 → 분석 → 해결 → 모니터링의 순환 과정을 지속적으로 수행해야 한다.
- 위험의 우선순위는 **발생 확률**과 **피해 규모**를 기준으로 결정하며, 고위험·고피해 요소를 최우선으로 관리한다.
