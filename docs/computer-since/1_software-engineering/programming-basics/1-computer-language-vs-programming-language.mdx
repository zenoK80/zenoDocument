# 💻 컴퓨터 언어와 프로그래밍 언어의 차이

프로그래밍을 처음 배우는 분들이 가장 혼란스러워하는 부분 중 하나가 바로 **"컴퓨터 언어"** 와 **"프로그래밍 언어"** 라는 두 용어입니다. 이름이 비슷해서 같은 말처럼 들리지만, 사실 이 둘은 엄밀하게 구분되는 개념입니다. 이 문서에서는 두 개념의 차이부터 시작해, 프로그래밍의 기본 원리, 언어의 종류와 역사, 그리고 실제 코딩 시 반드시 알아야 할 주의사항까지 초보자도 이해할 수 있도록 아주 친절하게 설명합니다.

---

## 🧩 컴퓨터 언어 vs 프로그래밍 언어 — 개념 소개

### 컴퓨터 언어란 무엇인가?

**컴퓨터 언어(Computer Language)** 란, 컴퓨터라는 시스템에서 소프트웨어가 동작하고 작동하도록 만들기 위한 언어입니다. 더 쉽게 말하면, 컴퓨터와 사람이 **서로 약속한 규칙으로 의사소통하는 수단** 이라고 볼 수 있습니다.

마치 한국 사람과 미국 사람이 대화하려면 서로 이해할 수 있는 공통 언어(예: 영어)가 필요하듯이, 사람과 컴퓨터 사이에서도 서로 이해할 수 있는 약속된 언어가 필요합니다. 그 약속된 언어 체계 전체를 **컴퓨터 언어** 라고 부릅니다.

> 💡 **핵심 정의**: 컴퓨터 언어 = 컴퓨터 시스템에서 소프트웨어를 만들고, 컴퓨터와 의사소통하기 위해 서로 약속된 규칙으로 만들어진 언어

### 프로그래밍 언어란 무엇인가?

**프로그래밍 언어(Programming Language)** 란, 말 그대로 **프로그램을 작성할 때 필요한 언어** 입니다. 소프트웨어(프로그램)를 만들기 위해 사용하는 도구적 언어라고 볼 수 있습니다.

컴퓨터 언어가 더 넓은 개념이라면, 프로그래밍 언어는 그 안에서 **"실제로 프로그램을 코딩(작성)하기 위한 언어"** 에 초점을 맞춘 개념입니다. 우리가 흔히 부르는 C 언어, Java, Python 같은 것들이 바로 프로그래밍 언어에 해당합니다.

### 두 개념의 차이를 표로 정리

| 구분 | 컴퓨터 언어 | 프로그래밍 언어 |
|------|------------|----------------|
| **의미** | 컴퓨터와 의사소통하기 위한 약속된 언어 전체 | 프로그램을 작성하기 위해 사용하는 언어 |
| **범위** | 더 넓은 개념 (상위 개념) | 컴퓨터 언어의 하위 개념 |
| **목적** | 컴퓨터 시스템과의 소통 | 소프트웨어(프로그램) 개발 |
| **예시** | 기계어, 어셈블리어, 각종 프로그래밍 언어 포함 | C, Java, Python, JavaScript 등 |

:::info 쉽게 기억하는 방법
"컴퓨터 언어"는 컴퓨터와 관련된 모든 언어를 아우르는 큰 바구니라고 생각하세요. "프로그래밍 언어"는 그 바구니 안에 들어있는 물건들 중 **"프로그램을 만드는 데 쓰는 언어"** 에 해당하는 것들입니다.
:::

## 🔍 왜 이 차이를 알아야 하는가? — 배경과 필요성

### 프로그래밍에는 문법(Syntax)이 존재한다

프로그래밍을 배우기 전에 이 개념을 구분해야 하는 이유는, **프로그래밍 언어에는 반드시 지켜야 할 문법적 규칙이 있기 때문** 입니다.

우리가 한국어를 쓸 때 "나는 밥을 먹었다"는 자연스럽지만 "밥을 나는 먹다 었"이라고 쓰면 이상하게 느껴지듯이, 프로그래밍 언어도 **정해진 문법 규칙(Syntax, 신택스)** 안에서만 작성해야 합니다. 이 규칙을 벗어나면 컴퓨터가 이해하지 못하고 오류(에러)가 발생합니다.

예를 들어, C 언어로 프로그래밍할 때는 C 언어만의 문법 규칙을 따라야 하고, Java로 프로그래밍할 때는 Java의 문법 규칙을 따라야 합니다. 이처럼 각 **프로그래밍 언어는 저마다 고유한 문법 체계** 를 가지고 있습니다.

### C 언어와 C 프로그램은 다르다

이 개념을 처음 접하는 분들이 자주 혼동하는 것이 있습니다. 바로 **"C 언어"** 와 **"C 프로그램(C 프로그래밍)"** 의 차이입니다.

- **C 언어**: C 언어로 프로그래밍할 수 있게 해주는 **컴파일러(번역 소프트웨어)** 와 그 언어 체계 자체를 의미합니다. 즉, 개발 도구이자 규칙 체계입니다.
- **C 프로그램**: C 언어를 사용해서 실제로 **개발된 소프트웨어** 를 의미합니다. 재고 관리 프로그램, 제어 프로그램 등이 그 예입니다.

비유하자면, **C 언어는 "붓과 물감"** 이고, **C 프로그램은 그것으로 그린 "그림"** 이라고 할 수 있습니다. 붓과 물감(C 언어)을 사용해서 그림(C 프로그램)을 그리는 것이 바로 **C 프로그래밍** 입니다.

## ⚙️ 핵심 원리 — 프로그래밍은 어떻게 동작하는가

### 소스 코드에서 실행 파일까지: 컴파일 과정

프로그래밍 언어로 작성한 코드(원시 코드, 소스 코드)가 컴퓨터에서 실제로 실행되려면 여러 단계를 거쳐야 합니다. 이 과정을 **컴파일(Compile)** 이라고 부릅니다.

마치 한국어로 쓴 문서를 외국인이 읽으려면 번역이 필요하듯, 사람이 이해하는 코드를 컴퓨터가 이해하는 언어로 **번역하는 과정** 이 바로 컴파일입니다.

전체 과정을 순서대로 살펴보겠습니다:

```
원시 프로그램 (소스 코드)
        ↓
    [컴파일러]  ← 문법 오류 체크
        ↓
  목적 프로그램 (기계어)
        ↓
     [링커]    ← 실행 가능한 형태로 연결
        ↓
  실행 파일 (.exe 등)
        ↓
     [로더]    ← 메모리에 올려서 실행
        ↓
    프로그램 실행!
```

각 단계를 자세히 살펴보겠습니다.

#### 1단계: 컴파일러(Compiler) — 문법 경찰

**컴파일러** 는 우리가 작성한 소스 코드에 **문법적 오류가 없는지 검사하는 역할** 을 합니다. 마치 맞춤법 검사기처럼, 약속된 문법 규칙을 벗어난 코드가 있으면 즉시 오류를 알려줍니다.

예를 들어, Java를 사용한다면:
- 변수는 어떻게 선언해야 하는지
- 반복문(for, while)은 어떤 형태로 작성해야 하는지
- 값을 반환할 때는 어떤 방식으로 해야 하는지
- 출력할 때는 어떤 함수를 써야 하는지

이 모든 것들이 이미 Java 언어의 규칙으로 **사전에 약속되어 정의** 되어 있습니다. 컴파일러는 이 규칙을 기준으로 코드를 검사합니다.

문법 오류가 없다면, 컴파일러는 소스 코드를 **기계어(컴퓨터가 이해하는 0과 1의 언어)로 변환** 합니다. 이 결과물을 **목적 프로그램(Object Program)** 이라고 합니다.

#### 2단계: 링커(Linker) — 조각을 하나로 연결

컴파일 후 만들어진 목적 프로그램은 아직 혼자서 실행될 수 없습니다. 프로그램은 보통 여러 개의 모듈(조각)로 이루어져 있고, 외부 라이브러리(미리 만들어진 기능 묶음)도 필요하기 때문입니다. **링커(Linker)** 는 이 조각들을 하나로 합쳐 **실제 실행 가능한 파일** 을 만들어줍니다.

레고 블록으로 비유하면, 컴파일러가 각각의 레고 조각을 만들어주고, 링커가 그 조각들을 **하나의 완성된 레고 작품으로 조립** 해주는 것입니다.

#### 3단계: 로더(Loader) — 실행 준비 완료

**로더(Loader)** 는 완성된 실행 파일을 컴퓨터의 메모리(기억 공간)에 올려서 실제로 프로그램이 실행되도록 합니다. 우리가 아이콘을 더블클릭해서 프로그램을 실행하면, 운영체제가 로더를 통해 이 작업을 수행합니다.

:::tip 개발자가 신경 써야 할 부분
컴파일 → 링킹 → 로딩의 전 과정 중, 개발자가 직접 신경 써야 할 부분은 **원시 코드(소스 코드)를 문법에 맞게 올바르게 작성하는 것** 뿐입니다. 나머지 과정은 C, Java, Python 등 각 언어의 개발 환경이 자동으로 처리해줍니다.
:::

### 실제 C 언어 코드 예시

아래는 간단한 C 언어 프로그램 예시입니다. 두 수를 더해서 출력하는 프로그램입니다.

```c
#include <stdio.h>  /* 표준 입출력 헤더 파일을 불러옴 (printf 함수 사용을 위해 필수) */

int main() {        /* 프로그램의 시작점 - main 함수는 반드시 있어야 함 */
    int a = 10;     /* 정수형 변수 a를 선언하고 10을 저장 */
    int b = 20;     /* 정수형 변수 b를 선언하고 20을 저장 */
    int sum;        /* 합계를 저장할 정수형 변수 sum 선언 */

    sum = a + b;    /* a와 b를 더해서 sum에 저장 (결과: 30) */

    printf("합계: %d\n", sum);  /* 결과를 화면에 출력. %d는 정수를 출력하는 형식 지정자 */

    return 0;       /* 프로그램이 정상적으로 종료됨을 운영체제에 알림 */
}
```

**실행 결과:**
```
합계: 30
```

이 코드를 컴파일하면 문법 오류가 없으므로 목적 프로그램이 생성되고, 링커를 통해 `test.exe` 같은 실행 파일이 만들어집니다. 실행하면 "합계: 30"이 출력됩니다.

## 🏗️ 코더 vs 프로그래머 — 그리고 구조적 프로그래밍

### 코더와 프로그래머의 차이

프로그래밍을 배우면서 반드시 이해해야 할 중요한 개념이 있습니다. 바로 **코더(Coder)** 와 **프로그래머(Programmer)** 의 차이입니다.

얼핏 같은 말처럼 보이지만, 이 둘은 실력과 역할에서 큰 차이가 있습니다.

| 구분 | 코더 (Coder) | 프로그래머 (Programmer) |
|------|-------------|------------------------|
| **하는 일** | 주어진 알고리즘대로 코드를 입력하는 작업 | 문제를 분석하고, 설계하고, 논리적으로 해결책을 만드는 작업 |
| **사고방식** | 기술적 코딩에 집중 | 논리적이고 체계적인 문제 해결에 집중 |
| **계획** | 별도의 계획 없이 주어진 대로 작업 | 기간, 비용, 방법을 치밀하게 계획 |
| **가치** | 반복 작업에 머무를 가능성 | 지속적으로 성장하고 가치가 높아짐 |

단순히 프로그래밍 언어를 알고, 누군가 던져준 알고리즘(문제 풀이 방법)에 따라 그냥 타이핑하는 사람이 코더입니다. 반면, **프로그래머** 는 논리적 두뇌를 가지고, 개발에 들어가는 인력·기간·비용을 고려하면서 체계적인 개발 방법을 스스로 설계하고 실행하는 사람입니다.

> 🎯 **핵심 메시지**: 단순히 코드를 많이 타이핑하는 것보다, 왜 이렇게 짜는지, 더 효율적인 방법은 없는지를 끊임없이 고민하는 습관이 훌륭한 프로그래머를 만듭니다.

### 프로그래밍의 목적과 과정

프로그래밍은 단순히 코드를 작성하는 행위가 아닙니다. **집을 짓는 것** 과 매우 비슷한 과정을 거칩니다.

집을 지으려면:
1. 어떤 집을 지을지 **요구사항 분석** (방은 몇 개? 층수는?)
2. 설계도(도면) 작성 → **설계 명세서**
3. 기초 공사부터 1층, 2층... **단계별 구현**
4. 완성 후 **검사 및 테스트**

프로그래밍도 똑같습니다:
1. **사용자 요구 조건 분석** — 무엇을 만들어야 하는가?
2. **설계** — 입력(Input)은 무엇으로 받을지, 처리(Process)는 어떻게 할지, 출력(Output)은 어떤 형태로 보여줄지
3. **코딩** — 설계를 바탕으로 실제 코드 작성
4. **테스트 및 디버깅** — 오류 발견 및 수정

특히 설계 단계에서 **입력-처리-출력(IPO: Input-Process-Output)** 을 명확히 정의하는 것이 매우 중요합니다.

- **입력(Input)**: 키보드로 받을지, 음성으로 받을지, 파일로 받을지
- **처리(Process)**: 어떤 알고리즘으로 데이터를 처리할지
- **출력(Output)**: 텍스트로 보여줄지, 이미지/그래프로 보여줄지, 파일로 저장할지

### 코딩 단계 — 언어 선택과 모듈화

코딩 단계에서 가장 먼저 해야 할 일은 **프로그래밍 언어를 선택** 하는 것입니다. 언어 선택은 단순히 취향의 문제가 아니라, **개발하려는 프로그램의 성격과 환경** 에 따라 달라집니다.

- **C 언어**: 절차적(순서대로 처리하는) 언어. 단순한 프로세스 순서로 동작하는 프로그램에 적합
- **C++ (C plus plus)**: C 언어를 확장하여 **객체지향(OOP)** 개념을 추가한 언어. 재사용성, 상속, 다형성 등이 필요할 때 적합
- **Java**: 웹 환경, 기업용 애플리케이션에 적합. 플랫폼 독립적
- **Python**: 배우기 쉽고 간결. 웹 개발, 데이터 분석 등에 널리 사용

언어를 선택했다면, 프로그램을 **모듈(Module)** 단위로 나누어 개발합니다. 모듈이란 **각각의 기능 단위** 를 의미합니다. 예를 들어 A 처리, B 처리, C 처리를 각각 별도의 모듈로 만들고, 이것들을 연결해서 하나의 완성된 프로그램을 만드는 방식입니다.

### 플로우차트 — 알고리즘을 시각적으로 표현하기

**플로우차트(Flowchart)** 는 프로그램의 수행 흐름을 시각적으로 표현하는 도구입니다. 알고리즘을 공부할 때 가장 기본적으로 배우는 표기법입니다.

기본 기호들:
- **타원(단말)**: 프로그램의 시작(START)과 끝(STOP)
- **평행사변형**: 입력(Input) 또는 출력(Output)
- **직사각형**: 처리(Process) — 계산이나 데이터 처리
- **마름모**: 조건 판단(Yes/No 분기)
- **화살표**: 흐름의 방향

플로우차트를 먼저 그리고 코딩을 시작하면, 논리적 흐름을 미리 파악할 수 있어 훨씬 효율적으로 프로그래밍할 수 있습니다.

### 구조적 프로그래밍(Structured Programming)

**구조적 프로그래밍** 이란, 프로그램의 흐름을 **입력 → 처리 → 출력** 의 구조로 명확하게 나누고, 복잡한 이동(goto문)을 최소화하여 읽기 쉽고 유지보수하기 쉬운 코드를 작성하는 방법론입니다.

#### goto문(고투문)의 문제점

과거에는 **goto문** 이라는 명령어를 자주 사용했습니다. goto문은 코드의 특정 위치로 강제로 이동하는 명령어입니다. 예를 들어 A를 처리하고 → B를 처리하다가 → 다시 A로 되돌아가고 → 또 C로 건너뛰고 하는 식입니다.

이렇게 하면 처음에는 편리해 보이지만, 코드가 점점 복잡해지면서 **마치 스파게티 면처럼 뒤엉킨 코드** 가 됩니다. 무한 루프(끝없이 돌다)에 빠지거나, 어디서 오류가 생겼는지 찾기 매우 어려워집니다.

#### 구조적 프로그래밍의 특징

- **단일 입구, 단일 출구**: 하나의 입력으로 받고, 하나의 출력으로 결과를 반환
- **제한된 제어 구조**: 순차(순서대로 실행), 선택(if/else 조건), 반복(for/while 반복) 세 가지만 사용
- **goto문 최소화**: 프로그램이 뒤죽박죽 이동하지 않도록 제한

**구조적 프로그래밍의 장점:**
- ✅ 코드를 처음 보는 사람도 이해하기 쉬움
- ✅ 유지보수(수정)가 쉬움
- ✅ 테스트하기 쉬움
- ✅ 오류(버그) 발생 시 찾아내기 쉬움
- ✅ 개발 비용이 줄어듦

**구조적 프로그래밍의 단점:**
- ❌ 성능이 상대적으로 낮을 수 있음
- ❌ 변수가 많이 필요할 수 있음
- ❌ 코드 중복이 발생할 수 있음
- ❌ 하위 모듈 호출이 많아져 구조가 복잡해질 수 있음

:::warning goto문 사용 주의
goto문이 완전히 나쁜 것은 아닙니다. 어떤 특수한 경우에는 반드시 필요하기도 합니다. 하지만 **가능하면 goto문 대신 구조화된 제어 구조(if, for, while)를 사용** 하는 것이 코드 품질을 높이는 데 훨씬 유리합니다.
:::

## 📚 프로그래밍 언어의 종류와 역사

### 저급 언어 vs 고급 언어

프로그래밍 언어는 크게 **저급 언어(Low-level Language)** 와 **고급 언어(High-level Language)** 로 나눌 수 있습니다.

- **저급 언어**: 기계(컴퓨터)가 이해하기 쉬운 언어. 사람이 이해하기 어려움
- **고급 언어**: 사람이 이해하기 쉬운 언어. 컴퓨터가 이해하려면 변환 과정이 필요

| 분류 | 종류 | 특징 |
|------|------|------|
| **저급 언어** | 기계어(0과 1) | 컴퓨터가 직접 이해. 사람이 읽기 매우 어려움 |
| **저급 언어** | 어셈블리어(Assembly) | 기계어와 1:1 대응되는 기호 언어 |
| **고급 언어 (컴파일)** | C, Java, C++ | 고급 명령어를 컴파일러가 기계어로 번역 |
| **고급 언어 (인터프리터)** | Basic, Visual Basic | 한 줄씩 즉시 실행. 컴파일 없이 동작 |

**컴파일러(Compiler) 방식** 은 전체 코드를 한 번에 번역해서 실행 파일을 만드는 방식이고, **인터프리터(Interpreter) 방식** 은 코드를 한 줄씩 읽어서 즉시 실행하는 방식입니다.

### 시대별 프로그래밍 언어의 역사

#### 1950~60년대: 컴퓨터의 탄생과 초기 언어

컴퓨터가 처음 등장했을 때는 **기계어와 어셈블리어** 만 존재했습니다. 컴퓨터를 직접 제어해야 했기 때문입니다. 이후 더 편리한 언어의 필요성이 생겨났습니다.

- **FORTRAN (포트란)**: 최초의 고급 프로그래밍 언어. IBM이 과학 계산과 수치 계산을 위해 개발
- **ALGOL (알골)**: 알고리즘을 위한 고급 언어. 이후 언어들의 기반을 만들어줬다고 평가받음
- **COBOL (코볼)**: 비즈니스(사무 처리)용 언어. 성적 관리, 재고 관리 프로그램 등에 사용
- **BASIC (베이직)**: 가장 쉬운 기본적인 인터프리터 언어. 프로그래밍 입문용으로 사용

#### 1970년대: C 언어의 등장

**C 언어** 는 데니스 리치(Dennis Ritchie)와 켄 톰슨(Ken Thompson)이 개발했습니다. 이식성(다른 환경에서도 쓸 수 있는 능력)과 효율성이 뛰어나 하드웨어, 네트워크 장비, 이미지 처리 등 다양한 분야에 사용됐습니다. 특히 유닉스(Unix) 운영체제가 개발되면서 널리 보급됐습니다.

C 언어는 오늘날에도 **모든 프로그래밍 언어의 기초** 로 여겨집니다. 개발자를 꿈꾼다면 C 언어는 반드시 공부해야 합니다.

#### 1980년대: 객체지향의 등장

- **C++ (C plus plus)**: C 언어에 **객체지향 프로그래밍(OOP)** 개념을 추가한 확장 언어. 재사용성, 상속, 다형성 등의 기능 제공
- **Pascal (파스칼)**: 교육 및 소프트웨어 개발에 주로 사용. 간단하고 구조적인 프로그래밍 언어

#### 1990년대: 웹의 시대

- **Java**: 웹 환경의 대표 언어. 객체지향 프로그래밍 기반. **플랫폼 독립성**(어떤 운영체제에서도 실행 가능)이 특징. 웹, 기업용 애플리케이션, 모바일 기기에 널리 사용
- **Python (파이썬)**: 간결하고 이해하기 쉬운 동적 언어. 웹 개발, 데이터 분석, 인공지능 등에 널리 사용. 현재 초급 개발자들이 가장 많이 배우는 언어
- **C# (C sharp)**: 마이크로소프트에서 개발. .NET 프레임워크 기반의 객체지향 언어
- **PHP**: 웹사이트 개발을 위한 스크립트 언어 (Personal Home Page Tools의 약자)
- **HTML**: 웹페이지를 만드는 마크업 언어. 우리가 보는 모든 홈페이지가 이것으로 만들어짐

#### 2000년대 이후: 다양화의 시대

- **JavaScript (자바스크립트)**: 웹 프론트엔드(화면)에서 동적인 기능을 만들 때 주로 사용. Java를 공부했다면 비교적 쉽게 배울 수 있음
- **Ruby (루비)**: 간결하고 직관적인 문법. 웹 개발과 스크립트 언어로 사용
- **Swift (스위프트)**: Apple(애플)에서 iOS/macOS 앱 개발을 위해 만든 언어
- **Kotlin (코틀린)**: Java 기반. Android 앱 개발에 주로 사용. Swift와 함께 모바일 앱 개발의 양대산맥
- **Go (고)**: 구글(Google)에서 만든 언어. 클라우드 기반 웹 환경에서 많이 사용
- **Rust (러스트)**: 고성능 언어. 메모리 관리, 다중 스레드 처리, 분산 환경에서 활용

:::info 어떤 언어를 먼저 배울까?
- 프로그래밍 기초를 다지려면: **C 언어**
- 웹 개발을 하고 싶다면: **HTML/CSS + JavaScript + Java 또는 Python**
- 데이터 분석/AI를 하고 싶다면: **Python**
- Android 앱을 만들고 싶다면: **Kotlin**
- iOS 앱을 만들고 싶다면: **Swift**
- 클라우드/백엔드 개발을 하고 싶다면: **Go 또는 Java**
:::

### 프로그래밍 언어 선택 시 고려 사항

좋은 개발자라면 단순히 자신이 아는 언어만 고집하지 않고, 상황에 맞는 언어를 선택할 줄 알아야 합니다. 언어 선택 시 고려해야 할 요소들은 다음과 같습니다.

1. **사용자 요구사항**: 웹 환경인지, 데스크탑인지, 모바일인지, 임베디드(가전제품 내부)인지에 따라 적합한 언어가 달라집니다.
2. **프로그래머의 지식**: 내가 잘 알고 있는 언어를 활용하는 것이 효율적입니다. Python을 한 번도 써보지 않은 사람이 갑자기 Python으로 개발하기는 어렵습니다.
3. **기존 프로젝트와의 연속성**: 이미 진행 중인 프로젝트에서 사용하는 언어를 이어서 쓰는 것이 유리합니다.
4. **컴파일러의 가용성과 품질**: 언어마다 컴파일러의 성능과 지원 환경이 다릅니다. 임베디드 분야에는 C 언어가, 웹 플랫폼 독립 환경에는 Java가 유리한 것처럼요.
5. **개발 도구 지원**: 요즘은 통합 개발 환경(IDE)을 사용해 코딩, 컴파일, 실행, 디버깅을 한 곳에서 처리합니다. 이 도구가 얼마나 잘 지원되는지도 중요합니다.
6. **호환성 및 이식성**: 개발한 프로그램이 다른 환경에서도 잘 동작하는지(이식성)를 고려해야 합니다.
7. **유지보수 용이성**: 나중에 수정하기 쉬운지도 언어 선택의 중요한 기준입니다.

## 🛡️ 프로그래밍의 기본 원칙과 주의사항

### 좋은 프로그래밍의 3대 원칙

아무리 복잡한 프로그래밍이라도 다음 세 가지 원칙을 지키면 훨씬 좋은 코드를 만들 수 있습니다.

#### 원칙 1: 설계 명세서를 완벽하게 반영하라

설계 명세서(설계 문서)는 집의 설계도면과 같습니다. 설계도면에 빠진 부분이 있으면 제대로 된 집을 지을 수 없듯이, **설계 명세서에 명시된 모든 요구사항이 코드에 완벽하게 반영** 되어야 합니다. 이 설계 명세서는 수많은 사용자 요구사항을 분석한 결과물이기 때문에 절대 무시해서는 안 됩니다.

#### 원칙 2: 원시 코드는 간단 명료하게

똑같은 기능을 구현하더라도, 코드가 **간단 명료** 할수록 좋은 프로그램입니다. 예를 들어, 1부터 100까지의 합을 구하는 프로그램을 100명의 개발자에게 짜달라고 하면 100가지 방법이 나올 것입니다. 그 중 가장 심플하고 누가 봐도 이해하기 쉬운 코드가 "잘 만든 코드"로 평가받습니다.

코드가 간단해야 하는 이유:
- **디버깅(오류 수정)이 쉬워집니다** — 오류가 어디서 발생했는지 찾기 쉬움
- **수정이 용이합니다** — 요구사항이 바뀌거나 기능이 추가될 때 빠르게 대응 가능
- **다른 개발자도 이해하기 쉽습니다** — 협업 시 생산성 향상
- **테스트하기 쉬워집니다** — 각 부분을 개별적으로 검증하기 편함

:::note 디버깅 vs 수정 — 다른 개념입니다
- **디버깅(Debugging)**: 코드에 오류(버그)가 발생했을 때, 그 오류를 찾아서 고치는 작업. 마치 기계에 생긴 고장을 수리하는 것
- **수정(Modification)**: 요구사항이 바뀌거나 기능이 추가될 때 코드를 변경하는 작업. 마치 기계의 기능을 업그레이드하는 것
이 두 가지는 완전히 다른 상황에서 이루어지는 작업입니다.
:::

#### 원칙 3: 디버깅과 테스트가 용이해야 한다

가전제품이나 자동차를 설계할 때 내부 구조를 복잡하게 만들면 고장 났을 때 수리하기 어렵습니다. 마찬가지로 프로그램도 **테스트하기 쉽고 오류를 찾기 쉬운 구조** 로 만들어야 합니다.

### 코딩 시 반드시 주의해야 할 오류들

아무리 문법에 맞게 코딩을 해도, 논리적 오류나 메모리 관련 오류는 컴파일러가 잡아주지 못합니다. 실제 실행 시에야 문제가 드러나는 경우가 많기 때문에, 개발자가 직접 주의해야 합니다.

#### ⚠️ 메모리 관련 오류

**메모리 누수(Memory Leak)**: 프로그램이 메모리를 사용한 후 해제(free)하지 않으면, 사용하지 않는 메모리가 계속 쌓입니다. 이렇게 되면 시스템 메모리가 가득 차서 컴퓨터가 느려지거나 다운됩니다. 스마트폰이나 컴퓨터가 오래 사용하면 점점 느려지는 현상이 이것과 관련이 있습니다.

```c
/* ❌ 잘못된 예시: 메모리를 할당했지만 해제하지 않음 */
char *ptr = (char *)malloc(10);  /* 10바이트 메모리 할당 */
/* 사용 후 free(ptr)를 호출하지 않음 → 메모리 누수 발생! */

/* ✅ 올바른 예시: 사용 후 반드시 메모리 해제 */
char *ptr = (char *)malloc(10);  /* 10바이트 메모리 할당 */
/* ... 메모리 사용 ... */
free(ptr);   /* 사용이 끝난 메모리를 반드시 해제해야 함 */
ptr = NULL;  /* 해제 후 포인터를 NULL로 초기화해서 이중 해제 방지 */
```

**이중 해제(Double Free)**: 이미 해제한 메모리를 또다시 해제하려고 하면 오류가 발생합니다.

```c
/* ❌ 잘못된 예시: 같은 메모리를 두 번 해제 */
char *ptr = (char *)malloc(10);  /* 메모리 할당 */
free(ptr);    /* 첫 번째 해제 */
free(ptr);    /* ❌ 두 번째 해제 → 오류 발생! */

/* ✅ 올바른 예시: 해제 후 NULL 체크 */
char *ptr = (char *)malloc(10);
free(ptr);
ptr = NULL;                       /* NULL로 초기화 */
if (ptr != NULL) {
    free(ptr);                    /* NULL이 아닐 때만 해제 */
}
```

#### ⚠️ 널 포인터(Null Pointer) 오류

**널(NULL)** 은 "값이 없음", "아무것도 가리키지 않음"을 의미합니다. 이미 해제된 메모리(NULL인 공간)에 접근하려고 하면 프로그램이 즉시 오류를 일으키고 종료됩니다.

```c
/* ❌ 잘못된 예시: NULL인 포인터를 역참조 */
char *ptr = NULL;          /* ptr은 아무것도 가리키지 않음 */
printf("%c\n", *ptr);      /* ❌ NULL을 역참조 → 프로그램 충돌! */

/* ✅ 올바른 예시: 사용 전 NULL 체크 필수 */
char *ptr = NULL;
if (ptr != NULL) {          /* NULL이 아닌지 먼저 확인 */
    printf("%c\n", *ptr);  /* 안전하게 접근 */
}
```

#### ⚠️ 배열 인덱스 범위 초과

배열은 **0부터 시작** 합니다. 크기가 80인 배열은 인덱스 0부터 79까지만 사용할 수 있습니다. 80을 사용하면 범위를 벗어난 접근이 되어 오류가 발생합니다.

```c
/* ❌ 잘못된 예시: 배열 범위 초과 접근 */
int arr[80];              /* 0 ~ 79까지, 총 80개 */
arr[80] = 100;            /* ❌ 인덱스 80은 존재하지 않음 → 범위 초과! */
arr[-1] = 50;             /* ❌ 음수 인덱스 → 오류! */

/* ✅ 올바른 예시: 배열 범위 안에서만 접근 */
int arr[80];
for (int i = 0; i < 80; i++) {  /* i < 80이므로 0 ~ 79까지만 접근 */
    arr[i] = i * 2;              /* 안전한 배열 접근 */
}
```

#### ⚠️ 하나 차이에 의한 오류 (Off-by-one Error)

`<`(보다 작다)와 `<=`(보다 작거나 같다)의 차이는 단 하나처럼 보이지만, 이 차이 하나가 완전히 다른 결과를 만들어냅니다. 컴파일러는 이것을 문법 오류로 인식하지 못하기 때문에, 실행 결과만 미묘하게 달라져서 찾아내기가 매우 어렵습니다.

```c
/* ❌ 잘못된 예시: 하나 차이 오류 */
int arr[5] = {10, 20, 30, 40, 50};
for (int i = 0; i <= 5; i++) {   /* ❌ i <= 5이면 i=5일 때 arr[5] 접근 → 범위 초과! */
    printf("%d\n", arr[i]);
}

/* ✅ 올바른 예시 */
int arr[5] = {10, 20, 30, 40, 50};
for (int i = 0; i < 5; i++) {    /* ✅ i < 5이면 i=0~4만 접근 → 안전 */
    printf("%d\n", arr[i]);
}
```

#### ⚠️ 동기화 오류 (Synchronization Error)

여러 개의 프로세스(실행 중인 프로그램)나 스레드(실행 흐름)가 동시에 **같은 데이터에 접근** 할 때 발생하는 오류입니다. 예를 들어, 프로세스 A는 변수를 "a에서 b로 수정"하고 있고, 프로세스 B는 그 변수를 동시에 "a로 읽으려고" 한다면 어떤 값이 맞는 것인지 충돌이 생깁니다.

이런 상황이 극단적으로 발전하면 **데드락(Deadlock)** 이 발생합니다. 데드락은 두 프로세스가 서로 상대방이 자원을 놓아주기를 기다리면서 둘 다 멈춰버리는 상황입니다.

#### ⚠️ 버퍼 오버플로우 (Buffer Overflow)

**버퍼(Buffer)** 는 데이터를 임시로 저장하는 메모리 공간입니다. 이 공간보다 더 큰 데이터를 억지로 밀어 넣으면 넘쳐흘러 인접한 메모리 공간을 덮어쓰게 됩니다. 이것이 **버퍼 오버플로우** 입니다.

이는 단순한 프로그램 오류를 넘어, **해킹 공격의 주요 수법** 으로도 활용됩니다. 공격자가 버퍼를 넘쳐흐르게 해서 인접 메모리에 악성 코드를 심고 실행할 수 있습니다. 이 때문에 버퍼 오버플로우 방어는 보안에서 매우 중요한 주제입니다.

### 프로그래밍 언어의 공학적/심리적 특성

좋은 프로그래밍 언어가 갖춰야 할 특성을 크게 두 가지로 나눌 수 있습니다.

**공학적 특성:**
- **코딩하기 쉬울 것**: 개발자가 빠르고 편리하게 코딩할 수 있어야 함
- **컴파일러 효율**: 컴파일 속도와 결과물의 품질이 좋아야 함
- **이식성(Portability)**: 다른 환경에서도 별 수정 없이 사용할 수 있어야 함
- **개발 도구 지원**: IDE 등 자동화된 개발 도구를 잘 지원해야 함
- **유지보수 용이성**: 수정과 업데이트가 쉬워야 함

**심리적 특성:**
- **일관성**: 표기법이나 문법 규칙이 언어 전체에서 일관되어야 함. C, Java, Python의 기본 문법이 비슷한 이유가 여기 있습니다
- **코드의 간결성**: 똑같은 처리를 하더라도 코드 양 자체를 줄일 수 있어야 함
- **직관성**: 코드를 읽었을 때 무엇을 하는지 직관적으로 이해할 수 있어야 함

### 주석과 문서화의 중요성

아무리 잘 짠 코드라도, **주석(Comment)** 이 없으면 나중에 다시 봤을 때 무슨 코드인지 이해하기 어렵습니다. 주석은 단순한 메모가 아니라, 개발자들 간의 **소통 수단** 입니다.

```python
# ✅ 좋은 주석의 예시 (Python)

# 학생 성적 평균을 계산하는 함수
# 입력: 성적 목록 (리스트 형태)
# 출력: 평균 점수 (소수점 2자리까지)
def calculate_average(scores):
    if not scores:               # 성적 목록이 비어 있으면 예외 처리
        return 0
    total = sum(scores)          # 모든 점수의 합계 계산
    count = len(scores)          # 성적 개수 파악
    average = total / count      # 평균 계산
    return round(average, 2)     # 소수점 2자리까지 반올림하여 반환

# ❌ 나쁜 주석의 예시 - 이렇게 하지 마세요
def calc(s):
    # 합 구함
    t = sum(s)
    # 나눔
    return t / len(s)
```

위 두 코드는 기능은 같지만, 유지보수성과 가독성에서 큰 차이가 납니다. 변수명도 의미 있는 이름을 사용하고(`scores`, `total`, `count` vs `s`, `t`), 각 단계마다 충분한 설명을 달아두는 것이 좋습니다.

문서화도 마찬가지입니다. 전체 코드에 대해 상세 설계, 파일명, 파일 크기, 성능 특성, 알고리즘 설명 등을 **문서로 만들어 관리** 하면 나중에 유지보수할 때 큰 도움이 됩니다.

:::danger 이것만은 절대 하지 마세요
- 메모리를 할당(malloc)한 후 해제(free)를 절대 빠뜨리지 마세요 → 메모리 누수
- 배열 크기를 벗어난 인덱스에 접근하지 마세요 → 예측 불가능한 오류 발생
- NULL 체크 없이 포인터를 역참조하지 마세요 → 프로그램 즉시 충돌
- 어떤 수를 0으로 나누는 코드를 작성하지 마세요 → 수학적 오류
- 버퍼 크기를 넘는 입력을 허용하지 마세요 → 보안 취약점 발생
:::

## 🎯 프로그래머로 성장하는 길

### 경력보다 중요한 것은 깊이

프로그래머의 몸값(전문성과 연봉)은 단순히 경력 연수에 비례하지 않습니다. 10년 경력이 있어도 단순 반복 작업만 해왔다면 성장이 제한적일 수 있습니다. 반면, 1~2년 차라도 다음과 같은 질문을 스스로에게 끊임없이 던지는 개발자는 빠르게 성장합니다.

- **"내가 짠 이 코드를 더 간단하게 만들 수는 없을까?"**
- **"이 코드는 다른 사람이 봐도 쉽게 이해할 수 있을까?"**
- **"오류가 생겼을 때 찾기 쉬운 구조인가?"**
- **"설계 명세서의 모든 요구사항이 빠짐없이 반영됐는가?"**
- **"메모리를 효율적으로 사용하고 있는가?"**

이런 질문을 습관화하면, 코드 작성 능력이 알게 모르게 조금씩 향상되고, 그것이 쌓여서 진정한 전문가로 성장하게 됩니다.

### 소프트웨어 공학의 관점에서 프로그래밍 보기

프로그래밍을 배울 때 단순히 "이 언어의 문법은 이렇다"를 외우는 것에 그치지 말고, **소프트웨어 공학적 관점** 에서 바라봐야 합니다. 소프트웨어 공학은 Java나 Python을 코딩하는 방법을 가르치는 것이 아닙니다. **소프트웨어 전반에 대한 이론적 기초** 를 다지는 것입니다.

나중에 어떤 언어를 배우든 — Java, Python, C++, HTML, JavaScript, PHP — 이 이론적 기반이 탄탄하다면 새로운 언어를 훨씬 빠르게, 더 깊이 있게 습득할 수 있습니다.

## ✅ 핵심 정리

- **컴퓨터 언어**는 컴퓨터와 의사소통하기 위한 약속된 규칙의 언어 체계 전체를 의미하며, **프로그래밍 언어**는 그 중 프로그램을 작성하기 위한 언어다.
- **C 언어**(개발 도구)와 **C 프로그램**(C 언어로 만든 소프트웨어)은 다른 개념이다.
- 소스 코드는 **컴파일러 → 목적 프로그램 → 링커 → 실행 파일 → 로더 → 실행**의 순서로 처리된다.
- **컴파일러**는 문법 오류를 체크하고 기계어로 변환하며, **링커**는 조각들을 실행 파일로 조립하고, **로더**는 메모리에 올려 실행시킨다.
- **코더**는 주어진 대로 코드를 입력하는 사람, **프로그래머**는 논리적으로 분석·설계·구현하는 사람이다.
- 좋은 프로그래밍의 3대 원칙: **설계 명세서 완벽 반영**, **코드 간단 명료**, **디버깅·테스트 용이**.
- **구조적 프로그래밍**은 단일 입구·단일 출구, 순차·선택·반복의 제한된 제어 구조를 사용하여 goto문을 최소화하는 방법론이다.
- 프로그래밍 언어는 **저급 언어(기계어, 어셈블리어)** 와 **고급 언어(컴파일러 방식, 인터프리터 방식)** 로 나뉜다.
- 언어 선택 시 **사용자 요구사항, 프로그래머 지식, 기존 프로젝트, 컴파일러 품질, 개발 도구 지원** 등을 고려해야 한다.
- 코딩 시 **메모리 누수, 이중 해제, 널 포인터, 배열 범위 초과, 하나 차이 오류, 동기화 오류, 버퍼 오버플로우** 를 반드시 주의해야 한다.
- **주석과 문서화**는 협업과 유지보수를 위한 필수 요소이며, 의미 있는 변수명 사용이 코드 가독성을 높인다.
- 프로그래머의 성장은 경력보다 **지속적인 자기 개발과 깊이 있는 고민** 에서 비롯된다.
