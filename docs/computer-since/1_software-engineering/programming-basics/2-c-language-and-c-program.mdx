# C 언어와 C 프로그램의 개념 및 컴파일 과정 🖥️

프로그래밍을 처음 시작하는 분들이라면 "C 언어", "C 프로그램", "컴파일" 같은 단어를 들었을 때 막막하게 느껴질 수 있습니다. 이번 문서에서는 컴퓨터 언어와 프로그래밍 언어의 차이부터 시작해서, C 언어가 무엇인지, 컴파일이 어떤 과정을 거쳐 실행 파일을 만들어 내는지, 그리고 좋은 프로그래머가 되려면 어떤 원칙을 지켜야 하는지까지 아주 자세하게 풀어서 설명하겠습니다. 처음 접하는 분도 충분히 이해할 수 있도록 비유와 예시를 풍부하게 담았으니 천천히 따라오세요.

---

## 1️⃣ 컴퓨터 언어 vs 프로그래밍 언어 — 비슷해 보이지만 다릅니다

### 🤔 왜 이 구분이 필요한가?

처음 공부를 시작하면 "컴퓨터 언어"와 "프로그래밍 언어"를 같은 말처럼 쓰는 경우가 많습니다. 실제로 일상적인 대화에서는 거의 구분 없이 사용하기도 합니다. 하지만 두 개념을 정확히 이해하고 있어야 이후에 배울 컴파일, 소스코드, 실행 파일 등의 개념이 훨씬 명확하게 자리를 잡습니다. 마치 "자동차"와 "승용차"처럼, 큰 범주와 세부 범주의 차이라고 생각하면 됩니다.

### 📖 컴퓨터 언어란?

**컴퓨터 언어(Computer Language)** 는 컴퓨터라는 시스템에서 동작하는 소프트웨어를 만들기 위해, 컴퓨터와 사람 사이에 **서로 약속된 규칙으로 이루어진 언어** 전체를 가리킵니다. 쉽게 말하면, 컴퓨터와 대화하기 위해 사용하는 모든 언어 체계를 통칭하는 넓은 개념입니다.

예를 들어 우리가 외국인과 대화하려면 그 나라의 언어를 배워야 하듯이, 컴퓨터와 소통하려면 컴퓨터가 이해하는 언어 체계를 알아야 합니다. 이 전체 체계를 컴퓨터 언어라고 부릅니다.

### 📖 프로그래밍 언어란?

**프로그래밍 언어(Programming Language)** 는 컴퓨터 언어의 한 종류로, 특히 **프로그램(소프트웨어)을 작성하기 위해 사용하는 언어** 를 말합니다. 우리가 흔히 말하는 C, Java, Python, JavaScript 같은 것들이 바로 프로그래밍 언어입니다.

> 💡 **정리하면**: 컴퓨터 언어는 컴퓨터와 관련된 모든 언어 체계를 의미하는 **큰 범주**이고, 프로그래밍 언어는 그 안에서 **소프트웨어를 개발하는 데 특화된 언어**입니다.

| 구분 | 컴퓨터 언어 | 프로그래밍 언어 |
|------|-------------|-----------------|
| **범위** | 컴퓨터와 관련된 모든 언어 체계 | 소프트웨어(프로그램)를 만들기 위한 언어 |
| **목적** | 컴퓨터 시스템과 의사소통 | 프로그램 코딩 및 개발 |
| **예시** | 기계어, 어셈블리어, 프로그래밍 언어 포함 | C, Java, Python, JavaScript 등 |

### 📌 프로그래밍 언어의 핵심 — 문법 규칙(Syntax)

프로그래밍 언어에는 반드시 지켜야 할 **문법(Syntax, 신택스)** 이 있습니다. 우리가 한국어로 글을 쓸 때 맞춤법과 문장 부호를 지켜야 하듯, 프로그래밍 언어도 정해진 규칙 안에서만 코드를 작성해야 합니다. 이 규칙을 어기면 컴퓨터가 이해하지 못하고 오류(에러)가 발생합니다.

:::info 문법 오류(Syntax Error)란?
프로그래밍 언어의 약속된 규칙을 지키지 않았을 때 발생하는 오류입니다. 예를 들어 한국어로 "나는 학교 갔다 을"이라고 쓰면 문법이 틀리듯, 프로그래밍에서도 규칙에 맞지 않는 코드를 작성하면 컴파일러가 "문법 오류!"라고 알려줍니다.
:::

## 2️⃣ C 언어와 C 프로그램 — 도구와 결과물의 차이 🔧

### 🤔 왜 이 둘을 구분해야 하는가?

많은 입문자들이 "C 언어"와 "C 프로그램"을 혼용해서 씁니다. 하지만 이 두 가지는 서로 다른 개념입니다. 이 차이를 명확히 알아두면 이후 개발 환경을 이해하는 데 큰 도움이 됩니다. 마치 "요리 도구(칼, 냄비 등)"와 "요리된 음식"이 다른 것처럼, 언어와 그 언어로 만들어진 프로그램은 다릅니다.

### 🔑 C 언어(C Language)

**C 언어** 는 C 프로그래밍을 가능하게 해주는 언어 체계 자체, 그리고 그것을 처리해주는 **컴파일러(compiler, 번역기) 소프트웨어** 를 포함한 개념입니다. 즉, C 언어는 개발자가 코드를 작성하고 실행 파일을 만들어 내기까지의 전체 언어 환경을 의미합니다.

C 언어는 모든 프로그래밍의 **가장 기본이 되는 언어** 입니다. 개발자를 꿈꾸거나 IT 전공을 공부하는 분이라면 반드시 C 언어를 먼저 공부해야 하는 이유가 여기에 있습니다. C 언어를 이해하면 다른 언어(Java, Python, C++ 등)를 배울 때 기초 개념이 이미 잡혀 있어 훨씬 수월합니다.

### 🖥️ C 프로그램(C Program)

**C 프로그램** 은 C 언어를 사용해서 실제로 **개발된 소프트웨어(결과물)** 를 말합니다. 예를 들어, C 언어를 이용해서 재고 관리 시스템을 만들었다면 그 완성된 소프트웨어가 바로 C 프로그램입니다.

> 📦 **비유**: C 언어는 "건축 도구와 설계 방법론"이고, C 프로그램은 그 도구와 방법론으로 실제로 완성된 "건물"입니다.

### 💻 간단한 C 프로그램 예시

아래는 두 수를 더해서 출력하는 아주 간단한 C 프로그램입니다. C 언어로 이렇게 코딩(coding, 코드 작성)을 하면 하나의 C 프로그램이 탄생합니다.

```c
/* 표준 입출력 헤더 파일을 불러옵니다 */
/* 이 파일이 있어야 printf 함수를 사용할 수 있습니다 */
#include <stdio.h>

/* 프로그램의 시작점, main 함수입니다 */
int main() {
    int a = 10;   /* 변수 a에 10을 저장합니다 */
    int b = 20;   /* 변수 b에 20을 저장합니다 */
    int sum;      /* 합계를 저장할 변수 sum을 선언합니다 */

    sum = a + b;  /* a와 b를 더한 값을 sum에 저장합니다 */

    /* 결과를 화면에 출력합니다 (결과: 30) */
    printf("합계: %d\n", sum);

    return 0; /* 프로그램이 정상적으로 끝났음을 알립니다 */
}
```

- `#include <stdio.h>` : 표준 입출력(Standard Input/Output) 헤더 파일을 불러옵니다. `printf` 같은 출력 함수를 쓰려면 반드시 필요합니다.
- `int main()` : 프로그램이 시작되는 지점입니다. C 언어는 항상 `main` 함수에서 실행이 시작됩니다.
- `int a = 10;` : 정수형 변수 `a`를 선언하고 10을 저장합니다.
- `sum = a + b;` : `a`와 `b`를 더한 값을 `sum`에 저장합니다. 결과는 30입니다.
- `printf(...)` : 화면에 결과를 출력합니다.
- `return 0;` : 프로그램이 정상 종료됨을 운영체제에 알립니다.

:::tip C 언어가 기본인 이유
C 언어는 변수, 조건문, 반복문, 함수, 배열, 포인터 등 모든 프로그래밍의 핵심 개념을 담고 있습니다. 이것들을 C 언어로 익혀 두면 Java, Python 등 다른 언어를 배울 때 "아, 이건 C에서 배운 그것과 비슷하구나"라고 빠르게 이해할 수 있습니다.
:::

## 3️⃣ 컴파일 과정 — 소스코드가 실행 파일이 되기까지 🏗️

### 🤔 왜 컴파일이 필요한가?

우리가 C 언어로 작성한 코드는 사람이 읽고 이해할 수 있는 언어입니다. 하지만 컴퓨터(CPU)는 사람의 언어를 직접 이해하지 못합니다. 컴퓨터는 오직 **0과 1로 이루어진 기계어(Machine Language)** 만 이해할 수 있습니다. 따라서 우리가 작성한 코드를 컴퓨터가 이해할 수 있는 기계어로 번역하는 과정이 필요합니다. 이 번역 과정을 **컴파일(Compile)** 이라고 합니다.

마치 한국어로 쓴 편지를 일본어를 모르는 일본인에게 보내려면 번역가가 일본어로 번역해 줘야 하듯, 우리의 코드를 컴퓨터가 이해할 수 있도록 번역해주는 것이 컴파일입니다.

### 🔄 컴파일 전체 과정 단계별 설명

컴파일은 크게 다음 4단계를 거칩니다.

```
[원시 프로그램 (소스 코드)]
         ↓
   [컴파일러(Compiler)]  ← 문법 오류 검사 및 기계어 번역
         ↓
   [목적 프로그램 (Object Program)]
         ↓
     [링커(Linker)]      ← 실행 가능한 형태로 연결
         ↓
   [실행 파일 (Executable File, .exe)]
         ↓
     [로더(Loader)]      ← 메모리에 올려서 실제 실행
         ↓
     [프로그램 실행 결과]
```

#### 1단계 — 원시 프로그램(Source Program)

개발자가 C 언어나 Java, Python 등의 프로그래밍 언어로 작성한 **소스 코드** 파일입니다. 확장자로는 `.c`(C 언어), `.java`(Java), `.py`(Python) 등이 있습니다. 이 단계는 오직 개발자(사람)만 읽고 이해할 수 있습니다.

#### 2단계 — 컴파일러(Compiler)

**컴파일러** 는 원시 프로그램을 받아서 두 가지 일을 합니다.

첫 번째로, **문법 오류(Syntax Error)를 검사** 합니다. 코드가 해당 언어의 문법 규칙을 올바르게 따르고 있는지 꼼꼼히 확인합니다. 문법 오류가 발견되면 "여기 문법이 틀렸어!"라고 오류 메시지를 개발자에게 알려주고 번역을 중단합니다.

두 번째로, 문법 오류가 없다면 소스 코드를 **컴퓨터가 이해할 수 있는 기계어로 번역** 합니다. 이렇게 번역된 결과물을 **목적 프로그램(Object Program)** 이라고 합니다.

```
[C 소스 코드 - 사람이 이해하는 언어]
int a = 10;
printf("hello");

         ↓  컴파일러가 번역

[목적 프로그램 - 컴퓨터가 이해하는 기계어]
01001000 10000011 11101100 ...
```

#### 3단계 — 링커(Linker)

목적 프로그램이 만들어졌다고 바로 실행할 수 있는 것은 아닙니다. 프로그램은 보통 여러 개의 파일로 나뉘어 개발되거나, 외부 라이브러리(미리 만들어진 기능 모음)를 사용하는 경우가 많습니다. **링커** 는 이렇게 분산되어 있는 목적 프로그램들과 라이브러리를 **하나의 실행 가능한 파일로 연결(Link)** 해주는 역할을 합니다.

마치 건물의 각 층(목적 프로그램)을 연결해서 하나의 완성된 건물(실행 파일)로 만드는 것과 같습니다.

#### 4단계 — 로더(Loader)

**로더** 는 링커가 만들어낸 실행 파일을 **컴퓨터의 메모리(RAM)에 올려서 실제로 실행** 시키는 역할을 합니다. 우리가 파일 탐색기에서 `.exe` 파일을 더블클릭하면 로더가 작동해서 프로그램이 메모리에 올라가고 실행되는 것입니다.

:::note 개발자가 신경 쓸 부분
이 전체 과정 중에서 개발자가 직접 신경 써야 하는 부분은 **원시 프로그램(소스 코드)** 입니다. 소스 코드를 문법에 맞게 올바르게 작성해 놓으면, 나머지 컴파일 → 링크 → 로드 과정은 개발 도구(IDE)가 자동으로 처리해 줍니다. 요즘은 하나의 통합 개발 환경(IDE, Integrated Development Environment) 안에서 이 모든 과정이 버튼 하나로 처리됩니다.
:::

## 4️⃣ 프로그래밍이란 무엇인가 — 코더와 프로그래머의 차이 👨‍💻

### 🤔 단순히 코드를 짜는 것이 전부일까?

"프로그래밍(Programming)"을 단순히 "코드를 타이핑하는 것"이라고 생각하는 분들이 많습니다. 하지만 진정한 프로그래밍은 그보다 훨씬 넓은 개념입니다. 이 차이를 이해하는 것이 여러분이 앞으로 어떤 개발자가 될지를 결정하는 중요한 첫걸음입니다.

### 👤 코더(Coder)와 프로그래머(Programmer)의 차이

**코더(Coder)** 는 누군가가 설계해 준 알고리즘(문제 해결 방법)을 받아서 그대로 코드로 옮기는 사람입니다. 또는 이미 만들어진 프로그램을 가져다가 단순히 유지보수하는 작업을 하는 사람을 말합니다. 기술적으로 코드를 작성하는 능력은 있지만, 전체 개발 방향을 설계하고 전략을 짜는 역할은 하지 않습니다.

**프로그래머(Programmer)** 는 단순히 코드를 작성하는 것을 넘어서, 논리적이고 치밀한 사고를 바탕으로 다음과 같은 일을 합니다.

- **사용자의 요구사항 분석** : 이 소프트웨어를 쓸 사람들이 무엇을 필요로 하는지 파악합니다
- **체계적인 개발 계획 수립** : 개발 기간, 인력, 비용 등을 계획합니다
- **설계** : 입력(Input), 처리(Process), 출력(Output)을 어떻게 구성할지 설계합니다
- **코딩** : 설계를 바탕으로 실제 코드를 작성합니다
- **테스트 및 유지보수** : 오류를 찾고 수정하며 기능을 개선합니다

> 🏠 **비유**: 코더는 도면을 보고 벽돌을 쌓는 사람이고, 프로그래머는 건물 전체를 설계하고 공사를 총괄하는 건축가입니다. 둘 다 중요하지만 역할과 책임이 다릅니다.

### 📐 프로그래밍의 전체 흐름

프로그래밍은 다음과 같은 흐름으로 진행됩니다.

```
[사용자 요구사항 분석]
         ↓
[설계 명세서(설계도) 작성]
   - 입력(Input): 키보드? 파일? 음성?
   - 처리(Process): 어떤 연산을 할 것인가?
   - 출력(Output): 텍스트? 그래프? 파일?
         ↓
[프로그래밍 언어 선택]
   - 절차적: C 언어
   - 객체지향: Java, C++
   - 웹: JavaScript, PHP
         ↓
[모듈(Module) 단위 코딩]
   - A 기능 모듈
   - B 기능 모듈
   - C 기능 모듈
         ↓
[컴파일 → 링크 → 실행]
         ↓
[테스트 및 디버깅(오류 수정)]
```

### 🔍 절차적 프로그래밍 vs 객체지향 프로그래밍

프로그래밍 언어를 선택할 때 개발하려는 소프트웨어의 성격에 따라 방식을 선택해야 합니다.

| 구분 | 절차적 프로그래밍 | 객체지향 프로그래밍 |
|------|------------------|---------------------|
| **대표 언어** | C 언어 | Java, C++ |
| **특징** | 정해진 순서대로 처리 | 객체(Object) 단위로 설계 |
| **장점** | 단순하고 빠름 | 재사용성, 유지보수성이 좋음 |
| **적합한 상황** | 단순한 처리 흐름, 임베디드 시스템 | 대규모 애플리케이션, 웹 서비스 |

:::tip 프로그래밍 언어 선택 기준
언어를 선택할 때는 다음을 고려해야 합니다.
- **사용자 환경**: 웹인지, 모바일인지, 데스크탑인지
- **개발자의 지식**: 내가 잘 아는 언어가 유리합니다
- **기존 프로젝트**: 이미 진행 중인 프로젝트의 언어를 그대로 유지하는 것이 효율적입니다
- **컴파일러의 품질**: 임베디드(세탁기, TV 등 내장 시스템)에는 C 언어, 플랫폼 독립적인 웹 환경에는 Java 등
- **개발 도구 지원**: 통합 개발 환경(IDE)이 잘 지원되는 언어가 개발 효율성을 높여 줍니다
:::

## 5️⃣ 좋은 프로그래밍의 기본 원칙 ✅

### 🤔 왜 원칙이 필요한가?

"1부터 100까지 합을 구하는 프로그램을 만들어라"고 하면 100명의 개발자가 100가지 방식으로 코드를 짤 수 있습니다. 결과는 모두 같을 수 있지만 **코드의 품질은 천차만별** 입니다. 어떤 코드가 더 좋은 코드일까요? 단순히 결과만 맞으면 되는 걸까요? 좋은 코드는 특정 원칙을 따릅니다.

### 📜 프로그래밍의 기본 원칙 3가지

#### 원칙 1: 설계 명세서를 완벽하게 반영할 것

**설계 명세서(Design Specification)** 는 프로그램을 만들기 전에 작성하는 설계도와 같습니다. 사용자들의 요구사항을 꼼꼼히 분석해서 만들어진 문서입니다. 집을 지을 때 설계도에 빠진 부분이 있으면 완성된 집에 문제가 생기듯, 프로그래밍도 설계 명세서의 내용이 빠짐없이 코드에 반영되어야 합니다. "아, 이 기능은 나중에 추가하지 뭐"라는 생각은 매우 위험합니다.

#### 원칙 2: 원시 코드는 간단하고 명료하게 작성할 것

코드가 간단하고 명료할수록 다음과 같은 이점이 생깁니다.

- **디버깅(Debugging, 오류 수정)이 쉬워집니다**: 어디서 오류가 발생했는지 빠르게 찾을 수 있습니다
- **수정(Modification)이 쉬워집니다**: 요구사항이 바뀌었을 때 코드를 쉽게 고칠 수 있습니다
- **테스트(Testing)가 쉬워집니다**: 각 기능이 제대로 동작하는지 검증하기 쉽습니다
- **다른 개발자도 이해할 수 있습니다**: 팀 협업에서 필수입니다

> 🔧 **비유**: 자동차나 스마트폰처럼 잘 만들어진 제품은 내부 구조가 복잡해 보여도, 부품들은 군더더기 없이 심플하게 설계되어 있습니다. 덕분에 고장이 났을 때 어디를 수리해야 하는지 쉽게 찾을 수 있는 것처럼, 코드도 심플하게 만들어야 유지보수가 쉬워집니다.

:::warning 디버깅(Debugging)과 수정(Modification)의 차이
- **디버깅**: 코드에 **오류(버그, Bug)가 발생했을 때** 그 오류를 찾아서 고치는 작업입니다
- **수정(Modification)**: 기능이 추가되거나 요구사항이 변경되었을 때 코드를 바꾸는 작업입니다

둘 다 "코드를 고친다"는 공통점이 있지만, 발생 원인이 다릅니다. 혼동하지 마세요!
:::

#### 원칙 3: 테스트하기 쉽게 만들 것

프로그램이 완성된 후에는 반드시 **테스트(Testing)** 를 거쳐야 합니다. 각 기능이 설계 명세서대로 동작하는지, 예외 상황에서도 오류 없이 처리되는지 꼼꼼히 확인해야 합니다. 코드가 복잡하게 얽혀 있으면 테스트 자체가 어려워집니다.

### 💡 코더가 아닌 프로그래머로 성장하려면

단순히 주어진 코드를 작업하는 것에서 벗어나, 항상 스스로에게 다음 질문을 던져 보세요.

- "내가 짠 이 코드를 더 간단하게 만들 수 없을까?"
- "이 코드에서 오류가 발생하면 쉽게 찾을 수 있을까?"
- "설계 명세서의 요구사항이 모두 반영되어 있는가?"
- "다른 개발자가 이 코드를 봤을 때 바로 이해할 수 있을까?"

이런 고민을 끊임없이 하고 코드를 개선하는 습관이 쌓이면, 자연스럽게 개발 역량이 올라갑니다. **경력이 쌓인다고 자동으로 몸값이 오르는 것이 아니라, 얼마나 깊이 고민하고 고난이도의 문제를 해결해봤느냐** 가 진짜 실력을 만들어 줍니다.

## 6️⃣ 구조적 프로그래밍과 흐름도(Flow Chart) 📊

### 🤔 구조적 프로그래밍이 왜 필요한가?

초기 프로그래밍에서는 **goto문(고투문)** 이라는 것을 사용했습니다. goto문은 코드 실행 중에 "이 줄에서 저 줄로 바로 점프!"를 명령하는 것입니다. 처음에는 편리해 보이지만, goto문이 많아질수록 코드가 이리저리 얽혀서 스파게티처럼 복잡해집니다. 어디서 어디로 흐르는지 파악하기 어렵고, 최악의 경우 무한 루프(프로그램이 끝나지 않고 계속 돌기만 하는 상태)에 빠질 수 있습니다.

이런 문제를 해결하기 위해 **구조적 프로그래밍(Structured Programming)** 이 등장했습니다.

### 📋 플로우 차트(Flow Chart, 흐름도)

**플로우 차트** 는 기본 기호들을 사용해서 프로그램의 실행 흐름을 시각적으로 나타내는 도구입니다. 알고리즘을 공부할 때 가장 기본적으로 익혀야 하는 표기법입니다.

| 기호 모양 | 의미 |
|-----------|------|
| 타원(둥근 직사각형) | 시작(Start) / 끝(End) |
| 직사각형 | 처리(Process): 연산, 대입 등 |
| 마름모 | 판단(Decision): 조건 비교 |
| 평행사변형 | 입력(Input) / 출력(Output) |
| 화살표 | 흐름 방향 |

### 🏗️ 구조적 프로그래밍의 핵심 특징

구조적 프로그래밍은 다음 원칙을 따릅니다.

1. **단일 입구(Single Entry)**: 프로그램은 하나의 시작점을 가져야 합니다
2. **단일 출구(Single Exit)**: 프로그램은 하나의 끝점을 가져야 합니다
3. **제한된 제어 구조 사용**: 순차(Sequential), 선택(Selection), 반복(Iteration) 세 가지 구조만 사용합니다
4. **goto문 최소화**: 앞뒤로 점프하는 goto문 사용을 최대한 줄입니다

```c
/* ✅ 구조적 프로그래밍 예시 - 순차, 선택, 반복을 사용 */
#include <stdio.h>

int main() {
    int i;      /* 반복에 사용할 변수 */
    int sum = 0; /* 합계를 저장할 변수, 0으로 초기화 */

    /* 반복 구조: 1부터 100까지 순서대로 처리합니다 */
    for (i = 1; i <= 100; i++) {
        sum = sum + i; /* sum에 i 값을 계속 더합니다 */
    }

    /* 출력 구조: 결과를 화면에 보여줍니다 */
    printf("1부터 100까지의 합: %d\n", sum);

    return 0; /* 프로그램 정상 종료 */
}
```

```c
/* ❌ goto문을 남용한 비구조적 프로그래밍 예시 */
#include <stdio.h>

int main() {
    int i = 1;
    int sum = 0;

start: /* 레이블(점프 목적지) 설정 */
    sum = sum + i;
    i++;
    if (i <= 100) goto start; /* goto로 위로 점프 - 코드 흐름이 복잡해집니다 */

    printf("합계: %d\n", sum);
    return 0;
}
```

### ✅ 구조적 프로그래밍의 장단점

| 구분 | 내용 |
|------|------|
| **장점** | 코드 이해가 쉬움, 유지보수 용이, 테스트하기 쉬움, 버그 찾기 편함 |
| **단점** | 성능이 다소 떨어질 수 있음, 변수가 많이 필요, 코드 중복 발생 가능 |

:::warning goto문에 대한 올바른 이해
goto문이 완전히 나쁜 것은 아닙니다. 특수한 상황에서는 goto문이 필요할 수도 있습니다. 다만 **가능하면 최소한으로 사용** 하고, 구조화된 방식으로 프로그래밍하는 것이 원칙입니다.
:::

## 7️⃣ 프로그래밍 언어의 분류와 역사 📚

### 🤔 왜 언어가 이렇게 많은가?

세상에는 수십 가지의 프로그래밍 언어가 존재합니다. 이유는 단순합니다. 각 언어마다 **특화된 분야** 가 있고, 시대의 기술 발전에 따라 새로운 언어들이 계속 탄생했기 때문입니다. 도구가 다양한 이유와 같습니다. 못을 박을 때는 망치, 나사를 조일 때는 드라이버를 쓰듯, 상황에 맞는 언어를 선택하는 것이 중요합니다.

### 📊 저급 언어 vs 고급 언어

프로그래밍 언어는 크게 **저급 언어(Low-Level Language)** 와 **고급 언어(High-Level Language)** 로 나뉩니다.

| 구분 | 저급 언어 | 고급 언어 |
|------|-----------|-----------|
| **이해 주체** | 기계(컴퓨터)가 이해하기 쉬운 언어 | 사람(개발자)이 이해하기 쉬운 언어 |
| **종류** | 기계어(0과 1), 어셈블리어 | C, Java, Python, JavaScript 등 |
| **특징** | 하드웨어 직접 제어 가능 | 문법이 인간 언어에 가까움 |

고급 언어는 다시 두 종류로 나뉩니다.

- **컴파일러 언어**: 전체 소스 코드를 한꺼번에 기계어로 번역한 후 실행합니다. C, Java, C++ 등이 해당됩니다.
- **인터프리터 언어(Interpreter Language, 통역 언어)**: 코드를 한 줄씩 읽어서 바로바로 실행합니다. Basic, 초기 Python 등이 해당됩니다. 마치 동시통역사가 한 문장씩 바로 번역해주는 것과 같습니다.

### 🕰️ 프로그래밍 언어의 역사 (시대별 흐름)

#### 1950~60년대 — 컴퓨터의 태동기

초기 컴퓨터는 기계어와 어셈블리어로 직접 제어해야 했습니다. 이후 과학 계산과 수치 처리를 위한 최초의 고급 프로그래밍 언어가 등장합니다.

- **포트란(FORTRAN)**: IBM이 과학 계산을 위해 개발한 최초의 고급 프로그래밍 언어입니다
- **코볼(COBOL)**: 기업의 사무 처리(재고 관리, 성적 관리 등)를 위해 개발된 언어입니다
- **알골(ALGOL)**: 알고리즘 구현을 위한 언어로, 이후 프로그래밍 언어들의 문법 기반을 만들었습니다
- **베이직(BASIC)**: 가장 배우기 쉬운 인터프리터 언어로, 프로그래밍 교육에 많이 활용되었습니다

#### 1970년대 — C 언어의 등장

- **C 언어**: 데니스 리치(Dennis Ritchie)와 켄 톰슨(Ken Thompson)이 개발했습니다. 이식성(다른 환경에서도 동작하는 능력)과 효율성이 뛰어나 하드웨어 제어, 네트워크 장비, 운영체제(유닉스 UNIX) 개발에 널리 사용되었으며, 현재도 임베디드 시스템(세탁기, TV 내장 프로그램)에서 핵심 언어로 사용됩니다

#### 1980년대 — 객체지향의 시작

- **C++(씨플플)**: C 언어에 **객체지향(Object-Oriented) 개념** 을 추가한 확장 언어입니다. C 언어가 절차적(순서대로 처리)이라면 C++는 객체 단위로 설계하고 재사용하고 상속할 수 있습니다
- **파스칼(Pascal)**: 교육용으로 많이 사용된 간결하고 구조적인 언어입니다

#### 1990년대 — 인터넷과 웹의 시대

- **Java(자바)**: 객체지향 언어로, 가장 큰 특징은 **플랫폼 독립성** 입니다. "한 번 작성하면 어디서나 실행된다(Write Once, Run Anywhere)"는 슬로건처럼, 운영체제에 상관없이 동작합니다. 웹, 기업용 애플리케이션, 안드로이드 앱 개발에 널리 사용됩니다
- **Python(파이썬)**: 문법이 간결하고 배우기 쉬워 입문자에게 인기가 높습니다. 웹 개발, 데이터 분석, 인공지능 분야에서 활발하게 사용됩니다
- **PHP**: 웹 서버에서 동작하는 스크립트 언어로, 동적인 웹 페이지를 만드는 데 사용됩니다
- **HTML**: 웹 페이지의 구조를 만드는 마크업 언어입니다. 엄밀히는 프로그래밍 언어는 아니지만 웹 개발의 기본입니다
- **C#(씨샵)**: 마이크로소프트(Microsoft)가 .NET 프레임워크 기반으로 개발한 객체지향 언어입니다

#### 2000년대 이후 — 다양성의 시대

- **JavaScript(자바스크립트)**: 웹 페이지에 동적인 효과를 주는 언어로, 현재는 프론트엔드(Front-end, 사용자가 보는 화면)의 핵심 언어입니다. Java와 이름이 비슷하지만 전혀 다른 언어입니다
- **Ruby(루비)**: 간결하고 직관적인 문법을 가진 동적 언어로, 웹 개발에 사용됩니다
- **Swift(스위프트)**: 애플(Apple)이 iOS와 macOS 앱 개발을 위해 만든 언어입니다
- **Kotlin(코틀린)**: 안드로이드 앱 개발에 주로 사용되며, Java와 호환됩니다
- **Go(고)**: Google이 만든 언어로, 클라우드 기반의 웹 서버 환경에서 많이 사용됩니다
- **Rust(러스트)**: 고성능 처리가 필요한 환경, 특히 메모리 관리와 다중 스레드(Multi-thread, 여러 작업을 동시에 처리) 처리에 강점이 있는 언어입니다

:::info 어떤 언어를 먼저 배울까?
목적에 따라 다릅니다.
- **프로그래밍 기초**: C 언어
- **웹 개발**: HTML/CSS → JavaScript → Python 또는 Java
- **안드로이드 앱**: Kotlin 또는 Java
- **iOS 앱**: Swift
- **데이터 분석/AI**: Python
- **게임 개발/시스템 프로그래밍**: C++
:::

## 8️⃣ 코딩 시 주의해야 할 오류 유형 ⚠️

### 🤔 왜 이런 오류들을 알아야 하는가?

코드를 작성할 때 문법적으로는 아무 문제가 없어도, 실제 실행 중에 예상치 못한 오류가 발생하는 경우가 많습니다. 특히 이런 오류들은 컴파일러가 잡아주지 못하기 때문에 개발자가 스스로 주의해야 합니다. 마치 문법적으로 맞는 문장이지만 의미가 이상한 것처럼, 코드도 문법은 맞지만 논리적으로 잘못된 경우가 있습니다.

### 🔴 메모리 관련 오류

#### 메모리 누수(Memory Leak)

**메모리 누수** 는 프로그램이 메모리를 할당(빌려서 사용)한 후, 사용이 끝났는데도 **반납(해제)하지 않는 경우** 발생합니다. 이 상태가 계속되면 사용 가능한 메모리가 점점 줄어들어 결국 시스템이 느려지거나 멈춰버립니다. 우리가 컴퓨터를 쓰다가 갑자기 느려지는 현상 중 하나가 이것 때문입니다.

```c
/* ❌ 잘못된 예시 - 메모리를 할당하고 해제하지 않음 */
#include <stdlib.h> /* malloc, free 함수를 사용하기 위한 헤더 */

void bad_function() {
    char *ptr = (char*)malloc(100); /* 메모리 100바이트를 빌림 */
    /* 작업을 수행함 */
    /* ❌ 함수가 끝나도 메모리를 반납하지 않음 → 메모리 누수 발생! */
}
```

```c
/* ✅ 올바른 예시 - 사용한 메모리를 반드시 해제함 */
#include <stdlib.h>

void good_function() {
    char *ptr = (char*)malloc(100); /* 메모리 100바이트를 빌림 */

    /* 작업을 수행함 */

    free(ptr); /* ✅ 사용이 끝나면 반드시 메모리를 반납합니다 */
    ptr = NULL; /* 반납 후 포인터를 NULL로 초기화하여 실수를 방지합니다 */
}
```

#### 널 포인터(Null Pointer) 오류

**널(NULL)** 은 "값이 없다" 또는 "가리키는 곳이 없다"를 의미합니다. 널 포인터가 가리키는 곳의 내용을 접근하려고 하면 프로그램이 충돌(Crash)합니다. 마치 주소가 없는 곳으로 택배를 보내려는 것과 같습니다.

```c
/* ❌ 잘못된 예시 - NULL 포인터 접근 */
#include <stdio.h>

int main() {
    char *ptr = NULL; /* ptr이 아무것도 가리키지 않음 (NULL) */

    /* ❌ NULL인 곳의 내용을 출력하려고 함 → 프로그램 충돌! */
    printf("%s\n", ptr);

    return 0;
}
```

```c
/* ✅ 올바른 예시 - NULL 여부를 먼저 확인 */
#include <stdio.h>

int main() {
    char *ptr = NULL; /* ptr이 NULL인 상태 */

    /* ✅ NULL인지 먼저 확인한 후 접근합니다 */
    if (ptr != NULL) {
        printf("%s\n", ptr);
    } else {
        printf("포인터가 NULL입니다. 접근할 수 없습니다.\n");
    }

    return 0;
}
```

### 🔴 배열 인덱스 범위 초과 오류

**배열(Array)** 은 같은 종류의 데이터를 연속으로 저장하는 공간입니다. C 언어에서 배열 인덱스는 항상 **0부터 시작** 하며, 크기가 80인 배열은 인덱스 0부터 79까지만 사용할 수 있습니다. 80을 인덱스로 사용하면 배열 범위를 벗어나 오류가 발생합니다.

```c
/* ❌ 잘못된 예시 - 배열 범위 초과 접근 */
#include <stdio.h>

int main() {
    int arr[80]; /* 크기 80짜리 배열 (인덱스 0~79 사용 가능) */

    /* ❌ 인덱스 80은 범위를 벗어납니다! (0~79까지만 유효) */
    arr[80] = 100;

    return 0;
}
```

```c
/* ✅ 올바른 예시 - 배열 범위 내에서만 접근 */
#include <stdio.h>
#define ARRAY_SIZE 80 /* 배열 크기를 상수로 정의합니다 */

int main() {
    int arr[ARRAY_SIZE]; /* 크기 80짜리 배열 */
    int i;

    /* ✅ 0부터 79까지, 즉 ARRAY_SIZE-1까지만 접근합니다 */
    for (i = 0; i < ARRAY_SIZE; i++) {
        arr[i] = i * 2; /* 각 칸에 값을 저장합니다 */
    }

    return 0;
}
```

### 🔴 하나 차이(Off-by-One) 오류

`<`(작다)와 `<=`(작거나 같다)의 차이처럼, 딱 하나의 차이가 완전히 다른 결과를 만들어 냅니다. 이런 오류는 컴파일러가 잡아주지 않아서 실행해 봐야 발견되는 경우가 많습니다.

```c
/* ❌ 잘못된 예시 - 1~100이 아니라 1~99까지만 더해버림 */
int sum = 0;
int i;
for (i = 1; i < 100; i++) { /* ❌ i < 100이므로 100은 포함되지 않습니다! */
    sum += i;
}
/* 결과: 4950 (1~99의 합) - 원하는 값이 아닙니다 */
```

```c
/* ✅ 올바른 예시 - 1~100까지 모두 더함 */
int sum = 0;
int i;
for (i = 1; i <= 100; i++) { /* ✅ i <= 100이므로 100까지 포함됩니다 */
    sum += i;
}
/* 결과: 5050 (1~100의 합) */
```

### 🔴 동기화 오류(Synchronization Error)

여러 프로세스나 스레드가 **공통 자원(데이터)에 동시에 접근** 할 때 발생하는 오류입니다. 예를 들어 A 스레드가 어떤 값을 B로 바꾸고 있는데, B 스레드가 동시에 그 값을 읽어가면 A는 "B로 바뀐 값"을 보고, B는 "아직 A인 값"을 읽어가게 되는 불일치가 발생합니다.

이런 충돌이 심각해지면 **데드락(Deadlock)** 이 발생합니다. 데드락은 두 프로세스가 서로 상대방의 자원을 기다리면서 영원히 기다리는 교착 상태입니다.

:::danger 버퍼 오버플로우(Buffer Overflow) 주의
버퍼(Buffer)란 입력값을 임시로 저장하는 메모리 공간입니다. 입력값이 버퍼의 크기를 초과하면 **버퍼 오버플로우** 가 발생하고, 인접한 메모리 공간을 침범하게 됩니다. 실제로 해커들이 이 취약점을 이용해서 악성 코드를 심고 시스템을 해킹하는 사례가 많습니다. 반드시 입력값의 크기를 검증하는 코드를 작성해야 합니다.
:::

### 📝 주석과 문서화의 중요성

좋은 코드는 **주석(Comment)** 이 잘 달려 있어야 합니다. 주석은 코드 자체에 설명을 덧붙여서, 나중에 그 코드를 다시 볼 때(혹은 다른 개발자가 볼 때) 쉽게 이해할 수 있게 해줍니다.

```c
/* ❌ 주석이 없어서 이해하기 어려운 코드 */
int f(int n) {
    if (n <= 1) return n;
    return f(n-1) + f(n-2);
}
```

```c
/* ✅ 주석이 있어서 이해하기 쉬운 코드 */

/*
 * 함수명: fibonacci
 * 목적: 피보나치 수열의 n번째 값을 반환합니다
 * 입력: n - 구하고자 하는 피보나치 수열의 위치 (0 이상의 정수)
 * 출력: n번째 피보나치 수
 */
int fibonacci(int n) {
    /* n이 0이나 1이면 n 자체를 반환합니다 (기저 조건) */
    if (n <= 1) return n;

    /* 이전 두 값의 합을 재귀적으로 계산합니다 */
    return fibonacci(n - 1) + fibonacci(n - 2);
}
```

:::tip 좋은 변수명 짓기
변수명도 직관적으로 지어야 합니다.
- ❌ `int a, b, c, x, y;`
- ✅ `int studentCount, totalScore, averageGrade;`

변수명만 봐도 무슨 데이터인지 알 수 있어야 좋은 코드입니다.
:::

## 📌 핵심 정리

이 문서에서 배운 내용을 한눈에 정리합니다.

- **컴퓨터 언어**는 컴퓨터와 의사소통하기 위한 모든 언어 체계를 의미하고, **프로그래밍 언어**는 소프트웨어를 만들기 위한 언어입니다
- **C 언어**는 C 프로그램을 개발할 수 있게 해주는 언어 환경(컴파일러 포함)이고, **C 프로그램**은 C 언어로 만든 소프트웨어 결과물입니다
- 컴파일 과정은 **원시 프로그램 → 컴파일러(문법 검사 + 기계어 번역) → 목적 프로그램 → 링커(연결) → 실행 파일 → 로더(실행)** 순서로 진행됩니다
- **코더**는 단순히 코드를 작성하는 사람이고, **프로그래머**는 요구사항 분석부터 설계, 코딩, 테스트, 유지보수까지 전체를 이끄는 사람입니다
- 좋은 프로그래밍의 3대 원칙은 **설계 명세서 완벽 반영**, **원시 코드의 간결 명료함**, **디버깅·테스트 용이성** 입니다
- **구조적 프로그래밍**은 단일 입구·출구와 순차·선택·반복의 제어 구조를 사용하며, goto문 사용을 최소화합니다
- 프로그래밍 언어는 **저급 언어(기계어, 어셈블리어)** 와 **고급 언어(컴파일러 언어, 인터프리터 언어)** 로 나뉩니다
- 언어 선택 기준은 **사용자 환경, 개발자 지식, 기존 프로젝트 언어, 컴파일러 품질, 개발 도구 지원** 등을 종합적으로 고려해야 합니다
- 코딩 시 주의해야 할 오류로는 **메모리 누수, 널 포인터 접근, 배열 범위 초과, 하나 차이 오류, 버퍼 오버플로우, 동기화 오류** 등이 있습니다
- **주석과 문서화**는 코드의 이해와 유지보수를 위해 반드시 습관화해야 합니다
