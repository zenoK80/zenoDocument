# 코딩과 프로그래밍의 차이: 코더와 프로그래머 👨‍💻

여러분은 "코딩"과 "프로그래밍"이라는 단어를 자주 들어보셨을 겁니다. 뉴스에서는 "코딩 교육 의무화", 취업 사이트에서는 "프로그래머 모집"이라는 말이 넘쳐나죠. 그런데 이 두 단어, 과연 같은 의미일까요? 아니면 다른 의미일까요?

이번 문서에서는 단순히 단어의 차이를 넘어서, **진짜 IT 전문가로 성장하기 위해 반드시 알아야 할 개념들**을 하나씩 풀어보겠습니다. 컴퓨터 언어와 프로그래밍 언어의 차이부터 시작해서, 코더와 프로그래머의 차이, 프로그램이 실행되는 원리, 프로그래밍 언어의 역사, 그리고 실제 코딩 시 반드시 주의해야 할 사항들까지 아주 자세하게 살펴보겠습니다.

---

## 🖥️ 컴퓨터 언어 vs 프로그래밍 언어: 비슷하지만 다르다

### 왜 이 두 용어를 구분해야 하는가?

"컴퓨터 언어"와 "프로그래밍 언어"는 일상적으로 거의 같은 의미로 쓰이지만, 엄밀하게 따지면 조금 다른 개념입니다. 마치 "자동차"와 "승용차"가 비슷하지만 완전히 같지는 않은 것처럼요. 자동차는 더 넓은 개념이고, 승용차는 그 안에 포함되는 특정 유형이죠. 컴퓨터 언어와 프로그래밍 언어도 이와 비슷한 관계입니다.

이 둘의 차이를 정확히 이해해야 나중에 "C 언어를 배운다"는 말과 "C 프로그래밍을 한다"는 말이 어떻게 다른지도 자연스럽게 이해할 수 있게 됩니다.

### 컴퓨터 언어란 무엇인가?

**컴퓨터 언어**란, 컴퓨터라는 시스템에서 소프트웨어가 동작하도록 만들기 위한 언어입니다. 더 쉽게 말하면, **컴퓨터와 사람이 서로 소통하기 위해 미리 약속된 규칙의 집합**이라고 할 수 있어요.

마치 한국 사람끼리는 한국어로, 미국 사람과는 영어로 소통하듯이, 컴퓨터와 소통하려면 컴퓨터가 이해할 수 있는 언어를 사용해야 합니다. 이 언어가 바로 컴퓨터 언어입니다. 컴퓨터 언어에는 기계어(0과 1로만 이루어진 언어), 어셈블리 언어, 그리고 우리가 흔히 배우는 고급 언어들이 모두 포함됩니다.

### 프로그래밍 언어란 무엇인가?

**프로그래밍 언어**는 좀 더 좁은 개념입니다. **프로그램(소프트웨어)을 작성하기 위해 특별히 설계된 언어**를 뜻합니다. 즉, 컴퓨터 언어 중에서도 "프로그램을 만드는 데 쓰이는 언어"를 프로그래밍 언어라고 부르는 것이죠.

예를 들어, C 언어, Java, Python, JavaScript 같은 것들이 바로 프로그래밍 언어입니다. 이 언어들은 모두 정해진 문법 규칙(Syntax, 신택스)이 있고, 그 규칙 안에서만 코드를 작성해야 컴퓨터가 올바르게 이해하고 실행할 수 있습니다.

| 구분 | 컴퓨터 언어 | 프로그래밍 언어 |
|------|-------------|-----------------|
| 정의 | 컴퓨터와 소통하기 위한 모든 언어 | 프로그램을 작성하기 위한 언어 |
| 범위 | 더 넓은 개념 | 컴퓨터 언어의 일부 |
| 예시 | 기계어, 어셈블리어, C, Java 등 | C, Java, Python, JavaScript 등 |
| 목적 | 컴퓨터 시스템과의 의사소통 | 소프트웨어(프로그램) 제작 |

:::info 쉽게 기억하는 법
컴퓨터 언어 ⊃ 프로그래밍 언어 (컴퓨터 언어가 더 큰 개념!)

프로그래밍 언어는 컴퓨터 언어의 한 종류입니다. 모든 프로그래밍 언어는 컴퓨터 언어이지만, 모든 컴퓨터 언어가 프로그래밍 언어는 아닐 수 있어요.
:::

### C 언어와 C 프로그램: 도구와 결과물의 차이

이 개념을 더 구체적으로 이해하기 위해, "C 언어"와 "C 프로그램"의 차이를 살펴봅시다.

- **C 언어**: C 프로그래밍을 가능하게 해주는 언어 도구 자체입니다. 마치 목수가 나무를 자르는 데 사용하는 "톱"에 해당합니다.
- **C 프로그램**: C 언어를 사용해서 실제로 개발된 소프트웨어입니다. 톱으로 만들어낸 "나무 의자"에 해당하죠.

즉, **C 언어를 이용해서 C 프로그래밍을 하고, 그 결과로 C 프로그램이 만들어지는 것**입니다. 재고 관리 프로그램을 만들든, 제어 프로그램을 만들든, 그것을 만드는 데 사용한 언어가 C 언어라면, 그 결과물이 바로 C 프로그램입니다.

아래는 간단한 C 언어 예시입니다:

```c
/* C 언어로 두 숫자의 합을 구하는 간단한 프로그램 */
#include <stdio.h>  /* 표준 입출력 헤더 파일을 불러옵니다 */

int main() {
    int a = 10;     /* 정수형 변수 a에 10을 저장합니다 */
    int b = 20;     /* 정수형 변수 b에 20을 저장합니다 */
    int sum;        /* 합계를 저장할 변수 sum을 선언합니다 */

    sum = a + b;    /* a와 b를 더한 결과를 sum에 저장합니다 */

    /* 결과를 화면에 출력합니다. 결과: 30 */
    printf("합계: %d\n", sum);

    return 0;       /* 프로그램이 정상적으로 종료되었음을 알립니다 */
}
```

이 코드가 바로 **원시 코드(소스 코드)** 입니다. 이것을 컴파일하면 컴퓨터가 이해할 수 있는 기계어로 변환되고, 최종적으로 실행 파일(test.exe)이 만들어집니다.

## 🎯 코딩 vs 프로그래밍: 코더가 아닌 프로그래머가 되어야 한다

### 왜 이 구분이 중요한가?

많은 사람들이 "코딩을 잘하면 훌륭한 개발자"라고 생각합니다. 하지만 실제 IT 업계에서는 **코딩을 잘하는 코더**와 **프로그래밍을 잘하는 프로그래머** 사이에 엄청난 차이가 있습니다. 연봉, 커리어 성장, 문제 해결 능력 모든 면에서요.

여러분이 앞으로 IT 전문가로 성장하고 싶다면, 반드시 코더를 넘어 프로그래머가 되는 것을 목표로 삼아야 합니다. 그러려면 먼저 이 두 가지가 어떻게 다른지 정확하게 알아야 하겠죠.

### 코더(Coder)란?

**코더**는 주어진 알고리즘(문제 해결 순서)이나 설계를 받아서 **그대로 코드로 옮기는 사람**입니다. 마치 건축 현장에서 설계도를 받아 벽돌을 쌓는 작업자와 비슷합니다. 물론 이 작업도 중요하고 전문성이 필요하지만, 설계 자체를 할 수는 없죠.

코더의 주요 특징:
- 누군가 알고리즘을 던져주면 그 알고리즘에 따라 코드를 작성합니다
- 이미 만들어진 프로그램을 가져와서 단순한 유지보수 작업을 합니다
- 기술적인 코딩 능력은 있지만, 전체 시스템을 설계하거나 기획하는 역량은 부족합니다

### 프로그래머(Programmer)란?

**프로그래머**는 단순히 코드를 작성하는 것을 넘어서, **문제를 분석하고, 논리적으로 해결책을 설계하며, 체계적인 개발 방법론을 적용하는 사람**입니다. 마치 건축 설계사처럼 전체 그림을 그리고 방향을 제시하는 역할을 합니다.

프로그래머의 주요 특징:
- 사용자의 요구사항을 분석하고 이해합니다
- 개발 계획(기간, 비용, 방법)을 논리적으로 수립합니다
- 어떤 프로그래밍 언어와 방법론이 가장 적합한지 판단합니다
- 코드의 효율성, 유지보수성, 확장성을 항상 고려합니다
- 문제가 생겼을 때 근본 원인을 분석하고 해결합니다

> 💡 **핵심 비유**: 코더는 레시피(조리법)를 보고 요리하는 사람이고, 프로그래머는 레시피 자체를 만들어내는 요리사입니다. 둘 다 요리를 할 수 있지만, 새로운 메뉴를 창조할 수 있는 사람은 프로그래머입니다.

| 구분 | 코더 (Coder) | 프로그래머 (Programmer) |
|------|-------------|------------------------|
| 역할 | 주어진 알고리즘을 코드로 변환 | 문제 분석부터 설계, 구현까지 전담 |
| 사고방식 | "어떻게 이걸 코드로 쓸까?" | "왜 이 문제가 생겼고, 최선의 해결책은 무엇인가?" |
| 계획 | 단기적, 주어진 작업 처리 | 장기적, 전체 시스템 고려 |
| 비용 인식 | 코딩 작업 자체에 집중 | 인건비, 개발 기간, 유지보수 비용까지 고려 |
| 성장 방향 | 코딩 속도, 문법 숙달 | 논리적 사고, 시스템 설계, 문제 해결 능력 |

:::warning 코더에 머물면 안 되는 이유
단순 코딩 작업은 자동화 도구나 AI의 발전으로 점점 대체될 가능성이 높습니다. 반면, 문제를 분석하고 논리적으로 해결책을 설계하는 프로그래머의 능력은 훨씬 오랫동안 가치를 유지합니다. 경력이 쌓였다고 해서 자동으로 몸값이 오르는 것이 아니라, **얼마나 깊이 있는 문제를 해결해 봤는가**, **얼마나 효율적이고 유지보수하기 좋은 코드를 작성할 수 있는가**에 따라 전문가의 가치가 결정됩니다.
:::

### 코딩(Coding)의 실제 의미

코딩은 프로그래밍 과정의 한 단계입니다. 구체적으로는 다음과 같은 작업들을 포함합니다:

1. **프로그래밍 언어 선택**: 개발하려는 프로그램의 성격에 맞는 언어를 고릅니다. 예를 들어, 절차적(순서대로 처리하는) 프로그램에는 C 언어가 적합하고, 객체지향(재사용과 확장이 중요한) 프로그래밍에는 C++나 Java가 적합합니다.

2. **코딩 스타일 표준 확립**: 팀에서 함께 작업할 때 코드가 일관성 있게 작성되도록 표준을 정합니다. 변수 이름을 짓는 방식, 들여쓰기 방식 등이 여기에 해당합니다.

3. **모듈(Module, 독립적인 기능 단위) 단위 코딩**: 전체 프로그램을 A, B, C처럼 작은 기능 단위로 나누고, 각 모듈을 따로 개발한 다음 연결해서 완성합니다. 마치 레고 블록을 하나씩 만들어서 조합하는 것처럼요.

## ⚙️ 프로그램이 실행되기까지: 컴파일러, 링커, 로더의 역할

### 왜 이 과정을 알아야 하는가?

우리가 작성한 코드가 어떻게 실행되는지 그 내부 흐름을 이해하면, 오류가 발생했을 때 어느 단계에서 문제가 생겼는지 훨씬 빠르게 파악할 수 있습니다. 마치 자동차의 구조를 이해하는 사람이 고장 났을 때 원인을 더 쉽게 찾는 것처럼요.

### 1단계: 원시 프로그램 (Source Program)

우리가 C, Java, Python 같은 언어로 작성한 코드를 **원시 프로그램(소스 코드)** 이라고 합니다. 이것은 사람이 이해할 수 있는 형태로 작성된 코드입니다. 하지만 컴퓨터는 이 코드를 그대로는 이해하지 못합니다. 컴퓨터는 오직 0과 1로 이루어진 기계어만 이해하기 때문이죠.

### 2단계: 컴파일러 (Compiler) - 문법 오류 검사

**컴파일러**는 원시 코드를 읽어서 **문법 오류(Syntax Error)가 있는지 검사**하는 도구입니다. 자바를 예로 들면, 자바에는 미리 약속된 문법 규칙들이 있습니다:

- 변수는 어떻게 선언해야 하는가
- 반복문(for, while)은 어떤 형태로 써야 하는가
- 출력할 때는 어떤 함수를 사용해야 하는가
- 값을 반환할 때는 어떻게 해야 하는가

이 규칙을 지키지 않으면 컴파일러가 "이건 문법에 맞지 않습니다!"라고 오류를 알려줍니다. 마치 학교에서 선생님이 맞춤법 검사를 해주는 것처럼요.

```java
// ❌ 잘못된 예시 - 문법 오류가 있는 코드
public class WrongExample {
    public static void main(String[] args) {
        int number = "안녕하세요"  // 정수형 변수에 문자열을 넣으려 함 (오류!)
        // 세미콜론도 빠져 있음 (오류!)
        System.out.println(number)
    }
}
```

```java
// ✅ 올바른 예시 - 문법 오류가 없는 코드
public class CorrectExample {
    public static void main(String[] args) {
        int number = 100;                        // 정수형 변수에 정수값 저장 (올바름)
        System.out.println("숫자: " + number);  // 세미콜론 포함, 올바른 출력 함수 사용
    }
}
```

### 3단계: 목적 프로그램 (Object Program) - 기계어로 변환

컴파일러가 문법 오류가 없다고 확인하면, 원시 코드를 **컴퓨터가 이해할 수 있는 기계어**로 번역합니다. 이것을 **목적 프로그램**이라고 합니다. 이제 이 코드는 사람이 읽기는 어렵지만, 컴퓨터가 이해할 수 있는 형태가 된 것입니다.

### 4단계: 링커 (Linker) - 실행 가능한 형태로 연결

목적 프로그램이 만들어졌다고 해서 바로 실행할 수 있는 것은 아닙니다. **링커**는 목적 프로그램을 실제로 실행 가능한 형태의 기계어로 연결해주는 도구입니다. 여러 모듈들을 하나로 합치고, 필요한 라이브러리들을 연결하는 작업을 합니다.

### 5단계: 로더 (Loader) - 실행

링커가 실행 파일(.exe)을 만들면, **로더**가 그 파일을 메모리에 올려서 실제로 실행합니다. 이 단계에서 드디어 우리 프로그램이 눈에 보이는 결과를 만들어냅니다.

```
[전체 흐름 요약]

원시 프로그램 (우리가 작성한 코드)
    ↓
  컴파일러 (문법 오류 검사 + 기계어 번역)
    ↓
목적 프로그램 (기계어로 번환된 코드)
    ↓
   링커 (실행 가능한 형태로 연결)
    ↓
  실행 파일 (.exe 등)
    ↓
   로더 (메모리에 올려서 실행)
    ↓
 프로그램 실행! (결과 출력)
```

:::tip 개발자가 신경 써야 할 부분
이 복잡한 과정에서 **개발자가 직접 신경 써야 하는 부분은 원시 프로그램(소스 코드)**뿐입니다. 문법에 맞게 올바른 코드를 작성하면, 나머지 컴파일, 링킹, 로딩 과정은 개발 도구(IDE)가 자동으로 처리해줍니다. C든 Java든 Python이든 마찬가지입니다.
:::

## 📐 프로그래밍의 기본 원칙과 구조적 프로그래밍

### 왜 원칙이 필요한가?

집을 지을 때 설계도 없이 마구 짓는다면 어떻게 될까요? 벽이 삐뚤어지고, 전기 배선이 엉키고, 나중에 수리할 때 어디가 어딘지 알 수 없게 됩니다. 프로그래밍도 마찬가지입니다. 원칙 없이 코드를 작성하면 처음에는 동작하는 것처럼 보여도, 나중에 문제가 생겼을 때 고치기가 너무 어려워집니다.

좋은 프로그래머는 항상 이 원칙들을 머릿속에 새기고 코드를 작성합니다.

### 프로그래밍의 3대 핵심 원칙

**첫 번째: 설계 명세서를 완벽하게 반영하라**

**설계 명세서(Design Specification, 어떤 기능을 어떻게 만들지 상세히 적어놓은 문서)**는 수많은 사용자 요구사항을 분석해서 만들어진 결과물입니다. 여기에는 입력은 어떻게 받을지(키보드? 음성? 파일?), 처리는 어떻게 할지, 출력은 어떤 형태로 보여줄지(텍스트? 그래프? 이미지?)가 모두 담겨있습니다. 이 명세서에 나온 요구사항을 단 하나도 빠짐없이 코드에 반영해야 합니다.

**두 번째: 원시 코드는 간단하고 명료하게 작성하라**

100명의 학생에게 "1부터 100까지 합을 구하는 프로그램을 만들어라"고 하면 100가지 다른 코드가 나옵니다. 그 중에서 **가장 좋은 코드는 가장 결과가 화려한 코드가 아니라, 가장 간단하고 누가 봐도 이해하기 쉬운 코드**입니다. 마치 잘 정비된 자동차 엔진처럼, 복잡해 보이지 않지만 완벽하게 동작하는 것이 최고입니다.

코드가 간단해야 하는 이유:
- **디버깅(Debugging, 오류를 찾아서 수정하는 작업)** 하기 쉽습니다
- 요구사항이 바뀌었을 때 **수정(Modification)** 하기 쉽습니다
- 다른 개발자가 코드를 이어받아서 작업하기 쉽습니다
- **테스트(Testing)** 하기 쉽습니다

:::note 디버깅 vs 수정의 차이
- **디버깅(Debugging)**: 오류(버그, Bug)가 발생했을 때 그 오류를 찾아서 고치는 작업입니다.
- **수정(Modification)**: 오류가 아니라, 요구사항이 바뀌거나 기능이 추가되어 코드를 변경하는 작업입니다.

이 두 가지는 완전히 다른 개념이므로 헷갈리지 않도록 주의하세요!
:::

**세 번째: 디버깅과 테스트가 용이하게 만들어라**

스마트폰, 세탁기, 자동차처럼 잘 만들어진 제품들은 공통적으로 설계가 심플합니다. 내부 부품이 복잡하게 얽혀있지 않아서 고장이 나도 어디가 문제인지 빠르게 찾을 수 있죠. 프로그램도 같습니다. 테스트하기 쉽고 오류를 찾기 쉬운 코드가 진짜 잘 만들어진 코드입니다.

### 구조적 프로그래밍 (Structured Programming)

**구조적 프로그래밍**은 프로그램을 체계적이고 읽기 쉽게 만들기 위한 프로그래밍 방법론입니다. 특히 **goto문(다른 줄로 갑자기 이동하는 명령어)을 최소화**하는 것이 핵심입니다.

goto문의 문제점을 비유로 설명하면, 이렇습니다: 여러분이 큰 백화점에서 쇼핑을 하는데, 안내방송이 계속 "3층으로 가세요", "다시 1층으로 오세요", "지금 당장 5층으로 뛰어가세요"를 반복한다면 어떨까요? 어디로 가야 할지 완전히 혼란스러워지죠. goto문이 많은 프로그램이 바로 이런 상태입니다.

구조적 프로그래밍의 핵심 특징:

1. **단일 입구 - 단일 출구**: 프로그램의 흐름이 한 곳에서 시작해서 한 곳에서 끝납니다
2. **제한된 제어 구조**: 순차(위에서 아래로), 선택(if-else), 반복(for, while) 세 가지만 사용합니다
3. **goto문 최소화**: 프로그램이 갑자기 앞뒤로 뛰어다니는 것을 최소화합니다

```c
/* ❌ 잘못된 예시 - goto문을 남발한 비구조적 코드 */
#include <stdio.h>

int main() {
    int i = 0;
start:                       /* 레이블(이동할 위치 표시) */
    if (i >= 5) goto end;    /* 조건이 맞으면 end로 이동 */
    printf("%d\n", i);
    i++;
    goto start;              /* 다시 start로 돌아감 - 코드가 엉킴! */
end:
    printf("완료\n");
    return 0;
}
```

```c
/* ✅ 올바른 예시 - 구조적 프로그래밍으로 작성한 코드 */
#include <stdio.h>

int main() {
    int i;

    /* for 반복문을 사용해 깔끔하게 표현 */
    for (i = 0; i < 5; i++) {
        printf("%d\n", i);   /* 0부터 4까지 출력 */
    }

    printf("완료\n");        /* 반복이 끝난 후 완료 메시지 출력 */
    return 0;                /* 정상 종료 */
}
```

구조적 프로그래밍의 장점과 단점을 정리하면:

| 구분 | 내용 |
|------|------|
| ✅ 장점 | 프로그램 이해하기 쉬움 |
| ✅ 장점 | 유지보수(수정) 쉬움 |
| ✅ 장점 | 테스트 쉬움 |
| ✅ 장점 | 개발 비용 절감 |
| ✅ 장점 | 프로그래밍 규율 강화 |
| ❌ 단점 | 성능이 다소 떨어질 수 있음 |
| ❌ 단점 | 변수가 많이 필요할 수 있음 |
| ❌ 단점 | 코드 중복이 발생할 수 있음 |
| ❌ 단점 | 하위 모듈 호출로 인한 오버헤드 발생 가능 |

## 🌍 프로그래밍 언어의 종류와 역사

### 왜 언어의 역사를 알아야 하는가?

프로그래밍 언어의 역사를 알면, 현재 우리가 쓰는 언어들이 왜 이런 특성을 갖고 있는지 자연스럽게 이해됩니다. 또한 어떤 언어가 어떤 상황에서 강점을 발휘하는지도 알게 되어, 프로젝트에 맞는 언어를 선택하는 안목이 생깁니다.

### 저급 언어 vs 고급 언어

프로그래밍 언어는 크게 **저급 언어**와 **고급 언어**로 나눌 수 있습니다.

- **저급 언어(Low-level Language)**: 기계가 이해하기 쉬운 언어입니다. 0과 1로만 이루어진 **기계어(Machine Language)**와, 기계어를 조금 더 읽기 쉽게 표현한 **어셈블리 언어(Assembly Language)**가 여기에 해당합니다.

- **고급 언어(High-level Language)**: 사람이 이해하기 쉬운 언어입니다. C, Java, Python 같은 언어들이 여기에 속합니다. 고급 언어는 다시 두 가지로 나뉩니다:
  - **컴파일러 언어**: 전체 코드를 한 번에 기계어로 번역한 후 실행합니다. C, Java, 포트란, 코볼 등이 해당합니다.
  - **인터프리터 언어(Interpreter Language, 해석 언어)**: 코드를 한 줄씩 읽어서 바로 실행합니다. 베이직(BASIC), Visual Basic 등이 해당합니다.

| 언어 종류 | 특징 | 예시 |
|-----------|------|------|
| 기계어 | 0과 1로만 구성, 컴퓨터가 직접 이해 | 01001000... |
| 어셈블리어 | 기계어 기반의 저급 언어, 약간 더 읽기 쉬움 | MOV AX, 1 |
| 컴파일러 언어 | 전체 코드를 번역 후 실행 | C, Java, Fortran |
| 인터프리터 언어 | 한 줄씩 해석하며 실행 | BASIC, Visual Basic |

### 연대별 프로그래밍 언어의 역사

**1950~60년대: 언어의 탄생**

컴퓨터가 처음 등장하던 시대입니다. 컴퓨터와 직접 소통해야 했기 때문에 기계어와 어셈블리어를 주로 사용했습니다. 이후 과학 계산과 수치 계산의 필요성이 커지면서 최초의 고급 프로그래밍 언어들이 등장했습니다.

- **포트란(Fortran)**: IBM이 과학 계산용으로 개발한 최초의 고급 프로그래밍 언어입니다
- **알골(ALGOL)**: 알고리즘 표현을 위한 언어로, 이후 프로그래밍 언어 문법의 기반을 마련했습니다
- **코볼(COBOL)**: 사무 처리(성적 관리, 재고 관리 등)에 특화된 언어입니다
- **베이직(BASIC)**: 가장 쉽고 기본적인 인터프리터 언어로 교육용으로 많이 사용됐습니다

**1970년대: C 언어의 등장**

- **C 언어**: 데니스 리치(Dennis Ritchie)와 켄 톰슨(Ken Thompson)이 개발한 언어로, 이식성과 효율성이 뛰어납니다. 유닉스(UNIX) 운영체제 개발에 사용되면서 널리 퍼졌으며, 임베디드 시스템(세탁기, TV 같은 기기에 들어가는 시스템), 네트워크 장비, 이미지 처리 등 다양한 분야에 사용됩니다. 현재까지도 모든 프로그래밍의 기초가 되는 가장 중요한 언어입니다.

**1980년대: 객체지향의 시작**

- **C++(C++ / Cpp)**: C 언어를 확장하여 **객체지향(Object-Oriented, 재사용과 확장이 용이한 프로그래밍 방식)** 개념을 도입한 언어입니다. C는 절차적(순서대로 처리) 언어이고, C++은 객체지향 언어라는 점이 가장 큰 차이입니다.
- **파스칼(Pascal)**: 구조적 프로그래밍과 교육용으로 주로 사용되었습니다.

**1990년대: 웹과 함께 폭발적 성장**

- **Java**: 웹 환경의 대표 언어로 자리 잡았습니다. 객체지향 프로그래밍을 기반으로 하며, **플랫폼 독립성(Platform Independence, 어떤 운영체제에서도 실행 가능)**이 가장 큰 강점입니다. 기업용 애플리케이션, 웹, 모바일(안드로이드) 개발에 널리 사용됩니다.
- **Python**: 문법이 간결하고 배우기 쉬워 초보자에게 특히 인기 있는 언어입니다. 웹 개발, 데이터 분석, 인공지능 분야에서 폭넓게 사용됩니다. 최근 통계에서 초급 개발자들이 가장 많이 선택하는 언어 1위를 차지하고 있습니다.
- **C#(C Sharp)**: 마이크로소프트(Microsoft)가 만든 객체지향 언어로, .NET 프레임워크를 기반으로 합니다.
- **PHP**: 웹 서버 사이드 스크립트 언어로, 웹사이트 개발에 특화되어 있습니다.
- **HTML**: 웹페이지를 구성하는 마크업 언어로, 우리가 보는 모든 웹사이트가 이것으로 만들어져 있습니다.

**2000년대 이후: 다양화와 전문화**

- **JavaScript**: 웹 프론트엔드(사용자가 보는 화면)의 동적 기능을 담당하는 언어입니다. Java와 이름이 비슷하지만 전혀 다른 언어입니다. Java를 먼저 공부하면 JavaScript를 배우기 더 쉽습니다.
- **Ruby**: 간결하고 직관적인 문법을 가진 동적 프로그래밍 언어로, 웹 개발에 많이 사용됩니다.
- **Swift**: 애플(Apple)이 iOS/macOS 앱 개발을 위해 만든 언어입니다.
- **Kotlin**: Java 기반으로 안드로이드 앱 개발에 주로 사용됩니다. Swift(iOS)와 Kotlin(Android)이 모바일 개발의 양대 축입니다.
- **Go(Golang)**: 구글(Google)이 만든 언어로, 클라우드 기반 웹 환경에서 많이 사용됩니다.
- **Rust**: 고성능 언어로, 메모리 관리와 다중 스레드(여러 작업을 동시에 처리) 환경에서 강점을 보입니다.

### 프로그래밍 언어 선택 시 고려할 사항

언어를 선택할 때는 다음 요소들을 고려해야 합니다:

1. **사용자 요구사항 및 환경**: 웹인지, 모바일인지, 데스크탑인지, 임베디드인지에 따라 적합한 언어가 다릅니다
2. **프로그래머의 지식**: 내가 잘 아는 언어로 개발하는 것이 생산성이 높습니다
3. **기존 프로젝트와의 호환성**: 이미 진행 중인 프로젝트에서 사용하던 언어를 유지하는 것이 유리합니다
4. **컴파일러의 가용성과 품질**: 언어마다 특성이 다르므로 환경에 맞는 컴파일러가 지원되는지 확인해야 합니다
5. **개발 도구(IDE) 지원**: 현대 개발에서는 코드 작성부터 컴파일, 테스트, 배포까지 하나의 통합 개발 도구에서 처리하므로, 원하는 언어를 잘 지원하는 IDE가 있어야 합니다
6. **호환성, 효율성, 유지보수 용이성**: 장기적으로 운영할 시스템이라면 반드시 고려해야 합니다

## ⚠️ 코딩 시 반드시 주의해야 할 오류 유형

### 왜 이 오류들을 미리 알아야 하는가?

이 오류들은 컴파일러가 잡아주지 못하는 경우가 많습니다. 문법적으로는 전혀 문제가 없어 보이지만, 실행했을 때 갑자기 시스템이 멈추거나, 잘못된 결과가 나오거나, 심각한 보안 문제가 생기는 원인이 됩니다. 미리 알고 주의하면 디버깅에 드는 시간을 크게 줄일 수 있습니다.

### 1. 메모리 누수 (Memory Leak)

**메모리 누수**는 프로그램이 메모리를 할당(빌려)한 뒤 다 쓰고 나서 반납하지 않는 문제입니다. 마치 도서관에서 책을 빌렸는데 계속 반납하지 않아서 다른 사람이 빌릴 수 없게 되는 것과 같습니다. 이 상태가 지속되면 결국 사용 가능한 메모리가 없어져서 시스템이 다운됩니다.

```c
/* ❌ 잘못된 예시 - 메모리를 할당하고 반납하지 않음 */
#include <stdlib.h>

void badFunction() {
    char *memory = (char *)malloc(100);  /* 100바이트 메모리 할당 */
    /* 사용 후 free()로 반납하지 않음 - 메모리 누수 발생! */
}
```

```c
/* ✅ 올바른 예시 - 메모리를 사용 후 반드시 반납 */
#include <stdlib.h>

void goodFunction() {
    char *memory = (char *)malloc(100);  /* 100바이트 메모리 할당 */

    if (memory != NULL) {               /* 할당이 성공했는지 확인 */
        /* 메모리 사용 */
        memory[0] = 'A';

        free(memory);                   /* 사용 후 반드시 메모리 반납! */
        memory = NULL;                  /* 반납 후 포인터를 NULL로 초기화 */
    }
}
```

### 2. 중복된 메모리 해제 (Double Free)

이미 반납(free)한 메모리를 또다시 반납하려는 오류입니다. 이미 반납한 책을 또 반납하러 가는 것과 같습니다. 이 경우 시스템이 어디서 오류가 생겼는지 파악하기 어려운 심각한 문제가 발생합니다.

```c
/* ❌ 잘못된 예시 - 이미 해제된 메모리를 또 해제 */
#include <stdlib.h>

int main() {
    char *ptr = (char *)malloc(10);  /* 메모리 할당 */
    free(ptr);                       /* 첫 번째 해제 (올바름) */
    free(ptr);                       /* ❌ 두 번째 해제 - 오류 발생! */
    return 0;
}
```

```c
/* ✅ 올바른 예시 - 해제 후 NULL 처리로 중복 해제 방지 */
#include <stdlib.h>

int main() {
    char *ptr = (char *)malloc(10);  /* 메모리 할당 */
    free(ptr);                       /* 메모리 해제 */
    ptr = NULL;                      /* NULL로 초기화하여 중복 해제 방지 */

    if (ptr != NULL) {               /* NULL 체크 후에만 해제 */
        free(ptr);
    }
    return 0;
}
```

### 3. 널 포인터 역참조 (Null Pointer Dereference)

**널(NULL)**은 "값이 없음"을 뜻합니다. 마치 빈 택배 상자에서 물건을 꺼내려는 것처럼, NULL 상태인 메모리에 접근하려 하면 시스템이 오류를 일으키고 심한 경우 다운됩니다.

```c
/* ❌ 잘못된 예시 - NULL 체크 없이 포인터 사용 */
#include <stdio.h>

int main() {
    char *ptr = NULL;          /* 포인터가 NULL 상태 */
    printf("%c\n", *ptr);     /* ❌ NULL 포인터 역참조 - 시스템 오류! */
    return 0;
}
```

```c
/* ✅ 올바른 예시 - 항상 NULL 체크 후 사용 */
#include <stdio.h>
#include <stdlib.h>

int main() {
    char *ptr = (char *)malloc(10);  /* 메모리 할당 */

    if (ptr != NULL) {               /* NULL이 아닌지 반드시 확인 */
        ptr[0] = 'A';
        printf("%c\n", ptr[0]);      /* 안전하게 출력 */
        free(ptr);
    } else {
        printf("메모리 할당 실패!\n");
    }
    return 0;
}
```

### 4. 배열 범위 초과 (Array Index Out of Bounds)

배열은 정해진 크기만큼만 사용할 수 있습니다. 크기가 80인 배열을 선언하면 인덱스(번호)는 0부터 79까지만 사용 가능합니다. 80번 인덱스에 접근하면 범위를 벗어난 것이므로 오류가 발생합니다.

```c
/* ❌ 잘못된 예시 - 배열 범위를 초과한 접근 */
#include <stdio.h>

int main() {
    int arr[80];         /* 0~79 인덱스 사용 가능 */
    arr[80] = 100;       /* ❌ 인덱스 80은 범위 초과! 오류 발생 */
    return 0;
}
```

```c
/* ✅ 올바른 예시 - 배열 크기 내에서만 접근 */
#include <stdio.h>
#define ARRAY_SIZE 80    /* 배열 크기를 상수로 정의 */

int main() {
    int arr[ARRAY_SIZE]; /* 80개짜리 배열 선언 */
    int i;

    for (i = 0; i < ARRAY_SIZE; i++) {  /* 0부터 79까지만 접근 */
        arr[i] = i * 2;                  /* 각 요소에 값 저장 */
    }

    printf("마지막 요소: %d\n", arr[ARRAY_SIZE - 1]);  /* 인덱스 79 접근 (올바름) */
    return 0;
}
```

### 5. Off-by-one 오류 (하나 차이에 의한 오류)

**Off-by-one 오류**는 "딱 하나 차이"로 생기는 오류입니다. `>` 대신 `>=`, `<` 대신 `<=`를 쓰는 것처럼 아주 사소한 차이지만, 전혀 다른 결과를 만들어냅니다. 컴파일러는 이 오류를 잡아주지 못해서 실제 실행할 때야 비로소 이상한 결과로 발견됩니다.

```c
/* ❌ 잘못된 예시 - off-by-one 오류 */
#include <stdio.h>

int main() {
    int arr[5] = {10, 20, 30, 40, 50};  /* 인덱스 0~4 */
    int i;

    /* ❌ i <= 5로 쓰면 arr[5]에 접근 - 범위 초과! */
    for (i = 0; i <= 5; i++) {
        printf("%d\n", arr[i]);
    }
    return 0;
}
```

```c
/* ✅ 올바른 예시 - 정확한 조건 사용 */
#include <stdio.h>

int main() {
    int arr[5] = {10, 20, 30, 40, 50};  /* 인덱스 0~4 */
    int i;

    /* ✅ i < 5로 써야 인덱스 0~4까지 올바르게 접근 */
    for (i = 0; i < 5; i++) {
        printf("%d\n", arr[i]);
    }
    return 0;
}
```

:::danger 절대 하면 안 되는 것: 0으로 나누기
수학에서도 0으로 나누는 것은 정의되지 않습니다. 프로그래밍에서도 마찬가지로, 어떤 숫자를 0으로 나누면 즉시 오류(런타임 에러)가 발생합니다. 나눗셈 연산을 할 때는 반드시 분모가 0인지 확인하는 조건을 추가해야 합니다.

```c
/* ❌ 절대 하면 안 되는 예시 */
int result = 100 / 0;  /* 런타임 오류 발생! */

/* ✅ 올바른 예시 */
int divisor = 0;
if (divisor != 0) {          /* 0인지 먼저 확인 */
    int result = 100 / divisor;
} else {
    printf("0으로 나눌 수 없습니다!\n");
}
```
:::

### 6. 버퍼 오버플로우 (Buffer Overflow)

**버퍼 오버플로우**는 정해진 메모리 공간(버퍼)보다 더 많은 데이터를 넣으려 할 때 발생합니다. 마치 10리터짜리 물통에 15리터를 넣으려는 것과 같습니다. 넘친 데이터가 다른 메모리 공간을 침범하게 되는데, 이것을 이용한 해킹 공격이 실제로 매우 많이 발생합니다.

```c
/* ❌ 잘못된 예시 - 버퍼 오버플로우 위험 */
#include <string.h>

int main() {
    char buffer[10];                         /* 10바이트 버퍼 */
    strcpy(buffer, "이 문자열은 10바이트보다 훨씬 깁니다!!");  /* ❌ 버퍼 초과! */
    return 0;
}
```

```c
/* ✅ 올바른 예시 - 크기 제한으로 버퍼 오버플로우 방지 */
#include <string.h>
#include <stdio.h>

int main() {
    char buffer[10];
    const char *src = "안녕하세요!";

    /* strncpy로 최대 복사 길이를 지정해 안전하게 복사 */
    strncpy(buffer, src, sizeof(buffer) - 1);  /* 최대 9글자만 복사 */
    buffer[sizeof(buffer) - 1] = '\0';          /* 마지막에 널 문자 추가 */

    printf("버퍼 내용: %s\n", buffer);
    return 0;
}
```

### 7. 동기화 오류 (Synchronization Error)와 데드락 (Deadlock)

여러 프로세스나 스레드(동시에 실행되는 작업 단위)가 같은 자원에 동시에 접근할 때 발생하는 오류입니다. 예를 들어, A 작업이 데이터를 "코리아"에서 "KOREA"로 바꾸는 도중에 B 작업이 그 데이터를 읽으면, B는 "코리아"를 읽을 수도 있고 "KOREA"를 읽을 수도 있어서 결과를 예측할 수 없습니다.

**데드락(Deadlock, 교착상태)**은 두 작업이 서로가 가진 자원을 기다리며 영원히 멈춰버리는 상태입니다. 마치 두 사람이 좁은 복도에서 만나 서로 비켜주기를 기다리며 아무도 못 지나가는 상황과 같습니다.

### 8. 주석과 문서화의 중요성

좋은 코드는 코드 자체도 중요하지만, 그 코드를 설명하는 **주석(Comment)**과 **문서화(Documentation)**도 매우 중요합니다. 주석이 잘 달린 코드는 나 자신도 나중에 이해하기 쉽고, 다른 개발자들과 협업할 때도 훨씬 원활합니다.

```c
/* ❌ 잘못된 예시 - 주석 없는 이해하기 어려운 코드 */
int f(int x, int y) {
    return x * x + 2 * x * y + y * y;
}
```

```c
/* ✅ 올바른 예시 - 명확한 이름과 주석이 있는 코드 */
/**
 * 두 정수 a와 b의 합의 제곱을 계산합니다.
 * 공식: (a + b)^2 = a^2 + 2ab + b^2
 *
 * @param a 첫 번째 정수
 * @param b 두 번째 정수
 * @return (a + b)의 제곱 값
 */
int calculateSquareOfSum(int a, int b) {
    /* (a + b)^2 공식을 전개한 형태로 계산 */
    return a * a + 2 * a * b + b * b;
}
```

:::tip 변수명과 주석 작성 팁
- 변수명은 `a`, `b`, `x` 같은 의미 없는 이름보다 `studentCount`, `totalPrice`, `maxRetry` 처럼 역할을 바로 알 수 있는 이름을 사용하세요.
- 주석은 "코드가 무엇을 하는가"보다 "왜 이렇게 했는가"를 설명할 때 더 유용합니다.
- 복잡한 알고리즘이나 특이한 로직에는 반드시 설명 주석을 달아두세요.
- 주석을 통해 개발자들 간의 소통이 훨씬 원활해집니다.
:::

## 📌 핵심 정리

이번 문서에서 배운 핵심 개념들을 정리합니다:

- **컴퓨터 언어**는 컴퓨터와 소통하기 위한 모든 언어를 뜻하며, **프로그래밍 언어**는 그 중에서 프로그램(소프트웨어)을 만들기 위한 언어입니다.
- **C 언어**는 언어 도구(톱)이고, **C 프로그램**은 그 언어로 만든 결과물(나무 의자)입니다.
- **코더**는 주어진 알고리즘을 코드로 옮기는 사람이고, **프로그래머**는 문제를 분석하고 설계부터 구현까지 주도하는 사람입니다.
- 프로그래머가 되려면 단순 코딩을 넘어 **논리적 사고, 시스템 설계, 비용 고려** 능력이 필요합니다.
- 소스 코드는 **컴파일러 → 목적 프로그램 → 링커 → 실행 파일 → 로더** 순서로 처리되어 실행됩니다.
- 프로그래밍의 핵심 원칙은 **설계 명세서 완벽 반영, 코드 간결성, 디버깅/테스트 용이성**입니다.
- **구조적 프로그래밍**은 goto문을 최소화하고 순차/선택/반복 구조를 활용해 코드를 읽기 쉽게 만드는 방법론입니다.
- 프로그래밍 언어는 **저급 언어(기계어, 어셈블리)** 와 **고급 언어(컴파일러 언어, 인터프리터 언어)** 로 나뉩니다.
- 언어 선택 시 **사용 환경, 개발자 역량, 기존 프로젝트, 컴파일러 품질, 개발 도구 지원** 등을 종합적으로 고려해야 합니다.
- 코딩 시 **메모리 누수, 중복 해제, 널 포인터, 배열 범위 초과, off-by-one 오류, 버퍼 오버플로우, 동기화 오류**를 반드시 주의해야 합니다.
- **주석과 문서화**는 코드 자체만큼 중요하며, 팀 협업과 유지보수 효율을 크게 높입니다.
- 개발자의 몸값은 경력 연수가 아니라 **얼마나 깊고 다양한 문제를 해결했느냐**에 따라 결정됩니다.
