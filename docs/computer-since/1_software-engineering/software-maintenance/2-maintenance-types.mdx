# 🔧 소프트웨어 유지보수의 모든 것: 완전·적응 유지보수부터 비용 계산까지

소프트웨어를 만들고 나면 끝일까요? 실제 개발 현장에서는 오히려 **소프트웨어가 완성된 그 순간부터 또 다른 긴 여정이 시작**됩니다. 이번 문서에서는 소프트웨어 유지보수(Maintenance)가 무엇인지, 어떻게 분류되는지, 그리고 어떻게 비용을 계산하는지까지 초보자도 이해할 수 있도록 아주 자세하게 살펴보겠습니다.

---

## 🚗 유지보수란 무엇인가? — 개념 소개

### 유지보수의 정의

**유지보수(Maintenance)** 란, 소프트웨어 혹은 시스템의 개발이 완료된 시점부터 그 소프트웨어가 더 이상 필요 없어져 폐기될 때까지, **사용하는 기간 내내 지속적으로 수행되는 소프트웨어의 생명을 연장시켜주는 작업**입니다.

이걸 딱 와닿게 이해하려면 **자동차**를 떠올려 보세요.

> 🚗 새 차를 샀습니다. 열심히 타고 다니면서 여행도 다니고, 출퇴근도 합니다. 10년, 15년, 20년을 탑니다. 그 과정에서 고장나면 고치고, 엔진오일도 갈고, 타이어도 교체합니다. 이렇게 관리를 하다가 어느 순간 "이제 이 차는 더 이상 쓸 수 없겠다"는 판단이 서면 폐차를 합니다. 폐차할 차에 타이어를 새로 달거나 엔진오일을 교체하지는 않죠.

소프트웨어도 똑같습니다. 개발이 완료된 시점부터 폐기되는 시점까지, 그 소프트웨어가 살아있는 동안에는 끊임없이 유지보수 작업이 이루어집니다. **한 번 하고 마는 것이 아니라, 계속해서 반복적으로 수행해야 하는 작업**입니다.

### "소프트웨어는 생물이다"

실무에서는 이런 말을 자주 합니다: **"소프트웨어는 생물이다."**

이 말의 의미는, 소프트웨어는 한 번 만들어 놓는다고 해서 그대로 변하지 않는 것이 아니라는 뜻입니다. 소프트웨어는 끊임없이 변합니다. 다음과 같은 이유들로요:

- **사용자 요구사항의 변경**: 처음에 없던 기능이 필요해지거나, 기존 기능이 불편해서 개선이 필요해집니다.
- **시스템 환경의 변화**: 운영체제(OS)가 새로운 버전으로 바뀌거나, 하드웨어 사양이 달라집니다.
- **예상하지 못한 오류 발생**: 아무리 꼼꼼하게 테스트해도, 실제 운영 환경에서는 예상치 못한 버그가 나타납니다.
- **경쟁 환경의 변화**: 경쟁사가 새로운 기능을 출시하면 우리도 그에 맞춰 기능을 추가해야 사용자 이탈을 막을 수 있습니다.

이처럼 소프트웨어는 살아 숨쉬는 생물처럼 계속 변화에 적응해야 하고, 그 적응을 도와주는 작업이 바로 **유지보수**입니다.

:::info 유지보수의 넓은 의미
유지보수는 단순히 "사용 중에 생기는 문제를 해결하는 것"에 그치지 않습니다. **미래의 변화에 대비해서 소프트웨어가 잘 적응할 수 있도록 지속적으로 관리하고 개선하는 과정** 전체를 포함합니다.
:::

### 실제 현장에서의 유지보수

실제 개발 현장에서는 10명이 참여한 프로젝트가 완성되고 나면, 그 중 2~3명은 별도로 남아서 유지보수를 담당합니다. 혹은 아예 전담 유지보수 팀이 따로 꾸려지기도 합니다. 그만큼 유지보수는 소프트웨어 개발만큼이나 중요하고, 결코 가볍게 여길 수 없는 작업입니다.

## 🗂️ 유지보수는 왜 필요한가? — 유지보수의 노력 분포

### 유지보수 작업의 통계적 비중

유지보수가 얼마나 중요한지를 숫자로 살펴보면 더욱 명확하게 와닿습니다. 소프트웨어 유지보수 작업 전체를 100%라고 했을 때, 각 활동은 다음과 같은 비율로 이루어집니다:

| 유지보수 유형 | 비율 | 설명 |
|---|---|---|
| **기능 개선** | 50% | 새로운 기능 추가 및 기존 기능 개선 |
| **환경 적응** | 25% | 운영체제·하드웨어 환경 변화에 대응 |
| **하자 수정 (A/S)** | 20% | 오류·버그 발견 후 수정 |
| **예방 유지보수** | 4% | 미래 문제를 대비한 구조 개선 |
| **기타** | 1% | 그 외 활동 |

가장 큰 비중을 차지하는 것은 **기능 개선(50%)** 입니다. 아무리 처음에 완벽하게 설계했다고 해도, 소프트웨어는 계속 변합니다. 환경이 바뀌고, 사용자 니즈가 달라지고, 경쟁자가 생기면서 새로운 기능을 추가해야만 하는 상황이 반드시 찾아옵니다.

### 기능 개선이 불가피한 이유

예를 들어, SNS 앱을 만들었는데 처음에는 우리 서비스만 유일하게 존재했다고 합시다. 그때는 굳이 새로운 기능을 추가하지 않아도 됐습니다. 그런데 경쟁사들이 속속 등장하면서 우리에게 없는 기능들을 탑재하기 시작하면, 사용자들은 이렇게 말합니다: *"A 앱은 이 기능이 좋은데, 저 기능이 없네. B 앱에는 그 기능이 있으니까 B로 옮겨가야겠다."* 이렇게 고객이 경쟁사로 이탈하는 것을 막으려면 결국 기능을 지속적으로 개선할 수밖에 없습니다.

### 기능 개선 작업의 세부 구성

기능 개선 작업 자체도 단순히 코드만 고치는 것이 아닙니다. 다음과 같은 여러 단계로 구성됩니다:

- **신규 요구사항 분석**: 사용자가 무엇을 원하는지 정확하게 파악합니다.
- **기존 코드 이해**: 누가 보더라도 쉽게 읽히는 코드가 이 단계를 빠르게 해줍니다.
- **문서 이해 및 관리**: 기존에 작성된 설계 문서, 요구사항 정의서 등을 파악합니다.
- **실제 코딩**: 변경사항을 코드로 구현합니다. (전체의 약 20% 비중)
- **테스트**: 개선된 기능이 오류 없이 잘 동작하는지 검증합니다.

:::warning 테스트 없는 배포는 금물
기능을 개선했다고 해서 바로 서비스에 올리면 안 됩니다. 새로운 기능이 추가된 후 오류가 발생하면, 기능 개선을 하지 않은 것보다 오히려 더 나쁜 상황이 됩니다. 무수히 많은 테스트를 거친 후에야 배포해야 합니다.
:::

## 📋 유지보수의 분류 — 4가지 핵심 유형

유지보수는 **작업의 목적**에 따라 크게 4가지로 분류됩니다. 각각의 유형을 자세히 살펴보겠습니다.

### 1️⃣ 완전 유지보수 (Perfective Maintenance)

**완전 유지보수**란, 기존 소프트웨어에 **새로운 기능을 추가**하거나 **기존 기능을 개선**하는 작업입니다. 영어로는 'Perfective(완전한, 퍼펙트한)'라는 단어에서 왔습니다.

쉽게 말하면 이렇습니다. 예전에 온라인 쇼핑몰의 결제 시스템에는 신용카드 결제, 계좌이체, 무통장입금 정도만 있었습니다. 그런데 최근 들어 **간편결제** (카카오페이, 네이버페이 등)가 대중화되었습니다. 기존에 없던 결제 방식이죠. 이 간편결제 기능을 기존 결제 시스템에 추가하는 것이 바로 완전 유지보수입니다.

또한, 기존에 있던 기능이 불편하거나 부족한 부분이 있다면 그것을 개선하는 것도 완전 유지보수에 해당합니다. 예를 들어 검색 기능이 너무 느려서 빠르게 개선한다든지, 사용자 인터페이스(UI)를 더 직관적으로 개편하는 작업 등이 있습니다.

> 💡 **완전 유지보수 = 기능 추가 + 기능 개선**
> 소프트웨어를 더 "완전"하게 만들기 위한 작업이라고 기억하세요.

### 2️⃣ 적응 유지보수 (Adaptive Maintenance)

**적응 유지보수**란, 소프트웨어가 **새로운 환경에 적응**할 수 있도록 변경하는 작업입니다. 여기서 '환경'이란 운영체제(OS), 하드웨어, 네트워크, 다양한 디바이스 등을 의미합니다.

우리가 새로운 곳으로 이사를 가거나 새로운 나라에서 생활하게 되면 그 환경에 적응해야 하듯이, 소프트웨어도 변화한 환경에 맞추어 적응해야 합니다.

예를 들어, 기존에 **Windows PC 환경에서만 동작하던 소프트웨어**가 있다고 합시다. 그런데 점점 더 많은 사용자들이 스마트폰으로 서비스를 이용하기 원합니다. 이때 안드로이드(Android)나 iOS 환경에서도 동작하도록 소프트웨어를 변경하는 것이 바로 적응 유지보수입니다.

이렇게 기존 소프트웨어를 다른 환경으로 옮겨주는 작업을 **포팅(Porting)** 이라고도 부릅니다. 마치 물고기를 다른 어항으로 옮겨주는 것처럼, 소프트웨어를 새로운 환경의 그릇에 담아주는 작업입니다.

적응 유지보수가 필요한 대표적인 상황들:

- Windows XP에서만 돌아가던 프로그램을 Windows 11에서도 동작하게 수정
- PC 전용 웹사이트를 모바일 반응형 웹으로 변경
- 32비트 환경에서 64비트 환경으로 전환
- 새로운 데이터베이스(DB) 시스템으로 마이그레이션(이전)

> 💡 **적응 유지보수 = 환경 변화에 대한 적응**
> 소프트웨어가 새로운 환경에서도 잘 살아남을 수 있도록 도와주는 작업입니다.

### 3️⃣ 수정 유지보수 (Corrective Maintenance)

**수정 유지보수**란, 소프트웨어를 사용하는 도중 발생하는 **오류(버그)나 하자를 찾아내고 수정**하는 작업입니다. 우리가 흔히 말하는 **A/S(After Service)** 와 같은 개념입니다.

예를 들어, 쇼핑몰 앱을 사용하다가 이런 문제들이 생길 수 있습니다:
- 장바구니에 담은 상품이 사라지는 오류
- 결제를 시도하면 오류 메시지가 뜨는 문제
- 배송 조회 화면이 정상적으로 표시되지 않는 현상
- 환불 신청 후 자동 처리가 안 되고 에러가 발생하는 문제

이런 문제들이 발생하면 원인을 찾아서 해결해야 하는데, 이 과정 전체가 수정 유지보수입니다.

핸드폰을 예로 들면, 사용 중에 갑자기 배터리가 빨리 소모되거나, 자꾸 꺼지거나, 특정 앱이 먹통이 된다면 서비스센터에 A/S를 요청하는 것과 같습니다.

:::note 수정 유지보수와 코드 품질의 관계
수정 유지보수가 얼마나 쉬운지는 처음부터 코드를 얼마나 잘 짰느냐에 달려있습니다. 마치 자동차 엔진룸이 복잡하게 설계되어 있으면 작은 부품 하나를 교체하려 해도 모든 것을 다 분해해야 하듯이, 코드가 복잡하게 얽혀있으면 간단한 버그 하나 고치는 데도 엄청난 시간이 걸립니다. 처음부터 수정이 쉽도록 코드를 작성하는 것이 중요합니다.
:::

**수정이 용이한 코드 vs 그렇지 않은 코드**를 비교해봅시다:

```python
# ❌ 나쁜 예시: 수정하기 어려운 코드
# 함수 하나에 너무 많은 일을 몰아넣음
# 중복 코드가 많고, 기능이 뒤섞여 있음
def process_order(user_id, product_id, quantity, payment_type, address, discount_code):
    # 유저 검증
    user = db.query("SELECT * FROM users WHERE id = " + str(user_id))
    if not user:
        print("유저 없음")
        return False
    # 상품 검증 (중복 코드 시작)
    product = db.query("SELECT * FROM products WHERE id = " + str(product_id))
    if not product:
        print("상품 없음")
        return False
    # 재고 확인 (또 다른 역할)
    if product['stock'] < quantity:
        print("재고 없음")
        return False
    # 할인 코드 처리 (또 다른 역할이 섞임)
    discount = 0
    if discount_code == "SALE10":
        discount = 0.1
    elif discount_code == "SALE20":
        discount = 0.2
    # 결제 처리 (또 다른 역할이 섞임)
    total = product['price'] * quantity * (1 - discount)
    # ... 모든 로직이 하나의 함수에 뒤섞여 있음
```

```python
# ✅ 좋은 예시: 수정하기 쉬운 코드
# 각 기능이 분리(모듈화)되어 있어서 수정이 용이함

def validate_user(user_id):
    """사용자 유효성 검증 - 이 함수만 수정하면 검증 로직 변경 가능"""
    user = UserRepository.find_by_id(user_id)  # DB 접근을 별도 계층으로 분리
    if not user:
        raise UserNotFoundException(f"사용자를 찾을 수 없습니다: {user_id}")
    return user

def validate_product_stock(product_id, quantity):
    """상품 재고 검증 - 재고 관련 로직만 독립적으로 관리"""
    product = ProductRepository.find_by_id(product_id)
    if not product:
        raise ProductNotFoundException(f"상품을 찾을 수 없습니다: {product_id}")
    if product.stock < quantity:
        raise InsufficientStockException("재고가 부족합니다.")
    return product

def calculate_discount(discount_code):
    """할인율 계산 - 할인 정책이 바뀌면 이 함수만 수정"""
    discount_rates = {
        "SALE10": 0.1,
        "SALE20": 0.2,
        "SALE30": 0.3
    }
    return discount_rates.get(discount_code, 0)  # 없는 코드면 0% 할인

def process_order(user_id, product_id, quantity, payment_type, address, discount_code):
    """주문 처리 - 각 역할이 분리되어 있어 읽기 쉽고 수정하기 쉬움"""
    user = validate_user(user_id)              # 사용자 검증
    product = validate_product_stock(product_id, quantity)  # 재고 검증
    discount = calculate_discount(discount_code)             # 할인율 계산
    # ... 이제 각 단계가 명확하게 분리됨
```

위의 좋은 예시처럼 코드가 모듈화(기능별로 잘 분리)되어 있으면, 나중에 할인 정책이 바뀌어도 `calculate_discount` 함수만 수정하면 됩니다. 다른 코드를 건드릴 필요가 없어서 추가 오류가 생길 위험이 줄어듭니다.

### 4️⃣ 예방 유지보수 (Preventive Maintenance)

**예방 유지보수**란, 현재는 문제가 없더라도 **미래에 발생할 수 있는 문제를 예방하기 위해 소프트웨어의 구조를 미리 개선**하는 작업입니다.

예를 들어, 현재 코드가 동작은 하지만 구조가 복잡하게 얽혀 있어서 나중에 기능을 추가하거나 수정할 때 문제가 생길 것 같다면, 미리 코드 구조를 깔끔하게 정리해두는 것입니다. 이를 **리팩토링(Refactoring)** 이라고도 부릅니다.

모듈화(기능별로 코드를 잘 나누는 것)가 되어있지 않은 코드를 객체지향(Object-Oriented) 개념으로 재구성하거나, 복잡하게 얽힌 반복문과 조건문을 정리하는 것이 대표적인 예입니다.

비율로 보면 예방 유지보수는 약 **4%** 정도로 낮지만, 이 작업을 꾸준히 해두면 나중에 훨씬 더 큰 비용과 노력을 절감할 수 있습니다.

## 🛠️ 유지보수를 쉽게 하는 방법 — 핵심 원리

### 표준화(Standard)가 최우선이다

유지보수를 쉽게 하기 위한 가장 중요한 방법은 바로 **표준화**입니다. 여기서 표준화란, 소프트웨어 개발과 운영에 관한 규칙, 절차, 문서 양식 등을 미리 정해두는 것을 의미합니다.

LMS(Learning Management System, 학습 관리 시스템) 프로젝트를 예로 들어볼까요? LMS 안에는 성적 처리, 레포트 제출, 결제, 토론, 시험 출제 등 수많은 기능이 있습니다. 이 각각의 기능에 대한 **표준 문서**가 잘 갖추어져 있다면, 새로운 개발자가 투입되어도 문서를 보면서 바로 이해하고 개발할 수 있습니다.

반면 표준이 없으면 어떻게 될까요? "레포트는 0점에서 100점까지 채점한다"는 한 줄짜리 규정만 있다면, 개발자 입장에서는 수많은 의문이 생깁니다.
- 소수점 처리는 어떻게 하나?
- 미제출은 0점인가, null 처리인가?
- 지각 제출은 감점 처리되는가?
- 채점 결과는 언제 공개되는가?

이런 불명확한 부분 때문에 개발자와 기획자, 운영팀 사이에서 갈등이 생기고, 결국 유지보수 비용이 올라가게 됩니다.

:::tip 표준 문서의 종류
유지보수를 위해 갖춰야 할 표준 문서들:
- **요구사항 정의서**: 시스템이 무엇을 해야 하는지 명세
- **기능 정의서**: 각 기능의 동작 방식 상세 기술
- **코드 정의서**: 코딩 스타일, 네이밍 규칙 등 코드 작성 표준
- **DB 설계서**: 데이터베이스 테이블 구조 및 관계 정의
- **API 명세서**: 서버와 클라이언트 간 통신 규칙 정의
:::

### 설계의 명확성과 모듈화

**설계가 명확**해야 하고, 코드가 **모듈화(Modularization)** 되어 있어야 합니다. 모듈화란 소프트웨어를 역할별로 잘게 나누어 각 조각이 독립적으로 동작하도록 하는 것입니다. 마치 레고 블록처럼, 각 블록이 독립적이기 때문에 특정 블록만 교체해도 전체가 잘 동작하는 구조입니다.

### 디렉터리(목차) 구조의 중요성

코드와 문서가 **디렉터리(폴더) 구조로 잘 정리**되어 있어야 합니다. 책에 목차가 있듯이, 소프트웨어도 어떤 코드가 어디에 있는지 한눈에 파악할 수 있는 구조가 필요합니다. 목차 없는 책을 상상해보세요. 특정 내용을 찾으려면 처음부터 끝까지 다 읽어야 할 것입니다. 소프트웨어도 마찬가지입니다.

### 단일 입구, 단일 출구 원칙

**단일 입구(Single Entry), 단일 출구(Single Exit)** 구조를 사용하면 유지보수가 훨씬 쉬워집니다. 데이터가 들어오는 곳은 하나, 나가는 곳도 하나로 유지하는 원칙입니다.

마치 공항의 입국심사처럼, 모든 여행객이 하나의 입국심사대를 통과하면 관리가 쉽지만, 여기저기 여러 곳에서 입국이 가능하다면 추적이 어려워집니다. 데이터 흐름도 마찬가지입니다.

```python
# ❌ 나쁜 예시: 출구가 여러 곳 (단일 출구 원칙 위반)
def calculate_grade(score):
    if score >= 90:
        return "A"   # 출구 1
    if score >= 80:
        return "B"   # 출구 2
    if score >= 70:
        return "C"   # 출구 3
    if score >= 60:
        return "D"   # 출구 4
    return "F"       # 출구 5
    # 출구가 5개라서 디버깅 시 어느 경로로 나갔는지 파악하기 어려움
```

```python
# ✅ 좋은 예시: 출구가 하나 (단일 출구 원칙 준수)
def calculate_grade(score):
    # 결과를 저장할 변수를 미리 선언
    grade = "F"  # 기본값 설정

    # 점수에 따라 등급 결정 (조건 판단만 수행)
    if score >= 90:
        grade = "A"
    elif score >= 80:
        grade = "B"
    elif score >= 70:
        grade = "C"
    elif score >= 60:
        grade = "D"

    return grade  # 출구는 항상 여기 하나만 존재
    # 어떤 경로를 타더라도 마지막에 이 출구로 나오므로 추적하기 쉬움
```

### 도메인 지식(Domain Knowledge)의 중요성

유지보수를 잘 하려면 해당 소프트웨어의 **업무 도메인(Domain, 분야)을 이해**해야 합니다. 회계 프로그램을 수정하려면 회계에 대한 기본 지식이 있어야 하고, 성적 처리 프로그램을 개발하려면 학사관리 제도를 알아야 합니다. 아무리 코딩 실력이 뛰어나도, 그 소프트웨어가 무엇을 위한 것인지 모른다면 제대로 된 유지보수를 할 수 없습니다.

## ⚠️ 유지보수 프로세스 모델 — 실제 작업 방식

### 즉시 수정 모델 (Quick-Fix Model)

**즉시 수정 모델**은 말 그대로 **임시방편으로 즉시 수정**하는 유지보수 방식입니다. 체계적인 분석이나 문서 작업 없이, 문제가 발생하면 바로 코드를 고쳐서 해결합니다.

예를 들어, 운영팀에서 전화가 옵니다: *"지금 동영상 재생이 안 돼요! 빨리 고쳐주세요!"* 이럴 때 유지보수 담당자가 즉시 코드를 열어 수정하고 배포하는 방식입니다.

이 방식이 효과적으로 동작하려면 유지보수 담당자가 해당 시스템을 매우 잘 알고 있어야 합니다. 시스템을 오랫동안 담당했거나, 직접 개발에 참여한 사람이라면 문서 없이도 빠르게 문제를 찾고 수정할 수 있습니다.

| 장점 | 단점 |
|---|---|
| 빠른 문제 해결 | 구조적 영향 분석 생략 |
| 즉각적인 서비스 복구 | 다른 기능에 부작용 발생 가능성 |
| 간단한 절차 | 반복 수정 시 코드 품질 저하 |

### 반복적 개선 모델 (Iterative Enhancement Model)

**반복적 개선 모델**은 소프트웨어 변경이 특정 단계에서만 일어나는 것이 아니라, **전체 생명주기(요구사항 분석 → 설계 → 코딩 → 테스트)에 걸쳐 반복적으로 이루어진다**는 개념을 기반으로 합니다.

즉, 문제가 생겼을 때 바로 코드를 뜯어고치는 것이 아니라, 다시 요구사항을 분석하고, 설계를 검토하고, 코드를 수정하고, 테스트하는 전체 과정을 반복합니다.

이 방식은 시간이 더 걸리지만, 더 안전하고 품질 높은 결과를 만들어냅니다. 단, 이것이 가능하려면 다음 조건이 필요합니다:
- **완벽한 문서화**가 되어있어야 합니다.
- 유지보수팀이 시스템을 **완전히 분석할 능력**이 있어야 합니다.

### 재사용 중심 모델 (Reuse-Oriented Model)

**재사용 중심 모델**은 소프트웨어를 구성하는 **컴포넌트(Component, 독립적인 기능 단위)들을 저장소에 모아두고, 필요할 때 꺼내어 재사용**하는 방식입니다.

마치 부품 창고처럼, 자주 쓰이는 기능 블록들을 미리 만들어 정리해두고, 유지보수 시에는 해당 컴포넌트를 꺼내서 수정하거나 조합하여 사용합니다. 최근의 프레임워크 기반 개발에서 많이 활용되는 방식입니다.

## ⚠️ 유지보수 시 주의사항 — 절대 하지 말아야 할 것들

### 코드를 이해하지 않고 수정하는 것

유지보수에서 가장 위험한 행동은 **코드를 제대로 이해하지 않은 상태에서 임의로 수정하는 것**입니다.

처음에 오류가 5개였는데, 코드를 이해하지 못한 채 수정하다 보면 오류가 10개, 15개로 늘어나는 경우가 생깁니다. 이것이 실무에서 종종 발생하는 "유지보수를 하면 할수록 오류가 더 생기는" 상황입니다.

자동차 비유로 설명하면 이렇습니다: 자동차 구조를 전혀 모르는 사람이 브레이크가 말을 안 듣는다고 해서 임의로 뜯어 고쳤더니, 주행 중에 엔진이 서버리거나 불이 나는 상황이 발생하는 것과 같습니다. 정확하게 이해한 상태에서만 수정을 해야 합니다.

:::danger 절대 하면 안 되는 것
소프트웨어의 구조와 코드를 충분히 이해하지 않은 상태에서 프로덕션(실제 서비스 운영) 코드를 직접 수정하는 것은 절대 금물입니다. 더 큰 장애를 만들어낼 수 있습니다.
:::

### 변경 영향 분석을 생략하는 것

어떤 기능을 변경할 때, 그 변경이 **다른 기능에 어떤 영향을 미치는지** 반드시 분석해야 합니다. 이것을 **형상 변경 관리(Configuration Change Management)** 라고 합니다.

예를 들어, 소프트웨어 버전이 A1.0 → A1.1 → A1.2로 바뀐다고 할 때, 각 버전에서 무엇이 어떻게 변경되었는지 기록하고 관리해야 합니다. 주문 처리 컴포넌트를 수정했더니 결제 컴포넌트에 오류가 생기는 상황을 미리 파악하고 대비해야 합니다.

유지보수를 담당하는 소프트웨어 엔지니어는 반드시 **의사결정에 적극적으로 참여**해야 합니다. 기능 변경 요청이 들어왔을 때, 그 변경이 가져올 수 있는 부작용과 리스크를 명확하게 설명하고, 올바른 방향으로 작업이 진행되도록 해야 합니다.

### 변경 요청을 무조건 수용하는 것

사용자의 요구라고 해서 모든 변경 요청을 무조건 수용하면 안 됩니다. 변경을 하기 전에 반드시 다음을 분석해야 합니다:

- **왜 이 변경이 필요한가?** (변경의 이유와 근거)
- **변경했을 때 어떤 영향이 있는가?** (다른 기능에 미치는 영향)
- **비용은 얼마나 드는가?** (개발 인력, 시간)
- **어떤 리스크가 있는가?** (보안, 성능, 안정성)

동영상 다운로드 기능 추가 요청을 예로 들어보겠습니다. 사용자들이 "오프라인에서도 강의를 볼 수 있도록 다운로드 기능을 추가해달라"고 요청합니다. 이때 단순히 "좋아요, 바로 추가할게요"가 아니라, **보안 리스크**를 먼저 분석해야 합니다. 다운로드된 파일이 암호화 없이 공유된다면, 유료 강의가 무료로 유통되어 저작권 침해와 수익 손실이 발생할 수 있습니다. 반드시 DRM(디지털 저작권 관리) 기술을 함께 검토해야 합니다.

## 🧮 유지보수 비용 계산 — 어떻게 산정하는가?

### 유지보수 비용의 특성

유지보수 비용은 개발 비용과 달리 **예측하기가 매우 어렵습니다.** 언제 어떤 오류가 발생할지, 사용자가 어떤 복잡한 기능을 요구할지 미리 알 수 없기 때문입니다.

일반적으로, 소프트웨어 개발 비용이 10억 원이라면 **유지보수 비용은 연간 약 2.5억 원(25% 수준)** 을 예상해야 합니다. 이 비용을 아끼려다 보면 결국 시스템이 제대로 관리되지 않아 더 큰 비용이 발생합니다.

또한, **시스템을 오래 사용할수록 유지보수 비용은 증가**합니다. 자동차를 오래 타면 고장이 잦아지고 수리비가 늘어나는 것처럼, 소프트웨어도 오래될수록 환경 변화 대응 비용, 구조적 부채(기술 부채) 해결 비용 등이 쌓입니다.

### 유지보수 비용에 영향을 미치는 요소

비용 계산 시 고려해야 할 요소들은 **관리적 요소**와 **기술적 요소**로 나눌 수 있습니다:

| 구분 | 요소 | 설명 |
|---|---|---|
| **관리적 요소** | 시스템 이해도 | 담당자가 시스템을 얼마나 잘 아는가 |
| **관리적 요소** | 소프트웨어 수명 | 앞으로 몇 년을 더 사용할 계획인가 |
| **관리적 요소** | 하드웨어 안정성 | 운영 환경이 얼마나 안정적인가 |
| **관리적 요소** | 외부 환경 종속성 | 외부 시스템에 얼마나 의존하는가 |
| **기술적 요소** | 모듈화 수준 | 코드가 기능별로 잘 분리되어 있는가 |
| **기술적 요소** | 응집도·결합도 | 모듈 내부는 강하게, 모듈 간은 느슨하게 연결되어 있는가 |
| **기술적 요소** | 프로그래밍 언어 | 어떤 언어로 개발되어 있는가 |
| **기술적 요소** | 코드 품질 | 코드가 읽기 쉽고 일관된 스타일인가 |
| **기술적 요소** | 문서화 수준 | 설계 문서, 주석 등이 잘 갖추어져 있는가 |

문서화가 잘 되어 있고, 모듈화가 잘 되어 있으며, 시스템이 안정적인 경우와 그렇지 않은 경우의 유지보수 비용 차이는 **최대 5배 이상** 날 수 있습니다. 예를 들어 같은 시스템인데 문서가 없는 경우에는 연간 1억 원이 필요하지만, 문서화가 잘 된 경우에는 2,000만 원이면 충분할 수도 있습니다.

### 비용 계산 방법들

**① 주먹구구식 방법**: 개발 비용을 기준으로 유지보수 비용을 간단히 추정하는 방법입니다. 정확도가 낮지만 빠르게 어림잡을 때 사용합니다.

**② 벨라디-레만(Belady-Lehman) 방법**: 유지보수에 들어가는 노력(한 달에 몇 명의 인력이 필요한지)을 수식으로 계산합니다. 생산적 활동에 들어가는 비용에 경험적 상수와 지수를 곱하고, 복잡도와 소프트웨어에 대한 친숙도를 반영합니다.

**③ COCOMO(코코모) 방법**: 다음 세 가지 값을 사용하여 인력을 계산합니다.

```
유지보수 인력(MM) = ACT × EAF × (KDSI)^지수

- ACT: 유지보수 작업량 (코드 한 줄에 가해지는 연간 변경 건수)
- EAF: 개발 노력 조정 계수 (한 달에 투입되는 인력)
- KDSI: 천 라인 단위의 코드 크기
```

**④ SMI(Software Maturity Index, 소프트웨어 성숙도 지수)**: 소프트웨어의 안정성을 측정하는 지수입니다. 각 릴리즈(버전) 간 변경사항을 분석하여 계산합니다.

```
SMI = (MT - (Fa + Fc + Fdel)) / MT

- MT: 현재 릴리즈의 총 모듈 수
- Fa: 추가된 모듈 수
- Fc: 변경된 모듈 수
- Fdel: 삭제된 모듈 수
```

SMI 값이 **1에 가까울수록 소프트웨어가 안정적**이라는 의미입니다. 변경이 거의 없다는 뜻이니까요.

### 개발 비용과 유지보수 비용의 노력 분포 차이

개발과 유지보수는 **어느 단계에서 가장 많은 비용이 들어가는지**가 다릅니다:

| 구분 | 가장 비용이 많이 드는 단계 | 이유 |
|---|---|---|
| **개발** | 구현(코딩) 단계 | 실제로 소프트웨어를 만들어내는 단계 |
| **유지보수** | 분석(이해) 단계 | 기존 시스템을 파악하는 데 시간이 가장 많이 소요 |

유지보수에서 가장 비용이 많이 드는 것은 **처음에 시스템을 이해하는 것**입니다. 내가 처음 접하는 시스템, 문서도 없고 개발자도 없는 상황에서 코드를 파악하는 데 가장 많은 시간과 비용이 들어갑니다. 일단 이해가 되면 실제 수정과 테스트는 상대적으로 빠르게 진행됩니다.

## 👻 외계인 코드와 개발자의 책임감

### 외계인 코드란?

**외계인 코드(Alien Code)** 란, 개발한 사람도 없고, 문서도 없고, 설계도도 없어서 **유지보수가 사실상 불가능에 가까운 코드**를 말합니다. 마치 외계어로 쓰인 것처럼 아무도 이해할 수 없다는 의미에서 붙여진 이름입니다.

이런 코드가 만들어지는 원인은 보통 이렇습니다:
- 최초 개발자가 회사를 떠나면서 인수인계를 제대로 하지 않음
- 개발 당시 표준화나 문서화를 하지 않음
- 개발 방법론을 적용하지 않고 주먹구구식으로 개발함
- 코드에 주석이 없고, 변수명이 의미 없는 단어로 가득함

### 욕 먹지 않는 개발자가 되려면

실무에서 이런 일이 실제로 벌어집니다. 프리랜서 개발자가 프로젝트를 완성하고 떠났는데, 남아있는 팀이 코드를 보니:
- 모듈화가 전혀 안 되어 있고
- 중복 코드가 곳곳에 넘쳐나며
- 5줄로 짤 수 있는 SQL 쿼리(데이터베이스 질의문)가 30줄로 복잡하게 얽혀있고
- 변수명이 `a`, `b`, `c`, `temp1` 같은 의미 없는 이름들뿐이라면

남아있는 개발자들은 그 사람을 절대 다시 부르지 않을 것입니다. 그 사람과 다시 일하겠다는 회사도 없을 것입니다.

반대로, 코드가 깔끔하게 모듈화되어 있고, 자세한 주석이 달려 있으며, 문서화가 잘 되어 있다면? 그 개발자는 언제 어디서나 환영받습니다.

```python
# ❌ 나쁜 예시: 외계인 코드가 될 수 있는 코드
# 변수명이 의미 없고, 주석이 없으며, 한 함수에 모든 것이 뒤섞임
def f(a, b, c, d):
    x = a * b
    if x > c:
        return x - d
    else:
        y = x * 0.1
        return x + y
```

```python
# ✅ 좋은 예시: 미래의 유지보수 개발자를 배려한 코드
def calculate_order_total(unit_price: float, quantity: int,
                          discount_threshold: float, shipping_fee: float) -> float:
    """
    주문 금액 계산 함수
    
    Args:
        unit_price: 상품 단가 (원)
        quantity: 주문 수량
        discount_threshold: 할인 기준 금액 (이 금액 초과 시 배송비 무료)
        shipping_fee: 기본 배송비 (원)
    
    Returns:
        최종 결제 금액 (원)
    """
    # 1단계: 주문 소계 계산 (단가 × 수량)
    subtotal = unit_price * quantity

    # 2단계: 배송비 결정 (할인 기준 금액 초과 시 배송비 무료)
    if subtotal > discount_threshold:
        final_shipping_fee = 0  # 무료 배송 조건 충족
    else:
        additional_discount = subtotal * 0.1  # 10% 할인 적용
        final_shipping_fee = shipping_fee + additional_discount

    # 3단계: 최종 금액 반환
    return subtotal + final_shipping_fee
```

## 💡 주의사항 및 실무 팁

### 유지보수 개발자에게 필요한 역량

유지보수를 담당하는 개발자는 단순히 "오류를 고치는 사람"이 아닙니다. 실제로 유지보수 작업에는 분석, 설계, 구현, 테스트가 모두 포함되기 때문에, **다방면의 기술을 갖춘 올라운드 플레이어**여야 합니다.

유지보수 개발자에게 필요한 역량:
- **코드 읽기 능력**: 남이 작성한 코드를 빠르게 이해하는 능력
- **분석 도구 활용 능력**: 디버깅 도구, 코드 분석 도구 사용 능력
- **도메인 지식**: 해당 비즈니스 분야에 대한 이해
- **커뮤니케이션 능력**: 운영팀, 기획팀과 원활하게 소통하는 능력
- **변경 영향 분석 능력**: 코드 변경이 미칠 파급 효과를 예측하는 능력

:::tip 신규 회사 또는 프로젝트에 합류할 때 확인할 것
새로운 회사나 프로젝트에 유지보수 개발자로 합류하게 되면, 가장 먼저 이것을 물어보세요:
- **"표준화된 문서가 있나요?"** 없다면 유지보수의 고난이 예고된 것입니다.
- **"코드 스타일 가이드나 코딩 컨벤션이 있나요?"** 없다면 코드가 뒤죽박죽일 가능성이 높습니다.
- **"이전 버전과의 변경 이력이 관리되고 있나요?"** 없다면 형상관리가 안 된다는 신호입니다.
이 세 가지만 확인해도 그 회사/프로젝트의 유지보수 환경 수준을 파악할 수 있습니다.
:::

### 언제 유지보수를 포기하고 새로 개발해야 하는가?

시스템을 오래 사용하다 보면, 유지보수 비용이 새로 개발하는 비용보다 더 커지는 시점이 옵니다. 이 시점에서는 기존 시스템을 고치는 것보다 새로 만드는 것이 더 경제적입니다.

다음과 같은 상황이 겹치면 새로 개발을 고려해야 합니다:
- 유지보수 비용이 연간 개발 비용을 초과할 때
- 기존 기술 스택(사용 언어, 프레임워크)이 너무 구식이 되어 전문가를 구하기 어려울 때
- 화면 디자인과 사용자 경험(UX)이 완전히 시대에 뒤처졌을 때
- 시스템 구조 자체가 새로운 요구사항을 수용하기 어려운 구조일 때

## 📌 핵심 정리

이번 문서에서 학습한 핵심 개념들을 정리합니다:

- **유지보수(Maintenance)** 는 소프트웨어 개발 완료 후 폐기될 때까지 지속적으로 수행되는 생명 연장 작업이다.
- **소프트웨어는 생물**과 같아서, 환경 변화·사용자 요구 변화·예상치 못한 오류 등으로 끊임없이 변한다.
- **완전 유지보수(Perfective)** 는 새로운 기능 추가와 기존 기능 개선 작업이다.
- **적응 유지보수(Adaptive)** 는 운영체제·하드웨어·네트워크 등 환경 변화에 소프트웨어를 적응시키는 작업이다.
- **수정 유지보수(Corrective)** 는 오류·버그를 찾아 수정하는 A/S 작업이다.
- **예방 유지보수(Preventive)** 는 미래의 문제를 사전에 방지하기 위해 구조를 개선하는 작업이다.
- 유지보수 비중은 **기능 개선 50%, 환경 적응 25%, 하자 수정 20%, 예방 4%** 순이다.
- **표준화·문서화·모듈화**는 유지보수 비용을 줄이는 가장 효과적인 방법이다.
- **코드를 이해하지 않고 수정하면** 오류가 오히려 더 늘어날 수 있다.
- 유지보수에서 가장 많은 비용이 드는 단계는 **분석(시스템 이해)** 단계다.
- 개발 비용이 10억이면 **연간 유지보수 비용은 약 25%(2.5억)** 를 예상해야 한다.
- **SMI(소프트웨어 성숙도 지수)** 가 1에 가까울수록 소프트웨어가 안정적이다.
- **외계인 코드**는 개발자도 없고, 문서도 없어 유지보수가 사실상 불가능한 코드를 말한다.
- 유지보수 개발자는 반드시 **변경 의사결정에 적극적으로 참여**해야 한다.
- **처음부터 유지보수를 염두에 두고 개발**하는 것이 장기적으로 비용을 절감하는 최선의 방법이다.
