# 🏗️ 운영, 유지보수 및 폐기 — 소프트웨어 생명주기의 마지막 여정

소프트웨어는 단순히 코드를 짜서 배포하는 것으로 끝나지 않습니다. 사람에게 생명주기(태어남 → 성장 → 죽음)가 있듯이, 소프트웨어에도 **탄생부터 폐기에 이르는 전체 변화 과정**이 있습니다. 이 문서에서는 그 생명주기의 마지막 단계인 **운영, 유지보수, 그리고 폐기**가 무엇인지, 왜 중요한지, 어떤 원리로 이루어지는지를 아주 쉽고 자세하게 설명합니다.

:::info 이 문서에서 배우는 것
- 소프트웨어 생명주기(SDLC)란 무엇인가
- 운영(Deployment), 유지보수(Maintenance), 폐기(Retirement) 단계의 개념과 역할
- 폭포수 모형을 중심으로 각 단계가 어떻게 연결되는가
- 실제 프로젝트에서 이 단계들이 어떻게 적용되는가
:::

## 🌱 소프트웨어 생명주기(SDLC)란 무엇인가?

### 생명주기라는 개념, 왜 소프트웨어에도 적용할까?

우리가 일상에서 "생명주기"라고 하면 생물이 태어나고, 자라고, 늙고, 결국 죽음에 이르는 전체 과정을 떠올립니다. 꽃씨가 땅에 심어지면 싹이 트고, 줄기가 자라며, 꽃이 피고, 열매를 맺고, 결국 시들어 흙으로 돌아가는 것처럼요.

소프트웨어도 이와 똑같습니다. **소프트웨어 생명주기(Software Development Life Cycle, SDLC)** 란, 소프트웨어가 처음 기획·개발되어 세상에 나오고, 사용되다가 더 이상 필요 없어져 폐기되기까지의 **전체 변화 과정**을 말합니다.

> 💡 **SDLC**는 "System Development Life Cycle"의 약자로, 앞으로 실무에서나 교재에서 이 용어를 자주 마주치게 됩니다. SDLC라는 단어를 보면 **"소프트웨어가 탄생해서 사라지기까지의 순서와 단계"** 라고 이해하면 됩니다.

### 왜 생명주기를 정의해야 할까?

단순히 "개발하고 쓰면 되는 거 아닌가?"라고 생각하실 수 있습니다. 하지만 실제 소프트웨어 프로젝트는 **수억 원의 예산**, **수십 명의 인력**, **수개월에서 수년의 기간**이 투입되는 대형 작업입니다.

이럴 때 생명주기를 명확히 정의해야 하는 이유는 다음과 같습니다.

| 이유 | 설명 |
|---|---|
| **비용 산정** | 각 단계별로 얼마의 예산이 필요한지 계획할 수 있다 |
| **인력 배분** | 어느 단계에 몇 명의 인력이 필요한지 파악할 수 있다 |
| **일정 관리** | 개발 기간이 얼마나 걸릴지 예측할 수 있다 |
| **진행 상황 파악** | 지금 프로젝트가 어느 단계에 있는지 명확히 알 수 있다 |
| **위험 관리** | 각 단계의 문제를 조기에 발견하고 대처할 수 있다 |

마치 집을 짓는 것과 같습니다. 무턱대고 벽돌부터 쌓는 것이 아니라, **부지 선정 → 설계 → 기초 공사 → 골조 공사 → 마감 공사 → 입주** 순서로 단계를 나누어 각 단계에서 필요한 자원과 인력을 배분하는 것이죠. 이 순서가 없으면 집은 언제 완성될지, 비용이 얼마나 들지 아무도 모르게 됩니다.

### SDLC의 전체 단계 한눈에 보기

일반적인 소프트웨어 생명주기는 아래 순서로 이루어집니다.

```
1️⃣ 타당성 조사
    ↓
2️⃣ 요구 분석
    ↓
3️⃣ 설계
    ↓
4️⃣ 개발(구현)
    ↓
5️⃣ 테스트
    ↓
6️⃣ 운영
    ↓
7️⃣ 유지보수
    ↓
8️⃣ 폐기
```

이 문서에서는 이 전체 흐름을 이해한 뒤, 특히 **6번 운영 → 7번 유지보수 → 8번 폐기** 단계에 집중하여 깊이 있게 살펴보겠습니다.

## 🔍 운영·유지보수·폐기 이전 단계: 왜 필요한가?

### 타당성 조사 — "정말 만들어야 할까?"

소프트웨어를 만들기 전에 가장 먼저 해야 할 일은 **"이걸 만드는 것이 정말 타당한가?"** 를 검토하는 것입니다. 이를 **타당성 조사(Feasibility Study)** 라고 합니다.

예를 들어, 어떤 회사에서 기존 쇼핑몰을 모바일에 최적화된 새 시스템으로 교체하려 합니다. 개발 비용이 2억 원이 든다고 할 때, 무조건 "개발하자!"고 달려들어선 안 됩니다. 우리 회사의 매출 규모는 어느 정도인지, 고객층의 특성은 모바일 친화적인지, 경쟁사는 이미 어떻게 하고 있는지를 분석해야 합니다. 이 분석 결과 "개발할 필요성이 충분하다"는 결론이 나와야 비로소 다음 단계로 넘어갈 수 있습니다.

:::tip 타당성 조사가 중요한 이유
타당성 조사를 건너뛰고 바로 개발에 들어가면, 수억 원을 투자하고도 아무도 사용하지 않는 시스템이 만들어질 수 있습니다. 마치 사막 한가운데에 눈썰매장을 짓는 것처럼, 수요가 없는 곳에 공급을 만드는 실수가 됩니다.
:::

### 요구 분석 — "누가 무엇을 원하는가?"

타당성 조사에서 "개발하자"는 결론이 나오면 다음은 **요구 분석(Requirements Analysis)** 입니다. 쇼핑몰 시스템을 예로 들면, 이해관계자(stakeholder)는 매우 다양합니다.

- **내부 고객(직원)**: 관리자 페이지, 재고 관리, 자동 알림 기능 필요
- **외부 고객(소비자)**: 쉬운 검색, 간편 결제, 빠른 배송 조회 필요
- **판매자(벤더)**: 상품 등록·관리 기능 필요
- **결제사(PG사)**: 결제 API 연동 필요
- **배송 업체**: 운송장 정보 연동 필요
- **창고/물류팀**: 재고 수량 실시간 반영 필요

이 모든 요구사항을 수집하고 정리한 문서를 **요구사항 정의서** 라고 합니다. 이 문서가 완성되어야 다음 단계인 설계로 넘어갈 수 있습니다.

### 설계 — "어떻게 만들 것인가?"

요구사항이 확정되면 이제 **설계(Design)** 를 합니다. 집 지을 때 설계도면이 필요하듯, 소프트웨어도 설계도가 있어야 합니다. 설계는 크게 세 단계로 나뉩니다.

- **개념적 설계**: 전체 시스템의 큰 그림을 그리는 단계. "어떤 기능이 필요하다"는 수준
- **논리적 설계**: 각 기능이 어떻게 연결되는지, 데이터 흐름이 어떻게 되는지를 정의
- **물리적 설계**: 실제 코드로 구현하기 위한 구체적인 데이터베이스 구조, 모듈 구성 등을 정의

### 개발 — "코드로 만든다"

설계서가 완성되면 **개발팀** 으로 넘어갑니다. 개발자들은 설계서를 바탕으로 Java, Python, PHP 등의 프로그래밍 언어로 실제 코드를 작성합니다. 이 과정에서 **코딩(Coding)** 과 **디버깅(Debugging, 오류 수정)** 이 반복됩니다. 마치 조각가가 조각을 하다가 실수하면 다듬고, 또 조각하면서 완성품에 가까워지듯이요.

```python
# 예시: 간단한 회원가입 기능 구현 (개발 단계의 산출물인 소스 코드)

def register_user(username, password, email):
    # 입력값이 비어 있는지 확인 (기본 유효성 검사)
    if not username or not password or not email:
        return {"success": False, "message": "모든 항목을 입력해 주세요."}

    # 이메일 형식 간단 확인 (@ 기호 포함 여부)
    if "@" not in email:
        return {"success": False, "message": "올바른 이메일 형식을 입력해 주세요."}

    # 실제 데이터베이스 저장 로직 (여기서는 예시로 생략)
    # db.save(username, password, email)

    return {"success": True, "message": f"{username}님, 회원가입이 완료되었습니다!"}
```

### 테스트 — "제대로 작동하는가?"

개발이 끝나면 **테스트(Test)** 단계로 넘어갑니다. 테스트는 단순히 "실행되는가?"를 확인하는 것이 아니라, 다양한 수준에서 검증합니다.

| 테스트 종류 | 설명 | 비유 |
|---|---|---|
| **단위 테스트(Unit Test)** | 개별 모듈·기능 하나하나를 따로 테스트 | 자동차 부품 하나하나를 개별 점검 |
| **통합 테스트(Integration Test)** | 여러 모듈을 합쳐서 연결이 잘 되는지 테스트 | 부품을 조립해 전체가 맞는지 확인 |
| **시스템 테스트(System Test)** | 전체 시스템이 실제 환경에서 잘 동작하는지 테스트 | 완성된 자동차를 실제 도로에서 시운전 |
| **인수 테스트(Acceptance Test)** | 실제 사용자가 직접 사용해 보며 요구사항을 충족하는지 확인 | 차를 구매자에게 인도 전 최종 점검 |

인수 테스트까지 모두 통과하면 드디어 **운영(Deployment)** 단계로 넘어갑니다.

## 🚀 운영(Deployment) 단계 — 세상에 나오는 순간

### 운영이란 무엇인가?

**운영(Deployment, 혹은 배포)** 이란, 테스트까지 완료된 소프트웨어를 실제 사용자들이 사용할 수 있도록 **실제 서비스 환경에 올리는 것** 을 말합니다. 쇼핑몰을 예로 들면, 새롭게 리뉴얼된 쇼핑몰 사이트를 대대적으로 홍보하고 일반 고객들이 접속할 수 있도록 **공식 오픈(런칭)** 하는 것입니다.

> 🎉 오픈(런칭)은 개발 여정의 '졸업식'과 같습니다. 수개월간 준비한 결과물이 처음으로 세상과 만나는 순간이죠.

### 운영 단계에서 일어나는 일들

소프트웨어가 실제로 운영되기 시작하면 다음과 같은 활동이 이루어집니다.

**1. 시스템 배포(Deploy)**
개발 서버에서 작동하던 소프트웨어를 **실제 운영 서버(Production Server)** 에 올립니다. 이때 데이터베이스 설정, 네트워크 설정, 보안 설정 등이 실제 환경에 맞게 구성됩니다.

**2. 사용자 교육 및 안내**
내부 직원들에게 새 시스템 사용법을 교육하고, 사용자 매뉴얼을 제공합니다. 아무리 잘 만든 시스템이라도 사용자가 사용법을 모르면 무용지물이 됩니다.

**3. 초기 모니터링(Monitoring)**
시스템이 처음 운영될 때는 예상치 못한 문제가 발생할 가능성이 높습니다. 따라서 서버 상태, 오류 로그(error log, 시스템이 기록하는 오류 내역), 접속자 수, 응답 속도 등을 면밀히 관찰합니다.

:::warning 운영 초기에 특히 주의할 것
소프트웨어는 테스트 환경과 실제 운영 환경이 다를 수 있습니다. 실제로 수천 명의 사용자가 동시에 접속하거나, 예상치 못한 데이터가 입력될 때 새로운 오류가 발생할 수 있습니다. 운영 초기에는 개발팀과 운영팀이 긴밀히 협력하며 문제에 즉각 대응해야 합니다.
:::

### 소프트웨어의 특성: 왜 운영 중에도 오류가 생길까?

소프트웨어의 중요한 특성 중 하나는 **처음에는 문제없이 작동하다가, 사용하다 보면 예상하지 못했던 오류가 나타난다는 것** 입니다. 이는 마치 새 자동차도 처음에는 문제없어 보이지만, 실제로 도로를 달리다 보면 미처 발견하지 못한 결함이 드러나는 것과 같습니다.

그 이유는 다음과 같습니다.

- **다양한 사용자 행동**: 개발자가 예상하지 못한 방식으로 시스템을 사용하는 사용자가 반드시 존재합니다.
- **데이터 증가**: 초기에는 데이터가 적어 빠르지만, 데이터가 쌓일수록 성능 문제가 나타날 수 있습니다.
- **환경 변화**: 운영 체제(OS) 업데이트, 브라우저 버전 변경 등 외부 환경이 바뀌면 기존 소프트웨어가 오동작할 수 있습니다.
- **동시 사용자**: 소수가 테스트할 때는 괜찮았던 기능이 수천 명이 동시에 사용하면 서버 과부하로 멈출 수 있습니다.

이런 이유 때문에 운영 단계는 "한 번 올리고 끝"이 아니라, 지속적인 관찰과 관리가 필요한 **살아있는 단계** 입니다.

## 🔧 유지보수(Maintenance) 단계 — 시스템을 살아있게 만드는 과정

### 유지보수란 무엇인가?

**유지보수(Maintenance)** 는 운영 중인 소프트웨어에서 발생하는 오류를 수정하고, 새로운 요구사항을 반영하며, 성능을 개선하는 **지속적인 관리 활동** 입니다. 이는 소프트웨어 생명주기에서 가장 오랜 시간이 소요되는 단계이기도 합니다.

마치 건물을 지은 뒤에도 지속적으로 청소하고, 고장난 곳을 수리하고, 새로운 입주자의 요구에 맞게 인테리어를 바꾸는 것처럼, 소프트웨어도 한 번 만들고 끝이 아니라 **꾸준한 유지보수** 가 필요합니다.

> 📊 소프트웨어 개발 비용의 상당 부분이 유지보수에 사용됩니다. 즉, 잘 만들어진 소프트웨어라도 유지보수 없이는 금방 낡고 쓸모없어집니다.

### 유지보수의 종류

유지보수는 목적에 따라 크게 네 가지로 나뉩니다.

#### ① 수정 유지보수 (Corrective Maintenance)

운영 중에 발견된 **버그(bug, 오류)를 고치는** 유지보수입니다. 예를 들어, 쇼핑몰에서 특정 상품을 장바구니에 담으면 수량이 두 배로 올라가는 오류가 발견됐을 때, 이를 수정하는 것이 수정 유지보수입니다.

```python
# ❌ 잘못된 예시: 장바구니에 상품을 추가할 때 수량이 두 배로 증가하는 버그

def add_to_cart(cart, product_id, quantity):
    if product_id in cart:
        # 버그: 기존 수량 + 새로운 수량 + 기존 수량 (중복 계산)
        cart[product_id] += cart[product_id] + quantity
    else:
        cart[product_id] = quantity
    return cart

# 결과: 원래 수량 2, 추가 수량 1 → 5가 돼버리는 문제 발생
```

```python
# ✅ 올바른 예시: 수정 유지보수 후 정상 동작

def add_to_cart(cart, product_id, quantity):
    # 이미 장바구니에 있는 상품이라면 수량만 추가
    if product_id in cart:
        cart[product_id] += quantity  # 기존 수량에 새로운 수량만 더함
    else:
        # 처음 담는 상품이라면 새로 추가
        cart[product_id] = quantity
    return cart

# 결과: 원래 수량 2, 추가 수량 1 → 정상적으로 3이 됨
```

**코드 설명:**
- `cart`: 현재 장바구니 상태를 담고 있는 딕셔너리(key: 상품 ID, value: 수량)
- `product_id in cart`: 해당 상품이 이미 장바구니에 있는지 확인
- `cart[product_id] += quantity`: 기존 수량에 새로운 수량만 더함 (잘못된 예시는 기존 수량을 한 번 더 더하는 실수)
- `cart[product_id] = quantity`: 처음 담는 상품이면 수량을 그대로 설정

#### ② 적응 유지보수 (Adaptive Maintenance)

외부 환경 변화에 맞게 소프트웨어를 **수정·적응시키는** 유지보수입니다. 예를 들어, 운영 체제가 Windows 10에서 Windows 11로 바뀌었거나, 정부의 개인정보 보호법이 개정되어 시스템을 그에 맞게 수정해야 하는 경우입니다.

또 다른 예로, PHP로 개발했던 쇼핑몰 시스템을 Java로 전환하는 경우도 적응 유지보수의 일환입니다. 기술 트렌드가 바뀌거나, 더 효율적인 언어·프레임워크가 나왔을 때 기존 시스템을 새로운 환경에 맞게 재구성하는 것이죠.

#### ③ 완전화 유지보수 (Perfective Maintenance)

기존 기능의 **성능을 향상**하거나 **새로운 기능을 추가**하는 유지보수입니다. 고객이 "이런 기능이 있으면 좋겠다"는 새로운 요구를 제시하거나, 더 빠르고 편리하게 시스템을 개선할 때 이루어집니다.

예를 들어, 기존 쇼핑몰에 **AI 기반 상품 추천 기능** 을 추가하거나, 검색 속도를 높이기 위해 데이터베이스 쿼리(query, 데이터 조회 명령)를 최적화하는 것이 완전화 유지보수입니다.

#### ④ 예방 유지보수 (Preventive Maintenance)

현재는 문제가 없지만 **미래에 발생할 수 있는 문제를 예방**하기 위해 소프트웨어를 점검하고 개선하는 유지보수입니다. 마치 자동차의 엔진오일을 문제가 생기기 전에 정기적으로 교체하는 것처럼, 소프트웨어도 주기적으로 코드를 점검하고 정리합니다.

이를 **리팩토링(Refactoring, 코드 재정비)** 이라고도 하는데, 기능은 그대로 유지하면서 코드의 구조를 더 읽기 쉽고 유지보수하기 쉽게 개선하는 작업입니다.

```python
# ❌ 유지보수가 어려운 코드 (예방 유지보수 전)

def p(u, p):
    # 변수명이 너무 짧고 주석도 없어서 무슨 기능인지 알 수 없음
    if u == "admin" and p == "1234":
        return True
    return False
```

```python
# ✅ 유지보수가 쉬운 코드 (예방 유지보수 후 리팩토링)

def check_admin_login(username, password):
    """
    관리자 로그인을 검증하는 함수
    - username: 로그인 시도 사용자 이름
    - password: 로그인 시도 비밀번호
    - 반환값: 로그인 성공 시 True, 실패 시 False
    """
    # 관리자 계정 정보 (실제 서비스에서는 데이터베이스에서 조회해야 함)
    ADMIN_USERNAME = "admin"
    ADMIN_PASSWORD = "1234"

    # 입력된 사용자 정보와 관리자 정보를 비교
    if username == ADMIN_USERNAME and password == ADMIN_PASSWORD:
        return True  # 로그인 성공
    return False  # 로그인 실패
```

**코드 설명:**
- 함수 이름을 `p`에서 `check_admin_login`으로 변경: 이름만 봐도 어떤 기능인지 알 수 있음
- 매개변수(parameter, 함수에 전달하는 값)도 `u`, `p`에서 `username`, `password`로 명확하게 변경
- `docstring`(함수 설명 문서)을 추가하여 다른 개발자가 이 함수를 이해하기 쉽게 함
- 상수(변하지 않는 값)를 대문자 변수로 분리하여 나중에 값을 바꿀 때 한 곳만 수정하면 됨

### 유지보수는 전체 개발 비용의 핵심

많은 분들이 소프트웨어 개발에서 비용이 가장 많이 드는 단계를 "개발 단계"라고 생각하지만, 실제로는 **유지보수 단계** 가 전체 생명주기에서 비용과 시간을 가장 많이 차지합니다.

그 이유는 소프트웨어는 한 번 배포되고 나면 수년, 심지어 수십 년 동안 운영될 수 있기 때문입니다. 그 긴 시간 동안 오류 수정, 기능 추가, 환경 변화 대응이 끊임없이 이어집니다. 따라서 처음 설계할 때부터 **"유지보수하기 쉬운 코드"** 를 작성하는 것이 매우 중요합니다.

:::tip 좋은 코드란?
"좋은 코드"는 단순히 실행이 잘 되는 코드가 아닙니다. **6개월 뒤 다른 개발자(또는 미래의 나)가 봐도 이해할 수 있는 코드** 가 좋은 코드입니다. 명확한 변수명, 충분한 주석, 모듈화(기능별로 나누기)가 유지보수 비용을 크게 줄여줍니다.
:::

## ♻️ 폐기(Retirement) 단계 — 소프트웨어의 마지막 여정

### 폐기란 무엇인가?

**폐기(Retirement, 혹은 Phase-out)** 는 더 이상 사용되지 않거나, 새로운 시스템으로 교체될 소프트웨어를 **공식적으로 사용 중단하는 것** 입니다. 소프트웨어의 생명주기에서 마지막 단계에 해당합니다.

마치 오래된 자동차를 폐차하고 새 차를 구입하는 것처럼, 오래되어 유지보수 비용이 너무 많이 들거나, 더 이상 현재의 기술 환경을 지원하지 못하는 소프트웨어는 결국 폐기됩니다.

> 🗂️ 소프트웨어 폐기는 단순히 "삭제"가 아닙니다. 기존 데이터의 이관, 사용자 공지, 새 시스템으로의 전환 등 체계적인 절차가 필요합니다.

### 언제 폐기를 결정하는가?

소프트웨어 폐기는 다음과 같은 상황에서 결정됩니다.

**1. 기술 노후화(Technology Obsolescence)**
PHP로 개발된 시스템을 운영하다가, 최신 기술 트렌드에 맞게 Java나 Python으로 완전히 새로 개발하기로 결정한 경우입니다. 기존 PHP 시스템은 새 시스템이 안정화되는 시점에 폐기됩니다.

**2. 유지보수 비용 과다**
시스템이 너무 오래되어 코드가 복잡하고 얽혀 있어서, 작은 수정 하나에도 엄청난 시간과 비용이 드는 경우입니다. 이를 **레거시 시스템(Legacy System, 오래된 유산 시스템)** 이라고 부르는데, 이런 경우 유지보수를 계속하는 것보다 새로 만드는 것이 더 경제적일 수 있습니다.

**3. 비즈니스 환경 변화**
서비스 자체가 폐지되거나, 회사가 해당 사업을 철수하는 경우 관련 소프트웨어도 함께 폐기됩니다.

**4. 보안 취약점 해결 불가**
너무 오래된 시스템은 최신 보안 위협에 대응하기 어렵습니다. 보안 패치(patch, 취약점을 막는 수정 프로그램)가 더 이상 지원되지 않는 경우 폐기를 결정하기도 합니다.

### 폐기 시 반드시 해야 할 것들

:::warning 폐기는 신중하게!
소프트웨어를 폐기할 때는 단순히 서버를 끄는 것이 아닙니다. 아래 절차를 반드시 따라야 합니다.
:::

**① 데이터 이관(Data Migration)**
기존 시스템에 저장된 중요한 데이터를 새 시스템이나 안전한 저장소로 옮겨야 합니다. 고객 정보, 주문 내역, 거래 기록 등은 법적으로도 일정 기간 보관해야 합니다.

**② 사용자 공지**
시스템이 종료된다는 사실을 사용자에게 미리 충분히 알려야 합니다. 갑작스러운 서비스 종료는 사용자 신뢰를 크게 떨어뜨립니다.

**③ 문서화**
폐기되는 시스템의 구조, 데이터 형식, 운영 노하우 등을 문서로 남겨야 합니다. 나중에 문제가 생겼을 때 참조할 수 있도록요.

**④ 점진적 전환(Gradual Transition)**
한 번에 모든 기능을 전환하는 것은 위험합니다. 보통은 **기존 시스템과 새 시스템을 병행 운영** 하면서 점차 사용자를 새 시스템으로 이동시키는 전략을 사용합니다.

```
[점진적 전환 전략 예시]

Phase 1: 새 시스템 개발 완료 → 내부 직원만 새 시스템 테스트
Phase 2: 신규 사용자는 새 시스템, 기존 사용자는 기존 시스템 사용
Phase 3: 모든 사용자를 새 시스템으로 이동
Phase 4: 기존 시스템 완전 종료(폐기)
```

:::danger 이것만은 절대 하지 마세요
- **데이터 백업 없이 시스템 삭제**: 복구 불가능한 데이터 손실이 발생합니다.
- **사용자 공지 없는 갑작스러운 종료**: 법적 분쟁 및 신뢰도 하락으로 이어집니다.
- **문서화 없는 폐기**: 새 시스템 개발 시 중요한 비즈니스 로직(업무 규칙)이 사라져 다시 파악하는 데 막대한 비용이 듭니다.
:::

## 🏛️ 폭포수 모형과 운영·유지보수·폐기의 관계

### 폭포수 모형(Waterfall Model)이란?

지금까지 설명한 SDLC의 각 단계를 **순서대로, 한 방향으로** 진행하는 가장 전통적인 개발 방법론을 **폭포수 모형(Waterfall Model)** 이라고 합니다.

폭포에서 물이 위에서 아래로 흐르듯이, **이전 단계가 완전히 끝나야 다음 단계로 넘어갈 수 있고, 역방향으로 돌아갈 수 없습니다.**

```
계획 → 분석 → 설계 → 개발 → 테스트 → 운영 → 유지보수 → (폐기)
  ↓      ↓      ↓      ↓       ↓       ↓        ↓
(각 단계에서 산출물이 완성되어야 다음 단계로 진행 가능)
```

### 폭포수 모형에서 각 단계의 산출물

폭포수 모형의 핵심은 **각 단계마다 명확한 산출물(deliverable, 결과 문서나 결과물)이 나와야 한다는 것** 입니다.

| 단계 | 주요 산출물 |
|---|---|
| 계획 | 프로젝트 계획서, 제안서 |
| 분석 | 요구사항 정의서 |
| 설계 | 기본 설계서, 상세 설계서 |
| 개발 | 소스 코드, 단위 테스트 결과서 |
| 테스트 | 통합 테스트 결과서, 시스템 테스트 결과서, 인수 테스트 결과서 |
| 운영 | 운영 매뉴얼, 시스템 구성도 |
| 유지보수 | 변경 요청서, 수정 이력서 |

이 산출물이 나오지 않으면 다음 단계로 진행할 수 없습니다. 마치 건물의 설계도면 없이 공사를 시작할 수 없는 것처럼요.

### 폭포수 모형의 장점과 단점

폭포수 모형이 수십 년간 가장 많이 사용된 이유도 있고, 단점도 명확합니다.

**✅ 장점**

- **단순하고 이해하기 쉽다**: 단계가 명확하게 나뉘어 있어 팀 전체가 현재 어느 단계에 있는지 파악하기 쉽습니다.
- **단계별 점검이 가능하다**: 각 단계마다 산출물로 진행 상황을 확인할 수 있습니다.
- **문서화가 잘 된다**: 각 단계의 산출물이 문서로 남아 나중에 참고하기 좋습니다.
- **적용 사례가 많다**: 가장 오래된 방법론으로, 실제 적용 경험과 노하우가 풍부합니다.

**❌ 단점**

- **역방향 불가**: 개발 도중 요구사항이 바뀌어도 이전 단계로 돌아갈 수 없습니다.
- **중간 결과물 확인 불가**: 고객은 프로젝트가 완전히 끝나야 결과물을 볼 수 있습니다.
- **마지막에 발견된 오류는 수정 비용이 엄청나다**: "이게 내가 원하던 게 아닌데?"를 마지막에 발견하면 처음부터 다시 해야 합니다.
- **변화 대응이 어렵다**: IT 환경은 빠르게 변하는데, 폭포수 모형은 그 변화에 유연하게 대응하기 어렵습니다.

:::note 폭포수 모형이 여전히 많이 사용되는 이유
단점이 있음에도 불구하고 폭포수 모형은 오늘날에도 많은 프로젝트에서 활용됩니다. 요구사항이 명확하고 변경 가능성이 낮은 경우, 예산과 기간이 정해진 대형 공공 프로젝트, 또는 기존 시스템을 보완하는 경우에는 폭포수 모형이 효율적입니다.
:::

## ⚠️ 주의사항과 실무 팁

### 유지보수를 염두에 둔 개발이 중요하다

많은 신입 개발자들이 "일단 작동하면 된다"는 생각으로 코드를 작성합니다. 하지만 실무에서는 **"6개월 뒤 다른 사람이 이 코드를 수정해야 한다면?"** 을 항상 고려해야 합니다.

유지보수를 어렵게 만드는 흔한 실수들을 살펴보겠습니다.

```python
# ❌ 유지보수하기 나쁜 코드의 예시

def calc(a, b, c):
    # 주석 없음, 변수명 불명확
    x = a * 0.1
    y = b * 0.2
    z = x + y + c
    return z
```

```python
# ✅ 유지보수하기 좋은 코드의 예시

# 세금 및 배송비를 포함한 최종 결제 금액을 계산하는 함수
def calculate_total_price(product_price, shipping_fee, discount_amount):
    """
    최종 결제 금액 계산
    - product_price: 상품 가격 (원)
    - shipping_fee: 배송비 (원)
    - discount_amount: 할인 금액 (원)
    - 반환값: 세금 포함 최종 결제 금액 (원)
    """
    # 세금 비율 (부가가치세 10%)
    TAX_RATE = 0.1

    # 배송비 할인 비율 (20% 할인 적용)
    SHIPPING_DISCOUNT_RATE = 0.2

    # 상품 가격에 세금 적용
    price_with_tax = product_price * (1 + TAX_RATE)

    # 배송비에서 할인 적용
    discounted_shipping = shipping_fee * (1 - SHIPPING_DISCOUNT_RATE)

    # 최종 금액 = (세금 포함 상품가) + (할인된 배송비) - 할인금액
    total = price_with_tax + discounted_shipping - discount_amount

    return total
```

**코드 설명:**
- 함수명 `calculate_total_price`: 이름만 봐도 "최종 가격 계산"임을 알 수 있음
- 매개변수명 `product_price`, `shipping_fee`, `discount_amount`: 각각 상품 가격, 배송비, 할인 금액을 의미
- `TAX_RATE = 0.1`: 세금 비율을 상수로 분리. 세금 정책이 바뀌어도 이 한 줄만 수정하면 됨
- `docstring`: 함수의 목적, 입력값, 반환값을 명확히 설명
- 각 계산 단계마다 중간 변수(`price_with_tax`, `discounted_shipping`)를 만들어 의미를 명확히 함

### 운영 중 모니터링의 중요성

소프트웨어가 운영되는 동안 **지속적인 모니터링** 은 선택이 아닌 필수입니다. 모니터링을 통해 문제가 커지기 전에 미리 감지하고 대응할 수 있습니다.

```python
# 실무에서 자주 사용하는 간단한 로깅(logging, 기록 남기기) 예시

import logging  # 파이썬 내장 로깅 모듈 불러오기
from datetime import datetime  # 날짜·시간 처리 모듈 불러오기

# 로그 설정: 파일에 기록하고, 날짜·시간·메시지를 함께 저장
logging.basicConfig(
    filename="app.log",  # 로그를 저장할 파일 이름
    level=logging.INFO,  # INFO 수준 이상의 로그만 기록
    format="%(asctime)s - %(levelname)s - %(message)s"  # 로그 형식 설정
)

def process_order(order_id, user_id, amount):
    """
    주문을 처리하는 함수
    - order_id: 주문 번호
    - user_id: 주문자 ID
    - amount: 결제 금액
    """
    # 주문 시작 시점 로그 기록 (나중에 문제 발생 시 추적 가능)
    logging.info(f"주문 처리 시작 - 주문번호: {order_id}, 사용자: {user_id}, 금액: {amount}원")

    try:
        # 실제 주문 처리 로직 (예시)
        if amount <= 0:
            # 오류 상황 로그 기록
            logging.error(f"잘못된 결제 금액 - 주문번호: {order_id}, 금액: {amount}원")
            return False

        # 주문 성공 로그 기록
        logging.info(f"주문 처리 완료 - 주문번호: {order_id}")
        return True

    except Exception as e:
        # 예상치 못한 오류 발생 시 로그 기록
        logging.error(f"주문 처리 중 오류 발생 - 주문번호: {order_id}, 오류: {str(e)}")
        return False
```

**코드 설명:**
- `logging.basicConfig()`: 로그 설정을 초기화. 파일명, 기록 수준, 형식을 지정
- `logging.INFO`: 일반적인 운영 정보를 기록하는 수준
- `logging.error()`: 오류가 발생했을 때 기록하는 함수. 운영팀이 문제를 빠르게 파악할 수 있게 해줌
- `try-except`: 예상치 못한 오류가 발생해도 프로그램이 멈추지 않고 오류를 처리하는 구조

:::tip 실무 꿀팁: 로그는 보험이다
로그(log, 시스템이 기록하는 작업 이력)는 문제가 생겼을 때 "언제, 어디서, 무슨 일이 있었는지"를 추적할 수 있게 해주는 일종의 블랙박스입니다. 귀찮더라도 중요한 작업마다 로그를 남겨두면, 나중에 문제를 해결할 때 큰 도움이 됩니다.
:::

### 폐기 계획은 미리 세워라

많은 팀들이 새 시스템 개발에만 집중하고 기존 시스템 폐기 계획을 늦게 세우는 실수를 합니다. 폐기 계획은 새 시스템 개발과 **동시에** 수립하는 것이 좋습니다.

- 어떤 데이터를 어떻게 이관할 것인가?
- 사용자에게 얼마나 미리 공지할 것인가?
- 기존 시스템과 새 시스템을 얼마나 병행 운영할 것인가?
- 폐기 후 발생할 수 있는 문의는 어떻게 처리할 것인가?

이러한 질문들에 대한 답을 미리 준비해 두어야 혼란 없이 부드럽게 전환할 수 있습니다.

## 📝 핵심 정리

이 문서에서 배운 핵심 개념을 정리합니다.

- **소프트웨어 생명주기(SDLC)** 는 소프트웨어가 탄생해서 폐기되기까지의 전체 변화 과정이다.
- SDLC는 **타당성 조사 → 요구 분석 → 설계 → 개발 → 테스트 → 운영 → 유지보수 → 폐기** 순서로 진행된다.
- **운영(Deployment)** 은 완성된 소프트웨어를 실제 사용자에게 서비스하는 단계로, 지속적인 모니터링이 필요하다.
- **유지보수(Maintenance)** 는 운영 중 발생하는 오류 수정, 기능 추가, 성능 향상 등의 지속적 관리 활동이다.
- 유지보수는 수정·적응·완전화·예방의 네 가지 유형으로 나뉜다.
- **폐기(Retirement)** 는 더 이상 필요 없거나 교체되는 소프트웨어를 공식 종료하는 단계로, 데이터 이관과 사용자 공지가 필수다.
- **폭포수 모형(Waterfall Model)** 은 SDLC를 선형 순차적으로 실행하는 가장 전통적인 개발 방법론이다.
- 폭포수 모형은 역방향이 불가능하므로, **앞 단계의 산출물이 정확해야** 전체 프로젝트가 성공할 수 있다.
- 유지보수 비용을 줄이려면 처음 개발할 때부터 **명확한 변수명, 충분한 주석, 모듈화** 를 실천해야 한다.
- 로그(log)는 운영 중 문제를 추적하는 **블랙박스** 역할을 하므로 반드시 기록해야 한다.
- 폐기 계획은 새 시스템 개발과 **동시에** 수립하여 점진적 전환 전략을 마련해야 한다.
