# 🏗️ 설계, 개발 및 테스트 — 소프트웨어는 어떻게 만들어지는가?

소프트웨어는 그냥 "코딩"만 해서 만들어지는 것이 아닙니다. 집을 지을 때 설계도 없이 벽돌부터 쌓는 사람이 없듯이, 소프트웨어도 치밀한 계획과 설계, 그리고 엄격한 테스트를 거쳐야만 비로소 완성됩니다. 이 문서에서는 소프트웨어가 어떤 단계를 거쳐 탄생하고, 각 단계에서 무슨 일이 일어나는지, 그리고 대표적인 개발 모형들은 어떤 특징을 갖는지 초보자도 이해할 수 있게 아주 자세하게 설명합니다.

---

## 🌱 소프트웨어 생명주기(SDLC)란 무엇인가?

### 생명주기라는 개념이 왜 필요한가?

우리 인간도 태어나서 유아기, 청소년기, 성인기, 노년기를 거쳐 생을 마감합니다. 이처럼 어떤 존재가 탄생해서 사망에 이르기까지 겪는 전체 변화 과정을 **생명주기(Life Cycle)** 라고 부릅니다. 소프트웨어도 마찬가지입니다. 어느 날 "이런 프로그램이 필요하다"는 아이디어에서 시작해서, 기획·설계·개발·테스트·운영 단계를 거쳐, 결국 더 이상 필요 없어지면 폐기됩니다. 이 전체 흐름을 **소프트웨어 생명주기(Software Development Life Cycle, SDLC)** 라고 합니다.

> **SDLC(Software Development Life Cycle)** 는 소프트웨어가 탄생해서 폐기될 때까지의 전체 변화 과정을 단계별로 정리한 것입니다.

SDLC가 중요한 이유는 단순히 "순서를 나열한 것"이 아니기 때문입니다. 실제 현장에서 프로젝트를 진행할 때 **예산이 얼마나 필요한지, 개발 기간은 얼마나 잡아야 하는지, 몇 명의 개발자가 투입되어야 하는지** 같은 자원 계획의 기본 골격이 바로 이 SDLC에서 나옵니다. 단계가 명확히 구분되어 있기 때문에 프로젝트 진행 상황을 한눈에 파악하기도 훨씬 쉽습니다.

:::info SDLC라는 약자를 기억하세요
앞으로 IT 현장이나 교재에서 **SDLC** 라는 약자를 자주 보게 됩니다. 이것이 나오면 "아, 소프트웨어 생명주기, 즉 개발 단계의 순서와 변화 과정을 말하는구나"라고 이해하시면 됩니다.
:::

### SDLC의 주요 단계 한눈에 보기

아래 표는 소프트웨어 생명주기의 주요 단계와 각 단계에서 하는 일을 정리한 것입니다.

| 단계 | 이름 | 주요 활동 | 산출물(결과물) |
|---|---|---|---|
| 1 | 타당성 조사 | 개발 필요성, 예산·기간 검토 | 타당성 분석 보고서 |
| 2 | 요구 분석 | 고객 요구사항 수집 및 분석 | 요구사항 정의서 |
| 3 | 설계 | 시스템 구조·화면·DB 설계 | 기본 설계서, 상세 설계서 |
| 4 | 개발(구현) | 프로그래밍(코딩), 디버깅 | 소스 코드 |
| 5 | 테스트 | 단위·통합·시스템·인수 테스트 | 각종 테스트 결과서 |
| 6 | 운영 | 서비스 오픈 및 실제 사용 | 운영 중인 시스템 |
| 7 | 유지보수 및 폐기 | 오류 수정, 기능 개선, 최종 폐기 | 수정된 시스템 또는 폐기 |

---

## 🔍 각 단계 자세히 살펴보기

### 1단계: 타당성 조사 — "정말 이 프로젝트를 해야 할까?"

타당성 조사는 소프트웨어 개발을 시작하기 **전에** 반드시 거치는 첫 번째 관문입니다. 마치 식당을 차리기 전에 "이 동네에 식당이 잘 될까? 경쟁 식당은 얼마나 많지? 초기 투자금 대비 수익이 날까?" 를 먼저 따져보는 것과 같습니다.

예를 들어 어떤 회사에서 "기존 쇼핑몰을 완전히 새롭게 모바일 최적화 버전으로 개발하자"는 아이디어가 나왔다고 해봅시다. 개발비가 약 2억 원, 기간은 1년이 예상됩니다. 이때 바로 "개발자 뽑아, 시작해!" 라고 하면 안 됩니다. 먼저 이런 질문들을 던져야 합니다.

- 우리 고객 중 모바일 사용자 비율이 얼마나 되는가?
- 경쟁사는 이미 모바일 서비스를 제공하고 있는가?
- 2억 원을 투자했을 때 기대 수익이 투자금을 넘는가?
- 현재 우리 팀의 기술력으로 1년 안에 완성 가능한가?

이런 것들을 꼼꼼히 따져보는 과정이 바로 **타당성 조사**입니다. 타당성 조사 결과 "해볼 만하다"는 결론이 나와야 비로소 다음 단계로 넘어갈 수 있습니다.

### 2단계: 요구 분석 — "고객이 진짜 원하는 게 뭐지?"

타당성 조사를 통과했다면, 이제 "무엇을 만들어야 하는가?"를 정밀하게 파악해야 합니다. 이것이 **요구 분석(Requirements Analysis)** 단계입니다.

쇼핑몰 예시를 계속 들어보겠습니다. 쇼핑몰 하나를 만들려면 정말 다양한 이해관계자(이 시스템을 사용하거나 관련된 모든 사람들)의 요구를 수집해야 합니다.

- **내부 고객(직원)**: 관리자 페이지는 어떻게 구성해야 편한지, 재고 관리는 어떻게 자동화할 수 있는지
- **일반 고객(소비자)**: 위치 기반 추천 기능, 쉬운 결제 과정, 배송 추적 기능
- **판매자(벤더)**: 상품 등록·수정이 편리한 인터페이스, 정산 기능
- **결제사(PG사, Payment Gateway)**: 이니시스 등 결제 시스템과의 연동 방식
- **배송 업체**: 주문 정보를 실시간으로 연동하는 API(연결 통로) 방식
- **물류 창고 팀**: 재고 수량 자동 반영, 출고 지시 시스템

이렇게 다양한 고객들의 요구사항을 모두 수집하고 정리한 결과물이 바로 **요구사항 정의서**입니다. 이 문서가 있어야 다음 단계인 설계가 가능합니다.

:::warning 요구사항 분석이 부실하면 전체가 무너진다
요구사항 정의서가 부정확하거나 빠진 내용이 많으면, 나중에 개발이 거의 끝난 시점에서 "이게 우리가 원하던 게 아닌데요?" 라는 상황이 발생합니다. 이 경우 처음부터 다시 만들어야 할 수도 있어 엄청난 비용과 시간이 낭비됩니다.
:::

### 3단계: 설계 — "어떻게 만들 것인가?"

요구사항이 정해지면 이제 본격적인 **설계(Design)** 단계에 들어갑니다. 집을 짓기 전에 건축 설계도를 먼저 그리는 것처럼, 소프트웨어도 설계도가 없으면 개발에 착수할 수 없습니다.

설계는 크게 **기본 설계(개념적·논리적 설계)** 와 **상세 설계(물리적 설계)** 로 나뉩니다.

- **기본 설계**: 사용자가 어떤 화면을 보게 될지, 어떤 버튼을 눌렀을 때 어떤 기능이 동작하는지, 전체적인 시스템 구조가 어떻게 될지를 큰 그림으로 잡는 단계입니다. 이 시점에서는 주로 기획팀이나 시스템 분석팀이 주도합니다.
- **상세 설계**: 실제 개발자가 코드를 작성할 수 있을 정도로 구체적인 내용을 담습니다. 데이터베이스(DB) 구조, 자료 구조(데이터를 어떻게 저장하고 관리할지), 데이터 흐름도, 모듈(기능 단위) 간의 관계 등이 포함됩니다.

상세 설계서가 완성되면 이 문서를 가지고 개발팀으로 넘어갑니다.

### 4단계: 개발(구현) — "실제로 코드를 작성한다"

설계서를 받은 개발팀은 이제 **프로그래밍 언어(Java, Python, PHP, Visual C# 등)** 를 사용해서 실제 소스 코드를 작성합니다. 이 작업을 **코딩(Coding)** 또는 **구현(Implementation)** 이라고 합니다.

개발 과정에서 중요한 개념이 하나 있습니다. 바로 **디버깅(Debugging)** 입니다. 디버깅은 코드를 실행했을 때 오류(버그)가 발생하면 그 원인을 찾아서 수정하는 작업입니다. 마치 요리하다가 간이 맞지 않으면 소금을 더 넣거나 빼면서 맛을 조정하는 것처럼, 개발자는 코드를 짜다가 오류가 나면 디버깅으로 오류를 잡고, 다시 개발을 이어가고, 또 오류가 나면 다시 디버깅하는 과정을 반복합니다.

```python
# ✅ 올바른 예시: 함수를 모듈 단위로 작성하고 주석을 달아 디버깅이 용이하게 함
def calculate_discount(price, discount_rate):
    """
    상품 가격과 할인율을 받아 최종 가격을 반환하는 함수
    :param price: 원래 상품 가격 (정수 또는 실수)
    :param discount_rate: 할인율 (0.0 ~ 1.0 사이의 실수, 예: 0.1은 10% 할인)
    :return: 할인 적용된 최종 가격
    """
    if discount_rate < 0 or discount_rate > 1:
        # 할인율이 유효 범위를 벗어난 경우 오류 메시지 출력
        raise ValueError("할인율은 0과 1 사이여야 합니다.")
    
    final_price = price * (1 - discount_rate)  # 최종 가격 계산
    return final_price  # 계산된 최종 가격 반환

# 함수 호출 예시
result = calculate_discount(10000, 0.1)  # 10,000원에서 10% 할인
print(f"최종 가격: {result}원")          # 출력: 최종 가격: 9000.0원
```

```python
# ❌ 잘못된 예시: 주석도 없고 오류 처리도 없어 디버깅이 어려움
def calc(p, d):
    return p * (1 - d)  # 할인율 범위 검증 없음, 변수명 의미 불명확
```

:::tip 좋은 코드는 읽기 쉬운 코드
코드를 작성할 때는 나중에 자신이나 동료가 다시 읽었을 때 쉽게 이해할 수 있도록 **변수 이름을 명확하게 짓고, 한국어 주석을 충분히 달아두는** 습관을 들이세요. 디버깅 시간을 크게 줄일 수 있습니다.
:::

### 5단계: 테스트 — "제대로 만들었는지 확인한다"

개발이 끝났다고 바로 서비스를 오픈하면 안 됩니다. 마치 자동차 공장에서 차를 만들고 나서 주행 테스트를 반드시 거치는 것처럼, 소프트웨어도 체계적인 테스트 과정이 필수입니다. 테스트는 크게 세 단계로 나뉩니다.

**① 단위 테스트(Unit Test)**

가장 작은 단위인 **모듈(특정 기능 하나)** 별로 진행하는 테스트입니다. 예를 들어 쇼핑몰에서 "SMS 발송 기능", "주문 처리 기능", "결제 기능"을 각각 따로따로 테스트합니다. 자동차로 치면 엔진, 브레이크, 핸들을 각각 개별적으로 점검하는 것과 같습니다.

**② 통합 테스트(Integration Test)**

각각의 모듈을 단위 테스트한 뒤, 이 모듈들을 하나로 합쳐서(통합해서) 서로 제대로 연동되는지 확인하는 테스트입니다. 엔진이 따로 멀쩡하고, 브레이크도 따로 멀쩡해도, 합쳐놓으면 문제가 생길 수 있습니다. 이 단계에서 그런 문제를 찾아냅니다.

**③ 시스템 테스트(System Test)**

전체 시스템을 실제 서버 환경에 올려서 **네트워크 속도, 성능, 데이터베이스 처리 속도** 등 실제 운영 환경과 동일한 조건에서 테스트합니다. "성능은 괜찮은가?", "동시에 1,000명이 접속해도 견딜 수 있는가?" 같은 것들을 확인합니다.

**④ 인수 테스트(Acceptance Test)**

개발팀이 아닌, **실제 사용자(각 부서 담당자)** 가 직접 시스템을 사용해보며 자신들의 요구사항이 제대로 반영됐는지 확인하는 마지막 테스트입니다. 이 테스트를 통과해야 비로소 시스템을 공식적으로 인수받고 서비스를 오픈할 수 있습니다.

| 테스트 종류 | 수행 주체 | 테스트 범위 | 확인 내용 |
|---|---|---|---|
| 단위 테스트 | 개발자 | 모듈(기능) 하나 | 개별 기능이 정상 동작하는가? |
| 통합 테스트 | 개발팀 | 여러 모듈 연결 | 모듈 간 연동이 정상인가? |
| 시스템 테스트 | QA팀 | 전체 시스템 | 성능·네트워크·DB가 정상인가? |
| 인수 테스트 | 실제 사용자 | 전체 시스템 | 요구사항이 실제로 충족됐는가? |

### 6단계: 운영 및 유지보수 — "오픈 이후에도 계속 관리한다"

모든 테스트를 통과하면 드디어 서비스를 오픈(런칭)합니다. 그런데 소프트웨어는 아무리 꼼꼼하게 테스트해도 실제 사용자가 쓰다 보면 **예상하지 못했던 오류** 가 발생할 수 있습니다. 이는 소프트웨어의 본질적인 특성입니다. 그래서 오픈 이후에도 지속적으로 오류를 수정하고, 고객의 새로운 요구를 반영하고, 기능을 개선하는 **유지보수(Maintenance)** 작업이 필요합니다.

그러다 시간이 지나면서 기술 자체가 너무 낡아버리거나(예: PHP로 개발된 시스템을 Java로 완전히 새로 개발), 더 이상 서비스가 필요 없어지면 시스템을 **폐기(Retirement)** 합니다. 이것이 SDLC의 마지막 단계입니다.

## 🏛️ 폭포수 모형(Waterfall Model) — 가장 전통적인 개발 방식

### 폭포수 모형이란?

**폭포수 모형(Waterfall Model)** 은 소프트웨어 생명주기의 각 단계를 **위에서 아래로, 한 방향으로만 순서대로 진행하는** 가장 오래되고 가장 널리 사용되는 개발 방법론입니다.

폭포를 상상해보세요. 물은 위에서 아래로만 떨어집니다. 아래에서 위로 거슬러 올라가는 폭포는 존재하지 않습니다. 폭포수 모형도 마찬가지입니다. **계획 → 분석 → 설계 → 개발 → 테스트 → 운영 → 유지보수** 의 순서로 한 방향으로만 흘러가며, 한번 지나간 단계로 되돌아가는 것이 원칙적으로 허용되지 않습니다.

> **폭포수 모형**은 소프트웨어 개발 프로세스에 체계적인 원리를 도입한 첫 번째 방법론으로, **선형 순차 모형**이라고도 불립니다.

### 폭포수 모형의 단계별 프로세스

```
[계획] → [분석] → [설계] → [개발(구현)] → [테스트] → [운영] → [유지보수]
  ↓          ↓        ↓           ↓              ↓          ↓          ↓
프로젝트    요구사항   설계서      소스코드       테스트     서비스     오류수정/
계획서      정의서  (기본+상세)               결과서       오픈      기능개선
```

각 단계를 구체적으로 살펴보겠습니다.

**계획 단계**: 프로젝트를 왜 해야 하는지 근거를 담은 **제안서(프로젝트 계획서)** 를 작성합니다. "현재 우리 고객의 70%가 모바일 사용자인데 우리 서비스는 PC 중심이라 경쟁력이 떨어진다. 따라서 모바일 최적화 쇼핑몰 개발이 시급하다. 예산 2억 원, 기간 1년이 필요하다." 같은 내용이 이 문서에 담깁니다. 이것이 경영진의 승인을 받아야 다음 단계로 넘어갑니다.

**분석 단계**: 승인이 떨어지면 이제 요구사항을 상세히 분석합니다. 내부 직원, 일반 고객, 판매자, 결제사, 배송사 등 모든 이해관계자의 요구를 수집해서 **요구사항 정의서** 를 작성합니다.

**설계 단계**: 요구사항 정의서를 바탕으로 기본 설계서와 상세 설계서를 만듭니다. 화면 구성, DB 구조, 모듈 간의 관계 등이 모두 이 설계서에 담깁니다.

**개발 단계**: 상세 설계서를 받은 개발팀이 Java, Python, PHP 등의 언어로 코딩을 시작합니다. 코딩과 디버깅을 반복하면서 모듈 하나씩 완성해 나갑니다.

**테스트 단계**: 단위 테스트 → 통합 테스트 → 시스템 테스트 → 인수 테스트 순으로 진행합니다. 각 테스트마다 결과서가 문서로 남습니다.

**운영 및 유지보수 단계**: 서비스를 오픈하고 실제 운영하면서 발생하는 문제를 수정하고, 고객의 새 요구를 반영합니다.

### 폭포수 모형의 장점과 단점

폭포수 모형은 전통적인 방식이지만 여전히 많은 현장에서 사용됩니다. 하지만 단점도 분명히 존재합니다.

**✅ 장점**

- **단순하고 이해하기 쉽다**: 단계가 명확하게 구분되어 있어 팀원 모두가 현재 어느 단계에 있는지 쉽게 파악할 수 있습니다.
- **단계별 점검이 가능하다**: 각 단계가 끝날 때마다 산출물(결과물)을 확인하므로 품질 관리가 용이합니다.
- **적용 사례가 많다**: 수십 년 동안 검증된 방법론이라 레퍼런스가 풍부합니다.
- **기존 시스템 보완에 적합하다**: 이미 잘 알고 있는 시스템을 개선할 때 특히 효과적입니다.

**❌ 단점**

- **되돌아갈 수 없다**: 테스트 단계에서 문제가 발견되어도 분석 단계로 되돌아가기 어렵습니다. 폭포는 거꾸로 흐르지 않으니까요.
- **중간 결과물을 볼 수 없다**: 고객은 프로젝트가 완전히 끝나기 전까지 결과물을 확인할 수 없습니다. 마지막에 보니 "이게 아닌데요?" 하면 처음부터 다시 해야 합니다.
- **새로운 요구사항 반영이 어렵다**: 개발 중에 고객이 새로운 기능을 추가해달라고 해도, 이미 지나간 단계를 되돌릴 수 없어 반영이 거의 불가능합니다.
- **수정 비용이 엄청나다**: 후반 단계에서 오류가 발견될수록 수정 비용이 기하급수적으로 증가합니다.

:::danger 앞 단계 산출물이 부실하면 재앙이 온다
폭포수 모형에서는 **앞 단계의 산출물이 정확하고 완전해야** 합니다. 요구사항 정의서가 부정확하면 설계가 잘못되고, 설계가 잘못되면 개발도 잘못되고, 결국 테스트 단계나 인수 단계에서 전부 뒤집어야 하는 최악의 상황이 발생합니다.
:::

## 🔄 프로토타이핑 모형(Prototyping Model) — 먼저 샘플을 만들어보자

### 프로토타이핑 모형이 왜 나왔는가?

폭포수 모형의 가장 큰 문제점은 "고객이 최종 결과물을 프로젝트가 끝날 때까지 볼 수 없다"는 것입니다. 그래서 나온 아이디어가 바로 **"미리 샘플을 만들어서 고객에게 보여주고 피드백을 받자"** 입니다. 이것이 **프로토타이핑 모형(Prototyping Model)** 의 핵심입니다.

프로토타입(Prototype)이란 **완성품이 아닌 견본품(샘플)** 을 말합니다. 마치 건축에서 실제 건물을 짓기 전에 작은 모형을 만들어 보거나, 자동차 회사에서 신차 출시 전에 클레이 모델(점토 모형)을 만들어보는 것과 같습니다.

> 프로토타이핑 모형은 **개발자가 만들어야 할 시스템의 샘플을 사전에 미리 만들어 고객에게 보여주고, 고객의 피드백을 받아 요구사항을 더 정확하게 파악하는** 방법론입니다.

### 프로토타이핑 모형의 진행 과정

프로토타이핑 모형은 다음과 같이 순환하는(반복하는) 구조를 가집니다.

```
[요구사항 수집] → [프로토타입 설계] → [프로토타입 개발]
        ↑                                        ↓
   [정제/수정]   ←    [고객 평가/피드백]   ←   [고객에게 시연]
        ↓
  [최종 프로토타입 확정] → [실제 제품 개발]
```

고객에게 프로토타입을 보여주고 "이 버튼을 눌렀을 때 이렇게 동작하면 될까요?"라고 물어봅니다. 고객이 "대략 맞는데, 이 부분은 좀 다르게 해줬으면 좋겠어요"라고 하면 그 피드백을 반영해서 프로토타입을 수정합니다. 이 과정을 반복하면서 최종 프로토타입이 완성되면, 그것을 기반으로 실제 제품 개발을 시작합니다.

### 프로토타이핑 모형의 장점과 단점

**✅ 장점**

- **요구사항을 정확하게 파악할 수 있다**: 고객이 직접 보고 느끼면서 피드백을 주기 때문에 오해나 착오가 줄어듭니다.
- **생각하지 못한 기능을 발견할 수 있다**: 프로토타입을 보면서 "이런 기능도 있으면 좋겠는데요?"라는 새로운 아이디어가 나올 수 있습니다.
- **의사소통 도구로 활용된다**: 말로 설명하기 어려운 내용을 직접 보여줌으로써 개발팀과 고객 사이의 소통이 원활해집니다.
- **개발 타당성을 미리 검증할 수 있다**: 실제로 만들기 전에 기술적으로 가능한지 미리 확인할 수 있습니다.

**❌ 단점**

- **산출물과 일정표가 없다**: 반복 수정 과정에서 공식 문서가 제대로 만들어지지 않을 수 있습니다.
- **기능이 제한적이고 신뢰성이 낮다**: 빠르게 만든 샘플이기 때문에 성능이나 완성도가 낮습니다.
- **고객이 완성품으로 착각할 수 있다**: 프로토타입을 너무 잘 만들면 고객이 "이게 거의 완성된 거 아닌가요?" 라고 오해하여 기대치가 지나치게 높아질 수 있습니다. 반대로 너무 허술하면 신뢰를 잃습니다.
- **적절한 수준의 완성도 조절이 어렵다**: 너무 좋아도, 너무 나빠도 문제입니다.

:::tip 프로토타입의 완성도는 "적당히"가 핵심
프로토타입은 실제 기능의 핵심 흐름을 보여줄 수 있을 정도의 완성도면 충분합니다. "이 버튼을 누르면 이런 화면이 나온다"는 수준의 인터페이스(화면) 중심으로 만드는 것이 일반적입니다.
:::

## 🌀 나선형 모형, V 모형, 점증적 모형, UP 모형 — 다양한 개발 방법론

### 나선형 모형(Spiral Model) — 위험을 줄이며 나선처럼 발전

**나선형 모형(Spiral Model)** 은 폭포수 모형의 **체계적인 단계 구조**와 프로토타이핑 모형의 **반복적인 피드백 구조**를 합쳐서 만든 방법론입니다. 위험(Risk)을 단계적으로 줄여나가면서 시스템을 점점 발전시키는 것이 핵심입니다.

나선이란 달팽이 껍데기처럼 중심에서 시작해서 바깥으로 점점 커지는 모양입니다. 나선형 모형도 이와 같습니다. 처음에는 작게 시작해서 매 사이클마다 계획 → 위험 분석 → 개발(프로토타입 제작) → 평가를 반복하면서 시스템이 점점 커지고 완성도가 높아집니다.

**나선형 모형의 한 사이클:**

1. **계획(Planning)**: 목표, 기능, 제약 조건을 결정합니다.
2. **위험 분석(Risk Analysis)**: 우선순위와 위험 요소를 분석하고, 프로젝트를 계속 진행할지 판단합니다.
3. **개발 및 검증(Engineering)**: 프로토타입을 만들거나 실제 개발을 진행합니다.
4. **평가(Evaluation)**: 고객이 결과를 평가하고 다음 사이클을 위한 피드백을 제공합니다.

이 사이클을 반복할수록 시스템은 점점 완성에 가까워집니다.

**✅ 장점**: 위험을 조기에 발견하고 관리할 수 있어, 대형 프로젝트에 적합합니다.

**❌ 단점**: 계속 시스템을 개선하다 보면 처음에 설계한 소프트웨어 구조가 훼손될 수 있습니다. 또한 사이클이 반복될수록 관리가 복잡해집니다.

### V 모형(V-Model) — 검증을 강화한 폭포수 모형

**V 모형(V-Model)** 은 폭포수 모형을 변형한 것으로, **개발의 각 단계와 그에 대응하는 테스트(검증) 단계를 명확하게 연결**한 방법론입니다. V라는 글자 모양처럼 왼쪽은 개발 단계, 오른쪽은 테스트 단계가 대응됩니다.

```
사용자 요구 명세서  ←→  인수 테스트
   시스템 요구 명세서  ←→  시스템 테스트
      시스템 설계  ←→  통합 테스트
         상세 설계  ←→  단위 테스트
               [구현(코딩)]
```

V 모형의 핵심은 **개발 초기 단계부터 테스트 계획을 함께 수립한다**는 것입니다. 예를 들어 시스템 요구사항을 분석하는 단계에서 이미 "이 요구사항을 어떻게 테스트할 것인가?"를 계획해 둡니다. 그러면 개발 단계에서 자연스럽게 테스트를 고려한 코드가 작성됩니다.

> V 모형은 **검증(Verification)** 과 **확인(Validation)** 을 강조하는 방법론으로, 의료 제어 시스템, 원자력 발전소 제어 시스템 등 **0.1%의 오류도 허용할 수 없는 고신뢰성 시스템**에 특히 적합합니다.

**✅ 장점**: 테스트가 개발과 함께 계획되어 품질이 높아집니다. 오류를 조기에 발견할 확률이 높습니다.

**❌ 단점**: 폭포수 모형과 마찬가지로 되돌아가기 어렵고, 새로운 요구사항을 중간에 반영하기 어렵습니다.

### 점증적 모형(Incremental Model) — 조금씩 나누어 개발하자

**점증적 모형(Incremental Model)** 은 전체 시스템을 한꺼번에 개발하지 않고, **작은 단위로 나누어 순차적으로 개발하고 릴리즈(출시)** 하는 방법론입니다. 마치 레고 블록을 조금씩 쌓아 올리듯이, 기본 기능부터 시작해서 기능을 하나씩 추가해 나갑니다.

**예시**: 온라인 교육 플랫폼을 만든다면,

```
증분 1: 학점 관리 시스템 → 개발·테스트·오픈·운영
증분 2: 자격증 과정 추가 → 개발·테스트·오픈·운영
증분 3: 모의고사 기능 추가 → 개발·테스트·오픈·운영
증분 4: 무료 강의 추가 → 개발·테스트·오픈·운영
  ↓
최종: 완전한 교육 플랫폼 완성
```

한꺼번에 전부 개발하는 것보다 **핵심 기능을 먼저 빠르게 사용자에게 제공**할 수 있고, 각 증분을 운영하면서 얻은 피드백을 다음 증분에 반영할 수 있다는 장점이 있습니다.

**✅ 장점**
- 핵심 기능을 빠르게 시장에 출시할 수 있습니다.
- 각 증분이 독립적으로 운영되므로 전체 프로젝트 실패 위험이 줄어듭니다.
- 개발팀마다 서로 다른 증분에 집중할 수 있어 병렬 개발이 가능합니다.

**❌ 단점**
- 각 증분의 크기가 너무 커지면 관리가 어려워집니다.
- 로그인, 결제처럼 공통으로 들어가야 하는 기능을 매 증분마다 반영해야 하는 추가 작업이 생깁니다.
- 너무 자주 업데이트되면 사용자 피로도가 쌓일 수 있습니다.

:::note 점증적 모형은 스타트업에서 자주 활용됩니다
아이디어가 있어도 전체를 한번에 만들기 어려운 스타트업 환경에서, 핵심 기능(MVP, Minimum Viable Product)부터 먼저 만들어 출시하고 사용자 반응을 보면서 기능을 추가하는 방식이 바로 점증적 모형의 철학과 일치합니다.
:::

### UP 모형(Unified Process) — 통합형 반복 점증 모형

**UP 모형(Unified Process, 통합 프로세스)** 은 반복적이고 점증적인 개발 방식을 체계적으로 통합한 방법론입니다. 한 사이클(도입 → 구체화 → 구축 → 전이)을 반복하면서 단계마다 모델링, 요구 조건 분석, 설계, 구현, 테스트를 모두 수행하되, **각 단계마다 어떤 작업에 더 집중할지(비중)가 달라지는** 것이 특징입니다.

| UP 모형 단계 | 주요 활동 | 집중 영역 |
|---|---|---|
| 도입(Inception) | 시스템 개념 확립, 범위 정의 | 요구사항 분석 비중 높음 |
| 구체화(Elaboration) | 개발 범위 확정, 아키텍처 설계 | 설계 비중 높음 |
| 구축(Construction) | 실제 시스템 구현, 점증적 개발 | 구현 비중 높음 |
| 전이(Transition) | 테스트, 설치, 다음 반복 준비 | 테스트 비중 높음 |

**✅ 장점**: 반복을 통해 위험도를 지속적으로 관리할 수 있습니다. 요구사항 변화에 유연하게 대응할 수 있으며, 소프트웨어 변경을 효율적으로 관리합니다.

**❌ 단점**: 개념 자체가 복잡하여 팀원 모두가 UP 모형을 정확히 이해하고 있어야 제대로 적용할 수 있습니다.

## ⚠️ 주의사항 및 실무 팁

### 어떤 모형을 선택해야 할까?

프로젝트마다 특성이 다르기 때문에 "이 모형이 무조건 최고"는 없습니다. 상황에 맞는 최적의 방법론을 선택하는 것이 중요합니다.

| 상황 | 추천 모형 |
|---|---|
| 요구사항이 명확하고 변경 가능성이 낮을 때 | 폭포수 모형 |
| 고객이 정확히 무엇을 원하는지 불명확할 때 | 프로토타이핑 모형 |
| 위험 요소가 많고 대형 프로젝트일 때 | 나선형 모형 |
| 의료·항공·원자력 등 고신뢰성이 요구될 때 | V 모형 |
| 핵심 기능부터 빠르게 출시해야 할 때 | 점증적 모형 |
| 복잡한 대형 시스템을 반복·점증적으로 개발할 때 | UP 모형 |

### 개발 프로세스를 모르면 어떻게 될까?

:::warning 프로세스를 모르는 개발자의 한계
코딩 실력만 있고 개발 프로세스를 모르는 개발자는 팀 내 의사소통에서 어려움을 겪습니다. "지금 우리가 어느 단계에 있는지", "내가 만든 코드가 전체 시스템에서 어떤 역할을 하는지", "테스트 결과서를 왜 써야 하는지" 를 이해하지 못한 채 그냥 "시키는 일만 하는 개발자"가 됩니다. 이는 성장에 분명한 한계를 만듭니다.
:::

소프트웨어 공학은 단순히 시험을 위한 학문이 아닙니다. 현업에서 프로젝트에 투입되었을 때, 기획자·PM(프로젝트 매니저)·QA 팀원과 원활하게 소통하고, 프로젝트 전체 그림을 이해하며, 자신이 맡은 부분의 중요성을 인식하는 데 직접적으로 필요한 지식입니다.

### 각 단계의 산출물을 소홀히 하지 마라

폭포수 모형이든 다른 모형이든, **각 단계의 산출물(문서, 설계서, 테스트 결과서 등)** 은 단순한 서류가 아닙니다. 다음 단계로 넘어가기 위한 필수 조건이자, 나중에 문제가 생겼을 때 원인을 추적할 수 있는 증거입니다. 귀찮더라도 꼼꼼하게 문서화하는 습관을 들이세요.

:::tip 문서화는 미래의 나를 위한 것
지금 작성한 요구사항 정의서, 설계서, 테스트 결과서는 6개월 뒤 같은 시스템을 유지보수할 때 엄청난 도움이 됩니다. 문서가 없으면 코드만 보고 모든 것을 다시 파악해야 하는 최악의 상황이 됩니다.
:::

## 📋 핵심 정리

- **SDLC(소프트웨어 생명주기)** 는 소프트웨어가 탄생해서 폐기될 때까지의 전체 단계를 의미하며, 타당성 조사 → 요구 분석 → 설계 → 개발 → 테스트 → 운영 → 유지보수/폐기 순으로 진행된다.
- **타당성 조사** 는 개발에 착수하기 전에 예산·기간·필요성을 검토하는 첫 번째 관문이다.
- **요구 분석** 은 내부 직원, 일반 고객, 결제사, 배송사 등 다양한 이해관계자의 요구를 수집하고 정리하는 단계이며, 결과물로 요구사항 정의서가 나온다.
- **설계** 는 기본 설계(큰 그림)와 상세 설계(구체적 구조)로 나뉘며, 개발팀에게 전달되는 핵심 설계 문서가 만들어진다.
- **테스트** 는 단위 테스트 → 통합 테스트 → 시스템 테스트 → 인수 테스트 순으로 진행되며 각각의 결과서가 문서로 남는다.
- **폭포수 모형** 은 가장 전통적인 선형 순차 모형으로, 위에서 아래로만 진행되며 되돌아가기 어렵다는 단점이 있다.
- **프로토타이핑 모형** 은 실제 개발 전에 샘플을 만들어 고객 피드백을 받음으로써 요구사항 오해를 줄이는 방법론이다.
- **나선형 모형** 은 폭포수와 프로토타이핑의 장점을 합쳐 위험을 반복적으로 줄여나가는 모형이며, 소프트웨어 구조 훼손이 단점이다.
- **V 모형** 은 개발 단계마다 대응하는 테스트 단계를 명확히 연결한 모형으로, 고신뢰성이 요구되는 시스템에 적합하다.
- **점증적 모형** 은 시스템을 작은 단위(증분)로 나누어 순차적으로 릴리즈하는 방법으로, 핵심 기능을 먼저 빠르게 제공할 수 있다.
- **UP 모형(통합 프로세스)** 은 반복적이고 점증적인 개발을 체계적으로 통합한 모형으로, 단계마다 비중이 달라지는 구조를 가진다.
- 어떤 개발 모형을 선택할지는 **프로젝트의 특성, 규모, 고객의 요구 명확도, 팀 역량** 에 따라 달라지므로 무조건 하나가 최선이 아니다.
- 소프트웨어 공학 지식은 단순히 이론이 아니라 **실무에서 팀 의사소통, 프로젝트 이해, 개발자로서의 성장** 에 직접적으로 필요한 필수 지식이다.
