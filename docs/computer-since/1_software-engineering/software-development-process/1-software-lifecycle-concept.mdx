# 🌱 소프트웨어 생명주기(SDLC) 개념 완전 정복

소프트웨어를 만든다는 것은 단순히 코드를 작성하는 일이 아닙니다. 집을 짓기 위해 설계도를 그리고, 기초 공사를 하고, 벽을 세우고, 마감하는 과정이 필요하듯이, 소프트웨어를 개발할 때도 **체계적인 순서와 단계**가 반드시 필요합니다. 이 체계적인 흐름을 **소프트웨어 생명주기(SDLC)** 라고 부릅니다.

이번 문서에서는 소프트웨어 생명주기가 무엇인지, 왜 필요한지, 그리고 이를 바탕으로 만들어진 다양한 개발 모형들을 초보자도 이해할 수 있도록 아주 쉽고 자세하게 설명드리겠습니다.

---

## 🌟 소프트웨어 생명주기(SDLC)란 무엇인가?

### 생명주기(Life Cycle)의 의미

우리가 살아있는 모든 것들에는 **생명주기**가 있습니다. 사람으로 따지면 태어나고, 자라고, 활동하다가, 나이 들어 생을 마감하는 과정이 바로 생명주기입니다. 나비도 알에서 애벌레가 되고, 번데기를 거쳐 나비가 되었다가 생을 마치는 과정이 있죠. 이처럼 탄생부터 소멸까지의 전체 변화 과정을 **생명주기**라고 합니다.

소프트웨어도 마찬가지입니다. 소프트웨어는 어느 날 갑자기 만들어지지 않습니다. 누군가가 "이런 시스템이 필요하다"는 아이디어를 내는 순간부터, 개발되고, 운영되다가, 더 이상 필요하지 않아 폐기될 때까지 하나의 **생명주기**를 갖습니다. 이것을 **소프트웨어 생명주기(Software Development Life Cycle, SDLC)** 라고 합니다.

> 📌 **소프트웨어 생명주기(SDLC)**란, 소프트웨어가 탄생해서 폐기에 이르기까지의 전체 변화 과정을 의미합니다.

### SDLC가 중요한 이유

앞으로 개발 관련 문서나 회사 내 기획서, 프로젝트 관리 도구 등 어디서든 **SDLC**라는 약어를 자주 마주치게 됩니다. 이 용어를 처음 보는 분들은 낯설겠지만, 내용을 이해하면 매우 자연스럽게 느껴질 것입니다. **SDLC**는 시스템을 개발하는 절차와 단계를 설명할 때 사용하는 핵심 용어입니다.

:::info SDLC란?
**SDLC**는 **S**oftware **D**evelopment **L**ife **C**ycle의 약자로, **소프트웨어 개발 생명주기** 또는 **시스템 개발 주기**를 의미합니다. 이 용어를 보면 "소프트웨어를 개발하는 순서와 변화 과정이구나"라고 이해하면 됩니다.
:::

SDLC가 중요한 이유는 단순히 개발 순서를 정해주는 것 이상의 역할을 하기 때문입니다. 프로젝트에 **돈(예산)은 얼마나 들지**, **개발 기간은 얼마나 필요한지**, **몇 명의 인력이 투입되어야 하는지**, **단계별로 어떤 자원을 어떻게 분배해야 하는지** 등을 수립하기 위한 **기본 골격**이 됩니다.

마치 집을 짓기 전에 공사 계획서를 만드는 것처럼, SDLC는 소프트웨어 프로젝트 전체를 운영하기 위한 **청사진(설계 계획)** 역할을 합니다. 단계별로 개발 진행 상황을 명확하게 파악할 수 있고, 각 단계가 뚜렷하게 구분되므로 팀 간 의사소통도 훨씬 수월해집니다.

## 🔍 소프트웨어 생명주기의 핵심 단계

소프트웨어 생명주기는 일반적으로 다음과 같은 단계로 구성됩니다. 실제 쇼핑몰을 새롭게 개발하는 상황을 예시로 들어 각 단계를 살펴보겠습니다.

### 1단계: 타당성 조사 🔎

**타당성 조사(Feasibility Study)** 는 소프트웨어 개발을 시작하기 전에 가장 먼저 이루어지는 단계입니다. 쉽게 말해 "이걸 정말 만들어야 할까?"를 검토하는 과정입니다.

예를 들어, 우리 회사에 기존 쇼핑몰이 있는데 모바일에 최적화된 새로운 쇼핑몰을 2억 원을 들여 개발하려고 한다고 가정해 봅시다. 이때 가장 먼저 해야 할 것은 "바로 개발팀을 꾸려서 개발 시작!"이 아닙니다. 다음과 같은 질문들을 먼저 던져야 합니다.

- 정말 이 시스템을 개발할 필요성이 있는가?
- 2억 원이라는 예산이 타당한가?
- 개발 기간 동안의 기회비용은 어떤가?
- 우리 고객의 특성과 현재 매출 규모를 고려했을 때 경제적으로 타당한가?

이런 질문들에 답하면서 "개발을 진행하는 것이 타당하다"는 결론이 나야 비로소 다음 단계로 넘어갈 수 있습니다. 많은 예산과 시간이 투입되는 만큼, 타당성 조사는 절대 건너뛸 수 없는 필수 단계입니다.

### 2단계: 요구 분석 📋

타당성 조사를 통해 "개발하자!"는 결론이 났다면, 다음은 **요구 분석(Requirements Analysis)** 단계입니다. 이 단계는 "고객이 원하는 것이 무엇인지"를 파악하고 분석하는 과정입니다.

여기서 **고객**은 단순히 물건을 사는 외부 소비자만을 의미하지 않습니다. 다양한 이해관계자(stakeholder, 프로젝트에 관련된 모든 사람)들이 고객이 될 수 있습니다.

| 고객 유형 | 설명 | 요구사항 예시 |
|---|---|---|
| **내부 고객 (직원)** | 시스템을 관리하는 우리 직원 | 관리자 페이지, 자동화 기능 |
| **외부 고객 (소비자)** | 실제 서비스를 이용하는 사용자 | 위치 기반 서비스, 간편 결제 |
| **판매자 (벤더)** | 상품을 올리는 공급자 | 상품 등록/관리 기능 |
| **결제사 (PG사)** | 이니시스 등 결제 시스템 운영사 | 결제 연동 API |
| **배송 업체** | 물류/배송을 담당하는 업체 | 배송 정보 연동 |
| **물류 창고팀** | 재고를 관리하는 팀 | 재고 관리 시스템 |

이처럼 다양한 이해관계자들의 요구사항을 수집하고 분석해야 합니다. 홍보팀에서는 "카카오톡 푸시 메시지 기능이 필요하다"고 할 수 있고, 배송팀에서는 "자동 배송 상태 업데이트 기능이 필요하다"고 할 수 있습니다. 이 모든 요구사항을 종합해서 **요구사항 정의서**라는 문서로 정리하는 것이 이 단계의 핵심 산출물(결과물)입니다.

### 3단계: 설계 📐

요구사항이 모두 분석되었다면 이제 **설계(Design)** 단계입니다. 집을 짓기 전에 설계도가 필요하듯이, 소프트웨어도 실제 개발을 시작하기 전에 반드시 설계도가 필요합니다.

설계는 크게 세 가지 단계로 나눌 수 있습니다.

- **개념적 설계**: 시스템 전체의 큰 그림을 그리는 단계. "어떤 기능들이 있고, 어떻게 연결될 것인가?"를 추상적으로 표현합니다.
- **논리적 설계**: 개념을 좀 더 구체화하는 단계. 데이터 흐름, 화면 구성, 기능 동작 방식을 논리적으로 표현합니다.
- **물리적 설계(상세 설계)**: 실제 개발이 가능한 수준까지 구체화하는 단계. 데이터베이스 구조, 자료 구조, 모듈 간 관계 등을 상세하게 설계합니다.

설계 단계까지는 주로 **기획팀** 또는 **시스템 분석/설계팀**이 담당합니다. 완성된 설계서(설계도)는 다음 단계인 개발팀으로 전달됩니다.

### 4단계: 개발(구현) 💻

설계서가 완성되면 이제 실제 **개발(Implementation)** 단계로 넘어갑니다. 이 단계에서 개발팀은 설계서를 바탕으로 Java, Python, PHP 등 프로그래밍 언어로 실제 코드를 작성합니다. 이 과정을 **코딩(Coding)** 이라고 하며, 결과물로 **소스 코드**가 나옵니다.

개발 과정에서는 **디버깅(Debugging, 오류 수정)** 작업이 반복적으로 이루어집니다. 코드를 작성하다 보면 오류가 발생하고, 오류를 수정하고, 또 개발하다가 또 오류가 발생하면 또 수정하는 과정이 반복됩니다. 마치 요리를 하다가 간을 보고 조절하는 것처럼, 개발과 디버깅은 끊임없이 함께 이루어집니다.

```java
// ✅ 올바른 예시: 주문 처리 모듈의 기본 구조
public class OrderService {
    
    // 주문을 처리하는 메서드
    public boolean processOrder(int userId, int productId, int quantity) {
        // 재고 확인
        if (!checkStock(productId, quantity)) {
            System.out.println("재고가 부족합니다.");
            return false; // 재고 부족 시 false 반환
        }
        
        // 결제 처리
        boolean paymentResult = processPayment(userId, productId, quantity);
        if (!paymentResult) {
            System.out.println("결제에 실패했습니다.");
            return false; // 결제 실패 시 false 반환
        }
        
        // 배송 정보 등록
        registerDelivery(userId, productId);
        System.out.println("주문이 완료되었습니다.");
        return true; // 모든 처리 성공 시 true 반환
    }
    
    // 재고 확인 메서드 (단위 테스트 대상)
    private boolean checkStock(int productId, int quantity) {
        // 실제 데이터베이스에서 재고를 확인하는 로직
        return true; // 예시용 반환값
    }
    
    // 결제 처리 메서드 (단위 테스트 대상)

        // PG사(결제사) 연동 로직
        return true; // 예시용 반환값
    }
    
    // 배송 등록 메서드 (단위 테스트 대상)
```

```java
public class BadOrderService {
    public void doEverything(int userId, int productId, int quantity) {
```

모듈(기능 단위)을 너무 크게 만들면 테스트와 유지보수가 어려워집니다. **하나의 모듈은 하나의 기능만** 담당하도록 설계하는 것이 좋습니다.
:::

### 5단계: 테스트 🧪

개발이 완료되면 **테스트(Test)** 단계를 거칩니다. 테스트는 한 번에 전체를 테스트하는 것이 아니라, 작은 단위부터 점차 큰 범위로 확장하며 진행합니다.

| 테스트 종류 | 설명 | 예시 |
|---|---|---|
| **단위 테스트 (Unit Test)** | 하나의 모듈(기능)을 개별적으로 테스트 | SMS 발송 모듈만 테스트, 결제 모듈만 테스트 |
| **통합 테스트 (Integration Test)** | 여러 모듈을 합쳐서 전체 연결이 잘 되는지 테스트 | 주문 → 결제 → 배송 연동 테스트 |
| **시스템 테스트 (System Test)** | 실제 서버/네트워크 환경에서 성능과 안정성 테스트 | 통신 속도, 동시 접속자 처리 성능 테스트 |
| **인수 테스트 (Acceptance Test)** | 실제 사용자가 직접 자신의 환경에서 테스트 | 각 부서 직원이 실제 업무 환경에서 직접 테스트 |

각 테스트 단계가 끝나면 반드시 **테스트 결과서**라는 문서가 산출물로 나와야 합니다. 인수 테스트까지 모두 완료되어야 비로소 실제 서비스를 오픈(런칭)할 수 있습니다.

### 6단계: 운영 및 유지보수 🔧

서비스가 오픈된 이후에도 소프트웨어의 생명주기는 계속됩니다. 실제로 사용하다 보면 처음에는 발견되지 않았던 오류가 생기거나, 고객이 새로운 기능을 요청하기도 합니다. 이런 문제들을 지속적으로 수정하고 개선하는 과정이 **유지보수(Maintenance)** 입니다.

소프트웨어의 특성상, 초기에는 문제가 없어 보이더라도 실제 사용 환경에서 다양한 데이터와 조건이 결합되면서 예상치 못한 오류가 발생할 수 있습니다. 따라서 유지보수는 선택이 아닌 필수입니다.

### 7단계: 폐기 🗑️

소프트웨어도 언젠가는 수명을 다합니다. 예를 들어, PHP로 개발된 시스템을 Java로 완전히 새롭게 재개발하기로 결정했다면, 기존 PHP 시스템은 폐기됩니다. 더 이상 필요하지 않은 시스템은 **폐기(Retirement)** 단계를 거쳐 생명주기를 마무리합니다.

## 🏗️ 폭포수 모형(Waterfall Model) - 가장 전통적인 개발 방식

### 폭포수 모형이란?

소프트웨어 생명주기를 실제 개발에 처음으로 체계적으로 적용한 방법론이 바로 **폭포수 모형(Waterfall Model)** 입니다. 이름에서도 알 수 있듯이, 폭포에서 물이 위에서 아래로 흐르듯이 **순차적으로(한 방향으로만)** 개발이 진행되는 방식입니다.

폭포를 상상해 보세요. 물은 위에서 아래로 떨어집니다. 아래서 위로 거슬러 올라갈 수 없죠. 폭포수 모형도 마찬가지입니다. 한 번 다음 단계로 넘어가면 이전 단계로 돌아갈 수 없습니다.

```
계획 단계
    ↓
분석 단계
    ↓
설계 단계
    ↓
개발 단계
    ↓
테스트 단계
    ↓
운영 단계
    ↓
유지보수 단계
```

> 📌 폭포수 모형은 **선형 순차 모형**이라고도 불립니다. 각 단계는 반드시 순서대로 진행되며, 이전 단계의 산출물이 완성되어야 다음 단계로 넘어갈 수 있습니다.

### 폭포수 모형의 각 단계별 세부 내용

**1. 계획 단계**: 프로젝트 계획서를 작성합니다. 여기에는 현재 시장 트렌드, 경쟁사 분석, 우리 서비스의 부족한 점, 개발 필요성, 예상 비용과 기간이 포함됩니다. 예를 들어, "우리 주요 고객층인 20~30대는 모바일 중심인데, 우리 서비스는 PC 위주라 경쟁력이 약하다. 따라서 모바일 최적화 서비스를 1년, 2억 원 규모로 개발해야 한다"는 내용이 계획서에 담깁니다. 이 계획서로 경영진(투자자)의 승인을 받는 것이 계획 단계의 핵심입니다.

**2. 분석 단계**: 앞서 설명한 요구 분석 단계로, **요구사항 정의서**가 산출물로 나옵니다.

**3. 설계 단계**: 요구사항 정의서를 바탕으로 화면 구성, 기능 설계, 데이터베이스 설계 등을 포함한 **기본 설계서**와 **상세 설계서**가 산출물로 나옵니다.

**4. 개발 단계**: 설계서를 받아 개발팀이 실제 코딩을 시작합니다. 모듈별 코딩과 디버깅이 반복적으로 이루어집니다.

**5. 테스트 단계**: 단위 테스트 → 통합 테스트 → 시스템 테스트 → 인수 테스트 순으로 진행되며, 각 단계별 **테스트 결과서**가 산출물로 나옵니다.

**6. 운영 및 유지보수 단계**: 서비스를 오픈하고 지속적으로 관리합니다.

### 폭포수 모형의 장점과 단점

| 구분 | 내용 |
|---|---|
| ✅ **장점** | 단계가 명확하여 이해하기 쉽다 |
| ✅ **장점** | 단계별 점검(체크포인트)이 가능하다 |
| ✅ **장점** | 기존 시스템을 보완하는 프로젝트에 적합하다 |
| ✅ **장점** | 가장 널리 사용되며 적용 사례가 풍부하다 |
| ❌ **단점** | 한 번 넘어간 단계로 돌아갈 수 없다 |
| ❌ **단점** | 개발 중간에 새로운 요구사항이 생겨도 반영하기 어렵다 |
| ❌ **단점** | 최종 단계에서야 결과물을 볼 수 있어 수정 비용이 크다 |
| ❌ **단점** | 앞 단계에서 작업이 완벽하지 않으면 전체에 영향을 미친다 |

:::danger 폭포수 모형의 치명적 단점
최종 단계에서 고객이 "이건 우리가 원하던 게 아니에요"라고 하면, 처음부터 다시 개발해야 할 수 있습니다. 초기 단계의 산출물이 정확하지 않으면 전체 프로젝트가 실패할 위험이 있습니다. **앞 단계의 산출물이 정확한지 철저히 검토하는 것**이 매우 중요합니다.
:::

## 🔄 다양한 소프트웨어 개발 모형

폭포수 모형의 단점을 보완하기 위해 다양한 개발 모형들이 등장했습니다. 프로젝트의 특성, 예산, 기간, 팀 구성에 따라 가장 적합한 모형을 선택하는 것이 중요합니다.

### 🧩 프로토타이핑 모형(Prototyping Model)

#### 왜 등장했는가?

폭포수 모형의 가장 큰 단점은 **최종 단계가 되어서야 결과물을 볼 수 있다**는 것이었습니다. 고객 입장에서는 몇 달, 길게는 1~2년을 기다렸다가 결과물을 보는데, 그게 자신이 원했던 것과 다르면 처음부터 다시 시작해야 합니다. 이 문제를 해결하기 위해 등장한 것이 바로 **프로토타이핑 모형(Prototyping Model)** 입니다.

#### 프로토타이핑 모형이란?

**프로토타입(Prototype)** 이란 **견본품** 또는 **시제품**을 말합니다. 가전제품을 만들기 전에 먼저 목업(mock-up, 실물 모형)을 만들어 보는 것처럼, 소프트웨어도 실제 완성품을 만들기 전에 **샘플을 먼저 만들어** 고객에게 "이렇게 만들면 될까요?"를 확인받는 방식입니다.

예를 들어, 개발자가 고객에게 이렇게 말할 수 있습니다.

> "고객님이 원하시는 기능이 이런 것 같은데, 제가 간단하게 샘플로 만들어 봤습니다. 이런 식으로 동작하면 괜찮으신가요?"

고객이 "거의 맞는데, 이 버튼을 눌렀을 때 이렇게 동작했으면 좋겠어요"라고 피드백을 주면, 그것을 반영해서 프로토타입을 수정합니다. 이 과정을 반복하면서 고객의 요구사항을 최대한 정확하게 파악한 뒤 실제 개발을 시작합니다.

```
요구사항 수집
    ↓
프로토타입 설계
    ↓
프로토타입 개발
    ↓
고객 평가 및 피드백
    ↓ (수정 사항 있으면 다시 위로)
최종 프로토타입 완성
    ↓
실제 제품 개발
```

#### 프로토타이핑 모형의 장단점

:::tip 프로토타이핑 모형의 장점
- 고객이 일찍 결과물을 확인할 수 있어 요구사항 오류를 미리 잡을 수 있습니다.
- 개발 초기에 생각하지 못했던 새로운 기능이나 서비스를 발견할 수 있습니다.
- 개발의 타당성을 사전에 검증할 수 있습니다.
- 의사소통 도구로 활용되어 개발팀과 고객 간의 소통이 원활해집니다.
:::

:::warning 프로토타이핑 모형의 단점
- 임시로 만드는 것이라 산출물, 일정표, 체계적인 문서가 부족합니다.
- 단기간에 제작해야 하므로 기능이 제한적이고 성능이나 신뢰성이 낮습니다.
- 고객이 프로토타입을 실제 완성품으로 착각할 수 있습니다.
- 너무 잘 만들면 기대치가 높아지고, 너무 허술하면 신뢰도가 떨어집니다. **적정한 수준**의 견본품을 만드는 것이 중요합니다.
:::

### 🌀 나선형 모형(Spiral Model)

#### 왜 등장했는가?

폭포수 모형과 프로토타이핑 모형의 **장점만을 모아서** 위험(리스크)을 최소화하면서 시스템을 점진적으로 발전시키는 방식이 필요했습니다. 이를 위해 등장한 것이 **나선형 모형(Spiral Model)** 입니다. 이 모형은 스파이럴 모델(Spiral Model)이라고도 불립니다.

#### 나선형 모형이란?

나선형 모형은 마치 달팽이 껍데기처럼 **안쪽에서 바깥쪽으로 점점 확장**되는 방식입니다. 처음에는 작은 프로토타입을 만들고, 그것을 평가하고, 기능을 추가하고, 또 평가하고, 또 기능을 추가하면서 점점 완성도 높은 시스템으로 발전시킵니다.

나선형 모형의 각 사이클은 다음 네 가지 활동으로 구성됩니다.

1. **계획(Planning)**: 목표, 기능, 제약 조건을 결정합니다.
2. **위험 분석(Risk Analysis)**: 우선순위와 위험 요소를 분석하고 타당성을 평가합니다.
3. **개발(Engineering)**: 프로토타입을 만들고 개발을 진행합니다.
4. **평가(Evaluation)**: 고객과 함께 결과를 평가하고 다음 사이클을 계획합니다.

이 네 가지 활동이 나선을 그리며 반복되면서 시스템이 점점 발전합니다. 첫 번째 사이클에서는 기본 기능의 프로토타입, 두 번째 사이클에서는 좀 더 발전된 버전, 세 번째 사이클에서는 거의 완성에 가까운 버전이 만들어집니다.

#### 나선형 모형의 장단점

나선형 모형은 **리스크(위험) 관리**에 특화된 모형입니다. 각 사이클마다 위험을 분석하고 줄여나가기 때문에, 대규모 프로젝트나 위험도가 높은 프로젝트에 특히 적합합니다.

| 구분 | 내용 |
|---|---|
| ✅ **장점** | 위험을 단계적으로 최소화할 수 있다 |
| ✅ **장점** | 폭포수와 프로토타이핑의 장점을 결합했다 |
| ✅ **장점** | 점진적으로 시스템을 발전시킬 수 있다 |
| ❌ **단점** | 계속 개선하다 보면 소프트웨어 구조가 훼손될 수 있다 |
| ❌ **단점** | 관리가 복잡하고 전문적인 리스크 분석 능력이 필요하다 |

:::warning 나선형 모형의 치명적 단점
계속해서 소프트웨어를 개선하는 과정에서 **최초에 설계한 소프트웨어 구조가 훼손**될 수 있습니다. 처음 설계의 원칙을 지키면서 개선을 이어가는 것이 매우 중요합니다.
:::

### ✅ V 모형(V-Model)

#### 왜 등장했는가?

폭포수 모형은 각 단계의 산출물이 중요했습니다. 하지만 V 모형은 산출물보다 **검증(Verification, 베리피케이션)** 이 더 중요합니다. 의료 시스템이나 원자력 발전소 제어 시스템처럼 **0.1%의 오류도 허용되지 않는 고신뢰성 시스템**을 개발할 때는 각 단계별로 완벽한 검증이 필수입니다. 이런 요구에 맞게 등장한 것이 **V 모형(V-Model)** 입니다.

#### V 모형이란?

V 모형은 폭포수 모형을 기반으로 **검증 단계를 대폭 강화**한 모델입니다. 알파벳 V자 모양처럼, 왼쪽 내려가는 방향에는 개발 단계가, 오른쪽 올라가는 방향에는 각 개발 단계에 대응하는 테스트/검증 단계가 배치됩니다.

```
사용자 요구 명세서 ──────────────────── 인수 테스트
        ↓                                    ↑
시스템 요구 명세서 ──────────────── 시스템 테스트
        ↓                                ↑
시스템 설계 ────────────────── 통합 테스트
        ↓                            ↑
상세 설계 ────────────── 단위 테스트
        ↓                        ↑
        └────── 코딩(구현) ──────┘
```

V 모형의 핵심은 **개발 초기 단계에서부터 테스트 계획을 함께 수립**한다는 점입니다. 예를 들어, 사용자 요구 명세서를 작성하는 단계에서 이미 **인수 테스트 계획**도 함께 작성합니다. 시스템 설계 단계에서 이미 **통합 테스트 계획**도 함께 수립합니다. 이렇게 하면 개발 단계에서 자연스럽게 테스트를 고려한 코드를 작성하게 됩니다.

:::info V 모형이 적합한 시스템
- 의료 기기 제어 시스템 (수술 로봇 등)
- 원자력 발전소 제어 시스템
- 항공기 제어 시스템
- 금융 핵심 시스템

이처럼 **생명이나 안전과 직결**되어 오류가 절대 허용되지 않는 시스템에 V 모형이 가장 적합합니다.
:::

### 📦 점증적 모형(Incremental Model)

#### 왜 등장했는가?

처음부터 전체 시스템을 한꺼번에 개발하려면 비용과 시간이 엄청나게 들고, 실패 위험도 높습니다. 이를 해결하기 위해 시스템을 **여러 번에 나누어 릴리스(출시)** 하는 방식이 등장했습니다. 이것이 **점증적 모형(Incremental Model)** 입니다.

#### 점증적 모형이란?

마치 레고 블록을 쌓듯이, 가장 핵심적인 기능을 먼저 만들고, 그것이 안정화되면 기능을 추가하고, 또 안정화되면 더 많은 기능을 추가하는 방식입니다.

예를 들어, 교육 플랫폼을 개발한다고 가정해봅시다.

```
[릴리스 1] 핵심 기능: 학점 관리 시스템 개발 → 운영 시작
    ↓
[릴리스 2] 자격증 관련 기능 추가 → 운영 계속
    ↓
[릴리스 3] 모의고사 자기주도학습 기능 추가 → 운영 계속
    ↓
[릴리스 4] 무료 강의 서비스 추가 → 완성된 교육 플랫폼
```

각 릴리스는 완전히 동작하는 소프트웨어여야 합니다. 반쪽짜리 기능을 출시하는 것이 아니라, 그 단계에서 제공할 수 있는 기능을 **완전하게 구현**한 후 출시하고, 이후에 기능을 추가해 나가는 방식입니다.

#### 점증적 모형의 장단점

| 구분 | 내용 |
|---|---|
| ✅ **장점** | 새로운 기능을 단계적으로 추가할 수 있다 |
| ✅ **장점** | 전체 프로젝트 실패 위험을 최소화할 수 있다 |
| ✅ **장점** | 개발팀마다 서로 다른 기능 영역에 초점을 맞출 수 있다 |
| ✅ **장점** | 핵심 기능을 먼저 출시해 빠르게 사용자 피드백을 받을 수 있다 |
| ❌ **단점** | 각 증분(추가 기능 단위)은 비교적 작아야 한다 |
| ❌ **단점** | 증분마다 공통 기능(로그인, 결제 등)을 반복 적용해야 해서 개발 공수가 늘어난다 |
| ❌ **단점** | 너무 잦은 업데이트는 사용자 피로도를 높이고 신뢰도를 낮출 수 있다 |

:::note 증분(Increment)이란?
**증분(Increment)** 이란 "추가되는 단위"를 의미합니다. 릴리스 1에서 릴리스 2로 넘어갈 때 추가되는 기능 묶음이 하나의 증분입니다. 이 증분이 너무 크면 별도 프로젝트가 되어버려 점증적 모형의 장점을 잃게 됩니다. 증분은 적당히 작고 관리 가능한 크기여야 합니다.
:::

### 🔁 UP 모형(Unified Process Model)

#### UP 모형이란?

**UP 모형(Unified Process Model, 통합 프로세스 모형)** 은 점증적 반복 개발 방식을 기반으로, 각 사이클이 끝날 때마다 테스트와 통합 수행을 통해 산출물을 검증하고 발전시키는 모형입니다.

UP 모형은 다음 네 가지 큰 단계로 구성됩니다.

1. **도입(Inception) 단계**: 시스템의 개념을 확립합니다. "우리가 만들 시스템은 이런 것이다"를 정의하는 단계입니다.
2. **전개(Elaboration) 단계**: 개념을 구체화하고 개발 범위를 확정합니다.
3. **구축(Construction) 단계**: 실제 개발을 진행하면서 시스템을 구축합니다.
4. **전이(Transition) 단계**: 테스트, 설치, 다음 반복 준비를 합니다.

UP 모형의 특징은 각 단계에서 **모델링, 요구조건 분석, 설계, 구현** 이 같은 활동들이 반복되지만, 각 단계마다 그 **비중이 다르다**는 것입니다. 도입 단계에서는 모델링과 요구 분석의 비중이 높고, 구축 단계에서는 구현의 비중이 높아집니다.

:::tip UP 모형의 핵심 장점
- **반복을 통해 리스크를 단계적으로 줄여나갈 수 있습니다.**
- 소프트웨어 변경을 효율적으로 관리할 수 있습니다.
- 반복 과정에서 고객의 요구사항을 지속적으로 반영할 수 있습니다.
- 개발 기간을 단축하고 품질을 높일 수 있습니다.
:::

## ⚠️ 개발 모형 선택 시 주의사항과 실무 팁

### 어떤 모형을 선택해야 할까?

프로젝트의 특성에 따라 가장 적합한 모형을 선택하는 것이 중요합니다. 다음 표를 참고하세요.

| 상황 | 추천 모형 | 이유 |
|---|---|---|
| 요구사항이 명확하고 변경 가능성이 낮다 | **폭포수 모형** | 단순하고 관리하기 쉽다 |
| 고객이 원하는 것을 먼저 확인해야 한다 | **프로토타이핑 모형** | 빠른 피드백이 가능하다 |
| 대규모 프로젝트에서 위험을 최소화해야 한다 | **나선형 모형** | 단계별 위험 분석이 가능하다 |
| 오류가 절대 허용되지 않는 고신뢰성 시스템 | **V 모형** | 각 단계별 철저한 검증이 가능하다 |
| 핵심 기능부터 단계적으로 출시하고 싶다 | **점증적 모형** | 리스크를 분산하여 점진적으로 개발 가능하다 |
| 반복적이고 점진적인 개발이 필요하다 | **UP 모형** | 반복 과정에서 지속적으로 품질을 향상시킬 수 있다 |

### 실무에서 자주 하는 실수들

:::warning 실무에서 자주 하는 실수 1: 요구 분석을 소홀히 하는 것
많은 프로젝트가 요구 분석 단계를 충분히 거치지 않고 빠르게 개발로 넘어가려 합니다. 하지만 요구 분석이 부실하면 개발이 끝나고 나서 "이건 우리가 원했던 게 아니에요"라는 상황이 발생합니다. **요구 분석에 충분한 시간을 투자하는 것이 전체 프로젝트 성공의 핵심**입니다.
:::

:::warning 실무에서 자주 하는 실수 2: 테스트를 마지막에 몰아서 하는 것
테스트는 개발이 모두 끝난 후에 한꺼번에 하는 것이 아닙니다. 모듈 단위로 개발이 완료될 때마다 **단위 테스트**를 해야 하고, 이후 통합 테스트, 시스템 테스트, 인수 테스트를 단계적으로 진행해야 합니다. 마지막에 몰아서 테스트하면 오류 발생 시 원인을 찾기 매우 어려워집니다.
:::

:::danger 절대 하면 안 되는 것: 산출물 없이 다음 단계로 넘어가기
어떤 개발 모형을 사용하든 각 단계별로 **산출물(결과 문서)** 이 반드시 있어야 합니다. 구두(말)로만 논의하고 문서 없이 다음 단계로 넘어가면, 나중에 분쟁이 발생했을 때 기준이 없어 큰 문제가 생깁니다. 요구사항 정의서, 설계서, 테스트 결과서 등 각 단계의 산출물을 반드시 문서화하세요.
:::

### 개발 모형을 이해해야 하는 이유

:::tip 왜 개발자도 SDLC를 알아야 할까요?
단순히 코딩만 잘하는 개발자와 소프트웨어 공학의 개념을 이해하는 개발자는 실무에서 확연히 다른 퍼포먼스를 보입니다. SDLC를 이해하는 개발자는 다음과 같은 능력을 갖게 됩니다.

- 지금 내가 어떤 단계에서 어떤 역할을 해야 하는지 명확히 알 수 있습니다.
- 팀원, 기획자, 고객과의 **의사소통**이 훨씬 원활해집니다.
- 프로젝트 전체 흐름을 이해하므로 더 나은 설계 결정을 내릴 수 있습니다.
- 결국 더 빠르게 성장하고 더 높은 가치를 인정받는 개발자가 됩니다.

소프트웨어 공학은 개발자의 몸값을 올리는 핵심 지식입니다!
:::

## 📝 핵심 정리

- **소프트웨어 생명주기(SDLC)** 란 소프트웨어의 탄생부터 폐기까지의 전체 변화 과정을 의미한다.
- SDLC는 프로젝트의 예산, 기간, 인력, 자원 분배를 계획하기 위한 기본 골격 역할을 한다.
- SDLC의 일반적인 단계는 **타당성 조사 → 요구 분석 → 설계 → 개발 → 테스트 → 운영 → 유지보수 → 폐기** 순이다.
- **폭포수 모형**은 가장 전통적이고 널리 사용되는 선형 순차 모형으로, 이전 단계로 돌아갈 수 없다.
- **프로토타이핑 모형**은 개발 전에 견본품을 만들어 고객의 요구사항을 미리 확인하는 방식이다.
- **나선형 모형**은 폭포수 모형과 프로토타이핑 모형의 장점을 결합해 리스크를 점진적으로 줄여나가는 방식이다.
- **V 모형**은 폭포수 모형에 검증 단계를 강화한 모델로, 고신뢰성 시스템에 적합하다.
- **점증적 모형**은 핵심 기능부터 단계적으로 출시하면서 기능을 확장해 나가는 방식이다.
- **UP 모형**은 반복적이고 점증적인 프로세스를 통해 리스크를 관리하고 품질을 높이는 통합 모형이다.
- 프로젝트 특성에 따라 가장 적합한 개발 모형을 선택하는 것이 프로젝트 성공의 핵심이다.
- 어떤 모형을 사용하든 각 단계별 산출물(문서)을 반드시 작성하고 유지해야 한다.
