# 소프트웨어 변경이 발생하는 원인 🔄

소프트웨어는 한 번 만들면 그걸로 끝이라고 생각하기 쉽습니다. 하지만 실제로는 완전히 반대입니다. 우리가 매일 쓰는 앱, 게임, 웹사이트는 끊임없이 업데이트되고 바뀌어 갑니다. 이번 글에서는 **소프트웨어가 왜 변경될 수밖에 없는지**, 그 근본적인 원인을 이해하고, 변경을 체계적으로 관리하는 **소프트웨어 형상관리(SCM)** 의 전체 그림을 함께 살펴봅니다.

---

## 🧩 소프트웨어 형상관리란 무엇인가?

### 형상관리의 기본 개념

**형상관리(Software Configuration Management, SCM)** 란, 소프트웨어 개발 과정에서 발생하는 **모든 변경 사항을 체계적으로 식별하고 기록하며 통제하는 활동 전체**를 말합니다.

"형상(形狀)"이라는 단어가 조금 낯설게 느껴질 수 있습니다. 쉽게 말하면 "소프트웨어의 현재 모습과 상태"라고 생각하면 됩니다. 마치 사람의 얼굴이 나이가 들면서 조금씩 달라지듯이, 소프트웨어도 시간이 지남에 따라 그 모습(형상)이 계속 바뀌어 갑니다. 그 변화를 빠짐없이 기록하고 관리하는 것이 바로 형상관리입니다.

> **소프트웨어는 완성 후에도 무수히 많은 변경이 발생합니다. 형상관리는 그 변경들을 추적하고 관리하여, 나중에 문제가 생겼을 때 원인을 파악하고 빠르게 대응할 수 있게 해줍니다.**

### 왜 형상관리가 필요한가?

개발 학자 **에릭 브로드(Eric Burd)** 는 이렇게 말했습니다.

> "프로젝트는 진행되어 가면서 새로운 산출물들이 만들어지고, 그 산출물들은 계속 쌓여가면서 버전업(version-up)된다."

즉, 소프트웨어는 처음 만든 그 상태로 영원히 유지되지 않습니다. 새로운 기능이 추가되고, 불필요한 기능이 빠지고, 디자인이 바뀌고, 버그(오류)가 수정됩니다. 우리가 즐겨 하는 게임을 떠올려 보세요. 업데이트가 전혀 없는 게임은 금방 지루해지고 사용자들이 떠나버립니다. 반대로, 꾸준히 새 콘텐츠와 기능을 추가하는 게임은 오랫동안 사랑받습니다.

또 다른 학자인 **퍼스트 오브올(Firstofal)** 은 이렇게 정의합니다.

> "시스템은 소프트웨어 개발 생명주기(SDLC)의 모든 단계에서 변경이 일어나고, 변경하고자 하는 욕구는 개발 기간 내내 지속적으로 발생한다."

이 말이 핵심입니다. 변경은 코드에만 국한된 것이 아닙니다. **요구사항 문서, 설계 문서, 테스트 계획서, 소스 코드** 등 소프트웨어 개발 과정에서 만들어지는 모든 결과물(산출물)이 변경 대상이 됩니다.

:::info 소프트웨어 개발 생명주기(SDLC)란?
SDLC(Software Development Life Cycle)는 소프트웨어가 기획·분석·설계·구현·테스트·배포·유지보수 단계를 거치는 전체 과정을 말합니다. 형상관리는 이 모든 단계에 걸쳐 이루어집니다.
:::

## 🌍 소프트웨어 변경이 발생하는 원인

### 변경의 원인을 알아야 하는 이유

변경의 원인을 정확히 이해하면, 개발자로서 변화에 여유롭게 대처할 수 있습니다. 흔히 "나는 개발자니까 코드만 잘 짜면 되지"라고 생각하기 쉽지만, 이는 큰 오해입니다.

최근 IT 업계에서는 개발자에게도 **인문학적 소양과 시장을 바라보는 시각**을 요구합니다. 왜냐하면 소프트웨어는 늘 다양한 요구에 의해 변경되기 때문입니다. 고객을 이해하고, 시장의 변화를 읽을 수 있는 개발자가 진정으로 사용자에게 맞는 성공적인 프로그램을 만들어낼 수 있습니다.

> **기술적으로는 완벽하지만 시장과 고객의 요구를 무시한 소프트웨어는 결국 시장에서 실패합니다.**

소프트웨어 변경의 원인은 크게 두 가지로 나눌 수 있습니다.

### 첫 번째 원인: 업무 환경의 변화 🏢

**업무 환경의 변화**는 소프트웨어 변경을 일으키는 가장 직접적인 원인입니다.

#### 고객의 새로운 기능 요구

고객은 처음에 요구하지 않았던 기능을 나중에 요구하는 경우가 매우 많습니다. "처음엔 이 정도면 됐는데, 써보니까 이런 기능이 있으면 더 좋겠어요"라는 피드백은 개발자라면 누구나 한 번쯤 들어본 말입니다. 이런 요구사항은 소프트웨어에 새로운 기능을 추가하거나 기존 기능을 수정하게 만듭니다.

#### 시장 환경의 변화

시장 환경이 바뀌면 소프트웨어도 그에 맞춰 변해야 합니다. 대표적인 예를 들어보겠습니다.

- **책의 디지털화**: 예전에는 지하철에서 종이책을 읽는 사람이 많았습니다. 지금은 스마트폰이나 태블릿으로 전자책(e-Book)을 읽고, 음성으로 책을 읽어주는 서비스까지 등장했습니다. 출판사들이 디지털 환경의 변화를 읽지 못했다면 시장에서 도태되었을 것입니다.
- **PC에서 모바일로의 전환**: 예전에는 PC 기반의 웹사이트만 있으면 됐지만, 지금은 스마트폰·태블릿 등 다양한 기기에서 최적화된 화면을 제공해야 합니다. 이 변화에 대응하지 못한 서비스는 사용자를 잃었습니다.

:::warning 코닥(Kodak)의 교훈
필름 카메라로 유명한 코닥(Kodak)은 놀랍게도 **디지털 카메라를 최초로 개발한 회사**였습니다. 하지만 "디지털로의 전환은 오래 걸릴 것"이라 판단하고 빠르게 대처하지 않았습니다. 결국 시장에서 도태되어 버렸습니다. 이 사례는 시장 환경의 변화를 무시했을 때 어떤 결과가 오는지를 잘 보여줍니다.
:::

#### 예산 및 일정의 변경

프로젝트 진행 중에 예산이 줄거나 늘 수 있고, 납기 일정이 앞당겨지거나 늦춰질 수 있습니다. 이러한 변화는 소프트웨어의 기능 범위를 조정하게 만들고, 결과적으로 소프트웨어 자체의 변경을 야기합니다.

### 두 번째 원인: 기술의 변화 💻

기술 환경의 변화 역시 소프트웨어 변경을 강요합니다.

#### 하드웨어 사양의 발전

스마트폰의 해상도, CPU 성능, 메모리 용량은 매년 눈에 띄게 향상되고 있습니다. 10년 전 스마트폰 기준으로 만들어진 앱은 지금 최신 스마트폰의 성능을 제대로 활용하지 못합니다. 새로운 하드웨어 환경에 맞춰 소프트웨어도 업데이트되어야 합니다.

#### 통신망 환경의 발전

인터넷 속도는 3G → 4G(LTE) → 5G → 6G로 급격히 빨라지고 있습니다. 이 변화 덕분에 스마트폰으로 고화질(4K, 8K) 동영상을 끊김 없이 시청할 수 있게 되었습니다. 유튜브를 비롯한 영상 서비스들이 이 기술 변화에 발맞춰 서비스를 지속적으로 개선해온 것이 좋은 예입니다. 통신망이 빨라지면, 그 속도를 활용하는 새로운 서비스 개발이 요구됩니다.

#### 운영체제 및 플랫폼의 변화

Windows, iOS, Android 등 운영체제(OS)는 꾸준히 버전이 업데이트됩니다. 새로운 OS 버전에서 기존 앱이 제대로 동작하지 않을 수 있고, 새 OS가 제공하는 기능을 활용하기 위해 앱을 업데이트해야 합니다.

| 변경 원인 분류 | 세부 원인 | 예시 |
|---|---|---|
| **업무 환경 변화** | 고객의 새로운 기능 요구 | "결제 기능 추가해 주세요" |
| **업무 환경 변화** | 시장 환경의 변화 | PC → 모바일 전환, 종이책 → 전자책 |
| **업무 환경 변화** | 예산·일정 변경 | 프로젝트 비용 축소로 기능 범위 조정 |
| **기술 변화** | 하드웨어 사양 향상 | 고해상도 디스플레이 지원 |
| **기술 변화** | 통신망 발전 | 5G 환경에 맞는 초고화질 스트리밍 |
| **기술 변화** | 운영체제 업데이트 | 새 Android 버전 대응 |

:::tip 개발자도 시장을 읽어야 합니다
"나는 자바(Java)만 잘하면 된다"는 생각은 이제 통하지 않습니다. 성공하는 개발자는 기술 역량과 함께 **시장의 흐름과 고객의 니즈를 이해하는 능력**을 갖추고 있습니다. 소프트웨어는 결국 사람을 위해 만들어지기 때문입니다.
:::

## 📦 버전 관리: 변경을 숫자로 기록하다

### 버전(Version)이란?

소프트웨어의 변경 상태를 관리하는 가장 기본적인 방법이 바로 **버전(Version) 관리**입니다. 버전은 소프트웨어가 얼마나 변했는지를 숫자로 표현한 것입니다.

우리가 잘 아는 **Windows 95, Windows 98, Windows XP, Windows 10, Windows 11**도 버전의 변화입니다. 버전이 바뀔 때마다 기능, 디자인, 성능이 달라집니다.

### 풀 모델 체인지 vs 마이너 체인지

자동차를 예로 들면 이해하기 쉽습니다.

**풀 모델 체인지(Full Model Change)** 는 자동차의 엔진, 외관 디자인, 내부 인테리어가 **전부 다 바뀌는 것**입니다. 마치 소나타 1에서 소나타 EF로 완전히 새로운 모델이 나온 것처럼요. 소프트웨어에서는 **버전 1.0 → 2.0** 처럼 앞자리 숫자(메이저 버전)가 바뀌는 것을 의미합니다. 홈페이지에 들어갔더니 메뉴 위치도 다 바뀌고, 없던 기능도 생기고, 완전히 새로운 느낌이 드는 것이 풀 모델 체인지입니다.

**마이너 체인지(Minor Change)** 는 자동차의 기본 틀은 유지하면서 **일부분만 바뀌는 것**입니다. 소프트웨어에서는 **버전 1.0 → 1.1 → 1.1.1 → 1.1.2** 처럼 뒷자리 숫자(마이너 버전)가 바뀌는 것을 의미합니다. 큰 틀은 그대로지만 특정 기능이 개선되거나 버그가 수정되는 수준의 변경입니다.

| 구분 | 자동차 비유 | 소프트웨어 버전 예시 | 변경 범위 |
|---|---|---|---|
| **풀 모델 체인지** | 엔진·디자인·인테리어 모두 교체 | 1.0 → **2.0** | 전체 구조·기능 대규모 변경 |
| **마이너 체인지** | 외관 일부 수정, 내부 인테리어만 교체 | 1.0 → **1.1** → **1.1.2** | 부분 기능 추가·수정·버그 수정 |

### 릴리스(Release)란?

**릴리스**는 여러 버전의 산출물(문서, 코드 등)을 하나의 패키지로 묶어 관리하는 단위입니다. 마치 택배 상자에 여러 물건을 담아 보내듯이, 릴리스는 특정 시점의 요구사항 문서 버전 + 설계 문서 버전 + 소스 코드 버전을 한데 묶어 하나의 릴리스로 관리합니다.

예를 들어 설명해보겠습니다.

```
요구사항 명세서: 버전 1.1
     ↓
설계 문서:       버전 1.2 (요구사항 v1.1 기반으로 작성)
     ↓
소스 코드:       버전 1.2 (설계 문서 v1.2 기반으로 개발)
     ↓
─────────────────────────────────────
📦 릴리스 1.0 = 요구사항 v1.1 + 설계 v1.2 + 코드 v1.2
```

이렇게 릴리스로 묶어서 관리하면 "릴리스 1.0 기준으로 어떤 요구사항이 반영됐고, 어떤 설계를 따랐으며, 어떤 코드가 배포됐는지"를 언제든지 명확하게 파악할 수 있습니다.

:::info 릴리스 관리가 중요한 이유
개발자는 한 회사에 영원히 다니지 않습니다. 이직하거나, 새 팀원이 오거나, 협업이 필요할 수 있습니다. 릴리스가 체계적으로 관리되어 있으면 새로운 개발자도 과거의 히스토리(이력)를 손쉽게 파악하고 업무에 빠르게 투입될 수 있습니다.
:::

## 🗂️ 소프트웨어 형상관리(SCM)의 핵심 원리

### 형상관리의 정의

**소프트웨어 형상관리(Software Configuration Management, SCM)** 의 공식적인 정의는 다음과 같습니다.

> 개발 중 발생하는 모든 산출물(요구사항 명세서, 설계 문서, 소스 코드 등)의 변경을 체계적으로 관리하고 유지하는 전체 활동. 소프트웨어 개발 생명주기(SDLC) 전체에 걸쳐 생성되는 모든 산출물을 통합하여 관리한다.

여기서 중요한 것은 "산출물"이 단순히 소스 코드만을 의미하지 않는다는 점입니다. **요구사항 명세서, 개념적 설계 문서, 논리적 설계 문서, 물리적 설계 문서, 테스트 계획서** 등 개발 과정에서 나오는 모든 문서가 포함됩니다.

### 형상관리의 네 가지 핵심 활동

형상관리는 단순히 코드를 저장하는 것이 아닙니다. 다음 네 가지 핵심 활동으로 구성됩니다.

#### 1️⃣ 형상 식별 (Configuration Identification)

형상 식별은 형상관리의 **가장 기본이 되는 활동**입니다. "무엇을 관리할 것인가?"를 결정하는 단계입니다.

- 형상 항목(관리 대상)을 선정합니다.
- 각 항목에 이름과 버전 번호 등의 규칙을 정합니다.
- 그 규칙을 기반으로 **베이스라인(Baseline, 기준점)** 을 설정합니다.

**베이스라인**은 "특정 시점에 만들어진 산출물들의 집합"입니다. 마치 건물의 설계 도면에서 "이 도면이 최종 승인된 기준 도면"이라고 도장을 찍어두는 것과 같습니다. 베이스라인이 설정되면, 이후 변경이 필요할 때 이 기준점을 기준으로 무엇이 얼마나 달라졌는지 명확히 파악할 수 있습니다.

```
예시: 쇼핑몰 소프트웨어의 베이스라인

베이스라인 v1.0 (2025년 1월 기준):
  - 결제 기능: 신용카드 결제 지원
  - 할인율: 회원 10%, 비회원 0%
  - 주문 처리: 실시간 재고 연동
  - 배송 추적: 외부 API 연동
```

#### 2️⃣ 형상 통제 (Configuration Control)

형상 통제는 **임의로 변경하는 것을 막고, 정해진 절차에 따라서만 변경이 이루어지도록 통제**하는 활동입니다.

개발자가 혼자서 몰래 코드를 고치고 기존 파일 위에 덮어쓰는 행동은 절대로 해서는 안 됩니다. 변경은 반드시 **형상통제위원회(CCB: Configuration Control Board)** 의 심사와 승인을 거쳐야 합니다.

:::danger 무단 변경은 절대 금지!
"버그를 발견했으니 내가 알아서 고치겠다"며 팀원 몰래 소스 코드를 수정하는 행위는 형상관리의 가장 기본 원칙을 위반하는 것입니다. 5개의 오류를 내 마음대로 고쳤더니 10개의 새로운 문제가 생기는 상황이 실제로 발생합니다. 모든 변경은 반드시 공식적인 승인 절차를 거쳐야 합니다.
:::

**형상통제위원회(CCB)** 의 구성원은 다음과 같습니다.

| 역할 | 설명 |
|---|---|
| **PM (프로젝트 매니저)** | 전체 프로젝트를 총괄하는 관리자 |
| **형상관리 담당자** | 주로 시니어 개발자가 맡음 |
| **품질 담당자** | 소프트웨어 품질 검증 담당 |
| **기술 담당자** | 기술적 타당성 검토 |
| **고객 측 담당자** | 고객 요구사항을 대변하는 역할 |

CCB에서는 변경 요청을 받았을 때 다음을 종합적으로 판단합니다.

- 이 변경이 **왜 필요한가?**
- 계약 기간과 **일정에 맞는가?**
- 추가 **비용은 얼마나 드는가?**
- 유지보수에 어떤 **영향을 미치는가?**
- 변경의 **결과가 적절한가?**

#### 3️⃣ 형상 상태 보고 (Configuration Status Accounting)

승인된 변경 사항이 어떻게 진행되고 있는지를 **관련자 모두에게 보고하고 기록**하는 활동입니다.

누가, 언제, 무엇을, 왜 바꿨는지에 대한 기록이 반드시 남아 있어야 합니다. 형상관리 도구(Git, SVN 등)를 사용하면 이 이력이 자동으로 남습니다. 1년 전에 바뀐 코드라도 기록만 잘 되어 있으면 "아, 그때 고객이 이런 니즈가 있어서 이 기능을 추가했구나"라고 바로 확인할 수 있습니다.

#### 4️⃣ 형상 감사 (Configuration Audit)

형상 감사는 **"승인한 대로 제대로 만들어졌는가?"를 검증하는 활동**입니다.

CCB에서 승인한 내용대로 실제 개발이 이루어졌는지, 요구사항에 맞는지, 정해진 일정을 준수했는지 등을 단계별로 확인합니다. 감사 결과 문제가 없으면 감사 보고서를 작성하고 최종 승인이 이루어집니다.

```
형상관리 전체 흐름 요약:

변경 요청 접수
     ↓
형상통제위원회(CCB) 심사
     ↓
승인 또는 거부 결정
     ↓
[승인된 경우] 개발자 변경 작업 수행
     ↓
테스트 및 검증
     ↓
형상 감사 (계획대로 수행됐는지 확인)
     ↓
감사 보고서 작성 및 배포(릴리스)
```

### 형상관리의 목적

형상관리를 왜 해야 하는지 세 가지로 정리할 수 있습니다.

**첫째, 가시성(Visibility) 확보** — 소프트웨어의 현재 상태가 한눈에 보여야 합니다. 어떤 기능이 추가됐고, 어떤 버그가 해결됐고, 어떤 버전이 배포 중인지를 누구나 즉시 확인할 수 있어야 합니다.

**둘째, 추적성(Traceability) 확보** — "누가 언제 무엇을 왜 바꿨는가?"에 대해 항상 답할 수 있어야 합니다. 개발자가 바뀌어도, 담당자가 퇴사해도, 모든 변경의 이력이 시스템에 남아 있어야 합니다.

**셋째, 안정성과 생산성 향상** — 무분별한 변경을 사전에 차단하여 변경으로 인한 부작용(버그, 시스템 오류)을 최소화하고, 전체 프로젝트를 효율적으로 관리할 수 있게 합니다.

## ♻️ 소프트웨어 재사용: 3R 개념

### 3R이란?

완성된 소프트웨어를 활용해 새로운 소프트웨어를 만드는 방법론을 **3R**이라고 합니다. 3R은 다음 세 가지 R을 의미합니다.

| 약어 | 영어 | 한국어 |
|---|---|---|
| **Reuse** | Re-use | 재사용 |
| **Reverse Engineering** | Reverse Engineering | 역공학 |
| **Re-engineering** | Re-engineering | 재공학 |

3R의 핵심 목적은 다음과 같습니다.
- **유지보수 비용 절감** — 기존에 검증된 코드를 재사용하면 유지보수가 쉬워집니다.
- **개발 생산성 향상** — 처음부터 새로 만드는 것보다 훨씬 빠릅니다.
- **품질 향상** — 이미 검증된 모듈을 재활용하므로 오류 가능성이 줄어듭니다.
- **빠른 변경 대응** — 다양한 변경 요구에 신속하게 대응할 수 있습니다.

### 재사용(Reuse): 있는 것을 다시 쓴다 🔁

**재사용**은 기존에 만들어진 소프트웨어의 일부(부품)를 새로운 소프트웨어를 개발할 때 **그대로 가져다 사용하는 것**입니다. 마치 레고 블록에서 이미 만들어진 특정 부품을 떼어다 새로운 작품을 만드는 것과 같습니다.

예를 들어, 기존 쇼핑몰 홈페이지에 결제 기능, 게시판 기능, 자료실 기능이 있다고 합시다. 새로운 홈페이지를 만들 때 이 세 기능이 여전히 필요하다면 처음부터 새로 개발하는 게 아니라 기존 것을 가져다 쓰면 됩니다. 새로 필요한 기능만 추가 개발하면 되므로 개발 속도와 효율성이 크게 올라갑니다.

재사용의 단위는 다양합니다.

- **서브루틴 / 라이브러리** — 특정 기능을 수행하는 함수나 라이브러리를 가져다 씁니다.
- **코드 블록** — 잘 작성된 코드 조각을 가져다 수정 없이 또는 최소한의 수정으로 씁니다.
- **모듈** — 독립적으로 동작하는 하나의 기능 단위를 통째로 가져다 붙입니다.

```python
# ✅ 재사용의 올바른 예시
# 기존 프로젝트에서 이미 검증된 결제 모듈을 import하여 재사용

from legacy_payment_module import PaymentProcessor  # 기존에 만들어진 결제 모듈 가져오기

class NewShoppingMall:
    def __init__(self):
        # 기존 결제 모듈 재사용 - 처음부터 새로 만들지 않음
        self.payment = PaymentProcessor()
    
    def checkout(self, cart_items, card_info):
        total = sum(item.price for item in cart_items)  # 장바구니 합계 계산
        # 검증된 기존 결제 모듈 그대로 사용
        result = self.payment.process(total, card_info)
        return result
```

```python
        if card_info.validate():
```

재사용 여부를 결정할 때는 다음 기준을 고려합니다.

- 모듈의 **복잡도**가 적절한가?
- **정규화(Normalization)** 가 잘 되어 있는가?
- **객체지향(OOP)** 언어로 작성되어 있는가? (재사용성이 높음)
- **라이브러리**가 잘 정비되어 있는가?

### 역공학(Reverse Engineering): 거꾸로 분석한다 🔍

**역공학**은 이름 그대로 "거꾸로(Reverse) 가는 공학"입니다. 완성된 결과물(소스 코드)을 가지고 거꾸로 분석하여 설계도, 요구사항 명세서 등의 문서를 복원해 내는 방법입니다.

자동차를 예로 들면, 설계 도면 없이 완성된 수입 자동차를 분해하여 그 구조를 파악하고, 역으로 설계도를 그려내는 것과 같습니다.

**언제 역공학이 필요한가?**

실무에서 이런 상황이 자주 발생합니다.

- 요구사항 정의서가 없습니다.
- 기능 명세서가 없습니다.
- 설계 문서가 없습니다.
- 심지어 코드 주석도 없습니다.

그렇다면 어떻게 해야 할까요? 기존에 만들어진 소스 코드를 분석하여 거꾸로 문서를 만들어 나가야 합니다. 이것이 역공학입니다.

```
역공학의 흐름:

소스 코드 분석
     ↓
결과물(컴포넌트, 관계) 추출
     ↓
DB에 저장
     ↓
다이어그램(설계도) 자동 생성
     ↓
설계 문서 복원
     ↓
(필요 시) 요구사항 명세서 복원
```

```java
// ✅ 역공학 예시 - 주석 없는 레거시(오래된) 코드를 분석하여 문서화하는 과정

// [역공학 전] 문서가 없는 기존 레거시 코드
public double calcDiscount(String memberType, double price) {
    if (memberType.equals("VIP")) {
        return price * 0.8;  // 20% 할인
    } else if (memberType.equals("REGULAR")) {
        return price * 0.9;  // 10% 할인
    }
    return price;  // 할인 없음
}

// [역공학 후] 코드를 분석하여 복원한 요구사항 명세
/*
 * 요구사항 명세서 (역공학으로 복원)
 * 기능명: 회원 등급별 할인 계산
 * 규칙:
 *   - VIP 회원: 정가의 20% 할인
 *   - 일반(REGULAR) 회원: 정가의 10% 할인
 *   - 비회원: 할인 없음 (정가 적용)
 * 복원일: 2026-03-03
 */
```

:::note 역공학을 방해하는 기술
일부 고급 소프트웨어나 하드웨어 장치는 역공학을 원천적으로 막기 위해 **분해 시 시스템이 자동으로 다운되거나 동작하지 않도록** 설계되어 있습니다. 이는 지적 재산권(IP)을 보호하기 위한 기술입니다.
:::

### 재공학(Re-engineering): 구조를 개선한다 🏗️

**재공학**은 역공학과 비슷하지만 목적이 다릅니다. 역공학이 "기존 코드로부터 문서를 복원"하는 것이라면, 재공학은 **기존 시스템의 구조를 개선하여 유지보수가 쉽고 효율적인 시스템으로 다시 만드는 것**입니다.

아파트 재건축을 떠올려 보세요. 낡은 아파트를 완전히 허물고 새로 짓는 것(신축)이 있고, 기존 구조를 최대한 살리면서 리모델링하는 것(재건축)이 있습니다. 소프트웨어 재공학은 후자에 가깝습니다. 기존 시스템의 기능은 유지하면서 **내부 구조(아키텍처)를 개선**하는 것입니다.

**언제 재공학이 필요한가?**

- 유지보수 비용이 너무 많이 들 때
- 코드 구조가 복잡하고 이해하기 어려울 때(스파게티 코드)
- 새로운 기능 추가 시 다른 기능에 영향을 많이 줄 때
- 시스템 성능이 저하되었을 때

재공학은 **PDCA 모델**을 따릅니다.

| 단계 | 영문 | 설명 |
|---|---|---|
| **P** | Plan (계획) | 어떤 부분을 어떻게 개선할지 전략 수립 |
| **D** | Do (실행) | 계획에 따라 구조 개선 작업 수행 |
| **C** | Check (검증) | 개선 결과가 목적에 맞는지 테스트 |
| **A** | Action (적용) | 검증이 완료된 시스템을 실제 서비스에 배포 |

```
재공학 프로세스:

현재 시스템 분석 → 개선 대상 식별
         ↓
개선 전략 수립 (어떻게 바꿀 것인가?)
         ↓
구조 개선 구현 (코드 리팩토링 등)
         ↓
개선 결과 평가 (목적 달성 여부 확인)
         ↓
실제 시스템에 적용 및 배포
```

```python
    total = 0

    if discount_type == 'VIP':
        total *= 0.8  # VIP 할인
    if payment_type == 'card':
        # ... 카드 결제 처리 (100줄짜리 코드)
        pass
    # ... 배송지 저장 (50줄짜리 코드)
    # ... 알림 발송 (30줄짜리 코드)
    # 모든 기능이 하나에 뒤섞여 수정 시 다른 기능에 영향을 줌

# ✅ 재공학 후 - 구조를 개선하여 역할별로 분리 (유지보수 용이)
class OrderProcessor:
    def __init__(self, payment_service, shipping_service, notification_service):
        # 각 역할을 담당하는 서비스를 주입받음 (역할 분리)
        self.payment = payment_service        # 결제만 담당
        self.shipping = shipping_service      # 배송만 담당
        self.notification = notification_service  # 알림만 담당

    def calculate_total(self, items, discount):
        # 금액 계산만 담당하는 독립된 메서드
        subtotal = sum(item.price * item.qty for item in items)  # 소계 계산
        return discount.apply(subtotal)  # 할인 적용

    def process(self, order):
        total = self.calculate_total(order.items, order.discount)  # 금액 계산
        self.payment.charge(order.user, total, order.payment_type)  # 결제 처리
        self.shipping.schedule(order.address, order.items)          # 배송 예약
        self.notification.send(order.user, order)                   # 알림 발송
        # 각 기능이 독립적으로 분리되어 있어 수정이 쉬움
```

## ⚠️ 주의사항과 실무 팁

### 형상관리를 생략했을 때의 위험성

소규모 팀이나 스타트업에서는 "문서 작성이 시간 낭비"라는 생각에 형상관리를 생략하는 경우가 많습니다. 하지만 이는 나중에 엄청난 비용을 치르게 됩니다.

형상관리가 안 된 프로젝트에서는 다음 문제가 발생합니다.

- 오류가 발생했을 때 **어떤 변경이 원인인지 알 수 없습니다**.
- 새로운 팀원이 왔을 때 **기존 코드를 이해하는 데 수개월이 걸립니다**.
- 특정 기능이 왜 그렇게 만들어졌는지 **아무도 기억하지 못합니다**.
- 고객이 "예전 기능으로 돌려달라"고 할 때 **이전 버전을 복구할 수 없습니다**.

:::warning 작은 팀이라도 형상관리는 필수입니다
"우리 팀은 3명뿐인데 굳이 이런 거 해야 하나요?"라고 묻는 개발자가 많습니다. 팀 규모에 상관없이 형상관리는 필수입니다. 오히려 소규모 팀일수록 한 사람이 나가면 지식이 통째로 사라지기 때문에 체계적인 관리가 더욱 중요합니다. Git을 활용하면 최소한의 노력으로도 기본적인 형상관리가 가능합니다.
:::

### 좋은 형상관리를 위한 실무 팁

**1. 커밋 메시지를 명확하게 작성하세요.**

```bash
# ❌ 나쁜 커밋 메시지 예시
git commit -m "수정함"
git commit -m "버그 고침"
git commit -m "기능 추가"

# ✅ 좋은 커밋 메시지 예시 (왜 바꿨는지가 명확함)
git commit -m "[기능추가] VIP 회원 20% 할인 기능 추가 - 고객 요구사항 #123 반영"
git commit -m "[버그수정] 결제 완료 후 재고 업데이트 안 되는 문제 수정"
git commit -m "[성능개선] 상품 목록 조회 쿼리 최적화 - 조회 속도 3배 향상"
```

**2. 변경 요청은 문서로 남기세요.**

아무리 작은 변경이라도 "누가, 왜, 무엇을 변경해달라고 요청했는지"를 기록해 두세요. 이슈 트래커(Jira, GitHub Issues 등)를 적극 활용하면 좋습니다.

**3. 베이스라인을 주기적으로 설정하세요.**

중요한 기능 개발이 완료될 때마다, 또는 일정 기간마다 베이스라인을 설정하고 태그(Tag)를 달아두세요. 나중에 문제가 생겼을 때 해당 시점으로 빠르게 돌아갈 수 있습니다.

```bash
# Git에서 베이스라인을 태그로 설정하는 예시
git tag -a v1.0.0 -m "릴리스 1.0: 기본 쇼핑몰 기능 완성 (2026-01-01)"
git tag -a v1.1.0 -m "릴리스 1.1: VIP 할인 기능 추가 (2026-02-01)"
```

### CASE 도구: 자동화로 생산성을 높이다 🛠️

과거에는 형상관리, 역공학, 재공학 모두 수작업으로 처리했습니다. 하지만 지금은 이 모든 것을 지원하는 **CASE(Computer-Aided Software Engineering) 도구**가 존재합니다.

CASE 도구는 소프트웨어 개발 과정 전체를 자동화·통합하는 도구의 총칭입니다.

- **버전 관리**: Git, SVN
- **소스코드 분석 및 다이어그램 자동 생성**: Rational Rose 등
- **자동 코드 생성**: 설계 문서에서 기본 코드 자동 생성
- **자료 흐름도(DFD), ER 다이어그램 자동 생성** 등

:::tip CASE 도구 활용 역량도 중요합니다
이론을 아는 것만큼, **다양한 개발 도구를 능숙하게 다루는 것**도 현대 개발자에게 매우 중요한 역량입니다. Git을 기본으로, 이슈 트래커, CI/CD 도구 등을 적극적으로 습득하고 업무에 도입하는 개발자가 더 높은 가치를 인정받습니다.
:::

## ✅ 핵심 정리

이번 글에서 배운 핵심 개념을 정리합니다.

- **소프트웨어는 완성 후에도 끊임없이 변경된다** — 기능 추가, 버그 수정, 환경 변화 등 수많은 이유로 변경이 발생한다.
- **소프트웨어 변경의 원인은 크게 두 가지** — ① 고객 요구·시장 환경·예산 변화 등 **업무 환경 변화**, ② 하드웨어 발전·통신망 고도화·OS 업데이트 등 **기술 변화**.
- **코닥의 실패는 시장 변화 무시의 결과** — 기술적 역량이 있어도 시장의 흐름을 읽지 못하면 도태된다.
- **버전 관리의 두 유형** — 메이저 버전 변경은 **풀 모델 체인지**, 마이너 버전 변경은 **마이너 체인지**.
- **릴리스(Release)** — 요구사항·설계·코드 각각의 버전을 하나의 패키지로 묶어 관리하는 단위.
- **형상관리(SCM)의 4대 활동** — ① 형상 식별 → ② 형상 통제(CCB 승인) → ③ 형상 상태 보고 → ④ 형상 감사.
- **베이스라인(Baseline)** — 특정 시점에 승인된 산출물 집합으로, 이후 변경의 기준점이 된다.
- **형상통제위원회(CCB)** — 모든 변경 요청을 심사하고 승인·거부를 결정하는 조직.
- **3R** — 재사용(Reuse), 역공학(Reverse Engineering), 재공학(Re-engineering).
- **재사용(Reuse)** — 기존 소프트웨어의 검증된 부품(모듈, 라이브러리 등)을 새 개발에 활용하는 것.
- **역공학(Reverse Engineering)** — 완성된 소스 코드를 거꾸로 분석하여 설계 문서·요구사항 명세서를 복원하는 것.
- **재공학(Re-engineering)** — 기존 시스템의 구조를 개선하여 유지보수 비용을 줄이고 효율성을 높이는 것. PDCA 모델 적용.
- **CASE 도구** — 소프트웨어 개발 전 과정을 자동화·통합하는 도구(Git, Rational Rose 등)로, 현대 개발자의 필수 역량.
- **형상관리 없이는 오류 추적이 불가능** — 무분별한 변경은 부작용을 키우고, 관리되지 않은 변경은 나중에 엄청난 비용을 치르게 한다.
