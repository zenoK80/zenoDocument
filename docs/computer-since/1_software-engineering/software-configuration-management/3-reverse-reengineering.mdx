# 🔧 역공학과 재공학의 개념 — 소프트웨어 형상관리부터 3R까지 완전 정복

소프트웨어는 한 번 만들면 끝일까요? 절대 그렇지 않습니다. 여러분이 매일 쓰는 스마트폰 앱, 게임, 웹사이트는 끊임없이 업데이트되고, 기능이 추가되고, 디자인이 바뀝니다. 이 글에서는 그 변화를 체계적으로 관리하는 **소프트웨어 형상관리(SCM)**의 개념부터, 기존 소프트웨어를 분석하고 개선하는 **역공학(Reverse Engineering)**과 **재공학(Reengineering)**까지, 초보자도 쉽게 이해할 수 있도록 하나하나 풀어서 설명합니다.

---

## 🏗️ 소프트웨어 형상관리란 무엇인가?

### 소프트웨어는 왜 계속 바뀌는가?

형상관리를 이해하려면 먼저 "왜 소프트웨어는 계속 바뀌는가?"라는 질문에 답할 수 있어야 합니다. 우리가 일상에서 쓰는 카카오톡, 인스타그램, 유튜브 앱을 생각해보세요. 이 앱들은 처음 출시된 모습 그대로 지금까지 유지되고 있을까요? 전혀 그렇지 않습니다. 새로운 기능이 추가되고, 불필요한 기능은 사라지고, 디자인이 바뀌고, 버그가 수정됩니다. 이처럼 **소프트웨어는 살아있는 생명체처럼 끊임없이 변화**합니다.

학자 **에릭 브로드(Eric Braude)**는 이렇게 말했습니다.

> 프로젝트는 진행되어 가면서 새로운 산출물(결과물)들이 만들어지게 되고, 그 산출물들은 계속 쌓여가면서 버전업(Version Up) 된다.

즉, 한 번 만들어 놓은 소프트웨어라도 시간이 지남에 따라 새로운 기능이 추가되고, 기존 기능이 개선되거나 제거되면서 계속 버전이 올라가게 됩니다. 여러분이 즐기는 온라인 게임도 마찬가지입니다. 새 아이템이 추가되고, 불균형한 스킬이 너프(약화)되거나 버프(강화)되고, 새로운 맵이 추가되지 않으면 유저들로부터 외면받게 됩니다.

또 다른 학자는 이런 말을 남겼습니다.

> 시스템은 소프트웨어 개발 생명주기(SDLC, Software Development Life Cycle)의 모든 단계에서 변경이 일어난다.

여기서 중요한 점은 **코드(프로그램 소스)만 바뀌는 게 아니라는 것**입니다. 요구사항 문서도 바뀌고, 설계도 바뀌고, 테스트 항목도 바뀝니다. 그리고 그 변경에 대한 욕구는 개발 생명주기 동안 **지속적으로 발생**합니다. 바꿔놓고 보니 더 좋은 아이디어가 생기고, 바꿔놓고 보니 새로운 문제가 생기고, 또 바꿔야 하는 상황이 반복됩니다.

### 변경은 왜 생기는가? — 두 가지 큰 원인

변경의 원인을 정확하게 알아야, 개발자로서 여유롭고 전략적으로 대처할 수 있습니다. 변경의 원인은 크게 **업무 환경의 변화**와 **기술의 변화** 두 가지로 나눌 수 있습니다.

**① 업무 환경의 변화**

고객이 새로운 기능을 요구하거나, 시장의 여건(환경)이 바뀌면 소프트웨어도 그에 맞춰 변해야 합니다. 예를 들어볼까요? 과거에는 사람들이 종이책을 주로 읽었지만, 지금은 스마트폰과 태블릿으로 전자책(e-Book)을 읽는 시대가 되었습니다. 지하철에서 종이책을 읽는 사람보다 스마트폰을 보는 사람이 훨씬 많죠. 그래서 출판사들은 단순히 종이책만 만들어서는 안 되고, **오디오북, 모바일 뷰어, 태블릿 필기 기능** 등 디지털 환경에 맞는 새로운 서비스를 제공해야 합니다. 이런 변화에 따라오지 못하면 시장에서 도태될 수밖에 없습니다.

대표적인 역사적 사례가 있습니다. **코닥(Kodak)**이라는 필름 카메라 회사는 놀랍게도 **디지털 카메라를 세계 최초로 개발한 회사**였습니다. 하지만 "디지털로 넘어가는 데는 오래 걸릴 것"이라고 판단하고 빠르게 대처하지 못했습니다. 결국 디지털 카메라 시대가 오자 시장에서 퇴출당하고 말았습니다. 이처럼 **시장 여건의 변화와 고객 요구사항의 변화**를 무시하면 아무리 기술이 뛰어난 회사도 살아남기 어렵습니다.

**② 기술의 변화**

하드웨어 성능이 향상되고, 인터넷 속도가 빨라지고, 새로운 디바이스(기기)가 등장하면 소프트웨어도 그에 맞게 변해야 합니다. 지금은 5G를 넘어 6G까지 개발되고 있고, 스마트폰 해상도는 점점 높아지고 있습니다. 덕분에 고화질 동영상을 어디서든 끊김 없이 볼 수 있게 되었고, 유튜브는 TV 시청 시간을 능가하는 미디어가 되었습니다. 이런 기술 변화에 맞춰 소프트웨어도 계속 업데이트되어야 합니다.

:::info 개발자에게도 인문학적 소양이 필요한 이유
많은 사람들이 "개발자는 코딩만 잘하면 되는 거 아닌가?"라고 생각합니다. 하지만 소프트웨어는 항상 고객의 다양한 요구에 따라 변경되기 때문에, 개발자도 **시장의 흐름, 고객의 니즈(필요), 업무 환경의 변화**를 이해하고 있어야 합니다. 기술적으로 완벽한 프로그램이라도 고객이 원하는 것과 다르면 시장에서 실패합니다. 반대로 시장과 고객을 잘 이해한 개발자가 만든 소프트웨어는 성공할 가능성이 훨씬 높습니다.
:::

## 📦 버전 관리와 릴리스(Release)의 개념

### 버전(Version)이란 무엇인가?

소프트웨어의 변경 이력을 관리하는 가장 기본적인 수단이 바로 **버전(Version)**입니다. 윈도우(Windows) 운영체제를 예로 들면, Windows 95 → 98 → 2000 → XP → 7 → 10 → 11처럼 계속 버전업이 되어왔습니다. 버전을 관리하면 "지금 내가 쓰고 있는 것이 어떤 상태인지", "이전 버전과 무엇이 다른지"를 명확하게 알 수 있습니다.

버전 변화는 크게 두 가지로 나뉩니다.

| 구분 | 설명 | 예시 |
|---|---|---|
| **풀 모델 체인지 (Major Change)** | 전체가 통째로 바뀌는 것. 디자인, 엔진, 기능 모두 교체 | 버전 1.0 → 2.0 (앞자리 숫자 변경) |
| **마이너 체인지 (Minor Change)** | 일부만 수정되는 것. 부분적인 개선 | 버전 1.0 → 1.1 → 1.1.1 (뒷자리 숫자 변경) |

자동차로 비유해보면 이해하기 쉽습니다. 현대자동차의 **소나타**는 1990년대부터 지금까지 같은 이름을 쓰고 있지만, 소나타 1 → EF 소나타 → NF 소나타처럼 계속 버전업이 되어왔습니다. 엔진도 바뀌고, 내부 인테리어도 바뀌고, 외관 디자인도 바뀐 것이 **풀 모델 체인지**입니다. 반면 외관은 그대로인데 내부 인테리어만 조금 바꾸거나, 엔진 성능을 약간 개선한 것은 **마이너 체인지**입니다.

소프트웨어도 똑같습니다. 여러분이 즐겨 쓰는 웹사이트에 들어갔더니 메뉴 위치가 완전히 바뀌고, 화면 구성이 전혀 달라졌다면 풀 모델 체인지가 된 것입니다. 반면 버튼 색상이 조금 바뀌거나 메뉴 하나가 추가된 정도라면 마이너 체인지입니다.

### 릴리스(Release)란 무엇인가?

버전을 이해했다면, 이제 **릴리스(Release)**라는 개념을 알아야 합니다. 릴리스는 단순히 코드의 버전만을 관리하는 것이 아니라, **관련된 모든 산출물(결과물)을 하나의 패키지로 묶어서 관리하는 것**입니다.

소프트웨어를 만들면 다양한 문서와 코드가 만들어집니다.

- **요구사항 분석 명세서**: 고객이 원하는 기능을 정리한 문서 (버전 1.0 → 1.1 → 1.2... 로 관리)
- **설계 문서**: 요구사항을 바탕으로 만든 설계도 (버전 1.0 → 1.1 → 1.4... 로 관리)
- **원시 코드(소스 코드)**: 설계를 기반으로 실제로 짠 프로그램 (버전 1.0 → 1.5... 로 관리)

이 세 가지가 각각 따로 버전 관리가 되고 있는데, 이것들을 서로 연결해서 **"이 릴리스는 요구사항 버전 1.1 + 설계 버전 1.2 + 코드 버전 1.2로 구성된 패키지입니다"** 라고 묶어서 관리하는 것이 바로 릴리스입니다.

예를 들어, 릴리스 1.0은 요구사항 v1.1 + 설계 v1.2 + 코드 v1.1을 묶은 것이고, 릴리스 3.0은 요구사항 v1.3 + 설계 v1.4 + 코드 v1.5를 묶은 것입니다. 이렇게 관리해야 "릴리스 2.0 때 어떤 요구사항이 반영됐고, 어떤 설계를 기반으로 어떤 코드가 만들어졌는지"를 명확하게 추적할 수 있습니다.

:::tip 왜 릴리스 관리가 중요한가?
개발자는 한 회사에서 영원히 일하지 않습니다. 언제든지 이직하거나, 새로운 개발자가 팀에 합류할 수 있습니다. 이때 릴리스 관리가 잘 되어 있으면, 새로 온 개발자도 "아, 이 버전에서는 이런 요구사항이 반영됐고, 이런 설계를 기반으로 코드가 만들어졌구나"를 바로 파악할 수 있습니다. 반대로 관리가 안 되어 있으면, 오류가 생겼을 때 원인을 찾는 것 자체가 엄청난 고생이 됩니다.
:::

## 🗂️ 소프트웨어 형상관리(SCM)의 정의와 원칙

### 형상관리(SCM)란?

**소프트웨어 형상관리(SCM, Software Configuration Management)**는 소프트웨어 개발 과정에서 발생하는 모든 산출물(요구사항 문서, 설계 문서, 소스 코드 등)의 변경을 체계적으로 관리하고 유지하는 전체 활동을 말합니다.

여기서 중요한 포인트는 두 가지입니다.

**첫째**, 관리 대상이 코드만이 아닙니다. 요구사항 명세서, 개념적 설계 문서, 논리적 설계 문서, 물리적 설계 문서, 테스트 계획서 등 **소프트웨어 개발 과정에서 만들어지는 모든 문서와 산출물**이 관리 대상입니다.

**둘째**, 특정 시점에만 하는 것이 아닙니다. **소프트웨어 개발 생명주기(SDLC) 전체에 걸쳐** 지속적으로 이루어집니다.

> 형상관리(SCM)는 변경 과정을 체계적으로 관리하고 유지하는 소프트웨어 개발 생명주기 전반의 통합 관리 활동입니다.

### 형상관리의 4가지 핵심 활동

형상관리는 단순히 "파일을 저장하고 백업하는 것"이 아닙니다. 다음 네 가지 핵심 활동으로 구성됩니다.

#### 1️⃣ 형상 식별 (Configuration Identification)

형상관리의 가장 기본이 되는 활동입니다. **"무엇을 관리할 것인가?"**를 정하는 것입니다. 프로젝트를 시작할 때 형상관리 계획을 세우고, 관리 대상이 되는 **형상 항목(Configuration Item)**을 식별합니다.

형상 항목을 식별한 후에는 **베이스라인(Baseline)**을 설정합니다. 베이스라인이란 특정 시점에 만들어진 산출물들의 집합, 즉 **기준점**을 의미합니다. 마치 건물을 지을 때 기준이 되는 바닥 면(기초 공사)처럼, 현재 시스템이 어떤 상태인지를 명확하게 정의한 기준선이라고 생각하면 됩니다.

예를 들어, 쇼핑몰 시스템의 베이스라인을 설정할 때는 이런 내용이 포함됩니다.

- 결제는 어떤 방식으로 처리하도록 했는가
- 주문은 어떤 방식으로 처리하도록 했는가
- 할인율은 몇 퍼센트가 적용되도록 했는가

A팀은 버전 1.2, B팀은 버전 1.4, C팀은 버전 1.3을 각각 개발하고 있다면, 이 모든 팀의 버전 정보를 통합해서 전체 시스템의 베이스라인을 관리하게 됩니다.

#### 2️⃣ 형상 통제 (Configuration Control)

형상 통제는 **"변경을 아무나 마음대로 할 수 없도록 통제하는 것"**입니다. 개발자가 오류를 발견했을 때 아무도 모르게 혼자 코드를 수정하고 덮어버리면 절대로 안 됩니다. 왜냐하면 그 변경이 다른 부분에 어떤 영향을 미칠지 모르기 때문입니다.

실제로 다섯 개의 오류를 수정했더니 열 개의 새로운 문제가 생기는 경우가 있습니다. 이런 **무분별한 변경이 가져오는 연쇄적인 부작용**을 막기 위해 형상 통제가 필요합니다.

변경은 다음과 같은 순서로 진행됩니다.

1. **변경 요청**: 개발팀, 운영팀, 홍보팀, 고객서비스팀 등에서 변경이 필요하다고 요청합니다.
2. **심사 및 승인**: **형상 통제위원회(CCB, Configuration Control Board)**에서 변경의 필요성, 비용, 일정, 영향도 등을 종합적으로 검토하여 승인 또는 거부합니다.
3. **구현**: 승인을 받은 후에야 개발자가 실제로 변경 작업을 수행합니다.
4. **검증**: 변경이 제대로 이루어졌는지 테스트합니다.

**형상 통제위원회(CCB)**는 거창한 이름이지만, 실제로는 프로젝트 관리자(PM), 형상 관리 담당자, 품질 담당자, 기술 담당자, 고객 측 담당자 등으로 구성된 의사결정 그룹입니다. 이들이 함께 모여 변경 요청을 검토하고, "이 변경이 정말 필요한가? 계약 기간과 예산은 맞는가? 유지보수에는 어떤 영향을 미치는가?"를 판단합니다.

#### 3️⃣ 형상 상태 보고 (Configuration Status Accounting)

변경이 이루어진 후에는 **변경 처리 상태를 반드시 기록하고 관련된 사람들에게 보고**해야 합니다. "누가, 언제, 무엇을, 왜 바꿨는가?"에 대한 명확한 기록이 남아 있어야 합니다.

:::warning 작은 회사일수록 이 단계가 생략되기 쉽습니다
규모가 작은 회사나 개발자 수가 적은 팀에서는 문서 작업과 보고 절차가 번거롭다는 이유로 이 단계를 생략하는 경우가 많습니다. 하지만 이것이 나중에 얼마나 큰 문제를 일으키는지 모릅니다. 변경 사항이 쌓여가는데 아무것도 기록이 없으면, 나중에 오류가 발생했을 때 "이 코드를 누가 왜 바꿨지?"를 역추적하는 것이 엄청난 고생이 됩니다.
:::

#### 4️⃣ 형상 감사 (Configuration Audit)

형상 감사는 **"계획한 대로 변경이 제대로 이루어졌는지 확인하는 것"**입니다. 위원회에서 승인을 내줬다고 해서 끝이 아닙니다. 실제로 그 계획대로 개발이 진행되고 있는지, 요구사항에 맞게 구현되고 있는지, 정해진 베이스라인 기준에 부합하는지를 단계별로 평가하고 검증합니다. 문제가 없다면 감사 보고서를 작성하여 "원하는 방향대로 문제없이 잘 만들어졌다"는 것을 공식적으로 확인합니다.

### 형상관리의 목적 — 왜 이렇게 복잡한 절차가 필요한가?

형상관리를 하는 이유는 크게 세 가지입니다.

| 목적 | 설명 |
|---|---|
| **가시성 확보** | 현재 소프트웨어가 어떤 상태인지, 어떤 기능이 추가됐고 어떤 오류가 해결됐는지를 한눈에 볼 수 있게 합니다. |
| **추적성 확보** | 누가, 언제, 무엇을, 왜 바꿨는지 언제든지 답변할 수 있도록 이력(히스토리)을 관리합니다. |
| **생산성·안전성 향상** | 무분별한 변경으로 인한 부작용을 최소화하고, 유지보수를 효율적으로 할 수 있게 합니다. |

## ♻️ 소프트웨어 재사용과 3R의 개념

### 3R이란 무엇인가?

소프트웨어 형상관리를 통해 버전을 관리하다 보면, 기존에 만들어진 소프트웨어를 새로운 방식으로 활용하는 개념이 등장합니다. 이것이 바로 **3R**입니다.

3R은 세 가지 R의 약자입니다.

| 약자 | 영어 | 한국어 |
|---|---|---|
| **1R** | Reverse Engineering | **역공학** |
| **2R** | Re-Engineering | **재공학** |
| **3R** | Reuse | **재사용** |

> 3R의 목표는 유지보수를 쉽게 하고, 개발 비용을 절감하며, 생산성을 높이고, 빠르게 변화하는 요구사항에 대응하기 위함입니다.

요즘 소프트웨어는 점점 더 크고 복잡해지고 있는 반면, 소프트웨어의 수명(라이프 사이클)은 점점 짧아지고 있습니다. 이런 상황에서 모든 것을 처음부터 새로 만드는 것은 시간과 비용 측면에서 매우 비효율적입니다. 그래서 기존 소프트웨어를 잘 활용하는 3R 전략이 중요합니다.

### 재사용(Reuse) — 기존 부품을 갖다 쓰기

**재사용(Reuse)**은 이미 만들어진 소프트웨어의 일부를 새로운 소프트웨어를 개발할 때 그대로 가져다 쓰는 것입니다. 마치 레고 블록처럼, 기존에 잘 만들어진 부품을 새로운 조합으로 사용하는 것입니다.

예를 들어, 기존 쇼핑몰 홈페이지에 **결제 모듈**, **게시판 기능**, **자료실 기능**이 있다고 가정해봅시다. 새로운 홈페이지를 만들 때 이 기능들이 그대로 필요하다면, 처음부터 다시 개발할 필요 없이 기존 코드를 가져와서 쓰면 됩니다. 이렇게 하면 개발 속도와 효율성이 크게 높아집니다.

재사용 가능한 단위는 다음과 같습니다.

- **서브루틴(Subroutine)이나 라이브러리(Library)**: 특정 기능을 수행하는 함수나 함수 묶음
- **코드 블록(Code Block)**: 특정 로직을 수행하는 코드 조각
- **모듈(Module)**: 독립적인 기능을 갖춘 코드 단위

```python
# ✅ 재사용 예시: 기존에 만들어둔 결제 모듈을 새 프로젝트에서 가져다 씀

# 기존 프로젝트에서 만들어둔 결제 처리 함수 (재사용 가능한 모듈)
def process_payment(user_id, amount, payment_method):
    """

    - payment_method: 결제 수단 (카드, 계좌이체 등)
    """
    # 결제 금액이 0보다 커야 처리 가능
    if amount <= 0:
        return {"status": "fail", "message": "결제 금액이 유효하지 않습니다."}
    
    # 결제 수단에 따라 처리 방식 분기
    if payment_method == "card":
        result = card_payment(user_id, amount)   # 카드 결제 처리
    elif payment_method == "bank":
        result = bank_transfer(user_id, amount)  # 계좌이체 처리
    else:
        return {"status": "fail", "message": "지원하지 않는 결제 수단입니다."}
    
    return result  # 결제 결과 반환

# ✅ 새 프로젝트(예: 새 쇼핑몰)에서 위 함수를 그대로 재사용
# from old_payment_module import process_payment  # 기존 모듈 임포트
result = process_payment("user_001", 50000, "card")  # 5만원 카드 결제
print(result)  # 결제 결과 출력
```

**재사용을 결정할 때 고려해야 할 기준**:
- 재사용할 부품(모듈)의 크기와 복잡도가 적절한가?
- 코드가 잘 정리(정규화)되어 있는가?
- 객체지향 언어로 작성되어 있어 재사용하기 좋은 구조인가?
- 라이브러리가 잘 갖춰져 있는가?

재사용을 하게 되면 **개발 품질 향상, 개발 속도 향상, 비용 절감, 팀 간 정보 공유**가 모두 좋아집니다.

## 🔍 역공학(Reverse Engineering)과 재공학(Reengineering)

### 역공학(Reverse Engineering) — 거꾸로 분석하기

**역공학(Reverse Engineering)**은 이름에서 알 수 있듯이, **완성된 결과물을 거꾸로 분석해서 설계도나 요구사항을 찾아내는 작업**입니다.

하드웨어 분야에서 역공학의 유명한 예시가 있습니다. 우리나라가 자동차 제조 기술이 없던 시절, 수입 자동차를 분해해서 설계도를 파악하고 이를 바탕으로 우리 기술로 자동차를 만들어낸 것이 바로 역공학의 원리입니다. 완성품으로부터 설계도를 역으로 해독해가는 것입니다.

소프트웨어 역공학이 필요한 가장 현실적인 상황을 생각해봅시다.

> 어느 회사에 새로 입사했더니, 요구사항 정의서도 없고, 기능 명세서도 없고, 설계 문서도 없습니다. 오직 옛날에 짜놓은 소스 코드만 있습니다. 이 소프트웨어를 유지보수해야 하는데, 어떻게 해야 할까요?

이때 바로 역공학이 필요합니다. **기존 소스 코드를 분석하여 설계 문서와 요구사항 명세서를 역으로 만들어내는 것**이 역공학입니다. 코드 → 설계도 → 요구사항 순서로 거꾸로 올라가는 것입니다.

역공학의 진행 과정을 간단히 나타내면 다음과 같습니다.

```
[원시 코드(소스 코드)]
       ↓ 분석
[결과물 추출 → DB에 저장]
       ↓
[다이어그램 레이아웃 생성 → DB에 저장]
       ↓
[설계 다이어그램 복원]
       ↓
[요구사항 명세서 복원]
```

```python
# 역공학 개념 예시
# 다음과 같은 소스 코드만 남아 있을 때, 역공학으로 기능을 파악함

# ✅ 역공학 전: 소스 코드만 있는 상태
def calculate_discount(price, member_grade):
    # 이 함수가 무엇을 하는지 문서가 전혀 없음
    if member_grade == "gold":
        return price * 0.8      # 골드 회원: 20% 할인
    elif member_grade == "silver":
        return price * 0.9     # 실버 회원: 10% 할인
    else:
        return price            # 일반 회원: 할인 없음

# ✅ 역공학 후: 코드를 분석해 요구사항 문서를 역으로 작성
# [역공학으로 도출한 요구사항]
# - 요구사항 1: 회원 등급에 따라 차등 할인율을 적용해야 한다.
# - 요구사항 2: 골드(GOLD) 등급 회원에게는 20% 할인을 제공해야 한다.
# - 요구사항 3: 실버(SILVER) 등급 회원에게는 10% 할인을 제공해야 한다.
# - 요구사항 4: 일반 회원에게는 할인을 제공하지 않는다.
```

:::note 역공학을 막는 기술도 있습니다
독일의 일부 고급 자동차는 분해하면 아예 동작하지 않도록 설계되어 있습니다. 마찬가지로 소프트웨어에서도 역공학을 방지하기 위한 **코드 난독화(obfuscation)** 기술이 있습니다. 코드를 일부러 읽기 어렵게 변환하여 역공학으로 원본 로직을 파악하지 못하도록 막는 것입니다. 상업적으로 중요한 소프트웨어는 이런 방법으로 지적재산권을 보호합니다.
:::

### 재공학(Reengineering) — 구조를 개선해서 다시 만들기

**재공학(Reengineering)**은 역공학과 비슷하게 들리지만 목적이 다릅니다. 재공학은 **기존 시스템의 구조를 개선하여 유지보수성을 높이는 작업**입니다.

아파트를 예로 들면 이해하기 쉽습니다. 오래된 아파트를 완전히 허물고 새로 짓는 것이 **재개발**이라면, 기존 골조(뼈대)는 유지하면서 내부를 대대적으로 리모델링하는 것이 **재건축**입니다. 소프트웨어 재공학은 후자에 해당합니다. 시스템을 완전히 새로 만드는 것이 아니라, **기존 시스템의 구조를 더 좋게 바꾸는 것**입니다.

재공학이 필요한 상황은 주로 **유지보수 비용이 너무 많이 들 때**입니다. 구조가 나쁜 소프트웨어는 작은 기능 하나를 바꿔도 다른 부분에 문제가 생기고, 오류를 수정하면 새로운 오류가 생기는 악순환이 반복됩니다. 이때 구조 자체를 개선하는 재공학이 해결책이 됩니다.

재공학의 목적을 정리하면 다음과 같습니다.

- **복잡도 감소**: 너무 복잡하게 얽혀있는 코드를 단순하게 정리
- **변경 적응성 향상**: 새로운 요구사항이 생겼을 때 쉽게 변경할 수 있는 구조로 개선
- **성능 및 자원 활용 개선**: 더 효율적으로 동작하도록 최적화
- **유지보수 용이성 향상**: 수정, 추가, 삭제가 쉬운 구조로 개선

재공학의 과정은 **PDCA 모델**로 설명할 수 있습니다.

| 단계 | 영어 | 의미 | 설명 |
|---|---|---|---|
| **P** | Plan | 계획 | 어느 부분을 어떻게 개선할지 전략을 수립 |
| **D** | Do | 구현 | 계획에 따라 실제로 구조를 변경하고 개선 |
| **C** | Check | 검증 | 변경이 목적대로 제대로 이루어졌는지 테스트하고 평가 |
| **A** | Action | 적용 | 검증이 완료되면 실제 서비스에 적용 |

```python
# ❌ 재공학 전: 구조가 나쁜 코드 (스파게티 코드 예시)
def handle_order(user, item, qty, pay_method, discount_code, addr):
    # 한 함수 안에 모든 로직이 뒤섞여 있음
    # 주문 유효성 검사, 할인 계산, 결제, 배송 주소 처리가 모두 한 곳에
    if user is None:
        print("사용자 없음")
        return
    if item is None:
        print("상품 없음")
        return
    if qty <= 0:
        print("수량 오류")
        return
    price = item['price'] * qty
    if discount_code == "SAVE10":
        price = price * 0.9
    if pay_method == "card":
        print(f"카드 결제: {price}원")
    elif pay_method == "bank":
        print(f"계좌이체: {price}원")
    if addr is None:
        print("배송 주소 없음")
        return
    print(f"{addr}로 배송 예정")
    # 수백 줄의 코드가 계속 이어짐... 유지보수 매우 어려움

# ✅ 재공학 후: 구조가 개선된 코드 (각 기능을 별도 모듈로 분리)
def validate_order(user, item, qty):
    """주문 유효성 검사 모듈 - 주문이 가능한지 확인"""
    if user is None:
        return False, "사용자 정보가 없습니다."
    if item is None:
        return False, "상품 정보가 없습니다."
    if qty <= 0:
        return False, "수량이 올바르지 않습니다."
    return True, "유효한 주문입니다."

def apply_discount(price, discount_code):
    """할인 적용 모듈 - 할인 코드에 따라 가격 계산"""
    discount_map = {
        "SAVE10": 0.9,   # 10% 할인
        "SAVE20": 0.8,   # 20% 할인
    }
    rate = discount_map.get(discount_code, 1.0)  # 해당 코드 없으면 할인 없음
    return price * rate

def process_payment(price, pay_method):
    """결제 처리 모듈 - 결제 수단에 따라 결제 처리"""
    if pay_method == "card":
        print(f"카드 결제 처리: {price}원")
    elif pay_method == "bank":
        print(f"계좌이체 처리: {price}원")

def schedule_delivery(addr):
    """배송 예약 모듈 - 배송 주소 확인 및 배송 예약"""
    if addr is None:
        print("배송 주소가 없습니다.")
        return
    print(f"{addr}로 배송 예약 완료")

# 메인 주문 처리 함수: 각 모듈을 순서대로 호출 (간결하고 이해하기 쉬움)
def handle_order_v2(user, item, qty, pay_method, discount_code, addr):
    """재공학 후: 역할이 분리된 깔끔한 주문 처리 함수"""
    is_valid, msg = validate_order(user, item, qty)  # 1. 유효성 검사
    if not is_valid:
        print(msg)
        return
    
    price = item['price'] * qty                      # 2. 기본 가격 계산
    final_price = apply_discount(price, discount_code)  # 3. 할인 적용
    process_payment(final_price, pay_method)         # 4. 결제 처리
    schedule_delivery(addr)                          # 5. 배송 예약
```

위 코드처럼 **하나의 거대한 함수**를 **역할별로 분리된 작은 함수들**로 재구성하는 것이 재공학의 핵심입니다. 이렇게 하면 나중에 결제 방식을 바꿀 때 `process_payment` 함수만 수정하면 되고, 할인 정책을 바꿀 때는 `apply_discount` 함수만 수정하면 됩니다.

### 역공학 vs. 재공학 — 무엇이 다른가?

두 개념이 비슷하게 들려서 헷갈릴 수 있습니다. 핵심 차이를 표로 정리해봅니다.

| 구분 | 역공학 (Reverse Engineering) | 재공학 (Reengineering) |
|---|---|---|
| **목적** | 설계 문서나 요구사항이 없을 때, 기존 코드를 분석해서 문서를 복원 | 유지보수 비용이 과다할 때, 시스템 구조 자체를 개선 |
| **방향** | 결과물(코드) → 설계도 → 요구사항 (역방향) | 현재 시스템 → 개선된 시스템 (순방향 개선) |
| **비유** | 완성된 요리를 먹어보고 레시피를 역으로 유추하는 것 | 요리 과정과 도구를 개선해서 더 맛있게 만드는 것 |
| **산출물** | 복원된 설계 다이어그램, 요구사항 명세서 | 구조가 개선된 새 시스템 |

## 🛠️ CASE 도구 — 자동화로 효율을 높이자

### CASE란 무엇인가?

형상관리, 역공학, 재공학을 모두 사람이 수작업으로 하는 것은 사실상 불가능합니다. 대규모 소프트웨어의 경우 수십만 줄의 코드와 수백 개의 문서가 있는데, 이를 일일이 손으로 관리하는 것은 시간적으로도 불가능합니다. 그래서 등장한 것이 **CASE(Computer-Aided Software Engineering, 컴퓨터 지원 소프트웨어 공학)** 도구입니다.

CASE는 **소프트웨어 개발 과정 전체를 자동화하고 통합 관리해주는 도구들의 집합**입니다. 과거에는 소프트웨어 개발이 완전한 수작업, 즉 노동 집약적인 방식이었다면, 이제는 다양한 자동화 도구를 활용하여 개발 생산성을 크게 높일 수 있습니다.

CASE 도구가 해주는 일은 다음과 같습니다.

- **자료 흐름도(DFD), ER 다이어그램** 등 구조도를 자동으로 생성
- **데이터베이스 기본 코드(DDL)** 자동 생성
- 설계 문서를 기반으로 **기본적인 프로그래밍 코드 자동 생성**
- 소스 코드의 **버전 관리와 이력 추적** 자동화

:::info 대표적인 CASE 도구 사례
설계 자동화 도구로는 **Rational Rose** 같은 툴이 있습니다. 이 도구는 UML(통합 모델링 언어)로 설계한 내용을 바탕으로 기본적인 프로그래밍 코드를 자동으로 생성해줍니다. 또한 **GitHub, GitLab** 같은 형상관리 도구는 소스 코드의 버전 관리, 변경 이력 추적, 팀원 간 협업을 자동화해줍니다. "누가 언제 어떤 코드를 왜 바꿨는지"가 모두 기록되어, 문제가 생겼을 때 언제든지 확인할 수 있습니다.
:::

:::tip 개발자로 성장하려면 도구 활용 능력이 필수입니다
소프트웨어 공학의 이론을 잘 아는 것도 중요하지만, 실제 개발 현장에서는 **다양한 도구를 얼마나 잘 활용하느냐**가 개발자의 역량을 크게 좌우합니다. 최신 개발 도구를 적극적으로 도입하고 익히는 것이 여러분의 시장 가치(몸값)를 높이는 가장 빠른 방법 중 하나입니다. Git, JIRA, Confluence, UML 도구 등 실무에서 많이 쓰이는 도구들을 꾸준히 학습하고 활용하는 습관을 기르세요.
:::

## ⚠️ 주의사항과 실무 팁

### 형상관리를 소홀히 하면 생기는 문제들

형상관리를 제대로 하지 않으면 어떤 일이 벌어지는지 구체적인 시나리오로 살펴봅시다.

**시나리오 1: 이직한 개발자의 흔적이 없는 경우**

A 개발자가 2년 동안 홀로 회사의 핵심 시스템을 개발했습니다. 그런데 형상관리 없이 자신만의 방식으로 코드를 관리했습니다. A 개발자가 이직하고 B 개발자가 새로 왔습니다. B는 코드를 봐도 "이 코드가 왜 이렇게 작성되어 있는지", "어떤 요구사항 때문에 이 로직이 들어간 건지" 전혀 알 수가 없습니다. 결국 오류 하나를 수정하는 데 몇 주가 걸리게 됩니다.

**시나리오 2: 무분별한 변경이 부른 연쇄 오류**

오류 다섯 개를 발견한 개발자가 승인 절차 없이 혼자 다 수정해버렸습니다. 그런데 그 수정이 다른 모듈에 영향을 미쳐 열 개의 새로운 오류가 발생했습니다. 어떤 변경이 어떤 오류를 일으켰는지 기록이 없어서, 원인을 찾는 데 더 오랜 시간이 걸렸습니다.

:::danger 절대 하면 안 되는 것
- 변경 승인 없이 임의로 프로덕션(실제 서비스) 코드를 수정하는 것
- 변경 사항을 기록하지 않고 덮어쓰는 것
- 버전 관리 시스템(Git 등) 없이 코드를 관리하는 것
- "내가 다 기억하고 있으니까 문서 안 써도 돼"라는 생각 — 기억은 반드시 휘발됩니다.
:::

### 역공학과 재공학을 적용할 때 주의할 점

**역공학**을 할 때는 코드가 너무 오래되거나 주석이 없으면 분석 시간이 기하급수적으로 늘어납니다. 역공학으로 복원한 문서가 실제 시스템의 동작과 일치하는지 반드시 검증해야 합니다.

**재공학**을 할 때는 구조를 바꾸는 과정에서 기존 기능이 망가지는 **회귀 오류(Regression Error)**가 발생할 수 있습니다. 반드시 자동화된 테스트를 먼저 충분히 갖춰놓고 재공학을 진행해야 합니다.

```python
# ✅ 재공학 진행 시 회귀 테스트(Regression Test) 먼저 작성하기

import unittest  # 파이썬 기본 테스트 프레임워크

class TestDiscountFunction(unittest.TestCase):
    """재공학 전후로 동일하게 실행하여 기능이 유지되는지 확인하는 테스트"""
    
    def test_gold_member_discount(self):
        """골드 회원은 20% 할인이 적용되어야 한다"""
        result = apply_discount(10000, "SAVE20")  # 1만원에 20% 할인 적용
        self.assertEqual(result, 8000)            # 8천원이어야 함

    def test_no_discount(self):
        """할인 코드가 없으면 원래 가격이어야 한다"""
        result = apply_discount(10000, "INVALID") # 잘못된 코드 입력
        self.assertEqual(result, 10000)           # 원가 그대로여야 함

    def test_silver_member_discount(self):
        """실버 회원은 10% 할인이 적용되어야 한다"""
        result = apply_discount(10000, "SAVE10")  # 1만원에 10% 할인 적용
        self.assertEqual(result, 9000)            # 9천원이어야 함

# 테스트 실행
if __name__ == '__main__':
    unittest.main()
```

이처럼 재공학을 시작하기 전에 현재 기능이 어떻게 동작하는지 테스트 코드로 먼저 정의해두면, 재공학 후에도 동일한 테스트를 실행해서 기능이 깨지지 않았음을 확인할 수 있습니다.

## 📋 핵심 정리

이 문서에서 배운 핵심 개념을 정리합니다.

- **소프트웨어는 완성 후에도 끊임없이 변경**된다. 이 변경을 체계적으로 관리하는 것이 형상관리(SCM)이다.
- 변경의 원인은 **업무 환경의 변화(고객 요구, 시장 여건)**와 **기술의 변화(하드웨어, 통신망, 디바이스)** 두 가지이다.
- **버전 관리**는 소프트웨어 변경 이력을 추적하는 기본 수단이며, 풀 모델 체인지(1.0→2.0)와 마이너 체인지(1.0→1.1)로 구분된다.
- **릴리스(Release)**는 요구사항 문서, 설계 문서, 소스 코드를 하나의 패키지로 묶어 관리하는 단위이다.
- 형상관리(SCM)의 4대 핵심 활동은 **형상 식별 → 형상 통제 → 형상 상태 보고 → 형상 감사**이다.
- **베이스라인(Baseline)**은 특정 시점의 산출물 집합으로, 형상관리의 기준점 역할을 한다.
- **형상 통제위원회(CCB)**는 변경 요청을 심사하고 승인/거부하는 의사결정 조직으로, 무분별한 변경을 방지한다.
- **3R**은 재사용(Reuse), 역공학(Reverse Engineering), 재공학(Reengineering)을 의미한다.
- **재사용(Reuse)**은 기존 소프트웨어의 부품(모듈, 라이브러리)을 새 프로젝트에 그대로 가져다 쓰는 것이다.
- **역공학(Reverse Engineering)**은 완성된 코드를 역방향으로 분석하여 설계 문서와 요구사항을 복원하는 작업이다.
- **재공학(Reengineering)**은 기존 시스템의 구조를 개선하여 유지보수성을 높이는 작업으로, PDCA(계획→구현→검증→적용) 모델로 진행된다.
- **CASE 도구**는 형상관리, 설계, 코드 생성 등 소프트웨어 개발 과정을 자동화해주는 통합 도구이며, 개발자에게 필수적인 역량이다.
- 형상관리를 소홀히 하면 **오류 추적 불가, 팀원 교체 시 업무 마비, 무분별한 변경으로 인한 연쇄 오류** 등 심각한 문제가 발생한다.
