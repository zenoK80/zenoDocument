# 🗂️ 소프트웨어 형상관리의 개념과 필요성

소프트웨어는 한 번 만들어지면 그것으로 끝이 아닙니다. 우리가 매일 쓰는 카카오톡, 유튜브, 게임 앱들을 생각해보세요. 매달 업데이트 알림이 뜨고, 새 기능이 추가되고, 버그가 수정되고, 디자인이 바뀝니다. 이처럼 **소프트웨어는 살아있는 생명체처럼 끊임없이 변화**합니다. 그렇다면 이 변화들을 어떻게 체계적으로 관리할 수 있을까요? 그 해답이 바로 **형상관리(Software Configuration Management, SCM)** 입니다.

이번 문서에서는 소프트웨어 형상관리가 무엇인지, 왜 필요한지, 어떻게 동작하는지를 비전공자도 이해할 수 있도록 최대한 쉽고 풍부하게 설명합니다. 또한 형상관리와 밀접하게 연관된 **역공학(Reverse Engineering)**, **재공학(Re-engineering)**, **재사용(Reuse)** 의 개념까지 함께 다룹니다.

## 🌱 소프트웨어 형상관리란 무엇인가?

### 형상(形狀)이란 무엇인가?

"형상관리"라는 단어가 처음에는 생소하게 느껴질 수 있습니다. 여기서 **형상(形狀)** 이란 소프트웨어를 구성하는 모든 산출물(결과물)의 모양, 즉 **현재 상태** 를 의미합니다. 코드 파일, 설계 문서, 요구사항 명세서, 테스트 케이스 등 소프트웨어 개발 과정에서 만들어지는 모든 것들이 형상의 대상입니다.

마치 사람의 얼굴이 나이가 들면서 조금씩 변해가듯, 소프트웨어도 시간이 흐르면서 조금씩, 때로는 크게 변해갑니다. 그 변화의 과정을 **기록하고, 추적하고, 통제하는 활동 전체** 를 우리는 형상관리라고 부릅니다.

### 형상관리의 정의

> **소프트웨어 형상관리(SCM)** 란, 소프트웨어 개발 생명주기(SDLC) 전반에 걸쳐 발생하는 모든 산출물의 변경 사항을 체계적으로 식별하고, 통제하고, 기록하고, 감사하는 방법론이다.

유명한 학자 **에릭 브로드(Eric Burd)** 는 이렇게 말했습니다.

> "프로젝트는 진행되면서 새로운 산출물들이 계속 만들어지고, 그 산출물들은 쌓여가면서 지속적으로 버전업된다."

또 다른 학자 **First of all** 은 다음과 같이 강조했습니다.

> "시스템은 소프트웨어 개발 생명주기의 모든 단계에서 변경이 일어나며, 그 변경하고자 하는 욕구는 개발 생명주기 동안 지속적으로 일어난다."

즉, 요구사항도 바뀌고, 설계도 바뀌고, 코드도 바뀌고, 테스트도 바뀝니다. 이처럼 **변경은 소프트웨어 개발의 숙명** 이며, 이 변경을 체계적으로 다루지 않으면 프로젝트 전체가 혼란에 빠집니다.

:::info 형상관리의 영어 표현
형상관리는 영어로 **SCM(Software Configuration Management)** 이라고 합니다. "Configuration"은 구성, 형상이라는 뜻으로, 소프트웨어를 이루는 모든 구성 요소의 상태를 관리한다는 의미입니다.
:::

## 🔍 왜 형상관리가 필요한가? — 변경의 원인을 이해하자

### 소프트웨어는 왜 계속 변하는가?

소프트웨어가 변하는 이유를 크게 두 가지로 나눌 수 있습니다. 바로 **업무 환경의 변화** 와 **기술의 변화** 입니다. 이 두 가지를 제대로 이해해야 개발자로서 변화에 능동적으로 대응할 수 있습니다.

### 업무 환경의 변화 📱

**첫 번째 원인은 고객이 새로운 기능을 요구하는 것** 입니다. 고객의 니즈는 시간이 지나면서 달라집니다. 예전에는 PC로만 인터넷을 하던 사람들이 이제는 스마트폰, 태블릿으로도 서비스를 이용합니다. 책을 예로 들어봅시다. 과거에는 종이책이 당연했지만, 이제는 전자책, 오디오북, 태블릿 필기형 북 등 다양한 형태로 콘텐츠가 제공됩니다. **출판사가 이 변화를 무시하고 종이책만 고집한다면 시장에서 도태될 것** 입니다.

**두 번째 원인은 시장 여건의 변경** 입니다. 이를 가장 잘 보여주는 사례가 바로 **코닥(Kodak)** 입니다. 코닥은 필름 카메라 시장의 절대 강자였습니다. 놀랍게도 디지털 카메라를 최초로 개발한 회사 역시 코닥이었습니다. 그러나 "디지털로 전환되기까지는 오랜 시간이 걸릴 것"이라고 판단하여 빠르게 대응하지 않았고, 결국 시장에서 사라지고 말았습니다. **시장의 변화를 읽지 못한 판단이 기업을 무너뜨린 것** 입니다.

**세 번째 원인은 예산과 일정의 변경** 입니다. 프로젝트를 진행하다 보면 처음 계획한 예산과 일정이 달라지는 경우가 많습니다. 이에 따라 개발 범위나 기능 목록도 조정되어야 하므로, 소프트웨어의 내용 역시 변경됩니다.

### 기술의 변화 💻

**하드웨어 사양의 발전** 도 소프트웨어 변경을 이끄는 중요한 요인입니다. 인터넷 속도가 5G를 넘어 6G를 향해 가고 있고, 스마트폰의 해상도와 멀티미디어 성능은 눈부시게 향상됐습니다. 이러한 기술 발전 덕분에 유튜브처럼 고해상도 동영상을 스마트폰으로 끊김 없이 볼 수 있게 되었습니다. 소프트웨어는 이런 새로운 하드웨어와 통신 환경에 맞춰 지속적으로 업데이트되어야 합니다.

:::warning 개발자도 시장을 알아야 합니다
"나는 자바(Java)랑 데이터베이스만 잘하면 되지"라는 생각은 위험합니다. 소프트웨어는 항상 다양한 요구사항에 의해 변경되기 때문에, **개발자도 고객을 이해하고 시장의 흐름을 파악하는 인문학적 소양이 필요합니다.** 기술만 잘 아는 개발자가 만든 소프트웨어는 시장에서 외면받을 수 있습니다.
:::

## 📦 버전 관리와 릴리스 — 형상관리의 기본 뼈대

### 버전(Version)이란?

형상관리의 가장 기본적인 도구는 **버전(Version)** 입니다. 소프트웨어는 버전이라는 숫자를 통해 자신의 상태를 표현합니다. 윈도우(Windows)를 예로 들면 Windows 95, 98, 2000, XP, 7, 10, 11로 이어지는 버전의 역사가 곧 윈도우의 변화 역사입니다.

버전 관리 개념은 자동차의 **모델 체인지** 와 비교하면 매우 이해하기 쉽습니다.

| 개념 | 자동차 비유 | 소프트웨어 예시 |
|---|---|---|
| **풀 모델 체인지** | 엔진, 디자인, 내부 인테리어 전부 교체 | 버전 1.0 → 2.0 (앞자리 변경) |
| **마이너 체인지** | 외형 디자인이나 내부 인테리어 일부 변경 | 버전 1.0 → 1.1 → 1.1.1 (뒷자리 변경) |

**풀 모델 체인지(Full Model Change)** 는 소프트웨어의 앞 버전 번호가 바뀌는 경우입니다. 예를 들어 `1.0`에서 `2.0`으로 바뀌면, 화면 구성이 완전히 달라지고, 기능 구조도 전면 개편되며, 사용자가 기존 메뉴를 찾기 어려울 정도로 바뀐 것입니다. 홈페이지를 오랜만에 방문했더니 레이아웃이 완전히 새로워진 경우가 이에 해당합니다.

**마이너 체인지(Minor Change)** 는 뒷자리 번호가 바뀌는 경우입니다. `1.0`에서 `1.1`, `1.1`에서 `1.1.1`로 바뀌는 것처럼, 전체 구조는 유지하되 일부 기능이나 디자인이 수정된 경우입니다.

### 왜 버전을 관리해야 하는가?

버전 1.0과 버전 2.0 사이에 어떤 차이가 있는지 정확하게 파악하고 있어야 합니다. 어떤 기능이 추가됐고, 어떤 기능이 제거됐고, 어떤 구조가 개선됐는지를 알아야 **유지보수(maintenance)** 를 효율적으로 할 수 있습니다. 이것을 관리하지 않으면, 새로운 개발자가 팀에 합류했을 때 "이 코드는 왜 이렇게 짜여 있나요?"라는 질문에 아무도 답하지 못하는 상황이 벌어집니다.

### 릴리스(Release)란?

**릴리스(Release)** 는 요구사항 명세서, 설계 문서, 원시 코드(소스 코드) 등 서로 다른 문서들의 특정 버전을 하나의 묶음으로 패키징하는 개념입니다. 마치 쇼핑몰에서 여러 상품을 하나의 상자에 담아 세트 상품으로 판매하는 것처럼, 특정 시점의 산출물들을 묶어서 하나의 릴리스로 관리합니다.

아래 표는 릴리스 구성 예시입니다.

| 릴리스 | 요구사항 명세서 버전 | 설계 문서 버전 | 원시 코드 버전 |
|---|---|---|---|
| **릴리스 1.0** | 요구사항 v1.0 | 설계 v1.0 | 코드 v1.0 |
| **릴리스 2.0** | 요구사항 v1.1 | 설계 v1.2 | 코드 v1.1 |
| **릴리스 3.0** | 요구사항 v1.3 | 설계 v1.4 | 코드 v1.5 |

이처럼 릴리스 3.0은 요구사항 v1.3, 설계 v1.4, 코드 v1.5 세 가지를 묶어서 하나의 완성된 패키지로 관리합니다. 이렇게 관리하면 "릴리스 2.0 기준으로 어떤 기능이 있었나?"라는 질문에 즉시 답할 수 있습니다.

:::tip 릴리스 관리가 중요한 이유
개발자는 한 회사에 영원히 있을 수 없습니다. 이직하거나, 새로운 개발자가 합류하거나, 팀원이 바뀌는 일은 반드시 발생합니다. 릴리스와 버전이 체계적으로 관리되어 있으면, 새로운 팀원도 히스토리(history, 변경 이력)를 보고 빠르게 프로젝트를 파악할 수 있습니다.
:::

## ⚙️ 형상관리의 핵심 원리 — SCM의 4가지 활동

형상관리는 단순히 파일을 저장하는 것이 아닙니다. **식별 → 통제 → 보고 → 감사** 라는 4단계 활동으로 이루어진 체계적인 방법론입니다.

### 1단계: 형상 식별(Configuration Identification) 🔎

**형상 식별** 은 형상관리의 가장 기본이 되는 활동입니다. "무엇을 관리할 것인가?"를 정하는 단계입니다.

형상 식별을 할 때는 먼저 **형상 항목(Configuration Item)** 을 선정합니다. 형상 항목이란 관리 대상이 되는 산출물을 말합니다. 요구사항 명세서, 설계 문서, 소스 코드, 테스트 케이스, 사용자 매뉴얼 등이 모두 형상 항목이 될 수 있습니다.

그런 다음 **베이스라인(Baseline, 기준선)** 을 설정합니다. 베이스라인이란 특정 시점에서의 산출물 집합으로, 변경 관리의 기준점 역할을 합니다. 예를 들어 쇼핑몰 서비스의 경우, "현재 버전에서는 결제 기능은 이렇게 구현됐고, 주문은 이 방식으로 처리되며, 할인율은 10%가 적용된다"라고 명확히 정해놓은 상태가 바로 베이스라인입니다.

베이스라인이 명확해야 나중에 변경이 일어났을 때 "기준에서 무엇이 달라졌는가?"를 비교할 수 있습니다.

```markdown
# 베이스라인 예시 (요구사항 명세서 v1.0 기준)

## 결제 기능
- 결제 수단: 신용카드, 계좌이체 지원
- 할인율: 10% 고정 적용
- 환불 정책: 7일 이내 전액 환불

## 주문 기능
- 주문 처리 방식: 선결제 후 배송
- 주문 취소: 배송 전까지 가능
```

### 2단계: 형상 통제(Configuration Control) 🛂

**형상 통제** 는 형상관리에서 가장 핵심적인 활동입니다. 아무나 마음대로 코드나 문서를 변경해서는 안 된다는 원칙을 실행하는 단계입니다.

마치 국가의 법률을 바꾸려면 국회의 심의와 의결을 거쳐야 하듯, 소프트웨어의 변경도 **형상통제위원회(CCB, Configuration Control Board)** 라는 의사결정 기구의 승인을 받아야 합니다.

형상 통제의 흐름은 다음과 같습니다.

```
[변경 요청]
고객, 개발팀, 운영팀, CS팀 등에서 변경 요청 제출
       ↓
[CCB 심사]
형상통제위원회에서 필요성, 비용, 일정, 영향도 검토
       ↓
[승인 또는 거부]
승인 → 개발자가 변경 작업 수행
거부 → 변경 요청 반려
       ↓
[변경 구현]
승인된 변경 사항을 개발자가 구현
       ↓
[테스트 및 검증]
변경된 내용이 제대로 동작하는지 확인
```

**형상통제위원회(CCB)** 는 보통 다음과 같은 구성원으로 이루어집니다.

- **PM(Project Manager, 프로젝트 관리자)**: 전체 프로젝트를 총괄하는 책임자
- **형상관리 담당자**: 실제 변경 작업을 수행하는 개발자
- **품질 담당자**: 변경 후 품질을 검증하는 사람
- **기술 담당자**: 기술적 타당성을 검토하는 전문가
- **고객 측 담당자**: 변경 요구사항의 당사자

CCB에서는 다음과 같은 사항들을 종합적으로 판단합니다.

- 이 변경이 왜 필요한가?
- 계약 기간과 일정에 맞는가?
- 추가 비용은 얼마나 발생하는가?
- 유지보수에 어떤 영향을 미치는가?
- 변경 결과물의 품질은 적절한가?

:::danger 무분별한 코드 변경은 절대 금지
오류가 발생했다고 해서 개발자가 단독으로 코드를 수정하고 아무도 모르게 덮어버리는 행위는 절대 해서는 안 됩니다. 다섯 가지 오류를 마음대로 고쳤더니 열 가지 새로운 문제가 생겨나는 상황이 발생할 수 있습니다. **모든 변경은 반드시 승인 절차를 거쳐야 합니다.**
:::

### 3단계: 형상 상태 보고(Configuration Status Accounting) 📋

**형상 상태 보고** 는 변경이 일어날 때마다 그 내용을 기록하고, 관련된 모든 사람들에게 공유하는 활동입니다. 누가, 언제, 무엇을, 왜 바꿨는지에 대한 기록을 남겨야 합니다.

이 기록이 없으면 다음과 같은 상황이 발생합니다.

> "1년 전에 이 기능을 왜 이렇게 만들었나요?"
> "글쎄요... 제가 그때 없어서 잘 모르겠는데요."

이런 상황은 매우 위험합니다. 형상관리 도구(Git, SVN 등)를 사용하면 변경 이력(history)이 자동으로 기록되므로, 언제든지 과거의 상태로 돌아가거나 변경 내용을 추적할 수 있습니다.

:::note 형상 상태 보고가 어려운 현실
규모가 작은 회사에서는 문서 작업과 보고 절차가 부담스러워 이를 생략하는 경우가 많습니다. 그러나 이를 생략하면 변경 사항이 쌓여가면서 나중에는 오류의 원인을 찾는 것 자체가 매우 힘들어집니다. 작은 수고가 나중의 큰 위기를 예방합니다.
:::

### 4단계: 형상 감사(Configuration Audit) 🔬

**형상 감사** 는 "CCB에서 승인한 대로 실제로 변경이 이루어졌는가?"를 확인하는 활동입니다. 마치 건물을 지을 때 설계 도면대로 시공이 되었는지 감리(監理)를 하는 것과 같습니다.

형상 감사에서는 다음 사항들을 점검합니다.

- 승인된 요구사항이 모두 반영되었는가?
- 베이스라인 기준에 적합한가?
- 일정을 준수했는가?
- 변경으로 인한 문제는 없는가?

감사 결과는 **감사 보고서** 로 작성되며, "계획한 방향대로 문제없이 구현되었다"는 확인이 이루어지면 해당 변경이 공식적으로 승인됩니다.

## 💡 형상관리의 목적 — 왜 이 모든 과정이 필요한가?

형상관리를 한다는 것은 분명히 추가적인 시간과 노력이 필요합니다. 그럼에도 불구하고 왜 해야 할까요? 크게 세 가지 목적이 있습니다.

### 목적 1: 가시성(Visibility) 확보 👁️

**가시성** 이란 현재 소프트웨어의 상태가 눈에 보이도록 만드는 것입니다. 형상관리가 잘 되어 있으면 다음 질문에 즉시 답할 수 있습니다.

- 현재 버전에는 어떤 기능이 포함되어 있는가?
- 어떤 오류가 수정되었고 어떤 오류가 아직 남아있는가?
- 어떤 기능이 추가되었고 어떤 기능이 제거되었는가?

반대로 형상관리가 안 되어 있으면, 소프트웨어의 현재 상태가 "블랙박스"처럼 되어버려 아무도 정확한 상태를 파악하지 못합니다.

### 목적 2: 변경 추적성(Traceability) 확보 🔗

**추적성** 이란 "누가, 언제, 무엇을, 왜 바꿨는가?"에 대해 언제든 답할 수 있는 능력입니다. 형상관리 도구에는 변경 이력이 모두 기록되어 있기 때문에, 새로운 개발자가 오더라도 히스토리를 보고 "아, 그때 고객이 이런 요구사항을 제시해서 이 기능을 추가했구나"라는 맥락을 파악할 수 있습니다.

### 목적 3: 생산성과 안정성 향상 🚀

체계적인 형상관리는 **무분별한 변경을 사전에 차단** 하고, **변경으로 인한 부작용을 최소화** 합니다. 이를 통해 소프트웨어의 품질이 높아지고, 유지보수가 쉬워지며, 개발 생산성도 향상됩니다. 결국 형상관리는 개발 비용을 줄이고 소프트웨어의 수명을 늘리는 가장 효과적인 방법입니다.

## 🔄 3R: 역공학, 재공학, 재사용

형상관리를 통해 소프트웨어의 변경을 관리한다면, **3R** 은 기존 소프트웨어를 활용하여 새로운 소프트웨어를 만들거나 개선하는 방법론입니다. 3R은 다음 세 가지를 의미합니다.

| 약자 | 영어 | 한국어 | 핵심 개념 |
|---|---|---|---|
| **R1** | Reverse Engineering | 역공학 | 완성품을 거꾸로 분석하여 설계도를 복원 |
| **R2** | Re-engineering | 재공학 | 기존 시스템의 구조를 개선하여 재구축 |
| **R3** | Reuse | 재사용 | 기존에 만들어진 부품을 새 프로젝트에 재활용 |

### 3R이 필요한 이유

3R이 필요한 배경을 이해하려면 현대 소프트웨어 개발의 현실을 알아야 합니다.

- 소프트웨어의 규모가 점점 커지고 복잡해지고 있습니다.
- 반면 소프트웨어의 **생명 주기(Life Cycle)** 는 점점 짧아지고 있습니다.
- 개발 비용은 높아지는데 유지보수는 더욱 어려워지고 있습니다.

이런 상황에서 처음부터 모든 것을 새로 만드는 것은 매우 비효율적입니다. 3R을 활용하면 **유지보수 비용 절감**, **개발 생산성 향상**, **소프트웨어 이해 및 변경 용이성**, **다양한 변경 요구에 빠른 대응** 이 가능합니다.

### R3: 재사용(Reuse) — 기존 부품을 다시 쓰자 ♻️

**재사용(Reuse)** 이란 기존에 개발된 소프트웨어의 부품(컴포넌트)을 새로운 프로젝트에 그대로 가져다 쓰는 것입니다. 마치 레고 블록처럼, 이미 만들어진 블록을 조합하여 새로운 구조물을 만드는 것과 같습니다.

홈페이지 개발을 예로 들어봅시다. 기존 홈페이지에 이미 잘 만들어진 결제 모듈, 게시판 기능, 자료실 기능이 있다면, 새로운 홈페이지를 만들 때 이 기능들을 처음부터 다시 개발할 필요가 없습니다. **기존 것을 가져다 쓰고, 없는 것만 새로 개발하면 개발 속도와 효율성이 크게 향상됩니다.**

재사용의 단위는 다양합니다.

- **서브루틴(Subroutine) / 라이브러리(Library)**: 특정 기능을 수행하는 함수 또는 함수 묶음
- **코드 블록(Code Block)**: 특정 기능을 수행하는 코드 조각
- **모듈(Module)**: 독립적으로 동작하는 소프트웨어 구성 단위

```python
# ✅ 재사용의 좋은 예: 기존에 만들어진 결제 모듈을 새 프로젝트에 가져다 사용
# 기존 프로젝트에서 이미 검증된 결제 처리 함수를 임포트하여 재사용
from legacy_payment_module import process_payment, refund_payment

def handle_new_order(order_id, amount):
    # 기존 결제 모듈을 재사용 (새로 개발하지 않음)
    result = process_payment(order_id, amount)
    if result.success:
        print(f"주문 {order_id} 결제 완료: {amount}원")
    return result
```

```python
# ❌ 재사용하지 않는 나쁜 예: 이미 존재하는 기능을 처음부터 다시 개발
def handle_new_order_bad(order_id, amount):
    # 기존에 검증된 결제 모듈이 있음에도 처음부터 다시 구현
    # → 개발 시간 낭비, 새로운 버그 발생 위험 증가
    import requests
    response = requests.post("https://payment.api", json={
        "order": order_id,
        "amount": amount
    })
    # ... 기존 코드와 중복되는 로직 반복 구현 ...
    return response
```

재사용 여부를 결정할 때는 다음 기준을 고려합니다.

- 모듈의 크기와 복잡도가 적절한가?
- 정규화(표준화)가 잘 되어 있는가?
- 라이브러리가 잘 정리되어 있는가?
- 객체지향 언어로 작성되어 재사용이 용이한가?

### R1: 역공학(Reverse Engineering) — 거꾸로 분석하기 🔙

**역공학(Reverse Engineering)** 은 완성된 결과물을 거꾸로 분석하여 설계도나 요구사항 명세서를 복원하는 과정입니다. 자동차로 비유하자면, 설계 도면 없이 완성된 자동차를 분해하여 어떻게 만들어졌는지 파악하는 것과 같습니다.

소프트웨어 현장에서 역공학이 필요한 상황은 매우 흔합니다. 오래된 시스템을 담당하게 됐는데 요구사항 정의서도 없고, 설계 문서도 없고, 기능 정의서도 없고, 코드 주석도 거의 없는 경우입니다. 이럴 때는 **기존 소스 코드를 거꾸로 분석하여 문서를 만들어 나가야 합니다.** 이것이 바로 역공학입니다.

역공학의 방향은 일반 개발과 정반대입니다.

```
[일반 개발 방향] ▶▶▶
요구사항 → 설계 → 코드 → 테스트 → 완성품

[역공학 방향] ◀◀◀
완성품 → 코드 분석 → 설계 복원 → 요구사항 추출
```

역공학의 기술적 절차는 다음과 같습니다.

```
1. 원시 코드(소스 코드) 결과물 추출
       ↓
2. 추출된 정보를 데이터베이스(DB)에 저장
       ↓
3. 저장된 데이터를 기반으로 다이어그램 레이아웃 생성
       ↓
4. 생성된 다이어그램을 다시 DB에 저장
       ↓
5. 설계 다이어그램 및 요구사항 문서 복원 완료
```

:::note 역공학 방지 기술
일부 고급 소프트웨어와 하드웨어 제품은 역공학을 의도적으로 막습니다. 예를 들어 일부 독일 자동차 브랜드의 ECU(전자제어장치)는 분해 시 자동으로 시스템이 다운되도록 설계되어 있어, 역공학을 통한 기술 유출을 원천 차단합니다.
:::

### R2: 재공학(Re-engineering) — 구조를 개선하자 🏗️

**재공학(Re-engineering)** 은 기존 시스템의 전체 틀은 유지하면서 내부 구조를 개선하는 과정입니다. 건물의 재건축(전체 철거 후 신축)이 아닌 **리모델링(구조를 유지하면서 내부 개선)** 에 가깝습니다.

재공학이 필요한 상황은 주로 **유지보수 비용이 과도하게 많이 드는 경우** 입니다. 유지보수 비용이 많이 든다는 것은 소프트웨어의 내부 구조가 복잡하고 나쁘다는 신호입니다. 이런 경우 구조 자체를 개선하여 유지보수를 쉽게 만드는 것이 재공학의 목표입니다.

재공학의 목표는 다음과 같습니다.

- **복잡도 감소**: 얽혀있는 코드 구조를 단순하게 정리
- **변경 적응성 향상**: 새로운 요구사항이 생겼을 때 쉽게 변경할 수 있도록 구조 개선
- **성능 효율 향상**: 같은 기능을 더 빠르고 적은 자원으로 처리
- **유지보수 용이성**: 오류 수정과 기능 추가가 쉬운 구조로 전환

```python
# ❌ 재공학 전: 유지보수가 어려운 나쁜 구조
# 모든 로직이 하나의 함수에 뒤엉켜 있어 수정하기 매우 어려움
def process_order(user_id, product_id, quantity, coupon_code):
    # 사용자 검증 로직
    user = db.query(f"SELECT * FROM users WHERE id = {user_id}")
    if not user:
        return "사용자 없음"
    # 상품 검증 로직
    product = db.query(f"SELECT * FROM products WHERE id = {product_id}")
    if not product or product['stock'] < quantity:
        return "재고 부족"
    # 할인 계산 로직
    price = product['price'] * quantity
    if coupon_code == "SALE10":
        price = price * 0.9
    # 결제 처리 로직
    payment_result = payment_api.charge(user['card_number'], price)
    # 재고 차감 로직
    db.query(f"UPDATE products SET stock = stock - {quantity} WHERE id = {product_id}")
    # 주문 저장 로직
    db.query(f"INSERT INTO orders VALUES ({user_id}, {product_id}, {quantity}, {price})")
    return "주문 완료"
```

```python
# ✅ 재공학 후: 각 책임을 분리하여 유지보수가 쉬운 구조로 개선
# 각 기능을 독립적인 함수/모듈로 분리 (단일 책임 원칙 적용)

def validate_user(user_id):
    """사용자 검증만 담당하는 함수"""
    user = UserRepository.find_by_id(user_id)
    if not user:
        raise ValueError("존재하지 않는 사용자입니다.")
    return user

def validate_stock(product_id, quantity):
    """재고 검증만 담당하는 함수"""
    product = ProductRepository.find_by_id(product_id)
    if not product or product.stock < quantity:
        raise ValueError("재고가 부족합니다.")
    return product

def calculate_price(product, quantity, coupon_code):
    """가격 계산만 담당하는 함수"""
    price = product.price * quantity
    return DiscountService.apply(price, coupon_code)  # 할인 로직 분리

def process_order(user_id, product_id, quantity, coupon_code):
    """주문 처리 흐름만 담당 (각 세부 로직은 분리된 함수에 위임)"""
    user = validate_user(user_id)               # 사용자 검증
    product = validate_stock(product_id, quantity)  # 재고 검증
    price = calculate_price(product, quantity, coupon_code)  # 가격 계산
    PaymentService.charge(user.card_number, price)  # 결제 처리
    OrderRepository.save(user_id, product_id, quantity, price)  # 주문 저장
    return "주문 완료"
```

재공학은 **PDCA 모델** 로 진행됩니다.

| 단계 | 약자 | 내용 |
|---|---|---|
| **Plan(계획)** | P | 어떤 부분을 개선할지 결정하고 전략 수립 |
| **Do(실행)** | D | 계획한 내용을 실제로 구현 |
| **Check(검증)** | C | 구현된 내용이 목표에 맞는지 검증 |
| **Action(적용)** | A | 검증이 완료된 내용을 실제 서비스에 적용 |

역공학과 재공학의 차이를 명확히 정리하면 다음과 같습니다.

| 구분 | 역공학 (Reverse Engineering) | 재공학 (Re-engineering) |
|---|---|---|
| **목적** | 없어진 문서(설계, 요구사항) 복원 | 기존 시스템의 구조 개선 |
| **방향** | 완성품 → 설계 → 요구사항 (역방향) | 현재 시스템 → 개선된 시스템 (순방향) |
| **결과물** | 설계 다이어그램, 요구사항 명세서 | 구조가 개선된 새 시스템 |
| **비유** | 완성된 케이크를 보고 레시피 추측 | 케이크 레시피를 더 맛있게 개선 |

## 🛠️ CASE 도구 — 형상관리를 자동화하자

### CASE란 무엇인가?

형상관리, 역공학, 재공학을 모두 수작업으로 진행하는 것은 현실적으로 불가능합니다. 이를 위해 다양한 자동화 도구들이 존재하는데, 이 도구들을 통칭하여 **CASE(Computer Aided Software Engineering, 컴퓨터 지원 소프트웨어 공학)** 도구라고 합니다.

과거에는 개발이 노동 집약적인 수작업 중심이었습니다. 하지만 최근에는 CASE 도구들이 개발 과정의 많은 부분을 자동화하고 있습니다. 형상관리 도구로는 **Git**, **SVN** 이 대표적이며, 설계 자동화 도구로는 **Rational Rose** 같은 도구들이 있습니다.

### CASE 도구의 주요 기능

- **자료 흐름도(DFD), ER 다이어그램** 등 구조도 자동 생성
- **데이터베이스 기본 코드 자동 생성**
- **설계 문서를 기반으로 기본 프로그래밍 코드 생성**
- **변경 이력(히스토리) 자동 기록 및 관리**
- **버전 간 차이점(diff) 자동 비교**

```bash
# ✅ Git을 활용한 형상관리 기본 명령어 예시

# 1. 변경 사항을 스테이징 영역에 추가 (변경된 파일을 추적 대상으로 등록)
git add payment_module.py

# 2. 변경 사항을 커밋 (변경 내용과 이유를 메시지로 기록)
git commit -m "feat: 결제 모듈에 카카오페이 결제 수단 추가 (CCB 승인 #234)"

# 3. 특정 파일의 변경 이력 확인 (역공학적 히스토리 조회)
git log --follow payment_module.py

# 4. 두 버전 간의 차이점 확인 (버전 비교)
git diff v1.0 v2.0 -- payment_module.py

# 5. 특정 버전으로 되돌리기 (이전 릴리스로 복구)
git checkout v1.5 -- payment_module.py
```

```bash
# ❌ 형상관리 도구 없이 파일명으로만 버전 관리하는 나쁜 예
# 이렇게 하면 변경 내용, 변경 이유, 변경자를 알 수 없음
payment_module_최종.py
payment_module_최종2.py
payment_module_진짜최종.py
payment_module_진짜진짜최종.py
payment_module_최종_수정본.py
```

:::tip 개발 도구 활용 능력도 핵심 역량입니다
소프트웨어 공학의 이론만 아는 것으로는 부족합니다. Git과 같은 형상관리 도구, 설계 자동화 도구 등 다양한 개발 도구를 실제로 잘 활용할 수 있어야 합니다. **최신 개발 도구를 업무에 적극적으로 도입하고 활용하는 능력이 개발자의 몸값을 결정합니다.**
:::

## ⚠️ 형상관리 주의사항과 실전 팁

### 자주 하는 실수들

**실수 1: 커밋 메시지를 대충 작성하기**

커밋 메시지(변경 내용 설명)를 "수정", "업데이트", "고침"처럼 모호하게 작성하면, 나중에 이력을 봤을 때 무엇이 왜 변경됐는지 전혀 알 수 없습니다.

```bash
# ❌ 나쁜 커밋 메시지 - 무엇을 왜 바꿨는지 전혀 알 수 없음
git commit -m "수정"
git commit -m "버그 고침"
git commit -m "업데이트"

# ✅ 좋은 커밋 메시지 - 무엇을(what), 왜(why) 명확히 기술
git commit -m "fix: 결제 시 할인율 중복 적용 버그 수정 (이슈 #145)"
git commit -m "feat: 모바일 결제를 위한 QR코드 생성 기능 추가"
git commit -m "refactor: 주문 처리 로직 단일 책임 원칙에 맞게 분리 (재공학)"
```

**실수 2: 베이스라인 없이 변경하기**

베이스라인(기준선)을 명확히 설정하지 않고 변경을 진행하면, 무엇이 기준이고 무엇이 변경된 것인지 구분할 수 없게 됩니다. 항상 변경 전 베이스라인을 태깅(tagging)하여 관리하세요.

```bash
# ✅ 변경 작업 시작 전 현재 상태를 베이스라인으로 태깅
git tag -a v1.0-baseline -m "릴리스 1.0 베이스라인 - CCB 승인 2026-01-15"
git tag -a v2.0-baseline -m "릴리스 2.0 베이스라인 - CCB 승인 2026-03-01"
```

**실수 3: 형상관리 없이 협업하기**

여러 개발자가 같은 파일을 동시에 수정하면 충돌이 발생합니다. 형상관리 도구 없이 이메일이나 메신저로 파일을 주고받으면 어떤 버전이 최신인지 알 수 없게 됩니다.

:::warning 형상관리 도구 없는 팀 협업은 재앙의 시작
"내 컴퓨터에서는 잘 되는데?"라는 말이 나온다면, 형상관리가 제대로 되지 않고 있다는 신호입니다. 팀원 모두가 같은 형상관리 도구를 사용하고, 동일한 규칙을 따르는 것이 중요합니다.
:::

### 형상관리를 잘 하기 위한 팁

**팁 1: 작은 단위로 자주 커밋하세요**

한 번에 많은 변경을 커밋하면 무엇이 변경됐는지 추적하기 어렵습니다. 하나의 기능이나 하나의 버그 수정 단위로 작게 나누어 자주 커밋하는 습관을 들이세요.

**팁 2: 브랜치(Branch)를 적극 활용하세요**

브랜치는 나뭇가지처럼 메인 코드에서 분기하여 독립적으로 작업할 수 있는 공간입니다. 새 기능 개발이나 버그 수정을 별도 브랜치에서 진행하면, 메인 코드의 안정성을 유지하면서 안전하게 변경 작업을 할 수 있습니다.

**팁 3: CCB 역할을 팀 내에서 명확히 정하세요**

팀 규모가 작더라도, 변경 사항에 대한 검토와 승인 절차를 간소화된 형태로라도 유지하세요. 예를 들어 "2명 이상의 코드 리뷰 승인 후 메인 브랜치에 병합"과 같은 규칙을 정하는 것만으로도 무분별한 변경을 막을 수 있습니다.

## 📝 핵심 정리

- **소프트웨어 형상관리(SCM)** 는 개발 생명주기 전반에 걸쳐 발생하는 모든 산출물의 변경을 체계적으로 관리하는 방법론이다.
- **소프트웨어 변경의 원인** 은 크게 업무 환경의 변화(고객 요구, 시장 변화, 예산·일정 변경)와 기술의 변화(하드웨어 발전, 통신 환경 변화)로 나뉜다.
- **버전 관리** 는 풀 모델 체인지(앞 자리 번호 변경)와 마이너 체인지(뒷 자리 번호 변경)로 구분되며, 변경의 범위를 표현한다.
- **릴리스(Release)** 는 특정 시점의 요구사항 명세서, 설계 문서, 원시 코드를 하나의 패키지로 묶어 관리하는 단위이다.
- **형상관리의 4대 활동** 은 형상 식별 → 형상 통제 → 형상 상태 보고 → 형상 감사 순서로 이루어진다.
- **베이스라인(Baseline)** 은 특정 시점의 산출물 집합으로, 변경 관리의 기준점 역할을 한다.
- **형상통제위원회(CCB)** 는 변경 요청을 심사하여 승인 또는 거부하는 의사결정 기구로, PM·품질 담당자·고객 측 담당자 등으로 구성된다.
- **형상관리의 목적** 은 가시성 확보, 변경 추적성 확보, 생산성과 안정성 향상이다.
- **3R** 은 역공학(Reverse Engineering), 재공학(Re-engineering), 재사용(Reuse)의 세 가지 소프트웨어 재활용 방법론을 총칭한다.
- **역공학(Reverse Engineering)** 은 완성된 소프트웨어를 거꾸로 분석하여 설계 문서나 요구사항 명세서를 복원하는 방법이다.
- **재공학(Re-engineering)** 은 기존 시스템의 전체 틀은 유지하면서 내부 구조를 개선하여 유지보수 비용을 줄이는 방법이며, PDCA 모델로 진행된다.
- **재사용(Reuse)** 은 기존에 개발된 소프트웨어 부품(서브루틴, 라이브러리, 모듈 등)을 새로운 프로젝트에 활용하여 개발 효율을 높이는 방법이다.
- **CASE(Computer Aided Software Engineering)** 도구는 소프트웨어 개발 과정을 자동화하는 통합 도구로, Git 같은 형상관리 도구와 설계 자동화 도구 등이 포함된다.
- **개발자도 시장과 고객을 이해하는 인문학적 소양** 이 필요하며, 기술 역량과 함께 다양한 개발 도구 활용 능력이 현대 개발자의 핵심 역량이다.
