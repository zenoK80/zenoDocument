# 코드 품질 향상 기법

## 코드 인스펙션 (Code Inspection)

### 코드 인스펙션이란?

**인스펙션** = 코드를 검사하여 결함을 찾는 작업

```
프로그램이 정상 컴파일된다고 해서 완벽한가?

NO! 컴파일은 문법만 확인
실제 논리 오류, 성능 문제, 설계 결함은 놓칠 수 있음

따라서 별도의 검사가 필요
```

### 인스펙션의 목적

```
✅ 코드의 결함 발견
✅ 성능 이슈 확인
✅ 코딩 표준 준수 여부 확인
✅ 로직 오류 탐지
✅ 인터페이스 문제 확인
✅ 데이터 처리 오류 탐지
✅ 타이밍 문제 확인
```

### 인스펙션의 순서

```
1단계: 컴파일 성공
└─ 문법 오류가 없음

2단계: 정적 분석 도구 사용
└─ 코드를 분석해 문제 찾기

3단계: 코드 리뷰
└─ 개발자가 직접 검토

4단계: 테스트 수행
└─ 실제 작동 확인

5단계: 품질 평가
└─ 최종 검증
```

## 정적 분석 (Static Analysis)

### 정적 분석이란?

**정적 분석** = 프로그램을 실행하지 않고 코드 자체를 분석하여 결함을 찾는 것

```
동적 분석:
└─ 프로그램을 실행해서 확인
└─ 특정 경로만 테스트 가능

정적 분석:
└─ 코드 자체를 분석
└─ 모든 가능한 결함 찾기
└─ 실행 전에 문제를 발견
```

### 정적 분석으로 찾을 수 있는 것

```
✅ 변수 초기화 전 사용
✅ 선언하지 않은 변수 사용
✅ 도달 불가능한 코드
✅ 메모리 누수 가능성
✅ 배열 범위 초과
✅ Null 포인터 접근
✅ 타입 불일치
```

### 정적 분석 도구

```
도구 예시:
├─ SonarQube: 코드 품질 분석
├─ Checkstyle: 코딩 스타일 검사
├─ PMD: 성능 및 버그 감지
├─ FindBugs: 잠재적 버그 찾기
└─ Lint: 가벼운 문법 검사
```

## 페어 프로그래밍 (Pair Programming)

### 페어 프로그래밍이란?

**페어 프로그래밍** = 두 명의 프로그래머가 함께 코드를 작성하는 방식

```
혼자 개발:
└─ 내가 모든 것을 책임
└─ 오류 발견 어려움
└─ 의견 충돌 불가능
└─ 학습 기회 제한

페어 프로그래밍:
├─ 서로 보완
├─ 오류를 즉시 발견
├─ 다양한 관점 확보
├─ 서로 배우는 기회
```

### 페어 프로그래밍의 역할

```
드라이버 (Driver):
├─ 키보드를 잡고 직접 코딩
├─ 세부 사항에 집중
└─ "어떻게"를 담당

네비게이터 (Navigator):
├─ 코드를 검토하며 관찰
├─ 전체 흐름에 집중
└─ "무엇을"을 담당
```

### 핑퐁 프로그래밍

**핑퐁 프로그래밍** = 개발과 테스트 역할을 계속 바꾸며 진행

```
1라운드:
├─ A가 기능 개발
└─ B가 테스트

2라운드:
├─ B가 기능 개발
└─ A가 테스트

3라운드:
└─ 반복...

비유: 탁구처럼 역할을 오가며 진행
```

### 페어 프로그래밍의 조건

```
✅ 효율적 운영:
   ├─ 필요한 만큼만 페어링 (항상 필요한 것은 아님)
   ├─ 너무 많은 대화로 시간 낭비하지 않기
   └─ 집중력 유지

✅ 물리적 환경:
   ├─ 같은 장소에 있어야 함 (원격도 가능하지만 효율 저하)
   ├─ 같은 모니터 공유 가능하게
   └─ 언제든 의사소통 가능하게

✅ 인지적 조화:
   ├─ 개발자의 실력이 균형잡혀야 함
   ├─ 너무 큰 실력 차이는 비효율
   └─ 서로 존중하는 태도 필요
```

### 페어 프로그래밍의 장점

```
장점:
✅ 코드 품질 향상
✅ 오류 조기 발견
✅ 지식 공유
✅ 의사소통 개선
✅ 팀 결속력 강화

단점:
❌ 시간 비용 증가 (초기에)
❌ 성격 충돌 가능
❌ 모든 작업에 필수는 아님
```

### 페어 프로그래밍이 효과적인 경우

```
✅ 복잡한 알고리즘 개발
✅ 핵심 기능 구현
✅ 새로운 기술 학습
✅ 리팩토링 작업
✅ 버그 수정

❌ 단순한 반복 작업
❌ 문서 작성
❌ 테스트 작성 (때로는 필요)
```

## 코드 리뷰

### 코드 리뷰의 목적

```
✅ 결함 발견
✅ 코드 표준 준수 확인
✅ 설계 개선
✅ 성능 최적화
✅ 지식 공유
✅ 팀 역량 향상
```

### 효과적인 코드 리뷰

```
리뷰어의 역할:
✅ 명확한 피드백 제시
✅ 개선점 제안
✅ 칭찬할 점도 언급
✅ 이유 설명

작성자의 태도:
✅ 열린 마음으로 피드백 받기
✅ 방어적 태도 피하기
✅ 토론을 통해 개선
✅ 배우는 자세
```

## 코드 품질 메트릭

### 측정 가능한 지표

```
✅ 코드 중복도
   └─ 얼마나 많은 코드가 반복되는가

✅ 순환 복잡도 (Cyclomatic Complexity)
   └─ 코드의 분기가 얼마나 많은가

✅ 결함 밀도
   └─ 코드 라인당 몇 개의 결함이 있는가

✅ 테스트 커버리지
   └─ 얼마나 많은 코드가 테스트되었는가

✅ 유지보수성 지수
   └─ 코드를 유지보수하기 얼마나 어려운가
```

## 개발자의 성장

### 프로그래밍 기술만으로는 부족

```
❌ 나쁜 개발자의 특징:
   ├─ 프로그래밍 언어만 잘함
   ├─ 코드 품질을 신경 쓰지 않음
   ├─ 리팩토링을 무시함
   └─ 혼자만의 방식 고집

✅ 좋은 개발자의 특징:
   ├─ 프로그래밍 언어 능력
   ├─ 코드 품질 의식
   ├─ 리팩토링 실천
   ├─ 코드 스멜 인식
   ├─ 팀과의 협력
   └─ 지속적 개선
```

### 경력 발전 과정

```
초급 개발자:
└─ "코드가 작동한다" ← 충분하다고 생각

중급 개발자:
└─ "코드가 작동하고 읽기 쉽다" ← 신경 쓰기 시작

고급 개발자:
└─ "코드가 작동하고, 읽기 쉽고, 유지보수하기 쉽다" ← 전문성

시니어 개발자:
└─ "코드뿐만 아니라 팀의 성장도 고려한다"
```

### 개발자로서의 자세

```
항상 생각해야 할 것:

1. "이 코드를 더 간단하게 만들 수는 없을까?"
2. "다른 개발자가 쉽게 이해할 수 있을까?"
3. "코드 스멜이 없을까?"
4. "유지보수하기 쉬울까?"
5. "팀의 성장에 기여하고 있나?"
```

## 정리

### 코드 품질 향상의 단계

```
1단계: 인스펙션 (결함 찾기)
2단계: 정적 분석 (자동 검사)
3단계: 코드 리뷰 (동료 검토)
4단계: 페어 프로그래밍 (협력 개발)
5단계: 리팩토링 (코드 개선)
```

### 좋은 개발자가 되기 위해

```
필수 사항:
✅ 프로그래밍 언어 숙달
✅ 리팩토링 기술 습득
✅ 코드 스멜 인식
✅ 코드 리뷰 참여
✅ 팀과의 협력

결과:
→ 개인의 성장
→ 팀의 역량 향상
→ 전문가로서의 인정
→ 경력 발전과 처우 개선
```

**핵심**: 프로그래밍은 기술일 뿐만 아니라 팀 활동입니다.
