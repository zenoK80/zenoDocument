# 🧪 드라이버와 스텁의 역할 — 소프트웨어 검사 전략 완전 정복

소프트웨어를 만들고 나서 "잘 돌아가겠지"라고 믿고 그냥 배포하면 어떻게 될까요? 실제 현업에서는 그 결과가 고스란히 사용자에게 오류로 전달됩니다. 결제가 두 번 되거나, 주문 데이터가 사라지거나, 시스템이 갑자기 다운되는 일이 벌어지죠. 이런 참사를 막기 위해 우리는 **소프트웨어 검사 전략(Software Testing Strategy)** 을 체계적으로 배워야 합니다.

이번 문서에서는 소프트웨어 테스트의 단계별 흐름을 처음부터 끝까지 살펴보고, 특히 **드라이버(Driver)** 와 **스텁(Stub)** 이라는 핵심 개념을 집중적으로 이해해 봅니다. 처음 들어보는 용어라도 걱정하지 마세요. 실생활 비유와 단계별 설명으로 누구든 이해할 수 있도록 풀어드리겠습니다.

---

## 🏗️ 소프트웨어 테스트란 무엇인가?

### 왜 테스트가 필요한가?

소프트웨어 테스트는 단순히 "오류를 찾는 작업"이 아닙니다. 마치 자동차를 출고하기 전에 공장에서 수천 번의 충돌 테스트, 주행 테스트, 제동 테스트를 거치는 것처럼, 소프트웨어도 실제 사용자에게 전달되기 전에 **체계적인 검증 과정**을 반드시 거쳐야 합니다.

프로그램은 수십, 수백 개의 **모듈(Module, 기능 단위 조각)** 로 이루어져 있습니다. 각 모듈이 혼자서는 잘 작동하더라도, 서로 연결되었을 때 문제가 생기는 경우가 매우 많습니다. 예를 들어 쇼핑몰을 생각해봅시다. 주문 모듈, 결제 모듈, 회원 모듈, 장바구니 모듈, 재고 관리 모듈이 각각 존재합니다. 주문이 완료됐을 때 결제 모듈이 그 데이터를 제대로 받아야 하고, 결제가 끝났을 때 재고 모듈이 수량을 줄여야 합니다. 이 **연결 고리** 중 하나라도 어긋나면 전체 시스템이 오동작합니다.

### 테스트의 단계별 흐름

소프트웨어 테스트는 한 번에 이루어지지 않습니다. 마치 건물을 지을 때 기초 → 골조 → 마감 → 준공검사 순서로 진행되듯, 소프트웨어 테스트도 단계가 있습니다.

| 테스트 단계 | 설명 | 주요 기법 |
|---|---|---|
| **단위 테스트 (Unit Test)** | 모듈 하나하나를 독립적으로 검증 | 화이트박스 기법 |
| **통합 테스트 (Integration Test)** | 모듈들을 연결해서 인터페이스 검증 | 블랙박스 기법 |
| **시스템 테스트 (System Test)** | 전체 시스템을 실제 서버에 올려 검증 | 기능/성능 종합 검사 |
| **인수 테스트 (Acceptance Test)** | 고객이 직접 검증 (알파/베타 테스트) | 사용자 중심 검사 |
| **설치 테스트 (Installation Test)** | 실제 운영 환경에 설치 후 검증 | 환경 호환성 검사 |

이 다섯 단계 중에서 가장 먼저, 그리고 가장 세밀하게 이루어지는 것이 바로 **단위 테스트**입니다. 그리고 단위 테스트를 이해하려면 **드라이버**와 **스텁**이라는 개념을 반드시 알아야 합니다.

## 🔬 단위 테스트와 모듈의 개념

### 모듈이란 무엇인가?

**모듈(Module)** 이란 소프트웨어를 구성하는 최소 단위의 기능 덩어리입니다. 마치 레고 블록처럼 각각의 블록(모듈)이 모여서 하나의 완성된 작품(시스템)을 만들어냅니다.

객체지향 프로그래밍(Object-Oriented Programming, OOP — 현실 세계의 사물처럼 소프트웨어를 설계하는 방법)의 대표 언어인 자바(Java)를 예로 들면, 클래스(Class) 하나가 하나의 모듈이 됩니다. 쇼핑몰 시스템을 예로 들어볼까요?

- **주문 모듈**: 고객이 상품을 선택하고 주문하는 기능 담당
- **결제 모듈**: 카드, 계좌이체 등 결제를 처리하는 기능 담당
- **회원 가입 모듈**: 신규 사용자를 등록하는 기능 담당
- **장바구니 모듈**: 구매 예정 상품을 보관하는 기능 담당
- **재고 관리 모듈**: 상품 수량을 실시간으로 관리하는 기능 담당

이 각각이 독립된 모듈입니다. 그리고 **단위 테스트(Unit Test)** 란 바로 이 모듈 하나하나를 개별적으로 검증하는 작업입니다.

### 단위 테스트의 핵심 특징

단위 테스트는 몇 가지 중요한 특징을 가지고 있습니다.

**첫째, 코딩 단계와 병행해서 수행합니다.** 개발자가 코드를 작성하면서 동시에 테스트도 진행합니다. 모든 개발이 끝난 후에 한꺼번에 테스트하면 오류를 찾기가 훨씬 어렵기 때문입니다.

**둘째, 화이트박스 기법(White-box Testing — 코드 내부를 보면서 하는 테스트)** 을 사용합니다. 개발자가 직접 코드를 들여다보면서 논리적 흐름이 맞는지 확인합니다. 반대로 블랙박스 기법은 내부 코드를 보지 않고 입력과 출력만으로 검증하는 방식입니다.

**셋째, 모듈들은 병렬로 테스트가 가능합니다.** 주문 모듈을 테스트하는 팀과 결제 모듈을 테스트하는 팀이 동시에 작업할 수 있습니다. 각 모듈이 독립적이기 때문에 한 모듈의 테스트가 끝나야만 다른 모듈을 테스트할 수 있는 것이 아닙니다.

> 💡 단위 테스트는 **설계 도면(화면 정의서, 기능 정의서 등 상세 설계 문서)** 을 기초로 하여, 프로그램이 그 설계대로 정상적으로 동작하는지 검증하는 과정입니다.

## 🎭 드라이버(Driver)와 스텁(Stub) — 핵심 개념

### 왜 드라이버와 스텁이 필요한가?

이제 핵심입니다. 단위 테스트를 하려면 한 가지 근본적인 문제가 생깁니다. 바로 **각 모듈은 혼자 존재하지 않는다**는 점입니다.

예를 들어 A 모듈, B 모듈, C 모듈이 순서대로 연결되어 있다고 생각해봅시다. 마치 공장의 컨베이어 벨트처럼, A가 처리한 결과를 B가 받아서 처리하고, B의 결과를 C가 받아서 최종 출력을 냅니다.

```
[A 모듈] → [B 모듈] → [C 모듈] → 최종 결과 출력
```

이때 **A 모듈만 단독으로 테스트**하고 싶다면 어떻게 해야 할까요? 두 가지 문제가 생깁니다.

**문제 1 — 입력 데이터가 없다**: A 모듈은 앞에서 누군가 데이터를 넘겨줘야 실행됩니다. 그런데 A 이전 단계가 없으면 입력 데이터가 없어서 A를 실행할 수 없습니다.

**문제 2 — 결과 확인이 안 된다**: A 모듈이 처리한 결과는 B로 넘어갑니다. 그런데 B가 없으면 A의 결과가 어디로도 가지 않아서, A가 제대로 동작하는지 눈으로 확인할 방법이 없습니다.

이 두 가지 문제를 해결하기 위해 등장한 것이 바로 **드라이버(Driver)** 와 **스텁(Stub)** 입니다.

### 드라이버(Driver) — 모듈을 가동시키는 주체

**드라이버(Driver)** 는 테스트 대상 모듈을 실제로 호출(실행)하고, 테스트에 필요한 데이터를 넣어주며, 결과를 출력해주는 역할을 하는 프로그램입니다.

자동차로 비유하면, **드라이버는 핸들을 잡고 실제로 차를 운전하는 사람**입니다. 차(모듈)가 아무리 좋아도 누군가 운전해주지 않으면 움직이지 않죠. 드라이버가 바로 그 역할을 합니다.

```java
// ✅ 드라이버 예시 — A 모듈을 테스트하기 위한 드라이버 프로그램
public class OrderModuleDriver {

    public static void main(String[] args) {
        // 1. 테스트용 입력 데이터를 직접 만들어서 넣어준다
        //    (실제로는 이전 단계에서 넘어와야 하지만, 드라이버가 대신 생성)
        String testUserId = "user_001";       // 가짜 사용자 ID
        String testProductId = "product_999"; // 가짜 상품 ID
        int testQuantity = 2;                 // 가짜 수량

        // 2. 실제 테스트 대상 모듈(A 모듈)을 호출한다
        OrderModule orderModule = new OrderModule();
        OrderResult result = orderModule.processOrder(testUserId, testProductId, testQuantity);

        // 3. 결과를 출력해서 정상 동작 여부를 확인한다
        System.out.println("주문 처리 결과: " + result.getStatus());
        System.out.println("주문 번호: " + result.getOrderId());
    }
}
```

:::info 드라이버의 역할 요약
드라이버는 테스트 대상 모듈을 **대신 실행시켜주는 가짜 상위 프로그램**입니다. 실제 환경에서는 이전 단계 모듈이 데이터를 넘겨주겠지만, 단위 테스트 시에는 드라이버가 그 역할을 대신합니다.
:::

### 스텁(Stub) — 다른 모듈을 흉내 내는 가짜 모듈

**스텁(Stub)** 은 테스트 대상 모듈이 호출하는 **다른 모듈을 흉내 내는 가짜 프로그램**입니다. 입력과 출력만 흉내 내는, 기능이 거의 없는 최소한의 가상 프로그램입니다.

마치 영화 촬영 현장에서 **스턴트맨(대역 배우)** 이 실제 배우 대신 위험한 장면을 찍는 것처럼, 스텁은 아직 완성되지 않은(또는 단위 테스트 시 함께 쓸 수 없는) 하위 모듈 대신 최소한의 응답만 해줍니다.

```java
// ✅ 스텁 예시 — 결제 모듈(B)이 아직 완성되지 않았을 때 사용하는 가짜 결제 모듈

// ❌ 잘못된 방식: B 모듈이 없어서 A 모듈 테스트 자체가 불가능
// -> 컴파일 오류 또는 NullPointerException 발생
public class OrderModule {
    public OrderResult processOrder(String userId, String productId, int quantity) {
        // 주문 처리 로직...
        PaymentModule payment = new PaymentModule(); // B 모듈이 없으면 여기서 오류!
        PaymentResult payResult = payment.pay(totalPrice);
        return new OrderResult(payResult);
    }
}

// ✅ 올바른 방식: 스텁으로 결제 모듈(B)을 흉내 낸다
public class PaymentModuleStub {
    // 실제 결제 로직은 없고, 테스트에 필요한 최소한의 응답만 반환
    public PaymentResult pay(int amount) {
        System.out.println("[스텁] 결제 요청 금액: " + amount + "원 — 가짜 승인 처리");
        // 항상 성공 응답을 돌려줌 (실제 카드사 연결 없음)
        return new PaymentResult("SUCCESS", "STUB-APPROVAL-0001");
    }
}

// ✅ A 모듈(주문)이 스텁을 사용해서 단독으로 테스트 가능
public class OrderModule {
    public OrderResult processOrder(String userId, String productId, int quantity) {
        int totalPrice = quantity * 10000; // 개당 10,000원 가정

        // 실제 결제 모듈 대신 스텁을 사용
        PaymentModuleStub paymentStub = new PaymentModuleStub();
        PaymentResult payResult = paymentStub.pay(totalPrice);

        // 주문 결과 생성 및 반환
        return new OrderResult("ORDER-" + System.currentTimeMillis(), payResult.getStatus());
    }
}
```

:::tip 스텁(Stub)을 왜 "가짜 모듈"이라고 하나요?
스텁은 실제 기능을 수행하지 않습니다. 그냥 **"나는 이런 입력을 받았고, 이런 출력을 돌려줄게"** 라는 최소한의 역할만 합니다. 덕분에 아직 완성되지 않은 하위 모듈이 있어도, 해당 모듈에 의존하는 상위 모듈을 먼저 테스트할 수 있습니다.
:::

### 드라이버 vs. 스텁 — 한눈에 비교

| 구분 | 드라이버 (Driver) | 스텁 (Stub) |
|---|---|---|
| **역할** | 테스트 대상 모듈을 실행시킴 | 하위 모듈을 흉내 냄 |
| **방향** | 위에서 아래로 (상위 → 대상 모듈) | 아래에서 위로 (대상 모듈 → 하위) |
| **실제 비유** | 차를 운전하는 사람 | 스턴트맨(대역 배우) |
| **필요한 시점** | 상위 모듈이 없을 때 | 하위 모듈이 없거나 사용 불가할 때 |
| **기능 수준** | 테스트 데이터 생성 + 호출 + 결과 확인 | 입출력만 흉내 내는 최소 기능 |

> 🔑 **핵심**: 스텁이 없으면 단위 테스트를 수행할 수 없습니다. 왜냐하면 모든 모듈은 다른 모듈과 연결되어 있기 때문에, 연결된 모듈 없이 홀로 테스트하려면 반드시 대역(스텁)이 필요합니다.

## 🔗 통합 테스트 — 모듈을 연결해서 검증하기

### 통합 테스트란?

단위 테스트가 모두 끝났다고 해서 시스템이 완성된 것은 아닙니다. 개별 레고 블록이 각각 멀쩡하다고 해도, 블록을 합쳤을 때 모양이 이상하게 나올 수 있는 것처럼, **각 모듈을 연결했을 때 제대로 동작하는지** 확인하는 것이 **통합 테스트(Integration Test)** 입니다.

통합 테스트에서 가장 중요한 것은 **인터페이스(Interface — 모듈과 모듈 사이의 연결 방식)** 가 제대로 동작하는지입니다. A 모듈이 처리한 결과가 B 모듈로 올바르게 넘어가는지, 데이터 형식이 맞는지, 빠지거나 손상된 데이터가 없는지를 꼼꼼히 확인합니다.

통합 테스트는 **블랙박스 기법(Black-box Testing — 내부 코드를 보지 않고 입력과 출력만으로 검증)** 을 주로 사용합니다. 단위 테스트에서 이미 내부 코드를 검증했으므로, 통합 테스트에서는 전체 흐름이 올바른지에 집중합니다.

### 인터페이스의 유형

모듈과 모듈 사이를 이어주는 인터페이스에는 여러 종류가 있습니다. 각각 어떻게 다른지 살펴봅시다.

**① 매개변수 인터페이스 (Parameter Interface)**

가장 일반적인 방식입니다. 함수(또는 메서드)를 호출할 때 **변수(매개변수)** 를 통해 데이터를 주고받는 방식입니다. 마치 택배 기사가 물건(데이터)을 집(함수) 안에 직접 전달하는 것처럼, 한 모듈이 다른 모듈을 호출하며 필요한 데이터를 직접 넘겨줍니다.

```java
// ✅ 매개변수 인터페이스 예시
// calculateTotal이라는 함수가 price와 quantity를 매개변수로 받아 결과를 리턴
public int calculateTotal(int price, int quantity) {
    return price * quantity; // 총액을 계산해서 반환 (리턴 = 인터페이스)
}

// 다른 모듈에서 이 함수를 호출
int total = calculateTotal(10000, 3); // 30000원이 다음 모듈로 전달됨
```

**② 공유 메모리 인터페이스 (Shared Memory Interface)**

여러 모듈이 **동일한 메모리 공간(물리적 저장 공간)** 을 함께 사용해서 데이터를 주고받는 방식입니다. 마치 회사 공용 게시판에 누군가 메모를 붙여두면 다른 사람이 그 메모를 읽어가는 것처럼, 메모리라는 공용 공간을 통해 데이터가 공유됩니다.

**③ 프로시저 인터페이스 (Procedure Interface)**

서브시스템(Subsystem — 시스템 내부의 독립적인 하위 시스템) 전체가 다른 서브시스템을 호출할 때 사용하는 방식입니다. 예를 들어 주문 서브시스템이 완료되면 그 결과(주문 내역 전체)가 마이페이지 서브시스템으로 전달되어야 합니다. 단순히 변수 하나를 넘기는 것이 아니라, 처리 결과 묶음 전체를 전달하는 방식입니다.

**④ 메시지 전달 인터페이스 (Message Passing Interface)**

컴포넌트(Component — 독립적으로 동작하는 소프트웨어 구성 요소)가 다른 컴포넌트에게 **메시지를 보내서** 서비스를 요청하는 방식입니다. 결제 버튼을 클릭했을 때, 우리 시스템이 외부 결제 서비스(예: 이니시스)에 결제 요청 메시지를 보내고, 결제 서비스가 승인 결과 메시지를 돌려주는 것이 대표적인 예입니다.

:::warning HTTP와 HTTPS 주의
메시지 전달 인터페이스와 관련해서 실무에서 자주 발생하는 문제가 있습니다. 우리 시스템이 **HTTP**(보안이 없는 일반 통신 방식)로 운영되는데, 결제 시스템이 **HTTPS**(Secure — 암호화된 보안 통신 방식)만 허용한다면 데이터를 주고받는 과정에서 오류가 발생합니다. 이런 환경 불일치 문제도 통합 테스트에서 반드시 확인해야 합니다.
:::

### 인터페이스 오류의 종류

통합 테스트에서 발견되는 인터페이스 오류는 크게 세 가지입니다.

**잘못된 사용 (Interface Misuse)**: 다른 컴포넌트를 호출할 때 인터페이스를 잘못 사용하는 경우입니다. 예를 들어 숫자를 넘겨야 하는데 문자열을 넘기거나, 데이터를 엉뚱한 모듈로 보내는 경우입니다.

**잘못된 이해 (Interface Misunderstanding)**: 인터페이스 명세서(인터페이스의 사용 방법을 기술한 문서)를 잘못 이해해서 원하는 데이터를 호출하지 못하거나, 데이터를 엉뚱한 곳으로 전송하는 경우입니다.

**타이밍 오류 (Timing Error)**: 호출하는 컴포넌트와 호출받는 컴포넌트의 **처리 속도 차이** 때문에 발생하는 오류입니다. 결제를 요청했는데 결제 서버의 응답이 늦어지면, 성격 급한 사용자는 결제 버튼을 한 번 더 누를 수 있습니다. 그러면 **이중 결제**가 발생하는 것이죠. 이런 타이밍 관련 문제도 통합 테스트에서 꼭 확인해야 합니다.

### 통합 테스트의 종류

통합 테스트는 어떤 순서로 모듈을 합쳐나가느냐에 따라 다음과 같이 나뉩니다.

**① 빅뱅 통합 (Big-Bang Integration) — 비점진적 방식**

모든 모듈을 한꺼번에 합쳐서 한 번에 테스트하는 방식입니다. 빅뱅(우주 대폭발)처럼 한꺼번에 터뜨리는 방식이죠. 그러나 오류가 발생했을 때 어느 모듈에서 문제가 생겼는지 찾기 매우 어렵고, 시간도 많이 걸립니다. **실무에서 권장하지 않는 방식**입니다.

**② 하향식 통합 (Top-Down Integration)**

최상위 주(Main) 프로그램에서 시작해 아래쪽 서브 모듈로 점점 내려가면서 통합하는 방식입니다. 마치 조직도의 대표이사부터 시작해서 팀장, 팀원 순으로 내려가는 것처럼요.

| 장점 | 단점 |
|---|---|
| 오류 원인을 찾기 쉬움 | 병렬 테스트가 어려움 |
| 실제 운영 환경과 유사 | 스텁이 필요함 |
| 주요 기능 테스트에 유리 | 최하위 모듈의 충분한 테스트 어려움 |
| 별도 드라이버 불필요 | 위에서 아래로만 진행 가능 |

하향식 통합에서는 아직 완성되지 않은 하위 모듈 대신 **스텁**을 사용하여 테스트를 진행합니다. 테스트가 끝나는 순서대로 스텁을 실제 모듈로 교체해 나갑니다.

**③ 상향식 통합 (Bottom-Up Integration)**

하향식과 반대로, 가장 아래의 기본 모듈부터 시작해서 위쪽으로 올라가면서 통합하는 방식입니다. 이때는 각 모듈을 실행시키는 **테스트 드라이버**가 필요합니다.

| 장점 | 단점 |
|---|---|
| 개발 초기에 병렬 작업 가능 | 많은 테스트 드라이버 필요 |
| 철저한 모듈 단위 테스트 가능 | 중요한 오류를 마지막에야 발견 |
| | 시스템 전체 검증은 마지막 단계에서만 가능 |

**④ 연쇄식 통합 / 샌드위치 통합 (Thread Integration / Sandwich Integration)**

하향식과 상향식을 혼합한 방식입니다. **중요하고 핵심적인 기능은 먼저 테스트**하고, 보조적인 기능은 나중에 테스트합니다. 예를 들어 쇼핑몰에서 "주문과 결제"가 가장 중요하다면 그것을 먼저 통합 테스트하고, 나머지 화면 표시나 회원 관리 같은 보조 기능은 나중에 테스트하는 방식입니다.

## 🖥️ 시스템 테스트, 인수 테스트, 설치 테스트

### 시스템 테스트 — 실제 서버에 올려서 검증

단위 테스트와 통합 테스트가 모두 끝나면, 이제 소프트웨어를 **실제 서버(하드웨어)에 올려서** 전체적으로 테스트하는 **시스템 테스트(System Test)** 를 수행합니다. 소프트웨어만 검증하는 것이 아니라, 소프트웨어와 하드웨어가 결합된 실제 환경에서 제대로 동작하는지 확인합니다.

시스템 테스트에서 확인하는 주요 항목들은 다음과 같습니다.

- **복구 테스트**: 시스템이 갑자기 다운됐을 때 정상적으로 복구되는지 확인
- **민감도 테스트**: 유효한 데이터인데도 불안정하게 처리되는 경우는 없는지 확인
- **배치 테스트**: 실제 운영 환경(백신, 특정 소프트웨어 등)에서 충돌 없이 동작하는지 확인
- **안전 테스트**: 해킹이나 외부 공격으로부터 시스템이 안전하게 보호되는지 확인
- **스트레스 테스트**: 최대 동시 접속자 수를 훨씬 초과하는 부하를 일부러 주어 서버가 다운되는지 확인
- **성능 테스트**: CPU, 메모리, 로그 파일 등 하드웨어 자원이 효율적으로 관리되는지 확인
- **부피 테스트**: 대용량 데이터가 들어왔을 때 정상 처리되는지 확인
- **구성 테스트**: 다양한 하드웨어 기종과 환경에서 정상 동작하는지 확인

### 인수 테스트 — 고객이 직접 검증하는 단계

시스템 테스트까지 통과했다면, 이제 개발된 소프트웨어를 고객에게 인도하기 전 고객이 직접 검증하는 **인수 테스트(Acceptance Test)** 단계입니다. 마치 집을 지은 후 집주인이 직접 돌아다니며 점검하는 것과 같습니다.

**알파 테스트 (Alpha Test)**: **개발자가 참석한 통제된 환경**에서 선택된 사용자가 테스트를 진행합니다. 개발 환경(개발사의 서버와 장비)에서 이루어지며, 개발자가 옆에서 지켜보고 바로 피드백을 받을 수 있습니다.

**베타 테스트 (Beta Test)**: 게임 출시 전 베타 테스트를 떠올리면 쉽습니다. **개발자 없이 고객의 환경(고객 집, 고객 기기)에서** 불특정 다수의 사용자가 자유롭게 테스트하는 방식입니다. 발견된 오류는 개발자에게 보고합니다. 실제 사용 환경을 그대로 재현하기 때문에 오류가 발견될 가능성이 가장 높은 방식입니다.

### 설치 테스트 — 실제 운영 환경에 설치 후 마지막 점검

모든 테스트가 끝나면 고객의 실제 운영 환경에 소프트웨어를 설치하고 마지막으로 점검하는 **설치 테스트(Installation Test)** 를 수행합니다. 이 단계에서도 예상치 못한 결함이 발견될 수 있습니다.

- 서버 사양이 요구사항에 맞는지 확인
- 파일, 로그 파일, 데이터베이스 설정이 올바른지 확인
- CPU 성능, 메모리, 저장장치 구조(RAID 등)가 적절히 구성됐는지 확인
- 다른 소프트웨어와의 연동(인터페이스)에 문제가 없는지 확인

:::info 백투백 테스트(Back-to-Back Test)
설치 테스트 과정에서 **시스템의 여러 버전을 동시에 테스트**하는 방법입니다. 버전 1.0과 버전 2.0에 동일한 테스트를 적용해서 두 결과가 서로 다르면, 둘 중 하나에 문제가 있다는 신호입니다. 버전 업그레이드나 마이그레이션(시스템 이전) 시 유용하게 활용됩니다.
:::

## 🛠️ 자동 검사 도구 — 정적 분석기와 동적 분석기

### 정적 분석기 (Static Analyzer) — 실행 없이 코드를 분석

**정적 분석기**란 프로그램을 실제로 실행하지 않고, **원시 코드(Source Code — 개발자가 작성한 프로그램 코드)를 그대로 분석**하는 도구입니다. 마치 국어 선생님이 작문을 첨삭해주는 것처럼, 코드를 실행하지 않고 읽으면서 문제점을 찾아냅니다.

정적 분석이 특히 유용한 경우는 다음과 같습니다.

- **무한 루프(Infinite Loop)**: 프로그램이 끝나지 않고 같은 동작을 무한히 반복하는 코드
- **초기화되지 않은 변수 사용**: 값을 할당하지 않은 변수를 그대로 사용하는 경우
- **실행 불가능한 코드**: 어떤 조건에서도 실행되지 않는 죽은 코드(Dead Code)

```python
# ❌ 잘못된 코드 — 정적 분석기가 잡아내는 문제들

def calculate_price(quantity):
    # 문제 1: total_price가 초기화되지 않은 채로 사용됨
    # 문제 2: 조건에 따라 total_price가 할당되지 않을 수도 있음
    if quantity > 0:
        total_price = quantity * 10000  # quantity가 0 이하면 total_price는 미정의 상태!

    return total_price  # 초기화되지 않은 변수 반환 → 런타임 오류 발생 가능

def never_reached():
    return "이 함수는 절대 호출되지 않음"  # Dead Code — 실행 불가 코드

# ✅ 올바른 코드 — 정적 분석 경고를 해결한 버전

def calculate_price(quantity):
    total_price = 0  # 반드시 초기화!

    if quantity > 0:
        total_price = quantity * 10000  # 수량이 양수일 때만 계산

    return total_price  # 항상 초기화된 값을 반환
```

정적 분석 도구의 종류는 다음과 같습니다.

- **코드 분석 도구**: 문법 오류나 잠재적 오류 가능성을 찾아주는 도구
- **구조 검사 도구**: 코드의 논리 흐름을 그래프(제어 흐름도)로 시각화해서 구조적 결함을 찾는 도구
- **데이터 분석 도구**: 변수가 제대로 초기화되어 있는지, 중복 데이터는 없는지, 잘못된 데이터가 사용되지 않는지 확인하는 도구

### 동적 분석기 (Dynamic Analyzer) — 실행 중에 동작을 분석

**동적 분석기**는 정적 분석기와 반대로, 프로그램을 **실제로 실행하면서** 그 동작과 반응을 추적하는 도구입니다. 마치 블랙박스 영상을 보면서 사고 원인을 분석하는 것처럼, 프로그램이 실행되는 과정을 실시간으로 관찰합니다.

동적 분석의 핵심 방법은 **원시 코드 중간에 검사 코드(인스트루먼트 코드)를 삽입**하고, 프로그램을 실행하면서 그 중간 결과들을 확인하는 것입니다. 특정 시점의 메모리 상태를 캡처(스냅샷)하거나, 특정 이벤트 발생 시 변수의 값을 덤프(Dump — 메모리 내용을 그대로 출력)해서 문제를 진단합니다.

```python
# ✅ 동적 분석을 위한 검사 코드(인스트루먼트 코드) 삽입 예시

def process_order(user_id, product_id, quantity):
    print(f"[DEBUG] 주문 시작 — 사용자: {user_id}, 상품: {product_id}, 수량: {quantity}")

    # 실제 주문 로직
    total_price = quantity * get_product_price(product_id)
    print(f"[DEBUG] 총액 계산 완료 — total_price: {total_price}")  # 중간 값 확인

    payment_result = request_payment(total_price)
    print(f"[DEBUG] 결제 요청 결과 — status: {payment_result['status']}")  # 결제 결과 확인

    if payment_result['status'] == 'SUCCESS':
        order_id = save_order(user_id, product_id, quantity, total_price)
        print(f"[DEBUG] 주문 저장 완료 — order_id: {order_id}")  # 저장 확인
        return {"success": True, "order_id": order_id}
    else:
        print(f"[DEBUG] 결제 실패 — 원인: {payment_result['reason']}")  # 실패 원인 확인
        return {"success": False, "reason": payment_result['reason']}
```

## 🐛 디버깅 — 오류를 찾아 수정하는 과정

### 디버깅이란?

**디버깅(Debugging)** 은 소프트웨어에서 발견된 오류(버그, Bug)를 찾아내고 수정하는 작업입니다. 개발자가 실무에서 가장 많이 하는 작업 중 하나입니다. "버그"라는 단어의 유래는 초창기 컴퓨터에 실제 벌레(bug)가 들어가 오동작을 일으킨 데서 비롯되었다는 이야기도 있습니다.

사용자가 오류를 신고하면 개발자는 해당 오류를 **분석 → 원인 파악 → 코드 수정 → 재배포** 순서로 처리합니다. 이 전체 과정이 디버깅입니다.

### 디버깅 방법의 종류

**① 맹목적 디버깅**: 문제 원인을 메모리에서 직접 찾는 방식입니다. 매우 비효율적이며 권장하지 않습니다.

**② 출력문 삽입 방식**: 코드 중간중간에 `print()` 문을 삽입해서 중간 결과를 확인하는 방식입니다. 간단하지만 삽입한 출력문을 나중에 모두 제거해야 하는 번거로움이 있습니다.

**③ 도구 이용 방식**: 디버깅 도구(IDE의 디버거)를 사용해서 코드 실행을 단계별로 추적하고, 특정 지점에서 실행을 일시 정지(브레이크포인트, Breakpoint)시켜 변수 값을 확인하는 방식입니다.

디버깅 전략은 크게 세 가지로 나뉩니다.

- **귀납적 방법**: 관련 정보를 수집해서 사실을 하나씩 따져가며 원인을 찾는 방식
- **연역적 방법**: 가능한 모든 원인을 미리 나열하고, 하나씩 제거해가며 원인을 좁히는 방식
- **역행 조사 (Back Tracking)**: 오류가 발생한 지점에서 원시 코드를 거꾸로 추적해 원인을 찾는 방식

### 디버깅 시 반드시 지켜야 할 원칙

:::warning 디버깅 원칙 — 반드시 숙지하세요

1. **자신의 힘으로 불가능하면 도움을 요청하라**: 끝까지 혼자 해결하려 하지 말고, 전문가나 동료의 도움을 받으세요.
2. **디버깅 도구는 만능이 아니다**: 도구는 차선책입니다. 처음부터 코드를 잘 작성하는 것이 최선입니다.
3. **시스템 전체 구조를 정확히 이해하라**: 전체 맥락 없이 부분만 보면 오류의 근본 원인을 놓칩니다.
4. **무작위 실험은 최후의 수단이다**: 이것저것 되는 대로 바꿔보는 방식은 가능한 한 피하세요.
5. **하나의 결함 옆에는 또 다른 결함이 있을 수 있다**: 오류 하나를 고쳤다고 안심하지 마세요.
:::

:::danger 교정이 새로운 결함을 만들 수 있다
A라는 오류를 고쳤는데, 며칠 후 B라는 새로운 오류가 발생하고, 분석해보니 B가 A의 수정 때문에 생겨난 것이라면? 이런 경우 **전체 시스템에 대한 포괄적인 재점검**이 필요합니다. 오류 수정이 또 다른 오류를 만들어낼 확률은 프로그램 규모가 클수록 높아집니다. 교정이 정확할 확률은 시스템 규모에 **반비례**합니다.
:::

디버깅 절차를 정리하면 다음과 같습니다.

1. **에러 위치 파악**: 디버깅 도구를 사용해 오류가 발생한 정확한 코드 위치 확인
2. **설계 오류 수정**: 설계 자체가 잘못된 경우 설계부터 수정
3. **코드 오류 수정**: 실제 소스 코드의 잘못된 부분 수정
4. **프로그램 재시작 및 배포**: 수정된 프로그램을 서버에 다시 올려 최신화

## ✅ 핵심 정리

- **모듈(Module)**: 소프트웨어를 구성하는 최소 기능 단위로, 단위 테스트의 기본 대상이다.
- **단위 테스트(Unit Test)**: 모듈 하나하나를 독립적으로 검증하는 테스트로, 화이트박스 기법을 사용한다.
- **드라이버(Driver)**: 테스트 대상 모듈을 실행시키고 데이터를 넣어주는 가짜 상위 프로그램이다.
- **스텁(Stub)**: 테스트 대상 모듈이 호출하는 하위 모듈을 흉내 내는 가짜 모듈로, 입출력만 흉내 낸다.
- **스텁이 없으면 단위 테스트 불가**: 모든 모듈은 다른 모듈과 연결되므로, 하위 모듈 대역(스텁)이 반드시 필요하다.
- **통합 테스트(Integration Test)**: 모듈을 연결해 인터페이스가 올바로 동작하는지 검증하며, 블랙박스 기법을 사용한다.
- **인터페이스 유형**: 매개변수, 공유 메모리, 프로시저, 메시지 전달 방식이 있다.
- **통합 방식**: 빅뱅(한꺼번에), 하향식(위→아래), 상향식(아래→위), 연쇄식(중요 기능 먼저) 네 가지가 있다.
- **하향식 통합**: 드라이버 불필요, 스텁 필요 / **상향식 통합**: 드라이버 필요, 스텁 불필요.
- **시스템 테스트**: 실제 서버에 소프트웨어를 올려 성능·복구·안전·스트레스 등을 종합 검증한다.
- **인수 테스트**: 알파(개발 환경에서 사용자 테스트)와 베타(고객 환경에서 자유 테스트)로 나뉜다.
- **설치 테스트**: 실제 운영 환경에 설치 후 하드웨어 호환성, 연동 문제 등을 최종 점검한다.
- **정적 분석기**: 코드를 실행하지 않고 소스 코드 자체를 분석해 잠재 오류를 찾는 도구다.
- **동적 분석기**: 프로그램을 실행하면서 동작과 반응을 실시간 추적해 오류를 찾는 도구다.
- **디버깅(Debugging)**: 발견된 오류를 분석하고 수정하는 작업으로, 오류 수정이 새로운 오류를 만들 수 있음을 항상 유의해야 한다.
