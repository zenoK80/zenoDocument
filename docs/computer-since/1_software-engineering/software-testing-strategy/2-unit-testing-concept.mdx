# 🧪 단위 테스트의 개념과 모듈 — 소프트웨어 검사 전략 완전 정복

소프트웨어를 개발할 때 "만들었으면 됐지, 왜 이렇게 테스트를 많이 해야 하나요?"라고 생각하신 적 있으신가요? 하지만 실제 현업에서는 테스트 단계가 개발 단계만큼이나, 어쩌면 그보다 더 중요합니다. 이번 문서에서는 소프트웨어 검사 전략의 핵심인 **단위 테스트(Unit Test)** 부터 시작해서, 통합 테스트, 시스템 테스트, 인수 테스트, 설치 테스트까지 전 과정을 초보자도 쉽게 이해할 수 있도록 상세히 안내합니다. 그리고 오류가 발생했을 때 어떻게 디버깅하는지까지 한 번에 마스터해 봅시다!

---

## 🏗️ 소프트웨어 테스트 단계 개요 — 왜 단계별로 테스트할까?

### 테스트를 단계별로 나누는 이유

집을 지을 때를 상상해 보세요. 벽돌 하나하나의 품질을 먼저 확인하고(단위 테스트), 그 벽돌들이 서로 잘 붙어 있는지 확인하고(통합 테스트), 완성된 집 전체가 튼튼한지 확인하고(시스템 테스트), 집주인이 직접 들어와 살아보면서 마음에 드는지 확인합니다(인수 테스트). 소프트웨어 테스트도 정확히 이런 방식으로 단계별로 이루어집니다.

소프트웨어 테스트는 크게 다음과 같은 단계로 구성됩니다.

| 테스트 단계 | 주요 목적 | 주요 기법 |
|---|---|---|
| **단위 테스트 (Unit Test)** | 모듈 단위의 개별 기능 검증 | 화이트박스 기법 |
| **통합 테스트 (Integration Test)** | 모듈 간 인터페이스 및 연결 검증 | 블랙박스 기법 |
| **시스템 테스트 (System Test)** | 전체 시스템의 기능·성능 검증 | 하드웨어+소프트웨어 통합 |
| **인수 테스트 (Acceptance Test)** | 고객 요구사항 충족 여부 확인 | 알파·베타 테스트 |
| **설치 테스트 (Installation Test)** | 실제 고객 환경에서의 정상 동작 확인 | 환경 구성 검증 |

이렇게 단계를 나누는 가장 큰 이유는 **오류를 일찍 발견할수록 수정 비용이 훨씬 적게 든다**는 원칙 때문입니다. 벽돌을 잘못 쌓은 걸 집 완공 후에 발견하면 집 전체를 허물어야 하지만, 벽돌 하나를 쌓는 도중에 발견하면 그 벽돌 하나만 교체하면 됩니다. 소프트웨어도 마찬가지입니다.

:::info 객체지향 프로그래밍과 테스트의 관계
소프트웨어 공학에서 주로 다루는 개발 방법론은 **객체지향 프로그래밍(OOP, Object-Oriented Programming)** 을 기반으로 합니다. 자바(Java)가 대표적인 객체지향 언어이며, 클래스(Class)와 모듈(Module)이라는 단위로 프로그램을 구조화합니다. 만약 프로그램이 처음부터 끝까지 하나로 쭉 이어져 있는 순차적 프로그래밍 방식이라면, 단위 테스트나 통합 테스트 같은 개념이 큰 의미가 없습니다. 하지만 현대 소프트웨어는 대부분 모듈 단위로 분리되어 있기 때문에, 이 테스트 단계 구분이 매우 중요합니다.
:::

## 🔬 단위 테스트(Unit Test) — 가장 작은 단위부터 검증하자

### 모듈(Module)이란 무엇인가?

단위 테스트를 이해하려면 먼저 **모듈(Module)** 이 무엇인지 알아야 합니다. 모듈은 소프트웨어 설계의 최소 단위입니다. 마치 레고 블록 하나하나와 같습니다. 예를 들어, 쇼핑몰 시스템을 생각해 봅시다.

- 🛒 **주문 모듈** — 고객이 상품을 주문하는 기능
- 👤 **회원 가입 모듈** — 신규 회원 등록 기능
- 💳 **결제 모듈** — 결제를 처리하는 기능
- 🛍️ **장바구니 모듈** — 상품을 담아두는 기능
- 📦 **재고 관리 모듈** — 재고 수량을 관리하는 기능

이 각각이 하나의 **독립된 모듈**입니다. 주문 모듈 안에는 주문 화면, 주문 데이터, 주문 절차에 관한 설계가 들어 있습니다. 단위 테스트는 바로 이 **각각의 모듈을 개별적으로 테스트하는 것**입니다.

> 💡 **단위 테스트 = 설계의 최소 단위인 모듈 하나하나를 독립적으로 검증하는 테스트**

### 단위 테스트의 주요 특징

**단위 테스트는 코딩 단계와 병행해서 수행합니다.** 개발자가 코드를 작성하면서 동시에 그 모듈이 잘 동작하는지 확인하는 방식입니다. 마치 요리사가 요리를 하면서 중간중간 맛을 보는 것과 같습니다. 완성 후에 한 번에 맛보는 것보다 과정 중에 확인하는 것이 훨씬 효율적입니다.

**단위 테스트는 화이트박스(White-box) 기법을 사용합니다.** 화이트박스 기법이란 코드의 내부 구조를 보면서 테스트하는 방식입니다. 마치 시계의 뚜껑을 열고 내부 톱니바퀴를 직접 들여다보면서 점검하는 것과 같습니다. 상세 설계도(화면 정의서, 기능 정의서 등)를 기초로 프로그램이 정상적으로 작동하는지 확인합니다.

**모듈은 독립적이기 때문에 병렬(동시에) 테스트가 가능합니다.** 주문 모듈 테스트가 끝나야 결제 모듈을 테스트할 수 있는 것이 아닙니다. 각 모듈이 독립적이기 때문에 주문 팀은 주문 모듈을, 결제 팀은 결제 모듈을 동시에 테스트할 수 있습니다. 이것이 개발 효율성을 크게 높여줍니다.

:::tip 단위 테스트의 핵심 세 가지
- **병렬 테스트 가능**: 모듈이 독립적이므로 여러 팀이 동시에 각각의 모듈을 테스트할 수 있음
- **화이트박스 기법**: 코드를 직접 보면서 테스트
- **코딩과 병행**: 개발 중에 함께 수행
:::

### 드라이버(Driver)와 스텁(Stub) — 단위 테스트의 필수 도구

여기서 아주 중요한 개념이 등장합니다. 바로 **드라이버(Driver)** 와 **스텁(Stub)** 입니다. 처음 들으면 낯선 용어지만, 이 개념을 이해하면 단위 테스트를 훨씬 깊이 이해할 수 있습니다.

**왜 드라이버와 스텁이 필요할까요?**

예를 들어 A 모듈, B 모듈, C 모듈이 순서대로 연결된 시스템이 있다고 가정합시다.

```
[A 모듈] → [B 모듈] → [C 모듈] → 최종 결과 출력
```

이 상황에서 A 모듈만 단독으로 테스트하려고 합니다. 그런데 문제가 있습니다.

- **입력 데이터 문제**: A 모듈은 앞에서 데이터를 받아야 하는데, 아직 앞 단계가 없습니다.
- **출력 결과 확인 문제**: A 모듈의 결과는 B, C를 거쳐야만 최종 화면에 나타납니다. A만 테스트해서는 결과를 눈으로 확인할 수가 없습니다.

이 문제를 해결하기 위해 **가짜 입력/출력 역할을 해주는 프로그램**이 필요합니다. 바로 이것이 스텁(Stub)입니다.

| 용어 | 역할 | 비유 |
|---|---|---|
| **드라이버 (Driver)** | 실제 모듈을 실행하고, 데이터를 넣고, 결과를 출력하는 실제 구동기 | 자동차의 실제 엔진 |
| **스텁 (Stub)** | 아직 개발되지 않았거나 연결되지 않은 다른 모듈을 흉내 내는 가짜 모듈 | 테스트용 가짜 배우(스턴트맨) |

**스텁(Stub)** 은 입출력만 흉내 내는 **무기능의 가상 하위 프로그램**입니다. 인터페이스(모듈 간 연결 방식)를 확인할 수 있는 최소한의 기능만 가진 가상의 루틴입니다. 실제 로직은 없지만, "데이터를 받아서 가짜 결과를 넘겨주는" 역할을 합니다.

```java
// ✅ 스텁(Stub) 예시 — 실제 결제 처리 모듈 대신 사용하는 가짜 모듈
public class PaymentModuleStub {
    
    /**
     * 실제 결제 처리 로직은 없고, 테스트용 가짜 결과만 반환하는 스텁
     * 실제 모듈이 완성되기 전까지 이 스텁이 결제 성공 여부를 흉내냄
     */
    public boolean processPayment(int amount) {
        // 실제 카드사 API를 호출하는 대신, 무조건 성공을 반환 (테스트용)
        System.out.println("[STUB] 결제 처리 흉내 중... 금액: " + amount + "원");
        return true; // 항상 결제 성공으로 처리 (가짜 결과)
    }
}
```

```java
// ✅ 드라이버(Driver) 예시 — 주문 모듈을 직접 실행시키는 구동기
public class OrderModuleDriver {
    
    public static void main(String[] args) {
        // 주문 모듈을 테스트하기 위해 드라이버가 직접 호출
        OrderModule orderModule = new OrderModule();
        
        // 테스트용 가짜 입력 데이터 생성
        String testProductId = "PROD-001";     // 테스트할 상품 ID
        int testQuantity = 2;                   // 테스트할 수량
        int testUserId = 12345;                 // 테스트할 사용자 ID
        
        // 주문 모듈 실행 및 결과 확인
        boolean result = orderModule.placeOrder(testProductId, testQuantity, testUserId);
        
        // 결과 출력 — 단위 테스트 결과 검증
        System.out.println("주문 처리 결과: " + (result ? "성공 ✅" : "실패 ❌"));
    }
}
```

:::warning 스텁 없이 단위 테스트는 불가능합니다
모듈이 혼자서는 입출력을 완성할 수 없을 때, 스텁이 없으면 단위 테스트 자체를 수행할 수 없습니다. 현업에서 "스텁도 만들어서 테스트해야 하는 거 아니야?"라는 말이 나오면, 이것은 단위 테스트를 제대로 하기 위한 준비 작업을 이야기하는 것입니다.
:::

## 🔗 통합 테스트(Integration Test) — 모듈들을 연결해서 확인하자

### 통합 테스트가 왜 필요한가?

단위 테스트에서 A 모듈, B 모듈, C 모듈 각각은 완벽하게 동작한다는 것을 확인했습니다. 그런데 이 세 모듈을 실제로 연결하면 어떤 일이 발생할까요? 마치 각각의 기계 부품이 단독으로는 완벽한데, 막상 조립하면 맞지 않는 경우처럼, **모듈 간 연결(인터페이스)에서 새로운 문제가 발생할 수 있습니다.**

통합 테스트는 이렇게 **여러 모듈을 결합해서 인터페이스가 올바르게 작동하는지 검증**하는 테스트입니다. 단위 테스트가 화이트박스 기법이었다면, 통합 테스트는 **블랙박스(Black-box) 기법**을 사용합니다. 블랙박스 기법이란 내부 코드는 보지 않고, 입력과 출력 결과만 보고 테스트하는 방식입니다.

### 인터페이스(Interface)의 유형 — 모듈이 서로 대화하는 방법들

통합 테스트에서 가장 중요한 것이 바로 **인터페이스(Interface, 모듈 간 연결 방식)** 입니다. 인터페이스의 유형은 크게 네 가지가 있습니다.

**① 매개변수 인터페이스 (Parameter Interface)**

하나의 모듈에서 다른 모듈로 데이터가 넘어갈 때, **변수(매개변수)를 통해서 값을 전달**하는 방식입니다. 마치 한 사람이 다른 사람에게 메모지에 숫자를 써서 전달하는 것과 같습니다. 우리가 흔히 함수를 호출하고 `return`으로 값을 돌려받는 것이 바로 매개변수 인터페이스입니다.

```java
// ✅ 매개변수 인터페이스 예시
public class OrderModule {
    
    // 주문 처리 후 결과를 매개변수를 통해 결제 모듈로 전달
    public int calculateOrderTotal(int price, int quantity) {
        int total = price * quantity;  // 주문 총액 계산
        return total;                  // 결과값을 반환 (매개변수 인터페이스)
    }
}

public class PaymentModule {
    
    // 주문 모듈에서 받은 총액을 매개변수로 받아서 결제 처리
    public void processPayment(int totalAmount) {
        // totalAmount가 바로 매개변수 인터페이스로 전달받은 값
        System.out.println("결제 처리 중: " + totalAmount + "원");
    }
}
```

**② 공유 메모리 인터페이스 (Shared Memory Interface)**

시스템들이 **물리적인 메모리를 공유**해서 데이터를 주고받는 방식입니다. 여러 모듈이 같은 메모리 공간에 데이터를 읽고 쓰면서 정보를 교환합니다. 마치 여러 사람이 하나의 화이트보드에 정보를 적고 읽는 것과 같습니다.

**③ 프로시저 인터페이스 (Procedure Interface)**

하나의 서브시스템(하위 시스템)이 다른 서브시스템을 **호출할 때** 발생하는 인터페이스입니다. 단순히 함수 하나를 호출하는 것이 아니라, 프로시저(처리 절차)의 집합 결과를 통째로 넘겨주는 방식입니다.

예를 들어, 주문 서브시스템이 완료되면 그 주문 내역 전체가 **마이페이지 서브시스템**에 그대로 전달되어야 합니다. 이때 주문 완료 처리 결과를 마이페이지로 한꺼번에 넘겨주는 것이 프로시저 인터페이스입니다.

**④ 메시지 전달 인터페이스 (Message Passing Interface)**

컴포넌트(구성 요소)가 다른 컴포넌트에게 **메시지를 전달해서 서비스를 요청**하는 방식입니다. 대표적인 예가 결제 시스템입니다.

```
[사용자] → 결제 버튼 클릭
    ↓
[우리 쇼핑몰 시스템] → 결제 정보 전송 (메시지 전달)
    ↓
[외부 결제 서비스 (예: 이니시스)] → 카드 승인 처리
    ↓
[외부 결제 서비스] → 승인 결과 반환 (메시지 전달)
    ↓
[우리 쇼핑몰 시스템] → "결제 완료" 화면 표시
```

:::info HTTP와 HTTPS — 메시지 전달의 보안 문제
메시지를 서로 전달할 때, **HTTP**(일반 통신)와 **HTTPS**(보안 통신, S = Secure) 두 가지 방식이 있습니다. HTTPS는 메시지를 암호화해서 제3자가 내용을 볼 수 없도록 보호합니다. 만약 우리 쇼핑몰은 HTTP를 사용하는데, 결제 시스템은 HTTPS만 허용한다면 통신 과정에서 오류가 발생하거나 데이터가 중복될 수 있습니다. 이런 부분을 통합 테스트에서 꼼꼼히 확인해야 합니다.
:::

### 인터페이스 오류의 종류

통합 테스트에서 발견되는 대표적인 오류 유형을 알아봅시다.

| 오류 유형 | 설명 | 예시 |
|---|---|---|
| **인터페이스 잘못 사용** | 다른 컴포넌트를 호출할 때 인터페이스를 잘못 사용한 경우 | 함수 파라미터 순서를 바꿔서 호출 |
| **인터페이스 오해** | 인터페이스 명세(설명서)를 잘못 이해하고 잘못 호출한 경우 | 원하는 데이터 대신 다른 데이터를 호출 |
| **타이밍 오류** | 호출하는 컴포넌트와 받는 컴포넌트의 처리 속도가 달라서 발생하는 오류 | 결제 완료 전에 결제 결과 화면을 먼저 확인하려다 오류 발생 |

**타이밍 오류**는 특히 주의가 필요합니다. 결제를 요청했는데 처리가 느려서 화면에 "결제 완료" 메시지가 바로 뜨지 않을 수 있습니다. 이때 성격이 급한 사용자가 결제 버튼을 한 번 더 누르면 **중복 결제**가 발생할 수 있습니다. 이런 시간 차(타이밍) 문제를 통합 테스트에서 반드시 점검해야 합니다.

### 통합 테스트의 종류 — 어떻게 합칠 것인가?

통합 테스트는 모듈들을 어떤 순서와 방식으로 합치느냐에 따라 크게 네 가지 방법으로 나뉩니다.

#### ① 빅뱅 통합 (Big Bang Integration) — 한꺼번에 다!

**빅뱅 방식**은 이름 그대로 모든 모듈을 **한꺼번에 전부 조합**해서 테스트하는 방식입니다. 비점진적(한 번에) 통합이라고도 합니다.

```
[A 모듈] ↘
[B 모듈] → 한꺼번에 통합 → 전체 테스트
[C 모듈] ↗
[D 모듈] ↗
```

- **장점**: 빠르게 전체를 볼 수 있음
- **단점**: 오류 찾기가 매우 어렵고, 혼란스럽고, 시간이 많이 걸리며, 결함 원인 발견이 어려움

:::danger 빅뱅 방식은 대부분 권장하지 않습니다
거대한 시스템을 한꺼번에 통합해서 테스트하면, 오류가 발생했을 때 어느 모듈에서 문제가 발생한 것인지 찾아내기가 매우 어렵습니다. 마치 수백 가지 재료를 한꺼번에 넣고 끓인 요리가 이상한 맛이 났을 때, 어떤 재료가 문제인지 알 수 없는 것과 같습니다.
:::

#### ② 하향식 통합 (Top-Down Integration) — 위에서 아래로

**하향식(탑다운, Top-Down)** 방식은 **주 프로그램(메인)부터 시작**해서, 그 메인이 호출하는 하위 모듈, 그 하위 모듈이 호출하는 더 하위 모듈 순서로 점진적으로 통합해 나가는 방식입니다.

```
      [메인 프로그램]
         ↓
    [주문 서브시스템]
       ↓         ↓
  [주문 처리]  [재고 확인]
       ↓
  [결제 처리]
```

**하향식 통합의 장점:**
- 실제 가동 환경과 유사한 방식으로 테스트 가능
- 주요 기능에 초점이 맞춰져 있어 기능 테스트에 용이
- 별도의 드라이버(구동기)가 필요 없음
- 오류 원인 찾기가 비교적 쉬움

**하향식 통합의 단점:**
- 병행 테스트(동시에 여러 모듈 테스트)가 어려움
- 최하위 모듈(맨 아래 단계)만 별도로 충분히 테스트하기 어려움
- 위에서 아래로만 흘러가므로, 하위 모듈의 결과를 상위로 다시 올려보낼 수 없음
- 하위 모듈로 데이터를 넘겨주는 **스텁(Stub)은 여전히 필요**

:::note 하향식에서 스텁이 필요한 이유
"하향식은 별도의 드라이버가 필요 없다"고 하지만, 메인에서 처리된 결과를 다음 단계로 넘겨줄 때 스텁은 여전히 필요합니다. 드라이버 자체(각 모듈을 독립적으로 실행시키는 구동기)는 필요 없지만, 데이터를 받아서 다음 단계로 전달해 주는 스텁은 하향식에서도 필요합니다.
:::

#### ③ 상향식 통합 (Bottom-Up Integration) — 아래서 위로

**상향식(바텀업, Bottom-Up)** 방식은 하향식과 반대로, **가장 하위 모듈부터 시작**해서 점점 상위 모듈로 통합해 올라가는 방식입니다. 이때는 각 하위 모듈을 실행시키는 **테스트 드라이버(Test Driver)** 가 필요합니다.

```
  [결제 처리]  [재고 확인]   ← 먼저 테스트
       ↑
    [주문 처리]              ← 다음 통합
       ↑
    [주문 서브시스템]         ← 그 다음 통합
       ↑
      [메인 프로그램]         ← 마지막 통합
```

**상향식 통합의 장점:**
- 개발 초기부터 병행 작업(여러 모듈 동시 테스트)이 가능
- 철저하게 모듈 단위로 테스트 가능 (하위 모듈을 충분히 검증 가능)

**상향식 통합의 단점:**
- 각 모듈을 실행시키는 **많은 양의 테스트 드라이버**가 필요
- 전체 시스템 구조(뼈대)가 갖춰지는 것은 마지막 단계
- 중요한 오류를 최후 단계에 가서야 발견할 가능성이 높음
- 아직 뼈대가 갖춰지지 않은 초기 시스템은 테스트 하기 어려움

#### ④ 연쇄식(샌드위치) 통합 (Thread/Sandwich Integration) — 중요한 것 먼저!

**연쇄식(쓰레드) 통합**은 하향식과 상향식을 합친 **샌드위치형 통합** 방식입니다. 모든 모듈을 순서대로 통합하는 것이 아니라, **특수하고 중요한 기능을 먼저** 테스트하고, **보조적인 기능은 나중에** 테스트합니다.

예를 들어 쇼핑몰 시스템에서:
- 🔴 **먼저 테스트**: 주문 모듈, 결제 모듈 (핵심 기능)
- 🟡 **나중에 테스트**: 상품 목록 표시, 회원 정보 관리 (보조 기능)

이 방식은 비즈니스적으로 가장 중요한 기능을 먼저 검증할 수 있다는 장점이 있습니다.

## 🖥️ 시스템 테스트와 인수 테스트 — 실제 환경에서 검증하자

### 시스템 테스트 (System Test) — 실제 서버에 올려서 확인

단위 테스트와 통합 테스트가 모두 끝났다면, 이제 **실제 서버(하드웨어) 위에 소프트웨어를 올려서 테스트**하는 단계입니다. 소프트웨어와 하드웨어를 함께 결합해서 전체 시스템의 기능과 성능을 점검합니다.

시스템 테스트에서 확인하는 내용들:

| 테스트 항목 | 설명 |
|---|---|
| **복구 테스트** | 시스템이 망가졌을 때 정상적으로 복구되는지 확인 |
| **보안 테스트 (안전시험)** | 해킹이나 외부 침입으로부터 시스템이 보호되는지 확인 |
| **스트레스 테스트** | 최대 접속 수 이상을 일부러 발생시켜 서버가 버티는지 확인 |
| **성능 테스트** | CPU, 메모리, 로그 관리 등 하드웨어 자원이 효율적으로 관리되는지 확인 |
| **부피 테스트** | 처리해야 할 데이터 양이 많을 때도 정상 동작하는지 확인 |
| **구성 테스트** | 다양한 하드웨어 기종에서 정상 동작하는지 확인 |
| **배치 테스트** | 실제 고객 환경(특정 프로그램 설치 여부 등)에서 정상 동작하는지 확인 |

:::warning 백신 프로그램과의 충돌 주의
배치 테스트에서 특히 주의해야 할 것이 백신(보안 프로그램)과의 충돌입니다. 특정 보안 프로그램이 설치된 환경에서는 소프트웨어가 제대로 동작하지 않을 수 있습니다. 실제 고객이 사용하는 환경을 최대한 재현해서 테스트해야 합니다.
:::

### 인수 테스트 (Acceptance Test) — 고객이 직접 확인

시스템 테스트까지 모두 통과했다면, 이제 **실제 의뢰 고객이 직접 검수**하는 단계입니다. 인수 테스트는 크게 **알파(Alpha) 테스트**와 **베타(Beta) 테스트** 두 가지로 나뉩니다.

**알파 테스트 (Alpha Test)**

> 선택된 사용자가 **개발자의 환경**에서, **개발자가 참석한 상태**로 진행하는 테스트

마치 자동차 공장에서 품질 검사원(고객 대표)이 공장 내부에 직접 와서 개발자와 함께 차량을 확인하는 것과 같습니다. 개발자가 옆에서 설명을 해줄 수 있고, 오류 발생 시 즉시 기록할 수 있습니다.

**베타 테스트 (Beta Test)**

> **고객의 환경에서**, **개발자 없이** 고객이 직접 사용해보고 결과를 개발자에게 보고하는 테스트

게임 서비스에서 "베타 버전 출시!"라고 하면, 수많은 일반 사용자들이 자기 집 컴퓨터에서 직접 다운받아 테스트해 보는 것이 바로 베타 테스트입니다. 개발자가 생각하지 못했던 다양한 환경에서 테스트가 이루어지기 때문에, **오류가 발견될 가능성이 가장 높은 단계**입니다.

| 구분 | 알파 테스트 | 베타 테스트 |
|---|---|---|
| **테스트 장소** | 개발자 환경 | 고객 환경 |
| **개발자 참여** | 참여 O | 참여 X |
| **참여 인원** | 선택된 소수 사용자 | 불특정 다수 사용자 |
| **오류 발견 가능성** | 보통 | 높음 |
| **현실성** | 통제된 환경 | 실제 현실 환경 |

### 설치 테스트 (Installation Test) — 실제 현장에 설치

베타 테스트까지 모두 완료됐다면, 이제 진짜 고객 현장에 가서 **실제 환경에 직접 설치**하고 교육하는 단계입니다. 설치 테스트에서는 다음과 같은 항목들을 점검합니다.

- 서버 사양이 요구 조건에 맞는지
- 파일, 로그 파일, 데이터베이스 구성이 올바른지
- CPU 성능, 메모리 성능, 저장장치(RAID) 구조가 맞는지
- 다른 소프트웨어와 연동할 때 문제가 없는지

**백투백 시험 (Back-to-Back Test)**

설치 단계에서 사용하는 특별한 테스트 방법입니다. **같은 시험을 두 개의 버전(예: 1.0과 2.0)에 동시에 적용**해서 결과를 비교합니다. 두 버전의 결과가 다르면, 둘 중 하나에 분명히 문제가 있다는 신호입니다. 마치 두 개의 저울로 같은 물건을 달았을 때 수치가 다르면 저울 하나가 고장났다는 것을 알 수 있는 것과 같습니다.

## 🔧 정적 분석과 동적 분석 — 자동화된 검사 도구

### 정적 분석기 (Static Analyzer) — 실행 없이 코드를 검사

**정적 분석**은 프로그램을 실제로 실행하지 않고, **원시 코드(소스 코드)를 직접 검사**하는 방식입니다. 마치 요리사가 실제로 요리를 하지 않고, 레시피만 읽어보면서 잘못된 부분을 찾아내는 것과 같습니다.

정적 분석이 특히 효과적인 경우:
- **무한 루프** (영원히 끝나지 않는 반복): 프로그램 실행 전에 코드 구조를 보고 감지
- **초기화되지 않은 변수 사용**: 값이 할당되지 않은 변수를 사용하는 코드 발견
- **실행 불가능한 코드**: 절대 실행될 수 없는 죽은 코드(Dead Code) 탐지

```java
// ❌ 정적 분석기가 잡아내는 잘못된 코드 예시

public class BadCodeExample {
    
    public void problematicMethod() {
        int uninitializedVariable; // 초기화되지 않은 변수 (⚠️ 정적 분석기가 경고)
        
        // 초기화되지 않은 변수를 바로 사용 — 오류 가능성 높음
        System.out.println(uninitializedVariable); // 컴파일 오류 또는 예측 불가 결과
        
        // 무한 루프 — 절대 끝나지 않음 (⚠️ 정적 분석기가 경고)
        while (true) {
            System.out.println("이 루프는 영원히 돌아갑니다...");
            // break 조건 없음
        }
        
        // 아래 코드는 절대 실행될 수 없음 — Dead Code (⚠️ 정적 분석기가 경고)
        System.out.println("이 코드는 절대 실행되지 않습니다");
    }
}
```

```java
// ✅ 정적 분석기의 경고를 반영한 올바른 코드

public class GoodCodeExample {
    
    public void correctMethod() {
        int initializedVariable = 0; // 변수를 반드시 초기화
        
        // 초기화된 변수를 안전하게 사용
        System.out.println("변수 값: " + initializedVariable);
        
        // 종료 조건이 있는 안전한 루프
        int count = 0;
        while (count < 10) {  // 명확한 종료 조건
            System.out.println("현재 카운트: " + count);
            count++;           // 루프 변수를 반드시 변경
        }
        
        // 루프 종료 후 실행 가능한 코드
        System.out.println("루프가 정상적으로 완료되었습니다");
    }
}
```

정적 분석 도구의 종류:

| 도구 유형 | 역할 |
|---|---|
| **코드 분석 도구** | 문법 오류, 오류 발생 가능성이 있는 코드 탐지 |
| **구조 검사 도구** | 원시 코드를 그래프로 만들어 논리 흐름과 구조적 결함 확인 |
| **데이터 분석 도구** | 데이터 구조 초기화 여부, 잘못된 데이터 사용, 중복 데이터 사용 확인 |

정적 분석에서 수행하는 주요 분석 항목:
- **제어 흐름 분석**: 코드의 실행 흐름이 올바른지 분석
- **데이터 사용 분석**: 변수와 데이터가 어떻게 사용되는지 분석
- **인터페이스 분석**: 모듈 간 연결이 제대로 되어 있는지 분석
- **경로 분석**: 데이터가 이동하는 경로와 그 과정의 명령어들을 분석

### 동적 분석기 (Dynamic Analyzer) — 실행하면서 검사

**동적 분석**은 정적 분석과 반대로, **프로그램을 실제로 실행하면서** 동작과 반응을 추적하는 방식입니다. 마치 요리를 실제로 해보면서 중간중간 맛을 보고 온도를 재는 것과 같습니다.

동적 분석기는 다음과 같은 방식으로 작동합니다:
- 원시 코드 중간중간에 **검사 코드(인스트루먼트 코드)를 삽입**
- 프로그램 실행 중 특정 시점의 메모리 상태를 **스냅샷**으로 덤프(복사)해서 확인
- 이벤트(사건) 발생 시 그 상태를 실시간으로 파악

```java
// ✅ 동적 분석을 위한 인스트루먼트 코드 삽입 예시

public class OrderProcessWithInstrumentation {
    
    public int processOrder(int productId, int quantity, int userId) {
        
        // [동적 분석 코드] 함수 진입 시 상태 기록
        System.out.println("[DEBUG] processOrder 시작 - productId: " + productId 
                          + ", quantity: " + quantity + ", userId: " + userId);
        
        // 재고 확인 로직
        int stockCount = checkStock(productId);
        
        // [동적 분석 코드] 재고 확인 후 상태 기록
        System.out.println("[DEBUG] 재고 확인 완료 - 현재 재고: " + stockCount);
        
        if (stockCount < quantity) {
            // [동적 분석 코드] 재고 부족 상황 기록
            System.out.println("[DEBUG] 재고 부족! 주문 실패");
            return -1; // 재고 부족으로 실패
        }
        
        // 주문 금액 계산
        int totalPrice = calculatePrice(productId, quantity);
        
        // [동적 분석 코드] 가격 계산 후 상태 기록
        System.out.println("[DEBUG] 주문 금액 계산 완료 - 총 금액: " + totalPrice + "원");
        
        return totalPrice; // 주문 총액 반환
    }
}
```

## 🐛 디버깅(Debugging) — 발견된 오류를 수정하는 방법

### 디버깅이란?

**디버깅(Debugging)** 은 소프트웨어에서 오류(버그)를 발견하고 수정하는 작업입니다. 개발자가 가장 많이 하는 작업 중 하나입니다. 사용자가 "오류가 있어요"라고 신고하면, 개발자는 그 오류를 분석해서 원인을 찾고 수정합니다. 이 전체 과정이 **디버깅**입니다.

:::tip 디버깅 과정 요약
1. **에러 위치 파악**: 디버깅 툴을 활용해서 오류가 발생한 위치를 찾기
2. **설계 에러 수정**: 잘못된 설계에서 비롯된 오류 수정
3. **코드 에러 수정**: 실제 코드의 잘못된 부분 수정
4. **재시작 및 배포**: 수정된 프로그램을 다시 서버에 올려 최신화
:::

### 디버깅 방법의 종류

**① 맹목적 디버깅**: 문제 원인을 메모리 덤프에서 찾는 방식. 효율성이 매우 떨어져 권장하지 않습니다.

**② 출력문 삽입 방식**: 코드의 임의 위치에 출력문을 삽입해서 중간 결과를 확인하는 방식. 간단하지만 체계적이지 않을 수 있습니다.

**③ 도구 활용 방식**: 디버깅 툴(IDE 내장 디버거 등)을 사용해서 실행을 추적하고, 특정 시점에서 실행을 멈추고(브레이크포인트) 상태를 확인하는 방식. 가장 효율적입니다.

**디버깅 전략의 종류:**

| 전략 | 설명 |
|---|---|
| **귀납적 방식** | 관련 정보를 입수해서 문제 원인을 하나씩 따져가며 사실을 찾아가는 방식 |
| **연역적 방식** | 모든 가능한 원인을 먼저 나열하고, 가정을 수립해서 원인을 규명하는 방식 |
| **역행 조사 (백트래킹)** | 원시 코드를 거꾸로 추적해서 오류 원인을 찾는 방식 |

### 디버깅 시 반드시 지켜야 할 원칙들

오랜 경험을 바탕으로 정리된 디버깅 원칙들입니다. 실제 현업에서 매우 유용합니다.

**1. 혼자 해결할 수 없으면 도움을 청해라**

자신의 힘으로 도저히 찾을 수 없으면 그만두고, 전문가의 도움을 요청해야 합니다. 끝까지 혼자만 하려다가 시간을 낭비하는 것은 팀 전체에 손해입니다.

**2. 디버깅 도구는 만능이 아니다**

디버깅 툴은 차선책일 뿐, 처음부터 코드를 잘 작성하는 것이 최선입니다. 전체 시스템 구조를 정확히 이해하고 있는 것이 무엇보다 중요합니다.

**3. 무작위 실험은 최후의 수단이다**

"이렇게 해볼까, 저렇게 해볼까" 식의 무작위 실험은 시간만 낭비할 수 있습니다. 분석을 먼저 충분히 한 후에 시도하세요.

**4. 하나의 결함 옆에는 또 다른 결함이 있다**

하나의 오류를 발견했다면, 그와 연관된 또 다른 오류가 있을 가능성이 높습니다. "하나 고쳤으니 이제 괜찮겠지"라고 방심하면 안 됩니다.

**5. 교정이 정확할 확률은 100%가 아니다**

오류를 고쳤다고 해서 완벽히 해결된 것이 아닐 수 있습니다. **프로그램의 규모가 클수록, 수정 후에도 완전히 해결되지 않을 확률이 높아집니다.** 규모가 작은 프로그램은 수정 정확도가 높지만, 거대한 시스템은 수정 후에도 50%밖에 해결되지 않을 수 있습니다.

**6. 결함의 교정이 또 다른 결함을 만들 수 있다**

A 모듈의 오류를 수정했는데, 며칠 후 전혀 다른 부분에서 새로운 오류가 발생하는 경우가 있습니다. 원인을 추적해 보면 A를 수정한 것이 영향을 줬다는 사실을 발견하게 됩니다. 이런 경우에는 부분 수정이 아닌 **전체 시스템 개선**이 필요해집니다.

:::danger 혼자 모든 것을 해결하려 하지 마세요
디버깅 원칙 중 가장 중요한 것 중 하나입니다. 막힐 때 전문가에게 도움을 요청하는 것은 능력이 없는 것이 아니라, 오히려 팀워크와 효율성을 아는 현명한 개발자의 자세입니다.
:::

## ⚠️ 주의사항 및 실전 팁 — 테스트할 때 꼭 알아야 할 것들

### 테스트 설계 전에 반드시 준비해야 할 것들

**설계 문서를 반드시 확인하세요.** 단위 테스트는 **화면 정의서, 기능 정의서** 등 상세 설계도를 기초로 수행합니다. 설계 문서 없이 테스트를 시작하면 무엇이 올바른 결과인지 판단할 기준이 없습니다. 마치 정답지 없이 채점하는 것과 같습니다.

**스텁(Stub)을 미리 준비하세요.** 단위 테스트 시작 전에 스텁이 없으면 테스트 자체가 불가능합니다. 모듈 간 의존 관계를 먼저 파악하고, 필요한 스텁을 미리 설계해 두는 것이 중요합니다.

### 통합 방식 선택 시 고려사항

```
프로젝트 특성에 따른 통합 방식 선택 가이드:

✅ 시스템이 크고 복잡한 경우 → 빅뱅 방식 피하기
✅ 핵심 기능을 일찍 검증하고 싶은 경우 → 하향식(Top-Down)
✅ 하위 모듈을 철저히 검증하고 싶은 경우 → 상향식(Bottom-Up)
✅ 중요 기능과 보조 기능이 명확히 구분되는 경우 → 연쇄식(Sandwich)
```

### HTTP vs HTTPS 인터페이스 주의

외부 서비스(결제, SMS 등)와 연동할 때 **HTTP/HTTPS 프로토콜 불일치** 문제가 자주 발생합니다. 우리 시스템이 HTTP를 사용하는데, 외부 결제 서비스가 HTTPS만 허용한다면 데이터 전송이 실패하거나 중복될 수 있습니다. 통합 테스트 전에 반드시 확인해야 합니다.

### 타이밍 오류 대응 방법

결제나 외부 API 호출처럼 시간이 걸리는 작업에서는 **로딩 표시(스피너)**와 **중복 요청 방지 로직**을 반드시 구현해야 합니다. 결제 버튼을 한 번 누른 후 처리가 완료될 때까지 버튼을 비활성화하는 것이 대표적인 방법입니다.

```java
// ✅ 중복 결제 방지를 위한 버튼 비활성화 예시 (개념적 코드)

public class PaymentController {
    
    private boolean isProcessing = false; // 처리 중 여부 플래그
    
    public void onPaymentButtonClick() {
        
        // 이미 처리 중이면 재요청 무시 (타이밍 오류 방지)
        if (isProcessing) {
            System.out.println("결제 처리 중입니다. 잠시 기다려 주세요.");
            return; // 중복 요청 차단
        }
        
        isProcessing = true; // 처리 시작 플래그 설정
        // (UI에서는 버튼 비활성화 처리)
        
        try {
            // 실제 결제 처리 로직 실행
            boolean result = processPayment();
            System.out.println("결제 완료: " + (result ? "성공" : "실패"));
        } finally {
            isProcessing = false; // 처리 완료 후 플래그 해제
            // (UI에서는 버튼 다시 활성화)
        }
    }
}
```

:::tip 디버깅 효율을 높이는 팁
- **IDE 내장 디버거를 적극 활용하세요**: 브레이크포인트(실행 중단점) 설정, 변수 값 실시간 확인, 스텝 실행(한 줄씩 실행) 기능을 활용하면 오류 원인을 빠르게 찾을 수 있습니다.
- **로그(Log)를 꼼꼼히 남기세요**: 운영 환경에서는 직접 디버거를 붙이기 어렵습니다. 충분한 로그를 남겨두면 오류 발생 시 원인 추적이 훨씬 쉬워집니다.
- **테스트 케이스를 문서화하세요**: 어떤 입력으로 테스트했고, 어떤 결과가 나왔는지 기록해두면 나중에 같은 오류가 재발했을 때 빠르게 대응할 수 있습니다.
:::

## 📝 핵심 정리

이 문서에서 배운 핵심 개념들을 정리합니다.

- **단위 테스트(Unit Test)**: 프로그램의 기본 단위인 모듈을 개별적으로 검증하는 테스트로, 화이트박스 기법을 사용하고 코딩 단계와 병행 수행
- **모듈(Module)**: 쇼핑몰의 주문, 결제, 장바구니처럼 시스템을 구성하는 설계의 최소 단위
- **드라이버(Driver)**: 실제 모듈을 실행·구동하는 프로그램
- **스텁(Stub)**: 아직 개발되지 않은 또는 연결되지 않은 다른 모듈을 흉내 내는 가짜 모듈로, 단위 테스트에 필수
- **통합 테스트(Integration Test)**: 여러 모듈을 결합해 인터페이스가 올바르게 작동하는지 검증하는 테스트로, 블랙박스 기법 사용
- **인터페이스 유형**: 매개변수, 공유 메모리, 프로시저, 메시지 전달 네 가지 방식으로 모듈 간 데이터를 주고받음
- **인터페이스 오류**: 잘못된 사용, 잘못된 이해, 타이밍 오류 세 가지 유형이 존재
- **빅뱅 통합**: 모든 모듈을 한꺼번에 통합하는 비점진적 방식으로, 오류 원인 찾기가 어려워 잘 사용하지 않음
- **하향식 통합(Top-Down)**: 메인에서 하위 모듈로 내려가며 통합, 드라이버 불필요, 스텁 필요
- **상향식 통합(Bottom-Up)**: 하위 모듈에서 상위로 올라가며 통합, 드라이버 필요, 병행 테스트 가능
- **연쇄식(샌드위치) 통합**: 중요한 기능 먼저, 보조 기능 나중에 통합하는 혼합 방식
- **시스템 테스트**: 실제 서버에 소프트웨어를 올려 하드웨어와 통합해서 전체 기능·성능을 검증
- **알파 테스트**: 개발자 환경에서 개발자가 참석한 상태로 선택된 사용자가 테스트
- **베타 테스트**: 고객 환경에서 개발자 없이 불특정 다수가 테스트, 오류 발견 가능성이 가장 높음
- **설치 테스트**: 실제 고객 현장에 설치하면서 서버 사양, 연동 등을 최종 확인
- **백투백 시험**: 두 버전을 같은 조건으로 테스트해 결과를 비교하는 방식
- **정적 분석**: 프로그램을 실행하지 않고 코드를 직접 검사해 무한 루프, 미초기화 변수 등을 탐지
- **동적 분석**: 프로그램을 실행하면서 이벤트 상태와 메모리를 실시간으로 추적
- **디버깅(Debugging)**: 발견된 오류의 원인을 찾아 수정하는 작업으로, 귀납적·연역적·역행조사 방식이 있음
- **디버깅 핵심 원칙**: 혼자 해결하려 하지 말고, 하나의 결함 수정이 또 다른 결함을 만들 수 있음을 항상 인식
