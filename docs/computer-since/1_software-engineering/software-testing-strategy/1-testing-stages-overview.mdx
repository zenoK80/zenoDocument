# 🧪 소프트웨어 테스트 단계 완벽 가이드

소프트웨어를 개발할 때, 코드를 작성하는 것만큼 중요한 것이 바로 **테스트**입니다. 하지만 "테스트를 해야 한다"는 것은 알아도, 어떤 순서로, 어떤 방법으로 해야 하는지 막막하게 느껴지는 분들이 많습니다. 이 문서에서는 소프트웨어 테스트를 단계별로 체계적으로 설명합니다. 처음 접하는 분도 이해할 수 있도록 비유와 예시를 풍부하게 사용하겠습니다.

---

## 🗺️ 테스트 단계, 왜 순서가 중요한가?

### 테스트를 한꺼번에 하면 안 되나요?

많은 입문자분들이 이런 질문을 합니다. "그냥 개발 다 끝내고 한 번에 테스트하면 안 되나요?" 충분히 이해할 수 있는 질문입니다. 하지만 이 방식은 마치 **자동차를 완전히 조립한 뒤에 부품이 불량인지 확인하는 것**과 같습니다. 조립이 다 끝난 후에 엔진 부품이 잘못되었다는 것을 발견하면, 전부 다 분해해야 하죠.

소프트웨어도 마찬가지입니다. 수십만 줄의 코드가 합쳐진 상태에서 오류를 찾으려면, 그 오류가 어디서 발생했는지 특정하기가 매우 어렵습니다. 그래서 소프트웨어 공학에서는 테스트를 **단계별로 나누어** 수행하는 전략을 사용합니다.

> **소프트웨어 테스트의 단계**: 단위 테스트 → 통합 테스트 → 시스템 테스트 → 인수 테스트 → 설치 테스트

이 다섯 단계는 각각 역할이 다르며, 앞 단계가 완료되어야 다음 단계로 넘어갈 수 있습니다. 마치 건물을 지을 때 기초 공사가 끝나야 벽을 세우고, 벽이 완성되어야 지붕을 올리는 것처럼, 테스트도 작은 단위에서 큰 단위로 점진적으로 확대되어 갑니다.

### 객체지향 프로그래밍(OOP)과 테스트의 관계

소프트웨어 공학에서 이야기하는 테스트 단계는 특히 **객체지향 프로그래밍(OOP, Object-Oriented Programming)** 환경을 전제로 합니다. Java, Python, C++ 같은 언어들이 대표적인 OOP 언어입니다.

OOP에서는 코드를 **모듈(Module)** 이라는 단위로 나누어 개발합니다. 예를 들어 쇼핑몰을 만든다면 다음과 같이 나뉩니다.

| 모듈 이름 | 역할 |
|---|---|
| 회원 모듈 | 회원 가입, 로그인, 정보 수정 |
| 주문 모듈 | 상품 주문, 주문 내역 조회 |
| 결제 모듈 | 카드 결제, 환불 처리 |
| 장바구니 모듈 | 상품 담기, 수량 변경 |
| 재고 모듈 | 재고 확인, 재고 수량 업데이트 |

이렇게 독립된 모듈로 나누어 개발하기 때문에, 테스트도 **모듈 단위**로 나누어 수행하는 것이 자연스럽습니다. 만약 코드가 분리 없이 하나로 쭉 이어진 순차적 프로그래밍 방식이라면 단위 테스트나 통합 테스트의 의미가 크지 않겠지만, 모듈 기반의 개발에서는 이 단계적 테스트가 매우 중요합니다.

## 🔬 1단계: 단위 테스트 (Unit Test)

### 단위 테스트란 무엇인가?

**단위 테스트(Unit Test)** 는 프로그램에서 가장 작은 단위인 **모듈 하나하나를 개별적으로 검증**하는 테스트입니다. 마치 공장에서 자동차를 만들 때, 완성된 자동차를 테스트하기 전에 먼저 **타이어, 엔진, 브레이크 같은 부품 하나씩을 검사**하는 것과 같습니다.

단위 테스트는 **상세 설계서(화면 정의서, 기능 정의서)** 를 기준으로 수행됩니다. 설계 문서에서 "회원 가입 기능은 이메일 형식을 검증해야 한다"고 명시되어 있다면, 회원 모듈이 실제로 이메일 형식을 올바르게 검증하는지 테스트합니다.

:::info 단위 테스트의 기법
단위 테스트는 **화이트박스(White-box) 기법**을 사용합니다. 화이트박스란 코드의 내부를 직접 들여다보면서 테스트하는 방식입니다. 마치 유리로 된 상자 안을 들여다보는 것처럼, 소스 코드를 직접 보면서 각 줄이 올바르게 동작하는지 확인합니다.
:::

### 단위 테스트의 특징

단위 테스트는 **코딩 단계와 병행하여** 수행됩니다. 즉, 개발자가 코드를 작성하면서 동시에 테스트를 진행합니다. 이것이 중요한 이유는, 나중에 모든 코드가 완성된 후에 오류를 찾으면 수정 비용이 엄청나게 커지기 때문입니다.

또한 모듈은 서로 **독립적**이기 때문에 **병렬로 테스트**가 가능합니다. 주문 모듈 테스트가 끝나야 결제 모듈 테스트를 시작해야 하는 것이 아닙니다. 주문 팀과 결제 팀이 동시에 각자의 모듈을 테스트할 수 있습니다. 이것이 단위 테스트의 큰 장점 중 하나입니다.

### 드라이버(Driver)와 스텁(Stub) — 단위 테스트의 핵심 도구

여기서 중요한 개념이 등장합니다. 단위 테스트를 할 때, 모듈은 혼자서 동작하지 않는 경우가 많습니다. 예를 들어 A 모듈, B 모듈, C 모듈이 순서대로 연결되어 동작하는 시스템이 있다고 가정해봅시다.

```
[A 모듈] → [B 모듈] → [C 모듈] → 최종 결과 출력
```

이 상황에서 A 모듈만 단독으로 테스트하려면 문제가 생깁니다.

- **입력 데이터 문제**: A 모듈은 앞에서 데이터를 받아야 하는데, 아직 앞 단계가 없습니다.
- **출력 결과 확인 문제**: A 모듈의 처리 결과는 B를 거쳐 C까지 가야 최종 결과가 나오는데, 혼자서는 결과를 확인할 수 없습니다.

이 문제를 해결하기 위해 두 가지 가짜 도구를 사용합니다.

| 용어 | 역할 | 비유 |
|---|---|---|
| **드라이버(Driver)** | 실제 모듈을 실행시키고 입력 데이터를 제공하며 결과를 출력해주는 실제 모듈 | 자동차의 운전대 — 모듈을 실제로 움직이게 해주는 것 |
| **스텁(Stub)** | 아직 완성되지 않은 다른 모듈을 흉내 내는 가짜 모듈 | 영화 촬영장의 대역 배우 — 진짜는 아니지만 역할을 대신함 |

:::tip 스텁(Stub)을 꼭 알아두세요
실무에서 "스텁 만들어서 테스트하자"는 말을 들었을 때 이해할 수 있어야 합니다. 스텁은 **입출력만 흉내 내는 최소한의 가짜 모듈**입니다. 실제 로직은 없지만, 특정 입력을 받으면 미리 정해진 가짜 결과를 돌려주는 역할을 합니다. 스텁이 없으면 단위 테스트를 제대로 수행할 수 없습니다.
:::

예를 들어 결제 모듈(B)을 단독으로 테스트한다고 해봅시다. 결제 모듈은 주문 모듈(A)로부터 주문 금액 데이터를 받아야 합니다. 하지만 주문 모듈이 아직 테스트 중이라면, 주문 모듈을 흉내 내는 **스텁**을 만들어서 "주문 금액: 50,000원"이라는 가짜 데이터를 결제 모듈에 넘겨줍니다. 그러면 결제 모듈은 실제처럼 동작하면서 테스트할 수 있습니다.

```java
// ✅ 올바른 예시: 스텁(Stub)을 활용한 단위 테스트
// 주문 모듈이 아직 완성되지 않았을 때, 가짜 주문 모듈(스텁)을 만들어서 테스트

// 가짜 주문 모듈 (Stub) - 실제 주문 처리는 하지 않고, 고정된 값만 반환
public class OrderModuleStub {
    // 실제 주문 모듈인 척하지만, 항상 50000원을 반환하는 가짜 모듈
    public int getOrderAmount() {
        return 50000; // 테스트용 고정 주문 금액
    }
}

// 테스트 대상: 결제 모듈
public class PaymentModule {
    // 주문 금액을 받아서 결제 처리하는 실제 모듈
    public boolean processPayment(int amount) {
        if (amount <= 0) {
            return false; // 금액이 0 이하이면 결제 실패
        }
        // 실제 결제 로직 처리
        System.out.println(amount + "원 결제가 완료되었습니다.");
        return true; // 결제 성공
    }
}

// 단위 테스트 실행
public class PaymentModuleTest {
    public static void main(String[] args) {
        // 스텁에서 가짜 주문 금액을 가져옴
        OrderModuleStub stub = new OrderModuleStub();
        int orderAmount = stub.getOrderAmount(); // 50000원 반환

        // 결제 모듈을 실제처럼 테스트
        PaymentModule payment = new PaymentModule();
        boolean result = payment.processPayment(orderAmount);

        // 테스트 결과 확인
        System.out.println("결제 테스트 결과: " + (result ? "성공" : "실패"));
    }
}
```

```java
// ❌ 잘못된 예시: 스텁 없이 단위 테스트를 시도하는 경우
// 주문 모듈이 없으면 결제 모듈에 넘겨줄 데이터가 없어서 테스트 자체가 불가능

public class PaymentModuleTest {
    public static void main(String[] args) {
        // 주문 모듈이 없으니 orderAmount를 어디서 가져올지 모름
        // int orderAmount = ???; // 데이터 출처 없음 → 테스트 불가
        
        PaymentModule payment = new PaymentModule();
        // 테스트할 수 없음 — 입력 데이터가 없기 때문
    }
}
```

위 올바른 예시에서 핵심을 정리하면:
- `OrderModuleStub` 클래스가 **스텁** 역할을 합니다. 진짜 주문 모듈은 아니지만, 테스트에 필요한 가짜 데이터(50,000원)를 제공합니다.
- `PaymentModule` 클래스가 **테스트 대상**입니다.
- `PaymentModuleTest` 클래스가 **드라이버** 역할을 하며, 전체 테스트를 실행합니다.

## 🔗 2단계: 통합 테스트 (Integration Test)

### 통합 테스트란 무엇인가?

단위 테스트가 끝나면 각 모듈이 제대로 동작한다는 것은 확인했습니다. 하지만 **개별 부품이 정상이라도 조립했을 때 제대로 맞물리지 않을 수 있습니다.** 마치 각각의 레고 블록은 완벽한데, 서로 다른 세트의 블록이라 결합이 안 되는 경우처럼요.

**통합 테스트(Integration Test)** 는 여러 모듈을 연결했을 때 **모듈 간의 인터페이스(연결 방식)** 가 올바르게 동작하는지를 검증합니다. A 모듈의 결과가 B 모듈로 올바르게 전달되는지, B 모듈에서 처리된 데이터가 C 모듈로 이상 없이 넘어가는지를 확인합니다.

:::info 통합 테스트의 기법
통합 테스트는 **블랙박스(Black-box) 기법**을 사용합니다. 블랙박스란 내부 코드는 보지 않고, **입력과 출력만 보고** 테스트하는 방식입니다. 단위 테스트가 끝난 뒤이므로 코드 내부는 이미 검증된 상태이기 때문입니다.
:::

### 인터페이스 유형 이해하기

통합 테스트에서 가장 중요한 것은 **인터페이스(Interface, 모듈 간의 연결 방식)** 입니다. 인터페이스의 유형은 크게 네 가지입니다.

**① 매개변수 인터페이스 (Parameter Interface)**

한 모듈에서 다른 모듈로 데이터를 넘길 때 **변수(매개변수)** 를 통해 전달하는 방식입니다. 함수를 호출할 때 값을 넘기고, 결과를 리턴으로 받는 방식이 바로 매개변수 인터페이스입니다.

```java
// ✅ 매개변수 인터페이스 예시
// 주문 모듈에서 결제 모듈로 주문 금액을 매개변수로 전달

public class IntegrationExample {
    // 주문 모듈: 주문을 처리하고 금액을 반환
    public int processOrder(String productName, int quantity, int price) {
        int totalAmount = quantity * price; // 총 금액 계산
        System.out.println(productName + " " + quantity + "개 주문 완료");
        return totalAmount; // 결제 모듈로 넘겨줄 금액 반환
    }

    // 결제 모듈: 주문 모듈에서 받은 금액으로 결제 처리
    public void processPayment(int amount) {
        // 매개변수(amount)로 데이터를 전달받음
        System.out.println("결제 금액: " + amount + "원 처리 중...");
    }

    public static void main(String[] args) {
        IntegrationExample system = new IntegrationExample();
        // 주문 모듈의 결과(매개변수)가 결제 모듈로 전달됨
        int orderAmount = system.processOrder("노트북", 1, 1500000);
        system.processPayment(orderAmount); // 1500000원이 매개변수로 전달
    }
}
```

**② 공유 메모리 인터페이스 (Shared Memory Interface)**

여러 모듈이 **같은 메모리 공간을 함께 사용**하여 데이터를 주고받는 방식입니다. 마치 사무실의 공용 게시판에 누군가 메모를 붙여 두면, 다른 사람이 와서 읽어가는 것과 같습니다. 어떤 시스템들이 물리적인 메모리를 어떻게 공유해서 데이터를 가져오는지를 테스트합니다.

**③ 프로시저 인터페이스 (Procedure Interface)**

하나의 **서브시스템이 다른 서브시스템을 호출**할 때 사용되는 방식입니다. 예를 들어, 주문 서브시스템이 완료되면 그 결과가 마이페이지 서브시스템에 그대로 반영되어야 합니다. 단순한 변수 값 전달이 아니라, 하나의 서브시스템 전체가 다른 서브시스템에 영향을 주는 방식입니다.

**④ 메시지 전달 인터페이스 (Message Passing Interface)**

컴포넌트(구성 요소)가 다른 컴포넌트에게 **메시지를 전송**하여 서비스를 요청하는 방식입니다. 실생활의 예를 들면, 쇼핑몰에서 결제 버튼을 누르면 우리 시스템이 카드 결제 서비스 업체(예: 이니시스)에 결제 요청 메시지를 보내고, 승인 결과 메시지를 다시 받아오는 것이 메시지 전달 방식입니다.

:::warning HTTP와 HTTPS 주의
메시지 전달 인터페이스 테스트 시 중요한 것이 있습니다. 우리 시스템은 `http`를 사용하는데, 결제 서비스는 `https`(보안 강화 버전)만 허용한다면 데이터 통신 과정에서 오류가 발생할 수 있습니다. `https`는 메시지를 암호화하여 전달하기 때문에, 보안 정책이 다른 시스템 간의 통신은 반드시 테스트해야 합니다.
:::

### 인터페이스 오류의 종류

통합 테스트에서 발견되는 대표적인 오류들을 살펴봅시다.

| 오류 유형 | 설명 | 예시 |
|---|---|---|
| **인터페이스 잘못 사용** | 다른 컴포넌트를 호출할 때 인터페이스를 잘못 사용함 | 결제 API를 호출할 때 필수 파라미터를 빠뜨림 |
| **인터페이스 오해** | 인터페이스 명세를 잘못 이해하고 호출함 | 주문 금액을 원 단위로 보내야 하는데 천 단위로 보냄 |
| **타이밍 오류** | 호출하는 쪽과 받는 쪽의 처리 속도가 달라 발생하는 오류 | 결제 요청 후 응답이 느려서 같은 결제가 두 번 실행됨 |

타이밍 오류는 특히 주의해야 합니다. 결제 버튼을 눌렀을 때 응답이 느리다고 해서 한 번 더 누르면 **이중 결제**가 발생할 수 있습니다. 이런 시간 차이로 인한 오류도 통합 테스트에서 반드시 점검해야 합니다.

### 통합 방식의 종류

통합 테스트를 어떤 방식으로 수행할지에 따라 네 가지 방법이 있습니다.

#### 🌋 빅뱅(Big Bang) 방식 — 비점진적 통합

모든 모듈을 **한꺼번에** 결합하여 한 번에 테스트하는 방식입니다. 우주가 한 번의 대폭발로 탄생한 빅뱅처럼, 모든 것을 동시에 합쳐버립니다.

- **단점**: 오류를 찾기가 매우 어렵고, 시간이 오래 걸립니다. 오류의 원인이 어느 모듈에 있는지 특정하기 어렵습니다. 시스템이 크면 클수록 더욱 혼란스럽습니다.
- **결론**: 현실적으로 거의 사용하지 않습니다.

#### ⬇️ 하향식(Top-Down) 통합 방식

**메인 프로그램(최상위)부터 시작**하여 아래 단계의 모듈을 점진적으로 추가해가며 테스트하는 방식입니다.

```
[메인 프로그램]
    ↓
[서브 모듈 A]
    ↓
[서브 모듈 B]
    ↓
[서브 모듈 C]
```

- **장점**
  - 실제 가동 환경과 유사하여 주요 기능 테스트에 용이합니다.
  - 오류의 원인을 찾아내기 쉽습니다. (메인부터 내려가면서 찾을 수 있으므로)
  - 별도의 드라이버가 불필요합니다.
  - 에러는 새로 추가된 모듈이나 인터페이스 범위 내로 한정됩니다.

- **단점**
  - 병렬 테스트가 어렵습니다. 위에서 아래로만 내려가기 때문입니다.
  - 하위 모듈로의 데이터 전달을 위해 스텁이 필요합니다.
  - 최하위 모듈만을 충분히 독립적으로 테스트하기 어렵습니다.

:::note 하향식에서 스텁이 필요한 이유
"하향식은 드라이버가 불필요하다"고 했지만, **스텁은 필요합니다.** 메인에서 서브 모듈로 데이터를 넘길 때, 아직 완성되지 않은 하위 모듈 대신 스텁이 그 역할을 대신합니다. 다만, 전체 흐름을 운전하는 드라이버는 메인 프로그램 자체가 그 역할을 하므로 별도의 드라이버는 필요 없습니다.
:::

#### ⬆️ 상향식(Bottom-Up) 통합 방식

**최하위 모듈부터 시작**하여 위 단계로 점진적으로 통합해가며 테스트하는 방식입니다. 하향식과 반대 방향입니다.

- **장점**
  - 개발 초기부터 병렬 작업이 가능합니다.
  - 철저한 모듈 단위 테스트가 가능합니다.

- **단점**
  - 각 모듈을 실행하기 위한 **테스트 드라이버**가 많이 필요합니다.
  - 최상위 통합은 가장 마지막에 이루어지므로, 중요한 오류를 늦게 발견할 수 있습니다.
  - 시스템의 전체 뼈대가 갖춰지는 것이 마지막이므로, 초기에 전체 시스템을 사용해볼 수 없습니다.

#### 🥪 연쇄식(Thread/Sandwich) 통합 방식

하향식과 상향식을 혼합한 **샌드위치 방식**입니다. 무조건 위에서 아래로 혹은 아래에서 위로가 아니라, **중요한 기능을 먼저 테스트하고 보조적인 기능은 나중에 테스트**합니다.

예를 들어, 쇼핑몰에서 주문과 결제가 가장 중요한 기능이라면 이 두 모듈을 먼저 통합 테스트하고, 상품 목록 화면이나 회원 관리 같은 보조적인 기능은 나중에 테스트합니다.

---

## 🖥️ 3단계: 시스템 테스트 (System Test)

### 시스템 테스트란 무엇인가?

단위 테스트와 통합 테스트가 완료되면, 이제 소프트웨어를 **실제 서버에 올려서** 테스트합니다. 이것이 **시스템 테스트(System Test)** 입니다. 소프트웨어와 하드웨어를 결합하여 전체 시스템이 제대로 동작하는지 확인합니다.

마치 자동차 부품 조립이 끝난 후, 실제 도로에서 주행 테스트를 하는 것과 같습니다. 각 부품은 완벽했지만, 실제 도로에서 달려봐야 알 수 있는 것들이 있기 때문입니다.

### 시스템 테스트의 종류

시스템 테스트는 다양한 관점에서 수행됩니다.

| 테스트 유형 | 내용 |
|---|---|
| **복구 시험** | 소프트웨어가 다운되었을 때 정상적으로 복구되는지 확인 |
| **안전 시험** | 해킹, 불법 접근으로부터 시스템이 보호되는지 확인 |
| **강도 시험 (스트레스 테스트)** | 최대 접속자 수 이상을 일부러 만들어 서버가 견디는지 확인 |
| **성능 시험** | CPU, 메모리, 로그 등이 효율적으로 관리되는지 확인 |
| **부피 시험** | 대용량 데이터를 처리할 때 문제가 없는지 확인 |
| **구성 시험** | 다양한 하드웨어 기종에서 정상 동작하는지 확인 |
| **배치 시험** | 특정 소프트웨어(예: 백신)가 설치된 환경에서 충돌이 없는지 확인 |
| **데이터 민감성 시험** | 유효한 입력 데이터가 불안정하거나 부적절한 처리를 일으키지 않는지 확인 |
| **기능 시험** | 사용자 요구사항 명세서의 기능들이 모두 충족되는지 확인 |

:::warning 강도 시험 시 주의사항
강도 시험(스트레스 테스트)은 일부러 비정상적인 상황을 만드는 것입니다. 최대 접속자 수를 훨씬 초과하는 가상 요청을 발생시켜서 서버가 다운되는지 확인합니다. 이 테스트는 반드시 **실제 운영 서버가 아닌 테스트 서버**에서 수행해야 합니다.
:::

시스템 테스트에서 특별히 중요한 것 중 하나는 **배치 시험**입니다. 고객이 사용하는 환경에는 다양한 소프트웨어가 설치되어 있을 수 있습니다. 우리 소프트웨어가 특정 백신 프로그램과 충돌한다면, 실제 사용 환경에서 오류가 발생할 수 있습니다. 그러므로 다양한 환경을 시뮬레이션하여 테스트해야 합니다.

---

## 👥 4단계: 인수 테스트 (Acceptance Test)

### 인수 테스트란 무엇인가?

시스템 테스트까지 모두 완료되었다면, 이제 개발된 소프트웨어를 **고객에게 납품**해야 합니다. 이때 고객이 "이 소프트웨어가 내가 원했던 대로 만들어졌는가?"를 확인하는 과정이 **인수 테스트(Acceptance Test)** 입니다.

마치 집을 계약대로 지었는지 건축주가 최종 점검하는 것처럼, 고객이 직접 소프트웨어를 사용해보고 요구사항이 충족되었는지 확인합니다.

### 알파 테스트 vs 베타 테스트

인수 테스트는 크게 두 가지로 나뉩니다.

**알파 테스트 (Alpha Test)**

선택된 사용자가 **개발자의 개발 환경**에 직접 참여하여 테스트하는 방식입니다. 개발자가 참석한 통제된 환경에서 진행되기 때문에, 오류가 발생하면 즉시 개발자에게 피드백할 수 있습니다.

**베타 테스트 (Beta Test)**

게임이나 앱에서 "베타 버전 출시"라는 말을 많이 들어보셨을 겁니다. 베타 테스트는 **개발자가 참여하지 않고**, 실제 고객들이 **자신의 환경(집, 회사 등)** 에서 직접 테스트하는 방식입니다. 오류를 발견하면 개발자에게 보고합니다.

| 구분 | 알파 테스트 | 베타 테스트 |
|---|---|---|
| **테스트 환경** | 개발자의 개발 환경 | 고객의 실제 사용 환경 |
| **개발자 참여** | 참여함 (통제된 환경) | 참여하지 않음 |
| **참여자** | 선택된 소수의 사용자 | 불특정 다수의 고객 |
| **오류 발견 가능성** | 보통 | 높음 (실제 환경이므로) |

베타 테스트는 실제 고객 환경에서 수행되기 때문에 **오류가 발견될 가능성이 가장 높은** 현실적인 테스트 방법입니다. 다양한 운영체제, 브라우저, 네트워크 환경에서 예상치 못한 오류가 발견될 수 있습니다.

---

## 🔧 5단계: 설치 테스트 (Installation Test)

### 설치 테스트란 무엇인가?

인수 테스트까지 완료되었다면, 이제 고객의 실제 운영 환경에 소프트웨어를 **직접 설치**합니다. 이 과정에서 또 다른 문제가 발견될 수 있으며, 이를 확인하는 것이 **설치 테스트(Installation Test)** 입니다.

설치 과정에서 확인해야 할 사항들은 다음과 같습니다.

- **서버 사양 확인**: CPU 성능, 메모리 용량, 저장 장치(RAID 구조) 등이 요구 사양을 충족하는지
- **파일 및 데이터베이스 설정**: 로그 파일, 데이터베이스 설정이 올바른지
- **연동 시스템 확인**: 다른 소프트웨어와 연동할 때 문제가 없는지
- **교육**: 사용자에게 소프트웨어 사용법을 교육

### 백투백(Back-to-Back) 시험

설치 테스트 과정에서 **백투백(Back-to-Back) 시험**이라는 방법이 사용됩니다. 이는 시스템에 **두 가지 버전(예: 버전 1.0과 버전 2.0)** 을 동시에 적용하여 같은 테스트를 수행하고 결과를 비교하는 방식입니다.

만약 두 버전의 결과가 서로 다르다면, 둘 중 하나에 문제가 있다는 신호입니다. 이 방법은 새로운 버전에서 기존 기능이 의도치 않게 변경되었는지를 빠르게 확인할 수 있는 장점이 있습니다.

---

## 🛠️ 자동 검사 도구: 정적 분석기와 동적 분석기

### 정적 분석기 (Static Analyzer)

**정적 분석기(Static Analyzer)** 는 프로그램을 **실제로 실행하지 않고** 소스 코드 자체를 검사하는 도구입니다. 마치 문서 편집기의 맞춤법 검사처럼, 코드를 실행하기 전에 잠재적 오류를 찾아냅니다.

정적 분석이 유용한 대표적인 경우:
- **무한 루프**: 반복문이 끝나지 않는 경우
- **미초기화 변수**: 값이 할당되지 않은 변수를 사용하는 경우
- **실행 불가능한 코드**: 절대 실행될 수 없는 코드가 존재하는 경우

```python
# ❌ 잘못된 예시: 정적 분석기가 찾아낼 수 있는 오류들

def calculate_discount(price):
    discount_rate = 0.1  # 할인율 설정
    
    # 초기화되지 않은 변수 사용 (정적 분석기가 경고)
    # final_price = price - (price * discount_rate)  # 이 줄은 주석 처리됨
    
    # 실행 불가능한 코드 (정적 분석기가 경고)
    if False:
        print("이 코드는 절대 실행되지 않음")  # Dead Code
    
    # 무한 루프 위험 (조건이 항상 참인 경우)
    count = 0
    while count >= 0:  # count는 항상 0 이상이므로 무한 루프
        count += 1
    
    return discount_rate  # final_price가 아닌 discount_rate를 반환하는 실수
```

```python
# ✅ 올바른 예시: 정적 분석기 경고를 해결한 코드

def calculate_discount(price):
    discount_rate = 0.1  # 할인율 설정
    
    # 변수를 올바르게 초기화하고 사용
    final_price = price - (price * discount_rate)  # 할인 가격 계산
    
    # 유효한 반복 조건 사용
    count = 0
    while count < 10:  # 10번만 반복하는 명확한 종료 조건
        count += 1
    
    return final_price  # 올바른 변수 반환
```

정적 분석 도구의 종류:

| 도구 종류 | 역할 |
|---|---|
| **코드 분석 도구** | 문법 오류, 오류 발생 가능성이 있는 코드를 찾아줌 |
| **구조 검사 도구** | 소스 코드의 흐름 그래프를 생성하고 논리 흐름을 시각화하여 구조적 결함 확인 |
| **데이터 분석 도구** | 데이터 구조의 올바른 초기화 여부, 잘못된 데이터 사용, 중복 데이터 확인 |

정적 분석에서 수행하는 분석 유형:
- **제어 흐름 분석**: 코드가 어떤 순서로 실행되는지 분석
- **데이터 사용 분석**: 변수와 데이터가 올바르게 사용되는지 분석
- **인터페이스 분석**: 모듈 간 연결이 제대로 되어 있는지 분석
- **경로 분석**: 데이터가 이동하는 경로와 그 과정의 명령어들을 분석

### 동적 분석기 (Dynamic Analyzer)

**동적 분석기(Dynamic Analyzer)** 는 정적 분석기와 반대로, **프로그램을 실제로 실행하면서** 동작과 반응을 추적하고 분석하는 도구입니다.

동적 분석기는 소스 코드의 중간중간에 **검사 코드(Instrument Code)** 를 삽입합니다. 그러면 프로그램이 실행되는 동안 특정 시점의 변수 값, 메모리 상태, 이벤트 등을 실시간으로 확인할 수 있습니다. 마치 의사가 환자에게 심장 모니터를 부착하고 실시간으로 심박수를 추적하는 것과 비슷합니다.

```python
# ✅ 동적 분석을 위해 검사 코드(instrument code)를 삽입한 예시

def process_order(items):
    # 검사 코드: 함수 시작 시 입력값 확인
    print(f"[DEBUG] process_order 시작 - 입력 아이템 수: {len(items)}")
    
    total = 0
    for i, item in enumerate(items):
        price = item['price']
        quantity = item['quantity']
        subtotal = price * quantity
        
        # 검사 코드: 각 반복마다 중간 결과 확인
        print(f"[DEBUG] 아이템 {i+1}: 가격={price}, 수량={quantity}, 소계={subtotal}")
        
        total += subtotal
    
    # 검사 코드: 함수 종료 시 최종 결과 확인
    print(f"[DEBUG] process_order 완료 - 총액: {total}")
    
    return total

# 테스트 실행
items = [
    {'price': 10000, 'quantity': 2},
    {'price': 5000, 'quantity': 3}
]
result = process_order(items)
# [DEBUG] process_order 시작 - 입력 아이템 수: 2
# [DEBUG] 아이템 1: 가격=10000, 수량=2, 소계=20000
# [DEBUG] 아이템 2: 가격=5000, 수량=3, 소계=15000
# [DEBUG] process_order 완료 - 총액: 35000
```

## 🐛 디버깅 (Debugging) — 오류를 찾고 수정하는 과정

### 디버깅이란?

**디버깅(Debugging)** 은 테스트 과정에서 발견된 오류를 **분석하고 수정하는 작업**입니다. 개발자가 가장 많이 하는 작업 중 하나이기도 합니다. 사용자가 "오류가 있어요"라고 보고하면, 개발자는 그 오류를 찾아서 고치는 디버깅 작업을 수행합니다.

### 디버깅 방법

| 방법 | 설명 | 효율성 |
|---|---|---|
| **맹목적 디버깅** | 문제 원인을 메모리에서 직접 찾으려는 방식 | 매우 낮음 |
| **출력문 삽입** | 임의의 위치에 출력문을 넣어 결과를 확인하는 방식 | 보통 |
| **디버깅 도구 사용** | 실행 추적, 중단점(Breakpoint) 설정 등 도구를 활용 | 높음 |

디버깅의 접근 방식:

- **귀납적(균압적) 방식**: 관련 정보를 수집하고 문제 원인을 하나씩 하나씩 찾아가는 방식
- **연역적 방식**: 가능한 모든 원인을 먼저 나열한 뒤, 하나씩 제거하며 실제 원인을 규명하는 방식
- **역행 조사(Back-tracking)**: 오류가 발생한 지점에서 소스 코드를 거꾸로 추적하여 원인을 찾는 방식

### 디버깅의 원칙

:::warning 디버깅 시 반드시 지켜야 할 원칙들
1. **혼자 해결할 수 없으면 도움을 요청하세요.** 끝까지 혼자만 해결하려는 고집은 시간과 비용을 낭비합니다.
2. **디버깅 도구는 만능이 아닙니다.** 도구는 차선책일 뿐, 처음부터 잘 만드는 것이 최선입니다.
3. **전체 시스템 구조를 이해하는 것이 가장 중요합니다.** 시스템 전체를 모르면 오류의 원인을 찾기 어렵습니다.
4. **무작위 실험은 최후의 수단입니다.** "이렇게 해볼까, 저렇게 해볼까" 식의 시도는 피하세요.
:::

:::danger 디버깅 시 절대 하면 안 되는 착각
- **"하나 고쳤으니까 끝이야"라고 생각하지 마세요.** 하나의 결함을 고치면, 그와 연관된 또 다른 결함이 나타날 수 있습니다. 교정이 정확할 확률은 프로그램 규모에 반비례합니다. 규모가 클수록 한 곳을 고쳐도 다른 곳에서 문제가 생길 가능성이 높습니다.
- **"A를 고쳤으니 B는 괜찮겠지"라는 가정 금지.** A를 수정한 것이 B에 예상치 못한 영향을 줄 수 있습니다. 이 경우 전체 시스템 개선이 필요한 대형 공사로 이어질 수 있습니다.
:::

디버깅의 올바른 절차를 정리하면 다음과 같습니다.

1. **에러 위치 파악**: 디버깅 툴을 사용하여 오류가 발생한 정확한 위치를 찾습니다.
2. **설계 에러 수정**: 설계 자체에 문제가 있다면 설계를 먼저 수정합니다.
3. **코드 에러 수정**: 실제 소스 코드의 오류를 수정합니다.
4. **재시작 및 배포**: 수정된 프로그램을 다시 빌드하고 서버에 올려 최신 버전으로 업데이트합니다.
5. **재테스트**: 수정 후 반드시 다시 테스트하여 수정이 올바르게 이루어졌는지, 새로운 오류가 생기지 않았는지 확인합니다.

## 📋 핵심 정리

이 문서에서 배운 핵심 개념을 정리합니다.

- **단위 테스트(Unit Test)**: 모듈 하나하나를 독립적으로 검증하는 테스트. 화이트박스 기법 사용, 코딩 단계와 병행 수행.
- **스텁(Stub)**: 단위 테스트 시 아직 완성되지 않은 다른 모듈을 흉내 내는 가짜 모듈. 없으면 단위 테스트 불가.
- **드라이버(Driver)**: 실제 모듈을 실행시키고 입력/출력을 처리하는 모듈.
- **통합 테스트(Integration Test)**: 모듈 간 인터페이스가 올바르게 동작하는지 검증. 블랙박스 기법 사용.
- **인터페이스 유형**: 매개변수, 공유 메모리, 프로시저, 메시지 전달 — 네 가지 방식으로 모듈 간 데이터를 주고받음.
- **통합 방식**: 빅뱅(비추), 하향식(탑다운), 상향식(바텀업), 연쇄식(샌드위치) 네 가지가 있음.
- **시스템 테스트**: 실제 서버에 소프트웨어를 올려 하드웨어와 결합하여 성능, 보안, 복구 등을 종합 검증.
- **인수 테스트**: 고객이 요구사항 충족 여부를 확인. 알파(개발 환경)와 베타(고객 환경) 두 종류.
- **설치 테스트**: 실제 운영 환경에 설치 시 서버 사양, 연동, 설정 등을 최종 확인.
- **백투백 시험**: 두 버전을 동시에 같은 테스트로 수행하여 결과를 비교하는 방식.
- **정적 분석기**: 프로그램을 실행하지 않고 소스 코드 자체를 분석. 무한 루프, 미초기화 변수 등 발견.
- **동적 분석기**: 프로그램을 실행하면서 동작을 추적. 검사 코드를 삽입하여 실시간 상태를 확인.
- **디버깅(Debugging)**: 발견된 오류를 분석하고 수정하는 작업. 하나 고쳤다고 끝이 아니며, 연관된 결함이 추가로 발생할 수 있음.
- **디버깅 원칙**: 혼자 해결 불가 시 도움 요청, 시스템 전체 구조 이해, 교정 후 반드시 재테스트.
