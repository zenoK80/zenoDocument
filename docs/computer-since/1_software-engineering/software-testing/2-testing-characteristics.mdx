# 🧪 소프트웨어 테스트의 특징과 수행 방식

소프트웨어를 개발하고 나면 "다 만들었으니까 이제 끝!"이라고 생각하기 쉽습니다. 하지만 아무리 훌륭한 요리사라도 음식을 손님에게 내기 전에 한 번 맛을 보듯이, 소프트웨어도 사용자에게 전달하기 전에 반드시 **테스트(Test)** 라는 과정을 거쳐야 합니다. 이번 문서에서는 소프트웨어 테스트가 무엇인지, 왜 필요한지, 어떤 방식으로 수행하는지를 처음 접하는 분들도 이해할 수 있도록 아주 친절하게 설명해 드리겠습니다.

---

## 🎯 소프트웨어 테스트란 무엇인가?

### 학교 시험과 소프트웨어 테스트는 어떻게 다를까?

우리가 일상에서 접하는 "시험"이라는 단어를 떠올려 보세요. 학교에서 중간고사나 기말고사를 볼 때, 그 시험은 **학생이 얼마나 공부를 잘 했는지**를 평가하는 도구입니다. 100점을 맞으면 잘 공부한 것이고, 50점을 맞으면 부족한 부분이 있다는 것을 알게 되죠.

그렇다면 소프트웨어 테스트는 어떨까요? 비슷해 보이지만 사실 목적이 조금 다릅니다. 소프트웨어 테스트는 **프로그램 개발이 완전히 끝난 후**, 문법적인 오류(컴파일 에러 등)가 없고 정상적으로 실행은 되지만, "과연 이 프로그램이 처음에 약속한 요구사항대로 올바르게 동작하는가?"를 **검사하고 평가하는 전체 과정**입니다.

> **소프트웨어 테스트는 단순히 오류를 찾는 것이 아닙니다.**
> 정해진 요구사항이 제대로 충족되고 있는지, 입력값에 대해 올바른 출력값이 나오는지를 체계적으로 검사하고 평가하는 과정 전체를 의미합니다.

예를 들어 계산 프로그램에서 `A`라는 값을 입력하면 항상 `100`이 나와야 한다고 합시다. 그런데 어떨 때는 `100`이 나오고, 어떨 때는 `120`이 나온다면? 눈으로는 프로그램이 잘 돌아가는 것처럼 보이지만, 실제로는 **숨겨진 결함(버그)** 이 존재하는 것입니다. 이런 숨어있는 문제를 찾아내는 것이 바로 소프트웨어 테스트입니다.

### 테스트가 다루는 핵심 질문들

테스트를 계획할 때는 단순히 "실행해보자"가 아니라, 아래와 같은 구체적인 질문들에 답을 준비해야 합니다.

- 이 테스트를 **수동으로** 할 것인가, **자동화 도구**를 이용할 것인가?
- **어느 범위까지** 검사할 것인가?
- **얼마나 오랜 기간** 동안 테스트를 진행할 것인가?
- **테스트 데이터의 양**은 얼마나 할 것인가?

마치 요리 대회 심사처럼, 어떤 기준으로 얼마나 평가할지를 미리 정해놓고 진행해야 공정하고 효율적인 테스트가 됩니다.

:::info 숨어있는 결함이란?
컴파일(번역) 과정에서 발생하는 문법 오류는 컴파일러(번역기)가 바로 알려줍니다. 하지만 **숨겨진 결함**은 프로그램이 정상적으로 실행되는 것처럼 보이다가, 특정 조건이나 입력값에서만 잘못된 결과를 내는 오류입니다. 이런 결함은 테스트를 충분히 하지 않으면 사용자가 실제로 쓰다가 발견하게 되고, 그 때는 수정 비용이 훨씬 크게 발생합니다.
:::

## 🔍 왜 테스트가 반드시 필요한가? — 배경과 중요성

### 100% 완벽한 프로그램은 존재하지 않는다

많은 입문자들이 "내가 열심히 만들었으니까 오류가 없겠지?"라고 생각하는 경우가 있습니다. 하지만 소프트웨어 업계의 현실은 다릅니다. **지구상에 100% 완벽한 소프트웨어는 존재하지 않습니다.** 어떤 예외 상황에서 오류가 발생할 가능성은 언제나 존재하기 때문입니다.

따라서 테스트의 목적은 "이 시스템은 완벽합니다"를 증명하는 것이 **아닙니다**. 테스트의 진짜 목적은 **"이런 결함들이 있었고, 우리가 이렇게 해결했습니다"** 라는 과정을 통해 결함을 찾아내고 개선해 나가는 것입니다.

> 테스트는 **결함이 없음을 증명**하는 활동이 아니라, **결함을 발견**하고 해결해 나가는 활동입니다.

### 40-20-40 법칙 — 테스트는 절반만큼 중요하다

소프트웨어 개발에 들어가는 노력과 비용의 이상적인 배분 비율이 있습니다.

| 단계 | 이상적인 비율 | 소규모 업체의 현실 |
|------|:---:|:---:|
| 분석 & 설계 | **40%** | 20% |
| 구현 (코딩) | **20%** | 70% |
| 테스트 | **40%** | 10% |

많은 소규모 IT 업체에서는 개발자가 코딩에 집중하다 보니, 분석·설계와 테스트가 소홀해집니다. 그 결과 요구사항도 제대로 반영되지 않고, 테스트도 부실해져서 **유지보수에 엄청난 추가 비용과 시간이 투입**되는 악순환이 생깁니다.

마치 집을 지을 때 기초 공사(분석/설계)와 마감 검사(테스트)를 대충 하면, 나중에 벽에 균열이 생기고 누수가 발생해 훨씬 큰 보수 비용이 드는 것과 같습니다. **처음과 끝이 중요합니다.** 분석·설계라는 첫 단추와 테스트라는 마지막 단추를 잘 꿰어야 합니다.

### 테스트는 모든 단계에서 수행된다

테스트는 개발이 완료된 후 마지막 단계에서만 하는 것이 아닙니다. 아래와 같이 모든 개발 단계에서 필요한 관점으로 수행됩니다.

| 역할 | 하는 일 |
|------|---------|
| **개발자** | 개발 + 디버깅(오류 수정) |
| **테스터** | 시나리오 기반 검증 |
| **사용자** | 실제 데이터를 넣어서 확인 |
| **품질 보증팀** | 소프트웨어가 품질 기준을 만족하는지 전반 보증 |

:::tip 테스트는 특정 단계에만 있는 것이 아닙니다
코딩 중간에도, 모듈 완성 후에도, 시스템 설치 과정에서도 테스트는 필요합니다. 소프트웨어 개발의 **모든 단계에서** 테스트가 필요하다는 점을 꼭 기억하세요.
:::

## 🧩 테스트의 핵심 원리와 수행 5단계

### 테스트는 순서대로 계획하고 실행한다

테스트를 그냥 "한번 돌려보자"는 식으로 주먹구구로 하면 안 됩니다. 체계적인 5단계 절차를 따라야 합니다.

#### 1단계: 목표 설정 (What — 무엇을 테스트할 것인가?)

가장 먼저 **테스트의 목표**를 명확하게 정해야 합니다. "이번 테스트는 기능이 제대로 구현되어 있는지를 확인하는 것이다", 또는 "데이터의 신뢰성을 보장하는 것이다"처럼 방향을 먼저 잡아야 합니다. 목표가 없으면 무엇을 어디까지 테스트해야 하는지 알 수 없습니다.

#### 2단계: 테스트 방법 결정 (How — 어떻게 할 것인가?)

목표가 정해지면 **어떤 방법으로 테스트할지** 결정합니다. 코드를 보면서 테스트하는 **화이트박스 테스트**를 할 것인지, 코드 없이 입출력만 보는 **블랙박스 테스트**를 할 것인지, 아니면 **자동화 도구**를 활용할 것인지를 선택합니다. 이 두 가지 방식은 뒤에서 자세히 설명합니다.

#### 3단계: 테스트 케이스 작성

방법이 결정되면 **구체적인 테스트 케이스(Test Case)** 를 만듭니다. 테스트 케이스란 "이 조건에서, 이 데이터를 입력하면, 이 결과가 나와야 한다"를 정리한 하나의 시나리오입니다. 마치 수학 문제처럼 문제(입력)와 정답(예상 출력)을 미리 준비하는 것입니다. 이때 예상 출력값을 **테스트 오라클(Test Oracle)** 이라고 합니다.

#### 4단계: 테스트 실행

준비가 완료되면 실제로 소프트웨어에 데이터를 입력하며 테스트를 수행합니다. 올바른 값만 넣는 것이 아니라, **의도적으로 잘못된 값이나 오류를 유발하는 값**도 넣어봐야 합니다. 이 과정에서 중간 결과를 화면에 출력해 주는 임시 코드를 삽입하기도 하는데, 이런 임시 테스트용 프로그램을 **테스트 하네스(Test Harness)** 라고 합니다. 테스트가 끝나면 이 임시 코드는 반드시 제거해야 합니다.

#### 5단계: 결과 분석 및 결함 보고

테스트 결과를 분석해서 **테스트 결과 보고서**를 작성하고, 발견된 결함을 개발팀에 전달합니다. 개발팀은 수정 후 다시 테스트를 반복합니다. 이 과정을 **회귀 테스트(Regression Test, 리그레션 테스트)** 라고 합니다.

:::warning 테스트 케이스에 오류가 없다고 안심하면 안 됩니다
테스트를 해서 오류가 발견되지 않았다고 해서 프로그램에 오류가 없다는 의미가 아닙니다. 테스트 케이스가 부족했을 가능성이 훨씬 높습니다. 모든 경우의 수를 무한대로 테스트할 수 없기 때문에, 가능한 한 다양하고 많은 케이스를 준비하는 것이 중요합니다.
:::

### 좋은 테스트 케이스의 조건

좋은 테스트 케이스는 다음 조건을 만족해야 합니다.

- **모듈 내의 모든 독립적인 경로가 최소 한 번씩 실행**되어야 합니다. 어느 하나의 경로를 빠뜨리면 그 부분의 결함을 발견할 기회가 없습니다.
- 가능하면 **복잡한 알고리즘은 배제**하고, 간단하고 명확한 조건으로 케이스를 만들어야 합니다.
- **내부 데이터 구조를 활용**하여 테스트하면 더욱 효과적입니다.
- **기대되지 않는 입력값(경계값, 오류값)** 에 대한 케이스도 반드시 포함해야 합니다.

## 🔬 Verification & Validation — 두 가지 관점의 테스트

### Verification(베리피케이션) — 개발자 관점의 점검

**Verification(검증)** 은 개발자 입장에서 "이 시스템이 요구사항 명세서대로 올바르게 만들어졌는가?"를 점검하는 것입니다. 즉, 기능적 요구사항과 비기능적 요구사항(성능, 트래픽, 응답시간 등)이 모두 충족되고 있는지를 확인합니다.

쉽게 말하면 **"제품을 올바르게 만들고 있는가?(Are we building the product right?)"** 에 대한 답을 구하는 과정입니다.

### Validation(베리데이션) — 사용자 관점의 점검

**Validation(확인)** 은 사용자 시각에서 "이 시스템이 사용자가 실제로 원하는 것을 충족하는가?"를 평가하는 것입니다. 기능이 기술적으로 잘 동작하는지가 아니라, **사용자의 실제 요구사항이 제대로 구현되어 있는지**를 보는 관점입니다.

쉽게 말하면 **"올바른 제품을 만들고 있는가?(Are we building the right product?)"** 에 대한 답을 구하는 과정입니다.

> 이 두 가지를 합쳐서 **V&V(Verification and Validation)** 라고 부르며, 둘 다 함께 수행되어야 완전한 테스트가 됩니다.

### Debugging(디버깅) — 오류 수정 작업

**디버깅(Debugging)** 은 테스트 과정에서 발견된 오류를 **찾아서 수정하는 작업**입니다. 테스트는 오류를 "발견"하는 것이고, 디버깅은 발견된 오류를 "수정"하는 것입니다. 이 둘은 서로 다른 활동이지만 밀접하게 연결되어 있습니다.

### 품질 보증(QA — Quality Assurance)

**품질 보증**은 품질 보증 부서에서 소프트웨어의 최종 결과물이 명세서와 일치하는지 확인하고, 품질이 일정 기준 이상을 만족한다는 것을 전반적으로 보증하는 활동입니다. 개별 기능 테스트를 넘어서 소프트웨어 전체의 품질 수준을 관리하는 활동이라고 이해하면 됩니다.

## 📋 테스트의 원칙과 개발자가 자신의 코드를 테스트하면 안 되는 이유

### 왜 개발자는 자기 코드를 직접 테스트하면 안 될까?

이것은 매우 중요한 원칙입니다. 예를 들어 계산기 프로그램을 만든 개발자가 있다고 합시다. 이 개발자는 코드를 짜면서 이미 "0으로 나누면 오류가 발생한다"는 것을 알고 있습니다. 그렇기 때문에 테스트할 때 **무의식적으로 0을 입력하는 것을 피하게 됩니다.**

반면 일반 사용자는 "100 ÷ 0을 계산하면 어떻게 되지?"라고 순수하게 궁금해서 입력해 봅니다. 그때 에러 메시지가 제대로 나오지 않거나 프로그램이 멈춰버린다면? 그것이 바로 숨어있는 결함입니다.

개발자가 직접 테스트하면 이런 문제가 생깁니다.

- 자신이 만든 코드이므로 **어떤 값을 넣으면 오류가 난다는 것을 이미 알고 있어** 그 값을 피하게 됩니다.
- 자신이 만든 코드에 대해 **심리적으로 너그러워** 오류를 쉽게 용납하려는 경향이 생깁니다.
- 결과적으로 테스트 데이터가 **제한적**이 되고, 오류 발견율이 크게 낮아집니다.

:::danger 개발자가 자신의 코드를 직접 테스트하는 것은 피해야 합니다
개발자가 만든 코드는 **개발과 무관한 별도의 독립된 팀**에서 테스트하는 것이 원칙입니다. 개발자는 무의식적으로 자신의 코드에 유리한 방향으로 테스트하게 되어 결함을 발견하지 못하는 경우가 많습니다.
:::

### 결함의 정비례 법칙

소프트웨어 테스트에는 흥미로운 법칙이 있습니다. **이미 발견된 결함의 수가 많을수록, 추가로 발견될 결함의 수도 많다**는 것입니다. 결함 10개가 발견된 시스템은 추가로 10개 이상의 결함이 더 있을 가능성이 높습니다. 반면 결함이 1~2개밖에 발견되지 않은 시스템은 상대적으로 안정적으로 개발된 것으로 볼 수 있습니다.

### Inspection(인스펙션) vs Testing(테스팅)

| 구분 | Inspection (검사) | Testing (테스팅) |
|------|-----------------|----------------|
| 방식 | **정적(Static)** — 실행 없이 코드 분석 | **동적(Dynamic)** — 실제 실행하며 확인 |
| 대상 | 프로그램 코드, 명세서 일치 여부 | 입력 데이터와 출력 결과 |
| 범위 | 제한적 (코드 검사 위주) | 광범위 (시스템 품질 전반) |
| 특징 | 자동화 도구로 소스코드 분석 시 활용 | 실제 시스템 품질 보장에 핵심적 |

**Inspection**은 마치 원고를 출판하기 전에 교정자가 글을 읽으며 오탈자를 찾는 것과 같습니다. **Testing**은 실제로 책을 독자에게 나눠주고 읽어보게 하면서 내용이 이해되는지 확인하는 것과 같습니다. 당연히 Testing이 훨씬 더 광범위하고 실질적인 품질 보장 활동입니다.

## 🏗️ 테스트 단계 — 단위 테스트부터 인수 테스트까지

소프트웨어 테스트는 한 번에 전체를 검사하는 것이 아니라, **개발 단계에 맞춰 점진적으로** 수행됩니다.

### 1️⃣ 단위 테스트 (Unit Test)

**단위 테스트**는 프로그램을 구성하는 **각각의 독립된 모듈(기능 단위)** 을 하나씩 따로 테스트하는 것입니다. 마치 자동차를 조립하기 전에 엔진, 바퀴, 브레이크를 각각 따로 점검하는 것과 같습니다.

예를 들어 쇼핑몰을 개발한다면 다음과 같은 독립 모듈이 있습니다.

- 상품 소개 화면 모듈
- 주문 처리 모듈
- 배송 관리 모듈
- 마이페이지(포인트, 주문내역, 회원정보) 모듈

이 각각을 **따로따로 테스트**해서 인터페이스, 자료 구조, 수행 경로, 오류 처리 등을 확인합니다. 단위 테스트는 **실제 코드를 기반**으로 수행합니다.

### 2️⃣ 통합 테스트 (Integration Test)

단위 테스트가 완료된 모듈들을 **서로 연결하여** 모듈 간의 인터페이스와 데이터 흐름이 올바른지 테스트합니다. 쇼핑몰 예시로 보면, "상품을 선택하고 → 주문하면 → 배송이 시작되고 → 마이페이지에 주문내역이 반영된다"는 전체 흐름이 올바르게 연결되는지를 확인하는 것입니다.

통합 테스트의 방식으로는 **빅뱅(Big Bang)**, **상향식(Bottom-Up)**, **하향식(Top-Down)**, **샌드위치(Sandwich)** 방식이 있습니다. 통합 테스트는 **구조 설계** 결과를 기반으로 수행합니다.

### 3️⃣ 시스템 테스트 (System Test)

통합된 전체 시스템을 실제 운영 환경(또는 유사 환경)에 올려서 테스트합니다. 이 단계에서는 다음과 같은 항목들을 확인합니다.

- **보안**: 외부 침입에 대한 방어가 되는가?
- **호환성**: 모바일, PC, 태블릿 등 다양한 환경에서 올바르게 동작하는가?
- **성능**: 동시에 많은 사용자가 접속해도 시스템이 정상 운영되는가?

시스템 테스트는 **요구사항 분석** 결과를 기반으로 수행합니다.

### 4️⃣ 인수 테스트 (Acceptance Test)

인수 테스트는 **고객(발주자)이 최종적으로 소프트웨어를 검수하고 받아들이는 과정**입니다. 마치 집을 지어준 건설사가 입주자에게 집을 인도하기 전에 함께 점검하는 것과 같습니다. 대표적으로 **알파(Alpha) 테스트**와 **베타(Beta) 테스트**가 있습니다.

- **알파 테스트**: 개발사 내부 또는 제한된 사용자가 테스트하는 방식
- **베타 테스트**: 실제 사용자들에게 공개하여 피드백을 받는 방식

### 5️⃣ 설치 테스트 (Installation Test)

모든 테스트가 끝난 후, 실제 고객의 서버나 환경에 소프트웨어를 **설치**하고 하드웨어 구성 및 설치 사항이 올바른지 확인하는 마지막 단계입니다.

### 개발 단계와 테스트 단계의 대응 관계

```
[개발 단계]                    [테스트 단계]
────────────────────────────────────────────
범위/목표 설정         ←→    인수 테스트 (전체 목표 부합 여부)
요구사항 분석          ←→    시스템 테스트 (요구사항 충족 여부)
구조 설계             ←→    통합 테스트 (모듈 연계 여부)
프로그램 설계/코딩     ←→    단위 테스트 (개별 모듈 동작)
```

## ⬜ 화이트박스 테스트 (White-Box Test)

### 화이트박스 테스트란?

**화이트박스 테스트**는 소프트웨어의 **내부 코드와 논리적 구조를 직접 보면서** 테스트하는 방식입니다. 상자 안이 투명하게 보인다고 해서 화이트박스(White-Box)라고 부릅니다. 코드를 기반으로 하기 때문에 **코드 기반 테스트**, **구현 기반 테스트**, **구조 기반 테스트**라고도 합니다.

마치 자동차 엔진 덮개를 열고 내부 부품을 하나하나 들여다보며 점검하는 것과 같습니다.

### 화이트박스 테스트의 방법들

#### 기본 경로 테스트 (Basis Path Testing)

원시 코드의 **모든 독립적인 경로가 최소 한 번씩 실행**되도록 테스트하는 방식입니다. 수행 순서는 다음과 같습니다.

1. 코드 작성 및 순서도(흐름도) 작성
2. 흐름 그래프(Flow Graph) 작성
3. 복잡도(Cyclomatic Complexity) 계산
4. 독립적인 경로 정의
5. 테스트 케이스 작성 및 수행

```python
# 예시: 1부터 n까지의 합을 구하는 함수
def sum_to_n(n):
    total = 0          # 합계를 저장할 변수 초기화
    i = 1              # 반복 시작 값
    while i <= n:      # i가 n 이하인 동안 반복 (반복 경로)
        total += i     # 합계에 i를 더함
        i += 1         # i를 1 증가
    return total       # 최종 합계 반환

# 기본 경로 테스트 케이스 예시
# - n=100 입력 시 → 예상 출력: 5050 (1+2+3+...+100)
# - n=0 입력 시 → 예상 출력: 0 (반복문 실행 안 됨, 경계값)
# - n=1 입력 시 → 예상 출력: 1 (반복문 1회 실행)
```

위 코드에서 `while` 반복문 내부의 경로와 반복문을 아예 실행하지 않는 경로(`n=0`인 경우)가 모두 최소 한 번씩 실행되어야 기본 경로 테스트가 완료됩니다.

#### 문장 검증 기준 (Statement Coverage)

프로그램 내의 **모든 문장(Statement)이 최소 한 번씩 실행**될 수 있도록 테스트 데이터를 설계하는 방식입니다. 어떤 코드 줄도 테스트에서 빠지면 안 된다는 원칙입니다.

```python
# 예시: 점수에 따른 학점 결정 함수
def get_grade(score):
    if score >= 90:        # 조건 1: 90점 이상인지 확인
        return "A"         # 문장 1: A 반환
    elif score >= 80:      # 조건 2: 80점 이상인지 확인
        return "B"         # 문장 2: B 반환
    elif score >= 70:      # 조건 3: 70점 이상인지 확인
        return "C"         # 문장 3: C 반환
    elif score >= 60:      # 조건 4: 60점 이상인지 확인
        return "D"         # 문장 4: D 반환
    else:
        return "F"         # 문장 5: F 반환

# 문장 검증 기준 테스트 케이스
# 모든 return 문장이 최소 한 번씩 실행되어야 함
# 케이스 1: score=95  → 예상 결과: "A" (문장 1 실행)
# 케이스 2: score=85  → 예상 결과: "B" (문장 2 실행)
# 케이스 3: score=75  → 예상 결과: "C" (문장 3 실행)
# 케이스 4: score=65  → 예상 결과: "D" (문장 4 실행)
# 케이스 5: score=50  → 예상 결과: "F" (문장 5 실행)
```

#### 분기 검증 기준 (Branch Coverage)

조건문의 **참(True)과 거짓(False) 결과를 모두 최소 한 번씩** 실행하도록 테스트하는 방식입니다. 분기점에서 오류가 발생할 가능성이 높기 때문에, 각 분기 경로를 모두 통과하는 것이 중요합니다.

```python
# 예시: 짝수/홀수 판별 후 처리
def process_number(n):
    if n % 2 == 0:          # 분기점: n이 짝수인가?
        result = n * 2      # True 경로: 짝수이면 2배
    else:
        result = n * 3      # False 경로: 홀수이면 3배
    return result           # 최종 결과 반환

# 분기 검증 기준 테스트 케이스
# True 경로: n=4 입력 → 예상 결과: 8   (짝수 분기 실행)
# False 경로: n=3 입력 → 예상 결과: 9  (홀수 분기 실행)
```

#### 조건 검증 기준 (Condition Coverage)

두 개 이상의 **개별 조건식**이 있을 때, 각 조건식의 참/거짓을 **모두 독립적으로** 최소 한 번씩 테스트하는 방식입니다.

```python
# 예시: 두 조건을 동시에 확인하는 함수
def check_access(is_member, age):
    # 조건 1: is_member가 True인가?
    # 조건 2: age가 18 이상인가?
    if is_member and age >= 18:  # 복합 조건
        return "접근 허용"
    else:
        return "접근 거부"

# 조건 검증 기준 테스트 케이스
# 케이스 1: is_member=True,  age=20 → "접근 허용" (조건1=T, 조건2=T)
# 케이스 2: is_member=True,  age=15 → "접근 거부" (조건1=T, 조건2=F)
# 케이스 3: is_member=False, age=20 → "접근 거부" (조건1=F, 조건2=T)
# 케이스 4: is_member=False, age=15 → "접근 거부" (조건1=F, 조건2=F)
# 모든 조건의 T/F 조합을 최소 한 번씩 테스트
```

## ⬛ 블랙박스 테스트 (Black-Box Test)

### 블랙박스 테스트란?

**블랙박스 테스트**는 소프트웨어의 **내부 코드나 알고리즘은 보지 않고**, 오직 **입력값과 출력값**만을 기준으로 테스트하는 방식입니다. 상자 안이 보이지 않는다고 해서 블랙박스(Black-Box)라고 부릅니다. **명세 기반 테스트**라고도 합니다.

마치 자판기에 돈을 넣고 버튼을 눌렀을 때 올바른 음료가 나오는지만 확인하는 것과 같습니다. 자판기 내부 기계 구조를 몰라도 테스트할 수 있습니다.

블랙박스 테스트는 **요구사항 명세서나 설계 사양서**에서 테스트 케이스를 추출합니다. 코드를 읽을 수 없는 일반 사용자나 전문 테스터들이 수행하기 적합한 방식입니다.

### Syntax 기법 (문법 기반 테스트)

**문법(규칙)을 정해 놓고** 그 규칙에 맞는 값과 맞지 않는 값을 모두 입력해서 예상 결과가 올바르게 나오는지 확인하는 방법입니다.

예를 들어 회원가입 기능을 테스트한다고 합시다.

**규칙 정의:**
- 아이디: 4~8자, 알파벳과 숫자만 허용
- 비밀번호: 7자 이상, 알파벳·숫자·특수문자 포함 필수

| 테스트 번호 | 아이디 입력 | 비밀번호 입력 | 예상 결과 |
|:---:|:---:|:---:|:---:|
| 케이스 1 | `user123` | `Pass@123` | ✅ 정상 등록 |
| 케이스 2 | `ab` | `Pass@123` | ❌ 아이디 길이 오류 |
| 케이스 3 | `user123` | `pass` | ❌ 비밀번호 길이 오류 |
| 케이스 4 | `user@12` | `Pass@123` | ❌ 아이디 특수문자 오류 |
| 케이스 5 | `user123` | `Password1` | ❌ 특수문자 미포함 오류 |

```python
# Syntax 기법 테스트 예시 코드
def validate_signup(user_id, password):
    # 아이디 길이 검사 (4~8자)
    if not (4 <= len(user_id) <= 8):
        return "오류: 아이디는 4~8자여야 합니다"
    
    # 아이디 문자 검사 (알파벳, 숫자만 허용)
    if not user_id.isalnum():
        return "오류: 아이디에 특수문자를 사용할 수 없습니다"
    
    # 비밀번호 길이 검사 (7자 이상)
    if len(password) < 7:
        return "오류: 비밀번호는 7자 이상이어야 합니다"
    
    # 비밀번호 특수문자 포함 여부 검사
    special_chars = "!@#$%^&*"
    if not any(c in special_chars for c in password):
        return "오류: 비밀번호에 특수문자를 포함해야 합니다"
    
    return "정상 등록 완료"  # 모든 조건 통과 시

# ✅ 올바른 테스트 - 다양한 케이스를 모두 검증
print(validate_signup("user123", "Pass@123"))  # 정상 등록 완료
print(validate_signup("ab", "Pass@123"))        # 오류: 아이디 길이
print(validate_signup("user123", "pass"))       # 오류: 비밀번호 길이

# ❌ 잘못된 테스트 - 정상 케이스만 테스트
# print(validate_signup("user123", "Pass@123"))  # 이것만 하면 오류 발견 불가
```

### 동등분할 기법 (Equivalence Partitioning)

**동등분할 기법**은 입력값의 범위를 여러 구간(파티션)으로 나누고, 각 구간에서 **대표값 하나씩**을 선택해 테스트하는 방법입니다. 같은 구간 내의 값들은 동일하게 처리된다고 가정합니다.

이 방법은 사용자도 쉽게 이해하고 작성할 수 있는 가장 직관적인 테스트 기법입니다.

| 점수 범위 | 학점 | 인센티브 | 대표 테스트 값 | 예상 결과 |
|:---:|:---:|:---:|:---:|:---:|
| 90점 이상 | A | 100만원 | 95점 | 100만원 |
| 80~89점 | B | 70만원 | 85점 | 70만원 |
| 70~79점 | C | 40만원 | 75점 | 40만원 |
| 60~69점 | D | 20만원 | 65점 | 20만원 |
| 59점 이하 | F | 없음 | 50점 | 0원 |

```python
# 동등분할 기법 예시
def calculate_incentive(score):
    if score >= 90:       # 구간 1: 90점 이상
        return 1000000    # 100만원
    elif score >= 80:     # 구간 2: 80~89점
        return 700000     # 70만원
    elif score >= 70:     # 구간 3: 70~79점
        return 400000     # 40만원
    elif score >= 60:     # 구간 4: 60~69점
        return 200000     # 20만원
    else:                 # 구간 5: 59점 이하
        return 0          # 인센티브 없음

# ✅ 동등분할 테스트 케이스 실행
test_cases = [
    (95, 1000000),   # 구간 1 대표값
    (85, 700000),    # 구간 2 대표값
    (75, 400000),    # 구간 3 대표값
    (65, 200000),    # 구간 4 대표값
    (50, 0),         # 구간 5 대표값
]

for score, expected in test_cases:
    actual = calculate_incentive(score)
    status = "✅ 통과" if actual == expected else "❌ 실패"
    print(f"입력: {score}점 | 예상: {expected:,}원 | 실제: {actual:,}원 | {status}")
```

### 경계값 분석 (Boundary Value Analysis)

**경계값 분석**은 각 구간의 **경계에 있는 값**을 중점적으로 테스트하는 방법입니다. 실제 시스템에서는 경계값 처리 실수로 인한 결함이 매우 자주 발생하기 때문에, 이 기법은 특히 중요합니다.

> 경계값에서 결함이 발생할 확률이 가장 높습니다. 경계값 바로 아래, 경계값 자체, 경계값 바로 위를 반드시 테스트해야 합니다.

예를 들어 점수 범위가 `0~100`인 시스템에서는 다음과 같은 경계값들을 테스트합니다.

| 테스트 케이스 | 입력값 | 테스트 목적 | 예상 결과 |
|:---:|:---:|:---:|:---:|
| 케이스 1 | -1 | 최소값 미달 (음수) | 오류 처리 |
| 케이스 2 | 0 | 최소 경계값 | 정상 처리 (F) |
| 케이스 3 | 1 | 최소 경계값 + 1 | 정상 처리 (F) |
| 케이스 4 | 69 | 구간 경계 미달 | D 또는 F |
| 케이스 5 | 70 | 구간 경계값 | C |
| 케이스 6 | 71 | 구간 경계 + 1 | C |
| 케이스 7 | 100 | 최대 경계값 | A |
| 케이스 8 | 101 | 최대값 초과 | 오류 처리 |

```python
# 경계값 분석 예시
def get_grade_boundary(score):
    # 유효 범위 검사 (경계값: 0과 100)
    if score < 0 or score > 100:
        return "오류: 유효하지 않은 점수입니다"  # 범위 초과 처리
    
    if score >= 90:
        return "A"
    elif score >= 80:
        return "B"
    elif score >= 70:
        return "C"
    elif score >= 60:
        return "D"
    else:
        return "F"

# ✅ 경계값 분석 테스트 케이스
boundary_tests = [
    (-1, "오류: 유효하지 않은 점수입니다"),   # 최소값 미달
    (0,   "F"),                               # 최소 경계값
    (1,   "F"),                               # 최소 경계값 + 1
    (59,  "F"),                               # D 경계 미달
    (60,  "D"),                               # D 경계값 (경계!)
    (61,  "D"),                               # D 경계값 + 1
    (69,  "D"),                               # C 경계 미달
    (70,  "C"),                               # C 경계값 (경계!)
    (100, "A"),                               # 최대 경계값
    (101, "오류: 유효하지 않은 점수입니다"),   # 최대값 초과
]

for score, expected in boundary_tests:
    actual = get_grade_boundary(score)
    status = "✅ 통과" if actual == expected else "❌ 실패"
    print(f"입력: {score:>4}점 | 예상: {expected:<10} | 실제: {actual:<10} | {status}")
```

:::warning 경계값은 개발자가 가장 실수하기 쉬운 부분입니다
`>=`(이상)과 `>`(초과), `<=`(이하)와 `<`(미만)의 차이로 인해 경계값에서 오류가 자주 발생합니다. 예를 들어 70점이 C학점의 시작인데, 코드에 `score > 70`이라고 쓰면 70점이 D로 처리되는 결함이 생깁니다. 경계값 테스트는 이런 실수를 잡아내는 데 매우 효과적입니다.
:::

### 원인결과 그래프 (Cause-Effect Graph)

**원인결과 그래프**는 동등분할이나 경계값 분석만으로는 다루기 어려운 **복합적인 입력 조건**을 테스트하기 위한 방법입니다. 여러 입력 조건이 조합될 때 어떤 결과가 나와야 하는지를 그래프로 표현하고, 이를 **의사결정 테이블(Decision Table)** 로 변환하여 최종 테스트 케이스를 만듭니다.

예를 들어 출입 통제 시스템에서 문이 열리는 조건이 다음과 같다면:

- 교수이고 + 유효한 카드이고 + 우리 구성원 → 출입 허용
- 학생이고 + 유효한 카드이고 + 우리 구성원 → 출입 허용
- 교수이고 + 카드 오류 + 우리 구성원 → "비밀번호 오류" 메시지

이처럼 여러 조건이 조합되는 경우를 **원인(Cause)** 과 **결과(Effect)** 로 정리하여 그래프를 그리고, 체계적으로 테스트합니다. 이 방법은 기존 기법의 단점인 **입력 환경의 복합성을 보완**한 테스트 기법입니다.

수행 순서:
1. 원인(입력 조건)과 결과(출력 조건)를 추출
2. 원인결과 그래프 작성
3. 그래프의 제한 조건 표시
4. 의사결정 테이블로 변환
5. 최종 테스트 시나리오 완성

## ⚠️ 주의사항과 실전 팁

### 성공적인 테스트란 오류를 많이 찾아내는 것이다

직관에 반하는 이야기처럼 들리지만, **테스트 관점에서 좋은 테스트는 더 많은 오류를 찾아내는 테스트**입니다. 테스트 결과 오류가 하나도 없다면, 두 가지 중 하나입니다. 첫째는 정말로 완벽한 프로그램이거나, 둘째는 테스트 케이스가 부족한 것입니다. 완벽한 프로그램은 현실적으로 존재하지 않으므로, 오류가 발견되지 않는다면 테스트를 더 다양하게 보강해야 한다는 신호로 받아들여야 합니다.

### 화이트박스 vs 블랙박스 — 어떤 것을 언제 써야 할까?

| 비교 항목 | 화이트박스 테스트 | 블랙박스 테스트 |
|:---:|:---:|:---:|
| **코드 열람** | 가능 (코드 분석) | 불가능 (코드 미열람) |
| **수행자** | 개발자, 전문 테스터 | 테스터, 일반 사용자 |
| **기반 자료** | 소스 코드, 알고리즘 | 요구사항 명세서, 설계 사양서 |
| **테스트 범위** | 내부 로직, 경로, 구조 | 기능, 성능, 입출력 결과 |
| **특징** | 숨겨진 내부 오류 발견에 유리 | 사용자 관점의 결함 발견에 유리 |

두 가지 방법은 서로 보완 관계입니다. 가장 효과적인 테스트는 **화이트박스와 블랙박스를 함께** 사용하는 것입니다.

### 테스트 목적에 따른 분류

| 테스트 종류 | 검사 내용 |
|:---:|:---:|
| **기능 테스트** | 입력에 따른 출력이 요구사항대로 동작하는가? |
| **성능 테스트** | 응답 시간, 처리량, 메모리, 처리속도가 기준을 충족하는가? |
| **스트레스 테스트** | 극한의 과부하나 최저 조건에서도 시스템이 버티는가? |
| **복잡도 테스트** | 데이터 처리 구조가 단순하여 유지보수가 용이한가? |

:::tip 스트레스 테스트는 왜 중요할까?
타이어 제조사가 타이어를 극한으로 돌려서 언제 터지는지를 확인하듯, 소프트웨어도 극한의 상황(대량 트래픽, 극단적 데이터 등)에서 어느 시점에 문제가 발생하는지 미리 파악해야 합니다. 이를 통해 실제 서비스에서 발생할 수 있는 최악의 상황에 대비할 수 있습니다.
:::

:::note 독립된 테스트팀이 없는 소규모 기업은 어떻게 해야 할까?
별도의 테스트 팀을 운영하기 어려운 소규모 기업에서는 **개발자끼리 서로 다른 사람의 코드를 테스트**하거나, **실제 요청한 사용자(내부 관계자)가 함께 테스트**에 참여하는 방식을 활용합니다. 완벽하지는 않지만, 최소한 자기 코드를 자기가 테스트하는 것은 피하는 것이 중요합니다.
:::

## 📌 핵심 정리

이번 문서에서 배운 소프트웨어 테스트의 핵심 개념을 정리합니다.

- **소프트웨어 테스트**는 단순히 오류를 찾는 것이 아니라, 요구사항 충족 여부와 입출력 정확성을 체계적으로 검사·평가하는 전체 과정이다.
- 테스트는 **결함이 없음을 증명**하는 것이 아니라, **결함을 발견하고 해결**해 나가는 과정이다.
- **100% 완벽한 테스트**는 불가능하며, 최대한 다양한 케이스로 결함을 최소화하는 것이 목표이다.
- 소프트웨어 개발 비용은 **40(분석/설계) : 20(구현) : 40(테스트)** 이 이상적인 비율이며, 테스트는 구현만큼 중요하다.
- **Verification(베리피케이션)** 은 개발자 관점에서 요구사항대로 만들었는지 점검하는 것이다.
- **Validation(베리데이션)** 은 사용자 관점에서 실제 요구사항이 올바르게 구현되었는지 평가하는 것이다.
- 개발자는 자신의 코드를 직접 테스트하면 **심리적 편향**으로 인해 오류를 발견하기 어렵다. 별도의 독립된 팀이 수행하는 것이 원칙이다.
- 테스트는 **단위 → 통합 → 시스템 → 인수 → 설치** 단계 순으로 진행된다.
- **화이트박스 테스트**는 내부 코드를 보며 논리적 구조를 검사하고, **블랙박스 테스트**는 코드 없이 입출력 결과만으로 검사한다.
- 화이트박스 기법: **기본 경로 테스트**, **문장 검증 기준**, **분기 검증 기준**, **조건 검증 기준**
- 블랙박스 기법: **Syntax 기법**, **동등분할 기법**, **경계값 분석**, **원인결과 그래프**
- **경계값**에서 결함이 가장 자주 발생하므로, 경계값 분석은 반드시 수행해야 한다.
- **Inspection(인스펙션)** 은 정적으로 코드를 분석하고, **Testing(테스팅)** 은 동적으로 실행하며 검사하는 방식으로, 둘의 차이를 명확히 알아야 한다.
- **성공적인 테스트**란 오류가 발견되지 않는 것이 아니라, 숨어있는 오류를 더 많이 찾아내는 것이다.
