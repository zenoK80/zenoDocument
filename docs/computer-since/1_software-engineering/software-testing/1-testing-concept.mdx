# 🧪 소프트웨어 테스트의 개념과 정의

소프트웨어를 개발하고 나서 "다 만들었으니 납품하면 되겠지!"라고 생각하면 큰일 납니다. 열심히 만든 프로그램이 현장에서 예상치 못한 오류를 일으키거나, 사용자가 원하는 기능이 제대로 동작하지 않는다면 그 피해는 엄청날 수 있습니다. 바로 이런 상황을 미리 막기 위해 **소프트웨어 테스트(Software Testing)**가 존재합니다.

이번 문서에서는 소프트웨어 테스트가 무엇인지, 왜 필요한지, 어떤 원리와 방법으로 수행되는지를 처음 배우는 분들도 쉽게 이해할 수 있도록 하나씩 풀어서 설명하겠습니다.

---

## 🎯 소프트웨어 테스트란 무엇인가?

### 일상 속 '테스트'와 소프트웨어 '테스트'의 차이

우리가 학교에서 치르는 중간고사나 기말고사를 떠올려 보세요. 시험은 내가 얼마나 공부했는지, 내용을 얼마나 잘 이해하고 있는지를 확인하는 과정입니다. 잘 공부했으면 100점을 받을 수 있고, 부족했다면 50점이 나올 수도 있죠.

그런데 **소프트웨어 테스트**는 조금 다릅니다. 학교 시험처럼 "얼마나 잘 만들었나"를 점수로 매기는 것이 아닙니다. 소프트웨어 테스트는 이미 개발이 완료된 프로그램이 **정해진 요구사항을 제대로 충족하고 있는지**, 그리고 **특정 입력값에 대해 올바른 결과를 반환하는지** 검사하고 평가하는 과정입니다.

예를 들어, "A를 입력하면 반드시 100이 나와야 한다"는 요구사항이 있다고 가정해봅시다. 그런데 어떤 경우에는 100이 나오고, 또 어떤 경우에는 120이 나온다면? 그 프로그램은 **결함(Defect)**이 있는 잘못된 프로그램입니다. 소프트웨어 테스트는 바로 이런 **숨어 있는 결함**을 찾아내는 과정입니다.

> **소프트웨어 테스트의 핵심 정의:**
> 노출되지 않고 숨어 있는 결함을 찾아내기 위해, 정해진 입력 데이터와 조건 및 기간, 방법을 결정하여 프로그램을 검사하고 평가하는 과정입니다.

### 테스트는 언제 하는가?

많은 초보 개발자들이 오해하는 것 중 하나가 "테스트는 개발이 다 끝난 후에 한 번만 하면 된다"는 생각입니다. 하지만 실제로는 그렇지 않습니다.

테스트는 **개발의 모든 단계**에서 필요합니다. 코딩 중간중간에도 할 수 있고, 개발이 완성된 다음에도 할 수 있으며, 시스템을 설치하는 과정에서도 해야 합니다. 어떤 단계에서든 시스템을 평가하는 데 필요하다면 반드시 테스트를 수행해야 합니다.

:::info 눈에 보이는 오류 vs 숨어 있는 결함
- **눈에 보이는 오류**: 코드 문법 오류(Syntax Error)처럼 컴파일(실행 준비) 과정에서 바로 발견됩니다. 컴파일러(코드를 컴퓨터가 이해하는 언어로 변환해주는 프로그램)가 오류를 알려주기 때문에 비교적 쉽게 찾고 수정할 수 있습니다.
- **숨어 있는 결함**: 프로그램은 정상적으로 실행되지만, 특정 조건에서만 잘못된 결과를 내놓는 경우입니다. 이런 결함은 납품 후 사용자가 발견하게 되는 경우가 많아 훨씬 더 위험합니다.
:::

## 🤔 왜 소프트웨어 테스트가 필요한가?

### 테스트 없이 납품하면 무슨 일이 생길까?

마치 요리사가 음식을 손님상에 내놓기 전에 맛을 보지 않는 것처럼, 소프트웨어를 테스트 없이 납품하는 것은 매우 위험한 행동입니다. 실제로 이런 상황이 벌어질 수 있습니다.

개발자가 A라는 값을 한 번 입력해보니 100이라는 정상적인 결과가 나왔습니다. "오케이, 잘 작동하네!" 하고 납품을 했습니다. 그런데 사용자가 실제로 사용하다 보니, 어떤 조건에서는 100이 나오고 또 다른 조건에서는 120이라는 잘못된 값이 출력됩니다. 결국 사용자가 연락을 해옵니다. "이거 값이 이상한 것 같아요. 확인해주세요." 개발자는 다시 원인을 찾아 수정해야 합니다. 이 과정에서 시간과 비용이 추가로 발생하고, 신뢰도까지 떨어지게 됩니다.

### 40-20-40 법칙: 테스트의 중요성

소프트웨어 개발에는 **40-20-40 법칙**이라는 것이 있습니다. 전체 소프트웨어 개발에 들어가는 노력과 비용을 분배하면 다음과 같이 나눌 수 있습니다.

| 단계 | 이상적인 비율 | 소규모 현장에서의 실제 비율 |
|---|---|---|
| 분석 및 설계 | **40%** | 약 20% |
| 구현(코딩) | **20%** | 약 70% |
| 테스트 | **40%** | 약 10% |

많은 소규모 IT 업체에서는 개발자가 코딩에 집중하다 보니 분석·설계에 20%, 코딩에 70%, 테스트에 10% 정도의 비중을 두는 경우가 많습니다. 그 결과 요구사항이 제대로 반영되지 않고, 테스트도 충분히 이루어지지 않아 유지보수에 엄청난 추가 비용과 시간이 필요하게 됩니다.

> 💡 첫 단추와 마지막 단추를 잘 껴야 합니다. **분석·설계(시작)**와 **테스트(마무리)**, 이 두 단계가 소프트웨어 개발의 성패를 가릅니다.

### 테스트는 완벽함을 증명하지 않는다

:::warning 중요한 오해
"테스트를 통과했으니 이 프로그램은 100% 완벽합니다!"라고 말할 수 없습니다.
:::

테스트는 결함이 **없음을 증명**하는 것이 아닙니다. 테스트는 결함을 **찾아내는** 과정입니다. 인간이 만든 소프트웨어인 이상 어떤 예외 상황에서 오류가 발생할 가능성은 언제나 존재합니다. 우리가 테스트를 무한정으로 수행할 수 없기 때문에, 테스트를 완료했다고 해서 완전무결하다고 단정 지을 수 없습니다.

따라서 **테스트의 목표**는 "이 시스템에는 결함이 있었고, 이러이러하게 해결했습니다"를 보여주는 것입니다. 많은 결함을 찾아내고 해결해 나가는 것이 테스트의 본질입니다.

## 🔑 소프트웨어 테스트의 핵심 원리

### 테스트의 5단계 원칙

소프트웨어 테스트를 수행할 때는 주먹구구식으로 진행해서는 안 됩니다. 체계적인 단계를 따라야 합니다.

**1단계: 목표(What) 설정**

가장 먼저 "이번 테스트에서 우리는 무엇을 검증할 것인가?"를 명확히 정해야 합니다. 기능이 올바르게 구현되었는지를 확인하는 **기능 테스트**인지, 데이터의 신뢰성을 보장하는 테스트인지, 아니면 시스템 성능을 확인하는 **성능 테스트**인지 목표를 먼저 결정합니다.

**2단계: 방법(How) 결정**

목표가 정해졌다면, 어떤 방법으로 테스트할지를 결정합니다. 코드를 직접 들여다보는 **화이트박스 테스트**를 할 것인지, 외부에서 입출력만 확인하는 **블랙박스 테스트**를 할 것인지, 혹은 자동화 도구를 활용할 것인지를 선택합니다.

**3단계: 테스트 케이스(Test Case) 작성**

방법이 정해졌다면 구체적인 테스트 사례를 만듭니다. 테스트 케이스(Test Case)란 "이 데이터를 이 조건에서 입력하면, 이런 결과가 나와야 한다"는 구체적인 시나리오를 말합니다. 이때 **예상 결과(Test Oracle, 테스트 오라클)**도 함께 작성해야 합니다.

**4단계: 테스트 실행**

테스트 케이스를 바탕으로 실제 데이터를 소프트웨어에 입력하여 테스트를 수행합니다. 이 과정에서 **테스트 하네스(Test Harness)**라는 테스트 전용 보조 프로그램을 활용하기도 합니다. 예를 들어, 중간 계산 결과를 화면에 출력해주는 임시 기능을 추가해 동작을 확인하고, 테스트가 끝나면 해당 기능을 제거합니다.

**5단계: 결과 비교 및 보고**

실행 결과와 예상 결과(테스트 오라클)를 비교하여 일치하지 않는 부분을 결함으로 기록하고, 테스트 결과 보고서를 작성하여 개발팀에 전달합니다.

### Verification(검증)과 Validation(확인)

테스트를 수행할 때는 두 가지 관점이 함께 필요합니다. 이를 **V&V(Verification and Validation)**라고 합니다.

| 구분 | Verification (검증) | Validation (확인) |
|---|---|---|
| **관점** | 개발자 관점 | 사용자 관점 |
| **질문** | "제품을 올바르게 만들고 있는가?" | "올바른 제품을 만들고 있는가?" |
| **확인 대상** | 요구사항 명세서 대로 기능이 구현되었는가 | 실제 사용자의 요구가 제대로 반영되었는가 |
| **예시** | 로그인 기능이 명세서대로 동작하는가 | 사용자가 실제로 편리하게 로그인할 수 있는가 |

기능적으로 잘 만들어졌다고(Verification) 해서 반드시 사용자가 원하는 결과(Validation)와 일치하지 않을 수 있습니다. 두 가지를 모두 수행해야 완성도 높은 테스트가 됩니다.

### 디버깅(Debugging)과 품질 보증(Quality Assurance)

테스트와 함께 언급되는 중요한 개념들이 있습니다.

- **디버깅(Debugging, 오류 수정)**: 개발자가 테스트 중 발견된 오류의 원인을 찾아내고 수정하는 작업입니다. 마치 시계가 고장났을 때 뚜껑을 열고 어떤 부품이 문제인지 찾아 고치는 것과 같습니다.
- **품질 보증(Quality Assurance, QA)**: 소프트웨어가 전반적으로 명세서 수준의 품질을 만족하는지 보증하는 활동입니다. 단순히 특정 기능의 오류를 찾는 것을 넘어, 소프트웨어 전체의 품질 수준을 평가합니다.

:::note 테스트 관련 역할 정리
- **개발자**: 개발 + 디버깅(오류 수정)
- **테스터**: 테스트 케이스를 활용한 시나리오 검증
- **사용자**: 실제 사용 환경에서 데이터를 넣어 확인
- **품질 보증팀(QA팀)**: 소프트웨어 전반의 품질 수준 보증
:::

## 📋 테스트의 주요 원칙과 특징

### 개발자는 자신의 코드를 테스트하면 안 된다

이것은 소프트웨어 테스트에서 매우 중요한 원칙입니다. 이유가 무엇일까요?

개발자는 자신이 만든 코드의 알고리즘(처리 방법)을 잘 알고 있습니다. 예를 들어, "100을 0으로 나누면 오류가 난다"는 것을 알고 있기 때문에, 테스트할 때 절대로 0을 입력하지 않습니다. 반면 일반 사용자는 그런 제약을 모르기 때문에 계산기에 `100 ÷ 0`을 자연스럽게 입력해볼 수 있습니다. 그럴 때 어떤 처리가 이루어지는지 반드시 검증해야 합니다.

또한 자신이 만든 코드에 대해 심리적으로 관대해지는 경향이 있어, 결함을 발견하더라도 "이 정도면 괜찮겠지"라고 넘어가기 쉽습니다. 따라서 **개발과 무관한 별도의 독립된 테스트팀**이 테스트를 수행하는 것이 가장 효율적입니다.

:::tip 독립된 테스트팀의 효과
독립된 팀이 테스트를 수행하면 개발자가 미처 생각하지 못했던 예외 상황(엣지 케이스, Edge Case)까지 발견할 가능성이 훨씬 높아집니다. 규모가 작은 회사에서는 별도 테스터를 두기 어려운 경우도 있으나, 가능하다면 개발자와 테스트 인력을 분리하는 것을 권장합니다.
:::

### 결함 발견 확률과 기발견 결함 수의 관계

소프트웨어 테스트에서는 흥미로운 법칙이 하나 있습니다. **"추가로 발견될 결함의 수는 이미 발견된 결함 수에 비례한다"**는 것입니다.

쉽게 말해, 테스트 중에 결함이 10개 발견되었다면 앞으로도 추가적인 결함이 발견될 가능성이 높고, 반대로 결함이 1개밖에 발견되지 않았다면 그 시스템은 훨씬 안정적으로 개발된 것이라고 볼 수 있습니다.

### 테스트 케이스의 좋은 조건

좋은 테스트 케이스를 만들기 위해서는 다음 조건을 충족해야 합니다.

- **모든 독립적인 경로를 최소 한 번씩 수행**: 어떤 모듈(독립된 기능 단위)의 모든 처리 경로가 테스트에서 한 번 이상 실행되어야 합니다. 특정 경로를 건너뛰고 테스트하면 그 경로에 숨어 있는 결함을 놓칠 수 있습니다.
- **간단하고 명확한 조건**: 복잡한 알고리즘보다는 명확한 조건을 설정하여 테스트해야 결과 해석이 쉽고 오류를 빠르게 찾을 수 있습니다.
- **내부 데이터 구조 활용**: 소프트웨어 내부에서 데이터가 어떻게 저장되고 처리되는지를 고려하여 테스트 케이스를 작성합니다.
- **예상 오류 및 비정상 입력값 포함**: 사용자가 잘못된 값을 입력하거나 예상하지 못한 상황이 발생했을 때의 처리도 반드시 포함해야 합니다.

## 🏗️ 테스트의 단계와 유형

### 테스트의 4단계

소프트웨어 테스트는 개발 단계와 맞물려 순차적으로 이루어집니다. 마치 건물을 지을 때 기초공사 → 골조공사 → 내장공사 → 최종 검수 순서로 진행하듯이, 소프트웨어 테스트도 단계별로 수행됩니다.

#### 1단계: 단위 테스트 (Unit Test)

단위 테스트는 소프트웨어를 구성하는 **각각의 독립된 모듈(기능 단위)**을 개별적으로 테스트합니다. 마치 자동차를 조립하기 전에 엔진, 바퀴, 브레이크를 각각 따로 점검하는 것과 같습니다.

예를 들어 쇼핑몰 시스템이라면 다음과 같은 모듈이 독립적으로 존재합니다.

- 상품 소개 화면 모듈
- 주문 처리 모듈
- 배송 관리 모듈
- 마이페이지(포인트, 주문 내역, 회원 정보) 모듈

이 각각의 모듈을 따로 떼어내어 인터페이스, 자료 구조, 수행 경로, 오류 처리 등을 점검하는 것이 단위 테스트입니다.

#### 2단계: 통합 테스트 (Integration Test)

단위 테스트가 끝난 모듈들을 서로 연결하여 **모듈 간의 데이터 흐름과 인터페이스(연결 방식)**를 테스트합니다. 쇼핑몰 예시를 계속 들면, 상품 소개 → 주문 → 배송 → 마이페이지 반영이라는 일련의 흐름이 끊기지 않고 올바르게 연결되는지를 확인합니다.

통합 테스트 방식으로는 **빅뱅 방식**, **상향식(Bottom-Up) 방식**, **하향식(Top-Down) 방식**, **샌드위치(Sandwich) 방식** 등이 있습니다.

#### 3단계: 시스템 테스트 (System Test)

통합된 전체 시스템을 실제 운영 환경과 유사한 환경에 올려 테스트합니다. 이 단계에서는 다음과 같은 것들을 확인합니다.

- 보안은 제대로 동작하는가?
- 모바일, PC, 태블릿 등 다양한 환경에서 정상 동작하는가?
- 동시에 많은 사용자가 접속했을 때 시스템이 견디는가?

#### 4단계: 인수 테스트 (Acceptance Test)

인수 테스트는 고객이 최종적으로 소프트웨어를 **수락(인수)**할 수 있는지 판단하는 테스트입니다. **알파 테스트(개발사 내부에서 실제 사용자처럼 테스트)**와 **베타 테스트(실제 사용자 일부에게 배포하여 테스트)**로 구분됩니다. 인수 테스트를 통과해야 최종 납품이 이루어집니다.

### 테스트 단계와 개발 단계의 연관 관계

| 개발 단계 | 대응하는 테스트 단계 | 참고 자료 |
|---|---|---|
| 코딩 + 프로그램 설계 | 단위 테스트 | 실제 소스 코드 |
| 구조 설계 | 통합 테스트 | 설계 문서 및 아키텍처 |
| 요구사항 분석 | 시스템 테스트 | 요구사항 명세서 |
| 전체 범위 및 목표 | 인수 테스트 | 전체 프로젝트 목표 문서 |

### 테스트 목적에 따른 유형

테스트는 무엇을 확인하느냐에 따라 다양한 유형으로 나눌 수 있습니다.

- **기능 테스트(Functional Test)**: 입력값에 따라 출력값이 올바르게 나오는지, 기능이 정상적으로 동작하는지 확인합니다.
- **성능 테스트(Performance Test)**: 응답 시간, 처리량, 메모리 사용량, 처리 속도 등을 측정합니다.
- **스트레스 테스트(Stress Test)**: 자동차 타이어를 극한 조건에서 수천 시간 돌려 내구성을 테스트하듯, 시스템에 과부하(과도한 데이터나 요청)를 일부러 가하거나 반대로 최저 조건 이하로 떨어뜨려 시스템이 어떻게 반응하는지 확인합니다.
- **복잡도 테스트(Complexity Test)**: 데이터 처리 과정이 얼마나 복잡한지 측정합니다. 구조가 복잡할수록 유지보수 비용이 증가하기 때문에 중요합니다.

## 🔬 화이트박스 테스트와 블랙박스 테스트

소프트웨어 테스트의 방법론은 크게 두 가지로 나뉩니다. 바로 **화이트박스 테스트(White-Box Test)**와 **블랙박스 테스트(Black-Box Test)**입니다.

### 화이트박스 테스트 (White-Box Test)

**화이트박스**라는 이름처럼, 상자(소프트웨어) 안이 **훤히 들여다보이는** 테스트입니다. 소프트웨어의 **내부 코드와 논리적 구조를 직접 보면서** 테스트를 수행합니다.

다른 이름으로는 **구현 기반 테스트**, **코드 기반 테스트(Code-Based Test)**, **구조 기반 테스트(Structural Test)**라고도 합니다.

```python
# 화이트박스 테스트 예시: 1부터 100까지 합계를 구하는 함수
def sum_to_n(n):
    total = 0          # 합계를 저장할 변수 초기화
    for i in range(1, n + 1):  # 1부터 n까지 반복
        total += i     # 현재 숫자를 합계에 더함
    return total       # 최종 합계 반환

# 화이트박스 테스트 케이스: 코드의 모든 경로가 실행되도록 설계
# 케이스 1: 정상 입력 (모든 반복문이 실행되는지 확인)
result = sum_to_n(100)
print(f"입력: 100, 예상 결과: 5050, 실제 결과: {result}")
# → 5050이 출력되어야 함

# 케이스 2: 경계값 - 최솟값 입력
result = sum_to_n(1)
print(f"입력: 1, 예상 결과: 1, 실제 결과: {result}")
# → 1이 출력되어야 함

# 케이스 3: 경계값 - 0 입력 (반복문이 실행되지 않는 경우)
result = sum_to_n(0)
print(f"입력: 0, 예상 결과: 0, 실제 결과: {result}")
# → 0이 출력되어야 함
```

#### 화이트박스 테스트의 주요 기법

**① 기본 경로 테스트 (Basis Path Testing)**

원시 코드의 독립적인 처리 경로가 최소한 한 번씩 모두 실행되도록 테스트 케이스를 설계하는 방법입니다. 순서도(Flowchart)와 흐름 그래프(Flow Graph)를 만들고, 복잡도를 계산하여 독립적인 경로를 정의한 뒤 테스트 케이스를 작성합니다.

**② 문장 검증 기준 (Statement Coverage)**

프로그램 내의 **모든 문장(코드 한 줄 한 줄)**이 최소한 한 번씩 실행될 수 있도록 테스트 데이터를 구성하는 기준입니다.

```python
# 문장 검증 기준 예시
def calculate_grade(score):
    if score >= 90:            # 조건문 1
        return "A"             # 문장 1: score가 90 이상일 때 실행
    elif score >= 80:          # 조건문 2
        return "B"             # 문장 2: score가 80~89일 때 실행
    elif score >= 70:          # 조건문 3
        return "C"             # 문장 3: score가 70~79일 때 실행
    else:
        return "F"             # 문장 4: score가 70 미만일 때 실행

# ✅ 올바른 테스트 케이스: 모든 문장이 최소 한 번씩 실행됨
test_cases = [95, 85, 75, 60]  # A, B, C, F 각각 한 번씩 실행

# ❌ 잘못된 테스트 케이스: 일부 문장이 실행되지 않음
test_cases_bad = [95, 85]      # C와 F 문장은 실행되지 않음
```

**③ 분기 검증 기준 (Branch Coverage)**

조건문(if-else)의 **참(True)과 거짓(False)** 두 가지 분기가 모두 최소 한 번씩 실행되도록 테스트하는 기준입니다. 분기점과 합류점에서 오류가 발생할 가능성이 높기 때문에 이 부분에 집중합니다.

```python
# 분기 검증 기준 예시
def check_eligibility(age):
    if age >= 18:              # 분기점: 참 또는 거짓
        return "성인 - 이용 가능"   # 참(True)일 때 실행되는 분기
    else:
        return "미성년자 - 이용 불가"  # 거짓(False)일 때 실행되는 분기

# ✅ 올바른 테스트: 참과 거짓 분기를 모두 테스트
print(check_eligibility(20))   # 참 분기 테스트 → "성인 - 이용 가능"
print(check_eligibility(15))   # 거짓 분기 테스트 → "미성년자 - 이용 불가"

# ❌ 잘못된 테스트: 한 가지 분기만 테스트
print(check_eligibility(25))   # 참 분기만 테스트 → 거짓 분기의 결함을 발견 못함
```

**④ 조건 검증 기준 (Condition Coverage)**

두 개 이상의 개별 조건식이 존재할 때, 각 조건식의 참/거짓을 모두 최소 한 번씩 테스트하는 방법입니다.

```python
# 조건 검증 기준 예시: 두 조건이 모두 충족해야 할인 적용
def apply_discount(is_member, purchase_amount):
    # 조건 1: is_member (회원 여부)
    # 조건 2: purchase_amount >= 50000 (5만원 이상 구매)
    if is_member and purchase_amount >= 50000:
        return purchase_amount * 0.9   # 10% 할인 적용
    else:
        return purchase_amount         # 할인 없음

# ✅ 올바른 테스트: 각 조건의 참/거짓을 모두 테스트
print(apply_discount(True, 60000))    # 조건1=참, 조건2=참 → 할인 적용
print(apply_discount(True, 30000))    # 조건1=참, 조건2=거짓 → 할인 없음
print(apply_discount(False, 60000))   # 조건1=거짓, 조건2=참 → 할인 없음
print(apply_discount(False, 30000))   # 조건1=거짓, 조건2=거짓 → 할인 없음
```

### 블랙박스 테스트 (Black-Box Test)

**블랙박스**라는 이름처럼, 상자(소프트웨어) 안이 **보이지 않는** 채로 테스트합니다. 코드나 내부 알고리즘을 보지 않고, **요구사항 명세서나 설계 사양서**를 기반으로 테스트 케이스를 만들어 입력값과 출력값만으로 정상 동작 여부를 확인합니다.

다른 이름으로는 **명세 기반 테스트(Specification-Based Test)**라고도 합니다.

일반 사용자나 코드를 이해하지 못하는 테스터도 블랙박스 테스트는 수행할 수 있습니다. 코드 내부를 몰라도, 무엇을 입력했을 때 무엇이 나와야 하는지만 알면 되기 때문입니다.

#### 블랙박스 테스트의 주요 기법

**① Syntax 기법 (문법 기반 테스트)**

정해진 문법(규칙)에 따라 적합한 입력값과 부적합한 입력값을 모두 넣어보고, 예상 결과가 올바르게 나오는지 확인하는 방법입니다.

아래는 회원가입 유효성 검사를 Syntax 기법으로 테스트하는 예시입니다.

```python
# Syntax 기법 예시: 회원가입 유효성 검사

# 테스트 규칙(명세서):
# - 아이디: 4~8자, 알파벳 또는 숫자만 사용 가능
# - 비밀번호: 7자 이상, 알파벳+숫자+특수문자 포함 필수

import re

def validate_signup(user_id, password):
    """회원가입 유효성 검사 함수"""
    errors = []

    # 아이디 검사: 4~8자, 알파벳+숫자만 허용
    if not (4 <= len(user_id) <= 8):
        errors.append("아이디는 4~8자여야 합니다.")
    if not re.match(r'^[a-zA-Z0-9]+$', user_id):
        errors.append("아이디는 알파벳과 숫자만 사용 가능합니다.")

    # 비밀번호 검사: 7자 이상, 특수문자 포함 필수
    if len(password) < 7:
        errors.append("비밀번호는 7자 이상이어야 합니다.")
    if not re.search(r'[!@#$%^&*]', password):
        errors.append("비밀번호에 특수문자가 포함되어야 합니다.")

    return errors if errors else ["정상 등록"]

# ✅ 테스트 케이스 실행
test_cases = [
    ("user1234", "Pass@123"),   # 적합한 입력 → 정상 등록
    ("ab",       "Pass@123"),   # 아이디 너무 짧음 → 오류
    ("user1234!", "Pass@123"),  # 아이디에 특수문자 포함 → 오류
    ("user1234", "pass"),       # 비밀번호 너무 짧음 → 오류
    ("user1234", "Password1"),  # 비밀번호 특수문자 없음 → 오류
]

for uid, pwd in test_cases:
    result = validate_signup(uid, pwd)
    print(f"아이디: {uid!r}, 비밀번호: {pwd!r} → {result}")
```

| 아이디 | 비밀번호 | 예상 결과 |
|---|---|---|
| user1234 | Pass@123 | ✅ 정상 등록 |
| ab | Pass@123 | ❌ 아이디 4자 미만 오류 |
| user1234! | Pass@123 | ❌ 아이디 특수문자 오류 |
| user1234 | pass | ❌ 비밀번호 7자 미만 오류 |
| user1234 | Password1 | ❌ 비밀번호 특수문자 없음 오류 |

**② 동등 분할 기법 (Equivalence Partitioning)**

입력값을 비슷한 성질의 그룹(영역)으로 나누고, 각 영역에서 대표값을 하나씩 선택하여 테스트하는 방법입니다. 사용자가 가장 이해하기 쉽고 활용하기 편한 기법입니다.

학점 계산 및 인센티브 지급 시스템을 예시로 살펴보겠습니다.

```python
# 동등 분할 기법 예시: 평가 점수별 인센티브 계산

def calculate_incentive(score):
    """평가 점수에 따른 인센티브(상여금) 계산"""
    if score >= 90:
        return 600    # 600만원 (A등급)
    elif score >= 80:
        return 400    # 400만원 (B등급)
    elif score >= 70:
        return 200    # 200만원 (C등급)
    elif score >= 60:
        return 100    # 100만원 (D등급)
    else:
        return 0      # 0원 (F등급)

# ✅ 동등 분할 테스트 케이스: 각 영역에서 대표값 선택
test_cases = [
    (95, 600),   # 90점대 대표값 → 600만원 예상
    (85, 400),   # 80점대 대표값 → 400만원 예상
    (75, 200),   # 70점대 대표값 → 200만원 예상
    (65, 100),   # 60점대 대표값 → 100만원 예상
    (50, 0),     # 60점 미만 대표값 → 0원 예상
]

for score, expected in test_cases:
    actual = calculate_incentive(score)
    status = "✅ 통과" if actual == expected else "❌ 실패"
    print(f"점수: {score}, 예상: {expected}만원, 실제: {actual}만원 → {status}")
```

**③ 경계값 분석 (Boundary Value Analysis)**

경계값 분석은 경계에 있는 값을 집중적으로 테스트하는 방법입니다. 실제 시스템에서 경계값 처리 오류는 매우 흔하고, 눈에 잘 띄지 않습니다. 마치 체중 제한이 100kg인 엘리베이터에 99kg, 100kg, 101kg을 각각 태워보는 것처럼, 경계 직전·경계·경계 직후 값을 모두 테스트합니다.

```python
# 경계값 분석 예시: 성적 입력 유효성 검사 (0~100점만 유효)

def validate_score(score):
    """성적 유효성 검사 - 0점 이상 100점 이하만 유효"""
    if 0 <= score <= 100:
        return f"유효한 점수: {score}점"
    else:
        return f"오류: {score}점은 유효 범위(0~100)를 벗어났습니다."

# ✅ 경계값 테스트 케이스: 경계 이전, 경계, 경계 이후 값 모두 테스트
boundary_test_cases = [
    -1,    # 하한 경계 이전값 (유효 범위 이탈) → 오류 예상
    0,     # 하한 경계값 (최솟값) → 유효 예상
    1,     # 하한 경계 이후값 → 유효 예상
    50,    # 중간값 → 유효 예상
    99,    # 상한 경계 이전값 → 유효 예상
    100,   # 상한 경계값 (최댓값) → 유효 예상
    101,   # 상한 경계 이후값 (유효 범위 이탈) → 오류 예상
]

# ❌ 잘못된 테스트 케이스: 경계값을 무시하고 중간값만 테스트
bad_test_cases = [50, 75, 90]  # 경계에서 발생하는 결함을 놓칠 수 있음

print("=== 경계값 분석 테스트 결과 ===")
for score in boundary_test_cases:
    print(validate_score(score))
```

| 입력값 | 분류 | 예상 결과 |
|---|---|---|
| -1 | 하한 경계 이전 | ❌ 오류 메시지 출력 |
| 0 | 하한 경계값 | ✅ 유효한 점수 |
| 1 | 하한 경계 이후 | ✅ 유효한 점수 |
| 100 | 상한 경계값 | ✅ 유효한 점수 |
| 101 | 상한 경계 이후 | ❌ 오류 메시지 출력 |

**④ 원인-결과 그래프 (Cause-Effect Graph)**

동등 분할이나 경계값 분석은 단일 입력값에 대한 테스트에는 강하지만, **여러 입력 조건이 복합적으로 맞물릴 때**는 완전한 테스트가 어렵습니다.

예를 들어, 인센티브 지급이 단순히 점수만이 아니라 "점수가 70점 이상이면서 지역이 서울이고 남성인 경우"처럼 여러 조건이 복합된 경우에는 원인-결과 그래프가 필요합니다.

원인-결과 그래프는 다음 순서로 작성합니다.

1. **원인(입력 조건)과 결과(출력 결과)를 파악**합니다.
2. **원인-결과 그래프를 작성**합니다.
3. 그래프에 **제한 조건을 표시**합니다.
4. **의사결정 테이블(Decision Table)**로 변환합니다.
5. **최종 테스트 시나리오를 완성**합니다.

예를 들어 출입 통제 시스템의 경우를 봅시다.

| 원인(조건) | 결과(출력) |
|---|---|
| 교수이고 + 정상 인증 + 내부 구성원 | ✅ 출입문 개방 |
| 학생이고 + 정상 발급 + 내부 구성원 | ✅ 출입문 개방 |
| 교수이고 + 인증 실패 + 내부 구성원 | ❌ 비밀번호 오류 메시지(555) 표시 |

이처럼 복수의 조건이 조합될 때의 결과를 체계적으로 정리하고 테스트하는 것이 원인-결과 그래프 기법입니다.

### 화이트박스 vs 블랙박스 비교 정리

| 구분 | 화이트박스 테스트 | 블랙박스 테스트 |
|---|---|---|
| **다른 이름** | 구현 기반, 코드 기반, 구조 기반 | 명세 기반 |
| **내부 코드 확인** | ✅ 가능 (코드를 직접 봄) | ❌ 불가 (코드를 보지 않음) |
| **테스트 기준** | 내부 논리 구조, 제어 흐름 | 요구사항 명세서, 입출력 명세 |
| **수행 주체** | 주로 개발자 또는 개발 관련 테스터 | 테스터, 사용자 등 누구나 가능 |
| **주요 기법** | 기본 경로, 문장/분기/조건 검증 | Syntax, 동등 분할, 경계값 분석, 원인-결과 그래프 |
| **발견 용이 결함** | 논리 오류, 경로 누락 | 기능 오류, 요구사항 미반영 |

## ⚠️ 주의사항과 실전 팁

### 인스펙션(Inspection, 정적 검사)과 테스팅(동적 테스트)의 차이

테스트를 수행하는 방식은 크게 두 가지로 나눌 수 있습니다.

- **인스펙션(Inspection, 정적 기술)**: 소프트웨어를 실제로 실행하지 않고, 코드나 명세서를 검토하여 오류를 찾는 방법입니다. 주로 자동화된 소스코드 분석 도구를 사용하며, 프로그램과 명세서의 일치 여부만 점검할 수 있습니다. 실행 없이 진행되기 때문에 한계가 있습니다.

- **테스팅(Testing, 동적 기술)**: 실제 데이터를 소프트웨어에 입력하고 실행하여 결과를 확인하는 방법입니다. 인스펙션보다 훨씬 광범위하고, 실제 시스템의 품질을 더 효과적으로 보장할 수 있습니다.

:::danger 절대 하지 말아야 할 것
- 개발자가 자신이 만든 코드를 혼자서 테스트하는 것
- 테스트 케이스 없이 즉흥적으로 테스트하는 것
- "오류가 안 나왔으니 완벽한 프로그램"이라고 결론 내리는 것
- 일부 경로만 테스트하고 모든 경로가 검증되었다고 판단하는 것
:::

### 리그레션 테스트 (Regression Test, 회귀 테스트)

결함을 수정한 후에는 반드시 **리그레션 테스트(회귀 테스트)**를 수행해야 합니다. 결함을 수정하면서 기존에 정상적으로 동작하던 다른 기능이 영향을 받아 새로운 오류가 발생할 수 있기 때문입니다. 마치 하나의 부품을 고쳤더니 다른 부품이 망가지는 것처럼요. 수정 후에는 반드시 이전에 통과했던 테스트 케이스를 다시 실행해 보아야 합니다.

:::tip 효율적인 테스트를 위한 실전 팁
1. **테스트 케이스를 문서로 남기세요**: 구두로 "이 값 넣어봤어요"보다 문서로 정리된 테스트 케이스가 훨씬 신뢰도 높습니다.
2. **예상 결과(테스트 오라클)를 반드시 명시하세요**: "어떤 결과가 나와야 한다"를 미리 정해두지 않으면 테스트 결과를 제대로 평가할 수 없습니다.
3. **비정상 입력값도 반드시 포함하세요**: 0 나누기, 음수 입력, 범위 초과 값 등 개발자가 피하고 싶은 값을 반드시 테스트해야 합니다.
4. **테스트 결과 보고서를 작성하세요**: 발견된 결함, 수정 여부, 재테스트 결과 등을 체계적으로 기록해 두어야 합니다.
5. **최소한 하나 이상의 독립 경로가 반드시 테스트되어야 합니다**: 어떤 모듈이든 모든 독립 경로가 최소 한 번은 실행되어야 신뢰할 수 있는 테스트라 할 수 있습니다.
:::

## 📌 핵심 정리

이번 문서에서 배운 소프트웨어 테스트의 핵심 개념을 정리합니다.

- **소프트웨어 테스트**는 숨어 있는 결함을 찾아내기 위해 정해진 조건으로 프로그램을 검사하고 평가하는 과정이다.
- 테스트는 결함이 없음을 증명하는 것이 아니라, **결함을 찾아내는** 과정이다.
- **40-20-40 법칙**: 소프트웨어 개발 노력의 40%는 분석·설계, 20%는 구현, 40%는 테스트에 투입해야 한다.
- **V&V(Verification and Validation)**: 개발자 관점의 검증(Verification)과 사용자 관점의 확인(Validation) 두 가지를 모두 수행해야 한다.
- **독립된 테스트팀**이 테스트를 수행하는 것이 가장 효율적이다. 개발자는 자신의 코드를 직접 테스트하면 결함을 놓치기 쉽다.
- 테스트는 특정 단계가 아니라 **개발의 모든 단계**에서 수행되어야 한다.
- 테스트 단계: **단위 테스트 → 통합 테스트 → 시스템 테스트 → 인수 테스트** 순서로 진행된다.
- **화이트박스 테스트**: 내부 코드를 보고 논리 구조를 검증. 주요 기법: 기본 경로, 문장/분기/조건 검증.
- **블랙박스 테스트**: 코드를 보지 않고 입출력으로 검증. 주요 기법: Syntax, 동등 분할, 경계값 분석, 원인-결과 그래프.
- **경계값 분석**은 경계 직전·경계·경계 직후 값을 집중 테스트하여 눈에 띄지 않는 결함을 발견하는 데 효과적이다.
- **인스펙션(정적)**은 실행 없이 코드를 검토하고, **테스팅(동적)**은 실제 실행을 통해 결과를 확인한다.
- 결함 수정 후에는 반드시 **리그레션 테스트(회귀 테스트)**로 새로운 오류 발생 여부를 확인해야 한다.
- 추가로 발견될 결함의 수는 이미 발견된 결함 수에 **비례**한다.
