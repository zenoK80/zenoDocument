# 프로그래밍의 기본 원칙

이번 시간은 **좋은 프로그래밍의 기본 원칙**을 배웁니다. 이 원칙들을 지키면 전문적인 프로그래머가 될 수 있습니다!

## 원칙 1️⃣ : 설계명세서를 완벽하게 반영

### 건물 짓기의 예시

건물을 지을 때를 생각해봅시다:
- 설계도면이 없으면 어떻게 건물을 지을까요?
- 설계도면에 빠진 부분이 있으면 어떻게 될까요?
- 완성된 건물이 설계도와 다르면?

**결과: 문제가 생깁니다!**

### 프로그래밍도 똑같습니다

**설계명세서란?**
사용자의 다양한 요구사항을 철저히 분석해서 만든 문서입니다.

**설계명세서에는:**
```
1. 입력(Input) 방식
2. 처리(Process) 방식
3. 출력(Output) 형태
4. 성능 요구사항
5. 예외 상황 처리 방법
```

:::warning
설계명세서를 완벽하게 반영하지 않으면 완성된 프로그램이 사용자의 요구와 맞지 않게 됩니다!
:::

## 원칙 2️⃣ : 원시코드는 간단해야 한다 (심플 is 베스트)

### 같은 결과를 다르게 만들 수 있다

**상황:** 1부터 100까지의 합을 구하는 프로그램

100명의 학생이 각각 10분씩 시간을 가지고 이 프로그램을 만들면:
- 학생 A의 코드: 간단하고 명확함 ✅
- 학생 B의 코드: 매우 복잡하고 이해 어려움 ❌
- 학생 C의 코드: 불필요한 부분이 너무 많음 ❌

**같은 결과인데 코드는 완전히 다릅니다!**

### 좋은 코드의 특징

```cpp
// ❌ 나쁜 예 (복잡함)
int a=0,b=1,c=2,d=3,e=4,f=5,g=6,h=7,i=8,j=9;
int result = a+b+c+d+e+f+g+h+i+j;
// ... 이런 식으로 100까지...

// ✅ 좋은 예 (간단함)
int sum = 0;
for (int i = 1; i <= 100; i++) {
    sum = sum + i;
}
```

**간단한 코드의 장점:**
- 📖 읽기 쉬움
- 🔧 수정하기 쉬움
- 🧪 테스트하기 쉬움
- 🐛 디버깅하기 쉬움
- 👥 다른 개발자가 이해하기 쉬움

### 복잡한 코드의 문제점

"나만 알아볼 수 있게" 복잡하게 만들면:
- 오류가 생겼을 때 **어디서 오류가 났는지 찾기 어려움**
- 6개월 뒤 자신도 **무엇을 한 건지 모름**
- 새로운 팀원이 **코드를 이해하지 못함**
- 유지보수 비용이 **엄청 증가**

:::tip
복잡한 것이 아니라 **간단하면서도 효율적인 코드**가 좋은 코드입니다!
:::

## 원칙 3️⃣ : 디버깅이 쉬워야 한다

### 디버깅이란?

**디버깅(Debugging)** = 오류(Bug)를 찾아서 수정하는 작업

:::info
"버그(Bug)"라는 단어의 유래: 옛날 컴퓨터에 나방이 들어가서 오류를 일으킨 사건에서 유래했다고 합니다!
:::

### 간단한 코드 = 디버깅 쉬움

```cpp
// ❌ 복잡해서 디버깅 어려운 코드
int x=0,y=0,z=0;z=x+y+z*x+y;if(z>0){...}

// ✅ 간단해서 디버깅 쉬운 코드
int firstValue = 0;
int secondValue = 0;
int result = firstValue + secondValue;
if (result > 0) { ... }
```

**간단한 코드는:**
- 어디가 잘못됐는지 **한눈에 보임**
- **빠르게 수정 가능**
- 다시 오류가 생길 **위험이 적음**

---

## 원칙 4️⃣ : 수정이 용이해야 한다

### 디버깅 vs 수정의 차이

:::info
이 둘은 다릅니다!
:::

| 항목 | 디버깅 | 수정(변경) |
|------|--------|---------|
| **목적** | 오류를 고침 | 기능을 변경 |
| **원인** | 코드의 버그 | 사용자 요구사항 변경 |
| **예시** | 계산 결과가 틀림 | 새로운 기능 추가 |

### 실제 상황

**상황:** 계산 프로그램을 만들었는데 고객이 새로운 기능을 요청함

- **단순하고 명확한 코드**: "네, 2일이면 가능합니다" ✅
- **복잡한 코드**: "코드를 분석해야 해서 1주일 필요합니다" ❌

**개발 비용 차이가 엄청남!** 💰

:::warning
프로그래머의 가치는 **빠르고 효율적으로 수정할 수 있는 능력**입니다!
:::

## 원칙 5️⃣ : 테스트가 쉬워야 한다

### 테스트(Testing)란?

프로그램이 제대로 작동하는지 확인하는 작업입니다.

### 좋은 코드는 테스트가 쉽다

```cpp
// ✅ 테스트하기 쉬운 코드
int Add(int a, int b) {
    return a + b;
}

// 테스트
if (Add(10, 20) == 30) printf("OK");  // 한 줄로 테스트 가능

// ❌ 테스트하기 어려운 코드
void DoSomething() {
    // 100줄의 복잡한 코드...
    // 이게 맞는지 확인하기 어려움
}
```

## 실제 사례: 가전제품 설계

### 복잡한 설계 vs 간단한 설계

**스마트폰, TV, 자동차** 같은 제품들을 봅시다:

- 내부 부품은 **아주 복잡**하지만
- 동작은 **아주 간단**합니다

### 왜 그럴까?

```
간단한 작동 = 고장 안 남 = 유지보수 쉬움
```

**결과:**
- ✅ 고객 만족도 높음
- ✅ 수리 비용 적음
- ✅ 신뢰도 높음

:::success
프로그래밍도 마찬가지입니다! 복잡한 기능도 **간단하게 보이도록** 만들어야 합니다.
:::

## 프로그래머가 되기 위한 자세

### 체크리스트 ✅

프로그래밍을 할 때마다 이 질문들을 스스로에게 던져보세요:

1. **"이 코드를 좀 더 간단하게 할 수는 없을까?"**
2. **"다른 개발자가 이 코드를 쉽게 이해할 수 있을까?"**
3. **"이 코드가 정말 필요한 기능만 하고 있을까?"**
4. **"이 코드는 수정하기 쉬울까?"**
5. **"설계명세서의 모든 요구사항을 반영했을까?"**

:::tip
시키는 일만 하는 사람은 **코더**이고, 이런 질문을 스스로 하는 사람이 **프로그래머**입니다!
:::

### 전문가의 길

```
단순한 코딩 일 처리
        ↓
내 코드를 끊임없이 개선
        ↓
설계 단계부터 고민
        ↓
전체 시스템을 생각
        ↓
IT 전문가로 성장 ⭐
```

## 정리하며 🎯

### 프로그래밍의 5가지 기본 원칙

| 원칙 | 설명 |
|------|------|
| **1. 명세서 반영** | 요구사항을 완벽하게 코드에 반영 |
| **2. 간단함** | 간단하고 명확한 코드 작성 |
| **3. 디버깅 용이** | 오류를 빨리 찾을 수 있게 |
| **4. 수정 용이** | 변경 요구사항을 빠르게 반영 |
| **5. 테스트 용이** | 제대로 작동하는지 쉽게 확인 |

:::success
이 5가지 원칙을 지키면, 여러분도 전문적인 프로그래머가 될 수 있습니다! 🎉
:::
