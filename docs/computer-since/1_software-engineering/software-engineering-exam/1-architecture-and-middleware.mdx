# 소프트웨어 아키텍처와 미들웨어 완전 정복 🏗️

소프트웨어 공학 시험에서 자주 출제되는 핵심 개념들을 하나씩 파헤쳐 봅시다. 이 문서에서는 SOA, WAS, MDD부터 시작해서 테스트 기법, 디자인 패턴, UML 다이어그램까지 초보자도 쉽게 이해할 수 있도록 차근차근 설명합니다. 단순 암기가 아니라 **"왜 이게 필요한가?"** 를 중심으로 이해하면서 공부해 봐요.

---

## 1️⃣ 소프트웨어 아키텍처 핵심 개념: SOA와 WAS

### 🧩 SOA란 무엇인가?

**SOA(Service Oriented Architecture, 서비스 지향 아키텍처)** 는 기존 애플리케이션의 기능들을 **비즈니스 단위로 묶어 표준화된 인터페이스(규격화된 통신 창구)로 서비스를 구현하고, 이를 조합하여 하나의 애플리케이션을 구성하는 소프트웨어 개발 방식**입니다.

비유를 들어볼게요. 여러분이 레고 블록으로 집을 짓는다고 상상해 보세요. 지붕 블록, 벽 블록, 창문 블록이 각각 독립적으로 존재하고, 이걸 조합해서 집을 완성합니다. SOA도 마찬가지예요. "로그인 기능", "결제 기능", "알림 기능" 같은 것들을 각각 독립적인 서비스로 만들어 두고, 필요할 때마다 조합해서 큰 시스템을 구성합니다. 덕분에 한 서비스를 고쳐도 다른 서비스에 영향을 최소화할 수 있어요.

> **핵심 포인트**: SOA는 "기능을 서비스 단위로 분리하고, 표준 인터페이스로 조합한다"는 철학입니다.

### 🏗️ SOA 아키텍처 단계 모델

SOA는 단계별로 발전하는 모델이 있습니다. 시험에서 자주 헷갈리는 부분이니 꼼꼼히 확인하세요.

| 단계 | 이름 | 특징 |
|------|------|------|
| 1단계 | **Fundamental SOA** | 기존 시스템들을 서비스화하여 통합. 시스템 크기가 커질수록 서비스 간 연결이 **매우 복잡**해짐 |
| 2단계 | **Networked SOA** | 복잡도 해결을 위해 **ESB(Enterprise Service Bus, 중앙 허브)**를 도입. 서비스들이 허브를 통해 연결됨 |
| 3단계 | **Process Oriented SOA** | 비즈니스 프로세스(업무 흐름) 중심으로 서비스를 조합 |
| 4단계 | **Business SOA** | 비즈니스 전략과 IT가 완전히 정렬된 최고 성숙 단계 |

:::warning ⚠️ 시험 함정 주의
**7번 문제**에서 많이 틀리는 부분입니다! "기존 시스템들을 서비스화하여 통합하는 단계이며, 서비스 간 연결이 복잡해지는 모델"은 **Networked SOA(②번)가 아니라 Fundamental SOA(①번)**입니다.

Networked SOA는 오히려 그 복잡도를 **해결**하기 위해 ESB라는 중앙 허브를 도입하는 단계예요. 문제에서 "복잡해진다"는 키워드가 나오면 Fundamental SOA를 선택하세요.
:::

### 🖥️ WAS(Web Application Server)란 무엇인가?

**WAS(Web Application Server, 웹 애플리케이션 서버)** 는 사용자 요구에 따라 변하는 **동적 콘텐츠(사람마다 다른 화면)**를 처리하기 위한 **미들웨어(중간에서 연결해주는 소프트웨어)**입니다.

일반 웹 서버(Web Server)와 WAS의 차이를 이해하는 게 핵심이에요. 비유를 들어볼게요.

- **웹 서버**: 도서관의 **책장**과 같습니다. "HTML 파일 주세요"라고 하면 항상 같은 파일을 그대로 꺼내서 줍니다. 누가 와도 똑같은 내용이에요. 이것이 **정적 콘텐츠(Static Content)**입니다.
- **WAS**: 도서관의 **사서 선생님**과 같습니다. "홍길동님, 지난번에 빌린 책 반납일이 내일이에요!"처럼 사람마다 다른 맞춤 응답을 해줍니다. 로그인 후 "홍길동님 환영합니다"처럼 **사람마다 다른 화면**을 만들어 내는 게 WAS가 하는 일입니다. 이것이 **동적 콘텐츠(Dynamic Content)**입니다.

| 구분 | 웹 서버 (Web Server) | WAS (Web Application Server) |
|------|---------------------|------------------------------|
| 처리 내용 | 정적 콘텐츠 (HTML, 이미지, CSS 등) | 동적 콘텐츠 (로그인, 결제, 개인화 화면 등) |
| 예시 | Apache, Nginx | Tomcat, JBoss, WebLogic |
| 특징 | 빠르고 단순함 | 비즈니스 로직(업무 처리 규칙) 포함 |

:::info 💡 미들웨어(Middleware)란?
**미들웨어**는 "가운데(Middle)"라는 말에서 왔어요. 클라이언트(사용자)와 데이터베이스(저장소) 사이에서 중간 역할을 하는 소프트웨어입니다. 마치 음식점에서 손님과 주방 사이에 있는 **서버(종업원)**처럼요. WAS는 대표적인 미들웨어 중 하나입니다.
:::

### 🔄 MDD(Model Driven Development)란 무엇인가?

**MDD(Model Driven Development, 모델 주도 개발)**는 코드를 직접 짜는 대신 **모델(설계도)**을 먼저 만들고, 그 모델에서 자동으로 코드를 생성하는 개발 방법론입니다.

MDD의 프로세스 순서는 시험에서 단골로 출제됩니다. 반드시 외워두세요!

```
CIM → PIM → PSM → Code
```

각 단계를 자세히 살펴볼게요.

| 단계 | 이름 | 의미 | 비유 |
|------|------|------|------|
| **CIM** | Computation Independent Model | 비즈니스 분석 단계. 시스템이 어떤 일을 해야 하는지 정의. 기술적 내용 없음 | 건물을 짓기 전에 "이 건물이 어떤 용도인가?" 결정 |
| **PIM** | Platform Independent Model | 플랫폼(특정 기술)에 독립적인 설계. 어떻게 동작할지 설계하되 Java인지 Python인지는 아직 결정 안 함 | "건물 평면도 그리기" – 재질은 미결정 |
| **PSM** | Platform Specific Model | 특정 플랫폼(Java, .NET 등)에 맞춘 설계 | "이 벽은 철근 콘크리트로 짓겠다" 결정 |
| **Code** | 코드 생성 | PSM을 바탕으로 실제 코드 자동 생성 | 실제 건물 시공 |

:::tip 💡 MDD 순서 외우는 꿀팁
**"CIM → PIM → PSM → Code"** 를 외울 때는 이렇게 기억하세요:
"**C**onceptual → **P**latform-**I**ndependent → **P**latform-**S**pecific → **C**ode"
비즈니스 개념 → 독립 설계 → 종속 설계 → 실제 코드 순서입니다.
:::

## 2️⃣ 소프트웨어 테스트의 모든 것 🧪

### 🔍 테스팅 vs 디버깅: 뭐가 다른가?

소프트웨어 개발 과정에서 **테스팅(Testing)**과 **디버깅(Debugging)**은 자주 혼동되는 개념입니다.

- **테스팅(Testing)**: 소프트웨어에 **에러(오류)가 있는지 찾아내는** 작업입니다. 마치 건강검진을 받는 것처럼 "어디가 나쁜지" 발견하는 역할입니다. 테스터(검사관)는 버그를 찾아 보고서를 씁니다.
- **디버깅(Debugging)**: 발견된 에러를 **실제로 수정하는** 작업입니다. 건강검진에서 병이 발견된 후 치료를 받는 것과 같습니다. 개발자(수리공)가 원인을 분석하고 코드를 고칩니다.

> **핵심 암기**: 테스팅 = 에러 **찾기** 🔍 / 디버깅 = 에러 **수정** 🔧

### 📦 블랙박스 테스트 vs 화이트박스 테스트

소프트웨어 테스트는 크게 두 가지 방식으로 나뉩니다.

**블랙박스 테스트(Black Box Test)**는 소프트웨어의 내부 구조를 모르는 상태에서 **입력과 출력**만으로 테스트하는 방법입니다. 마치 냉장고를 사용할 때 내부 회로를 모르고 "버튼을 누르면 얼음이 나온다"는 결과만 확인하는 것처럼요.

**화이트박스 테스트(White Box Test)**는 소프트웨어의 내부 구조(코드, 로직 흐름)를 보면서 테스트하는 방법입니다. 마치 자동차 엔지니어가 엔진 덮개를 열고 각 부품을 하나씩 점검하는 것과 같아요.

| 구분 | 블랙박스 테스트 | 화이트박스 테스트 |
|------|----------------|-----------------|
| 내부 구조 확인 | ❌ 모름 | ✅ 알고 있음 |
| 주요 기법 | 동등 분할, 경곗값 분석, 문법 오류 검사 | **기본 경로 검사**, 조건 검사, 루프 검사 |
| 담당자 | 주로 테스터(비개발자 가능) | 주로 개발자 |

:::warning ⚠️ 시험 함정 주의
**12번 문제**에서 "블랙박스 테스트 기법이 아닌 것"을 묻는 문제에서 **기본 경로 검사(Basic Path Testing)**는 **화이트박스 테스트 기법**입니다! 동등 분할, 경곗값 분석, 문법 오류 검사는 모두 블랙박스 기법이에요.
:::

### 🔼🔽 상향식 vs 하향식 통합 테스트

여러 모듈(기능 단위)을 합칠 때 어떤 순서로 테스트하느냐에 따라 두 가지 방식이 있습니다.

**하향식 통합 테스트(Top-Down Integration Test)**는 상위 모듈(가장 중요한 메인 모듈)부터 시작해서 아래로 내려가며 테스트하는 방식입니다. 이때 아직 개발되지 않은 하위 모듈 대신 **스텁(Stub, 가짜 하위 모듈)**을 사용합니다. 스텁은 "나 여기 있어요!"라고 빈 응답만 해주는 임시 대역 배우 같은 존재입니다.

**상향식 통합 테스트(Bottom-Up Integration Test)**는 하위 모듈부터 시작해서 위로 올라가며 테스트하는 방식입니다. 이때 상위 모듈을 대신하는 **드라이버(Driver, 가짜 상위 모듈)**가 필요합니다. 드라이버는 하위 모듈을 실행시켜주는 "운전사" 역할입니다.

| 구분 | 하향식 통합 | 상향식 통합 |
|------|------------|------------|
| 시작 방향 | 상위(메인) → 하위 | 하위(세부) → 상위 |
| 필요한 도구 | **스텁(Stub)** | **드라이버(Driver)** |
| 특징 | 전체 골격을 조기에 테스트 가능 | 실제 모듈을 바로 테스트 가능 |

:::danger ❌ 절대 혼동하지 마세요
- **드라이버(Driver)** = **상향식** 통합 테스트에서 사용
- **스텁(Stub)** = **하향식** 통합 테스트에서 사용

13번 문제에서 "하향식 통합 테스트를 위해 **드라이버** 모듈이 필요하다"는 보기는 **완전히 틀린 설명**입니다. 하향식에는 스텁이 필요합니다!
:::

### 🛠️ 테스트 하니스(Test Harness)란?

**테스트 하니스(Test Harness)**는 시스템의 일부 기능만 시험하기 위하여 소프트웨어를 **임시로 변경**하는 것으로, 검사가 끝나면 **제거**하는 기법입니다.

"하니스"는 원래 말(馬)의 마구(끈과 고리 장치)를 뜻하는 단어입니다. 테스트를 위해 시스템에 임시로 "장치"를 달아두었다가 테스트 후 떼어내는 것과 같아요. 마치 자동차 충돌 테스트를 할 때 더미(인형)를 태우고 테스트하고, 테스트 후에는 더미를 제거하는 것과 비슷합니다.

:::note 📝 테스트 스텁과의 차이점
- **테스트 스텁(Test Stub)**: 하향식 통합 테스트에서 아직 구현되지 않은 **하위 모듈**을 대신하는 가짜 모듈입니다.
- **테스트 하니스(Test Harness)**: 특정 기능을 테스트하기 위해 **임시로 소프트웨어를 변경**하고 테스트 후 제거하는 더 포괄적인 개념입니다.

3번 문제의 정답은 **테스트 스텁이 아니라 테스트 하니스**입니다!
:::

## 3️⃣ 코드 품질과 유지보수 🔧

### ✨ 리팩토링(Refactoring)이란?

**리팩토링(Refactoring)**은 **결과(외부 동작)의 변경 없이** 코드의 내부 구조를 재조정하여 **가독성을 높이고 유지보수를 쉽게 하는 기술**입니다.

음식점 비유로 이해해 봐요. 손님에게 "된장찌개"를 제공한다는 결과는 똑같은데, 주방 내부를 더 효율적으로 정리하고 조리 순서를 개선하는 것과 같습니다. 손님 입장에서는 똑같은 맛의 된장찌개를 먹지만, 주방장 입장에서는 훨씬 깔끔하고 빠르게 요리할 수 있게 됩니다.

```python
# ❌ 리팩토링 전: 읽기 어렵고 중복이 많은 코드
def calc(a, b, c):
    x = a * a + b * b  # 뭘 계산하는지 불명확
    y = c * c
    if x == y:
        return True
    else:
        return False

# ✅ 리팩토링 후: 명확하고 간결한 코드
def is_pythagorean_triple(side_a, side_b, hypotenuse):
    """피타고라스 정리 검증: a² + b² = c² 인지 확인"""
    sum_of_squares = side_a ** 2 + side_b ** 2  # 두 변의 제곱합
    hypotenuse_square = hypotenuse ** 2          # 빗변의 제곱
    return sum_of_squares == hypotenuse_square   # 같으면 True 반환
```

> **핵심**: 리팩토링은 기능은 그대로, 코드 구조만 개선합니다. "같은 결과, 더 나은 코드" 💡

### 🔬 역공학(Reverse Engineering)이란?

**역공학(Reverse Engineering)**은 원래 하드웨어 분야에서 **완제품으로부터 설계를 해독해내는 작업**을 의미하며, 소프트웨어에서는 **시스템을 분석하여 컴포넌트와 관계를 찾는 작업**입니다.

마치 마트에서 산 과자를 먹어보고 "이 성분이 뭘까?" 분석해서 레시피를 역으로 추측하는 것처럼, **완성된 시스템 → 설계도 파악** 방향으로 진행하는 것이 역공학입니다.

반면 **재공학(Re-Engineering)**은 기존 시스템을 분석한 후 **더 나은 시스템으로 다시 만드는** 것으로, 역공학 + 개선이 결합된 개념입니다.

| 구분 | 역공학 (Reverse Engineering) | 재공학 (Re-Engineering) |
|------|------------------------------|------------------------|
| 방향 | 완제품 → 설계도 파악 | 분석 → 개선된 시스템 재구축 |
| 목적 | 설계 이해, 문서화 | 시스템 현대화, 개선 |
| 결과물 | 설계 문서, 명세서 | 새로운(개선된) 시스템 |

### 💸 소프트웨어 유지보수 비용

소프트웨어 유지보수 비용에 대한 중요한 사실이 있습니다.

> **소프트웨어는 오래 사용할수록 유지보수 비용이 증가합니다!**

많은 사람이 "오래 쓰면 익숙해지니까 비용이 줄지 않을까?" 라고 생각하지만, 실제로는 반대입니다. 시간이 지날수록 코드 구조가 복잡해지고, 원래 개발자가 퇴사하거나, 기술이 노후화되어 유지보수가 점점 어렵고 비용이 많이 들게 됩니다.

또한 분석(요구사항 정의) 단계에서 사용자가 원하는 것을 **정확히 파악하는 것**이 유지보수 비용을 줄이는 데 가장 중요합니다. 처음에 잘못 만들면 나중에 고치는 비용이 몇 배로 늘어납니다.

## 4️⃣ 디자인 패턴 완전 이해 🎨

### 🏷️ 디자인 패턴이란?

**디자인 패턴(Design Pattern)**은 소프트웨어 설계에서 자주 반복되는 문제들에 대한 **검증된 해결책을 유형별로 템플릿화한 것**입니다. 마치 건축에서 "아파트 평면도 표준안"처럼, 비슷한 문제가 발생했을 때 이미 검증된 설계 방식을 재사용하는 것입니다.

디자인 패턴의 장점:
- ✅ 재사용을 통해 **개발 시간 단축**
- ✅ 설계 변경 요청에 대한 **유지보수가 쉬움** (시험에서 "어렵다"고 나오면 오답!)
- ✅ 설계자 간 **원활한 의사소통** 도구로 활용 가능

:::warning ⚠️ 시험 함정 주의
23번 문제에서 "설계 변경 요청에 대한 유지보수가 **어렵다**"는 보기는 **완전히 틀린 설명**입니다. 디자인 패턴의 가장 큰 장점 중 하나가 바로 **유지보수가 쉬워진다**는 것입니다!
:::

### 🗂️ 디자인 패턴의 3가지 분류

디자인 패턴은 크게 세 가지로 분류됩니다. 시험에서 어느 패턴이 어느 분류에 속하는지 자주 출제됩니다.

| 분류 | 목적 | 주요 패턴 |
|------|------|-----------|
| **생성(Creational) 패턴** | 객체를 어떻게 만들지 | Factory Method, Abstract Factory, **Singleton**, Builder, Prototype |
| **구조(Structural) 패턴** | 클래스/객체를 어떻게 조합할지 | **Flyweight**, Adapter, Bridge, Composite, Decorator, Facade, Proxy |
| **행위(Behavioral) 패턴** | 객체 간 의사소통 방식 | **State**, Interpreter, **Command**, **Iterator**, Observer, Strategy, Template Method |

:::danger ❌ 절대 혼동 금지
- **Flyweight** = **구조(Structural)** 패턴 → 생성 패턴이 아님! (24번 정답)
- **Factory Method** = **생성(Creational)** 패턴 → 행위 패턴이 아님! (25번 정답)

"24번: 생성 패턴이 아닌 것 = Flyweight(구조 패턴)"
"25번: 행위 패턴이 아닌 것 = Factory Method(생성 패턴)"
:::

### 🚦 상태 패턴(State Pattern) 자세히 알기

**상태 패턴(State Pattern)**은 같은 동작을 **객체의 상태에 따라 다르게 처리**하는 디자인 패턴으로, **객체 상태를 클래스로 생성하여 참조**하게 합니다.

신호등을 예로 들어볼게요. 신호등에 "신호 변경" 이벤트가 발생했을 때:
- **빨간 불 상태**일 때 → 초록 불로 바뀜
- **초록 불 상태**일 때 → 노란 불로 바뀜
- **노란 불 상태**일 때 → 빨간 불로 바뀜

같은 "신호 변경" 동작인데 **현재 상태에 따라 결과가 달라집니다**. 상태 패턴을 사용하면 각 상태를 별도의 클래스로 만들어 관리하기 때문에 코드가 훨씬 깔끔해집니다.

```python
# ✅ 상태 패턴 예시: 신호등
class State:
    """상태 기본 클래스 (인터페이스 역할)"""
    def change(self, traffic_light):
        pass  # 각 상태 클래스에서 구현

class RedState(State):
    """빨간 불 상태"""
    def change(self, traffic_light):
        print("빨간 불 → 초록 불로 변경")
        traffic_light.state = GreenState()  # 다음 상태로 전환

class GreenState(State):
    """초록 불 상태"""
    def change(self, traffic_light):
        print("초록 불 → 노란 불로 변경")
        traffic_light.state = YellowState()  # 다음 상태로 전환

class YellowState(State):
    """노란 불 상태"""
    def change(self, traffic_light):
        print("노란 불 → 빨간 불로 변경")
        traffic_light.state = RedState()  # 다음 상태로 전환

class TrafficLight:
    """신호등 클래스 - 현재 상태를 참조함"""
    def __init__(self):
        self.state = RedState()  # 초기 상태: 빨간 불

    def change_signal(self):
        self.state.change(self)  # 현재 상태에게 change() 위임
```

## 5️⃣ UML 다이어그램 완전 정리 📐

### 📊 UML이란?

**UML(Unified Modeling Language, 통합 모델링 언어)**은 소프트웨어 시스템을 시각적으로 표현하기 위한 **표준 다이어그램 언어**입니다. 마치 건축가가 건물을 짓기 전에 설계도를 그리듯, 개발자들이 시스템 구조와 동작을 그림으로 표현하는 도구입니다.

### 🗺️ 주요 UML 다이어그램 정리

시험에서 각 다이어그램의 역할을 혼동하지 않도록 꼼꼼히 정리해 봅시다.

| 다이어그램 | 핵심 목적 | 핵심 키워드 |
|-----------|----------|------------|
| **클래스 다이어그램** | 시스템 구조를 클래스 속성 및 관계로 표현 | 정적 구조, 클래스 간 관계 |
| **순서(시퀀스) 다이어그램** | 객체 간 메시지 순서를 시간 흐름에 따라 표현 | 시간 흐름, 메시지 순서 |
| **사용 사례 다이어그램** | 사용자 관점의 시스템 기능 표현 | 사용자 시점, 기능 목록 |
| **액티비티 다이어그램** | 업무 흐름과 작업 순서 표현 | 업무 흐름, 조건 분기 |
| **상태 다이어그램** | 단일 객체의 상태와 상태 전환 표현 | 상태 변화, 이벤트 |
| **컴포넌트 다이어그램** | 소프트웨어 간 종속 관계 표현 | 컴포넌트, 의존성 |
| **배치 다이어그램** | 소프트웨어가 하드웨어에 배치되는 구조 표현 | 하드웨어 배치, 물리적 구조 |

:::warning ⚠️ 자주 틀리는 UML 함정들

**20번 문제**: "액티비티 다이어그램 = 객체의 상태와 상태 전환으로 단일 객체의 동작을 나타낸다" → **틀림!**
→ 이것은 **상태 다이어그램(State Diagram)**의 설명입니다.
→ **액티비티 다이어그램**은 **업무 흐름과 작업 순서**를 표현합니다.

**22번 문제**: "배치 다이어그램 = 두 개 이상의 클래스 객체 간 제어 흐름을 보여준다" → **틀림!**
→ **배치 다이어그램**은 소프트웨어가 **하드웨어에 어떻게 배치되는지** 물리적 구조를 표현합니다.
→ 제어 흐름을 보여주는 것은 **액티비티 다이어그램**입니다.
:::

### 🔗 UML 관계 종류

UML에서 클래스(또는 인터페이스) 간의 관계도 자주 출제됩니다.

| 관계 | 설명 | 예시 |
|------|------|------|
| **일반화(Generalization)** | 상속 관계. 부모-자식 클래스 관계 | Animal → Dog |
| **실체화(Realization)** | 인터페이스의 스펙(명세)만 있는 메서드를 오버라이딩(재정의)하여 실제 기능으로 구현 | Flyable 인터페이스 → Bird 클래스가 구현 |
| **의존(Dependency)** | 한 클래스가 다른 클래스를 일시적으로 사용 | 메서드 내에서 다른 클래스 객체 사용 |
| **연관(Association)** | 한 클래스가 다른 클래스와 지속적인 관계 | 학생-학교 관계 |

:::info 💡 실체화(Realization) 핵심 포인트
**실체화**는 **인터페이스(약속된 명세)**를 **클래스가 실제로 구현하는 관계**입니다.
인터페이스는 "날 수 있어야 한다(fly())"는 약속만 있고 실제 구현이 없는데, Bird 클래스가 fly()를 실제로 구현하면 이 관계가 실체화입니다. UML에서 점선 + 화살표로 표현합니다.
:::

## 6️⃣ 컴포넌트와 프로그래밍 언어 선택 기준 💎

### 🧱 컴포넌트(Component)란?

**컴포넌트(Component)**는 소프트웨어 시스템에서 독립적으로 교체하거나 재사용할 수 있는 **소프트웨어 단위**입니다. 마치 컴퓨터의 RAM, CPU, SSD처럼 독립적으로 업그레이드 가능한 부품과 같은 개념입니다.

컴포넌트의 주요 특징:

1. **독립성**: 다른 것들과 명확히 구별되어야 합니다. 각 컴포넌트는 고유한 역할이 있어야 합니다.
2. **교체 가능성**: 다른 버전의 컴포넌트로 교체 가능해야 합니다. 예를 들어 결제 모듈을 v1에서 v2로 교체할 수 있어야 합니다.
3. **인터페이스 접근**: 인터페이스(규격화된 창구)를 통해서만 접근 가능해야 합니다. 내부 구현은 숨기고 정해진 방법으로만 소통합니다.
4. **문서화 필수**: 컴포넌트는 **반드시 별도의 문서로 기록해야 합니다**. 다른 개발자가 사용할 수 있도록 명세서가 필요합니다.

:::danger ❌ 절대 틀리면 안 되는 포인트
"인터페이스를 통해 서비스를 제공하므로 별도의 문서로 기록할 **필요가 없다**" → **완전히 틀린 설명!**

컴포넌트는 **반드시 문서로 기록해야** 합니다. 인터페이스가 있다고 해서 문서가 필요 없는 것이 아닙니다. 오히려 다른 개발자들이 올바르게 사용할 수 있도록 인터페이스 명세를 문서로 작성해야 합니다. (6번 문제 정답: ④)
:::

### 💻 프로그래밍 언어 선택 기준

프로젝트에 맞는 프로그래밍 언어를 선택할 때 고려해야 할 기준들:

1. **명료성, 호환성, 효율성, 유지보수성**: 언어 자체가 명확하고 다른 시스템과 호환되며 효율적으로 동작해야 합니다.
2. **사용자의 요구사항 반영 용이성**: 개발하려는 시스템의 특성에 맞는 언어를 선택해야 합니다. 예를 들어 데이터 분석에는 Python, 시스템 프로그래밍에는 C가 적합합니다.
3. **다양한 개발 도구 지원**: IDE(개발 환경), 라이브러리, 프레임워크 등 생태계가 풍부한 언어가 개발 효율을 높입니다.
4. **검증의 단순성**: 검증(테스트, 검사)은 **복잡하고 어려울수록 좋은 것이 아니라**, 오히려 **단순하고 쉬울수록 좋습니다**. 코드가 단순하면 버그도 적고 테스트도 쉬워집니다.

:::warning ⚠️ 함정 주의
15번 문제에서 "검증은 복잡하고 어려울수록 좋다"는 보기는 **틀린 설명**입니다. 검증은 단순하고 쉬울수록 좋은 특성입니다.
:::

## 7️⃣ 코드 품질 분석 기법과 UI 평가 기준 🖥️

### 🔬 정적 분석 vs 동적 분석

코드의 품질을 검사하는 방법에는 두 가지가 있습니다.

**정적 분석(Static Analysis)**은 코드를 **실행하지 않고** 소스 코드 자체를 검사하는 방법입니다. 마치 요리사가 음식을 만들기 전에 레시피를 읽으면서 "이 부분이 이상한데?"라고 발견하는 것처럼요. 비정상적인 패턴이나 잠재적 오류를 찾아내는 방법입니다.

**동적 분석(Dynamic Analysis)**은 코드를 **실제로 실행**하면서 검사하는 방법입니다. 음식을 실제로 만들어 먹어보면서 "맛이 이상하다"고 발견하는 것과 같습니다.

:::warning ⚠️ 시험 함정 주의
17번 문제에서 "비정상적 패턴이나 읽히지 않는 패턴을 찾는 방법을 **동적 분석**이라 한다" → **틀린 설명!**

코드를 실행하지 않고 패턴을 분석하는 것은 **정적 분석**입니다. 동적 분석은 실제 실행 중에 메모리 누수, 성능 문제 등을 찾는 방법입니다.
:::

**코드 인스펙션(Code Inspection)**은 개발자들이 모여 코드를 직접 검토하는 정적 분석 기법입니다. 컴파일(소스 코드를 기계 코드로 변환) 및 정적 분석 **이후**에 수행되어야 합니다.

**코드 품질 결함 유형**:
- **로직 문제**: 알고리즘이나 조건문의 논리적 오류
- **컴퓨팅 문제**: 연산 오류, 오버플로우, 0으로 나누기
- **인터페이스 문제**: 함수 간 데이터 전달 오류, 매개변수 불일치

### 🖱️ UI 평가 기준 5가지

**UI(User Interface, 사용자 인터페이스)** 품질을 평가하는 5가지 기준입니다. 각 기준의 정의를 정확히 알아야 합니다.

| 기준 | 올바른 정의 |
|------|-----------|
| **용이성(Learnability)** | 처음 사용자도 빠르게 배울 수 있는 정도 |
| **효율성(Efficiency)** | 숙련 사용자도 고수준 작업이 가능해야 함 |
| **기억성(Memorability)** | 오래 안 써도 다시 사용할 수 있도록 기억하기 쉬워야 함 |
| **오류성(Errors)** | 오류를 **적게 범하고 오류 회복이 쉬운** 것 |
| **만족성(Satisfaction)** | 시스템 고장 없이 사용자가 만족하고 즐겁게 사용할 수 있어야 함 |

:::danger ❌ 18번 문제 함정
"오류성: 시스템 고장 없이 사용자가 만족하고 즐겁게 사용할 수 있어야 함" → **완전히 틀린 설명!**

이것은 **만족성(Satisfaction)**의 설명입니다. **오류성(Errors)**은 "사용자가 오류를 적게 범하고, 오류가 발생했을 때 쉽게 회복할 수 있는 것"을 말합니다.
:::

### 🎨 효과적인 UI 제작 방법

효과적인 UI를 만들기 위한 원칙들:

1. **시선 흐름 고려**: 사용자는 화면을 볼 때 **좌측 상단 → 우측 하단** 방향으로 훑어봅니다. 중요한 정보는 좌측 상단에 배치해야 합니다.
2. **입력 필드 안내**: 입력 필드 옆에 글자 수 제한을 표시하여 사용자가 미리 알 수 있게 합니다.
3. **친숙한 언어와 아이콘**: 사용자가 이미 알고 있는 언어와 아이콘을 사용해 학습 부담을 줄입니다.
4. **지속적인 개선**: 기존 UI를 무조건 유지하는 것이 아니라, 더 나은 방향으로 개선해야 합니다.

:::warning ⚠️ 19번 문제 함정
"기존의 것들과 차이가 나지 않도록 하여 혼동을 줄인다" → **틀린 원칙!**

UI는 항상 **개선 방향으로 나아가야** 합니다. 기존 것만 고집하면 발전이 없습니다. 물론 기존 패턴을 완전히 무시하면 혼란을 줄 수 있지만, 더 나은 UX를 위해 개선은 반드시 필요합니다.
:::

## 8️⃣ 소프트웨어 행위 모델링: 상태 전이도 🔄

### 📈 상태 전이도(State Transition Diagram)란?

**상태 전이도(STD, State Transition Diagram)**는 소프트웨어의 **행위(behavior)를 모델링**하는 데 사용됩니다. 시스템이 어떤 **상태(State)**에 있고, 어떤 **이벤트**가 발생했을 때 다른 상태로 **전환(Transition)**되는지를 시각적으로 표현합니다.

예를 들어 ATM기의 상태를 생각해 볼게요:
- **초기 상태**: 대기 중
- **카드 삽입** 이벤트 → 비밀번호 입력 상태로 전환
- **비밀번호 입력** 이벤트 → 거래 선택 상태로 전환
- **거래 완료** 이벤트 → 다시 대기 상태로 전환

이런 시스템의 동적 행위(상태 변화)를 표현하는 것이 바로 상태 전이도입니다.

| 소프트웨어 분석 도구 | 역할 |
|--------------------|------|
| **자료 사전(Data Dictionary)** | 데이터 요소의 의미와 구조 정의 |
| **자료 흐름도(DFD, Data Flow Diagram)** | 시스템 내 데이터의 흐름과 처리 과정 표현 |
| **처리 명세(Process Specification)** | 각 처리 단계의 세부 로직 기술 |
| **상태 전이도(STD)** | **시스템의 동적 행위(상태 변화) 표현** ✅ |

## 📌 핵심 정리

이 문서에서 배운 핵심 개념을 정리합니다.

- **SOA(Service Oriented Architecture)**: 기능을 독립 서비스로 만들어 레고처럼 조합하는 아키텍처
- **Fundamental SOA**: SOA 1단계, 서비스 간 연결이 복잡해지는 단계 (Networked SOA와 혼동 주의)
- **WAS(Web Application Server)**: 로그인 후 개인화 화면처럼 동적 콘텐츠를 처리하는 미들웨어
- **MDD 순서**: CIM → PIM → PSM → Code (비즈니스 분석 → 독립 설계 → 종속 설계 → 코드)
- **테스팅**: 에러를 **찾는** 것 / **디버깅**: 에러를 **수정**하는 것
- **스텁(Stub)**: 하향식 통합 테스트에서 사용 / **드라이버(Driver)**: 상향식 통합 테스트에서 사용
- **테스트 하니스**: 일부 기능 테스트를 위해 임시 변경하고 테스트 후 제거하는 기법
- **리팩토링**: 결과는 그대로, 코드 구조만 개선하여 가독성과 유지보수성 향상
- **역공학**: 완제품 → 설계도 파악 방향의 분석
- **상태 패턴**: 객체 상태에 따라 같은 동작이 다르게 처리되는 디자인 패턴
- **Flyweight**: 구조(Structural) 패턴 (생성 패턴 아님!)
- **Factory Method**: 생성(Creational) 패턴 (행위 패턴 아님!)
- **디자인 패턴**: 유지보수가 **쉬워지는** 것이 장점 (어렵다는 설명은 오답!)
- **실체화(Realization)**: 인터페이스를 클래스가 실제로 구현하는 UML 관계
- **액티비티 다이어그램**: 업무 흐름/작업 순서 표현 (상태 전환 표현이 아님!)
- **배치 다이어그램**: 소프트웨어의 하드웨어 배치 구조 표현 (제어 흐름 표현이 아님!)
- **정적 분석**: 코드 실행 없이 패턴을 검사하는 방법 (동적 분석과 혼동 주의!)
- **오류성(Errors)**: 오류를 적게 범하고 회복이 쉬운 것 (만족성과 혼동 주의!)
- **상태 전이도(STD)**: 소프트웨어의 동적 **행위(behavior)** 모델링에 사용
- **컴포넌트**: 반드시 별도의 문서로 기록해야 함 (문서 불필요 설명은 오답!)
- **유지보수 비용**: 소프트웨어는 오래 사용할수록 비용이 **증가**함
