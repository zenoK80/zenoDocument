# 🧩 디자인 패턴, UML, 컴포넌트 및 프로그래밍 언어 선택

소프트웨어 공학 시험에서 단골로 등장하는 핵심 개념들을 한 번에 정리합니다. 디자인 패턴이 무엇인지, UML 다이어그램을 어떻게 읽는지, 컴포넌트란 무엇인지, 그리고 프로그래밍 언어는 어떤 기준으로 선택하는지까지 — 초보자도 이해할 수 있도록 실생활 비유와 함께 아주 자세하게 설명합니다.

---

## 🏗️ 소프트웨어 아키텍처 — SOA와 WAS

### SOA란 무엇인가?

**SOA(Service Oriented Architecture, 서비스 지향 아키텍처)** 는 기존 애플리케이션의 기능들을 **비즈니스 단위로 묶어 독립적인 서비스로 만들고**, 표준화된 인터페이스(서로 소통할 수 있는 규격화된 창구)를 통해 이 서비스들을 조합하여 하나의 애플리케이션을 완성하는 소프트웨어 개발 방식입니다.

쉽게 말하면, 마치 **레고 블록**처럼 각 기능(로그인, 결제, 배송 조회 등)을 독립적인 블록으로 만들어 놓고, 필요에 따라 원하는 블록만 골라 조립하는 방식입니다. 하나의 거대한 프로그램 덩어리 대신, 잘게 나누어진 서비스들을 조합하므로 유연성과 재사용성이 크게 높아집니다.

> 💡 **핵심 비유**: 레스토랑 주방처럼 생각해보세요. 요리사가 하나씩 전부 다 하는 게 아니라, 전채 담당, 메인 요리 담당, 디저트 담당이 따로 있고, 주문에 따라 적절히 조합해 한 상을 차려내는 것입니다.

### SOA의 3단계 발전 모델

SOA는 단계별로 발전하는 구조를 가집니다. 시험에서 자주 혼동하는 부분이니 꼼꼼히 정리해두세요.

| 단계 | 모델명 | 특징 |
|------|--------|------|
| 1단계 | **Fundamental SOA** | 기존 시스템들을 서비스화하여 단순 연결. 시스템 규모가 커질수록 서비스 간 연결이 복잡해지는 문제 발생 |
| 2단계 | **Networked SOA** | 복잡도 해결을 위해 ESB(Enterprise Service Bus, 중앙 허브 역할의 연결 버스)를 도입하여 서비스 간 통신을 일원화 |
| 3단계 | **Process Oriented SOA** | 비즈니스 프로세스 중심으로 서비스를 오케스트레이션(조율)하는 단계 |
| 4단계 | **Business SOA** | 비즈니스 전략 자체와 IT를 완전히 통합하는 최고 수준의 SOA |

:::warning ⚠️ 자주 틀리는 포인트
"시스템 규모 증가에 따라 서비스 간 연결이 복잡해지는 모델"은 **① Fundamental SOA**입니다.  
Networked SOA는 그 **복잡도를 해결하기 위해** ESB를 도입한 **2단계**입니다. 순서를 꼭 기억하세요!
:::

### WAS란 무엇인가?

**WAS(Web Application Server, 웹 애플리케이션 서버)** 는 정적 콘텐츠(변하지 않는 HTML, 이미지 파일 등)를 단순히 전달하는 일반 웹 서버와 달리, **사용자의 요청에 따라 동적으로 변하는 콘텐츠를 처리**하는 미들웨어(Middleware, 서로 다른 소프트웨어 사이에서 연결 역할을 하는 소프트웨어)입니다.

예를 들어, 로그인 후 나타나는 **"홍길동님, 환영합니다!"** 라는 문구는 사람마다 다르게 보여야 합니다. 이렇게 사용자별, 상황별로 다른 화면을 만들어 응답하는 것이 바로 WAS의 역할입니다.

:::info 웹 서버 vs WAS 비교
- **웹 서버(Web Server)**: 정적인 파일(HTML, CSS, 이미지)을 그대로 전달. 마치 책장에서 책을 꺼내주는 것.
- **WAS(Web Application Server)**: 사용자 요청에 맞춰 동적으로 콘텐츠를 생성. 마치 요청받은 대로 맞춤 보고서를 작성해주는 것.
:::

## 🔧 소프트웨어 개발 핵심 기법 — 테스트 하니스, 리팩토링, 역공학

### 테스트 하니스 (Test Harness)

**테스트 하니스(Test Harness)** 는 시스템의 일부 기능만 시험하기 위해 소프트웨어를 **임시로 변경**하고, 테스트가 끝나면 **다시 제거**하는 것을 말합니다. 마치 자동차 공장에서 특정 부품만 테스트하기 위해 임시 장치를 달았다가 검사 후 제거하는 것과 같습니다.

:::warning ⚠️ 테스트 스텁(Test Stub)과 혼동 주의!
- **테스트 하니스**: 시스템 일부 기능 테스트를 위해 소프트웨어를 임시 변경 후 제거하는 것
- **테스트 스텁(Test Stub)**: 하향식 통합 테스트에서 아직 개발되지 않은 하위 모듈을 대신하는 **가짜 모듈**

두 개념은 전혀 다릅니다! 시험에서 "임시 변경 후 제거"라는 표현이 나오면 반드시 **테스트 하니스**를 떠올리세요.
:::

### 테스팅 vs 디버깅 — 헷갈리지 마세요

이 두 개념은 자주 혼동하지만 목적이 완전히 다릅니다.

| 구분 | 테스팅(Testing) | 디버깅(Debugging) |
|------|----------------|------------------|
| **목적** | 에러(오류)를 **찾는** 것 | 에러(오류)를 **수정하는** 것 |
| **역할 비유** | 검사관 (문제를 발견) | 수리공 (문제를 고침) |
| **수행 시점** | 개발 완료 후 또는 중간 점검 | 테스팅 후 오류 발견 시 |

> 💡 **쉬운 비유**: 테스팅은 "자동차에 이상 있는지 검사하는 것"이고, 디버깅은 "발견된 이상을 실제로 고치는 것"입니다. 검사관이 문제를 발견하면, 수리공이 고치는 것처럼 순서대로 이어집니다.

### 리팩토링 (Refactoring)

**리팩토링(Refactoring)** 이란 코드가 실행되어 나오는 **결과는 전혀 바꾸지 않으면서**, 코드의 내부 구조를 더 읽기 쉽고 관리하기 좋게 재정리하는 기술입니다. 기능 추가나 버그 수정이 아닌, 순수하게 코드의 **품질과 가독성**을 높이는 데 집중합니다.

> 💡 **쉬운 비유**: 맛은 똑같은 요리인데 주방을 더 깔끔하게, 동선이 편하게 정리하는 것입니다. 손님이 먹는 음식은 변하지 않지만, 주방이 정돈되어 이후 요리가 훨씬 수월해집니다.

```python
# ❌ 리팩토링 전: 변수명이 불명확하고 코드 읽기 어려움
def calc(a, b, c):
    # a = 단가, b = 수량, c = 할인율 (주석 없으면 모름)
    return a * b * (1 - c)

# ✅ 리팩토링 후: 변수명을 의미 있게 변경하여 가독성 향상
def calculate_discounted_price(unit_price, quantity, discount_rate):
    # 단가 × 수량 × (1 - 할인율) = 최종 할인 금액 계산
    total_price = unit_price * quantity          # 총 가격 계산
    discounted_price = total_price * (1 - discount_rate)  # 할인 적용
    return discounted_price
```

위 두 함수는 **완전히 동일한 결과**를 냅니다. 하지만 아래 코드는 누가 봐도 무엇을 계산하는 함수인지 바로 알 수 있죠. 이것이 리팩토링의 핵심입니다.

### 역공학 (Reverse Engineering)

**역공학(Reverse Engineering)** 은 원래 하드웨어 분야에서 완성된 제품을 분해하여 설계 방식을 파악하는 작업에서 유래했습니다. 소프트웨어에서는 **완성된 시스템을 분석하여 그 안의 컴포넌트(구성 요소)와 관계를 파악**하는 작업을 의미합니다. 즉, 설계도 → 완제품이 아니라, **완제품 → 설계도** 방향으로 거슬러 올라가는 것입니다.

:::info 역공학 vs 재공학 차이
- **역공학(Reverse Engineering)**: 완성품을 분석해서 설계 정보를 **파악**하는 것 (설계도를 읽어내는 것)
- **재공학(Re-Engineering)**: 기존 시스템을 분석하고 더 나은 형태로 **재구축**하는 것 (설계도를 읽고 더 좋게 다시 짓는 것)
:::

## 📐 UML — 소프트웨어 설계의 공통 언어

### UML이란 무엇인가?

**UML(Unified Modeling Language, 통합 모델링 언어)** 은 소프트웨어 시스템의 구조와 동작을 **시각적인 다이어그램(그림)으로 표현하기 위한 표준화된 언어**입니다. 건축가가 집을 짓기 전에 설계 도면을 그리듯, 개발자들은 UML로 시스템의 설계도를 그립니다. 전 세계 어느 개발자가 보아도 같은 의미로 해석할 수 있도록 표준화되어 있다는 것이 가장 큰 장점입니다.

### UML 주요 다이어그램 종류

UML 다이어그램은 크게 **구조 다이어그램**과 **행위 다이어그램**으로 나뉩니다. 시험에서는 각 다이어그램의 정의를 서로 바꿔서 출제하는 경우가 많으니 확실하게 구분해두세요.

| 다이어그램 | 종류 | 설명 |
|-----------|------|------|
| **클래스 다이어그램** | 구조 | 시스템의 클래스(설계 틀), 속성, 관계를 나타내는 핵심 다이어그램 |
| **컴포넌트 다이어그램** | 구조 | 소프트웨어 컴포넌트 간의 종속 관계를 보여주는 다이어그램 |
| **배치 다이어그램** | 구조 | 소프트웨어가 실제 **하드웨어에 어떻게 배치**되는지를 표현. 서버, 네트워크 장비 등 물리적 구성을 나타냄 |
| **순서(시퀀스) 다이어그램** | 행위 | 객체 간 메시지(통신)의 순서를 **시간 흐름**에 따라 표현 |
| **액티비티 다이어그램** | 행위 | 업무 흐름, 작업 순서, 활동의 흐름을 표현. **여러 객체에 걸친 프로세스 흐름** 묘사 |
| **상태 다이어그램** | 행위 | 단일 객체의 **상태 변화**와 전환을 표현. 예: 주문 상태(접수→처리중→완료) |
| **사용 사례 다이어그램** | 행위 | **사용자(액터) 관점**에서 시스템이 제공하는 기능을 표현 |

:::warning ⚠️ 가장 많이 혼동하는 쌍
**액티비티 다이어그램 vs 상태 다이어그램**

- **액티비티 다이어그램**: 업무 흐름, 여러 활동의 **순서와 흐름** 표현 (여러 객체에 걸친 작업 흐름)
- **상태 다이어그램**: 하나의 객체가 가지는 **상태(State)와 그 전환** 표현 (단일 객체의 생명주기)

시험에서 "객체의 상태 전환을 나타내는 것"은 **상태 다이어그램**이고, "업무 흐름과 작업 순서를 나타내는 것"은 **액티비티 다이어그램**입니다!

**배치 다이어그램 vs 액티비티 다이어그램**

- 배치 다이어그램은 "제어 흐름"을 보여주는 게 아니라 소프트웨어가 **하드웨어 노드에 배치되는 물리적 구조**를 보여줍니다. "두 클래스 간 제어 흐름"은 액티비티 다이어그램의 설명입니다.
:::

### UML 관계의 종류

UML에서 클래스 간의 관계를 표현하는 방법도 시험에 자주 나옵니다.

| 관계 | 이름 | 설명 |
|------|------|------|
| `─────>` | **연관(Association)** | 두 클래스가 서로 알고 있는 관계. "A는 B를 참조한다" |
| `- - ->` | **의존(Dependency)** | 한 클래스가 다른 클래스를 일시적으로 사용하는 관계 |
| `──────▷` | **일반화(Generalization)** | 상속 관계. "자식 클래스는 부모 클래스다(is-a)" |
| `- - -▷` | **실체화(Realization)** | 인터페이스를 실제로 구현하는 관계 |

**실체화(Realization)** 는 특히 중요합니다. 인터페이스(Interface, 기능의 명세서만 있고 실제 구현은 없는 것)에 정의된 메서드를 클래스가 **실제로 구현(오버라이딩)** 하는 것이 바로 실체화입니다. 마치 "로봇 사용 설명서(인터페이스)"에 적힌 기능 목록을 실제 로봇(클래스)이 구현해내는 것과 같습니다.

```java
// 인터페이스 - 기능의 명세(스펙)만 있고 구현은 없음
interface Payable {
    // 결제를 수행하라는 명세만 있음 (실제 구현 없음)
    void pay(int amount);
}

// ✅ 실체화(Realization) - 인터페이스를 실제로 구현하는 클래스
class CreditCard implements Payable {
    @Override
    public void pay(int amount) {
        // 인터페이스의 pay() 메서드를 실제 신용카드 결제로 구현
        System.out.println("신용카드로 " + amount + "원 결제 완료");
    }
}

class KakaoPay implements Payable {
    @Override
    public void pay(int amount) {
        // 동일한 인터페이스를 카카오페이 방식으로 구현
        System.out.println("카카오페이로 " + amount + "원 결제 완료");
    }
}
```

위 코드에서 `CreditCard`와 `KakaoPay` 클래스가 `Payable` 인터페이스를 `implements`하여 실제 기능을 구현하는 것, 이것이 바로 **실체화(Realization)** 관계입니다.

### 상태 전이도 (STD)와 소프트웨어 행위 모델링

**상태 전이도(State Transition Diagram, STD)** 는 시스템 또는 객체의 **동적 행위(behavior, 시간에 따라 변하는 동작)**를 모델링하는 데 사용됩니다. 시스템이 어떤 상태에 있고, 어떤 조건이 생기면 다른 상태로 이동하는지를 시각적으로 표현합니다.

> 💡 **쉬운 비유**: 신호등을 생각해보세요. 신호등은 "빨강 → 초록 → 노랑 → 빨강"이라는 상태 변화를 순환합니다. 특정 조건(타이머)에 의해 상태가 전환되는 이 흐름을 도식화한 것이 상태 전이도입니다.

:::note 행위 모델링에 사용되는 도구
소프트웨어의 행위(behavior)를 모델링할 때는 **상태 전이도(STD)** 가 핵심 도구입니다. 자료 사전, 자료 흐름도, 처리 명세는 구조적 분석 도구이며, 동적 행위 모델링에는 상태 전이도가 가장 적합합니다.
:::

## 🎨 디자인 패턴 — 검증된 설계 템플릿

### 디자인 패턴이란?

**디자인 패턴(Design Pattern)** 은 소프트웨어 설계에서 자주 반복되어 나타나는 문제들에 대한 **검증된 해결책을 유형별로 정리한 템플릿(설계 양식)** 입니다. 마치 건축에서 "아파트 평면도 표준 양식"처럼, 개발자들이 설계 문제를 해결할 때 매번 처음부터 고민하지 않고 **이미 검증된 패턴을 재사용**할 수 있도록 도와줍니다.

:::tip 디자인 패턴의 장점
- ✅ 자주 사용하는 설계 형태를 유형별로 **템플릿화**하여 재사용 가능
- ✅ **재사용**을 통해 개발 시간을 단축할 수 있음
- ✅ 설계자 간 **원활한 의사소통**에 활용 가능 (공통 언어 역할)
- ✅ 설계 변경 요청에 대한 **유지보수가 쉬움** (유연한 구조)
:::

:::danger ❌ 절대 혼동하지 마세요
"디자인 패턴은 유지보수가 어렵다"는 설명은 **완전히 틀린 설명**입니다. 디자인 패턴의 핵심 장점 중 하나가 바로 **유지보수를 쉽게 만든다**는 것입니다. 시험에서 이 선택지를 고르면 안 됩니다!
:::

### 디자인 패턴의 3가지 분류

GoF(Gang of Four, 디자인 패턴을 체계화한 4명의 저자) 디자인 패턴은 크게 세 가지로 분류됩니다. 시험에서는 각 패턴이 어느 분류에 속하는지를 묻는 문제가 자주 출제됩니다.

#### 1️⃣ 생성 패턴 (Creational Pattern) — 객체를 어떻게 만들 것인가

객체의 **생성 방식**에 관한 패턴입니다. 객체를 직접 `new`로 생성하지 않고, 상황에 맞는 방식으로 유연하게 생성할 수 있도록 해줍니다.

| 패턴 이름 | 설명 |
|-----------|------|
| **Singleton** | 클래스의 인스턴스(실제 객체)를 오직 하나만 생성하도록 제한. 예: 데이터베이스 연결 객체 |
| **Factory Method** | 객체 생성을 서브클래스에 위임. 어떤 객체를 만들지 서브클래스가 결정 |
| **Abstract Factory** | 관련된 객체들의 집합을 생성하는 인터페이스 제공. 예: UI 테마 전환(다크/라이트 모드) |
| **Builder** | 복잡한 객체를 단계적으로 조립하여 생성 |
| **Prototype** | 기존 객체를 복사(clone)하여 새 객체 생성 |

```python
# ✅ Singleton 패턴 예시 - 오직 하나의 인스턴스만 존재
class DatabaseConnection:
    _instance = None  # 클래스 변수로 유일한 인스턴스 저장
    
    @classmethod
    def get_instance(cls):
        # 인스턴스가 아직 없을 때만 새로 생성
        if cls._instance is None:
            cls._instance = cls()  # 처음 한 번만 객체 생성
            print("DB 연결 생성!")
        return cls._instance  # 이미 있으면 기존 인스턴스 반환

# 사용 예시
db1 = DatabaseConnection.get_instance()  # "DB 연결 생성!" 출력
db2 = DatabaseConnection.get_instance()  # 아무것도 출력 안 됨 (기존 것 반환)
print(db1 is db2)  # True - 완전히 동일한 객체임을 확인
```

#### 2️⃣ 구조 패턴 (Structural Pattern) — 객체를 어떻게 조합할 것인가

클래스나 객체를 **조합하여 더 큰 구조**를 만드는 방법에 관한 패턴입니다.

| 패턴 이름 | 설명 |
|-----------|------|
| **Adapter** | 서로 다른 인터페이스를 연결해주는 어댑터. 마치 돼지코 어댑터처럼 |
| **Facade** | 복잡한 서브시스템을 단순한 인터페이스로 감싸줌. 리모컨처럼 복잡한 내부를 단순하게 |
| **Composite** | 개별 객체와 복합 객체를 동일하게 처리 |
| **Decorator** | 기존 객체에 새 기능을 동적으로 추가 |
| **Proxy** | 다른 객체에 대한 접근을 제어하는 대리자 역할 |
| **Flyweight** | 많은 수의 작은 객체들을 공유하여 메모리 효율을 높임 |
| **Bridge** | 구현부에서 추상층을 분리하여 독립적으로 변형 가능 |

:::warning ⚠️ Flyweight는 구조 패턴!
**Flyweight**는 **구조(Structural) 패턴**입니다. Factory Method, Abstract Factory, Singleton과 같은 **생성(Creational) 패턴**과 혼동하지 마세요. 시험에서 "객체 생성 패턴이 아닌 것"을 묻는 문제에서 Flyweight가 정답으로 자주 출제됩니다!
:::

#### 3️⃣ 행위 패턴 (Behavioral Pattern) — 객체 간 협력을 어떻게 할 것인가

객체들 사이의 **알고리즘과 책임 분담**에 관한 패턴입니다. 객체들이 어떻게 서로 통신하고 협력할지를 정의합니다.

| 패턴 이름 | 설명 |
|-----------|------|
| **State** | 객체 내부 상태에 따라 행동을 다르게 처리 |
| **Command** | 요청을 객체로 캡슐화하여 취소/재실행 가능하게 함 |
| **Iterator** | 컬렉션 내부 구조를 노출하지 않고 순차적으로 접근 |
| **Observer** | 한 객체의 상태 변화를 다른 객체들에게 자동 통보 |
| **Strategy** | 알고리즘을 캡슐화하여 교체 가능하게 함 |
| **Template Method** | 알고리즘의 골격을 정의하고 세부 단계는 서브클래스가 구현 |
| **Interpreter** | 언어의 문법을 클래스로 표현하고 해석 |
| **Chain of Responsibility** | 요청을 처리할 객체를 연쇄적으로 전달 |
| **Mediator** | 객체들 간의 복잡한 통신을 중재자 객체가 담당 |
| **Memento** | 객체의 상태를 저장하고 복원 (되돌리기 기능) |
| **Visitor** | 객체 구조를 변경하지 않고 새 기능 추가 |

:::warning ⚠️ Factory Method는 생성 패턴!
**Factory Method**는 **생성(Creational) 패턴**입니다. Interpreter, Command, Iterator 같은 **행위(Behavioral) 패턴**과 혼동하지 마세요. "행위 패턴이 아닌 것"을 묻는 문제에서 Factory Method가 정답으로 자주 출제됩니다!
:::

### 상태 패턴 (State Pattern) 자세히 보기

**상태 패턴(State Pattern)** 은 객체의 **내부 상태에 따라 같은 동작이 다르게 처리**되도록 하는 패턴입니다. 핵심은 객체의 각 상태를 별도의 클래스로 만들고, 현재 상태 클래스에게 실행을 위임하는 것입니다.

> 💡 **신호등 비유**: 신호등에 "신호를 보여줘(show)" 라는 동작이 있다고 가정합니다. 현재 상태가 "빨강"이면 정지 신호를, "초록"이면 진행 신호를 보여줍니다. 동작의 이름은 같지만 **상태에 따라 완전히 다른 결과**가 나옵니다. 이게 상태 패턴입니다.

```python
# ✅ 상태 패턴 (State Pattern) 예시 - 신호등
from abc import ABC, abstractmethod

# 상태를 표현하는 추상 클래스
class TrafficLightState(ABC):
    @abstractmethod
    def show_signal(self):
        pass  # 각 상태 클래스가 구체적으로 구현해야 함

# 빨간 불 상태 클래스
class RedState(TrafficLightState):
    def show_signal(self):
        print("🔴 정지! 멈추세요.")  # 빨간 불 상태의 동작

# 초록 불 상태 클래스
class GreenState(TrafficLightState):
    def show_signal(self):
        print("🟢 진행! 가도 됩니다.")  # 초록 불 상태의 동작

# 신호등 클래스 - 현재 상태를 참조
class TrafficLight:
    def __init__(self):
        self.state = RedState()  # 초기 상태는 빨간 불

    def set_state(self, state: TrafficLightState):
        self.state = state  # 상태 변경

    def show(self):
        self.state.show_signal()  # 현재 상태에게 동작을 위임

# 사용 예시
light = TrafficLight()
light.show()  # 🔴 정지! 멈추세요.

light.set_state(GreenState())  # 상태를 초록으로 변경
light.show()  # 🟢 진행! 가도 됩니다.
```

위 코드에서 `TrafficLight` 객체의 `show()` 메서드는 항상 같은 이름이지만, **현재 상태(RedState 또는 GreenState)에 따라 전혀 다른 동작을 수행**합니다. 이것이 상태 패턴의 핵심입니다.

## 🧱 컴포넌트 (Component) 이해하기

### 컴포넌트란 무엇인가?

**컴포넌트(Component)** 는 소프트웨어 시스템을 구성하는 **독립적이고 재사용 가능한 소프트웨어 단위**입니다. 마치 PC를 조립할 때 CPU, RAM, 하드디스크 각각이 독립적인 부품으로 존재하면서 정해진 규격(인터페이스)으로 연결되는 것처럼, 소프트웨어도 컴포넌트 단위로 나누어 조립할 수 있습니다.

### 컴포넌트의 올바른 특징

컴포넌트가 제대로 된 컴포넌트이려면 다음 조건을 만족해야 합니다.

| 특징 | 설명 |
|------|------|
| **독립성** | 다른 것들과 명확히 구별되어야 한다. 경계가 뚜렷해야 함 |
| **교체 가능성** | 다른 버전의 컴포넌트로 교체 가능해야 한다. 업그레이드가 쉬워야 함 |
| **인터페이스 접근** | 인터페이스를 통해서만 접근 가능해야 한다. 내부 구현이 숨겨져야 함 |
| **문서화 필수** | 인터페이스와 기능이 반드시 문서로 기록되어야 한다 |

:::danger ❌ 절대 틀리지 마세요
"컴포넌트는 인터페이스를 통해 서비스를 제공하므로 **별도의 문서로 기록할 필요가 없다**"는 설명은 **완전히 틀렸습니다!**

컴포넌트는 다른 개발자나 시스템이 사용해야 하므로, 어떤 인터페이스를 제공하는지, 어떻게 사용하는지를 **반드시 문서로 기록해야 합니다**. 문서 없는 컴포넌트는 사용할 수 없는 블랙박스에 불과합니다.
:::

## 🔬 테스트 기법과 코드 품질

### 블랙박스 테스트 vs 화이트박스 테스트

소프트웨어 테스트는 **내부 코드를 알고 하느냐, 모르고 하느냐**에 따라 두 가지로 나뉩니다.

| 구분 | 블랙박스 테스트 | 화이트박스 테스트 |
|------|---------------|-----------------|
| **개념** | 내부 코드를 모르는 상태에서 입출력만으로 테스트 | 내부 코드 구조를 알고 논리 흐름을 테스트 |
| **비유** | 요리 맛만 보고 레시피를 평가 | 레시피(코드)를 직접 보고 검증 |
| **주요 기법** | 동등 분할, 경곗값 분석, 원인-결과 그래프 | **기본 경로 검사**, 조건 검사, 루프 검사 |

:::warning ⚠️ 기본 경로 검사(Basic Path Testing)는 화이트박스!
**기본 경로 검사**는 코드의 내부 흐름(경로)을 분석하는 기법이므로 **화이트박스 테스트**에 속합니다. 블랙박스 테스트 기법 목록에서는 절대 나타나지 않습니다. 시험에서 "블랙박스 테스트 기법이 아닌 것"을 고르는 문제에서 기본 경로 검사가 정답입니다!
:::

### 하향식 통합 테스트 vs 상향식 통합 테스트

시스템을 단위 모듈로 나누어 테스트할 때, **어느 방향으로 조립해 가느냐**에 따라 두 가지 방식이 있습니다.

| 구분 | 하향식 통합 테스트 | 상향식 통합 테스트 |
|------|-----------------|-----------------|
| **방향** | 최상위 모듈부터 아래로 | 최하위 모듈부터 위로 |
| **필요한 가짜 모듈** | **스텁(Stub)** - 아직 없는 하위 모듈 대체 | **드라이버(Driver)** - 아직 없는 상위 모듈 대체 |
| **장점** | 전체 시스템 골격을 조기에 확인 가능 | 하위 레벨 모듈을 충분히 테스트 가능 |

> 💡 **쉬운 비유**:
> - **하향식**: 건물을 지을 때 1층 뼈대부터 세우고, 2층 3층으로 올라가는 방식. 아직 없는 아래층은 임시 기둥(스텁)으로 대체.
> - **상향식**: 건물의 최상층 엘리베이터 기계실부터 만들고 아래로 내려오는 방식. 아직 없는 위층은 임시 제어장치(드라이버)로 대체.

:::warning ⚠️ 드라이버 vs 스텁 혼동 주의
- **드라이버(Driver)**: **상향식** 통합 테스트에서 사용. 아직 개발되지 않은 **상위 모듈**을 대신하는 가짜 모듈.
- **스텁(Stub)**: **하향식** 통합 테스트에서 사용. 아직 개발되지 않은 **하위 모듈**을 대신하는 가짜 모듈.

"하향식에서는 드라이버가 필요하다"는 설명은 **틀렸습니다**. 하향식에서는 **스텁**이 필요합니다!
:::

### 흔한 코딩 오류의 종류

코딩 시 자주 발생하는 오류들은 다음과 같습니다.

- **메모리 누수(Memory Leak)**: 메모리를 할당받은 후 해제(free)하지 않아 메모리가 계속 쌓이는 현상
- **배열 인덱스 오류**: 배열 인덱스가 음수이거나 배열의 범위를 벗어나는 경우
- **0으로 나누기(Division by Zero)**: 분모가 0인 나눗셈 시도

반면, **매개변수 값이 `null`이 아닌 경우**는 정상적인 상황이므로 코딩 오류가 아닙니다. `null`인 경우가 예외 상황이고, `null`이 아닌 것은 매개변수가 올바르게 전달된 것입니다.

### 코드 품질 향상 기법

```python
# ❌ 정적 분석(Static Analysis)이 잡을 수 있는 잠재적 오류 예시
def divide(a, b):
    return a / b  # b가 0이면 런타임 오류 발생 - 정적 분석으로 경고 가능

# ✅ 방어 코드 추가 후 개선
def divide(a, b):
    if b == 0:
        raise ValueError("0으로 나눌 수 없습니다!")  # 사전에 예외 처리
    return a / b
```

코드 품질 향상을 위한 기법들의 차이를 명확히 알아두세요:

- **정적 분석(Static Analysis)**: 코드를 **실행하지 않고** 소스 코드를 읽어서 비정상 패턴이나 잠재적 오류를 찾는 방법. 컴파일 전에 수행됨.
- **동적 분석(Dynamic Analysis)**: 코드를 **실행하면서** 메모리 누수, 성능 문제 등을 탐지하는 방법.
- **코드 인스펙션(Code Inspection)**: 동료 개발자들이 코드를 직접 읽으며 검토하는 방법. 컴파일 및 정적 분석 **이후**에 수행.

:::warning ⚠️ 정적 vs 동적 분석 헷갈리지 마세요
"비정상적 패턴이나 읽히지 않는 패턴을 찾는 방법"은 코드를 실행하지 않고 분석하므로 **정적 분석**입니다.  
동적 분석은 실제로 프로그램을 **실행하면서** 분석하는 방법입니다.
:::

## 🖥️ 프로그래밍 언어 선택과 UI 설계

### 프로그래밍 언어 선택 기준

새로운 프로젝트를 시작할 때 어떤 프로그래밍 언어를 선택할지는 매우 중요한 결정입니다. 올바른 선택 기준은 다음과 같습니다.

| 선택 기준 | 설명 |
|-----------|------|
| **명료성** | 코드를 명확하게 표현할 수 있어야 함 |
| **호환성** | 다른 시스템/환경과 잘 연동되어야 함 |
| **효율성** | 성능이 우수해야 함 |
| **유지보수성** | 나중에 고치고 발전시키기 쉬워야 함 |
| **요구사항 반영** | 사용자의 요구사항이 쉽게 반영될 수 있어야 함 |
| **도구 지원** | 다양한 개발 도구와 라이브러리의 지원이 있어야 함 |
| **검증 용이성** | 검증은 **쉽고 단순**할수록 좋음 |

:::danger ❌ 절대 틀리지 마세요
"검증은 **복잡하고 어려울수록 좋다**"는 선택 기준은 **완전히 틀렸습니다.**  
검증(테스트, 검사)은 **쉽고 단순할수록** 오류를 발견하기 쉽고 개발 효율이 높아집니다. 복잡한 검증 과정은 오히려 개발 비용과 시간을 늘립니다.
:::

### MDD (Model Driven Development) 프로세스 순서

**MDD(모델 주도 개발)** 는 모델을 중심으로 소프트웨어를 개발하는 방법론입니다. 프로세스 순서가 시험에 자주 출제됩니다.

```
CIM → PIM → PSM → Code
```

각 단계를 자세히 이해해보겠습니다:

1. **CIM(Computation Independent Model, 계산 독립 모델)**: 비즈니스 관점의 모델. "무엇을 해야 하는가"를 기술. 기술 세부사항 없음.
2. **PIM(Platform Independent Model, 플랫폼 독립 모델)**: 특정 플랫폼(자바, .NET 등)에 종속되지 않는 설계 모델.
3. **PSM(Platform Specific Model, 플랫폼 종속 모델)**: 특정 플랫폼에 맞게 구체화된 설계 모델.
4. **Code**: PSM을 기반으로 실제 소스 코드를 생성.

> 💡 **쉬운 비유**: 건물을 지을 때 "어떤 건물이 필요한가(CIM)" → "일반적인 건축 설계도(PIM)" → "이 지역 건축 법규에 맞는 설계도(PSM)" → "실제 시공(Code)" 순서로 진행하는 것과 같습니다.

### UI 평가 기준

사용자 인터페이스(UI)의 품질을 평가하는 기준들을 알아봅시다.

| 평가 기준 | 영어 | 설명 |
|-----------|------|------|
| **용이성** | Learnability | 처음 사용하는 사람도 빠르게 배울 수 있는 정도 |
| **효율성** | Efficiency | 숙련된 사용자가 고수준 작업도 효율적으로 할 수 있어야 함 |
| **기억성** | Memorability | 오래 사용하지 않다가 다시 사용할 때도 기억하기 쉬워야 함 |
| **오류성** | Errors | 사용자가 오류를 적게 범하고, 오류 발생 시 회복이 쉬워야 함 |
| **만족성** | Satisfaction | 시스템 고장 없이 사용자가 만족하고 즐겁게 사용할 수 있어야 함 |

:::warning ⚠️ 오류성(Errors)과 만족성(Satisfaction) 혼동 주의
- **오류성**: "오류를 적게 범하고, 발생한 오류에서 회복이 쉬운 것"
- **만족성**: "시스템을 안정적으로 사용하면서 사용자가 만족하고 즐겁게 느끼는 것"

"시스템 고장 없이 사용자가 만족하고 즐겁게 사용할 수 있어야 한다"는 설명은 **만족성(Satisfaction)** 의 설명이지, 오류성의 설명이 아닙니다.
:::

### 효과적인 UI 제작 방법

좋은 UI를 만들기 위한 검증된 원칙들은 다음과 같습니다.

- **시선 흐름 고려**: 사용자는 일반적으로 화면을 **좌측 상단 → 우측 하단** 방향으로 훑어봅니다. 중요한 정보는 좌측 상단에 배치하세요.
- **즉각적인 피드백 제공**: 입력 필드 옆에 허용 글자 수 표시, 비밀번호 강도 표시 등 사용자에게 즉각적인 정보를 제공합니다.
- **친숙한 언어와 아이콘 사용**: 사용자가 이미 알고 있는 용어와 아이콘을 사용하여 학습 비용을 줄입니다.
- **지속적인 개선**: UI는 기존 것만 고집하면 발전이 없습니다. 사용자 경험을 분석하고 더 나은 방향으로 끊임없이 개선해야 합니다.

:::tip 💡 UI 설계의 핵심 원칙
"기존의 것들과 차이가 나지 않도록 하여 혼동을 줄인다"는 것은 완전히 올바른 UI 설계 원칙이 아닙니다. 기존 방식을 무조건 고수하면 발전할 수 없습니다. UI는 사용자의 피드백을 바탕으로 **더 나은 방향으로 개선**해 나가야 합니다.
:::

### 유지보수 비용과 소프트웨어 수명

소프트웨어의 **유지보수 비용**은 개발이 완료된 이후에도 지속적으로 발생합니다. 중요한 사실은, 소프트웨어를 **오래 사용할수록 유지보수 비용은 증가**한다는 것입니다.

시간이 지남에 따라 코드는 복잡해지고, 개발 담당자가 교체되며, 새로운 요구사항이 기존 구조와 충돌하고, 기술 부채(Technical Debt, 빠른 개발을 위해 미뤄뒀던 설계 개선 비용)가 쌓입니다. 따라서 **분석 단계에서부터 사용자가 원하는 것을 정확히 파악**하는 것이 나중의 유지보수 비용을 줄이는 가장 효과적인 방법입니다.

:::danger ❌ "오래 쓸수록 유지보수 비용이 감소한다"는 완전히 틀린 말
소프트웨어는 하드웨어처럼 새로운 상태로 고정되어 있지 않습니다. 시간이 지날수록 요구사항 변경, 인력 교체, 기술 노후화 등으로 인해 유지보수 비용은 **지속적으로 증가**합니다.
:::

## 📊 핵심 개념 총정리 비교표

시험 직전 빠르게 확인할 수 있도록 핵심 개념들을 한눈에 정리합니다.

### 디자인 패턴 분류 총정리

| 패턴 | 분류 | 핵심 키워드 |
|------|------|------------|
| Singleton | 생성(Creational) | 인스턴스 하나만 |
| Factory Method | 생성(Creational) | 객체 생성을 서브클래스에 위임 |
| Abstract Factory | 생성(Creational) | 관련 객체 집합 생성 |
| Builder | 생성(Creational) | 단계적 객체 조립 |
| Prototype | 생성(Creational) | 복사(clone)로 생성 |
| **Flyweight** | **구조(Structural)** | **메모리 효율, 공유** |
| Adapter | 구조(Structural) | 인터페이스 변환 |
| Facade | 구조(Structural) | 복잡성 단순화 |
| State | 행위(Behavioral) | 상태에 따른 동작 변화 |
| Command | 행위(Behavioral) | 요청 객체화 |
| Iterator | 행위(Behavioral) | 순차 접근 |
| Interpreter | 행위(Behavioral) | 언어 문법 해석 |
| Observer | 행위(Behavioral) | 상태 변화 통보 |

### 테스트 관련 개념 총정리

| 개념 | 설명 | 혼동 포인트 |
|------|------|------------|
| 테스팅 | 에러를 **찾는** 것 | 수정이 아님 |
| 디버깅 | 에러를 **수정하는** 것 | 찾는 게 아님 |
| 테스트 하니스 | 임시 변경 후 제거 | 테스트 스텁과 다름 |
| 테스트 스텁 | 하향식의 가짜 하위 모듈 | 드라이버와 다름 |
| 드라이버 | 상향식의 가짜 상위 모듈 | 스텁과 다름 |
| 기본 경로 검사 | **화이트박스** 기법 | 블랙박스 아님 |
| 정적 분석 | 코드 실행 없이 검사 | 동적 분석과 다름 |
| 동적 분석 | 코드 실행하면서 검사 | 정적 분석과 다름 |

## ✅ 핵심 정리

- **SOA(서비스 지향 아키텍처)** 는 기능을 독립 서비스로 만들어 조합하는 아키텍처로, 레고 블록처럼 서비스를 조립한다.
- **Fundamental SOA**는 서비스를 단순 연결하는 1단계로, 규모가 커질수록 복잡도가 증가한다.
- **WAS(웹 애플리케이션 서버)** 는 사용자마다 다르게 생성되는 동적 콘텐츠를 처리하는 미들웨어다.
- **테스트 하니스**는 임시로 소프트웨어를 변경하여 특정 기능을 테스트하고 완료 후 제거하는 것이다.
- **리팩토링**은 코드의 실행 결과를 바꾸지 않으면서 내부 구조를 개선하여 가독성과 유지보수성을 높이는 기법이다.
- **역공학**은 완성된 시스템에서 역방향으로 설계 정보를 파악하는 것이다 (완제품 → 설계도).
- **테스팅은 에러를 찾고, 디버깅은 에러를 수정**한다. 두 개념의 목적이 다르다.
- **기본 경로 검사**는 화이트박스 테스트 기법이며, 블랙박스 테스트 기법이 아니다.
- **하향식 통합 테스트에서는 스텁(Stub)**, 상향식 통합 테스트에서는 드라이버(Driver)가 필요하다.
- **상태 패턴(State Pattern)** 은 객체 내부 상태를 클래스로 표현하여, 상태에 따라 동일한 동작이 다르게 처리되게 한다.
- **디자인 패턴은 유지보수를 쉽게** 만드는 장점이 있다. 어렵게 만든다는 설명은 틀렸다.
- **Flyweight**는 구조(Structural) 패턴이며, 생성(Creational) 패턴이 아니다.
- **Factory Method**는 생성(Creational) 패턴이며, 행위(Behavioral) 패턴이 아니다.
- **실체화(Realization)** 는 인터페이스의 스펙만 있는 메서드를 오버라이딩하여 실제로 구현하는 UML 관계다.
- **액티비티 다이어그램**은 업무 흐름과 작업 순서를 표현하며, 단일 객체의 상태 전환은 **상태 다이어그램**이 담당한다.
- **배치 다이어그램**은 소프트웨어가 하드웨어에 배치되는 물리적 구조를 표현하며, 제어 흐름이 아니다.
- **컴포넌트**는 반드시 문서로 기록되어야 하며, "문서 불필요"라는 설명은 틀렸다.
- **MDD 프로세스 순서**는 CIM → PIM → PSM → Code 이다.
- **유지보수 비용**은 소프트웨어를 오래 사용할수록 **증가**한다. 감소하지 않는다.
- **프로그래밍 언어 선택 시 검증은 쉽고 단순**할수록 좋다. 복잡할수록 좋다는 것은 틀렸다.
- **정적 분석**은 코드를 실행하지 않고 비정상 패턴을 탐지하며, 실행하면서 분석하는 것은 동적 분석이다.
- **UI 오류성(Errors)** 은 오류를 적게 범하고 회복이 쉬운 것을 의미하며, 만족감과는 다른 개념이다.
- **상태 전이도(STD)** 는 소프트웨어의 동적 행위(behavior)를 모델링하는 데 사용된다.
