# 🛠️ 유지보수, 리팩토링, 역공학 및 코드 품질

소프트웨어를 만드는 것도 중요하지만, **만든 후에 잘 관리하고 개선하는 것**이 더 어렵고 오래 걸리는 작업입니다. 이 문서에서는 소프트웨어 공학 시험에서 자주 등장하는 핵심 개념들을 하나씩 뜯어보며, 왜 필요한지부터 어떻게 적용하는지까지 초보자도 이해할 수 있도록 자세히 설명합니다.

---

## 🏗️ 소프트웨어 아키텍처 개념 — SOA, WAS, MDD

### SOA (Service Oriented Architecture, 서비스 지향 아키텍처)란?

**SOA**는 기존 애플리케이션의 기능들을 **비즈니스 단위로 묶어** 표준화된 인터페이스로 서비스를 구현하고, 이를 조합하여 하나의 큰 애플리케이션을 구성하는 소프트웨어 개발 아키텍처입니다.

쉽게 말해, **레고 블록**을 생각하면 됩니다. 각 레고 블록은 독립적으로 존재하지만, 블록끼리 표준 방식으로 연결(조합)되어 거대한 성이나 자동차를 만들 수 있죠. SOA도 마찬가지입니다. "결제 서비스", "로그인 서비스", "상품 조회 서비스"처럼 각 기능을 독립적인 서비스로 만들어두고, 필요할 때마다 조합해서 원하는 애플리케이션을 완성합니다.

> **SOA의 핵심 키워드**: 비즈니스 단위 분리 + 표준화된 인터페이스 + 서비스 조합

SOA 아키텍처는 발전 단계가 있습니다. 시험에서 자주 나오므로 꼭 기억하세요.

| 단계 | 이름 | 특징 |
|------|------|------|
| 1단계 | **Fundamental SOA** | 기존 시스템들을 서비스화하여 통합. 시스템 크기가 커질수록 서비스 간 연결이 매우 복잡해짐 |
| 2단계 | **Networked SOA** | 복잡도 해결을 위해 ESB(Enterprise Service Bus, 중앙 허브)를 도입. 스파게티 같은 연결을 정리 |
| 3단계 | **Process Oriented SOA** | 비즈니스 프로세스 중심으로 서비스를 조합 |
| 4단계 | **Business SOA** | 완전한 비즈니스 가치 중심의 서비스 구성 |

:::warning ⚠️ 시험 함정 주의
"시스템 크기 증가에 따라 서비스 간 연결이 매우 복잡해지는 모델"은 **Networked SOA(②)가 아니라 Fundamental SOA(①)** 입니다. Networked SOA는 오히려 그 복잡도를 ESB로 해결하는 단계입니다.
:::

### WAS (Web Application Server, 웹 애플리케이션 서버)란?

일반적인 웹 서버(Web Server)는 **정적(static) 콘텐츠**, 즉 누가 접속해도 항상 똑같이 보이는 HTML 파일, 이미지 등을 처리합니다. 마치 책방의 진열대처럼, 누가 가져가도 같은 책입니다.

반면 **WAS(Web Application Server)** 는 **동적(dynamic) 콘텐츠**를 처리합니다. 예를 들어 쇼핑몰에 로그인하면 "홍길동님 환영합니다"라는 화면이 뜨죠. 이건 사람마다 다릅니다. 김철수가 로그인하면 "김철수님 환영합니다"가 뜹니다. 이처럼 **사용자 요구에 따라 변하는 콘텐츠를 처리하는 미들웨어(middleware, 중간 소프트웨어)** 가 바로 WAS입니다.

> **WAS의 핵심 키워드**: 동적 콘텐츠 처리 + 미들웨어 + 사용자별 다른 응답

### MDD (Model Driven Development, 모델 주도 개발)의 프로세스 순서

MDD는 코드를 직접 작성하기 전에 **모델(설계도)**을 먼저 만들고, 그 모델로부터 자동으로 코드를 생성하는 개발 방식입니다. 건축으로 비유하면, 집을 짓기 전에 청사진(모델)을 그리고, 그 청사진에서 자재 목록(코드)을 자동으로 뽑아내는 것과 같습니다.

MDD 프로세스 순서는 다음과 같습니다:

```
CIM → PIM → PSM → Code
```

- **CIM (Computation Independent Model, 계산 독립 모델)**: 비즈니스 분석 단계. "우리가 어떤 일을 해야 하는가?"를 기술. 기술적인 내용은 없고 순수하게 비즈니스 요구사항만 담음.
- **PIM (Platform Independent Model, 플랫폼 독립 모델)**: 특정 기술 플랫폼에 종속되지 않는 설계. "어떤 구조로 만들 것인가?"를 기술.
- **PSM (Platform Specific Model, 플랫폼 종속 모델)**: 특정 플랫폼(Java, .NET 등)에 맞는 구체적인 설계.
- **Code**: PSM을 바탕으로 실제 코드를 생성.

:::info 암기 팁
**"비평플코"** — 비즈니스(CIM) → 플랫폼 독립(PIM) → 플랫폼 종속(PSM) → 코드(Code)
:::

## 🔧 컴포넌트와 인터페이스 — 재사용 가능한 소프트웨어의 기본

### 컴포넌트(Component)란?

**컴포넌트**는 소프트웨어에서 독립적으로 동작하며 재사용 가능한 단위 모듈입니다. 마치 자동차의 부품처럼, 엔진은 엔진대로, 바퀴는 바퀴대로 독립적으로 존재하다가 조립하면 하나의 자동차가 되는 것과 같습니다.

컴포넌트의 주요 특징은 다음과 같습니다:

- **독립성**: 다른 것들과 명확히 구별되어야 합니다. 엔진과 바퀴가 섞여 있으면 안 되듯이.
- **교체 가능성**: 다른 버전의 컴포넌트로 교체 가능해야 합니다. 타이어를 새 타이어로 바꿀 수 있듯이.
- **인터페이스 접근**: 인터페이스를 통해서만 접근 가능해야 합니다. 자동차 엔진에 직접 손을 넣는 것이 아니라, 핸들·가속 페달 같은 인터페이스로 제어하듯이.
- **문서화 필수**: 반드시 별도의 문서로 기록해야 합니다. ← **"문서가 필요 없다"는 말은 틀렸습니다!**

:::danger ❌ 절대 틀리면 안 되는 함정
"인터페이스를 통해 서비스를 제공하므로 **별도의 문서로 기록할 필요가 없다**"는 설명은 **완전히 틀렸습니다.** 컴포넌트는 다른 개발자들이 재사용해야 하므로 반드시 문서화가 필요합니다.
:::

### UML 관계 — 실체화(Realization)란?

UML(Unified Modeling Language, 통합 모델링 언어)에서 **실체화(Realization)** 는 인터페이스의 스펙(명세)만 있는 메서드를 오버라이딩(overriding, 재정의)하여 실제 기능으로 구현하는 관계를 말합니다.

쉬운 비유를 들면, "날 수 있다"는 인터페이스(능력 명세)를 새(Bird) 클래스와 비행기(Airplane) 클래스가 각자의 방식으로 구체적으로 구현하는 것이 바로 실체화입니다.

```java
// 인터페이스(명세만 존재, 구현 없음)
interface Flyable {
    void fly(); // 스펙(명세)만 존재, 내용은 없음
}

// 실체화(Realization): 실제 기능으로 구현
class Bird implements Flyable {
    @Override
    public void fly() {
        // 새는 날개를 퍼덕여서 날기 - 실제 구현
        System.out.println("날개를 퍼덕여서 날아갑니다.");
    }
}

// 또 다른 실체화
class Airplane implements Flyable {
    @Override
    public void fly() {
        // 비행기는 엔진 추력으로 날기 - 실제 구현
        System.out.println("엔진 추력으로 날아갑니다.");
    }
}
```

UML 주요 관계를 헷갈리지 않도록 표로 정리합니다:

| 관계 | 이름 | 설명 | 예시 |
|------|------|------|------|
| 실선 + 빈 화살표 | **일반화 (Generalization)** | 부모-자식 상속 관계 | 동물 → 개 |
| 점선 + 빈 화살표 | **실체화 (Realization)** | 인터페이스를 실제로 구현 | Flyable ← Bird |
| 점선 + 화살표 | **의존 (Dependency)** | 한 클래스가 다른 클래스를 일시적으로 사용 | 주문 → 배송 |
| 실선 | **연관 (Association)** | 클래스 간 지속적인 관계 | 학생 — 강의 |

## 🔄 유지보수와 비용 — 소프트웨어는 완성 후가 더 중요하다

### 유지보수(Maintenance)란 왜 중요한가?

소프트웨어 개발은 출시(릴리즈)로 끝나지 않습니다. 오히려 **출시 이후의 유지보수 단계**가 전체 소프트웨어 생명주기에서 가장 긴 시간과 비용을 차지합니다. 자동차를 예로 들면, 자동차를 사는 비용보다 수십 년간의 유지·보수·수리 비용이 훨씬 더 크게 드는 것과 같습니다.

**유지보수 비용의 주요 특징:**

- 유지보수 비용은 **예측하기 매우 어렵습니다.** 언제 어떤 버그가 발생할지, 요구사항이 어떻게 바뀔지 사전에 정확히 알 수 없기 때문입니다.
- **분석 단계에서 사용자의 요구사항을 정확히 파악**하는 것이 중요합니다. 처음부터 잘못 만들어진 소프트웨어는 나중에 고치는 비용이 기하급수적으로 증가합니다.
- **복잡도가 높은 프로그램일수록 유지보수 비용이 더 큽니다.** 실타래처럼 얽힌 코드는 한 부분을 고치면 다른 부분이 망가지는 일이 빈번하게 발생합니다.
- **소프트웨어는 오래 사용할수록 유지보수 비용이 증가합니다.** ← 이것이 핵심입니다!

:::warning ⚠️ 자주 틀리는 함정
"소프트웨어는 오래 사용할수록 유지보수 비용이 **감소**된다"는 설명은 **틀렸습니다.** 실제로는 오래 사용할수록 코드 구조는 복잡해지고, 원래 담당자는 퇴사하고, 레거시(legacy, 오래된) 코드에 새로운 기능이 덧붙여지면서 비용이 **증가**합니다.
:::

### 역공학(Reverse Engineering)과 재공학(Re-Engineering)

**역공학(Reverse Engineering)** 은 원래 하드웨어 분야에서 완제품으로부터 설계도를 해독해내는 작업을 의미했습니다. 소프트웨어에서는 완성된 시스템(소스코드 또는 실행파일)을 분석하여 컴포넌트와 관계, 설계 구조를 파악하는 작업입니다.

마치 완성된 케이크를 먹어보고 레시피를 역으로 추적하는 것과 같습니다. 또는 경쟁사의 제품을 분해해서 어떻게 만들었는지 파악하는 것이죠.

> **역공학의 방향**: 완성된 코드(결과) → 설계도(원인) 파악 (역방향)

반면 **재공학(Re-Engineering)** 은 기존 시스템을 분석하여 더 나은 시스템으로 다시 만드는 과정으로, 역공학을 포함하는 더 넓은 개념입니다.

| 구분 | 역공학 (Reverse Engineering) | 재공학 (Re-Engineering) |
|------|------|------|
| 방향 | 결과물 → 설계 (역방향) | 기존 시스템 → 새 시스템 (전방향 포함) |
| 목적 | 설계/구조 이해 | 시스템 개선 및 재구성 |
| 범위 | 분석에 집중 | 분석 + 재설계 + 재구현 |
| 비유 | 케이크 → 레시피 파악 | 낡은 건물 철거 후 새 건물 건축 |

## 🧪 테스팅, 디버깅 및 통합 테스트 — 버그를 잡는 두 가지 방법

### 테스팅(Testing)과 디버깅(Debugging)의 차이

많은 초보자들이 테스팅과 디버깅을 혼동하는데, 이 둘은 명확히 다릅니다.

- **테스팅(Testing)** 은 소프트웨어에 **에러(오류)가 존재하는지 찾는(발견하는)** 작업입니다. 마치 건강검진을 통해 "어디가 이상한지 검사하는 것"과 같습니다. 테스팅의 역할은 **검사관**입니다.
- **디버깅(Debugging)** 은 발견된 에러를 **수정하는** 작업입니다. 건강검진 결과를 바탕으로 "이상 부위를 치료하는 것"과 같습니다. 디버깅의 역할은 **수리공**입니다.

> 테스팅 = 에러를 **찾는** 것 / 디버깅 = 에러를 **수정하는** 것

### 블랙박스 테스트와 화이트박스 테스트

테스트 기법은 크게 **블랙박스(Black Box)** 테스트와 **화이트박스(White Box)** 테스트로 나뉩니다.

**블랙박스 테스트**는 내부 구조를 모르는 상태에서 입력과 출력만 보고 테스트하는 방법입니다. 마치 전자레인지를 사용할 때 내부 회로를 모르고 "음식을 넣고 버튼을 눌렀을 때 음식이 데워지는가?"만 확인하는 것과 같습니다.

**화이트박스 테스트**는 내부 코드 구조를 보면서 테스트하는 방법입니다. 전자레인지 내부 회로를 직접 들여다보며 각 부품이 제대로 작동하는지 확인하는 것과 같습니다.

| 구분 | 블랙박스 테스트 | 화이트박스 테스트 |
|------|------|------|
| 관점 | 외부(사용자 관점) | 내부(개발자 관점) |
| 대상 | 입력/출력 | 코드 구조/경로 |
| 주요 기법 | 동등 분할, 경곗값 분석, 문법 오류 검사 | **기본 경로 검사**, 조건 검사, 루프 검사 |

:::tip 💡 기억 포인트
**기본 경로 검사(Basis Path Testing)** 는 코드 내부의 실행 경로를 분석하는 것이므로 **화이트박스 테스트** 기법입니다. 블랙박스가 아닙니다!
:::

### 하향식 통합 테스트 vs 상향식 통합 테스트

통합 테스트(Integration Test)는 여러 모듈을 합쳐서 테스트하는 방식입니다.

**하향식(Top-Down) 통합 테스트**: 최상위 모듈부터 아래 방향으로 테스트를 진행합니다. 상위 모듈을 먼저 테스트하고 하위 모듈을 붙여나가는 방식입니다. 이때 아직 개발되지 않은 하위 모듈 대신 **스텁(Stub, 가짜 하위 모듈)** 을 사용합니다.

**상향식(Bottom-Up) 통합 테스트**: 최하위 모듈부터 위 방향으로 테스트를 진행합니다. 이때 상위 모듈 대신 **드라이버(Driver, 가짜 상위 모듈)** 를 사용합니다.

```
하향식 테스트:
[최상위 모듈] ← 먼저 테스트
      ↓
[중간 모듈]
      ↓
[하위 모듈] ← 스텁(Stub)으로 대체

상향식 테스트:
[최상위 모듈] ← 드라이버(Driver)로 대체
      ↑
[중간 모듈]
      ↑
[하위 모듈] ← 먼저 테스트
```

:::danger ❌ 절대 혼동 금지
- **하향식 통합 테스트** → **스텁(Stub)** 필요 (가짜 하위 모듈)
- **상향식 통합 테스트** → **드라이버(Driver)** 필요 (가짜 상위 모듈)

"하향식 테스트에서 드라이버가 필요하다"는 말은 **틀렸습니다!**
:::

---

### 테스트 하니스(Test Harness)란?

**테스트 하니스(Test Harness)** 는 시스템의 일부 기능만 시험하기 위해 소프트웨어를 임시로 변경하는 것으로, 검사가 끝나면 제거하는 도구입니다.

쉬운 비유로는 자동차 공장에서 새 모델을 테스트할 때 임시로 특수 센서와 장치를 달았다가, 테스트가 끝나면 모두 제거하는 것과 같습니다. **테스트 스텁(Test Stub)** 과는 다른 개념입니다. 테스트 스텁은 하향식 통합 테스트에서 사용하는 가짜 하위 모듈이고, 테스트 하니스는 테스트 목적으로 소프트웨어 자체를 임시 변경하는 더 넓은 개념입니다.

## ♻️ 리팩토링과 코드 품질 — 더 좋은 코드로 만드는 기술

### 리팩토링(Refactoring)이란?

**리팩토링(Refactoring)** 은 소프트웨어의 외부 동작(결과)은 변경하지 않으면서 내부 코드 구조를 개선하여 가독성을 높이고 유지보수를 쉽게 만드는 기술입니다.

요리를 예시로 들면, 같은 맛의 요리를 만들되 주방을 더 깔끔하게 정리하고 조리 순서를 효율적으로 바꾸는 것입니다. 손님에게 나가는 음식(결과)은 동일하지만, 주방(코드 구조)은 훨씬 깨끗해집니다.

> **리팩토링의 핵심**: 결과(동작)는 그대로 + 구조(코드)는 개선

```python
# ❌ 리팩토링 전: 읽기 어렵고 중복이 많은 코드
def calc(a, b, t):
    if t == 1:
        return a + b  # 더하기
    if t == 2:
        return a - b  # 빼기
    if t == 3:
        return a * b  # 곱하기

# 위 코드 문제점:
# 1. 함수 이름이 calc로 너무 모호함
# 2. t라는 변수명이 무엇을 의미하는지 불명확
# 3. 확장 시 if문이 계속 늘어나는 구조

# ✅ 리팩토링 후: 읽기 쉽고 구조가 명확한 코드
def calculate(first_number, second_number, operation_type):
    """두 수를 주어진 연산 타입으로 계산하는 함수"""
    
    # 연산 타입별 함수를 딕셔너리로 관리 (확장성 향상)
    operations = {
        'add': lambda a, b: a + b,        # 더하기 연산
        'subtract': lambda a, b: a - b,   # 빼기 연산
        'multiply': lambda a, b: a * b,   # 곱하기 연산
    }
    
    # 연산 타입이 유효한지 확인
    if operation_type not in operations:
        raise ValueError(f"지원하지 않는 연산: {operation_type}")
    
    # 해당 연산 실행 및 반환 (결과는 동일하지만 구조가 명확)
    return operations[operation_type](first_number, second_number)

# 사용 예시
result = calculate(10, 5, 'add')       # 10 + 5 = 15
result = calculate(10, 5, 'subtract')  # 10 - 5 = 5
```

위 두 코드는 **동일한 결과**를 냅니다. 하지만 리팩토링 후의 코드는 변수명이 명확하고, 새 연산을 추가하기 쉽고, 읽기도 훨씬 편합니다. 이것이 리팩토링입니다.

---

### 코드 품질 향상 기법 — 정적 분석 vs 동적 분석

코드 품질을 높이는 대표적인 방법으로 **코드 인스펙션(Code Inspection)** 과 **분석 기법**이 있습니다.

**정적 분석(Static Analysis)**: 코드를 **실행하지 않고** 검사하는 방법입니다. 비정상적인 패턴, 읽히지 않는 코드(dead code), 잠재적 버그 등을 탐지합니다. 마치 비행기가 이륙하기 전에 지상에서 점검하는 것과 같습니다.

**동적 분석(Dynamic Analysis)**: 코드를 **실행하면서** 검사하는 방법입니다. 실제 실행 중에 발생하는 메모리 누수, 성능 문제 등을 탐지합니다. 비행기를 실제로 띄워보면서 문제를 찾는 것과 같습니다.

:::warning ⚠️ 자주 틀리는 함정
"비정상적 패턴이나 읽히지 않는 패턴을 찾는 방법을 **동적 분석**이라 한다"는 설명은 **틀렸습니다.** 이는 **정적 분석**의 설명입니다. 정적 분석은 코드 실행 없이 코드를 읽고 분석하는 방법입니다.
:::

| 구분 | 정적 분석 | 동적 분석 |
|------|------|------|
| 실행 여부 | 코드 실행 없음 | 코드 실행 중 |
| 탐지 대상 | 코드 패턴, 죽은 코드, 잠재 버그 | 메모리 누수, 런타임 오류, 성능 |
| 시점 | 컴파일 전후 | 런타임(실행 중) |
| 비유 | 이륙 전 지상 점검 | 실제 비행 중 점검 |

**코드 인스펙션(Code Inspection)** 은 컴파일 및 정적 분석 **이후**에 이루어져야 합니다. 먼저 기계적으로 검사할 수 있는 것들을 다 처리하고, 그다음 사람이 직접 코드를 꼼꼼히 읽어보는 순서가 효율적입니다.

코드에서 발생하는 결함의 대표 타입은 다음과 같습니다:
- **로직 문제**: 조건문이 잘못되거나 알고리즘이 틀린 경우
- **컴퓨팅 문제**: 0으로 나누기, 배열 인덱스 초과, 메모리 해제 누락 등
- **인터페이스 문제**: 함수 간 데이터 전달 방식의 오류

### 흔한 코딩 오류들

시험에서 "흔한 코딩 오류"를 고르는 문제가 나옵니다. 흔한 코딩 오류의 예시는 다음과 같습니다:

```c
// ❌ 흔한 오류 1: 메모리 해제 누락 (메모리 누수, Memory Leak)
void bad_function() {
    int* ptr = malloc(100 * sizeof(int));  // 메모리 할당
    // 사용 후 free(ptr)를 호출하지 않음 → 메모리가 계속 쌓임
    return;
}

// ✅ 올바른 코드
void good_function() {
    int* ptr = malloc(100 * sizeof(int));  // 메모리 할당
    // ... 사용 ...
    free(ptr);  // 반드시 해제 필요
    return;
}

// ❌ 흔한 오류 2: 배열 인덱스 음수
int arr[10];
int index = -1;       // 음수 인덱스는 오류!
arr[index] = 5;       // 잘못된 메모리 접근 → 프로그램 충돌

// ❌ 흔한 오류 3: 0으로 나누기
int a = 10;
int b = 0;
int result = a / b;   // 0으로 나누기 → 런타임 오류!
```

:::note 📝 참고
"매개변수 값이 null이 아닌 경우"는 **정상적인 상황**입니다. 매개변수가 null이 아닌 것은 오류가 아니라 오히려 올바른 입력입니다. 따라서 이는 흔한 코딩 오류에 해당하지 않습니다.
:::

## 🎨 디자인 패턴 — 검증된 설계의 레시피

### 디자인 패턴(Design Pattern)이란?

**디자인 패턴(Design Pattern)** 은 소프트웨어 설계에서 자주 발생하는 문제들에 대한 **검증된 해결책**을 유형별로 템플릿화한 것입니다. 마치 요리 레시피처럼, 처음부터 혼자 고민하지 않고 이미 검증된 방법을 따라 하는 것입니다.

디자인 패턴의 주요 장점:
- 검증된 설계를 **재사용**하여 개발 시간을 단축할 수 있습니다.
- 설계자 간 **원활한 의사소통**에 활용됩니다. "이 부분은 싱글톤 패턴으로 구현했어요"라고 하면 모든 개발자가 바로 이해합니다.
- 설계 변경 요청에 대한 **유지보수가 쉽습니다.** ← 매우 중요!

:::danger ❌ 절대 혼동 금지
"디자인 패턴은 설계 변경 요청에 대한 유지보수가 **어려운** 단점이 있다"는 설명은 **완전히 틀렸습니다.** 디자인 패턴의 핵심 장점 중 하나가 바로 **유지보수가 쉽다**는 것입니다.
:::

---

### 디자인 패턴의 3가지 분류

디자인 패턴은 목적에 따라 **생성(Creational)**, **구조(Structural)**, **행위(Behavioral)** 세 가지로 분류됩니다.

| 분류 | 목적 | 주요 패턴 |
|------|------|------|
| **생성(Creational)** | 객체를 어떻게 생성할 것인가 | Factory Method, Abstract Factory, Singleton, Builder, Prototype |
| **구조(Structural)** | 객체와 클래스를 어떻게 조합할 것인가 | **Flyweight**, Adapter, Bridge, Composite, Decorator, Facade, Proxy |
| **행위(Behavioral)** | 객체 간 상호작용을 어떻게 할 것인가 | **Interpreter**, Command, Iterator, Observer, **State**, Strategy, Template Method 등 |

:::warning ⚠️ 시험 함정 주의
- **Flyweight** 패턴은 **구조(Structural)** 패턴입니다. 생성 패턴이 아닙니다!
- **Factory Method** 패턴은 **생성(Creational)** 패턴입니다. 행위 패턴이 아닙니다!
:::

---

### 상태 패턴(State Pattern)이란?

**상태 패턴(State Pattern)** 은 같은 동작이라도 **객체의 현재 상태에 따라 다르게 처리**되도록 하는 행위(Behavioral) 디자인 패턴입니다. 핵심은 **객체의 상태를 클래스로 생성하여 참조**하게 만드는 것입니다.

신호등을 예로 들면, 신호등의 "신호 변경" 동작은 항상 같지만, 현재 상태(빨간불, 초록불, 노란불)에 따라 다음 상태가 달라집니다.

```java
// 상태 인터페이스 정의
interface TrafficLightState {
    void changeSignal(TrafficLight light); // 신호 변경 동작
}

// 빨간불 상태 클래스
class RedLight implements TrafficLightState {
    @Override
    public void changeSignal(TrafficLight light) {
        System.out.println("빨간불 → 초록불로 변경");
        light.setState(new GreenLight()); // 다음 상태로 전환
    }
}

// 초록불 상태 클래스
class GreenLight implements TrafficLightState {
    @Override
    public void changeSignal(TrafficLight light) {
        System.out.println("초록불 → 노란불로 변경");
        light.setState(new YellowLight()); // 다음 상태로 전환
    }
}

// 노란불 상태 클래스
class YellowLight implements TrafficLightState {
    @Override
    public void changeSignal(TrafficLight light) {
        System.out.println("노란불 → 빨간불로 변경");
        light.setState(new RedLight()); // 다음 상태로 전환
    }
}

// 신호등 클래스 (현재 상태를 참조)
class TrafficLight {
    private TrafficLightState currentState; // 현재 상태를 클래스로 참조
    
    public TrafficLight() {
        this.currentState = new RedLight(); // 초기 상태는 빨간불
    }
    
    public void setState(TrafficLightState state) {
        this.currentState = state; // 상태 변경
    }
    
    public void changeSignal() {
        currentState.changeSignal(this); // 현재 상태에게 동작 위임
    }
}
```

상태 패턴의 핵심은 상태(RedLight, GreenLight, YellowLight)를 **별도의 클래스**로 만들어, 신호등 객체가 현재 상태에 따라 자동으로 다른 동작을 하도록 만드는 것입니다.

## 🖥️ UML 다이어그램과 UI 평가 기준

### UML 다이어그램 완전 정리

UML(Unified Modeling Language)에는 다양한 다이어그램이 있으며, 각각의 목적과 표현 대상이 명확히 다릅니다.

| 다이어그램 | 목적 | 핵심 키워드 |
|------|------|------|
| **클래스 다이어그램** | 시스템 구조를 클래스 속성 및 관계로 표현 | 구조, 속성, 관계 |
| **순서(시퀀스) 다이어그램** | 객체 간 상호작용에 의한 시스템 동작 표현, 시간 흐름에 따른 메시지 순서 | 시간 흐름, 메시지 순서 |
| **사용 사례(Use Case) 다이어그램** | 사용자(Actor) 관점의 시스템 기능 표현 | 사용자 관점, 기능 |
| **액티비티 다이어그램** | 업무 흐름, 작업 순서, 활동 간 제어 흐름 표현 | 업무 흐름, 작업 순서 |
| **상태 다이어그램** | 객체의 상태와 상태 전환으로 단일 객체의 동작 표현 | 상태, 상태 전환 |
| **컴포넌트 다이어그램** | 소프트웨어 간 종속 관계 표현 | 소프트웨어 종속성 |
| **배치(Deployment) 다이어그램** | 소프트웨어가 하드웨어에 배치되는 구조 표현 | 하드웨어 배치, 물리적 구조 |

:::warning ⚠️ 자주 틀리는 함정 2가지
1. **액티비티 다이어그램** ≠ 상태 전환 / "객체의 상태와 상태 전환"은 **상태 다이어그램**의 설명입니다.
2. **배치 다이어그램** ≠ 제어 흐름 / "두 클래스 간 제어 흐름"은 **액티비티 다이어그램**에 해당합니다. 배치 다이어그램은 소프트웨어가 어떤 하드웨어 서버에 배치되는지를 표현합니다.
:::

---

### 소프트웨어 행위 모델링 — 상태 전이도(STD)

소프트웨어의 **행위(behavior)** 를 모델링하는 데 사용되는 것은 **상태 전이도(State Transition Diagram, STD)** 입니다. 상태 전이도는 시스템이 어떤 상태에 있을 때 어떤 이벤트가 발생하면 어떤 상태로 전이하는지를 표현합니다.

- **자료 사전(Data Dictionary)**: 데이터 항목의 정의와 설명
- **자료 흐름도(Data Flow Diagram)**: 데이터가 시스템 내에서 어떻게 흐르는지 표현
- **처리 명세(Process Specification)**: 각 처리 과정의 알고리즘 기술
- **상태 전이도(State Transition Diagram)**: 시스템의 **동적 행위(상태 변화)** 표현 ✅

### UI 평가 기준 — 사용성(Usability)의 5가지 요소

좋은 UI(User Interface, 사용자 인터페이스)를 평가하는 기준으로 다음 5가지가 있습니다. 각 항목의 정확한 정의를 기억하는 것이 중요합니다.

| 평가 기준 | 설명 |
|------|------|
| **용이성 (Learnability)** | 처음 사용자도 빠르게 배울 수 있는 정도 |
| **효율성 (Efficiency)** | 숙련 사용자도 고수준 작업이 가능해야 함 |
| **기억성 (Memorability)** | 오래 안 써도 다시 사용할 수 있도록 기억하기 쉬워야 함 |
| **오류성 (Errors)** | 오류를 적게 범하고, 오류 발생 시 회복이 쉬워야 함 |
| **만족성 (Satisfaction)** | 시스템 고장 없이 사용자가 만족하고 즐겁게 사용할 수 있어야 함 |

:::warning ⚠️ 자주 틀리는 함정
"시스템 고장 없이 사용자가 만족하고 즐겁게 사용할 수 있어야 함"은 **만족성(Satisfaction)** 의 설명입니다. **오류성(Errors)** 은 "오류를 적게 범하고 오류 발생 시 회복이 쉬운 것"입니다. 둘을 혼동하지 마세요!
:::

### 효과적인 UI 제작 방법

효과적인 UI를 만들기 위한 방법들입니다:

- **사용자의 시선 흐름**: 사람은 보통 **좌측 상단 → 우측 하단** 순서로 화면을 훑어봅니다. 중요한 요소는 좌측 상단에 배치하는 것이 효과적입니다.
- **입력 피드백 제공**: 입력 필드 옆에 글자 수를 표시하는 등 사용자에게 즉각적인 피드백을 줍니다.
- **친숙한 언어와 아이콘 사용**: 사용자가 쉽게 이해할 수 있는 언어와 아이콘을 사용합니다.
- **지속적인 개선**: UI는 기존 것에만 안주하지 않고 개선 방향으로 나아가야 합니다. "기존의 것들과 차이가 나지 않도록 하여 혼동을 줄인다"는 설명은 잘못된 것입니다. UI는 발전해야 합니다.

### 프로그래밍 언어 선택 기준

개발 프로젝트에서 적합한 프로그래밍 언어를 선택할 때 고려해야 할 기준들입니다:

- **명료성, 호환성, 효율성, 유지보수성**: 코드가 명확하고, 다른 시스템과 호환되며, 효율적이고 유지보수가 쉬워야 합니다.
- **사용자의 요구사항 반영**: 사용자가 원하는 기능을 구현하기 쉬운 언어여야 합니다. (예: 과학 계산은 Python/MATLAB, 모바일은 Swift/Kotlin)
- **다양한 개발 도구 지원**: IDE, 라이브러리, 프레임워크 등 풍부한 생태계가 있는 언어가 유리합니다.
- **검증의 용이성**: **검증은 단순하고 쉬울수록 좋습니다.** "검증이 복잡하고 어려울수록 좋다"는 완전히 잘못된 기준입니다.

## 📌 핵심 정리

이 문서에서 학습한 핵심 개념들을 정리합니다.

- **SOA(서비스 지향 아키텍처)**: 기능을 비즈니스 단위로 묶어 표준화된 인터페이스로 서비스화하고 조합하는 아키텍처
- **Fundamental SOA**: SOA 1단계, 시스템 크기 증가에 따라 서비스 간 연결이 복잡해지는 단계
- **WAS(웹 애플리케이션 서버)**: 사용자 요구에 따라 변하는 동적 콘텐츠를 처리하는 미들웨어
- **MDD 순서**: CIM → PIM → PSM → Code (비즈니스 분석 → 플랫폼 독립 설계 → 플랫폼 종속 설계 → 코드 생성)
- **컴포넌트**: 독립적이고 교체 가능하며 인터페이스로 접근하고, 반드시 문서화가 필요함
- **실체화(Realization)**: 인터페이스의 명세를 실제로 구현하는 UML 관계
- **역공학(Reverse Engineering)**: 완성된 시스템으로부터 설계 구조를 역방향으로 분석하는 기법
- **유지보수 비용**: 소프트웨어는 오래 사용할수록 비용이 증가함 (감소 아님!)
- **테스팅 vs 디버깅**: 테스팅은 에러를 찾는 것, 디버깅은 에러를 수정하는 것
- **기본 경로 검사**: 화이트박스 테스트 기법 (블랙박스 아님!)
- **하향식 통합 테스트**: 스텁(Stub) 필요 / 상향식 통합 테스트: 드라이버(Driver) 필요
- **테스트 하니스**: 일부 기능 테스트를 위해 임시 변경 후 제거하는 도구
- **리팩토링**: 결과(동작)는 그대로, 내부 구조를 개선하는 기술
- **정적 분석**: 코드 실행 없이 비정상 패턴 탐지 / **동적 분석**: 실행 중 탐지
- **디자인 패턴**: 유지보수가 쉬운 것이 장점 (어렵다는 설명은 오답!)
- **Flyweight**: 구조(Structural) 패턴 / Factory Method, Singleton: 생성(Creational) 패턴
- **상태 패턴(State Pattern)**: 객체 상태를 클래스로 만들어 상태별 다른 동작을 처리하는 행위 패턴
- **배치 다이어그램**: 소프트웨어가 하드웨어에 배치되는 구조 표현 (제어 흐름 아님!)
- **액티비티 다이어그램**: 업무 흐름과 작업 순서 표현 (상태 전환 아님!)
- **상태 전이도(STD)**: 소프트웨어의 동적 행위(상태 변화)를 모델링
- **UI 오류성(Errors)**: 오류를 적게 범하고 회복이 쉬운 것 / 만족성과 혼동 금지!
- **프로그래밍 언어 선택**: 검증은 단순하고 쉬울수록 좋음
