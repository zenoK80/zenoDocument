# 🖥️ 클라이언트-서버 아키텍처의 장점과 단점

현대 소프트웨어 개발 환경에서 **클라이언트-서버 아키텍처(Client-Server Architecture)**는 거의 모든 서비스의 근간을 이루고 있습니다. 우리가 매일 사용하는 네이버, 구글, 카카오톡, 유튜브 모두 이 구조 위에서 동작합니다. 이번 문서에서는 클라이언트-서버 아키텍처가 무엇인지, 왜 필요한지, 어떤 장점과 단점이 있는지, 그리고 실제로 어떤 계층 구조와 미들웨어로 구성되어 있는지를 초보자도 이해할 수 있도록 아주 자세하게 설명합니다.

- 미들웨어(Middleware)의 역할과 종류

## 🧩 클라이언트-서버 아키텍처란 무엇인가?

### 기본 개념부터 잡기

**클라이언트-서버 아키텍처**란, 이름 그대로 **클라이언트(서비스를 요청하는 쪽)**와 **서버(서비스를 제공하는 쪽)**로 역할을 나누어 시스템을 구성하는 설계 방식입니다. 공식적인 정의를 빌리자면, "서비스, 서버, 클라이언트의 집합으로 구성되는 시스템 모델"이라고 할 수 있습니다.

비유를 들어볼까요? 마치 **음식점**과 비슷합니다. 손님(클라이언트)은 메뉴를 보고 음식을 주문(요청)하고, 주방(서버)은 그 주문을 받아서 음식을 만들어 제공(응답)합니다. 손님은 주방이 어떻게 음식을 만드는지 알 필요가 없고, 단지 음식이 나오기만 하면 됩니다. 그리고 주방이 손님한테 "재료 좀 가져다주세요"라고 요청하지는 않죠. 이것이 클라이언트-서버의 기본 관계입니다.

### 클라이언트와 서버의 역할 구분

| 구분 | 역할 | 예시 |
|------|------|------|
| **클라이언트** | 서비스를 요청하는 쪽 | 웹 브라우저, 모바일 앱, PC 프로그램 |
| **서버** | 서비스를 제공하는 쪽 | 웹 서버, 데이터베이스 서버, 파일 서버 |

### 클라이언트가 알아야 하는 것과 몰라도 되는 것

클라이언트는 **어떤 서버가 운영되고 있는지(서버의 존재)**는 알아야 합니다. 내가 네이버에 접속하려면 네이버 서버의 주소(URL)는 알아야 하죠. 그러나 **현재 그 서버에 몇 명의 사용자가 접속해 있는지**는 전혀 알 필요가 없고, 실제로도 알지 못합니다.

여러분이 네이버에 접속할 때 "지금 몇 명이 동시에 접속 중이에요?"라는 화면이 나오지 않죠? 나는 그냥 나 혼자 서버를 쓰는 것처럼 느낍니다. 이처럼 클라이언트는 다른 클라이언트의 존재를 인식하지 못합니다.

### 서버도 클라이언트가 될 수 있다?

흥미로운 점이 있습니다. **서버도 상황에 따라 클라이언트 역할을 할 수 있습니다.** 단, 서버가 다른 서버에게 무언가를 요청할 때만 해당됩니다. 예를 들어, 쇼핑몰 서버가 결제 처리를 위해 카드사 서버에 결제 승인을 요청할 때, 쇼핑몰 서버는 카드사 서버 입장에서 **클라이언트**가 됩니다.

> 🔑 핵심 원칙: **서버는 클라이언트에게 서비스를 요청하지 않습니다.** 서버는 항상 서비스를 제공하는 입장이며, 필요 시 다른 서버에게만 요청할 수 있습니다.

## ✅ 클라이언트-서버 아키텍처의 장점

### 왜 클라이언트-서버 구조가 등장했는가?

과거에는 **메인프레임(Mainframe)**이라는 거대하고 비싼 중앙 컴퓨터 한 대가 모든 업무를 처리했습니다. 마치 도시 전체에 전기를 공급하는 발전소가 하나뿐인 것처럼요. 이 방식은 비용이 엄청나게 비쌌고, 그 한 대가 고장나면 모든 업무가 멈추는 치명적인 단점이 있었습니다.

클라이언트-서버 아키텍처는 이 문제를 해결하기 위해 등장했습니다. 비싼 메인프레임 대신, **여러 대의 작고 저렴한 서버들을 분산 배치**하여 동일한 효과를 얻는 것이 목표였습니다.

### 장점 ① 비용 절감 💰

**메인프레임급 중대형 컴퓨터를 설치하는 것보다 훨씬 적은 비용**으로 동일한 수준의 서비스를 제공할 수 있습니다. 수십억 원짜리 메인프레임 한 대 대신, 수백만 원짜리 서버 여러 대를 분산시켜 배치하면 훨씬 경제적입니다. 기업 입장에서는 초기 투자 비용을 획기적으로 줄일 수 있는 매우 매력적인 구조입니다.

### 장점 ② 지역 분산 처리 🌏

**처리할 자료가 발생한 그 지역에서 바로 처리**할 수 있습니다. 예를 들어, 서울 지사, 부산 지사, 제주 지사가 있는 회사라면 각 지사 근처에 서버를 하나씩 두고, 각 지역의 업무를 그 지역 서버에서 처리할 수 있습니다. 모든 데이터를 서울 본사의 메인프레임 한 대로 보내서 처리하는 것보다 훨씬 빠르고 효율적입니다.

마치 전국에 지점을 둔 편의점처럼, 중앙 창고(본사) 하나에서 모든 걸 처리하는 것보다 각 지역 물류 창고(지역 서버)를 두는 것이 훨씬 빠르게 배송(처리)할 수 있는 것과 같습니다.

### 장점 ③ 고성능 GUI 지원 🖱️

**고성능 워크스테이션에서 가능한 그래픽 사용자 인터페이스(GUI)**를 지원합니다. 클라이언트 측 컴퓨터의 성능을 활용하여 풍부하고 반응성 높은 화면을 제공할 수 있습니다. 텍스트만 나오던 과거의 터미널 화면에서 벗어나, 아이콘, 버튼, 애니메이션이 있는 현대적인 UI를 제공할 수 있게 된 것입니다.

:::tip 장점 한 줄 요약
클라이언트-서버 아키텍처의 핵심 장점은 **"저렴한 비용으로 분산된 서버를 통해 고성능 서비스를 빠르고 효율적으로 제공할 수 있다"**는 것입니다.
:::

## ⚠️ 클라이언트-서버 아키텍처의 단점

장점만 있는 기술은 없습니다. 클라이언트-서버 아키텍처에도 분명한 단점이 존재합니다. 이 단점들을 제대로 이해해야 실무에서 적절한 대응 방안을 마련할 수 있습니다.

### 단점 ① 서버 부하 집중 문제 🔥

처음에는 클라이언트가 많은 역할을 분담했습니다. 그런데 시간이 지나면서 **응용 처리(애플리케이션 로직)의 많은 부분들이 서버로 점점 옮겨가게 되었습니다.** 그 결과, 서버에 걸리는 부하(처리 부담)가 과거 메인프레임 수준만큼 커져버렸습니다.

마치 회사에서 처음에는 팀장과 팀원이 일을 나눠서 하다가, 팀원이 점점 일을 팀장에게 떠넘기는 상황과 같습니다. 결국 팀장(서버) 혼자 모든 일을 처리하게 되면서 과부하가 걸리는 것이죠.

### 단점 ② 시스템 복잡성 증가 🌀

서버를 **분산 환경(여러 곳에 흩어진 서버들이 네트워크로 연결된 환경)**에서 운영하기 때문에 시스템 구조가 매우 복잡해집니다. 서버 수십, 수백 대가 네트워크로 연결되어 병렬로 처리하고, 처리 결과를 합치는 작업이 동시에 이루어집니다. 이 복잡한 구조를 설계하고, 운영하고, 문제 발생 시 원인을 찾아 해결하는 것은 매우 어려운 일입니다.

### 단점 ③ 보안 취약성 🔓

이것이 가장 심각한 단점입니다. **분산된 서버 환경에서는 보안을 유지하기가 극도로 어렵습니다.**

예를 들어, 서버 100대를 분산 배치했다면, 100대 중 **단 1대에만 보안 구멍이 뚫려도** 전체 시스템의 보안이 위협받습니다. 마치 집에 100개의 창문이 있는데, 그 중 창문 1개만 잠그지 않아도 도둑이 들어올 수 있는 것과 같습니다.

최근 사회적으로 문제가 되는 **해킹 사고들도 대부분 이러한 분산 환경의 취약점을 노린 공격**에서 비롯됩니다. 분산 서버들 사이의 틈새를 찾아 침투하는 방식이죠.

:::warning 보안에 각별한 주의가 필요합니다
클라이언트-서버 아키텍처를 구성할 때 **각각의 서버 모두에 동일한 수준의 보안 정책**을 적용해야 합니다. 단 하나의 서버라도 보안이 허술하면 전체 시스템이 위험해집니다.
:::

| 단점 | 설명 | 비유 |
|------|------|------|
| **서버 부하** | 점점 많은 처리가 서버에 집중됨 | 팀장 혼자 모든 일을 처리하는 상황 |
| **복잡성** | 분산 환경의 구성과 관리가 어려움 | 100개의 조각으로 나뉜 퍼즐 맞추기 |
| **보안 취약성** | 분산된 서버 중 하나만 뚫려도 전체 위험 | 100개의 창문 중 1개만 열려 있어도 도둑이 들어옴 |

## 🏗️ 클라이언트-서버의 4계층 구조

### 왜 계층을 나누는가?

복잡한 시스템을 한 번에 이해하고 관리하기란 불가능합니다. 그래서 역할에 따라 **계층(Layer)**으로 나누어 각 계층이 자신의 역할만 담당하도록 설계합니다. 마치 건물을 지을 때 기초 공사, 골조 공사, 내장 공사, 인테리어를 단계적으로 나눠서 하는 것처럼요.

클라이언트-서버 시스템은 크게 **4개의 계층**으로 구성됩니다. 위쪽이 사용자(클라이언트)에 가깝고, 아래쪽이 하드웨어(데이터베이스)에 가깝습니다.

```
┌─────────────────────────────────┐
│  표현 계층 (Presentation Layer) │  ← 사용자에게 가장 가까운 쪽
├─────────────────────────────────┤
│  데이터 처리 계층               │
├─────────────────────────────────┤
│  애플리케이션 처리 계층         │
├─────────────────────────────────┤
│  데이터베이스 계층 (DB Layer)   │  ← 하드웨어/데이터에 가장 가까운 쪽
└─────────────────────────────────┘
```

### 계층 ① 표현 계층 (Presentation Layer) 🖥️

**표현 계층(프레젠테이션 계층)**은 사용자와 직접 인터페이스(소통)하는 계층입니다. 사용자가 화면에서 보는 모든 것, 즉 버튼, 입력창, 텍스트, 이미지 등을 처리합니다.

쉽게 말해, 여러분이 웹사이트에 접속했을 때 눈에 보이는 화면 그 자체가 표현 계층의 결과물입니다. 사용자가 입력한 내용을 받아서 아래 계층으로 전달하고, 아래 계층에서 처리된 결과를 다시 사용자에게 보여주는 역할을 합니다.

### 계층 ② 데이터 처리 계층 📦

**데이터 처리 계층**은 클라이언트(사용자)가 요구한 것들을 실제로 처리하는 계층입니다. 구체적으로는 다음과 같은 일을 합니다:

- 사용자에게 보여줄 **웹 페이지를 생성(조립)**합니다
- 사용자가 입력한 데이터가 **올바른지 검증(Validation)**합니다
- 처리된 데이터를 **클라이언트에게 전달**합니다

예를 들어, 회원 가입 폼에서 이메일 형식이 올바른지 확인하고, 비밀번호가 조건에 맞는지 체크하는 것이 이 계층에서 이루어집니다.

### 계층 ③ 애플리케이션 처리 계층 ⚙️

**애플리케이션 처리 계층**은 실제 비즈니스 로직(업무 규칙)을 수행하는 계층입니다. 예를 들어, 쇼핑몰에서 "10만 원 이상 구매 시 무료 배송"이라는 규칙을 처리하거나, 은행에서 "잔액보다 많은 금액은 출금 불가"라는 규칙을 처리하는 것이 이 계층의 역할입니다.

실제 계산을 하거나, 요구된 기능에 대한 결과물을 만들어내기 위한 핵심 로직들이 여기서 동작합니다. 마치 요리사가 레시피대로 재료를 조리하는 것과 같습니다. 재료(데이터)를 받아서 레시피(비즈니스 로직)대로 처리하여 결과물(응답)을 만들어냅니다.

### 계층 ④ 데이터베이스 계층 (DB Layer) 🗄️

**데이터베이스 계층**은 시스템의 가장 밑바닥에서 실제 데이터를 저장하고 관리하는 계층입니다. 이 계층의 주요 역할은 다음과 같습니다:

- 실제 데이터를 **저장**하고 **조회**합니다
- 여러 사용자가 동시에 데이터에 접근할 때 **충돌 없이 관리**합니다
- **트랜잭션(작업 묶음)**을 관리합니다 (예: 계좌 이체 시 출금과 입금이 반드시 함께 성공하거나 함께 실패해야 함)
- 다양한 사용자의 **데이터베이스 질의(Query)에 대한 서비스**를 제공합니다

:::note 4계층 구조 정리
1. **표현 계층**: 사용자 화면 처리 (보여주기)
2. **데이터 처리 계층**: 사용자 요청 처리 및 웹 페이지 생성, 데이터 검증
3. **애플리케이션 처리 계층**: 비즈니스 로직 수행 (실제 업무 규칙 처리)
4. **데이터베이스 계층**: 데이터 저장, 관리, 트랜잭션 처리
:::

## 🔌 미들웨어(Middleware)란 무엇인가?

### 미들웨어가 왜 필요한가?

분산 환경에서 클라이언트와 서버는 서로 다른 운영체제, 다른 프로그래밍 언어, 다른 하드웨어 환경에서 동작할 수 있습니다. 마치 한국인과 일본인이 서로 대화를 나누려는데, 언어가 달라서 소통이 어려운 상황과 같습니다.

이럴 때 필요한 것이 바로 **통역사(미들웨어)**입니다. **미들웨어(Middleware)**는 클라이언트와 서버 사이에 위치하는 소프트웨어로, 서로 다른 환경 간의 차이를 극복하고 원활하게 연결해주는 역할을 합니다.

```
[클라이언트 A]  ─────┐
[클라이언트 B]  ──── [미들웨어] ──── [서버]
[클라이언트 C]  ─────┘
```

> 미들웨어는 마치 콘센트 어댑터와 같습니다. 나라마다 콘센트 모양이 달라도 어댑터(미들웨어)를 끼우면 어디서든 같은 전자기기를 쓸 수 있는 것처럼, 미들웨어는 서로 다른 환경의 시스템들을 연결해줍니다.

### 미들웨어의 장점

**① 표준화된 인터페이스 제공**: 미들웨어는 표준화되어 있어, 어떤 클라이언트든 동일한 방식으로 서버와 통신할 수 있습니다.

**② 데이터 일관성 보장**: 서로 다른 시스템 간에 데이터를 교환할 때 데이터의 정확성과 일관성을 보장합니다.

**③ 이기종 환경 연동**: 클라이언트 환경과 서버 환경이 달라도 서로 다른 업무를 연동하여 처리할 수 있습니다.

**④ 실시간 분산 처리**: 데이터를 앞뒤로 주고받으며 기다리는 방식이 아니라, 미들웨어를 통해 실시간으로 변환하고 바로 처리할 수 있어 효율적입니다.

### 미들웨어의 종류

미들웨어는 기술적으로 크게 **트랜스포트(Transport), 애플리케이션 서버, 메시지 브로커, 비즈니스 프로세스 관리자**로 나눌 수 있습니다.

| 미들웨어 종류 | 역할 | 대표 제품 |
|------|------|------|
| **데이터베이스 미들웨어** | 애플리케이션과 DB 서버 연결 | MS ODBC, Oracle GLUE |
| **웹 애플리케이션 서버(WAS)** | 동적 웹 컨텐츠 처리 | WebLogic, WebSphere |
| **메시지 지향 미들웨어(MOM)** | 비동기 메시지 처리 | Oracle MQ, IBM MQ, JMS |
| **트랜잭션 처리 모니터(TP Monitor)** | 트랜잭션 관리 및 실시간 응답 | Oracle Tuxedo, Tmax |
| **원격 프로시저 호출(RPC)** | 원격 서버의 함수를 로컬처럼 호출 | RPC |
| **객체 요청 브로커(ORB)** | 객체 형태의 분산 통신 | CORBA |

## 🔧 미들웨어 상세 설명

### 데이터베이스 미들웨어 (DB Middleware) 🗃️

**데이터베이스 미들웨어**는 애플리케이션과 데이터베이스 서버를 연결해주는 미들웨어입니다. 데이터베이스에 직접 연결하여 데이터를 주고받을 수 있도록 중간에서 연결 다리 역할을 합니다.

데이터베이스를 만드는 회사(벤더)마다 연결 방식이 다를 수 있기 때문에, 각 벤더가 자신의 DB에 맞는 미들웨어를 제공합니다. Microsoft는 **ODBC**, Oracle은 **GLUE** 같은 것들이 대표적입니다.

단순히 데이터베이스만 사용하는 경우(서버가 DB 관리만 하는 경우)를 **2티어 아키텍처**라고 부릅니다.

### 웹 애플리케이션 서버 (WAS) 🌐

**WAS(Web Application Server, 웹 애플리케이션 서버)**는 실무에서 가장 많이 접하게 되는 미들웨어입니다. 개발 현장에서 "WAS 재시작해야 해", "WAS에 배포했어"라는 말을 자주 듣게 됩니다.

일반적인 **웹 서버**는 이미 만들어진 HTML 파일처럼 **정적인 콘텐츠**를 처리합니다. 반면 WAS는 사용자 요청에 따라 다르게 생성되는 **동적인 콘텐츠**를 처리합니다. 예를 들어, "홍길동 님의 주문 목록"처럼 사용자마다 다른 내용을 보여주는 페이지는 WAS가 처리합니다.

요즘 대부분의 서비스들이 웹 기반으로 동작하기 때문에, WAS는 개발 환경에서 **필수 요소**입니다. 대표적인 WAS 제품으로는 **WebLogic**, **WebSphere** 등이 있습니다.

:::danger 웹 개발자라면 WAS를 반드시 알아야 합니다
웹 개발자로 실무에 투입되었는데 WAS가 무엇인지 모른다면 실제 개발 환경에서 큰 어려움을 겪게 됩니다. WAS는 현대 웹 개발 환경의 핵심입니다.
:::

### 메시지 지향 미들웨어 (MOM) 📬

**MOM(Message Oriented Middleware, 메시지 지향 미들웨어)**은 클라이언트가 생성한 요청(메시지)을 저장소에 쌓아두고, 서버가 차례대로 처리하는 방식입니다.

일반적인 방식에서는 클라이언트가 요청을 보내면, 서버가 그 요청을 처리할 때까지 **기다려야** 합니다. 마치 음식점에서 주문 하나가 나올 때까지 기다렸다가, 나온 다음에 또 주문하는 것처럼요.

MOM 방식에서는 클라이언트가 요청을 **메시지 큐(대기열)**에 모두 집어넣고, 서버는 큐에서 하나씩 꺼내서 처리합니다. 이 방식을 **비동기식(Asynchronous) 처리**라고 합니다. 클라이언트는 기다리지 않고 다른 일을 계속할 수 있습니다.

```
클라이언트 → [메시지 큐: 요청1, 요청2, 요청3, 요청4] → 서버가 하나씩 처리
                    ↑
             클라이언트는 다른 일을 계속 할 수 있음
```

대표 제품: **Oracle Message Q**, **IBM MQ**, **JMS**

### 트랜잭션 처리 모니터 (TP Monitor) ⏱️

**트랜잭션 처리 모니터(Transaction Processing Monitor)**는 트랜잭션(작업 묶음)을 관리하고, 수많은 사용자가 동시에 접속해도 빠른 응답 속도를 유지하도록 해주는 미들웨어입니다.

실생활 예시로 이해해봅시다. 추석이나 설 명절 전날, 수만 명이 동시에 기차표를 예매하려고 접속합니다. 이때 시스템이 느려지거나 다운되지 않고 빠르게 처리되도록 해주는 것이 바로 TP 모니터입니다. 사용자 수가 폭발적으로 증가하더라도 안정적인 응답 속도를 유지해야 하는 **온라인 실시간 처리 시스템**에 주로 사용됩니다.

대표 제품: **Oracle Tuxedo**, **Tmax(티맥스)**

### 원격 프로시저 호출 (RPC) 📞

**RPC(Remote Procedure Call, 원격 프로시저 호출)**는 원격 서버에 있는 함수(프로시저)를 마치 내 컴퓨터에 있는 함수를 호출하듯이 사용할 수 있게 해주는 방식입니다.

전화로 비유하면, 멀리 있는 전문가에게 전화해서 "이 문제 어떻게 해결하나요?"라고 묻고 답변을 받는 것과 같습니다. 실제로 그 전문가는 멀리 있지만, 내 옆에 있는 것처럼 즉각적으로 도움을 받을 수 있습니다.

### 객체 요청 브로커 (ORB / CORBA) 🔗

**CORBA(Common Object Request Broker Architecture, 공통 객체 요청 브로커 아키텍처)**는 분산 환경에서 객체(Object) 형태로 데이터를 서로 주고받기 위한 표준 규격입니다. **OMG(Object Management Group, 객체 관리 그룹)**에서 개발한 표준입니다.

CORBA의 핵심 구성 요소인 **ORB(Object Request Broker, 객체 요청 중계자)**는 서비스를 요청하는 객체를 찾고, 요청을 준비하고, 요청을 전달하고, 결과를 받아 돌려주는 중간 역할을 합니다.

쉽게 말하면, CORBA는 서로 다른 환경에 있는 시스템들이 **객체 형태로 데이터를 주고받을 수 있게 해주는 국제 표준**이라고 이해하시면 됩니다.

## 🏛️ 2티어, 3티어, N티어 아키텍처

### 왜 티어(Tier)를 구분하는가?

앞서 설명한 4계층 구조에서, 각 계층을 **어떻게 물리적으로 나누어 배치하느냐**에 따라 2티어, 3티어, N티어로 구분합니다. 계층은 논리적 구분이고, 티어는 물리적(실제 서버) 구분이라고 생각하면 됩니다.

### 2티어 아키텍처 (클라이언트 + 서버)

**2티어 아키텍처**는 가장 단순한 형태로, **클라이언트**와 **서버** 딱 두 개로만 구성됩니다.

- **클라이언트**: 화면 표현(표현 계층) + 비즈니스 로직(애플리케이션 계층) 모두 담당
- **서버**: 데이터베이스 처리만 담당

```
[클라이언트: 표현 + 비즈니스 로직] ←────→ [서버: DB 처리]
```

이 구조에서 클라이언트가 많은 일을 담당하기 때문에 **팻 클라이언트(Fat Client, 뚱뚱한 클라이언트)** 모델이라고도 합니다. 클라이언트가 "뚱뚱하게" 많은 역할을 갖고 있다는 의미입니다.

단점은 비즈니스 로직이 바뀌면 **모든 클라이언트(PC)를 업데이트**해야 한다는 점입니다. 100대의 PC에 설치된 프로그램을 모두 업데이트하는 것은 매우 번거롭습니다.

### 3티어 아키텍처 (클라이언트 + 웹서버 + DB서버)

**3티어 아키텍처**는 현재 가장 많이 사용되는 구조입니다. 서버를 두 개로 나눕니다.

- **클라이언트**: 화면 표현(표현 계층)만 담당
- **웹 서버 (WAS)**: 비즈니스 로직 처리 담당
- **DB 서버**: 데이터베이스 처리 담당

```
[클라이언트: 표현만] ←──→ [웹 서버: 비즈니스 로직] ←──→ [DB 서버: 데이터 처리]
```

이 구조에서 클라이언트는 화면만 보여주는 **씬 클라이언트(Thin Client, 날씬한 클라이언트)** 역할을 합니다. 클라이언트의 역할이 최소화된 것이죠.

3티어의 큰 장점은 **비즈니스 로직이 바뀌면 웹 서버만 업데이트**하면 된다는 점입니다. 클라이언트는 그냥 화면만 보여주니까요. 운영 및 유지보수가 훨씬 편리합니다.

### N티어 아키텍처 (복잡한 엔터프라이즈 시스템)

**N티어 아키텍처**는 3티어를 확장하여, 처리해야 할 응용이 매우 다양하고 복잡할 때 사용하는 구조입니다. 3티어의 웹 서버와 DB 서버 사이에 **비즈니스 컴포넌트 계층** 등을 추가하는 방식입니다.

```
[클라이언트] ←──→ [웹 계층] ←──→ [비즈니스 컴포넌트 계층] ←──→ [엔터프라이즈 정보 시스템(DB)]
```

N티어 아키텍처의 각 계층 역할:
- **웹 계층**: JSP, ASP, 서블릿 같은 동적 웹 페이지 처리 (WAS에서 수행)
- **비즈니스 컴포넌트 계층**: EJB, CORBA 같은 기술로 비즈니스 로직 처리
- **엔터프라이즈 정보 시스템 계층**: 실제 데이터베이스 처리

N티어는 대규모 트랜잭션과 복잡한 응용 처리가 필요한 **대기업 시스템(엔터프라이즈 시스템)**에서 주로 사용합니다. 은행 시스템, 항공사 예약 시스템 등이 대표적입니다.

:::tip 2티어 vs 3티어 vs N티어 선택 기준
- **소규모 단순 시스템** → 2티어
- **일반적인 웹 서비스** → 3티어 (현재 가장 보편적)
- **대규모 복잡한 엔터프라이즈 시스템** → N티어
:::

### 마스터-슬레이브 아키텍처 🔄

추가로 알아두면 좋은 구조로 **마스터-슬레이브(Master-Slave) 아키텍처**가 있습니다. 마스터(주인)와 슬레이브(종속)로 역할을 나누어 실시간 분산 환경에서 처리하는 방식입니다. **최소한의 응답 시간을 보장해야 하는 실시간 시스템**에서 주로 사용됩니다.

## 📋 주의사항 및 실무 팁

### 실무에서 꼭 알아야 할 것들

**① WAS는 반드시 익혀야 합니다**

웹 개발자라면 WAS(WebLogic, WebSphere 등)가 무엇인지, 어떻게 동작하는지 반드시 알아야 합니다. 실제 개발 환경에서 WAS 위에서 코드가 실행되기 때문에, WAS를 모르면 문제 발생 시 원인조차 찾기 어렵습니다.

**② 보안은 모든 서버에 동일하게 적용해야 합니다**

분산 환경에서 서버 하나의 보안이 뚫리면 전체 시스템이 위협받습니다. 서버 수가 많을수록 보안 관리의 복잡도도 함께 증가합니다. 각 서버에 동일한 수준의 보안 정책을 일관되게 적용해야 합니다.

**③ 각 미들웨어의 특성을 파악하고 적절히 선택해야 합니다**

모든 상황에 최적인 미들웨어는 없습니다. 실시간 처리가 중요하면 TP 모니터, 비동기 처리가 필요하면 MOM, 단순 DB 연결이면 DB 미들웨어를 선택하는 등 상황에 맞는 선택이 필요합니다.

:::warning 흔히 하는 실수
클라이언트-서버 구조를 설계할 때 처음부터 무조건 N티어로 만들려는 경우가 있습니다. 복잡성과 비용을 고려하여 실제 서비스 규모에 맞는 구조를 선택하는 것이 중요합니다. 작은 서비스에 N티어를 적용하면 오히려 관리 부담만 늘어납니다.
:::

:::danger 절대 하면 안 되는 것
분산 환경에서 일부 서버에만 보안 패치를 적용하거나, 일부 서버의 보안 설정을 다르게 유지하는 것은 매우 위험합니다. 보안 패치와 보안 설정은 모든 서버에 동시에, 동일하게 적용해야 합니다.
:::

### 각 티어별 기술 스택 예시

```
【3티어 아키텍처 기술 스택 예시】

클라이언트 계층:
  - HTML, CSS, JavaScript
  - React, Vue.js, Angular

웹 서버 / WAS 계층:
  - Apache Tomcat (가장 많이 사용되는 WAS)

```

## 🔬 정형 기법(Formal Method) 간략 소개

### 소프트웨어 위기와 정형 기법의 등장 배경

클라이언트-서버 아키텍처와 함께 최신 소프트웨어 공학에서 중요하게 다루는 개념이 바로 **정형 기법(Formal Method)**입니다. 소프트웨어 규모가 커지면서 다양한 자동화 도구와 새로운 언어들이 등장했음에도 불구하고, 개발된 소프트웨어가 명세서(설계도)와 다르게 만들어지는 실패 사례가 빈번하게 발생했습니다. "요구사항대로 만들었다고 했는데 왜 결과가 다르지?"라는 문제가 반복된 것이죠.

이 문제를 해결하기 위해 등장한 것이 **수학적·논리적 방법론을 이용하는 정형 기법**입니다.

### 정형 기법이란?

**정형 기법(Formal Method)**은 수학적 논리와 형식 언어를 이용하여 컴퓨터 시스템이나 소프트웨어를 개발하고 검증하는 방법입니다. 수학적으로 증명된 것은 논리적으로 오류가 없다는 원리를 소프트웨어 개발에 적용한 것입니다.

정형 기법은 크게 두 가지로 나뉩니다:

- **정형 명세(Formal Specification)**: 수학적 기호와 논리를 이용해 시스템의 요구사항과 설계를 기술하는 것
- **정형 검증(Formal Verification)**: 수학적으로 시스템이 요구사항을 만족하는지 증명하는 것

### 정형 명세의 종류

| 종류 | 설명 | 사용 언어/기법 |
|------|------|------|
| **논리 기반 명세** | 수리 논리를 이용해 시스템 명세 | First-order Logic, Temporal Logic |
| **대수 기반 명세** | 수학적 공리와 규칙을 이용해 명세 | π-Calculus, ACSR |
| **오토마타 기반 명세** | 상태와 전이(상태 변화)로 시스템 행위 표현 | Finite State Machine, Statechart |

### 정형 검증의 종류

- **의미론적 검증(Semantic Verification)**: 오토마타 이론 기반으로 자동화하여 시스템이 특정 속성을 만족하는지 검사합니다.
- **구문적 검증(Syntactic Verification)**: 구문 규칙을 이용해 사람이 직접 단계별로 증명합니다. 도구의 도움을 받을 수 있지만 기본적으로 수동 작업입니다.

:::info 정형 기법은 언제 주로 사용하는가?
정형 기법은 특히 **원자력 발전소 제어 시스템, 항공기 자동 비행 시스템, 자동차 안전 시스템**처럼 오류가 발생하면 매우 큰 피해를 초래하는 **고신뢰성 시스템**을 개발할 때 주로 사용됩니다. 일반 웹 서비스보다는 인명 피해와 직결되는 시스템에서 더욱 중요합니다.
:::

## 📌 핵심 정리

이번 문서에서 학습한 핵심 내용을 정리합니다.

- **클라이언트-서버 아키텍처**는 서비스를 요청하는 클라이언트와 서비스를 제공하는 서버로 역할을 분리한 시스템 모델이다
- **클라이언트는 서버의 존재를 알지만, 다른 클라이언트의 존재는 알지 못한다**
- **서버는 클라이언트에게 서비스를 요청하지 않으며**, 필요 시 다른 서버에게만 요청(클라이언트 역할)한다
- **장점**: 메인프레임 대비 저렴한 비용, 지역 분산 처리, 고성능 GUI 지원
- **단점**: 서버 부하 집중, 분산 시스템의 복잡성 증가, 보안 취약성
- 클라이언트-서버 시스템은 **표현 계층 → 데이터 처리 계층 → 애플리케이션 처리 계층 → DB 계층**의 4계층 구조를 가진다
- **미들웨어(Middleware)**는 서로 다른 환경의 클라이언트와 서버를 연결하고 차이를 극복해주는 중간 소프트웨어이다
- **WAS(웹 애플리케이션 서버)**는 동적 웹 콘텐츠를 처리하는 미들웨어로, 웹 개발에서 필수적이다
- **MOM(메시지 지향 미들웨어)**은 메시지 큐를 이용한 비동기식 처리로 클라이언트가 기다리지 않고 다른 일을 할 수 있게 한다
- **TP 모니터**는 온라인 실시간 트랜잭션(기차표 예매 등)을 빠르게 처리하기 위한 미들웨어이다
- **2티어**: 클라이언트(표현+비즈니스 로직) + 서버(DB) / **팻 클라이언트** 모델
- **3티어**: 클라이언트(표현) + WAS(비즈니스 로직) + DB서버 / **씬 클라이언트** 모델, 가장 보편적
- **N티어**: 3티어를 확장한 형태로 대규모 엔터프라이즈 시스템에 적용
- **정형 기법(Formal Method)**은 수학적 논리를 이용해 소프트웨어를 명세하고 검증하여 오류를 줄이는 최신 소프트웨어 공학 방법론이다
