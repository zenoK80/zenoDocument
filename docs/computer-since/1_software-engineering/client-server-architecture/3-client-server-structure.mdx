# 🏗️ 클라이언트-서버 아키텍처의 계층 구조

현대 소프트웨어 시스템의 뼈대를 이루는 클라이언트-서버 구조, 과연 어떻게 생겼을까요? 우리가 매일 사용하는 네이버, 구글, 인터넷 뱅킹 뒤에는 이 구조가 숨어 있습니다. 이번 문서에서는 클라이언트-서버 아키텍처의 계층 구조부터 미들웨어, 티어(Tier) 구조, 정형 기법까지 처음 배우는 분도 충분히 이해할 수 있도록 차근차근 풀어드리겠습니다.

---

## 🖥️ 클라이언트-서버 아키텍처란 무엇인가?

### 기본 개념 이해하기

**클라이언트-서버 아키텍처(Client-Server Architecture)** 란, 서비스를 요청하는 쪽(클라이언트)과 그 서비스를 제공하는 쪽(서버)으로 역할을 나눈 시스템 모델입니다. 마치 식당처럼 생각해보세요. 손님(클라이언트)이 "된장찌개 주세요"라고 요청하면, 주방(서버)에서 요리를 만들어 제공합니다. 손님은 다른 손님이 몇 명인지 알 필요가 없고, 그냥 자기가 원하는 것을 주문하면 됩니다.

> **핵심 정의**: 클라이언트-서버 아키텍처는 서비스(Service), 서버(Server), 클라이언트(Client)의 집합으로 구성되는 시스템 모델입니다.

이 구조에서 **서비스(Service)** 는 서버가 제공하는 것이고, **클라이언트(Client)** 는 그 서비스를 이용하는 주체입니다. 우리가 브라우저를 열고 네이버 주소를 입력하는 순간, 우리의 컴퓨터(클라이언트)가 네이버 서버에 "메인 페이지를 보여주세요"라고 요청을 보내는 것입니다.

### 클라이언트와 서버의 관계

클라이언트-서버 관계에서 몇 가지 중요한 특징이 있습니다. 하나씩 살펴보겠습니다.

**① 클라이언트는 서버를 인식하지만, 다른 클라이언트의 존재는 모른다**

우리가 구글에 접속할 때, 우리는 마치 구글 서버를 나 혼자 쓰는 것처럼 느낍니다. 하지만 실제로는 전 세계 수억 명이 동시에 접속하고 있죠. 그러나 클라이언트(내 브라우저) 입장에서는 다른 사용자들이 얼마나 몰려 있는지 알 필요도 없고, 알지도 못합니다. 이것이 클라이언트-서버 구조의 특징입니다.

**② 서버는 클라이언트에게 서비스를 요청하지 않는다**

서버는 항상 서비스를 "제공하는" 쪽입니다. 서버가 클라이언트에게 "나한테 데이터 좀 줘"라고 요청하는 일은 없습니다. 요청은 언제나 클라이언트→서버 방향으로 흐릅니다.

**③ 서버도 다른 서버에게는 클라이언트가 될 수 있다**

이 부분이 조금 헷갈릴 수 있습니다. 예를 들어, 쇼핑몰 서버가 결제 처리를 위해 PG(Payment Gateway, 결제 대행) 서버에 요청을 보낼 때, 쇼핑몰 서버는 PG 서버 입장에서 **클라이언트** 역할을 합니다. 서버라는 이름표가 붙어 있어도, 다른 서버에게 서비스를 요청할 때는 클라이언트처럼 동작하는 것입니다.

:::info 서버도 클라이언트가 될 수 있다
서버는 고정된 역할이 아닙니다. A 서버가 B 서버에게 데이터를 요청할 때, A는 클라이언트 역할을 합니다. 역할은 **관계에 따라** 달라집니다.
:::

## ⚖️ 클라이언트-서버 아키텍처의 장점과 단점

### 장점: 왜 이 구조를 선택하는가?

클라이언트-서버 구조가 왜 현대 IT의 기본이 됐는지, 그 이유를 살펴보겠습니다.

**💰 비용 절감**

과거에는 모든 연산을 처리하는 거대한 메인프레임(Mainframe, 대형 중앙 컴퓨터) 한 대를 구축해야 했습니다. 이는 엄청난 비용이 들었습니다. 클라이언트-서버 구조에서는 상대적으로 저렴한 서버들을 분산 배치해서 동일한 효과를 낼 수 있습니다. 마치 대형 공장 하나를 짓는 대신, 여러 군데에 소규모 공장을 분산 배치하는 것과 같습니다.

**📍 지역 분산 처리**

데이터가 발생한 그 지역 근처에 서버를 배치할 수 있기 때문에, 처리 속도가 훨씬 빠릅니다. 예를 들어, 서울에서 발생한 데이터를 굳이 미국 서버까지 보냈다가 받아오지 않아도 됩니다. 서울 근처의 서버가 바로 처리해주면 되는 것이죠.

**🖼️ 고성능 UI(사용자 인터페이스) 지원**

고성능 워크스테이션(Workstation, 전문 작업용 컴퓨터)이나 PC에서 GUI(Graphical User Interface, 그래픽 사용자 인터페이스)를 풍부하게 지원할 수 있습니다. 버튼을 클릭하고, 화면이 예쁘게 보이고, 다양한 시각적 요소들이 동작하는 것 모두 클라이언트-서버 구조 덕분입니다.

> 한마디로 정리하면: **저렴한 비용**으로 **고성능 분산 처리**를 실현할 수 있다는 것이 클라이언트-서버 아키텍처의 핵심 장점입니다.

### 단점: 만능은 없다

장점이 있으면 반드시 단점도 있습니다. 클라이언트-서버 구조의 단점을 솔직하게 짚어보겠습니다.

**📈 서버 부하 집중**

처음에는 클라이언트와 서버가 역할을 나눠서 처리했습니다. 그런데 시간이 지날수록 서버에게 요구하는 응용 처리(비즈니스 로직, 데이터 처리 등)가 점점 많아집니다. 결국 서버에 엄청난 부하(Load, 작업량)가 걸리게 되고, 예전의 비싼 메인프레임과 다를 바 없는 상황이 됩니다.

**🔀 분산 환경의 복잡성**

서버가 한 군데가 아니라 전국, 전 세계에 흩어져 있습니다. 이 서버들이 네트워크로 연결돼서 동시에(병렬로) 데이터를 처리하고 합치는 작업을 합니다. 이 구조 자체가 매우 복잡하기 때문에, 관리와 운영이 어렵습니다.

**🔓 보안 취약성**

이것이 가장 심각한 단점입니다. 서버를 100개 분산해놓으면, 그 100개 서버 중 단 하나에만 보안 구멍이 뚫려도 전체 시스템이 위험해집니다. 마치 100개의 자물쇠가 달린 금고인데, 자물쇠 하나가 고장나면 그 문으로 들어올 수 있는 것과 같습니다. 최근에 빈번하게 발생하는 해킹 사고의 상당수가 바로 이런 분산 환경의 취약점을 이용한 것입니다.

| 구분 | 내용 |
|---|---|
| ✅ 장점 | 저렴한 비용으로 메인프레임급 성능 실현 |
| ✅ 장점 | 데이터 발생 지역 근처에서 처리 가능 |
| ✅ 장점 | 고성능 GUI 지원 |
| ❌ 단점 | 서버 부하 집중 (메인프레임과 유사한 문제 재발) |
| ❌ 단점 | 분산 환경 특성상 시스템 복잡도 증가 |
| ❌ 단점 | 보안 유지 어려움 (분산 환경의 취약점) |

## 🧱 클라이언트-서버의 4계층 구조

### 왜 계층을 나누는가?

클라이언트-서버 시스템은 단순히 "요청하는 쪽"과 "제공하는 쪽"으로만 나뉘지 않습니다. 내부적으로는 **4개의 계층(Layer)** 으로 구성되어 있습니다. 왜 굳이 계층을 나눌까요?

건물을 지을 때 기초공사, 골조공사, 인테리어, 마감 공사를 각각 다른 전문가가 담당하듯이, 소프트웨어 시스템도 각 역할에 맞는 계층을 나눠서 관리하면 유지보수와 확장이 훨씬 쉬워집니다. 어느 층에서 문제가 생겼는지 바로 파악할 수 있고, 한 층만 수정해도 전체 시스템에 영향을 최소화할 수 있습니다.

### 4계층 구조 상세 설명

클라이언트-서버 시스템의 4계층을 사용자와 가까운 쪽(위)부터 하드웨어와 가까운 쪽(아래)으로 살펴보겠습니다.

```
┌─────────────────────────────────────┐
│  표현 계층 (Presentation Layer)      │  ← 사용자와 직접 상호작용
├─────────────────────────────────────┤
│  데이터 처리 계층 (Data Processing)  │  ← 사용자 요구사항 처리
├─────────────────────────────────────┤
│  애플리케이션 계층 (Application)     │  ← 비즈니스 로직 처리
├─────────────────────────────────────┤
│  데이터베이스 계층 (Database Layer)  │  ← 데이터 저장 및 관리
└─────────────────────────────────────┘
```

**① 표현 계층 (Presentation Layer) — 사용자의 눈**

표현 계층은 사용자와 직접 인터페이스(Interface, 접점)하는 계층입니다. 쉽게 말해, 사용자가 눈으로 보고 손으로 조작하는 화면이 여기에 해당합니다. 우리가 웹사이트에서 보는 버튼, 텍스트 박스, 이미지, 메뉴 같은 것들이 모두 표현 계층에서 만들어집니다. 사용자가 뭔가를 입력하면 받아서 아래 계층으로 전달하고, 처리 결과를 다시 화면에 보기 좋게 보여주는 역할을 합니다.

**② 데이터 처리 계층 (Data Processing Layer) — 요청의 관문**

사용자(클라이언트)가 요청한 내용을 실제로 처리하기 위한 준비를 하는 계층입니다. 예를 들어, 웹페이지 형태로 데이터를 만들어내고, 입력된 데이터에 오류가 없는지 검증(Validation, 유효성 검사)하고, 클라이언트에게 보여줄 데이터를 준비합니다. 마치 식당의 홀 직원처럼, 손님의 주문을 받아 주방에 전달하고 결과를 다시 손님에게 가져다주는 역할입니다.

**③ 애플리케이션 처리 계층 (Application Processing Layer) — 두뇌**

실제 비즈니스 로직(Business Logic, 업무 규칙 및 처리 논리)이 수행되는 계층입니다. 예를 들어 "회원 등급에 따른 할인율 계산", "재고 수량 확인 후 주문 처리", "날짜 계산" 같은 실질적인 연산과 처리가 여기서 이루어집니다. 식당으로 치면 주방장이 요리를 만드는 공간입니다.

**④ 데이터베이스 계층 (Database Layer) — 창고**

가장 아래에 위치한 계층으로, 실제 데이터를 저장하고 관리하는 역할을 합니다. 애플리케이션 계층에서 처리한 결과를 저장하거나, 필요한 데이터를 꺼내주는 역할을 합니다. 다양한 사용자의 데이터베이스 질의(Query, 데이터 조회/수정 요청)에 응답하고, 트랜잭션(Transaction, 데이터베이스 작업 단위)을 관리합니다. 마치 식재료 창고처럼, 필요한 재료를 꺼내주고 새로 들어온 재료를 정리해서 보관합니다.

:::tip 4계층 구조 기억법
**표→데→애→데** (표현 → 데이터처리 → 애플리케이션 → 데이터베이스)
위에서 아래로 내려갈수록 사용자에서 멀어지고 데이터(저장소)에 가까워진다고 기억하세요!
:::

## 🔗 미들웨어(Middleware): 서로 다른 환경을 연결하는 다리

### 미들웨어가 필요한 이유

분산 환경에서는 클라이언트의 환경과 서버의 환경이 서로 다를 수 있습니다. 예를 들어, 클라이언트는 Windows에서 동작하고, 서버는 Linux에서 동작할 수 있습니다. 또는 클라이언트는 Java로 만들어졌고, 서버는 .NET으로 만들어졌을 수도 있습니다. 이렇게 서로 다른 환경에서 데이터를 주고받으려면 중간에 "통역사"가 필요합니다.

**미들웨어(Middleware)** 가 바로 그 역할입니다. 이름 그대로 "중간(Middle)에 있는 소프트웨어(ware)"입니다. 클라이언트와 서버 사이에 위치해서, 서로 다른 환경의 차이를 극복하고 원활하게 통신할 수 있도록 도와줍니다.

> 미들웨어는 마치 **국제 무역에서의 통관 서비스**와 같습니다. 한국에서 보낸 택배가 미국으로 가려면 세관, 통관, 언어 표기 변환 등의 과정이 필요한데, 미들웨어가 바로 이 과정을 소프트웨어 세계에서 담당합니다.

### 미들웨어의 주요 장점

**표준화된 인터페이스 제공**: 미들웨어는 표준화되어 있기 때문에, 어떤 환경에서도 동일한 방식으로 사용할 수 있습니다. 매번 새로운 연결 방법을 만들 필요가 없습니다.

**데이터 일관성 보장**: 서로 다른 시스템 간에 데이터를 교환할 때, 데이터가 정확하게 전달되고 일관성(Consistency, 데이터가 모순 없이 정확한 상태)이 유지되도록 보장합니다.

**이기종 환경 연동**: 서로 다른 운영체제, 다른 프로그래밍 언어, 다른 데이터베이스를 사용하는 시스템끼리도 미들웨어를 통해 원활하게 업무를 연동할 수 있습니다.

**실시간 분산 처리**: 데이터를 앞뒤로 왔다갔다 보내지 않고, 분산 환경 중간중간에서 미들웨어가 데이터를 변환하고 넘겨줌으로써 빠른 실시간 처리가 가능합니다.

### 미들웨어의 기술적 분류

미들웨어를 기술적으로 분류하면 크게 4가지로 나눌 수 있습니다.

| 분류 | 역할 | 예시 |
|---|---|---|
| 트랜스포트(Transport) | 데이터 전송, 기본 파이프 역할 | 메시지 기반 미들웨어, SOAP |
| 애플리케이션 서버 | 응용 처리 (트랜잭션, 보안, 멀티스레드) | J2EE, .NET |
| 메시지 브로커(Message Broker) | 메시지 교환 및 라우팅 | WebSphere Message Broker |
| 비즈니스 프로세스 관리자(BPM) | 워크플로우 방식 응용 처리 | BizTalk, TIBCO |

**트랜스포트(Transport) 계층**: 소프트웨어 컴포넌트(Component, 독립된 기능 단위) 사이에서 데이터를 주고받기 위한 기본 파이프(Pipe, 통로) 역할을 합니다. 분산 환경에서 데이터를 서로 전달하기 위한 기반 환경을 제공합니다.

**애플리케이션 서버**: 실제 응용 처리를 담당합니다. 트랜잭션(Transaction, 작업 단위) 관리, 보안 권한 체크, 디렉터리 형태의 데이터 접근 관리, 멀티스레드(Multi-Thread, 여러 작업을 동시에 처리하는 방식) 기반의 응용 프로그램 처리 모델을 제공합니다.

**메시지 브로커(Message Broker)**: 트랜스포트 계층과 애플리케이션 서버 계층 사이에서 메시지를 처리하고, 애플리케이션 간 메시지 교환 및 **라우팅(Routing, 경로 설정)**을 담당합니다. A 지점에서 B 지점으로 메시지를 보낼 때 어떤 경로로 보낼지 결정하는 것이 라우팅입니다.

**비즈니스 프로세스 관리자(BPM)**: 메시지 브로커의 기능에 더해, 워크플로우(Workflow, 업무 흐름) 방식의 애플리케이션 지원 기능이 추가된 것입니다. 실제 비즈니스 프로세스 전체를 관리하고 실행합니다.

## 🗂️ 미들웨어의 종류별 상세 설명

### 1️⃣ 데이터베이스 미들웨어 (DB Middleware)

데이터베이스 미들웨어는 **애플리케이션과 데이터베이스 서버를 직접 연결해주는 미들웨어**입니다. 애플리케이션이 데이터베이스에 "이 데이터 주세요"라고 요청할 때 그 연결 다리 역할을 합니다.

데이터베이스 벤더(Vendor, 소프트웨어 공급업체)마다 각자의 DB 미들웨어를 제공합니다. 예를 들어 Microsoft에서 제공하는 **ODBC(Open Database Connectivity)**, Borland의 **IDAPI**, Oracle의 **Glue** 같은 것들이 있습니다.

단순히 클라이언트가 데이터베이스에 직접 연결해서 데이터를 읽고 쓰는 구조, 즉 **2-Tier(투 티어) 아키텍처**에서 주로 사용됩니다.

:::note DB 미들웨어와 티어 구조
DB 미들웨어만 사용하는 단순한 구조는 클라이언트 ↔ 데이터베이스 서버의 2-Tier 구조입니다. 여기서는 서버가 데이터 관리만 담당합니다.
:::

### 2️⃣ 웹 애플리케이션 서버 (WAS, Web Application Server)

현장에서 **"WAS(와스)"** 라고 흔히 부르는 미들웨어입니다. 웹 개발자라면 반드시 알아야 하는 핵심 미들웨어입니다.

**정적(Static) 콘텐츠**는 어떤 사용자에게나 항상 동일한 내용을 보여주는 것(예: 회사 소개 페이지의 고정된 텍스트)입니다. 이런 것은 일반 웹 서버가 처리합니다. 반면 **동적(Dynamic) 콘텐츠**는 사용자마다, 시간마다, 상황마다 다른 내용을 보여줘야 하는 것(예: 내 장바구니 목록, 실시간 검색어 순위)입니다. 이런 동적 콘텐츠를 처리하기 위해 WAS가 필요합니다.

요즘 거의 모든 웹 서비스는 동적 콘텐츠로 구성되어 있습니다. 따라서 WAS는 현대 웹 개발 환경에서 필수적인 요소입니다. 대표적인 WAS 제품으로는 **WebLogic**, **WebSphere**, **JBoss** 등이 있습니다.

WAS는 단순히 HTTP(Hypertext Transfer Protocol, 웹 통신 규약) 세션 처리만 하는 것이 아니라, 기업의 핵심 업무(크리티컬한 업무)를 처리하고, Java EJB(Enterprise JavaBeans, 기업용 자바 컴포넌트)나 .NET 컴포넌트 기반의 구현도 모두 가능합니다.

:::warning WAS를 모르면 웹 개발이 어렵다
실무 웹 개발 환경에서는 WAS가 기본으로 구축되어 있습니다. WebLogic이나 WebSphere 같은 WAS를 모르면 개발 환경에 투입됐을 때 매우 어려움을 겪을 수 있습니다. WAS의 개념과 기본 동작 원리를 반드시 숙지하세요.
:::

### 3️⃣ 메시지 지향 미들웨어 (MOM, Message-Oriented Middleware)

**MOM(맘)**은 클라이언트가 생성한 메시지(요청)를 **저장소에 쌓아두고**, 서버가 하나씩 꺼내서 처리하는 방식의 미들웨어입니다.

일반적인 방식(동기식)으로는, 클라이언트가 요청을 보내면 서버가 처리하고 응답을 줄 때까지 기다려야 합니다. 하지만 MOM 방식(비동기식)은 다릅니다. 클라이언트는 처리해야 할 요청들을 중간 저장소에 전부 넣어두고, 다른 작업을 계속합니다. 서버는 저장소에서 하나씩 꺼내서 처리하면 됩니다.

> 비유하자면 **우체통**과 같습니다. 편지를 보내는 사람(클라이언트)은 우체통(저장소)에 편지를 넣고 바로 다른 일을 할 수 있습니다. 우체부(서버)는 나중에 편지를 수거해서 배달합니다. 편지를 넣는 사람이 우체부가 올 때까지 기다릴 필요가 없죠.

이런 방식을 **비동기식(Asynchronous)** 처리라고 합니다. 온라인 실시간 업무보다는, 분산 환경에서 대량의 데이터를 순서대로 처리해야 할 때 주로 사용합니다.

대표적인 MOM 제품으로는 **Oracle Message Q**, **IBM MQ**, **JMS(Java Message Service)** 등이 있습니다.

### 4️⃣ 트랜잭션 처리 모니터 (TP Monitor, Transaction Processing Monitor)

**트랜잭션(Transaction)**이란 데이터베이스의 최소 처리 단위입니다. 예를 들어 "계좌 이체" 작업은 하나의 트랜잭션입니다. 내 계좌에서 돈이 나가고, 상대방 계좌로 돈이 들어오는 두 가지 작업이 하나의 묶음으로 처리되어야 합니다. 중간에 하나만 되고 실패하면 안 되죠.

**TP 모니터**는 이런 트랜잭션을 관리하고 감시(Monitor)하는 미들웨어입니다. 특히 **실시간 온라인 트랜잭션(OLTP, Online Transaction Processing)**을 처리할 때 사용합니다.

대표적인 예가 바로 **기차표 예매**입니다. 명절 연휴에 수십만 명이 동시에 열차 예매를 시도합니다. 이때 서버가 멈추지 않고, 빠른 응답 속도를 유지하려면 TP 모니터가 필요합니다. 사용자 수가 폭발적으로 증가해도 안정적으로 서비스를 제공할 수 있게 해줍니다.

대표적인 TP 모니터 제품으로는 **Oracle Tuxedo(턱시도)**, **TmaxSoft의 Tmax(티맥스)** 등이 있습니다.

### 5️⃣ 원격 프로시저 호출 (RPC, Remote Procedure Call)

**RPC(Remote Procedure Call)**는 원격에 있는 프로시저(Procedure, 함수나 서브루틴)를 마치 로컬(Local, 내 컴퓨터)에 있는 것처럼 호출해서 처리하는 방식입니다.

예를 들어, 내 프로그램에서 함수를 호출하듯이 `calculateTax(amount)` 라고 작성하면, 실제로는 원격 서버에 있는 세금 계산 함수가 실행되고 결과가 돌아옵니다. 개발자 입장에서는 원격 서버를 신경 쓸 필요 없이, 마치 로컬 함수를 쓰듯이 코드를 작성할 수 있습니다.

### 6️⃣ 객체 요청 브로커 (ORB, Object Request Broker) — CORBA

**CORBA(Common Object Request Broker Architecture, 코바)**는 객체 지향 방식으로 분산 환경을 구현하기 위한 표준 미들웨어입니다. **OMG(Object Management Group, 객체 관리 그룹)**이라는 표준화 기관에서 만든 표준 스펙을 기반으로 합니다.

CORBA의 핵심은 **ORB(Object Request Broker, 객체 요청 중계자)**입니다. ORB는 서로 다른 환경에 있는 객체들 간의 통신을 중계하고, 요청을 받아 전달하는 역할을 합니다.

CORBA의 주요 구성 요소를 정리하면 다음과 같습니다.

| 구성 요소 | 설명 |
|---|---|
| 객체 모델 | 객체를 캡슐화(Encapsulation)하고 상태를 나타냄 |
| ORB (객체 요청 중계자) | 서비스 요청 객체를 찾고, 요청을 준비하고, 전달하는 중간 역할 |
| 객체 서비스 집합 | 자주 요청되는 서비스를 디렉터리 형태로 관리 |
| 공통 컴포넌트 | 특정 도메인 컴포넌트(수직)와 범용 컴포넌트(수평)로 구성 |

CORBA는 서로 다른 환경에서 동작하는 시스템 간의 **상호 운용성(Interoperability)**을 제공하는 것이 핵심입니다. 플랫폼, 언어, 네트워크가 달라도 객체 형태로 데이터를 주고받을 수 있도록 해줍니다.

## 🏢 2-Tier, 3-Tier, N-Tier 아키텍처

### 티어(Tier)란 무엇인가?

**티어(Tier)**는 시스템을 구성하는 **논리적인 계층의 수**를 의미합니다. 클라이언트와 서버 사이에 몇 개의 계층이 있느냐에 따라 2-Tier, 3-Tier, N-Tier로 구분합니다.

### 2-Tier 아키텍처 (팻 클라이언트 모델)

```
[클라이언트]  ←→  [데이터베이스 서버]
 표현 계층         데이터 처리
 비즈니스 로직
```

**2-Tier(투 티어)**는 클라이언트와 서버, 딱 두 개의 계층으로 구성된 단순한 구조입니다. 클라이언트가 화면 표시(표현 계층)와 비즈니스 로직 처리를 모두 담당하고, 서버는 데이터베이스 관리만 합니다.

이 구조를 **팻 클라이언트 모델(Fat Client Model)**이라고도 합니다. "팻(Fat, 뚱뚱한)"이라는 이름처럼 클라이언트가 많은 일을 담당하기 때문입니다. 구조가 단순하다는 장점이 있지만, 클라이언트 쪽에서 처리해야 할 것이 많고, 비즈니스 로직이 변경되면 모든 클라이언트를 다 업데이트해야 하는 단점이 있습니다.

### 3-Tier 아키텍처 (씬 클라이언트 모델)

```
[클라이언트]  ←→  [웹/앱 서버]  ←→  [데이터베이스 서버]
 표현 계층         비즈니스 로직       데이터 저장/관리
```

**3-Tier(쓰리 티어)**는 클라이언트, 웹/앱 서버, 데이터베이스 서버의 세 계층으로 나눈 구조입니다. 현재 대부분의 웹 서비스가 이 구조를 기반으로 합니다.

클라이언트는 화면 표시(표현 계층)만 담당하고, 비즈니스 로직은 웹/앱 서버(미들웨어)에서, 데이터 저장은 DB 서버에서 처리합니다. 클라이언트가 최소한의 일만 하기 때문에 **씬 클라이언트 모델(Thin Client Model)**이라고도 합니다.

> 3-Tier의 가장 큰 장점은 **비즈니스 로직의 중앙 관리**입니다. 로직이 변경되면 서버만 수정하면 됩니다. 클라이언트(사용자 단말)를 하나하나 업데이트할 필요가 없습니다.

### N-Tier 아키텍처

```
[클라이언트] ←→ [웹 서버] ←→ [비즈니스 컴포넌트] ←→ [EIS(엔터프라이즈 정보 시스템)]
  표현 계층      웹 처리      비즈니스 로직 처리       데이터베이스 처리
```

**N-Tier(엔 티어)**는 3-Tier를 확장한 구조로, 처리해야 할 업무가 복잡하고 많을 때 사용합니다. 여기서 N은 "여러 개"를 의미합니다. 대용량 트랜잭션 처리, 다양한 형태의 응용 처리가 필요한 대규모 엔터프라이즈(Enterprise, 기업 규모) 시스템에서 사용됩니다.

N-Tier 구조의 계층별 역할:

- **클라이언트 층**: HTML 웹페이지 표현 처리
- **웹 층(WAS)**: JSP, ASP, 서블릿(Servlet) 등 동적 웹페이지 처리
- **비즈니스 컴포넌트 층**: EJB, .NET, CORBA 등으로 비즈니스 로직 처리
- **엔터프라이즈 정보 시스템(EIS) 층**: 데이터베이스 접근 및 데이터 처리

| 구조 | 구성 | 특징 | 사용 상황 |
|---|---|---|---|
| 2-Tier | 클라이언트 + DB 서버 | 단순, 클라이언트 부담 큼 | 소규모 시스템 |
| 3-Tier | 클라이언트 + 앱 서버 + DB 서버 | 균형, 유지보수 용이 | 일반 웹 서비스 |
| N-Tier | 클라이언트 + 여러 서버 계층 | 유연, 복잡, 확장성 우수 | 대규모 엔터프라이즈 |

## ⚙️ 마스터-슬레이브 아키텍처와 주요 미들웨어 기술 정리

### 마스터-슬레이브 아키텍처 (Master-Slave Architecture)

**마스터-슬레이브(Master-Slave)** 구조는 마스터(Master, 주인 역할)와 슬레이브(Slave, 일꾼 역할)로 역할을 나눈 분산 구조입니다. 주로 **최소한의 응답 시간을 보장해야 하는 실시간 시스템**에서 사용됩니다. 예를 들어 자동차 제어 시스템, 항공기 제어 시스템처럼 밀리초(1/1000초) 단위의 응답이 필요한 환경에서 활용됩니다.

### MOM(메시지 지향 미들웨어)의 동작 방식 심화

MOM이 실제로 어떻게 동작하는지 좀 더 구체적으로 살펴보겠습니다.

```
[보내는 쪽 (Producer)]
        ↓
[메시지 처리 스레드풀 (Thread Pool) — 저장소]
  메시지1, 메시지2, 메시지3, 메시지4 ...
        ↓
[받는 쪽 (Consumer)]
```

보내는 쪽(Producer, 생산자)은 처리할 메시지들을 **스레드풀(Thread Pool, 미리 만들어진 작업 대기열)**에 모아둡니다. 받는 쪽(Consumer, 소비자)은 스레드풀에서 메시지를 하나씩 꺼내 처리합니다. 보내는 쪽은 메시지를 넣고 나서 바로 다른 작업을 할 수 있으므로, 서버의 응답을 기다리지 않아도 됩니다. 이것이 **비동기(Asynchronous) 방식**의 핵심입니다.

대규모 시스템 통합(예: 은행 전산 시스템 구축, 전사적 시스템 통합)에서 서로 다른 독립 애플리케이션들을 하나로 묶는 **접착제 역할**을 하는 것이 MOM의 진정한 가치입니다.

:::tip MOM은 언제 쓰나?
MOM은 실시간 즉각 응답보다는 대량 데이터를 **순서대로 안정적으로 처리**해야 할 때 적합합니다. 은행 일괄 처리, 로그 수집 시스템, 대용량 주문 처리 시스템 등에 활용됩니다.
:::

## ⚠️ 주의사항 및 실전 팁

### 계층 구조 설계 시 주의사항

:::warning 계층 간 역할 혼재 금지
표현 계층에 비즈니스 로직을 넣거나, 데이터베이스 계층에서 화면 처리를 하는 것은 유지보수를 극도로 어렵게 만듭니다. 각 계층은 자신의 역할에만 집중해야 합니다.
:::

다음은 계층 구조를 잘못 설계한 예시와 올바른 예시입니다.

```python
# ❌ 잘못된 예시: 표현 계층(화면 처리 함수)에 비즈니스 로직이 섞인 경우
def show_order_page(user_id):
    # 화면을 보여주는 함수인데 할인율 계산까지 하고 있음 (역할 혼재)
    discount = 0
    if user_id in vip_list:       # 비즈니스 로직: VIP 여부 판단
        discount = 0.2            # 비즈니스 로직: 할인율 계산
    price = base_price * (1 - discount)
    render_html(price)            # 화면 표시

# ✅ 올바른 예시: 계층별 역할 분리
# 비즈니스 로직은 애플리케이션 계층에서 처리
def calculate_discount(user_id):
    # 애플리케이션 계층: 비즈니스 로직만 담당
    if user_id in vip_list:       # VIP 여부 판단
        return 0.2                # 할인율 반환
    return 0.0                    # 일반 고객 할인율

# 표현 계층은 화면 표시만 담당
def show_order_page(user_id):
    discount = calculate_discount(user_id)  # 비즈니스 계층 호출
    price = base_price * (1 - discount)
    render_html(price)            # 화면 표시만 수행
```

### 미들웨어 선택 시 고려사항

:::danger 미들웨어를 잘못 선택하면 시스템 전체가 위험
미들웨어는 시스템의 핵심 인프라입니다. 잘못 선택하면 성능 병목, 보안 취약점, 유지보수 불가능 상황이 발생할 수 있습니다. 반드시 시스템 요구사항을 충분히 분석한 후 선택하세요.
:::

```
미들웨어 선택 기준:

1. 실시간 트랜잭션 처리 필요?
   → TP 모니터 (Tuxedo, Tmax)

2. 동적 웹 콘텐츠 처리 필요?
   → WAS (WebLogic, WebSphere, JBoss)

3. 비동기 대량 메시지 처리 필요?
   → MOM (IBM MQ, Oracle Message Q, JMS)

4. 단순 DB 연결만 필요?
   → DB 미들웨어 (ODBC, JDBC)

5. 이기종 분산 객체 통신 필요?
   → CORBA (ORB 기반 미들웨어)
```

### 분산 환경 보안 주의사항

:::warning 분산 서버는 보안이 가장 취약한 지점을 기준으로 평가된다
서버 100개를 운영할 때, 99개가 완벽해도 1개에 보안 구멍이 있으면 전체 시스템이 위험합니다. 분산 환경에서는 **모든 노드(Node, 서버 개체)의 보안 수준이 균일**해야 합니다.
:::

## 📝 핵심 정리

이 문서에서 배운 핵심 개념들을 정리합니다.

- **클라이언트-서버 아키텍처**는 서비스를 요청하는 클라이언트와 제공하는 서버로 역할을 나눈 시스템 모델이다.
- 클라이언트는 서버를 인식하지만 다른 클라이언트의 존재는 알지 못하며, 서버는 클라이언트에게 서비스를 요청하지 않는다.
- 서버도 다른 서버에게 서비스를 요청할 때는 클라이언트 역할을 수행한다.
- 장점: 저렴한 비용, 분산 처리, 고성능 GUI 지원 / 단점: 서버 부하, 복잡성, 보안 취약성.
- 클라이언트-서버 시스템은 **표현 계층 → 데이터 처리 계층 → 애플리케이션 계층 → 데이터베이스 계층**의 4계층 구조로 이루어진다.
- **미들웨어(Middleware)**는 클라이언트와 서버 사이에서 서로 다른 환경의 차이를 극복하고 연결해주는 소프트웨어다.
- 미들웨어는 기술적으로 트랜스포트, 애플리케이션 서버, 메시지 브로커, BPM으로 분류된다.
- **DB 미들웨어**는 애플리케이션과 데이터베이스를 직접 연결하며 2-Tier 환경에서 주로 사용된다.
- **WAS(웹 애플리케이션 서버)**는 동적 웹 콘텐츠를 처리하는 현대 웹 개발의 필수 미들웨어다.
- **MOM(메시지 지향 미들웨어)**은 비동기 방식으로 메시지를 저장소에 쌓아두고 처리하는 미들웨어다.
- **TP 모니터**는 실시간 온라인 트랜잭션을 관리하고 대용량 동시 접속에도 빠른 응답을 보장한다.
- **RPC**는 원격의 함수를 로컬처럼 호출할 수 있게 해주는 방식이다.
- **CORBA**는 이기종 분산 환경에서 객체 형태로 통신하기 위한 OMG 표준 미들웨어다.
- **2-Tier**는 클라이언트가 비즈니스 로직을 처리하는 팻 클라이언트 구조다.
- **3-Tier**는 클라이언트·앱 서버·DB 서버로 나눈 씬 클라이언트 구조로 현재 가장 많이 사용된다.
- **N-Tier**는 3-Tier의 확장으로, 복잡한 대규모 엔터프라이즈 시스템에 적용된다.
- **마스터-슬레이브 아키텍처**는 실시간 응답이 필수인 시스템에서 사용하는 분산 구조다.
