# 🏗️ 클라이언트-서버 아키텍처 개념 및 정의

소프트웨어 공학의 최신 동향을 이야기할 때 가장 먼저 등장하는 주제가 바로 **클라이언트-서버 아키텍처(Client-Server Architecture)** 입니다. 이 개념은 현대 소프트웨어 개발의 근간을 이루는 구조로, 여러분이 매일 사용하는 웹사이트, 앱, 온라인 서비스의 뼈대가 바로 이 구조 위에 세워져 있습니다. 처음 접하면 다소 어렵게 느껴질 수 있지만, 실생활 예시와 비유를 통해 차근차근 이해해 나가다 보면 "아, 이게 이런 거였구나!" 하는 순간이 반드시 옵니다. 이 문서에서는 클라이언트-서버 아키텍처의 기본 개념부터 시작해서 미들웨어(Middleware), 다양한 계층 구조, 그리고 정형 기법(Formal Method)까지 넓은 범위를 초보자도 이해할 수 있도록 아주 자세하게 안내해 드리겠습니다.

---

## 🧩 클라이언트-서버 아키텍처란 무엇인가?

### 기본 개념 소개

**클라이언트-서버 아키텍처(Client-Server Architecture)** 란, 말 그대로 **서비스를 요청하는 쪽(클라이언트)** 과 **서비스를 제공하는 쪽(서버)** 으로 역할을 나누어 시스템을 구성하는 소프트웨어 설계 모델입니다.

> 💡 **한 줄 정의**: 클라이언트-서버 아키텍처란 "서비스(Service), 서버(Server), 클라이언트(Client)의 집합으로 구성되는 시스템 모델"입니다.

이것을 실생활에 비유하면 **음식점**과 매우 비슷합니다. 음식점에서 손님(클라이언트)은 메뉴를 보고 음식을 주문(요청)하고, 주방(서버)은 그 주문을 받아 음식을 만들어 제공(응답)합니다. 손님은 주방이 어떤 방식으로 음식을 만드는지 전혀 알 필요가 없고, 그냥 맛있는 음식이 나오면 됩니다. 마찬가지로 우리가 네이버나 구글에 접속할 때, 서버가 어떤 방식으로 데이터를 처리해 우리에게 결과를 보여주는지 몰라도 됩니다. 그냥 검색 결과가 화면에 나오면 그것으로 충분합니다.

### 클라이언트와 서버의 역할 구분

| 구분 | 역할 | 예시 |
|---|---|---|
| **클라이언트(Client)** | 서비스를 요청하는 주체 | 웹 브라우저, 모바일 앱, PC 프로그램 |
| **서버(Server)** | 서비스를 제공하는 주체 | 웹 서버, 데이터베이스 서버, 파일 서버 |
| **서비스(Service)** | 서버가 클라이언트에게 제공하는 기능 | 검색, 로그인, 파일 다운로드 등 |

**클라이언트**는 자신이 이용할 수 있는 서버가 무엇인지는 알아야 합니다. 예를 들어 네이버에 접속하려면 네이버 서버의 주소를 알아야 하죠. 하지만 현재 같은 서버에 동시에 접속 중인 **다른 클라이언트(사용자)들의 존재는 알 필요도 없고, 실제로 알지도 못합니다**. 여러분이 네이버에 접속할 때, 지금 전국에서 몇 백만 명이 동시에 접속하고 있어도 여러분 입장에서는 마치 혼자 서버를 사용하는 것처럼 느껴지는 것이 바로 이 때문입니다.

**서버**에 대해서도 중요한 특성이 있습니다. 서버는 기본적으로 클라이언트에게 서비스를 **제공**하는 역할을 합니다. 따라서 서버가 클라이언트에게 "서비스를 달라"고 요청하는 일은 없습니다. 그러나 **서버가 다른 서버에게 무언가를 요청할 때는 클라이언트 역할을 할 수 있습니다**. 예를 들어, 웹 서버가 데이터베이스 서버에 "이 사용자의 정보를 조회해줘"라고 요청할 때, 웹 서버는 데이터베이스 서버 입장에서 클라이언트가 되는 것입니다. 이처럼 하나의 시스템이 상황에 따라 서버도 되고 클라이언트도 될 수 있다는 점이 중요합니다.

:::info 핵심 포인트
- 클라이언트는 서버의 존재를 알지만, 다른 클라이언트의 존재는 모릅니다.
- 서버는 클라이언트에게 서비스를 제공하지만, 클라이언트에게 서비스를 요청하지는 않습니다.
- 서버도 다른 서버에게 요청할 때는 클라이언트 역할을 합니다.
:::

## 🤔 왜 클라이언트-서버 아키텍처가 필요한가?

### 등장 배경: 메인프레임의 시대에서 분산 환경으로

클라이언트-서버 아키텍처가 등장하기 전에는 **메인프레임(Mainframe)** 이라고 불리는 초대형 중앙 컴퓨터가 모든 처리를 담당했습니다. 메인프레임은 마치 혼자 모든 것을 처리하는 슈퍼 요리사와 같습니다. 모든 주문을 혼자 받고, 혼자 요리하고, 혼자 서빙까지 해야 하니 비용이 엄청나게 많이 들고, 한 명의 요리사가 과부하(Overload)에 걸리는 문제가 생깁니다.

클라이언트-서버 아키텍처는 이 문제를 해결하기 위해 등장했습니다. **여러 대의 일반 컴퓨터(서버)를 네트워크로 연결하여 역할을 나누어 처리**함으로써, 훨씬 적은 비용으로 동일하거나 더 높은 성능을 낼 수 있게 된 것입니다.

### 클라이언트-서버 아키텍처의 장점 🌟

**첫째, 비용 절감 효과입니다.** 메인프레임 한 대를 구매하는 비용보다 훨씬 저렴한 비용으로 동일한 서비스를 제공할 수 있습니다. 마치 한 명의 스타 셰프보다 여러 명의 일반 요리사를 고용하는 것이 비용 면에서 효율적인 것과 같습니다.

**둘째, 지역 분산 처리가 가능합니다.** 서버를 여러 지역에 분산시켜 배치할 수 있기 때문에, 데이터가 발생하는 그 지역에서 즉시 처리할 수 있습니다. 서울 사용자의 데이터는 서울 서버에서, 부산 사용자의 데이터는 부산 서버에서 처리하면 응답 속도가 훨씬 빨라집니다.

**셋째, 고성능 그래픽 사용자 인터페이스(GUI)를 지원합니다.** 고성능 워크스테이션(Workstation, 고사양 업무용 컴퓨터)에서 가능한 풍부한 시각적 인터페이스를 클라이언트 측에서 제공할 수 있습니다.

### 클라이언트-서버 아키텍처의 단점 ⚠️

장점만 있으면 완벽한 기술이겠지만, 현실에서는 항상 트레이드오프(Trade-off, 장점을 얻으면 단점도 생기는 상충관계)가 존재합니다.

**첫째, 서버 과부하 문제입니다.** 점점 더 많은 응용 처리 로직(비즈니스 로직)이 서버 쪽으로 집중되면서, 결국 서버도 메인프레임과 비슷한 수준의 높은 부하(Load)를 받게 됩니다. 서버가 처리해야 할 일이 너무 많아지면 성능이 저하되는 문제가 생깁니다.

**둘째, 분산 시스템의 복잡성입니다.** 서버를 여러 곳에 분산시켜 놓으면 관리가 매우 복잡해집니다. 수십, 수백 대의 서버가 네트워크로 연결되어 동시에 병렬로 작업을 처리하고 데이터를 합치는 과정은 매우 복잡한 구조를 만들어냅니다.

**셋째, 보안 취약성입니다.** 서버를 100개 분산시켜 운영한다면, 100개 중 단 하나에만 보안 구멍이 뚫려도 전체 시스템의 보안이 위협받습니다. 최근 해킹 사고들의 많은 부분이 이러한 분산 환경의 취약점을 노린 것입니다.

:::warning 분산 환경의 보안 위협
서버가 많아질수록 관리해야 할 보안 취약점도 늘어납니다. 분산 환경에서는 단 하나의 서버만 뚫려도 전체 시스템이 위험해질 수 있으므로, 모든 서버에 대한 균일한 보안 관리가 필수입니다.
:::

## 🏢 클라이언트-서버 시스템의 4계층 구조

### 왜 계층(Layer)으로 나누는가?

소프트웨어 시스템을 설계할 때 역할별로 계층을 나누는 이유는 **각 계층이 자신의 역할에만 집중할 수 있도록 하여 유지보수와 확장을 쉽게 만들기 위해서**입니다. 마치 회사 조직도처럼 영업팀, 개발팀, 인사팀이 각자의 역할에 집중하고 서로 협력하는 구조와 같습니다.

클라이언트-서버 시스템은 일반적으로 다음 4개의 계층으로 구성됩니다.

### 4개 계층 상세 설명

```
┌─────────────────────────────────────┐
│   표현 계층 (Presentation Layer)    │  ← 사용자와 가장 가까운 계층
├─────────────────────────────────────┤
│  데이터 처리 계층 (Data Layer)      │
├─────────────────────────────────────┤
│  애플리케이션 계층 (Application     │
│           Processing Layer)         │
├─────────────────────────────────────┤
│  데이터베이스 계층 (Database Layer) │  ← 하드웨어와 가장 가까운 계층
└─────────────────────────────────────┘
```

**① 표현 계층 (Presentation Layer) 🖥️**

표현 계층은 사용자와 직접 소통하는 계층입니다. 사용자가 눈으로 보고 손으로 입력하는 모든 화면 관련 처리가 이 계층에서 이루어집니다. 웹사이트의 버튼, 입력창, 메뉴, 이미지 등 여러분이 화면에서 보는 모든 시각적 요소들이 표현 계층의 산물입니다. 프레젠테이션 계층(Presentation Layer)이라고도 불리며, 사용자 인터페이스(UI)를 담당합니다.

**② 데이터 처리 계층 (Data Processing Layer) 🔄**

클라이언트가 요청한 내용 중 데이터와 관련된 처리가 이 계층에서 이루어집니다. 예를 들어, 사용자가 회원가입 폼을 제출했을 때 입력된 이메일 형식이 올바른지 검증하거나, 웹페이지 형태로 데이터를 구성해서 클라이언트에게 전달하는 작업이 여기에 해당합니다. 데이터의 유효성 검사, 데이터 변환, 웹페이지 생성 등의 작업이 이 계층의 역할입니다.

**③ 애플리케이션 처리 계층 (Application Processing Layer) ⚙️**

실제 비즈니스 로직(Business Logic, 업무 규칙과 처리 절차)이 수행되는 계층입니다. 예를 들어, 쇼핑몰에서 "장바구니에 담긴 상품의 총 금액을 계산하고, 할인 쿠폰을 적용하고, 배송비를 추가하는" 일련의 계산 처리가 바로 애플리케이션 계층에서 이루어집니다. 요구된 기능들을 실제로 실행하고 결과를 만들어내는 핵심 처리 계층입니다.

**④ 데이터베이스 계층 (Database Layer) 🗄️**

맨 아래에 위치하며, 실제 데이터를 저장하고 관리하는 계층입니다. 데이터베이스에 직접 접근하여 데이터를 저장, 조회, 수정, 삭제하는 작업을 수행합니다. 또한 트랜잭션(Transaction, 작업의 최소 단위 묶음)을 관리하고, 다양한 사용자의 데이터베이스 질의(Query, 데이터 검색 요청)에 응답하는 서비스를 제공합니다.

| 계층 | 주요 역할 | 실생활 비유 |
|---|---|---|
| **표현 계층** | 화면 표시, 사용자 입력 처리 | 음식점 홀 서빙 직원 |
| **데이터 처리 계층** | 데이터 검증, 웹페이지 생성 | 주문서를 정리하는 캐셔 |
| **애플리케이션 계층** | 비즈니스 로직 처리, 계산 수행 | 실제 요리하는 주방장 |
| **데이터베이스 계층** | 데이터 저장, 조회, 트랜잭션 관리 | 식재료를 보관하는 냉장고와 창고 |

:::note 계층 구조 이해의 핵심
각 계층은 자신의 바로 위/아래 계층과만 소통하는 것이 원칙입니다. 표현 계층이 직접 데이터베이스에 접근하는 것은 좋은 설계가 아닙니다. 이렇게 계층을 나누면 하나의 계층을 변경해도 다른 계층에 영향을 최소화할 수 있어 유지보수가 훨씬 쉬워집니다.
:::

## 🔌 미들웨어(Middleware)란 무엇인가?

### 미들웨어가 왜 필요한가?

분산 환경에서는 클라이언트와 서버가 각각 다른 운영체제, 다른 프로그래밍 언어, 다른 하드웨어 위에서 동작하는 경우가 흔합니다. 예를 들어, 클라이언트는 Windows 환경에서 Java로 개발된 앱을 사용하고, 서버는 Linux 환경에서 Python으로 구동될 수 있습니다. 이렇게 서로 다른 환경 사이에서 원활하게 통신하려면 **중간에서 통역해주는 존재**가 필요합니다.

> 🌉 **미들웨어(Middleware)** 란, 클라이언트와 서버 사이에 존재하는 소프트웨어로, 서로 다른 환경의 시스템을 연결해주고 데이터를 주고받을 수 있도록 매개(중간 다리) 역할을 하는 소프트웨어입니다.

미들웨어는 마치 **국제 무역에서의 통관 대행업체**와 같습니다. 한국에서 미국으로 물건을 보낼 때, 각 나라의 법규와 형식이 다르기 때문에 통관 대행업체가 중간에서 서류를 맞춰주고, 형식을 변환해주고, 안전하게 전달되도록 관리합니다. 미들웨어가 바로 이런 역할을 합니다.

### 미들웨어의 장점

**① 표준화된 동일한 인터페이스 제공:** 미들웨어는 표준화되어 있기 때문에 어떤 환경에서든 동일한 방식으로 시스템을 연결할 수 있습니다. 개발자는 서로 다른 환경의 차이를 직접 다룰 필요 없이 미들웨어가 제공하는 표준 인터페이스만 사용하면 됩니다.

**② 데이터 일관성 보장:** 서로 다른 환경에서 데이터를 교환할 때 데이터의 형식이나 내용이 왜곡되지 않고 정확하게 전달될 수 있도록 일관성을 보장합니다.

**③ 서로 다른 환경의 업무 연동:** 클라이언트 환경과 서버 환경이 달라도 서로 다른 업무 시스템을 연동하여 함께 처리할 수 있게 해줍니다.

**④ 분산 처리 효율 향상:** 실시간으로 발생하는 작업들을 분산 환경에서 중간중간 미들웨어를 통해 변환하고 즉시 전달하여 빠른 처리가 가능합니다.

### 미들웨어의 종류 📦

미들웨어는 기능과 목적에 따라 다양한 종류로 나뉩니다.

#### 1. 데이터베이스 미들웨어 (DB Middleware) 🗃️

데이터베이스 미들웨어는 **애플리케이션과 데이터베이스 서버를 연결해주는 미들웨어**입니다. 마치 전화 교환기처럼 애플리케이션의 데이터 요청을 데이터베이스가 이해할 수 있는 방식으로 변환하여 전달합니다. 데이터베이스를 제공하는 벤더(Vendor, 소프트웨어 공급업체)마다 다른 클라이언트 라이브러리를 제공합니다. 대표적인 예로 Microsoft의 **ODBC(Open Database Connectivity)**, Oracle의 **OCI(Oracle Call Interface)** 등이 있습니다. 단순히 클라이언트와 데이터베이스 서버만 있는 구조, 즉 **2-티어 아키텍처(2-Tier Architecture)** 에서 주로 사용됩니다.

#### 2. 웹 애플리케이션 서버 미들웨어 (WAS: Web Application Server) 🌐

현업에서 가장 많이 언급되는 미들웨어입니다. 개발 현장에서 "WAS(와스) 재시작해야 해", "웹로직(WebLogic) 서버 올려줘" 라는 말을 자주 듣게 되는데, 이것이 바로 **웹 애플리케이션 서버(WAS)** 입니다.

정적인 HTML 파일만 제공하는 일반 웹 서버와 달리, WAS는 **동적인 웹 콘텐츠(사용자마다 다른 내용을 보여주는 페이지)** 를 처리하기 위해 반드시 필요한 미들웨어입니다. 예를 들어 쇼핑몰에서 로그인하면 "홍길동 님, 환영합니다"라고 내 이름이 나오는 것처럼 사용자마다 다른 화면을 만들어내는 것이 WAS의 역할입니다. 대표적인 제품으로 **Oracle WebLogic**, **IBM WebSphere**, Apache Tomcat 등이 있습니다.

:::warning 웹 개발자라면 WAS는 필수!
웹 개발자로 실무에 투입된다면 WAS에 대한 이해는 선택이 아닌 필수입니다. 동적 웹 페이지 처리, 트랜잭션 관리, 보안 처리 등 현대 웹 서비스의 핵심 기능이 WAS 위에서 돌아가기 때문입니다.
:::

#### 3. 메시지 지향 미들웨어 (MOM: Message-Oriented Middleware) 📨

메시지 지향 미들웨어(MOM)는 **클라이언트가 생성한 메시지(작업 요청)를 일단 저장소에 모아두고, 서버가 준비됐을 때 하나씩 처리하는 방식**입니다. 이것을 **비동기식(Asynchronous) 처리 방식**이라고 합니다.

이것을 카페 주문에 비유하면 이해하기 쉽습니다. 스타벅스처럼 주문을 받는 직원(클라이언트)이 계속 주문을 받아 주문표를 만들어 붙여놓고, 바리스타(서버)가 준비된 순서대로 커피를 만드는 방식입니다. 주문 직원은 바리스타가 커피를 다 만들 때까지 기다리지 않고 계속 다음 주문을 받을 수 있습니다.

```
클라이언트 → [메시지 저장소 Queue] → 서버
             ↑                    ↑
         메시지 적재            순서대로 처리
```

MOM은 온라인 실시간 처리보다는 **분산 환경에서 대량의 데이터를 순차적으로 처리해야 할 때** 주로 사용합니다. 대표적인 제품으로 Oracle의 **Message Q**, IBM의 **MQ**, **JMS(Java Message Service)** 등이 있습니다.

#### 4. 트랜잭션 처리 모니터 (TP Monitor: Transaction Processing Monitor) 🎯

트랜잭션(Transaction, 작업의 최소 처리 단위)을 관리하고 감시하는 미들웨어입니다. 주로 **실시간 온라인 트랜잭션 처리(OLTP, Online Transaction Processing)** 환경에서 사용합니다.

가장 쉬운 예시는 **명절 기차표 예매**입니다. 설날이나 추석에 동시에 수십만 명이 KTX 표를 예매하려고 접속합니다. 이때 서버가 과부하로 멈추지 않도록 트랜잭션을 효율적으로 관리하고, 사용자 수가 폭발적으로 증가하더라도 빠른 응답 속도를 유지하는 역할을 트랜잭션 처리 모니터가 담당합니다. 대표적인 제품으로 Oracle의 **Tuxedo**, 티맥스소프트의 **Tmax** 등이 있습니다.

#### 5. 원격 프로시저 호출 (RPC: Remote Procedure Call) 📞

**원격 프로시저 호출(RPC)** 은 네트워크 너머 원격 컴퓨터에 있는 함수(프로시저)를 마치 내 컴퓨터에 있는 함수처럼 호출하여 실행하는 방식입니다. 실제로는 원격에서 처리되지만, 개발자 입장에서는 로컬(내 컴퓨터)에서 처리하는 것처럼 느껴집니다. 마치 전화로 친구에게 "이거 좀 확인해줘"라고 부탁하면, 친구가 직접 처리해서 결과를 알려주는 것과 같습니다.

#### 6. 객체 요청 브로커 (ORB: Object Request Broker) 🔁

**객체 요청 브로커(ORB)** 는 객체 지향(Object-Oriented) 방식으로 분산 환경에서 서로 다른 시스템 간의 통신을 가능하게 하는 미들웨어입니다. **CORBA(Common Object Request Broker Architecture, 코바)** 라는 국제 표준 스펙을 기반으로 만들어집니다. CORBA는 서로 다른 환경의 시스템 간에 객체(데이터와 기능이 묶인 단위) 형태로 정보를 주고받을 수 있게 해주는 분산 객체 컴퓨팅의 표준입니다.

:::info CORBA와 OMG
CORBA를 만든 표준화 기구가 **OMG(Object Management Group)** 입니다. OMG는 객체 지향 기술의 표준을 만들고 관리하는 국제 단체로, CORBA 외에도 UML 등 다양한 소프트웨어 표준을 제정합니다.
:::

## 🏛️ 2-티어, 3-티어, N-티어 아키텍처

### 티어(Tier)란 무엇인가?

**티어(Tier)** 는 시스템을 구성하는 **물리적 또는 논리적 층(Layer)의 수**를 의미합니다. 클라이언트와 서버 사이에 몇 개의 독립적인 처리 단위가 있느냐에 따라 2-티어, 3-티어, N-티어로 구분합니다.

### 2-티어 아키텍처 (2-Tier Architecture)

2-티어는 가장 단순한 형태의 클라이언트-서버 구조입니다. 클라이언트와 서버, 딱 두 개의 층으로만 이루어져 있습니다.

```
[클라이언트]  ←→  [서버(데이터베이스)]
 - 화면 표시        - 데이터 저장/조회
 - 비즈니스 로직
 - 데이터 처리
```

2-티어 구조에서는 클라이언트가 **화면 표시, 비즈니스 로직 처리, 데이터 처리**를 모두 담당하고, 서버는 주로 데이터베이스 관리만 담당합니다. 이 구조를 **팻 클라이언트 모델(Fat Client Model)** 이라고도 부릅니다. "팻(Fat, 뚱뚱한)"이라는 표현은 클라이언트가 많은 일을 처리하기 때문에 그만큼 "무거운" 클라이언트라는 의미입니다.

**2-티어의 단점:** 비즈니스 로직이 클라이언트 측에 있기 때문에 업무 규칙이 변경되면 모든 클라이언트 프로그램을 업데이트해야 하는 번거로움이 있습니다.

### 3-티어 아키텍처 (3-Tier Architecture)

3-티어는 현재 가장 보편적으로 사용되는 구조입니다. 클라이언트와 데이터베이스 서버 사이에 **웹 서버(미들웨어 역할)** 가 추가됩니다.

```
[클라이언트]  ←→  [웹 서버(WAS)]  ←→  [DB 서버]
 - 화면 표시        - 비즈니스 로직    - 데이터 저장/조회
                   - 데이터 처리      - 트랜잭션 관리
```

3-티어 구조에서는 **비즈니스 로직이 웹 서버(WAS) 측으로 이동**합니다. 이로 인해 업무 규칙이 변경되면 서버 한 곳만 수정하면 되므로 **유지보수가 훨씬 편리**해집니다. 또한 클라이언트는 화면 표시만 담당하므로 가벼워집니다. 이 구조를 **씬 클라이언트 모델(Thin Client Model)** 이라고도 합니다. "씬(Thin, 가벼운)"이라는 표현은 클라이언트가 할 일이 적어 가볍다는 의미입니다.

| 구조 | 클라이언트 역할 | 서버 구성 | 특징 |
|---|---|---|---|
| **2-티어 (팻 클라이언트)** | 화면 + 비즈니스 로직 + 데이터 처리 | DB 서버 1개 | 단순하지만 유지보수 어려움 |
| **3-티어 (씬 클라이언트)** | 화면 표시만 | 웹 서버 + DB 서버 | 유지보수 용이, 보안 강화 |
| **N-티어** | 화면 표시만 | 여러 층의 서버 | 대규모 복잡한 시스템에 적합 |

### N-티어 아키텍처 (N-Tier Architecture)

N-티어는 3-티어를 더욱 확장한 개념으로, 처리해야 할 응용 서비스가 매우 복잡하고 다양할 때 사용합니다. N은 2, 3, 4, 5... 어떤 숫자든 될 수 있습니다.

```
[클라이언트] ←→ [웹 서버] ←→ [비즈니스 컴포넌트] ←→ [EIS(엔터프라이즈 정보 시스템)/DB]
```

**N-티어 구조의 각 계층:**
- **클라이언트 층:** HTML 웹페이지를 처리하고 화면에 표시
- **웹 층(WAS):** JSP, ASP, 서블릿(Servlet) 등 동적 웹페이지 생성
- **비즈니스 컴포넌트 층:** 핵심 비즈니스 로직 처리 (EJB, .NET, CORBA 등 활용)
- **EIS 층(Enterprise Information System, 기업 정보 시스템):** 데이터베이스 처리

N-티어 아키텍처는 **대용량 트랜잭션 처리**, **복잡한 비즈니스 로직**, **높은 확장성**이 필요한 대규모 기업 시스템에 적합합니다.

:::tip 마스터-슬레이브 아키텍처
N-티어 외에도 **마스터-슬레이브(Master-Slave) 아키텍처**라는 구조도 있습니다. 마스터(Master, 주 서버)가 전체 작업을 조율하고 슬레이브(Slave, 보조 서버)들이 실제 처리를 분담하는 구조로, 최소한의 응답 시간을 보장해야 하는 **실시간 시스템(Real-time System)** 에서 주로 사용합니다.
:::

## 🔬 미들웨어의 기술적 분류와 CORBA 심층 이해

### 기술적 분류: 트랜스포트, 애플리케이션 서버, 메시지 브로커, BPM

미들웨어를 기술적인 관점에서 분류하면 크게 4가지로 나눌 수 있습니다.

#### ① 트랜스포트 층 (Transport Layer) 🚚

**트랜스포트(Transport, 전송)** 층은 데이터를 분산 환경에서 이동시키는 역할을 하는 기본 파이프(Pipe, 통로)입니다. 소프트웨어 컴포넌트(Component, 독립적인 소프트웨어 단위) 사이의 요청을 보내고 데이터를 전달하기 위한 기반 인프라를 제공합니다. 마치 택배 회사의 물류 네트워크처럼 어디서 어디로 데이터를 전달할지 경로를 확보해주는 역할입니다. 이 층에서는 **메시지 기반 미들웨어**, **분산 객체 시스템(SOAP)** 등의 기술이 사용됩니다.

#### ② 애플리케이션 서버 (Application Server) ⚙️

실제 **응용 처리(Application Processing)** 를 담당하는 계층입니다. 트랜잭션(Transaction, 작업 묶음) 처리, 보안 및 권한 관리, 데이터 접근 및 디렉터리(Directory, 자원 목록) 관리, 그리고 **멀티스레드(Multi-thread, 여러 작업을 동시에 처리하는 방식)** 기반의 프로그램 모델을 지원합니다. 대표적인 제품으로 **J2EE(Java 2 Enterprise Edition)** 기반의 서버들과 **.NET** 환경의 서버들이 있습니다.

#### ③ 메시지 브로커 (Message Broker) 📬

**메시지 브로커(Message Broker)** 는 트랜스포트 층과 애플리케이션 서버 층 사이에서 **메시지를 중개(Brokering)** 하는 역할을 합니다. 브로커(Broker)란 중개인이라는 뜻으로, 부동산 중개인처럼 메시지를 받아 적절한 대상에게 전달합니다. 메시지 브로커의 핵심 기능은 **라우팅(Routing, 경로 설정)** 입니다.

예를 들어, 서버 A에서 서버 B로 메시지를 보낼 때 여러 경로가 존재할 수 있습니다. 메시지 브로커는 현재 네트워크 상황에서 가장 효율적인 경로를 선택하여 메시지를 전달합니다. 마치 네비게이션이 출발지에서 목적지까지 여러 경로 중 가장 빠른 길을 안내해주는 것과 같습니다. 대표 제품으로 **IBM WebSphere Message Broker** 등이 있습니다.

#### ④ 비즈니스 프로세스 관리자 (BPM: Business Process Manager) 📋

메시지 브로커의 기능에 **워크플로우(Workflow, 업무 흐름) 스타일 애플리케이션 지원 기능**이 추가된 계층입니다. 실제 비즈니스 프로세스(Business Process, 업무 절차)를 실행하고 관리하는 역할을 담당합니다. 예를 들어 "주문 접수 → 결제 확인 → 재고 확인 → 배송 시작" 같은 일련의 업무 흐름 전체를 자동으로 관리합니다. 대표 제품으로 **BizTalk**, **TIBCO**, **Staffware** 등이 있습니다.

### CORBA와 ORB 심층 이해 🔎

**CORBA(Common Object Request Broker Architecture, 코바)** 는 서로 다른 플랫폼, 서로 다른 언어로 개발된 객체(Object)들이 네트워크를 통해 서로 통신할 수 있게 해주는 분산 객체 컴퓨팅의 국제 표준 아키텍처입니다.

CORBA의 주요 구성 요소는 다음과 같습니다.

**① 객체 모델 (Object Model):** 데이터와 기능을 캡슐화(Encapsulation, 내부 구현을 숨기고 외부에 인터페이스만 제공하는 방식)한 객체의 구조와 상태를 정의합니다.

**② ORB (Object Request Broker, 객체 요청 중계자):** CORBA의 핵심으로, 클라이언트가 원격 객체에 서비스를 요청하면 그 요청을 받아서 적절한 서버의 객체에 전달하고 결과를 돌려주는 중계 역할을 합니다. 쉽게 말해 "요청을 받아 올바른 대상에게 전달하고 결과를 다시 가져다주는 택배 기사"입니다.

**③ 객체 서비스 집합 (Object Services):** 많은 분산 애플리케이션이 공통으로 필요로 하는 서비스들을 디렉터리 형태로 모아놓은 것입니다. 트랜잭션 서비스, 지속성(Persistence, 데이터 영구 저장) 서비스 등이 포함됩니다.

**④ 공통 컴포넌트 (Common Facilities):** 특정 도메인(Domain, 특정 업무 영역)의 컴포넌트와 여러 애플리케이션이 공통으로 사용하는 범용 컴포넌트로 구분됩니다.

```
[클라이언트 객체]
      ↓  요청
   [ORB (객체 요청 중계자)]  ← 핵심 중간 다리
      ↓  전달
[서버 객체 (다른 환경, 다른 언어)]
      ↓  결과 반환
   [ORB]
      ↓
[클라이언트 객체] ← 결과 수신
```

ORB가 필요한 이유는 **서비스를 제공하는 객체들이 네트워크상 어디에 있든 동적으로 찾아서 연결하고, 서로 다른 환경 간의 차이를 극복하여 통신**할 수 있게 해주기 때문입니다.

## ⚡ 정형 기법(Formal Method): 수학으로 소프트웨어를 검증한다

### 정형 기법이 등장한 배경

소프트웨어의 규모가 점점 커지면서 개발 현장에서는 심각한 문제가 반복되기 시작했습니다. 요구사항 명세서(Requirements Specification, 고객이 원하는 것을 문서로 정리한 것)를 작성했는데, 실제로 개발된 소프트웨어는 그 명세서와 다른 결과물이 나오는 일이 빈번해진 것입니다. 이를 **소프트웨어 위기(Software Crisis)** 라고 합니다.

> 💬 자동화 도구를 아무리 잘 만들어도, 새로운 프로그래밍 언어가 나와도, 명세서 자체가 부정확하거나 모호하면 결국 엉뚱한 소프트웨어가 만들어집니다.

이 문제를 해결하기 위해 "아예 수학적으로 증명이 가능한 방식으로 요구사항을 작성하고 검증하자"는 아이디어에서 **정형 기법(Formal Method)** 이 탄생했습니다.

### 정형 기법이란?

**정형 기법(Formal Method)** 은 **수학적 논리(Mathematical Logic)** 와 **형식 언어(Formal Language)** 를 사용하여 소프트웨어의 요구사항, 설계, 동작을 정확하게 기술하고 검증하는 소프트웨어 개발 방법론입니다.

수학이라는 것은 논리적으로 증명된 것은 옳다는 것이 보장됩니다. 마찬가지로, 수학적으로 검증된 소프트웨어 명세(Specification, 상세 규격)를 그대로 구현하면 이론적으로 오류(Error)가 없는 소프트웨어를 만들 수 있습니다.

### 정형 기법의 두 가지 축: 정형 명세와 정형 검증

**① 정형 명세 (Formal Specification):** 수리 논리나 수학적 언어를 이용하여 시스템의 환경, 요구사항, 설계를 정확하게 기술하는 것입니다. 크게 **요구 명세(Requirements Specification)** 와 **설계 명세(Design Specification)** 로 나뉩니다.

**② 정형 검증 (Formal Verification):** 작성된 명세가 실제로 요구사항을 만족하는지를 수학적·논리적 방법으로 증명하는 것입니다.

### 정형 명세의 종류

**① 논리 기반 명세 (Logic-based Specification):** 수리 논리에 기반하여 시스템을 명세하는 방법입니다. 시스템의 상태를 나타내는 변수와 타입을 정의하고, 기능이 실행되기 전(Pre-condition, 사전 조건)과 실행된 후(Post-condition, 사후 조건)의 상태를 수학적 형태로 표현합니다. 사용 언어로는 **First Order Logic(일차 술어 논리)**, **Propositional Logic(명제 논리)**, **Temporal Logic(시제 논리)** 등이 있습니다.

**② 대수 기반 명세 (Algebraic Specification):** 대수학(Algebra)의 공리(Axiom, 증명 없이 참으로 인정되는 기본 규칙) 규칙들을 이용하여 시스템을 명세하는 방법입니다. 수학적으로 검증된 공리 체계를 기반으로 하기 때문에 논리적으로 오류가 없는 명세를 작성할 수 있습니다. 특히 **병렬성(Parallelism)** 과 **동시성(Concurrency, 여러 작업이 동시에 실행되는 것)** 을 수학적으로 기술하는 데 유용합니다. 사용 언어로는 **π-Calculus(파이 계산법)**, **ACSR** 등이 있습니다.

**③ 오토마타 기반 명세 (Automata-based Specification):** 오토마타(Automata, 상태와 전이로 시스템을 모델링하는 수학적 기계) 이론을 기반으로 시스템의 **상태(State)** 와 **전이(Transition, 상태 간의 변화)** 를 시각적으로 표현하는 방법입니다. 시스템의 동적인 행동(Behavior)을 표현하는 데 탁월하며, **원자력 발전소**, **자동차 제어 시스템** 등 안전이 매우 중요한 실시간 시스템 설계에 널리 사용됩니다. 사용 언어로는 **Finite State Machine(유한 상태 기계)**, **Statechart** 등이 있습니다.

### 정형 검증의 종류

**① 의미론적 검증 (Semantic Verification):** 시스템 모델이 특정 속성을 만족하는지 오토마타 이론 등을 기반으로 **자동으로** 분석하는 방법입니다. 즉, 컴퓨터가 자동으로 검증해줄 수 있어 빠르고 효율적입니다.

**② 구문론적 검증 (Syntactic Verification):** 구문(Syntax, 문법 규칙)을 이용하여 논리적으로 증명하는 방법입니다. 의미론적 검증과 달리 **사람이 직접 단계별(Step by Step)로 증명**해나가야 합니다. 이 과정에서 일부 도구의 도움을 받을 수는 있지만, 본질적으로 사람의 논리적 사고가 필요합니다.

:::note 정형 기법의 현실적 위치
정형 기법은 이론적으로는 매우 강력하지만, 수학적 지식이 깊이 필요하고 적용하는 데 많은 비용과 시간이 들기 때문에 모든 소프트웨어 개발에 적용되지는 않습니다. 주로 항공기 제어 소프트웨어, 의료 기기, 원자력 시스템 등 **오류가 발생했을 때 인명 피해나 막대한 손실이 생기는 안전 필수 시스템(Safety-Critical System)** 에서 중점적으로 사용됩니다.
:::

## 🧭 주의사항 및 실무 팁

### 실무에서 이 개념들을 어떻게 활용하는가?

**① WAS에 대한 이해는 필수입니다.** 웹 개발자로 일하다 보면 "WAS 재시작", "WebLogic 설정 변경" 같은 업무 지시를 받게 됩니다. WAS가 무엇인지, 왜 필요한지 이해하고 있어야 실무 환경에서 당황하지 않습니다. WAS의 기본 개념인 동적 웹 처리, 트랜잭션 관리, 세션 관리 등을 반드시 학습하세요.

**② 2-티어, 3-티어 구조 중 어느 것을 선택할지 판단하는 눈을 길러야 합니다.** 작은 규모의 사내 시스템이라면 2-티어도 충분할 수 있습니다. 그러나 수천 명 이상이 사용하는 공개 서비스라면 3-티어 이상의 구조가 필수입니다.

**③ 분산 환경에서의 보안에 특별히 주의하세요.** 클라이언트-서버 아키텍처의 단점 중 하나가 분산 환경에서의 보안 취약성이라고 설명했습니다. 실무에서는 각 서버의 보안 패치 관리, 방화벽(Firewall) 설정, 접근 권한 관리에 세심한 주의를 기울여야 합니다.

:::danger 절대 하면 안 되는 실수: 클라이언트에 비즈니스 로직 노출
팻 클라이언트(Fat Client) 구조에서 비즈니스 로직을 클라이언트 측 코드에 직접 구현하면 사용자가 해당 로직을 분석하고 악용할 수 있습니다. 예를 들어 할인율 계산 로직이 클라이언트 앱에 그대로 들어있으면, 사용자가 이를 분석해 비정상적인 할인을 적용받을 수 있습니다. 반드시 핵심 비즈니스 로직은 서버(WAS) 측에서 처리하세요.
:::

### 자주 혼동하는 개념 정리

**웹 서버 vs WAS (웹 애플리케이션 서버):**

```
웹 서버 (Web Server)
  → 정적 파일(HTML, CSS, 이미지 등) 제공
  → 예: Apache, Nginx

WAS (Web Application Server)
  → 동적 콘텐츠 처리 (JSP, Servlet, Spring 등)
  → 비즈니스 로직 처리
  → 예: Tomcat, WebLogic, WebSphere
```

실제 운영 환경에서는 웹 서버와 WAS를 함께 사용하는 경우가 많습니다. 웹 서버가 정적 파일을 빠르게 처리하고, 동적 처리가 필요한 요청만 WAS로 전달하는 방식이 일반적입니다.

**동기식 처리 vs 비동기식 처리:**

```
동기식 (Synchronous):
  클라이언트 → 요청 → 서버 처리 완료까지 대기 → 응답 수신
  (전화 통화처럼 상대방이 응답할 때까지 기다림)

비동기식 (Asynchronous):
  클라이언트 → 요청을 메시지 큐에 저장 → 다른 작업 진행
  서버 → 준비되면 메시지 큐에서 꺼내 처리 → 결과 전달
  (문자 메시지처럼 보내고 나서 다른 일을 하다가 나중에 답장 확인)
```

MOM(메시지 지향 미들웨어)은 비동기식 처리 방식을 사용하므로, 대량 처리나 시스템 간 느슨한 결합(Loose Coupling, 서로 독립적으로 운영되는 구조)이 필요할 때 적합합니다.

## 📌 핵심 정리

이 문서에서 배운 클라이언트-서버 아키텍처의 핵심 개념을 정리합니다.

- **클라이언트-서버 아키텍처**는 서비스를 요청하는 클라이언트와 서비스를 제공하는 서버의 집합으로 구성된 시스템 모델이다.
- **클라이언트**는 서버의 존재는 알지만 다른 클라이언트의 존재는 알 필요가 없고, **서버는 클라이언트에게 서비스를 요청하지 않는다**는 것이 핵심 원칙이다.
- 클라이언트-서버 아키텍처의 **장점**은 저렴한 비용, 지역 분산 처리, GUI 지원이며, **단점**은 서버 과부하, 시스템 복잡성, 보안 취약성이다.
- 클라이언트-서버 시스템은 **표현 계층 → 데이터 처리 계층 → 애플리케이션 처리 계층 → 데이터베이스 계층**의 4계층으로 구성된다.
- **미들웨어(Middleware)** 는 클라이언트와 서버 사이에서 서로 다른 환경을 연결하고 통신을 가능하게 해주는 소프트웨어다.
- 미들웨어의 종류에는 **DB 미들웨어, WAS(웹 애플리케이션 서버), MOM(메시지 지향 미들웨어), TP 모니터, RPC, ORB**가 있다.
- **WAS(와스)** 는 동적 웹 콘텐츠 처리를 위한 핵심 미들웨어로, 웹 개발자에게 필수적인 기술이다.
- **MOM(메시지 지향 미들웨어)** 은 비동기식 처리 방식으로 대량의 메시지를 큐에 모아두고 순차적으로 처리한다.
- **2-티어(팻 클라이언트)** 는 클라이언트가 비즈니스 로직까지 처리하고, **3-티어(씬 클라이언트)** 는 비즈니스 로직이 서버로 이동하여 유지보수가 용이하다.
- **N-티어 아키텍처**는 복잡한 대규모 시스템에서 여러 계층을 나누어 역할을 분리한 구조다.
- **CORBA**는 서로 다른 환경의 분산 객체들이 통신할 수 있게 하는 국제 표준이며, **ORB(객체 요청 중계자)** 가 핵심 역할을 한다.
- **정형 기법(Formal Method)** 은 수학적 논리를 이용해 소프트웨어를 명세하고 검증하는 방법으로, 소프트웨어 위기를 해결하기 위해 등장했다.
- 정형 기법은 **정형 명세**(논리 기반, 대수 기반, 오토마타 기반)와 **정형 검증**(의미론적, 구문론적)으로 나뉜다.
