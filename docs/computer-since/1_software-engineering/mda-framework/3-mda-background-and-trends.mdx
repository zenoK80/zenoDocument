# 🏗️ MDA 등장 배경: 기존 미들웨어의 한계와 개발 패러다임 변화

소프트웨어 공학을 공부하다 보면 "MDA", "SOA", "컴포넌트 기반 개발"처럼 다소 낯설고 추상적인 용어들을 접하게 됩니다. 이런 개념들은 실제로 시스템을 구축하면서 익히면 가장 빠르게 이해할 수 있지만, 지금 이 시점에서 "아, 이런 것이 존재하는구나, 이런 이유로 필요한 거구나"를 파악해 두는 것만으로도 충분히 가치 있습니다. 나중에 실무에서 이 용어들을 마주쳤을 때 "소프트웨어 공학에서 배웠던 그것!" 하고 자신 있게 연결지을 수 있기 때문입니다.

이 문서에서는 **MDA(Model Driven Architecture, 모델 기반 아키텍처)** 가 왜 등장했는지, 기존 미들웨어(소프트웨어와 소프트웨어 사이를 연결해주는 중간 소프트웨어)의 어떤 한계가 있었는지, 그리고 최신 개발 패러다임(개발의 사고방식이나 접근 방법)이 어떻게 변화하고 있는지를 초보자도 이해할 수 있도록 차근차근 풀어봅니다.

---

## 🧩 MDA란 무엇인가? — 개념 소개

### 모델이 코드를 만들어낸다는 발상

**MDA(Model Driven Architecture)**를 한 문장으로 표현하면, **"모델을 기반으로 플랫폼과 독립적으로 소프트웨어를 자동으로 생성해내는 기술"** 입니다.

여기서 "모델(model)"이란 실제 코드를 짜기 전에 시스템의 구조와 동작 방식을 추상적으로 표현해 놓은 설계도 같은 것입니다. 마치 건축가가 실제 건물을 짓기 전에 도면을 그리듯이, 소프트웨어에서도 코드를 짜기 전에 모델이라는 설계도를 먼저 그립니다.

그런데 MDA가 특별한 이유는 이 모델이 단순한 그림이 아니라는 점입니다. **MDA에서의 모델은 정해진 표준에 따라 작성되어, 나중에 실제 코드로 자동 변환(자동 생성)될 수 있습니다.** 마치 공장에서 설계도만 입력하면 기계가 알아서 부품을 찍어내듯이, MDA에서는 모델을 입력하면 프레임워크(개발의 전체 틀)가 알아서 코드를 찍어냅니다.

> 💡 **핵심 포인트:** MDA는 코드를 자동으로 생성하기 위한 기술입니다. 이를 위해 모델이 특정 플랫폼(예: Windows, Linux, Java 환경 등)에 종속되지 않고 독립적으로 설계됩니다. 그 독립적인 모델을 나중에 원하는 플랫폼에 맞게 매핑(연결·변환)해서 최종 코드가 만들어지는 구조입니다.

### 플랫폼 독립성이 왜 중요한가?

"플랫폼에 종속되지 않는다"는 말이 왜 중요할까요? 예를 들어 어떤 쇼핑몰 시스템을 Java 환경으로 개발했다가 나중에 Python 환경으로 바꾸고 싶다고 가정해봅시다. 기존 방식대로라면 처음부터 다시 개발해야 할 수도 있습니다. 하지만 모델이 플랫폼과 독립적으로 만들어져 있다면, 같은 모델을 Java용으로도, Python용으로도, 모바일용으로도 각각 매핑해서 코드를 생성할 수 있습니다.

즉, 설계도 하나로 여러 환경에 맞는 코드를 찍어낼 수 있다는 것이 MDA의 핵심 강점입니다.

:::info MDA의 정식 명칭
MDA는 **Model Driven Architecture**의 약자로, OMG(Object Management Group, 객체 관리 그룹)라는 국제 표준화 단체가 제안한 소프트웨어 개발 프레임워크입니다. "모델 주도 아키텍처"라고 번역하기도 합니다.
:::

## ❗ 왜 MDA가 필요해졌는가? — 기존 미들웨어의 한계와 배경

### 기존 미들웨어가 가진 문제들

MDA가 등장하게 된 배경을 이해하려면 먼저 **기존 미들웨어(Middleware)**의 한계를 알아야 합니다. 미들웨어란 운영체제(OS)와 응용 프로그램 사이에서 다리 역할을 해주는 소프트웨어입니다. 쉽게 말하면, 앱과 앱 사이, 서버와 클라이언트 사이에서 "연결고리" 역할을 하는 소프트웨어입니다.

문제는 **각 미들웨어가 자기만의 표준으로 만들어진다**는 점입니다. A사 미들웨어는 A사 표준으로, B사 미들웨어는 B사 표준으로 동작하다 보니, A 미들웨어와 B 미들웨어를 함께 쓰고 싶어도 서로 호환이 되지 않는 문제가 생깁니다.

마치 각기 다른 나라에서 만들어진 충전기들이 플러그 모양이 제각각이라 맞지 않는 것처럼, 미들웨어들도 서로 다른 "플러그 규격"을 갖고 있어서 연결이 안 되는 상황이 빈번하게 발생했습니다.

> **미들웨어 간 상호 운용성(서로 다른 시스템끼리 막힘 없이 통신하고 동작하는 능력) 문제는 기존 개발 환경에서 오랜 기간 해결되지 않은 숙제였습니다.**

### CORBA의 복잡성 문제

MDA 이전에는 **CORBA(Common Object Request Broker Architecture, 공통 객체 요청 브로커 아키텍처)**라는 기술이 있었습니다. CORBA는 OMG라는 단체가 만든 개방형 객체 표준으로, 원격에서 객체(데이터와 기능을 묶은 단위)를 다룰 수 있게 해주는 기술입니다.

하지만 CORBA는 **"무겁고 복잡하다"** 는 단점이 있었습니다. 설정도 어렵고, 구조도 복잡하며, 빠르게 변화하는 현대 개발 환경에서 사용하기가 쉽지 않다는 평가를 받으면서 점차 외면받기 시작했습니다.

### 빠르게 변화하는 개발 패러다임

기술적 한계뿐만 아니라, 시장 환경 자체가 빠르게 변화하면서 개발 방식에 대한 요구사항도 달라졌습니다. 이를 몇 가지 키워드로 정리하면 다음과 같습니다.

| 문제 | 설명 |
|------|------|
| **타임 투 마켓(Time to Market)** | 소비자가 원하는 시점에 제품이 나와야 합니다. 개발에 6개월이 걸리면, 6개월 뒤에는 소비자의 니즈가 이미 바뀌어 있을 수 있습니다. |
| **상호 운용성 부족** | 서로 다른 플랫폼, 서로 다른 미들웨어 사이에서 원활하게 데이터를 주고받지 못하는 문제가 지속됩니다. |
| **높은 인건비와 낮은 생산성** | 개발자 인건비는 계속 올라가는데, 생산성은 그만큼 따라가지 못하는 상황이 발생합니다. |
| **유지보수의 어려움** | 요구사항이 계속 바뀌는데, 기존 코드를 고치기 어려워 유지보수 비용이 천문학적으로 늘어납니다. |
| **표준 모델의 부재** | 개발 방법이나 모델에 대한 공통 표준이 부족해서, 팀마다 다른 방식으로 개발하는 혼란이 생깁니다. |

:::warning 개발 패러다임 변화의 핵심
요즘 앱이나 서비스들을 보면 정말 빠르게 등장하고, 빠르게 사라집니다. 보안 인증, IoT(사물인터넷) 서비스, 금융 앱 등 모든 분야에서 "빠른 개발 → 빠른 배포 → 빠른 대응"이 핵심 경쟁력이 됐습니다. 기존의 무겁고 느린 미들웨어 구조로는 이 속도를 따라가기 어려웠습니다.
:::

## 🔬 MDA의 핵심 원리 — 메타모델과 MDD 프로세스

### 표준 메타모델(Meta Model)이란?

MDA의 핵심 원리 중 하나는 **표준 메타모델(Standard Meta Model)**입니다. 메타모델이란 "모델을 위한 모델", 즉 **모델을 만들기 위한 기본 규칙과 구성 요소를 정의해 놓은 것**입니다.

조금 더 쉽게 비유하면, 레고(LEGO)와 같습니다. 레고 블록은 정해진 규격(크기, 연결 방식)이 있기 때문에, 어떤 블록이든 서로 연결할 수 있습니다. 만약 블록마다 규격이 다르다면 조립이 불가능하겠죠. **메타모델이 바로 이 레고 블록의 규격 역할**을 합니다. 메타모델 덕분에 서로 다른 시스템, 서로 다른 컴포넌트들이 호환되고 자동으로 연동될 수 있습니다.

메타모델을 기반으로 개발하면 다음 두 가지가 보장됩니다:

1. **호환성(Compatibility):** 서로 다른 시스템 간에도 같은 메타모델을 따르므로 연동이 쉬워집니다.
2. **자동 생성(Auto Generation):** 메타모델에 따라 정의된 구성 요소들로부터 코드나 구조가 자동으로 만들어집니다.

### MDA 메타모델의 구성 요소

MDA의 메타모델 구조는 몇 가지 핵심 기술 표준으로 이루어져 있습니다.

| 구성 요소 | 풀네임 | 역할 |
|-----------|--------|------|
| **MOF** | Meta Object Facility (메타 객체 기능) | 모델 정보에 대한 표준 저장소를 제공하고, 표준화된 모델 구조를 정의합니다. |
| **UML** | Unified Modeling Language (통합 모델링 언어) | 구현 환경과 무관하게 표준화된 방법으로 시스템을 모델링할 때 사용하는 표준 언어입니다. |
| **CWM** | Common Warehouse Metamodel (공통 데이터 창고 메타모델) | 데이터베이스 모델, 스키마 변환, OLAP(대용량 데이터 분석), 데이터 마이닝 등의 표준을 정의합니다. |
| **XMI** | XML Metadata Interchange (XML 메타데이터 교환) | UML로 기술된 모델 정보를 XML(데이터 교환 표준 언어) 형식으로 매핑하기 위한 표준 인터페이스입니다. |

:::note XML이란?
XML(eXtensible Markup Language, 확장 가능한 마크업 언어)은 HTML처럼 태그를 사용하는 언어인데, HTML보다 더 유연하게 데이터 구조를 표현할 수 있어서 시스템 간 데이터 교환 표준으로 널리 쓰입니다. XMI는 이 XML을 이용해서 MDA의 모델 데이터를 교환할 수 있도록 해주는 인터페이스입니다.
:::

### MDD 프로세스 — 실제 개발은 어떻게 이루어지나?

MDA를 이용해서 실제 소프트웨어를 개발하는 방식을 **MDD(Model Driven Development, 모델 주도 개발)**라고 합니다. MDD 프로세스는 세 가지 모델 단계를 거칩니다.

```
[CIM] → 매핑 → [PIM] → 매핑 → [PSM] → 매핑 → [실제 소스 코드]
```

각 단계를 자세히 살펴보겠습니다.

**① CIM (Computation Independent Model, 계산 독립적 모델)**

- 비즈니스 분석 단계입니다.
- 시스템이 어떤 기술로 구현될지와는 완전히 독립적으로, 비즈니스 요구사항만을 분석합니다.
- "이 시스템이 어떤 업무를 처리해야 하는가?"에만 집중합니다.
- **주로 비즈니스 분석가(Business Analyst)가 담당합니다.**

**② PIM (Platform Independent Model, 플랫폼 독립적 모델)**

- 특정 기술 플랫폼(Java, .NET, Python 등)과 무관하게 시스템의 구조와 동작을 설계합니다.
- "어떻게 설계할 것인가?"에 집중하되, "어떤 기술로 구현할 것인가?"는 고려하지 않습니다.
- **주로 아키텍처 설계자(Architect)가 담당합니다.**

**③ PSM (Platform Specific Model, 플랫폼 종속적 모델)**

- PIM을 특정 플랫폼에 맞게 변환한 모델입니다.
- 이제 "Java로 만들 것이다", "Oracle DB를 사용할 것이다" 같은 기술적 결정이 반영됩니다.
- **주로 시스템 아키텍처 설계자가 담당합니다.**

**④ 소스 코드 자동 생성**

- PSM을 바탕으로 코드 생성기(Code Generator)가 실제 코드를 자동으로 만들어냅니다.
- **개발자(Developer)가 최종 확인 및 테스트를 담당합니다.**

:::tip MDD 프로세스를 건축에 비유하면?
- **CIM** = 건축주가 "방 3개짜리 집이 필요하다"고 요구하는 단계
- **PIM** = 건축가가 집의 구조와 설계도를 그리는 단계 (재료는 아직 미정)
- **PSM** = 시공업자가 "이 지역 기후와 예산에 맞게 벽돌로 짓겠다"고 결정하는 단계
- **소스 코드** = 실제로 집을 짓는 단계
:::

## 🏭 SOA와 컴포넌트 기반 개발 — 실제 적용 예시

### SOA(서비스 지향 아키텍처)란?

**SOA(Service Oriented Architecture, 서비스 지향 아키텍처)**는 응용 프로그램을 개발할 때, 전체 로직을 하나로 묶어서 처리하는 것이 아니라 **기업의 업무를 의미 있는 서비스 단위로 잘게 나누어 표준화하는 방법**입니다.

예를 들어 학교 시스템을 만든다고 하면, 기존 방식은 "성적 처리 전체 프로그램"을 통째로 개발하는 것입니다. 반면 SOA 방식은 "성적 조회 서비스", "장학금 산출 서비스", "수강 신청 서비스"처럼 업무를 서비스 단위로 쪼개고, 각 서비스를 표준 인터페이스를 통해 제공하는 것입니다.

SOA가 필요한 이유는 다음과 같습니다:

- 비즈니스 프로세스가 바뀌어도 해당 서비스만 교체하면 되기 때문에 **변화에 빠르게 대응**할 수 있습니다.
- 서비스 단위로 나뉘어 있으므로 **재사용이 용이**합니다.
- 복잡한 기업 시스템을 서비스로 분리하면 **관리가 쉬워집니다.**

> **서비스(Service)의 정의:** 플랫폼에 종속되지 않은 표준 인터페이스를 통해 기업의 업무를 표현한, 느슨하게 결합되어(Loosely Coupled) 상호 조합이 가능한 소프트웨어 컴포넌트입니다.

"느슨하게 결합(Loosely Coupled)"이라는 표현이 중요합니다. 레고 블록을 생각해보세요. 레고는 어느 방향으로든 끼웠다 뺄 수 있습니다. 반면 접착제로 붙인 블록은 절대 분리할 수 없죠. SOA의 서비스들은 레고처럼 느슨하게 연결되어 있어서, 필요에 따라 다른 서비스와 자유롭게 조합할 수 있습니다.

### SOA의 발전 단계

SOA는 세 가지 발전 단계로 구분됩니다.

**① 펀더멘탈 SOA (Fundamental SOA)**

기존 시스템들을 모두 서비스 형태로 변환하고, 각 서비스들을 통합하는 가장 기본적인 형태입니다. 단점은 서비스가 많아질수록 서비스 간 연결이 매우 복잡해진다는 점입니다. 마치 전화기가 늘어날수록 전화선이 거미줄처럼 엉키는 것과 같습니다.

**② 네트워크 SOA (Network SOA)**

서비스들 사이에 **ESB(Enterprise Service Bus, 엔터프라이즈 서비스 버스)**라는 중앙 허브를 두어, 모든 서비스 간 통신이 이 허브를 통하도록 하는 방식입니다. ESB는 서비스 간의 변환, 로깅(기록), 통제, 순서 관리 등을 담당합니다. 마치 교통 경찰처럼 "너 먼저 가, 나는 잠깐 기다려" 역할을 합니다. 덕분에 중앙 통제력과 유연성이 강화됩니다.

**③ 프로세스 중심 SOA (Process-Centric SOA)**

비즈니스 흐름(Business Flow)이 있을 때 사용하는 방식입니다. 예를 들어 대출 신청 프로세스를 보면, "대출 신청 → 심사 → 승인 → 이체"라는 흐름이 있습니다. 각 단계가 서비스이고, 이 서비스들이 ESB를 통해 프로세스 순서대로 연결되어 동작합니다. 업무 흐름이 변경되면 서비스의 순서나 구성만 바꾸면 되므로 **업무 변화에 매우 민첩하게 반응**할 수 있습니다.

### 컴포넌트 기반 소프트웨어 공학(CBD)

**컴포넌트(Component)**라는 말을 들으면 자동차 부품을 떠올리시면 됩니다. 자동차는 바퀴, 엔진, 차체, 시트가 각각 독립적인 부품으로 만들어집니다. 자동차 공장에서는 이 부품들을 조립하기만 합니다. 사용자가 시트 색깔을 바꾸고 싶다면? 시트 부품만 교체하면 됩니다. 엔진을 바꾸고 싶다면? 엔진 부품만 교체하면 됩니다.

**컴포넌트 기반 소프트웨어 공학(Component-Based Development, CBD)**은 바로 이 원리를 소프트웨어에 적용한 것입니다. 즉, **독립적인 기능을 갖는 소프트웨어 부품(컴포넌트)을 미리 만들어두고, 이를 조립해서 새로운 시스템을 만드는 방법**입니다.

:::tip 컴포넌트의 핵심 특징: 인터페이스를 통해서만 접근 가능
컴포넌트는 반드시 **인터페이스(Interface)**를 통해서만 외부와 연결됩니다. 컴포넌트 내부 구현이 어떻게 되어 있든 상관없이, 정해진 인터페이스 규격대로 연결하면 됩니다. 마치 USB 포트처럼, 어떤 USB 기기든 USB 규격만 맞으면 연결되는 것과 같습니다.
:::

### 컴포넌트의 분류 구조

컴포넌트는 크기와 역할에 따라 계층적으로 분류됩니다.

```
요소 컴포넌트 (가장 작은 단위, 더 이상 쪼갤 수 없음)
    ↓ 여러 개가 묶이면
기능 컴포넌트 (특정 기능을 수행하는 단위)
    ↓ 여러 개가 묶이면
서비스 컴포넌트 (사용자에게 제공되는 서비스 단위)
    ↓ 여러 개가 묶이면
애플리케이션 컴포넌트 (전체 시스템)
```

예를 들어 재고 관리 시스템을 만든다면:
- **요소 컴포넌트:** "재고 수량 계산" (더 쪼갤 수 없는 최소 단위)
- **기능 컴포넌트:** "입출고 처리" (입고 + 출고 + 재고 수량 계산이 묶인 것)
- **서비스 컴포넌트:** "자재 관리 서비스" (입출고 처리 + 발주 관리 등이 묶인 것)
- **애플리케이션:** "통합 ERP 시스템" (모든 서비스가 묶인 전체 시스템)

### CBD가 등장한 이유

CBD가 등장하기 전, **객체 지향 프로그래밍(Object-Oriented Programming, OOP)**이 주류였습니다. OOP는 객체의 재사용이 장점이었지만, 실제로는 재사용이 쉽지 않았습니다. 같은 컴파일러 환경에서만 객체를 재사용할 수 있었고, 확장성도 부족했으며, 복잡도가 높아질수록 유지보수가 어려워졌습니다.

이런 문제들을 해결하기 위해 CBD가 등장했습니다. CBD 방식으로 개발하면 다음과 같은 이점이 있습니다:

- **개발 기간 단축:** 이미 만들어진 컴포넌트를 조립하면 되므로 빠릅니다.
- **생산성 향상:** 처음부터 구현하는 것이 아니라 조립하는 것이므로 효율적입니다.
- **품질 향상:** 이미 검증된 컴포넌트를 사용하므로 품질이 보장됩니다.
- **변화 대응 용이:** 특정 컴포넌트만 교체하면 되므로 요구사항 변화에 빠르게 대응합니다.
- **복잡성 최소화:** 각 컴포넌트가 독립적이므로 전체 시스템의 복잡도가 줄어듭니다.
- **사용자 중심 개발:** 개발자 편의가 아닌 사용자 요구 중심으로 서비스를 설계합니다.

### 객체 지향 개발 vs 컴포넌트 기반 개발 비교

| 비교 항목 | 객체 지향 개발 (OOD) | 컴포넌트 기반 개발 (CBD) |
|-----------|---------------------|------------------------|
| **개발 방식** | 객체 독립성 중심으로 처음부터 구현 | 기존 컴포넌트를 검색·조립 |
| **필요 역량** | 높은 수준의 전문 개발 역량 필요 | 상대적으로 접근이 용이함 |
| **상호 운용성** | 상대적으로 낮음 | 높음 |
| **복잡성** | 높음 | 낮음 |
| **재사용성** | 이론적으로 가능하지만 실제는 어려움 | 설계 자체가 재사용을 전제로 함 |
| **유지보수** | 어려움 | 해당 컴포넌트만 교체하면 됨 |
| **유통** | 소프트웨어 내부에서만 사용 | 컴포넌트 전문 회사를 통해 사고팔 수 있음 |

## 🔍 관점 지향 프로그래밍(AOP) — 또 다른 개발 패러다임

### AOP가 왜 필요한가?

객체 지향 프로그래밍을 하다 보면 한 가지 불편한 점이 생깁니다. 예를 들어 은행 시스템을 개발한다고 해봅시다. 계좌 이체, 입출금, 이자 계산이라는 핵심 기능들이 있습니다. 그런데 이 세 가지 기능 모두 공통으로 "로그인 확인", "보안 처리", "트랜잭션(작업 묶음) 처리"를 해야 합니다.

기존 방식이라면, 계좌 이체 코드에도 보안 처리 코드가 들어가고, 입출금 코드에도 보안 처리 코드가 들어가고, 이자 계산 코드에도 보안 처리 코드가 들어가야 합니다. **같은 코드가 여러 곳에 중복 등장**하는 것이죠. 이렇게 되면 나중에 보안 처리 방식이 바뀌면, 모든 곳을 찾아다니며 수정해야 합니다. 불편하고, 실수가 생길 위험도 높습니다.

**AOP(Aspect Oriented Programming, 관점 지향 프로그래밍)**는 이 문제를 해결합니다. 핵심 기능과 공통으로 필요한 기능(로그인, 보안, 트랜잭션 등)을 **별도 모듈로 분리**하고, 나중에 **위빙(Weaving)**이라는 과정을 통해 필요한 지점에 삽입하는 방식입니다.

```
[핵심 관심 모듈]          [횡단 관심 모듈]
- 계좌 이체               - 로그인 처리
- 입출금                  - 보안 처리
- 이자 계산               - 트랜잭션 처리

            ↓ 위빙(Weaving) / 크로스커팅(Cross-Cutting)

[완성된 시스템]
- 계좌 이체 + 로그인 + 보안 + 트랜잭션
- 입출금 + 로그인 + 보안 + 트랜잭션
- 이자 계산 + 로그인 + 보안 + 트랜잭션
```

이렇게 하면 보안 처리 코드가 한 곳(횡단 관심 모듈)에만 존재하고, 나중에 변경도 그 한 곳만 수정하면 됩니다.

### AOP 주요 용어 정리

| 용어 | 설명 |
|------|------|
| **핵심 관심(Core Concern)** | 계좌 이체, 입출금처럼 시스템의 핵심 기능 |
| **횡단 관심(Cross-Cutting Concern)** | 로그인, 보안, 트랜잭션처럼 여러 핵심 기능에 공통으로 필요한 기능 |
| **조인 포인트(Join Point)** | 횡단 관심 모듈의 기능이 삽입될 수 있는 특정 위치(결합 지점) |
| **포인트 컷(Point Cut)** | 어떤 클래스의 어떤 조인 포인트에 횡단 관심을 적용할지 결정하는 것 |
| **어드바이스(Advice)** | 각 조인 포인트에서 실행될 실제 코드 |
| **위빙(Weaving)** | 핵심 모듈과 횡단 관심 모듈을 실제로 결합·삽입하는 과정 |
| **크로스커팅(Cross-Cutting)** | 위빙과 유사하게, 횡단 관심을 핵심 로직에 가로질러 적용하는 과정 |

AOP의 장점은 다음과 같습니다:

- **분할과 정복(Divide and Conquer):** 복잡한 시스템을 관심사별로 분리하여 관리합니다.
- **모듈화(Modularity):** 각 관심사가 독립적인 모듈로 관리됩니다.
- **느슨한 결합(Loose Coupling):** 모듈 간 의존성이 낮아집니다.
- **재사용성(Reusability):** 횡단 관심 모듈을 여러 핵심 기능에 재사용할 수 있습니다.

## ⚠️ 주의사항과 실무 팁

### MDA/MDD 적용 시 주의할 점

:::danger 자동 생성 = 마법이 아닙니다
MDA가 코드를 자동으로 생성해준다고 해서, 아무것도 입력하지 않아도 AI처럼 알아서 만들어주는 것이 아닙니다. "학사 관리 프로그램을 만들어줘"라고 명령하면 알아서 만들어주는 수준이 아닙니다. 반드시 처리 조건, 명세(Specification, 요구 사항 및 조건을 문서화한 것), 처리 방법을 표준화된 형식으로 정확하게 입력해야만 그것을 기반으로 코드가 자동 생성됩니다. **"GIGO(Garbage In, Garbage Out)"** — 쓰레기를 넣으면 쓰레기가 나옵니다.
:::

:::warning 표준이 없으면 자동화도 없습니다
MDA/MDD 방식에서 가장 중요한 것은 **표준(Standard)**입니다. 모델이 표준화되어 있어야만 자동 변환이 가능합니다. 팀마다 모델 작성 방식이 다르거나, 메타모델 규약을 따르지 않으면 자동 생성 자체가 불가능합니다. 처음에 표준을 잡는 데 많은 노력이 들어가지만, 그 이후에는 개발 속도가 크게 향상됩니다.
:::

### CBD 적용 시 주의할 점

컴포넌트 기반 개발에서 도메인 분석(Domain Analysis, 해당 업무 영역에 대한 분석)이 가장 중요합니다. 은행 시스템을 개발하려면, 은행 업무를 모르는 개발자가 혼자서는 절대 제대로 된 컴포넌트를 설계할 수 없습니다. **도메인 전문가(해당 분야 업무 전문가)와의 협업이 필수**입니다.

또한, 컴포넌트들은 **리포지터리(Repository, 컴포넌트 저장소)**에 잘 정리되고 문서화되어 있어야 합니다. 컴포넌트가 어떤 기능을 하는지, 인터페이스는 어떻게 생겼는지, 어떤 버전인지가 문서로 명확히 기록되어야 검색하고 재사용할 수 있습니다.

:::tip 형상 관리(Configuration Management)를 반드시 병행하세요
컴포넌트들은 시간이 지나면서 추가, 수정, 삭제됩니다. 이때 어떤 컴포넌트가 언제 어떻게 변경됐는지를 추적·관리하는 것이 **형상 관리**입니다. Git 같은 버전 관리 도구를 사용하는 것이 대표적인 예입니다. 형상 관리 없이는 컴포넌트들이 뒤죽박죽되어 재사용 자체가 불가능해집니다.
:::

### 최신 소프트웨어 개발 트렌드와 개발자의 자세

현재 소프트웨어 개발 트렌드를 간단히 정리하면 다음과 같습니다.

| 트렌드 | 핵심 내용 |
|--------|-----------|
| **AI 기반 개발** | AI가 코드 검증, 자동 생성, 오류 탐지 등에 활발히 활용되고 있습니다. |
| **CI/CD** (지속적 통합/배포) | 개발, 테스트, 배포가 분리되지 않고 동시에 병렬로 이루어집니다. |
| **컨테이너화** | Docker, Kubernetes 같은 도구로 배포와 관리를 단순화하고 확장합니다. |
| **마이크로서비스** | 서비스를 점점 더 작고 독립적으로 만들어 확장성과 유지보수성을 높입니다. |
| **보안 강화(DevSecOps)** | 개발 단계부터 보안을 내재화하는 접근 방식입니다. |
| **서버리스** | 서버 관리 없이 코드 실행에만 집중할 수 있는 클라우드 기반 개발 방식입니다. |
| **블록체인** | 스마트 계약, 디지털 자산 등 보안과 투명성이 강화된 앱 개발이 확대됩니다. |
| **사용자 경험(UX) 중심** | 모든 개발의 출발점은 "사용자가 원하는 것인가?"입니다. |

:::note 개발자로서 가장 중요한 태도
AI가 코딩을 도와주는 시대라고 해서 개발자가 필요 없어지는 것이 아닙니다. 오히려 **AI를 얼마나 잘 활용하여 개발 효율과 생산성을 높이느냐**가 핵심 역량이 됩니다. AI와 경쟁하는 것이 아니라, AI를 도구로 삼아 더 가치 있는 일에 집중하는 개발자가 되는 것이 중요합니다.
:::

## 📋 보안 공학 — 소프트웨어 안전성을 지키는 학문

### 보안 공학이란?

**보안 공학(Security Engineering)**은 데이터의 정확성과 시스템의 안정성을 유지하기 위해 보안, 제어, 방어 메커니즘을 연구하는 학문입니다. 단순히 기술적인 방화벽이나 암호화만을 다루는 것이 아니라, **물리학, 수학, 사회과학, 심리학, 경제학** 등 다양한 분야의 요소가 복합적으로 적용되는 융합 학문입니다.

보안 공학의 역할은 현재 시스템이 가진 보안 위험 상황을 파악하고, 그 위험을 정량적으로 평가하며, 고객에게 "당신의 시스템은 이런 위험이 있으니 이렇게 대응하세요"라고 조언하는 것입니다. 보안 정책 수립, 동기 유발(조직 내 보안 문화 조성), 기술적 메커니즘 적용, 보증(Assurance, 보안 조치가 실제로 효과 있음을 검증) 등이 보안 공학의 핵심 활동입니다.

보안 공학이 적용되지 않는 분야는 이제 거의 없습니다. 금융, 국방, 의료, 가정용 IoT 기기까지 모든 분야에서 보안은 필수입니다. 보안 관련 전문가 수요는 계속 증가하고 있으며, 보안 분야로의 진로도 매우 밝다고 할 수 있습니다.

## 📌 핵심 정리

이 문서에서 배운 핵심 개념들을 정리합니다.

- **MDA(Model Driven Architecture)** 는 모델을 기반으로 플랫폼과 독립적으로 소프트웨어를 자동 생성하는 기술이다.
- **기존 미들웨어의 한계** 는 각자의 표준으로 만들어져 미들웨어 간 호환성(상호 운용성)이 부족하다는 점이다.
- **CORBA** 는 개방형 객체 표준이었으나 너무 무겁고 복잡해서 현대 개발 환경에서 외면받고 있다.
- **MDA의 핵심 원리** 는 표준 메타모델을 기반으로 플랫폼 독립적 모델을 설계하고, 나중에 매핑을 통해 종속적 코드로 변환하는 것이다.
- **MDD 프로세스** 는 CIM(비즈니스 분석) → PIM(플랫폼 독립 설계) → PSM(플랫폼 종속 설계) → 소스 코드 순으로 진행된다.
- **MOF, UML, CWM, XMI** 는 MDA 메타모델 구조를 이루는 핵심 구성 요소들이다.
- **SOA(서비스 지향 아키텍처)** 는 업무를 서비스 단위로 분리·표준화하여 변화에 빠르게 대응하는 개발 방식이다.
- **SOA는 세 단계** 로 발전한다: 펀더멘탈 SOA → 네트워크 SOA(ESB 활용) → 프로세스 중심 SOA.
- **컴포넌트(Component)** 는 독립적인 기능을 갖는 소프트웨어 부품이며, 반드시 인터페이스를 통해서만 접근한다.
- **CBD(컴포넌트 기반 개발)** 는 기존 컴포넌트를 조립해 시스템을 구축하는 방식으로 재사용성과 생산성이 높다.
- **AOP(관점 지향 프로그래밍)** 는 핵심 관심과 횡단 관심을 분리하고, 위빙으로 결합하여 코드 중복을 제거한다.
- **보안 공학** 은 기술뿐 아니라 물리, 심리, 경제 등 복합 학문으로 모든 분야에서 수요가 증가하고 있다.
- **최신 트렌드** 는 AI 기반 개발, CI/CD, 컨테이너화, 마이크로서비스, 보안 강화, 서버리스, 사용자 경험 중심으로 빠르게 변화하고 있다.
- **개발자의 핵심 자세** 는 AI와 경쟁하는 것이 아니라, AI를 도구로 활용하며 항상 사용자 중심으로 생각하는 것이다.
