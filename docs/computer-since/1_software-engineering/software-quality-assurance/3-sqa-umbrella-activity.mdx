# 🛡️ 소프트웨어 품질 보증(SQA)과 보호 활동

소프트웨어를 개발한다는 것, 단순히 코드를 잘 짜면 끝일까요? 실제 현장에서 일해보면 그렇지 않다는 것을 금방 깨닫게 됩니다. "내가 만든 이 소프트웨어가 정말 좋은 품질인가?"라는 질문에 당당하게 대답하려면, **소프트웨어 품질 보증(SQA, Software Quality Assurance)** 이라는 개념을 반드시 이해해야 합니다. 이번 문서에서는 초보자도 쉽게 이해할 수 있도록, 품질이란 무엇인지부터 시작해서 국제 표준까지 차근차근 살펴보겠습니다.

---

## 🤔 소프트웨어 품질이란 무엇인가?

### 눈에 보이는 제품 vs. 눈에 보이지 않는 소프트웨어

우리 주변에는 품질을 쉽게 판단할 수 있는 제품들이 많습니다. 자동차, 신발, 가방, 스마트폰처럼 **눈에 보이고 손으로 만질 수 있는 제품**은 그 품질을 어렵지 않게 측정할 수 있습니다. 자동차라면 주행 성능, 연비, 내구성, 디자인 등을 기준으로 "이 차는 품질이 좋다"고 말할 수 있죠.

그런데 **소프트웨어는 다릅니다.** 눈에 보이지 않고, 손으로 만질 수도 없습니다. 그렇다면 "이 소프트웨어는 품질이 좋다"는 것을 어떻게 증명할 수 있을까요? 어떻게 보증할 수 있을까요?

자동차는 "3년 또는 6만 킬로미터 이내에 고장이 나면 무상 수리해 드립니다"처럼 명확한 보증 기준이 있습니다. 그런데 소프트웨어에는 그런 킬로미터 단위 같은 측정 기준이 없습니다. 사용자가 값을 잘못 입력했거나, 실수로 잘못 클릭한 경우는 사용자 부주의이므로 소프트웨어의 오류가 아닙니다. 그럼 어디서부터 어디까지가 오류이고, 얼마 동안 보증해 줄 것인가? 이 질문에 답하기 위해 **소프트웨어 품질 보증**이라는 개념이 필요합니다.

### 현실에서 벌어지는 개발사와 클라이언트의 충돌

실제 개발 현장에서 자주 발생하는 상황을 한번 살펴봅시다. 한 개발사가 클라이언트(고객사)로부터 3억 원짜리 프로젝트를 수주해서 기능 A, B, C를 개발해 납품했습니다. 이후 유지보수 계약 기간(보통 1년)이 시작됩니다.

- 기능 A, B, C에서 오류가 발생하면 → **당연히 유지보수 대상** ✅
- 클라이언트가 "기능 A를 조금 바꾼 A1을 만들어 달라"고 요청하면 → **신규 개발? 유지보수?** ❓

여기서 갈등이 시작됩니다. 개발사는 "A1은 처음 계약에 없던 기능이니 신규 개발 비용을 추가로 주셔야 합니다"라고 하고, 클라이언트는 "A랑 비슷한 건데 왜 돈을 더 내야 하냐"고 맞섭니다.

> 💬 "화장실 들어갈 때와 나올 때 얘기가 다르다"는 속담처럼, 개발 수주할 때는 "다 해드릴게요!"라고 했던 개발사가 개발 완료 후에는 "이건 유지보수가 아니라 신규 개발입니다"라고 말을 바꾸는 경우가 실제로 매우 많습니다.

이런 충돌을 방지하려면, **개발 과정 전체에서 품질을 어떻게 관리하고 어디까지 보증할 것인지**를 명확히 해두는 것이 중요합니다. 이것이 바로 소프트웨어 품질 보증(SQA)의 핵심 존재 이유입니다.

---

## ☂️ SQA란 무엇인가? — 보호 활동(Umbrella Activity)의 개념

### SQA의 정의

**SQA(Software Quality Assurance, 소프트웨어 품질 보증)** 는 소프트웨어를 개발하는 **과정 전체**에 적용되는 **보호 활동(Umbrella Activity)** 입니다.

여기서 **Umbrella(엄브렐러)** 는 영어로 '우산'을 뜻합니다. 비가 내릴 때 우산을 펼치면 비를 맞지 않듯이, 소프트웨어 개발 전 과정에 SQA라는 우산을 씌워서 품질 문제라는 비를 맞지 않도록 보호하는 활동이라는 의미입니다.

:::info SQA의 적용 범위
SQA는 소프트웨어 개발의 **전체 과정**에 걸쳐 적용됩니다:
- **요구사항 분석** 단계
- **설계** 단계
- **구현(코딩)** 단계
- **테스트** 단계
- **유지보수** 단계

즉, 개발이 끝난 결과물만 검토하는 것이 아니라, 처음 기획부터 마지막 유지보수까지 모든 과정을 포함합니다.
:::

### SQA는 결과물 검토가 아니다

많은 초보 개발자들이 이런 착각을 합니다. *"소프트웨어 품질 보증이란, 개발이 끝난 소프트웨어를 놓고 기능이 제대로 구현됐는지 확인하면 되는 거 아닌가요?"* 하지만 이는 **잘못된 이해**입니다.

SQA는 단순히 완성된 결과물을 평가하는 것이 주목적이 아닙니다. 오히려 **품질 개선의 노력을 통해 개발 경험, 작업 방식, 프로세스 전체를 지속적으로 개선해 나가는 하나의 과정**입니다.

| 잘못된 이해 ❌ | 올바른 이해 ✅ |
|---|---|
| 개발 완료 후 결과물만 검토 | 개발 전 과정에 걸쳐 지속적으로 관리 |
| 기능 구현 여부만 확인 | 요구사항, 설계, 코딩, 테스트, 유지보수 전체 포함 |
| 한 번의 검토로 끝 | 지속적인 개선 활동 |
| 개발자만의 책임 | 팀 전체의 운영 프로세스, 방법, 도구 활용 포함 |

다시 말해, SQA는 **"이 소프트웨어 개발 과정이 체계적으로 잘 이루어졌는가?"** 를 전체적으로 보증하는 활동입니다. 개발 과정, 작업 방식, 팀 운영, 프로세스, 방법론, 도구의 활용 등 **모든 영역**이 제대로 갖춰졌을 때 비로소 품질 보증이 이루어진다고 할 수 있습니다.

## 🔍 핵심 원리 — 확인(Validation) vs. 검증(Verification)

### 왜 이 두 개념을 구분해야 할까?

SQA를 이해하는 데 있어 가장 중요한 개념 중 하나가 **확인(Validation)** 과 **검증(Verification)** 의 차이입니다. 이 두 단어는 비슷해 보이지만, 의미와 맥락이 완전히 다릅니다. 실무에서도 혼용되어 큰 문제를 일으키는 경우가 많으므로, 반드시 명확히 구분해서 이해해야 합니다.

### 검증(Verification) — "제대로 만들었는가?"

**검증(Verification)** 은 개발자 또는 테스터가 **미리 세팅된 시험 환경**에서 소프트웨어가 **명세서(설계 문서)대로** 제대로 개발되었는지를 확인하는 활동입니다. 쉽게 말해, "우리가 설계한 대로 만들었나?"를 검토하는 것입니다.

예를 들어, 동영상 앱을 개발했다면 테스터는 다음과 같은 환경에서 테스트합니다:
- 5G 환경에서 동영상이 끊기지 않고 재생되는가?
- LTE 환경에서도 정상 재생되는가?
- 3G 환경에서도 성능 기준을 만족하는가?
- Wi-Fi 환경에서는 어떠한가?

이처럼 **사전에 계획된 시험 환경**에서 명세서의 요구사항을 하나씩 체크하는 것이 검증입니다.

### 확인(Validation) — "올바른 것을 만들었는가?"

**확인(Validation)** 은 분석가나 **실제 사용자**가 **실제 운영 환경**에서 소프트웨어가 정말로 사용자의 요구를 만족하는지 확인하는 활동입니다. "우리가 사용자가 원하는 것을 만들었나?"를 검토하는 것이죠.

:::warning 검증과 확인의 차이를 헷갈리면 안 됩니다
검증(Verification)은 **통제된 시험 환경**에서 명세서 기준으로 체크하고, 확인(Validation)은 **실제 운영 환경**에서 사용자 요구 기준으로 체크합니다. 이 둘은 환경도 다르고, 평가 기준도 다릅니다.
:::

### 왜 이 차이가 중요한가? — 지리산 꼭대기 사례

테스터가 5G, LTE, 3G, Wi-Fi 환경 모두에서 테스트했고, 아무런 문제가 없었습니다. 검증은 통과한 겁니다. 그런데 사용자가 지리산 꼭대기에서 앱을 사용하다가 인터넷이 끊겼다 연결됐다를 반복하는 환경에서 앱이 제대로 동작하지 않아 클레임을 제기했다면? 이것은 검증 환경에서는 발견하지 못했던 문제입니다.

사용자 환경은 정말 다양합니다:
- KTX보다 빠른 속도로 이동하면서 사용
- 수영장이나 물 속 근처에서 사용
- 사우나처럼 온도가 높은 환경에서 사용
- 섬이나 지하실처럼 신호가 약한 곳에서 사용

**모든 환경을 다 테스트할 수 없다**는 것이 소프트웨어 품질 보증의 현실적인 한계입니다. 그렇기 때문에 처음 **요구사항 분석 단계**에서 "어느 수준까지의 환경을 지원하겠다"는 범위를 명확히 정의하는 것이 매우 중요합니다.

| 구분 | 검증(Verification) | 확인(Validation) |
|---|---|---|
| **누가** | 개발자, 테스터 | 분석가, 실제 사용자 |
| **어디서** | 시험(테스트) 환경 | 실제 운영 환경 |
| **기준** | 명세서(설계 문서) | 사용자 요구사항 |
| **질문** | 제대로 만들었는가? | 올바른 것을 만들었는가? |
| **예시** | 3G에서 동영상 재생 테스트 | 실제 사용자가 지리산에서 앱 사용 |

## 👥 다양한 관점에서 본 소프트웨어 품질

### 왜 여러 관점을 고려해야 할까?

소프트웨어 품질을 이야기할 때 중요한 점은, **입장에 따라 품질의 기준이 완전히 달라진다**는 것입니다. 사용자, 개발자, 유지보수자, 발주자 모두가 서로 다른 기준으로 품질을 평가합니다. 한쪽의 관점만 보고 품질을 평가하면 실패할 가능성이 높습니다.

### 🙋 사용자 입장

사용자는 소프트웨어를 쓸 때 이런 것을 원합니다:
- **배우기 쉬워야 한다** — 복잡한 매뉴얼 없이도 직관적으로 사용 가능해야 합니다.
- **사용하기 편해야 한다** — 버튼 몇 번으로 원하는 기능이 동작해야 합니다.
- **기능이 다양했으면 좋겠다** — 표 그리기, 이미지 삽입, 다양한 글꼴 등 기능이 풍부하면 좋습니다.
- **성능이 좋아야 한다** — 파일이 커져도 느려지거나 멈추지 않아야 합니다.

그런데 사실 이 네 가지 요구 사항은 서로 충돌합니다. 기능이 많아지면 성능이 떨어질 수밖에 없고, 기능이 복잡해지면 배우기도 어려워집니다. 사용자는 이 모든 것을 동시에 원하지만, 이를 전부 만족시키는 것은 매우 어렵습니다.

### 👨‍💻 개발자 입장

개발자의 관점은 훨씬 단순합니다. "프로그램이 제대로 실행되고, 입력이 정상적으로 처리되고, 출력이 올바르게 나오면 끝!" 이것이 개발자의 기준입니다. 사용자 요구와의 **갭(차이)** 이 여기서 발생합니다.

> 🎯 성공한 소프트웨어의 비결은 이 갭을 얼마나 좁히느냐에 있습니다. 여러분이 매일 쓰는 인기 앱이나 게임들이 성공한 이유는 바로 이 갭이 작기 때문입니다.

### 🔧 유지보수자 입장

유지보수자는 이미 만들어진 소프트웨어를 관리하고 수정하는 사람입니다. 이들에게 중요한 것은:
- **오류가 적어야 한다** — 오류가 적을수록 유지보수할 일이 줄어듭니다.
- **문서화가 잘 되어 있어야 한다** — 오류가 발생했을 때 문서를 보고 빠르게 원인을 찾을 수 있어야 합니다.
- **코드가 이해하기 쉬워야 한다** — 10줄로 짤 수 있는 코드를 200줄로 복잡하게 짜놓으면 유지보수자가 고생합니다.

:::danger 나쁜 코드 작성은 팀 전체에 피해를 줍니다
코드를 일부러 난해하게 작성하거나, 문서화를 소홀히 하면 나중에 유지보수하는 사람이 엄청난 고생을 하게 됩니다. 이는 단순히 동료에게 피해를 주는 것을 넘어, 회사의 신뢰도와 재계약에까지 영향을 미칩니다.
:::

### 💰 발주자(클라이언트) 입장

발주자에게 가장 중요한 것은 딱 하나입니다. **비용이 적게 들어야 한다.** 3억짜리 프로젝트를 발주했는데, 개발 이후 유지보수 비용으로 5억이 추가로 들어간다면 발주자 입장에서는 완전한 실패입니다.

| 이해관계자 | 가장 중요하게 생각하는 품질 요소 |
|---|---|
| **사용자** | 사용 편의성, 기능 다양성, 성능 |
| **개발자** | 정상 실행, 정확한 입출력 처리 |
| **유지보수자** | 낮은 오류율, 문서화, 코드 가독성 |
| **발주자** | 비용 절감, 일정 준수 |

## 💰 품질 보증 비용의 구조

### 품질 보증에도 비용이 든다

자동차 리콜 뉴스를 본 적 있으신가요? 전기차 배터리 결함으로 전량 리콜을 하면 몇 조 원의 비용이 든다는 뉴스가 나오죠. 소프트웨어도 마찬가지입니다. **품질 보증은 비용과 직결**됩니다.

실제로 개발 비용보다 유지보수 비용이 더 많이 들어가는 경우가 현장에서 자주 발생합니다. 3억짜리 프로젝트를 잘못 개발하면 유지보수에 5억이 들어가기도 합니다. 품질 보증 비용은 크게 세 가지로 나눌 수 있습니다.

### 1️⃣ 예방 비용(Prevention Cost)

**결함이 발생하지 않도록 미리 조치하는 데 드는 비용**입니다. 마치 건강검진을 미리 받아서 큰 병을 예방하는 것과 같습니다.

- 개발자 교육 및 훈련
- 요구사항 분석 및 기획 강화
- 계약자 역량 사전 조사
- 개발 프로세스 표준화

### 2️⃣ 평가 비용(Appraisal Cost)

**소프트웨어가 고객의 요구사항을 올바르게 구현하고 있는지 평가하는 데 드는 비용**입니다.

- 코드 리뷰 및 검토
- 테스트 계획 수립 및 수행
- 품질 감사(Quality Audit)
- 사용자 수용 테스트(UAT)

### 3️⃣ 장애 비용(Failure Cost)

**실제로 결함이 발생했을 때 수정하고 시정조치 하는 데 드는 비용**입니다. 이것이 세 가지 중 가장 많은 비용이 들어갑니다. 자동차 전량 리콜처럼, 소프트웨어를 대규모로 수정해야 하는 상황이 생기면 비용이 폭발적으로 커집니다.

:::tip 예방이 최선입니다
예방 비용과 평가 비용을 아끼면 장애 비용이 폭발적으로 늘어납니다. 반대로 예방과 평가에 충분히 투자하면 장애 비용을 크게 절약할 수 있습니다. "나중에 고치면 되지"라는 생각이 결국 더 큰 비용을 만들어냅니다.
:::

## 📏 맥콜(McCall)의 소프트웨어 품질 특성

### 소프트웨어 품질을 어떤 기준으로 평가할까?

맥콜(McCall)이라는 학자는 소프트웨어 품질을 체계적으로 구분하여 평가 기준을 제시했습니다. 이를 세 가지 관점(**제품 운영**, **제품 개조**, **제품 전이**)으로 나누어 살펴보겠습니다.

### 제품 운영(Product Operation) — 실제 사용 시 품질

**정확성(Correctness)**: 소프트웨어가 사용자의 요구사항을 정확하게 충족시키는지에 관한 특성입니다. 저장 버튼을 눌렀을 때 항상 저장이 되어야 하고, 10번 중 9번만 되고 1번은 안 된다면 정확성이 낮은 것입니다.

**신뢰성(Reliability)**: 정해진 조건에서 요구되는 기능을 정확하게 수행할 수 있는 능력입니다. 오랫동안 사용해도 안정적으로 동작해야 합니다.

**사용 용이성(Usability)**: 쉽게 배우고 쉽게 사용할 수 있어야 합니다. 처음 접하는 사용자도 짧은 시간 안에 기본 기능을 익힐 수 있어야 합니다.

**무결성(Integrity)**: 권한이 없는 사용자가 데이터를 변경하거나 시스템을 제어할 수 없어야 합니다. 예를 들어, 결제를 하지 않은 사용자가 유료 콘텐츠를 볼 수 있다면 무결성이 깨진 것입니다.

**효율성(Efficiency)**: 최소한의 시간, 메모리, CPU 자원을 사용하면서 서비스가 제공되어야 합니다. 게임을 설치했더니 컴퓨터 전체가 느려진다면 효율성이 낮은 것입니다.

### 제품 개조(Product Revision) — 변경 시 품질

**유지보수성(Maintainability)**: 오류를 발견하고 수정하는 것이 용이해야 합니다. 문서화가 잘 되어 있고, 코드가 깔끔하게 작성되어 있으면 유지보수성이 높습니다.

**유연성(Flexibility)**: 기능을 추가하거나 환경이 변경되었을 때 쉽게 수정할 수 있어야 합니다. 변경 사항이 생겼을 때 코드 전체를 뜯어고쳐야 한다면 유연성이 낮은 것입니다.

**시험성(Testability)**: 언제 어디서든 쉽게 테스트할 수 있어야 합니다. 테스트 환경을 구성하기 복잡하거나 특정 조건에서만 테스트 가능하다면 시험성이 낮습니다.

### 제품 전이(Product Transition) — 이식 및 확장 시 품질

**이식성(Portability)**: 리눅스, 윈도우, macOS 등 다양한 운영체제 환경에서도 동작할 수 있어야 합니다. 윈도우에서만 동작하는 소프트웨어는 이식성이 낮습니다.

**재사용성(Reusability)**: 소프트웨어의 전체 또는 일부 모듈을 다른 응용 프로그램에서도 재사용할 수 있어야 합니다. 잘 설계된 모듈은 다른 프로젝트에 그대로 가져다 쓸 수 있습니다.

**상호 운용성(Interoperability)**: 다른 소프트웨어와 정보를 서로 주고받을 수 있어야 합니다. 아래아한글에서 PDF로 저장하거나, 엑셀 파일을 이미지로 내보내는 기능 등이 그 예입니다.

## 📐 소프트웨어 품질 평가 원칙

### 주먹구구식이 아닌 체계적인 기준이 필요하다

소프트웨어 품질을 평가할 때는 명확한 원칙이 있어야 합니다. "느낌상 좋은 것 같다"는 주관적 평가는 전혀 의미가 없습니다. 표준화된 평가 원칙은 다음과 같습니다.

### 반복성(Repeatability)

**동일한 제품에 대해 동일한 평가자가 동일한 기준으로 평가하면 항상 같은 결과가 나와야 합니다.** 아래아한글에서 저장 버튼을 누르면 항상 저장이 되어야 하고, 프린트 버튼을 누르면 항상 인쇄 팝업창이 떠야 합니다. 10번 중 1번이라도 다른 결과가 나온다면 반복성이 낮은 것입니다.

### 재생 가능성(Reproducibility)

**동일한 제품에 대해 다른 사용자가 동일한 방법으로 평가해도 같은 결과가 나와야 합니다.** 반복성과 차이는 딱 하나, **평가자가 같은 사람이냐 다른 사람이냐**입니다.

- **반복성**: 같은 사람이 여러 번 해도 같은 결과
- **재생 가능성**: 다른 사람이 같은 방법으로 해도 같은 결과

내가 아래아한글에서 저장하는 것과, 옆 사람이 아래아한글에서 저장하는 것의 결과가 동일해야 한다는 것입니다.

### 공평성(Fairness)

**평가는 특정 결과를 유도하기 위한 불공정한 편향(Bias) 없이 이루어져야 합니다.** 어떤 팀의 소프트웨어는 5G 환경에서 테스트하고, 다른 팀의 소프트웨어는 3G 환경에서 테스트한다면 공평하지 않습니다. 모든 평가 대상에 동일한 조건을 적용해야 합니다.

### 객관성(Objectivity)

**평가 결과는 주관적인 판단을 최소화하고 객관적이어야 합니다.** 평가자가 특정 업체와 친하다고 해서 쉽게 통과시켜 주거나, 자주 써본 솔루션이라서 눈감아 주는 행위는 절대 금지입니다. 항상 정해진 기준에 따라 공정하게 평가해야 합니다.

## 🌐 국제 품질 표준(ISO) 체계

### ISO란 무엇인가?

**ISO(International Organization for Standardization, 국제표준화기구)** 는 전 세계적으로 공통된 표준 규격을 정의하고 배포하는 국제 기구입니다. 1942년 6월에 정식 발족했으며, 우리나라의 KS(코리아 스탠다드)처럼 각 나라에 국가 표준이 있듯이, ISO는 전 세계가 공통으로 사용하는 국제 표준입니다.

우리가 어디서든 동일한 전원 코드 규격을 사용할 수 있는 것, 다른 나라에서 만든 볼트와 너트가 규격에 맞아 함께 사용할 수 있는 것 모두 ISO 표준 덕분입니다.

:::info ISO의 주요 활동
- 국제 표준 제정 및 배포
- 관련 기관 및 위원회 간 업무 정보 교환
- 국제 협력 증진 및 조정
:::

### ISO 9000 — 품질 관리 시스템의 기초

**ISO 9000**은 ISO가 1987년에 제정한 **품질 관리 및 품질 보증에 관한 기본 규격**입니다. 여기서 중요한 점은, ISO 9000이 **제품 자체가 좋다는 것을 인증하는 것이 아니라**, 그 제품을 만들어 내는 **기업의 품질 보증 체계와 프로세스**가 잘 갖춰져 있음을 인증한다는 것입니다.

쉽게 말하면, "이 회사는 제품을 만들 때 인력 관리, 공정 관리, 환경 관리 등을 체계적으로 하고 있다"는 것을 인증하는 겁니다. 모든 비즈니스 분야에 적용할 수 있는 일반적인 품질 관리 기준입니다.

**ISO 9001**은 ISO 9000 시리즈 중 하나로, 고객에게 제공되는 제품이나 서비스가 제대로 유지 관리되고 있음을 인증하는 제도입니다. ISO 9001 인증을 받기 위해서는 다음 원칙들을 만족해야 합니다:

- 고객 중심(Customer Focus)
- 리더십(Leadership)
- 전원 참여(Involvement of People)
- 프로세스 접근 방법(Process Approach)
- 지속적 개선(Continual Improvement)
- 의사결정에 대한 사실적 접근 방법(Factual Approach to Decision Making)
- 상호 유익한 공급자 관계(Mutually Beneficial Supplier Relationships)

### ISO/IEC 9126 — 소프트웨어 제품 품질 표준

**ISO/IEC 9126**은 **소프트웨어 제품의 품질을 정량적으로 평가하기 위한 표준**입니다. ISO 9000이 모든 제품에 일반적으로 적용되는 기준이라면, 9126은 소프트웨어라는 특수한 분야를 위해 별도로 만들어진 표준입니다.

소프트웨어 품질을 **객관적이고 정량적으로 평가**할 수 있는 기본 틀을 제공하며, 다음 4개의 세부 표준으로 구성됩니다:

| 번호 | 내용 |
|---|---|
| **ISO/IEC 9126-1** | 소프트웨어 품질의 주 특성(기능성, 신뢰성, 사용성, 효율성, 유지보수성, 이식성)과 서브 특성 정의 |
| **ISO/IEC 9126-2** | 외부 품질 측정 기준 (운영 및 테스트 시 측정 방법) |
| **ISO/IEC 9126-3** | 내부 품질 측정 기준 (설계 및 코딩 단계의 특성 측정) |
| **ISO/IEC 9126-4** | 사용 품질 측정 기준 (최종 사용 결과 품질 측정) |

### ISO/IEC 12119 — 요구사항과 테스팅 표준

**ISO/IEC 12119**는 소프트웨어의 **요구사항 정의와 테스트 절차**에 관한 표준입니다. 이 인증을 받았다는 것은 "우리는 이 규정에 따라 요구사항을 수집했고, 이 절차에 따라 테스트를 수행했다"는 것을 공식적으로 인정받은 것입니다.

### ISO/IEC 14598 — 소프트웨어 평가 절차 표준

**ISO/IEC 14598**은 소프트웨어 품질을 **평가하고 관리하는 방법과 절차**에 관한 표준입니다. 이 인증은 "우리는 이 기준에 부합하게 소프트웨어를 평가하고 관리하고 있다"는 것을 인증받은 것입니다.

### ISO/IEC 25000 — 통합 품질 평가 모델

**ISO/IEC 25000(SQuaRE)** 은 앞서 설명한 9126, 14598, 12119 등 각각 흩어져 있던 소프트웨어 품질 관련 표준들을 **하나의 통합된 틀로 묶어서 관리**하는 표준입니다.

ISO/IEC 25000은 다음 다섯 가지 구성 요소로 이루어져 있습니다:
1. **품질 관리(Quality Management)** — 전체 품질 관리 계획 및 조정
2. **품질 모형(Quality Model)** — 품질 특성 정의 (기존 9126-1 기반)
3. **품질 측정(Quality Measurement)** — 측정 방법 및 기준 (기존 9126-2, 3, 4 기반)
4. **품질 요구사항(Quality Requirements)** — 요구사항 명세 방법 (기존 12119 기반)
5. **품질 평가(Quality Evaluation)** — 평가 절차 및 방법 (기존 14598 기반)

:::note ISO 표준은 외울 필요 없습니다
이 ISO 번호들을 전부 외울 필요는 없습니다. "아, 소프트웨어 품질을 이렇게 다양한 국제 기준으로 체계적으로 평가하고 관리하는구나"는 정도로 이해하시면 충분합니다. 실무에서는 해당 표준의 상세 매뉴얼을 참조하면서 작업하게 됩니다.
:::

## ⚠️ 주의사항 및 실무 팁

### 개발자가 놓치기 쉬운 품질 관련 실수들

```python
# ❌ 잘못된 예시: 오류 처리 없이 바로 결과만 반환
def divide(a, b):
    return a / b  # b가 0이면 ZeroDivisionError 발생 — 품질(신뢰성) 저하

# ✅ 올바른 예시: 다양한 예외 상황을 고려한 방어적 코딩
def divide(a, b):
    # 입력값 유효성 검사 (무결성, 신뢰성 향상)
    if not isinstance(a, (int, float)) or not isinstance(b, (int, float)):
        raise TypeError("숫자 타입만 입력 가능합니다.")  # 사용자 오입력 대비
    
    if b == 0:
        raise ValueError("0으로 나눌 수 없습니다.")  # 0 나눗셈 방지
    
    return a / b  # 정상적인 나눗셈 결과 반환
```

위 코드 예시를 통해 알 수 있듯이, 단순히 기능이 동작하는 코드를 짜는 것과 **품질을 고려한 코드**를 짜는 것은 전혀 다릅니다. 다양한 예외 상황을 고려한 방어적 코딩이 소프트웨어의 신뢰성과 무결성을 높입니다.

```python
# ❌ 잘못된 예시: 문서화 없는 복잡한 코드 (유지보수성 저하)
def p(d, f, t):
    return [x for x in d if x[f] == t]

# ✅ 올바른 예시: 명확한 함수명과 주석, 타입 힌트 포함 (유지보수성 향상)
def filter_data_by_field(data: list, field_name: str, target_value) -> list:
    """
    주어진 데이터 목록에서 특정 필드가 목표 값과 일치하는 항목만 필터링합니다.
    
    Args:
        data (list): 딕셔너리로 구성된 데이터 목록
        field_name (str): 필터링 기준이 되는 필드 이름
        target_value: 일치 여부를 확인할 목표 값
    
    Returns:
        list: 조건에 맞는 항목들만 포함된 새 목록
    """
    # 각 항목을 순회하며 필드 값이 목표 값과 일치하는 항목만 반환
    return [item for item in data if item[field_name] == target_value]
```

이처럼 함수명을 명확하게 짓고, 주석과 문서화(docstring)를 잘 작성하는 것이 유지보수자를 위한 배려이며 품질의 일부입니다.

### 요구사항 정의의 중요성

# ❌ 잘못된 예시: 요구사항 범위 불명확 (나중에 분쟁 발생 가능)

#           단, 3G 이하 환경 및 신호 불안정 지역에서의 동작은 보증 범위에서 제외한다."

:::

### 품질 보증은 개발자 혼자의 책임이 아니다

품질 보증은 개발자 한 명이 혼자서 담당하는 것이 아닙니다. 기획자, 디자이너, 개발자, 테스터, 운영자, 관리자 등 **프로젝트에 참여하는 모든 사람이 함께 책임지는 활동**입니다. 각자의 역할에서 품질을 의식하고 개선하려는 노력이 모여야 비로소 좋은 품질의 소프트웨어가 탄생합니다.

## 📌 핵심 정리

이번 문서에서 배운 소프트웨어 품질 보증(SQA)의 핵심 개념들을 정리합니다.

- **소프트웨어 품질**이란 상품이나 서비스가 그 목적을 만족시키는 고유의 성질이나 성능을 의미한다.
- **SQA(소프트웨어 품질 보증)** 는 소프트웨어 개발 과정 전체에 적용되는 **보호 활동(Umbrella Activity)** 이다.
- SQA는 결과물만 검토하는 것이 아니라, **개발 과정, 작업 방식, 프로세스, 도구 활용 전체**를 포함하는 지속적 개선 활동이다.
- **검증(Verification)** 은 시험 환경에서 명세서 기준으로 "제대로 만들었는가?"를 확인하고, **확인(Validation)** 은 실제 운영 환경에서 사용자 기준으로 "올바른 것을 만들었는가?"를 확인한다.
- 소프트웨어 품질은 **사용자, 개발자, 유지보수자, 발주자** 관점에서 각각 다르게 평가되므로, 모든 관점을 고려해야 한다.
- 품질 보증 비용은 **예방 비용, 평가 비용, 장애 비용** 세 가지로 구성되며, 예방과 평가에 투자할수록 장애 비용을 줄일 수 있다.
- **맥콜(McCall)** 은 소프트웨어 품질을 제품 운영(정확성, 신뢰성, 사용 용이성, 무결성, 효율성), 제품 개조(유지보수성, 유연성, 시험성), 제품 전이(이식성, 재사용성, 상호 운용성)로 구분하였다.
- 품질 평가 원칙으로는 **반복성(같은 평가자가 항상 같은 결과), 재생 가능성(다른 평가자도 같은 결과), 공평성(편향 없는 평가), 객관성(주관 배제)** 이 있다.
- **ISO 9000** 은 제품이 아닌 기업의 품질 관리 체계를, **ISO/IEC 9126** 은 소프트웨어 제품 품질을, **ISO/IEC 14598** 은 평가 절차를, **ISO/IEC 12119** 는 요구사항과 테스팅 절차를 표준화한 것이다.
- **ISO/IEC 25000** 은 기존의 분산된 소프트웨어 품질 표준들을 하나의 통합 모델로 묶은 것으로, 품질 관리·모형·측정·요구사항·평가 5개 요소로 구성된다.
- 요구사항 명세를 **구체적이고 명확하게** 정의하는 것이 이후 발생할 수 있는 분쟁과 추가 비용을 막는 가장 효과적인 방법이다.
