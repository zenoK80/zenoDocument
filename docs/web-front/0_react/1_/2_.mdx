## 두 가지 선택지

React 앱을 만드는 방법은 크게 **두 가지**입니다:

```
1. 프레임워크 사용 (추천! ⭐⭐⭐⭐⭐)
   └─ 이미 만들어진 틀 사용
   └─ 빠르고 쉬움
   └─ 초보자 추천

2. 처음부터 직접 만들기 (고급)
   └─ 모든 것을 직접 설정
   └─ 어렵지만 유연함
   └─ 학습 목적이나 특수한 경우
```

**비유로 이해하기:**

```
집을 짓는다면?

프레임워크 사용:
🏠 조립식 주택 키트 구매
   - 설계도 포함
   - 필요한 자재 모두 포함
   - 조립만 하면 됨
   - 시간: 1개월
   - 난이도: 보통

처음부터 만들기:
🧱 벽돌과 시멘트 구매
   - 직접 설계
   - 자재 하나하나 선택
   - 전부 직접 시공
   - 시간: 6개월
   - 난이도: 매우 어려움
```

## 1부: 풀스택 프레임워크 (추천!)

### 풀스택이 뭔가요?

**풀스택 (Full-stack)** = 전체 (앞 + 뒤)

```
웹 애플리케이션의 구조:

프론트엔드 (Front-end) - 사용자가 보는 화면
├─ HTML (구조)
├─ CSS (디자인)
└─ JavaScript (동작)

백엔드 (Back-end) - 서버, 데이터베이스
├─ 데이터 저장
├─ 로그인 처리
└─ API 제공

풀스택 = 프론트엔드 + 백엔드 모두!
```

**실생활 비유:**

```
레스토랑:

프론트엔드 = 홀 (손님이 있는 곳)
└─ 메뉴판
└─ 테이블
└─ 웨이터

백엔드 = 주방 (보이지 않는 곳)
└─ 요리사
└─ 재료 보관
└─ 요리 과정

풀스택 = 홀 + 주방 전체!
```

### 왜 프레임워크를 써야 하나요?

**순수 React만 사용하면 생기는 문제:**

```
문제 1: 페이지 이동이 없어요!
- React만으로는 /about, /contact 같은 URL 만들기 어려움
- 직접 라우팅 시스템을 만들어야 함
→ 해결: 프레임워크가 라우팅 제공!

문제 2: 서버에서 데이터 어떻게 가져와요?
- API 호출 코드를 직접 작성
- 로딩 상태, 에러 처리 직접 구현
→ 해결: 프레임워크가 데이터 페칭 도구 제공!

문제 3: 코드를 어떻게 배포해요?
- 번들링, 최적화 직접 설정
- 서버 설정 직접
→ 해결: 프레임워크가 빌드 시스템 포함!

문제 4: 검색엔진에 안 잡혀요! (SEO)
- React는 JavaScript로 렌더링
- 구글 봇이 빈 페이지만 봄
→ 해결: 프레임워크가 SSR 지원!
```

### 프레임워크가 제공하는 것들

| 기능 | 설명 | 직접 구현 난이도 |
|------|------|-----------------|
| **라우팅** | URL 관리 (/home, /about) | ⭐⭐⭐ |
| **데이터 페칭** | 서버에서 데이터 가져오기 | ⭐⭐⭐⭐ |
| **SSR/SSG** | 서버 렌더링, 정적 생성 | ⭐⭐⭐⭐⭐ |
| **빌드 최적화** | 코드 압축, 최적화 | ⭐⭐⭐⭐⭐ |
| **개발 서버** | 핫 리로딩 등 | ⭐⭐⭐ |
| **배포** | 프로덕션 빌드 | ⭐⭐⭐⭐ |

:::tip 초보자에게 전하는 조언
프레임워크를 사용하세요! 

처음부터 만들기는:
- ❌ 시간 낭비 (3개월 걸림)
- ❌ 버그 많음
- ❌ 유지보수 어려움

프레임워크 사용은:
- ✅ 빠른 시작 (하루면 충분)
- ✅ 검증된 코드
- ✅ 커뮤니티 지원
:::

## 중요한 오해 바로잡기

### "풀스택 = 서버 필요"는 잘못된 생각!

```
❌ 잘못된 생각:
"풀스택 프레임워크를 쓰면 서버를 꼭 운영해야 한다"

✅ 진실:
"풀스택 프레임워크는 선택권을 준다!"
```

**풀스택 프레임워크로 할 수 있는 것들:**

```
1. CSR (Client-Side Rendering)
   - 서버 없이 CDN만으로 배포
   - 일반 React 앱처럼 작동
   - 예: Vercel, Netlify 무료 호스팅

2. SSG (Static Site Generation)
   - 빌드할 때 HTML 미리 생성
   - 서버 없이 정적 파일 호스팅
   - 예: GitHub Pages

3. SSR (Server-Side Rendering)
   - 필요한 페이지만 서버 렌더링
   - 나머지는 클라이언트 렌더링
   - 선택적으로 사용!

결론: 서버 없어도 완벽하게 작동!
```

**실제 시나리오:**

```
시작 단계 (서버 없음):
- CSR로 앱 개발
- Vercel 무료 호스팅
- 비용: $0

성장 단계 (일부 서버 기능):
- 홈페이지만 SSR 적용 (SEO)
- 나머지는 여전히 CSR
- 비용: $0~20/월

대규모 단계 (풀 서버):
- 모든 페이지 SSR
- 서버 API 추가
- 비용: 사용량에 따라
```

### 라우트별로 렌더링 방식을 선택할 수 있어요!

**"라우트 (Route)"가 뭔가요?**

```
라우트 = 경로 = URL 주소

예시:
https://mywebsite.com/         ← 홈 라우트
https://mywebsite.com/about    ← About 라우트
https://mywebsite.com/blog     ← Blog 라우트
https://mywebsite.com/contact  ← Contact 라우트

각 라우트마다 다른 페이지가 보임
```

**라우트별로 다른 렌더링 방식 적용:**

```jsx
// Next.js 예시
my-website/
├── app/
│   ├── page.tsx           → / (홈)
│   │   └─ SSR 사용 (SEO 중요!)
│   │
│   ├── about/
│   │   └── page.tsx       → /about
│   │       └─ SSG 사용 (내용 고정)
│   │
│   ├── blog/
│   │   └── page.tsx       → /blog
│   │       └─ SSR 사용 (최신 글 표시)
│   │
│   └── dashboard/
│       └── page.tsx       → /dashboard
│           └─ CSR 사용 (로그인 필요, SEO 불필요)
```

**왜 이렇게 섞어 쓰나요?**

```
페이지 특성에 따라 최적의 방식 선택:

홈페이지 (/):
- SEO 중요 → SSR
- 실시간 데이터 표시 → SSR

회사 소개 (/about):
- 내용 거의 안 바뀜 → SSG
- 빌드 시 생성 → 빠름

대시보드 (/dashboard):
- 로그인 필요 → CSR
- SEO 불필요 → CSR
- 사용자마다 다른 데이터 → CSR
```

**코드 예시 (Next.js):**

```jsx
// app/page.tsx (홈 - SSR)
export default async function Home() {
  // 서버에서 실행됨!
  const data = await fetch('https://api.example.com/latest');
  const posts = await data.json();
  
  return (
    <div>
      <h1>최신 글</h1>
      {posts.map(post => (
        <article key={post.id}>{post.title}</article>
      ))}
    </div>
  );
}

// app/about/page.tsx (About - SSG)
export default function About() {
  // 빌드 시 한 번만 실행됨!
  return (
    <div>
      <h1>회사 소개</h1>
      <p>우리는 2020년에 설립되었습니다...</p>
    </div>
  );
}

// app/dashboard/page.tsx (대시보드 - CSR)
'use client';  // ← 클라이언트 컴포넌트 표시!

import { useState, useEffect } from 'react';

export default function Dashboard() {
  const [userData, setUserData] = useState(null);
  
  useEffect(() => {
    // 브라우저에서 실행됨!
    fetchUserData().then(setUserData);
  }, []);
  
  return (
    <div>
      <h1>{userData?.name}님의 대시보드</h1>
    </div>
  );
}
```

**마이그레이션 시나리오:**

```
1단계: 전체 CSR로 시작
my-website/
└── 모든 페이지 CSR
    └─ 개발 빠름
    └─ SEO 나쁨

2단계: 중요한 페이지만 SSR
my-website/
├── 홈 → SSR로 변경 (SEO)
├── About → SSG로 변경 (빠름)
└── 나머지 → 여전히 CSR

3단계: 필요한 페이지 더 추가
my-website/
├── 홈 → SSR
├── About → SSG
├── Blog → SSR로 변경
└── Dashboard → CSR

앱 전체를 다시 작성할 필요 없음!
```

:::tip 렌더링 방식 선택 가이드

**SSR을 사용하세요:**
- ✅ SEO가 중요한 페이지 (홈, 상품 페이지)
- ✅ 실시간 데이터 표시
- ✅ 개인화된 콘텐츠

**SSG를 사용하세요:**
- ✅ 내용이 거의 안 바뀜 (About, 문서)
- ✅ 모든 사용자에게 같은 내용
- ✅ 빠른 로딩 속도 필요

**CSR을 사용하세요:**
- ✅ SEO 불필요 (대시보드, 관리자 페이지)
- ✅ 로그인 필요
- ✅ 사용자별 다른 데이터

**혼합 사용 (추천!):**
- ✅ 각 페이지 특성에 맞게
- ✅ 필요할 때만 SSR
- ✅ 성능과 SEO 최적화
:::

## 추천 프레임워크 상세 가이드

### 1. Next.js (가장 추천! ⭐⭐⭐⭐⭐)

**Next.js가 뭔가요?**

```
Next.js = React를 위한 풀스택 프레임워크

만든 곳: Vercel (배포 플랫폼 회사)
첫 출시: 2016년
사용하는 곳: Netflix, Uber, TikTok, Twitch...
```

**Next.js의 "App Router"란?**

```
Next.js에는 두 가지 방식이 있었어요:

1. Pages Router (구버전)
   - 2016년부터 사용
   - pages/ 폴더 사용
   - 단순하지만 기능 제한

2. App Router (신버전, 추천!)
   - 2023년 정식 출시
   - app/ 폴더 사용
   - React 최신 기능 활용
   - 서버 컴포넌트, 스트리밍 등

→ 새 프로젝트는 App Router 사용!
```

#### 설치 방법

```bash
# Next.js 프로젝트 생성
npx create-next-app@latest

# 설치 중 질문들 (화살표로 선택, Enter로 확인):

? What is your project named? › my-app
  → 프로젝트 이름 입력

? Would you like to use TypeScript? › No / Yes
  → TypeScript 쓸까요? (초보자는 No)

? Would you like to use ESLint? › No / Yes
  → 코드 검사 도구 쓸까요? (Yes 추천)

? Would you like to use Tailwind CSS? › No / Yes
  → Tailwind CSS 쓸까요? (선택)

? Would you like your code inside a `src/` directory? › No / Yes
  → src/ 폴더 쓸까요? (Yes 추천)

? Would you like to use App Router? › No / Yes
  → App Router 쓸까요? (Yes!)

? Would you like to use Turbopack for `next dev`? › No / Yes
  → 빠른 개발 서버 쓸까요? (Yes 추천)

? Would you like to customize the import alias? › No / Yes
  → import 경로 커스터마이징? (No)

# 설치 완료!
```

**각 질문 상세 설명:**

```
1. Project name
   - 프로젝트 폴더 이름
   - 예: my-blog, my-shop, portfolio

2. TypeScript
   - Yes: 타입 안정성 (나중에 배우기)
   - No: 일반 JavaScript (초보자 추천)

3. ESLint
   - 코드 오류를 미리 찾아줌
   - Yes 강력 추천!

4. Tailwind CSS
   - CSS 프레임워크
   - Yes: 빠른 스타일링
   - No: 직접 CSS 작성

5. src/ directory
   - Yes: app/ 폴더가 src/ 안으로
   - 프로젝트 구조가 깔끔해짐

6. App Router
   - 무조건 Yes!
   - 최신 기능들

7. Turbopack
   - 개발 서버 속도 향상
   - Yes 추천

8. Import alias
   - @/ 같은 경로 단축키
   - 초보자는 No
```

#### 페이지 추가하기

**파일 기반 라우팅:**

```
Next.js는 폴더 구조 = URL 구조

app/
├── page.tsx              → /
├── about/
│   └── page.tsx         → /about
├── blog/
│   ├── page.tsx         → /blog
│   └── [id]/
│       └── page.tsx     → /blog/1, /blog/2, /blog/3...
└── contact/
    └── page.tsx         → /contact

폴더만 만들면 자동으로 라우트 생성!
```

**About 페이지 만들기:**

```bash
# 1. 폴더 생성
mkdir app/about

# 2. 파일 생성
# app/about/page.tsx
```

```jsx
// app/about/page.tsx
export default function AboutPage() {
  return (
    <main>
      <h1>회사 소개</h1>
      <p>우리는 2020년에 설립된 IT 기업입니다.</p>
    </main>
  );
}

// 저장하면 자동으로 /about 페이지 생성됨!
```

**동적 라우트 (Dynamic Route):**

```
동적 라우트 = 변하는 URL

예시:
/blog/1     ← 1번 글
/blog/2     ← 2번 글
/blog/99    ← 99번 글

1, 2, 99가 동적으로 변함!
```

```jsx
// app/blog/[id]/page.tsx
// ↑ [id]는 동적 파라미터

export default function BlogPost({ params }: { params: { id: string } }) {
  return (
    <main>
      <h1>{params.id}번 글</h1>
      <p>블로그 글 내용...</p>
    </main>
  );
}

// /blog/1 접속 → params.id = "1"
// /blog/2 접속 → params.id = "2"
```

#### 데이터 가져오기 (Data Fetching)

**서버 컴포넌트에서 데이터 가져오기:**

```jsx
// app/blog/page.tsx
async function getPosts() {
  // 서버에서 실행됨!
  const res = await fetch('https://api.example.com/posts');
  
  if (!res.ok) {
    throw new Error('Failed to fetch');
  }
  
  return res.json();
}

export default async function BlogPage() {
  // await 사용 가능!
  const posts = await getPosts();
  
  return (
    <main>
      <h1>블로그</h1>
      {posts.map((post) => (
        <article key={post.id}>
          <h2>{post.title}</h2>
          <p>{post.excerpt}</p>
        </article>
      ))}
    </main>
  );
}
```

**코드 설명:**

```
1. getPosts() 함수
   - 서버에서 실행됨
   - API 호출
   - 데이터 반환

2. BlogPage 컴포넌트
   - async function으로 선언
   - await로 데이터 기다림
   - 데이터 받은 후 렌더링

3. 장점
   - 로딩 상태 관리 불필요
   - useEffect 불필요
   - 서버에서 미리 렌더링 → 빠름!
```

#### Next.js의 장점 정리

```
✅ 장점:

1. 파일 기반 라우팅
   - 폴더만 만들면 페이지 생성
   - 라우터 코드 작성 불필요

   - 서버에서 데이터 가져오기
   - 번들 크기 감소

3. 이미지 최적화
   - next/image 컴포넌트
   - 자동 최적화, lazy loading

4. 폰트 최적화
   - next/font로 폰트 로드
   - 성능 향상

5. API 라우트
   - 백엔드 코드 같이 작성
   - app/api/ 폴더

6. 배포 쉬움
   - Vercel에 무료 배포
   - 1클릭 배포

7. 커뮤니티 큼

   - 튜토리얼 많음

#### Next.js를 사용해야 하는 경우

### 3. Expo (모바일 앱!)

**Expo가 뭔가요?**

```
Expo = React로 모바일 앱을 만드는 프레임워크

만들 수 있는 것:
- iOS 앱 (아이폰)
- Android 앱 (갤럭시 등)
- 웹 앱

**React Native와의 관계:**

```
React Native:
- React로 네이티브 앱 만드는 라이브러리
- Facebook이 만듦
- 복잡한 설정 필요

Expo:
- React Native를 쉽게 사용하게 해주는 도구
- 설정 거의 없음
- SDK, 도구 제공

비유:
React Native = 재료
Expo = 요리 키트
```

#### Expo의 장점

```

2. 실제 기기 테스트

3. 풍부한 SDK

5. 웹도 지원
   - 같은 코드로 웹 버전도 생성

## 2부: 처음부터 만들기 (From Scratch)

### 왜 처음부터 만들까요?

```
프레임워크가 맞지 않는 경우:

1. 특수한 요구사항
   - 프레임워크가 지원 안 하는 기능
   - 매우 특별한 구조 필요

2. 학습 목적
   - React 내부 동작 이해
   - 빌드 도구 학습
   - 라우팅 구현 학습

3. 프레임워크 제작
   - 자신만의 프레임워크 만들기
   - 회사 내부 프레임워크

4. 마이크로 프론트엔드
   - 여러 앱을 하나로 합치기
   - 레거시 통합
```

:::warning 주의!
처음부터 만들기는:
- ⏰ 시간이 매우 오래 걸림 (수개월)
- 😰 어려움
- 🐛 버그 많음
- 🔧 유지보수 힘듦

**초보자는 절대 비추!**

프레임워크 사용 → 익숙해지면 → 처음부터 만들기 시도
:::

### 처음부터 만들 때 필요한 것들

```
1. 번들러 (Bundler)
   └─ 여러 파일을 하나로 합치기
   └─ 선택: Vite, Webpack, Parcel, Rspack

2. 컴파일러 (Compiler)
   └─ JSX를 일반 JS로 변환
   └─ 선택: Babel, SWC

3. 린터 (Linter)
   └─ 코드 오류 검사
   └─ 선택: ESLint

4. 포매터 (Formatter)
   └─ 코드 스타일 통일
   └─ 선택: Prettier

5. 라우터 (Router)
   └─ 페이지 이동 관리
   └─ 선택: React Router, TanStack Router

6. 상태 관리 (State Management)
   └─ 전역 상태 관리
   └─ 선택: Zustand, Redux, Jotai

7. 데이터 페칭 (Data Fetching)
   └─ API 호출 관리
   └─ 선택: TanStack Query, SWR

8. 스타일링 (Styling)
   └─ CSS 관리
   └─ 선택: CSS Modules, Styled-components, Tailwind

9. 테스팅 (Testing)
   └─ 테스트 작성
   └─ 선택: Jest, Vitest, Testing Library

10. 빌드 설정 (Build Config)
    └─ 개발/프로덕션 환경 설정
    └─ 직접 설정
```

**각 선택이 중요합니다!**

```
프레임워크 사용:
- 위 10가지가 이미 설정됨
- 즉시 시작
- 시간: 1일

처음부터 만들기:
- 위 10가지를 직접 선택하고 설정
- 각각 설정 방법 학습 필요
- 시간: 1-3개월
```

### Q1: 프레임워크를 꼭 써야 하나요?

```
A: 실전 프로젝트라면 YES!

이유:

- 검증된 코드
- 보안 업데이트

- 채용 시 유리

예외:
- 학습 목적 (처음부터 만들어보기)
- 매우 특수한 요구사항
- 프레임워크 제작자
```

### Q2: Next.js vs Vite, 뭘 배워야 하나요?

```
A: 둘 다 배우세요! (순서대로)

추천 학습 경로:

1단계: Vite로 React 기초 (1-2주)
   - 컴포넌트, Props, State
   - Hooks
   - 라우팅 (React Router)

2단계: Next.js로 전환 (1-2주)
   - 파일 기반 라우팅
   - SSR, SSG
   - API 라우트

3단계: 프로젝트 경험 (계속)
   - 블로그 만들기
   - 포트폴리오 만들기
   - 쇼핑몰 만들기
```

### Q3: TypeScript도 배워야 하나요?

```
A: React 익숙해진 후에!

학습 순서:
1. JavaScript 기초 ✅
2. React 기초 ✅
3. 프레임워크 (Next.js/Vite) ✅
4. 작은 프로젝트 완성 ✅
5. TypeScript 시작 ← 여기!

TypeScript 장점:
✅ 버그 사전 방지
✅ 자동완성 향상
✅ 리팩토링 쉬움
✅ 큰 프로젝트에 필수

TypeScript 단점:
❌ 초반 학습 곡선
❌ 코드 작성 시간 증가
```

A: 프레임워크마다 다릅니다!

가장 쉬운 방법:
  Next.js → Vercel (추천!)
```

### Q5: Create React App은 어떻게 되나요?

```
A: 사용하지 마세요!

이유:
- Deprecated (지원 중단)
- 느림
- 무거움
- 업데이트 없음

대신 사용:
✅ Next.js (풀스택)
✅ Vite (SPA)
✅ Gatsby (정적 사이트)

기존 CRA 프로젝트:
→ Vite로 마이그레이션
→ Next.js로 마이그레이션
```

### Q6: 서버가 없어도 되나요?

```
A: Yes! 대부분의 경우 서버 불필요!

서버 없이 배포:
1. CSR 모드
   - Vercel, Netlify 무료 호스팅
   - 정적 파일만 업로드

2. SSG 모드
   - 빌드 시 HTML 생성
   - 정적 파일 호스팅

3. 무료 호스팅 서비스:
   - Vercel (추천!)
   - Netlify
   - GitHub Pages
   - Cloudflare Pages

서버 필요한 경우:
- SSR (모든 페이지)
- 실시간 기능 (채팅 등)
- 데이터베이스 직접 연결
- 민감한 API 키 사용

하지만 이것도:
- Vercel의 Serverless Functions
- Netlify Functions
- 로 해결 가능! (무료!)
```

### Q7: 프레임워크 전환이 어렵나요?

```
A: 생각보다 쉽습니다!

React 기초를 안다면:
- 컴포넌트 코드는 거의 같음
- 프레임워크는 "추가 기능"만 제공
- 핵심 React 지식은 동일

예시:
```jsx
// 이 코드는 Vite, Next.js, Remix 모두 동일!
function Button({ children, onClick }) {
  return (
    <button onClick={onClick}>
      {children}
    </button>
  );
}
```

차이점:
- 라우팅 방식
- 데이터 가져오기
- 설정 파일

→ 문서 보면 금방 익숙해짐!
```

배포:

- Expo: 앱 등록 무료

- 상업용 대규모 서비스

하지만 대부분의 개인 프로젝트:
→ 100% 무료!
